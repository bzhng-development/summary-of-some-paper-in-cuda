## 1. Executive Summary

This paper introduces Agentic Reinforced Policy Optimization (ARPO), a novel reinforcement learning algorithm that addresses the limitation of current trajectory-level methods in balancing long-horizon reasoning with multi-turn tool interactions for LLM-based agents. By leveraging an entropy-based adaptive rollout mechanism and advantage attribution estimation, ARPO dynamically explores high-uncertainty steps following tool usage, achieving superior performance across 13 benchmarks including GAIA and AIME2024 on backbones like Qwen2.5 and Qwen3. Notably, ARPO delivers these improvements using only half the tool-use budget required by existing methods like GRPO, offering a scalable and efficient solution for aligning agents with dynamic environments.

## 2. Context and Motivation

### The Core Problem: Multi-Turn Tool Interaction vs. Single-Turn Reasoning

This paper addresses a fundamental mismatch between how current reinforcement learning algorithms train LLM-based agents and how these agents actually operate in realistic scenarios. The specific gap is this: **existing RL methods treat agent reasoning as a single trajectory to be optimized end-to-end, but real-world agent tasks require dynamic, multi-turn interactions with external tools where uncertainty spikes unpredictably after each tool response**.

To understand why this matters, consider the difference between two scenarios:

1. **Single-turn reasoning** (what current RLVR methods excel at): The model receives a math problem, thinks internally, and produces an answer. The uncertainty distribution across tokens remains relatively stable throughout generation.

2. **Multi-turn tool interaction** (what real agents face): The model generates a search query, receives external feedback from a search engine, then must interpret that feedback and decide the next action. Critically, the paper observes that **token entropy rises sharply in the first 10–50 tokens following each tool call** (Section 2.2, Figure 2). This entropy spike indicates the model encounters genuine uncertainty when processing external information—uncertainty that trajectory-level RL methods fail to adequately explore.

The problem is not merely academic. As stated in Section 1:

> "Unlike the single-turn reasoning paradigm, multi-turn tool-use loops offer LLMs diverse and informative feedback in real-time. This characteristic underscores the necessity of discovering effective stepwise tool-use behaviors."

When an agent receives search results or code execution output, it faces a distributional shift between its internal reasoning and external feedback. Current methods treat this as just another token sequence, missing the opportunity to specifically explore the high-uncertainty decision points that determine whether the agent recovers from ambiguous tool responses or spirals into errors.

### Why This Problem Matters: Both Practical and Theoretical Significance

**Real-World Impact:** The benchmarks evaluated in this work span three critical domains (Section 4.1):

| Domain | Example Benchmarks | Real-World Analogue |
|--------|-------------------|---------------------|
| Mathematical Reasoning | AIME2024, MATH500, GSM8K | Scientific computation, financial modeling |
| Knowledge-Intensive Reasoning | HotpotQA, 2WikiMultihopQA, Musique | Research assistance, fact-checking |
| Deep Search | GAIA, HLE, xbench-DeepSearch | Open-ended information gathering, competitive intelligence |

In deep search scenarios especially, agents must navigate ambiguous information, verify sources, and synthesize findings across multiple tool calls. The paper notes that even advanced models like GPT-4o and DeepSeek-R1-671B achieve only 2% and 8.6% respectively on the Humanity's Last Exam (HLE) benchmark (Section 4.5, Table 2). This performance ceiling directly reflects the inability of current training methods to teach agents how to handle uncertainty during multi-turn interactions.

**Theoretical Significance:** The entropy observations challenge a core assumption in trajectory-level RL: that sampling complete trajectories and comparing their final rewards provides sufficient signal for learning step-level behaviors. Section 2.2 demonstrates that **search feedback introduces more uncertainty than Python feedback** because search engines return informative textual content while Python outputs consist of deterministic numbers. This means different tool types create different uncertainty profiles—something a uniform trajectory-level approach cannot adapt to.

### Prior Approaches and Their Limitations

The paper identifies three categories of prior work, each with specific shortcomings:

**1. Reinforcement Learning with Verifiable Rewards (RLVR)**

RLVR methods like those from OpenAI o1, DeepSeek-R1, and QwQ have demonstrated strong performance on single-turn reasoning tasks (Section 1, Section 5). These approaches:
- Sample complete trajectories from the policy
- Compute rewards based on final answer correctness
- Apply policy gradient updates across all tokens uniformly

*Limitation:* RLVR was designed for closed-ended reasoning where the model never interacts with external systems. The reward signal arrives only at trajectory end, providing no guidance on which intermediate decisions (particularly after tool calls) were beneficial or harmful.

**2. Trajectory-Level Agentic RL Algorithms**

Methods like GRPO (Shao et al., 2024), DAPO (Yu et al., 2025), and REINFORCE++ (Hu, 2025) extend RLVR to tool-using agents (Section 1, Section 4.2). These approaches:
- Sample multiple complete tool-use trajectories using predefined special tokens
- Provide reward signals based on final output
- May incorporate reward shaping to address tool overuse

*Limitation:* As the paper states in Section 1:

> "These optimizations often overlook a crucial aspect of training LLM-based agents: the multi-turn interaction loops between the LLM and the tool environment."

Trajectory-level methods compare complete rollouts but **neglect exploration of fine-grained behavior at each tool-use step**. When entropy spikes after a tool call, these methods continue sampling uniformly rather than allocating additional exploration budget to the high-uncertainty region where better behaviors might be discovered.

**3. Reward Function Engineering**

Several studies attempted to design elegant reward functions to better align tool-use behavior, addressing issues like tool overuse and sparse rewards (Section 1, citing Wang et al., 2025b; Sha et al., 2025; Bai et al., 2025).

*Limitation:* Reward engineering addresses *what* to optimize but not *how* to explore the action space effectively. Even with perfect reward signals, if the sampling mechanism doesn't allocate exploration budget to high-uncertainty steps, the policy cannot discover superior step-level behaviors.

### How ARPO Positions Itself Relative to Existing Work

ARPO positions itself as a **step-level agentic RL algorithm** that fundamentally changes the sampling and credit assignment mechanisms, not just the reward function. The key distinctions are:

| Aspect | Trajectory-Level RL (GRPO, DAPO) | ARPO (This Work) |
|--------|----------------------------------|------------------|
| Sampling | Uniform trajectory sampling | Entropy-based adaptive branching |
| Exploration | Global trajectory comparison | Step-level branching at high-entropy tool calls |
| Advantage Assignment | Per-trajectory advantage | Advantage attribution distinguishing shared vs. individual tokens |
| Tool Efficiency | Full tool budget per trajectory | ~50% tool budget with comparable/better performance |

The paper's positioning is explicit in Section 1:

> "To bridge this gap, we propose Agentic Reinforced Policy Optimization (ARPO), an agentic RL algorithm tailored for training multi-turn LLM-based agent. The core principle of ARPO is to encourage the policy model to adaptively branch sampling during high-entropy tool-call rounds, thereby efficiently aligning step-level tool-use behaviors."

This is not an incremental improvement to reward shaping—it is a rethinking of the rollout mechanism itself. The entropy-based adaptive rollout (Section 3.1) monitors token entropy in real-time and triggers additional partial sampling when entropy changes exceed a threshold. The advantage attribution estimation (Section 3.2) then ensures that tokens on branched paths receive distinct advantage values while shared tokens receive averaged advantages, allowing the model to internalize which step-level decisions led to better outcomes.

Critically, the paper provides theoretical grounding for this approach through the Generalized Policy Gradient (GPG) Theorem (Section 3.3), showing that optimization can be conducted using macro actions (partial rollout segments) rather than only single-token actions. This theoretical foundation distinguishes ARPO from purely heuristic approaches and establishes it as a principled extension of policy gradient methods to the multi-turn agent setting.

The experimental results validate this positioning: ARPO achieves superior performance across 13 benchmarks while using **only half the tool-call budget** required by trajectory-level methods (Section 1, Figure 1 right; Section 4.6, Figure 7). This efficiency gain directly results from the entropy-guided exploration—rather than wasting tool calls on low-uncertainty steps, ARPO concentrates exploration where the model actually needs to learn.

## 3. Technical Approach

This is an algorithm design paper that proposes a new reinforcement learning training method for LLM-based agents. The core idea is to use token-level entropy measurements to dynamically decide when to explore alternative reasoning paths during multi-turn tool interactions, rather than sampling complete trajectories uniformly.

### 3.1 Reader orientation (approachable technical breakdown)

ARPO is a training algorithm that teaches language models to become better agents by selectively exploring uncertain decision points that occur after the model receives tool responses. The solution works by measuring how "unsure" the model becomes after each tool call, then branching into multiple alternative continuations only at those high-uncertainty moments, which allows the model to learn better step-level behaviors without wasting computation on steps where it is already confident.

### 3.2 Big-picture architecture (diagram in words)

The ARPO system consists of four major components that operate in sequence during the training rollout phase. First, the **Rollout Module** takes an input question and generates an initial set of N reasoning trajectories using the current policy model, while reserving budget for M − N additional partial samples. Second, the **Entropy Monitoring System** computes token-level entropy for the first k tokens after each tool response, comparing this against the initial entropy baseline to detect uncertainty spikes. Third, the **Adaptive Beaming Controller** uses the normalized entropy change to compute a branching probability, triggering additional partial rollouts when this probability exceeds a threshold τ. Fourth, the **Advantage Attribution Estimator** assigns different advantage values to tokens depending on whether they belong to shared reasoning segments or branched paths, enabling the policy update to distinguish which step-level decisions led to better outcomes. Information flows from the policy model through entropy calculation, to branching decisions, to reward computation, and finally to the GRPO-style policy update that modifies the model weights.

### 3.3 Roadmap for the deep dive

- We first explain the entropy calculation mechanism because understanding how uncertainty is measured is prerequisite to understanding why branching occurs at specific points.
- We then describe the adaptive rollout process in chronological order (initialization → monitoring → branching → termination) to show how exploration budget is dynamically allocated during a single training episode.
- We next detail the advantage attribution estimation because this component determines how the learning signal is assigned to tokens after adaptive sampling has produced trajectories with shared and divergent segments.
- We then present the hierarchical reward design to clarify what optimization objective the policy is actually trying to maximize.
- We conclude with the theoretical foundation (Generalized Policy Gradient Theorem) to establish why optimizing over macro-actions (partial rollout segments) is mathematically valid.

### 3.4 Detailed, sentence-based technical breakdown

#### 3.4.1 Token Entropy as an Uncertainty Signal

Before explaining how ARPO uses entropy to guide exploration, we must define what token entropy means in this context. Token entropy measures the uncertainty in the model's probability distribution over the vocabulary at a given generation step, not the uncertainty about any particular token choice. The paper computes token-level generation entropy at step t using the standard Shannon entropy formula:

$$H_t = -\sum_{j=1}^{V} p_{t,j} \log p_{t,j}$$

where $V$ is the vocabulary size, and $p_{t,j}$ represents the probability assigned to vocabulary token $j$ at step $t$. These probabilities are obtained by applying a softmax function to the model's pre-softmax logits $z_t \in \mathbb{R}^V$ scaled by a decoding temperature $\tau$:

$$p_t = \pi_\theta(\cdot \mid R_{&lt;t}, x; T) = \text{Softmax}\left(\frac{z_t}{\tau}\right)$$

Here, $R_{&lt;t}$ denotes the reasoning trajectory up to step $t$, $x$ is the input question, and $T$ represents the set of available tools. The key insight from the pilot experiments in Section 2.2 is that entropy rises sharply in the first 10–50 tokens following each tool call, indicating that external tool feedback introduces genuine uncertainty into the model's reasoning process. This entropy spike is larger for search engine feedback than for Python interpreter feedback because search results contain informative textual content while Python outputs consist of deterministic numbers. ARPO leverages this observation by treating high entropy after tool calls as a signal that additional exploration at that step could discover better behaviors.

#### 3.4.2 Entropy-Based Adaptive Rollout Mechanism

The adaptive rollout mechanism is the core innovation that distinguishes ARPO from trajectory-level RL methods like GRPO or DAPO. This mechanism operates in four sequential steps during each training iteration, and we describe each step with its associated configurations and hyperparameters.

**Step 1: Rollout Initialization**

Given a global rollout budget of $M$ trajectories, the language model first generates $N$ complete trajectories via standard trajectory-level sampling based on the input question $q$. The remaining $M - N$ trajectory budget is reserved for partial sampling that will be triggered adaptively during the rollout. For the experiments reported in this paper, the configuration is $M = 16$ and $N = 8$, meaning half the budget is reserved for adaptive branching (Appendix C.2). After generating these $N$ initial trajectories, the system computes the entropy of the first $k$ tokens in each trajectory using Equation 3 from Section 2.2, forming an initial entropy matrix denoted as $H_{\text{initial}} \in \mathbb{R}^{1 \times k}$. This baseline entropy represents the model's uncertainty before any tool interaction occurs.

**Step 2: Entropy Variation Monitoring**

After recording the initial entropy, the model performs agentic reasoning with tools as defined in Equation 2 of Section 2.1, which decomposes the rollout sampling into agentic reasoning segments interleaved with answer generation segments. To continuously monitor entropy dynamics following each tool invocation, the model generates $k$ additional tokens after concatenating the response from each tool call. For tool-call step $t$, the system computes a step-level entropy matrix denoted as $H_t \in \mathbb{R}^{1 \times k}$. The normalized change in entropy relative to the initial state is then quantified using:

$$\Delta H_t = \text{Normalize}(H_t - H_{\text{initial}})$$

where normalization means summing all the values of $\Delta H$ and dividing by the vocabulary size $V$. A positive $\Delta H$ indicates an increase in uncertainty after the tool-call step, whereas a negative value reflects a reduction in uncertainty. This normalized entropy change serves as the branching criterion for the next step.

**Step 3: Entropy-Based Adaptive Beaming**

To encourage adaptive exploration along tool-use paths that exhibit beneficial entropy variations, ARPO defines the partial sampling probability at tool-call step $t$ as:

$$P_t = \alpha + \beta \cdot \Delta H_t$$

where $\alpha$ is a base sampling probability and $\beta$ is a stability entropy value (Section 3.1, Equation 5). The model uses $P_t$ to determine its branching behavior through the following action rule:

$$\text{Action}(P_t) = \begin{cases} \text{Branch}(Z), & \text{if } P_t > \tau \\ \text{Continue}, & \text{otherwise} \end{cases}$$

When $P_t$ exceeds a predefined threshold $\tau$, the system initiates $\text{Branch}(Z)$, which forks $Z$ partial reasoning paths from the current node; otherwise, it continues along the current trajectory. For the experiments in this paper, the hyperparameters are set as $\alpha = 0.5$, $\beta = 0.2$, and $\tau = 0.5$ (Appendix C.2). This mechanism enables the model to adaptively allocate exploration resources to regions of the reasoning space where rising entropy indicates a higher potential for informative outcomes. Figure 4(a) in the paper illustrates this principle visually, showing how high-entropy tool-call steps trigger branching while low-entropy steps continue without additional sampling.

**Step 4: Termination**

The adaptive rollout process iterates until one of two conditions is satisfied. First, if the total number of forked paths $\hat{Z}$ reaches the partial sampling budget $M - N$, branching stops and sampling continues until a final answer is produced. Second, if all paths terminate before reaching $M - N$, the system supplements with $M - N - \hat{Z}$ additional trajectory-level samples to satisfy the global rollout budget condition. By leveraging this efficient rollout mechanism, ARPO facilitates uncertainty-aware exploration, allowing LLMs to more effectively identify step-level tool-calling behavior. The paper notes in Section 3.1 that assuming the global expansion size and the number of tokens per trajectory are $n$, ARPO reduces the computational complexity of each rollout from the trajectory-level RL's $O(n^2)$ to between $O(n \log n)$ and $O(n^2)$, neglecting the minor overhead from token-level entropy calculations.

#### 3.4.3 Advantage Attribution Estimation

The entropy-based adaptive rollout mechanism naturally produces trajectories containing both shared reasoning token segments and distinct beam paths, as illustrated in Figure 4(b) of the paper. This structure motivates a more principled agentic RL policy update strategy that distinguishes between tokens that appear on multiple trajectories versus tokens that appear only on branched paths. The paper considers two advantage assignment settings.

**Hard Advantage Estimation**

In the hard advantage setting, the system explicitly distinguishes the shared and individual parts of each trajectory at the advantage level. Given $d$ trajectories that share certain tokens while diverging in others, the advantage for individual tokens is computed using the normalized reward $R_i$:

$$\hat{A}_{i,t} = \frac{r_i - \text{mean}(\{R_i\}_{i=1}^G)}{\text{std}(\{R_i\}_{i=1}^G)}$$

For shared tokens, the system assigns the average advantage across all $d$ trajectories that contain the shared segment:

$$\hat{A}^{\text{shared}}_{i,t} = \frac{1}{d} \sum_{i=1}^{d} \hat{A}_{i,t}$$

This approach explicitly encodes the intuition that tokens appearing on multiple paths should receive credit proportional to the average performance of all paths containing them, while tokens on unique branches should receive credit based only on their specific trajectory's outcome.

**Soft Advantage Estimation**

The soft advantage setting provides an elegant alternative by integrating the distinction between shared and individual token segments latently during policy optimization through the GRPO objective. For each input question $x$, the Group Relative Policy Optimization (GRPO) enables the reference policy $\pi_{\text{ref}}$ to generate a set of responses $\{y_1, y_2, \ldots, y_G\}$ and optimizes the policy by maximizing:

$$\mathcal{J}_{\text{GRPO}}(\theta) = \mathbb{E}_{(q,a) \sim \mathcal{D}, \{y_i\}_{i=1}^G \sim \pi_{\theta_{\text{old}}}(\cdot|q)} \left[ \frac{1}{G} \sum_{i=1}^{G} \frac{1}{|y_i|} \sum_{t=1}^{|y_i|} \min\left(r_{i,t}(\theta) \hat{A}_{i,t}, \text{clip}(r_{i,t}(\theta), 1 - \epsilon, 1 + \epsilon) \hat{A}_{i,t}\right) - \beta D_{\text{KL}}(\pi_\theta \parallel \pi_{\text{ref}}) \right]$$

The key mechanism that enables soft advantage estimation is the importance sampling ratio $r_{i,t}(\theta)$:

$$r_{i,t}(\theta) = \frac{\pi_\theta(y_{i,t} \mid x, y_{i,&lt;t})}{\pi_{\text{ref}}(y_{i,t} \mid x, y_{i,&lt;t})}$$

This ratio has a critical property: when trajectories $y_i$ and $y_j$ undergo a partial rollout at token $t$, they share the same response prefix tokens (i.e., $y_{i,&lt;t} = y_{j,&lt;t}$), and consequently the shared prefix tokens in both trajectories are assigned the same importance weight $r_{i,t}(\theta)$. When trajectories diverge after branching, the importance ratios differ ($r_{i,t}(\theta) \neq r_{j,t}(\theta)$). In the GRPO formulation, the mathematical interpretation is that the policy update is guided by the average advantage of tokens within each group, which serves as the loss signal. Since shared tokens have identical $r_{i,t}(\theta)$, their advantage contributions are effectively aligned and closely approximate the advantage $\hat{A}^{\text{shared}}_{i,t}$ in a hard estimation setting.

The paper provides empirical comparison between hard and soft advantage estimation in Figure 5 of Section 3.2, showing that the soft setting achieves consistently higher rewards with greater stability during ARPO training. Consequently, ARPO defaults to using the soft setting for advantage estimation in all reported experiments.

#### 3.4.4 Hierarchical Reward Design

The reward function serves as the optimization objective that guides the policy model's behavior during training. ARPO follows the Tool-Star framework (Dong et al., 2025), considering both correctness and format rewards, along with a multi-tool collaboration reward mechanism. The overall reward $R$ is formally defined as:

$$R = \begin{cases} \max(\text{Acc.} + r_M, \text{Acc.}) & \text{If Format is Good \& Acc.} > 0 \\ 0 & \text{If Format is Good \& Acc.} = 0 \\ -1 & \text{Otherwise} \end{cases}$$

where the multi-tool collaboration reward $r_M$ is:

$$r_M = \begin{cases} 0.1 & \text{If } \exists (\text{&lt;search>} \text{ \& } \text{&lt;python>}) \\ 0 & \text{Otherwise} \end{cases}$$

This hierarchical design provides three levels of feedback. First, the model receives a negative reward of $-1$ if the output format is incorrect, which enforces proper tool invocation syntax. Second, if the format is correct but the answer accuracy is zero, the model receives zero reward, providing a neutral signal that distinguishes format compliance from task success. Third, if both format and accuracy are correct, the model receives the accuracy score plus an additional $0.1$ bonus if it used multiple tool types (both search and Python) during reasoning. This multi-tool bonus encourages the model to learn when different tools are complementary rather than relying on a single tool type. The detailed flowchart for the ARPO algorithm can be found in Algorithm 1 in Appendix E of the paper.

#### 3.4.5 Theoretical Foundation: Generalized Policy Gradient Theorem

The adaptive partial rollout mechanism involves branching at high-entropy tool-use steps, which segments the Transformer-based policy's output tokens into multiple segments rather than treating the entire trajectory as a single unit. To establish the theoretical validity of this approach, the paper derives the Generalized Policy Gradient (GPG) Theorem applicable to all Transformer-based policies.

The adaptive partial rollout mechanism dynamically segments the Transformer-based policy's output tokens $\langle OT_1, OT_2, \ldots, OT_{|\text{output}|} \rangle$ into $K$ segments. Each segment is defined as a macro action:

$$MA_i \triangleq \langle OT_m, OT_{m+1}, \ldots, OT_{m+n} \rangle$$

The corresponding macro states are defined as $MS_1 \triangleq \langle IT_1, IT_2, \ldots, IT_{|\text{input}|} \rangle$ for the initial state, and $MS_i \triangleq \langle MS_{i-1}, MA_{i-1} \rangle$ for subsequent states. This segmentation allows derivation of the GPG Theorem:

$$\nabla_\theta \mathcal{J}(\theta) = \mathbb{E}_{\tau \sim \pi_\theta} \left\{ \sum_{T=1}^{K} \left[ \nabla_\theta \log \pi_\theta(MA_T \mid MS_T) A_T(\tau) \right] \right\}$$

In this equation, $T$ represents the macro step (not individual token step), and $A_T(\tau)$ denotes the advantage of trajectory $\tau$ at macro step $T$. The GPG Theorem asserts that for any differentiable Transformer-based policy $\pi_\theta$ and any objective function $\mathcal{J}(\theta)$, optimization can be effectively conducted using macro actions (i.e., partial rollout segments) rather than only single-token actions. This generalization encompasses the traditional Policy Gradient Theorem as a specific instance where each macro action consists of exactly one token.

The proof relies on two key observations about Transformer-based policies. First, for Transformer architectures, the next state $s_{t+1}$ is deterministically determined by the current state $s_t$ and action $a_t$, so $P(s_{t+1} \mid s_t, a_t) = 1$. Second, the chain rule allows decomposition of the full trajectory probability into a product of macro-action probabilities:

$$\pi_\theta(MA \mid MS_1) = \prod_{T=1}^{K} \pi_\theta(MA_T \mid MS_T)$$

The formal proof is presented in Appendix D.2 of the paper, with key steps showing the transition from traditional policy gradient (summing over individual token steps $H$) to generalized policy gradient (summing over macro steps $K$). Consequently, ARPO as an advanced implementation of the GPG Theorem provides a robust theoretical foundation for optimizing over partial rollout segments rather than complete trajectories.

#### 3.4.6 Design Choices and Alternative Considerations

Several design choices in ARPO warrant explanation regarding why this approach was selected over alternatives.

**Why entropy-based branching instead of fixed-step branching?** The pilot experiments in Section 2.2 demonstrate that entropy spikes occur unpredictably after tool calls, with the magnitude varying by tool type (search vs. Python). A fixed-step branching schedule would waste computation on low-uncertainty steps while potentially missing critical high-uncertainty decision points. The entropy-based approach adapts to the actual uncertainty profile of each episode.

**Why soft advantage estimation instead of hard advantage estimation?** Figure 5 in Section 3.2 shows that the soft setting achieves consistently higher rewards with greater stability during training. The soft approach leverages the existing GRPO infrastructure while implicitly encoding the shared vs. individual token distinction through importance sampling ratios, avoiding the need for explicit advantage averaging logic.

**Why reserve half the rollout budget for adaptive sampling?** The configuration $M = 16$, $N = 8$ means 50% of the budget is reserved for adaptive branching. The scaling analysis in Section 4.7 (Figure 8, middle) shows that increasing the initial sampling size from 0 to 8 improves performance, but increasing to 16 (complete global sampling) causes a great performance decline. This confirms that balancing global and partial sampling proportions is critical—too little initial sampling provides insufficient baseline trajectories, while too much eliminates the adaptive exploration benefit.

**Why use normalized entropy change instead of raw entropy?** The normalization in Equation 4 (dividing by vocabulary size $V$) ensures that the entropy change signal is on a consistent scale regardless of vocabulary size or model architecture. This makes the threshold $\tau = 0.5$ portable across different model backbones without requiring per-model tuning.

**Why include a multi-tool collaboration bonus?** The $r_M = 0.1$ bonus for using both search and Python tools encourages the model to learn complementary tool usage rather than specializing in a single tool type. This is particularly important for deep search tasks where different tools provide different types of information (textual search results vs. computational verification).

## 4. Key Insights and Innovations

This section identifies the fundamental contributions that distinguish ARPO from prior agentic RL work. We distinguish between **incremental improvements** (refinements to existing methods) and **fundamental innovations** (conceptual shifts that enable new capabilities).

---

### 4.1 Fundamental Innovation: Step-Level Learning Paradigm vs. Trajectory-Level Optimization

**What Prior Work Does:** All existing agentic RL methods—GRPO, DAPO, REINFORCE++—treat agent training as a **trajectory-level optimization problem**. They sample complete reasoning paths, compute final rewards, and apply uniform policy updates across all tokens. As noted in Section 1, these approaches "independently sample complete tool-use trajectories using predefined special tokens and provide reward signals based on the final output."

**What ARPO Does Differently:** ARPO reframes agent training as a **step-level learning problem** where exploration budget is dynamically allocated to specific decision points based on real-time uncertainty signals. The key conceptual shift is recognizing that **not all tokens deserve equal exploration attention**—tokens following tool responses carry fundamentally different learning signal than tokens in stable reasoning segments.

**Why This Matters:**

| Aspect | Trajectory-Level (Prior) | Step-Level (ARPO) |
|--------|-------------------------|-------------------|
| Exploration Target | Complete trajectories | High-uncertainty decision points |
| Learning Granularity | Episode-wide credit | Token-level credit attribution |
| Budget Efficiency | Uniform allocation | Adaptive allocation based on entropy |
| Tool Interaction Modeling | Implicit (via special tokens) | Explicit (via entropy monitoring) |

This is not an incremental improvement to reward shaping or sampling size—it is a **rethinking of what constitutes a meaningful learning unit** in agentic RL. The paper's empirical results validate this: ARPO achieves superior performance while using only half the tool-call budget (Section 4.6, Figure 7), which would be impossible under a trajectory-level paradigm where every trajectory consumes the full tool budget.

**Significance:** This paradigm shift enables ARPO to scale more efficiently than prior methods. As agent tasks become more complex (more tool calls, longer horizons), trajectory-level methods face exponential growth in required samples. ARPO's step-level focus concentrates learning signal where it matters most, offering a path to scalable agent training.

---

### 4.2 Fundamental Innovation: Entropy as a Dynamic Exploration Signal

**What Prior Work Does:** Existing entropy-based RL studies (Wang et al., 2025c;d; Cheng et al., 2025; Zheng et al., 2025) use entropy primarily for **regularization** or **diversity promotion** during sampling. Some works identify high-entropy tokens as important for learning (Lin et al., 2024; Wang et al., 2025c), but none use entropy as a **real-time branching criterion** during rollout.

**What ARPO Does Differently:** ARPO treats entropy as a **dynamic exploration signal** that triggers additional sampling only when uncertainty exceeds a threshold. The key insight from Section 2.2 is that **entropy spikes predictably after tool calls** (first 10–50 tokens), and the magnitude varies by tool type (search > Python). ARPO leverages this pattern to allocate exploration budget adaptively rather than uniformly.

**Why This Matters:**

1. **Uncertainty-Aware Exploration:** Prior methods explore blindly—every trajectory gets the same sampling budget regardless of where uncertainty actually lies. ARPO explores strategically, concentrating samples where the model genuinely needs to learn.

2. **Tool-Type Adaptation:** The observation that search feedback introduces more uncertainty than Python feedback (Section 2.2) means different tools require different exploration strategies. ARPO's entropy-based mechanism automatically adapts to this without manual tuning per tool type.

3. **Real-Time Responsiveness:** Unlike static exploration schedules (e.g., fixed branching at every tool call), ARPO's entropy monitoring responds to the **actual uncertainty** encountered in each episode. Some tool responses may be unambiguous (low entropy → no branching), while others create genuine decision points (high entropy → branching).

**Significance:** This is the first work to demonstrate that **token entropy can serve as a reliable proxy for exploration value** in agentic settings. The scaling analysis in Section 4.7 (Figure 8, left) shows performance peaks at entropy value 0.4 and declines at 1.0, confirming that moderate entropy-guided exploration is optimal—too little ignores useful signals, too much reduces sampling diversity. This establishes entropy as a tunable exploration hyperparameter rather than just a regularization term.

---

### 4.3 Fundamental Innovation: Advantage Attribution for Branched Trajectories

**What Prior Work Does:** Standard policy gradient methods (including GRPO) assume all sampled trajectories are **independent**—each trajectory receives its own advantage estimate based on its final reward. When trajectories share prefix tokens (as occurs naturally in beam search or partial rollouts), prior methods either ignore this structure or treat shared tokens identically without explicit credit assignment logic.

**What ARPO Does Differently:** ARPO's advantage attribution estimation (Section 3.2) explicitly distinguishes between **shared tokens** (appearing on multiple trajectories) and **individual tokens** (appearing only on branched paths). The soft advantage setting leverages the GRPO importance sampling ratio to implicitly encode this distinction: shared prefixes have identical $r_{i,t}(\theta)$ values, causing their advantage contributions to align naturally.

**Why This Matters:**

1. **Correct Credit Assignment:** Without advantage attribution, a token that appears on both successful and failed trajectories would receive ambiguous learning signal. ARPO's mechanism ensures shared tokens receive averaged advantage (reflecting their presence on multiple paths), while branched tokens receive trajectory-specific advantage.

2. **Enables Adaptive Rollout:** The advantage attribution mechanism is what makes the entropy-based adaptive rollout mathematically sound. Without it, branching would produce trajectories with incompatible credit structures, destabilizing training.

3. **Empirical Superiority:** Figure 5 in Section 3.2 shows soft advantage estimation achieves **consistently higher rewards with greater stability** than hard advantage estimation. This validates that the implicit encoding through importance ratios is more effective than explicit advantage averaging.

**Significance:** This is the first work to address the **credit assignment problem specific to branched rollout structures** in LLM-based agent training. Prior agentic RL work assumes independent trajectories; ARPO recognizes that adaptive sampling creates dependent trajectory structures requiring specialized advantage computation.

---

### 4.4 Practical Innovation: 50% Tool-Use Budget Reduction

**What Prior Work Does:** Existing agentic RL methods consume the full tool budget on every sampled trajectory. For deep search tasks requiring frequent tool interactions, this creates substantial financial costs (API calls) and computational overhead (waiting for tool responses).

**What ARPO Does Differently:** ARPO achieves comparable or superior performance using **only half the tool-call budget** required by trajectory-level methods (Section 1, Figure 1 right; Section 4.6, Figure 7). This efficiency gain comes directly from the entropy-based adaptive rollout: instead of sampling full trajectories uniformly, ARPO branches only at high-entropy steps, reducing redundant tool calls on low-uncertainty paths.

**Why This Matters:**

| Metric | GRPO (Baseline) | ARPO | Improvement |
|--------|-----------------|------|-------------|
| Tool Calls (Training) | 100% | ~50% | 2× efficiency |
| GAIA Performance (Qwen3-8B) | 32.0% | 38.8% | +6.8% |
| HLE Performance (Qwen3-8B) | 7.8% | 10.0% | +2.2% |

*Data from Table 2, Section 4.5*

The efficiency gain is not just cost savings—it enables **larger-scale training** within the same budget constraints. A research team with limited API budget can train 2× more episodes with ARPO, or allocate saved budget to harder tasks requiring more tool calls per episode.

**Significance:** This is a **practical breakthrough** with immediate real-world implications. Prior agentic RL work focuses on accuracy improvements without addressing the cost barrier to deployment. ARPO demonstrates that algorithmic efficiency can reduce operational costs without sacrificing performance—a critical consideration for production agent systems.

---

### 4.5 Theoretical Innovation: Generalized Policy Gradient (GPG) Theorem

**What Prior Work Does:** Traditional policy gradient methods (Sutton et al., 1999) operate on **single-token actions**, treating each generated token as an independent decision point. This formulation assumes the action space is fixed at the vocabulary level and does not accommodate macro-actions (sequences of tokens treated as single decisions).

**What ARPO Does Differently:** ARPO derives the **Generalized Policy Gradient (GPG) Theorem** (Section 3.3), which proves that policy optimization can be conducted over **macro-actions** (partial rollout segments) rather than only single tokens. The theorem shows that the traditional Policy Gradient Theorem is a special case where each macro-action consists of exactly one token.

**Why This Matters:**

1. **Mathematical Legitimacy:** The GPG Theorem provides formal justification for ARPO's partial rollout mechanism. Without this proof, entropy-based branching could be dismissed as a heuristic without theoretical grounding.

2. **Flexibility:** The theorem allows macro-action segmentation of **arbitrary length**, enabling trajectory splitting based on special tokens, tool-call boundaries, or entropy thresholds. This flexibility is what makes ARPO adaptable to different agent architectures.

3. **Broader Applicability:** The GPG Theorem applies to **all Transformer-based policies**, not just ARPO. This means future agentic RL work can build on this theoretical foundation to design other macro-action optimization methods.

**Significance:** This is the first theoretical framework specifically designed for **multi-turn Transformer-based agent training**. Prior theoretical work on policy gradients assumes single-token actions or fixed-horizon MDPs. The GPG Theorem bridges this gap, establishing ARPO as a principled extension of policy gradient methods rather than an ad-hoc engineering solution.

---

### 4.6 Distinguishing Incremental vs. Fundamental Contributions

To clarify the novelty landscape, we categorize ARPO's contributions:

| Contribution | Type | Rationale |
|-------------|------|-----------|
| Step-level learning paradigm | **Fundamental** | Reframes what constitutes a learning unit in agentic RL |
| Entropy as exploration signal | **Fundamental** | First use of entropy as dynamic branching criterion (not just regularization) |
| Advantage attribution for branched trajectories | **Fundamental** | First solution to credit assignment in dependent trajectory structures |
| 50% tool-budget reduction | **Practical** | Empirical outcome of fundamental innovations, not a separate algorithmic contribution |
| GPG Theorem | **Theoretical** | First formal framework for macro-action optimization in Transformer policies |
| Hierarchical reward design | **Incremental** | Builds on Tool-Star (Dong et al., 2025); adds multi-tool bonus but follows existing reward engineering patterns |

The hierarchical reward design (Section 3.2) is the only **incremental** contribution—it extends existing reward engineering practices rather than introducing a new conceptual framework. All other contributions represent fundamental shifts in how agentic RL is formulated, executed, or theoretically grounded.

---

### 4.7 Summary of Key Innovations

ARPO's significance lies not in any single technique, but in the **coherent integration** of three fundamental innovations:

1. **Entropy-guided exploration** identifies *where* to explore (high-uncertainty steps after tool calls)
2. **Advantage attribution** determines *how* to assign credit (shared vs. individual tokens)
3. **GPG Theorem** establishes *why* this is mathematically valid (macro-action optimization)

Together, these enable the practical breakthrough of **50% tool-budget reduction** while improving performance across 13 benchmarks. This combination of theoretical grounding, algorithmic innovation, and practical efficiency distinguishes ARPO from prior agentic RL work that typically excels in only one dimension.

## 5. Experimental Analysis

This section examines how the authors validate ARPO's claims through empirical evaluation. We will walk through the experimental design, unpack the quantitative results with specific numbers, and critically assess whether the evidence supports the paper's central assertions about step-level learning and tool-use efficiency.

### 5.1 Evaluation Methodology

#### 5.1.1 Dataset Coverage Across Three Reasoning Domains

The authors evaluate ARPO across **13 benchmarks** spanning three distinct reasoning domains (Section 4.1). This multi-domain coverage is critical for establishing that ARPO's improvements are not task-specific artifacts but reflect genuine algorithmic advances.

**Mathematical Reasoning (5 benchmarks):**
- `AIME2024` and `AIME2025`: 30 challenging problems each from the American Invitational Mathematics Examination, covering algebra, geometry, number theory, and combinatorics
- `MATH500`: 500 high-difficulty problems at college level or above
- `MATH`: Multi-level dataset covering abstract algebra, calculus, and discrete mathematics
- `GSM8K`: Elementary school math problems requiring 2–8 solution steps

**Knowledge-Intensive Reasoning (5 benchmarks):**
- `WebWalker`: Web traversal tasks requiring multi-hop information gathering
- `HotpotQA`, `2WikiMultihopQA`, `Musique`: Wikipedia-based open-domain QA requiring multi-hop reasoning
- `Bamboogle`: Multi-hop question answering with diverse reasoning patterns

**Deep Search (4 benchmarks):**
- `GAIA` (General AI Assistant): 466 questions testing reasoning, web browsing, and tool use
- `HLE` (Humanity's Last Exam): Highly challenging interdisciplinary problems requiring abstract thinking
- `WebWalkerQA`: 680 question-answer pairs for web traversal evaluation
- `xbench-DeepSearch`: Deep search capability assessment with Chinese-language questions

The inclusion of `HLE` is particularly noteworthy—Section 4.5 reports that even `GPT-4o` and `DeepSeek-R1-671B` achieve only 2% and 8.6% respectively on this benchmark. This establishes a challenging ceiling that makes any improvement meaningful.

#### 5.1.2 Baseline Categories

The authors compare ARPO against three distinct baseline categories (Section 4.2):

| Category | Methods | Purpose |
|----------|---------|---------|
| Direct Reasoning | Qwen2.5, Llama3.1, Qwen3, QwQ, DeepSeek-R1, GPT-4o, o1-preview | Establish performance without RL training |
| Trajectory-Level RL | GRPO, DAPO, REINFORCE++ | Compare against standard agentic RL algorithms |
| LLM-Based Search Agents | RAG, Search-o1, WebThinker, ReAct | Compare against workflow-based agent systems |

This three-tier comparison structure is important: it isolates whether improvements come from (1) RL training itself, (2) ARPO's algorithmic innovations over other RL methods, or (3) ARPO's advantages over non-RL agent architectures.

#### 5.1.3 Training Configuration

The experimental setup follows a **cold-start SFT with RL paradigm** (Section 4.3, Appendix C):

**Supervised Fine-Tuning Phase:**
- Framework: LLaMAFactory with DeepSpeed ZeRO-3 and FlashAttention2
- Training data: 54K Tool-Star samples + 0.8K STILL dataset for mathematical reasoning enrichment
- Learning rate: $7 \times 10^{-6}$
- Batch size: 128, trained for 3 epochs
- Maximum input length: 4096 tokens

**RL Phase:**
- Deep Reasoning Tasks: 10K Tool-Star samples, 2 epochs, 8 NVIDIA H800 GPUs
- Deep Search Tasks: 1K mixed hard search samples, 5 epochs (due to limited data), 8–16 NVIDIA H800 GPUs
- Response length cap: 4096 tokens (Deep Reasoning), 8192 tokens (Deep Search)

**ARPO-Specific Hyperparameters (Appendix C.2):**
- Global rollout size $M = 16$
- Initial sampling size $N = 8$ (50% of budget reserved for adaptive branching)
- Entropy weight $\beta = 0.2$
- Base sampling probability $\alpha = 0.5$
- Branching threshold $\tau = 0.5$

**Evaluation Metrics (Section 4.4):**
- F1 scores for the four Knowledge-Intensive QA tasks
- LLM-as-Judge using `Qwen2.5-72B-instruct` for other tasks
- `pass@1` evaluation with temperature 0.6 and top-p 0.95
- Answers extracted from `\boxed{}` delimiters in model output

**Entropy Value ($\Delta H_t$):**
- Performance increases with rising entropy values, **peaking at 0.4**
- At entropy value 1.0, performance declines

**Initial Sampling Size ($N$):**
- Performance improves as $N$ increases from 0 to 8, **peaking at 8**

**Global Rollout Size ($M$):**
- Increasing $M$ enhances model performance
- This indicates "ARPO algorithm is scalable and can improve generalization performance with larger sizes"

**Critical Assessment:**

The hyperparameter analysis provides important practical guidance:
1. The entropy threshold is not "higher is better"—there is an optimal range around 0.4
2. The 50/50 split between global and partial sampling ($M=16, N=8$) is near-optimal
3. ARPO benefits from increased compute budget (larger $M$), suggesting it scales well with resources

However, all scaling experiments use Qwen2.5-7B. The paper does not show whether these optimal hyperparameters transfer to larger models (Qwen3-14B) or different domains (mathematical vs. deep search). This is a limitation for practitioners seeking to deploy ARPO on different backbones.

**Evidence Strength: Strong**

- **5.0 percentage point improvement** on deep search tasks (Table 2)

#### 5.4.2 Claim: ARPO Uses Half the Tool-Use Budget

**Evidence Strength: Moderate**

1. **No exact numbers:** The paper does not provide numerical tool-call counts in tables—only the visual representation in Figure 7.
2. **Single model:** The efficiency analysis is shown only for Qwen2.5-7B, not for other backbones.
3. **Training vs. inference:** The efficiency claim applies to training-time tool calls. It is unclear whether ARPO-trained models use fewer tool calls during inference.

**Assessment:** The claim is plausible given ARPO's adaptive branching mechanism, but the lack of exact numerical data limits reproducibility and verification.

#### 5.4.3 Claim: ARPO Generalizes Across Domains

**Evidence Strength: Strong**

The 13-benchmark evaluation spans three distinct domains (mathematical, knowledge-intensive, deep search), and ARPO shows improvements in all three. The consistency across domains suggests the entropy-based exploration mechanism captures a general property of tool-use uncertainty rather than domain-specific patterns.

**Caveat:** All benchmarks are English-language except `xbench-DeepSearch` (Chinese). The paper does not evaluate ARPO on non-English reasoning tasks beyond xbench, limiting claims about multilingual generalization.

#### 5.4.4 Claim: Step-Level Learning is the Key Innovation

**Evidence Strength: Moderate to Strong**

The paper's central thesis is that step-level learning (via entropy-based branching) drives ARPO's improvements. The evidence supporting this includes:

1. **Entropy observations (Section 2.2):** Token entropy rises sharply after tool calls, validating the premise that these are high-uncertainty decision points.
2. **Hyperparameter analysis (Figure 8):** Performance peaks at balanced global/partial sampling ratios, confirming that adaptive branching (not just more sampling) drives improvements.
3. **Ablation implications:** The browser agent ablation (Table 3) shows performance scales with information extraction capability, suggesting step-level information processing matters.

**Missing Evidence:** The paper does not include an ablation that removes the entropy-based branching while keeping other ARPO components. A direct comparison between "ARPO with entropy branching" vs. "ARPO with fixed-step branching" would more conclusively isolate the entropy mechanism's contribution.

#### 5.5.1 Training Data Scale

The deep search experiments use only **1K RL training samples** (Section 4.3). While this demonstrates sample efficiency, it raises questions:
- Would ARPO's advantages persist or widen with larger training sets (10K, 100K samples)?
- Is the 1K sample result partly due to careful sample selection ("mixed hard search samples") rather than algorithmic superiority?

- The paper does not quantify the entropy calculation overhead in wall-clock time
- The adaptive branching mechanism requires real-time entropy monitoring, which may introduce latency during training
- No comparison of training time or GPU hours between ARPO and baselines is provided

For practitioners, training time is often as important as sample efficiency. The lack of timing data limits practical deployment decisions.

#### 5.5.3 Failure Cases

The paper does not present systematic failure case analysis. Specific unanswered questions:
- On which types of tasks does ARPO underperform relative to baselines?
- Are there scenarios where entropy-based branching triggers incorrectly (branching on low-value steps or failing to branch on high-value steps)?
- Does ARPO exhibit any training instability or reward collapse patterns?

The case studies in Appendix F show successful examples but do not illustrate failure modes.

#### 5.5.4 Comparison to Non-RL Methods

Table 2 compares ARPO against workflow-based agents (RAG, Search-o1, WebThinker, ReAct). ARPO outperforms these methods, but:

- These comparisons use different model backbones in some cases (e.g., WebThinker results may use different base models)
- The workflow-based methods may not be optimized with the same training data or infrastructure
- It is unclear whether ARPO's advantages come from RL training itself or the step-level exploration mechanism

A more controlled comparison (same backbone, same data, RL vs. non-RL) would clarify the source of improvements.

## 6. Limitations and Trade-offs

While ARPO demonstrates compelling improvements over trajectory-level RL methods, a critical examination reveals several assumptions, constraints, and open questions that practitioners should consider before deployment. This section identifies limitations grounded in the paper's evidence, distinguishing between inherent algorithmic trade-offs and gaps in empirical validation.

---

### 6.1 Core Assumptions Underlying ARPO

ARPO's effectiveness depends on several assumptions that may not hold universally across all agent training scenarios.

#### 6.1.1 Entropy as a Reliable Proxy for Exploration Value

**Assumption:** Token entropy spikes after tool calls indicate decision points where additional exploration will discover better behaviors.

**Evidence from Paper:** Section 2.2 establishes that "entropy rises sharply in the first 10–50 tokens following each tool call," and Section 4.7 (Figure 8, left) shows performance peaks at entropy value 0.4 and declines at 1.0.

**Limitation:** This relationship is empirically observed but not theoretically guaranteed. The paper acknowledges the trade-off:

> "Over-reliance on entropy may reduce sampling diversity, confirming the necessity of balancing base sampling probabilities α with entropy in ARPO." (Section 4.7)

**When This Fails:**
- **Low-uncertainty tool responses:** If a tool returns deterministic output (e.g., a database query with exact match), entropy may remain low even though the step is critical for task success.
- **High-uncertainty but irrelevant steps:** Entropy may spike due to ambiguous phrasing in tool responses that don't actually affect the final answer, causing ARPO to waste exploration budget on non-critical decision points.
- **Domain-specific entropy profiles:** The paper observes that "search feedback introduces more uncertainty than Python feedback" (Section 2.2), but does not establish whether this pattern generalizes to other tool types (e.g., API calls, robotic actions, database queries).

**Practical Implication:** Practitioners deploying ARPO on novel tool environments should validate the entropy-uncertainty relationship before assuming the adaptive branching mechanism will function as intended.

#### 6.1.2 Availability of Capable Browser Agents

**Assumption:** External browser agents with sufficient information extraction capability are available for deep search tasks.

**Evidence from Paper:** Table 3 in Section 4.6 shows dramatic performance differences based on browser capability:

| Browser Setting | Qwen3-14B Average |
|-----------------|-------------------|
| Snippet only | 24.8% |
| Same-scale browser | 29.9% |
| QWQ-32B browser | **39.4%** |

**Limitation:** The paper states:

> "The capability of the external browser agent is highly correlated with the accuracy of the Deepsearch task and shows a clear upward trend as its scale increases." (Section 4.6)

This reveals a critical dependency: ARPO's reported deep search results (e.g., 43.7% on GAIA for Qwen3-14B) assume access to a QWQ-32B browser agent. Practitioners without access to such capable browsers should expect performance closer to the "snippet only" baseline (35.0% on GAIA for Qwen3-14B)—a **20% relative performance drop**.

**When This Fails:**
- **Resource-constrained deployments:** Teams without access to large-parameter browser models cannot achieve the reported state-of-the-art results.
- **Proprietary tool environments:** If browser agents must be trained in-house, the additional training cost and data requirements are not accounted for in ARPO's efficiency claims.

**Practical Implication:** The 50% tool-budget savings during ARPO training may be offset by the computational cost of running a larger browser agent during inference.

#### 6.1.3 Stable Entropy Thresholds Across Model Scales

**Assumption:** The entropy threshold $\tau = 0.5$ and related hyperparameters transfer across different model backbones without extensive retuning.

**Evidence from Paper:** All scaling experiments in Section 4.7 use Qwen2.5-7B. The paper does not show whether optimal hyperparameters for Qwen2.5-7B transfer to Qwen3-8B, Qwen3-14B, or Llama3.1-8B.

**Limitation:** The normalization in Equation 4 divides by vocabulary size $V$, which the paper claims "makes the threshold $\tau = 0.5$ portable across different model backbones without requiring per-model tuning" (Section 3.4.6). However, this claim is not empirically validated—the paper reports results using these hyperparameters but does not include an ablation showing hyperparameter sensitivity across different backbones.

**When This Fails:**
- **Models with different vocabulary distributions:** Models trained on different corpora may have different baseline entropy profiles, requiring threshold recalibration.
- **Multilingual models:** The paper evaluates primarily on English benchmarks except xbench-DeepSearch. Entropy distributions may differ significantly for non-Latin scripts or morphologically rich languages.

**Practical Implication:** Practitioners should budget time for hyperparameter tuning when deploying ARPO on models other than Qwen2.5-7B.

### 6.2 Unaddressed Scenarios and Edge Cases

The paper's evaluation covers 13 benchmarks across three domains, but several important scenarios remain unexplored.

#### 6.2.1 Long-Horizon Tasks with Many Tool Calls

**Gap:** The deepest search tasks in the evaluation (GAIA, HLE) involve multiple tool calls, but the paper does not analyze how ARPO scales when tool calls exceed 10–20 per episode.

**Evidence from Paper:** Section 4.3 notes that deep search tasks use 8192 token response length caps, but does not report average tool-call counts per episode. Figure 7 shows tool-call efficiency but does not stratify by episode length.

**Why This Matters:**
- **Cumulative entropy drift:** If entropy spikes occur at every tool call, the branching mechanism could trigger excessively on long trajectories, consuming the partial sampling budget early and leaving later decision points unexplored.
- **Budget exhaustion:** With $M - N = 8$ partial sampling slots, an episode with 15 tool calls could exhaust the branching budget before reaching critical late-stage decisions.

**Open Question:** Does ARPO's adaptive mechanism include any budget-aware allocation that prioritizes later tool calls, or does it branch greedily based on local entropy signals?

#### 6.2.2 Multi-Agent or Collaborative Settings

**Gap:** ARPO is evaluated on single-agent reasoning tasks. The paper does not address scenarios where multiple agents must coordinate tool usage or share information.

**Evidence from Paper:** The agentic RL formulation in Section 2.1 defines a single policy $\pi_\theta$ interacting with tools $T$. There is no discussion of multi-agent credit assignment or coordinated exploration.

**Why This Matters:**
- **Inter-agent entropy correlations:** In collaborative settings, one agent's tool call may reduce uncertainty for another agent, creating cross-agent entropy dependencies that ARPO's single-agent formulation cannot capture.
- **Shared tool budgets:** If multiple agents share a tool-call budget, the adaptive branching mechanism would need to coordinate across agents to avoid budget exhaustion.

**Open Question:** Can ARPO's entropy-based branching be extended to multi-agent settings, or does the theoretical foundation (GPG Theorem) require modification?

#### 6.2.3 Real-Time or Streaming Environments

**Gap:** All benchmarks use static datasets where tool responses are available immediately. The paper does not evaluate ARPO in environments where tool responses arrive asynchronously or with significant latency.

**Evidence from Paper:** Section 4.3 mentions using "a Python compiler within a sandbox environment" and "top-10 snippets from the Bing search engine," but does not discuss response latency or asynchronous tool execution.

**Why This Matters:**
- **Real-time entropy monitoring:** ARPO's adaptive rollout requires computing entropy after each tool response. If tool responses arrive asynchronously, the rollout mechanism must buffer and synchronize entropy calculations across multiple in-flight tool calls.
- **Training throughput:** High-latency tool calls could significantly slow training, potentially offsetting the tool-budget savings with increased wall-clock time.

**Open Question:** What is the actual training time comparison between ARPO and baseline methods when accounting for tool-call latency?

#### 6.2.4 Failure Mode Analysis

**Gap:** Appendix F provides case studies of successful ARPO executions but does not systematically analyze failure cases.

**Evidence from Paper:** The case studies in Tables 4–9 all show correct final answers. There is no discussion of episodes where ARPO underperformed relative to baselines or where the branching mechanism triggered incorrectly.

**Why This Matters:**
- **False positive branching:** If entropy spikes occur on non-critical steps, ARPO wastes exploration budget. The paper does not quantify how often this occurs.
- **False negative branching:** If critical decision points have low entropy, ARPO may fail to explore alternative behaviors that would improve performance.
- **Training instability:** The paper notes that soft advantage estimation achieves "greater stability" than hard estimation (Figure 5), but does not report whether ARPO exhibits any reward collapse or policy degradation patterns during training.

**Open Question:** On which specific task types or difficulty levels does ARPO underperform relative to trajectory-level methods?

### 6.3 Computational and Data Constraints

ARPO introduces several resource requirements that may limit deployment in resource-constrained settings.

#### 6.3.1 Entropy Calculation Overhead

**Constraint:** Token-level entropy must be computed in real-time during rollout for every tool-call step.

**Evidence from Paper:** Section 3.1 states that ARPO reduces computational complexity "from the trajectory-level RL's $O(n^2)$ to between $O(n \log n)$ and $O(n^2)$," but adds the caveat "neglecting the minor overhead from token-level entropy calculations."

**Limitation:** The paper does not quantify this overhead in:
- **Wall-clock time:** How many additional seconds per training episode?
- **GPU utilization:** Does entropy computation require additional GPU memory or compute cycles?
- **Scalability:** Does the overhead grow linearly with vocabulary size $V$, or is there a fixed cost per token?

**Practical Implication:** For teams training on tight GPU budgets, the unquantified entropy overhead could meaningfully impact training throughput. The claimed $O(n \log n)$ complexity advantage may be partially offset by entropy calculation costs.

#### 6.3.2 Training Data Requirements

**Constraint:** The deep search experiments use only 1K RL training samples, which demonstrates sample efficiency but raises questions about scalability.

**Evidence from Paper:** Section 4.3 states:

> "We use only 1K mixed hard search samples from SimpleDeepSearcher and WebSailor for training."

**Limitation:** The paper does not address:
- **Data selection bias:** The 1K samples are described as "mixed hard search samples," suggesting careful curation. It is unclear whether ARPO performs as well with randomly selected training data.
- **Scaling with data size:** Would ARPO's advantages persist, widen, or diminish with 10K or 100K training samples? The mathematical reasoning experiments use 10K samples, but the paper does not show a data scaling curve.
- **Domain transfer:** The 1K deep search samples come from specific datasets (SimpleDeepSearcher, WebSailor). It is unclear whether a model trained on these samples generalizes to other deep search domains without additional fine-tuning.

**Practical Implication:** Teams with access to large-scale tool-use datasets may not see the same sample efficiency benefits reported in the paper.

#### 6.3.3 Memory Requirements for Partial Rollouts

**Constraint:** The adaptive branching mechanism must maintain multiple partial trajectories simultaneously during rollout.

**Evidence from Paper:** Section 3.1 describes branching $Z$ partial reasoning paths from high-entropy nodes, with a total partial sampling budget of $M - N = 8$ trajectories.

**Limitation:** The paper does not discuss:
- **KV cache memory:** Maintaining multiple branched trajectories requires storing separate key-value caches for each path. For large models (e.g., Qwen3-14B), this could significantly increase GPU memory requirements.
- **Batching efficiency:** Partial rollouts may reduce batching efficiency compared to uniform trajectory sampling, as branched paths may have different lengths and completion times.

**Practical Implication:** Teams deploying ARPO on memory-constrained hardware may need to reduce the partial sampling budget ($M - N$), potentially diminishing the adaptive exploration benefits.

### 6.4 Unvalidated Claims and Open Questions

Several claims in the paper lack complete empirical validation, leaving open questions for future work.

#### 6.4.1 Tool-Efficiency Claim Lacks Numerical Precision

**Claim:** ARPO uses "only half the tool-call budget required by existing methods" (Section 1, Abstract).

**Evidence from Paper:** Figure 7 in Section 4.6 provides a visual comparison of tool-call efficiency between GRPO and ARPO on Qwen2.5-7B. However:
- **No exact numbers:** The figure shows relative bars but does not provide numerical tool-call counts in any table.
- **Single model:** The efficiency analysis is shown only for Qwen2.5-7B, not for Qwen3-8B, Qwen3-14B, or Llama3.1-8B.
- **Training vs. inference:** The claim applies to training-time tool calls. It is unclear whether ARPO-trained models use fewer tool calls during inference.

**Open Question:** Does the 50% tool-budget reduction persist across different model scales and task domains, or is it specific to the Qwen2.5-7B deep reasoning configuration?

#### 6.4.2 GPG Theorem Practical Impact Not Isolated

**Claim:** The Generalized Policy Gradient Theorem provides theoretical foundation for ARPO's macro-action optimization (Section 3.3).

**Evidence from Paper:** The GPG Theorem is proven in Appendix D.2, showing that policy optimization can be conducted over macro-actions rather than single tokens.

**Limitation:** The paper does not include an ablation comparing:
- **ARPO with GPG-based updates** vs. **ARPO with standard policy gradient updates**
- **Macro-action segmentation** vs. **Single-token actions** with the same entropy-based branching

**Open Question:** How much of ARPO's improvement comes from the GPG Theorem's macro-action formulation versus the entropy-based adaptive rollout mechanism? The two components are tightly coupled in the implementation, making it difficult to isolate their individual contributions.

#### 6.4.3 Long-Term Training Stability Not Assessed

**Claim:** ARPO achieves "consistently higher rewards with greater stability" than hard advantage estimation (Figure 5, Section 3.2).

**Evidence from Paper:** Figure 5 shows reward curves during ARPO training, comparing hard and soft advantage estimation over the training period reported in the experiments.

**Limitation:** The paper does not address:
- **Extended training:** Do ARPO's advantages persist over 10+ epochs, or does performance plateau or degrade?
- **Reward collapse:** Some RLVR methods exhibit reward hacking where the model optimizes for reward signals without genuine capability improvement. The paper does not report whether ARPO exhibits this behavior.
- **Catastrophic forgetting:** When fine-tuning a pretrained model with ARPO, does the model retain its general capabilities, or does it overfit to the tool-use training distribution?

**Open Question:** What is the long-term training dynamics profile of ARPO, and are there any failure modes that emerge only after extended training?

#### 6.4.4 Comparison to Non-RL Methods Lacks Controls

**Claim:** ARPO outperforms workflow-based agents like RAG, Search-o1, and WebThinker (Table 2, Section 4.5).

**Evidence from Paper:** Table 2 compares ARPO-trained models against various baseline methods on deep search benchmarks.

**Limitation:** The comparison lacks controls for:
- **Same backbone:** Some baseline results may use different base models than the ARPO experiments.
- **Same training data:** Workflow-based methods may not have been trained on the same 1K deep search samples.
- **Same evaluation protocol:** The paper notes that xbench-DR results use Chinese prompts for Pass@K analysis, "leading to a performance improvement compared to Table 2" (footnote 4, Section 4.6).

**Open Question:** How much of ARPO's advantage over non-RL methods comes from the RL training itself versus the step-level exploration mechanism? A controlled comparison (same backbone, same data, RL vs. non-RL) would clarify this.

We distinguish between immediate implications (what practitioners can act on now) and longer-term research opportunities (what the community should explore next).

### 7.1 Landscape Impact: How ARPO Changes Agentic RL

ARPO represents a **paradigm shift** in how the field conceptualizes agent training. Rather than treating this as an incremental algorithmic improvement, we identify three fundamental changes ARPO introduces to the agentic RL landscape.

#### 7.1.1 From Trajectory-Centric to Step-Centric Learning

**Before ARPO:** The dominant paradigm treated complete trajectories as the atomic learning unit. Methods like GRPO, DAPO, and REINFORCE++ sample full episodes, compute final rewards, and apply uniform policy updates across all tokens (Section 1, Section 4.2). This approach assumes uncertainty is distributed evenly throughout generation.

**After ARPO:** The paper demonstrates that **uncertainty concentrates at specific decision points**—particularly the 10–50 tokens following tool responses (Section 2.2, Figure 2). By treating these high-entropy steps as distinct learning units, ARPO enables:

| Dimension | Trajectory-Level Paradigm | Step-Level Paradigm (ARPO) |
|-----------|--------------------------|---------------------------|
| Exploration Budget | Uniform per trajectory | Adaptive per decision point |
| Credit Assignment | Episode-wide advantage | Token-level attribution |
| Sampling Efficiency | Fixed trajectories | Dynamic branching |
| Tool Efficiency | 100% budget per episode | ~50% budget (Section 4.6, Figure 7) |

**Why This Matters:** This shift enables **non-linear scaling** of agent capabilities. As tasks become more complex (more tool calls, longer horizons), trajectory-level methods face exponential sample requirements. ARPO's step-level focus concentrates learning signal where it matters most, offering a path to scalable agent training that was previously unavailable.

#### 7.1.2 Entropy as a First-Class Exploration Signal

**Before ARPO:** Entropy-based RL studies used entropy primarily for **regularization** or **diversity promotion** (Wang et al., 2025c;d; Cheng et al., 2025; Zheng et al., 2025, cited in Section 2.2). Some works identified high-entropy tokens as important for learning (Lin et al., 2024), but none used entropy as a **real-time branching criterion**.

**After ARPO:** The paper establishes entropy as a **dynamic exploration trigger** with empirically validated thresholds:

- Performance peaks at entropy value **0.4** (Section 4.7, Figure 8 left)
- Performance declines at entropy value **1.0** (over-reliance reduces diversity)
- Threshold $\tau = 0.5$ provides portable branching criterion across models (Section 3.4.6)

**Why This Matters:** This transforms entropy from a passive regularization term into an **active decision mechanism**. Practitioners can now tune exploration behavior through entropy hyperparameters rather than relying on fixed sampling schedules. The scaling analysis in Section 4.7 provides concrete guidance: entropy weight $\beta = 0.2$ and base probability $\alpha = 0.5$ serve as reasonable starting points.

#### 7.1.3 Theoretical Foundation for Macro-Action Optimization

**Before ARPO:** Traditional Policy Gradient Theorem (Sutton et al., 1999) operates on **single-token actions**, treating each generated token as an independent decision point. This formulation does not accommodate macro-actions (sequences of tokens treated as single decisions).

**After ARPO:** The Generalized Policy Gradient (GPG) Theorem (Section 3.3, Appendix D.2) proves that policy optimization can be conducted over **macro-actions** (partial rollout segments) rather than only single tokens. The theorem shows:

$$\nabla_\theta \mathcal{J}(\theta) = \mathbb{E}_{\tau \sim \pi_\theta} \left\{ \sum_{T=1}^{K} \left[ \nabla_\theta \log \pi_\theta(MA_T \mid MS_T) A_T(\tau) \right] \right\}$$

where $T$ represents macro steps (not individual token steps) and $MA_T$ represents macro actions (partial rollout segments).

**Why This Matters:** This provides **mathematical legitimacy** for a broad class of future agentic RL methods. Any approach that segments trajectories by tool-call boundaries, special tokens, or uncertainty thresholds can now build on the GPG Theorem rather than treating such segmentation as heuristic engineering. This elevates ARPO from an ad-hoc solution to a **principled framework** for multi-turn agent training.

### 7.2 Follow-Up Research Directions

ARPO opens numerous research questions that the community should explore. We categorize these by timeline and feasibility.

#### 7.2.1 Immediate Research Opportunities (6–12 months)

**1. Entropy-Uncertainty Validation Across Tool Types**

**Question:** Does the entropy-uncertainty relationship observed for search and Python tools (Section 2.2) generalize to other tool types?

**Why This Matters:** ARPO's effectiveness depends on entropy reliably indicating exploration value. The paper notes that "search feedback introduces more uncertainty than Python feedback" because search returns textual content while Python returns deterministic numbers. But what about:
- API calls with structured JSON responses?
- Database queries with exact-match results?
- Robotic action execution with sensor feedback?
- Human-in-the-loop interactions?

**Suggested Approach:** Replicate the entropy analysis from Section 2.2 (Figure 2) across diverse tool environments. Measure whether entropy spikes consistently occur in the first 10–50 tokens after tool responses, and whether the magnitude correlates with task difficulty.

**2. Hyperparameter Transfer Across Model Scales**

**Question:** Do ARPO's optimal hyperparameters ($\alpha = 0.5$, $\beta = 0.2$, $\tau = 0.5$, $M = 16$, $N = 8$) transfer to models beyond Qwen2.5-7B?

**Why This Matters:** All scaling experiments in Section 4.7 use Qwen2.5-7B. The paper claims normalization by vocabulary size makes $\tau = 0.5$ "portable across different model backbones" (Section 3.4.6), but this is not empirically validated. Practitioners deploying ARPO on Qwen3-14B, Llama3.1-70B, or other architectures need guidance on whether retuning is necessary.

**Suggested Approach:** Conduct hyperparameter sweeps on at least two additional model scales (e.g., 3B and 70B) to establish whether optimal values shift with model size. Report whether the 50/50 global-to-partial sampling ratio ($M = 16$, $N = 8$) remains optimal.

**3. Long-Horizon Task Scaling**

**Question:** How does ARPO perform on tasks requiring 10–20+ tool calls per episode?

**Why This Matters:** The deepest search tasks in the evaluation (GAIA, HLE) involve multiple tool calls, but the paper does not report average tool-call counts per episode (Section 6.2.1). With a partial sampling budget of $M - N = 8$ trajectories, an episode with 15 tool calls could exhaust the branching budget before reaching critical late-stage decisions.

**Suggested Approach:** Stratify results by episode length (number of tool calls). Analyze whether branching occurs uniformly across tool calls or concentrates on early calls. Consider budget-aware allocation strategies that reserve branching capacity for later decision points.

#### 7.2.2 Medium-Term Research Opportunities (12–24 months)

**4. Multi-Agent Extensions of ARPO**

**Question:** Can ARPO's entropy-based branching be extended to multi-agent settings where agents coordinate tool usage?

**Why This Matters:** The agentic RL formulation in Section 2.1 defines a single policy $\pi_\theta$ interacting with tools $T$. Real-world scenarios often involve multiple agents sharing information or coordinating actions (e.g., research teams, distributed sensing networks). In such settings:
- One agent's tool call may reduce uncertainty for another agent
- Cross-agent entropy dependencies could inform coordinated exploration
- Shared tool budgets would require inter-agent coordination

**Suggested Approach:** Extend the GPG Theorem to multi-agent settings. Define cross-agent entropy signals that trigger coordinated branching. Evaluate on collaborative benchmarks (e.g., multi-agent QA, distributed information gathering).

**5. Alternative Uncertainty Signals Beyond Entropy**

**Question:** Are there uncertainty signals that complement or outperform token entropy for triggering exploration?

**Why This Matters:** Entropy measures uncertainty in the token generation distribution, but other signals might better capture decision-point importance:
- **Gradient magnitude:** Large gradients may indicate high-impact decisions
- **Attention variance:** High variance in attention weights may signal ambiguous context
- **Self-consistency disagreement:** Multiple samples producing different answers indicates uncertainty
- **Tool response complexity:** Longer or more structured tool responses may warrant more exploration

**Suggested Approach:** Compare entropy against alternative signals in controlled ablation studies. Consider hybrid approaches that combine multiple signals (e.g., entropy + attention variance).

**6. Inference-Time Efficiency of ARPO-Trained Models**

**Question:** Do models trained with ARPO use fewer tool calls during inference, or does the efficiency gain apply only to training?

**Why This Matters:** The 50% tool-budget reduction claim (Section 4.6, Figure 7) applies to **training-time** tool calls. For production deployments, inference-time tool costs matter more. If ARPO-trained models learn more efficient tool-use behaviors, they might reduce inference costs as well.

**Suggested Approach:** Compare inference-time tool-call counts between ARPO-trained and GRPO-trained models on the same benchmarks. Measure whether ARPO models achieve comparable accuracy with fewer tool calls during evaluation.

#### 7.2.3 Long-Term Research Opportunities (24+ months)

**7. Theoretical Analysis of Entropy-Guided Exploration**

**Question:** Under what conditions does entropy-guided exploration provably improve sample efficiency?

**Why This Matters:** The GPG Theorem (Section 3.3) establishes mathematical validity for macro-action optimization, but does not prove that entropy-based branching improves sample efficiency. A theoretical analysis could establish:
- Conditions under which entropy correlates with exploration value
- Bounds on sample efficiency gains from adaptive vs. uniform sampling
- Relationships between entropy thresholds and optimal exploration budgets

**Suggested Approach:** Develop formal analysis connecting entropy dynamics to regret bounds in multi-turn MDPs. Leverage existing literature on entropy-regularized RL and adaptive exploration.

**8. ARPO for Non-Text Modalities**

**Question:** Can ARPO's principles extend to multimodal agents (vision, audio, robotics)?

**Why This Matters:** The paper focuses on text-based tool interactions (search, Python, browser). But many agent applications involve:
- Vision-language agents interpreting images
- Robotics agents executing physical actions
- Audio agents processing speech

In these domains, "token entropy" may not be the appropriate uncertainty signal. Alternative formulations (e.g., action-space entropy for robotics, visual attention entropy for vision) may be needed.

**Suggested Approach:** Generalize the entropy-based adaptive rollout to non-text action spaces. Define appropriate uncertainty signals for each modality. Evaluate on multimodal benchmarks.
