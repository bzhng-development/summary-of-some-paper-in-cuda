## 1. Executive Summary
This paper tackles the critical bottleneck of running large language models (LLMs) on devices with limited DRAM by storing model parameters in flash memory and loading them on-demand during inference. The authors introduce a hardware-aware framework featuring "windowing" to reuse activated neurons and "row-column bundling" to maximize flash read throughput, leveraging activation sparsity via a low-rank predictor. These techniques enable running models like **OPT 6.7B** and **Falcon 7B** up to **2x the size of available DRAM**, achieving inference speedups of up to **4x on CPU** and **20x on GPU** compared to naive loading approaches.

## 2. Context and Motivation

To understand the significance of this work, we must first establish the hardware constraints facing modern Large Language Model (LLM) deployment and why standard optimization techniques fail to solve them on personal devices. This section dissects the memory bottleneck, the physical characteristics of storage hardware, and the specific gaps in prior research that this paper aims to fill.

### 2.1 The Memory Capacity Gap
The primary challenge addressed is the mismatch between model size and available **DRAM** (Dynamic Random Access Memory). DRAM is the high-speed memory used by CPUs and GPUs to store active model weights during inference.

*   **Model Size vs. Device Capacity:** A standard 7 billion parameter model (e.g., Llama 2-7B or OPT-6.7B) requires over `$14\text{GB}$` of memory just to store parameters in half-precision floating point format (Section 1).
*   **Device Limitations:** Most personal devices, such as smartphones or standard laptops, possess less DRAM than this threshold. For instance, the paper notes that `$14\text{GB}$` exceeds the capabilities of most personal devices (Section 1).
*   **The Standard Assumption:** Conventional inference engines assume the entire model can reside in DRAM (Section 1). When this assumption fails, inference becomes impossible without offloading or compression.

While techniques like **quantization** (reducing numerical precision) can shrink model size, the authors argue this "cannot address the main limitation of loading the entire model into DRAM" (Section 1). Even compressed, large models may still exceed the strict memory budgets of edge devices, necessitating a fundamentally different storage strategy.

### 2.2 The Flash Memory Opportunity and Constraint
The proposed solution shifts model storage from DRAM to **flash memory** (e.g., SSDs), which offers significantly higher capacity at lower cost. However, this substitution introduces severe performance penalties due to physical hardware characteristics.

*   **Capacity vs. Bandwidth:** Figure 2a illustrates the trade-off. Flash memory offers capacity an order of magnitude larger than DRAM (e.g., `$\sim 100\text{GB}$` vs. `$\sim 10\text{GB}$` in the diagram), but suffers from much lower bandwidth. The diagram labels flash bandwidth at `$\sim 1\text{GB/s}$` compared to `$\sim 100\text{GB/s}$` for CPU/GPU interfaces (Figure 2a).
*   **Sequential vs. Random Access:** Flash memory is not uniformly slow. It performs optimally during large, sequential reads. Benchmarks on an Apple MacBook M1 Max show speeds exceeding `$6\text{GiB/s}$` for a `$1\text{GiB}$` linear read (Section 2.2).
*   **The Random Read Penalty:** Performance degrades drastically for smaller, random reads due to "latency to first byte." This latency includes overhead from the operating system, drivers, and flash controllers. Figure 2b demonstrates that random read throughput increases with chunk size and thread count. Small chunks incur high latency penalties, reducing overall throughput.

> "For smaller blocks, a substantial part of the overall read time is spent waiting for data transfer to begin. This is often referred to as latency to first byte." (Section 2.2)

This hardware reality dictates the algorithmic design: simply loading weights on-demand is insufficient if the access pattern is random and the chunks are small. The system must minimize data volume *and* maximize chunk size to mitigate the bandwidth gap.

### 2.3 Limitations of Prior Approaches
Existing literature generally falls into two categories, both of which leave a gap for devices with strictly limited DRAM.

*   **Model Compression:** Techniques like pruning and quantization reduce model size but often still require the compressed model to fit entirely in memory during inference (Section 6).
*   **Selective Execution:** Methods like **Dejavu** (Liu et al., 2023b) exploit activation sparsity to load subsets of weights. However, Dejavu still requires loading from GPU memory, assuming the weights reside there initially (Section 6).
*   **Offloading Systems:** Systems like **FlexGen** (Sheng et al., 2023) offload weights and KV-cache from GPU memory to DRAM, and DRAM to flash. However, the authors note that FlexGen is "theoretically bound by the slow throughput of flash to DRAM" in scenarios where the full model cannot reside in DRAM (Section 6).

The critical gap identified is the assumption that the model can fully reside in GPU memory or system DRAM at some stage. As stated in Section 6: "Overall, the primary assumption in the literature is that the model can fully reside in the GPU memory or system DRAM. However, considering the limited resources available on personal devices, we do not share this assumption in this work."

### 2.4 Positioning of This Work
This paper positions itself at the intersection of hardware-aware systems design and sparse machine learning. It differs from prior work in three key dimensions:

1.  **Storage Hierarchy:** Unlike Dejavu or standard inference, the full model weights are stored in flash, not DRAM. DRAM is treated as a limited cache (Section 1).
2.  **Hardware-Informed Optimization:** The algorithm is explicitly designed around flash physics. It prioritizes large contiguous reads (to maximize throughput per Figure 2b) and minimizes total volume (to reduce latency per Figure 2a).
3.  **Edge-First Constraint:** The work targets scenarios where available DRAM is approximately half the model size (Section 4.1). This is a stricter constraint than offloading systems that rely on large DRAM buffers.

By acknowledging that flash cannot match DRAM speeds, the authors argue that efficiency must come from **selective loading** (loading only what is needed) and **access pattern optimization** (reading in large bundles), rather than raw bandwidth improvements. This shifts the problem from a hardware limitation to an algorithmic optimization challenge.

## 3. Technical Approach

### 3.1 Reader orientation (approachable technical breakdown)

This paper presents a **hardware-aware inference system** that stores large language model weights in flash memory and loads only the necessary subsets into DRAM during token generation. The solution addresses the memory capacity problem by combining **activation sparsity prediction** (loading only neurons that will fire), **temporal reuse** (caching recently-used neurons across tokens), and **storage-optimized data layout** (reading weights in large contiguous chunks) to achieve practical inference speeds despite flash memory's bandwidth limitations.

### 3.2 Big-picture architecture (diagram in words)

The system consists of four interconnected components that operate during each token generation step. First, a **low-rank activation predictor** analyzes the current layer's attention output to forecast which Feed-Forward Network (FFN) neurons will activate, producing a binary mask indicating which weights are needed. Second, a **sliding window cache manager** maintains a DRAM buffer containing weights for neurons activated by the recent `$k$` tokens, computing the incremental difference between what is already cached and what is newly required. Third, a **flash read orchestrator** bundles corresponding rows from the up-projection matrix with columns from the down-projection matrix into contiguous chunks, then issues parallelized read requests across 32 threads to maximize throughput. Fourth, an **in-place memory manager** uses a preallocated matrix with pointer indirection to insert new weights and delete stale weights without reallocating DRAM, enabling efficient updates as the window slides forward. Information flows sequentially: the predictor outputs a mask, the cache manager computes the delta, the orchestrator reads from flash, and the memory manager updates DRAM before computation proceeds.

### 3.3 Roadmap for the deep dive

- **Activation sparsity and prediction**: We first explain why FFN layers exhibit natural sparsity and how the low-rank predictor identifies active neurons without loading full weights, since this determines *what* data must be transferred.
- **Sliding window technique**: We then describe how temporal locality across tokens reduces incremental transfers, since this determines *how much* new data is needed per token.
- **Row-column bundling**: We next detail how weights are physically organized in flash to enable larger read chunks, since this determines *how fast* data can be transferred.
- **DRAM memory management**: We then explain the preallocated data structure that minimizes memory copy overhead during cache updates, since this determines the *management cost* of loading new weights.
- **Integration and latency breakdown**: Finally, we show how these components combine to produce the end-to-end latency improvements reported in the experiments.

### 3.4 Detailed, sentence-based technical breakdown

This is a **systems optimization paper** that redesigns the LLM inference pipeline to operate under strict DRAM constraints by treating flash memory as the primary weight store and DRAM as a carefully managed cache. The core idea is that not all model parameters are needed for every token generation step, and by predicting which neurons will activate, caching recently-used weights, and reading data in flash-optimized patterns, the system can achieve practical inference speeds despite loading weights from slower storage.

#### 3.4.1 Exploiting Activation Sparsity in Feed-Forward Networks

The foundation of this approach rests on the observation that Feed-Forward Network (FFN) layers in transformer-based LLMs exhibit high **activation sparsity**, meaning that for any given input token, only a small fraction of neurons produce non-zero outputs after the activation function. The paper reports that the OPT 6.7B model exhibits approximately **97% sparsity** within its FFN layers, while the Falcon 7B model adapted with ReLU activation achieves **95% sparsity** while maintaining similar accuracy (Section 3.1). This sparsity arises naturally from the ReLU activation function, which zeroes out all negative pre-activation values, but the authors note that models using other activations (like Llama 2's SwiGLU) can be fine-tuned with **FATReLU** to achieve **90% sparsity** (Section 3.1, citing Song et al., 2024).

The critical insight is that if a neuron's output will be zero after activation, then the corresponding weights connecting to that neuron need not be loaded from flash memory for that token. However, a naive approach would require loading the full **up-projection matrix** to compute which neurons activate, defeating the purpose of selective loading. To resolve this circular dependency, the authors employ a **low-rank predictor** that forecasts neuron activation using only the attention module's output from the current layer, without needing the previous layer's FFN output (Section 3.1, building on Liu et al., 2023b).

The predictor architecture, illustrated in **Figure 3b**, consists of a small low-rank matrix that takes the attention output as input and produces a binary prediction for each FFN neuron. Mathematically, if `$\mathbf{h} \in \mathbb{R}^{d_{\text{model}}}$` denotes the attention output for a token, the predictor computes:

$$\mathbf{p} = \sigma(\mathbf{W}_{\text{pred}} \mathbf{h} + \mathbf{b}_{\text{pred}})$$

where `$\mathbf{W}_{\text{pred}} \in \mathbb{R}^{r \times d_{\text{model}}}$` is a low-rank weight matrix with rank `$r$` much smaller than `$d_{\text{model}}$`, `$\mathbf{b}_{\text{pred}}$` is a bias vector, and `$\sigma$` is a sigmoid function. The output `$\mathbf{p}$` is then thresholded (typically at 0.5) to produce a binary mask indicating which neurons are predicted to activate. The predictor is trained using a **balanced loss** over positive and negative samples from each layer, using 10,000 samples from the C4 training dataset for 2 epochs, which takes approximately 4 hours on an A100 GPU per predictor (Section 3.1).

The predictor's rank varies by model and layer to balance prediction accuracy against overhead. For OPT 6.7B, the initial 28 layers use predictors with rank `$r = 128$`, while the final 4 layers use `$r = 1024$` to reduce false negatives (Appendix C.1). For Falcon 7B, the initial 28 layers use `$r = 256$` and the last 4 layers use `$r = 1152$` (Appendix C.2). For Persimmon 8B, the initial 32 layers use `$r = 256$` and the last 4 layers use `$r = 1152$` (Appendix C.3). For Phi-2, which exhibits different sparsity patterns with middle layers being less sparse, the authors train predictors of size 160 for layers 1-4, 480 for layers 9-12, and 800 for middle layers, with no predictors for the last group (Appendix C.4). The predictor overhead is minimal: for OPT 6.7B, predictors comprise less than **2.4% of non-embedding weights and FLOPs**, and account for only **2.75% of inference time on M1 Max CPU** and **4.8% on RTX GPU** (Appendix B.3).

**Table 1** demonstrates that using predictors does not adversely affect model performance on zero-shot tasks: OPT 6.7B achieves 66.1% on Arc Easy without predictors and 66.2% with predictors, while Arc Challenge remains at 30.6% and HellaSwag drops minimally from 50.3% to 49.8%. The authors note that false negatives (neurons the predictor missed that should have activated) constitute a small portion of elements, and as shown in **Figure 3a**, most false positives have output magnitudes close to zero, meaning their exclusion does not significantly alter the final output (Section 3.1).

#### 3.4.2 The Sliding Window Technique for Temporal Reuse

While the predictor identifies which neurons activate for a single token, the **sliding window technique** exploits the observation that consecutive tokens in a sequence tend to activate overlapping sets of neurons. This temporal locality allows the system to cache weights for recently-used neurons in DRAM, reducing the incremental data that must be loaded for each new token.

The authors define `$s_{\text{agg}}(k)$` as the **cumulative use of neuron data** across a sequence of `$k$` input tokens (Section 3.1). The memory architecture is designed to store an average of `$s_{\text{agg}}(k)$` in DRAM, representing the union of all neurons activated by the most recent `$k$` tokens. As each new token is processed, the system loads only the **incremental neuron data**, mathematically represented as:

$$\Delta s = s_{\text{agg}}(k + 1) - s_{\text{agg}}(k)$$

This incremental transfer is significantly smaller than loading all active neurons for each token independently, because many neurons remain active across consecutive tokens. **Figure 4a** illustrates this phenomenon for the tenth layer of Falcon 7B, showing that the slope of aggregated neuron usage decreases as window size `$k$` increases, meaning larger windows result in less data being loaded per new token.

**Figure 4b** provides a concrete visualization of the sliding window mechanism. The diagram shows an initial window covering tokens "Once Upon A Time There Was A Kid" with certain neurons cached in DRAM. When processing the new token "Who", the system identifies which neurons are newly active (shown in red), which neurons from the initial window remain active (shown in green), and which neurons are no longer needed and can be deleted (shown with strike-through). Rather than deleting neurons immediately when they fall out of the window, the system maintains the active neurons of the past `$k$` tokens, and only when a new token arrives does it compute the difference.

The window size `$k$` is a critical hyperparameter that balances DRAM usage against transfer volume. Larger values of `$k$` result in more neurons cached in DRAM, reducing the incremental load per token but consuming more memory. Smaller values of `$k$` conserve DRAM but require more frequent transfers. The authors state that "in determining the size of the sliding window, the aim is to maximize it within the constraints imposed by the available memory capacity" (Section 3.1). In practice, the paper uses `$k = 4$` or `$k = 5$` for most models: OPT 6.7B uses `$k = 4$` (Appendix C.1), Falcon 7B uses `$k = 4$` (Appendix C.2), Persimmon 8B uses `$k = 4$` with dynamic reduction when memory exceeds 25% of FFN size (Appendix C.3), and Llama 2 uses `$k = 2$` (Appendix C.5).

Using the windowing method with `$k = 4$` in OPT 6.7B significantly reduces fresh data loading requirements. While using active neurons from the predictor alone would require about **10% of DRAM memory capacity** on average, the windowing method reduces this to **2.4%** (Appendix C.1). The overall memory retained in DRAM comprises embeddings (3%), attention model weights (32.3%), the predictor (1.25%), and the loaded FFN layer (15.5%, calculated as `$0.24 \times 64.62$`), summing to **52.1% of the model's total size** (Appendix C.1). This confirms the paper's claim that models up to **2x the size of available DRAM** can be run (Abstract, Section 1).

#### 3.4.3 Row-Column Bundling for Flash Throughput Optimization

While the predictor and windowing reduce the *volume* of data transferred, the **row-column bundling** technique addresses the *throughput* of data transfer by optimizing how weights are physically organized and read from flash memory. As established in Section 2.2 and **Figure 2b**, flash memory achieves much higher throughput for large, sequential reads compared to small, random reads due to the "latency to first byte" overhead.

The key observation enabling bundling is that in the FFN layer, the usage of the `$i$`-th column from the up-projection matrix and the `$i$`-th row from the down-projection matrix **coincides with the activation of the `$i$`-th intermediate neuron** (Section 3.2). This means that whenever neuron `$i$` is predicted to activate, both its up-projection column weights and down-projection row weights are needed simultaneously. By storing these corresponding columns and rows together contiguously in flash memory, the system can consolidate what would be two separate reads into one larger read.

**Figure 5** illustrates this bundling approach. Without bundling, reading a single neuron's weights would require two separate flash reads: one for the up-projection column (size `$d_{\text{model}} \times \text{num\_bytes}$`) and one for the down-projection row (size `$d_{\text{model}} \times \text{num\_bytes}$`). With bundling, these are stored adjacently, doubling the chunk size from `$d_{\text{model}} \times \text{num\_bytes}$` to `$2 \times d_{\text{model}} \times \text{num\_bytes}$`. For a 32-bit model like OPT 6.7B, this results in a data chunk size per read of `$2 \times d_{\text{model}} \times 4 \text{ bytes} = 32 \text{ KiB}$` (Appendix C.1). For Falcon 7B, the chunk size is `$35.5 \text{ KiB}$` per read (Appendix C.2).

**Table 2** quantifies the impact of bundling on throughput and latency. Without bundling (row 3), the system achieves **1.25 GB/s** throughput with **738 ms** I/O latency for OPT 6.7B on M1 Max. With bundling enabled (row 5), throughput increases to **2.25 GB/s** and I/O latency drops to **87 ms**, representing a **1.8x throughput improvement** and **8.5x latency reduction** from the bundling optimization alone. The authors note that "while this reduces the throughput, the bundling technique can alleviate this by doubling the data transfer chunk size and hence the throughput which leads to reducing the overall latency to half" (Table 2 caption).

The authors also explored an alternative bundling strategy based on **neuron co-activation patterns**, hypothesizing that neurons with correlated activity could be bundled together for even larger chunk sizes. Analysis of the C4 validation dataset revealed a **power law distribution of co-activations**, where each neuron has a "closest friend" that co-activates with it at least **95% of the time** (Appendix D, **Figure 12b**). However, this approach led to **multiple loadings of highly active neurons**, because very active neurons are the "closest friends" of many others, counteracting the goal of minimizing data transfer. The authors present this as a negative result to inspire future research (Section 3.2, Appendix D).

To further maximize throughput, the system employs **parallelized reads across 32 threads** (Section 4.1, Implementation Details). This multithreaded approach amortizes the latency to first byte by not waiting for each read sequentially, and maximizes read throughput by reading multiple streams simultaneously, as suggested by **Figure 2b** which shows throughput increasing with thread count. The authors conducted benchmarks **without operating system caching** (using `F_NOCACHE` on macOS and DirectIO on Linux) to accurately measure flash throughput under limited DRAM conditions (Appendix C, Caching Considerations).

#### 3.4.4 Efficient DRAM Memory Management with Preallocated Structures

Even with optimized flash reads, managing data within DRAM incurs non-negligible overhead. When introducing weights for new neurons, a naive approach would reallocate the matrix and append new data, requiring significant memory copies. The authors note that "when a substantial portion (approximately 25%) of the Feed-Forward Networks (FFNs) in DRAM needs to be rewritten," this overhead becomes particularly costly (Section 3.3).

To address this, the system adopts a **preallocated memory management strategy** with a specialized data structure, illustrated in **Figure 6**. The data structure comprises several components:

- **Pointer vector**: Indicates the original neuron index corresponding to each row in the matrix, allowing the system to track which neurons are stored where.
- **Matrix**: Preallocated with size `$\text{Req}_i \times 2d_{\text{model}}$` for the `$i$`-th layer, where `$\text{Req}_i$` denotes the maximum number of neurons required for the specified window size based on analysis of the C4 validation set (Section 3.3).
- **Bias vector**: Stores the up-projection bias from the original model for each neuron.
- **`num_used`**: Tracks the number of rows currently utilized in the matrix, initially set to zero.
- **`last_k_active`**: Identifies which neurons from the original model were most recently activated using the last `$k$` tokens, enabling efficient deletion decisions.

The memory update process follows three steps, as depicted in **Figure 6**:

1. **Deleting Neurons**: Neurons no longer required are identified in linear time using the `last_k_active` value and current prediction. The matrix rows, pointers, and scalars for these redundant neurons are **replaced with the most recent elements** (swapping with the last rows), and their count is subtracted from `num_rows`. For `$c$` neurons to be deleted, this requires a memory rewrite of order `$O(c \times d_{\text{model}})$` rather than reallocating the entire matrix.

2. **Bringing in New Neurons**: Required weights are retrieved from flash memory. The corresponding pointers and scalars are read from DRAM, and these rows are inserted into the matrix extending from `num_row` to `num_row + num_new`. This approach **eliminates the need for reallocating memory in DRAM and copying existing data**, reducing inference latency.

3. **Inference Process**: For computation, the first half of `matrix[:num_rows, :d_model]` serves as the up-projection, and the transposed second half `matrix[:num_rows, d_model:].transpose()` serves as the down-projection (Section 3.3). This configuration is valid because "the order of neurons in the intermediate output of the FFN does not alter the final output, allowing for a streamlined inference process."

**Table 3** breaks down the latency contribution of memory management. For OPT 6.7B on CPU, memory management accounts for **58 ms** out of 669 ms total latency (8.7%). On Metal M1, it accounts for **35 ms** out of 565 ms (6.2%). On Metal M2, it accounts for only **8 ms** out of 305 ms (2.6%). On GPU, it accounts for **34 ms** out of 84 ms (40.5%), reflecting the higher overhead of CPU-to-GPU data transfers. These numbers confirm that while memory management is not the dominant cost, it is significant enough to warrant optimization.

#### 3.4.5 Selective Persistence Strategy for Attention vs. FFN Weights

A critical design choice in the system is which model components remain permanently in DRAM versus which are loaded on-demand. The authors adopt a **selective persistence strategy** where embeddings and attention mechanism matrices are retained constantly in DRAM, while only the non-sparse segments of the Feed-Forward Network (FFN) are dynamically loaded (Section 3.1).

This decision is motivated by several factors. First, **attention weights constitute approximately one-third of the model's size**, and keeping them in memory "allows for more efficient computation and quicker access, thereby enhancing inference performance without the need for full model loading" (Section 3.1). Second, the attention mechanism does not exhibit the same activation sparsity as the FFN layer, making selective loading less beneficial. Third, the **up-projection matrix must be fully present in memory** to compute which neurons activate, but the predictor circumvents this by using attention output instead (Section 3.1).

For OPT 6.7B, the memory breakdown is: embeddings (3%), attention model weights (32.3%), predictor (1.25%), and loaded FFN (15.5%), totaling 52.1% (Appendix C.1). For Falcon 7B: embeddings (4.2%), attention weights (19.4%), predictors (4%), and active FFN portion (25.3%), totaling 52.93% (Appendix C.2). This confirms that attention and embeddings together occupy roughly **35-40% of model size**, leaving the remaining DRAM budget for the cached FFN weights.

#### 3.4.6 Integration: End-to-End Latency Composition

The three optimization techniques—predictor-based sparsity, windowing, and bundling—combine to produce the end-to-end latency improvements reported in the experiments. **Table 2** provides an ablation study showing the cumulative effect for OPT 6.7B on M1 Max:

| Configuration | Predictor | Windowing | Bundling | DRAM (GB) | Flash→DRAM (GB) | Throughput (GB/s) | I/O Latency (ms) |
|--------------|-----------|-----------|----------|-----------|-----------------|-------------------|------------------|
| Naive        | ✗         | ✗         | ✗        | 0         | 13.4            | 6.10              | 2196             |
| Hybrid       | ✓         | ✗         | ✗        | 6.7       | 6.7             | 6.10              | 1090             |
| + Windowing  | ✓         | ✓         | ✗        | 4.8       | 0.9             | 1.25              | 738              |
| + Bundling   | ✓         | ✓         | ✓        | 6.5       | 0.2             | 2.25              | 87               |

The naive baseline loads the entire model (13.4 GB) for each token, resulting in 2196 ms I/O latency. The hybrid baseline persists half the model in DRAM (6.7 GB) and loads the other half on-demand without sparsity, achieving 1090 ms. Adding the predictor reduces transfer to 6.7 GB. Adding windowing further reduces transfer to 0.9 GB but drops throughput to 1.25 GB/s due to scattered reads. Finally, adding bundling restores throughput to 2.25 GB/s and reduces transfer to 0.2 GB, achieving 87 ms I/O latency— a **25x improvement over naive** and **12.5x over hybrid**.

**Table 3** shows end-to-end latency including compute. For OPT 6.7B on GPU, the naive approach takes 2218 ms total (2196 ms I/O + 22 ms compute), while the optimized "All" approach takes 84 ms total (30 ms I/O + 34 ms memory + 20 ms compute), representing a **26x speedup**. With speculative decoding added, latency drops further to 60 ms (Table 3, row "OPT 6.7B Speculative GPU").

The latency breakdown reveals that under the optimized system, **compute becomes a larger fraction of total latency** compared to the naive approach. For OPT 6.7B GPU naive, compute is only 1% of total time (22/2218). For OPT 6.7B GPU "All", compute is 24% of total time (20/84). This shift indicates the flash loading optimizations successfully reduced I/O from the dominant bottleneck to a manageable component, though I/O remains significant at 36% (30/84).

#### 3.4.7 Design Rationale: Why This Approach Over Alternatives

The authors' design choices reflect careful trade-offs between competing constraints. The decision to use a **low-rank predictor** rather than loading the full up-projection to compute activation is motivated by the circular dependency problem: you cannot know which neurons activate without computing the up-projection, but computing the up-projection requires loading all weights. The predictor breaks this cycle by approximating activation from attention output alone, with the trade-off of introducing false positives (loading some unnecessary weights) and false negatives (missing some necessary weights). The authors accept this trade-off because **Table 1** shows negligible accuracy impact, and false positives near zero magnitude do not significantly affect output (Figure 3a).

The choice of **sliding window over per-token caching** exploits temporal locality in token sequences. An alternative would be to cache only the current token's active neurons, but **Figure 4a** shows that aggregated neuron usage has a decreasing slope, meaning consecutive tokens share many active neurons. The window size `$k = 4$` or `$k = 5$` was chosen empirically to balance DRAM usage against transfer reduction, though the authors note this is not a hard constraint and can be adjusted based on available memory (Section 4.3, **Figure 7**).

The **row-column bundling** design choice reflects the flash memory physics established in Section 2.2. An alternative would be to read rows and columns separately, but **Figure 2b** demonstrates that throughput increases with chunk size. Bundling doubles chunk size with no accuracy cost, since the up-projection column and down-projection row for neuron `$i$` are always needed together. The rejected co-activation bundling alternative demonstrates the authors' willingness to explore and report negative results, providing valuable guidance for future work (Appendix D).

The **preallocated memory structure** choice avoids the overhead of dynamic memory allocation during inference. An alternative would be to allocate and deallocate memory for each token's neuron set, but this would incur significant copy overhead when 25% of FFN weights need rewriting (Section 3.3). The pointer-indirection approach allows in-place updates with minimal copying, though it requires upfront memory reservation based on worst-case neuron requirements.

Finally, the decision to **persist attention weights in DRAM** while loading FFN weights on-demand reflects the sparsity differential between these components. The FFN exhibits 90-97% sparsity amenable to selective loading, while attention does not. Additionally, attention weights are accessed for every token regardless of sparsity, making them better candidates for permanent residency. This hybrid approach achieves the 50% DRAM utilization target while maximizing the benefit of selective loading where it matters most.

## 4. Key Insights and Innovations

This paper contributes more than just a set of optimization tricks; it introduces a fundamental shift in how we conceptualize the memory hierarchy for Large Language Model (LLM) inference. Below, we dissect the core innovations, distinguishing between foundational paradigm shifts and critical engineering advancements that make the system viable.

### 4.1 Fundamental Innovation: Flash as Primary Storage, Not Just Overflow
The most significant conceptual contribution of this work is the reversal of the standard memory hierarchy assumption. Prior systems like **FlexGen** (Sheng et al., 2023) or **Dejavu** (Liu et al., 2023b) treat flash memory as a secondary overflow store, assuming the primary working set (or at least the active weights) resides in DRAM or GPU memory (Section 6).

*   **The Shift:** This paper operates under the constraint that the **full model cannot reside in DRAM**. Instead, flash is treated as the primary weight store, and DRAM is treated strictly as a limited cache (Section 1).
*   **Why It Matters:** This unlocks the ability to run models **up to twice the size of the available DRAM** (Abstract). For example, on a device with `$16\text{GB}$` of RAM, standard approaches might cap out at a `$7\text{B}$` model (requiring `$\sim 14\text{GB}$`). This approach allows running a `$13\text{B}$` class model by keeping only attention weights and a sparse subset of FFN weights in DRAM.
*   **The Cost Model:** The innovation is driven by a hardware-informed **inference cost model** mentioned in the Abstract. Rather than optimizing purely for FLOPs (compute), the algorithm optimizes for **I/O latency** and **throughput**, explicitly accounting for the "latency to first byte" penalty described in Section 2.2. This co-design ensures that algorithmic choices (like sparsity) are validated against physical hardware constraints (like flash chunk sizes).

### 4.2 Algorithmic Innovation: Temporal Sparsity via Sliding Window
While activation sparsity (loading only active neurons) has been explored in prior work (e.g., **Dejavu**), this paper introduces a novel dimension: **temporal locality across tokens**.

*   **Prior Work Limitation:** Previous sparse inference methods typically optimize for a single forward pass. They identify active neurons for token `$t$`, load them, compute, and then discard or reload for token `$t+1$`.
*   **The Innovation:** The **Sliding Window Technique** (Section 3.1) recognizes that consecutive tokens in a sequence often activate overlapping sets of neurons. By maintaining a cache of neurons active over the last `$k$` tokens, the system loads only the **incremental difference** `$\Delta s = s_{\text{agg}}(k + 1) - s_{\text{agg}}(k)$` rather than the full active set.
*   **Significance:** This reduces the data transfer volume dramatically. As shown in **Table 2**, adding windowing to the predictor reduces flash transfer from `$6.7\text{GB}$` to `$0.9\text{GB}$` for OPT 6.7B. This is not just an optimization; it changes the unit of optimization from the **layer** to the **token sequence window**, enabling stable inference over long generations without thrashing the DRAM cache.

### 4.3 Systems Innovation: Structure-Aware Data Layout (Row-Column Bundling)
The **Row-Column Bundling** technique (Section 3.2) is a critical engineering innovation that bridges the gap between model architecture and storage physics.

*   **The Problem:** Sparse loading inherently creates random access patterns, which suffer from low throughput on flash memory (**Figure 2b**). Simply loading less data (sparsity) is insufficient if the access pattern is inefficient.
*   **The Solution:** The authors observe that the `$i$`-th column of the up-projection matrix and the `$i$`-th row of the down-projection matrix are always needed together for neuron `$i$`. By physically storing these weights contiguously on flash, they double the read chunk size from `$d_{\text{model}} \times \text{num\_bytes}$` to `$2d_{\text{model}} \times \text{num\_bytes}$` (**Figure 5**).
*   **Impact:** This is an **incremental but high-leverage innovation**. It does not change the model architecture or accuracy, but it directly addresses the hardware bottleneck. **Table 2** shows that bundling increases throughput from `$1.25\text{GB/s}$` to `$2.25\text{GB/s}$` and cuts I/O latency by more than half (from `$738\text{ms}$` to `$87\text{ms}$`). Without this layout change, the sparsity gains would be negated by flash latency penalties.

### 4.4 Methodological Innovation: Transparency via Negative Results
A rare and valuable contribution of this paper is the explicit documentation of a failed optimization strategy in **Appendix D**.

*   **The Experiment:** The authors hypothesized that neurons with high **co-activation** (neurons that fire together often) could be bundled to increase chunk sizes further. They identified "closest friend" neurons that co-activate `$\ge 95\%$` of the time (**Figure 12b**).
*   **The Failure:** Bundling based on co-activation led to **multiple loadings of highly active neurons**. Because "very active neurons are the closest friends of many others," this strategy increased total data transfer rather than reducing it (Section 3.2).
*   **Significance:** By publishing this negative result, the authors save the research community significant time and provide insight into the **power law distribution of neuron co-activation**. It highlights that sparsity patterns are complex and that intuitive optimizations (grouping correlated items) can fail in systems constrained by total volume rather than just access count. This level of transparency is uncommon in systems papers and elevates the work's scientific value.

### 4.5 Capability Innovation: Viable Inference at 50% DRAM Capacity
Finally, the paper demonstrates a new capability boundary for edge devices.

*   **The Benchmark:** The experiments are conducted under the strict condition that **approximately half of the model size is available in DRAM** (Section 4.1).
*   **The Result:** Despite this severe constraint, the system achieves end-to-end latencies competitive with naive baselines that assume full residency. For OPT 6.7B on GPU, the optimized system achieves `$84\text{ms}$` per token compared to `$2218\text{ms}$` for a naive flash loading baseline (**Table 3**).
*   **Why It's Novel:** Prior offloading systems often incur order-of-magnitude latency penalties when exceeding DRAM capacity. This work proves that with hardware-aware sparsity and loading strategies, the latency penalty can be reduced to a factor of `$\sim 4\text{x}$` on CPU and `$\sim 20\text{x}$` on GPU compared to naive loading, making large model inference on memory-constrained devices practically feasible for the first time.

## 5. Experimental Analysis

The authors validate their hardware-aware framework through a rigorous evaluation strategy designed to stress-test the system under the specific constraint that **DRAM capacity is approximately half the model size**. This section dissects the experimental methodology, quantifies the performance gains, and critically assesses whether the results support the claim of viable large model inference on memory-constrained devices.

### 5.1 Evaluation Methodology

To ensure the results reflect real-world edge device constraints, the experimental setup deviates from standard server-based benchmarks in three critical ways: strict memory budgeting, diverse hardware backends, and specific baseline comparisons.

*   **Hardware Configurations:** Experiments were conducted across three distinct setups to verify generalizability:
    *   **Apple M1 Max:** 1TB SSD, running on CPU (float32) or GPU (Metal, float16).
    *   **Apple M2 Ultra:** 2TB SSD, running on GPU (Metal).
    *   **Linux NVIDIA RTX 4090:** 24GB VRAM, running on GPU (bfloat16).
    *   *Note:* Across all setups, the authors assume **nearly half of the total memory** is allocated for model computations, leaving the rest for the OS and other processes (Section 4.1).

*   **Models and Data:** The evaluation covers a range of model architectures and sizes to test robustness:
    *   **Models:** OPT 6.7B, Falcon 7B (sparsified), Persimmon 8B, Phi-2 2.7B, and Llama 2 7B (sparsified via FATReLU).
    *   **Dataset:** A subset of the **C4 validation dataset** is used for latency measurements. Prompts consist of the first 128 tokens, followed by the generation of 256 new tokens (Section 4.1).
    *   **Sparsity:** Models exhibit varying sparsity rates. OPT 6.7B shows **97% sparsity** in FFN layers, while Phi-2 required a **65% DRAM limit** due to lower observed sparsity (Section 4.2).

*   **Baselines:** The authors define two primary baselines to isolate the value of their contributions:
    1.  **Naive:** Loads the entire model from flash for every forward pass. This represents the theoretical worst-case I/O scenario.
    2.  **Hybrid:** Persists half the model in DRAM and loads the other half on-demand *without* using sparsity or windowing. This represents the theoretical lower bound for non-sparse approaches under the 50% memory constraint (Section 4.1).

*   **Metrics:** The primary metric is **inference latency per token** (ms), decomposed into three components:
    *   **I/O:** Time to load weights from flash to DRAM.
    *   **Memory Management:** Overhead for updating the DRAM cache (deleting/inserting weights).
    *   **Compute:** Time for matrix multiplications and activation functions.

### 5.2 Core Performance Results

The central claim of the paper is that efficient flash loading can enable models **2x larger than available DRAM** with minimal latency penalties. The quantitative results strongly support this, particularly on GPU backends.

**Table 3** presents the end-to-end inference latency across different setups. The speedups are substantial when comparing the optimized ("All") method against the Naive baseline:

*   **OPT 6.7B (GPU):** Latency drops from **2218 ms** (Naive) to **84 ms** (Ours), a **~26x speedup**.
*   **OPT 6.7B (CPU):** Latency drops from **3182 ms** (Naive) to **669 ms** (Ours), a **~4.7x speedup**.
*   **Falcon 7B (CPU):** Latency drops from **3095 ms** (Naive) to **706 ms** (Ours), a **~4.4x speedup**.
*   **Persimmon 8B (CPU):** Latency drops from **3806 ms** (Naive) to **1041 ms** (Ours), a **~3.7x speedup**.

> "Our efficient implementation... that employs the predictor, windowing, and bundling can lead to significant latency reduction." (Table 3 Caption)

Crucially, the optimized system also outperforms the **Hybrid** baseline, which does not use sparsity. For example, on Falcon 7B (CPU), the Hybrid approach achieves **1947 ms**, while the optimized approach achieves **706 ms**. This confirms that sparsity awareness provides benefits beyond simple memory offloading.

**Throughput Gains:**
**Table 2** isolates the I/O performance on the M1 Max for OPT 6.7B. It demonstrates how the techniques stack:
*   **Naive I/O:** 2196 ms (loading 13.4 GB).
*   **With Predictor & Windowing:** Reduces data transfer to **0.9 GB**, but throughput drops to **1.25 GB/s** due to scattered reads, resulting in 738 ms latency.
*   **With Bundling:** Restores throughput to **2.25 GB/s** by doubling chunk size, reducing I/O latency further to **87 ms**.

This data validates the design choice in Section 3.2: reducing volume is insufficient without optimizing the access pattern (bundling) to maintain flash throughput.

### 5.3 Accuracy and Model Integrity

A critical risk in sparse inference is accuracy degradation. If the predictor misses active neurons (false negatives), the model output could diverge significantly from the dense baseline. The authors address this through zero-shot benchmarks and perplexity measurements.

**Zero-Shot Tasks:**
**Table 1** compares OPT 6.7B performance with and without the low-rank predictor:
*   **Arc Easy:** 66.1% (Baseline) vs. **66.2%** (With Predictor).
*   **Arc Challenge:** 30.6% (Baseline) vs. **30.6%** (With Predictor).
*   **HellaSwag:** 50.3% (Baseline) vs. **49.8%** (With Predictor).

The results indicate **negligible accuracy loss**. The authors explain in Section 3.1 that most false positives (unnecessary weights loaded) have output magnitudes close to zero (Figure 3a), and false negatives are rare enough not to impact the aggregate output.

**Perplexity and Relufication:**
For models requiring sparsification (like Phi-2 or Llama 2), the process of replacing activation functions with ReLU (or FATReLU) can impact quality.
*   **Phi-2:** MMLU accuracy drops from **57** (Original) to **54.3** (Relufied with Distillation). Using predictors further drops it slightly but keeps it at **52** (Figure 10b, Appendix B.2).
*   **Persimmon 8B:** Using larger predictors (rank 1152) for the last 4 layers prevents MMLU accuracy drops compared to using uniform smaller predictors (Figure 10a).

This suggests a trade-off: achieving high sparsity may require architectural tweaks (like ReLU substitution) that incur a small accuracy penalty, but the predictor itself does not significantly worsen this.

### 5.4 Ablation Studies and Component Contribution

The paper provides a clear ablation study in **Table 2**, allowing us to quantify the contribution of each technical component introduced in Section 3.

| Component | Data Transfer | Throughput | I/O Latency |
| :--- | :--- | :--- | :--- |
| **Naive** | 13.4 GB | 6.10 GB/s | 2196 ms |
| **+ Predictor** | 6.7 GB | 6.10 GB/s | 1090 ms |
| **+ Windowing** | 0.9 GB | 1.25 GB/s | 738 ms |
| **+ Bundling** | 0.2 GB | 2.25 GB/s | **87 ms** |

*   **Predictor:** Reduces data volume by **50%** (13.4 GB → 6.7 GB) by exploiting activation sparsity.
*   **Windowing:** Reduces data volume by **87%** (6.7 GB → 0.9 GB) by exploiting temporal locality across tokens.
*   **Bundling:** Increases throughput by **80%** (1.25 → 2.25 GB/s) and reduces final latency by **88%** (738 ms → 87 ms).

This breakdown confirms that **all three techniques are necessary**. Without windowing, the data volume remains too high. Without bundling, the throughput collapses due to random read penalties.

**Memory-Latency Trade-off:**
**Figure 7** explores the relationship between DRAM usage and latency. By increasing the percentage of the model kept in DRAM (relaxing the 50% constraint), latency decreases.
*   At **50% DRAM**, latency is higher due to more frequent flash loads.
*   At **75% DRAM**, latency drops significantly as fewer weights need swapping.
This validates the "sliding window" hyperparameter `$k$`: larger windows consume more DRAM but reduce I/O, allowing users to tune based on available hardware (Section 4.3).

### 5.5 Robustness and Edge Cases

The authors test the system under conditions that typically degrade performance in offloading systems, such as long generation sequences and non-greedy sampling.

*   **Long Generation:** A common concern with flash storage is thermal throttling or performance degradation over time. **Figure 8** plots weight loading latency over **1000 tokens** for OPT 6.7B. The latency remains stable, and notably, the flash latency for the *first few tokens* is higher because the DRAM cache is initially empty. As the cache fills, latency stabilizes.
*   **Sampling Methods:** Non-greedy sampling (e.g., **Nucleus sampling** with `$p=0.9$`) can lead to more diverse activation patterns, potentially reducing the effectiveness of the sliding window. The authors report that "Nucleus sampling doesn't lead to lower performance in long generation in neither cpu or gpu" (Section 5.1).
*   **Speculative Decoding:** To further improve speed, the authors integrate speculative decoding (using a draft model to propose tokens). **Table 3** shows that for OPT 6.7B on GPU, speculative decoding reduces total latency from **84 ms** to **60 ms**. Section 5.2 notes a **1.4x speedup**, which is close to the theoretical 1.58x speedup of speculative decoding, indicating their memory management does not bottleneck this advanced technique.

### 5.6 Critical Assessment

While the experimental results are compelling, a critical analysis reveals specific limitations and trade-offs that users must consider.

*   **Power Consumption:** Section 5.3 acknowledges a significant drawback. While the sparse model has lower **power usage** (energy per unit time) than a dense model, the **total energy consumption** is higher because the inference takes longer.
    > "The extended duration required for token generation resulted in the sparse model having a higher total energy consumption." (Section 5.3)
    This makes the technique ideal for latency-constrained devices with power availability, but potentially suboptimal for battery-critical scenarios where total energy matters more than peak power.

*   **Batch Size Limitations:** The experiments are limited to **single-batch inference** (processing one sequence at a time). Section 8 explicitly states: "Currently, our study is limited to single-batch inference." This limits applicability in server settings where throughput via batching is key, though it aligns with the personal device use case.

*   **Sparsity Dependency:** The performance gains are contingent on high activation sparsity. Phi-2, which exhibited lower sparsity, required allocating **65% of DRAM** rather than 50% to maintain performance (Section 4.2). Models without natural sparsity (or those not fine-tuned for it, like standard Llama 2) would require the FATReLU modification, which incurs a training cost and slight accuracy drop (Appendix C.5).

*   **Hardware Specifics:** The throughput benefits rely on specific flash characteristics (high sequential read speed). While the authors argue these characteristics apply to smartphones (Appendix F), actual performance will vary based on the specific SSD controller and NAND type in the target device.

**Conclusion on Experiments:**
The experiments convincingly support the paper's primary claim: **LLMs larger than DRAM can run efficiently on personal devices.** The ablation studies (Table 2) rigorously prove that each component (predictor, windowing, bundling) is necessary to achieve the reported speedups. However, the trade-offs in total energy consumption and single-batch limitations indicate this is a specialized solution for **interactive, single-user edge inference** rather than a universal replacement for server-side dense inference.

## 6. Limitations and Trade-offs

No systems optimization comes without costs. This section critically examines the assumptions underpinning the "LLM in a Flash" approach, identifies scenarios where the method may underperform, and highlights open questions that remain unresolved. Understanding these limitations is essential for practitioners considering deployment in real-world settings.

### 6.1 Energy Efficiency Trade-off: Lower Power, Higher Total Consumption

One of the most significant trade-offs acknowledged by the authors is the **energy consumption paradox**. While the sparse, flash-based approach consumes less power *per unit of time* compared to dense model inference, the **total energy consumed per token is higher** due to the extended inference duration.

> "While the power usage (energy per unit of time) of the sparse model was lower than that of the dense model, the extended duration required for token generation resulted in the sparse model having a higher total energy consumption." (Section 5.3)

This distinction matters for different deployment scenarios:

| Scenario | Priority | Suitability |
|----------|----------|-------------|
| **Plugged-in devices** (laptops, desktops) | Latency > Energy | ✅ Well-suited |
| **Battery-constrained devices** (smartphones) | Energy > Latency | ⚠️ Potential concern |
| **Thermal-constrained devices** | Power density | ⚠️ Needs evaluation |

The authors explicitly state that "a systematic and quantitative evaluation of the exact power usage pattern is left as a future work" (Section 5.3). This gap means practitioners cannot precisely estimate battery life impact without empirical testing on target hardware.

### 6.2 Single-Batch Inference Constraint

The experimental evaluation is strictly limited to **single-sequence inference**. Section 8 states:

> "Currently, our study is limited to single-batch inference. We provide some preliminary results on combining our proposed idea with speculative decoding, however, expanding this to include more complex scenarios like prompt processing and multi-batch inference are valuable areas for further investigation." (Section 8)

This limitation has practical implications:

*   **Server Deployments:** Production LLM services typically batch multiple requests to maximize GPU utilization. The current system does not support this optimization, limiting its applicability to cloud inference scenarios.
*   **Prompt Processing:** The initial prompt processing phase (processing hundreds of tokens before generation begins) is not explicitly optimized. While Figure 8 shows stable latency over 1000 token generations, the **first few tokens incur higher flash latency** because "the allocated memory in DRAM is empty and needs to be filled in with neurons" (Section 5.1). For long prompts, this initial cache-warming cost could be substantial.
*   **Throughput vs. Latency:** The system optimizes for **per-token latency** (important for interactive applications) rather than **tokens-per-second throughput** (important for batch processing). These are distinct optimization goals.

### 6.3 Sparsity Dependency and Model Modification Requirements

The approach fundamentally relies on **high activation sparsity** in the Feed-Forward Network (FFN) layers. This dependency creates several constraints:

*   **Not All Models Are Naturally Sparse:** The OPT 6.7B model exhibits **97% sparsity** naturally (Section 3.1). However, Phi-2 showed **lower sparsity rates**, requiring the authors to allocate **65% of DRAM** instead of the standard 50% (Section 4.2). Models without inherent sparsity would see diminished benefits.

*   **Architectural Modification Required:** For models using activation functions other than ReLU (e.g., Llama 2's SwiGLU), achieving high sparsity requires **fine-tuning with FATReLU** (Section 3.1, citing Song et al., 2024). This introduces:
    *   **Training Cost:** Models must be retrained or fine-tuned, which is computationally expensive.
    *   **Accuracy Degradation:** Appendix B.2 and Figure 10b show that Phi-2's MMLU accuracy drops from **57** (original) to **54.3** (relufied with distillation), and further to **52** with predictors applied.
    *   **Appendix C.5** notes: "For models that have used Swi-GLU activation function... replacing Swish with ReLU within the FFN doesn't ensure high amount of sparsity."

*   **Layer-Wise Sparsity Variation:** Sparsity is not uniform across layers. **Figure 9** shows that for OPT-6.7B, later layers have more active neurons, while for Phi-2, middle layers are less sparse (Appendix B.1). This requires **layer-specific predictor configurations** (different ranks for different layers), adding implementation complexity.

### 6.4 Predictor Accuracy and False Negatives

The low-rank activation predictor is critical to the system's efficiency, but it is **not perfect**. The authors acknowledge two types of prediction errors:

*   **False Positives:** Neurons predicted as active that are actually inactive. These result in **unnecessary weight loads**, wasting I/O bandwidth. The authors note that "most of the False Positives are close to 0" (Figure 3a caption), minimizing accuracy impact but still incurring transfer cost.

*   **False Negatives:** Neurons predicted as inactive that should have activated. These are more concerning as they **directly affect model output**. Appendix C.1 states the predictor achieves "an average of **5% false negatives** and **7% false positives** in the OPT 6.7B model."

While Table 1 shows negligible accuracy impact on zero-shot tasks, the authors note that "to reduce false negatives, the predictor has to 'over-predict', which results in loading neurons that are redundant" (Appendix B.1). This creates a **tuning dilemma**: higher predictor accuracy requires larger predictor ranks (more compute overhead) or lower thresholds (more data transfer).

**Table 4** demonstrates this trade-off for Persimmon 8B:

| Threshold | Arc Easy | Arc Challenge | HellaSwag |
|-----------|----------|---------------|-----------|
| 0.50      | 67.26    | 33.87         | 50.51     |
| 0.70      | 66.30    | 34.40         | 52.70     |

Higher thresholds reduce data load but can affect accuracy differently across benchmarks.

### 6.5 Hardware-Specific Performance Characteristics

The optimization techniques are explicitly **tuned to flash memory physics**, which vary across devices:

*   **Flash Throughput Variability:** The reported throughput of **2.25 GB/s** with bundling (Table 2) was measured on an **Apple M1 Max with 1TB SSD**. Different SSD controllers, NAND types (TLC vs. QLC), and interface standards (NVMe vs. eMMC) will yield different performance characteristics.

*   **Sequential Read Dependency:** The bundling technique relies on flash achieving high throughput for large sequential reads. **Figure 2b** shows throughput increases with chunk size, but the exact curve is hardware-dependent. Devices with poorer random read performance would benefit more from bundling, while devices with better random access might see diminished returns.

*   **Unified Memory Architecture:** The Apple experiments leverage **unified memory** where CPU and GPU share the same memory pool. Appendix C notes: "We made use of the unified memory architecture available on Apple systems, which we exploit to maintain the weights cache using the GPU allocator." The NVIDIA RTX 4090 setup requires **CPU-to-GPU data transfers**, which adds overhead (Table 3 shows memory management is **40.5%** of total latency on GPU vs. **8.7%** on CPU for OPT 6.7B).

*   **Smartphone Applicability:** Appendix F argues the technique applies to smartphones by loading 4-bit values instead of 16-bit. However, the authors acknowledge: "Actual implementation would require special 4-bit compute kernels on device, which falls outside the scope of this paper." This means the claimed smartphone benefits are **theoretical** without additional engineering work.

### 6.6 Memory Capacity Assumptions

The experiments operate under a specific constraint: **approximately half of the model size is available in DRAM** (Section 4.1). While Section 4.3 and Figure 7 explore relaxing this constraint, several scenarios remain unaddressed:

*   **Less Than 50% DRAM:** The paper does not evaluate scenarios where available DRAM is **25% or less** of model size. At extreme memory constraints, the sliding window would need to shrink significantly, potentially negating the temporal reuse benefits.

*   **More Than 50% DRAM:** Figure 7 shows latency decreases as more model parameters are kept in DRAM (from 50% to 75%). However, the **diminishing returns curve** is not fully characterized. At what point does keeping more weights in DRAM cease to provide meaningful latency improvements?

*   **Dynamic Memory Availability:** Personal devices often have **variable available memory** depending on background processes. The system assumes a fixed memory budget, but real-world conditions may require dynamic adjustment of the window size `$k$`, which is not implemented.

### 6.7 Flash Storage Wear and Longevity

A notable omission in the paper is any discussion of **flash memory wear**. Flash storage has a finite number of program/erase cycles, and while reads are less damaging than writes, **intensive read operations** can still contribute to wear over time.

*   **Read Amplification:** The selective loading approach means the same weights may be read multiple times across token generations. Over thousands of inference sessions, this could accumulate significant read volume.
*   **Thermal Considerations:** Section 8 notes that "systematic analysis of power consumption and thermal limitations inherent in the methods we propose, particularly for on-device deployment" is left for future work. While Figure 8 shows no thermal throttling over 1000 tokens, **longer sessions** or **higher ambient temperatures** could present issues.

### 6.8 Failed Optimization: Co-activation Bundling

The paper transparently reports a **negative result** in Appendix D regarding neuron co-activation bundling. While intellectually valuable, this failure reveals limitations in the optimization space:

*   **Power Law Distribution:** Neuron co-activations follow a power law, where highly active neurons are the "closest friends" of many others (Appendix D, Figure 12).
*   **Multiple Loading Problem:** Bundling based on co-activation "led to multiple loadings of highly active neurons, counteracting our goal" (Section 3.2).
*   **Implication:** This suggests that **simple correlation-based optimizations** may not work for sparse inference. More sophisticated bundling strategies would need to account for **global activation frequency**, not just pairwise correlation.

The authors present this to "inspire future research on effective neuron bundling" (Section 3.2), acknowledging that the current bundling approach (row-column pairing) may not be optimal.

### 6.9 Summary of Trade-offs

| Trade-off | Benefit | Cost |
|-----------|---------|------|
| **Flash Storage** | Enables 2x model size vs. DRAM | Higher latency than DRAM-resident inference |
| **Activation Sparsity** | Reduces data transfer volume | Requires model modification (ReLU/FATReLU) for some architectures |
| **Sliding Window** | Exploits temporal locality | Consumes additional DRAM for caching |
| **Row-Column Bundling** | Improves flash throughput | Requires custom weight storage layout |
| **Low-Rank Predictor** | Avoids loading full up-projection | Adds 2-5% compute overhead, introduces false negatives |
| **Single-Batch Design** | Optimized for interactive latency | Not suitable for high-throughput server deployment |
| **Sparse Inference** | Lower power per unit time | Higher total energy per token |

### 6.10 Open Questions for Future Research

The authors identify several directions for future work in Section 8:

1.  **Multi-Batch Inference:** "Expanding this to include more complex scenarios like prompt processing and multi-batch inference are valuable areas for further investigation."
2.  **Variable Memory Sizes:** "Exploring the dynamics of working with varying memory sizes—both larger and smaller—introduces a fascinating balance between latency and accuracy."
3.  **Power and Thermal Analysis:** "A critical aspect for future exploration is the systematic analysis of power consumption and thermal limitations inherent in the methods we propose."
4.  **Improved Bundling Strategies:** The failed co-activation bundling suggests "more optimized techniques of weight bundling and data structures can be crafted" (Section 7).
5.  **Non-Sparse Adaptation:** "The underlying concept holds potential for broader applications. It can be adapted to selectively load weights in non-sparse networks" based on input prompt characteristics (Section 8).

These open questions indicate that while the paper establishes a viable foundation for flash-based LLM inference, significant optimization headroom remains for specialized deployment scenarios.

## 7. Implications and Future Directions

This paper does more than solve a specific engineering bottleneck—it establishes a new paradigm for thinking about memory hierarchies in Large Language Model (LLM) inference. By demonstrating that flash memory can serve as the **primary weight store** rather than a secondary overflow, the authors open a design space that was previously considered impractical. This section examines how this work reshapes the field, what research directions it enables, where it can be practically deployed, and when practitioners should consider adopting this approach over alternatives.

### 7.1 Paradigm Shift: Redefining the Memory Hierarchy

The most profound implication of this work is the **reversal of the standard memory hierarchy assumption** for LLM inference. Prior systems operate under the constraint that model weights must reside in DRAM or GPU memory during inference, with flash serving only as persistent storage for loading and checkpointing (Section 6). This paper demonstrates that flash can be an **active participant in the inference loop**, not just a passive archive.

**Why This Changes the Field:**

*   **Capacity Decoupling:** Model size is no longer strictly bound by DRAM capacity. The paper demonstrates running models **up to 2x the size of available DRAM** (Abstract, Section 1). For a device with `$16\text{GB}$` of RAM, this shifts the feasible model class from `$7\text{B}$` parameters to potentially `$13\text{B}$` parameters.
*   **Hardware-Algorithms Co-Design:** The work establishes that inference algorithms must be **explicitly designed around storage physics**. The row-column bundling technique (Section 3.2) shows that data layout on flash is as important as the model architecture itself. This elevates storage optimization from a systems concern to a core ML systems research direction.
*   **Edge-First Inference:** By targeting the **50% DRAM availability constraint** (Section 4.1), the paper shifts focus from server-scale optimization to personal device deployment. This aligns with growing industry interest in on-device AI for privacy, latency, and cost reasons.

> "Overall, the primary assumption in the literature is that the model can fully reside in the GPU memory or system DRAM. However, considering the limited resources available on personal devices, we do not share this assumption in this work." (Section 6)

This statement marks a clear departure from prior work and establishes a new research frontier: **inference under strict memory constraints**.

### 7.2 Follow-Up Research Directions

The authors explicitly identify several open problems in Section 8 (Limitations) and Section 7 (Discussion). These represent concrete opportunities for future research:

#### 7.2.1 Multi-Batch and Prompt Processing Optimization

**Current Limitation:** The system is limited to **single-batch inference** (Section 8). Production LLM services typically batch multiple requests to maximize hardware utilization.

**Research Opportunity:** Extending the sliding window and memory management techniques to handle **multiple concurrent sequences** would require:
*   Developing a **multi-sequence cache eviction policy** that balances neuron reuse across different prompts.
*   Optimizing the **prompt processing phase** (initial token ingestion), which currently incurs higher flash latency because "the allocated memory in DRAM is empty and needs to be filled in with neurons" (Section 5.1).
*   Investigating whether **batched flash reads** can amortize the "latency to first byte" penalty across multiple sequences.

This is a high-impact direction because it would extend applicability from personal devices to **edge server deployments** where throughput matters more than per-token latency.

#### 7.2.2 Power and Thermal Characterization

**Current Limitation:** The paper acknowledges that "a systematic and quantitative evaluation of the exact power usage pattern is left as a future work" (Section 5.3). While the sparse model has lower **power per unit time**, it has higher **total energy per token** due to extended inference duration.

**Research Opportunity:** A comprehensive power analysis would enable:
*   **Battery Life Estimation:** Practicians could predict how many tokens can be generated on a single charge for smartphones or laptops.
*   **Thermal Throttling Models:** Understanding when sustained inference triggers thermal limits on mobile SoCs.
*   **Energy-Latency Trade-off Curves:** Optimizing for **energy efficiency** rather than just latency, which may favor different window sizes `$k$` or predictor thresholds.

This is critical for **battery-constrained deployments** where total energy matters more than peak power (Section 6.1).

#### 7.2.3 Improved Weight Bundling Strategies

**Current Limitation:** The row-column bundling technique (Section 3.2) doubles chunk size but is relatively simple. The authors explored **co-activation bundling** (Appendix D) but found it led to "multiple loadings of highly active neurons" due to power law distributions in neuron activity (Figure 12).

**Research Opportunity:** Future work could explore:
*   **Frequency-Aware Bundling:** Instead of pairing neurons by correlation, bundle based on **global activation frequency** to avoid over-loading highly active neurons.
*   **Layer-Specific Layouts:** Different layers exhibit different sparsity patterns (Figure 9). OPT has more active neurons in later layers, while Phi-2 has more in middle layers (Appendix B.1). Custom layouts per layer could optimize chunk sizes further.
*   **Dynamic Reorganization:** Weights could be **reorganized on flash** based on observed activation patterns during initial inference sessions, creating a personalized layout for specific use cases.

The authors explicitly state: "from the algorithmic perspective, more optimized techniques of weight bundling and data structures can be crafted" (Section 7).

#### 7.2.4 Variable Memory Size Adaptation

**Current Limitation:** Experiments assume **approximately half of the model size is available in DRAM** (Section 4.1). While Figure 7 explores relaxing this constraint, the system does not dynamically adapt to **variable memory availability**.

**Research Opportunity:** Personal devices have **fluctuating memory** depending on background processes. A production system would need:
*   **Dynamic Window Adjustment:** Automatically shrinking or expanding the sliding window `$k$` based on available DRAM.
*   **Graceful Degradation:** Maintaining inference when available memory drops below 50%, potentially by increasing predictor thresholds or reducing cached neurons.
*   **Memory Pressure Signals:** Integrating with OS-level memory management to evict weights proactively before system thrashing occurs.

The authors note: "Exploring the dynamics of working with varying memory sizes—both larger and smaller—introduces a fascinating balance between latency and accuracy" (Section 8).

#### 7.2.5 Non-Sparse Network Adaptation

**Current Limitation:** The approach relies on **high activation sparsity** (90-97% for OPT, Falcon). Models without natural sparsity require **fine-tuning with FATReLU** (Section 3.1), which incurs training costs and accuracy penalties (Appendix B.2).

**Research Opportunity:** The authors suggest the concept "can be adapted to selectively load weights in non-sparse networks" based on input characteristics (Section 8). This could involve:
*   **Context-Adaptive Loading:** Predicting which weights are needed based on **prompt semantics** rather than activation sparsity.
*   **Attention-Based Sparsity:** Exploiting sparsity in the **attention mechanism** rather than just FFN layers.
*   **Hybrid Approaches:** Combining this method with **quantization** (Appendix F mentions 4-bit loading for smartphones) to reduce transfer volume further.

This direction would broaden applicability to models that cannot be easily sparsified.

### 7.3 Practical Applications and Use Cases

The techniques in this paper are specifically designed for **memory-constrained edge devices**. Below are concrete deployment scenarios where this approach provides unique value:

#### 7.3.1 Personal Device LLM Assistants

**Scenario:** Running a `$7\text{B}$` to `$13\text{B}$` parameter model on a laptop or tablet with `$16\text{GB}$` of RAM.

**Why This Method:** Standard inference requires `$\sim 14\text{GB}$` just for model weights (Section 1), leaving insufficient memory for the OS, applications, and KV cache. This approach enables:
*   **Multitasking:** Users can run the LLM alongside other applications without swapping.
*   **Larger Models:** Access to more capable models than would fit in DRAM alone.
*   **Privacy:** On-device inference avoids sending data to cloud APIs.

**Evidence:** The paper demonstrates OPT 6.7B running with **52.1% of model size in DRAM** (Appendix C.1), leaving the rest of RAM available for other processes.

#### 7.3.2 Smartphone Deployment

**Scenario:** Running LLMs on smartphones with `$4\text{GB}$` to `$8\text{GB}$` of RAM.

**Why This Method:** Appendix F explicitly discusses smartphone applicability:
*   **4-Bit Loading:** Instead of 16-bit values, the system could load **4-bit quantized weights**, reducing DRAM requirements further.
*   **Memory Budget:** The authors estimate that "rather than the baseline's `$3.5\text{GB}$` of required DRAM, our model would necessitate **less than `$2\text{GB}$`** of memory for operation" (Appendix F).

**Caveat:** This requires "special 4-bit compute kernels on device, which falls outside the scope of this paper" (Appendix F). Practitioners would need to implement or integrate quantized kernels.

#### 7.3.3 Embedded and IoT Devices

**Scenario:** Running LLMs on devices with **severely limited DRAM** (e.g., `$1\text{GB}$` to `$2\text{GB}$`).

**Why This Method:** The sliding window technique (Section 3.1) allows trading DRAM for flash bandwidth. By reducing window size `$k$`, the system can operate with **less than 50% DRAM availability**, though with higher latency.

**Evidence:** Figure 7 shows latency decreases as more model parameters are kept in DRAM, but the system remains functional at lower percentages.

#### 7.3.4 Cost-Effective Edge Servers

**Scenario:** Deploying LLM inference on **low-cost hardware** with limited GPU memory (e.g., consumer GPUs with `$8\text{GB}$` to `$12\text{GB}$` VRAM).

**Why This Method:** Instead of purchasing expensive GPUs with `$24\text{GB}$`+ VRAM, organizations can use **commodity hardware** with large SSDs. The paper demonstrates **20x speedup on GPU** compared to naive flash loading (Abstract, Table 3).

**Evidence:** OPT 6.7B on RTX 4090 achieves **84 ms per token** with this method vs. **2218 ms** for naive loading (Table 3).

### 7.4 Reproducibility and Integration Guidance

Practitioners considering this approach should evaluate whether their use case aligns with the method's strengths and constraints. Below is guidance on **when to adopt this method** versus alternatives.

#### 7.4.1 When to Prefer This Method

| Condition | Recommendation | Rationale |
|-----------|----------------|-----------|
| **DRAM < 50% of model size** | ✅ Use this method | Standard inference is impossible; this enables 2x model capacity (Section 1) |
| **Single-user interactive inference** | ✅ Use this method | Optimized for per-token latency, not batch throughput (Section 8) |
| **Model has high FFN sparsity** | ✅ Use this method | OPT (97%), Falcon (95%), Persimmon (90%+) benefit most (Section 3.1) |
| **Flash storage available** | ✅ Use this method | Requires SSD/NVMe with good sequential read performance (Figure 2b) |
| **Latency > Energy priority** | ✅ Use this method | Lower power per unit time, but higher total energy per token (Section 5.3) |

#### 7.4.2 When to Consider Alternatives

| Condition | Alternative | Rationale |
|-----------|-------------|-----------|
| **DRAM ≥ 100% of model size** | Standard inference | No need for flash loading; DRAM is faster (Section 2.1) |
| **Multi-batch server deployment** | FlexGen, vLLM | These support batching; this method is single-sequence only (Section 8) |
| **Battery-critical deployment** | Quantization-only | Lower total energy per token, though smaller model capacity |
| **Model lacks sparsity** | Dejavu, standard offloading | This method requires 90%+ sparsity or FATReLU fine-tuning (Section 3.1) |
| **No flash storage access** | DRAM-only methods | Requires SSD/NVMe; not applicable to memory-only systems |

#### 7.4.3 Implementation Requirements

To reproduce or integrate this approach, practitioners need:

1.  **Sparsified Model:** Either a naturally sparse model (OPT, Falcon) or one fine-tuned with **FATReLU** (Section 3.1, citing Song et al., 2024).
2.  **Low-Rank Predictors:** Trained per layer using **10,000 samples from C4** for 2 epochs (~4 hours per predictor on A100) (Section 3.1).
3.  **Custom Weight Layout:** Weights must be stored with **row-column bundling** on flash (Figure 5).
4.  **Memory Management System:** Preallocated DRAM structures with pointer indirection (Figure 6).
5.  **Parallel Flash Reads:** **32 threads** for optimal throughput (Section 4.1, Implementation Details).
6.  **OS-Level Cache Control:** Use `F_NOCACHE` (macOS) or DirectIO (Linux) to bypass OS caching for accurate throughput (Appendix C).

**Code Availability:** The paper does not explicitly mention open-source code release. Practitioners would need to implement the techniques based on the descriptions in Section 3 and Appendix C.

#### 7.4.4 Integration with Existing Frameworks

The authors built their implementation using **HuggingFace Transformers** and **PyTorch** (Section 4.1), with custom Metal kernels borrowed from Apple's **MLX framework** (Appendix C). For integration:

*   **PyTorch:** Requires custom operators for dynamic weight loading and memory management.
*   **Metal (Apple):** Leverages unified memory architecture; the authors used `MTLStorageModeShared` for CPU-GPU access without copies (Appendix C).
*   **NVIDIA GPU:** Requires explicit CPU-to-GPU transfers, which adds overhead (Table 3 shows memory management is **40.5%** of latency on GPU vs. **8.7%** on CPU for OPT 6.7B).

**Speculative Decoding Compatibility:** The method integrates with speculative decoding, achieving **1.4x speedup** (Section 5.2, Table 3). This suggests compatibility with other decoding optimizations.

### 7.5 Summary: A Foundation for Edge LLM Inference

This paper establishes a **viable foundation** for running large language models on memory-constrained devices. The key contributions are:

1.  **Proof of Concept:** Models **2x larger than DRAM** can run with acceptable latency (Abstract, Section 1).
2.  **Hardware-Aware Design:** Algorithms must be co-designed with storage physics (Section 2, Figure 2).
3.  **Modular Techniques:** Predictor, windowing, and bundling can be adopted independently or together (Table 2).
4.  **Transparent Limitations:** The authors clearly document trade-offs in energy, batching, and sparsity dependency (Section 8).

**Future Impact:** As LLMs continue to grow and edge devices become more capable, the techniques in this paper will become increasingly relevant. The paradigm shift from "fit the model in memory" to "load what you need from flash" opens new possibilities for **democratizing LLM access** on personal devices (Section 8).

> "Our study represents an initial endeavor in the pursuit of democratizing Large Language Model (LLM) inference, making it accessible to a wider array of individuals and devices." (Section 8)

This statement captures the broader vision: not just optimizing inference, but **expanding who can run LLMs** and **where they can run**. The techniques in this paper are a critical step toward that goal.
