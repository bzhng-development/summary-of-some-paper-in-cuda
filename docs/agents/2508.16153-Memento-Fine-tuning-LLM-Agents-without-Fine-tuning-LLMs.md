## 1. Executive Summary

This paper introduces Memento, a novel learning paradigm that enables Adaptive Large Language Model (LLM) agents to achieve continual adaptation through memory-based online reinforcement learning, eliminating the need for computationally intensive fine-tuning of the underlying LLM parameters. By formalizing agent behavior as a Memory-augmented Markov Decision Process (M-MDP) with a neural case-selection policy, Memento attains top-1 performance on the GAIA validation set (87.88% Pass@3, Table 2) and achieves 66.6% F1 on the DeepResearcher dataset, outperforming state-of-the-art training-based methods. This approach provides a scalable pathway for developing generalist agents capable of real-time learning, demonstrated by absolute accuracy gains of 4.7% to 9.6% on out-of-distribution tasks through efficient memory reading and rewriting mechanisms (Figure 1d).

## 2. Context and Motivation

### The Core Problem: Continuous Adaptation Without Parameter Updates

The paper addresses a fundamental tension in building capable LLM agents: **how to enable continuous learning from experience without the prohibitive computational cost of fine-tuning the underlying language model**. This is not merely an efficiency concern—it represents a structural limitation in how current agent systems are designed.

To understand why this matters, consider what happens when an LLM agent encounters a novel task after deployment. The paper identifies two prevailing paradigms, each with critical shortcomings (Section 1):

1. **Static, handcrafted workflows**: Agents operate with fixed reasoning pipelines that work well for narrow, predefined tasks but cannot incorporate new information or adapt to unfamiliar situations. Once deployed, these systems are essentially frozen—they neither learn from successes nor avoid repeating failures.

2. **Parameter-tuning approaches**: Agents update the LLM itself through supervised fine-tuning (SFT) or reinforcement learning (RL). While this enables more flexible behavior, it comes at substantial computational cost and requires large volumes of human-annotated training data. As the paper notes in Section 2.1, *"when tackling long-horizon, complex tasks, LLM agent systems must spend substantial time rolling out trajectories to gather training data, and they additionally depend on large volumes of human-annotated questions."*

The central research challenge, explicitly stated in the Introduction, is:

> *"How can we build LLM agents that learn continuously from a changing environment without the prohibitive cost of fine-tuning the underlying LLMs?"*

This question matters because real-world agent deployments—particularly in **deep research scenarios**—require handling open-ended, evolving tasks where pre-training on all possible situations is infeasible.

### Why This Problem Matters: Real-World and Theoretical Significance

The importance of this problem extends across both practical deployment concerns and theoretical machine learning advances:

**Real-World Impact:**

- **Deep research agents** must navigate dynamic information environments where facts change, new tools emerge, and task structures vary widely. The paper evaluates on benchmarks like GAIA (requiring up to 50 steps with unrestricted tool use, Section 5.1) and DeepResearcher (real-time web research across seven QA datasets). Static agents cannot adapt to such variability.

- **Computational efficiency** directly affects deployability. Fine-tuning large models requires significant GPU resources and time, making continuous adaptation impractical for many organizations. A method that achieves improvement without gradient updates offers a fundamentally different cost structure.

- **Catastrophic forgetting** remains a persistent risk in parametric approaches. Section 2.1 explicitly notes that parameter-updating methods face *"the danger of catastrophic forgetting"*—where learning new tasks degrades performance on previously mastered ones. Non-parametric memory avoids this by keeping the base model frozen.

**Theoretical Significance:**

- The paper frames agent learning through a **Memory-augmented Markov Decision Process (M-MDP)** (Definition 3.1, Section 3), extending standard MDP formulations with an explicit memory space $\mathcal{M} = (\mathcal{S} \times \mathcal{A} \times \mathcal{R})^*$. This formalization enables rigorous analysis of how past experiences should influence future decisions.

- By drawing inspiration from **human memory mechanisms** (Section 1), the work connects AI agent design to cognitive science principles: episodic trace encoding, sleep-dependent consolidation, dopamine-driven credit assignment, and case-based retrieval. This interdisciplinary grounding provides theoretical justification for the architectural choices.

### Prior Approaches and Their Limitations

The Related Work section (Section 2) provides a comprehensive taxonomy of existing methods. Understanding where they fall short clarifies Memento's contributions.

**Parametric Continual-Learning Approaches:**

Methods like START (Li et al., 2025a) and various RL-based agent training frameworks update model weights directly. The paper identifies three key limitations:

- **High computational cost**: Full fine-tuning requires substantial compute resources for each adaptation cycle.
- **Data inefficiency**: These approaches depend on *"large volumes of human-annotated questions"* (Section 2.1), which are expensive and slow to collect.
- **Catastrophic forgetting**: Parameter updates risk degrading previously learned capabilities.

**Non-Parametric Memory Approaches:**

Methods that freeze the LLM and attach external memory include RAG systems, reflection pipelines, and various memory-augmented agents. However, the paper identifies critical gaps:

- **Static retrieval corpora**: Modern RAG systems *"typically query static document corpora and lack mechanisms for continual adaptation"* (Section 2.1). They retrieve from fixed knowledge bases rather than learning from interaction trajectories.

- **Pre-defined heuristics**: ReAct-style agents and reflective prompting pipelines (Shinn et al., 2023; Yao et al., 2023) demonstrate improvement through feedback but *"remain constrained by pre-defined heuristics and do not achieve true lifelong learning"* (Section 2.3).

- **The swamping problem**: A classic issue in case-based reasoning where *"most systems keep adding cases without selective curation, leading to the classic swamping problem where retrieval costs outweigh utility"* (Section 2.3, citing Francis and Ram, 1993). Without intelligent case selection, memory grows unbounded and retrieval becomes inefficient.

**Tool-Augmented and Multi-Agent Systems:**

Frameworks like AutoGen, OWL, and DeerFlow coordinate specialized agents via dialogue. While powerful, they face a bottleneck: *"without explicit planning, deciding when and which tools to invoke remains a major bottleneck for long-horizon tasks"* (Section 2.2). These systems often lack mechanisms to learn from past tool invocation successes and failures.

**Recent Memory Mechanisms:**

Several contemporary works address memory design:
- **Memory-Bank** (Zhong et al., 2024) uses Ebbinghaus-style forgetting schedules.
- **SAGE** (Liang et al., 2024) unifies reflection with memory optimization.
- **Mem0** (Chhikara et al., 2025) adopts structured memory operations (ADD, UPDATE, DELETE, NOOP).
- **ExpeL** (Zhao et al., 2024) converts trajectories into natural-language insights.

However, the paper positions itself distinctly: *"Differently, we formulate planning as a memory-augmented MDP and learn a neural case-selection policy over an episodic case bank via online soft Q-learning, enabling continual adaptation without fine-tuning the underlying LLM parameters"* (Section 2.3).

### How Memento Positions Itself

Memento occupies a specific niche in the design space of LLM agent systems. Table 7 (Appendix B) provides a detailed comparison of memory mechanisms, but the key differentiators are:

| Dimension | Prior Non-Parametric Methods | Memento |
|-----------|------------------------------|---------|
| **Memory Update** | Append-only or heuristic-based | Online Q-function learning via TD updates |
| **Case Selection** | Similarity-based or fixed rules | Neural policy $\mu(c \mid s, M)$ optimized via soft Q-learning |
| **Learning Signal** | None or offline | Online environmental feedback (rewards) |
| **Base Model** | Frozen | Frozen (no gradient updates to LLM) |

The paper's positioning rests on three claims:

1. **First to cast case-based continual learning for LLM agents** with formal M-MDP grounding (Section 1, final paragraph). While CBR exists in classical AI, integrating it with modern LLM agents under a reinforcement learning framework is novel.

2. **Achieves top-level performance without parameter tuning**. The GAIA validation result of 87.88% Pass@3 (Table 2) demonstrates that memory-based adaptation can match or exceed training-based methods.

3. **Provides principled framework for deep research agents**. Unlike ad-hoc memory systems, Memento offers a mathematically grounded approach with derivable optimal policies (Eq. 7, Section 3) and clear separation between planning (case retrieval) and execution (tool invocation).

The ablation studies in Section 5.5 validate this positioning. Figure 1c shows continual learning curves where parametric CBR outperforms non-parametric CBR, which in turn outperforms no CBR—demonstrating that the learned case-selection policy provides measurable value beyond simple similarity-based retrieval. Figure 1d shows OOD generalization gains of 4.7% to 9.6% absolute points, confirming that memory-based learning transfers to unseen task distributions.

In essence, Memento argues that **the path to adaptable agents need not pass through parameter space**. By treating past experiences as a learnable memory structure with an optimized retrieval policy, agents can achieve continual improvement while keeping the base LLM frozen—a fundamentally different approach to agent learning that balances performance, efficiency, and flexibility.

## 3. Technical Approach

This is a **methodology paper** that proposes a memory-based reinforcement learning framework for LLM agents, with the core idea being: **instead of updating the LLM's weights through gradient descent, we learn a retrieval policy over past experiences stored in external memory, allowing the agent to improve continuously while keeping the base model frozen**.

### 3.1 Reader orientation (approachable technical breakdown)

The *system* being built is an LLM agent that learns from its own past successes and failures by storing them in a searchable memory bank and learning which past cases to consult when facing new problems. The problem it solves is **continuous adaptation without parameter updates**, and the "shape" of the solution is a **two-stage planner–executor architecture** where the planner retrieves relevant past cases from memory before generating plans, and a learned policy determines which cases are most useful to retrieve.

### 3.2 Big-picture architecture (diagram in words)

Memento consists of three major components operating in a loop:

1. **Case Memory (Write/Read Module)**: Stores past trajectories as case triplets $(s_t, a_t, r_t)$ and supports two operations—*Write* appends new experiences after each task, and *Read* retrieves the most relevant past cases using either similarity matching (non-parametric) or a learned Q-function (parametric).

2. **Planner (LLM-driven CBR Agent)**: Receives the current task instruction, queries the Case Memory for $K$ relevant cases, concatenates them with the task prompt, and generates a decomposed plan consisting of subtasks.

3. **Executor (MCP Client)**: Takes each subtask from the planner, invokes external tools via the Model Context Protocol (MCP), records tool outputs in Tool Memory, and returns results back to the planner for evaluation and potential replanning.

Information flows as follows: User query → Planner retrieves cases from Case Memory → Planner generates subtask plan → Executor executes subtasks using MCP tools → Results stored in Subtask/Tool Memory → Planner evaluates completion → If incomplete, replan; if complete, Write new case to Case Memory.

### 3.4 Detailed, sentence-based technical breakdown

#### Formal Foundation: Memory-augmented Markov Decision Process

The paper formalizes agent behavior as a **Memory-augmented Markov Decision Process (M-MDP)**, which extends the standard MDP tuple by adding an explicit memory space. Definition 3.1 states that an M-MDP is a tuple $\langle \mathcal{S}, \mathcal{A}, \mathcal{P}, \mathcal{R}, \gamma, \mathcal{M} \rangle$, where $\mathcal{S}$ is the state space (all finite-length sequences over a vocabulary), $\mathcal{A}$ is the action space (also finite-length sequences), $\mathcal{P}: \mathcal{S} \times \mathcal{A} \rightarrow \Delta(\mathcal{S})$ is the transition dynamics, $\mathcal{R}: \mathcal{S} \times \mathcal{A} \rightarrow \mathbb{R}$ is the reward function, $\gamma \in [0, 1)$ is the discount factor, and critically, $\mathcal{M} = (\mathcal{S} \times \mathcal{A} \times \mathcal{R})^*$ is the **memory space** consisting of all possible sequences of past experience tuples.

The key difference from a standard MDP is that the agent's policy now conditions on both the current state $s_t$ *and* the accumulated memory $M_t$, rather than state alone. At each timestep $t$, the agent maintains a case bank $M_t = \{c_i\}_{i=1}^{N_t}$, where each case $c_i$ is a tuple $(s_i, a_i, r_i)$ containing a past state, action, and reward, and $N_t$ is the number of cases stored so far.

Definition 3.2 formalizes the **Case-Based Reasoning Agent** policy as a two-stage process. Given current state $s$ and case bank $M$, the agent first retrieves a case $c$ from memory according to a retrieval policy $\mu(c \mid s, M)$, which assigns a probability distribution over all cases in $M$. Then, the LLM generates an action $a$ conditioned on both the state and the retrieved case, with likelihood $p_{\text{LLM}}(a \mid s, c)$. The overall policy $\pi$ is the marginal over all possible retrieved cases:

$$
\pi(a \mid s, M) = \sum_{c \in M} \mu(c \mid s, M) p_{\text{LLM}}(a \mid s, c)
$$

This equation (Eq. 1, Section 3) is fundamental: it shows that the agent's behavior is a **mixture** of LLM responses, weighted by how likely each past case is to be retrieved. The paper's core innovation is learning $\mu$ while keeping $p_{\text{LLM}}$ frozen—this is what enables adaptation without fine-tuning.

The full trajectory $\tau$ of the CBR agent includes memory states at each step: $\tau = \{M_0, s_0, c_0, a_0, r_0, M_1, s_1, c_1, a_1, r_1, \cdots\}$. Equation 2 decomposes the trajectory probability into five components: (1) Retrieve—sampling a case from memory, (2) Reuse & Revise—LLM generating action given the case, (3) Evaluation—receiving reward from environment, (4) Retain—updating memory with the new experience, and (5) Transition—environment moving to next state. This decomposition makes explicit where learning signal comes from (the reward) and where adaptation happens (the memory update).

#### Learning the Retrieval Policy via Soft Q-Learning

The paper optimizes the retrieval policy $\mu$ using **maximum entropy reinforcement learning**, which encourages diversity in retrieved cases while maximizing expected reward. The objective function (Eq. 3) is:

$$
J(\pi) = \mathbb{E}_{\tau \sim p} \left[ \sum_{t=0}^{T-1} \left[ \mathcal{R}(s_t, a_t) + \alpha \mathcal{H}(\mu(\cdot \mid s_t, M_t)) \right] \right]
$$

where $\mathcal{H}$ denotes entropy and $\alpha$ is a hyperparameter controlling the entropy weight. The entropy term prevents the retrieval policy from collapsing to always selecting the same case, which would reduce the agent's ability to explore diverse strategies.

The value function $V^\pi(s_t, M_t)$ represents the expected return from state $s_t$ with memory $M_t$ under policy $\pi$, and is defined as (Eq. 4):

$$
V^\pi(s_t, M_t) = \sum_{c \in M_t} \mu(c \mid s_t, M_t) \left[ Q^\pi(s_t, M_t, c) - \alpha \log \mu(c \mid s_t, M_t) \right]
$$

The Q-function $Q^\pi(s_t, M_t, c_t)$ represents the expected return of selecting case $c_t$ at state $s_t$ with memory $M_t$, followed by optimal behavior thereafter (Eq. 5):

$$
Q^\pi(s_t, M_t, c_t) = \mathbb{E}_{a \sim p_{\text{LLM}}(\cdot \mid s_t, c_t), s_{t+1} \sim \mathcal{P}(\cdot \mid s_t, a_t)} \left[ \mathcal{R}(s_t, a_t) + \gamma V^\pi(s_{t+1}, M_{t+1}) \right]
$$

Note that $M_{t+1}$ denotes the updated memory after adding the new experience $(s_t, a_t, r_t)$—this is critical because the memory grows over time, and future decisions can leverage newly acquired experiences.

The paper derives a **closed-form solution** for the optimal retrieval policy $\mu^*$ as a softmax over optimal Q-values (Eq. 7):

$$
\mu^*(c \mid s, M) = \frac{\exp(Q^*(s, M, c) / \alpha)}{\sum_{c' \in M} \exp(Q^*(s, M, c') / \alpha)}
$$

This is a key result: the optimal way to retrieve cases is to assign higher probability to cases with higher Q-values, with temperature controlled by $\alpha$. The derivation uses Lagrange multipliers to enforce that $\mu$ sums to 1 (see Appendix A for full derivation).

To learn the Q-function, the paper uses **temporal difference (TD) learning** from soft Q-learning (Eq. 8):

$$
Q(s_t, M_t, c_t) \leftarrow Q(s_t, M_t, c_t) + \eta \left[ r_t + \gamma \alpha \log \sum_{c' \in M_{t+1}} \exp(Q(s_{t+1}, M_{t+1}, c_{t+1})) - Q(s_t, M_t, c_t) \right]
$$

where $\eta$ is the learning rate. This update rule adjusts Q-values toward the TD target, which includes the immediate reward $r_t$ plus the discounted soft value of the next state.

#### Kernel-based Q-function Approximation for Natural Language States

Directly learning the Q-function via TD updates is challenging because states and cases are described in natural language, making the state space effectively infinite and unstructured. To address this, the paper proposes **kernel-based estimation** following episodic control algorithms (Pritzel et al., 2017).

The system maintains an **episodic memory** $\mathcal{D} = \{(s, c, Q)\}$ that stores past interactions, including the state, retrieved case, and Q-value. The Q-function is approximated using a kernel network $k_\theta(\cdot, \cdot)$ parameterized by $\theta$ (Eq. 9):

$$
Q_{\text{EC}}(s, M, c; \theta) = \frac{\sum_{(s', c', Q') \in \mathcal{D}_c} k_\theta(s, s') Q'}{\sum_{(\hat{s}, \hat{c}, \hat{Q}) \in \mathcal{D}_c} k_\theta(s, \hat{s})}
$$

where $\mathcal{D}_c = \{(s_i, c_i, Q_i) \in \mathcal{D} : c_i = c\}$ denotes past interactions with the same retrieved case $c$. Intuitively, this estimates the Q-value for a new state $s$ by taking a **weighted average** of Q-values from similar past states, where similarity is determined by the kernel $k_\theta(s, s')$.

The kernel parameters $\theta$ are learned by minimizing the TD loss (Eq. 10):

$$
\mathcal{L}(\theta) = \mathbb{E}_{(s, c, r, s', M, M')} \left[ \left( Q_{\text{EC}}(s, M, c; \theta) - \left[ r + \gamma \alpha \log \sum_{c' \in M'} \exp(Q_{\text{EC}}(s', M', c'; \bar{\theta})) \right] \right)^2 \right]
$$

where $\bar{\theta}$ denotes the target kernel network (updated periodically for stability, following standard DQN practice). The gradient with respect to $\theta$ is given in Eq. 11, which involves weighting terms $w_i = \frac{k_\theta(s, s_i)}{\sum_{s_j \in \mathcal{D}_c} k_\theta(s, s_j)}$ that determine how much each past experience contributes to the current Q-estimate.

Algorithm 1 provides the complete training loop: at each timestep, the agent retrieves a case using the current policy, executes the action, observes reward and next state, stores the transition in a replay buffer $\mathcal{B}$, samples a mini-batch, and updates $\theta$ via gradient descent. The target network $\bar{\theta}$ is updated periodically (every $K$ steps) using Polyak averaging with weight $\beta$.

#### Implementation: Planner–Executor Architecture with Three Memory Modules

Figure 3 illustrates the concrete system architecture, which instantiates the M-MDP framework in a **planner–executor** design alternating between Case-Based Planning (Stage 1) and Tool-Based Execution (Stage 2).

**Stage 1: Case-Based Planning.** The Planner, powered by GPT-4.1 (Section 5.3), receives the user query and queries the Case Memory for $K$ relevant case triplets $(s_i, a_i, r_i)_{i=1}^K$, where $s_i$ is the task description, $a_i$ is the plan that was executed, and $r_i$ indicates success (binary reward). The Case Memory supports two retrieval modes: non-parametric (similarity-based) and parametric (Q-function-based), detailed below. The retrieved cases are concatenated with the current task instruction to form the prompt, guiding the LLM to generate a **decomposed plan** consisting of multiple subtasks.

**Stage 2: Tool-Based Execution.** The Executor, powered by o3 for GAIA or o4-mini for other datasets (Section 5.3), executes each subtask as an autonomous episode using the **Model Context Protocol (MCP)**. MCP serves as a standardized, model-agnostic interface that enables flexible coordination with diverse external tools. For each subtask, the executor consults the Tool Memory (which logs tool interactions scoped per subtask), determines the appropriate tool invocation, and updates results. The executor reads pending subtasks from Subtask Memory, which orchestrates the interaction between planner and executor by recording generated subtasks and their execution outcomes.

**Three Memory Modules.** The system integrates three distinct memory types (Section 4.1):

1. **Case Memory**: Vectorized storage of prior cases $(s_t, a_t, r_t)$ for high-level planning. This is the core learning module that grows over time and supports both Write and Read operations.

2. **Subtask Memory**: Text-based storage of active subtasks and their results. This enables the planner to track progress across multiple execution iterations and decide when to replan.

3. **Tool Memory**: Text-based logs of tool interactions for each subtask. This provides fine-grained traceability of which tools were called with which arguments and what results they produced.

After each iteration, the planner uses accumulated execution history to assess task completion. If unfinished, the planner replans based on updated context; if complete, the final result is returned and the Case Memory is updated with the new experience.

#### Case Memory Management: Non-Parametric vs. Parametric Retrieval

The Case Memory is an online-growing case bank $M_t$ operated with **Write** and **Read** operations, available in two variants that represent different points on the complexity–performance tradeoff.

**Write Operation (Both Variants).** Following Definition 3.2, the Write operation appends each historical case $(s_t, a_t, r_t)$ to the case bank after each timestep (Eq. 12):

$$
\text{Write}(s_t, a_t, r_t, M_t) = M_{t+1} = M_t \cup \{(s_t, a_t, r_t)\}
$$

The state $s_t$ is encoded using a **frozen text encoder** (SimCSE, Section 5.3), while the action $a_t$ and reward $r_t$ are preserved in original form. Only the state representation requires vectorization for subsequent retrieval. Critically, the paper notes that *"only the state, action, and reward from the final step of each trajectory are written to memory, ensuring that the case bank remains both compact and informative"* (Section 5.3). This design choice prevents memory from growing too rapidly—instead of storing every intermediate step, only completed trajectories are retained.

**Non-Parametric Retrieval.** This variant retrieves cases by computing semantic similarity between the current state and past states, following the mainstream CBR paradigm that assumes similar problems should have similar solutions. The Read operator is defined as (Eq. 13):

$$
\text{Read}_{\text{NP}}(s_t, M_t) = \underset{(s_i, a_i, r_i) \in M_t}{\text{TopK}} \ \text{sim}(\text{enc}(s_t), \text{enc}(s_i))
$$

where $\text{enc}(\cdot)$ is the pretrained textual encoder (SimCSE) and $\text{sim}(\cdot)$ is cosine similarity. This approach is computationally efficient and requires no training, but it assumes that **surface-level semantic similarity** correlates with solution utility—which may not always hold.

**Parametric Retrieval.** This variant learns a Q-function $Q(s, c; \theta)$ that predicts the utility of retrieving case $c$ given state $s$. Because CBR is applied only for planning in Memento (not multi-step execution), the setting simplifies to **single-step** rather than multi-step M-MDP. This collapses the TD target in Eq. 10 to the immediate reward, avoiding bootstrapping and non-stationary targets. The learning objective becomes supervised (Eq. 14):

$$
\mathcal{L}(\theta) = \mathbb{E}_{(s, c, r)} \left[ (Q(s, c; \theta) - r)^2 \right]
$$

Since the reward signal in deep research tasks is binary ($r \in \{0, 1\}$), the paper replaces Mean Squared Error with **cross-entropy loss** for numerical stability (Eq. 15):

$$
\mathcal{L}(\theta) = \mathbb{E}_{(s, c, r)} \left[ -r \log Q(s, c; \theta) - (1 - r) \log(1 - Q(s, c; \theta)) \right]
$$

The Q-function is implemented as a **two-layer MLP** (Section 5.3), and $Q(s, c; \theta)$ can be interpreted as the probability $p(r = 1 \mid s, c; \theta)$ that the retrieved case is a good reference. During retrieval, the system applies a TopK operator to select the $K$ cases with highest Q-values (Eq. 16):

$$
\text{Read}_{\text{P}}(s_t, M_t) = \underset{c_i \in M_t}{\text{TopK}} \ Q(s_t, c_i; \theta)
$$

This is more interpretable than sampling from the softmax policy (Eq. 7) and reduces randomness in case selection.

**Design Choice: Why Two Variants?** The paper provides both options because they serve different use cases. Non-parametric retrieval is simpler, requires no training, and works well when semantic similarity correlates with task similarity. Parametric retrieval learns latent patterns between states and cases, producing a closer approximation to the optimal retrieval policy $\mu^*$, but requires training data and compute. Table 7 (Appendix B) compares these against other memory mechanisms, showing that Memento's parametric variant uniquely combines neural Q-learning with episodic memory storage.

#### Tool Usage via Model Context Protocol

Deep research tasks require accessing external information and processing heterogeneous data, so Memento implements a suite of tools accessible via MCP (Section 4.3).

**External Information Acquisition.** The search toolkit integrates **searxng**, a self-hosted metasearch engine that aggregates results from Google, Bing, DuckDuckGo, and Brave. Retrieved candidates are re-ranked based on semantic similarity to the query context. For deeper understanding, the system incorporates **Crawl4AI** to fetch and parse full web content. The search tool functions as a coarse filter (keyword matching), while the crawler serves as a fine-grained mechanism for detailed extraction.

**Multimodal Heterogeneous Information Processing.** The document processing toolkit handles diverse file types: images are captioned using a vision-language model, audio is transcribed via ASR, PowerPoint files are parsed slide-by-slide, spreadsheets are converted to row-wise layout, archives are unpacked, JSON/XML are parsed into structured objects, Word documents are translated to Markdown, and videos receive natural-language summaries from VLMs. For PDFs or unsupported formats, fallback extraction via Chunkr AI or plain-text parsing is used.

**Reasoning Tools.** The Code tool provides a sandboxed environment for writing, running, and managing Python code within a persistent task directory, with validation against a security whitelist (supporting numpy, pandas, torch). The Math tool handles fundamental arithmetic operations. These tools enable data analysis, automation, and dynamic code generation tasks.

#### Why This Approach Over Alternatives?

The paper's design choices reflect deliberate tradeoffs:

1. **Frozen LLM + External Memory vs. Fine-tuning**: Fine-tuning requires substantial compute and risks catastrophic forgetting. By keeping the LLM frozen and learning only the retrieval policy, Memento achieves adaptation at a fraction of the cost while avoiding forgetting (Section 2.1).

2. **Learned Retrieval Policy vs. Similarity-only**: Pure similarity-based retrieval assumes surface-level similarity correlates with solution utility, which may not hold. Learning a Q-function captures latent patterns between states and cases, as evidenced by parametric CBR outperforming non-parametric CBR in Figure 1c and Table 4.

3. **Single-Step Q-Learning vs. Multi-Step TD**: Because CBR is applied only for planning (not execution), the multi-step M-MDP collapses to single-step, eliminating bootstrapping and non-stationary targets. This simplifies training and improves stability (Section 4.2).

4. **Cross-Entropy vs. MSE for Binary Rewards**: MSE suffers from vanishing gradients near 0/1 boundaries. Cross-entropy provides more numerically stable signals for binary rewards, which is the case in deep research tasks (Section 4.2).

5. **TopK Selection vs. Softmax Sampling**: While the optimal policy is a softmax (Eq. 7), the paper applies TopK during inference to reduce randomness and enhance interpretability. This is a practical engineering choice that trades some exploration for consistency (Section 4.2).

6. **Store Only Final Trajectory Steps**: Instead of storing every intermediate step, only the final state, action, and reward from each trajectory are written to memory. This keeps the case bank compact and prevents rapid saturation—important because *"with only about 3k training data, the Case Bank saturates quickly"* (Section 5.5.3).

These choices collectively enable Memento to achieve continual adaptation without gradient updates to the base LLM, addressing the core research challenge stated in the Introduction.

## 4. Key Insights and Innovations

This section identifies the fundamental contributions that distinguish Memento from prior work. Rather than cataloging technical components (covered in Section 3), I focus on **why** these innovations matter—what conceptual barriers they overcome and what new capabilities they enable.

### 4.1 Fundamental Innovation: Learning Adaptation Without Parameter Space Traversal

The most significant contribution is the **paradigm shift from parameter-based to memory-based adaptation**. Prior continual-learning approaches for LLM agents operate under an implicit assumption: *to improve, the model must change its weights*. This assumption drives the entire design space of methods like START (Li et al., 2025a), Search-r1 (Jin et al., 2025), and various RL-based agent training frameworks—all of which require gradient updates to the underlying LLM.

Memento challenges this assumption directly. By formalizing adaptation as **learning a retrieval policy $\mu$ over external memory** while keeping $p_{\text{LLM}}$ frozen (Eq. 1, Section 3), the paper demonstrates that performance improvement need not pass through parameter space. This is not merely an efficiency optimization—it represents a fundamentally different conception of what "learning" means for an agent system.

**Why this matters:**

- **Avoids catastrophic forgetting**: Parameter-updating methods face the persistent risk that learning new tasks degrades previously mastered capabilities (Section 2.1). By freezing the base model, Memento sidesteps this problem entirely—past knowledge remains intact because the LLM itself never changes.

- **Enables real-time adaptation**: Fine-tuning requires collecting trajectories, computing gradients, and updating weights—a process that takes hours or days. Memento's memory updates occur **online during inference**, as each completed task immediately becomes available for future retrieval (Eq. 12, Section 4.2). This transforms adaptation from a batch training concern into an inference-time capability.

- **Decouples capability from compute**: Organizations without resources for large-scale fine-tuning can still deploy adaptive agents. The computational burden shifts from GPU-intensive gradient computation to memory retrieval and case selection—operations that scale more gracefully.

The empirical validation is striking: Memento achieves **87.88% Pass@3 on GAIA validation** (Table 2), outperforming training-based methods like Search-r1-instruct (39.6% F1 on DeepResearcher vs. Memento's 66.6% F1, Table 1). This demonstrates that memory-based adaptation is not a compromise—it can **exceed** parameter-based approaches on complex, long-horizon tasks.

**Distinction from prior work:** While RAG systems also freeze the LLM and use external knowledge, they retrieve from *static* corpora without learning from interaction outcomes (Section 2.1). Memento's memory *grows and adapts* based on environmental feedback, with a learned policy determining which experiences are worth retrieving. This is the difference between reading a textbook (RAG) and learning from your own mistakes (Memento).

### 4.2 Theoretical Innovation: M-MDP as a Unifying Framework for Case-Based Reasoning

The **Memory-augmented Markov Decision Process (M-MDP)** formulation (Definition 3.1, Section 3) provides the first rigorous mathematical grounding for case-based reasoning in LLM agents. Prior CBR systems in AI operated largely on heuristic principles—"retrieve similar cases, adapt solutions"—without formal guarantees about optimality or convergence.

By extending the standard MDP tuple with an explicit memory space $\mathcal{M} = (\mathcal{S} \times \mathcal{A} \times \mathcal{R})^*$, the paper enables derivation of an **optimal retrieval policy** (Eq. 7, Section 3) through soft Q-learning. This is not just notation—it provides a principled answer to the question: *which past experiences should influence current decisions, and how strongly?*

**Why this matters:**

- **Derivable optimality**: The softmax form of $\mu^*$ (Eq. 7) is not a design choice but a **mathematical consequence** of the maximum entropy RL objective. This distinguishes Memento from ad-hoc memory systems that rely on intuition rather than derivation.

- **Bridges classical AI and modern LLMs**: Case-based reasoning has existed since the 1990s (Aamodt and Plaza, 1994), but integrating it with transformer-based agents required new formalism. The M-MDP provides this bridge, showing how episodic memory can be rigorously incorporated into sequential decision-making.

- **Enables analysis of memory dynamics**: With a formal framework, the paper can analyze questions like: How does memory growth affect policy quality? When does additional experience yield diminishing returns? Section 5.5.3 addresses this empirically, noting that *"with only about 3k training data, the Case Bank saturates quickly"*—an observation grounded in the M-MDP's treatment of memory as part of the state.

**Distinction from prior work:** Memory-Bank (Zhong et al., 2024) and SAGE (Liang et al., 2024) implement memory mechanisms but lack formal optimization objectives for *which* memories to retrieve. They use heuristics like Ebbinghaus forgetting schedules or reflection triggers. Memento's retrieval policy is **learned from rewards**, not prescribed by design—making it adaptive to task structure rather than fixed by assumption.

### 4.3 Practical Innovation: Single-Step Q-Learning for Planning-Only CBR

A subtle but important design choice is the **collapse from multi-step to single-step Q-learning** when CBR applies only to planning (Section 4.2). Because the executor handles tool invocation independently, the planner's case retrieval becomes a single decision per task rather than a sequential chain. This simplifies the TD target (Eq. 10 → Eq. 14) and eliminates bootstrapping.

**Why this matters:**

- **Training stability**: Multi-step TD learning suffers from non-stationary targets—the Q-values being learned depend on other Q-values that are also changing. By reducing to single-step supervised learning, Memento avoids this instability without sacrificing performance.

- **Reduced data requirements**: Bootstrapping requires sufficient trajectory coverage to propagate rewards backward through time. Single-step learning needs only **task-level outcomes** (success/failure), which are easier to collect than step-by-step annotations.

- **Numerical stability with binary rewards**: The paper replaces MSE with cross-entropy loss (Eq. 15) because *"MSE loss suffers from vanishing gradients near 0/1"* (Section 4.2). This is a practical insight that improves convergence when rewards are binary—a common case in deep research tasks where answers are either correct or incorrect.

**Distinction from prior work:** Neural Episodic Control (Pritzel et al., 2017) uses kernel-based Q-estimation but operates in multi-step RL settings with complex bootstrapping. Memento's single-step simplification is tailored to the planner–executor architecture, where planning and execution are decoupled. This architectural insight enables more stable learning with less data.

### 4.4 Empirical Innovation: Demonstrated OOD Generalization Through Memory

The **out-of-distribution (OOD) generalization results** (Figure 1d, Section 5.5.4) provide evidence that memory-based learning transfers better than parameter-based methods to unseen task distributions. Memento achieves **4.7% to 9.6% absolute gains** on OOD datasets (Musique, Bamboogle, PopQA) when trained on different datasets (NQ, TQ, HotpotQA, 2Wiki).

**Why this matters:**

- **Memory as a generalization mechanism**: Unlike fine-tuning, which risks overfitting to training task distributions, memory-based retrieval allows the agent to **match new tasks to analogous past experiences** regardless of domain. A successful web research strategy from one domain can inform decisions in another if the underlying structure is similar.

- **Validates the CBR hypothesis**: The core claim of case-based reasoning is that *"similar problems should have similar solutions"* (Section 4.2). The OOD results empirically validate this—retrieved cases from training domains improve performance on held-out domains, confirming that semantic similarity captures transferable structure.

- **Addresses a known weakness of RAG**: Standard RAG systems struggle with OOD queries because they retrieve from static corpora that may not cover novel situations (Section 2.1). Memento's memory *grows* during deployment, continuously expanding coverage to include new task types.

**Distinction from prior work:** Training-based methods like Search-r1 (Jin et al., 2025) and DeepResearcher (Zheng et al., 2025) show strong in-distribution performance but lack published OOD evaluations. Memento's explicit OOD analysis (Section 5.5.4) fills this gap and demonstrates that memory-based adaptation offers a complementary pathway to generalization—particularly valuable in open-ended scenarios where test distributions cannot be anticipated.

### 4.5 Architectural Insight: Fast Planning Outperforms Deliberative Planning

An unexpected finding from the ablation studies (Table 6, Section 6.3) is that **fast, non-deliberative planners outperform slow, deliberative ones** when paired with capable executors. GPT-4.1 as planner with o3 as executor achieves 70.91% average accuracy on GAIA, while o3 as both planner and executor achieves only 63.03%.

**Why this matters:**

- **Role specialization matters**: The paper's analysis reveals that deliberative planners often *"answer directly—skipping plan generation altogether—or produce overly verbose plans, which can mislead the executor"* (Section 6.3). Fast planners produce **concise, structured decompositions** that the executor can act on without confusion.

- **Challenges conventional wisdom**: The prevailing assumption in agent design is that more reasoning (slow think, chain-of-thought) always improves performance. Memento shows that in modular architectures, **planning quality depends on appropriate scope**—the planner should decompose, not solve.

- **Cost implications**: Fast planners consume fewer tokens and latency. Table 6 shows that slower planners not only underperform but also incur higher computational costs (Figure 6, Section 6.2 shows input tokens growing from 26k at Level 1 to 121k at Level 3). Efficient planning directly reduces operational expenses.

**Distinction from prior work:** Most agent frameworks (AutoGen, OWL, DeerFlow) do not systematically analyze planner model selection. They often use the same model for planning and execution, or assume stronger models always improve planning. Memento's ablation provides empirical guidance: **planner and executor should be optimized for different capabilities**, not uniformly maximized.

### Summary: Incremental vs. Fundamental Contributions

| Contribution | Type | Significance |
|--------------|------|--------------|
| Frozen LLM + learned retrieval policy | **Fundamental** | Paradigm shift from parameter-based to memory-based adaptation |
| M-MDP formalization | **Fundamental** | First rigorous RL framework for CBR in LLM agents |
| Single-step Q-learning simplification | **Incremental** | Practical stability improvement tailored to planner–executor architecture |
| OOD generalization demonstration | **Fundamental** | Empirical validation that memory enables transfer beyond training distribution |
| Fast planner > slow planner finding | **Incremental** | Architectural insight with cost and performance implications |

The fundamental innovations (frozen LLM adaptation, M-MDP formalization, OOD generalization) collectively establish Memento as a **new learning paradigm** rather than an incremental improvement. They demonstrate that continual adaptation need not require gradient updates—a claim with implications for how we design, deploy, and scale LLM agent systems in open-ended environments.

## 5. Experimental Analysis

This section evaluates whether Memento's empirical results substantiate its core claims: that memory-based adaptation without parameter fine-tuning can achieve state-of-the-art performance, enable continual learning, and generalize to out-of-distribution tasks. I examine the evaluation methodology, summarize quantitative results with specific numbers, assess the strength of evidence, and analyze ablation studies that isolate individual contributions.

### 5.1 Evaluation Methodology: Datasets, Metrics, and Baselines

The paper evaluates Memento across **four distinct benchmark categories**, each targeting a different capability dimension of deep research agents (Section 5.1):

| Benchmark | Purpose | Scale | Difficulty Stratification |
|-----------|---------|-------|--------------------------|
| **GAIA** | Long-horizon tool use and planning | 450 questions (150 validation, 300 test) | 3 levels: Level 1 (~5 steps, single tool), Level 2 (5–10 steps, multiple tools), Level 3 (up to 50 steps, unrestricted tools) |
| **DeepResearcher** | Real-time web-based research | 7 datasets × 512 examples (except Bamboogle: 125) | Open-domain QA with multi-hop reasoning |
| **SimpleQA** | Factual precision and hallucination resistance | 4,330 fact-seeking questions | Single-hop factual queries |
| **HLE** | Long-tail academic reasoning at knowledge frontier | 2,500 questions across academic subjects | Expert-level domain knowledge |

**Evaluation Metrics (Section 5.2):**

- **GAIA**: Exact Match (EM) — a prediction is correct only if it exactly matches the ground-truth answer after standard normalization (lowercasing, punctuation and article removal, whitespace normalization).
- **DeepResearcher, SimpleQA, HLE**: Macro-F1 score and Partial Match (PM). PM indicates partial semantic match scores between generated and gold answers, evaluated using **GPT-4o-mini** as the answer evaluator (following DeepResearcher protocol).

**Baseline Categories (Section 5.3):**

The paper compares against three categories of systems:

1. **Prompt-Based Methods**: CoT, CoT + RAG, Search-o1 (Web) — these freeze the LLM and rely on prompting strategies without parameter updates.
2. **Training-Based Methods**: Search-r1-base, Search-r1-instruct, R1-Searcher, DeepResearcher — these update model weights through supervised fine-tuning or reinforcement learning.
3. **Agent Frameworks**: Manus, Aworld, OWL, OpenAI DeepResearch — production agent systems with various architectures.

**Model Configuration:**

- **Planner**: GPT-4.1
- **Executor**: o3 for GAIA, o4-mini for other datasets
- **Image Processing**: GPT-4o
- **Video Agent**: Gemini 2.5 Pro
- **Audio Agent**: Assembly AI
- **Non-parametric CBR**: SimCSE encoding with cosine similarity ranking
- **Parametric CBR**: Two-layer MLP initialized with SimCSE representations

The paper defines three ablation configurations (Section 5.3):
- **Offline Executor**: Static executor without planner, case memory, or external tools — reflects only raw parametric knowledge from LLMs.
- **Online Executor**: Same executor connected to live search and MCP tools — reflects value of real-time retrieval without planning or memory.
- **Memento (w/o CBR)**: Episodic memory disabled — measures gain specifically from case-based reasoning.

### 5.3 Ablation Studies: Isolating Component Contributions

The paper conducts comprehensive ablation studies (Section 5.5) to validate that each architectural component contributes measurably to performance.

#### 5.3.1 Hyperparameter Selection: Optimal K for Case Retrieval

Table 3 evaluates the effect of varying $K$ (number of retrieved cases) on DeepResearcher:

| K | F1 | PM |
|---|----|----|
| 0 (no CBR) | 59.9 | 72.2 |
| 1 | 63.6 | 77.9 |
| 2 | 63.7 | 78.1 |
| **4** | **64.5** | **78.5** |
| 8 | 64.1 | 78.2 |
| 16 | 63.9 | 78.1 |
| 32 | 63.9 | 78.1 |

**Key Finding**: Performance peaks at **$K = 4$** and plateaus or slightly declines for larger $K$. This contradicts few-shot prompting intuition, where more examples often help (Agarwal et al., 2024).

> *"This suggests that CBR benefits from a small, high-quality memory, unlike few-shot prompting, where more examples often help. Careful case selection and memory curation are thus crucial for continual learning"* (Section 5.5.1).

The decline for $K > 4$ suggests **noise introduction**: irrelevant cases dilute the signal from high-utility cases. This validates the paper's emphasis on **learned case selection** (parametric CBR) over simple similarity-based retrieval.

#### 5.3.2 Component-Wise Analysis: Additive Benefits of Tools, Planning, and CBR

Table 5 presents ablation results across three benchmarks (HLE, SimpleQA, DeepResearcher), comparing four configurations:

1. **Offline Executor**: No tools, no planning, no memory
2. **Online Executor**: Tools enabled, no planning, no memory
3. **Memento w/o CBR**: Tools + planning, no case memory
4. **Memento (Full)**: Tools + planning + case memory

**DeepResearcher Results (Table 5c):**
- Offline Executor: 48.8% F1 / 62.8% PM
- Online Executor: 30.8% F1 / 60.7% PM (**−18.0 F1, −2.1 PM** vs. Offline)
- Memento w/o CBR: 59.9% F1 / 72.2% PM (**+29.1 F1, +11.5 PM** vs. Online)
- Memento (Full): 66.6% F1 / 80.4% PM (**+6.7 F1, +8.2 PM** vs. w/o CBR)

**Critical Observation**: Online Executor **underperforms** Offline Executor on DeepResearcher (−18.0 F1). The paper explains: *"we also identify data contamination across the seven evaluated benchmarks, evidenced by a noticeable drop in both F1 and PM when moving from the offline executor to the online executor without planning... This aligns with broader findings in the field: simply using external knowledge can sometimes negatively affect the model, while the internal knowledge within the model plays an important role in QA tasks and can even outperform RAG"* (Section 5.5.2).

This is a **counterintuitive but important finding**: adding tools without proper planning can **hurt** performance due to distraction or contamination. Planning (task decomposition) provides a **+29.1 F1** boost, and CBR provides an additional **+6.7 F1** boost.

**SimpleQA Results (Table 5b):**
- Offline Executor: 19.7% F1 / 21.5% PM
- Online Executor: 48.5% F1 / 84.8% PM (**+28.8 F1, +63.3 PM**)

- Memento (Full): 84.7% F1 / 95.0% PM (**+3.7 F1, +5.3 PM** vs. w/o CBR)

Here, tools provide massive gains (+28.8 F1), planning adds substantial value (+32.5 F1), and CBR provides consistent improvement (+3.7 F1). The pattern differs from DeepResearcher because SimpleQA benefits more from real-time retrieval (factual lookup) without the contamination risk.

**HLE Results (Table 5a):**
- Offline Executor: 6.4% F1 / 8.7% PM
- Online Executor: 11.2% F1 / 15.8% PM (**+4.8 F1, +7.1 PM**)

- Memento (Full): 26.7% F1 / 24.4% PM (**+4.5 F1, +7.0 PM** vs. w/o CBR)

For HLE, the paper notes: *"without sufficient domain knowledge encoded in the backbone model, neither tool usage nor planning alone can reliably produce correct answers on long-tail, expert-level tasks"* (Section 5.5.2). This explains why gains are smaller in absolute terms — the base model's parametric knowledge limits ceiling performance.

### 5.4 Critical Assessment: Do Experiments Support the Claims?

#### Claims and Evidence Mapping

| Claim | Supporting Evidence | Strength |
|-------|--------------------|----------|
| **Memory-based adaptation matches/exceeds fine-tuning** | GAIA Top-1 (87.88%), DeepResearcher 66.6% F1 vs. 51.8% for training-based DeepResearcher | **Strong** — direct comparison with training-based methods |
| **Continual learning without parameter updates** | Figure 1c shows 5-iteration improvement; Table 4 shows parametric CBR gains | **Moderate** — gains are real but diminish after ~3 iterations |
| **OOD generalization through memory** | Figure 1d shows 4.7%–9.6% gains on held-out datasets | **Strong** — consistent across three OOD benchmarks |
| **Efficiency advantage over fine-tuning** | No gradient updates; token costs documented in Figure 6 | **Moderate** — token costs are high for Level 3, but no fine-tuning compute |
| **Fast planner > slow planner** | Table 6 shows 7.88–13.33 point advantages | **Strong** — consistent across model families |

#### Limitations and Caveats

**1. Memory Saturation:** The paper acknowledges that *"with only about 3k training data, the Case Bank saturates quickly"* (Section 5.5.3). This raises questions about scalability: does Memento's advantage persist with 10k, 100k, or 1M experiences? The diminishing returns after 5 iterations suggest **memory curation** (not just growth) becomes critical.

**2. Level 3 GAIA Performance Gap:** While Memento achieves 61.54% on Level 3 validation, there's still a **29.16 point gap** from Level 1 (96.23%). The paper admits *"challenges remain for Level 3 tasks that require extended reasoning horizons and advanced tool coordination"* (Section 5.4). This suggests memory-based adaptation alone cannot solve all long-horizon challenges.

**3. Data Contamination Risk:** The Online Executor underperforming Offline Executor on DeepResearcher (−18.0 F1) reveals that **tool use without planning can hurt performance**. This is a critical caveat: Memento's benefits depend on proper architectural integration, not just adding memory.

**4. Binary Reward Simplification:** The parametric CBR uses binary rewards ($r \in \{0, 1\}$) and cross-entropy loss (Eq. 15). This works for QA tasks but may not generalize to settings with **dense or continuous rewards** (e.g., negotiation, creative tasks).

**5. Model Dependency:** Memento uses GPT-4.1, o3, o4-mini, GPT-4o, and Gemini 2.5 Pro across components. While the paper claims the approach is model-agnostic, performance may depend on **base model capabilities** — particularly for the executor, which must interpret plans and invoke tools correctly.

#### Overall Assessment

The experiments **convincingly support** the paper's core claims:

1. **Memory-based adaptation achieves competitive performance** without fine-tuning — validated by GAIA Top-1 and DeepResearcher outperforming training-based methods.
2. **Continual learning occurs through memory updates** — validated by learning curves in Figure 1c and Table 4.
3. **OOD generalization improves through CBR** — validated by 4.7%–9.6% gains on held-out datasets.
4. **Architectural choices matter** — validated by fast planner > slow planner and planning > no planning ablations.

However, the experiments also reveal **boundary conditions**: memory saturation limits long-term gains, Level 3 tasks remain challenging, and improper tool integration can degrade performance. These are not failures but **important scoping information** for practitioners considering Memento for deployment.

The ablation studies are particularly strong: they isolate the contribution of each component (tools, planning, CBR) and show **additive benefits** rather than redundant mechanisms. The OOD evaluation is also rigorous, using standard held-out dataset protocols rather than ad-hoc splits.

One gap: the paper does not report **wall-clock time** or **dollar costs** for inference, only token counts (Figure 6). For practitioners, knowing that Level 3 tasks consume 121k input tokens is useful, but translating this to latency and cost would strengthen the efficiency argument.

## 6. Limitations and Trade-offs

While Memento demonstrates compelling performance across multiple benchmarks, a critical examination reveals important assumptions, boundary conditions, and unresolved challenges. This section identifies where the approach may struggle, what scenarios remain untested, and what trade-offs practitioners should anticipate when deploying memory-based adaptation.

### 6.1 Memory Saturation and Diminishing Returns

The most significant scalability concern is **memory saturation**. The paper explicitly acknowledges:

> *"With only about 3k training data, the Case Bank saturates quickly. Each additional iteration, therefore, contains progressively fewer previously unseen (and thus potentially failing) cases... adding many more iterations yields diminishing returns"* (Section 5.5.3).

This observation has profound implications:

- **Learning plateaus early**: Figure 1c and Table 4 show accuracy improvements across 5 iterations, but the gains diminish progressively (e.g., parametric CBR improves from 80.46% at Iteration 1 to 85.44% at Iteration 5—a 4.98 point total gain, with most improvement occurring in the first 2–3 iterations).

- **Unknown behavior at scale**: The experiments evaluate on ~3k trajectories. It remains unclear whether Memento's advantages persist with 10k, 100k, or 1M experiences. At what point does retrieval latency outweigh the utility of additional cases? The paper does not address **memory pruning strategies** beyond the design choice to store only final trajectory steps (Section 5.3).

- **The swamping problem resurfaces**: Section 2.3 cites Francis and Ram (1993) on the classic CBR issue where *"retrieval costs outweigh utility"* when cases accumulate without curation. While Memento's learned Q-function mitigates this by prioritizing high-utility cases, the fundamental tension between memory growth and retrieval efficiency remains unresolved.

**Trade-off**: Memento trades parameter updates for memory growth. This avoids catastrophic forgetting but introduces a different bottleneck—**retrieval latency and memory management** become the limiting factors rather than gradient computation.

### 6.2 Performance Gaps on Complex, Long-Horizon Tasks

Despite top-1 GAIA validation performance, there is a **substantial difficulty gradient** that reveals limitations:

| GAIA Level | Validation Accuracy | Steps Required |
|------------|--------------------|----------------|
| Level 1 | 96.23% | ~5 steps, single tool |
| Level 2 | 90.70% | 5–10 steps, multiple tools |
| Level 3 | 61.54% | Up to 50 steps, unrestricted tools |

The **34.69 percentage point drop** from Level 1 to Level 3 indicates that memory-based adaptation alone cannot fully compensate for extreme reasoning horizons. The paper acknowledges:

> *"Challenges remain for Level 3 tasks that require extended reasoning horizons and advanced tool coordination"* (Section 5.4).

**Why this matters:**

- **Error accumulation**: In 50-step trajectories, small planning errors compound. Memory retrieval can guide individual decisions but cannot prevent cascading failures when early subtasks are misidentified.

- **Memory relevance decay**: Cases from simpler tasks may not transfer well to Level 3 complexity. The assumption that *"similar problems should have similar solutions"* (Section 4.2) weakens when task structures diverge significantly.

- **Context window limits**: Figure 6 shows Level 3 tasks consume 121k input tokens—accumulated tool outputs, intermediate reasoning, and retrieved cases. Even with Memento's memory architecture, the **LLM's context window** becomes a bottleneck for extremely long trajectories.

**Unaddressed scenario**: Tasks requiring **hierarchical memory** (e.g., remembering high-level strategies separately from low-level tool sequences) are not evaluated. Memento stores flat case triplets $(s_t, a_t, r_t)$, which may be insufficient for tasks with nested subgoal structures.

### 6.3 Binary Reward Assumption and Limited Reward Structures

The parametric CBR formulation relies on **binary rewards** ($r \in \{0, 1\}$), which simplifies the learning objective to cross-entropy loss (Eq. 15, Section 4.2):

$$
\mathcal{L}(\theta) = \mathbb{E}_{(s, c, r)} \left[ -r \log Q(s, c; \theta) - (1 - r) \log(1 - Q(s, c; \theta)) \right]
$$

**Limitations of this design:**

- **No partial credit**: Tasks with graded outcomes (e.g., "partially correct answer," "efficient but incomplete solution") cannot be naturally represented. The binary signal discards information about *how well* a case performed, only whether it succeeded.

- **Delayed reward attribution**: In multi-step tasks, the reward arrives only at trajectory completion. Memento stores only the final step (Section 5.3), which means **intermediate decisions receive no direct learning signal**. This contrasts with step-level RL methods that can assign credit to individual actions.

- **Domain specificity**: QA and deep research tasks naturally produce binary outcomes (correct/incorrect). However, domains like **negotiation, creative writing, or code debugging** lack clear binary success criteria. The paper does not evaluate Memento on tasks with continuous or multi-dimensional reward signals.

**Open question**: Can Memento's Q-function learning generalize to **dense reward settings** where feedback arrives at each step rather than only at trajectory completion? The single-step simplification (Section 4.2) may not apply.

### 6.4 Tool Integration Risks: When External Knowledge Hurts

A counterintuitive finding from the ablation studies (Table 5c, Section 5.5.2) is that **Online Executor underperforms Offline Executor** on DeepResearcher:

| Configuration | F1 | PM |
|--------------|-----|-----|
| Offline Executor | 48.8% | 62.8% |
| Online Executor | 30.8% | 60.7% |
| **Difference** | **−18.0%** | **−2.1%** |

The paper attributes this to **data contamination**:

> *"We also identify data contamination across the seven evaluated benchmarks, evidenced by a noticeable drop in both F1 and PM when moving from the offline executor to the online executor without planning... simply using external knowledge can sometimes negatively affect the model"* (Section 5.5.2).

**Implications:**

- **Tool use is not universally beneficial**: Adding search and retrieval capabilities can **degrade performance** if the external information conflicts with the model's parametric knowledge or introduces distraction.

- **Planning is essential**: The +29.1 F1 gain from adding planning (Memento w/o CBR vs. Online Executor) shows that **structured task decomposition** is necessary to leverage tools effectively. Memory alone cannot compensate for poor orchestration.

- **Contamination detection is manual**: The paper identifies contamination post-hoc but does not propose automated mechanisms to detect when external retrieval is harmful. Practitioners must manually evaluate whether tool integration helps or hurts for their specific domain.

**Unaddressed edge case**: Scenarios where external information is **outdated, contradictory, or adversarial** (e.g., search results containing misinformation) are not evaluated. Memento assumes retrieved tool outputs are reliable.

### 6.5 Model Dependency and Cost Considerations

While Memento avoids fine-tuning costs, it relies on **multiple high-end proprietary models**:

| Component | Model(s) Used |
|-----------|---------------|
| Planner | GPT-4.1 |
| Executor | o3 (GAIA), o4-mini (other datasets) |
| Image Processing | GPT-4o |
| Video Agent | Gemini 2.5 Pro |
| Audio Agent | Assembly AI |

**Cost implications:**

- **Token consumption scales with difficulty**: Figure 6 (Section 6.2) shows Level 3 tasks consume 121k input tokens vs. 26k for Level 1—a **4.6× increase**. While output tokens remain stable (4.7k → 9.8k), the input burden grows due to accumulated tool outputs and retrieved context.

- **No dollar cost reporting**: The paper reports token counts but does not translate these to **inference latency or monetary cost**. For practitioners, knowing that Level 3 tasks consume 121k tokens is useful, but without pricing information, the efficiency claim ("low-cost continual adaptation," Abstract) cannot be fully evaluated.

- **Model availability**: GPT-4.1, o3, and o4-mini are not universally accessible. The paper does not evaluate Memento with **open-weight models** (e.g., Llama, Qwen) as the base LLM, limiting reproducibility for organizations without API access.

**Trade-off**: Memento eliminates fine-tuning compute but incurs **high inference-time token costs** for complex tasks. The total cost of ownership depends on task distribution—if most tasks are Level 1–2, efficiency gains may outweigh token costs; if Level 3 dominates, inference expenses could exceed fine-tuning costs.

### 6.6 Fast Planner Finding: Architecture-Specific or General?

Table 6 (Section 6.3) shows that **fast planners outperform slow planners**:

| Planner | Executor | Average Accuracy |
|---------|----------|-----------------|
| GPT-4.1 (fast) | o3 | 70.91% |
| o3 (slow) | o3 | 63.03% |
| Qwen3-32B-Fast | o4-mini | 53.94% |
| Qwen3-32B-Slow | o4-mini | 40.61% |

The paper explains that slow planners *"often either answer directly—skipping plan generation altogether—or produce overly verbose plans, which can mislead the executor"* (Section 6.3).

**Limitations of this finding:**

- **Architecture dependency**: This result applies specifically to the **planner–executor separation**. In monolithic agent designs where planning and execution are not decoupled, deliberative reasoning may still provide benefits.

- **Task type dependency**: Fast planning works well for deep research tasks with clear subtask structures. For tasks requiring **creative synthesis or open-ended exploration** (e.g., scientific hypothesis generation), deliberative planning may be necessary.

- **Model-specific behavior**: The finding is based on GPT-4.1 vs. o3 and Qwen3 variants. It is unclear whether this generalizes to other model families or future model generations.

**Open question**: Is there a **sweet spot** for planner capability? The paper compares "fast" vs. "slow" but does not systematically vary planner capacity to identify optimal planner–executor pairings.

### 6.7 Retrieval Hyperparameter Sensitivity

Table 3 (Section 5.5.1) shows performance is sensitive to $K$, the number of retrieved cases:

| K | F1 | PM |
|---|----|-----|
| 0 (no CBR) | 59.9% | 72.2% |
| 1 | 63.6% | 77.9% |
| **4** | **64.5%** | **78.5%** |
| 8 | 64.1% | 78.2% |
| 32 | 63.9% | 78.1% |

**Implications:**

- **Optimal K is small**: Performance peaks at $K = 4$ and declines for larger values. This contradicts few-shot prompting intuition, where more examples often help (Agarwal et al., 2024, cited in Section 5.5.1).

- **Tuning required**: Practitioners must tune $K$ for their specific domain. The optimal value may vary across task types, memory sizes, and retrieval methods.

- **No adaptive K**: Memento uses a fixed $K$ across all queries. An **adaptive retrieval policy** that varies $K$ based on task complexity or confidence could improve performance but is not implemented.

### 6.8 Unaddressed Scenarios and Edge Cases

Several important scenarios are not evaluated in the paper:

| Scenario | Status | Why It Matters |
|----------|--------|----------------|
| **Multi-turn user interaction** | Not evaluated | Real-world agents often engage in extended dialogues with users, requiring memory of conversation history beyond task trajectories. |
| **Adversarial environments** | Not evaluated | Search results or tool outputs may be manipulated or misleading. Memento assumes reliable external information. |
| **Cross-domain transfer** | Partially evaluated | OOD gains (Figure 1d) are measured on QA datasets. Transfer to fundamentally different domains (e.g., code generation → web research) is untested. |
| **Long-term memory decay** | Not addressed | The paper does not implement forgetting mechanisms (unlike Memory-Bank's Ebbinghaus schedule, Section 2.3). Old, irrelevant cases persist indefinitely. |
| **Collaborative multi-agent settings** | Not evaluated | Memento is a single-agent system. How memory sharing works across multiple agents is unexplored. |
| **Real-time latency constraints** | Not reported | Token counts are provided, but wall-clock inference time is not. Applications with strict latency requirements (e.g., customer support) may find Memento too slow. |

### 6.9 Summary of Key Trade-offs

| Trade-off | Benefit | Cost |
|-----------|---------|------|
| **Frozen LLM + Memory** | No catastrophic forgetting, no fine-tuning compute | Retrieval latency, memory management overhead |
| **Learned Q-function** | Better case selection than similarity-only | Requires training data, adds model complexity |
| **Single-step Q-learning** | Training stability, reduced data needs | Cannot capture multi-step credit assignment |
| **Fast planner** | Concise plans, lower token costs | May miss nuanced reasoning opportunities |
| **Store only final trajectory** | Compact memory, slower saturation | Loses intermediate step learning signals |
| **Binary rewards** | Numerically stable training | Cannot represent partial success or graded outcomes |

### 6.10 Open Questions for Future Work

The paper identifies several directions that remain unresolved:

1. **Memory capacity limits**: At what case bank size does retrieval latency outweigh utility? What pruning or compression strategies are effective?

2. **Dense reward generalization**: Can Memento's Q-learning framework extend to tasks with continuous or multi-dimensional rewards?

3. **Hierarchical memory**: Would storing cases at multiple abstraction levels (strategies, subroutines, primitive actions) improve performance on Level 3 tasks?

4. **Automated contamination detection**: Can the system learn to identify when external retrieval is harmful vs. helpful?

5. **Open-weight model evaluation**: Does Memento's advantage persist with fully open-source model stacks, or does it depend on proprietary model capabilities?

6. **Cost-benefit analysis**: What is the total cost of ownership (inference tokens + memory storage) compared to fine-tuning approaches over extended deployment periods?

These limitations do not invalidate Memento's contributions but rather **scope its applicability**. The approach excels in deep research scenarios with binary outcomes, moderate trajectory lengths, and reliable tool access. Practitioners should carefully evaluate whether their use case aligns with these conditions before adoption.

## 7. Implications and Future Directions

This section examines how Memento reshapes the landscape of LLM agent development, identifies concrete research directions it enables, outlines practical deployment scenarios, and provides actionable guidance for practitioners deciding whether to adopt this approach.

### 7.1 Landscape Impact: Redefining What "Learning" Means for LLM Agents

Memento's core contribution extends beyond performance metrics—it challenges a foundational assumption in the field: **that agent improvement requires parameter updates**. This paradigm shift has cascading implications for how we design, evaluate, and deploy LLM agents.

**Theoretical Reframing:**

Prior work implicitly treats the LLM's parameter space as the primary locus of adaptation. Methods like Search-r1 (Jin et al., 2025) and DeepResearcher (Zheng et al., 2025) operate under the assumption that *"achieving the capability to solve complex reasoning problems requires substantial changes to the model's parameters"* (Section 2.1). Memento demonstrates this assumption is **not necessary** for strong performance.

By formalizing adaptation as learning a retrieval policy $\mu(c \mid s, M)$ over external memory while keeping $p_{\text{LLM}}$ frozen (Eq. 1, Section 3), the paper establishes that:

- **Learning can occur in memory space rather than parameter space**. This decouples capability improvement from gradient computation.
- **The optimal policy is derivable** (Eq. 7, Section 3) through maximum entropy RL, providing theoretical grounding absent in heuristic memory systems.
- **Continual adaptation becomes an inference-time capability** rather than a training-time concern—each completed task immediately enriches the case bank (Eq. 12, Section 4.2).

**Field-Wide Consequences:**

| Dimension | Before Memento | After Memento |
|-----------|---------------|---------------|
| **Adaptation Mechanism** | Fine-tuning, RLHF, SFT | Memory-based policy learning |
| **Compute Bottleneck** | GPU-intensive gradient updates | Memory retrieval and case selection |
| **Forgetting Risk** | Catastrophic forgetting during parameter updates | No forgetting (base model frozen) |
| **Deployment Model** | Train → Deploy → Static | Deploy → Learn Continuously |
| **Evaluation Focus** | In-distribution performance | OOD generalization through memory |

This reframing positions Memento alongside other **non-parametric adaptation** approaches (e.g., RAG, prompt engineering) but distinguishes itself through **learned retrieval** rather than static retrieval. The 87.88% GAIA validation score (Table 2, Section 5.4) proves this is not a compromise—it is a **competitive alternative** to parameter-based methods.

**Connection to Broader ML Trends:**

Memento aligns with several emerging directions in machine learning:

1. **Test-time computation**: Like chain-of-thought prompting and self-consistency methods, Memento shifts computational burden from training to inference. However, unlike these methods, Memento's computation **accumulates value over time** through memory growth.

2. **Modular architectures**: The planner–executor separation (Figure 3, Section 4.1) reflects a broader trend toward decomposing monolithic models into specialized components. The finding that **fast planners outperform slow planners** (Table 6, Section 6.3) provides empirical guidance for this architectural pattern.

3. **Memory-augmented neural networks**: Memento builds on classical work like Neural Episodic Control (Pritzel et al., 2017) but adapts it to the transformer era. The kernel-based Q-approximation (Eq. 9, Section 3) shows how classical RL techniques can be revitalized for LLM agents.

### 7.2 Research Directions Enabled

Memento opens several concrete avenues for follow-up research. Each direction addresses a limitation identified in Section 6 or extends a strength demonstrated in Section 5.

**7.2.1 Memory Architecture Innovations**

The current implementation uses a **flat case bank** storing triplets $(s_t, a_t, r_t)$. Future work could explore:

- **Hierarchical memory structures**: Store cases at multiple abstraction levels (strategies, subroutines, primitive actions). This could address the Level 3 GAIA performance gap (61.54% vs. 96.23% for Level 1, Table 2) by enabling transfer of high-level patterns independent of low-level tool sequences.

- **Compressed representations**: Instead of storing raw trajectories, distill cases into **abstracted schemas** (e.g., "when task requires X, try approach Y"). This could mitigate memory saturation (Section 6.1) while preserving transferable knowledge.

- **Forgetting mechanisms**: The paper does not implement memory decay (unlike Memory-Bank's Ebbinghaus schedule, Section 2.3). Research on **adaptive forgetting**—removing low-utility cases while retaining high-utility ones—could improve long-term scalability.

**7.2.2 Reward Structure Generalization**

The parametric CBR uses **binary rewards** ($r \in \{0, 1\}$) with cross-entropy loss (Eq. 15, Section 4.2). Extensions could address:

- **Dense reward signals**: Tasks like code generation, negotiation, or creative writing lack clear binary outcomes. Learning Q-functions with **continuous or multi-dimensional rewards** would broaden applicability.

- **Step-level credit assignment**: Currently, only final trajectory outcomes are stored (Section 5.3). Incorporating **intermediate rewards** could improve learning efficiency for long-horizon tasks where early decisions disproportionately affect outcomes.

- **Inverse reward design**: Instead of assuming rewards are given, learn reward functions from human feedback or implicit signals (e.g., user satisfaction, task completion time).

**7.2.3 Multi-Agent and Collaborative Extensions**

Memento is evaluated as a **single-agent system**. Natural extensions include:

- **Shared case banks**: Multiple agents could contribute to and retrieve from a common memory, enabling **collective learning**. This raises questions about case attribution, conflict resolution, and privacy.

- **Specialized memory pools**: Different agents could maintain domain-specific case banks (e.g., one for web research, one for code execution) with a meta-retrieval policy selecting which pool to query.

- **Adversarial memory**: In multi-agent settings, agents could learn to **identify and avoid cases** that led to failures in competitive environments (e.g., negotiation, game playing).

**7.2.4 Theoretical Analysis Opportunities**

The M-MDP formalization (Definition 3.1, Section 3) enables rigorous analysis:

- **Convergence guarantees**: Under what conditions does the learned retrieval policy $\mu$ converge to $\mu^*$? How does memory growth affect convergence rates?

- **Sample complexity bounds**: How many cases are needed to achieve a target performance level? The observation that *"the Case Bank saturates quickly"* with ~3k trajectories (Section 5.5.3) suggests favorable sample efficiency, but formal bounds are absent.

- **Generalization theory**: Why does CBR improve OOD performance (4.7%–9.6% gains, Figure 1d)? A theoretical account of **when and why** retrieved cases transfer across domains would strengthen the approach.

**7.2.5 Open-Weight Model Evaluation**

All experiments use **proprietary models** (GPT-4.1, o3, o4-mini, GPT-4o, Gemini 2.5 Pro, Section 5.3). Critical open questions:

- Does Memento's advantage persist with **fully open-source stacks** (e.g., Llama 3 + open tools)?
- How does base model capability affect the **planner–executor pairing**? The fast planner finding (Table 6) may be model-specific.
- Can smaller models (7B–13B parameters) achieve competitive performance with Memento's memory augmentation, reducing inference costs?

### 7.3 Practical Applications and Downstream Use Cases

Memento's architecture is well-suited for specific deployment scenarios. Understanding where it excels—and where it may not—helps practitioners make informed adoption decisions.

**7.3.1 Deep Research and Information Synthesis**

**Best fit**: Tasks requiring multi-step information gathering, cross-source synthesis, and tool orchestration.

- **Academic literature review**: Agents can accumulate cases from prior review tasks, learning which search strategies yield relevant papers for different query types.
- **Market intelligence**: Memory of successful data collection pipelines (e.g., "for competitor analysis, start with SEC filings, then news articles") accelerates future research.
- **Due diligence investigations**: Cases encoding successful verification workflows transfer across domains (financial, legal, technical).

**Why Memento excels**: The 66.6% F1 on DeepResearcher (Table 1, Section 5.4) demonstrates strong performance on real-time web research. The OOD gains (4.7%–9.6%, Figure 1d) suggest transferability across research domains.

**7.3.2 Enterprise Knowledge Work Automation**

**Best fit**: Repetitive knowledge tasks with clear success criteria and access to internal tools.

- **Customer support escalation**: Cases encode successful resolution paths for different issue types. The binary reward structure aligns well with ticket resolution (solved/unsolved).
- **Internal IT operations**: Memory of successful troubleshooting sequences (e.g., "for database connectivity issues, check X, then Y, then Z") reduces mean time to resolution.
- **Compliance auditing**: Cases capture successful audit workflows, ensuring consistency across auditors and reducing reliance on individual expertise.

**Why Memento excels**: Freezing the base model avoids **data privacy concerns** associated with fine-tuning on sensitive internal data. Memory can be scoped to organizational boundaries without modifying model weights.

**7.3.3 Educational Tutoring Systems**

**Best fit**: Adaptive tutoring where student interactions generate learning signals.

- **Personalized learning paths**: Cases encode successful tutoring strategies for different student profiles (e.g., "for visual learners struggling with algebra, use diagram-based explanations").
- **Error pattern recognition**: Memory of common misconceptions and effective corrections enables proactive intervention.
- **Curriculum adaptation**: Cases track which instructional sequences lead to mastery for different topic areas.

**Why Memento excels**: The continual learning capability allows the system to **improve with each student interaction** without requiring periodic retraining. Binary rewards (concept mastered/not mastered) align with the parametric CBR formulation.

**7.3.4 Scenarios Where Memento May Not Be Ideal**

Practitioners should consider alternatives when:

| Scenario | Concern | Alternative Approach |
|----------|---------|---------------------|
| **Real-time latency constraints** | Memory retrieval adds inference overhead (121k input tokens for Level 3 GAIA, Figure 6) | Direct fine-tuning for low-latency inference |
| **Continuous reward signals** | Binary rewards limit expressiveness (Section 6.3) | RLHF or direct preference optimization |
| **Extremely long-horizon tasks** | Level 3 GAIA gap (34.69 points from Level 1, Table 2) suggests limits | Hierarchical planning with explicit subgoal memory |
| **Adversarial environments** | Assumes reliable tool outputs (Section 6.4) | Robust RAG with source verification |
| **Limited memory infrastructure** | Case bank growth requires storage and retrieval systems | Prompt-based methods with fixed context |

### 7.4 Integration Guidance: Decision Framework for Practitioners

This subsection provides actionable criteria for deciding whether to adopt Memento versus alternative approaches.

**7.4.1 When to Prefer Memento**

Choose Memento when **all** of the following conditions hold:

1. **Task structure is episodic**: Tasks have clear boundaries with definable success/failure outcomes. The binary reward formulation (Eq. 15) works best when outcomes are unambiguous.

2. **Continual adaptation is required**: The deployment environment changes over time, requiring the agent to learn from new experiences post-deployment. Memento's online memory updates (Eq. 12) enable this without retraining.

3. **Base model capabilities are sufficient**: The frozen LLM must have adequate reasoning and tool-use capabilities. Memento augments but does not replace base model competence—HLE performance (24.4% PM, Figure 4) is bounded by the backbone model's knowledge.

4. **Memory infrastructure is available**: Organizations must support case bank storage, retrieval, and (for parametric CBR) Q-function training. The ~3k case saturation point (Section 5.5.3) suggests modest initial requirements.

5. **OOD generalization is valued**: If test distributions differ from training distributions, Memento's 4.7%–9.6% OOD gains (Figure 1d) provide measurable advantage over fine-tuning approaches that may overfit.

**7.4.2 When to Consider Alternatives**

| Alternative | Prefer When | Trade-off |
|-------------|-------------|-----------|
| **Fine-tuning (SFT/RL)** | Static deployment environment, abundant labeled data, no OOD concerns | Higher upfront cost, risk of catastrophic forgetting |
| **Standard RAG** | Static knowledge domains, no interaction trajectories available | No continual learning, limited to document retrieval |
| **Prompt engineering only** | Extremely low-latency requirements, minimal infrastructure | No adaptation, performance ceiling fixed by base model |
| **Multi-agent orchestration** | Tasks requiring specialized expertise across domains | Higher coordination overhead, potential role confusion (Section 6.3) |

**7.4.3 Deployment Checklist**

Before deploying Memento in production:

- [ ] **Validate reward structure**: Ensure tasks produce clear success/failure signals. If rewards are ambiguous, consider human-in-the-loop labeling.
- [ ] **Tune retrieval hyperparameters**: Table 3 (Section 5.5.1) shows $K = 4$ is optimal for DeepResearcher, but this may vary by domain. Run ablation studies on your specific task distribution.
- [ ] **Monitor memory growth**: Track case bank size and retrieval latency. Implement pruning strategies if saturation occurs (Section 6.1).
- [ ] **Evaluate planner–executor pairing**: Table 6 (Section 6.3) shows fast planners outperform slow ones, but this is model-dependent. Test multiple configurations.
- [ ] **Assess contamination risk**: Section 5.5.2 found Online Executor underperformed Offline Executor on DeepResearcher due to data contamination. Validate that external tools improve rather than degrade performance.
- [ ] **Plan for Level 3 tasks**: If long-horizon tasks (>20 steps) are common, consider hierarchical memory extensions beyond the current flat case bank design.

**7.4.4 Cost-Benefit Considerations**

While Memento eliminates fine-tuning costs, practitioners should evaluate **total cost of ownership**:

| Cost Component | Memento | Fine-tuning Alternative |
|----------------|---------|------------------------|
| **Initial setup** | Memory infrastructure, Q-function training | GPU clusters, labeled data collection |
| **Ongoing compute** | Inference tokens (121k for Level 3, Figure 6) | Inference tokens (similar) |
| **Adaptation cycles** | Online (per-task) | Batch retraining (hours to days) |
| **Storage** | Case bank (~3k cases saturates, Section 5.5.3) | Model checkpoints (multiple versions) |
| **Engineering overhead** | Memory management, retrieval optimization | Training pipeline maintenance |

For organizations with **frequent task distribution shifts**, Memento's online adaptation likely provides better ROI despite higher per-inference token costs. For **stable deployments**, fine-tuning may be more cost-effective long-term.

### 7.5 Summary: Memento's Position in the Agent Development Landscape

Memento establishes that **memory-based adaptation is a viable alternative to parameter-based learning** for LLM agents. The empirical results—top-1 GAIA validation (87.88%), 66.6% F1 on DeepResearcher, 4.7%–9.6% OOD gains—demonstrate competitive performance without gradient updates.

**Key takeaways for the field:**

1. **Learning need not traverse parameter space**. The M-MDP formalization (Definition 3.1) and derived optimal policy (Eq. 7) provide theoretical grounding for memory-based adaptation.

2. **Architectural choices matter**. Fast planners outperform slow planners (Table 6), planning is essential for effective tool use (Table 5), and small curated memories outperform large uncurated ones (Table 3).

3. **Continual learning is achievable at inference time**. The 5-iteration learning curves (Figure 1c, Table 4) show measurable improvement without retraining, though diminishing returns suggest memory curation becomes critical.

4. **OOD generalization improves through CBR**. The consistent 4.7%–9.6% gains on held-out datasets (Figure 1d) validate that retrieved cases capture transferable structure beyond surface similarity.

**Open challenges remain**: memory saturation at ~3k cases, Level 3 long-horizon gaps, binary reward limitations, and proprietary model dependency. Addressing these limitations constitutes a rich research agenda that extends Memento's contributions while acknowledging its boundary conditions.

For practitioners, the decision framework in Section 7.4 provides concrete guidance: Memento excels in episodic, continually-adapting environments with clear success signals and available memory infrastructure. For other scenarios, alternative approaches may be more appropriate.

The broader implication is that **the agent development landscape is expanding**. Rather than a binary choice between static prompting and expensive fine-tuning, Memento demonstrates a third path: **frozen models with learnable memory**. This paradigm shift enables new deployment models, reduces barriers to adaptive agent development, and opens theoretical questions about the nature of learning in language-based agents.
