## 1. Executive Summary

PATHWAYS presents a new large-scale orchestration layer using asynchronous distributed dataflow that enables flexible programming models like pipelining and multi-tenancy while retaining state-of-the-art performance for current SPMD models. It solves the trade-off between the flexibility of single-controller systems and the performance of multi-controller architectures, demonstrating ~100% accelerator utilization on 2048 TPUs and matching JAX throughput (e.g., 84.8k tokens/s for T5-11B) even when pipelining Transformer models across 16 stages or multiple TPU islands. This capability is significant because it supports the heterogeneous computation and resource sharing required for future foundation models without compromising efficiency on existing workloads.

## 2. Context and Motivation

### The Core Tension: Flexibility vs. Performance in Distributed ML

The PATHWAYS paper addresses a fundamental architectural dilemma in distributed machine learning systems: **how to support flexible, heterogeneous computation patterns without sacrificing the raw performance needed for large-scale training**. This tension arises from the co-evolution of ML models, accelerator hardware, and the software systems connecting them (Section 1). As models grow more complex, systems risk becoming over-specialized to current workloads and failing to anticipate future needs.

To understand why this matters, we must first distinguish between two dominant system architectures:

**Multi-controller systems** (e.g., JAX, PyTorch, MPI-style configurations) run identical copies of user code directly on each accelerator host. As shown in **Figure 1a**, dispatch happens over fast PCIe links, achieving low latency. However, this architecture assumes exclusive ownership of hardware resources and requires all accelerators to run the same computation in lockstep using collectives like AllReduce.

**Single-controller systems** (e.g., TensorFlow v1) use a central coordinator that builds a computation graph and partitions it across workers. As shown in **Figure 1b**, this offers flexibility but introduces higher dispatch latency because control messages traverse the slower datacenter network (DCN) rather than PCIe—typically an order of magnitude slower.

### Why This Problem Matters: Three Converging Trends

The paper identifies three specific trends that make the flexibility-performance trade-off increasingly urgent:

**1. Models are outgrowing SPMD.** Very large language models now require pipelining rather than pure data-parallelism (Section 1, citing Narayanan et al., 2019; Rasley et al., 2020). Models like Mixture of Experts (MoE) explore computational sparsity that is "most naturally expressed using fine-grain control flow and heterogeneous computation across accelerators." The MPI-style programming model is "too restrictive both for users and for the underlying system" for these patterns.

**2. Hardware is becoming heterogeneous.** ML clusters increasingly consist of multiple "islands" of accelerators connected over datacenter networks rather than single homogeneous pods. Providing exclusive access to large islands is "expensive, and often wasteful as a single user program must try to keep all of the accelerators continuously busy" (Section 1). This drives researchers toward MPMD (multiple program multiple data) computations that map sub-parts of computation to smaller, more readily available accelerator islands.

**3. Foundation models enable resource multiplexing.** Researchers are standardizing on foundation models that are "trained on large data at scale and are adaptable to multiple downstream tasks" (Section 1). This creates opportunities for improving cluster utilization by multiplexing resources across many tasks—for example, several researchers concurrently fine-tuning a foundation model for different tasks while sharing the fixed foundation model layers on the same accelerators.

### Where Prior Approaches Fall Short

The paper systematically analyzes why existing systems cannot simultaneously address all three trends:

**Multi-controller limitations (Section 2):**
- Any communication beyond standard collectives requires users to implement their own coordination primitives
- Exclusive ownership of hardware resources shifts the responsibility of ensuring high utilization onto the user
- Complicates resource virtualization and multiplexing needed for efficient cluster-wide infrastructure
- Cannot scale beyond a single TPU pod because XLA collectives are only available over ICI (inter-chip interconnect), not DCN

**TensorFlow v1 single-controller limitations (Section 2):**
- Dispatch latency accumulates in programs with many cross-host transfers (e.g., pipelined models with many stages), "leading to inefficient accelerator utilization"
- Lack of centralized scheduler makes it "impossible to ensure consistent ordering between computations across programs" (critical for gang-scheduling on TPUs)
- Materializes the full sharded computation graph, introducing "substantial overhead in both graph serialization and execution when the number of shards reaches into the thousands, leading to millions of graph edges"
- Implementation was "over-specialized to assume a single, smallish, exclusively-owned island of accelerators"

**Resource management systems (Section 7):**
Conventional cluster schedulers optimize for heterogeneity and fairness, but "resources are still exclusively dedicated to single jobs at long time scales (seconds or more)." Recent fine-grained sharing techniques (virtualizing accelerators, GPU memory virtualization, concurrent task execution) "demonstrate opportunities for sharing accelerators that are hard to capitalize on at scale without a single-controller system like PATHWAYS."

### PATHWAYS's Positioning: Best of Both Worlds

PATHWAYS positions itself as combining "the flexibility of single-controller frameworks with the performance of multi-controllers" (Section 2). The key design choices that enable this are:

| Aspect | Multi-controller (JAX) | Single-controller (TF v1) | PATHWAYS |
|--------|----------------------|--------------------------|----------|
| Dispatch latency | Low (PCIe) | High (DCN) | Low (parallel async dispatch) |
| Programming model | Restrictive (SPMD) | Flexible | Flexible |
| Resource management | Exclusive ownership | Limited virtualization | Centralized, virtualized |
| Cross-pod scaling | No | Inefficient | Yes (efficient) |
| Gang-scheduling | Implicit | Per-program only | Cross-program |

The paper's core innovation is an **asynchronous distributed dataflow design that lets the control plane execute in parallel despite dependencies in the data plane** (Abstract). This enables a single-controller model that "makes it easier to express complex new parallelism patterns" while achieving "performance parity (~100% accelerator utilization) with state-of-the-art systems when running SPMD computations over 2048 TPUs."

### Critical Design Insight: Regular Compiled Functions

A subtle but crucial enabler for PATHWAYS's approach is the observation that most ML computations are "regular compiled functions" with known resource requirements before execution (Section 3, Appendix B). These functions have:
- Known input/output types and shapes
- Bounded loops (or specified maximum trip counts)
- Functional conditionals where both branches have the same output type

This regularity allows PATHWAYS to perform **parallel asynchronous dispatch** (Section 4.5): since a successor node's input shapes can be computed before the predecessor computation completes, most host-side work can run in parallel rather than serially. **Figure 4** illustrates this: sequential dispatch causes stalls when computation time is shorter than host-side coordination time, while parallel dispatch overcomes this bottleneck.

The paper acknowledges this is an optimization that "falls back to the traditional model when a node's resource requirements are not known until a predecessor computation has completed (e.g., due to data-dependent control flow)" (Section 4.5). This fallback capability is important for future work on data-dependent vectorized control flow (Section 6.3).

### What This Means for Practitioners

For someone building or deploying ML systems, the motivation translates to concrete capabilities:

1. **Scale beyond pod boundaries:** JAX "cannot scale beyond a single TPU pod" natively, but PATHWAYS "allows JAX programs to scale for the first time to multiple TPU pods, containing many thousands of TPU cores" (Section 3).

2. **Express non-SPMD patterns naturally:** Pipelined models can be expressed without awkward workarounds. Table 2 shows PATHWAYS achieves 131.4k tokens/sec with 16 pipeline stages—competitive with SPMD's 125.7k tokens/sec on the same 128 cores.

3. **Share resources efficiently:** Figure 8 demonstrates PATHWAYS time-multiplexes accelerators between concurrent programs "incurring no overhead to context switch," with aggregate throughput matching or exceeding JAX depending on computation size.

4. **Maintain performance on existing workloads:** Table 1 shows identical throughput between JAX and PATHWAYS for T5 models up to 11B parameters (e.g., 84.8k tokens/s on 512 cores for T5-11B), proving the single-controller overhead is masked for realistic computations.

The paper's claim is not that PATHWAYS enables entirely new model classes, but that it "matches the functionality and performance of state of the art ML systems, while providing the capabilities needed to support future ML workloads" (Section 1). This positioning is critical: rather than requiring researchers to choose between flexibility and performance, PATHWAYS aims to deliver both simultaneously through careful systems engineering.

## 3. Technical Approach

This is a systems paper presenting a new orchestration layer for distributed machine learning. The core idea is an **asynchronous distributed dataflow architecture** that enables a single-controller programming model while matching the performance of multi-controller systems through parallel dispatch and sharded graph representation.

### 3.1 Reader orientation (approachable technical breakdown)

PATHWAYS is a centralized coordination system that manages thousands of accelerator devices across multiple network-connected "islands" and schedules computation graphs on behalf of many concurrent clients. It solves the problem that single-controller systems traditionally suffer from high dispatch latency over the datacenter network by exploiting the fact that most ML computations have known resource requirements ahead of time, allowing host-side coordination work to execute in parallel rather than waiting for each computation to complete before scheduling the next.

### 3.2 Big-picture architecture (diagram in words)

The system consists of six major components that work together to execute distributed ML programs:

- **Client Library** (runs on user machines): Accepts Python code (JAX or TensorFlow), traces compiled functions, constructs a device location-agnostic intermediate representation (IR), and submits programs to the coordinator via RPC.

- **Resource Manager** (global, centralized): Maintains a registry of all available physical devices across all islands, allocates "virtual slices" of accelerators to clients based on their topology requirements, and tracks device availability dynamically.

- **PLAQUE Coordination Substrate** (distributed across hosts): A sharded dataflow engine that handles all cross-host communication over the datacenter network (DCN), passing data handles and scheduling messages between islands.

- **Per-Island Scheduler** (one per accelerator island): Receives computation requests from PLAQUE, determines a consistent execution order for all computations on that island (gang-scheduling), and enqueues work to executors at millisecond timescales.

- **Per-Device Executor** (one per accelerator): Receives scheduled computations from the island scheduler, manages local HBM buffers, and dispatches compiled functions to the physical TPU cores.

- **Sharded Object Store** (one per host): Tracks buffers held in both host DRAM and accelerator HBM, enables remote references via opaque handles, and supports garbage collection and migration if needed.

Information flows as follows: The client traces Python code into a PATHWAYS IR, the Resource Manager assigns virtual devices, the IR is lowered to include physical device locations, PLAQUE coordinates cross-island data transfers, per-island schedulers order computations consistently, and executors run the actual accelerator kernels while the object store manages intermediate data.

### 3.3 Roadmap for the deep dive

- **Compiled functions and the programming model (Section 3)**: We start here because understanding what PATHWAYS executes (regular compiled functions with known shapes) explains why parallel dispatch is possible.

- **Resource management and virtual devices (Section 4.1)**: Next we explain how physical accelerators are abstracted, since this enables the single-controller model to manage heterogeneous islands.

- **Client IR construction and lowering (Section 4.2)**: We then trace how user code becomes an executable program, including the sharded buffer abstraction that scales the client.

- **PLAQUE coordination substrate (Section 4.3)**: We explain the distributed dataflow engine that handles cross-host messaging, since this is the backbone of the control plane.

- **Gang-scheduling and parallel async dispatch (Sections 4.4–4.5)**: We dive into the two key scheduling mechanisms that eliminate single-controller overhead, as this is the core performance innovation.

- **Data management and object stores (Section 4.6)**: Finally we cover how intermediate values are stored and tracked, since this completes the execution picture.

### 3.4 Detailed, sentence-based technical breakdown

#### 3.4.1 The Programming Model: Compiled Functions as Atomic Units

PATHWAYS operates on a fundamental abstraction called **compiled functions**, which are sub-computations with statically known resource requirements before execution begins. According to Appendix B, a compiled function must satisfy three constraints: input and output types and tensor shapes are known before input data is computed, loop bounds are either known at scheduling time or specified as a maximum trip count with potential early termination, and conditionals are "functional" where both branches have the same output type with resources allocated in advance sufficient for either branch. This regularity is not an arbitrary restriction but reflects how modern ML frameworks already operate: JAX users wrap standard Python code with decorators to indicate fragments that should be compiled into XLA computations (Section 3), and these XLA computations are "characterized by known input and output types and shapes, bounded loops, and with few (if any) conditionals."

Each compiled function maps to a **single sharded computation node** in the PATHWAYS dataflow graph, regardless of how many accelerator shards execute it. This is a critical design choice that distinguishes PATHWAYS from TensorFlow v1, which "materializes the full sharded computation graph" leading to "millions of graph edges between sub-computations" when scaling to thousands of shards (Section 2). In PATHWAYS's representation, a chained execution of two computations $A$ and $B$ with $N$ computation shards each requires only 4 nodes in the dataflow graph: $\text{Arg} \rightarrow \text{Compute}(A) \rightarrow \text{Compute}(B) \rightarrow \text{Result}$, regardless of the choice of $N$ (Section 4.3). The PLAQUE runtime implementation handles the sharding internally: each node generates output data tuples tagged with a destination shard, so when performing data-parallel execution, $N$ data tuples flow between each adjacent pair of IR nodes.

The programming model supports two modes of operation. In the default mode, each compiled function becomes a standalone PATHWAYS program containing just one sharded computation, meaning "if a user wants to run many functions back to back, a separate Python call and RPC from client to coordinator is required for each function" (Section 3). To avoid this overhead, PATHWAYS implements a **program tracer** (shown in Figure 2) that a user can wrap around a block of Python code calling many compiled functions; the tracer generates a single PATHWAYS program where each compiled function is represented by a computation node in a dataflow graph. This tracing capability is essential for expressing pipelined models: the authors note they "have written a library that automatically converts a FLAX model into a pipelined PATHWAYS program" (Section 3).

PATHWAYS serves as a "plug-in replacement for the JAX backend" (Section 3), allowing unmodified JAX code to run with access to "not just to the locally connected TPU cores, but to as many cores as are provisioned in the system." This is significant because JAX "cannot scale beyond a single TPU pod" natively, since JAX programs running in multi-controller configurations "transfer all data using XLA collectives, and these are only currently available over ICI on TPU" (Section 3). PATHWAYS can communicate over both ICI (inter-chip interconnect within a pod) and DCN (datacenter network between pods), enabling JAX programs "to scale for the first time to multiple TPU pods, containing many thousands of TPU cores" (Section 3).

#### 3.4.2 Resource Management: Virtual Devices and Physical Allocation

The Resource Manager (Section 4.1) implements a layer of indirection between **virtual devices** that clients request and **physical devices** that actually execute computations. A PATHWAYS backend consists of accelerators grouped into "tightly-coupled islands that are in turn connected to each other over DCN" (Figure 3). Clients may ask for "virtual slices" of an island with specific 2D or 3D mesh shapes that suit their communication pattern, and each virtual slice contains virtual devices that allow the client to express how computations are laid out on the mesh. The Resource Manager dynamically assigns physical devices for virtual devices, satisfying constraints like desired interconnect topology and memory capacity.

The initial implementation uses a simple heuristic: it "attempts to statically balance load by spreading computations across all available devices, and keeps a one to one mapping between virtual and physical devices" (Section 4.1). This one-to-one mapping is important for performance predictability, though the paper notes that "if future workloads require it we can adopt a more sophisticated allocation algorithm, for example taking into account the resource requirements of all client computations and the current state of the system to approximate an optimal allocation of physical devices to computations" (Section 4.1). The layer of indirection enables future features like "transparent suspend/resume and migration, where a client's virtual devices are temporarily reclaimed or reassigned without the need for cooperation from the user program" (Section 4.1).

PATHWAYS allows backend compute resources to be added and removed dynamically, with the Resource Manager tracking available devices (Section 4.1). This dynamic capability is essential for multi-tenancy: Figure 9 shows traces where PATHWAYS performs "gang-scheduling of programs submitted by 4 independent clients while controlling allocation of accelerator time for fairness," with the scheduler able to "enforce proportional share in this multi-tenancy setting" with ratios like 1:1:1:1 or 1:2:4:8 between clients (Section 5.2).

#### 3.4.3 Client-Side IR Construction and Sharded Buffer Abstraction

When a user wants to run a traced program, the PATHWAYS client library performs several steps in sequence (Section 4.2). First, it "assigns virtual devices to any computations that have not been run before, and registers the computations with the resource manager, triggering the servers to compile the computations in the background." Second, the client "constructs a device location-agnostic PATHWAYS intermediate representation (IR) for the program, expressed as a custom MLIR dialect." Third, the IR is progressively lowered via a series of standard compiler passes, which eventually output a low-level representation that includes the physical device locations. This low-level program "takes into account the network connectivity between physical devices and includes operations to transfer outputs from a source computation shard to the locations of its destination shards, including scatter and gather operations when a data exchange is required" (Section 4.2).

The client uses a **sharded buffer abstraction** to represent a logical buffer that may be distributed over multiple devices (Section 4.2). This abstraction is critical for scaling: it "helps the client scale by amortizing the cost of bookkeeping tasks (including reference counting) at the granularity of logical buffers instead of individual shards" (Section 4.2). Without this abstraction, the client would need to track millions of individual shard buffers when running on thousands of accelerators, becoming a performance bottleneck as it "coordinates thousands of individual computations and data buffers corresponding to each shard of computations spread across thousands of accelerators" (Section 4.2).

It is efficient to repeatedly run the low-level program "in the common case that the virtual device locations do not change, and the program can be re-lowered if the resource manager changes the mapping between virtual and physical devices" (Section 4.2). This re-lowering capability enables the dynamic resource management described above while maintaining performance for stable allocations.

#### 3.4.4 PLAQUE: The Coordination Substrate

PATHWAYS relies on **PLAQUE**, an existing closed-source production sharded dataflow system used at Google, for all cross-host coordination that uses DCN (Section 4.3). The low-level PATHWAYS IR is converted directly to a PLAQUE program, represented as a dataflow graph. PLAQUE meets four stringent requirements that PATHWAYS needs:

**First**, the representation must contain a single node for each sharded computation to ensure compact representation for computations spanning many shards (Section 4.3). As noted above, this means $N$ shards do not require $N$ graph nodes.

**Second**, the coordination runtime must support **sparse data exchanges** along sharded edges, in which messages can be sent between a dynamically chosen subset of shards (Section 4.3). This uses "standard progress tracking mechanisms to detect when all messages for a shard have been received" (citing Akidau et al., 2013; Murray et al., 2013). Efficient sparse communication is "a requirement to avoid the DCN becoming a bottleneck for data-dependent control flow on accelerators, which is one of the key capabilities that we want PATHWAYS to enable" (Section 4.3).

**Third**, the coordination substrate sends DCN messages that are "in the critical path for transmitting scheduling messages and data handles" (Figure 4), so it must send critical messages with low latency and batch messages destined for the same host when high throughput is required (Section 4.3).

**Fourth**, using an extensible general-purpose dataflow engine means PATHWAYS can also use PLAQUE for "background housekeeping tasks such as distributing configuration information, monitoring programs, cleaning them up, delivering errors on failures, and so on" (Section 4.3).

The paper notes that "it would be feasible to re-implement the full PATHWAYS design using other distributed frameworks such as Ray rather than PLAQUE" (Section 4.3), though some additions might be required to attain comparable performance because Ray lacks features like an HBM object store or primitives to efficiently transfer remote objects over the GPU interconnect.

#### 3.4.5 Gang-Scheduling: Consistent Ordering Across Programs

As discussed in Section 2, supporting SPMD computations on a shared set of accelerators requires efficient **gang-scheduling**, where communicating computations are enqueued in a consistent order across all devices. This is essential for TPUs because they are "single-threaded and only run non-preemptible kernels, so the system will deadlock if communicating computations are not enqueued in a consistent order" (Section 2). Even for GPUs that can execute concurrent computations, gang-scheduling "allows more efficient execution of collectives" (citing Feitelson and Rudolph, 1992).

PATHWAYS includes a **centralized scheduler per island** that consistently orders all computations in the island (Section 4.4). As PATHWAYS enqueues a program for execution, the PLAQUE dataflow program is responsible for three tasks: (i) enqueueing the execution of local compiled functions at each accelerator with buffer futures as inputs, (ii) enqueueing network sends to remote accelerators for the buffer futures output by function executions, and (iii) communicating with the scheduler to determine a consistent order of function executions across all programs running on the island (Section 4.4). The scheduler must implement policies for allocating accelerators at a timescale of milliseconds. The current implementation "simply enqueues work in FIFO order, but more sophisticated schedulers might for example reorder computations based on estimated execution times" (Section 4.4).

Figure 9 demonstrates gang-scheduling in action: traces show "interleaving of gang-scheduled concurrent programs with proportional-share ratios of 1:1:1:1 (Upper) and 1:2:4:8 (Lower) between 4 clients" (Section 5.2). This proves PATHWAYS can time-multiplex accelerators between concurrent programs efficiently, "incurring no overhead to context switch" (Section 5.2).

#### 3.4.6 Parallel Asynchronous Dispatch: The Core Performance Innovation

The most significant technical innovation in PATHWAYS is **parallel asynchronous dispatch**, which overcomes the traditional single-controller performance bottleneck. To understand why this matters, consider the sequential dispatch model shown in Figure 4a: when a computation's on-device execution time is shorter than the time spent in scheduling, resource allocation, and coordination, "the asynchronous pipeline stalls because of host-side work in sequential dispatch" (Section 4.5).

PATHWAYS introduces a novel parallel asynchronous dispatch design shown in Figure 4b, which "exploits the statically known resource usage of regular compiled functions to run most of the host-side work for a computation's nodes in parallel, rather than serializing the work for a node to happen after its predecessors have been enqueued" (Section 4.5). The key insight is that "since the compiled functions are all regular, a successor node's input shapes can in practice be computed before the predecessor computation was even enqueued" (Section 4.5).

When a subgraph of a computation can be scheduled statically, "the program sends a single message (describing the entire subgraph) to the scheduler, which is able to sequence the execution of all the active shards in the subgraph back to back" (Section 4.5). The use of a single message is designed to minimize network traffic, but does not require the scheduler to actually enqueue all the subgraph's shards as a batch: "computations may still be interleaved with those submitted by other concurrently executing programs" (Section 4.5).

PATHWAYS treats parallel scheduling as an optimization and "falls back to the traditional model when a node's resource requirements are not known until a predecessor computation has completed (e.g., due to data-dependent control flow)" (Section 4.5). This fallback is important for future work on data-dependent vectorized control flow (Section 6.3).

Figure 7 quantifies the benefit: with parallel async dispatch, PATHWAYS "amortizes the fixed client overhead and the scheduling overhead for large number of pipeline stages" (Section 5.1). The figure shows three phases: first the fixed client overhead is amortized as the number of hosts increases, then the increasing transfer costs of adding more stages begin to dominate, and finally the system starts to amortize the fixed scheduling overhead. For comparison, the figure also shows performance when PATHWAYS is forced to use sequential asynchronous dispatch, waiting for one computation to be enqueued before enqueueing the next.

#### 3.4.7 Data Management: Sharded Object Stores

Each host manages a **sharded object store** that is similar to Ray's object stores but extended to also track buffers held in accelerator HBM at each shard (Section 4.6). Client programs can hold references to objects in remote host or accelerator memory, and the client and servers refer to them using opaque handles that allow the system to migrate them if needed. Intermediate program values are kept in the object stores, for example while the system is waiting to transfer them between accelerators or pass them to a subsequent computation.

The objects are tagged with ownership labels so that they can be garbage collected if a program or client fails (Section 4.6). The system can use simple back-pressure to stall a computation if it cannot allocate memory because other computations' buffers are temporarily occupying HBM (Section 4.6). This is important because TPUs have limited HBM capacity (16GB per accelerator for TPUv3 in the experiments, Section 5.3), and fine-grained context-switching where data is paged out to host DRAM over PCIe would waste accelerator cycles since "PCIe bandwidth is much smaller than HBM- or accelerator interconnect-bandwidth" (Appendix A.4).

#### 3.4.8 System Configurations and Scale Parameters

The evaluation uses three distinct hardware configurations (Section 5):

**Configuration (A)**: 4 TPUs per host, with the largest instance having 512 hosts, resulting in 2048 total TPUs connected via ICI. This configuration tests scaling to thousands of accelerators within a single pod.

**Configuration (B)**: 8 TPUs per host, with the largest instance having 64 hosts and a total of 512 TPUs. This configuration is used for most micro-benchmarks and multi-tenancy experiments.

**Configuration (C)**: Four islands of TPUs, where each island has 4 hosts and 32 TPUs (128 TPUs total). This configuration tests cross-island communication over DCN.

The paper reports specific thresholds for when PATHWAYS matches JAX throughput. For 16 hosts with 128 TPUs on configuration (B), parity is reached with a computation size of only 2.3 ms (Figure 6). Even for 512 hosts with 2048 TPUs on configuration (A), a computation of at least 35 ms masks all of PATHWAYS's single-controller overhead (Figure 6). These numbers are critical: they show that realistic ML computations (which typically run for tens to hundreds of milliseconds) easily exceed these thresholds, meaning the single-controller overhead is negligible in practice.

For large-scale models, PATHWAYS trains a 3B parameter Transformer with 62 layers, model dimension 2048, and hidden dimension 8192 (Section 5.3). The pipelined configuration splits this into $S$ stages with $M$ micro-batches: Table 2 shows results for $S=4, 8, 16$ stages with $M=16, 32, 64$ micro-batches respectively. The number of examples per micro-batch is fixed at 4, giving a global batch size per step of 2048 for 128-core configurations and 8192 for the 512-core configuration (Section 5.3, footnote 2).

For even larger models, PATHWAYS scales to 64B and 136B parameters using two islands of accelerators (Section 5.3). For the 136B model, training uses two islands of 1024 cores each with DCN transfer of 1030GB for global reduction. For the 64B model, training uses two islands of 512 cores each with DCN transfer of 457GB. PATHWAYS achieves approximately 97% of the throughput compared to a single island with twice as many devices (Section 5.3).

#### 3.4.9 Why This Design Over Alternatives?

The paper makes several deliberate design choices that distinguish PATHWAYS from prior systems:

**Single-controller over multi-controller**: The authors adopt a single-controller model because they believe it "offers much better opportunities than multi-controller for novel and efficient ML computation, both by exploiting computational sparsity and heterogeneity, and by enabling cluster management systems that promote sharing and virtualizing resources" (Section 2). Multi-controller systems require exclusive ownership of hardware, which "shifts the responsibility of ensuring high utilization of the expensive accelerators on to the user" and "complicates the design of features like resource virtualization and multiplexing" (Section 2).

**Sharded graph over materialized graph**: Unlike TensorFlow v1 which materializes the full sharded computation graph, PATHWAYS uses a compact representation with one node per sharded computation regardless of shard count. This avoids the "substantial overhead in both graph serialization and execution when the number of shards reaches into the thousands, leading to millions of graph edges" (Section 2).

**Parallel dispatch over sequential dispatch**: Traditional single-controller systems serialize host-side work, waiting for each computation to complete before scheduling the next. PATHWAYS exploits the regularity of compiled functions to run host-side work in parallel, overcoming the bottleneck shown in Figure 4a.

**Centralized gang-scheduling over per-program scheduling**: TensorFlow v1 can enforce consistent ordering within a single program using control edges, but "the lack of a centralized scheduler in single-controller systems like TF v1 makes it impossible to ensure consistent ordering between computations across programs" (Section 2). PATHWAYS's per-island scheduler enables true multi-tenancy with fair resource allocation.

**Virtual devices over physical device assignment**: The indirection between virtual and physical devices enables dynamic resource management, migration, and suspend/resume capabilities that would be impossible with direct physical device assignment.

The paper acknowledges that TPU-specific characteristics influence some design decisions: TPUs support "long-running and more complex computations" fused into a single kernel because XLA can compile functions with collective communication, and TPUs have "large islands of high-performance TPU interconnects" that allow flexible scheduling (Section 6.1). However, the authors believe "most of the high-level architectural choices we made in PATHWAYS would also be valid for large-scale GPU systems" (Section 6.1), noting that gang-scheduling is "highly advantageous for GPU efficiency" even though GPUs can execute concurrent programs without centralized scheduling (Appendix A.5).

## 4. Key Insights and Innovations

PATHWAYS introduces several contributions that range from fundamental architectural innovations to careful engineering optimizations. This section distinguishes between the two and explains why each matters for the future of distributed ML systems.

### 4.1 Parallel Asynchronous Dispatch: Exploiting Computational Regularity

**Innovation type:** Fundamental algorithmic insight

The most significant technical contribution is the observation that **ML computation regularity can be exploited to eliminate single-controller dispatch latency**. Prior single-controller systems like TensorFlow v1 serialized host-side coordination work: a successor computation could not be prepared until its predecessor's outputs were available (Section 2, Figure 1b). PATHWAYS recognizes that for "regular compiled functions" (Appendix B), input shapes and resource requirements are known *before* execution begins, enabling parallel preparation of multiple computation nodes.

**What makes this different:**

| Aspect | Traditional Single-Controller | PATHWAYS |
|--------|------------------------------|----------|
| Dispatch ordering | Sequential (wait for predecessor) | Parallel (prepare all nodes upfront) |
| Critical path | Host coordination + compute | Compute only (coordination overlapped) |
| Fallback | N/A | Sequential dispatch for irregular functions |

The significance is quantified in **Figure 7**: parallel dispatch amortizes fixed scheduling overhead across many pipeline stages, while sequential dispatch shows diminishing returns as stage count increases. This is not merely an optimization—it fundamentally changes the performance characteristics of single-controller systems.

**Why this matters beyond PATHWAYS:**

This insight generalizes to any system where computation graphs have statically analyzable resource requirements. The paper explicitly notes this is treated as "an optimization" that "falls back to the traditional model when a node's resource requirements are not known until a predecessor computation has completed" (Section 4.5). This fallback capability is crucial: it means PATHWAYS can support future research on data-dependent control flow (Section 6.3) without sacrificing performance on current workloads.

**Critical limitation:** The approach depends on the regularity of compiled functions. If ML research shifts toward highly dynamic, data-dependent architectures where shapes cannot be determined ahead of time, the parallel dispatch advantage diminishes. The paper acknowledges this tension but argues that "almost all ML models currently update every model weight based on every training example in every step" (Section 6.3), suggesting regularity will persist for foreseeable workloads.

### 4.2 Sharded Graph Representation: Avoiding Combinatorial Edge Explosion

**Innovation type:** Fundamental representation innovation

PATHWAYS introduces a **compact sharded dataflow graph** where each sharded computation is represented as a single node regardless of shard count. This contrasts sharply with TensorFlow v1, which "materializes the full sharded computation graph" leading to "millions of graph edges between sub-computations" at scale (Section 2).

**The mathematical difference:**

For a computation with $N$ shards connected to another computation with $M$ shards:

- **TensorFlow v1 representation:** Requires $N + M$ nodes and $N \times M$ edges
- **PATHWAYS representation:** Requires 2 nodes and 1 logical edge (sharding handled internally by PLAQUE)

At 2048 TPUs with typical model parallelism, this reduces graph complexity from $O(N^2)$ to $O(1)$ per computation pair. The paper states that a chained execution of computations $A$ and $B$ with $N$ shards each requires only 4 nodes: $\text{Arg} \rightarrow \text{Compute}(A) \rightarrow \text{Compute}(B) \rightarrow \text{Result}$, "regardless of the choice of $N$" (Section 4.3).

**Why this is fundamental:**

This is not an implementation detail—it enables scaling to thousands of accelerators that would be infeasible with materialized graphs. Graph serialization, distribution, and execution overhead all scale with graph size. By keeping the logical graph compact, PATHWAYS avoids the "substantial overhead in both graph serialization and execution" that plagued TensorFlow v1 at scale (Section 2).

**Trade-off acknowledged:** The compact representation requires PLAQUE to handle sharding internally, meaning the coordination substrate must be aware of shard topology. This couples PATHWAYS more tightly to its coordination layer than a system with explicit shard nodes would be. The paper notes that re-implementing with Ray would require "additions... to attain comparable performance" because Ray lacks primitives for efficient inter-accelerator transfer (Section 4.3).

### 4.3 Cross-Pod Scaling with Single-Controller: Breaking the ICI Boundary

**Innovation type:** New capability (previously impossible)

PATHWAYS enables JAX programs to "scale for the first time to multiple TPU pods, containing many thousands of TPU cores" (Section 3). This is not an incremental improvement—**native JAX cannot do this at all** because "JAX programs that run in multi-controller configurations transfer all data using XLA collectives, and these are only currently available over ICI on TPU" (Section 3).

**What this enables:**

- **Model sizes beyond single-pod memory:** A single TPU pod has finite aggregate HBM. Multi-pod training enables models that exceed this capacity.
- **Resource pooling across datacenter topology:** Organizations can utilize available accelerator islands rather than requiring contiguous pod allocation.
- **Foundation model fine-tuning at scale:** Multiple researchers can share a foundation model while fine-tuning on different tasks (Section 1).

**Evidence of success:**

**Table 2** shows PATHWAYS training a 3B Transformer across 4 islands of 32 TPUs each (Configuration C) achieves identical throughput (131.4k tokens/sec) to a single island of 128 TPUs (Configuration B). **Figure 10** confirms that "DCN transfers occur between every group of 8 rows in the trace, and are not visible in the trace because communication time is effectively overlapped with computation."

For even larger models, PATHWAYS trains 64B and 136B parameter models across two islands with "∼97% of the throughput as compared to a single island with twice as many devices" (Section 5.3). The DCN transfer overhead is 1030GB for the 136B model and 457GB for the 64B model, yet throughput degradation remains minimal.

**Why this matters:** This capability addresses the hardware heterogeneity trend identified in Section 1. As clusters become more heterogeneous, the ability to span multiple islands without performance degradation becomes essential. Prior systems forced a choice: use multi-controller (high performance, single pod only) or single-controller (flexible, poor cross-pod performance). PATHWAYS eliminates this trade-off.

### 4.4 Centralized Gang-Scheduling for True Multi-Tenancy

**Innovation type:** Fundamental systems innovation

PATHWAYS introduces **cross-program gang-scheduling** via per-island centralized schedulers. TensorFlow v1 could enforce consistent ordering "within a single program, by using control edges," but "the lack of a centralized scheduler in single-controller systems like TF v1 makes it impossible to ensure consistent ordering between computations across programs" (Section 2).

**What this enables:**

- **Time-multiplexing without context-switch overhead:** **Figure 8** shows PATHWAYS achieves aggregate throughput matching or exceeding JAX when multiple clients concurrently submit programs, "incurring no overhead to context switch" (Section 5.2).
- **Fair resource allocation:** **Figure 9** demonstrates proportional-share scheduling with ratios like 1:1:1:1 and 1:2:4:8 between 4 clients.
- **Dynamic resource reclamation:** The virtual device abstraction enables "transparent suspend/resume and migration, where a client's virtual devices are temporarily reclaimed or reassigned without the need for cooperation from the user program" (Section 4.1).

**Why this is innovative:**

Prior cluster schedulers operate at "long time scales (seconds or more)" with resources "exclusively dedicated to single jobs" (Section 7). PATHWAYS operates at "millisecond timescales" (Section 4.4), enabling fine-grained sharing that was previously infeasible. This is not merely faster scheduling—it enables fundamentally different resource utilization patterns.

**Evidence:**

**Figure 11** (Appendix D) shows traces with 1, 4, 8, and 16 concurrent clients. With a single client using 0.33 ms compute time, accelerators are underutilized. With 16 clients, "device utilization to ∼100%" is achieved through interleaving at "millisecond scale or less, showing little context-switch overhead."

**Critical insight:** The paper notes that "the maximum throughput of PATHWAYS exceeds that of JAX for very small computations" (Section 5.2). This is counterintuitive: a single-controller system outperforming a multi-controller system. The explanation is that "a PATHWAYS worker can accept more computations from remote clients than JAX can dispatch using Python locally." This reveals a hidden bottleneck in multi-controller systems: Python dispatch overhead limits throughput for small computations, which PATHWAYS avoids through C++ execution paths.

### 4.5 Distinguishing Innovation from Engineering

Not all contributions are equally novel. The table below distinguishes fundamental innovations from engineering optimizations:

| Contribution | Type | Significance |
|--------------|------|--------------|
| Parallel async dispatch | Fundamental | Changes single-controller performance characteristics |
| Sharded graph representation | Fundamental | Enables scaling to thousands of accelerators |
| Cross-pod JAX scaling | New capability | Previously impossible with native JAX |
| Centralized gang-scheduling | Fundamental | Enables millisecond-scale multi-tenancy |
| Virtual device abstraction | Engineering | Enables dynamic resource management (known pattern, new application) |
| PLAQUE integration | Engineering | Leverages existing infrastructure (though PLAQUE itself is closed-source) |
| Sharded object store | Engineering | Extension of Ray-style object stores to HBM |

**The core insight unifying these contributions:** PATHWAYS succeeds by recognizing that **ML workloads have structural properties** (regular compiled functions, predictable resource requirements, batch-oriented execution) that can be exploited for system optimization. Rather than building a general-purpose distributed system and hoping ML workloads perform well, PATHWAYS is purpose-built for ML's specific characteristics.

**What remains unresolved:**

The paper acknowledges several open challenges:

1. **Data-dependent control flow:** Future work must support "fine-grain control flow so that different model weights can be updated per example" (Section 6.3). PATHWAYS currently falls back to sequential dispatch for irregular functions.

2. **Resource management sophistication:** The initial resource manager uses "a simple heuristic" with one-to-one virtual-to-physical mapping. More sophisticated allocation algorithms are planned but not implemented (Section 4.1).

3. **GPU applicability:** While the authors believe "most of the high-level architectural choices... would also be valid for large-scale GPU systems" (Section 6.1), the evaluation is TPU-specific. GPU systems have different characteristics (smaller islands, different interconnects, more dynamic kernels) that may require adaptation.

These limitations do not diminish PATHWAYS's contributions but clarify the boundary between what is achieved and what remains for future work.

## 5. Experimental Analysis

The PATHWAYS evaluation is structured to answer three distinct questions: (1) Does the single-controller architecture introduce unacceptable overhead compared to multi-controller systems? (2) Can PATHWAYS efficiently support non-SPMD patterns like pipelining and multi-tenancy? (3) Does performance hold at production scale with real ML models? The experiments progress from micro-benchmarks that isolate specific system mechanisms to end-to-end model training that validates real-world applicability.

### 5.1 Evaluation Methodology

**Hardware Configurations (Section 5):**

The paper uses three distinct TPU configurations to test different scaling dimensions:

| Configuration | TPUs per Host | Max Hosts | Total TPUs | Interconnect | Purpose |
|--------------|---------------|-----------|------------|--------------|---------|
| (A) | 4 | 512 | 2048 | ICI (within pod) | Scale to thousands of accelerators |
| (B) | 8 | 64 | 512 | ICI | Micro-benchmarks and multi-tenancy |
| (C) | 8 (4 islands × 4 hosts) | 16 | 128 | DCN (between islands) | Cross-pod communication |

For GPU comparisons, the paper uses Ray v1.3 with PyTorch 1.8.1 on `p3.2xlarge` VMs, each with 1×V100 GPU and 8×CPU cores, connected via datacenter network with Amazon placement groups (Section 5, footnote 1).

**Baseline Systems:**

- **JAX multi-controller:** The primary baseline, chosen because it "has demonstrated state of the art performance in industry standard benchmarks" (Section 5) and can run on identical TPU hardware.
- **TensorFlow v1:** Represents traditional single-controller systems with known dispatch latency issues.
- **Ray:** Represents general-purpose distributed execution frameworks (GPU-based).

**Key Metrics:**

- **Throughput:** Computations per second (micro-benchmarks) or tokens per second (model training)
- **Dispatch overhead:** Time spent in host-side coordination vs. on-device computation
- **Utilization:** Percentage of accelerator time spent computing vs. idle
- **Scaling efficiency:** Throughput ratio when doubling resources

**Enqueue Modes (Section 5.1):**

The micro-benchmarks test three ways of submitting work, which stress different parts of the system:

- **OpByOp (-O):** Each computation requires a separate Python call and RPC. This is "a worst-case experiment that is not idiomatic for any of the frameworks" but stresses dispatch overhead.
- **Chained (-C):** A single client call executes a chain of 128 nodes, where each node is a separate computation. Tests the program tracer and multi-node graph execution.
- **Fused (-F):** A single call executes one computation node that internally contains a chain of 128 computations. Tests the ability to mask overhead with larger kernels.

### 5.2 Dispatch Overhead: Single-Controller Viability

The first critical question is whether PATHWAYS's single-controller architecture introduces unacceptable overhead. **Figure 5** presents the dispatch overhead micro-benchmark, measuring computations per second across 2 to 512 hosts.

**Key Findings:**

- **OpByOp performance:** As expected, JAX multi-controller significantly outperforms all single-controller systems (PATHWAYS, TensorFlow, Ray) in OpByOp mode, particularly as accelerator count increases. This is because "most of PATHWAYS's overhead comes from the fact that the client waits until the coordinator has enqueued one computation and returned its output handles before enqueueing the next" (Section 5.1).

- **Fused performance:** "Once enough work is Fused into a single node PATHWAYS matches JAX's performance up to 1000 TPU cores" (Section 5.1). This is the critical result: realistic ML workloads typically fuse many operations into single kernels, so this is the relevant operating regime.

- **Chained performance:** "PATHWAYS Chained outperforms JAX OpByOp up to 256 cores, because PATHWAYS can execute back-to-back accelerator computations directly from C++ while JAX OpByOp transitions to Python for every computation" (Section 5.1).

- **TensorFlow limitation:** "TensorFlow is slow when running over many cores because it uses a centralized barrier, implemented with control edges, to serialize the gang-scheduled computations" (Section 5.1). This validates the design motivation from Section 2.

- **Ray comparison:** "Out of the box, Ray shows about an order of magnitude worse performance per computation than PATHWAYS" (Section 5.1). The paper notes this is not directly comparable (different hardware), but suggests the gap comes from Ray lacking "an on-GPU object store and primitives to transfer objects efficiently over the GPU interconnect."

**Figure 6** answers the threshold question: what is the minimum computation size needed to mask single-controller overhead?

| Configuration | Hosts | TPUs | Minimum Computation Time for Parity |
|--------------|-------|------|-------------------------------------|
| (B) | 16 | 128 | 2.3 ms |
| (A) | 512 | 2048 | 35 ms |

> "For 16 hosts with 128 TPUs on configuration (B), parity is reached with only 2.3 ms, and even for 512 hosts with 2048 TPUs on configuration (A), a computation of at least 35 ms masks all of PATHWAYS's single-controller overhead" (Section 5.1, Figure 6 caption).

**Critical Assessment:**

This is a convincing result for the target workload class. Modern Transformer layers typically execute for tens to hundreds of milliseconds, well above the 35 ms threshold even at 2048 TPUs. However, this does reveal a **conditional limitation**: workloads with very fine-grained computations (e.g., certain reinforcement learning or graph neural network patterns with sub-millisecond kernels) would experience overhead. The paper acknowledges this implicitly by focusing evaluation on Transformer models where computation granularity is favorable.

### 5.3 Parallel vs. Sequential Dispatch: Ablation Study

**Figure 7** presents what is effectively an ablation study, comparing parallel asynchronous dispatch (the innovation from Section 4.5) against forced sequential dispatch. The experiment uses a pipeline benchmark where "each computation runs on a different set of 4 TPU cores, each on a different host, and data output from one computation must be sent via ICI before the next computation can execute" (Section 5.1).

**Three Phases Observed:**

1. **Client overhead amortization (1–8 stages):** As the number of hosts increases, the fixed client overhead is spread across more stages, improving throughput.

2. **Transfer cost dominance (8–32 stages):** "The increasing transfer costs of adding more stages begin to dominate" (Section 5.1). Throughput plateaus or decreases slightly.

3. **Scheduling overhead amortization (32–128 stages):** "Finally the system starts to amortize the fixed scheduling overhead" (Section 5.1). Throughput improves again.

**Quantitative Gap:**

While the figure does not provide exact numbers in the text, the visual gap between parallel and sequential dispatch is substantial at high stage counts. The sequential line shows diminishing returns beyond ~16 stages, while the parallel line continues improving. This validates the core insight from Section 4.5: when computation time is shorter than host-side coordination time, sequential dispatch creates stalls that parallel dispatch eliminates.

**Critical Assessment:**

This experiment directly isolates the contribution of parallel dispatch. However, it uses a synthetic pipeline benchmark rather than a real model. The follow-up experiments in Section 5.3 (Table 2) show that this benefit translates to real workloads, but the ablation would be stronger with a direct parallel-vs-sequential comparison on an actual pipelined Transformer.

### 5.4 Multi-Tenancy: Resource Sharing Efficiency

**Figure 8** tests whether PATHWAYS can time-multiplex accelerators between concurrent programs without overhead. The experiment varies the number of clients (1 to 256) and computation sizes (0.04, 0.33, 1.04, 2.4 ms).

**Key Results:**

- **Aggregate throughput:** "PATHWAYS can achieve at least the same aggregated throughput as JAX when multiple clients concurrently submit different PATHWAYS programs" (Section 5.2). There is "no overhead to context switch between programs from different clients, at least when their resources concurrently fit in HBM" (Section 5.2).

- **Small computation advantage:** "The maximum throughput of PATHWAYS exceeds that of JAX for very small computations, achieving higher TPU utilization" (Section 5.2). The explanation: "a PATHWAYS worker can accept more computations from remote clients than JAX can dispatch using Python locally" (Section 5.2).

- **Concurrency threshold:** "The degree of concurrency required to match the throughput is lower for larger computation sizes since the TPU cores reach full utilization sooner" (Section 5.2, referencing Figure 6).

**Figure 9** and **Figure 11** (Appendix D) provide trace evidence of gang-scheduling in action:

- **Figure 9:** Shows "interleaving of gang-scheduled concurrent programs with proportional-share ratios of 1:1:1:1 (Upper) and 1:2:4:8 (Lower) between 4 clients" (Section 5.2). This demonstrates fair resource allocation at millisecond granularity.

- **Figure 11:** Shows traces with 1, 4, 8, and 16 concurrent clients. "A single client uses a very small per-program compute time of 0.33 ms that is insufficient to saturate accelerators. With PATHWAYS's multi-tenancy support, using multiple clients increases the device utilization to ~100%" (Appendix D).

**Critical Assessment:**

This is one of the strongest experimental contributions. Prior cluster schedulers operate at "long time scales (seconds or more)" with exclusive resource dedication (Section 7). PATHWAYS demonstrates millisecond-scale multiplexing with no measurable overhead. The trace evidence (Figures 9 and 11) is particularly convincing—it shows the actual interleaving pattern rather than just aggregate throughput.

**Limitation:** The experiments assume resources "concurrently fit in HBM" (Section 5.2). If concurrent programs exceed aggregate HBM capacity, the paper does not report performance. Appendix A.4 notes that paging HBM data to host DRAM over PCIe "would waste a significant fraction of accelerator cycles," so this is a real constraint.

### 5.5 Large-Scale Model Performance: End-to-End Validation

The final question is whether PATHWAYS performs well on production ML workloads. The paper evaluates two model classes: Encoder-Decoder Transformers (T5 family) and Decoder-only Transformers (GPT-style).

**Table 1: T5 Model Training (JAX vs. PATHWAYS)**

| Model | Parameters | TPU Cores | JAX (tokens/s) | PATHWAYS (tokens/s) |
|-------|------------|-----------|----------------|---------------------|
| T5-Base | 270M | 32 | 618k | 618k |
| T5-Large | 770M | 32 | 90.4k | 90.4k |
| T5-3B | 3B | 512 | 282.8k | 282.8k |
| T5-11B | 11B | 512 | 84.8k | 84.8k |

> "Over all tested model sizes, the two systems show identical performance since realistic computations are large enough to mask single-controller overheads" (Section 5.3).

This is a critical result: PATHWAYS achieves **exact numerical parity** with JAX multi-controller on SPMD workloads. The paper notes that "the models trained on JAX and PATHWAYS achieve the same perplexity in the same number of steps" (Section 5.3), confirming numerical correctness.

**Table 2: 3B Transformer with Pipelining (PATHWAYS only)**

| Configuration | TPU Cores | Stages (S) | Micro-batches (M) | Throughput (tokens/s) |
|--------------|-----------|------------|-------------------|----------------------|
| SPMD | 128 | N/A | N/A | 125.7k |
| Pipeline | 128 | 4 | 16 | 133.7k |
| Pipeline | 128 | 8 | 32 | 132.7k |
| Pipeline | 128 | 16 | 64 | 131.4k |
| Pipeline | 512 | 16 | 64 | 507.8k |

**Key Observations:**

- **Pipeline vs. SPMD:** "At least in this instance, the pipeline has competitive performance to SPMD, since collective communication within the SPMD computation incurs higher overhead than pipeline bubble overhead" (Section 5.3). The 4-stage pipeline (133.7k tokens/s) actually **exceeds** SPMD (125.7k tokens/s) on the same 128 cores.

- **Scaling with stages:** "Increasing the number of pipeline stages adds minimal overhead, the throughput being reduced from 133.7k tokens/sec to 131.4k tokens/sec when the number of stages increases from 4 to 16" (Section 5.3). This is only a 1.7% degradation despite 4× more stages.

- **Scaling with cores:** "PATHWAYS's training throughput increases proportionally with the number of TPU cores per pipeline stage" (Section 5.3). Going from 128 to 512 cores (4×) increases throughput from 131.4k to 507.8k tokens/s (3.86×), showing near-linear scaling.

**Cross-Island Training (Figure 10):**

> "We measure the same throughput (131.4k tokens/sec) using a single island of 128 cores on configuration (B), or 4 islands of 32 cores each on configuration (C)" (Section 5.3).

**Figure 10** shows the execution trace: "DCN transfers occur between every group of 8 rows in the trace, and are not visible in the trace because communication time is effectively overlapped with computation" (Section 5.3). This validates that cross-pod communication does not degrade performance when properly overlapped.

**64B and 136B Parameter Models:**

> "When trained using PATHWAYS over two islands of compute connected over DCN, PATHWAYS achieves ~97% of the throughput as compared to a single island with twice as many devices" (Section 5.3).

- **136B model:** Two islands of 1024 cores each, DCN transfer of 1030GB for global reduction
- **64B model:** Two islands of 512 cores each, DCN transfer of 457GB for global reduction

**Figure 12** (Appendix D) shows the trace: "DCN transfers incur minimal overhead even at the scale of pairs of 128 hosts resulting in 97.2% training throughput compared to an SPMD configuration that uses ICI communication over total equivalent number of chips" (Appendix D).

**Critical Assessment:**

These results strongly support the paper's claims:

1. **SPMD parity:** Table 1 shows exact match with JAX across model sizes and scales.

2. **Pipelining efficiency:** Table 2 shows pipelining is competitive with (and sometimes exceeds) SPMD, with minimal overhead from adding stages.

3. **Cross-pod scaling:** Figure 10 and the 64B/136B results show ~97% efficiency when spanning islands, which is remarkable given DCN is orders of magnitude slower than ICI.

**Potential Concerns:**

- **Model diversity:** All end-to-end experiments use Transformer variants. The paper does not evaluate other model classes (CNNs, GNNs, MoE with dynamic routing). Section 6.3 acknowledges that "data-dependent vectorized control flow" is future work.

- **Pipeline configuration:** The pipelined experiments use GPipe-like scheduling with balanced stages. The paper notes they "took out one Transformer layer from the first and last stage to balance the amount of compute per stage" (Section 5.3). Real-world models may have less regular structures.

- **Failure modes:** The paper does not report behavior under failure conditions (e.g., host crashes, network partitions). Section 4.6 mentions ownership labels for garbage collection "if a program or client fails," but no fault tolerance experiments are presented.

### 5.6 Summary of Experimental Evidence

The table below summarizes how each experiment maps to the paper's claims:

| Claim | Supporting Experiment | Strength of Evidence |
|-------|----------------------|---------------------|
| Single-controller overhead is masked for realistic workloads | Figure 5, Figure 6 | Strong (clear thresholds established) |
| Parallel dispatch improves pipeline performance | Figure 7 | Moderate (synthetic benchmark, not real model) |
| Multi-tenancy incurs no context-switch overhead | Figure 8, Figure 9, Figure 11 | Strong (trace evidence + throughput data) |
| SPMD performance matches JAX | Table 1 | Strong (exact numerical parity across 4 model sizes) |
| Pipelining is competitive with SPMD | Table 2 | Strong (multiple stage counts, scaling data) |
| Cross-pod training is efficient | Figure 10, 64B/136B results | Strong (97% efficiency at massive scale) |

**Overall Assessment:**

The experiments are well-structured and convincingly support the paper's core claims. The progression from micro-benchmarks (isolating mechanisms) to end-to-end models (validating real-world applicability) is methodologically sound. The use of JAX as the primary baseline is appropriate given its state-of-the-art performance on TPU.

**Key Limitations:**

1. **TPU-specific evaluation:** All experiments use TPUs. While Section 6.1 argues the design applies to GPUs, no GPU data is presented for PATHWAYS (only Ray on GPU as a separate baseline).

2. **Workload narrowness:** All end-to-end models are Transformers. The paper's motivation includes MoE and sparse models (Section 1, 6.3), but these are not evaluated.

3. **Resource manager simplicity:** The evaluation uses the "simple heuristic" resource manager with one-to-one virtual-to-physical mapping (Section 4.1). More sophisticated allocation policies are noted as future work but not tested.

4. **Fault tolerance:** No experiments address failure recovery, which is critical for long-running training jobs at this scale.

Despite these limitations, the experimental evidence strongly supports PATHWAYS's central thesis: a single-controller system can match multi-controller performance on current workloads while enabling capabilities (cross-pod scaling, fine-grained multi-tenancy, flexible pipelining) that multi-controller systems cannot provide.

## 6. Limitations and Trade-offs

PATHWAYS represents a significant architectural advance, but like all systems, it makes deliberate trade-offs and rests on assumptions that constrain its applicability. This section examines what the system does *not* do well, what scenarios remain unaddressed, and what open questions persist. Understanding these limitations is crucial for practitioners deciding whether PATHWAYS fits their use case, and for researchers identifying opportunities for future work.

### 6.1 Dependence on Computational Regularity

The most fundamental assumption underlying PATHWAYS's performance is that **ML computations are "regular"**—meaning their resource requirements (input/output shapes, memory needs, execution time) are known before execution begins. This assumption enables the parallel asynchronous dispatch optimization described in Section 4.5.

**Where this breaks down:**

> "PATHWAYS treats parallel scheduling as an optimization and falls back to the traditional model when a node's resource requirements are not known until a predecessor computation has completed (e.g., due to data-dependent control flow)" (Section 4.5).

This fallback has concrete performance implications:

- **Sequential dispatch penalty:** When irregular functions are encountered, PATHWAYS reverts to the sequential dispatch model shown in **Figure 4a**, where "the asynchronous pipeline stalls because of host-side work" (Section 4.5). The performance benefit from parallel dispatch disappears.

- **Threshold sensitivity:** **Figure 6** shows that PATHWAYS needs computations of at least 2.3 ms (128 TPUs) to 35 ms (2048 TPUs) to match JAX throughput. Irregular functions that execute faster than these thresholds will experience measurable overhead.

**Real-world impact:**

Section 6.3 acknowledges that "almost all ML models currently update every model weight based on every training example in every step." However, the paper explicitly identifies emerging research directions that violate this regularity:

- **Mixture of Experts (MoE):** Models that "route" different examples to different accelerators based on learned functions require "fine-grain data-dependent data exchanges between nodes" (Section 6.3).

- **Routed capsule networks:** These exploit "computational sparsity by 'routing' different (sub-)examples to the accelerators hosting different subsets of model weights" (Section 6.3).

- **Per-example control flow:** The authors want to enable research where "different model weights can be updated per example, or even per sub-example (patch of an image, or word of a sentence)" (Section 6.3).

**Critical assessment:**

This is not a minor limitation—it defines the boundary of PATHWAYS's current applicability. The system is optimized for *today's* dominant workload patterns (regular Transformers) but explicitly defers support for *tomorrow's* research directions (sparse, data-dependent architectures). The paper frames this as "future work" (Section 6.3), but practitioners should recognize that adopting PATHWAYS may constrain experimentation with irregular model architectures until this capability is implemented.

### 6.2 TPU-Specific Design and GPU Applicability

All experimental evaluation in PATHWAYS uses **TPU accelerators exclusively**. While the authors argue their design applies to GPUs, this claim is not empirically validated.

**TPU-specific characteristics that influence design (Appendix A.5):**

- TPUs support "long-running and more complex computations" fused into single XLA kernels
- TPUs have "thousands of devices connected all-to-all" within pods
- TPUs execute non-preemptible kernels, making gang-scheduling *essential* to avoid deadlock
- TPUs have limited support for dynamic shapes due to static buffer assignment

**GPU differences that may require adaptation:**

- GPUs typically have "small islands of NVLink-connected devices (e.g., 8 GPUs within one host)" with larger aggregations over InfiniBand (Appendix A.5)
- GPUs dispatch "many small pre-compiled kernels" rather than long-running XLA functions
- GPUs support concurrent kernel execution without centralized scheduling
- GPU communication is "initiated by the host" via NCCL, unlike TPU's device-side collective execution

**Paper's claim vs. evidence:**

> "We believe that most of the high-level architectural choices we made in PATHWAYS and describe in this paper would also be valid for large-scale GPU systems" (Section 6.1).

However, the paper acknowledges GPU-specific challenges:

> "Some additions might be required to attain comparable performance [if implemented with Ray] because Ray lacks, for example, an HBM object store, or primitives to efficiently transfer remote objects over the GPU interconnect" (Section 4.3).

**Critical assessment:**

The GPU applicability claim rests on architectural reasoning rather than empirical evidence. Key uncertainties include:

1. **Dispatch overhead:** GPU kernels are typically shorter than TPU XLA computations. The 2.3–35 ms thresholds from **Figure 6** may not hold for GPU workloads with sub-millisecond kernels.

2. **Gang-scheduling necessity:** TPUs *require* gang-scheduling to avoid deadlock; GPUs do not. The performance benefit of centralized scheduling on GPUs is asserted but not measured.

3. **Interconnect topology:** TPU pods have uniform all-to-all ICI connectivity; GPU clusters often have heterogeneous topology (NVLink within hosts, InfiniBand between hosts). PATHWAYS's resource manager may need topology-aware allocation algorithms not present in the initial implementation (Section 4.1).

Practitioners deploying on GPU clusters should treat PATHWAYS's GPU applicability as a *hypothesis* rather than a proven capability.

### 6.3 Workload Narrowness in Evaluation

The experimental evaluation focuses almost exclusively on **Transformer models**. This is appropriate for validating performance on current state-of-the-art workloads, but it leaves open questions about other model classes.

**What was evaluated (Section 5.3):**

- **T5 family:** Encoder-Decoder Transformers (270M to 11B parameters) — **Table 1**
- **Decoder-only Transformer:** 3B parameters with pipelining — **Table 2**
- **Large Decoder-only:** 64B and 136B parameters across two islands — **Section 5.3**

**What was NOT evaluated:**

| Model Class | Mentioned in Paper | Evaluated? |
|-------------|-------------------|------------|
| Mixture of Experts (MoE) | Yes (Sections 1, 6.3, 7) | No |
| Graph Neural Networks | Yes (Section 7) | No |
| Neural Architecture Search | Yes (Section 7) | No |
| Multi-modal / Multi-task | Yes (Section 7) | No |
| CNNs / Vision Models | No | No |
| Reinforcement Learning | No | No |

**Why this matters:**

The paper's motivation explicitly cites MoE and sparse models as drivers for PATHWAYS's design:

> "Models such as Mixture of Experts (MoE) have started to explore computational sparsity that is most naturally expressed using fine-grain control flow and heterogeneous computation across accelerators" (Section 1).

However, Section 6.3 acknowledges:

> "It is the subject of future work to support data-dependent vectorized control flow with both a clean programming model and good performance."

**Critical assessment:**

There is a tension between the paper's motivation and its evaluation. PATHWAYS is motivated by the need to support heterogeneous, sparse, non-SPMD workloads, but the evaluation demonstrates performance only on homogeneous SPMD and pipelined Transformer workloads. This does not invalidate the results, but it means:

1. **Unproven claims:** The assertion that PATHWAYS enables "complex new parallelism patterns" (Abstract) is not directly validated for the patterns that motivated the design.

2. **Risk of over-specialization:** The paper warns that systems "become over-specialized to current workloads" (Section 1). There is some irony that PATHWAYS's evaluation focuses heavily on the very workloads (Transformers) that current systems already handle well.

3. **Pipeline assumptions:** The pipelined experiments use "balanced computation in each stage" with manual adjustment ("took out one Transformer layer from the first and last stage to balance the amount of compute per stage," Section 5.3). Real-world models may have less regular structures that challenge the load-balancing assumptions.

### 6.4 Resource Management Simplicity

The Resource Manager implementation is explicitly described as a **first-step heuristic** rather than a production-ready allocation system.

**Current implementation (Section 4.1):**

> "Our initial resource manager implementation uses a simple heuristic that attempts to statically balance load by spreading computations across all available devices, and keeps a one to one mapping between virtual and physical devices."

**Acknowledged limitations:**

> "If future workloads require it we can adopt a more sophisticated allocation algorithm, for example taking into account the resource requirements of all client computations and the current state of the system to approximate an optimal allocation of physical devices to computations" (Section 4.1).

**What this means in practice:**

- **No topology awareness:** The initial heuristic does not explicitly optimize for interconnect topology (e.g., placing communicating computations on physically proximate devices). This could matter for cross-pod training where DCN latency varies.

- **No preemption:** While the virtual device abstraction *enables* "transparent suspend/resume and migration" (Section 4.1), the current implementation does not demonstrate these capabilities. The multi-tenancy experiments in **Figure 8** and **Figure 9** show time-multiplexing but not preemption of running computations.

- **Static allocation:** Resources are allocated at program submission time. Dynamic reallocation during execution (e.g., scaling a computation up or down based on load) is not demonstrated.

**Critical assessment:**

The Resource Manager is the component most clearly marked as incomplete. For practitioners, this means:

1. **Multi-tenancy is limited:** While PATHWAYS can time-multiplex between concurrent programs, it cannot dynamically reclaim resources from a running computation without cooperation from the user program (Section 4.1 notes migration requires "future" support).

2. **Allocation quality is unproven:** The simple heuristic may work for balanced workloads but could produce suboptimal placements for heterogeneous workloads with varying communication patterns.

3. **Cluster utilization claims are conditional:** The paper's claims about improved cluster utilization through virtualization (Section 1) depend on Resource Manager capabilities not yet implemented.

### 6.5 HBM Capacity Constraints in Multi-Tenancy

The multi-tenancy experiments have an important caveat that is easy to overlook:

> "PATHWAYS can achieve at least the same aggregated throughput as JAX when multiple clients concurrently submit different PATHWAYS programs, i.e., there is no overhead to context switch between programs from different clients, **at least when their resources concurrently fit in HBM**" (Section 5.2, emphasis added).

**Why this constraint matters:**

Appendix A.4 explains the underlying hardware limitation:

> "PCIe bandwidth is much smaller than HBM- or accelerator interconnect-bandwidth. This means that fine-grained context-switching (where much of the data in HBM is paged out to host DRAM over PCIe) results in wasting a significant fraction of accelerator cycles."

**Quantitative context:**

- TPUv3 accelerators used in the experiments have **16GB HBM per accelerator** (Section 5.3)
- A 3B parameter Transformer model requires substantial memory for parameters, gradients, and optimizer states
- If concurrent programs exceed aggregate HBM capacity, the system must either:
  1. Reject new programs (reducing multiplexing benefits)
  2. Page data to host DRAM (wasting accelerator cycles)
  3. Preempt running programs (not yet implemented)

**Critical assessment:**

This constraint significantly limits the practical multi-tenancy scenarios PATHWAYS can support:

1. **Small computations only:** The multi-tenancy experiments use computations of 0.04–2.4 ms (**Figure 8**), which are much smaller than production model training steps. Large models may not fit concurrently in HBM.

2. **Utilization trade-off:** The benefit of multi-tenancy is highest when individual programs underutilize accelerators (as shown in **Figure 11**, Appendix D, where a single client with 0.33 ms compute time leaves accelerators idle). However, production training jobs typically aim for ~100% utilization of their allocated resources, leaving little idle capacity to multiplex.

3. **Memory fragmentation:** Even if aggregate HBM capacity is sufficient, fragmentation across shards could prevent efficient packing of concurrent programs. The paper does not address this.

Practitioners should interpret the multi-tenancy results as demonstrating *feasibility* rather than *universal applicability*. The HBM constraint means PATHWAYS's multi-tenancy benefits are most relevant for:
- Inference workloads (smaller memory footprint)
- Fine-tuning experiments (sharing foundation model layers)
- Development/debugging scenarios (multiple small jobs)

### 6.6 Fault Tolerance: Mentioned but Not Validated

PATHWAYS includes mechanisms for fault handling, but **no experimental evaluation** of fault tolerance is presented.

**What the paper describes:**

- **Ownership labels:** "Objects are tagged with ownership labels so that they can be garbage collected if a program or client fails" (Section 4.6)
- **Error delivery:** PLAQUE handles "delivering errors on failures" as part of background housekeeping (Section 4.3)
- **Back-pressure:** "Simple back-pressure to stall a computation if it cannot allocate memory" (Section 4.6)

**What is NOT described:**

| Fault Scenario | Addressed? |
|---------------|------------|
| Host crash during computation | No details |
| Network partition between islands | No details |
| TPU hardware failure | No details |
| Checkpointing and recovery | Not mentioned |
| Graceful degradation | Not mentioned |

**Why this matters:**

At the scales PATHWAYS targets (2048+ TPUs), failures are not rare events. Training jobs can run for days or weeks. A system without robust fault tolerance is not production-viable at this scale.

**Critical assessment:**

The absence of fault tolerance evaluation is a significant gap. The paper mentions mechanisms but provides no evidence of:

1. **Recovery time:** How long does it take to recover from a host failure?
2. **Progress preservation:** Can training resume from a checkpoint, or must it restart?
3. **Failure frequency:** How does the system behave under realistic failure rates?
4. **Cascading failures:** Does a single host failure cascade to other islands?

This limitation is particularly notable because the multi-controller systems PATHWAYS competes with (JAX, PyTorch) also have limited fault tolerance—they typically require job restart on failure. However, PATHWAYS's centralized architecture could *enable* better fault tolerance (e.g., migrating virtual devices to healthy hosts), but this capability is not demonstrated.

### 6.7 PLAQUE Dependency and Reproducibility

PATHWAYS relies on **PLAQUE**, described as "an existing (closed-source) production sharded dataflow system used at Google" (Section 4.3). This dependency has implications for reproducibility and external adoption.

**What this means:**

- **Not independently verifiable:** Researchers outside Google cannot inspect, modify, or benchmark PLAQUE directly.
- **Performance attribution:** It is difficult to determine how much of PATHWAYS's performance comes from PATHWAYS's design vs. PLAQUE's underlying implementation.
- **Alternative implementations:** The paper suggests Ray as an alternative (Section 4.3) but notes "some additions might be required to attain comparable performance."

**Paper's acknowledgment:**

> "We believe that it would be feasible to re-implement the full PATHWAYS design using other distributed frameworks such as Ray rather than PLAQUE to realize the low-level coordination framework" (Section 4.3).

However, the Ray comparison in **Figure 5** shows "about an order of magnitude worse performance per computation than PATHWAYS" (Section 5.1). The paper attributes this to Ray lacking "an on-GPU object store and primitives to transfer objects efficiently over the GPU interconnect," but this comparison conflates:
1. PATHWAYS design vs. Ray design
2. TPU vs. GPU hardware
3. PLAQUE substrate vs. Ray substrate

**Critical assessment:**

The PLAQUE dependency creates uncertainty about:

1. **Generalizability:** Would PATHWAYS achieve similar performance with an open-source coordination substrate?
2. **Adoption barrier:** Organizations without access to PLAQUE (i.e., everyone outside Google) cannot directly deploy PATHWAYS as described.
3. **Research reproducibility:** Other researchers cannot replicate the experiments or build on the implementation.

This does not invalidate PATHWAYS's architectural contributions, but it does limit the paper's practical impact outside Google. The open-source community would need to implement a PLAQUE equivalent to realize PATHWAYS's benefits.

### 6.8 Cross-Pod Efficiency: The 3% Gap

PATHWAYS achieves approximately **97% efficiency** when training across multiple islands compared to a single island with equivalent resources (Section 5.3). While this is impressive, the 3% degradation represents a real cost that warrants examination.

**Where the overhead comes from:**

- **DCN latency:** Datacenter network is "typically an order of magnitude slower than PCIe" (Section 2)
- **Transfer volume:** 1030GB for 136B model, 457GB for 64B model (Section 5.3)
- **Synchronization:** Global reduction requires coordination across islands

**Paper's explanation:**

> "DCN transfers occur between every group of 8 rows in the trace, and are not visible in the trace because communication time is effectively overlapped with computation" (Section 5.3, discussing **Figure 10**).

**Critical assessment:**

The 3% overhead is small but not negligible at scale:

1. **Cost implications:** For expensive training runs (weeks on thousands of accelerators), a 3% efficiency loss translates to meaningful compute costs.

2. **Scaling uncertainty:** The experiments test 2 islands (64B/136B models) and 4 islands (3B model). It is unclear whether the 3% overhead scales linearly, sublinearly, or superlinearly with more islands.

3. **Workload dependence:** The overhead may vary based on communication patterns. Models with more frequent cross-island communication (e.g., certain MoE configurations) could experience higher degradation.

4. **Network conditions:** The experiments likely use well-provisioned datacenter networks. Real-world deployments with network congestion could see higher overhead.

The 3% figure should be interpreted as a *best-case* bound under favorable conditions rather than a universal guarantee.

### 6.9 Summary: Trade-off Table

The table below summarizes the key trade-offs PATHWAYS makes:

| Trade-off | Benefit | Cost |
|-----------|---------|------|
| Single-controller architecture | Flexible programming, centralized resource management | Higher dispatch latency (masked for regular workloads) |
| Parallel async dispatch | Eliminates host-side bottleneck | Only works for regular compiled functions |
| Sharded graph representation | Scales to thousands of accelerators | Requires PLAQUE-aware coordination substrate |
| Virtual device abstraction | Enables migration, suspend/resume | Initial implementation uses simple 1:1 mapping |
| TPU-optimized design | Achieves state-of-the-art performance on TPUs | GPU applicability unproven |
| Multi-tenancy support | Improves cluster utilization | Limited by HBM capacity constraints |
| PLAQUE integration | Production-grade coordination | Closed-source, limits reproducibility |

### 6.10 Open Questions for Future Work

Based on the limitations identified above, several important questions remain unresolved:

1. **Irregular workloads:** Can PATHWAYS support data-dependent control flow (MoE, routed networks) without sacrificing the performance benefits demonstrated on regular workloads?

2. **GPU deployment:** What modifications are required to achieve comparable performance on GPU clusters with heterogeneous interconnect topology?

3. **Fault tolerance:** How does PATHWAYS handle host failures, network partitions, and hardware errors at production scale? What is the recovery time objective?

4. **Resource allocation:** Can sophisticated allocation algorithms (topology-aware, load-balanced) improve upon the initial heuristic without adding scheduling overhead?

5. **Preemption:** Can the virtual device abstraction enable transparent preemption and migration of running computations, and what is the performance cost?

6. **Open-source implementation:** Can an open-source coordination substrate achieve PLAQUE-like performance, enabling broader adoption?

7. **Scaling beyond 4 islands:** How does cross-pod efficiency degrade (if at all) when spanning 8, 16, or more islands?

These questions define the research agenda for extending PATHWAYS beyond its current capabilities. For practitioners, they also define the risk profile of adopting PATHWAYS: the system excels at today's workloads but has unproven capabilities for tomorrow's research directions.

## 7. Implications and Future Directions

PATHWAYS demonstrates that the historical trade-off between system flexibility and execution performance is not fundamental, but rather a consequence of specific implementation choices regarding dispatch and graph representation. By validating a single-controller architecture that matches multi-controller throughput, this work reshapes the design space for distributed ML systems. This section outlines how these findings influence the broader field, identifies concrete research opportunities opened by this architecture, and provides guidance for practitioners evaluating similar systems.

### 7.1 Shifting the Systems Landscape: Flexibility Without Penalty

The dominant paradigm in high-performance ML training prior to PATHWAYS was the **multi-controller model** (e.g., JAX, PyTorch distributed), chosen primarily to minimize dispatch latency over PCIe (Section 2). PATHWAYS challenges this orthodoxy by proving that a **single-controller model** can achieve parity (~100% accelerator utilization) through **parallel asynchronous dispatch** (Section 4.5).

**Key Landscape Shifts:**

*   **Control Plane Decoupling:** The success of PATHWAYS validates decoupling the control plane (coordination, scheduling) from the data plane (accelerator execution) even at scale. Prior single-controller systems like TensorFlow v1 struggled because control messages traversed the datacenter network (DCN) sequentially (Figure 1b). PATHWAYS shows that by exploiting the static knowledge of **compiled functions** (Appendix B), control messages can be batched and dispatched in parallel, masking DCN latency.
*   **Graph Representation Standards:** The paper demonstrates that materializing sharded graphs (one node per shard) is unnecessary overhead. The **sharded dataflow representation** (Section 4.3), where one logical node represents $N$ physical shards, reduces graph complexity from $O(N^2)$ edges to $O(1)$ logical edges between computations. This suggests future systems should prioritize compact logical representations over explicit physical graph materialization to scale beyond thousands of accelerators.
*   **Resource Virtualization as Default:** By introducing **virtual devices** mapped to physical hardware by a Resource Manager (Section 4.1), PATHWAYS moves resource management from a user responsibility (as in multi-controller exclusive ownership) to a system responsibility. This enables cluster-wide optimizations like multi-tenancy and migration that are impossible when users hold exclusive locks on physical hardware.

**Implication:** System designers should no longer assume multi-controller architectures are required for performance. The burden of proof has shifted: new systems should justify why they *cannot* exploit workload regularity to achieve single-controller performance.

### 7.2 Enabling Future Model Architectures

PATHWAYS is explicitly designed to support workloads that are "poorly supported by state-of-the-art systems" (Section 1). While the evaluation focuses on Transformers (Section 5.3), the architecture unlocks specific research directions that were previously constrained by system limitations.

**Research Opportunities Enabled:**

*   **Heterogeneous Parallelism (MPMD):** Current systems force homogeneous SPMD (Single Program Multiple Data) even when models are heterogeneous. PATHWAYS supports **Multiple Program Multiple Data (MPMD)** natively (Section 1). This enables research into models where different layers or components run different code on different accelerator subsets without awkward workarounds.
*   **Computational Sparsity and MoE:** The paper identifies **Mixture of Experts (MoE)** and routed networks as key targets (Section 6.3). These models require "fine-grain data-dependent data exchanges between nodes" based on learned routing functions. PATHWAYS's support for **sparse data exchanges** via PLAQUE (Section 4.3) provides the necessary substrate for this research, though full support for data-dependent control flow is marked as future work.
*   **Foundation Model Multiplexing:** The architecture supports sharing state between tasks (Section 1). This enables research into **parameter-efficient fine-tuning** where multiple users share fixed foundation model layers on the same accelerators while training task-specific heads. The multi-tenancy results (Figure 8) show this can be done with "no overhead to context switch," making shared infrastructure economically viable.
*   **Cross-Pod Scaling:** By efficiently spanning multiple TPU islands over DCN (Section 5.3), PATHWAYS enables training models that exceed the memory capacity of a single pod. The 97% throughput efficiency when scaling across islands (Section 5.3) suggests that physical topology constraints (pod boundaries) need not limit model size.

**Follow-Up Research Questions:**

1.  **Irregular Control Flow Optimization:** Section 6.3 notes that PATHWAYS currently falls back to sequential dispatch for data-dependent control flow. Research is needed to extend parallel dispatch techniques to irregular workloads, perhaps via speculative execution or improved shape inference.
2.  **Topology-Aware Allocation:** The initial Resource Manager uses a "simple heuristic" (Section 4.1). Future work should explore allocation algorithms that optimize for physical interconnect topology (ICI vs. DCN) to minimize the 3% overhead observed in cross-island training (Section 5.3).
3.  **Fault Tolerance at Scale:** The paper mentions garbage collection on failure (Section 4.6) but lacks evaluation. Research into checkpointing and recovery mechanisms that leverage the centralized state of the single-controller model could yield faster recovery times than multi-controller restarts.

### 7.3 Practical Applications and Downstream Use Cases

For practitioners and infrastructure teams, PATHWAYS defines a set of use cases where its architecture provides distinct advantages over traditional multi-controller systems.

**Preferred Scenarios:**

*   **Training Models > Single Pod Capacity:** When model parameters or activation memory exceed the aggregate HBM of a single accelerator pod (e.g., the 136B model in Section 5.3), PATHWAYS is preferable because it efficiently spans islands over DCN. Native JAX multi-controller cannot do this (Section 3).
*   **High-Utilization Clusters:** In environments where accelerator cost is high and utilization is critical, PATHWAYS's **gang-scheduling** and **multi-tenancy** (Section 5.2) allow multiple jobs to share hardware without exclusive ownership. This is ideal for internal cloud providers managing heterogeneous workloads.
*   **Pipelined Model Training:** For models requiring pipeline parallelism (e.g., large language models), PATHWAYS offers native support with competitive performance (Table 2). The **parallel async dispatch** mechanism specifically amortizes overhead across pipeline stages (Figure 7), making deep pipelines viable.
*   **Foundation Model Serving:** The ability to share fixed model layers across multiple fine-tuning tasks (Section 1) makes PATHWAYS suitable for serving infrastructure where multiple tenants access a common base model.

**Scenarios Where Caution is Needed:**

*   **Highly Dynamic Workloads:** If computations have shapes or resource requirements that are not known until runtime (violating the "regular compiled function" constraints in Appendix B), PATHWAYS falls back to sequential dispatch (Section 4.5). In these cases, the single-controller overhead may not be masked (Figure 6).
*   **GPU Clusters (Currently):** While the authors believe the design applies to GPUs (Section 6.1), the evaluation is TPU-specific. GPU kernels are typically shorter, and interconnects (NVLink/InfiniBand) differ from TPU ICI. Practitioners should verify performance thresholds (like the 35 ms mask at 2048 cores in Figure 6) hold for their specific GPU hardware.
*   **Small-Batch, Low-Latency Inference:** The multi-tenancy benefits are most pronounced when aggregating work to fill idle capacity (Figure 11). For strict low-latency inference where jobs cannot be batched or interleaved, the DCN dispatch latency might be prohibitive compared to local multi-controller dispatch.

### 7.4 Integration and Reproducibility Guidance

Adopting PATHWAYS or its principles requires navigating specific implementation dependencies and reproducibility constraints.

**Integration Pathways:**

*   **For JAX Users:** PATHWAYS is implemented as a "plug-in replacement for the JAX backend" (Section 3). Users can run unmodified JAX code but gain access to multiple TPU pods. Integration requires using the PATHWAYS client library to request **virtual devices** and optionally using the **program tracer** (Figure 2) to fuse multiple compiled functions into a single graph for optimal performance.
*   **For System Builders:** Implementing a PATHWAYS-like system does not require replicating the closed-source **PLAQUE** substrate. The paper suggests frameworks like **Ray** could serve as the coordination layer (Section 4.3), provided they are extended with:
    *   An HBM object store (to avoid PCIe transfers for intermediate data).
    *   Primitives for efficient inter-accelerator transfer (e.g., GPUDirect).
    *   Support for gang-scheduling semantics to prevent deadlocks on non-preemptible accelerators.

**Reproducibility Constraints:**

*   **PLAQUE Dependency:** PATHWAYS relies on PLAQUE, a "closed-source production sharded dataflow system used at Google" (Section 4.3). External researchers cannot directly replicate the coordination substrate. Performance comparisons against Ray (Figure 5) show an order-of-magnitude difference, partly attributed to Ray lacking specific hardware integrations (Section 5.1).
*   **Hardware Specificity:** All evaluations use TPUs (Configurations A, B, C in Section 5). The specific thresholds for overhead masking (e.g., 2.3 ms at 128 TPUs, 35 ms at 2048 TPUs in Figure 6) are tied to TPU kernel dispatch characteristics and ICI bandwidth. These numbers will not transfer directly to GPU clusters.
*   **Resource Manager Simplicity:** The evaluated Resource Manager uses a "simple heuristic" with one-to-one virtual-to-physical mapping (Section 4.1). Production deployments may require more sophisticated allocation logic to achieve the same efficiency in heterogeneous environments.

**Recommendation:** Practitioners should adopt the **architectural principles** of PATHWAYS—specifically parallel asynchronous dispatch, compact sharded graph representation, and centralized gang-scheduling—even if the exact system is not available open-source. When evaluating alternative systems, verify that they support **parallel dispatch** for regular workloads and **cross-pod communication** without forcing data through host memory, as these are the critical mechanisms that enable PATHWAYS's performance parity (Section 4.5, Section 5.3).
