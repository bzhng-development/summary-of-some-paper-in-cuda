## 1. Executive Summary

FlashAttention-2 addresses the computational bottleneck of scaling Transformers to longer sequence lengths by optimizing GPU work partitioning and parallelism, overcoming the suboptimal 25-40% theoretical FLOPs/s utilization of the original FlashAttention. Through algorithmic tweaks and sequence-level parallelism, it achieves around 2× speedup over FlashAttention, reaching up to 73% of theoretical maximum throughput on A100 GPUs and enabling end-to-end training speeds of up to 225 TFLOPs/s per GPU on GPT-style models. This efficiency gain allows training models with 16k context for the same cost as 8k context, facilitating applications in long-document understanding and high-resolution media.

## 2. Context and Motivation

To understand the necessity of FlashAttention-2, we must first examine why scaling Transformer models to longer sequences is difficult, why existing solutions fall short, and where the specific hardware bottlenecks lie. This section unpacks the computational landscape that motivated the authors to revisit the attention mechanism.

### 2.1 The Barrier to Long-Context Transformers

The core ambition of modern Transformer research is to scale context length—the number of tokens a model can process at once. The paper highlights that while standard models often cap at 2,000 tokens, emerging applications demand significantly more:
*   **Language:** Understanding books or long documents (e.g., GPT-4 with 32k context, Claude with 100k context).
*   **Multimedia:** High-resolution image understanding and long-form video generation.

However, the attention layer at the heart of Transformers presents a fundamental computational barrier. Given an input sequence length $N$ and head dimension $d$, standard attention computes an intermediate similarity matrix $S = QK^\top$ of size $N \times N$. This leads to:
*   **Quadratic Memory:** Storing $S$ and the attention probabilities $P$ requires $O(N^2)$ memory.
*   **Quadratic Runtime:** Computing $S$ and $P$ requires $O(N^2)$ operations.

As noted in the Introduction, when $N$ grows from 2k to 100k, the memory requirement grows by a factor of 2,500. This quadratic scaling makes training on long sequences prohibitively expensive or impossible on standard hardware due to memory capacity limits and memory bandwidth bottlenecks.

### 2.2 Limitations of Prior Attention Implementations

Before FlashAttention-2, researchers attempted to solve this bottleneck through two main avenues: algorithmic approximation and IO-aware optimization.

#### Approximate Attention Methods
Numerous methods proposed approximating the attention mechanism to reduce complexity (e.g., linear attention, sparse attention). The paper cites several such approaches [2, 3, 4, 8, 9, 14, 19, 20]. While these methods reduce computational load, they alter the mathematical definition of attention. The authors observe that despite their theoretical benefits, "most large-scale training runs still use standard attention." This suggests that approximations often incur performance penalties or stability issues that outweigh their speed benefits in production settings.

#### FlashAttention (Original)
The immediate predecessor, FlashAttention [5], addressed the memory bottleneck without approximation. It utilized two key techniques:
1.  **Tiling:** Loading blocks of inputs from high-bandwidth memory (HBM) to fast on-chip SRAM, computing attention on those blocks, and updating the output incrementally.
2.  **Recomputation:** Avoiding the storage of the $N \times N$ matrix $P$ during the backward pass by recomputing it from saved statistics.

FlashAttention successfully reduced memory usage from $O(N^2)$ to $O(N)$ and achieved a 2-4× wall-clock speedup over optimized baselines. However, the authors identify a critical remaining gap: **FlashAttention was still not compute-efficient.**

### 2.3 The Hardware Efficiency Gap

The primary motivation for FlashAttention-2 is not memory savings (which FlashAttention already solved), but **compute utilization**. The paper provides a stark comparison of hardware throughput on Nvidia A100 GPUs:

*   **Optimized Matrix Multiply (GEMM):** Can reach 80-90% of the theoretical maximum FLOPs/s (floating-point operations per second).
*   **FlashAttention (Original):**
    *   Forward pass: Only 30-50% of theoretical maximum throughput (Fig. 5).
    *   Backward pass: Only 25-35% of theoretical maximum throughput (Fig. 6).

> "Through careful profiling, we observe that FlashAttention still has suboptimal work partitioning between different thread blocks and warps on the GPU, causing either low-occupancy or unnecessary shared memory reads/writes." (Section 1)

This inefficiency stems from how the algorithm maps to GPU architecture. GPUs execute kernels using **thread blocks** scheduled on **Streaming Multiprocessors (SMs)**. Within thread blocks, threads are grouped into **warps** (32 threads). The original FlashAttention implementation suffered from:
1.  **Low Occupancy:** In scenarios with long sequences but small batch sizes, there were not enough thread blocks to keep all SMs busy.
2.  **Non-Matmul Overhead:** GPUs have specialized Tensor Cores for matrix multiplication (matmul) that are significantly faster than standard arithmetic units. On an A100, FP16 matmul throughput is 312 TFLOPs/s, while non-matmul FP32 throughput is only 19.5 TFLOPs/s. This means **each non-matmul FLOP is effectively 16× more expensive** than a matmul FLOP (Section 3.1).
3.  **Shared Memory Contention:** The original work partitioning between warps required frequent synchronization and data exchange via shared memory, creating bottlenecks.

### 2.4 Positioning FlashAttention-2

FlashAttention-2 positions itself not as a new mathematical approximation, but as a **hardware-aware algorithmic refinement**. It retains the exact mathematical output of standard attention (no approximation) and builds directly upon the IO-aware foundations of FlashAttention.

The key distinction lies in the optimization target:
*   **FlashAttention:** Optimized for **Memory IO** (minimizing reads/writes to HBM).
*   **FlashAttention-2:** Optimized for **Compute Parallelism** (maximizing utilization of Tensor Cores and GPU occupancy).

The paper positions this work as essential for the next stage of scaling. As context lengths increase, batch sizes often decrease (due to memory constraints), exacerbating the occupancy issues of the original implementation. By addressing work partitioning at the thread block and warp level, FlashAttention-2 aims to close the gap between attention performance and the theoretical efficiency of standard matrix multiplication primitives.

## 3. Technical Approach

### 3.1 Reader orientation (approachable technical breakdown)

FlashAttention-2 is a GPU kernel implementation that reorganizes how the attention computation is distributed across GPU processing units to maximize hardware utilization. It solves the problem of underutilized GPU compute resources in the original FlashAttention by restructuring work assignment at three levels: algorithmic operations, thread block parallelization, and warp-level task distribution.

### 3.2 Big-picture architecture (diagram in words)

The FlashAttention-2 system consists of three interconnected optimization layers that operate on the same fundamental attention computation. First, the **Algorithm Layer** modifies the mathematical sequence of operations to minimize expensive non-matrix-multiply operations while preserving exact output. Second, the **Parallelism Layer** distributes attention computation across multiple thread blocks along the sequence length dimension, not just across batch and head dimensions. Third, the **Work Partitioning Layer** reorganizes how individual warps (groups of 32 threads) within each thread block share data, reducing shared memory communication overhead. Information flows from input matrices Q, K, V in high-bandwidth memory (HBM) through on-chip SRAM in tiled blocks, undergoes attention computation with online softmax statistics tracking, and produces output O and logsumexp statistics L back in HBM for use in the backward pass.

### 3.3 Roadmap for the deep dive

- We begin with the algorithmic tweaks to the forward pass because understanding how online softmax works is prerequisite to grasping why certain operations can be eliminated.
- We then examine the backward pass modifications, which build on forward pass statistics and demonstrate how recomputation avoids memory storage.
- We analyze the sequence-level parallelism strategy next, as this requires understanding the tiled computation structure from the algorithm section.
- We dive into warp-level work partitioning last, since this is the most granular optimization and depends on understanding thread block organization.
- We conclude with block size tuning considerations, which tie together all three optimization layers and explain practical deployment choices.

### 3.4 Detailed, sentence-based technical breakdown

This is an algorithmic systems paper that presents hardware-aware optimizations for exact attention computation, with the core idea being that GPU throughput is limited not by memory IO (which FlashAttention solved) but by how work is partitioned across GPU compute units.

#### The GPU Execution Model Foundation

To understand FlashAttention-2's innovations, we must first establish how GPUs execute computation. A GPU kernel runs across many **thread blocks**, where each thread block is a group of threads scheduled together on a **Streaming Multiprocessor (SM)**. The A100 GPU contains 108 SMs, and each SM can handle multiple thread blocks concurrently depending on resource availability. Within each thread block, threads are organized into **warps**, where a warp consists of exactly 32 threads that execute in lockstep. Threads within a warp can communicate efficiently through **shuffle instructions** that exchange register values, while warps within a thread block communicate through **shared memory** (on-chip SRAM). The A100 provides 192KB of shared memory per SM with approximately 19TB/s bandwidth, compared to 1.5-2.0TB/s for high-bandwidth memory (HBM) (Section 2.1).

The critical performance insight driving FlashAttention-2 is that modern GPUs have specialized **Tensor Cores** for matrix multiplication that achieve dramatically higher throughput than general arithmetic units. On the A100, FP16/BF16 matrix multiply throughput reaches 312 TFLOPs/s, while non-matrix-multiply FP32 throughput is only 19.5 TFLOPs/s (Section 3.1). This 16× difference means that each non-matmul FLOP effectively costs 16× more time than a matmul FLOP, making it essential to minimize non-matmul operations even if they represent a small fraction of total FLOPs.

#### Algorithmic Tweaks: Reducing Non-Matmul FLOPs

The first optimization layer modifies the online softmax algorithm to reduce expensive elementwise operations while maintaining mathematical equivalence. To understand this, we must first review how online softmax enables tiling in the attention forward pass.

Standard attention computes the softmax row-wise across the full $N \times N$ similarity matrix $S = QK^\top$, which requires materializing the entire matrix. Online softmax allows computing softmax incrementally across blocks by tracking running statistics. Consider splitting a row of $S$ into two blocks $S^{(1)}$ and $S^{(2)}$. Standard softmax would compute:

$$m = \max(\text{rowmax}(S^{(1)}), \text{rowmax}(S^{(2)}))$$
$$\ell = \text{rowsum}(e^{S^{(1)} - m}) + \text{rowsum}(e^{S^{(2)} - m})$$
$$P = \text{diag}(\ell)^{-1} \begin{bmatrix} e^{S^{(1)} - m} & e^{S^{(2)} - m} \end{bmatrix}$$
$$O = P \begin{bmatrix} V^{(1)} \\ V^{(2)} \end{bmatrix}$$

where $m$ is the row-wise maximum for numerical stability, $\ell$ is the sum of exponentials (the normalizer), and $O$ is the attention output (Section 2.3.1).

Online softmax instead processes blocks sequentially, updating statistics as each block arrives. For the first block:

$$m^{(1)} = \text{rowmax}(S^{(1)})$$
$$\ell^{(1)} = \text{rowsum}(e^{S^{(1)} - m^{(1)}})$$
$$\tilde{O}^{(1)} = e^{S^{(1)} - m^{(1)}} V^{(1)}$$

For the second block, we update the running maximum and rescale previous outputs:

$$m^{(2)} = \max(m^{(1)}, \text{rowmax}(S^{(2)})) = m$$
$$\ell^{(2)} = e^{m^{(1)} - m^{(2)}} \ell^{(1)} + \text{rowsum}(e^{S^{(2)} - m^{(2)}}) = \ell$$
$$O^{(2)} = \text{diag}(\ell^{(1)}/\ell^{(2)})^{-1} O^{(1)} + \text{diag}(\ell^{(2)})^{-1} e^{S^{(2)} - m^{(2)}} V^{(2)}$$

The original FlashAttention implementation performed the rescaling $\text{diag}(\ell^{(2)})^{-1}$ on both terms at each step, which requires expensive elementwise division operations (Section 3.1.1).

FlashAttention-2 makes two key algorithmic modifications. First, it maintains an **unscaled accumulator** $\tilde{O}$ throughout the block loop, only applying the final normalization $\text{diag}(\ell^{(T_c)})^{-1}$ after processing all $T_c$ column blocks. The update rule becomes:

$$\tilde{O}^{(j)} = \text{diag}(e^{m^{(j-1)} - m^{(j)}})^{-1} \tilde{O}^{(j-1)} + e^{S^{(j)} - m^{(j)}} V^{(j)}$$

Then at the end: $O = \text{diag}(\ell^{(T_c)})^{-1} \tilde{O}^{(T_c)}$. This eliminates one scaling operation per block iteration, reducing non-matmul FLOPs significantly when $T_c$ is large.

Second, FlashAttention-2 stores only the **logsumexp** statistic $L^{(j)} = m^{(j)} + \log(\ell^{(j)})$ for the backward pass, rather than storing both $m^{(j)}$ and $\ell^{(j)}$ separately (Section 3.1.1). This reduces memory storage and simplifies the backward pass computation, since the backward pass can recover necessary statistics from $L$ directly. Algorithm 1 in the paper provides the complete forward pass pseudocode showing these modifications integrated into the tiled loop structure.

#### Backward Pass with Recomputation

The backward pass computes gradients dQ, dK, dV given the output gradient dO, following the chain rule through the attention computation. From Section 2.2, the standard backward pass equations are:

$$dV = P^\top dO$$
$$dP = dO V^\top$$
$$dS = d\text{softmax}(dP) = P \circ (dP - D)$$
$$dQ = dS K$$
$$dK = Q dS^\top$$

where $D = \text{rowsum}(dO \circ O)$ is computed once upfront, and $\circ$ denotes elementwise multiplication. The gradient of softmax involves the term $(\text{diag}(P) - PP^\top)dP$, which simplifies to $P \circ (dP - D)$ when $D$ captures the row-wise dot product (Section 2.2).

FlashAttention-2's backward pass (Algorithm 2) leverages the saved logsumexp $L$ from the forward pass to recompute $P$ on-chip without storing the $N \times N$ matrix. For each block pair $(i, j)$:

$$P^{(j)}_i = \exp(S^{(j)}_i - L_i)$$

This recomputation is feasible because $S^{(j)}_i = Q_i K_j^\top$ can be computed from tiled inputs in SRAM, and $L_i$ is stored in HBM with only $O(N)$ memory (Section 3.1.2). The backward pass performs 5 matrix multiplications per block pair compared to 2 in the forward pass, making it more computationally intensive and more sensitive to work partitioning inefficiencies.

A minor but important modification in FlashAttention-2 is that the backward pass uses only the logsumexp $L$ rather than separate $m$ and $\ell$ statistics, consistent with the forward pass change. This simplifies the interface between forward and backward passes and reduces the memory footprint of saved activations.

#### Sequence-Level Parallelism

The second optimization layer addresses GPU occupancy by parallelizing across the sequence length dimension, not just across batch size and number of heads. In the original FlashAttention, one thread block processes one attention head, yielding $\text{batch\_size} \times \text{num\_heads}$ thread blocks total (Section 3.2). This works well when this product is large (e.g., ≥80 thread blocks to utilize 108 SMs on A100), but fails when sequences are long and batch sizes are small.

FlashAttention-2 parallelizes the outer loop over row blocks across multiple thread blocks. In the forward pass, each thread block handles a subset of row blocks $Q_i$, processing all column blocks $K_j, V_j$ for those rows independently (Figure 2, left). Since row blocks do not need to communicate with each other during the forward pass, this parallelization is **embarrassingly parallel**—thread blocks operate completely independently.

In the backward pass, parallelization is more complex because updating dQ requires accumulation across column blocks. From Algorithm 2, line 15 shows that dQ is updated as:

$$dQ_i \leftarrow dQ_i + dS^{(j)}_i K_j$$

Each column block $j$ contributes to the same dQ gradient, requiring synchronization. FlashAttention-2 assigns one thread block per column block $j$, and uses **atomic adds** to safely accumulate dQ updates from multiple thread blocks writing to the same memory locations (Section 3.2, Backward pass). This design choice trades some atomic operation overhead for significantly improved occupancy when sequence length is large.

The parallelization strategy was inspired by Phil Tillet's Triton implementation of FlashAttention, which first demonstrated swapping the loop order (outer loop over row blocks instead of column blocks) and parallelizing over sequence length (Section 3.2, footnote 3). FlashAttention-2 adopts and extends this approach with additional warp-level optimizations.

#### Warp-Level Work Partitioning

The third optimization layer reorganizes how warps within a thread block divide the matrix multiplication work, which is the most granular and technically subtle improvement. To understand this, we must distinguish between two partitioning schemes.

The original FlashAttention uses a **split-K** scheme for the $QK^\top$ computation within each thread block. As shown in Figure 3(a), K and V are split across 4 warps while Q is accessible to all warps. Each warp computes a slice of $QK^\top$, then multiplies with its slice of V. However, all warps must write their intermediate results to shared memory, synchronize, and then add up the partial results to produce the final output (Section 3.3, Forward pass). This shared memory communication creates a bottleneck, as warps must wait for each other during synchronization points.

FlashAttention-2 reverses the partitioning: Q is split across 4 warps while K and V are accessible to all warps (Figure 3(b)). Each warp computes its slice of $QK^\top$, then immediately multiplies with the shared V to produce its corresponding slice of the output O. Critically, **no inter-warp communication is required** because each warp produces a disjoint portion of the output. This eliminates shared memory reads/writes for intermediate accumulation and removes synchronization overhead between warps.

The backward pass similarly avoids split-K partitioning, though it requires some synchronization due to more complex dependencies between Q, K, V, O, dO, dQ, dK, and dV gradients (Section 3.3, Backward pass). The key principle remains: minimize shared memory traffic by assigning each warp independent work that does not require reduction with other warps.

This design choice reflects a deeper understanding of GPU microarchitecture. Shared memory bandwidth (19TB/s) is much higher than HBM bandwidth (1.5-2.0TB/s), but shared memory access still incurs latency and requires synchronization primitives that stall warps. By eliminating inter-warp communication entirely in the forward pass, FlashAttention-2 keeps warps busy with compute rather than waiting on memory operations.

#### Block Size Tuning and Practical Configuration

FlashAttention-2 uses tiled computation with block sizes $B_r \times B_c$ for row and column partitions. The paper reports typical block sizes of $\{64, 128\} \times \{64, 128\}$, chosen based on head dimension $d$ and available shared memory (Section 3.3, Tuning block sizes). Larger block sizes reduce the number of shared memory loads/stores per FLOP but increase register pressure and total shared memory consumption.

The tuning process involves a tradeoff: if blocks are too large, **register spilling** occurs (registers exceed availability and spill to slower memory), causing significant slowdown. If blocks are too small, the overhead of loading tiles dominates computation. The authors manually tune block sizes for each head dimension, noting that only 4 choices exist but that auto-tuning could eliminate this manual effort in future work.

For causal masking (required in autoregressive language modeling), FlashAttention-2 exploits the block structure to skip computation. When all column indices in a block exceed row indices (approximately half of blocks for large sequences), that entire block computation is skipped, yielding 1.7-1.8× speedup compared to non-causal attention (Section 3.1.1, Causal masking). For blocks where some entries are masked, the causal mask is applied to only 1 block per row (assuming square blocks), minimizing the overhead of mask application.

#### Memory and Complexity Characteristics

FlashAttention-2 maintains the same theoretical complexity as FlashAttention: $O(N^2 d)$ FLOPs for computation and $O(N)$ additional memory beyond inputs and outputs (Section 3.1.1, Correctness). The $O(N)$ memory stores the logsumexp statistics $L$ of size $N \times 1$ per head, compared to $O(N^2)$ for storing the full attention matrix $P$ in standard implementations.

The correctness guarantee is critical: FlashAttention-2 produces **exact** attention output with no approximation, matching standard attention mathematically. The proof follows from the online softmax derivation and is essentially identical to Dao et al. [5, Theorem 1], so the paper omits it (Section 3.1.1, Correctness). This distinguishes FlashAttention-2 from approximate attention methods that trade accuracy for speed.

#### Integration with Attention Variants

FlashAttention-2 supports **Multi-Query Attention (MQA)** and **Grouped-Query Attention (GQA)**, where multiple query heads attend to the same key/value heads to reduce KV cache size during inference (Section 3.1.2, Multi-query attention). Instead of duplicating K and V for each query head, the implementation implicitly manipulates head indices to perform the same computation. In the backward pass, gradients dK and dV must be summed across different query heads that shared the same key/value head, which the algorithm handles through accumulation during the tiled loop.

This support is important for practical deployment, as MQA and GQA are increasingly common in production models for their inference efficiency benefits. The FlashAttention-2 optimizations apply equally to these variants since the core tiled computation structure remains unchanged.

## 4. Key Insights and Innovations

FlashAttention-2 introduces several contributions that range from fundamental algorithmic insights to careful engineering optimizations. Understanding which innovations are conceptual breakthroughs versus incremental refinements helps clarify what makes this work significant and what aspects might generalize to other GPU kernels.

### 4.1 The Non-Matmul FLOP Cost Disparity Insight

**What makes this novel:** While prior work recognized that Tensor Cores accelerate matrix multiplication, FlashAttention-2 quantifies the *magnitude* of this asymmetry and treats it as a first-order optimization constraint. The paper reveals that on A100 GPUs, non-matmul FLOPs are effectively **16× more expensive** than matmul FLOPs (312 TFLOPs/s vs. 19.5 TFLOPs/s for FP16/BF16 vs. FP32) (Section 3.1).

**Why this matters:** This insight reframes how we should think about kernel optimization. Traditional performance analysis often counts total FLOPs without distinguishing operation types. FlashAttention-2 demonstrates that:

- A kernel with fewer total FLOPs can be *slower* if those FLOPs are non-matmul operations
- Optimization should prioritize **matmul utilization** over raw FLOP reduction
- Even small reductions in non-matmul operations yield disproportionate speedups

> "While the non-matmul FLOPs only account for a small fraction of the total FLOPs, they take longer to perform as GPUs have specialized units for matrix multiply, and as a result the matmul throughput can be up to 16× higher than non-matmul throughput." (Section 1)

**Distinction from prior work:** The original FlashAttention optimized for **memory IO** (minimizing HBM reads/writes). FlashAttention-2 optimizes for **compute type** (maximizing Tensor Core usage). This represents a shift in optimization philosophy—from IO-bound thinking to compute-heterogeneity-aware thinking.

**Significance:** This insight generalizes beyond attention. Any GPU kernel mixing Tensor Core operations with elementwise arithmetic should weight non-matmul operations heavily in performance models. This could influence compiler design and automatic kernel generation strategies.

### 4.2 Warp-Level Work Partitioning Reversal

**What makes this novel:** The decision to split **Q across warps** instead of **K and V** (reversing the split-K scheme) is a fundamental architectural change in how parallel reduction is handled within a thread block. As detailed in Section 3.3 and Figure 3, this eliminates inter-warp communication entirely in the forward pass.

**Why this matters:** This innovation addresses a subtle but critical bottleneck that profiling revealed but prior implementations accepted:

- **Original FlashAttention:** Warps compute partial results → write to shared memory → synchronize → reduce partial results → write output
- **FlashAttention-2:** Warps compute independent output slices → write output directly (no reduction needed)

The key insight is recognizing that the reduction operation (summing partial results across warps) was unnecessary if the output partitioning aligns with the query partitioning. Each warp can produce a **disjoint portion of the final output** without needing to combine with other warps.

**Distinction from prior work:** This is not an incremental tuning parameter—it's a structural reorganization of the computation graph. The split-K pattern is common in GEMM implementations for load balancing, but FlashAttention-2 demonstrates that for attention specifically, split-Q yields better performance due to the asymmetric role of Q (appears once) versus K and V (used across multiple row blocks in some parallelization schemes).

**Significance:** This pattern may apply to other attention-like computations where one input matrix determines output partitioning. It suggests that work partitioning decisions should consider **output ownership** rather than just input load balancing.

### 4.3 Sequence-Length Parallelism for Long-Context Regimes

**What makes this novel:** Parallelizing across the sequence length dimension (in addition to batch and head dimensions) addresses a specific performance cliff that occurs when long sequences force small batch sizes. Section 3.2 explains that the original FlashAttention uses one thread block per head, yielding `batch_size × num_heads` thread blocks. When sequence length increases, batch size typically decreases, leaving GPU SMs underutilized.

**Why this matters:** This innovation solves an **occupancy problem** that becomes worse as the very use case FlashAttention enables (long context) becomes more extreme:

| Scenario | Batch Size | Sequence Length | Thread Blocks (Original) | Thread Blocks (FA-2) |
|----------|-----------|-----------------|-------------------------|---------------------|
| Short context | 64 | 512 | 64 × 32 = 2,048 | 2,048 + sequence parallelism |
| Long context | 2 | 16k | 2 × 32 = 64 | 64 + sequence parallelism |

For the long-context case, 64 thread blocks cannot saturate 108 SMs on A100, causing significant idle compute capacity. FlashAttention-2's sequence parallelism creates additional thread blocks from the sequence dimension itself.

**Distinction from prior work:** The paper acknowledges this idea was "first suggested and implemented by Phil Tillet in the Triton implementation" (Section 3.2, footnote 3). FlashAttention-2's contribution is **integrating this with warp-level optimizations** and providing the CUDA implementation with full backward pass support. This is more of an adoption and extension than original discovery, but the integration is non-trivial—particularly the atomic add strategy for dQ accumulation in the backward pass.

**Significance:** This demonstrates that kernel parallelism strategies must adapt to the **workload regime**. A parallelization scheme optimal for short sequences becomes suboptimal for long sequences, even when the underlying algorithm is identical.

### 4.4 Unscaled Accumulator for Online Softmax

**What makes this novel:** The observation that softmax rescaling can be deferred until the end of the block loop, rather than applied at each iteration, reduces elementwise operations proportionally to the number of column blocks $T_c$. Section 3.1.1 shows the mathematical reformulation:

Original per-iteration update:
$$O^{(j)} = \text{diag}(\ell^{(1)}/\ell^{(2)})^{-1} O^{(1)} + \text{diag}(\ell^{(2)})^{-1} e^{S^{(2)} - m^{(2)}} V^{(2)}$$

FlashAttention-2 update:
$$\tilde{O}^{(j)} = \text{diag}(e^{m^{(j-1)} - m^{(j)}})^{-1} \tilde{O}^{(j-1)} + e^{S^{(j)} - m^{(j)}} V^{(j)}$$

With final scaling: $O = \text{diag}(\ell^{(T_c)})^{-1} \tilde{O}^{(T_c)}$

**Why this matters:** This is a clever algebraic observation that reduces $T_c - 1$ scaling operations per row block. For long sequences with many column blocks, this accumulates to meaningful savings. Combined with the non-matmul cost insight (4.1), eliminating these elementwise divisions has outsized impact.

**Distinction from prior work:** This builds directly on the online softmax technique from prior work [11, 13] used in FlashAttention. It's an **algorithmic refinement** rather than a fundamental breakthrough—the mathematical equivalence is straightforward once identified. However, it demonstrates careful attention to operation counting that prior implementations overlooked.

**Significance:** This type of optimization—deferring normalization until the end of an iterative process—may apply to other numerically stable iterative algorithms where intermediate scaling is conventionally applied at each step.

### 4.5 LogSumExp as the Single Forward-Backward Interface

**What makes this novel:** Storing only the logsumexp statistic $L = m + \log(\ell)$ instead of separate max $m$ and sum $\ell$ values simplifies the forward-backward pass interface. Section 3.1.1 and 3.1.2 show this reduces memory storage and enables direct computation of $P = \exp(S - L)$ in the backward pass.

**Why this matters:** This innovation provides three benefits:

- **Memory reduction:** One value per row instead of two (50% reduction in activation storage for this statistic)
- **Numerical stability:** LogSumExp is a well-studied numerically stable operation
- **Interface simplification:** Backward pass needs only one statistic to recompute attention probabilities

**Distinction from prior work:** The original FlashAttention stored both $m$ and $\ell$ separately. This change is relatively minor in isolation but reflects a broader design principle: **minimize the forward-backward interface surface area**. Every value saved during forward pass for backward use represents memory bandwidth and capacity cost.

**Significance:** This principle generalizes to any training algorithm with recomputation strategies. Minimizing saved activations while enabling exact gradient computation is a recurring theme in memory-efficient training (e.g., activation checkpointing, reversible networks).

### Summary: Incremental vs. Fundamental

| Innovation | Type | Generalizability |
|------------|------|-----------------|
| Non-matmul cost awareness | Fundamental insight | High—applies to any Tensor Core kernel |
| Split-Q work partitioning | Fundamental architectural change | Medium—attention-specific but pattern may transfer |
| Sequence-length parallelism | Regime-specific optimization | Medium—applies to long-sequence workloads |
| Unscaled accumulator | Algorithmic refinement | Medium—iterative normalization algorithms |
| LogSumExp interface | Interface optimization | High—any recomputation-based training |

The most significant contribution is arguably the **non-matmul cost awareness**, as it changes how practitioners should think about GPU kernel optimization fundamentally. The **warp partitioning reversal** delivers the largest measured performance gain but is more specific to attention's computational structure. Together, these innovations demonstrate that even well-studied operations like attention still have optimization headroom when analyzed through the lens of hardware heterogeneity.

## 5. Experimental Analysis

This section examines how the authors validate FlashAttention-2's claims. We will walk through their evaluation methodology, unpack the specific numbers they report, and critically assess whether the experiments convincingly demonstrate the paper's contributions. Understanding *how* they measured performance is as important as the results themselves, since GPU benchmarking has many pitfalls that can mislead readers.

### 5.1 Evaluation Methodology

The authors employ a two-pronged evaluation strategy: **micro-benchmarks** that isolate attention kernel performance, and **end-to-end training** that measures real-world impact on model training throughput. This dual approach is important because kernel-level speedups do not always translate proportionally to end-to-end training speedups due to other bottlenecks (e.g., optimizer steps, communication overhead).

#### Hardware Setup

All primary experiments run on **Nvidia A100 80GB SXM4 GPUs**, which was the state-of-the-art datacenter GPU at the time of publication (July 2023). The A100 specifications are critical context for interpreting the results:

- **108 Streaming Multiprocessors (SMs)** per GPU
- **192KB shared memory** per SM
- **1.5-2.0 TB/s HBM bandwidth**
- **312 TFLOPs/s theoretical FP16/BF16 matmul throughput** (Section 2.1)

The paper also reports results on **H100 80GB SXM5 GPUs** (Figure 7), but with an important caveat: the implementation does *not* use H100-specific features like TMA (Tensor Memory Accelerator) or 4th-generation Tensor Cores. The authors explicitly state they expect "another 1.5×-2× speedup on H100 GPUs" with proper optimization (Section 4.1). This is honest reporting but means the H100 numbers are *lower bounds* on what FlashAttention-2 could achieve on that hardware.

For end-to-end training, they use **8×A100 GPUs** in parallel, which is a standard configuration for medium-scale model training.

#### Baselines and Comparisons

The authors compare against four baselines, each representing a different point in the attention implementation design space:

| Baseline | Description | Optimization Focus |
|----------|-------------|-------------------|
| **PyTorch standard** | Built-in `torch.nn.functional.scaled_dot_product_attention` | General-purpose, not attention-specific |
| **FlashAttention (original)** | Dao et al. [5] CUDA implementation | Memory IO optimization |
| **FlashAttention in xformers** | CUTLASS-based implementation | Memory IO with different kernel backend |
| **FlashAttention in Triton** | Phil Tillet's Triton implementation | Memory IO with higher-level language |

This is a comprehensive set of baselines. Comparing against the original FlashAttention isolates the *algorithmic and parallelism improvements* of FlashAttention-2. Comparing against Triton and xformers tests whether the CUDA implementation advantages hold against alternative programming models. Comparing against PyTorch standard attention shows the absolute improvement over what most practitioners would use without specialized libraries.

#### Benchmark Configuration

For micro-benchmarks (Figures 4-6), the authors use a controlled setup:

- **Sequence lengths:** 512, 1k, 2k, 4k, 8k, 16k tokens
- **Batch size:** Adjusted so total tokens = 16k (e.g., batch=32 for 512 seq len, batch=1 for 16k seq len)
- **Hidden dimension:** 2048
- **Head dimension:** 64 or 128 (corresponding to 32 or 16 heads)
- **Causal mask:** Tested both with and without

> "To calculate the FLOPs of the forward pass, we use: $4 \cdot \text{seqlen}^2 \cdot \text{head dimension} \cdot \text{number of heads}$. With causal mask, we divide this number by 2." (Section 4.1)

This FLOPs formula is standard for attention, but note the causal mask adjustment: since approximately half the attention matrix is masked out, they count only half the FLOPs. This is important for fair comparison—without this adjustment, causal attention would appear artificially less efficient.

For the backward pass, they multiply forward FLOPs by **2.5**, accounting for the 5 matrix multiplications in backward versus 2 in forward (due to recomputation) (Section 4.1).

#### End-to-End Training Configuration

For end-to-end validation (Table 1), they train **GPT-style models** with:

- **Model sizes:** 1.3B and 2.7B parameters
- **Context lengths:** 2k and 8k tokens
- **Metric:** TFLOPs/s per GPU, plus Model FLOPs Utilization (MFU)

The MFU metric is calculated as:

$$\text{MFU} = \frac{\text{Achieved TFLOPs/s}}{\text{Theoretical Maximum TFLOPs/s}}$$

They report up to **72% MFU**, which is exceptionally high for Transformer training. For context, many production training runs achieve 40-60% MFU due to communication overhead, optimizer steps, and data loading bottlenecks.

### 5.2 Quantitative Results

We now walk through the specific numbers, organized by experiment type. Pay attention to *where* FlashAttention-2 shows the largest gains—this reveals which optimizations matter most under which conditions.

#### Micro-Benchmark: Combined Forward + Backward (Figure 4)

Figure 4 shows the most comprehensive performance picture, combining forward and backward pass throughput. The key takeaway is that FlashAttention-2's advantage *grows with sequence length*, which is exactly the regime where it matters most.

**Without causal mask, head dimension 64 (Figure 4a):**
- At 16k sequence length, PyTorch standard attention hits **OOM** (Out Of Memory)
- FlashAttention-2 reaches **176 TFLOPs/s** at 16k
- FlashAttention (original) reaches **110 TFLOPs/s** at 16k
- This is a **1.6× speedup** at the longest sequence length

**Without causal mask, head dimension 128 (Figure 4b):**
- FlashAttention-2 reaches **203 TFLOPs/s** at 16k
- FlashAttention (original) reaches **83 TFLOPs/s** at 16k
- This is a **2.4× speedup**—larger than the head-dim-64 case

**With causal mask, head dimension 128 (Figure 4d):**
- FlashAttention-2 reaches **189 TFLOPs/s** at 16k
- FlashAttention (original) reaches **83 TFLOPs/s** at 16k
- This is a **2.3× speedup**

The pattern is clear: **larger head dimensions and longer sequences yield bigger speedups**. This makes sense because:
1. Longer sequences mean more column blocks ($T_c$), amplifying the benefit of reduced non-matmul FLOPs per block
2. Larger head dimensions mean more matmul work per block, better amortizing any remaining overhead

The paper summarizes: "FlashAttention-2 is around 2× faster than FlashAttention and FlashAttention in xformers... Compared to a standard attention implementation in PyTorch, FlashAttention-2 can be up to 10× faster" (Section 4.1). The 10× figure comes from short sequence lengths where PyTorch's $O(N^2)$ memory access pattern is particularly inefficient.

#### Micro-Benchmark: Forward Pass Only (Figure 5)

The forward pass results show higher absolute throughput than combined forward+backward, which is expected since the backward pass has more matrix multiplications (5 vs 2).

**Key numbers from Figure 5:**
- **Maximum forward throughput:** 230 TFLOPs/s (Figure 5b, head dim 128, no causal, 2k seq len)
- **Theoretical utilization:** 73% of A100's 312 TFLOPs/s maximum
- **FlashAttention (original) comparison:** ~100-110 TFLOPs/s in similar settings, meaning **~2× improvement**

An important observation: the forward pass peaks at **2k-4k sequence length**, then slightly declines at 8k-16k. This is likely due to reduced occupancy as sequence length grows—each thread block does more work, so fewer thread blocks can run concurrently. This is a subtle trade-off that the sequence-level parallelism (Section 3.2) partially mitigates but does not eliminate.

#### Micro-Benchmark: Backward Pass Only (Figure 6)

The backward pass is where FlashAttention-2's improvements matter most, since the backward pass is more computationally intensive and was less optimized in the original FlashAttention.

**Key numbers from Figure 6:**
- **Maximum backward throughput:** 196 TFLOPs/s (Figure 6b, head dim 128, no causal, 8k seq len)
- **Theoretical utilization:** 63% of A100's 312 TFLOPs/s maximum
- **FlashAttention (original) comparison:** ~80-90 TFLOPs/s in similar settings, meaning **~2.2× improvement**

> "The backward pass is even more challenging, reaching only 25-35% of maximum throughput on A100 GPU" (Section 1, describing original FlashAttention)

FlashAttention-2 nearly **doubles** the backward pass efficiency, closing the gap between forward and backward performance. This is significant because training time is dominated by the backward pass (typically 60-70% of iteration time).

#### H100 GPU Results (Figure 7)

The H100 results demonstrate FlashAttention-2's scalability to newer hardware, though with the caveat that H100-specific features are not yet exploited.

**Key numbers from Figure 7:**
- **Maximum throughput:** 338 TFLOPs/s (Figure 7b, head dim 128, no causal, 8k seq len)
- **Comparison to A100:** ~1.5× higher absolute throughput
- **FlashAttention (original) on H100:** ~140-170 TFLOPs/s in similar settings

The authors note: "We expect that by using new instructions, we can obtain another 1.5×-2× speedup on H100 GPUs" (Section 4.1). This is an important qualification—the reported H100 numbers are *not* the ceiling for FlashAttention-2 on that hardware.

#### End-to-End Training (Table 1)

Table 1 provides the most practically relevant numbers: how much faster can you actually *train models* with FlashAttention-2?

| Model | Context | Without FA | FlashAttention | FlashAttention-2 | FA-2 vs Baseline | FA-2 vs FA |
|-------|---------|------------|----------------|------------------|------------------|------------|
| GPT3-1.3B | 2k | 142 TFLOPs/s | 189 TFLOPs/s | **196 TFLOPs/s** | 1.38× | 1.04× |
| GPT3-1.3B | 8k | 72 TFLOPs/s | 170 TFLOPs/s | **220 TFLOPs/s** | 3.06× | 1.29× |
| GPT3-2.7B | 2k | 149 TFLOPs/s | 189 TFLOPs/s | **205 TFLOPs/s** | 1.38× | 1.08× |
| GPT3-2.7B | 8k | 80 TFLOPs/s | 175 TFLOPs/s | **225 TFLOPs/s** | 2.81× | 1.29× |

Several observations from this table:

1. **Longer context = bigger gains:** At 2k context, FlashAttention-2 is only 1.04-1.08× faster than FlashAttention. At 8k context, it's 1.29× faster. This confirms that the optimizations target the long-sequence regime.

2. **Baseline collapses at long context:** Without FlashAttention, training speed drops from 142 to 72 TFLOPs/s (1.3B model) when going from 2k to 8k context. FlashAttention-2 *increases* from 196 to 220 TFLOPs/s. This is because standard attention becomes memory-bandwidth-bound at long sequences, while FlashAttention-2 becomes more compute-efficient.

3. **Absolute throughput is exceptional:** 225 TFLOPs/s per A100 represents **72% MFU**. For comparison, many production training runs report 40-50% MFU. This suggests FlashAttention-2 removes attention as a bottleneck entirely—the remaining overhead comes from other sources (communication, optimizer, data loading).

The paper summarizes: "FlashAttention-2 yields 2.8× speedup compared to a baseline without FlashAttention and 1.3× speedup compared to FlashAttention" (Section 4.2). These figures are consistent with Table 1's 8k context results.

### 5.3 Critical Assessment of Experimental Claims

Now we evaluate whether the experiments convincingly support the paper's central claims.

#### Claim 1: "2× Speedup Over FlashAttention"

**Supported?** Yes, with conditions.

The micro-benchmarks (Figures 4-6) consistently show ~2× speedup at long sequence lengths (8k-16k). However, at short sequences (512-2k), the speedup is smaller (~1.3-1.5×). The end-to-end training results (Table 1) show 1.29× at 8k context but only 1.04-1.08× at 2k context.

**Why the discrepancy?** The optimizations in FlashAttention-2 (sequence-level parallelism, reduced non-matmul FLOPs) have more impact when there are many blocks to process. At short sequences, there are fewer blocks, so the overhead reductions matter less. This is not a weakness—it's an honest reflection of where the technique applies.

#### Claim 2: "73% of Theoretical Maximum Throughput"

**Supported?** Yes, for forward pass.

Figure 5 shows forward pass throughput reaching 230 TFLOPs/s on A100, which is 73% of the 312 TFLOPs/s theoretical maximum. The backward pass reaches 63% (Figure 6), which is still a dramatic improvement over FlashAttention's 25-35%.

**Caveat:** The theoretical maximum assumes perfect matmul utilization. Since attention includes non-matmul operations (softmax, exponentiation), 100% utilization is theoretically impossible. The 73% figure should be interpreted as "73% of what a perfect GEMM could achieve," not "73% of what attention could theoretically achieve."

#### Claim 3: "Enables 16k Context Training"

**Supported?** Yes, with evidence.

Figure 4 shows PyTorch standard attention hitting **OOM at 16k sequence length** across all configurations. FlashAttention-2 runs successfully at 16k with strong throughput (176-203 TFLOPs/s depending on head dimension). The end-to-end experiments only go to 8k context (Table 1), but the micro-benchmarks demonstrate the kernel works at 16k.

**Missing evidence:** The paper does not show end-to-end training at 16k context. This is a gap—we know the *kernel* works at 16k, but we don't have direct evidence of *model training* at that length. The authors state "we can train models with 16k longer context for the same price as previously training a 8k context model" (Section 5), but this is extrapolated from the micro-benchmarks.

#### Claim 4: "No Approximation, Exact Attention"

**Supported?** Yes, by construction.

This is not an empirical claim but a mathematical one. The paper states the proof "is almost the same as the proof of Dao et al. [5, Theorem 1]" (Section 3.1.1). Since FlashAttention was already proven exact, and FlashAttention-2 only modifies the *order* and *partitioning* of operations (not the mathematical formulas), exactness is preserved.

**Verification:** The authors do not provide numerical comparison showing FlashAttention-2 output matches standard attention to machine precision. This would have been a useful sanity check, but given the mathematical derivation, it's reasonable to accept the claim.

### 5.4 Ablation Studies and Limitations

#### Missing Ablation Studies

A notable gap in the experimental section is the **absence of ablation studies** isolating individual optimizations. FlashAttention-2 has three main improvements:

1. Algorithmic tweaks (reduced non-matmul FLOPs)
2. Sequence-level parallelism
3. Warp-level work partitioning

The paper does not report experiments like "FlashAttention-2 without sequence parallelism" or "FlashAttention-2 with original warp partitioning." This makes it impossible to quantify how much each optimization contributes to the overall speedup.

**Why this matters:** Without ablations, we cannot tell if one optimization dominates (e.g., warp partitioning provides 90% of the gain) or if they are complementary. This limits the paper's utility for researchers who might want to apply similar techniques to other kernels.

The authors acknowledge some optimizations were inspired by prior work (Triton implementation for sequence parallelism, Section 3.2), but do not credit specific performance contributions to each source.

#### Conditional Results and Trade-offs

Several results are conditional on specific hardware or configuration choices:

**Head dimension sensitivity:** FlashAttention-2 performs better with head dimension 128 than 64 (compare Figure 4a vs 4b). At 16k sequence length without causal mask:
- Head dim 64: 176 TFLOPs/s
- Head dim 128: 203 TFLOPs/s

This is a **15% difference** in throughput. The reason is that larger head dimensions provide more matmul work per block, better amortizing the fixed overhead of loading tiles and managing parallelism.

**Causal mask benefit:** The paper states causal masking provides "around 1.7-1.8× speedup compared to attention without the causal mask" (Section 3.1.1) because approximately half the blocks can be skipped. This is visible in Figure 4—compare 4a (no causal, 176 TFLOPs/s at 16k) with 4c (causal, 171 TFLOPs/s at 16k for head dim 64). Wait—the numbers don't match the claimed 1.7-1.8×.

**Discrepancy note:** Looking at Figure 4a vs 4c at 16k, head dim 64:
- No causal: 176 TFLOPs/s
- With causal: 171 TFLOPs/s

This is essentially the same throughput, not 1.7-1.8×. The 1.7-1.8× claim likely refers to *runtime* (wall-clock time), not TFLOPs/s. Since causal attention computes half the FLOPs, achieving similar TFLOPs/s means completing in half the time. This is a subtle but important distinction that the paper could clarify.

**Block size tuning:** The authors state they "manually tune for each head dimensions since there are essentially only 4 choices for block sizes" (Section 3.3). This is a practical limitation—deployment requires per-configuration tuning. They note "this could benefit from auto-tuning to avoid this manual labor" but leave it to future work.

#### Robustness Checks

The experiments test multiple sequence lengths, head dimensions, and causal/non-causal settings, which provides good coverage of the design space. However, several robustness aspects are not addressed:

- **Numerical stability:** No comparison of training loss curves or convergence behavior between FlashAttention and FlashAttention-2. Given the algorithmic changes to online softmax, this would reassure readers that the optimizations don't affect model quality.

- **Multi-GPU scaling:** The end-to-end experiments use 8 GPUs, but there's no analysis of how FlashAttention-2 interacts with distributed training (e.g., tensor parallelism, pipeline parallelism). The sequence-level parallelism within a kernel could potentially conflict with sequence parallelism across GPUs.

- **Different GPU architectures:** Only A100 and H100 are tested. AMD GPUs, older Nvidia GPUs (V100, A10), and consumer GPUs (RTX 4090) are not evaluated. The authors mention plans to support "AMD GPUs" in future work (Section 5), acknowledging this gap.

### 5.5 Summary of Experimental Evidence

| Claim | Evidence Strength | Key Supporting Data |
|-------|------------------|---------------------|
| 2× speedup over FlashAttention | Strong (conditional) | Figures 4-6, Table 1 (8k context) |
| 73% theoretical throughput | Strong | Figure 5 (230 TFLOPs/s on A100) |
| Enables 16k context | Moderate | Figure 4 (kernel works), no end-to-end 16k training |
| Exact attention (no approximation) | Strong (by construction) | Mathematical derivation from [5] |
| Works across head dimensions | Strong | Tested 64 and 128 head dim |
| Generalizes to H100 | Moderate | Figure 7 (without H100-specific features) |

The experiments are **convincing for the core claims** but leave some practical questions unanswered. The micro-benchmarks are thorough and well-controlled. The end-to-end training results demonstrate real-world impact but only cover 1.3B-2.7B parameter models—larger models (7B, 70B+) are not tested.

The most significant gap is the **lack of ablation studies**. Without isolating individual optimizations, readers cannot determine which techniques are most valuable or whether they would transfer to other kernels. This is a common limitation in systems papers but reduces the paper's instructional value for researchers.

Overall, the experimental section successfully demonstrates that FlashAttention-2 achieves its stated goals: faster attention with better GPU utilization, enabling longer context training at lower cost. The numbers are specific, the baselines are appropriate, and the hardware context is well-explained. The caveats (no H100-specific optimization, no ablations, limited end-to-end model sizes) are honestly acknowledged, allowing readers to interpret the results appropriately.

## 6. Limitations and Trade-offs

While FlashAttention-2 delivers substantial performance improvements, the approach rests on specific assumptions and exhibits constraints that practitioners should understand before deployment. This section examines what the method does not address, where performance degrades, and what open questions remain.

### 6.1 Hardware-Specific Optimization Assumptions

FlashAttention-2 is explicitly tuned for Nvidia A100 GPU architecture, which limits its immediate portability and raises questions about generalization to other hardware platforms.

**A100-Centric Tuning:**
The block size choices ($\{64, 128\} \times \{64, 128\}$) and warp partitioning strategies are optimized for A100's specific memory hierarchy:
- 192KB shared memory per SM
- 108 Streaming Multiprocessors
- 3rd-generation Tensor Cores

> "Typically we choose blocks of size {64, 128} × {64, 128}, depending on the head dimension $d$ and the device shared memory size." (Section 3.3)

This means deployment on GPUs with different shared memory capacities or SM counts requires re-tuning. The paper acknowledges this limitation explicitly:

> "We manually tune for each head dimensions since there are essentially only 4 choices for block sizes, but this could benefit from auto-tuning to avoid this manual labor. We leave this to future work." (Section 3.3)

**H100 Underutilization:**
The H100 results (Figure 7) do not exploit H100-specific hardware features:
- **TMA (Tensor Memory Accelerator)**: Hardware-assisted memory transfers
- **4th-generation Tensor Cores**: Higher matmul throughput
- **FP8 support**: Lower-precision computation

The authors state they expect "another 1.5×-2× speedup on H100 GPUs" with proper optimization (Section 4.1). This means the reported 338 TFLOPs/s on H100 is a **lower bound**, not the achievable ceiling. Practicians deploying on H100 should anticipate further gains but also recognize that additional engineering effort is required.

**No AMD GPU Support:**
The paper explicitly mentions plans to support "AMD GPUs" in future work (Section 5), confirming that FlashAttention-2 currently does not run on non-Nvidia hardware. This is a significant limitation for organizations with heterogeneous GPU fleets or cost constraints favoring AMD alternatives.

### 6.2 Sequence Length and Batch Size Dependencies

The performance benefits of FlashAttention-2 are **not uniform** across all sequence lengths and batch sizes. The optimizations target a specific workload regime.

**Diminishing Returns at Short Sequences:**
Table 1 reveals that at 2k context length, FlashAttention-2 provides only marginal improvement over the original FlashAttention:

| Model | Context | FlashAttention | FlashAttention-2 | Speedup |
|-------|---------|----------------|------------------|---------|
| GPT3-1.3B | 2k | 189 TFLOPs/s | 196 TFLOPs/s | 1.04× |
| GPT3-2.7B | 2k | 189 TFLOPs/s | 205 TFLOPs/s | 1.08× |

At 8k context, the speedup increases to 1.29×. This pattern reflects that the sequence-level parallelism and reduced non-matmul FLOPs have more impact when there are many blocks to process. For applications that primarily use short contexts (e.g., 512-2k tokens), the migration cost to FlashAttention-2 may not be justified.

**Long Sequence, Small Batch Regime:**
The sequence-level parallelism optimization specifically addresses the case where long sequences force small batch sizes, leaving GPU SMs underutilized. However, this creates an implicit assumption: **the workload must be memory-constrained enough that long sequences reduce batch size**. If an application can maintain large batch sizes even at long sequences (e.g., through gradient accumulation or multi-GPU data parallelism), the occupancy benefits diminish.

**16k Context Gap:**
While micro-benchmarks demonstrate FlashAttention-2 works at 16k sequence length (Figures 4-6), the end-to-end training experiments only validate up to 8k context (Table 1). The paper claims "we can train models with 16k longer context for the same price as previously training a 8k context model" (Section 5), but this is **extrapolated from kernel benchmarks**, not demonstrated with actual model training. There may be unanticipated bottlenecks at 16k+ context in full training pipelines (e.g., optimizer state, gradient synchronization, checkpointing).

### 6.3 Computational and Memory Trade-offs

FlashAttention-2 introduces specific computational trade-offs that affect different aspects of the training pipeline.

**Backward Pass Efficiency Gap:**
Despite improvements, the backward pass remains less efficient than the forward pass:
- **Forward pass**: Up to 73% of theoretical maximum throughput (Figure 5)
- **Backward pass**: Up to 63% of theoretical maximum throughput (Figure 6)

This 10 percentage point gap reflects the inherent complexity of the backward pass, which requires 5 matrix multiplications versus 2 in the forward pass (Section 4.1). Since training time is typically dominated by the backward pass (60-70% of iteration time), this efficiency gap means FlashAttention-2 does not fully close the performance disparity between forward and backward computation.

**Atomic Add Overhead in Backward Pass:**
The sequence-level parallelism in the backward pass requires **atomic adds** to safely accumulate dQ gradients across thread blocks (Section 3.2):

> "We use atomic adds to communicate between different thread blocks to update dQ." (Section 3.2)

Atomic operations serialize memory access and can become a bottleneck when many thread blocks contend for the same memory locations. The paper does not quantify this overhead, but it represents a fundamental trade-off: sequence parallelism improves occupancy but introduces synchronization costs. At extreme sequence lengths with many column blocks, atomic contention could theoretically degrade performance, though the benchmarks do not show this occurring up to 16k.

**Register Spilling Risk:**
Larger block sizes reduce shared memory traffic but increase register pressure:

> "Past a certain block size, register spilling causes significant slowdown, or the amount of shared memory required is larger than what the GPU has available, and the kernel cannot run at all." (Section 3.3)

This creates a **narrow tuning window** for block sizes. The manual tuning process (4 choices per head dimension) suggests limited flexibility. Applications with non-standard head dimensions or custom attention variants may require additional tuning effort that the paper does not address.

### 6.4 Unaddressed Scenarios and Edge Cases

Several important use cases and edge cases are not evaluated in the paper.

**Numerical Stability and Convergence:**
The paper provides mathematical proof that FlashAttention-2 produces exact attention output (Section 3.1.1), but it does not include:
- Training loss curves comparing FlashAttention vs. FlashAttention-2
- Convergence behavior across training steps
- Gradient norm comparisons

While the mathematical equivalence suggests identical convergence, the different order of operations (unscaled accumulator, logsumexp storage) could theoretically affect numerical stability in edge cases (e.g., very long sequences, extreme attention scores). The absence of empirical validation leaves this as an open question.

**Multi-GPU Distributed Training:**
The end-to-end experiments use 8×A100 GPUs, but the paper does not analyze how FlashAttention-2 interacts with distributed training paradigms:
- **Tensor Parallelism**: Splitting attention heads across GPUs
- **Pipeline Parallelism**: Splitting layers across GPUs
- **Sequence Parallelism**: Splitting sequence dimension across GPUs (distinct from FlashAttention-2's intra-kernel sequence parallelism)

The intra-kernel sequence parallelism (Section 3.2) could potentially conflict with inter-GPU sequence parallelism strategies used in frameworks like Megatron-LM. This interaction is not explored, creating uncertainty for large-scale distributed training deployments.

**Larger Model Sizes:**
The end-to-end validation is limited to 1.3B and 2.7B parameter models (Table 1). Modern large language models often exceed 7B, 70B, or even 175B parameters. The paper does not demonstrate whether FlashAttention-2's benefits scale proportionally to these larger models. At extreme model sizes, other bottlenecks (e.g., inter-GPU communication, optimizer state) may dominate, diminishing the relative impact of attention optimization.

**Attention Variants Beyond MQA/GQA:**
The paper supports Multi-Query Attention (MQA) and Grouped-Query Attention (GQA) (Section 3.1.2), but does not address:
- **Sliding Window Attention**: Local attention with fixed window size
- **Block-Sparse Attention**: Structured sparsity patterns
- **Cross-Attention**: Encoder-decoder attention in seq2seq models

The authors mention combining FlashAttention-2 with "local, dilated, block-sparse attention" as future work (Section 5), acknowledging these variants are not currently supported.

### 6.5 Missing Ablation Analysis

A significant methodological limitation is the **absence of ablation studies** isolating individual optimization contributions. FlashAttention-2 combines three main improvements:

1. Algorithmic tweaks (reduced non-matmul FLOPs, unscaled accumulator)
2. Sequence-level parallelism
3. Warp-level work partitioning (split-Q instead of split-K)

The paper does not report experiments like:
- "FlashAttention-2 without sequence parallelism"
- "FlashAttention-2 with original warp partitioning"
- "FlashAttention-2 with original online softmax scaling"

Without these ablations, readers cannot determine:
- Which optimization contributes most to the 2× speedup
- Whether optimizations are complementary or redundant
- Which techniques would transfer to other GPU kernels

This limits the paper's utility for researchers seeking to apply similar optimization principles to different problems. The acknowledgment that sequence parallelism was "first suggested and implemented by Phil Tillet in the Triton implementation" (Section 3.2, footnote 3) further complicates attribution of specific performance gains.

### 6.7 Open Questions for Future Work

Based on the limitations identified, several questions remain unanswered:

1. **Auto-tuning feasibility**: Can block sizes and warp configurations be automatically optimized per device, eliminating manual tuning?

2. **Numerical equivalence at scale**: Does FlashAttention-2 produce identical training trajectories to standard attention at 100k+ token sequences?

3. **Distributed training integration**: How does intra-kernel sequence parallelism interact with inter-GPU sequence parallelism in frameworks like Megatron-LM or DeepSpeed?

4. **H100-specific optimization**: What absolute throughput is achievable when exploiting TMA, 4th-gen Tensor Cores, and FP8?

5. **Larger model validation**: Do the 1.3× speedup figures hold for 7B, 70B, and 175B parameter models?

6. **Attention variant extension**: Can the work partitioning principles apply to sliding window, block-sparse, or cross-attention patterns?

These open questions represent both limitations of the current work and opportunities for future research. The FlashAttention-2 authors acknowledge several of these directions (Section 5), suggesting active development is ongoing.

## 7. Implications and Future Directions

FlashAttention-2 represents more than an incremental speedup—it signals a maturation in how the machine learning community approaches hardware-aware algorithm design. This section examines how the work reshapes the landscape of efficient Transformer training, what research directions it unlocks, where it applies in practice, and how practitioners should decide whether to adopt it.

### 7.1 Shifting the Optimization Paradigm

The most profound implication of FlashAttention-2 is its demonstration that **memory optimization alone is insufficient** for maximum hardware efficiency. The original FlashAttention solved the $O(N^2)$ memory bottleneck, yet still achieved only 25-40% of theoretical FLOPs/s utilization (Section 1). FlashAttention-2 closes this gap by treating **compute heterogeneity** as a first-order constraint.

This paradigm shift has three concrete implications for the field:

**1. Operation-Type Awareness Becomes Essential**

Prior kernel optimization often counted total FLOPs without distinguishing operation types. FlashAttention-2 reveals that on modern GPUs, **non-matmul FLOPs are 16× more expensive** than matmul FLOPs (312 TFLOPs/s vs. 19.5 TFLOPs/s on A100, Section 3.1). This means:

> "While the non-matmul FLOPs only account for a small fraction of the total FLOPs, they take longer to perform as GPUs have specialized units for matrix multiply." (Section 1)

Future kernel design must weight operations by their **hardware cost**, not just their FLOP count. This affects compiler design, automatic kernel generation, and performance modeling. A kernel with fewer total FLOPs can be slower if those FLOPs are non-matmul operations.

**2. Parallelism Strategies Must Adapt to Workload Regime**

FlashAttention-2 demonstrates that optimal parallelism depends on the **workload characteristics**, not just the algorithm. The original FlashAttention parallelized over batch and head dimensions, which works when `batch_size × num_heads` is large. However, long sequences force small batch sizes, creating an occupancy cliff.

> "In the case of long sequences (which usually means small batch sizes or small number of heads), to make better use of the multiprocessors on the GPU, we now additionally parallelize over the sequence length dimension." (Section 3.2)

This insight generalizes: kernel parallelism strategies should be **adaptive** to sequence length, batch size, and hardware configuration. Static parallelism schemes will increasingly become suboptimal as workloads diversify.

**3. Warp-Level Design Matters as Much as Algorithm Design**

The split-Q versus split-K work partitioning (Figure 3) delivers measurable speedup by eliminating inter-warp communication. This granular optimization—often left to hardware vendors or library authors—becomes a **research variable** that algorithm designers must consider.

The implication is that high-performance ML systems now require **co-design across three levels**:
- Algorithm level (online softmax reformulation)
- Thread-block level (sequence parallelism)
- Warp level (work partitioning)

Ignoring any level leaves performance on the table.

### 7.2 Enabled Research Directions

FlashAttention-2 opens several concrete research avenues, some explicitly mentioned by the authors and others implied by the technical approach.

#### 7.2.1 Hardware-Specific Optimization (Explicit Direction)

The authors state clear plans to optimize for newer hardware:

> "As an immediate next step, we plan to optimize FlashAttention-2 for H100 GPUs to use new hardware features (TMA, 4th-gen Tensor Cores, fp8)." (Section 5)

This suggests a research program of **hardware-aware kernel specialization**:

| Hardware Feature | Expected Impact | Research Question |
|-----------------|-----------------|-------------------|
| **TMA (Tensor Memory Accelerator)** | Hardware-assisted memory transfers | Can TMA eliminate manual shared memory management? |
| **4th-gen Tensor Cores** | Higher matmul throughput | What block sizes maximize 4th-gen utilization? |
| **FP8 support** | 2× memory bandwidth efficiency | Does FP8 attention maintain model quality? |

The paper reports 338 TFLOPs/s on H100 **without** using these features (Figure 7), with authors expecting "another 1.5×-2× speedup" with proper optimization (Section 4.1). This represents a clear research target: **500-650 TFLOPs/s per H100** for attention forward+backward.

#### 7.2.2 Cross-Architecture Portability (Explicit Direction)

The authors acknowledge FlashAttention-2 currently runs only on Nvidia GPUs:

> "In the near future, we plan to collaborate with researchers and engineers to make FlashAttention widely applicable in different kinds of devices (e.g., H100 GPUs, AMD GPUs)." (Section 5)

This raises research questions about **portable high-performance kernels**:
- Can the optimization principles (split-Q partitioning, sequence parallelism) transfer to AMD's CDNA architecture?
- What abstractions enable hardware-specific tuning without code duplication?
- Should compilers (e.g., Triton, TVM) automatically generate these optimizations?

The acknowledgment that sequence parallelism was "first suggested and implemented by Phil Tillet in the Triton implementation" (Section 3.2, footnote 3) suggests higher-level languages may facilitate cross-architecture portability.

#### 7.2.3 Algorithmic Combinations (Explicit Direction)

The authors propose combining FlashAttention-2 with high-level attention variants:

> "Combining the low-level optimizations in FlashAttention-2 with high-level algorithmic changes (e.g., local, dilated, block-sparse attention) could allow us to train AI models with much longer context." (Section 5)

This suggests a **layered optimization strategy**:

```
Hardware Efficiency (FlashAttention-2)
        ↓
Algorithmic Sparsity (local, block-sparse)
        ↓
Application-Specific Patterns (dilated, sliding window)
```

Research questions include:
- How do sparse attention patterns interact with warp-level work partitioning?
- Can FlashAttention-2's tiling strategy adapt to irregular sparsity patterns?
- What is the combined speedup of hardware + algorithmic optimization?

The paper currently supports MQA and GQA (Section 3.1.2), but explicitly leaves sliding window and block-sparse attention to future work.

#### 7.2.4 Compiler Integration (Explicit Direction)

The authors express interest in making optimizations "easily programmable":

> "We are also excited to work with compiler researchers to make these optimization techniques easily programmable." (Section 5)

This points to a **compiler research agenda**:
- Can compilers automatically infer optimal block sizes from hardware specs?
- Can warp partitioning decisions be abstracted from kernel authors?
- Should online softmax reformulations be compiler transformations rather than manual rewrites?

The manual tuning requirement ("we manually tune for each head dimensions," Section 3.3) represents technical debt that compiler automation could resolve.

#### 7.2.5 Implicit Research Directions

Beyond the authors' stated plans, FlashAttention-2 suggests several additional research avenues:

**Numerical Stability at Extreme Scales**

The unscaled accumulator and logsumexp storage (Section 3.1.1) change the numerical properties of attention computation. While mathematically equivalent, the **order of operations** differs from standard attention. Research questions:

- Does FlashAttention-2 produce identical training trajectories at 100k+ token sequences?
- Are there edge cases (extreme attention scores, very long sequences) where numerical divergence occurs?
- Should practitioners validate convergence empirically for new model architectures?

The paper provides no training loss curves comparing FlashAttention vs. FlashAttention-2, leaving this as an open empirical question.

**Distributed Training Interactions**

The end-to-end experiments use 8×A100 GPUs (Table 1), but the paper does not analyze interactions with distributed training paradigms:

- How does intra-kernel sequence parallelism interact with **inter-GPU sequence parallelism** (e.g., Megatron-LM's sequence parallel)?
- Does the atomic add strategy for dQ accumulation (Section 3.2) create contention in multi-GPU settings?
- Can FlashAttention-2's occupancy improvements reduce communication-to-computation ratios?

Large-scale training (7B+ parameters) may reveal bottlenecks not visible in the 1.3B-2.7B models tested.

**Inference Optimization**

The paper focuses on training throughput, but FlashAttention-2 has implications for inference:

- MQA and GQA support (Section 3.1.2) directly targets inference KV cache efficiency
- Does the 2× speedup translate to **latency reduction** for single-request inference?
- How does FlashAttention-2 interact with **continuous batching** and **speculative decoding**?

Inference workloads have different occupancy characteristics than training, potentially requiring different tuning.

### 7.3 Practical Applications and Use Cases

FlashAttention-2's performance improvements enable concrete applications that were previously cost-prohibitive or technically infeasible.

#### 7.3.1 Long-Context Language Models

The most direct application is training models with **16k+ token context**:

> "FlashAttention-2 is 2× faster than FlashAttention, which means that we can train models with 16k longer context for the same price as previously training a 8k context model." (Section 5)

Specific use cases include:

| Application | Context Requirement | FlashAttention-2 Benefit |
|-------------|--------------------|-------------------------|
| **Book understanding** | 50k-100k tokens | Makes training feasible on existing hardware |
| **Legal document analysis** | 20k-50k tokens | Reduces training cost by ~2× |
| **Code generation** | 8k-32k tokens | Enables full-file context modeling |
| **Long-form story writing** | 10k-50k tokens | Maintains coherence across chapters |

The micro-benchmarks demonstrate FlashAttention-2 works at 16k sequence length (Figures 4-6), while standard PyTorch attention hits **OOM** at this length. This is not merely a speedup—it **enables capabilities** that were previously impossible.

#### 7.3.2 High-Resolution Multimodal Models

The Introduction mentions applications beyond language:

> "Ideally, we would like to go beyond the standard 2k sequence length limit to train models to understand books, high resolution images, and long-form videos." (Section 1)

For vision transformers, sequence length scales with image resolution:
- A 1024×1024 image with 16×16 patches = 4,096 tokens
- A 2048×2048 image = 16,384 tokens

FlashAttention-2 enables **high-resolution image understanding** without aggressive downsampling or windowed attention that loses global context.

For video models, temporal sequences compound the problem:
- 30 fps video × 60 seconds = 1,800 frames
- With spatial tokens per frame, sequence length easily exceeds 100k

FlashAttention-2's linear memory scaling makes such models **trainable** rather than requiring architectural compromises.

#### 7.3.3 Existing Model Acceleration

Beyond enabling new capabilities, FlashAttention-2 accelerates **existing models**:

> "FlashAttention-2 will also speed up training, finetuning, and inference of existing models." (Section 5)

Practical scenarios:

- **Finetuning large models**: A 7B model finetuned on 8k context runs 1.3× faster (Table 1 extrapolation), reducing cloud compute costs proportionally
- **Hyperparameter sweeps**: Faster iteration enables more experiments in the same budget
- **Production inference**: Lower latency for long-context queries (e.g., document Q&A)

The 72% model FLOPs utilization (MFU) achieved (Section 4.2) is exceptionally high—many production training runs achieve 40-50% MFU. Adopting FlashAttention-2 could close this efficiency gap without algorithmic changes.

#### 7.3.4 Research Accessibility

By reducing the computational cost of long-context training, FlashAttention-2 **democratizes access** to capabilities previously limited to well-funded labs:

- Academic researchers can experiment with 8k-16k context on modest GPU clusters
- Startups can train competitive models without massive compute budgets
- Open-source model development becomes more feasible

The paper notes FlashAttention (original) saw "wide adoption in large-scale training and inference of Transformers" (Section 1). FlashAttention-2 continues this trajectory with broader impact due to larger performance gains.

### 7.4 Integration and Adoption Guidance

Practitioners should consider FlashAttention-2 when specific conditions are met. This section provides decision criteria for adoption.

#### 7.4.1 When to Use FlashAttention-2

**Strong Candidates:**

| Condition | Rationale | Expected Benefit |
|-----------|-----------|------------------|
| **Sequence length ≥ 4k** | Optimizations amplify with more blocks | 1.5-2× speedup over FlashAttention |
| **Training (not just inference)** | Backward pass benefits significantly | 2.2× backward speedup (Figure 6) |
| **Nvidia A100/H100 GPUs** | Implementation is CUDA-optimized | Full performance realized |
| **Memory-constrained workloads** | Linear memory enables longer context | Avoids OOM at 16k+ sequences |
| **MQA/GQA architectures** | Explicitly supported (Section 3.1.2) | No modification needed |

**Marginal Candidates:**

| Condition | Rationale | Expected Benefit |
|-----------|-----------|------------------|
| **Sequence length ≤ 2k** | Fewer blocks reduce optimization impact | 1.04-1.08× speedup (Table 1) |
| **Inference-only workloads** | Backward pass optimizations unused | Forward speedup only (~2×) |
| **Non-Nvidia hardware** | No AMD/Intel GPU support yet | Requires waiting for ports |
| **Very large models (70B+)** | Other bottlenecks may dominate | Untested at this scale |

#### 7.4.2 Integration Considerations

**Software Stack Requirements:**

- **PyTorch integration**: The authors thank "Driss Guessous for integrating FlashAttention to PyTorch" (Acknowledgments), indicating official PyTorch support exists or is planned
- **CUDA version**: Requires CUDA-capable GPUs with Tensor Core support (A100, H100, potentially A10, RTX 4090)
- **Precision**: Tested with FP16/BF16; FP8 support is future work (Section 5)

**Tuning Requirements:**

> "We manually tune for each head dimensions since there are essentially only 4 choices for block sizes." (Section 3.3)

Practitioners should expect to:
1. Test block sizes {64, 128} × {64, 128} for their specific head dimension
2. Monitor for register spilling (causes significant slowdown)
3. Validate numerical stability for their specific model architecture

**Backward Compatibility:**

FlashAttention-2 produces **exact attention output** with no approximation (Section 3.1.1). This means:
- Drop-in replacement for FlashAttention in most cases
- No model architecture changes required
- Training convergence should match standard attention (though not empirically validated in the paper)

#### 7.4.3 Migration Path

For teams currently using standard attention:

```
Standard Attention → FlashAttention → FlashAttention-2
      ↓                    ↓                  ↓
   OOM at 8k          8k feasible       16k feasible
   40% MFU            55% MFU           72% MFU
```

For teams already using FlashAttention:

- **If sequence length ≥ 4k**: Migration is strongly recommended (1.5-2× speedup)
- **If sequence length ≤ 2k**: Migration provides marginal benefit (1.04-1.08×)
- **If using H100**: Wait for H100-specific optimization (potential 1.5-2× additional gain)

#### 7.4.4 Risk Mitigation

Before production deployment, practitioners should:

1. **Validate numerical equivalence**: Compare training loss curves between FlashAttention and FlashAttention-2 for your specific model
2. **Test at target sequence length**: Performance scales with sequence length—benchmark at your actual workload
3. **Monitor for register spilling**: Larger block sizes may fail on GPUs with less shared memory
4. **Plan for hardware portability**: If AMD GPU support is required, budget for future migration when ports become available

### 7.5 Summary: A Foundation for the Next Scaling Phase

FlashAttention-2 does not merely improve attention—it **redefines the efficiency baseline** for Transformer training. The 2× speedup over FlashAttention and 72% MFU achievement mean that attention is no longer the primary bottleneck in long-context training.

The implications extend beyond this specific kernel:

- **Hardware-aware algorithm design** becomes a core competency for ML researchers
- **Operation-type costing** should inform performance models and compiler design
- **Adaptive parallelism** strategies will become standard as workloads diversify

The authors' forward-looking statements (Section 5) suggest this is not an endpoint but a **foundation** for further optimization. Combining FlashAttention-2's low-level efficiency with high-level algorithmic innovations (sparse attention, new architectures) could enable models with 100k+ token context at practical cost.

For practitioners, the decision framework is clear: if you train Transformers with sequence lengths above 4k on Nvidia GPUs, FlashAttention-2 should be your default attention implementation. The performance gains are substantial, the mathematical equivalence is guaranteed, and the integration path is well-trodden following FlashAttention's wide adoption.

The work exemplifies a broader trend in ML systems research: as models scale, **every layer of the stack**—from algorithm design to warp-level partitioning—becomes a lever for improvement. FlashAttention-2 demonstrates what's possible when all levers are pulled in concert.
