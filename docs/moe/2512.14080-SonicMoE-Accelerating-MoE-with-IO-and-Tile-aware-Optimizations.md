## 1. Executive Summary

SonicMoE addresses the hardware inefficiency of fine-grained and sparse Mixture of Experts (MoE) models, which suffer from increased activation memory and wasted compute due to IO costs and Grouped GEMM padding, by co-designing a memory-efficient algorithm, IO-aware GPU kernels, and a novel token rounding routing method. This approach reduces activation memory by 45% and achieves a 1.86x compute throughput improvement on Hopper GPUs for a fine-grained 7B MoE compared to ScatterMoE, enabling 64 H100s to match the training throughput of 96 H100s (213 vs. 225 billion tokens per day). Additionally, the tile-aware token rounding algorithm yields an additional 1.16x speedup on kernel execution time under high sparsity settings without sacrificing downstream performance.

## 2. Context and Motivation

### 2.1 The Core Problem: Hardware Inefficiency in Modern MoE Architectures

Mixture of Experts (MoE) models have become the standard approach for scaling language models to hundreds of billions or even trillions of parameters without proportionally increasing training compute costs. The fundamental idea is straightforward: instead of processing every token through all model parameters, a router selects only a subset of "expert" subnetworks for each token. This sparse activation means the model can have vastly more parameters than the compute budget would normally allow.

However, a critical trend has emerged in recent MoE designs that creates a hardware efficiency crisis. As shown in **Table 1** of the paper, frontier models are moving toward two simultaneous directions:

1. **Higher expert granularity**: The ratio between embedding dimension ($d$) and expert intermediate size ($n$) is increasing. For example, Mixtral 8x22B has granularity $6144/16384 = 0.38$, while Qwen3-Next-80B reaches $2048/512 = 4.00$. Smaller experts mean more diverse computation per token but also more frequent memory accesses.

2. **Higher sparsity**: The activation ratio (activated experts $K$ / total experts $E$) is decreasing. DeepSeek-V3 activates only 8 out of 256 experts (3.13%), and Kimi K2 activates 8 out of 384 experts (2.08%). This means tokens are routed to a smaller fraction of the available experts.

MoE scaling laws predict that both trends improve model quality per FLOP. Yet **Figure 3** reveals the hidden cost: as expert granularity increases, the IO cost of the forward pass scales linearly. For a 7B MoE, moving from $d/n = 1$ to $d/n = 16$ can increase IO from roughly 2 GiB to over 8 GiB per layer. This pushes MoE training into the **memory-bound regime**, where GPU Tensor Cores sit idle waiting for data from High Bandwidth Memory (HBM).

The paper identifies three specific hardware inefficiencies that compound this problem:

| Inefficiency | Root Cause | Impact |
|-------------|------------|--------|
| Activation memory explosion | Activations scale with number of activated experts ($TKd$) | Limits batch size, causes OOM |
| Low arithmetic intensity | More IO per FLOP as granularity increases | GPU compute underutilized |
| Wasted compute from padding | Grouped GEMM requires tile-aligned dimensions | 10-25% FLOPs wasted on padding |

### 2.2 Why This Matters: The Gap Between Algorithmic and Hardware Efficiency

This problem sits at the intersection of algorithmic design and systems engineering. From an algorithmic perspective, fine-grained and sparse MoEs are desirable: they offer better model quality per FLOP, enable larger parameter counts, and provide more diverse expert specialization. Recent models like DeepSeek-V3, Qwen3 MoE, and Kimi K2 demonstrate this empirically.

However, **reducing FLOPs does not directly translate to faster training**. The paper emphasizes a crucial distinction:

> "Reducing FLOPs does not directly translate to better hardware utilization since MoE computation features more dynamic IO accesses when each expert needs to gather token embeddings from different positions, and also scatter the results back to the original positions."

This creates a real-world bottleneck. Consider the training throughput comparison in the abstract: ScatterMoE requires 96 H100 GPUs to achieve 225 billion tokens per day, while SonicMoE achieves 213 billion tokens per day on only 64 H100s. For organizations training large MoE models, this 33% reduction in GPU requirements translates to millions of dollars in infrastructure savings and significantly faster iteration cycles.

The theoretical significance extends beyond cost. If hardware inefficiency limits the practical batch sizes and sequence lengths that can fit in memory, researchers cannot fully exploit the algorithmic benefits of fine-grained MoEs. The architecture becomes constrained by systems limitations rather than modeling considerations.

### 2.3 Prior Approaches and Their Limitations

Several MoE kernel implementations existed before SonicMoE, each addressing parts of the efficiency problem but falling short on the combined challenges of fine-grained, sparse MoEs.

**ScatterMoE** (Tan et al. 2024) was among the first to implement gather fusion for the forward pass, reducing the need for a separate gather kernel. However, as documented in **Table 2** and **Appendix B**, ScatterMoE has critical gaps:
- It fuses gather only for varlen-M Grouped GEMM (forward), not varlen-K (backward weight gradients)
- It does not overlap Matrix Multiply-Accumulate (MMA) operations with memory IO
- It computes the router gradient $dS$ as $\langle dO, Y \rangle$, which requires caching the down-projection output $Y$ in HBM. This causes activation memory to scale linearly with expert granularity

**MoMoE** (Costin et al. 2025) shares similar limitations. While it fuses some operations, it still computes $dS$ using the memory-intensive path and does not leverage asynchronous IO overlapping. **Figure 6** shows that MoMoE's backward pass takes 4.678ms compared to SonicMoE's 2.173ms for a 7B MoE configuration—more than 2x slower.

**MegaBlocks** (Gale et al. 2023) takes a block-sparse matrix multiplication approach. This requires explicit gather and scatter kernels with total IO cost of $8TKd$ bytes, which becomes a bottleneck for fine-grained MoEs. **Figure 6** shows MegaBlocks at 2.983ms forward pass time versus SonicMoE's 1.237ms.

**Megatron-LM** (Shoeybi et al. 2019) offers GroupedMLP using CUTLASS Grouped GEMM, but it assumes contiguously-packed inputs and does not fuse gather with the prologue. This requires a separate preprocessing step that adds latency.

**DeepGEMM** (Zhao et al. 2025b) provides highly optimized Grouped GEMM kernels but assumes inputs are already contiguous and padded to tile multiples. It does not implement Ping-Pong scheduling for overlapping MMA with heavy epilogue operations, which **Figure 19** shows results in 10% lower TFLOPS on down-projection compared to SonicMoE.

A common thread across all prior work: they were designed for coarser-grained MoEs where IO costs were less dominant. None simultaneously address (1) activation memory that scales with granularity, (2) IO-compute overlapping for memory-bound regimes, and (3) padding waste from tile quantization in sparse settings.

### 2.4 SonicMoE's Positioning: Co-Design Across Algorithm, Kernel, and Routing

SonicMoE positions itself as a **hardware-model co-design** solution rather than an incremental kernel optimization. The paper explicitly states:

> "We propose to co-design the MoE architecture with a GPU kernel tailored to NVIDIA Blackwell and Hopper generation GPUs and a novel routing method."

This co-design manifests in three integrated contributions:

1. **Memory-efficient backward algorithm**: By reformulating the computation of $dS$ as $\langle dA'_e, A_e \rangle$ instead of $\langle dO, Y \rangle$, SonicMoE eliminates the need to cache $Y$ activations. This keeps activation memory constant regardless of expert granularity—**Figure 1** shows SonicMoE's memory footprint staying flat while ScatterMoE and MoMoE increase linearly.

2. **IO-aware kernel design**: SonicMoE exploits Hopper and Blackwell GPU features (asynchronous TMA, Ping-Pong warpgroup scheduling, Tensor Memory on Blackwell) to overlap memory transfers with computation. **Figure 7** illustrates how one warpgroup performs MMA while another handles IO, then roles switch. This is particularly critical for heavy epilogue operations like the backward down-projection activation gradient kernel.

3. **Token rounding routing**: Rather than accepting padding waste from Grouped GEMM tile alignment, SonicMoE introduces a routing algorithm that rounds per-expert token counts to tile-size multiples (e.g., 128). **Figure 12** demonstrates how this eliminates the "tile quantization effect" where experts receive non-aligned token counts requiring padding. **Figure 16** shows this yields up to 16% higher TFLOPS in highly sparse regimes.

Critically, SonicMoE is **routing-agnostic** for its core computation kernels. The paper notes: "SonicMoE's MoE computation is independent of the MoE router choice and is thus compatible with arbitrary router logic." This means the token rounding method is an optional enhancement rather than a requirement, allowing practitioners to adopt the kernel optimizations without changing their routing strategy.

The positioning relative to prior work is summarized in **Table 2**, which shows SonicMoE as the only implementation that combines all key features: gather fusion for both forward and backward, epilogue fusion for activation and router gradients, MMA-IO overlapping, and efficient top-K sorting. This comprehensive approach is necessary because the inefficiencies in fine-grained, sparse MoEs are interconnected—optimizing one dimension without addressing others leaves significant performance on the table.

## 3. Technical Approach

### 3.1 Reader orientation (approachable technical breakdown)

SonicMoE is a software system that rewrites how Mixture of Experts (MoE) layers execute on NVIDIA GPUs, specifically targeting the Hopper (H100) and Blackwell architectures. It solves the problem of memory-bound MoE training by restructuring the computation graph to minimize activation caching, fusing memory operations with computation to hide IO latency, and rounding token counts to eliminate wasted compute from padding.

### 3.2 Big-picture architecture (diagram in words)

The SonicMoE system consists of three interconnected layers that operate during MoE training. First, the **memory-efficient algorithm layer** reformulates the backward pass computation to avoid caching large intermediate activations, reducing memory footprint from linear to constant with respect to expert granularity. Second, the **IO-aware kernel layer** implements eight specialized GPU kernels that fuse gather operations with memory loads, overlap matrix multiplication with asynchronous IO transfers, and combine activation functions with gradient computations in the epilogue stage. Third, the **token rounding routing layer** sits above the computation kernels and adjusts how tokens are assigned to experts so that each expert receives a tile-aligned number of tokens, eliminating padding waste in Grouped GEMM operations. Information flows from the router (which produces token-to-expert assignments) through the token rounding module (which adjusts counts to tile multiples) into the forward kernels (up-projection, down-projection, expert aggregation) and then through the backward kernels (activation gradients, weight gradients) during backpropagation.

#### 3.4.1 Memory-Efficient Backward Algorithm

The core insight behind SonicMoE's memory efficiency is that standard MoE implementations cache intermediate activations that scale linearly with expert granularity, but these activations are not mathematically necessary for computing gradients. In a standard MoE forward pass, each token $t$ receives routing scores $S_{t,e}$ for each expert $e$, and the top-$K$ experts are selected via a binary mask $\pi_{t,e}$ where $\pi_{t,e} = 1$ if token $t$ is routed to expert $e$ and $0$ otherwise. The forward computation for expert $e$ proceeds through three stages: up-projection produces $H_e = X_e W_{1,e}$ where $X_e \in \mathbb{R}^{T_e \times d}$ contains the tokens routed to expert $e$, SwiGLU activation produces $A_e = \text{SwiGLU}(H_e)$, and down-projection produces $Y_e = A_e W_{2,e}$ where $Y_e \in \mathbb{R}^{T_e \times d}$. The final output for token $t$ aggregates across experts as $O_t = \sum_{e \in [E]} \pi_{t,e} S_{t,e} Y_{e,t}$.

During backpropagation, the gradient with respect to the router scores $dS$ must be computed. Standard implementations like ScatterMoE and MoMoE compute this as $dS_{t,e} = \langle dO_t, Y_{e,t} \rangle$, which requires both the output gradient $dO$ and the down-projection output $Y$ to be loaded from HBM. The problem is that $Y$ has size $TKd$ bytes, and since $K$ (number of activated experts) increases proportionally with expert granularity to maintain constant FLOPs, the activation memory for $Y$ scales linearly with granularity. SonicMoE instead derives an equivalent computation path that avoids $Y$ entirely. By substituting $dY_{e,t} = S_{t,e} dO_t$ into the gradient chain rule, SonicMoE computes an intermediate value $dA'_e = dO_e W_{2,e}^\top$ and then calculates $dS_{t,e} = \langle dA'_{e,t}, A_{e,t} \rangle$. This reformulation requires only $A_e$ (size $TKn$) and $dA'_e$ (size $Tn$) instead of $Y_e$ (size $TKd$). Since $n$ is the expert intermediate dimension and $d$ is the embedding dimension, and typical MoE configurations have $d > n$ (often $d/n = 2$ to $4$ as shown in Table 1), this reduces activation memory substantially.

The complete set of cached activations in SonicMoE totals $2Td + 4TKn$ bytes per layer: $X$ and $H$ for the forward pass, plus routing metadata $\pi$ and $S$. Critically, this activation memory is independent of expert granularity because $Kn$ remains constant under iso-FLOPs scaling (when granularity $d/n$ increases, $n$ decreases and $K$ increases proportionally). Figure 13 demonstrates this empirically: for a 7B MoE with $n=256$, SonicMoE reduces activation memory by 45% compared to ScatterMoE, and the memory footprint stays constant as $n$ varies from 256 to 1024 while ScatterMoE's memory increases linearly.

SonicMoE also avoids materializing certain intermediate gradients in HBM. The gradient $dY$ (activation gradient for down-projection output) and $dO_e$ (gathered output gradient per expert) are computed on-the-fly rather than written to HBM. For $X$ and $dO$, SonicMoE fuses the gather operation with the HBM-to-SMEM load, eliminating the need to materialize gathered tensors in global memory. This gather fusion reduces IO cost by $2TKd$ bytes and is particularly important for fine-grained MoEs where $K$ is large.

#### 3.4.2 Eight-Kernel Computational Workflow

SonicMoE launches exactly eight CUDA kernels per MoE layer during training: three for the forward pass and five for the backward pass. Figure 4 illustrates this computational workflow with yellow boxes grouping related operations.

**Forward pass kernels (3 total):**
1. **Up-proj A kernel**: This kernel performs gather + varlen-M Grouped GEMM + SwiGLU activation. It loads input tokens $X$, router mask $\pi$, and up-projection weights $W_1$ from HBM. For each expert $e$, it gathers the tokens assigned to that expert via $X_e = \text{Gather}(X, \pi_{:,e})$, computes $H_e = X_e W_{1,e}$ using varlen-M Grouped GEMM (variable token dimension $M$, fixed expert dimensions), applies SwiGLU activation to produce $A_e$, and stores both $H_e$ and $A_e$ to HBM for use in backward pass. The fusion of gather with GEMM prologue eliminates a separate gather kernel launch.
2. **Down-proj Y kernel**: This kernel performs varlen-M Grouped GEMM for the down-projection. It loads $A_e$ and down-projection weights $W_2$ from HBM, computes $Y_e = A_e W_{2,e}$, and stores $Y_e$ to HBM. Unlike the up-proj kernel, this does not require gather fusion because $A_e$ is already contiguously packed per expert from the previous kernel.
3. **Expert aggregation O kernel**: This kernel computes the final output by gathering and summing expert outputs per token. For each token $t$, it loads $Y_{e,t}$, $S_{t,e}$, and $\pi_{t,e}$ for all activated experts, computes $O_t = \sum_{e \in [E]} \pi_{t,e} S_{t,e} Y_{e,t}$, and stores $O_t$ to HBM. This is a gather-and-sum operation rather than a scatter operation, which SonicMoE chooses for reasons explained in Section 3.4.4.

**Backward pass kernels (5 total):**
1. **Down-proj act dH kernel**: This is SonicMoE's most heavily fused backward kernel. It computes the activation gradient $dH$, the router gradient $dS$, and an intermediate value $A'$ needed for weight gradients. The kernel loads $dO$, $W_2$, $S$, and $\pi$ from HBM. For each expert, it gathers $dO_e = \text{Gather}(dO, \pi_{:,e})$, computes $dA'_e = dO_e W_{2,e}^\top$, applies broadcast scaling $dA_e = \text{Broadcast}(s_e) dA'_e$, then simultaneously computes the forward activation $A_e$ (recomputed from cached $H_e$) and backward activation gradient $dH_e = \text{dSwiGLU}(dA_e, H_e)$. It also computes $dS_{e,t} = \langle dA'_{e,t}, A_{e,t} \rangle$ by reducing over the $n$ dimension. The kernel stores $dH_e$, $dS$, and $A'_e$ to HBM. This heavy epilogue fusion is what enables SonicMoE's 0.47ms runtime versus 0.99ms for ScatterMoE's separate kernels (Figure 6).
2. **Down-proj weight dW2 kernel**: This kernel computes weight gradients for $W_2$ using varlen-K Grouped GEMM. It loads $dO_e$, $A'_e$, and $\pi$ from HBM, gathers $dO_e$, and computes $dW_{2,e} = A'^\top_e dO^\top_e$. The varlen-K designation indicates that the reduction dimension $K$ (tokens per expert) varies across experts while the other dimensions are fixed.
3. **Up-proj act dX̃ kernel**: This kernel computes the activation gradient for the up-projection input. It loads $dH_e$ and $W_1$ from HBM and computes $d\tilde{X}_e = dH_e W_{1,e}^\top$ using varlen-M Grouped GEMM. The tilde notation indicates this is the per-expert gradient before aggregation across experts.
4. **Up-proj weight dW1 kernel**: This kernel computes weight gradients for $W_1$ using varlen-K Grouped GEMM with gather fusion. It loads $X$, $\pi$, and $dH_e$ from HBM, gathers $X_e = \text{Gather}(X, \pi_{:,e})$, and computes $dW_{1,e} = X^\top_e dH_e$. This is one of the two backward kernels that requires gather fusion (the other being dH).
5. **Expert aggregation dX kernel**: This kernel aggregates the per-expert input gradients back to per-token gradients. For each token $t$, it loads $d\tilde{X}_{e,t}$ and $\pi_{t,e}$ for all activated experts, computes $dX_t = \sum_{e \in [E]} \pi_{t,e} d\tilde{X}_{e,t}$, and stores $dX_t$ to HBM.

Algorithm 2, Algorithm 3, and Algorithm 5 in the paper provide pseudocode for the forward pass, down-projection backward, and up-projection backward respectively. The key observation is that all eight kernels are modular and can be dispatched with optimal GEMM configurations and load/store strategies depending on the specific MoE configuration (values of $T$, $d$, $n$, $E$, $K$).

#### 3.4.3 IO-Aware Kernel Design: Gather Fusion

SonicMoE's Grouped GEMM kernels accept either contiguously-packed inputs or inputs gathered from different positions in the input tensor. For the latter case, SonicMoE fuses the input gather operation with the input loads from global memory (GMEM, which is physically HBM) to shared memory (SMEM). This fusion is critical because a separate gather kernel would require writing $2TKd$ bytes to HBM and then reading them back for the GEMM computation.

The gather fusion mechanism works as follows. During the GEMM prologue stage, producer warpgroups fetch both the routed token indices (from $\pi$) and the corresponding activation values (from $X$ or $dO$) using NVIDIA's `cp.async` instruction. On Hopper GPUs, the `cp.async` instruction allows asynchronous copying from global memory to shared memory, which can proceed in parallel with GEMM computation in the consumer warpgroups. The producer warpgroups prefetch token indices cooperatively, then use these indices to issue `cp.async` calls that gather the activation values directly into SMEM tiles. Figure 18 in Appendix D illustrates the index prefetching strategies: for gather on the $M$ dimension (varlen-M GEMM), each thread independently prefetches indices to its own registers before the mainloop; for gather on the $K$ dimension (varlen-K GEMM), four producer warps cooperatively prefetch indices to an SMEM buffer, and each producer thread reads from this buffer to its registers.

On Blackwell GPUs, gather fusion encounters an architectural challenge when using 2-CTA (Cooperative Thread Array) clusters for GEMM computation. The `cp.async` instruction can only signal completion within the same CTA, but Blackwell's 2-CTA GEMM requires the MMA (Matrix Multiply-Accumulate) instruction in the leader CTA (CTA 0) to wait for gather completion from both CTAs. SonicMoE work around this by dedicating a relay warp in CTA 1 that receives the `cp.async` completion signal and forwards it to CTA 0's MMA warp using cluster-level synchronization primitives like `mbarrier` with cluster scope. Figure 5 illustrates this pipeline structure. This relay mechanism adds scheduling complexity but enables efficient gather fusion across the 2-CTA cluster.

The performance impact of gather fusion is substantial. Figure 20 shows that for varlen-M Grouped GEMM (forward up-projection), SonicMoE with gather fusion achieves an average 6.3% higher TFLOPS than without gather fusion. More importantly, when comparing against baselines that use separate gather kernels, SonicMoE achieves 38.3% higher TFLOPS than DeepGEMM, 30.9% higher than MoMoE, and 9.7% higher than ScatterMoE on average. For varlen-K Grouped GEMM (backward weight gradients), the advantage is even larger: SonicMoE with gather fusion achieves 71.8% higher TFLOPS than DeepGEMM, 42.4% higher than MoMoE, and 55.1% higher than ScatterMoE on average. The gap widens as expert granularity increases (moving from right to left in Figure 20's bar groups) because higher granularity means larger $K$ and thus more gather operations.

#### 3.4.4 IO-Aware Kernel Design: Epilogue Fusion and MMA-IO Overlap

The epilogue stage of a GEMM kernel handles post-processing operations on the matrix multiplication results, such as applying activation functions, computing gradients, and writing results back to HBM. In MoE kernels, the epilogue can be heavy relative to the mainloop computation, especially for fine-grained MoEs where the GEMM tile sizes are small. SonicMoE exploits two techniques to handle heavy epilogues efficiently: epilogue operation fusion and Ping-Pong warpgroup scheduling for overlapping MMA with IO.

**Epilogue fusion:** SonicMoE fuses multiple operations into the epilogue stage to reduce unnecessary HBM accesses. For the forward up-projection, SwiGLU activation is fused with the GEMM epilogue rather than launched as a separate kernel. Figure 6 shows that DeepGEMM++ requires 0.60ms total for separate up-proj (0.49ms) and SwiGLU (0.11ms) kernels, while SonicMoE completes both in 0.55ms despite also including gather fusion. For the backward down-projection activation gradient (dH kernel), SonicMoE fuses the computation of $dH$, $dS$, and $A'$ into a single epilogue. This is the "heavy epilogue fusion" that enables SonicMoE's 0.47ms runtime versus 0.99ms for ScatterMoE's three separate kernels (down-proj act, dS, and dSwiGLU). The mathematical basis for computing $dS$ in the epilogue is derived in Appendix C: $dS_{t,e} = \langle dA'_{e,t}, A_{e,t} \rangle$ can be computed from values already in registers or SMEM during the dH kernel, avoiding a separate $2TKd$ byte HBM load that ScatterMoE and MoMoE require for their $\langle dO, Y \rangle$ formulation.

**Ping-Pong warpgroup scheduling:** On Hopper GPUs, GEMM is performed asynchronously with a producer-consumer paradigm where producer warpgroups load data tiles from HBM to SMEM while consumer warpgroups perform MMA computation. SonicMoE uses Ping-Pong scheduling when the epilogue is heavy relative to the mainloop. Suppose there are two consumer warpgroups. In Ping-Pong scheduling, one warpgroup performs MMA computation while the other warpgroup handles epilogue IO (loading or storing data from/to HBM). Once the MMA warpgroup finishes, it signals the epilogue warpgroup to start, and then the roles switch. Figure 7 illustrates this scheduling pattern. This is particularly useful for the down-projection forward Y kernel (heavy HBM store IO of $2TKd$ bytes) and the backward dH kernel (heavy HBM load of $H$ and $S$, plus multiple activation and reduction operations).

The alternative to Ping-Pong scheduling is cooperative scheduling, where both warpgroups cooperatively issue the WGMMA (Warp Group Matrix Multiply-Accumulate) instruction with a larger tile size. Cooperative scheduling is better for kernels with long mainloops and light epilogues (like dW1 and dW2 weight gradient kernels), while Ping-Pong is better for kernels with heavy epilogues (like Y and dH). SonicMoE dynamically selects between these scheduling modes based on the kernel characteristics.

**Asynchronous TMA operations:** SonicMoE relies on asynchronous TMA (Tensor Memory Access) operations for GMEM-to-SMEM loads and SMEM-to-GMEM stores. TMA is a hardware unit on Hopper and Blackwell GPUs that handles memory transfers independently from the compute units, enabling true overlap of IO and computation. SonicMoE overlaps the following asynchronous IO with MMA operations:
- Asynchronous TMA load during dH kernel's epilogue: A dedicated pipeline loads $H$ to compute $dH$ from $dA$, overlapping with other epilogue operations.
- Asynchronous TMA store in forward down-proj Y and backward up-proj activation gradient dX̃ kernels: SonicMoE applies asynchronous TMA store for all six Grouped GEMMs.

Critically, SonicMoE does **not** fuse scatter operations with HBM store in the GEMM epilogue, unlike ScatterMoE and MoMoE. The reason is that scatter fusion requires synchronous index fetching and address calculations, and on Hopper GPUs it requires the synchronous `st.global` PTX instruction for SMEM-to-GMEM store. This synchronous store blocks the execution of MMA for the next tile and degrades TFLOPS by approximately 20% in the case of heavy HBM store. Figure 8 illustrates this: asynchronous TMA store (top) has higher memory bandwidth and can naturally overlap with Tensor Core MMA, while synchronous `st.global` (bottom) blocks the execution of the next Tensor Core MMA tile. Instead of scatter fusion, SonicMoE lets each expert directly store contiguously-packed outputs via TMA in the GEMM epilogue, then performs a separate expert aggregation kernel where each token gathers and sums over activated expert outputs. Figure 22 shows this strategy achieves 17% speedup over the scatter-fusion approach, and SonicMoE achieves 1.75x and 3.11x speedup respectively over ScatterMoE and MoMoE on the forward down-projection kernel.

**Blackwell adaptations:** On Blackwell GPUs, GEMM kernels use a similar Ping-Pong scheduling concept but with different implementation due to Tensor Memory (TMEM). TMEM is a dedicated 256KB on-chip memory per SM organized as 512 columns × 128 rows of 32-bit cells. Accumulator results from matrix multiplication are stored directly in TMEM rather than in registers, with the 512-column structure enabling a two-stage accumulator pipeline. Each stage uses 256 columns: while one stage performs MMA operations via the UMMA (Unified Matrix Multiply-Accumulate) instruction, the other stage executes the epilogue. Unlike Hopper's WGMMA which required warpgroup-level coordination and consumed significant register memory, Blackwell's UMMA is a single-threaded asynchronous operation that eliminates register pressure for accumulation. This allows epilogue warps to read and process results from one TMEM stage concurrently with MMA warps accumulating into the other stage, enabling better overlap of epilogue and MMA operations compared to Hopper's Ping-Pong scheduling.

#### 3.4.5 Token Rounding Routing Algorithm

The token rounding (TR) routing method addresses wasted compute from padding in Grouped GEMM operations. Modern GPUs compute GEMM in tiles, and if any dimension ($M$, $N$, or $K$) is not divisible by the tile size, the kernel must pad to the next tile multiple. For MoE, the token dimension per expert ($T_e$) varies across experts, and when $T_e$ is not a multiple of the tile size $M_{\text{tile}}$ (typically 128), the kernel wastes FLOPs on padding. Figure 12 illustrates this "tile quantization effect": if an expert receives 337 tokens and $M_{\text{tile}} = 128$, the kernel must pad to 384 tokens (3 tiles), wasting computation on 47 padding tokens. As MoEs become sparser (smaller activation ratio $\rho = K/E$), the expected tokens per expert $\bar{T}_e = T\rho$ decreases, making the padding waste proportionally larger. Figure 11 shows that wasted FLOPs from padding can reach 15-20% for highly sparse configurations.

Algorithm 4 describes the token rounding algorithm as a two-step sorting procedure:

**Step 1: Top-K token choice sorting.** The algorithm first computes vanilla token-choice (TC) top-$K$ routing results $(S_{\text{topK}}, I_{\text{topK}}) = \text{TopK}(S, K)$ where $S \in [0, 1]^{T \times E}$ is the router score matrix and $K$ is the number of activated experts per token. This produces the initial token-to-expert assignments.

**Step 2: Calculate expert frequencies and tile-rounded multiples.** For each expert $e$, the algorithm calculates the received token frequency $f_e = \sum_t \mathbb{1}\{e \in I_{\text{topK}, t}\}$. It then computes the floor and ceiling tile-multiples: $\lfloor f_e \rfloor_{M_{\text{tile}}} = \lfloor f_e / M_{\text{tile}} \rfloor \cdot M_{\text{tile}}$ and $\lceil f_e \rceil_{M_{\text{tile}}} = \lceil f_e / M_{\text{tile}} \rceil \cdot M_{\text{tile}}$.

**Step 3: Build Top-K-preferred scores.** The algorithm constructs a modified score matrix $S'$ where non-top-$K$ entries are set to $S_e - 1$ (ensuring they are smaller than top-$K$ entries), and top-$K$ entries retain their original scores. This ensures that during the second sorting step, tokens originally selected by TC routing are preferred over tokens that would be added via expert-choice (EC) padding.

**Step 4: Token rounding per expert.** For each expert $e$, the algorithm sorts tokens by their modified scores to get an ordering $\pi_e$ and sorted scores $s_e$. It then calls a `round and sparsify` subroutine that makes a binary decision: either discard tokens from the TC selection (rounding down to $\lfloor f_e \rfloor_{M_{\text{tile}}}$) or pad with additional EC-selected tokens (rounding up to $\lceil f_e \rceil_{M_{\text{tile}}}$). The default rounding subroutine is "nearest rounding via expert frequency" (NR-f): pad EC tokens if $\lceil f_e \rceil_{M_{\text{tile}}} - f_e &lt; f_e - \lfloor f_e \rfloor_{M_{\text{tile}}}$, otherwise discard TC tokens.

The key property of token rounding is that for each expert, the maximum deviation from the original token-choice routing is at most one tile ($M_{\text{tile}}$ tokens). This bounded deviation preserves model quality while eliminating padding waste. Table 3 validates this empirically across multiple model scales (0.5B to 1.8B parameters) and sparsity levels (activation ratios from 8/64 = 12.5% to 2/128 = 1.56%). In extremely sparse settings (Table 3c and 3e with $K/E &lt; 1/32$), TR actually achieves slightly lower validation perplexity and higher average accuracy than TC top-$K$ routing.

Table 6 ablates different rounding subroutines: NR-f (nearest rounding via frequency), SR-f (stochastic rounding via frequency), NR-s (nearest rounding via scores), Balance-f (balanced rounding that preserves total token count), UP (always round up), and DOWN (always round down). The results show that TR is robust to the specific rounding subroutine, with NR-f providing the best balance of training efficiency and model quality. Always discarding tokens (DOWN) yields shorter kernel runtime but higher validation perplexity, while always padding (UP) yields longer runtime without necessarily better quality.

The training throughput benefit of token rounding increases with sparsity. Figure 16 benchmarks TR against TC top-$K$ routing under iso-FLOPs settings (keeping $T$, $n$, and $K$ constant while increasing $E$). For a MoE with 128 experts ($K/E = 1/64$) and $n=1024$, TR achieves 16.5% higher model TFLOPS in forward pass and 6.1% in backward pass, resulting in 9.4% end-to-end improvement. For 256 experts ($K/E = 1/128$), the improvement reaches 25.7% forward and 11.8% backward, or 15.9% end-to-end. Figure 17 shows similar trends for configurations from recent open-source MoEs: for Qwen3-Next-80B-A3B-Thinking ($K/E = 10/512$), TR achieves 19.6% and 7.9% speedup over TC top-$K$ during forward and backward pass respectively.

Token rounding is applied at the microbatch level, so the microbatch size $T$ affects the qualitative results. Table 7 ablates microbatch size while keeping minibatch size (tokens per optimization step) constant. The results show that TR preserves model quality when $\bar{T}_e / M_{\text{tile}} \geq 2$, but there is noticeable degradation when $\bar{T}_e / M_{\text{tile}} = 1$ (e.g., $\bar{T}_e = 128$ with $M_{\text{tile}} = 128$). Even in this edge case, TR still outperforms EC routing with finetuned TC router. Table 8 ablates tile size $M_{\text{tile}}$ and finds similar robustness when $\bar{T}_e / M_{\text{tile}} \geq 2$.

#### 3.4.6 Efficient Top-K Sorting Kernel

SonicMoE implements a custom top-$K$ sorting kernel to reduce router computation overhead. Existing MoE approaches like ScatterMoE, MoMoE, and MegaBlocks use PyTorch's `torch.topk`, which can take approximately 40% of the router's computation time. SonicMoE's top-$K$ kernel supports $E \leq 4096$ experts and $K \leq 16$ activated experts, optimized for large token counts $T$.

The kernel accepts router output with shape $(T, E)$ and parallelizes over $T$ (tokens). It uses bitonic sort (a parallel sorting algorithm) over every row to sort $E$ values, then selects the first $K$ columns as the output. After loading the input, the kernel packs the column indices of the first $K$ columns into the lower $\log_2(E)$ mantissa bits of FP32 values in registers. Figure 10 illustrates this value format: the raw FP32 score is modified so that the lower mantissa bits encode the column index. This ensures stable sorting because no two values will be equal after packing (each token-expert pair has a unique index). The bitonic compare and merge operations occur within the same thread or warp via warp-shuffle instructions, using only intra-thread or intra-warp registers without SMEM access. This achieves higher memory bandwidth than alternative implementations like PyTorch TopK, Triton official examples, Tilelang official examples, and RTop-K, as shown in Figure 23.

The kernel also offers optional softmax fusion on top-$K$ values within the same kernel, further reducing memory traffic. For base sorting cases with $\leq 64$ values, the kernel follows comparison strategies from optimal low-latency sorting networks, which provide the minimum number of parallel operation steps and compare-swap calls.

## 4. Key Insights and Innovations

SonicMoE's contributions extend beyond incremental kernel optimizations. The paper reveals fundamental insights about the relationship between MoE architecture design and hardware execution that challenge conventional assumptions. This section distinguishes between what is genuinely novel versus what represents engineering refinement, and explains why these distinctions matter for future MoE system design.

### 4.1 Breaking the Granularity-Memory Coupling: A Mathematical Reformulation

**Prior assumption:** Activation memory for MoE backward pass must scale linearly with the number of activated experts $K$. Since fine-grained MoEs require larger $K$ to maintain constant FLOPs (as expert intermediate dimension $n$ decreases), activation memory was considered an unavoidable cost of granularity.

**SonicMoE's insight:** This coupling is a *computational choice*, not a *mathematical necessity*. The paper demonstrates that the router gradient $dS$ can be computed via $\langle dA'_e, A_e \rangle$ instead of $\langle dO, Y \rangle$, eliminating the need to cache $Y$ activations whose size $TKd$ scales with $K$.

**Why this is fundamental:** This is not an incremental optimization but a reformulation of the computation graph. Prior kernels (ScatterMoE, MoMoE, MegaBlocks) all compute $dS$ using the memory-intensive path because they follow the standard autograd chain rule without questioning whether alternative paths exist. SonicMoE's derivation in **Appendix C** shows that multiple mathematically equivalent computation paths exist, and the choice between them has profound implications for memory scaling.

**Significance:** The 45% activation memory reduction for 7B MoE (**Figure 13**) is not the full story. The critical insight is that activation memory now stays *constant* as granularity increases, whereas all prior implementations show linear growth (**Figure 1**, left). This removes a hard constraint that previously limited how fine-grained MoE architectures could practically become. Researchers can now pursue higher granularity for model quality without hitting memory walls—a capability that did not exist before.

**Incremental vs. Fundamental:** Memory-efficient kernels existed before (e.g., activation checkpointing), but those trade compute for memory by recomputing activations. SonicMoE achieves memory efficiency *without* additional FLOPs, making this a fundamental algorithmic innovation rather than an engineering trade-off.

### 4.2 Token Rounding: Reframing Routing as Hardware Constraint Satisfaction

**Prior assumption:** Routing algorithms should optimize purely for model quality—select the top-$K$ experts per token based on router scores. Hardware inefficiencies (padding, tile misalignment) are accepted as unavoidable overhead.

**SonicMoE's insight:** Routing can be formulated as a *constrained optimization problem* where token assignments must satisfy hardware constraints (tile alignment) while minimizing deviation from the ideal top-$K$ selection. The token rounding algorithm treats the GEMM tile size $M_{\text{tile}}$ as a first-class constraint in the routing decision.

**Why this is fundamental:** This represents a paradigm shift in how we think about MoE routing. Prior work treats routing and kernel execution as separate concerns: the router produces assignments, and the kernel executes them (with padding waste). Token rounding *integrates* hardware awareness into the routing logic itself. **Algorithm 4** shows that the algorithm makes binary decisions (pad or discard) at the expert level to ensure tile-aligned token counts, with bounded deviation (at most one tile per expert).

**Significance:** The 16% end-to-end throughput improvement in highly sparse regimes (**Figure 16**) comes without model quality degradation (**Table 3**). More importantly, this demonstrates that hardware-aware routing can *improve* effective training efficiency without compromising the algorithmic benefits of sparse MoEs. The insight generalizes beyond MoEs: any system with dynamic computation graphs and tile-based hardware execution could benefit from similar constraint-aware scheduling.

**Incremental vs. Fundamental:** Prior work like Rectify-Router (Zeng et al. 2024a) drops and reroutes tokens, but does not consider tile structure. TMA-adaptive FP8 Grouped GEMM (Fu et al. 2025) reduces padding-related load traffic but does not address wasted FLOPs from non-aligned tile sizes. Token rounding is the first method to explicitly optimize for GEMM tile alignment at the routing level, making this a novel contribution rather than an extension of existing token-dropping techniques.

### 4.3 IO-Compute Overlap as a First-Class Design Principle

**Prior assumption:** GEMM kernels should maximize compute throughput (TFLOPS). Memory IO is handled in prologue/epilogue stages, with overlapping considered an implementation detail.

**SonicMoE's insight:** For fine-grained MoEs, the system is *memory-bound*, not compute-bound. Kernel design should prioritize hiding IO latency over maximizing raw TFLOPS. This requires treating IO-compute overlap as a central design principle, not an optimization pass.

**Why this is fundamental:** The paper's analysis in **Section 2.2** and **Figure 3** shows that arithmetic intensity decreases linearly with expert granularity. When arithmetic intensity drops below a threshold (approximately 2-3 FLOPs/byte for H100), the kernel becomes memory-bound, and TFLOPS become irrelevant—what matters is memory bandwidth utilization. SonicMoE's Ping-Pong scheduling (**Figure 7**) and asynchronous TMA operations are designed specifically for this regime.

**Significance:** The 83% and 115% backward pass speedup over ScatterMoE and MoMoE respectively (**Abstract**) comes not from faster GEMM but from better IO hiding. **Figure 6** shows that SonicMoE's dH kernel achieves 328 TFLOPS while ScatterMoE's separate kernels achieve higher individual TFLOPS but take 2x longer total time. This demonstrates that optimizing for memory-bound workloads requires different metrics than compute-bound workloads—a insight that applies broadly to sparse model training beyond MoEs.

**Incremental vs. Fundamental:** Ping-Pong scheduling exists in CUTLASS and FlashAttention-3. However, applying it specifically to MoE kernels with heavy epilogue operations (dS computation, activation gradients) and combining it with gather fusion is novel. The key innovation is recognizing that MoE's unique computation pattern (variable token counts per expert, heavy gather/scatter) requires customized overlap strategies rather than generic GEMM optimizations.

### 4.4 Routing-Computation Decoupling: Modular Adoption Path

**Prior assumption:** Kernel optimizations and routing algorithms are tightly coupled. Adopting a new kernel often requires changing the routing logic, and vice versa.

**SonicMoE's insight:** The MoE computation kernels can be made *routing-agnostic*. As stated in **Section 3.1**: "SonicMoE's MoE computation is independent of the MoE router choice and is thus compatible with arbitrary router logic." Token rounding is an optional enhancement layered on top of the core kernels.

**Why this is fundamental:** This decoupling enables incremental adoption. Practitioners can adopt SonicMoE's memory-efficient kernels without changing their routing strategy, or adopt token rounding without rewriting their entire training pipeline. This contrasts with prior systems where kernel and routing optimizations were monolithic.

**Significance:** The open-source release includes a PyTorch interface that accepts arbitrary routing input. This design choice lowers the barrier to adoption and enables the community to experiment with token rounding on existing MoE models without full retraining. The ablation in **Table 3** shows that models trained with token rounding can switch to vanilla top-$K$ routing at inference time with minimal quality loss, further increasing practical utility.

**Incremental vs. Fundamental:** Modular kernel design is common in deep learning frameworks. However, explicitly designing MoE kernels to be routing-agnostic while still achieving state-of-the-art performance is novel. Prior MoE kernels (ScatterMoE, MoMoE) have implicit assumptions about routing that make them less flexible. SonicMoE's modularity is a deliberate architectural choice that enables broader impact.

### 4.5 Synthesis: Quality per Compute Hour vs. Quality per FLOP

The paper's conclusion articulates a broader vision that synthesizes these insights:

> "We envision future model architecture designs that optimize for quality per compute hour rather than just quality per FLOP—jointly considering algorithmic and hardware efficiency."

This statement captures the fundamental shift SonicMoE represents. Prior MoE research optimizes for *quality per FLOP* (algorithmic efficiency), assuming that reducing FLOPs automatically translates to faster training. SonicMoE demonstrates that *quality per compute hour* (hardware efficiency) requires co-design across algorithm, kernel, and routing layers.

**Why this matters:** As MoE models continue scaling toward trillions of parameters, the gap between algorithmic FLOPs and hardware execution time will only widen. SonicMoE's co-design approach provides a template for future architectures: identify hardware bottlenecks (memory bandwidth, tile alignment, activation memory), then reformulate algorithms and kernels to address them jointly rather than sequentially.

**Limitations and open questions:** The paper acknowledges that token rounding is not directly compatible with autoregressive inference (requires switching to top-$K$ at evaluation time, **Footnote 28**). Additionally, the evaluation focuses on BF16 precision; extending to FP8 and microscaling formats (MXFP8, MXFP4) is listed as future work (**Section 7**). These limitations highlight that SonicMoE is a foundational step rather than a complete solution—the insights are transferable, but specific implementations will need adaptation for different hardware generations and precision formats.

## 5. Experimental Analysis

This section examines how SonicMoE validates its three core claims: (1) activation memory independent of expert granularity, (2) superior training throughput via IO-aware kernels, and (3) token rounding that eliminates padding waste without quality loss. The experiments span kernel-level microbenchmarks, end-to-end training throughput measurements, and downstream task quality evaluations across multiple model scales.

### 5.1 Evaluation Methodology

**Datasets and Training Setup.** All quality experiments use a deduplicated version of FineWeb-Edu as the pretraining corpus (Appendix H). Models are trained with context length 4096 tokens, SwiGLU activation for MoE layers, and auxiliary load balancing loss with coefficient 0.01 (no router Z-loss). The OLMoE codebase serves as the foundation, with models ranging from 0.5B to 1.8B parameters for quality evaluation and 1.4B to 120B for kernel benchmarks. Table 10 in Appendix H provides hyperparameter details: learning rates range from 4e-4 to 6e-4, weight decay is 0.01, and learning rate follows cosine schedule with 10% warmup steps.

**Baselines.** The paper compares against five prior MoE kernel implementations:
- **ScatterMoE** (Tan et al. 2024): Gather fusion for forward pass only, computes $dS = \langle dO, Y \rangle$
- **MoMoE** (Costin et al. 2025): Similar to ScatterMoE with different fusion strategy
- **MegaBlocks** (Gale et al. 2023): Block-sparse matrix multiplication approach
- **Megatron-LM** (Shoeybi et al. 2019): GroupedMLP using CUTLASS Grouped GEMM
- **DeepGEMM** (Zhao et al. 2025b): Optimized Grouped GEMM for contiguous inputs

For token rounding evaluation, the paper includes three routing baselines from Table 3: vanilla **TC top-K** routing, **EC (aux router)** (expert choice with auxiliary router for inference), and **EC (ft TC router)** (expert choice with finetuned token-choice router). A **TC (token drop)** baseline always rounds down expert frequencies.

**Metrics.** The evaluation uses four primary metric categories:
1. **Activation memory** (GiB per layer): Peak memory required for backward pass
2. **Training throughput** (TFLOPS, tokens/day): Kernel-level and end-to-end measurements
3. **Kernel runtime** (ms): Breakdown of forward and backward pass components
4. **Model quality** (validation perplexity, downstream task accuracy): 11 tasks including WinoGrande, Social IQA, SciQ, PIQA, OpenBookQA, HellaSwag, COPA, CommonsenseQA, BoolQ, Arc-Easy, and Arc-Challenge

**Hardware.** All kernel benchmarks run on NVIDIA H100 GPUs (Hopper architecture). The end-to-end throughput measurement uses 64 H100s for SonicMoE versus 96 H100s for ScatterMoE, with FSDP-2 sharding via the lm-engine codebase.

### 5.2 Activation Memory Results

**Claim:** SonicMoE reduces activation memory by 45% for fine-grained 7B MoE and keeps memory constant regardless of expert granularity.

**Evidence.** Figure 13 presents peak activation memory per layer across four model scales (1.4B, 7B, 30B, 120B) with varying expert intermediate dimensions $n \in \{256, 512, 1024\}$. For the 7B model with $n=256$:

| Method | Memory per Layer (GiB) | Relative to SonicMoE |
|--------|----------------------|---------------------|
| SonicMoE | ~1.0 | 1.0x |
| ScatterMoE | ~1.8 | 1.8x |
| MoMoE | ~2.5 | 2.5x |
| MegaBlocks | N/A (doesn't support small $n$) | — |

The paper states explicitly in Section 6.1:

> "For the 7B model with $n = 256$, our approach reduces memory usage by 45% compared to ScatterMoE, and more significantly compared to MoMoE."

At 120B scale, the gap widens: SonicMoE saves more than 3 GiB memory per layer compared to MoMoE. Figure 1 (left) shows the critical trend: as expert granularity $d/n$ increases from 2 to 16, SonicMoE's memory footprint stays flat at approximately 1 GiB, while ScatterMoE increases from ~2 GiB to ~8 GiB and MoMoE from ~3 GiB to ~10 GiB.

**Assessment.** The activation memory results are convincing and directly support the paper's mathematical reformulation claim. The constant memory profile across granularity levels demonstrates that SonicMoE successfully decouples activation caching from the number of activated experts $K$. The 45% reduction is substantial enough to enable larger batch sizes or longer sequences on the same hardware—a practical improvement rather than a marginal optimization.

**Limitation.** The paper notes in Footnote 9 that SonicMoE still materializes a temporary $Y$ variable that can be recycled after each layer. Removing this entirely would require atomic add operations to global memory, which introduces non-determinism and numerical accuracy issues for BF16. This is an acceptable trade-off, but it means the memory footprint is not theoretically minimal.

### 5.3 Training Throughput Results

**Claim:** SonicMoE achieves 1.86x compute throughput improvement on H100 for 7B MoE compared to ScatterMoE's BF16 MoE kernel.

**Evidence.** Figure 6 provides detailed kernel runtime breakdown for a 7B MoE configuration $(T, d, n, E, K) = (24576, 1536, 256, 128, 8)$:

|-----------|--------------|-----------------|------------|-----------------|
| Forward pass total | 1.237 | 2.255 | 2.347 | 1.770 |
| Backward pass total | 2.173 | 3.968 | 4.678 | 3.303 |
| **End-to-end** | **3.410** | **6.223** | **7.025** | **5.073** |

The backward pass shows the largest gains: SonicMoE is 83% faster than ScatterMoE and 115% faster than MoMoE. The dH kernel (down-projection activation gradient) is the primary contributor: SonicMoE completes it in 0.47ms versus 0.99ms for ScatterMoE's three separate kernels (down-proj act 0.43ms + dS 0.24ms + dSwiGLU 0.33ms).

Figure 14 extends this analysis across model scales. For the 7B model with $n=256$:

| Method | Forward TFLOPS | Backward TFLOPS |
|--------|---------------|-----------------|
| SonicMoE | ~550 | ~500 |
| ScatterMoE | ~400 | ~350 |
| MoMoE | ~380 | ~300 |
| DeepGEMM++ | ~480 | ~420 |

The paper reports in Section 6.2.1:

> "In 1.4B and 7B settings, our approach improves TFLOPS by 40% compared to ScatterMoE and MoMoE."

For larger models (30B, 120B), SonicMoE achieves over 500 TFLOPS in both forward and backward passes, while baselines either fail to support certain $n$ sizes (MegaBlocks) or suffer significant degradation (MoMoE).

**End-to-end training throughput.** The abstract provides the most practically relevant metric:

> "SonicMoE on 64 H100s achieves a training throughput of 213 billion tokens per day comparable to ScatterMoE's 225 billion tokens per day on 96 H100s for a 7B MoE model training with FSDP-2 using the lm-engine codebase."

This represents a 33% reduction in GPU requirements for equivalent throughput, or approximately 1.5x throughput per GPU.

**Assessment.** The throughput results are robust and well-supported by kernel-level profiling. The 1.86x speedup claim is conservative—the actual end-to-end improvement is closer to 1.82x (6.223ms / 3.410ms) for the 7B configuration in Figure 6. The fact that 64 H100s with SonicMoE match 96 H100s with ScatterMoE demonstrates real-world impact beyond microbenchmarks.

**Condition.** The speedup is most pronounced for fine-grained MoEs (small $n$). As Figure 14 shows, the gap between SonicMoE and baselines widens as $n$ decreases from 1024 to 256. This is expected: fine-grained MoEs are more memory-bound, so IO-aware optimizations yield greater relative benefits.

### 5.4 Token Rounding Results

**Claim:** Token rounding eliminates padding waste from Grouped GEMM tile quantization, yielding up to 16% higher TFLOPS in highly sparse regimes without quality loss.

**Evidence.** Figure 11 quantifies the padding problem: for a configuration with $T=16k$, $d=4k$, $n=1k$, $K=4$, wasted FLOPs from padding reach 15-20% as the number of experts increases from 64 to 256. Token rounding eliminates this waste by ensuring each expert receives a tile-multiple number of tokens.

Figure 16 benchmarks token rounding (TR) against vanilla TC top-K routing under iso-FLOPs settings (keeping $T$, $n$, and $K$ constant while increasing $E$):

| Configuration | Experts | K/E | Forward TFLOPS Gain | Backward TFLOPS Gain | End-to-End Gain |
|--------------|---------|-----|---------------------|---------------------|-----------------|
| 7B-style | 128 | 1/64 | 16.5% | 6.1% | 9.4% |
| 7B-style | 256 | 1/128 | 25.7% | 11.8% | 15.9% |

The paper states in Section 6.3.3:

> "For the 3rd and 4th column on the bottom row in Figure 16, when we have a MoE with 256 experts ($K/E = 1/128$), token rounding routing achieves a 25.7% TFLOPS improvement in forward and 11.8% in backward resulting in an end-to-end improvement of 15.9%."

Figure 17 extends this to configurations from recent open-source MoEs. For Qwen3-Next-80B-A3B-Thinking ($K/E = 10/512$), TR achieves 19.6% and 7.9% speedup over TC top-K during forward and backward pass respectively.

**Quality preservation.** Table 3 validates that token rounding does not degrade model quality. For the 1.4B model with 8/128 activation (Table 3d):

| Method | Train PPL | Val PPL | Avg Accuracy (11 tasks) |
|--------|-----------|---------|------------------------|

**Assessment.** The token rounding results are compelling. The throughput gains scale with sparsity as expected (higher $E$ with constant $K$ means more padding waste to eliminate), and the quality metrics show TR is not just neutral but occasionally slightly better than TC top-K. This suggests the bounded deviation (at most one tile per expert) does not harm the routing signal.

When $\bar{T}_e / M_{\text{tile}} = 1$ (e.g., $\bar{T}_e = 128$), there is noticeable degradation in validation perplexity and downstream accuracy, though TR still outperforms EC baselines. - **Epilogue fusion (Figure 6):** 0.47ms vs 0.99ms for dH kernel (53% reduction)
- **Ping-Pong scheduling (Figure 19):** 10% higher TFLOPS on down-projection vs DeepGEMM
- **Top-K kernel (Figure 23):** SonicMoE achieves 1.5-2.0 TB/s vs PyTorch's 0.5-1.0 TB/s for large $T$

**Assessment.** The ablation studies are thorough and support the paper's design choices. The rounding subroutine ablation shows NR-f is a reasonable default, though the differences are small enough that practitioners could experiment with alternatives. The granularity ablation validates the core motivation for SonicMoE: fine-grained MoEs are worth accelerating because they improve quality.

### 5.6 Failure Cases and Limitations

**Inference incompatibility.** Token rounding is not directly compatible with autoregressive inference. Footnote 28 states:

> "Token rounding is not a token-choice routing method which creates difficulty for autoregressive generation. Here we do not apply any adaptation and switch to vanilla token choice top-K routing during evaluation/validation."

This means models trained with TR must switch to TC top-K at inference time. Table 3 validates this works with minimal quality loss, but it introduces a train-test mismatch that could theoretically cause issues in production deployments.

**Microbatch size sensitivity.** As noted in Section 5.4, token rounding degrades when $\bar{T}_e / M_{\text{tile}} &lt; 2$. For very small microbatches or extremely high sparsity ($K/E &lt; 1/100$), practitioners may need to increase batch size or accept quality degradation. Table 7 shows Val PPL increases from 13.10 to 13.55 when $\bar{T}_e$ drops from 512 to 128 for the 1.8B model.

**Precision limitations.** The evaluation focuses on BF16 precision. Section 7 lists extending to FP8 and microscaling formats (MXFP8, MXFP4) as future work. As the community moves toward lower precision training, SonicMoE's kernels may need adaptation to maintain their advantages.

**Blackwell support incomplete.** The paper mentions Blackwell support but provides limited experimental results on Blackwell GPUs. Section 4.2 describes the TMEM-based Ping-Pong scheduling for Blackwell, but all throughput numbers in Figures 6, 14, 15, 16, 17 are from H100 (Hopper). Practitioners with Blackwell hardware should verify the claimed benefits empirically.

**Expert parallelism not evaluated.** The end-to-end throughput measurement uses FSDP-2 within nodes but does not evaluate expert parallelism (all2all communication across nodes). Section 7 lists "overlapping communication with computation in distributed settings like expert parallelism" as future work. For multi-node MoE training, the IO-aware optimizations may interact differently with communication overhead.

### 5.7 Overall Assessment

The experiments convincingly support SonicMoE's three core claims:

1. **Memory efficiency:** The 45% activation memory reduction is well-documented in Figure 13, and the constant memory profile across granularity levels in Figure 1 directly validates the mathematical reformulation.

2. **Throughput improvement:** The 1.86x speedup claim is conservative relative to the kernel-level measurements in Figure 6 (1.82x end-to-end). The end-to-end training throughput (213B tokens/day on 64 H100s vs 225B on 96 H100s) demonstrates real-world impact.

3. **Token rounding quality:** Table 3 shows TR matches or slightly exceeds TC top-K quality across multiple model scales and sparsity levels, while Figure 16 demonstrates up to 16% throughput gains in sparse regimes.

The ablation studies are comprehensive, covering rounding subroutines, microbatch sizes, tile sizes, and kernel-level optimizations. The identified limitations (inference incompatibility, microbatch sensitivity, precision constraints) are acknowledged transparently rather than hidden.

**Weaknesses.** The Blackwell experimental results are limited despite architectural descriptions. The expert parallelism evaluation is absent, which is a significant gap for large-scale MoE training. The token rounding inference mismatch, while validated, introduces complexity that practitioners must manage.

**Strengths.** The kernel-level profiling in Figure 6 is exceptionally detailed, breaking down runtime by component and comparing against multiple baselines. The quality evaluation spans 11 downstream tasks across 5 model configurations (Table 3a-e), providing robust evidence that TR does not harm model capability. The open-source release with PyTorch interface lowers adoption barriers.

Overall, the experimental analysis is rigorous and supports the paper's contributions. The results demonstrate that SonicMoE is not just an incremental optimization but a meaningful advance in MoE training efficiency that enables finer-grained and sparser architectures without hardware penalties.

## 6. Limitations and Trade-offs

SonicMoE represents a significant advance in MoE training efficiency, but it is not a universal solution. This section examines the assumptions, constraints, and open questions that practitioners must consider when adopting SonicMoE. We ground each limitation in specific evidence from the paper to enable informed deployment decisions.

### 6.1 Train-Test Routing Mismatch

**The Problem:** Token rounding (TR) is fundamentally incompatible with autoregressive inference. The paper states explicitly in **Footnote 28**:

> "Token rounding is not a token-choice routing method which creates difficulty for autoregressive generation. Here we do not apply any adaptation and switch to vanilla token choice top-K routing during evaluation/validation."

This creates a **train-test distribution mismatch**: models are trained with tile-aligned token assignments but must inference with standard top-K routing. While **Table 3** validates that this switch works with minimal quality loss (TR often matches or slightly exceeds TC top-K on downstream tasks), the mismatch introduces theoretical uncertainty.

**Why This Matters:** During training, TR may route tokens to experts based on tile-alignment constraints rather than pure router scores. At inference, tokens follow pure score-based routing. If the model learns to rely on TR's padding behavior (e.g., certain experts consistently receive padded tokens), the inference behavior could diverge from training.

**Evidence of Risk:** **Table 3** shows that Expert Choice (EC) routing suffers significant train-test gaps when switching to TC at inference. For the 1.4B model with 2/128 activation (**Table 3e**), EC achieves validation perplexity of 24.79 during training but the EC (aux router) baseline reaches 37.52 at evaluation—a 51% degradation. While TR does not show this extreme gap, the underlying concern remains: any routing modification during training that is not present at inference introduces potential mismatch.

**Mitigation:** The paper validates TR-to-TC switching across five model configurations in **Table 3a–e**, finding TR consistently matches or exceeds TC top-K quality. However, these evaluations are on models up to 1.8B parameters. Whether this holds at 70B+ scale remains untested.

### 6.2 Microbatch Size Sensitivity

**The Problem:** Token rounding requires sufficient tokens per expert to maintain quality. **Tables 7 and 8** identify a critical threshold:

> "TR preserves its trained MoE quality when $\bar{T}_e / M_{\text{tile}} \geq 2$, but if $\bar{T}_e / M_{\text{tile}} = 1$... there is a noticeable quality degradation."

Where $\bar{T}_e$ is the average tokens per expert and $M_{\text{tile}} = 128$ is the GEMM tile size.

**Concrete Constraint:** With $M_{\text{tile}} = 128$, practitioners need $\bar{T}_e \geq 256$ tokens per expert on average. For a highly sparse MoE with $K/E = 1/128$ and microbatch size $T = 16384$ tokens:
- $\bar{T}_e = T \times (K/E) = 16384 \times (1/128) = 128$ tokens per expert
- This violates the $\bar{T}_e \geq 256$ threshold

**Evidence:** **Table 7b** shows the degradation empirically for a 1.8B model:

| $\bar{T}_e$ | $\bar{T}_e / M_{\text{tile}}$ | Val PPL | Avg Accuracy |
|-------------|------------------------------|---------|--------------|
| 512 | 4 | 13.10 | 53.5% |
| 256 | 2 | 13.12 | 53.2% |
| 128 | 1 | 13.55 | 52.7% |
| TC top-K | — | 13.12 | 52.8% |

At $\bar{T}_e = 128$, validation perplexity increases by 0.43 points (3.3% degradation) compared to $\bar{T}_e = 512$. The model still outperforms EC baselines, but the gap to TC top-K narrows significantly.

**Practical Impact:** This constraint limits token rounding's applicability to:
- Very small microbatches (memory-constrained training)
- Extremely sparse configurations ($K/E &lt; 1/100$)
- Early training stages with small effective batch sizes

Practitioners must either increase microbatch size (requiring more GPU memory) or accept potential quality degradation in these regimes.

### 6.3 Precision and Format Limitations

**The Problem:** All experimental evaluation uses **BF16 precision**. **Section 7 (Conclusion)** explicitly lists extending to lower precision as future work:

> "Future directions include extending to low-precision and microscaling formats (FP8, MXFP8, MXFP4) for further memory savings."

**Why This Matters:** The MoE training landscape is rapidly shifting toward FP8 and microscaling formats for memory and throughput benefits. DeepSeek-V3 and other frontier models already use FP8 for MoE layers. SonicMoE's kernel optimizations are designed around BF16 data paths, and several components may not transfer directly:

1. **Gather fusion:** FP8 has different alignment requirements than BF16
2. **Token rounding:** The rounding behavior may interact differently with FP8's dynamic range
3. **Epilogue fusion:** Activation functions may need reformulation for FP8

**Evidence of Gap:** The paper provides no FP8 benchmarks or analysis. **Figure 6**, **Figure 14**, and **Figure 16** all report BF16 TFLOPS. While the algorithmic contributions (memory-efficient backward, token rounding) are precision-agnostic in principle, the kernel implementations are not.

**Risk:** Practitioners training with FP8 cannot directly adopt SonicMoE's kernels without modification. The IO-aware optimizations may still apply, but the specific tile sizes, pipeline stages, and fusion strategies may need redesign for FP8's different memory access patterns.

### 6.4 Incomplete Blackwell Validation

**The Problem:** While **Section 4.2** describes Blackwell-specific adaptations (TMEM-based Ping-Pong scheduling, 2-CTA cluster relay mechanism), **all experimental results are from H100 (Hopper) GPUs**.

**Evidence:** Every throughput figure (**Figure 6**, **Figure 14**, **Figure 15**, **Figure 16**, **Figure 17**, **Figure 19–23**) specifies H100 in the caption or text. The abstract states "1.86x compute throughput improvement on Hopper GPUs" with no Blackwell equivalent claim.

**Why This Matters:** Blackwell GPUs (B100, B200) introduce architectural changes that could affect SonicMoE's optimizations:
- **Tensor Memory (TMEM):** Changes how accumulators are stored, potentially affecting epilogue fusion
- **UMMA instruction:** Single-threaded asynchronous operation vs. Hopper's warpgroup-level WGMMA
- **2-CTA clusters:** Require the relay warp mechanism described in **Figure 5**, adding scheduling complexity

**Open Question:** Does SonicMoE achieve similar speedups on Blackwell? The paper describes the adaptation but provides no empirical validation. Practitioners with Blackwell hardware cannot verify the claimed benefits without running their own benchmarks.

**Risk:** The Blackwell adaptations add implementation complexity (relay warps, cluster-level synchronization) without proven benefit. If the speedups do not translate, the engineering effort may not be justified.

### 6.5 Expert Parallelism Not Evaluated

**The Problem:** The end-to-end throughput measurement uses **FSDP-2 within nodes** but does not evaluate **expert parallelism (EP)** across nodes. **Section 7** lists this as future work:

> "Future directions include... overlapping communication with computation in distributed settings like expert parallelism."

**Why This Matters:** Large-scale MoE training (70B+ parameters) typically requires expert parallelism to distribute experts across multiple GPUs. EP introduces all2all communication overhead that interacts with SonicMoE's IO-aware optimizations:

1. **Gather fusion:** EP requires all2all before expert computation, potentially conflicting with SonicMoE's gather-with-load strategy
2. **Token rounding:** EP may require token redistribution across nodes, complicating the tile-alignment guarantees
3. **Ping-Pong scheduling:** Communication latency may dominate compute latency, reducing the benefit of IO-compute overlap

**Evidence of Gap:** The 64 H100 vs. 96 H100 comparison in the abstract uses FSDP-2 sharding "within a single node (8x H100s) and replicate this sharded unit across nodes." This is **not** expert parallelism—it's data parallelism with model sharding. True EP would route tokens to experts on different nodes, requiring all2all communication.

**Practical Impact:** Organizations training 100B+ MoEs with EP cannot directly apply SonicMoE's end-to-end throughput claims. The kernel-level optimizations (gather fusion, epilogue fusion) should still apply within each node, but the distributed training overhead is unmeasured.

### 6.6 Temporary Y Materialization

**The Problem:** Despite the memory-efficient backward algorithm, SonicMoE still materializes a temporary $Y$ variable. **Footnote 9** explains:

> "Although SonicMoE still materializes a temporary Y variable, we can recycle Y after each layer. As long as the number of MoE layers (typically 32+ for 7B+ MoE) is larger than K, the transient memory usage of Y will be overshadowed."

**Why This Is a Trade-off:** Removing $Y$ entirely would require atomic add operations to global memory, which introduces three issues:
1. **Non-determinism:** Atomic operations may produce different results across runs
2. **Numerical accuracy:** BF16 atomic add has known precision issues
3. **Expert parallelism incompatibility:** Atomic adds conflict with all2all communication patterns

**Evidence:** The paper chooses determinism and numerical stability over theoretically minimal memory. This is a reasonable trade-off, but practitioners should understand that SonicMoE's activation memory is not the absolute minimum possible—it's the minimum *without* sacrificing determinism.

**Quantitative Impact:** For a 7B MoE with $T = 24576$, $K = 8$, $d = 1536$:
- $Y$ size = $TKd = 24576 \times 8 \times 1536 \times 2$ bytes (BF16) ≈ 600 MB per layer
- This is recycled after each layer, so peak memory is not affected
- But it does contribute to transient memory pressure during layer execution

### 6.7 Top-K Kernel Constraints

**The Problem:** SonicMoE's custom top-K sorting kernel has hard limits on expert count and activation count. **Section 4.3** states:

> "Our top-K kernel supports E and K when $E \leq 4096$ and $K \leq 16$."

**Why This Matters:** While these limits cover most current MoE configurations (**Table 1** shows maximum $E = 512$ for Qwen3-Next-80B), future models may exceed these bounds:
- Kimi K2 has 384 experts (within limit)
- But models with 1000+ experts would require fallback to PyTorch's top-K

**Evidence:** **Footnote 24** explains the constraint:

> "We restrict this E and K so that we only need thread and warp-level primitives. Larger E > 4k requires a SMEM buffer and block-level synchronization and is often too large for most MoE models."

**Practical Impact:** For models within the limits, SonicMoE's top-K achieves 1.5–2.0 TB/s vs. PyTorch's 0.5–1.0 TB/s (**Figure 23**). For models exceeding the limits, practitioners must use PyTorch's top-K, losing the 40% router computation time reduction mentioned in **Section 4.3**.

### 6.8 Tile Size Hardware Dependency

**The Problem:** Token rounding depends on the GEMM tile size $M_{\text{tile}}$, which is **GPU-dependent and potentially (d, n)-dependent**. **Footnote 26** acknowledges:

> "For simplicity, we always use $M_{\text{tile}}$ as 128 in Table 3 and Figure 16, but we acknowledge that $M_{\text{tile}}$ is GPU-dependent and in some cases (d, n)-dependent."

**Why This Matters:** Different GPU architectures have different optimal tile sizes:
- Hopper (H100): Typically 128 for BF16
- Blackwell: May differ due to TMEM structure
- Future architectures: Unknown

**Practical Impact:** Practitioners must tune $M_{\text{tile}}$ for their specific hardware and model configuration. Using $M_{\text{tile}} = 128$ on a GPU where the optimal tile is 64 or 256 could reduce the token rounding benefits or even cause quality degradation if $\bar{T}_e / M_{\text{tile}}$ falls below the threshold.

**Open Question:** The paper provides no guidance on how to select $M_{\text{tile}}$ for a given hardware/model combination. This requires empirical tuning, adding complexity to adoption.

### 6.9 Router Load Balancing Considerations

**The Problem:** The training setup uses **auxiliary load balancing loss with coefficient 0.01** but **does not use router Z-loss**. **Appendix H** states:

> "We always use MoE with SwiGLU for the MoE layers and we use an auxiliary load balancing loss (Shazeer et al. 2017) with coefficient 0.01 but we do not use the router Z loss (Zoph et al. 2022)."

**Why This Matters:** Router Z-loss is commonly used in large-scale MoE training to stabilize router logits and prevent collapse. Not using it may affect:
- Training stability at larger scales (70B+)
- Load balancing quality under extreme sparsity
- Interaction with token rounding (which modifies token assignments)

**Evidence:** The evaluation is on models up to 1.8B parameters. Whether the load balancing strategy scales to 100B+ models without Z-loss is untested.

**Risk:** Practitioners training larger models may need to add Z-loss, which could interact with token rounding in unforeseen ways. The paper provides no analysis of this interaction.

### 6.10 Routing-Agnostic Design Trade-off

**The Problem:** SonicMoE's kernels are **routing-agnostic** by design (**Section 3.1**), enabling modular adoption but potentially leaving optimization opportunities on the table.

**Why This Is a Trade-off:**
- **Benefit:** Practitioners can adopt kernel optimizations without changing routing logic
- **Cost:** Kernels cannot exploit routing-specific patterns (e.g., if certain experts consistently receive more tokens, the kernel could optimize for that distribution)

**Evidence:** The paper does not explore routing-aware kernel optimizations. All kernels treat experts uniformly regardless of token distribution patterns.

**Open Question:** Could tighter integration between routing and kernel execution yield additional speedups? For example, if token rounding produces predictable token counts per expert, could the GEMM kernel pre-allocate resources accordingly? The paper does not investigate this.

### 6.11 Summary of Limitations

| Limitation | Severity | Mitigation |
|------------|----------|------------|
| Train-test routing mismatch | Medium | Validated in Table 3; monitor at larger scales |
| Microbatch size sensitivity ($\bar{T}_e / M_{\text{tile}} \geq 2$) | High | Increase microbatch size or accept quality degradation |
| BF16-only evaluation | Medium | Await FP8 kernel updates |
| No Blackwell empirical results | Medium | Run own benchmarks on Blackwell hardware |
| No expert parallelism evaluation | High | Kernel optimizations apply; distributed overhead unmeasured |
| Temporary Y materialization | Low | Acceptable trade-off for determinism |
| Top-K kernel limits (E ≤ 4096, K ≤ 16) | Low | Covers most current models; fallback to PyTorch if exceeded |
| Tile size hardware dependency | Medium | Tune $M_{\text{tile}}$ for specific hardware |
| No router Z-loss | Low | Add Z-loss if needed for larger scales |
| Routing-agnostic design | Low | Enables modularity; may leave optimization on table |

**Overall Assessment:** SonicMoE's limitations are primarily **deployment constraints** rather than fundamental flaws. The core contributions (memory-efficient backward, IO-aware kernels, token rounding) are sound, but practitioners must verify applicability to their specific hardware, model scale, and training configuration. The most significant gaps are the lack of expert parallelism evaluation (critical for 100B+ models) and BF16-only validation (as the field moves toward FP8). These are acknowledged as future work rather than hidden weaknesses, enabling informed adoption decisions.

## 7. Implications and Future Directions

SonicMoE represents more than a collection of kernel optimizations; it signals a paradigm shift in how we approach large-scale model training. By demonstrating that algorithmic efficiency (FLOPs) and hardware efficiency (throughput) must be co-designed, this work redefines the optimization frontier for sparse architectures. This section explores how SonicMoE changes the research landscape, outlines concrete follow-up directions, identifies high-value practical applications, and provides guidance for integrating these techniques into existing workflows.

### 7.1 Shifting the Optimization Frontier: From FLOPs to Compute Hours

Historically, MoE research has optimized for **quality per FLOP**. The assumption was that reducing computational complexity (e.g., activating fewer parameters per token) would automatically yield faster training. SonicMoE dismantles this assumption by showing that hardware bottlenecks (memory bandwidth, tile alignment) can negate algorithmic gains.

The paper's conclusion articulates a new north star for the field:

> "We envision future model architecture designs that optimize for quality per compute hour rather than just quality per FLOP—jointly considering algorithmic and hardware efficiency."

This shift has three profound implications:

1.  **Architecture Search Constraints Change:** When selecting MoE configurations (number of experts $E$, activated experts $K$, intermediate size $n$), researchers can no longer look at FLOP counts alone. They must consider **arithmetic intensity** (FLOPs per byte of IO). As shown in **Figure 3**, increasing expert granularity $d/n$ linearly increases IO cost. SonicMoE enables higher granularity than previously possible, but future architecture search should treat hardware constraints as first-class citizens in the loss function.
2.  **Hardware-Aware Routing Becomes Standard:** Token rounding (**Section 5**) demonstrates that routing algorithms can be modified to satisfy hardware constraints (tile alignment) without sacrificing model quality. This suggests future routers should not just maximize prediction accuracy but also minimize padding waste and memory fragmentation. The bounded deviation property (at most one tile per expert) provides a theoretical guarantee that makes this safe.
3.  **Memory Becomes the Primary Constraint:** With IO-aware kernels hiding latency, the remaining bottleneck is activation memory capacity. SonicMoE's memory-efficient backward pass (**Section 3.4.1**) decouples activation memory from expert granularity. This unlocks architectural choices that were previously impossible due to Out-Of-Memory (OOM) errors, allowing models to scale granularity without hitting memory walls.

### 7.2 Enabled Research Directions

SonicMoE opens several concrete avenues for follow-up research, explicitly identified in the paper's conclusion and implied by its limitations.

#### 7.2.1 Low-Precision and Microscaling Formats
The paper evaluates SonicMoE exclusively in **BF16 precision**. However, **Section 7** explicitly lists extending to low-precision formats as a primary future direction:

> "Future directions include extending to low-precision and microscaling formats (FP8, MXFP8, MXFP4) for further memory savings."

**Why this matters:** The core bottleneck identified in **Section 2.2** is memory bandwidth. Moving from BF16 (16 bits) to FP8 (8 bits) or MXFP4 (4 bits) theoretically doubles or quadruples effective bandwidth. However, this is non-trivial:
*   **Gather Fusion:** FP8 has different alignment requirements than BF16. The `cp.async` instructions used in **Section 4.1.1** may need adjustment for FP8 tensor cores.
*   **Token Rounding:** Rounding behavior may interact differently with FP8's dynamic range. The "nearest rounding" subroutine (**Table 6**) assumes BF16 precision stability.
*   **Epilogue Fusion:** Activation functions like SwiGLU may require reformulation to maintain numerical stability in lower precision.

**Research Question:** Can SonicMoE's IO-overlap techniques sustain the same relative speedup (1.86x) when the compute-to-memory ratio changes drastically with FP8?

#### 7.2.2 Distributed Expert Parallelism (EP)
The end-to-end throughput results (**Abstract**) use FSDP-2 within nodes but do not evaluate **Expert Parallelism** across nodes. **Section 7** states:

> "Future directions include... overlapping communication with computation in distributed settings like expert parallelism."

**Why this matters:** For models larger than 70B parameters, single-node training is insufficient. EP requires `all2all` communication to route tokens to experts on different GPUs. This introduces new IO costs that interact with SonicMoE's optimizations:
*   **Communication-Compute Overlap:** SonicMoE's Ping-Pong scheduling (**Figure 7**) overlaps memory IO with compute. Can this be extended to overlap network communication (NCCL) with compute?
*   **Token Rounding in EP:** Token rounding ensures tile alignment per expert. In EP, tokens are redistributed across nodes. Does the rounding constraint survive the `all2all` operation, or does it require global coordination?
*   **Gather Fusion Conflict:** EP typically requires gathering tokens before communication. SonicMoE fuses gather with GEMM load. These strategies may conflict, requiring a redesign of the communication pipeline.

**Research Question:** How does SonicMoE's gather fusion strategy integrate with expert parallelism's `all2all` communication primitives? Can the "relay warp" mechanism (**Figure 5**) be adapted for network synchronization?

#### 7.2.3 Dynamic Tile Sizes and Adaptive Routing
SonicMoE uses a fixed tile size $M_{\text{tile}} = 128$ for token rounding (**Footnote 26**). However, optimal tile sizes depend on GPU architecture and model dimensions $(d, n)$.

**Why this matters:** A fixed tile size may not be optimal for all configurations. For example, Blackwell's Tensor Memory (TMEM) structure (**Section 4.2**) might favor different tile dimensions than Hopper's SMEM.
*   **Adaptive Rounding:** Could the router dynamically select $M_{\text{tile}}$ based on current load balancing?
*   **Hardware Profiling:** Automated tools could profile optimal tile sizes for specific $(d, n, E)$ combinations, feeding this back into the routing algorithm.

**Research Question:** Can we develop a runtime profiler that selects $M_{\text{tile}}$ dynamically to maximize throughput while maintaining the $\bar{T}_e / M_{\text{tile}} \geq 2$ quality threshold (**Table 7**)?

### 7.3 Practical Applications and Downstream Use Cases

SonicMoE is not universally superior in all scenarios. Its benefits are concentrated in specific regimes. Practitioners should evaluate their use case against the following profiles.

#### 7.3.1 High-Value Scenarios (Adopt SonicMoE)
*   **Fine-Grained MoE Training:** If your model has expert granularity $d/n > 2$ (e.g., $d=4096, n=1024$), SonicMoE's memory efficiency is critical. **Figure 1** shows baseline memory usage scaling linearly here, while SonicMoE stays constant.
*   **High Sparsity Regimes:** If your activation ratio $K/E &lt; 1/32$ (e.g., 8 experts activated out of 256+), token rounding yields significant throughput gains. **Figure 16** shows up to 16% end-to-end improvement in these settings due to eliminated padding.
*   **Memory-Constrained Clusters:** If you are limited by GPU count rather than time, SonicMoE allows matching larger clusters with fewer GPUs. The **Abstract** demonstrates 64 H100s matching 96 H100s throughput—a 33% infrastructure cost reduction.
*   **Hopper/Blackwell Hardware:** The kernels are specifically optimized for NVIDIA Hopper (H100) and Blackwell architectures. If you possess this hardware, SonicMoE extracts maximum performance via TMA and Ping-Pong scheduling.

#### 7.3.2 Low-Value Scenarios (Stick to Baselines)
*   **Coarse-Grained MoEs:** If $d/n \approx 1$ (e.g., Mixtral 8x7B style), the IO bottleneck is less severe. Baselines like ScatterMoE may suffice without the integration complexity.
*   **Small Microbatches:** If your memory constraints force microbatch size $T$ such that $\bar{T}_e &lt; 256$ (with $M_{\text{tile}}=128$), token rounding may degrade quality (**Table 7**).
*   **FP8 Training Pipelines:** Until SonicMoE supports FP8 natively, mixing BF16 kernels with FP8 training loops adds complexity and may negate precision benefits.
*   **Pure Inference Deployment:** Token rounding is not compatible with autoregressive inference without switching routers (**Footnote 28**). If you need identical train-test behavior, standard top-K routing is simpler.

### 7.4 Reproduction and Integration Guidance

The authors have open-sourced SonicMoE with a PyTorch interface (**Section 1**). Below is practical guidance for integrating these kernels into your training stack.

#### 7.4.1 Integration Steps
1.  **Kernel Replacement:** SonicMoE's computation kernels are **routing-agnostic** (**Section 3.1**). You can replace your existing MoE kernel (e.g., from Megatron or DeepSpeed) without changing your router logic initially.
    *   *Action:* Swap the MoE layer implementation to use SonicMoE's `forward` and `backward` kernels.
    *   *Benefit:* Immediate memory reduction (45% for 7B MoE) and throughput gain (1.86x) without retraining.
2.  **Enable Token Rounding:** Once kernels are stable, enable the token rounding router (**Algorithm 4**).
    *   *Action:* Replace `torch.topk` with SonicMoE's custom top-K kernel and apply the rounding subroutine.
    *   *Constraint:* Ensure $\bar{T}_e \geq 256$ tokens per expert on average. If not, increase microbatch size.
3.  **Inference Switch:** Configure your inference pipeline to switch back to vanilla top-K routing.
    *   *Action:* Load the trained model weights but use standard top-K selection during generation.
    *   *Validation:* Verify quality on a held-out set (**Table 3** shows minimal degradation, but monitor your specific task).

#### 7.4.2 Configuration Recommendations
Based on the paper's ablation studies (**Section 5.5**, **Appendix F**):

| Parameter | Recommended Value | Rationale |
| :--- | :--- | :--- |
| **Tile Size ($M_{\text{tile}}$)** | `128` (Hopper) | Default used in all experiments (**Footnote 26**). Tune for Blackwell. |
| **Rounding Subroutine** | `NR-f` (Nearest via Frequency) | Best balance of quality and efficiency (**Table 6**). |
| **Microbatch Size ($T$)** | Ensure $\bar{T}_e \geq 256$ | Critical for token rounding quality (**Table 7**). |
| **Precision** | `BF16` | Only supported precision currently. |
| **Expert Count ($E$)** | $\leq 4096$ | Limit for efficient top-K kernel (**Section 4.3**). |
| **Activated Experts ($K$)** | $\leq 16$ | Limit for efficient top-K kernel (**Section 4.3**). |

#### 7.4.3 Debugging and Monitoring
*   **Memory Profiling:** Use `torch.cuda.memory_summary()` to verify activation memory stays constant as you increase granularity. If it scales linearly, the memory-efficient backward path is not active.
*   **Throughput Monitoring:** Track TFLOPS per layer. If forward pass TFLOPS drop significantly when enabling token rounding, check that padding waste was actually present (i.e., ensure sparsity is high enough).
*   **Quality Checks:** Monitor validation perplexity closely when $\bar{T}_e / M_{\text{tile}} \approx 2$. This is the stability boundary (**Table 7**).

### 7.5 Summary of Impact

SonicMoE demonstrates that the next generation of MoE efficiency will not come from algorithmic sparsity alone, but from **hardware-aware co-design**. By treating memory bandwidth, tile alignment, and activation caching as optimization variables alongside model loss, SonicMoE achieves what prior work could not: enabling finer-grained, sparser models without hardware penalties.

For practitioners, the immediate value is clear: **train faster with less memory on existing hardware**. For researchers, the invitation is broader: **rethink architecture design through the lens of hardware constraints**. As the field moves toward trillion-parameter models, the lessons from SonicMoE—overlap IO, minimize caching, and align computation with hardware tiles—will become foundational principles rather than optional optimizations.
