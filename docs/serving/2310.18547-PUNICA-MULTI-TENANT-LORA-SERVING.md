## 1. Executive Summary

Punica addresses the inefficiency of serving multiple Low-Rank Adaptation (LoRA) models by introducing a system that consolidates multi-tenant workloads on a shared GPU cluster using a novel CUDA kernel, Segmented Gather Matrix-Vector Multiplication (`SGMV`). This design allows a GPU to hold only a single copy of the underlying pre-trained model (e.g., Llama2 7B, 13B, 70B) while batching operations for different LoRA adapters, significantly improving memory and computation efficiency. Evaluations on NVIDIA A100 clusters show Punica achieves 12x higher throughput compared to state-of-the-art LLM serving systems while adding only 2ms latency per token.

## 2. Context and Motivation

To understand the necessity of Punica, we must first examine the shifting landscape of Large Language Model (LLM) deployment. The industry is moving away from serving a single, monolithic model toward serving many specialized variants of a base model. This section explains the technical constraints that make this transition difficult and why existing serving systems fail to address them efficiently.

### 2.1 The Shift to Parameter-Efficient Specialization

Historically, adapting a pre-trained LLM to a specific domain (e.g., medical diagnosis, legal contract analysis) required full fine-tuning. This involves updating all parameters of the model, which is computationally expensive and storage-intensive. 

**Low-Rank Adaptation (LoRA)** has emerged as a dominant alternative. As described in **Section 2.2**, LoRA operates on the observation that the weight update required to adapt a model has a low intrinsic rank. Instead of updating the full weight matrix $W \in \mathbb{R}^{h_1 \times h_2}$, LoRA trains two small dense matrices $A \in \mathbb{R}^{h_1 \times r}$ and $B \in \mathbb{R}^{r \times h_2}$, where $r$ is the LoRA rank (typically much smaller than $h_1$, e.g., $r=16$ vs $h_1=4096$). The effective weight during inference becomes:

$$W_{\text{effective}} = W + AB$$

This approach reduces the number of trainable parameters significantly and adds minimal storage overhead. The paper notes that each fine-tuned LoRA model adds only **0.1% to 1%** of the underlying pre-trained model's weight (**Section 2.2**). 

**The Real-World Impact:** This efficiency allows ML providers to host hundreds of specialized models for different tenants. However, this creates a new systems challenge: while *training* is cheap, *serving* these models efficiently at scale remains unsolved. If a provider has 100 tenants each requiring a unique LoRA adapter on top of a Llama2-7B backbone, how should these be deployed on a GPU cluster?

### 2.2 The Inefficiency of Naive Serving Strategies

The most straightforward approach to serving multiple LoRA models is to treat each as an independent entity. This is the "naive" approach criticized in **Section 1**. 

*   **Resource Explosion:** If serving one LoRA model requires $k$ GPUs, serving $n$ different LoRA models would seemingly require $k \times n$ GPUs. 
*   **Redundant Memory Usage:** This approach overlooks the weight correlations among models. Since every LoRA model shares the same pre-trained backbone $W$, storing $n$ copies of $W$ across the cluster wastes massive amounts of GPU memory. 

A slightly better approach is to share the backbone model $W$ across tenants but load different LoRA adapters ($A, B$ matrices) on demand. However, this introduces a computational bottleneck. Standard GPU operations are optimized for **batching**, where multiple requests undergo the exact same matrix multiplications simultaneously. If Request 1 uses LoRA Adapter $\alpha$ and Request 2 uses LoRA Adapter $\beta$, their computation graphs diverge at the layers where $AB$ is added. 

**The Core Gap:** Existing high-performance serving systems cannot efficiently batch requests that require *different* weight matrices within the same kernel launch. They are forced to serialize these requests or separate them onto different GPUs, leading to low utilization.

### 2.3 The Critical Role of the Decode Stage

To understand why batching heterogeneous requests is so critical, we must look at the mechanics of LLM text generation. As detailed in **Section 2.1**, generation consists of two stages:

1.  **Prefill Stage:** The model processes the user's input prompt all at once. This is compute-bound and highly parallelizable.
2.  **Decode Stage:** The model generates tokens one by one autoregressively. Each step takes the previous token and the **Key-Value cache (KvCache)** to generate the next token.

The decode stage is the predominant factor in serving cost for long responses. **Figure 1** illustrates the latency behavior of these stages. 
*   **Prefill:** Latency is proportional to batch size because the GPU is fully utilized.
*   **Decode:** Latency increases very slowly with batch size. For short sequences, increasing the batch size from 1 to 32 increases latency only from **11ms to 13ms**. For longer sequences, it increases from **17ms to 34ms**.

> "The decode stage has low GPU utilization because the input is a single vector." (**Section 2.1**)

This disparity creates a massive opportunity for **batching**. By grouping many requests together during the decode stage, we can amortize the memory access cost across many tokens, significantly improving throughput without drastically increasing latency. Systems like **Orca** (**Yu et al., 2022**) leveraged this for standard models. However, Orca assumes all requests in a batch use the *same* model weights. Punica identifies that to serve multi-tenant LoRA efficiently, we must enable this batching benefit even when requests use *different* LoRA adapters (**Section 1, Guideline G2**).

### 2.4 Limitations of Prior Serving Systems

Punica positions itself against two categories of prior work, both of which fall short in the multi-tenant LoRA scenario.

**1. General LLM Serving Systems (e.g., vLLM, Orca, FasterTransformer):**
These systems optimize inference for a single model instance. 
*   **vLLM** (**Kwon et al., 2023**) introduces PagedAttention to manage KvCache memory efficiently but does not natively support batching different LoRA adapters concurrently. 
*   **FasterTransformer** (**Hsueh, 2021**) and **DeepSpeed** (**Aminabadi et al., 2022**) suffer from inefficient KvCache layouts that prevent flexible batching of requests with different sequence lengths or models (**Section 5.4**).
*   **Shortcoming:** As shown in **Section 7.2**, when these systems serve multiple LoRA models, they cannot batch heterogeneous requests. In the "Distinct" workload (where every request uses a different LoRA model), baseline systems effectively run with a **batch size of 1**, destroying throughput.

**2. Parameter-Efficient Serving (e.g., PetS):**
Some work has attempted to address adapter serving.
*   **PetS** (**Zhou et al., 2022**) allows GPU memory sharing of the pre-trained model for different downstream tasks.
*   **Shortcoming:** According to **Section 8**, PetS "does not enable multiple different models to run concurrently." It lacks the kernel-level support to interleave operations for different adapters within the same batch execution window.

### 2.5 Punica's Design Positioning

Punica positions itself as the infrastructure layer specifically for **multi-tenant LoRA serving**. It adheres to three design guidelines established in **Section 1**:

*   **(G1) Consolidation:** Workloads must be consolidated to a small number of GPUs to increase utilization.
*   **(G2) Heterogeneous Batching:** The system must enable batching for *different* LoRA models, not just identical ones.
*   **(G3) Decode Focus:** Optimization efforts should prioritize the decode stage, as it dominates serving latency.

By focusing on these guidelines, Punica differentiates itself from general inference engines (which ignore adapter heterogeneity) and prior adapter servers (which lack concurrent execution capabilities). The system assumes that LoRA weights can be loaded on-demand with negligible latency (**Section 5.2** notes loading an entire model takes ~**2ms** over PCIe Gen4), allowing the scheduler to dynamically place requests on any GPU regardless of which adapters are currently resident. This flexibility is the key to achieving the high consolidation rates demonstrated in the evaluation.

## 3. Technical Approach

This is a systems paper that presents a serving infrastructure for multi-tenant LoRA models. The core idea is to design a CUDA kernel that enables batching of GPU operations across different LoRA adapters while maintaining a single copy of the base model in memory.

### 3.1 Reader orientation (approachable technical breakdown)

Punica is a GPU cluster management system that allows multiple users to run different LoRA-adapted versions of the same base model on shared hardware without each user needing dedicated GPUs. It solves the problem of GPU resource waste by introducing a custom kernel (`SGMV`) that can process requests with different LoRA weights in the same batch, combined with a scheduler that dynamically consolidates workloads onto the minimum number of active GPUs.

### 3.2 Big-picture architecture (diagram in words)

The system consists of four major components that work together to serve requests efficiently. **Frontend servers** expose RESTful APIs to end-users and forward serving requests (containing a LoRA model identifier and a prompt) to the **Scheduler**. The **Scheduler** maintains a global view of all GPU states, including their current working sets and available KvCache memory, and dispatches requests to specific GPUs based on load. Each **GPU Runner** controls one or more GPUs, communicates with the scheduler, and executes the actual model inference using the custom CUDA kernels. Finally, the **CUDA Kernel Layer** (containing `SGMV` and attention kernels) performs the actual computation, streaming generated tokens back through the runners to the scheduler, then to frontends, and finally to end-users. This architecture is illustrated in **Figure 2** of the paper.

### 3.3 Roadmap for the deep dive

- First, we explain the `SGMV` kernel design because it is the foundational innovation that enables heterogeneous batching — without this kernel, the rest of the system cannot function.
- Second, we describe the scheduling mechanisms that decide which requests go to which GPUs, since the kernel's capabilities directly inform scheduling decisions.
- Third, we cover on-demand model loading and request migration, which are enabled by the small size of LoRA weights and the kernel's flexibility.
- Fourth, we explain the KvCache memory layout, which is critical for supporting continuous batching across requests with different sequence lengths.
- Finally, we discuss implementation details including the Python library and system components written in Rust.

### 3.4 Detailed, sentence-based technical breakdown

**The SGMV Kernel: Enabling Heterogeneous Batching**

The fundamental technical contribution of Punica is the Segmented Gather Matrix-Vector Multiplication (`SGMV`) kernel, which allows a single GPU to compute LoRA adaptations for multiple different models within the same batch. To understand why this is necessary, recall that a LoRA-adapted layer computes $y = xW + xAB$, where $W$ is the base model weight and $AB$ is the LoRA adaptation. The $xW$ term can be batched normally using standard GEMM (General Matrix Multiply) because all requests share the same $W$. However, the $xAB$ term requires different $A$ and $B$ matrices for different LoRA models, which standard kernels cannot handle in a single launch.

The `SGMV` operator computes the LoRA addon term by separating it into two kernel launches. First, the system initializes an intermediate vector $v = 0$, then computes $v = xA$ (the shrink operation), and finally computes $y = vB$ (the expand operation). **Figure 3** illustrates the semantics: for a batch containing $n$ different LoRA models, the input vectors are partitioned by model, and each partition is multiplied by its corresponding LoRA weight matrix. Formally, if $s_i$ denotes the last element index for the $i$-th model within the batch (where $s_0 = 0$ and $s_n$ is the total batch size), the output is computed as:

$$
\begin{pmatrix}
\begin{pmatrix}
y_1 \\
\vdots \\
y_{s_1}
\end{pmatrix} \\
\vdots \\
\begin{pmatrix}
y_{s_{n-1}+1} \\
\vdots \\
y_{s_n}
\end{pmatrix}
\end{pmatrix}
:=
\begin{pmatrix}
\begin{pmatrix}
x_1 \\
\vdots \\
x_{s_1}
\end{pmatrix} \\
\vdots \\
\begin{pmatrix}
x_{s_{n-1}+1} \\
\vdots \\
x_{s_n}
\end{pmatrix}
\end{pmatrix}
W +
\begin{pmatrix}
\begin{pmatrix}
x_1 \\
\vdots \\
x_{s_1}
\end{pmatrix} A_1 B_1 \\
\vdots \\
\begin{pmatrix}
x_{s_{n-1}+1} \\
\vdots \\
x_{s_n}
\end{pmatrix} A_n B_n
\end{pmatrix}
$$

The left-hand multiplication is the backbone computation handled by regular GEMM, while the right-hand side is the LoRA addon computed by `SGMV`.

The kernel is classified into two categories based on dimensionality: `SGMV-shrink` and `SGMV-expand`. The shrink operation ($v = xA$) reduces a high-dimensional input feature to a low-rank output, while the expand operation ($y = vB$) expands the low-rank feature back to a high-dimensional output. **Figure 4** shows how these are scheduled differently on the GPU. For the expand kernel, the system splits matrix $A$ on the output feature dimension as $A = [A^{(1)} \cdots A^{(n)}]$ and dispatches different $v^{(i)} = xA^{(i)}$ to different threadblocks, with concatenation forming the final result. For the shrink kernel, the output dimension is too thin to provide sufficient parallelism, so the system adopts a Split-K strategy where matrix $B$ is split on the input feature dimension as $B = [B^{(1)} \cdots B^{(k)}]^T$. Different partial results $y^{(i)} = vB^{(i)}$ are dispatched to different threadblocks, and after all partial sums are computed, a grid synchronization followed by cross-threadblock reduction aggregates the results as $y = \sum_{i=1}^{k} y^{(i)}$. Both kernels use GPU Tensor Cores to accelerate matrix multiplication when operational intensity is sufficient.

A critical design choice emerges when each request has a distinct LoRA index. In this case, the computation degrades to matrix-vector multiplication, which is entirely I/O-bound with low operational intensity. For this scenario, Punica uses a specific schedule that maximizes memory bandwidth utilization and deliberately does not use Tensor Cores, since the overhead of Tensor Core operations would outweigh their benefits for such small computations. The roofline analysis in **Figure 7** validates this design: in the "Distinct" case, arithmetic intensity does not change with batch size because FLOP and I/O grow at the same rate, confirming the I/O-bound nature.

**Scheduler Design: Consolidating Multi-Tenant Workloads**

The Punica scheduler operates at two levels: it schedules new user requests at a per-request level and migrates old requests between GPUs at a per-iteration level. The scheduler maintains a global view of all GPU states, specifically tracking the working set of requests (the batch input for LLM invocation) and the available KvCache memory on each GPU. When a new request arrives, the scheduler routes it to the GPU that currently has the largest working set of requests while satisfying two constraints: the GPU has not reached the maximum batch size limit, and it has enough memory for the new request's KvCache. When multiple candidate GPUs exist, the one with the highest GPU UUID receives the request. This "busy stays busy" scheduling logic ensures that active GPUs remain at high utilization while lightly loaded GPUs naturally drain as requests terminate.

The maximum batch size is a critical hyperparameter that balances cluster throughput against per-token latency. Oversized batches significantly slow down latency while providing only marginal throughput gains. Through profiling on A100 GPUs, the authors set the maximum batch size to **32** (**Section 5.1**). When all GPUs are fully occupied (either reaching maximum batch size or insufficient memory), new requests are queued and scheduled in a first-come-first-serve manner when GPUs become available.

This scheduling approach has important implications for cluster scaling. Because the scheduler consolidates workloads onto the minimum number of active GPUs, idle GPUs (those with batch size = 0) are likely to stay idle. In a cloud setting, this enables straightforward scaling decisions: if no lightly loaded GPU exists in the cluster, Punica requests more GPUs; if GPU servers have no load, Punica can return those resources to the cloud provider. **Figure 13** demonstrates this behavior in practice, showing that GPUs usually run at maximum batch size when active, and when a GPU becomes idle, it remains idle and can be released.

**On-Demand Model Loading: Leveraging LoRA's Small Footprint**

Weight sharing between LoRA models and the underlying pre-trained model makes model loading exceptionally fast. The LoRA model (matrices $A$ and $B$) is only **1%** of the underlying pre-trained model's size (**Section 5.2**). Loading a LoRA model from main memory to GPU memory is merely an asynchronous host-to-device memory copy bounded by PCIe bandwidth. On PCIe Gen4 x16, it takes approximately **50µs** to load a single layer and **2ms** to load the entire model.

The system design leverages the fact that each decode step takes around **30ms** to complete, and each request might need thousands of decode steps. Rather than implementing sophisticated layer-by-layer loading, Punica uses a simpler method: when a new request is added to a GPU and its LoRA model is not already loaded, the system issues an asynchronous memory copy to load the LoRA weight while the GPU continues running other inputs in the batch. By the end of the model execution, the weight has finished loading, and the new request can join the batch naturally. This overlapping of memory copy with GPU computation eliminates loading delay from the critical path.

**Request Migration: Enabling Dynamic Consolidation**

As each request generates more tokens, its KvCache occupies increasing GPU memory. When a GPU runs out of space for KvCache, it must migrate some requests to other GPUs. Request migration consists of two steps: evict and add. The scheduler evicts the newest request from the GPU to preserve FCFS semantics, and the scheduling for the evicted request follows the same process as adding a new request.

**Figure 5** illustrates the migration workflow for a request R3 moving from GPU 1 to GPU 2. The scheduler first sends a cancellation command to GPU 1. After GPU 1 finishes running the previous batch, it processes the cancellation, releases the KvCache for R3, and omits R3's new token generated in the previous batch. Immediately after sending the cancellation, the scheduler adds R3 to GPU 2. GPU 2 runs a prefill step on R3's original prompt plus all previously generated tokens, which reestablishes R3's KvCache on GPU 2. GPU 2 then starts streaming R3's new tokens to the scheduler.

The system chooses recomputation over moving the KvCache for simplicity. PagedAttention (**Kwon et al., 2023**) has shown that recomputation's latency is equal to or better than moving the KvCache in most cases, which agrees with Punica's observations. This design choice avoids the complexity of implementing KvCache transfer mechanisms while maintaining equivalent performance.

**KvCache Memory Layout: Enabling Continuous Batching**

The memory layout for KvCache is critical for text generation batching throughput. The HuggingFace Transformers library uses a KvCache layout that can be viewed as shape $[L, 2, B, N, S, D]$, where $L$ is the number of layers, $2$ is for Key and Value projection, $B$ is batch size, $N$ is the number of heads, $S$ is the sequence length, and $D$ is the head dimension. This layout has two critical problems. First, in each decode step, the library concatenates one tensor along the sequence length dimension, which is inefficient because it needs to read the whole KvCache and write a new copy, whereas the new tensor is only $1/S$ of the KvCache. Second, and more importantly, the batching dimension is not the outermost dimension, which means requests in the batch are hard to separate in the KvCache.

**Figure 6** illustrates the consequence of this inseparable layout. When consecutive requests are batched together, shorter requests in the batch must run additional decode steps until all requests meet their stopping condition, resulting in wasted computation. FasterTransformer and DeepSpeed suffer from similar problems (**Section 5.4**).

Punica instead uses a separable KvCache layout with shape $[\sum_i \lceil S_i / P \rceil, L, 2, N, P, D]$, where $S_i$ is the length of sequence $i$ and $P$ is the page size. The system uses paged KvCache (**Kwon et al., 2023**) to minimize memory fragmentation. Critically, the batching dimension is placed on the outermost dimension to enable continuous batching. This allows requests to enter and leave the batch independently without forcing other requests to wait, eliminating the wasted decode steps shown in **Figure 6**.

**Implementation Details: Python Library and System Components**

Punica's implementation consists of two parts: a Python library on top of PyTorch for single-GPU execution and system components for cluster-wide serving. The CUDA kernels are exposed as a PyTorch Extension using PyBind11. The Llama model implementation is adapted from the HuggingFace Transformers library. For self-attention computation, Punica uses the FlashInfer open-source project (**Ye, 2023**), which supports batch decoding without padding and paged KvCache similar to PagedAttention. The system also fuses LayerNorm operations, reducing latency from **110µs** to **4µs** (**Section 6**).

A key implementation detail is how Punica mixes new requests in the Prefill stage with existing requests in the Decode stage within one batch. All inputs are concatenated along the sequence length dimension, with Prefill requests placed at the beginning and Decode requests at the latter part. The system passes a `BatchLen` struct to distinguish different requests, containing a list of indices indicating the starting index of each Prefill request and a number indicating the count of Decode requests. In the self-attention layer, indices and leading input states are passed to the BatchPrefill kernel, while trailing input states are passed to the BatchDecode kernel. Within a batch, requests sharing the same LoRA model are organized to be consecutive, and the tail of Prefill requests can share a LoRA model with the head of Decode requests if possible. Segment indices for the `SGMV` kernel are generated before each batched model invocation, and both `BatchLen` and `SGMV` segment indices remain constant for the entire model invocation. This design avoids recomputation (L times for `BatchLen` and 7L times for `SGMV` segment indices, where L is the number of layers).

The scheduler, frontend, and runner components are written in Rust. Unary RPC and streaming text chunks are implemented via WebSocket, with I/O handled asynchronously. The Runner spawns a Python subprocess for each GPU, which acts as a thin wrapper around the PyTorch library. The Runner main process communicates with these subprocesses using pipes.

**Evaluation Configuration and Workload Characteristics**

The evaluation uses two testbeds with specific configurations that are important for interpreting results. Testbed #1 is a single server with one NVIDIA A100 80GB GPU, which allows study of the LoRA batching effect with large GPU memory. Testbed #2 consists of two NVIDIA HGX A100 40GB servers with 8 GPUs on each server, equipped with fast NvSwitch technology for studying tensor parallelism and cluster deployment. The experiments use Llama-2 models with 7B, 13B, and 70B parameters, with LoRA rank set to **16** for all experiments. LoRA is applied to all dense projections in the transformer layers.

Four workload distributions are considered to evaluate different multi-tenant scenarios. The "Distinct" distribution means each request is for a distinct LoRA model. The "Uniform" distribution means all LoRA models are equally popular, using $\lceil \sqrt{n} \rceil$ models for $n$ requests. The "Skewed" distribution follows a Zipf-$\alpha$ distribution where the number of requests to the $i$-th most popular model is $\alpha$ times that of the $(i+1)$-th model, with $\alpha = 1.5$ in the experiments. The "Identical" distribution means all requests are for the same LoRA model. These distributions are critical for understanding the performance results in **Section 7**, as they represent different real-world deployment scenarios ranging from worst-case (Distinct) to best-case (Identical) for heterogeneous batching.

## 4. Key Insights and Innovations

Punica's contributions extend beyond standard system optimizations; they represent fundamental shifts in how multi-tenant LLM serving is architected. While prior work focused on optimizing single-model inference or static multi-model hosting, Punica introduces capabilities that decouple model identity from batch execution. Below, we analyze the three most significant innovations, distinguishing between incremental improvements and fundamental changes to the serving paradigm.

### 4.1 Fundamental Innovation: Heterogeneous Batching via SGMV

The most critical innovation in Punica is the ability to batch requests that use **different** LoRA adapters within the same GPU kernel launch. This stands in stark contrast to prior high-performance serving systems like **vLLM** or **Orca**, which assume **homogeneous batching**—all requests in a batch must share identical weights.

*   **The Prior Limitation:** In standard serving systems, if Request A uses LoRA Adapter $\alpha$ and Request B uses LoRA Adapter $\beta$, they cannot be processed in the same batch. The system must either serialize them (running batch size 1 for each) or separate them onto different GPUs. As shown in **Section 7.2**, this limitation causes baseline systems to collapse to **batch size 1** in the "Distinct" workload scenario, drastically reducing throughput.
*   **The Punica Breakthrough:** The `SGMV` kernel (detailed in **Section 3**) breaks the coupling between batch membership and weight identity. By segmenting the input vector and gathering specific LoRA weights for each segment, Punica allows a single GPU to process diverse adapters concurrently.
*   **Significance:** This is not an incremental speedup; it is a capability expansion. It enables the system to maintain high GPU utilization even when tenant demand is highly fragmented. The evaluation in **Figure 11** demonstrates this impact: in the "Distinct" workload, Punica maintains high throughput while baselines degrade significantly. The roofline analysis in **Figure 7** confirms that even in this heterogeneous case, the kernel achieves high arithmetic intensity by leveraging parallelism across the batch, rather than stalling on memory bandwidth.

### 4.2 Empirical Insight: LoRA-Model-Agnostic Performance

A subtle but powerful finding drives Punica's simplified scheduler design: **the latency of a transformer layer is roughly independent of which LoRA model is being served.**

*   **The Expectation:** One might assume that serving different LoRA models would introduce variable overheads due to memory access patterns or weight loading times, necessitating complex affinity-based scheduling (e.g., "keep Request A on GPU 1 because Adapter $\alpha$ is already loaded").
*   **The Reality:** **Figure 10** (Transformer Layer Benchmark) shows that layer latency remains consistent across "Distinct," "Uniform," "Skewed," and "Identical" workloads. The computation time for the LoRA add-on is negligible compared to the backbone dense projection and self-attention.
*   **Significance:** This empirical insight fundamentally simplifies the scheduling problem. Because performance is **LoRA-model-agnostic**, the scheduler does not need to track *which* adapters are resident on which GPUs. It only needs to track *how many* requests each GPU is handling. This allows Punica to implement the "busy stays busy" logic described in **Section 5.1** without worrying about model placement constraints. This contrasts with systems like **PetS** (**Zhou et al., 2022**), which share memory but lack concurrent execution, or systems that require explicit model loading management before scheduling.

### 4.3 System-Level Innovation: Elastic Consolidation via Recomputation

Punica introduces a novel approach to resource elasticity in GPU clusters by combining request migration with recomputation, enabling dynamic scaling of GPU resources.

*   **The Challenge:** In multi-tenant serving, workload demand fluctuates. To save costs, a system should release idle GPUs. However, migrating active requests between GPUs is difficult because of the **KvCache** state (the memory of past tokens). Moving KvCache physically is complex and bandwidth-intensive.
*   **The Punica Solution:** Instead of moving KvCache, Punica migrates requests by **recomputing** the prefill stage on the destination GPU. As described in **Section 5.3**, the scheduler cancels the request on the source GPU (releasing KvCache) and adds it to the destination GPU, which runs a prefill step on the original prompt plus generated tokens.
*   **Significance:** This design choice trades a small amount of compute (re-running prefill) for significant system simplicity and flexibility. The paper notes that recomputation latency is equal to or better than moving KvCache, aligning with findings from **PagedAttention** (**Kwon et al., 2023**). The impact is visible in **Figure 13** (Cluster Deployment): as load decreases, GPUs become fully idle (batch size = 0) and stay idle, allowing them to be returned to the cloud provider. This enables true **elastic scaling**, where the cluster size adapts to demand without manual intervention or complex state transfer protocols.

### 4.4 Summary of Contributions vs. Prior Work

The table below summarizes how Punica's innovations differentiate it from existing approaches.

| Feature | Prior Systems (vLLM, Orca) | Adapter Systems (PetS) | **Punica** |
| :--- | :--- | :--- | :--- |
| **Batching Semantics** | Homogeneous (Same Model) | Sequential / Non-concurrent | **Heterogeneous (Different LoRA)** |
| **Scheduler Complexity** | High (Model Affinity) | High (Memory Sharing) | **Low (Model-Agnostic)** |
| **Resource Elasticity** | Static / Manual Scaling | Static | **Dynamic (Migration + Recompute)** |
| **Throughput (Distinct)** | Low (Batch Size 1) | Moderate | **High (Batch Size 32)** |

These innovations collectively enable Punica to achieve the **12x higher throughput** claimed in the Abstract while adding only **2ms latency per token**. The system does not merely optimize existing kernels; it redefines the constraints under which multi-tenant serving operates, allowing providers to consolidate workloads that were previously forced onto separate hardware.

## 5. Experimental Analysis

This section dissects Punica's evaluation methodology and results. Understanding *how* the authors measured performance is as important as the numbers themselves, because the experimental design reveals what claims are well-supported and which remain conditional. We will walk through the testbeds, workloads, baselines, and then analyze each result category with specific numbers from the paper.

### 5.1 Evaluation Methodology

**Testbed Configuration**

The authors evaluate Punica on two distinct hardware configurations, each serving a specific analytical purpose (**Section 7**):

*   **Testbed #1 (Single-GPU Analysis):** One NVIDIA A100 80GB GPU. This configuration isolates the LoRA batching effect without complications from tensor parallelism or inter-GPU communication. The large 80GB memory allows the authors to study maximum batch sizes without memory constraints.
*   **Testbed #2 (Cluster Analysis):** Two NVIDIA HGX A100 40GB servers with 8 GPUs each (16 GPUs total), connected via NvSwitch. This configuration enables evaluation of tensor parallelism for large models (70B) and cluster-wide scheduling behavior.

**Model Configurations**

All experiments use Llama-2 models (**Touvron et al., 2023**) at three scales:
*   **7B parameters:** Fits on a single A100 80GB
*   **13B parameters:** Fits on a single A100 80GB
*   **70B parameters:** Requires 8 GPUs with tensor parallelism

For all experiments, the LoRA rank is fixed at **$r = 16$**, and LoRA is applied to **all dense projections** in the transformer layers (Query-Key-Value-Output projections in attention and MLP). The authors note they use **random weights for LoRA models** because weight values do not affect latency performance—only the matrix dimensions matter for kernel execution time.

**Workload Distributions**

The evaluation considers four request distribution patterns across LoRA models (**Section 7**):

| Distribution | Description | Real-World Analogy |
| :--- | :--- | :--- |
| **Distinct** | Each request uses a different LoRA model | Worst-case: 100 tenants, 100 unique adapters |
| **Uniform** | All LoRA models equally popular; uses $\lceil \sqrt{n} \rceil$ models for $n$ requests | Moderate diversity: many tenants, shared adapters |
| **Skewed** | Zipf-$\alpha$ distribution with $\alpha = 1.5$ | Realistic: some adapters more popular than others |
| **Identical** | All requests use the same LoRA model | Best-case: single tenant or all tenants share one adapter |

These distributions are critical because they represent the spectrum from **heterogeneous batching** (Distinct) to **homogeneous batching** (Identical). Punica's advantage should be most pronounced in the Distinct case, where baseline systems cannot batch at all.

**Baseline Systems**

Since no well-known multi-LoRA serving system exists, the authors compare against popular LLM backbone serving systems with varying degrees of relaxation (**Section 7**):

*   **HuggingFace Transformers + PEFT:** Full LoRA support but no CUDA kernel optimizations
*   **DeepSpeed:** Full LoRA support with some optimizations
*   **FasterTransformer:** Backbone-only (no LoRA support); LoRA costs omitted
*   **vLLM:** Backbone-only (no LoRA support); LoRA costs omitted

> "We omit the model switching costs for baseline systems." (**Section 7**)

This is an important caveat: baseline systems are given favorable treatment by not accounting for the time needed to load different LoRA adapters. This makes Punica's performance advantage more conservative—if anything, the real-world gap would be larger.

**Metrics**

The primary metrics are:
*   **Throughput:** Tokens per second (tok/s)
*   **Latency:** Milliseconds per token (ms/token)
*   **GPU Utilization:** Batch size over time

### 5.2 Microbenchmark Results: Validating the SGMV Kernel

Before evaluating end-to-end text generation, the authors validate the core `SGMV` kernel through microbenchmarks. These results establish whether the kernel-level innovation actually delivers the expected performance characteristics.

**Roofline Analysis (Figure 7)**

**Figure 7** presents a roofline model plot for the `SGMV` kernel, showing achieved FLOP/s against arithmetic intensity (FLOP/I/O ratio). The plot includes four workload distributions and two reference lines:
*   **Diagonal dotted line:** Memory bandwidth limit (1.935 TB/s for A100)
*   **Top dotted line:** Peak FP16 performance (312 TFLOP/s for A100)

Key observations:
*   In the **Distinct** case, arithmetic intensity does not change with batch size because FLOP and I/O grow at the same rate. Performance increases with batch size due to better parallelism, but the kernel remains I/O-bound.
*   In the **Identical** case, the line follows the memory bandwidth slope, confirming `SGMV` is memory-bandwidth bounded when all requests share the same LoRA weights.
*   **Uniform** and **Skewed** cases sit between these extremes, showing a combination of increasing parallelism and increasing arithmetic intensity.

This analysis validates the kernel design choice discussed in **Section 3.4**: for Distinct workloads, the system deliberately avoids Tensor Cores and maximizes memory bandwidth utilization instead.

**LoRA Operator Comparison (Figure 8)**

**Figure 8** compares three implementations of the batched LoRA operator across all four workload distributions:
1.  **Loop:** Sequential for-loop over each LoRA model
2.  **Gather-BMM:** Stack weights into a single matrix, then use `torch.bmm()`
3.  **SGMV:** Punica's custom kernel

The results show dramatic differences:

*   **Distinct workload:** Loop performs terribly (multiple rounds of batch size 1). Gather-BMM latency increases rapidly from **37µs to 116µs** as batch size grows from 1 to 64 due to Gather slowdown. SGMV increases gradually from **37µs to 46µs** over the same range.
*   **Identical workload:** All implementations have the same semantics (standard BMM). SGMV latency remains nearly constant from **37µs to 40µs**, indicating it implements BMM more efficiently than `torch.bmm()` for LoRA use cases.

> "Overall, SGMV significantly outperforms baseline implementations regardless of workloads." (**Section 7.1**)

**LoRA Rank Sensitivity (Figure 9)**

**Figure 9** tests LoRA ranks of 8, 16, 32, and 64. In the Distinct case, single-request batch latency is around **42µs** for all ranks, but batch size 64 latency increases with rank:
*   $r=8$: **72µs**
*   $r=16$: **75µs**
*   $r=32$: **89µs**
*   $r=64$: **118µs**

When weight sharing exists (Uniform, Skewed, Identical), latency remains stable at **42µs to 45µs** across all batch sizes and ranks. This confirms that the LoRA add-on computation is small relative to the backbone, supporting the **LoRA-model-agnostic performance** insight from **Section 4.2**.

**Transformer Layer Benchmark (Figure 10)**

**Figure 10** evaluates full transformer layer performance (including self-attention and dense projections) for 7B and 13B models at sequence lengths 512 and 2048. Key findings:

*   **Batching effect is stronger for shorter sequences:** At sequence length 512, latency increases only **72%** when batch size grows from 1 to 32. At sequence length 2048, self-attention dominates, reducing the layer-wise batching benefit.
*   **Workload-agnostic performance:** Layer latency is roughly the same across Distinct, Uniform, Skewed, and Identical workloads. This is because LoRA add-on computation is small compared to backbone dense projection and self-attention.

> "This LoRA-model-agnostic performance property enables us to schedule different LoRA models as if one model." (**Section 7.1**)

This result directly justifies Punica's simplified scheduler design: the system does not need to track which adapters are on which GPUs because performance is invariant to adapter identity.

### 5.3 End-to-End Text Generation Results

The most critical evaluation is end-to-end text generation throughput, as this reflects real-world serving performance.

**Single-GPU Performance (7B and 13B Models)**

**Figure 11** shows throughput results for 7B and 13B models on a single A100 80GB GPU. All systems use a maximum batch size of **32**, and 1000 requests are generated (approximately 101k tokens total).

| Model | System | Distinct | Uniform | Skewed | Identical |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **7B** | Punica | **1044 tok/s** | **1044 tok/s** | **1044 tok/s** | **1044 tok/s** |
| **7B** | vLLM (backbone) | ~100 tok/s | ~300 tok/s | ~500 tok/s | **1140 tok/s** |
| **13B** | Punica | **693 tok/s** | **693 tok/s** | **693 tok/s** | **693 tok/s** |
| **13B** | vLLM (backbone) | ~70 tok/s | ~200 tok/s | ~400 tok/s | **789 tok/s** |

Key observations:

*   **Punica maintains consistent throughput** across all workload distributions because it can batch different LoRA models together. The throughput is identical for Distinct, Uniform, Skewed, and Identical cases.
*   **Baselines degrade severely in Distinct case:** All baseline systems run with **batch size 1** when every request uses a different LoRA model, because they cannot batch heterogeneous requests. This explains the ~10x throughput gap in the Distinct workload.
*   **vLLM slightly outperforms Punica in Identical case:** vLLM achieves **1140 tok/s** vs Punica's **1044 tok/s** for 7B, and **789 tok/s** vs **693 tok/s** for 13B. This is because vLLM runs backbone-only (no LoRA overhead), while Punica includes the LoRA add-on computation. The overhead is approximately **8-12%**.

> "In the Distinct case, all baseline systems run with a batch size of 1, and thus, the throughput is low." (**Section 7.2**)

**70B Model with Tensor Parallelism**

**Figure 12** evaluates the 70B model on 8 GPUs using Megatron's tensor parallel scheme. Both Punica and vLLM use the same parallelization strategy, making this a fair comparison.

| System | Distinct | Uniform | Skewed | Identical |
| :--- | :--- | :--- | :--- | :--- |
| **Punica** | **441-446 tok/s** | **441-446 tok/s** | **441-446 tok/s** | **441-446 tok/s** |
| **vLLM (backbone)** | **21-25 tok/s** | **21-25 tok/s** | **21-25 tok/s** | **457 tok/s** |

The results mirror the single-GPU findings but at larger scale:
*   Punica delivers **441-446 tok/s** consistently regardless of LoRA popularity distribution.
*   vLLM achieves **457 tok/s** in the Identical case (backbone-only) but collapses to **21-25 tok/s** when serving multiple LoRA models.
*   This represents approximately **18x throughput improvement** for Punica in multi-LoRA scenarios at the 70B scale.

**Latency Overhead**

The Abstract claims Punica adds only **2ms latency per token**. This comes from two sources:
1.  **LoRA computation overhead:** The `SGMV` kernel adds minimal latency compared to backbone computation (as shown in **Figure 10**).
2.  **On-demand model loading:** Loading a full LoRA model takes approximately **2ms** over PCIe Gen4 x16 (**Section 5.2**), but this is overlapped with GPU computation and does not appear on the critical path.

The evaluation does not provide a detailed latency breakdown table, but the microbenchmark results in **Figure 8** and **Figure 9** support the claim that LoRA add-on latency is in the tens of microseconds range per layer, which aggregates to a few milliseconds per token across all layers.

### 5.4 Cluster Deployment Behavior

**Figure 13** shows Punica's behavior over a one-hour experiment on 16 GPUs (Testbed #2) with time-varying load. The request rate follows a pattern that gradually increases and then decreases, with gaps between arrivals following an exponential distribution (Poisson process). LoRA model popularity follows Zipf-1.5 distribution.

The figure has three panels:
1.  **Top:** Request rate over time (Req/s)
2.  **Middle:** Text generation throughput (Tok/s)
3.  **Bottom:** Batch size per GPU over time

Key observations:
*   **GPUs run at maximum batch size when active:** The scheduler's "busy stays busy" logic ensures active GPUs operate at or near the maximum batch size of **32**.
*   **Idle GPUs stay idle:** When a GPU's batch size drops to 0, it remains idle. This enables clean resource release in cloud settings.
*   **Occasional smaller batches:** Sometimes a GPU runs with batch size &lt; 32 because it runs out of KvCache space and migrates requests to other GPUs.

> "When a GPU becomes idle (batch size = 0), it is likely that it stays idle, which can then be released to the cloud provider if necessary." (**Section 7.3**)

This result validates the **elastic consolidation** claim from **Section 4.3**: Punica can dynamically scale the cluster size based on demand without manual intervention.

### 5.5 Critical Assessment of Experimental Claims

**Do the experiments support the 12x throughput claim?**

The Abstract claims **12x higher throughput** compared to state-of-the-art LLM serving systems. Looking at the data:

*   **7B model, Distinct workload:** Punica achieves **1044 tok/s** vs baseline ~100 tok/s = **~10x improvement**
*   **13B model, Distinct workload:** Punica achieves **693 tok/s** vs baseline ~70 tok/s = **~10x improvement**
*   **70B model, Distinct workload:** Punica achieves **441 tok/s** vs vLLM **21-25 tok/s** = **~18x improvement**

The 12x claim is **conservative and well-supported** by the data. In some cases (70B model), the improvement exceeds 12x.

**Do the experiments support the 2ms latency claim?**

The latency claim is **plausible but less rigorously documented**. The paper provides:
*   LoRA model loading time: **2ms** total (**Section 5.2**)
*   LoRA operator latency: **37-116µs** per operator (**Figure 8**)
*   LayerNorm fusion: reduces latency from **110µs to 4µs** (**Section 6**)

However, the paper does not provide a detailed end-to-end latency breakdown showing per-token latency across different workloads. The claim appears to be based on the observation that LoRA computation is small relative to backbone computation, plus the overlapped model loading. This is a **minor gap** in the evaluation rigor.

**Are there any failure cases or limitations?**

The evaluation reveals several conditional results:

1.  **Identical workload disadvantage:** When all requests use the same LoRA model, vLLM (backbone-only) outperforms Punica by **8-12%** because Punica includes LoRA overhead. This is expected and acceptable—Punica optimizes for the multi-tenant case, not single-tenant.

2.  **Sequence length sensitivity:** **Figure 10** shows that batching benefits diminish for longer sequences (2048 vs 512) because self-attention dominates. Punica's advantage is most pronounced for **shorter to medium sequence lengths**.

3.  **KvCache migration overhead:** Request migration uses recomputation rather than KvCache transfer. While the paper cites PagedAttention showing recomputation is equal or better, there is no direct measurement of migration latency in Punica's specific configuration. This is a **minor gap**.

4.  **No comparison to multi-adapter systems:** The paper notes PetS (**Zhou et al., 2022**) does not support concurrent execution, so no direct comparison is made. A reader might wonder how Punica compares to other adapter-serving approaches beyond backbone-only systems.

**Robustness Checks**

The authors perform several robustness checks:
*   **Multiple model scales:** 7B, 13B, and 70B models tested
*   **Multiple workload distributions:** Distinct, Uniform, Skewed, Identical
*   **Multiple LoRA ranks:** 8, 16, 32, 64 tested in microbenchmarks
*   **Multiple sequence lengths:** 512 and 2048 tested

However, the evaluation does not include:
*   **Different LoRA application patterns:** LoRA is applied to all dense projections; results for partial LoRA application (e.g., attention only) are not shown.
*   **Quantized models:** The paper mentions QLoRA (**Dettmers et al., 2023**) in Related Work but does not evaluate Punica with quantized backbones.
*   **Longer-running cluster experiments:** The cluster deployment experiment runs for **one hour**; longer-term stability and fragmentation effects are not measured.

### 5.6 Summary of Experimental Findings

The evaluation convincingly demonstrates that Punica achieves its core design goals:

| Claim | Evidence | Confidence |
| :--- | :--- | :--- |
| **12x throughput improvement** | Figures 11, 12 show 10-18x improvement in Distinct workload | **High** |
| **2ms latency overhead** | Microbenchmarks show µs-level operator latency; loading is overlapped | **Medium** (less detailed breakdown) |
| **LoRA-model-agnostic scheduling** | Figure 10 shows consistent layer latency across workloads | **High** |
| **Elastic consolidation** | Figure 13 shows idle GPUs stay idle, enabling resource release | **High** |
| **SGMV kernel efficiency** | Figures 7, 8, 9 validate kernel performance across configurations | **High** |

The experiments are well-designed to isolate Punica's contributions, and the results consistently support the paper's claims. The main limitation is the lack of detailed end-to-end latency breakdowns, but the microbenchmark data provides strong indirect evidence for the latency claims.

## 6. Limitations and Trade-offs

No system design is without compromises. Punica makes specific architectural choices that optimize for multi-tenant LoRA serving but introduce constraints and leave certain scenarios unaddressed. This section critically examines the assumptions underlying Punica's approach, the problem settings it does not cover, and the trade-offs inherent in its design decisions. Understanding these limitations is essential for determining when Punica is the appropriate solution and when alternative approaches may be preferable.

### 6.1 Core Architectural Assumptions

Punica's design rests on several foundational assumptions that, if violated, would undermine its effectiveness.

**Assumption 1: All LoRA Models Share the Same Pre-trained Backbone**

The entire system architecture depends on weight sharing across tenants. As stated in **Section 3**, "each GPU loads the backbone pre-trained large language model" and "only the LoRA components of models are swapped in from remote storage when needed." This assumption enables the memory efficiency gains but creates a hard constraint:

*   **Implication:** Punica cannot serve LoRA adapters derived from different base models (e.g., Llama2-7B adapters mixed with Llama2-13B adapters) on the same GPU. Each GPU must be dedicated to a single backbone model family.
*   **Evidence:** The evaluation only tests LoRA models adapted from Llama2 7B, 13B, and 70B separately (**Section 7**). There is no discussion of cross-backbone serving or how the scheduler would handle heterogeneous backbone requirements.
*   **Real-World Impact:** In production environments where tenants may request adapters for different model sizes or architectures, the cluster must be partitioned by backbone type, reducing the consolidation benefits.

**Assumption 2: LoRA Weights Are Small Enough for On-Demand Loading**

Punica's scheduling flexibility depends on the claim that LoRA weights can be loaded with negligible latency. **Section 5.2** states that LoRA models are only **1%** of the underlying pre-trained model's size, and loading takes approximately **2ms** over PCIe Gen4 x16.

*   **Implication:** If LoRA ranks increase significantly or if adapters include additional components beyond the standard $A$ and $B$ matrices, loading times could become a bottleneck.
*   **Evidence:** **Figure 9** shows that LoRA operator latency increases with rank: for batch size 64 in the Distinct case, latency grows from **72µs** at $r=8$ to **118µs** at $r=64$. While this is still small relative to backbone computation, the paper does not evaluate ranks beyond 64.
*   **Trade-off:** Higher LoRA ranks improve adaptation quality but increase both computation time and memory bandwidth requirements for loading. Punica's benefits diminish as the LoRA component grows relative to the backbone.

**Assumption 3: Decode Stage Dominates Serving Cost**

The design guidelines in **Section 1** explicitly state "(G3) The decode stage is the predominant factor in the cost of model serving. We thus only need to focus on the decode stage performance."

*   **Implication:** Optimization efforts are concentrated on decode-stage batching, with prefill treated as a secondary concern.
*   **Evidence:** **Section 5.1** notes, "To minimize latency penalty, we limit the prefill batch size to 1 for each batch." This means prefill operations do not benefit from the same batching optimizations as decode.
*   **Trade-off:** Workloads with very long prompts (large prefill cost) relative to output length may not see the full benefits of Punica's optimizations. The system is optimized for chat-like interactions with short prompts and longer responses, not document summarization with long inputs and short outputs.

### 6.2 Unaddressed Scenarios and Edge Cases

The evaluation covers common multi-tenant scenarios but leaves several important edge cases unexplored.

**Scenario 1: Quantized Backbone Models**

The paper mentions quantization in **Section 8** (Related Work), citing QLoRA (**Dettmers et al., 2023**) and various quantization techniques. However, the evaluation does not test Punica with quantized backbones.

*   **Gap:** **Section 7** explicitly states, "We use random weights for LoRA models as the weight does not affect latency performance." There is no mention of quantized backbone weights (e.g., INT8, FP8) in the experimental setup.
*   **Potential Issue:** Quantized backbones change the memory bandwidth vs. compute balance. If the backbone computation becomes memory-bound due to quantization, the relative overhead of LoRA add-ons may increase, potentially reducing Punica's advantage.
*   **Open Question:** Does `SGMV` maintain its performance characteristics when the backbone uses quantized weights? The paper does not provide evidence.

**Scenario 2: Partial LoRA Application**

The evaluation applies LoRA to **all dense projections** in the transformer layers (**Section 7**). However, some fine-tuning strategies apply LoRA selectively (e.g., only to attention projections, not MLP).

*   **Gap:** **Section 7** states, "LoRA is applied to all dense projections." No results are shown for partial LoRA application patterns.
*   **Potential Issue:** If LoRA is applied to fewer layers, the relative overhead of the `SGMV` kernel setup (segment index generation, kernel launches) may become more significant compared to the computation saved.
*   **Open Question:** What is the minimum LoRA coverage required for Punica's batching benefits to outweigh the kernel management overhead?

**Scenario 3: Long-Term Cluster Stability**

The cluster deployment experiment in **Section 7.3** runs for **one hour** with time-varying load. This is insufficient to capture long-term system behavior.

*   **Gap:** **Figure 13** shows one hour of operation. The paper does not discuss memory fragmentation over extended periods, KvCache page allocation patterns, or scheduler state growth.
*   **Potential Issue:** Paged KvCache (**Section 5.4**) minimizes fragmentation, but over days or weeks of operation, fragmentation could accumulate, requiring periodic compaction or GPU restarts.
*   **Open Question:** How does Punica handle memory fragmentation over extended deployment periods? What maintenance operations are required?

**Scenario 4: Request Migration Latency**

Punica uses recomputation for request migration rather than KvCache transfer (**Section 5.3**). While the paper cites PagedAttention showing recomputation is "equal to or better than moving the KvCache," there is no direct measurement in Punica's configuration.

*   **Gap:** **Section 5.3** states, "We opt for recomputation instead of moving the KvCache for its simplicity." However, **Section 5.5** (Critical Assessment) notes, "there is no direct measurement of migration latency in Punica's specific configuration."
*   **Potential Issue:** For requests with very long generated sequences, recomputing the prefill stage (original prompt + all generated tokens) could become expensive compared to a direct KvCache transfer.
*   **Trade-off:** Simplicity vs. optimal migration performance. The system favors implementation simplicity over potentially better migration latency for edge cases.

### 6.3 Computational and Scalability Constraints

Punica's design introduces specific computational constraints that limit its applicability in certain scenarios.

**Constraint 1: Maximum Batch Size Limit**

The system enforces a maximum batch size of **32** based on A100 profiling (**Section 5.1**).

*   **Evidence:** "We profile A100 GPUs and decide to set the maximum batch size to 32."
*   **Implication:** This limit balances throughput against latency. Oversized batches "greatly slow down latency while providing marginal throughput gains."
*   **Trade-off:** In scenarios where latency is less critical than raw throughput (e.g., batch offline processing), this limit may be suboptimal. The system is optimized for interactive serving, not batch processing.
*   **Scalability Question:** Does this batch size limit hold for newer GPU architectures (e.g., H100, B200) with different memory bandwidth and compute characteristics? The paper only evaluates on A100.

**Constraint 2: Tensor Core Utilization in Distinct Workloads**

**Section 4** explains that when each request has a distinct LoRA index, the computation "degrades to matrix-vector multiplication, which is totally IO-bound" and the system "does not use Tensor Cores because of the low operational intensity of the operator."

*   **Evidence:** **Figure 7** (Roofline plot) shows the Distinct case remains I/O-bound regardless of batch size.
*   **Implication:** In the worst-case scenario (every request uses a different LoRA model), Punica cannot leverage Tensor Cores for the LoRA computation, limiting the absolute performance ceiling.
*   **Trade-off:** The system sacrifices peak compute utilization for flexibility. This is acceptable because the LoRA add-on is small relative to backbone computation (**Figure 10**), but it represents a fundamental limitation in the Distinct workload case.

**Constraint 3: Single-Backbone Per GPU**

As noted in **Section 6.1**, each GPU holds a single pre-trained backbone model. This creates a scalability constraint for clusters serving many different base models.

*   **Evidence:** **Section 3** states, "In Punica, each GPU loads the backbone pre-trained large language model."
*   **Implication:** If a provider needs to serve LoRA adapters for 10 different base models, they need at least 10 GPUs (one per backbone), regardless of actual load.
*   **Trade-off:** Memory efficiency within a backbone family vs. flexibility across backbone families. This is a fundamental constraint of the weight-sharing approach.

### 6.4 Performance Trade-offs vs. Baseline Systems

Punica's optimizations come with measurable trade-offs compared to specialized baseline systems.

**Trade-off 1: Single-Tenant Performance Penalty**

When all requests use the same LoRA model (Identical workload), Punica incurs overhead compared to backbone-only systems.

*   **Evidence:** **Section 7.2** and **Figure 11** show vLLM (backbone-only) achieves **1140 tok/s** vs Punica's **1044 tok/s** for 7B models, and **789 tok/s** vs **693 tok/s** for 13B models.
*   **Magnitude:** This represents an **8-12% throughput penalty** for Punica in the single-tenant case.
*   **Implication:** If a deployment scenario involves only a single LoRA adapter (or all tenants share one adapter), Punica is suboptimal compared to specialized single-model serving systems.
*   **Design Justification:** This trade-off is intentional—Punica optimizes for the multi-tenant case, not the single-tenant case.

**Trade-off 2: Sequence Length Sensitivity**

The batching benefits diminish for longer sequences because self-attention computation dominates.

*   **Evidence:** **Figure 10** shows that at sequence length 512, latency increases only **72%** when batch size grows from 1 to 32. At sequence length 2048, "self-attention takes longer time, which reduces the layer-wise batching effect."
*   **Implication:** Punica's advantages are most pronounced for **shorter to medium sequence lengths**. For very long context windows (e.g., 8k, 32k tokens), the relative benefit of LoRA batching decreases.
*   **Open Question:** How does Punica perform with the extended context lengths now common in modern LLMs (e.g., Llama2-7B with 4k+ context)? The evaluation only tests up to 2048 tokens.

**Trade-off 3: Scheduler Complexity vs. Model Affinity**

Punica's scheduler is simplified by the LoRA-model-agnostic performance property (**Section 4.2**), but this comes at the cost of not optimizing for model affinity.

*   **Evidence:** **Section 5.1** states the scheduler "does not need to track *which* adapters are resident on which GPUs. It only needs to track *how many* requests each GPU is handling."
*   **Implication:** The system does not attempt to keep requests for the same LoRA model on the same GPU to maximize cache hits. Instead, it relies on fast on-demand loading.
*   **Trade-off:** Simplicity and flexibility vs. potential cache optimization. If model loading times increase (e.g., slower PCIe, larger adapters), this design choice could become suboptimal.

### 6.5 Open Questions and Future Directions

Several questions remain unanswered by the current evaluation, representing opportunities for future work.

**Open Question 1: Interaction with Speculative Decoding**

**Section 8** mentions Speculative Decoding (**Leviathan et al., 2023; Miao et al., 2023**) as orthogonal to Punica's optimizations. However, no evaluation is provided.

*   **Gap:** The paper states, "Punica is orthogonal to these optimizations," but does not demonstrate this empirically.
*   **Potential Issue:** Speculative decoding uses a draft model to propose tokens, which are then verified by the main model. If the draft model also uses LoRA, how does `SGMV` handle the interaction? Does the verification step introduce additional heterogeneity that complicates batching?
*   **Future Work:** Empirical evaluation of Punica combined with speculative decoding would clarify whether the optimizations are truly composable.

**Open Question 2: Multi-Adapter Comparison**

The paper notes that **PetS** (**Zhou et al., 2022**) "does not enable multiple different models to run concurrently" (**Section 8**), so no direct comparison is made.

*   **Gap:** There is no quantitative comparison against other adapter-serving approaches beyond backbone-only systems.
*   **Implication:** It is unclear whether Punica's approach is superior to other multi-adapter serving strategies that may exist but were not evaluated.
*   **Future Work:** A comparison against systems that support concurrent adapter execution (if any exist) would strengthen the claims about Punica's uniqueness.

**Open Question 3: Energy Efficiency**

The evaluation focuses on throughput and latency but does not measure energy consumption.

*   **Gap:** No power or energy metrics are reported in **Section 7**.
*   **Implication:** While Punica improves GPU utilization, it is unclear whether this translates to proportional energy savings. Higher utilization could mean more efficient energy use per token, or it could mean GPUs run at higher power states for longer periods.
*   **Future Work:** Energy-per-token measurements would be valuable for cloud providers optimizing for cost and sustainability.

**Open Question 4: Fault Tolerance and Recovery**

The paper does not discuss fault tolerance mechanisms.

*   **Gap:** **Section 3** (System Architecture) and **Section 5** (Punica in Detail) do not address GPU failures, request recovery, or scheduler failover.
*   **Implication:** In a production cluster, GPU failures are inevitable. How does Punica handle a GPU crash mid-generation? Are in-flight requests lost, or is there a recovery mechanism?
*   **Future Work:** Fault tolerance design and evaluation would be necessary for production deployment.

### 6.6 Summary of Limitations

The table below summarizes the key limitations and their implications:

| Limitation | Evidence | Severity | Mitigation |
| :--- | :--- | :--- | :--- |
| **Single backbone per GPU** | Section 3 | High | Partition cluster by backbone type |
| **8-12% single-tenant overhead** | Figure 11 | Medium | Use vLLM for single-tenant workloads |
| **No quantized backbone evaluation** | Section 7, 8 | Medium | Future evaluation needed |
| **Sequence length sensitivity** | Figure 10 | Medium | Benefits decrease for long contexts |
| **No long-term stability data** | Figure 13 (1 hour) | Low | Monitor fragmentation in production |
| **No migration latency measurement** | Section 5.3 | Low | Recomputation generally efficient |
| **No fault tolerance discussion** | N/A | Medium | Requires production hardening |

**Final Assessment:** Punica's limitations are largely **design trade-offs** rather than fundamental flaws. The system explicitly optimizes for multi-tenant LoRA serving with shared backbones, and it achieves this goal effectively. The limitations primarily define the **boundary conditions** where Punica is most appropriate:

*   **Best suited for:** Multi-tenant deployments with many LoRA adapters on the same backbone, interactive workloads with moderate sequence lengths, and cloud environments requiring elastic scaling.
*   **Less suited for:** Single-tenant deployments, cross-backbone serving, very long context windows, or environments where quantized backbones are required.

Understanding these boundaries is essential for practitioners deciding whether to adopt Punica for their specific use case.

## 7. Implications and Future Directions

Punica does more than optimize a specific kernel; it fundamentally alters the economic and architectural assumptions of Large Language Model (LLM) serving. By decoupling the base model from the adapter during execution, it enables a shift from static, isolated deployments to dynamic, shared infrastructure. This section explores how this work changes the field, what research questions it opens, where it applies practically, and how engineers should approach integrating these ideas.

### 7.1 Shifting the Serving Paradigm

The most profound implication of Punica is the decoupling of **model identity** from **batch execution**. Prior to this work, high-performance serving systems (like vLLM or Orca) operated under the constraint that a batch must be homogeneous—all requests must share the exact same weight matrix $W$. Punica demonstrates that this constraint is unnecessary for Low-Rank Adaptation (LoRA) workloads.

*   **Economic Impact:** The evaluation in **Section 7.2** shows a **12x throughput improvement** in multi-tenant scenarios compared to state-of-the-art baselines. In practical terms, this means a service provider can serve 12 times as many tenants on the same hardware, or reduce their GPU cluster size by 92% for the same load. This changes the unit economics of offering specialized LLM APIs.
*   **Resource Utilization:** By allowing heterogeneous batching, Punica eliminates the "straggler" problem where GPUs sit idle waiting for a specific model's requests to accumulate. **Figure 13** illustrates this: GPUs run at maximum batch size (32) when active, and idle GPUs stay idle, enabling clean resource release. This moves the industry closer to true **multi-tenancy** where isolation is logical (via adapters) rather than physical (via separate GPUs).
*   **Latency Transparency:** Crucially, this efficiency gain does not come at the cost of user experience. The Abstract and **Section 5.2** note that the system adds only **2ms latency per token**. For most interactive applications (where human reading speed is the bottleneck), this overhead is negligible, making the throughput gains "free" from the user's perspective.

### 7.2 Enabling Follow-up Research

Punica solves the serving problem for standard LoRA, but its design choices open several specific avenues for future research. The system establishes a baseline for **adapter-aware serving**, suggesting that future optimizations should focus on the interaction between adapters and hardware constraints.

*   **Quantized Backbones:** As noted in **Section 6**, Punica does not evaluate quantized backbones (e.g., INT8 or FP4). Since quantization reduces memory bandwidth pressure for the backbone $W$, the relative cost of the LoRA add-on ($AB$) might increase. Future work should investigate whether `SGMV` needs modification to handle quantized weights efficiently. The **Roofline Analysis** in **Figure 7** suggests that in the "Distinct" case, the kernel is I/O-bound; quantization could shift this balance, potentially allowing Tensor Core usage even for heterogeneous batches.
*   **Beyond LoRA:** LoRA is not the only Parameter-Efficient Fine-Tuning (PEFT) method. **Section 8** mentions Adapters, Prefix Tuning, and BitFit. Each has a different mathematical structure. For example, Prefix Tuning adds tokens to the sequence rather than modifying weights. Research is needed to determine if a generalized "Segmented Gather" kernel can support these other structures without sacrificing the batching benefits Punica achieved for LoRA.
*   **Speculative Decoding Integration:** **Section 8** states that Punica is orthogonal to Speculative Decoding (**Leviathan et al., 2023**). However, this claim is not empirically validated. Speculative decoding involves a draft model verifying tokens with the main model. If the draft model also uses LoRA, the verification step becomes complex. Future work should measure whether Punica's heterogeneous batching can accelerate the verification phase of speculative decoding, where multiple candidate tokens must be evaluated simultaneously.
*   **Advanced Scheduling:** Punica's scheduler uses a simple "busy stays busy" heuristic (**Section 5.1**). It does not predict future request arrivals or optimize for model affinity beyond simple consolidation. Machine learning-based schedulers could potentially predict which LoRA adapters will be popular next and pre-load them, further reducing the already small **2ms** loading latency.

### 7.3 Practical Applications and Downstream Use Cases

Punica is not a general-purpose inference engine; it is specialized for a specific deployment pattern. Understanding this specialization is key to identifying where it delivers value.

*   **Multi-Tenant SaaS Platforms:** The primary use case is a platform serving hundreds of customers, each with their own fine-tuned model (e.g., a legal AI service where each law firm has a model trained on their own case history). Punica allows these firms to share the same Llama2-7B backbone while maintaining data isolation via unique LoRA adapters. The **1% storage overhead** per model (**Section 2.2**) makes storing thousands of adapters feasible on a single server's disk.
*   **Enterprise Internal Tools:** Large organizations often have multiple departments (HR, Engineering, Support) requiring specialized models. Instead of provisioning separate GPU clusters for each department, Punica allows a central IT team to host all adapters on a shared cluster, dynamically scaling resources based on departmental demand as shown in **Figure 13**.
*   **A/B Testing for Model Updates:** When updating a LoRA adapter (e.g., retraining with new data), providers can route a fraction of traffic to the new adapter version while keeping the rest on the old version. Because Punica handles heterogeneous batching efficiently, this split traffic does not degrade performance for the majority users, enabling safer, gradual rollouts.

### 7.4 Reproduction and Integration Guidance

For engineers considering adopting Punica or similar architectures, the following guidance clarifies when this approach is appropriate and what integration challenges to expect.

*   **When to Prefer Punica:**
    *   **High Adapter Count:** Use this architecture when serving **$>10$ distinct LoRA adapters** on the same backbone. Below this threshold, the complexity of `SGMV` may not outweigh the benefits of simpler systems like vLLM.
    *   **Shared Backbone:** Ensure all adapters derive from the **exact same pre-trained weights** (e.g., all `Llama-2-7B`). As emphasized in **Section 6**, Punica cannot mix backbones (e.g., 7B and 13B) on the same GPU.
    *   **Interactive Workloads:** The system is optimized for the **decode stage** (**Section 1, Guideline G3**). If your workload is primarily prefill-heavy (e.g., large document summarization with short outputs), the benefits may be reduced.

*   **When to Avoid:**
    *   **Single Tenant:** If serving a single model, standard systems like vLLM achieve **8-12% higher throughput** (**Section 7.2**) because they avoid the LoRA add-on overhead entirely.
    *   **Long Context:** Performance gains diminish for sequence lengths **$>2048$** (**Figure 10**) because self-attention dominates computation. For very long contexts, memory bandwidth optimizations for KvCache (like vLLM's PagedAttention) become more critical than LoRA batching.
    *   **Quantized Requirements:** If your deployment requires INT8/FP8 quantization to fit the model in memory, Punica's current implementation does not support this natively (**Section 6**).

*   **Integration Complexity:**
    *   **Kernel Dependencies:** Punica is not a pure Python library. It requires compiling custom CUDA kernels (`SGMV`) via PyBind11 (**Section 6**). Integrating this into an existing pipeline requires C++/CUDA engineering expertise.
    *   **Scheduler Replacement:** You cannot simply drop Punica's kernel into an existing scheduler. The scheduler must be aware of the `SGMV` capabilities to route heterogeneous requests correctly. The paper provides a Rust-based scheduler (**Section 6**), but integrating this with existing Kubernetes or cloud orchestration tools requires additional engineering effort.
    *   **Migration Handling:** Be prepared to handle request migration. Punica migrates requests by **recomputing the prefill stage** (**Section 5.3**). While this simplifies memory management, applications sensitive to transient latency spikes during migration should account for this recomputation cost.

In summary, Punica represents a maturation of LLM serving infrastructure, moving from "serving models" to "serving capabilities." By treating LoRA adapters as lightweight, swappable components rather than monolithic models, it enables a future where specialized AI is as scalable and elastic as general-purpose cloud compute.
