## 1. Executive Summary

This survey addresses the increasing conceptual fragmentation in agent memory research by proposing a unified taxonomy organized through the lenses of **Forms** (token-level, parametric, latent), **Functions** (factual, experiential, working), and **Dynamics** (formation, evolution, retrieval). It consolidates representative benchmarks such as **LoCoMo** and **LongMemEval** alongside open-source frameworks like **MemGPT** and **Mem0** to support empirical research and practical development. This work matters because it establishes a conceptual foundation for rethinking memory as a first-class primitive in future agentic intelligence, guiding emerging frontiers such as automation-oriented memory design and the deep integration of reinforcement learning.

## 2. Context and Motivation

To understand why this survey is necessary, we must first examine the rapid evolution of Large Language Model (LLM) systems into **AI agents** and the specific bottleneck that memory addresses in this transition. This section details the conceptual fragmentation plaguing the field, the limitations of existing taxonomies, and the critical distinction between agent memory and related technologies like Retrieval-Augmented Generation (RAG).

### 2.1 The Shift from Static Models to Adaptive Agents
Historically, LLMs functioned as **static conditional generators**. Once trained, their knowledge was fixed within their parameters, meaning they could not rapidly update their behavior based on new interactions without costly retraining. The field has since shifted toward **foundation model-based agents**, which are equipped with capabilities such as reasoning, planning, tool-use, and memory (Section 1).

Memory is the specific mechanism that enables this transformation. It allows agents to:
*   **Adapt Continually:** Unlike model parameters, memory can be updated instantly based on environmental feedback (Section 1).
*   **Reason over Long Horizons:** Memory underpins the ability to maintain context across extended interaction sequences, which is essential for tasks like deep research or software engineering (Abstract).
*   **Interact with Complex Environments:** Effective interaction requires retaining state information about users, tools, and environmental constraints (Abstract).

Without memory, agents remain "forgetful," limiting their utility in applications requiring persistence, such as personalized chatbots, recommender systems, social simulations, and financial investigations (Section 1). Consequently, memory is not merely an auxiliary feature but a **cornerstone capability** for achieving Artificial General Intelligence (AGI) aspirations related to continual evolution (Section 1).

### 2.2 The Problem of Conceptual Fragmentation
Despite the growing significance of memory, the research landscape has become **increasingly fragmented** (Abstract). This fragmentation manifests in three critical ways that hinder progress:

1.  **Loose Terminology:** There is a proliferation of loosely defined terminologies (e.g., declarative, episodic, semantic, parametric memory) that obscure conceptual clarity (Section 1). Researchers often find that papers claiming to study "agent memory" differ drastically in their motivations, implementations, and assumptions (Section 1).
2.  **Insufficient Traditional Taxonomies:** Conventional classifications, such as the binary split between **long-term** and **short-term memory**, have proven insufficient to capture the diversity and dynamics of contemporary systems (Abstract). These traditional labels fail to account for modern architectural variations like latent memory or parametric internalization.
3.  **Conflation with Related Concepts:** The community often conflates **agent memory** with distinct but related constructs, specifically:
    *   **LLM Memory:** Mechanisms internal to the model architecture (e.g., KV cache management) rather than agentic decision loops (Section 2.3.1).
    *   **Retrieval-Augmented Generation (RAG):** Systems that access static external knowledge rather than evolving internal memory states (Section 2.3.2).
    *   **Context Engineering:** Methodologies focused on optimizing the context window as a constrained resource rather than maintaining a persistent cognitive state (Section 2.3.3).

This ambiguity makes it difficult to compare systems, reproduce results, or build upon prior work effectively.

### 2.3 Limitations of Prior Surveys
Several recent surveys have attempted to overview agent memory (e.g., Zhang et al., 2025s; Wu et al., 2025g). However, this paper identifies two specific gaps in those works (Section 1, "Agent Memory Needs A New Taxonomy"):

*   **Temporal Obsolescence:** Existing taxonomies were developed prior to rapid methodological advances occurring around 2025. Consequently, they do not fully reflect the current breadth of the research landscape (Section 1).
*   **Underrepresentation of Emerging Directions:** Specific modern techniques remain underrepresented in earlier classification schemes. Notable examples include:
    *   Memory frameworks that distill **reusable tools** from past experiences (Section 1).
    *   **Memory-augmented test-time scaling** methods (Section 1).

Because these emerging directions fundamentally change how memory is formed and utilized, a new survey is required to reconcile existing definitions with these new capabilities.

### 2.4 Positioning of This Survey
This paper positions itself as a **conceptual foundation** for rethinking memory as a first-class primitive in agentic design (Abstract). It addresses the fragmentation problem by proposing a unified taxonomy organized through three lenses (Abstract, Section 1):

1.  **Forms:** What carries the memory? (e.g., token-level, parametric, latent).
2.  **Functions:** Why is the memory needed? (e.g., factual, experiential, working).
3.  **Dynamics:** How does the memory operate and evolve? (e.g., formation, evolution, retrieval).

By delineating the scope of agent memory and distinguishing it from related concepts (Section 2.3), this survey aims to provide an up-to-date landscape that supports both empirical research and practical development. It moves beyond consolidation to articulate forward-looking perspectives on frontiers such as automation-oriented memory design, reinforcement learning integration, and trustworthiness (Abstract, Section 7). This positioning ensures the work serves not only as a reference for existing literature but as a guide for future agentic intelligence design.

## 3. Technical Approach

This survey paper proposes a unified conceptual framework for understanding agent memory systems through three interconnected dimensions: **Forms** (what carries memory), **Functions** (why memory is needed), and **Dynamics** (how memory operates and evolves). Rather than presenting a single algorithm or system, this work establishes a technical taxonomy that organizes over 200 representative methods into a coherent structure, enabling researchers to compare approaches, identify gaps, and design new memory architectures systematically.

### 3.1 Reader orientation (approachable technical breakdown)

This survey builds a conceptual framework that classifies all agent memory systems by what form they take, what function they serve, and how they change over time. It solves the problem of conceptual fragmentation in the field by providing a unified vocabulary and structure that lets researchers compare fundamentally different memory mechanisms on equal footing.

### 3.2 Big-picture architecture (diagram in words)

The framework consists of three primary analytical dimensions that intersect to characterize any memory system. **Forms** describes the representational substrate (token-level, parametric, or latent memory). **Functions** describes the cognitive role (factual, experiential, or working memory). **Dynamics** describes the lifecycle operations (formation, evolution, and retrieval). Information flows through the framework as follows: first, a memory system is characterized by its form (Section 3); second, its functional purpose is identified (Section 4); third, its operational dynamics are analyzed (Section 5). These three dimensions are not independent—each form enables certain functions, and each function requires specific dynamic operations. The framework also includes supporting infrastructure: benchmarks and datasets for evaluation (Section 6.1) and open-source frameworks for implementation (Section 6.2).

### 3.4 Detailed, sentence-based technical breakdown

This survey establishes a technical framework for agent memory that operates at the conceptual and architectural level rather than presenting a single implementable algorithm. The core idea is that any agent memory system can be fully characterized by specifying its position along three dimensions: the **Form** of representation, the **Function** it serves, and the **Dynamics** of its operation. This section provides a complete technical breakdown of each dimension, the formal relationships between them, and the design principles that emerge from this taxonomy.

#### 3.4.1 Formal Foundation: Agent-Memory Coupling

Before analyzing memory systems themselves, we must establish the formal context in which they operate. The paper defines an LLM-based agent system through several key mathematical constructs that determine how memory integrates with decision-making.

**Agent and Environment Formalization.** The system consists of an index set of agents $\mathcal{I} = \{1, \ldots, N\}$, where $N = 1$ corresponds to single-agent settings and $N > 1$ represents multi-agent configurations (Section 2.1). The environment is characterized by a state space $\mathcal{S}$, and at each time step $t$, the environment evolves according to a controlled stochastic transition model:

$$s_{t+1} \sim \Psi(s_{t+1} | s_t, a_t)$$

where $a_t$ denotes the action executed at time $t$. Each agent $i \in \mathcal{I}$ receives an observation defined as:

$$o_t^i = \mathcal{O}_i(s_t, h_t^i, \mathcal{Q})$$

where $h_t^i$ denotes the portion of interaction history visible to agent $i$, and $\mathcal{Q}$ denotes the task specification (e.g., user instruction or goal description). The action space is heterogeneous, including natural-language generation, tool invocation, planning actions, environment-control actions, and communication actions (Section 2.1).

**Memory System Integration.** The agent policy is formally written as:

$$a_t = \pi_i(o_t^i, m_t^i, \mathcal{Q})$$

where $m_t^i$ is a memory-derived signal that augments the observation. The memory system itself is represented as an evolving memory state:

$$M_t \in \mathcal{M}$$

where $\mathcal{M}$ denotes the space of admissible memory configurations. No specific internal structure is imposed on $M_t$—it may take the form of a text buffer, key-value store, vector database, graph structure, or any hybrid representation (Section 2.2).

**Memory Lifecycle Operators.** The dynamics of the memory system are characterized by three conceptual operators that form the foundation of the Dynamics dimension:

1. **Memory Formation:** At time step $t$, the agent produces informational artifacts $\phi_t$ (e.g., tool outputs, reasoning traces, environmental feedback). A formation operator selectively transforms these artifacts into memory candidates:

$$M_{t+1}^{\text{form}} = \mathcal{F}(M_t, \phi_t)$$

This extracts information with potential future utility rather than storing the entire interaction history verbatim (Section 2.2).

2. **Memory Evolution:** Formed memory candidates are integrated into the existing memory base through an evolution operator:

$$M_{t+1} = \mathcal{E}(M_{t+1}^{\text{form}})$$

This may consolidate redundant entries, resolve conflicts, discard low-utility information, or restructure memory for efficient retrieval (Section 2.2).

3. **Memory Retrieval:** When selecting an action, agent $i$ retrieves a context-dependent memory signal:

$$m_t^i = \mathcal{R}(M_t, o_t^i, \mathcal{Q})$$

where $\mathcal{R}$ denotes a retrieval operator that constructs a task-aware query and returns relevant memory content formatted for direct consumption by the LLM policy (Section 2.2).

**Temporal Roles.** Short-term and long-term memory phenomena emerge not from discrete architectural modules but from the temporal patterns with which formation, evolution, and retrieval are engaged. For instance, some systems perform retrieval only once at task initialization:

$$m_t^i = \begin{cases} \mathcal{R}(M_0, o_0^i, \mathcal{Q}), & t = 0 \\ \perp, & t > 0 \end{cases}$$

where $\perp$ denotes null retrieval. Others may retrieve memory intermittently or continuously based on contextual triggers (Section 2.2).

This formal foundation establishes that memory is not a standalone component but an integrated subsystem that couples with the agent's decision process through read/write interactions. The three lifecycle operators ($\mathcal{F}$, $\mathcal{E}$, $\mathcal{R}$) need not be invoked at every time step—different memory effects arise from distinct temporal invocation patterns.

#### 3.4.2 Dimension 1: Forms — What Carries Memory?

The Forms dimension categorizes memory by where it resides and in what representational format it is stored. This is the most fundamental architectural choice, as the form determines what operations are possible, what information can be preserved, and how the memory integrates with the agent's policy. The paper identifies three major memory forms, each with distinct sub-types.

**Token-level Memory (Section 3.1).** Token-level memory stores information as persistent, discrete units that are externally accessible and inspectable. The term "token" here is a broad representational notion: beyond text tokens, it includes visual tokens, audio frames, or any discrete element that can be written, retrieved, reorganized, and revised outside model parameters. Because these units are explicit, token-level memory is typically transparent, easy to edit, and straightforward to interpret, making it a natural layer for retrieval, routing, conflict handling, and coordination with parametric and latent memory.

Token-level memory is further categorized by inter-unit structural organization, moving from no explicit topology to multi-layer topologies (Figure 3, Section 3.1):

1. **Flat Memory (1D):** No explicit inter-unit topology exists. Memories are accumulated as sequences or bags of units (e.g., snippets, trajectories, chunks). Representative methods include MemoryBank (Zhong et al., 2024) for dialogue history, MemGPT (Packer et al., 2023a) for virtual memory management, and Reflexion (Shinn et al., 2023b) for trajectory and feedback storage. Table 1 lists 47 flat memory methods across factual, experiential, and working memory types.

2. **Planar Memory (2D):** A structured but single-layer organization exists within one plane. Units are related by a graph, tree, table, or implicit connection structure, with no cross-layer relations. The structure is explicit but not layered. Representative methods include A-Mem (Xu et al., 2025c) for card-based connected memory, MemTree (Rezazadeh et al., 2025c) for dynamic hierarchical conversation trees, and M3-Agent (Long et al., 2025) for multimodal nodes in graph structures. Table 1 lists 18 planar memory methods.

3. **Hierarchical Memory (3D):** Structured across multiple layers with inter-layer links, forming a volumetric or stratified memory. This enables vertical abstraction and cross-layer reasoning between different data granularities. Representative methods include GraphRAG (Edge et al., 2025) for multi-level community graph indices, G-Memory (Zhang et al., 2025c) for query-centric three-layer graph structures, and HiAgent (Hu et al., 2025a) for goal graphs with recursive clusters. Table 1 lists 17 hierarchical memory methods.

The primary advantage of flat memory is simplicity and scalability: memory can be appended or pruned with minimal cost, and retrieval methods such as similarity search allow flexible access without requiring predefined structure. However, the lack of explicit relational organization means that coherence and relevance depend heavily on retrieval quality. Planar memory establishes links between nodes, enabling memories to leverage collective synergies and encode more comprehensive contextual knowledge, but faces high construction and search costs. Hierarchical memory allows different memories to interact and form multi-dimensional synergies, supporting complex multi-path queries, but the structure's complexity creates challenges for retrieval efficiency and optimal three-dimensional layout design (Section 3.1.1–3.1.3).

**Parametric Memory (Section 3.2).** Parametric memory stores information directly in the model's parameters, in contrast to token-level memory which stores information as visible and editable discrete units. This form embeds memory into learnable parameter spaces, allowing the model to internalize and recall information without referring to external storage. The paper distinguishes two primary forms based on where the memory is stored relative to the core model parameters (Table 2, Section 3.2):

1. **Internal Parametric Memory:** Memory is encoded within the original parameters of the model (e.g., weights, biases). These methods directly adjust the base model to incorporate new knowledge or behavior. The timing of memory injection can be the pre-training phase, continued pre-training phase, mid-training phase, or post-training phase. Representative methods include:
   - Pre-Train Phase: LMLM (Zhao et al., 2025b) and HierMemLM (Pouransari et al., 2025) store memory for knowledge retrieval during pre-training.
   - Mid-Train Phase: Agent-Founder (Su et al., 2025) and Early Experience (Zhang et al., 2025k) integrate agent experience during continued pre-training.
   - Post-Train Phase: Character-LM (Shao et al., 2023) and CharacterGLM (Zhou et al., 2024a) fine-tune the LLM into different characteristics; KnowledgeEditor (Cao et al., 2021) and MEND (Mitchell et al., 2022) modify internal parameters for knowledge editing.

2. **External Parametric Memory:** Memory is stored in additional or auxiliary parameter sets, such as adapters, LoRA modules, or lightweight proxy models. These methods introduce new parameters to carry memory without modifying the original model weights. Representative methods include:
   - Adapter-based Modules: MLP-Memory (Wei et al., 2025d) integrates RAG knowledge with Transformer decoders through MLP; K-Adapter (Wang et al., 2021) injects new knowledge by training task-specific adapter modules; WISE (Wang et al., 2024e) introduces a dual-parameter memory setup with a routing mechanism.
   - Auxiliary LM-based Modules: MAC (Tack et al., 2024) compresses information from a new document into compact modulation through an amortization network; Retroformer (Yao et al., 2024a) proposes a learning paradigm for memorizing experiences of successes or failures.

The advantages of internal parametric memory lie in their simple structure, which does not add extra inference overhead or deployment costs to the vanilla model. Their drawback is the difficulty in updating internal parameters: storing new memory requires retraining, which is costly and prone to forgetting old memory. External parametric memory provides a balance between adaptability and model stability—memory can be added, removed, or replaced without interfering with the base model's pre-trained representation space, but the effectiveness of memory injection depends on how well the external parameters can interface with internal parametric knowledge (Section 3.2.1–3.2.2).

**Latent Memory (Section 3.3).** Latent memory refers to memory that is carried implicitly in the model's internal representations (e.g., KV cache, activations, hidden states, latent embeddings), rather than being stored as explicit, human-readable tokens or dedicated parameter sets. Latent memory avoids exposing memory in plaintext and introduces practically less inference latency, while potentially offering better performance gains by preserving fine-grained contextual signals within the model's own representational space.

The paper organizes latent memory by the origin of the latent state, which means how the latent state is formed and introduced into the agent (Figure 4, Table 3, Section 3.3):

1. **Generate:** Latent memory is produced by an independent model or a module, and then supplied to the agent as reusable internal representations. The model or an auxiliary encoder creates compact continuous states that may appear as special tokens in the sequence or as standalone vectors. Representative methods include:
   - Single Modal: Gist (Mu et al., 2023) trains a language model to produce a set of gist tokens after processing a long prompt; AutoCompressor (Chevalier et al., 2023) encodes entire long documents into a small number of summary vectors; MemoryLLM (Wang et al., 2024j) embeds a set of dedicated memory tokens within the model's latent space; MemGen (Zhang et al., 2025d) dynamically generates latent memory during decoding using two LoRA adapters.
   - Multi-Modal: CoMem (Wu et al., 2025d) uses a VLM to compress multimodal knowledge into a set of embeddings; Time-VLM (Zhong et al., 2025) divides video or interaction streams into patches and generates a latent embedding for each patch; XMem (Cheng and Schwing, 2022) encodes each frame into key-value latent embeddings for long-video object segmentation.

2. **Reuse:** Latent memory is directly carried over from prior computation, most prominently KV-cache reuse (within or across turns), as well as recurrent or stateful controllers that propagate hidden states. These approaches do not transform (modify, compress) the stored KV pairs and instead treat the raw activations from forward passes as reusable memory entries. Representative methods include Memorizing Transformers (Wu et al., 2022) which explicitly store past KV pairs and retrieve them via K-nearest-neighbor search; FOT (Tworkowski et al., 2023) which introduces memory-attention layers that perform KNN-based retrieval over additional KV memories; LONGMEM (Wang et al., 2023b) which employs a lightweight residual SideNet that treats historical KV embeddings as a persistent memory store.

3. **Transform:** Existing latent state is transformed into new representations (e.g., distillation, pooling, or compression), so the agent can retain essentials while reducing latency and context footprint. These approaches treat KV caches and hidden activations as malleable memory units, reshaping them through selection, aggregation, or structural transformation. Representative methods include:
   - Scissorhands (Liu et al., 2023b) prunes tokens based on attention scores when cache capacity is exceeded.
   - SnapKV (Li et al., 2024b) aggregates high-importance prefix KV representations via a head-wise voting mechanism.
   - PyramidKV (Cai et al., 2024) reallocates KV budgets across layers.
   - H2O (Zhang et al., 2023) adopts a simpler eviction strategy, retaining only the most recent tokens along with special H2 tokens.

The generation process itself may introduce information loss or bias, and the states can drift or accumulate errors over multiple read-write cycles. Reuse-type methods preserve the full fidelity of the model's internal activations but raw KV caches grow rapidly with context length. Transformation introduces the risk of information loss, and the compressed states can become harder to interpret or verify compared with directly reused KV caches (Section 3.3.1–3.3.3).

**Adaptation Guidelines (Section 3.4).** The choice of memory type in an agent system reflects how designers expect the agent to behave in a given task. Figure 5 summarizes the features and suitable applications for each memory form:

- **Token-level Memory** is symbolic, addressable, and transparent, with swift adding/deleting/updating capabilities. It is suitable for multi-turn chatbots, personalized agents, recommender systems, and high-stake domains (law, finance, medical) where explicit reasoning, controllability, and accountability are essential.
- **Parametric Memory** is implicit, abstract, and generalizable, with slower memory update but typically better performance gain. It is suitable for role-playing, reasoning-intensive tasks, and tasks that require fundamentally new capabilities where conceptual understanding and broad pattern induction are needed.
- **Latent Memory** is implicit and human-unreadable, offering a trade-off between efficiency and flexibility with convenient modality fusion. It is suitable for multimodal memory, on-device or edge deployment, and low-resource or small-data settings where performance and scalability are prioritized over interpretability.

#### 3.4.3 Dimension 2: Functions — Why Agents Need Memory?

The Functions dimension categorizes memory by the cognitive role it serves in the agent system. This dimension addresses the fundamental purpose of memory, prioritizing the question of why it is essential over how it is implemented. The paper moves beyond the coarse long-term versus short-term dichotomy prevalent in prior surveys and proposes a more fine-grained taxonomy that distinguishes three primary functional pillars (Figure 6, Section 4).

**Factual Memory (Section 4.1).** Factual memory refers to the capacity of an agent to store and retrieve explicit, declarative facts about past events, user-specific information, and the state of the external environment. This information encompasses dialogue history, user preferences, and relevant properties of the external world. Functionally, this architecture ensures that the agent exhibits three fundamental properties during interaction: consistency (stable behavior and self-presentation over time), coherence (robust context awareness), and adaptability (personalization based on stored user profiles and historical feedback).

The paper further organizes factual memory according to the primary entity it refers to (Table 4, Section 4.1):

1. **User Factual Memory (Section 4.1.1):** Denotes facts that sustain the consistency of interactions between humans and agents, including identities, stable preferences, task constraints, and historical commitments. This memory persists verifiable facts about a specific user across sessions and tasks. Its primary function is to prevent characteristic failure modes of stateless interaction, such as coreference drift, repeated elicitation, and contradictory responses. Engineering practice typically comprises selection and compression, structured organization, retrieval and reuse, and consistency governance. Representative methods include:
   - Dialogue Coherence: MemGPT (Packer et al., 2023b) uses token-level 1D structure for long-term dialogue; MemoryBank (Zhong et al., 2024) organizes dialogue history and event summaries by timestamp; mem0 (Chhikara et al., 2025) establishes standardized operations for memory maintenance.
   - Goal Consistency: RecurrentGPT (Zhou et al., 2023b) maintains explicit task representation over time; A-Mem (Xu et al., 2025c) organizes memories as an interconnected graph of linked notes.

2. **Environment Factual Memory (Section 4.1.2):** Denotes facts that sustain consistency with respect to the external world, such as document states, resource availability, and the capabilities of other agents. This memory paradigm addresses incomplete factual recall and unverifiable provenance, minimizes contradictions and redundancy in multi-agent collaboration, and stabilizes long-horizon tasks in heterogeneous environments. Representative methods include:
   - Knowledge Persistence: HippoRAG (Gutierrez et al., 2024) utilizes knowledge graphs to facilitate evidence propagation; MemTree (Rezazadeh et al., 2025c) employs a dynamic hierarchical structure to optimize aggregation and targeted access.
   - Shared Access: MetaGPT (Hong et al., 2024) and GameGPT (Chen et al., 2023b) utilize shared message pools as central workspaces; G-Memory (Zhang et al., 2025e) employs hierarchical memory graphs as a unified coordination medium.

**Experiential Memory (Section 4.2).** Experiential memory encapsulates the mechanism by which agents encode historical trajectories, distilled strategies, and interaction outcomes into durable, retrievable representations. Unlike working memory, which manages transient context, experiential memory focuses on the long-term accumulation and transfer of knowledge across distinct episodes. Theoretically grounded in cognitive science, this paradigm parallels human nondeclarative memory, specifically the procedural and habit systems. Crucially, experiential memory serves as a foundation for continual learning and self-evolution, enabling agents to achieve a non-parametric path to adaptation and avoid the prohibitive costs of frequent parametric updates.

The paper classifies experiential memory based on the abstraction level of the stored information (Figure 7, Table 5, Section 4.2):

1. **Case-based Memory (Section 4.2.1):** Stores minimally processed records of historical episodes, prioritizing high informational fidelity to support direct replay and imitation. By retaining the original alignment between situations and outcomes, it serves as a repository of concrete, verifiable evidence that functions as in-context exemplars for evidence-driven learning. Representative methods include:
   - Trajectories: Memento (Zhou et al., 2025a) employs soft Q-learning to dynamically refine the probability of selecting high-utility past trajectories; JARVIS-1 (Wang et al., 2025q) stores survival experiences in Minecraft.
   - Solutions: ExpeL (Zhao et al., 2024) autonomously gathers experience through trial-and-error, storing successful trajectories as exemplars; MapCoder (Islam et al., 2024) keeps relevant example code as a playbook-like case.

2. **Strategy-based Memory (Section 4.2.2):** Distills transferable reasoning patterns, workflows, and high-level insights from past trajectories to guide planning across diverse scenarios. Acting as a cognitive scaffold, it decouples decision-making logic from specific contexts, thereby enhancing cross-task generalization and constraining the search space for complex reasoning. Based on the granularity and structural complexity of the retained knowledge, the paper categorizes strategy-based memory into three distinct types:
   - Insights: H2R (Ye et al., 2025b) explicitly decouples planning-level and execution-level memories; R2D2 (Huang et al., 2025c) integrates remembering, reflecting, and dynamic decision-making for web navigation.
   - Workflows: Agent Workflow Memory (AWM) (Wang et al., 2024m) induces reusable workflows on Mind2Web and WebArena; Agent KB (Tang et al., 2025d) establishes a unified knowledge base that treats workflows as transferable procedural knowledge.
   - Patterns: Buffer of Thoughts (Yang et al., 2024b) maintains a meta-buffer of thought templates; ReasoningBank (Ouyang et al., 2025) abstracts both successes and failures into reusable reasoning units.

3. **Skill-based Memory (Section 4.2.3):** Encapsulates executable procedural capacities, ranging from atomic code snippets to standardized API protocols, that operationalize abstract strategies into verifiable actions. This category serves as the agent's active execution substrate, enabling the modular expansion of capabilities and the efficient handling of tool-use environments. Skill memory spans a continuum from internal, fine-grained code to externalized, standardized interfaces. Representative methods include:
   - Code Snippets: Voyager (Wang et al., 2024b) exemplifies this pattern with an ever-growing skill library; Darwin Gödel Machine (Zhang et al., 2025i) safely rewrites its own code under empirical validation.
   - Functions and Scripts: CREATOR (Qian et al., 2023) empowers agents to autonomously create specialized tools; LEGOMem (Han et al., 2025a) distills execution trajectories into retrievable scripts.
   - APIs: Gorilla (Patil et al., 2024) and ToolLLM (Qin et al., 2024a) focus on fine-tuning models to correctly invoke tools.
   - MCPs (Model Context Protocol): Alita (Qiu et al., 2025c) provides an open standard that unifies how agents discover and use tools and data.

4. **Hybrid Memory (Section 4.2.4):** Advanced agent architectures increasingly adopt a hybrid design that integrates multiple forms of experiential memory to balance grounded evidence with generalizable logic. Representative methods include ExpeL (Zhao et al., 2024) which synergizes concrete trajectories with abstract textual insights; Agent KB (Tang et al., 2025d) which employs a hierarchical structure where high-level workflows guide planning and specific solution paths provide execution details; G-Memory (Zhang et al., 2025c) and Memp (Fang et al., 2025d) which implement dynamic transitions where repeated successful cases are gradually compiled into efficient skills.

**Working Memory (Section 4.3).** In cognitive science, working memory is defined as a capacity-limited, dynamically controlled mechanism that supports higher-order cognition by selecting, maintaining, and transforming task-relevant information in the moment. When transposed to LLMs, the standard context window functions primarily as a passive, read-only buffer. The paper defines working memory as the set of mechanisms for the active management and manipulation of context within a single episode. The objective is to transform the context window from a passive buffer into a controllable, updatable, and interference-resistant workspace.

The paper categorizes working memory mechanisms based on the interaction dynamics (Table 6, Section 4.3):

1. **Single-turn Working Memory (Section 4.3.1):** Focuses on input condensation and abstraction. In this setting, the system must process massive immediate inputs such as long documents or high-dimensional multimodal streams within a single forward pass. The goal is to dynamically filter and rewrite evidence to construct a bounded computational scratchpad, thereby maximizing the effective information payload per token. Mechanisms include:
   - Input Condensation: LLMLingua (Jiang et al., 2023) and LongLLMLingua (Jiang et al., 2024) estimate token perplexity to discard predictable or task-irrelevant content; Gist (Mu et al., 2023) and AutoCompressors (Chevalier et al., 2023) train models to compress prompts into valid summary tokens or distinct memory embeddings.
   - Observation Abstraction: Synapse (Zheng et al., 2024a) rewrites unstructured HTML DOM trees into task-relevant state summaries; VideoAgent (Wang et al., 2024g) converts video streams into temporal event descriptions.

2. **Multi-turn Working Memory (Section 4.3.2):** Addresses temporal state maintenance. In sequential interactions, the challenge is to prevent historical accumulation from overwhelming the attention mechanism. This involves maintaining task states, goals, and constraints through a continuous loop of reading, executing, and updating, ensuring that intermediate artifacts are folded and consolidated across turns. Mechanisms include:
   - State Consolidation: MemAgent (Yu et al., 2025a) and MemSearcher (Yuan et al., 2025a) employ recurrent mechanisms to update fixed-budget memory and discard redundancy; ReSum (Wu et al., 2025f) periodically distills history into reasoning states using reinforcement learning.
   - Hierarchical Folding: HiAgent (Hu et al., 2025a) uses subgoals as memory units, retaining only active action-observation pairs and writing back a summary after subgoal completion; Context-Folding (Sun et al., 2025b) and AgentFold (Ye et al., 2025a) make the folding operation a learnable policy.
   - Cognitive Planning: PRIME (Tran et al., 2025) integrates retrieval directly into the planning loop; SayPlan (Rana et al., 2023) employs 3D scene graphs as queryable environmental memory.

#### 3.4.4 Dimension 3: Dynamics — How Memory Operates and Evolves?

The Dynamics dimension investigates the paradigm shift from static storage to dynamic memory management and utilization. This dimension examines the complete memory lifecycle by decomposing it into three fundamental processes that drive the system's adaptability and self-evolution (Figure 8, Section 5).

**Memory Formation (Section 5.1).** Memory formation is defined as the process of encoding raw contexts (e.g., dialogues or images) into compact knowledge. The necessity for memory formation emerges from the scaling limitations inherent in processing lengthy, noisy, and highly redundant raw contexts. Based on the granularity of information compression and the logic of encoding, the paper categorizes the memory formation process into five distinct types (Table 7, Section 5.1):

1. **Semantic Summarization (Section 5.1.1):** Transforms lengthy raw data into compact summaries, filtering out redundancy while preserving global, high-level semantic information to reduce contextual overhead. The compression process can be implemented in two primary ways:
   - Incremental Semantic Summarization: Employs a temporal integration mechanism that continuously fuses newly observed information with the existing summary. MemGPT (Packer et al., 2023a) and Mem0 (Chhikara et al., 2025) directly merged new chunks with existing summaries; Mem1 (Zhou et al., 2025b) and MemAgent (Yu et al., 2025a) enhanced the LLM's summarization capability through reinforcement learning with PPO and GRPO.
   - Partitioned Semantic Summarization: Adopts a spatial decomposition mechanism, dividing information into distinct semantic partitions and generating separate summaries for each. MemoryBank (Zhong et al., 2024) summarizes long-term dialogues by treating each day or session as a basic unit; ReadAgent (Lee et al., 2024a) and LightMem (Fang et al., 2025b) introduce semantic or topical clustering before summarization.

2. **Knowledge Distillation (Section 5.1.2):** Extracts specific cognitive assets, ranging from factual details to experiential planning strategies. This process focuses on transforming raw interactions and documents into explicit, declarative knowledge regarding users and environmental states. Representative methods include:
   - Distilling Factual Memory: TiM (Liu et al., 2023a) and RMM (Tan et al., 2025c) employ abstraction mechanisms to convert dialogue turns into high-level thoughts; MemGuide (Du et al., 2025b) extracts user intent descriptions from dialogues.
   - Distilling Experiential Memory: AgentRR (Feng et al., 2025) and AWM (Wang et al., 2024m) summarize overall task plans from successful cases; ExpeL (Zhao et al., 2024) contrasts successful and failed experiences to uncover holistic planning insights; H2R (Ye et al., 2025b) introduces a two-tier reflection mechanism for step-wise execution insights.

3. **Structured Construction (Section 5.1.3):** Organizes amorphous source data into explicit topological representations, such as knowledge graphs or hierarchical trees, to enhance the explainability of memory and support multi-hop reasoning. Based on the operational granularity of how the underlying structure is derived, the paper categorizes existing methods into two paradigms:
   - Entity-Level Construction: Derives underlying topology by atomizing text into entities and relations. KGT (Sun et al., 2024) encodes user preferences and feedback as nodes and edges within a user-specific knowledge graph; GraphRAG (Edge et al., 2025) applies community detection algorithms to extract graph communities and generate community summaries iteratively; Zep (Rasmussen et al., 2025) formalizes a three-layer temporal graph architecture.
   - Chunk-Level Construction: Builds structure by organizing intact text segments or memory items. MemTree (Rezazadeh et al., 2025c) adopts a bottom-up approach where new text fragments retrieve the most similar nodes and are inserted as children; CAM (Li et al., 2025g) establishes edges between text blocks based on semantic relevance and narrative coherence.

4. **Latent Representation (Section 5.1.4):** Encodes raw experiences directly into machine-native formats (e.g., vector embeddings or KV states) within a continuous latent space. Unlike semantic compression and structured extraction, which summarize experiences before embedding them into vectors, latent encoding inherently stores experiences in latent space, thereby reducing information loss during summarization and text embedding. Representative methods include MEMORYLLM (Wang et al., 2024j) and M+ (Wang et al., 2025n) which represent memory as self-updatable latent embeddings; MemGen (Zhang et al., 2025d) which introduces a memory trigger to monitor the agent's reasoning state; CoMEM (Wu et al., 2025d) which compresses vision-language inputs into fixed-length tokens via a Q-Former.

5. **Parametric Internalization (Section 5.1.5):** Consolidates external memories directly into the model's weight space through parameter updates, effectively transforming retrievable information into the agent's intrinsic competence and instincts. This paradigm fundamentally enhances the model's intrinsic competence, eliminating the overhead of external storage and retrieval while seamlessly supporting continual updates. The paper separates this into knowledge internalization and capability internalization:
   - Knowledge Internalization: MEND (Mitchell et al., 2022) introduced an auxiliary network that enables rapid, single-step edits by decomposing fine-tuning gradients; ROME (Meng et al., 2022) refined the editing process by using causal tracing to precisely locate the MLP layers; MEMIT (Meng et al., 2023) supported batch edits for thousands of facts simultaneously.
   - Capability Internalization: Achieved by learning from reasoning traces through supervised fine-tuning or preference-guided optimization methods such as DPO and GRPO.

**Memory Evolution (Section 5.2).** Memory evolution represents the dynamic evolution of the memory system, focusing on integrating newly formed memories with the existing memory base. Through mechanisms such as the consolidation of correlated entries, conflict resolution, and adaptive pruning, the system ensures that the memory remains generalizable, coherent, and efficient in an ever-changing environment. The paper categorizes memory evolution into three mechanisms (Figure 9, Section 5.2):

1. **Memory Consolidation (Section 5.2.1):** Merges new and existing memories and performs reflective integration, forming more generalized insights. This ensures that learning is cumulative rather than isolated. Prior work spans a spectrum of consolidation strategies:
   - Local Consolidation: RMM (Tan et al., 2025c) retrieves top-K most similar candidates for each new topic memory, and an LLM decides whether merging is appropriate; VLN (Song et al., 2025b) triggers a pooling mechanism when capacity is saturated.
   - Cluster-level Fusion: PREMem (Kim et al., 2025b) aligns new memory clusters with similar existing ones and applies fusion modes such as generalization and refinement; TiM (Liu et al., 2023a) periodically invokes an LLM to examine memories that share the same hashing bucket and merges semantically redundant entries.
   - Global Integration: MOOM (Chen et al., 2025e) constructs stable role profiles by integrating temporary role snapshots with historical traces; Matrix (Liu et al., 2024) performs iterative optimization to combine execution trajectories and reflective insights with global memory.

2. **Memory Updating (Section 5.2.2):** Resolves conflicts between new and existing memories, correcting and supplementing the repository to maintain accuracy and relevance. It allows the agent to adapt to changes in the environment or task requirements. Depending on where the memory resides, updates fall into two categories:
   - External Memory Update: Entries in vector databases or knowledge graphs are revised whenever contradictions or new facts emerge. MemGPT (Packer et al., 2023a), D-SMART (Lei et al., 2025), and Mem0g (Chhikara et al., 2025) follow a straightforward pipeline in which the LLM detects conflicts and then invokes replace or delete operations; Zep (Rasmussen et al., 2025) introduced temporal annotations, marking conflicting facts with invalid timestamps rather than deleting them; Mem-α (Wang et al., 2025p) formulated memory updating as a policy-learning problem using reinforcement learning.
   - Model Editing: Performs direct modifications within the model's parameter space. ROME (Tan et al., 2025b) identifies the parameter region encoding specific knowledge via gradient tracing and performs targeted weight updates; MEMORYLLM (Xu et al., 2025c) embeds a memory pool within Transformer layers, periodically replacing memory tokens to integrate new knowledge.

3. **Memory Forgetting (Section 5.2.3):** Removes outdated or redundant information, freeing capacity and improving efficiency. This prevents performance degradation due to knowledge overload and ensures that the memory repository remains focused on actionable and current knowledge. Forgetting mechanisms can be categorized into three types:
   - Time-based Forgetting: MemGPT (Packer et al., 2023a) evicts the earliest messages upon context overflow; MAICC (Jiang et al., 2025c) implements soft forgetting by gradually decaying the weights of memories over time.
   - Frequency-based Forgetting: XMem (Cheng and Schwing, 2022) employs an LFU policy to remove low-frequency entries; KARMA (Wang et al., 2025r) uses counting Bloom filters to track access frequency; MemOS (Li et al., 2025l) applies an LRU strategy.
   - Importance-driven Forgetting: VLN (Song et al., 2025b) pools semantically redundant memories via similarity clustering; Livia (Xi and Wang, 2025) incorporates emotional salience and contextual relevance to model emotion-driven selective forgetting; TiM (Liu et al., 2023a) and MemTool (Lumer et al., 2025) leverage LLMs to assess memory importance and explicitly prune less important memories.

**Memory Retrieval (Section 5.3).** Memory retrieval is defined as the process of retrieving relevant and concise knowledge fragments from a certain memory repository to support current reasoning tasks at the right moment. The key challenge lies in efficiently and accurately locating the required knowledge fragments within a large-scale memory store. Based on the execution order of retrieval, this process can be decomposed into four aspects (Figure 10, Section 5.3):

1. **Retrieval Timing and Intent (Section 5.3.1):** Determines the specific moments and objectives for memory retrieval, shifting from passive, instruction-driven triggers to autonomous, self-regulated decisions. The paper reviews literature from two complementary perspectives:
   - Automated Retrieval Timing: MemGPT (Packer et al., 2023a) and MemTool (Lumer et al., 2025) allow the LLM itself to invoke retrieval functions; ComoRAG (Wang et al., 2025f) and PRIME (Tran et al., 2025) first produce a fast response and then let the agent evaluate its adequacy before triggering deeper retrieval; MemGen (Zhang et al., 2025d) converts the explicit agent-level decision into a latent, trainable process.
   - Automated Retrieval Intent: MemOS (Li et al., 2025l) employs a MemScheduler that dynamically selects among parametric, activation, and plaintext memory; H-MEM (Sun and Zeng, 2025) introduces an index-based routing mechanism which performs coarse-to-fine retrieval.

2. **Query Construction (Section 5.3.2):** Bridges the semantic gap between the user's raw input and the stored memory index by decomposing or rewriting queries into effective retrieval signals. The paper identifies two paradigms:
   - Query Decomposition: Visconde (Pereira et al., 2023) and ChemAgent (Tang et al., 2025c) employ LLMs to decompose the original question into sub-problems; PRIME (Tran et al., 2025) and MA-RAG (Nguyen et al., 2025) introduce a Planner Agent that first formulates a global retrieval plan; Agent KB (Tang et al., 2025d) adopts a two-stage retrieval process in which a teacher model observes the student model's failures and generates fine-grained sub-queries.
   - Query Rewriting: HyDE (Gao et al., 2023b) instructs the LLM to generate a hypothetical document in a zero-shot manner; MemoRAG (Qian et al., 2025) incorporates global memory into hypothetical document generation; Rewrite-Retrieve-Read (Ma et al., 2023b) trains a small language model as a dedicated rewriter through reinforcement learning.

3. **Retrieval Strategies (Section 5.3.3):** Executes the search over the memory repository, employing paradigms ranging from sparse lexical matching to dense semantic embedding and structure-aware graph traversal. The paper systematically reviews various retrieval paradigms:
   - Lexical Retrieval: Relies on keyword matching with representative methods including TF-IDF and BM25. Such methods are often employed in precision-oriented retrieval scenarios but struggle to capture semantic variations.
   - Semantic Retrieval: Encodes queries and memory entries into a shared embedding space and matches them based on semantic similarity. Representative approaches utilize semantic encoders including Sentence-BERT and CLIP. This is the default choice in most agentic memory frameworks.
   - Graph Retrieval: Leverages not only semantic signals but also the explicit topological structure of graphs. AriGraph (Anokhin et al., 2024), EMG-RAG (Wang et al., 2024l), Mem0g (Chhikara et al., 2025), and SGMem (Wu et al., 2025h) first identify the most relevant nodes or triples and then expand to their semantically related K-hop neighbors; HippoRAG (Gutierrez et al., 2024) performs personalized PageRank seeded on the retrieved nodes.
   - Generative Retrieval: Replaces lexical or semantic retrieval with a model that directly generates the identifiers of relevant documents. This paradigm often outperforms traditional retrieval methods in small-scale settings but requires additional training to internalize the semantics of all candidate documents.
   - Hybrid Retrieval: Integrates the strengths of multiple retrieval paradigms. Agent KB (Tang et al., 2025d) and MIRIX (Wang and Chen, 2025) combine lexical and semantic retrieval; Generative Agents (Kaiya et al., 2023) illustrate a multi-factor approach through a scoring scheme that accumulates recency, importance, and relevance.

4. **Post-Retrieval Processing (Section 5.3.4):** Refines the retrieved raw fragments through re-ranking, filtering, and aggregation, ensuring that the final context provided to the model is concise and coherent. Two components are central:
   - Re-ranking and Filtering: Semantic Anchoring (Chatterjee and Agarwal, 2025) integrates vector similarity with entity- and discourse-level alignment; learn-to-memorize (Zhang et al., 2025u) formulates score aggregation as a reinforcement-learning problem; Memory-R1 (Yan et al., 2025c) and Westhäußer et al. (2025) introduce LLM-based evaluators that filter retrieved content.
   - Aggregation and Compression: ComoRAG (Wang et al., 2025f) illustrates this through its Integration Agent which identifies historical signals semantically aligned with the query; MA-RAG (Nguyen et al., 2025) performs fine-grained content selection over retrieved documents; G-Memory (Zhang et al., 2025c) consolidates retrieved high-level insights and sparsified trajectories for multi-agent personalization.

To support empirical research and practical development, the paper compiles a comprehensive summary of representative benchmarks and open-source memory frameworks (Section 6).

**Benchmarks and Datasets (Section 6.1).** The paper classifies benchmarks into two broad categories (Table 8, Section 6.1.1–6.1.2):

1. **Benchmarks for Memory / Lifelong / Self-Evolving Agents:** These focus primarily on how well an agent can construct, maintain, and exploit an explicit memory of past interactions or world facts. Representative examples include:

- MIRIX (Wang and Chen, 2025): Structured memory with multimodal support, evaluated on LoCoMo and MemoryAgentBench.

#### 3.4.6 Design Principles and Trade-offs

The taxonomy reveals several critical design principles that emerge from analyzing the interplay between forms, functions, and dynamics.

**Form-Function Alignment.** The choice of memory form should align with the intended function. Token-level memory excels in factual memory scenarios where explicit reasoning, controllability, and accountability are essential (Section 3.4). Parametric memory is better aligned with tasks demanding structural insight, robust abstraction, and deeply ingrained behavioral or stylistic patterns. Latent memory prioritizes performance and scalability over interpretability, achieving high knowledge density and compression ideal for constrained or highly dynamic environments.

**Dynamic Adaptation.** Memory systems must support dynamic adaptation through the three lifecycle operators. The paper emphasizes that short-term and long-term memory phenomena emerge not from discrete architectural modules but from the temporal patterns with which formation, evolution, and retrieval are engaged (Section 2.2). This insight challenges traditional binary classifications and suggests that memory systems should be designed with flexible invocation patterns rather than rigid architectural separations.

**Scalability vs. Interpretability Trade-off.** Token-level memory offers high interpretability but may face scalability challenges as memory grows. Parametric memory offers better performance gains but suffers from catastrophic forgetting and high update costs. Latent memory provides a balance but introduces opaqueness that makes debugging and verification difficult. The adaptation guidelines in Section 3.4 help designers navigate these trade-offs based on application requirements.

**Automation Trajectory.** The paper identifies a clear trajectory toward automated memory management. Figure 11 (Section 7.3) illustrates the evolution from RL-free memory systems based on heuristic or prompt-driven pipelines, to partially RL-involved designs where reinforcement learning governs selected memory operations, and finally to fully RL-driven memory systems in which memory architectures and control policies are learned end-to-end. This evolution reflects a broader paradigm shift from manually engineered memory pipelines toward model-native, self-optimizing memory management.

**Trustworthiness Requirements.** As memory systems become more deeply embedded into LLM-based agents, trustworthiness becomes paramount. The paper argues that trustworthy memory must be built around three interlinked pillars: privacy preservation (granular permissioned memory, user-governed retention policies, encrypted or on-device storage), explainability (traceable access paths, self-rationalizing retrievals, visualizations of memory attention), and hallucination robustness (conflict detection, multi-document reasoning, uncertainty-aware generation) (Section 7.7).

This comprehensive technical framework provides researchers and practitioners with a structured lens through which to understand current developments in agent memory, evaluate existing systems, and design new memory architectures that are aligned with specific agentic objectives.

## 4. Key Insights and Innovations

This survey does not propose a single new algorithm but instead offers a conceptual restructuring of the field. The following insights distill the most significant departures from prior work, explaining why these shifts matter for designing future agents. We distinguish between **fundamental innovations** (changes to how we define or architect memory) and **incremental improvements** (optimizations within existing paradigms).

### 4.1 Memory Roles Emerge from Temporal Dynamics, Not Architectural Modules

**The Insight:**
Traditional systems often hardcode "short-term memory" (STM) and "long-term memory" (LTM) as separate physical buffers (e.g., context window vs. vector database). This survey argues that these roles are not architectural constants but **emergent properties of operator invocation frequency** (Section 2.2).

**Why It Differs from Prior Work:**
Previous taxonomies typically classify memory by storage location or duration (e.g., "context window" vs. "external database"). This paper formalizes memory through three lifecycle operators: Formation ($\mathcal{F}$), Evolution ($\mathcal{E}$), and Retrieval ($\mathcal{R}$). The distinction between STM and LTM arises from *when* these operators are called:
*   **Short-Term Effect:** Emerges when retrieval is continuous or formation is verbatim logging (e.g., $m_t^i = \mathcal{R}(M_t, \dots)$ at every step).
*   **Long-Term Effect:** Emerges when retrieval is intermittent and formation involves abstraction (e.g., updating $M_t$ only at task boundaries) (Section 2.2).

**Significance:**
This is a **fundamental theoretical advance**. It liberates designers from rigid architectural constraints. Instead of building separate STM and LTM modules, engineers can design a single memory state $M_t$ and tune the *policy* governing $\mathcal{F}, \mathcal{E}, \mathcal{R}$. For example, a system might behave like STM during a complex reasoning step (high-frequency retrieval) and like LTM across user sessions (low-frequency consolidation). This flexibility is crucial for **long-horizon agents** that must adapt their memory strategy dynamically rather than relying on static buffers (Section 5).

### 4.2 The Forms–Functions–Dynamics Taxonomy Resolves Conceptual Fragmentation

**The Insight:**
The field suffers from fragmented terminology where "memory" might refer to KV caches in one paper and user profiles in another. This survey introduces a unified **3D taxonomy** (Forms, Functions, Dynamics) that maps over 200 representative methods into a coherent landscape (Section 1, Section 3).

**Why It Differs from Prior Work:**
Existing surveys (e.g., Zhang et al., 2025s; Wu et al., 2025g) often rely on binary splits (long/short-term) or focus narrowly on specific implementations (e.g., RAG vs. Fine-tuning). This taxonomy decouples **what carries memory** (Forms: Token-level, Parametric, Latent) from **why it is needed** (Functions: Factual, Experiential, Working) and **how it changes** (Dynamics) (Section 3, Section 4).
*   **Example:** A method like `MemGPT` is not just "long-term memory"; it is specifically **Token-level Form** (virtual memory), serving **Factual Function** (dialogue coherence), via **Semantic Summarization Dynamics** (Section 3.1.1, Table 1).

**Significance:**
This is a **fundamental conceptual innovation**. It allows for precise comparison across disparate systems. For instance, researchers can now identify that `Latent Memory` (Section 3.3) might be better suited for **Working Memory Functions** (Section 4.3) due to efficiency, while `Token-level Memory` is superior for **Factual Memory** requiring auditability (Section 3.4). By providing a common vocabulary, this framework reduces ambiguity in experimental design and helps practitioners select the right memory substrate for their specific agentic objective (Figure 5).

### 4.3 The Paradigm Shift from Retrieval-Centric to Generative Memory

**The Insight:**
Historically, agent memory has been **retrieval-centric**: finding the most relevant static fragment from a store. The survey identifies a critical shift toward **generative memory**, where agents actively synthesize new memory representations on demand (Section 7.1).

**Why It Differs from Prior Work:**
Standard RAG and early memory systems treat memory as a static repository to be queried (Section 7.1.1). Generative approaches (e.g., `MemGen`, `ComoRAG`) treat memory as a **reconstructive process**. Instead of concatenating retrieved snippets, the agent generates a refined memory representation tailored to the current context (Section 7.1.1).
*   **Mechanism:** This often involves **Latent Memory** forms (Section 3.3) where the agent generates continuous vectors or tokens that summarize past interactions without explicit lookup (Section 7.1.1).

**Significance:**
This represents a **paradigm shift** in capability. Retrieval is limited by what was explicitly stored; generation allows for **abstraction and recomposition**. An agent can generate a memory state that combines multiple past experiences into a new strategy, rather than just recalling one past example. This is essential for **self-evolving agents** that need to generalize beyond their stored history (Section 7.1.2). However, this introduces risks of hallucination, requiring new verification mechanisms (Section 7.7).

### 4.4 Automation Trajectory: From Heuristic Rules to RL-Driven Management

**The Insight:**
Current memory systems largely rely on **hand-crafted heuristics** (e.g., "summarize every 10 turns" or "delete oldest entry"). The survey outlines a clear trajectory toward **fully RL-driven memory management**, where the agent learns *how* to manage memory as part of its policy (Section 7.3).

**Why It Differs from Prior Work:**
Most existing frameworks (e.g., `Mem0`, `MemGPT`) use fixed rules or prompt-based instructions for memory operations (Section 7.3.1). Emerging work (e.g., `Memory-R1`, `Mem-α`) uses Reinforcement Learning (RL) to optimize memory construction, updating, and retrieval policies (Section 7.3.1, Figure 11).
*   **Progression:** The field is moving from **RL-free** (heuristic thresholds) $\rightarrow$ **RL-assisted** (RL for reranking or summarization) $\rightarrow$ **Fully RL-driven** (end-to-end learning of memory architecture) (Figure 11).

**Significance:**
This is a **methodological evolution** with high potential impact. Hand-crafted rules do not scale to open-ended environments where interaction patterns are unpredictable. RL-driven management allows the agent to discover optimal memory strategies (e.g., when to forget, what to abstract) based on task performance rather than human intuition (Section 7.3.2). This reduces the engineering burden and enables **self-organizing memory architectures** that adapt to specific domains without manual redesign. However, it introduces complexity in training stability and reward design, which remains an open challenge (Section 7.3.2).

#### 5.1.2 Evaluation Metrics Across the Field

The survey reveals that evaluation metrics vary significantly across memory research, reflecting the functional diversity identified in the taxonomy (Section 4). Common metric categories include:

**Factual Memory Metrics (Section 4.1):**
- **Accuracy**: Correct retrieval of user preferences, identity facts, or environmental states
- **Consistency**: Stability of responses across multiple turns or sessions
- **Coherence**: Logical continuity in multi-turn dialogue
- **Personalization Quality**: Alignment with stored user profiles

**Experiential Memory Metrics (Section 4.2):**
- **Task Success Rate**: Completion rate on tasks requiring learned strategies or skills
- **Sample Efficiency**: Reduction in trials needed after memory accumulation
- **Transfer Performance**: Success on novel tasks using distilled experience
- **Skill Reuse Rate**: Frequency of successfully invoking stored procedures

**Working Memory Metrics (Section 4.3):**
- **Context Retention**: Ability to maintain task-relevant information within episode
- **Compression Ratio**: Reduction in token count while preserving essential information
- **Latency**: Inference time with memory operations
- **Goal Drift**: Deviation from original task objectives over long horizons

**Dynamics Metrics (Section 5):**
- **Formation Efficiency**: Quality of compressed/summarized memory units
- **Evolution Stability**: Resistance to catastrophic forgetting during updates
- **Retrieval Precision**: Relevance of retrieved memory to current context
- **Forgetting Appropriateness**: Removal of obsolete vs. retention of valuable information

### 5.2 Quantitative Trends from Surveyed Works

While this survey does not aggregate performance numbers across all methods (due to heterogeneous evaluation protocols), several quantitative patterns emerge from the surveyed literature:

#### 5.2.1 Memory Form Performance Trade-offs

Based on the adaptation guidelines in Section 3.4 and Figure 5, the survey identifies consistent trade-offs across memory forms:

**Token-level Memory:**
- **Strength**: High interpretability and editability
- **Typical Use**: Factual memory scenarios requiring auditability (Section 3.4)
- **Limitation**: Scalability challenges as memory grows; retrieval quality becomes critical (Section 3.1.1)
- **Representative Scale**: MemoryBank handles 194 samples for user memory updating (Table 8)

**Parametric Memory:**
- **Strength**: Better performance gain through internalization; zero-latency access (Section 3.2, Section 5.1.5)
- **Typical Use**: Role-playing, reasoning-intensive tasks (Section 3.4)
- **Limitation**: High update costs; prone to catastrophic forgetting (Section 3.2.1)
- **Representative Methods**: KnowledgeEditor, MEND, ROME, MEMIT support batch edits for thousands of facts (Section 5.1.5)

**Latent Memory:**
- **Strength**: High knowledge density; efficient inference-time retrieval (Section 3.3)
- **Typical Use**: Multimodal memory, on-device deployment (Section 3.4)
- **Limitation**: Opaqueness makes debugging and verification difficult (Section 3.3, Section 5.1.4)
- **Representative Methods**: Gist, AutoCompressor achieve high compression ratios but require additional training (Section 3.3.1)

#### 5.2.2 Function-Specific Evaluation Patterns

The survey reveals that different memory functions are evaluated with different emphases (Section 4):

**Factual Memory Evaluation:**
- Focus on **consistency** and **coherence** metrics (Section 4.1)
- Benchmarks like LoCoMo (300 samples) and LongMemEval (500 samples) target conversational memory
- Methods like MemGPT, MemoryBank, mem0 report improvements in dialogue coherence but specific accuracy numbers vary by benchmark

**Experiential Memory Evaluation:**
- Focus on **task success rate** and **sample efficiency** (Section 4.2)
- Benchmarks like StreamBench (9,702 samples) target continuous learning
- Methods like ExpeL, Voyager, AWM report reduced steps and improved success rates through experience reuse (Section 4.2.1–4.2.3)

**Working Memory Evaluation:**
- Focus on **compression ratio** and **context retention** (Section 4.3)
- Methods like LLMLingua, HiAgent, Context-Folding report token reduction while maintaining task performance (Section 4.3.1–4.3.2)
- RL-based methods (MemSearcher, ReSum) optimize summary-conditioned behavior for indefinite exploration (Section 4.3.2)

#### 5.2.3 Dynamics Operation Effectiveness

The survey identifies quantitative patterns in how memory dynamics affect performance (Section 5):

**Memory Formation:**
- **Semantic Summarization**: Incremental approaches (MemGPT, Mem0) vs. partitioned approaches (MemoryBank, ReadAgent) show different efficiency profiles (Section 5.1.1)
- **Knowledge Distillation**: Methods like ExpeL, H2R, AWM extract insights from trajectories, with RL-trained methods (Mem-α, Memory-R1) showing improved extraction policies (Section 5.1.2)
- **Structured Construction**: Graph-based methods (GraphRAG, Zep, A-Mem) support multi-hop reasoning but incur higher construction costs (Section 5.1.3)

**Memory Evolution:**
- **Consolidation**: Local (RMM, VLN), cluster-level (PREMem, TiM), and global (MOOM, Matrix) strategies show different generalization capabilities (Section 5.2.1)
- **Updating**: External memory updates (MemGPT, Zep, Mem-α) vs. model editing (ROME, MEMORYLLM, M+) represent different trade-offs between flexibility and stability (Section 5.2.2)
- **Forgetting**: Time-based (MemGPT, MAICC), frequency-based (XMem, KARMA, MemOS), and importance-driven (VLN, Livia, TiM) mechanisms address different obsolescence patterns (Section 5.2.3)

**Memory Retrieval:**
- **Timing**: Automated timing methods (MemGPT, ComoRAG, MemGen) reduce unnecessary retrieval overhead (Section 5.3.1)
- **Query Construction**: Decomposition (Visconde, PRIME, Agent KB) vs. rewriting (HyDE, MemoRAG, Rewrite-Retrieve-Read) address different semantic gap challenges (Section 5.3.2)
- **Strategies**: Lexical, semantic, graph, generative, and hybrid retrieval show different precision-recall trade-offs (Section 5.3.3)
- **Post-Processing**: Re-ranking (Semantic Anchoring, Memory-R1) and aggregation (ComoRAG, MA-RAG, G-Memory) improve context quality (Section 5.3.4)

### 5.4 Conditions and Trade-offs in Reported Results

The survey reveals that memory system performance is highly **conditional** on several factors:

**Parametric Memory** performs best when:
- Conceptual understanding and pattern induction are needed (Section 3.4)

**Latent Memory** performs best when:
- Performance and scalability are prioritized over interpretability (Section 3.4)

#### 5.4.2 Scale-Dependent Performance

**Medium-Scale Settings (1,000–10,000 samples):**

#### 5.4.3 Dynamics-Dependent Performance

**High-Frequency Retrieval:**
- Benefits working memory functions (Section 4.3)
- Requires efficient latent or token-level forms (Section 3.1, 3.3)

**Low-Frequency Consolidation:**
- Benefits long-term factual and experiential memory (Section 4.1–4.2)

- Risks outdated information without proper updating (Section 5.2.2)

**Automated vs. Manual Dynamics:**
- Manual heuristics work for narrow, well-defined tasks (Section 7.2.1)
- Automated (RL-driven) dynamics scale better to open-ended environments (Section 7.3.2)
- Hybrid approaches (RL-assisted) offer practical middle ground (Section 7.3.1, Figure 11)

## 6. Limitations and Trade-offs

While this survey establishes a unified framework for understanding agent memory, it also reveals significant constraints inherent in current designs. No single memory architecture serves all purposes; every design choice involves a trade-off between efficiency, interpretability, adaptability, and scalability. This section critically examines the assumptions underlying current approaches, the scenarios they fail to address, and the open questions that remain unresolved.

### 6.1 Architectural Trade-offs: The Form-Function Mismatch

The choice of memory **Form** (Section 3) dictates the fundamental capabilities and limitations of an agent. There is no "universal" memory substrate; each form excels in specific functional roles while failing in others.

*   **Token-level Memory: Interpretability vs. Scalability**
    Token-level memory (Section 3.1) stores information as explicit, discrete units (e.g., text snippets, graph nodes).
    *   **Strength:** It is **symbolic and addressable**, allowing for swift adding, deleting, and updating (Figure 5). This makes it ideal for high-stake domains like law or finance where auditability is required (Section 3.4).
    *   **Limitation:** As memory grows, the lack of explicit relational organization in **Flat Memory (1D)** means coherence depends heavily on retrieval quality (Section 3.1.1). Conversely, **Hierarchical Memory (3D)** supports complex reasoning but incurs high construction and search costs, making optimal three-dimensional layout design a difficult problem (Section 3.1.3).
    *   **Trade-off:** Designers must choose between the scalability of flat storage and the reasoning power of structured hierarchies.

*   **Parametric Memory: Performance vs. Plasticity**
    Parametric memory (Section 3.2) embeds information directly into model weights or adapters.
    *   **Strength:** It offers **zero-latency access** and better performance gains for tasks requiring conceptual understanding (Section 3.4). Internal parametric memory adds no extra inference overhead (Section 3.2.1).
    *   **Limitation:** It suffers from the **stability-plasticity dilemma**. Storing new memory requires retraining or editing, which is costly and prone to **catastrophic forgetting** of old memory (Section 3.2.1). External parametric memory (e.g., adapters) mitigates this but relies on indirect influence mediated through attention pathways, limiting effectiveness (Section 3.2.2).
    *   **Trade-off:** You gain inference speed and generalization but lose the ability to rapidly update specific facts without risking model stability.

*   **Latent Memory: Efficiency vs. Opacity**
    Latent memory (Section 3.3) operates within internal representations (e.g., KV cache, embeddings).
    *   **Strength:** It provides **high knowledge density** and convenient modality fusion, suitable for on-device deployment (Section 3.4).
    *   **Limitation:** It is **human-unreadable**. The latent memory is a black box, making it difficult to debug, edit, or verify the knowledge it stores (Section 5.1.4). Furthermore, generation processes may introduce information loss or bias, and states can drift over multiple read-write cycles (Section 3.3.1).
    *   **Trade-off:** You gain compression and efficiency but sacrifice transparency and control.

### 6.2 Operational Constraints: Dynamics and Failure Modes

The **Dynamics** of memory (Section 5)—how it is formed, evolved, and retrieved—introduce specific failure modes that current systems struggle to mitigate.

*   **Lossy Compression in Formation**
    Memory formation often relies on **Semantic Summarization** (Section 5.1.1) to reduce context length.
    *   **Constraint:** This is a **lossy compression mechanism**. While it prioritizes global semantic coherence, specific details or subtle cues may be smoothed out (Section 5.1.1 Summary).
    *   **Risk:** This limits utility in **evidence-critical tasks** where factual precision is paramount. For example, incremental summarization poses computational bottlenecks and potential information forgetting (Section 5.1.1).

*   **The Forgetting Dilemma**
    Memory forgetting (Section 5.2.3) is necessary to prevent overload but risky.
    *   **Constraint:** Heuristic forgetting mechanisms (e.g., LRU, time-based decay) may eliminate **long-tail knowledge** that is seldom accessed but essential for correct decision-making (Section 5.2.3 Summary).
    *   **Risk:** Overly aggressive pruning harms reasoning continuity in long-term contexts. Conversely, retaining everything leads to retrieval delays and interference from outdated knowledge (Section 5.2.3).

*   **Silent Failure in Retrieval**
    Automated retrieval timing (Section 5.3.1) allows agents to decide when to access memory.
    *   **Constraint:** If an agent overestimates its internal knowledge and fails to initiate retrieval, the system can fall into a **silent failure mode** (Section 5.3.1 Summary).
    *   **Risk:** Knowledge gaps may lead to hallucinated outputs without explicit error signals. Balancing essential information access against excessive noise introduction remains an unsolved control problem (Section 5.3.1).

*   **Schema Rigidity in Structured Construction**
    Structured Construction (Section 5.1.3) organizes data into graphs or trees.
    *   **Constraint:** Pre-defined structures may fail to represent **nuanced or ambiguous information** (Section 5.1.3 Summary).
    *   **Risk:** The extraction and maintenance costs are typically high, and rigid schemas may not adapt well to open-ended environments where information types evolve dynamically.

### 6.3 Evaluation Gaps: Benchmarking Limitations

The survey identifies critical weaknesses in how the field evaluates memory systems (Section 6.1, Section 5.3). The current experimental landscape limits our ability to generalize findings.

*   **Scale Imbalance**
    Many memory-specific benchmarks are **small-scale**, limiting statistical power.
    *   **Evidence:** **LoCoMo** has only 300 samples, **MemoryBank** has 194 samples, and **PersonaMem** has 180 samples (Table 8).
    *   **Implication:** Results on these benchmarks may not generalize to large-scale, real-world deployments where memory stores contain millions of entries. In contrast, non-memory benchmarks like **ToolBench** have 126,486 samples (Table 8), highlighting a disparity in evaluation rigor.

*   **Benchmark Concentration**
    There is heavy reliance on a single benchmark for framework evaluation.
    *   **Evidence:** **LoCoMo** appears in 6 out of 7 frameworks with reported evaluations (Table 9).
    *   **Implication:** This concentration risks **overfitting** to benchmark characteristics. Frameworks may be optimized specifically for LoCoMo's conversational memory tasks rather than general memory capabilities.

*   **Metric Heterogeneity**
    Different papers use different metrics for similar capabilities, making cross-paper comparison difficult (Section 5.3.2).
    *   **Evidence:** Some report accuracy, others report success rate; some measure token compression, others measure latency (Section 5.3.2).
    *   **Implication:** Without standardized metrics, it is impossible to determine if a new method is genuinely superior or simply optimized for a different metric.

*   **Missing Trustworthiness Evaluation**
    Despite identified trustworthiness requirements (Section 7.7), few benchmarks explicitly evaluate privacy, explainability, or robustness.
    *   **Evidence:** Only **HaluMem** (3,467 samples) specifically targets memory hallucinations (Table 8).
    *   **Implication:** We lack empirical data on how memory systems behave under adversarial conditions or privacy constraints, which is critical for real-world deployment (Section 7.7.2).

### 6.4 Unaddressed Scenarios and Open Frontiers

Several critical scenarios remain underexplored or unresolved in the current literature.

*   **Multimodal Memory Gaps**
    While text-based memory is mature, multimodal memory is nascent.
    *   **Constraint:** There is currently **no memory system that provides truly omnimodal support** (Section 7.4.2). Most approaches remain specialized to individual modalities (e.g., visual vs. audio) or loosely coupled (Section 7.4.2).
    *   **Open Question:** How to design memory representations that flexibly accommodate diverse modalities while preserving semantic alignment and temporal coherence remains a key challenge (Section 7.4.2).

*   **Automation and RL Challenges**
    The field is moving toward automated memory management via Reinforcement Learning (RL) (Section 7.3).
    *   **Constraint:** Current RL-assisted approaches typically intervene in only a **subset of the memory lifecycle** (e.g., only writing or only retrieval) (Section 7.3.2).
    *   **Open Question:** Fully agentic memory systems require end-to-end RL training to coordinate complex interactions among formation, evolution, and retrieval. However, this introduces complexity in **training stability and reward design** (Section 7.3.2).

*   **Human-Cognitive Discrepancies**
    Agent memory systems diverge from human cognition in critical ways.
    *   **Constraint:** Agents rely on **verbatim retrieval** mechanisms (e.g., RAG), treating memory as immutable tokens (Section 7.8.1). Humans, conversely, operate via **constructive processes**, actively reconstructing past events (Section 7.8.1).
    *   **Implication:** Agents lack the biological capacity for memory distortion and dynamic remodeling that characterizes human intelligence. Bridging this gap requires evolving beyond exclusive online updating to incorporate **offline consolidation mechanisms** analogous to biological sleep (Section 7.8.2).

*   **Multi-Agent Shared Memory**
    Shared memory for multi-agent systems is an emerging frontier.
    *   **Constraint:** Naive global sharing exposes challenges including **memory clutter, write contention, and lack of role-aware access control** (Section 7.5.1).
    *   **Open Question:** Developing agent-aware shared memory where read/write behaviors are conditioned on agent roles and trust remains an open research direction (Section 7.5.2).

Addressing these limitations requires not only architectural innovation but also a maturation of evaluation standards to ensure trustworthy and scalable memory systems.

## 7. Implications and Future Directions

This survey does more than catalog existing methods; it fundamentally reorients how we conceptualize memory in AI agents. By shifting the focus from static storage buffers to dynamic, learnable cognitive substrates, this work changes the landscape of agent design. This section outlines how the **Forms–Functions–Dynamics** taxonomy reshapes research priorities, identifies concrete frontiers for follow-up work, and provides a practical decision matrix for practitioners deploying memory systems today.

### 7.1 Reshaping the Research Landscape

The primary implication of this survey is the transition of memory from an **auxiliary feature** to a **first-class primitive** in agentic design (Abstract). Historically, memory was treated as an external database queried via Retrieval-Augmented Generation (RAG). This survey argues that such a view is insufficient for agents that must adapt continually over long horizons.

**Theoretical Reorientation:**
The survey challenges the traditional binary of **Short-Term Memory (STM)** and **Long-Term Memory (LTM)** as fixed architectural modules. Instead, it posits that these roles emerge from the **temporal invocation patterns** of three lifecycle operators: Formation ($\mathcal{F}$), Evolution ($\mathcal{E}$), and Retrieval ($\mathcal{R}$) (Section 2.2 of the survey).
> "Short-term and long-term memory phenomena therefore emerge not from discrete architectural modules but from the temporal patterns with which formation, evolution, and retrieval are engaged." (Section 2.2)

**Implication for Researchers:**
This theoretical shift implies that future research should not focus on building separate STM/LTM buffers. Instead, efforts should target **policy learning** for the operators $\mathcal{F}, \mathcal{E}, \mathcal{R}$. For example, a single memory state $M_t$ can serve both roles if the retrieval policy $\mathcal{R}$ is high-frequency during a task (acting as STM) and low-frequency across sessions (acting as LTM). This unifies the architecture and reduces engineering complexity.

**Conceptual Unification:**
By distinguishing **Agent Memory** from related concepts like **LLM Memory** (internal KV cache) and **Context Engineering** (resource optimization), the survey clears conceptual ambiguity (Section 2.3).
> "Agent Memory is uniquely characterized by its focus on maintaining a persistent and self-evolving cognitive state that integrates factual knowledge and experience." (Figure 2 Caption)

**Implication for the Field:**
This distinction prevents the misapplication of techniques. For instance, optimizing KV cache compression (LLM Memory) does not solve the problem of cross-session user preference retention (Agent Memory). Researchers must now explicitly state which scope they are addressing, ensuring that benchmarks and metrics align with the claimed contribution.

### 7.2 Priority Research Frontiers

The survey identifies eight specific frontiers where significant innovation is needed (Section 7 of the survey). Based on the taxonomy, we can prioritize these into three high-impact categories for follow-up research.

#### 7.2.1 From Retrieval to Generative Memory
The most significant paradigm shift identified is the move from **retrieval-centric** to **generative memory** (Section 7.1).
*   **Current State:** Most systems retrieve static fragments (e.g., text snippets) and concatenate them into the context.
*   **Future Direction:** Agents should **synthesize** new memory representations on demand. Instead of recalling a past trajectory verbatim, the agent generates a latent vector or summary that abstracts the *lesson* from that trajectory.
*   **Why It Matters:** Retrieval is limited by what was explicitly stored. Generation allows for **abstraction and recomposition**, enabling agents to generalize beyond their stored history (Section 7.1.2).
*   **Technical Path:** This aligns with **Latent Memory** forms (Section 3.3). Research should focus on methods like `MemGen` (Section 3.3.1) that dynamically generate latent memory tokens during decoding, reducing reliance on explicit lookup.

#### 7.2.2 Automated Management via Reinforcement Learning
Current memory systems rely heavily on **hand-crafted heuristics** (e.g., "summarize every 10 turns"). The survey outlines a trajectory toward **fully RL-driven memory management** (Section 7.3).
*   **Current State:** **RL-free** systems use fixed thresholds or prompt-based instructions (Figure 11).
*   **Future Direction:** **Fully RL-driven** systems where the agent learns *when* to forget, *what* to abstract, and *how* to retrieve as part of its policy (Section 7.3.2).
*   **Why It Matters:** Hand-crafted rules do not scale to open-ended environments. RL-driven management allows the agent to discover optimal memory strategies based on task performance rather than human intuition.
*   **Technical Path:** Research should explore end-to-end training of memory operators. For example, using RL to optimize the forgetting policy $\mathcal{E}_{forget}$ to maximize long-horizon task success while minimizing context cost (Section 7.3.1).

#### 7.2.3 Trustworthiness and Cognitive Alignment
As memory systems become persistent, they introduce new risks regarding privacy and reliability.
*   **Trustworthiness:** Future systems must support **granular permissioned memory**, **verifiable forgetting**, and **auditable updates** (Section 7.7.2). Current benchmarks lack evaluation for these properties (Section 6.3).
*   **Human-Cognitive Connections:** Agents currently rely on verbatim retrieval, unlike human **constructive memory** which actively reconstructs past events (Section 7.8.1). Future architectures should incorporate **offline consolidation mechanisms** analogous to biological sleep to reorganize memory without latency constraints (Section 7.8.2).
*   **Why It Matters:** Without trustworthiness, persistent memory becomes a liability (e.g., leaking private user data). Without cognitive alignment, agents remain rigid archivists rather than adaptive learners.

### 7.3 Practical Deployment Guide

For practitioners building agent systems today, the survey's taxonomy provides a decision matrix for selecting memory architectures. The choice depends on the required **Function** (Factual, Experiential, Working) and the constraints of the **Environment** (Interpretability, Efficiency, Modality).

#### 7.3.1 Selecting Memory Forms
Based on the adaptation guidelines in **Section 3.4** and **Figure 5** of the survey, use the following criteria:

| Requirement | Recommended Form | Rationale | Representative Methods |
| :--- | :--- | :--- | :--- |
| **Auditability & Control** | **Token-level** | Symbolic, addressable, and transparent. Allows swift adding/deleting/updating. | `MemGPT`, `MemoryBank`, `Mem0` |
| **Performance & Generalization** | **Parametric** | Implicit and abstract. Better for tasks requiring conceptual understanding. | `Character-LM`, `WISE`, `LoRA` adapters |
| **Efficiency & Multimodal** | **Latent** | High knowledge density. Convenient modality fusion. Human-unreadable. | `MemGen`, `AutoCompressor`, `XMem` |

**Practical Context:**
*   **Choose Token-level** for high-stake domains (law, finance) where you must trace *why* an agent made a decision (Section 3.4).
*   **Choose Parametric** for role-playing or stylized responses where consistency of persona is more important than factual precision (Section 3.4).
*   **Choose Latent** for on-device deployment or multimodal tasks (video/audio) where context window limits are strict (Section 3.4).

#### 7.3.2 Designing Memory Dynamics
The survey emphasizes that dynamics (Formation, Evolution, Retrieval) should be tuned to the task horizon (Section 5).

*   **Short-Horizon Tasks (Single Session):**
    *   **Focus:** **Working Memory** (Section 4.3).
    *   **Strategy:** Use **Input Condensation** (e.g., `LLMLingua`) to maximize information density within the context window (Section 5.1.1).
    *   **Retrieval:** High-frequency, continuous retrieval to maintain state (Section 2.2).

*   **Long-Horizon Tasks (Multi-Session):**
    *   **Focus:** **Factual & Experiential Memory** (Section 4.1, 4.2).
    *   **Strategy:** Use **Semantic Summarization** (e.g., `Mem0`) for formation and **Consolidation** for evolution (Section 5.1.1, 5.2.1).
    *   **Retrieval:** Intermittent, intent-driven retrieval to avoid noise (Section 5.3.1).

*   **Open-Ended Environments:**
    *   **Focus:** **Automated Management**.
    *   **Strategy:** Move toward **RL-assisted** dynamics (e.g., `Memory-R1`) to learn forgetting and updating policies (Section 7.3.1).
    *   **Retrieval:** Automated timing and intent detection to prevent silent failures (Section 5.3.1).

### 7.4 Integration and Reproducibility Guidance

To accelerate adoption and ensure rigorous evaluation, practitioners should leverage the existing infrastructure cataloged in the survey while adhering to emerging best practices.

#### 7.4.1 Leveraging Open-Source Frameworks
The survey identifies a mature ecosystem of frameworks that implement the taxonomy's concepts (Section 6.2, Table 9).
*   **For Hierarchical Memory:** Use **MemGPT** or **MemoryOS** which implement virtual memory management with separate short/long-term stores (Table 9).
*   **For Graph-Structured Memory:** Use **Zep** or **Mem0g** which support temporal knowledge graphs for relational reasoning (Table 9, Section 3.1.2).
*   **For Modular Integration:** Use **MemEngine** or **LangMem** which provide unified APIs for developing advanced memory modules (Table 9).

**Integration Tip:** Do not treat these frameworks as black boxes. Align their internal configurations with your target **Function**. For example, if you need **Experiential Memory** (Section 4.2), configure the framework to store trajectories and insights (e.g., `ExpeL` style) rather than just dialogue history.

#### 7.4.2 Benchmarking Best Practices
The survey reveals significant gaps in evaluation, particularly benchmark concentration (e.g., over-reliance on `LoCoMo`) and scale imbalance (Section 6.3).
*   **Avoid Overfitting:** Do not evaluate solely on `LoCoMo` (300 samples). Complement it with larger-scale benchmarks like **MemBench** (53,000 samples) or **StreamBench** (9,702 samples) for statistical power (Table 8).
*   **Match Metrics to Function:**
    *   For **Factual Memory**, report **Consistency** and **Coherence** (Section 4.1).
    *   For **Experiential Memory**, report **Task Success Rate** and **Sample Efficiency** (Section 4.2).
    *   For **Working Memory**, report **Compression Ratio** and **Context Retention** (Section 4.3).
*   **Report Trade-offs:** Always report performance against cost (e.g., accuracy vs. latency vs. memory size). The survey highlights that no single method dominates all metrics (Section 6.1).

#### 7.4.3 Addressing Trustworthiness
Given the risks identified in **Section 7.7**, practitioners must implement safeguards even if benchmarks do not yet require them.
*   **Privacy:** Implement **user-governed retention policies** and **encrypted storage** for sensitive user data (Section 7.7.2).
*   **Explainability:** Use **Token-level Memory** where possible to enable traceable access paths (Section 3.4).
*   **Robustness:** Implement **conflict detection** mechanisms during Memory Evolution to prevent contradictory facts from corrupting the memory base (Section 5.2.2).

### 7.5 Summary of Strategic Directions

The survey concludes that memory is the essential substrate through which agents achieve temporal coherence and continual adaptation (Section 8). To advance the field, researchers and practitioners should prioritize the following strategic directions:

1.  **Architectural Unification:** Design single memory states $M_t$ with flexible operator policies ($\mathcal{F}, \mathcal{E}, \mathcal{R}$) rather than rigid STM/LTM modules (Section 2.2).
2.  **Generative Capabilities:** Invest in **Latent Memory** and generative retrieval to enable abstraction beyond verbatim recall (Section 7.1).
3.  **Learned Management:** Transition from heuristic rules to **RL-driven memory policies** for open-ended adaptability (Section 7.3).
4.  **Trustworthy Deployment:** Prioritize **privacy, explainability, and robustness** as foundational requirements, not add-ons (Section 7.7).

By adopting this structured perspective, the community can move beyond fragmented experiments toward robust, scalable, and trustworthy agentic intelligence.
