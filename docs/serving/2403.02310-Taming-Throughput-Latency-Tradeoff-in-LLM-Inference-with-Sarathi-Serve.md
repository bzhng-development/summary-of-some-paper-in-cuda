## 1. Executive Summary

This paper introduces **Sarathi-Serve**, an LLM inference scheduler that resolves the critical tradeoff between high throughput and low tail latency by replacing eager prefill scheduling with **chunked-prefills** and **stall-free batching**. By splitting large input prompts into fixed-size compute chunks and coalescing them with ongoing decode iterations, the system eliminates "generation stalls" that plague state-of-the-art systems like vLLM, where long prefills delay token generation for seconds. This approach yields up to **3.7× higher serving capacity** for the **Yi-34B** model on two A100 GPUs and up to **5.6× gain** for the **Falcon-180B** model under pipeline parallelism, all while strictly meeting tight Time-Between-Tokens (TBT) Service Level Objectives.

## 2. Context and Motivation

To understand why **Sarathi-Serve** is necessary, we must first dissect the unique computational anatomy of Large Language Model (LLM) inference and why standard scheduling strategies fail to optimize it for real-time serving. The core problem addressed by this paper is the **throughput-latency tradeoff** inherent in current LLM serving systems, specifically caused by the interference between two distinct phases of inference: **prefill** and **decode**.

### The Fundamental Asymmetry of LLM Inference

LLM inference is not a uniform process; it consists of two phases with diametrically opposite hardware characteristics:

1.  **The Prefill Phase:** This phase processes the user's entire input prompt (which can range from hundreds to thousands of tokens) in parallel to generate the first output token.
    *   **Compute Characteristic:** It is **compute-bound**. Because it processes many tokens simultaneously, it saturates the GPU's arithmetic logic units (ALUs).
    *   **Batching Sensitivity:** As shown in **Figure 3**, increasing the batch size has a marginal effect on prefill throughput. A single request with a long prompt already utilizes the GPU efficiently; adding more requests does not significantly speed up the operation because the compute is already maxed out.
2.  **The Decode Phase:** This phase generates subsequent tokens one-by-one in an autoregressive manner. Each iteration takes the previously generated token and the entire history (stored in the **KV-cache**) to predict the next token.
    *   **Compute Characteristic:** It is **memory-bound**. Since only one token is processed per request per iteration, the GPU spends most of its time waiting for model weights to be fetched from High Bandwidth Memory (HBM) rather than performing math.
    *   **Batching Sensitivity:** Batching is critical here. **Figure 3** demonstrates that decode throughput scales almost linearly with batch size. By grouping many requests, the system amortizes the cost of fetching weights across multiple tokens, drastically improving efficiency.

This asymmetry creates a scheduling dilemma: to maximize throughput, we want large batches (favoring decodes), but large batches inevitably mix prefill and decode operations, which have vastly different execution times.

### The Failure of Existing Scheduling Paradigms

Current LLM serving systems generally fall into two categories based on how they prioritize these phases. The paper argues that both approaches possess fundamental pitfalls that make them unsuitable for high-performance online serving.

#### 1. Decode-Prioritizing (Request-Level Batching)
Systems like **FasterTransformer** employ a conservative strategy known as request-level batching.
*   **Mechanism:** The scheduler admits a batch of new requests, runs their prefill phases, and then runs their decode phases. Crucially, **no new requests are admitted** until every request in the current batch has finished generating its entire output sequence.
*   **The Flaw:** While this guarantees low latency for ongoing requests (since new work never interrupts them), it destroys throughput. If one request in a batch of 32 finishes early, the GPU sits idle or runs with a reduced batch size while waiting for the longest request to complete. This leads to severe under-utilization of GPU compute resources.

#### 2. Prefill-Prioritizing (Iteration-Level Batching)
State-of-the-art systems like **Orca** and **vLLM** utilize **iteration-level batching**, which allows requests to dynamically enter or exit the batch at every iteration. These systems typically employ **prefill-prioritizing** scheduling.
*   **Mechanism:** Whenever GPU memory becomes available (e.g., when a request finishes), the scheduler eagerly admits new requests and immediately executes their **full prefill phase**.
*   **The Flaw:** This strategy maximizes throughput by ensuring decode batches are always large. However, it introduces a catastrophic latency phenomenon termed **generation stalls**.
    *   Because prefill times depend on prompt length, a single long prompt (e.g., 8,000 tokens) can take seconds to process.
    *   When the scheduler eagerly inserts this long prefill into the running batch, it pauses the decode iterations of all other active requests.
    *   **Figure 1a** illustrates this vividly: in vLLM, generation stalls can last for several seconds, causing the **Time-Between-Tokens (TBT)** latency to spike unpredictably. For interactive applications like chatbots, a multi-second pause between words renders the user experience unacceptable.

**Figure 2** summarizes this tradeoff: decode-prioritizing optimizes latency but sacrifices throughput, while prefill-prioritizing optimizes throughput but sacrifices tail latency (TBT).

### The Hidden Cost: Pipeline Bubbles in Multi-GPU Deployments

The problem compounds when scaling to larger models (e.g., **Falcon-180B**) that require **Pipeline Parallelism (PP)**. In PP, the model layers are split across multiple GPUs, and data flows through them like an assembly line.
*   **The Expectation:** Ideally, micro-batching should keep all pipeline stages busy.
*   **The Reality:** Because current schedulers create batches with wildly varying compute requirements (e.g., one micro-batch has a huge prefill, the next has only decodes), the execution time of each iteration fluctuates significantly.
*   **Pipeline Bubbles:** These fluctuations cause **pipeline bubbles**—periods where downstream GPUs sit idle waiting for upstream GPUs to finish a slow iteration. **Figure 8** shows that in systems like Orca, these bubbles can waste nearly a second of GPU time per iteration when mixing long prefills with short decodes. This directly degrades system throughput and increases latency, making PP inefficient on commodity networks where communication overhead is already a concern.

### Positioning Sarathi-Serve

Sarathi-Serve positions itself as the solution that breaks the deadlock between throughput and latency. It does not choose between "prefill-first" or "decode-first"; instead, it fundamentally changes the unit of work.

1.  **Reframing the Prefill:** Instead of treating a prefill as a monolithic, long-running task, Sarathi-Serve introduces **chunked-prefills**. It splits a large prompt into small, fixed-size chunks (e.g., 512 tokens).
2.  **Exploiting Arithmetic Slack:** The authors observe that decode batches are so memory-bound that there is "slack" in the GPU's compute capacity. **Figure 5** shows that decode batches operate in a region of low arithmetic intensity. Sarathi-Serve fills this slack by coalescing a **chunk** of a prefill with ongoing decodes.
3.  **Stall-Free Scheduling:** By limiting the amount of prefill work added to any single iteration, the system ensures that the iteration time remains bounded and predictable. This eliminates generation stalls entirely (**Figure 7**), allowing new requests to enter the system without pausing ongoing decodes.

In essence, while prior work treats the variability of prompt lengths as an unavoidable source of latency spikes, Sarathi-Serve treats it as a scheduling constraint that can be smoothed out via chunking. This allows the system to achieve the high throughput of prefill-prioritizing systems (by keeping batches full) with the low, predictable tail latency of decode-prioritizing systems (by preventing long prefills from blocking decodes).

## 3. Technical Approach

This section details the architectural design and algorithmic mechanisms of **Sarathi-Serve**, a system that resolves the throughput-latency tradeoff in LLM inference by decoupling prefill execution from decode latency through **chunked-prefills** and **stall-free batching**.

### 3.1 Reader orientation (approachable technical breakdown)
Sarathi-Serve is an intelligent inference scheduler that sits between incoming user requests and the GPU execution engine, dynamically constructing computation batches to ensure no single operation blocks another. It solves the problem of unpredictable latency spikes (generation stalls) caused by long input prompts by breaking those prompts into small, fixed-size pieces and interleaving them with ongoing token generation, effectively smoothing out computational variance.

### 3.2 Big-picture architecture (diagram in words)
The system architecture consists of three logical components operating in a tight feedback loop:
1.  **The Profiler & Budget Calculator:** This offline or initialization-time component analyzes the specific hardware (e.g., A100 GPU) and model configuration to determine a global `token_budget` ($\tau$), which represents the maximum number of tokens that can be processed in a single iteration without violating the user-defined Time-Between-Tokens (TBT) Service Level Objective (SLO).
2.  **The Stall-Free Scheduler:** This is the core runtime logic that executes at every iteration. It accepts the current set of active requests (both those in prefill and those in decode), queries the `token_budget`, and constructs a new hybrid batch by prioritizing ongoing decodes first, then filling the remaining capacity with chunks of new prefills.
3.  **The Chunked Execution Engine:** This modified inference backend (built on top of vLLM and FlashAttention) receives the hybrid batch. Unlike standard engines that expect full prompts, it accepts partial prefill sequences, manages the incremental update of the KV-cache for these chunks, and executes the matrix operations for the mixed batch of decode tokens and prefill chunks.

### 3.4 Detailed, sentence-based technical breakdown

#### The Core Concept: Bounding Iteration Time
The fundamental insight of Sarathi-Serve is that latency spikes occur because the duration of an inference iteration is unbounded when full prefills are allowed; therefore, the system enforces a hard cap on the computational load of every single iteration.
Instead of asking "How many requests can we fit?", Sarathi-Serve asks "How many *tokens* can we process in this specific iteration such that the runtime remains below our latency target?"
This cap is defined as the **Token Budget** ($\tau$), a hyperparameter determined via profiling that balances the need for high throughput (larger $\tau$) against the need for low tail latency (smaller $\tau$).

#### Determining the Token Budget ($\tau$)
The token budget is not a static constant but a carefully tuned parameter derived from the relationship between batch size, token count, and execution time on specific hardware.
To determine $\tau$, the system performs a one-time profiling run where it measures the execution time of batches with varying numbers of tokens to find the maximum count that satisfies the TBT SLO.
Mathematically, if $T_{SLO}$ is the maximum allowable time-between-tokens and $T_{iter}(n)$ is the execution time for a batch containing $n$ tokens, the budget is defined as:
$$ \tau = \max \{ n \mid T_{iter}(n) \leq T_{SLO} \} $$
This selection process must account for two competing overheads:
1.  **Chunking Overhead:** If $\tau$ is too small, large prompts are split into many tiny chunks. Since each chunk requires re-loading the KV-cache of all previous chunks for the attention calculation, excessive splitting increases memory bandwidth usage. Specifically, if a prompt is split into $N$ chunks, the KV-cache for the first chunk is loaded $N-1$ times, the second $N-2$ times, and so on.
2.  **Tile Quantization Effects:** GPU matrix multiplications operate on fixed-size tiles (e.g., 16x16 or 32x32 threads). If the token count $\tau$ is not a multiple of the tile size, the GPU performs wasted "padding" computations. The paper notes that a chunk size of 257 can increase prefill time by 32% compared to 256 due to this effect, so $\tau$ is chosen to align with hardware tile dimensions.
3.  **Pipeline Bubble Minimization:** In pipeline-parallel settings, $\tau$ also influences the variance in execution time between micro-batches. A larger $\tau$ allows more tokens per iteration but risks higher variance if prompt lengths differ wildly, while a smaller $\tau$ ensures uniformity at the cost of potential kernel launch overheads.
For the evaluations in this paper, the authors use a token budget of **512** for strict SLO scenarios (interactive chat) and **2048** for relaxed SLO scenarios (summarization), with a specific adjustment to **1536** for the LLaMA2-70B relaxed configuration to optimize pipeline performance.

#### Mechanism 1: Chunked-Prefills
**Chunked-prefills** is the technique of splitting a single large input prompt (which might be thousands of tokens) into a sequence of smaller, equal-sized chunks, each containing at most $\tau$ tokens.
In a traditional system, a prompt of 4,000 tokens would be processed in one monolithic iteration taking perhaps 1,500ms, blocking all other traffic.
In Sarathi-Serve, this same prompt is divided into, for example, eight chunks of 512 tokens each, and these chunks are processed over eight separate iterations.
Crucially, the attention mechanism for each chunk must still attend to all previous tokens in the prompt to maintain correctness.
When processing chunk $k$ of a prompt, the system loads the KV-cache entries generated by chunks $1$ through $k-1$ from GPU memory, computes the attention for the new tokens in chunk $k$, and then appends the new KV entries to the cache.
Although this introduces extra memory reads compared to a single-shot prefill, the paper argues that prefill operations are **compute-bound** even at modest sequence lengths (e.g., 512 tokens), meaning the GPU is already saturated with math operations and the extra memory bandwidth cost is negligible relative to the total runtime.
This is supported by **Figure 4**, which shows that linear layers (compute-heavy) dominate runtime, and **Figure 9**, which demonstrates that the latency penalty of chunking is minimal compared to the massive stalls caused by full prefills in other systems.
By processing prefills in chunks, the system transforms a long, unpredictable blocking operation into a series of short, predictable tasks that can be easily interleaved with other work.

#### Mechanism 2: Stall-Free Batching Algorithm
The **Stall-Free Batching** algorithm is the scheduling logic that constructs each iteration's batch to ensure that ongoing decode requests are never delayed by new prefill work.
The algorithm operates on a strict priority hierarchy, ensuring that the "critical path" of token generation for existing users is never interrupted.
We can describe the algorithm's execution flow in three distinct phases, as formalized in **Algorithm 3** of the paper:

**Phase 1: Admit All Ongoing Decodes**
The scheduler first iterates through all currently active requests that have completed their prefill phase and are in the decode stage.
Every single token required for the next step of these decode requests is unconditionally added to the new batch.
This guarantees that if a user is currently receiving a response, their next token will be generated in this exact iteration, regardless of how many new users are trying to join the system.
Let $B_{decode}$ be the set of tokens from these ongoing decode requests; the current token count is initialized as $n_t = |B_{decode}|$.

**Phase 2: Admit Partial Prefills**
Next, the scheduler checks for any requests that are currently in the middle of a chunked-prefill operation (i.e., they have started but not finished processing their prompt).
For each such request, the scheduler calculates the size of the next chunk, denoted as $c$, such that adding it does not exceed the token budget $\tau$.
The chunk size is computed as $c = \min(\text{remaining\_prompt\_tokens}, \tau - n_t)$.
If there is remaining budget ($\tau - n_t > 0$), this chunk is added to the batch, and $n_t$ is updated.
This ensures that progress is made on incomplete prefills without starving the decodes, as the decodes were already fully accommodated in Phase 1.

**Phase 3: Admit New Requests (Chunked)**
Only after all ongoing decodes and partial prefills are accommodated does the scheduler attempt to admit brand new requests from the queue.
For each new request, the scheduler again calculates a chunk size $c$ that fits within the remaining budget $\tau - n_t$.
If the new request's prompt is larger than the remaining budget, only the first chunk (of size $c$) is admitted, and the rest of the prompt waits for the next iteration.
If the prompt fits entirely within the remaining budget, the full prefill is admitted (though this is rare under high load).
This admission loop continues until either the request queue is empty or the token budget $\tau$ is completely exhausted.
The resulting batch $B$ is a **hybrid batch** containing a mix of decode tokens and prefill chunks, with a total token count $n_t \leq \tau$.

**Effect on Latency**
Because the total number of tokens in any iteration is bounded by $\tau$, the execution time of every iteration is also bounded.
This eliminates the "generation stall" phenomenon seen in **Figure 1a**, where vLLM pauses decodes for seconds to process a long prompt.
In Sarathi-Serve, a long prompt merely occupies a small fraction of the budget in several consecutive iterations, allowing decodes to proceed in every single one of those iterations without interruption.
**Figure 7** visually contrasts this: while vLLM creates a gap in the timeline for requests A and B while processing C and D, Sarathi-Serve interleaves chunks of C and D ($C_{p1}, C_{p2}$) alongside the decodes of A and B ($A_d, B_d$), resulting in a continuous, stall-free timeline.

#### Handling Pipeline Parallelism and Bubbles
When deploying large models like **Falcon-180B** across multiple GPUs using Pipeline Parallelism (PP), the system faces the challenge of **pipeline bubbles**, which are idle cycles where a GPU stage waits for data from the previous stage.
In traditional iteration-level batching, bubbles arise because different micro-batches have wildly different execution times; for instance, a micro-batch containing a 4,000-token prefill might take 1,150ms, while the next micro-batch containing only decodes takes 200ms.
This variance forces downstream stages to wait for the slowest upstream stage, creating bubbles as large as 950ms (**Section 3.3**).
Sarathi-Serve mitigates this by leveraging the uniformity of its chunked approach.
Since every batch is constructed to have a token count close to the budget $\tau$, the computational load of each micro-batch becomes nearly identical.
Whether a batch contains 32 decode tokens or a mix of 20 decode tokens and a 512-token prefill chunk, the total compute time remains stable because the system operates in a regime where the linear layer execution time is dominated by the total token count rather than the specific mix of prefill/decode.
**Figure 8** illustrates this: while Orca shows significant gaps (bubbles) between pipeline stages due to runtime variance, Sarathi-Serve's schedule shows tightly packed operations with minimal bubbles.
This uniformity allows the pipeline to flow smoothly, making Pipeline Parallelism viable even on commodity networks (like 100 Gbps Ethernet) where communication latency would otherwise exacerbate the inefficiency of irregular schedules.

#### Implementation Specifics and Hyperparameters
The authors implemented Sarathi-Serve by modifying the open-source **vLLM** codebase, specifically extending its scheduling logic and memory management.
*   **Kernel Support:** The system utilizes **FlashAttention v2** and **FlashInfer** kernels to efficiently handle the attention computation for chunked prefills. These kernels are optimized to handle the varying sequence lengths and incremental KV-cache updates required by the chunking strategy.
*   **Memory Management:** The system retains vLLM's **PagedAttention** mechanism to manage the KV-cache efficiently, preventing memory fragmentation. However, it adds logic to track the "completion status" of prefills, distinguishing between "fully completed," "partially completed (chunked)," and "not started."
*   **Parallelism Configuration:** The evaluation covers diverse setups, including:
    *   **Tensor Parallelism (TP):** Used for models like Yi-34B (2-way TP) and LLaMA2-70B (4-way TP within a node).
    *   **Hybrid Parallelism:** For Falcon-180B, the system uses 4-way TP within each node and 2-way Pipeline Parallelism (PP) across two nodes connected by Ethernet.
*   **SLO Configuration:** The system allows users to specify a target P99 TBT SLO. For example, a "strict" SLO for Mistral-7B is set to 0.1 seconds, while a "relaxed" SLO is 0.5 seconds (**Table 3**). The profiler uses these targets to automatically select the appropriate token budget ($\tau$).
*   **Telemetry:** An extensive telemetry system was added to track the composition of every batch (number of prefill vs. decode tokens) and the resulting latency, enabling the dynamic adjustment strategies discussed in the evaluation.

#### Why This Design Works: The Arithmetic Intensity Argument
The viability of mixing prefills and decodes in the same batch relies on the concept of **Arithmetic Intensity** (FLOPs per byte of memory access).
As shown in **Figure 5**, pure decode batches have very low arithmetic intensity; they are **memory-bound**, meaning the GPU spends most of its time waiting for weights to arrive from HBM, leaving its compute units underutilized.
Pure prefill batches, conversely, are **compute-bound**, saturating the math units but potentially underutilizing memory bandwidth.
Sarathi-Serve's hybrid batches exploit the "slack" in the decode phase.
By adding a chunk of prefill tokens to a batch of decodes, the system adds significant compute work (matrix multiplications for the new tokens) without proportionally increasing the memory bottleneck (since the model weights are already being fetched for the decodes).
This moves the combined batch closer to the "balanced" region in **Figure 5**, where both compute and memory bandwidth are utilized efficiently.
Consequently, the latency of the hybrid batch is only marginally higher than a decode-only batch, yet the throughput is significantly higher because useful prefill work is accomplished "for free" during the memory-wait cycles of the decodes.
This physical property of GPU hardware is the fundamental reason why stall-free batching can achieve high throughput without sacrificing the low latency of the decode stream.

## 4. Key Insights and Innovations

The contributions of Sarathi-Serve extend beyond simple engineering optimizations; they represent a fundamental shift in how we conceptualize the scheduling unit in LLM inference. While prior work treated the variability of prompt lengths as an unavoidable source of latency, Sarathi-Serve treats it as a solvable scheduling constraint. Below are the core innovations that enable this paradigm shift.

### 4.1 The Decoupling of Prefill Atomicity via Chunked-Prefills
**The Innovation:**
Prior systems (e.g., vLLM, Orca) treat the **prefill phase** as an atomic, indivisible unit of work. A request enters the system, and the scheduler must commit to running its *entire* prompt processing sequence before yielding the GPU to other tasks. Sarathi-Serve introduces the novel concept of **chunked-prefills**, which breaks this atomicity. It redefines the prefill not as a single monolithic operation, but as a sequence of smaller, independent compute tasks that can be interleaved with other workloads.

**Distinction from Prior Work:**
*   **Prior Approach:** In iteration-level batching systems like Orca, if a 4,000-token prompt arrives, the scheduler blocks all other activities for the duration of that single long iteration (potentially >1 second). This creates the "generation stall" phenomenon described in **Figure 1a**.
*   **Sarathi-Serve Approach:** By splitting that same 4,000-token prompt into eight 512-token chunks, the system transforms one blocking event into eight non-blocking events. Crucially, this is distinct from simply reducing batch sizes; the system maintains high throughput by filling the "compute slack" of decode iterations with these chunks.

**Significance:**
This is a **fundamental innovation** rather than an incremental improvement because it breaks the direct coupling between *prompt length* and *iteration latency*.
*   **Latency Guarantee:** It mathematically bounds the maximum interference a new request can impose on existing requests. No single incoming request can ever delay an ongoing decode by more than the time it takes to process one chunk ($\tau$).
*   **Performance Impact:** This mechanism directly enables the **3.7× higher serving capacity** observed for Yi-34B under strict SLOs (**Figure 10**). Without chunking, the system would be forced to either reject new requests (lowering throughput) or accept them and violate latency SLOs.

### 4.2 The "Stall-Free" Priority Inversion Strategy
**The Innovation:**
Sarathi-Serve implements a rigorous **priority inversion** strategy formalized in **Algorithm 3**, which we term **stall-free batching**. Unlike traditional schedulers that use First-Come-First-Served (FCFS) or greedy "fill-the-gap" policies, Sarathi-Serve enforces a strict hierarchy: **(1) Ongoing Decodes $\rightarrow$ (2) Partial Prefills $\rightarrow$ (3) New Requests**.

**Distinction from Prior Work:**
*   **Decode-Prioritizing (FasterTransformer):** These systems prioritize decodes by *blocking* new prefills entirely until the batch empties. This eliminates stalls but kills throughput because the GPU runs with small batch sizes for long periods.
*   **Prefill-Prioritizing (vLLM/Orca):** These systems prioritize throughput by eagerly admitting new prefills whenever memory allows, often preempting or delaying decodes. This maximizes batch size but introduces unbounded latency spikes.
*   **Sarathi-Serve Approach:** It achieves the "best of both worlds" by admitting new work *only if* it fits within the residual capacity left by ongoing decodes. It never pauses a decode to start a prefill; instead, it piggybacks the prefill chunk onto the decode iteration.

**Significance:**
This represents a **theoretical advance** in scheduling logic for autoregressive models. It proves that high throughput and low tail latency are not mutually exclusive if the unit of work is sufficiently granular.
*   **Elimination of Stalls:** As shown in **Figure 7**, this strategy completely eliminates the generation stalls seen in vLLM, ensuring that the Time-Between-Tokens (TBT) remains stable regardless of incoming prompt lengths.
*   **Resource Efficiency:** It allows the system to sustain near-maximum batch sizes (high throughput) without the penalty of latency spikes, effectively decoupling the throughput-latency tradeoff curve.

### 4.3 Uniformity as a Mechanism for Pipeline Efficiency
**The Innovation:**
Sarathi-Serve leverages the uniformity of chunked batches to solve the **pipeline bubble** problem in multi-GPU deployments. The key insight is that pipeline efficiency depends not just on having enough work, but on having *predictable* work durations across stages. By constraining every batch to a fixed token budget ($\tau$), Sarathi-Serve ensures that every micro-batch sent down the pipeline has nearly identical execution time.

**Distinction from Prior Work:**
*   **Prior Approach:** Existing pipeline-parallel systems (like Orca with PP) suffer from severe bubbles because iteration times vary wildly. A micro-batch with a long prefill might take 1,150ms, while the next (decode-only) takes 200ms. Downstream GPUs sit idle for ~950ms waiting for the slow stage (**Section 3.3**).
*   **Sarathi-Serve Approach:** Because every batch is capped at $\tau$ tokens, the variance in execution time is minimized. Whether a batch is 100% decode or a mix of decode and prefill chunks, the total compute load remains constant.

**Significance:**
This is a **critical systems capability** that makes Pipeline Parallelism (PP) viable on commodity hardware.
*   **Enabling Commodity Clusters:** It allows large models (like Falcon-180B) to be served efficiently across nodes connected by standard Ethernet (100 Gbps) rather than requiring expensive, high-bandwidth NVLink interconnects.
*   **Performance Gain:** This uniformity drives the massive **5.6× gain in serving capacity** for Falcon-180B (**Abstract**, **Figure 11**). Without this innovation, the communication overhead and pipeline bubbles of PP on commodity networks would render such deployments impractical for low-latency serving.

### 4.4 Exploiting Arithmetic Intensity Slack for "Free" Computation
**The Innovation:**
The paper provides a rigorous quantitative justification for hybrid batching based on **arithmetic intensity**. It identifies a specific region of GPU operation where decode batches are so memory-bound that there is significant unused compute capacity ("slack"). Sarathi-Serve explicitly targets this slack to perform prefill computations with negligible latency penalty.

**Distinction from Prior Work:**
*   **Prior Approach:** Most systems view hybrid batching (mixing prefill and decode) as a compromise that inevitably increases latency. They assume adding *any* prefill work to a decode batch will linearly increase the iteration time.
*   **Sarathi-Serve Approach:** By analyzing the hardware characteristics (**Figure 5**), the authors demonstrate that adding a chunk of prefill tokens to a memory-bound decode batch moves the operation closer to the "balanced" regime (optimal FLOPs/byte) without significantly increasing the wall-clock time. The prefill work is effectively performed "for free" during the memory wait cycles of the decode.

**Significance:**
This insight transforms hybrid batching from a heuristic into a **hardware-aware optimization**.
*   **Efficiency:** It explains *why* the system can achieve higher throughput without latency penalties: it is maximizing both Model FLOPs Utilization (MFU) and Model Bandwidth Utilization (MBU) simultaneously.
*   **Generalizability:** This principle applies across different GPU architectures and model sizes, making the approach robust and future-proof as hardware characteristics evolve.

## 5. Experimental Analysis

This section dissects the empirical evaluation of **Sarathi-Serve**, moving beyond the headline numbers to understand *how* the authors validated their claims, the specific conditions under which the system excels, and the trade-offs revealed by their ablation studies. The evaluation is designed to stress-test the system across the two primary axes identified in the motivation: **throughput-latency tradeoffs** and **scalability via parallelism**.

### 5.1 Evaluation Methodology: Rigor in Workloads and Metrics

To ensure the results reflect real-world serving scenarios rather than synthetic best-cases, the authors constructed a rigorous experimental framework spanning diverse models, hardware configurations, and workload characteristics.

#### Models and Hardware Configurations
The evaluation covers a spectrum of model sizes and parallelism strategies, ensuring the findings are not limited to a specific architecture. **Table 1** details the four key configurations:
*   **Mistral-7B:** Runs on a **single NVIDIA A100 (80GB)**. This represents a standard single-node, single-GPU deployment for smaller models.
*   **Yi-34B:** Runs on **two A100 GPUs** with **2-way Tensor Parallelism (TP-2)**. This tests the system's ability to handle moderate model sizes requiring intra-node communication.
*   **LLaMA2-70B:** Runs on **eight NVIDIA A40 GPUs (48GB each)** using a hybrid **4-way TP and 2-way Pipeline Parallelism (TP4-PP2)**. This configuration targets large models constrained by memory capacity per GPU.
*   **Falcon-180B:** Runs across **two nodes** (4 A100s per node) using **4-way TP within nodes and 2-way PP across nodes**. Crucially, these nodes are connected via **100 Gbps Ethernet**, a commodity network. This setup explicitly tests the system's ability to mitigate pipeline bubbles in high-latency, low-bandwidth interconnects where Tensor Parallelism alone would fail.

#### Workloads: Capturing Real-World Variance
The authors utilize two distinct datasets to emulate different production use cases, as detailed in **Table 2**:
1.  **`openchat_sharegpt4`:** Derived from user conversations with ChatGPT-4. This dataset features **high variance** in prompt lengths (Median: 1,730 tokens; P90: 5,696 tokens) and represents interactive chatbot workloads where low latency is critical.
2.  **`arxiv_summarization`:** Derived from scientific papers and their abstracts. This dataset features **very long prompts** (Median: 7,059 tokens; P90: 12,985 tokens) but lower variance in output length. This represents batch-processing or summarization tasks where long prefills are the norm.

By filtering outliers (removing requests >8,192 or >16,384 tokens), the authors ensure the evaluation focuses on the 99th percentile of realistic traffic without being skewed by pathological edge cases. Request arrivals follow a **Poisson distribution**, mimicking the stochastic nature of real user traffic.

#### Metrics and SLO Definitions
The evaluation focuses on three core metrics:
*   **Capacity (Queries Per Second - QPS):** The maximum load the system can sustain while meeting a specific latency target. This is the primary throughput metric.
*   **Time-To-First-Token (TTFT):** Measured at the **median (P50)**. This reflects the initial responsiveness for a new request.
*   **Time-Between-Tokens (TBT):** Measured at the **99th percentile (P99)**. This is the critical metric for user-perceived fluency; a high P99 TBT indicates "stalls" or freezing during generation.

The authors define **Service Level Objectives (SLOs)** relative to the baseline execution time of a decode iteration to account for hardware-specific limits. **Table 3** outlines the thresholds:
*   **Strict SLO:** Set to **5×** the baseline decode time (e.g., 0.1s for Mistral-7B). This represents interactive applications like chatbots where any stutter is noticeable.
*   **Relaxed SLO:** Set to **25×** the baseline decode time (e.g., 0.5s for Mistral-7B). This represents applications where total completion time matters more than per-token fluidity.

#### Baselines
Sarathi-Serve is compared against two state-of-the-art systems:
*   **vLLM:** Represents the current industry standard for iteration-level batching with PagedAttention. It uses a **prefill-prioritizing** schedule.
*   **Orca:** A research prototype that also uses iteration-level batching but supports hybrid batches (mixing prefill and decode). It serves as a baseline for understanding the impact of chunking versus simple hybrid batching.

### 5.2 Main Quantitative Results: Breaking the Tradeoff

The experimental results provide compelling evidence that Sarathi-Serve successfully decouples throughput from latency, achieving gains that are impossible for baseline systems under strict constraints.

#### Capacity Gains Under Strict Latency Constraints
The most significant findings appear under **Strict SLO** conditions, where baseline systems struggle with generation stalls.
*   **Single Node (Mistral-7B & Yi-34B):** As shown in **Figure 10a** (`openchat_sharegpt4`), Sarathi-Serve achieves **2.78×** higher capacity than vLLM for Mistral-7B and **4.00×** higher capacity for Yi-34B under strict SLOs.
    *   *Why this matters:* In vLLM, increasing the load introduces long prefills that stall ongoing decodes, violating the strict 0.1s–0.2s TBT target. Sarathi-Serve, by chunking these prefills, maintains the TBT target even at 4× the load.
*   **Long Prompt Workloads:** In **Figure 10b** (`arxiv_summarization`), where prompts are significantly longer, the gains persist but are slightly moderated (**1.82×** for Mistral, **1.69×** for Yi). This confirms that while chunking helps, extremely long prompts still impose overhead, yet Sarathi-Serve remains superior.

#### Scaling to Massive Models: The Pipeline Parallelism Breakthrough
The evaluation of **Falcon-180B** (**Figure 11** and **Figure 13**) demonstrates Sarathi-Serve's ability to make Pipeline Parallelism (PP) viable on commodity networks.
*   **The TP Bottleneck:** **Figure 13a** shows that using pure Tensor Parallelism (TP-8) across nodes for Falcon-180B results in a median TBT **>2× higher** than a hybrid TP-PP configuration. The all-reduce operations over Ethernet are simply too slow.
*   **The PP Bubble Problem:** While hybrid TP-PP reduces communication cost, standard schedulers (vLLM) suffer from pipeline bubbles due to irregular batch times. **Figure 13b** shows that under strict SLOs, vLLM's capacity drops precipitously.
*   **Sarathi-Serve's Solution:** By enforcing uniform batch sizes via chunking, Sarathi-Serve eliminates these bubbles.
    *   Result: Sarathi-Serve achieves a **5.54×** capacity gain over Orca and **4.69×** over vLLM for Falcon-180B on `openchat_sharegpt4` under strict SLOs (**Figure 11a**).
    *   This is a transformative result: it allows serving a 180B parameter model on commodity Ethernet with interactive latencies, a feat previously impractical.

#### The Throughput-Latency Curve
**Figure 12** provides a granular view of the tradeoff by sweeping the TBT SLO from 0.1s to 0.5s.
*   **vLLM's Plateau:** The capacity of vLLM remains flat regardless of the max batch size setting (32, 64, or 128). This indicates that **generation stalls** are the bottleneck, not memory capacity. Even with PagedAttention allowing large batches, the scheduler cannot *use* those batches without violating latency targets.
*   **Sarathi-Serve's Scalability:** Sarathi-Serve shows a clear upward trajectory.
    *   At a stringent **100ms SLO** (Mistral-7B), Sarathi-Serve (with token budget $\tau=512$) delivers **3.5×** higher capacity than vLLM.
    *   At a relaxed **1s SLO** (Yi-34B), increasing the token budget to $\tau=2048$ allows Sarathi-Serve to operate more efficiently, yielding **1.65×** higher capacity.
    *   *Key Insight:* The system allows operators to *tune* the tradeoff via the token budget parameter, whereas vLLM is stuck on a suboptimal curve.

### 5.3 Ablation Studies: Isolating the Mechanisms

To prove that the performance gains stem from the specific combination of **chunked-prefills** and **stall-free batching**, the authors conducted ablation studies isolating each component.

#### The Cost of Chunking (Overhead Analysis)
Does splitting prefills introduce prohibitive overhead due to repeated KV-cache loads?
*   **Methodology:** The authors measured the prefill runtime for Yi-34B with varying chunk sizes (512, 1024, 2048) against a no-chunking baseline (**Figure 14**).
*   **Result:**
    *   With a chunk size of **512**, the overhead is moderate, peaking at **~25%** for shorter prompts.
    *   With a chunk size of **2048**, the overhead becomes **negligible**.
*   **Interpretation:** This validates the hypothesis in Section 3.1: since prefills are compute-bound, the extra memory bandwidth required to reload KV-cache for chunks does not significantly impact total runtime. The "free" compute slack in decode batches easily absorbs this minor overhead.

#### Component Isolation: Why Both Are Needed
**Table 4** presents a critical ablation comparing three configurations on Yi-34B:
1.  **Hybrid-Batching-Only:** Mixing full prefills and decodes (like Orca) without chunking.
2.  **Chunked-Prefills-Only:** Using chunks but without the strict stall-free priority logic.
3.  **Sarathi-Serve (Combined):** Both techniques applied.

| Configuration | P50 TTFT (s) | P99 TBT (s) | Analysis |
| :--- | :--- | :--- | :--- |
| **Hybrid-Batching-Only** | 0.53 | **0.68** | **Fails on Latency:** Without chunking, long full-prefills still cause generation stalls, spiking the P99 TBT. |
| **Chunked-Prefills-Only** | **1.04** | 0.17 | **Fails on TTFT:** Without prioritizing decodes, new chunked prefills can delay the *start* of generation, hurting Time-To-First-Token. |
| **Sarathi-Serve** | **0.76** | **0.14** | **Optimal:** Combines low TBT (from chunking) with balanced TTFT (from priority scheduling). |

*   **Conclusion:** Neither technique alone is sufficient. Hybrid batching without chunking suffers from stalls; chunking without strict prioritization hurts initial responsiveness. Only the combination achieves the dual goals.

### 5.4 Critical Assessment and Limitations

While the results are robust, a critical reading reveals specific conditions and trade-offs inherent to the design.

#### Strengths of the Evaluation
*   **Realistic Stress Testing:** The use of `arxiv_summarization` effectively stresses the "long prompt" weakness of prior systems. The fact that Sarathi-Serve maintains gains even with 7k+ token median prompts is strong evidence of robustness.
*   **Commodity Network Validation:** The Falcon-180B experiment on 100 Gbps Ethernet is a standout contribution. Many systems assume NVLink-level bandwidth; proving that PP can work efficiently on Ethernet via bubble elimination is a significant practical advancement.
*   **SLO-Driven Metrics:** By defining capacity relative to SLO violations rather than raw throughput, the evaluation aligns with actual production requirements.

#### Conditions and Trade-offs
*   **The Token Budget Tuning Burden:** The system's performance hinges on selecting the correct token budget ($\tau$).
    *   If $\tau$ is too small, chunking overheads (KV-cache reloads) accumulate, reducing efficiency.
    *   If $\tau$ is too large, latency SLOs may be violated.
    *   The paper relies on offline profiling (using the `Vidur` simulator) to set this. In highly dynamic environments where workload characteristics shift rapidly, a static $\tau$ might be suboptimal. The authors acknowledge this, leaving dynamic adjustment for future work.
*   **Tile Quantization Sensitivity:** The evaluation notes a **32% performance penalty** if the chunk size misaligns with GPU tile dimensions (e.g., 257 vs. 256 tokens). This implies that operators must be careful when configuring $\tau$ to match hardware specifics, adding a layer of operational complexity.
*   **TTFT vs. TBT Balance:** As seen in **Table 4**, there is a slight tension between TTFT and TBT. While Sarathi-Serve improves both compared to baselines, the "Chunked-Prefills-Only" variant showed that aggressive chunking can delay the first token if not managed by the priority scheduler. Users prioritizing *instant* first-token response above all else might need to tune the admission policy for new requests carefully.

#### Missing Elements
*   **Preemption Comparison:** The paper mentions **FastServe** [72] in related work, which uses preemption to handle head-of-line blocking. The evaluation does not include a direct quantitative comparison against a preemptive scheduler. It is unclear if Sarathi-Serve's "non-preemptive chunking" offers advantages over "preemptive full-prefills" in terms of context switch overhead.
*   **Disaggregated Architectures:** The authors explicitly defer comparison to disaggregated systems (like **SplitWise** or **DistServe**) where prefill and decode run on separate hardware clusters. While Sarathi-Serve optimizes *co-located* serving, it is unknown how it compares to the architectural approach of physically separating the phases, which eliminates interference entirely but requires KV-cache migration.

### Final Verdict on Experimental Claims
The experiments convincingly support the paper's central claim: **Sarathi-Serve breaks the throughput-latency tradeoff.**
The data clearly shows that baseline systems hit a "latency wall" where increasing load causes SLO violations due to generation stalls. Sarathi-Serve pushes this wall significantly further (up to **5.6×** capacity) by smoothing the computational load. The ablation studies rigorously prove that this gain is not accidental but the direct result of the synergy between **chunked-prefills** and **stall-free scheduling**. The successful deployment of Falcon-180B on commodity Ethernet stands as a particularly powerful validation of the system's ability to optimize pipeline utilization.

## 6. Limitations and Trade-offs

While Sarathi-Serve demonstrates significant improvements in serving capacity and latency stability, its design relies on specific hardware characteristics and workload assumptions that introduce trade-offs. Understanding these limitations is crucial for operators deciding whether to deploy this system in production environments.

### 6.1 The Overhead of Repeated KV-Cache Access
The most direct computational cost of **chunked-prefills** is the increased memory bandwidth consumption required to re-process attention contexts.
*   **The Mechanism:** In a standard monolithic prefill, the KV-cache for a prompt is computed once in a single pass. In Sarathi-Serve, if a prompt is split into $N$ chunks, the system must reload the KV-cache entries of all previous chunks for every subsequent chunk to compute self-attention correctly.
*   **The Cost:** As detailed in **Section 4.3**, if a prompt is split into $N$ chunks, the KV-cache for the first chunk is loaded $N-1$ times, the second chunk $N-2$ times, and so on. This results in redundant memory reads from High Bandwidth Memory (HBM).
*   **The Trade-off:** The paper argues this overhead is acceptable because prefill operations are **compute-bound** (saturating GPU math units) rather than memory-bound. **Figure 14** quantifies this trade-off:
    *   With a small chunk size (e.g., **512 tokens**), the overhead can reach **~25%** for shorter prompts compared to a non-chunked baseline.
    *   With larger chunk sizes (e.g., **2048 tokens**), the overhead becomes negligible.
*   **Implication:** Operators facing extremely memory-constrained scenarios or models with exceptionally large KV-cache footprints might find that the bandwidth penalty of aggressive chunking (small $\tau$) outweighs the latency benefits, particularly if the workload consists mostly of short prompts where the relative overhead is highest.

### 6.2 Sensitivity to Hardware Tile Quantization
Sarathi-Serve's performance is highly sensitive to the alignment of its **token budget** ($\tau$) with the underlying GPU's matrix multiplication tile sizes.
*   **The Mechanism:** GPUs execute matrix multiplications by partitioning work into fixed-size tiles (thread blocks). If the input dimension (number of tokens) is not a multiple of the tile size, the GPU must perform "padding" computations that yield no useful result, wasting cycles.
*   **The Evidence:** **Section 4.3** highlights a stark example where choosing a chunk size of **257 tokens** instead of **256 tokens** increases prefill computation time by **32%** due to tile quantization effects.
*   **The Constraint:** This imposes a rigid operational constraint: the token budget $\tau$ cannot be chosen solely based on latency SLOs; it must also be an integer multiple of the hardware's optimal tile dimension.
*   **Implication:** This reduces the flexibility of the system. An operator cannot simply tune $\tau$ to any arbitrary value to meet a specific millisecond-level SLO. They must profile the specific GPU architecture (A100 vs. A40 vs. H100) to find the "sweet spots" that avoid quantization penalties, adding complexity to the deployment configuration.

### 6.3 Reliance on Offline Profiling and Static Budgets
The system's ability to guarantee latency SLOs depends entirely on the accuracy of the **token budget** ($\tau$), which is currently determined via **offline profiling**.
*   **The Assumption:** The approach assumes that the relationship between token count and iteration latency ($T_{iter}(n)$) is stable and predictable across time. The paper states in **Section 4.3** that $\tau$ is set using a one-time profiling run (leveraging the `Vidur` simulator) to find the maximum tokens that satisfy the SLO.
*   **The Limitation:** The paper explicitly acknowledges in **Section 5.1** that "dynamic varying of the token budget based on workload characteristics" is left for future work.
*   **The Risk:** In real-world production environments, noise from co-located jobs, thermal throttling, or sudden shifts in workload characteristics (e.g., a shift from short chat prompts to long code-generation prompts) could render a statically profiled $\tau$ suboptimal.
    *   If the system becomes slower than predicted, the fixed $\tau$ might cause SLO violations.
    *   If the system becomes faster, the fixed $\tau$ might be too conservative, leaving throughput on the table.
*   **Implication:** Sarathi-Serve lacks an online feedback loop to adapt $\tau$ in real-time, making it potentially less robust to dynamic environmental changes compared to systems that adjust batching parameters continuously.

### 6.4 Unaddressed Architectural Alternatives: Disaggregation
The paper positions Sarathi-Serve as a solution for **co-located** serving (running prefill and decode on the same GPU replicas), but it explicitly defers comparison to **disaggregated architectures**.
*   **The Missing Comparison:** In **Section 6 (Related Work)**, the authors discuss systems like **SplitWise** [58], **DistServe** [77], and **TetriInfer** [47], which physically separate prefill and decode phases onto different clusters of GPUs.
*   **The Trade-off:**
    *   **Disaggregation Pros:** These systems can eliminate interference entirely (no need for chunking) and optimize each phase independently (e.g., using large batches for prefill on specialized hardware). They may achieve better Time-To-First-Token (TTFT) for massive prompts.
    *   **Disaggregation Cons:** They require migrating the KV-cache from the prefill cluster to the decode cluster, which can be challenging over commodity networks, and they may under-utilize memory on the prefill nodes.
*   **The Gap:** The paper does not provide quantitative data comparing Sarathi-Serve's efficiency against a well-tuned disaggregated system. It remains an open question whether the complexity of managing chunked schedules on a unified cluster offers better cost-performance than simply buying separate hardware for prefill and decode. The authors state, "We leave a quantitative comparison between Sarathi-Serve and disaggregation-based solutions for future work."

### 6.5 Scope of Preemption and Fairness
While Sarathi-Serve solves the problem of *generation stalls* caused by long prefills, it does not address all forms of head-of-line blocking or fairness issues inherent in multi-tenant environments.
*   **No Preemption:** Unlike **FastServe** [72], which uses preemptive scheduling to pause and resume requests, Sarathi-Serve relies on non-preemptive chunking. Once a chunk is started, it runs to completion. While chunks are small, this is still a coarse-grained approach compared to fine-grained preemption.
*   **Fairness:** The paper mentions in **Section 6** that recent work by Sheng et al. [62] addresses fairness among clients. Sarathi-Serve's primary goal is aggregate throughput and tail latency; it does not explicitly introduce mechanisms to prevent starvation of specific users or to guarantee fairness across different tenants in a multi-tenant cloud setting. The "stall-free" guarantee applies to the *batch* as a whole, not necessarily to individual request prioritization beyond the decode-vs-prefill hierarchy.

### 6.6 Summary of Constraints

| Constraint Category | Specific Limitation | Evidence Source |
| :--- | :--- | :--- |
| **Computational** | Redundant KV-cache loads increase memory bandwidth usage by up to **25%** with small chunks. | **Section 4.3**, **Figure 14** |
| **Hardware** | Performance degrades sharply (**~32%**) if chunk size misaligns with GPU tile dimensions. | **Section 4.3** |
| **Operational** | Requires **offline profiling**; lacks dynamic adaptation to runtime noise or workload shifts. | **Section 4.3**, **Section 5.1** |
| **Architectural** | No comparison to **disaggregated** (prefill/decode split) architectures; co-location assumption may not be optimal for all scales. | **Section 6** |
| **Scheduling** | Does not implement **preemption** or explicit multi-tenant **fairness** mechanisms. | **Section 6** |

In conclusion, Sarathi-Serve is not a universal panacea but a highly optimized solution for a specific regime: **co-located serving on commodity hardware where predictable tail latency is paramount.** Its effectiveness diminishes if the overhead of chunking exceeds the available compute slack, if the operator cannot correctly align hyperparameters with hardware tiles, or if the deployment strategy favors physical separation of inference phases.

## 7. Implications and Future Directions

The introduction of **Sarathi-Serve** represents a paradigm shift in LLM serving architecture, moving the field away from the binary choice between "throughput-optimized" and "latency-optimized" systems. By demonstrating that **chunked-prefills** and **stall-free batching** can simultaneously maximize GPU utilization and guarantee strict tail latency Service Level Objectives (SLOs), this work fundamentally alters how we approach the scheduling of autoregressive models. The implications extend from immediate cost reductions in cloud deployments to enabling new classes of distributed inference on commodity hardware.

### 7.1 Reshaping the Serving Landscape: Breaking the Tradeoff Curve
Prior to Sarathi-Serve, the industry operated under the assumption that the **throughput-latency tradeoff** was an immutable law of LLM inference. Operators were forced to choose:
*   **High Throughput:** Use prefill-prioritizing systems (like vLLM) and accept unpredictable latency spikes (generation stalls) that degrade user experience during peak load.
*   **Low Latency:** Use decode-prioritizing systems (like FasterTransformer) and accept severely under-utilized GPUs, leading to exorbitant serving costs.

Sarathi-Serve breaks this curve. The experimental results showing up to **3.7× higher serving capacity** for Yi-34B and **5.6× for Falcon-180B** under strict SLOs (**Figure 10**, **Figure 11**) prove that high efficiency and predictable responsiveness are not mutually exclusive. This changes the economic model of LLM serving:
*   **Cost Efficiency:** Cloud providers can serve the same volume of traffic with significantly fewer GPUs, directly reducing the cost-per-token.
*   **Quality of Service (QoS):** Interactive applications (chatbots, coding assistants) can now guarantee smooth token generation even under heavy load, eliminating the "freezing" behavior that plagues current state-of-the-art systems when long prompts arrive.

### 7.2 Enabling Commodity Clusters for Massive Models
Perhaps the most transformative implication of this work is the viability of **Pipeline Parallelism (PP)** on commodity networks.
*   **The Barrier:** Previously, scaling massive models (e.g., **Falcon-180B**) across multiple nodes required expensive, high-bandwidth interconnects (like NVLink or InfiniBand) to support Tensor Parallelism (TP). Using TP over standard Ethernet resulted in prohibitive latency due to all-reduce operations (**Figure 13a**). While PP was an alternative, it suffered from severe **pipeline bubbles** caused by irregular batch execution times, rendering it inefficient for online serving.
*   **The Sarathi-Serve Solution:** By enforcing uniform compute loads via chunked-prefills, Sarathi-Serve eliminates these bubbles. The result is a **5.6× capacity gain** for Falcon-180B running on **100 Gbps Ethernet** (**Section 5.3**).
*   **Impact:** This democratizes access to massive models. Organizations no longer need to invest in specialized, hyper-cluster infrastructure to serve 100B+ parameter models with low latency. Standard data center Ethernet fabrics become sufficient, drastically lowering the barrier to entry for deploying frontier models.

### 7.3 Future Research Directions
Sarathi-Serve opens several promising avenues for future research, addressing the limitations identified in the paper and extending its core concepts.

#### Dynamic and Adaptive Token Budgeting
The current implementation relies on **offline profiling** to determine a static token budget ($\tau$) (**Section 4.3**). Future work should focus on **online adaptive control**:
*   **Real-Time Adjustment:** Developing feedback loops that dynamically adjust $\tau$ based on real-time telemetry (e.g., observed iteration latency, queue depth, and hardware thermal states). This would allow the system to react to noise, co-located workloads, or sudden shifts in prompt length distributions without violating SLOs.
*   **Request-Aware Sizing:** Instead of a global $\tau$, investigating per-request or per-batch dynamic chunk sizing that optimizes for the specific mixture of prefill and decode tokens currently in the queue.

#### Integration with Disaggregated Architectures
The paper explicitly defers comparison with **disaggregated systems** (e.g., SplitWise, DistServe) where prefill and decode run on separate hardware clusters (**Section 6**). A critical future direction is a **hybrid architecture**:
*   **The Question:** Can Sarathi-Serve's chunking logic be applied *within* a disaggregated setting? For instance, could the "decode cluster" use stall-free batching to handle variable-length outputs more efficiently, while the "prefill cluster" uses chunking to smooth out ingestion bursts before KV-cache migration?
*   **KV-Cache Migration Optimization:** Research into optimizing the transfer of incremental KV-cache chunks between nodes could combine the interference-free benefits of disaggregation with the hardware efficiency of Sarathi-Serve.

#### Fairness and Multi-Tenant Preemption
While Sarathi-Serve solves generation stalls, it does not explicitly address **fairness** among competing tenants or fine-grained **preemption** (**Section 6**).
*   **Fairness Mechanisms:** Integrating Sarathi-Serve with fairness-aware schedulers (like those proposed by Sheng et al. [62]) to ensure that no single user or tenant monopolizes the "chunk slots" in a hybrid batch.
*   **Fine-Grained Preemption:** Exploring whether the chunking mechanism can be combined with preemptive scheduling (like FastServe [72]) to pause a chunk mid-execution if a higher-priority request arrives, further reducing tail latency for critical tasks.

#### Hardware-Aware Kernel Co-Design
The sensitivity to **tile quantization** (**Section 4.3**) suggests a need for tighter co-design between scheduling and kernel implementation.
*   **Flexible Kernels:** Developing attention kernels that are less sensitive to non-aligned dimensions or that can dynamically pad inputs without the 32% performance penalty observed.
*   **Async Chunk Loading:** Optimizing the memory subsystem to overlap the loading of previous KV-cache chunks with the computation of the current chunk, potentially hiding the remaining memory bandwidth overhead of chunked-prefills.

### 7.4 Practical Applications and Downstream Use Cases
The capabilities unlocked by Sarathi-Serve directly benefit several high-value application domains:

*   **Interactive Chatbots and Agents:** Applications requiring fluid, human-like conversation (e.g., customer support, personal assistants) benefit most from the elimination of **generation stalls**. Users will no longer experience multi-second pauses between words when the system processes a long context window.
*   **Code Generation and IDE Assistants:** Coding tools often deal with large input contexts (entire files or repositories). Sarathi-Serve allows these tools to maintain low **Time-Between-Tokens (TBT)** even when analyzing thousands of lines of code, ensuring the developer's flow state is not interrupted.
*   **Long-Context Summarization and RAG:** Retrieval-Augmented Generation (RAG) systems and summarization tools frequently process documents ranging from 4k to 32k tokens. Sarathi-Serve enables these systems to run on cost-effective commodity clusters while maintaining interactive response times, making real-time document analysis feasible at scale.
*   **Multi-Tenant SaaS Platforms:** Cloud providers offering LLM APIs can achieve higher **queries-per-second (QPS)** densities on the same hardware fleet, significantly improving profit margins while adhering to strict SLAs for enterprise customers.

### 7.5 Reproduction and Integration Guidance
For practitioners and researchers looking to adopt or build upon Sarathi-Serve, the following guidance outlines when and how to deploy this approach:

*   **When to Prefer Sarathi-Serve:**
    *   **Strict Latency SLOs:** If your application requires guaranteed low tail latency (e.g., P99 TBT &lt; 200ms) for interactive use cases.
    *   **Variable Prompt Lengths:** If your workload includes a mix of short chats and long documents (high variance), where traditional schedulers would suffer from frequent stalls.
    *   **Commodity Hardware:** If you are deploying large models (70B+) across multiple nodes without access to high-bandwidth NVLink/InfiniBand interconnects. Sarathi-Serve is the superior choice for Pipeline Parallelism in these environments.
    *   **Co-located Deployment:** If you must run prefill and decode on the same GPU replicas due to memory constraints or architectural simplicity.

*   **When Alternatives Might Suffice:**
    *   **Offline Batch Processing:** If latency is irrelevant and only total throughput matters (e.g., overnight dataset processing), simpler request-level batching or specialized offline engines (like FlexGen) might offer comparable throughput with less implementation complexity.
    *   **Disaggregated Infrastructure:** If you already have a dedicated, high-bandwidth infrastructure for separating prefill and decode clusters, a disaggregated approach might offer better isolation, though a quantitative comparison is still an open research question.

*   **Integration Steps:**
    1.  **Profiling:** Before deployment, perform the necessary offline profiling (using tools like `Vidur` mentioned in **Section 4.3**) to determine the optimal **token budget ($\tau$)** for your specific hardware (A100, A40, H100) and model. Ensure $\tau$ aligns with GPU tile dimensions to avoid quantization penalties.
    2.  **SLO Definition:** Clearly define your Strict and Relaxed SLO targets based on user experience requirements (e.g., 5× vs. 25× baseline decode time).
    3.  **Codebase Adoption:** The source code is available as a fork of **vLLM**. Integrators should look for the `sarathi` directory and the custom scheduling logic in the scheduler module. Note that as a research prototype, it may lack some production features of the main vLLM branch, so careful testing in a staging environment is essential.
    4.  **Monitoring:** Implement telemetry to track the ratio of prefill chunks to decode tokens in each batch. A healthy system should show a steady stream of hybrid batches; a dominance of decode-only batches might indicate the token budget is too conservative, leaving throughput on the table.

By adopting Sarathi-Serve, the field moves closer to a future where LLM serving is both economically efficient and universally responsive, removing the technical barriers that currently limit the scale and quality of AI applications.
