## 1. Executive Summary

This paper addresses the misalignment between standard functional code evaluation (`pass@k`) and human preference in "vibe coding" by introducing `VeriCode`, a taxonomy of 30 verifiable non-functional instructions, and `Vibe Checker`, a testbed augmenting `BigCodeBench` and `LiveCodeBench` to measure instruction following. Evaluating 31 LLMs including `Gemini 2.5 Pro` and `Claude 4 Opus`, the authors demonstrate that adding five non-functional instructions causes an average `pass@1` drop of 5.85% to 6.61%, while a composite score of functional correctness and instruction following correlates best with `LMArena` human preference data. This work establishes instruction following as a critical, under-measured component for developing models that align with real-world user needs beyond mere functionality.

## 2. Context and Motivation

### The Rise of "Vibe Coding" and Its Evaluation Gap

The paper identifies a fundamental shift in how humans write code with AI assistance. The authors introduce the term **"vibe coding"** (Section 1) to describe a workflow where users leverage LLMs to generate and iteratively refine code through multi-turn natural language interactions until the solution "passes their vibe check." This is distinct from traditional one-shot code completion: the AI becomes an interactive collaborator rather than a static tool, and the user's accept/reject choice serves as real-time evaluation.

What makes a solution pass the vibe check? According to the authors, it is a **subjective preference** based on whether the code:

- Feels right and reads cleanly
- Avoids obvious issues or anti-patterns
- Preserves intent across interactions
- Maintains correct functionality

This matters because tools like GitHub Copilot and Cursor have made this collaborative workflow "rapidly becoming standard practice in modern software development" (Section 1). The problem is that **current evaluation metrics do not measure what users actually care about**.

### Why Functional Correctness Alone Is Insufficient

The dominant metric in code evaluation is `pass@k` (Section 1), which measures whether generated code passes unit tests. This captures **functional correctness** but abstracts away non-functional expectations that users routinely apply when selecting a response, including:

- Adherence to project conventions
- Documentation clarity
- Minimal and targeted edits
- Preservation of prior intent across interactions

The paper provides concrete evidence that this disconnect has real consequences. The authors cite **Copilot Arena** (Chi et al., 2025), a large-scale vibe-checking scenario where human programmers choose preferred candidate snippets. Strikingly, "rankings of code LLMs from Copilot Arena exhibit weak or negative correlations with functional scores on popular benchmarks" (Section 1). This means a model can achieve high leaderboard scores yet fail the vibe check in practice, producing code that is technically correct but misaligned with user preferences.

The problem extends beyond evaluation into training. The authors note that `pass@k` "remains a dominant verifiable reward signal in RLVR training" (Section 1), steering optimization toward an incomplete notion of code quality. If models are optimized solely for functional correctness, they will not learn to satisfy the non-functional constraints that determine whether users actually accept their output.

### Prior Approaches and Their Limitations

The paper positions itself against two branches of existing work: **instruction following research** and **code evaluation research** (Section 5).

**Instruction Following Research.** General instruction following work focuses on stress-testing models with synthetic constraints (e.g., forced word repetition) and evaluates with either deterministic checkers or LLM-as-a-judge. A prevailing trend leverages large-scale, verifiable instructions to boost capabilities via post-training (SFT and RL). However, the authors argue that "instructions in the coding domain are tied to practical software development, concerning aspects such as logic patterns, coding style, and library usage" (Section 5). General instruction following benchmarks do not capture these domain-specific constraints.

**Code Evaluation Research.** Functional correctness dominates code evaluation, from snippet-level functions to repository-level tasks. Research on non-functional requirements exists but is "a relatively small branch" (Section 5), covering aspects like adherence to task-oriented instructions, runtime efficiency, maintainability, and security. The critical limitation is **verifiability**. Prior benchmarks for code instructions "lack verifiability" (Section 5): they typically compare to ground truth with DiffBLEU or use LLM and human judgment, which is "unreliable and hard to scale."

This creates a methodological gap: without deterministic verifiers, researchers cannot reliably measure instruction following at scale, cannot use it as a training signal, and cannot establish clear correlations with human preference.

### How This Paper Positions Itself

The paper positions itself as bridging the gap between functional evaluation and human preference through **verifiable measurement**. The authors hypothesize that "instruction following is the missing piece underlying vibe check that represents human preference in coding besides functional correctness" (Abstract).

Their approach differs from prior work in three key ways:

1. **Verifiability.** Each of the 30 instructions in their `VeriCode` taxonomy is paired with an automated, deterministic verifier that returns a binary pass/fail signal (Section 2.1). This enables objective and scalable evaluation, unlike prior work relying on LLM judges or human annotation.

2. **Practice Grounding.** Instructions reflect common developer expectations and conventions, drawing on "widely used standards rather than synthetic or adversarial constraints" (Section 2.1). The initial candidate pool sources from Ruff, an industry-standard Python linter aggregating 800+ rules from popular tools (Section 2.2).

3. **Unified Measurement.** The `Vibe Checker` testbed measures both functional correctness and instruction following within the same evaluation framework, allowing the authors to study the trade-off between these dimensions and correlate them with human preference data from LMArena (Section 4.5).

The paper does not claim to replace functional evaluation but to **augment** it. As stated in the conclusion: "Our work calls for moving beyond pass@k and for optimizing both functional and non-functional qualities in future research for coding" (Section 6). This positioning is critical—the authors are not arguing that functionality is unimportant, but that it is insufficient for capturing real-world user preference in coding tasks.

## 3. Technical Approach

This is a **benchmark construction and evaluation paper** that builds a measurement system for code instruction following. The core idea is to create deterministic, verifiable non-functional instructions that can be automatically checked, then use them to augment existing code benchmarks so models can be evaluated on both functionality and instruction adherence simultaneously.

### 3.1 Reader orientation (approachable technical breakdown)

The system being built is a **testbed called Vibe Checker** that takes code generation tasks and adds verifiable non-functional constraints (like "use Google-style docstrings" or "keep lines under 88 characters"), then automatically checks whether the generated code satisfies both the original functionality requirements and the added instructions. This solves the problem that current code benchmarks only measure whether code works, not whether it follows the stylistic and structural preferences that real users care about, by creating a two-axis evaluation framework that produces binary, deterministic scores for instruction adherence.

### 3.2 Big-picture architecture (diagram in words)

The Vibe Checker system consists of four major components that operate in sequence. First, the **VeriCode Taxonomy** serves as a curated library of 30 non-functional code instructions, each paired with a deterministic verifier that returns pass/fail. Second, the **Benchmark Augmentation Pipeline** takes existing code problems from BigCodeBench and LiveCodeBench, uses an LLM selector to choose relevant instructions from VeriCode, and assigns specific parameter values to create augmented test cases. Third, the **Evaluation Engine** runs models in two modes (single-turn generation and multi-turn editing), collects their code outputs, and executes both unit tests and instruction verifiers. Fourth, the **Metrics Calculator** computes functional regression rates and instruction following scores at both instruction-level and task-level granularity, then correlates these with human preference data from LMArena. Information flows from the taxonomy through augmentation to evaluation to final correlation analysis.

### 3.3 Roadmap for the deep dive

- **VeriCode taxonomy construction** comes first because it is the foundational artifact—without understanding how the 30 instructions were selected and verified, the rest of the system lacks context.
- **Benchmark augmentation process** follows next because it explains how raw benchmark problems become Vibe Checker test cases with attached instructions.
- **Evaluation protocols (single-turn vs. multi-turn)** come third because they define how models interact with the augmented benchmarks.
- **Metric definitions and formulas** are fourth because they specify exactly what is being measured from the evaluation outputs.
- **Experimental configuration details** close the section because they ground the methodology in concrete hyperparameters and implementation choices.

### 3.4 Detailed, sentence-based technical breakdown

#### VeriCode Taxonomy Construction

The VeriCode taxonomy is constructed through a four-stage filtering pipeline that transforms 800+ linter rules into 30 verifiable instructions. The process begins with **candidate pool sourcing**, where the authors extract rules from Ruff, an industry-standard Python linter that aggregates more than 800 rules drawn from popular tools (Section 2.2). This provides high-coverage inventory of practices that users routinely follow and check in real development workflows. However, static linting inspects only code and cannot evaluate instructions that target the entire response, such as appending a JSON explanation after the code block. To close this gap, the authors add a set of instructions focusing on documentation outside the code blocks, extending coverage beyond what static analysis can capture.

The second stage is **scope and relevance filtering**, which applies a top-down consolidation to address rule overlap. The authors prioritize broader instructions over their more specific subsets, ensuring that each instruction is broadly applicable across common coding tasks and not confined to niche scenarios. This stage removes instructions that would only apply to specialized domains or rare edge cases.

The third stage is **difficulty filtering**, which screens remaining candidates to ensure they are meaningfully challenging for advanced models. Using Gemini 2.5 Flash on BigCodeBench-Hard, the authors measure instruction following rate alongside functional correctness at pass@1. Any instruction with a success rate above 90% and no degradation in pass@1 is removed, as such instructions would not discriminate between model capabilities. Borderline cases are flagged for manual review. This step focuses on non-trivial constraints that challenge state-of-the-art LLMs rather than trivial constraints that all models satisfy.

The fourth and final stage is **review and verifier implementation**, where domain experts on the author team with coding-research experience manually review the final instruction set to ensure clarity and real-world relevance. For verification, the authors prioritize linter-backed checks when available and implement deterministic tests using Abstract Syntax Tree (AST) analysis and regular expressions when no direct rule exists. All verifiers share a common interface: a testing function that returns a binary pass or fail, enabling scalable evaluation and reproducibility.

The resulting taxonomy contains **30 instructions organized into five categories** (Section 2.3). Coding Style & Conventions contains 9 instructions covering aspects like line length limits. Logic & Code Patterns contains 9 instructions covering aspects like maximum branch counts per function. Documentation & Commenting contains 6 instructions covering docstring format requirements. Error Handling & Exception Management contains 4 instructions covering exception type usage. Library & API Constraints contains 2 instructions covering library substitution requirements. The taxonomy is organized hierarchically: the root represents the overall concept of verifiable code instructions, the five categories form the top-level nodes, and the 30 individual instructions are the leaf nodes.

Each instruction specifies five necessary elements (Section 2.3). First, the **category** places the instruction within the taxonomy structure. Second, the **description** explains what the instruction requires in natural language. Third, **distinct prompts** are provided for both single-turn generation and multi-turn editing scenarios. Fourth, **configurable parameters** with recommended or supported values allow the instruction to be instantiated at different difficulty levels. Fifth, the **verification code** returns a binary score indicating pass or fail. A key feature is extensibility through the Parameters field: parameters such as `line_length`, `max_branches`, or documentation conventions allow a single instruction to generate multiple variants with different difficulty levels. This flexibility enables the set of 30 core instructions to be programmatically expanded into hundreds of distinct and checkable constraints.

Table 1 shows refined examples from the taxonomy. For Coding Style & Conventions, the prompt "Write code ensuring all lines are no longer than {line_length} characters" maps to E501 Rule with `line_length` as an integer parameter (recommended: 79 classic, 88 modern). For Logic & Code Patterns, "Ensure each function has at most {max_branches} branches" maps to PLR0912 Rule with `max_branches` as an integer parameter (recommended: 2–4). For Documentation & Commenting, "Document your code using the {convention} docstring format" maps to D Rule with `convention` as a string parameter (supported: Google, NumPy, PEP 257). For Error Handling, "Replace all aliases with the canonical OSError exception" maps to UP024 Rule with no parameters. For Library & API Constraints, "Replace all os, os.path, glob, and open with their pathlib equivalents" maps to PTH Rule with no parameters.

#### Benchmark Augmentation Pipeline

The Vibe Checker testbed augments two established benchmarks to create BigVibeBench and LiveVibeBench. BigVibeBench is adapted from BigCodeBench to cover real-world programming tasks, containing 1,140 instances. LiveVibeBench is adapted from LiveCodeBench v1–v6 to cover algorithmic and contest problems, containing 1,055 problems from May 2023 to May 2025. Each instance across both benchmarks is augmented with 5 instructions from the VeriCode taxonomy, resulting in a total of over 10,000 instruction-level evaluations (Section 4.1).

The augmentation process involves three sequential stages. First is **instruction selection**, where for each user query, the system randomly permutes the full set of 30 taxonomy instructions to form an ordered list (Section 3.1). An LLM-based selector then scans this permuted list once, deciding whether to keep or discard each instruction based on two criteria. The first criterion is **relevance**: the instruction must pertain to the query and plausibly influence the implementation. The second criterion is **non-conflict**: the instruction must not contradict any instruction already selected earlier in the pass. The accepted instructions, in this permuted order, constitute the constraint set used to evaluate all models. This ensures that each benchmark problem receives a consistent set of instructions across all model evaluations.

Second is **parameter selection and validation**, where the system prompts an LLM to assign specific parameter values to each selected instruction (Section 3.1). To guide this generation, the prompt includes the supported keys, types, ranges, and recommended values from the taxonomy, as well as the context of the user query, aiming for parameters that are both achievable and challenging. Finally, the generated parameters undergo a rule-based validation step: any parameter keys not explicitly defined for that instruction are removed, and any invalid values are reverted to predefined defaults. This two-step process (LLM generation followed by rule-based validation) balances flexibility with correctness.

Both Gemini 2.5 Pro and Claude 4 Opus were tested as selectors in the augmentation pipeline, yielding similar instruction-category distributions (Section 3.1). The final benchmark is augmented by Claude 4 Opus, chosen for its lower invalid-parameter rate of 0.96% versus 2.47% for Gemini 2.5 Pro. The resulting distributions show that instructions for Coding Logic, Coding Style, and Documentation are most prevalent, with Coding Logic being particularly frequent in the algorithm-focused LiveCodeBench (Figure 12 in Appendix C.1 shows full breakdown: BigVibeBench has 35.9% Coding Logic, 30.6% Coding Style, 25.0% Documentation, 6.3% Error Management, 2.2% Library Constraint; LiveVibeBench has 42.3% Coding Logic, 30.5% Coding Style, 26.3% Documentation, 0.9% Error Management, 0.1% Library Constraint).

#### Evaluation Protocols

The evaluation protocol mirrors real-world usage by providing two interactive settings that differ in how instructions are presented to the model (Section 3.2, Figure 2). **Single-turn generation** presents all selected instructions after the original query within one prompt, and the model returns a single implementation that must satisfy all constraints simultaneously. This setting tests the model's ability to integrate multiple constraints in one pass without iterative refinement. **Multi-turn editing** first elicits an initial implementation in response to the original query, then reveals the selected instructions one at a time. At each round, the model sees the full interaction history and updates the solution. The code from the last round is used for evaluation. This setting tests the model's ability to incrementally refine code while preserving prior functionality and intent.

The multi-turn setting introduces instructions sequentially rather than all at once. For a task with 5 instructions, there are 5 editing rounds after the initial generation. In round 1, the model sees the base prompt plus instruction 1 and produces revised code 1. In round 2, the model sees the base prompt, instruction 1, instruction 2, and revised code 1, then produces revised code 2. This continues through round 5. The key difference from single-turn is that the model can focus on satisfying each new instruction while building on previous revisions, rather than needing to satisfy all constraints in one generation pass.

#### Metric Definitions

For both settings, the system evaluates code on two axes: functionality and instruction following (Section 3.2). **Functionality** is measured with unit tests, reporting functional regression $FR_k$ from adding $k$ instructions. Let $S_k$ denote the functional score (typically pass@1) after injecting $k$ instructions, with $S_0$ the score on the original prompt without added instructions. The regression rate is calculated as:

$$FR_k = \frac{S_0 - S_k}{S_0}$$

This formula expresses the proportional drop in functional correctness when instructions are added. A positive $FR_k$ indicates degradation (the model performs worse with instructions), while a negative value indicates improvement (rare, but possible if instructions inadvertently help). For example, if $S_0 = 50\%$ and $S_5 = 45\%$, then $FR_5 = \frac{50 - 45}{50} = 10\%$ regression.

**Instruction Following** is reported at two granularities (Section 3.2). For a task with $k$ instructions, let $I_j \in \{0, 1\}$ indicate whether instruction $j$ passes its verifier, where 1 means pass and 0 means fail. The **instruction-level score** averages per-instruction passes:

$$IF_{instruction} = \frac{1}{k} \sum_{j=1}^{k} I_j$$

This metric measures what fraction of individual instructions the model satisfies, allowing partial credit. The **task-level score** requires all passes for success:

$$IF_{task} = \mathbb{1}\left[\sum_{j=1}^{k} I_j = k\right]$$

where $\mathbb{1}[\cdot]$ is the indicator function that returns 1 if the condition is true and 0 otherwise. This metric is stricter: a task counts as successful only if every instruction passes. The task-level score is more aligned with real-world expectations where users typically reject code that violates any stated constraint.

Here, a **task** refers to a benchmark problem together with its selected instruction set. Each task produces one $IF_{task}$ score and $k$ instruction-level signals that aggregate into one $IF_{instruction}$ score.

#### Experimental Configuration

The authors evaluate 31 LLMs spanning 10 distinct model families (Section 4.1). The families include Gemini, Claude, OpenAI (GPT and o-series), DeepSeek, Qwen, Grok, Gemma, Mistral, MiniMax, and Kimi. Table 4 in Appendix D.1 lists comprehensive details including LMArena designations and Elo ratings from the September 18, 2025 leaderboard, presented under two conditions: with Style Control (w/ SC) and without (w/o SC).

All models are queried via the Vertex AI and OpenRouter APIs (Section 4.1). During benchmark augmentation, the system uses a deterministic temperature of 0.0 to ensure consistent instruction selection. During evaluation, the system follows the defaults of the underlying benchmarks: 0.0 for BigVibeBench and 0.2 for LiveCodeBench. For Claude models with thinking mode enabled, the API requires temperature 1.0, so the authors set it accordingly; all other models use the benchmark defaults. The system enables thinking mode on all models that support it, as this affects reasoning behavior on complex constraints. The context length is set to each model's supported maximum, capped at 32,768 tokens to ensure all models can process the full prompt including base query, instructions, and interaction history.

On LiveCodeBench, models demonstrate a significantly higher rate of failure to generate complete responses (Appendix D.1). These failures are attributed to either OpenRouter provider errors or exceeding the 32,768-token limit. In the experiments, each task is attempted up to three times, and a persistent failure is recorded as an error. To ensure reliability of results, the authors exclude models with an error rate exceeding 10%. Consequently, the LiveCodeBench analysis is conducted on the remaining 24 LLMs rather than the full 31.

#### Human Preference Correlation Methodology

To correlate Vibe Checker metrics with human preference, the authors use LMArena coding subset, which contains over 800,000 human votes aggregated into Elo ratings for each model (Section 4.5). They take the latest default Elo ratings from this subset and compute correlations against two metrics derived from Vibe Checker. **Func** is defined as pass@1 on the original problems without added instructions. **IF** is taken from the single-turn setting under one instruction (to isolate instruction following capability from multi-instruction complexity).

The authors then evaluate a composite score $\alpha IF + (1 - \alpha) Func$ with $\alpha \in [0, 1]$, where $\alpha$ represents the weight on instruction following and $(1 - \alpha)$ represents the weight on functionality. They report both Pearson and Spearman correlation coefficients between this composite score and LMArena Elo ratings. The optimal $\alpha$ is identified as the value that maximizes correlation, indicating the blend of functionality and instruction following that best predicts human preference.

Figure 5 shows that across both benchmarks, the peak correlation occurs at intermediate $\alpha$ values, indicating that neither IF nor Func alone explains preference as well as their combination. On BigVibeBench, the optimum for Pearson correlation places a 40% weight on IF ($\alpha = 0.4$), while for Spearman correlation, the weight on IF rises to 70% ($\alpha = 0.7$). The optimal blend for LiveVibeBench is remarkably similar. In all cases, the mixture outperforms either isolated metric by a clear margin. Additional correlation types and results with LMArena style control disabled are reported in Appendix E.2, with conclusions remaining consistent.

## 4. Key Insights and Innovations

This section distills the paper's most significant contributions — not just what the authors built, but what they *discovered* about code evaluation and model behavior. These insights challenge prevailing assumptions in the code LLM community and provide actionable direction for future research.

### 4.1 The Functionality-Instruction Following Trade-off Is Real and Measurable

**What prior work assumed:** The code evaluation community has operated under an implicit assumption that functional correctness and non-functional quality are independent dimensions. A model that passes unit tests should be able to also follow stylistic constraints without degradation — or at worst, the trade-off was considered negligible.

**What this paper reveals:** Adding non-functional instructions systematically degrades functional performance, even though those instructions do not target functionality at all. This is not a minor effect. As shown in **Table 2**, under five instructions in multi-turn editing on BigVibeBench, *every* top model shows regression above 5% except Gemini 2.5 Flash and Claude 4 Opus. On LiveVibeBench, the effect is amplified: models like `o4 mini` and `Kimi K2` exceed 10% regression in more than half of test configurations.

The average pass@1 drop of **5.85% to 6.61%** across benchmarks (Section 4.2) represents a meaningful performance penalty. To put this in perspective: if a model scores 50% pass@1 on a benchmark, adding five non-functional instructions reduces it to approximately 47% — a gap large enough to change leaderboard rankings.

**Why this is a fundamental innovation:** This is the first work to *quantify* the cost of instruction following in code generation. Prior instruction following research focused on general domains (e.g., word repetition constraints) without measuring downstream task performance. Prior code evaluation research measured functionality in isolation. Vibe Checker reveals these dimensions are coupled: optimizing for one degrades the other.

This finding has direct implications for training. If `pass@k` remains the dominant reward signal in RLVR training (as the authors note in Section 1), models will learn to sacrifice instruction adherence for functional correctness — precisely the behavior that causes them to fail the "vibe check" in practice. The trade-off must be explicitly modeled during optimization.

### 4.2 Instruction Following Emerges as the Primary Differentiator Among Advanced Models

**What prior work assumed:** Leaderboards and benchmarks rank models primarily by functional correctness. The assumption is that as models improve, functionality separates the best from the rest. Non-functional qualities are treated as secondary polish.

**What this paper reveals:** On real-world programming tasks (BigVibeBench), instruction following is *more* correlated with human preference than functional correctness alone. As shown in **Figure 5**, for Spearman correlation on BigVibeBench, pure IF ($\alpha = 1$) correlates over **0.1 points higher** with human preference than pure Func ($\alpha = 0$). This is a substantial gap in correlation analysis.

However, the insight is nuanced by task type. On algorithmic programming tasks (LiveVibeBench), the opposite holds: pure functionality maintains a clear advantage. This suggests that **user priorities shift based on context**:
- Real-world development: users value adherence to conventions, documentation, and style
- Competitive programming: users prioritize correctness above all

**Why this is a fundamental innovation:** This is the first empirical evidence that instruction following — not functionality — distinguishes top models on practical coding tasks. The finding validates the "vibe check" hypothesis: users reject technically correct code that violates their non-functional expectations.

The implication is that model developers optimizing for real-world deployment should prioritize instruction following capabilities, not just pass@k. A model that scores slightly lower on functionality but significantly higher on IF may be more useful in practice.

### 4.3 Interaction Mode Fundamentally Alters Model Behavior, Not Just Convenience

**What prior work assumed:** Single-turn and multi-turn evaluation are treated as interchangeable protocols — different ways of querying the same capability. Multi-turn is often viewed as a convenience feature for users, not a fundamentally different evaluation regime.

**What this paper reveals:** The two modes produce systematically different trade-offs. As illustrated in **Figure 3**:
- **Single-turn generation** better preserves functionality (average regression 5.76% with five instructions on BigVibeBench) but follows fewer instructions
- **Multi-turn editing** achieves higher instruction following (3% to 8% advantage in task-level IF score) at the cost of more functional regressions (9.31% average with five instructions)

This is not a marginal difference — it represents a **qualitative shift in model behavior**. In single-turn, models appear to prioritize maintaining overall correctness, potentially ignoring some constraints. In multi-turn, models focus on satisfying each new instruction but accumulate functional errors across iterations.

**Position bias compounds the effect.** As shown in **Figure 4**, models exhibit a "lost-in-the-middle" pattern on BigVibeBench despite prompts being only a few hundred tokens long. Single-turn shows primacy bias (best on first instruction), while multi-turn shows recency bias (best on last instruction). Mid-position instructions are followed less reliably in both modes.

**Why this is a fundamental innovation:** This finding challenges how the community designs evaluation protocols. Choosing single-turn vs. multi-turn is not neutral — it measures different capabilities and produces different model rankings. A model that excels in single-turn may underperform in multi-turn, and vice versa.

For practitioners, this means evaluation should match deployment context. If a tool will be used iteratively (like Copilot or Cursor), multi-turn evaluation is more predictive of real-world performance. If one-shot generation is the use case, single-turn metrics matter more.

### 4.4 Verifiable Non-Functional Metrics Can Predict Human Preference at Scale

**What prior work assumed:** Human preference in coding is subjective and difficult to measure at scale. Prior benchmarks either rely on LLM-as-a-judge (unreliable) or human annotation (expensive and unscalable). Non-functional qualities were considered too nuanced for deterministic verification.

**What this paper reveals:** A composite of functional correctness and verifiable instruction following correlates better with LMArena human preference than either metric alone. The optimal blend varies by correlation type but consistently falls at intermediate $\alpha$ values (**Figure 5**):
- BigVibeBench Pearson: $\alpha = 0.4$ (40% IF, 60% Func)
- BigVibeBench Spearman: $\alpha = 0.7$ (70% IF, 30% Func)
- LiveVibeBench shows remarkably similar optima

This demonstrates that **deterministic verifiers can capture subjective preference**. The 30 instructions in VeriCode, though narrow in scope, collectively approximate what users mean when code "feels right."

**Why this is a fundamental innovation:** This is the first work to bridge verifiable metrics and human preference in code evaluation. Prior work treated these as separate tracks: either build verifiable benchmarks (focused on functionality) or collect human preference data (expensive, noisy). Vibe Checker shows these can be unified.

The practical implication is significant: researchers can now optimize models for human preference *without* collecting new human annotations. The VeriCode verifiers provide a scalable reward signal for training that correlates with what users actually want. This opens a path for RLVR training that optimizes for vibe check alignment, not just pass@k.

### 4.5 Summary: Incremental vs. Fundamental Contributions

| Contribution | Type | Significance |
|-------------|------|--------------|
| VeriCode taxonomy (30 verifiable instructions) | Incremental build on prior linter work | Enables deterministic IF measurement |
| Vibe Checker testbed | Incremental augmentation of existing benchmarks | Provides unified evaluation framework |
| Functionality-IF trade-off quantification | **Fundamental discovery** | First evidence that non-functional constraints degrade functionality |
| IF as primary differentiator on real-world tasks | **Fundamental discovery** | Challenges functionality-centric model ranking |
| Interaction mode behavioral differences | **Fundamental discovery** | Shows single-turn and multi-turn measure different capabilities |
| Verifiable metrics predict human preference | **Fundamental discovery** | Bridges objective measurement and subjective preference |

The taxonomy and testbed are engineering contributions — necessary infrastructure but not conceptually novel. The **four empirical discoveries** above are the paper's true innovations. They reshape how we understand code evaluation, model behavior, and the relationship between measurable metrics and human judgment.

## 5. Experimental Analysis

This section walks through the experimental evidence that supports the paper's central claims. We examine not just what numbers were reported, but what they reveal about model behavior, where the evidence is strongest, and where limitations remain. The analysis is structured to mirror the evaluation pipeline: setup, functionality results, instruction following results, position effects, and human preference correlation.

### 5.1 Experimental Setup and Scope

**Model Coverage.** The authors evaluate 31 LLMs spanning 10 distinct model families (Section 4.1). This is a comprehensive cohort that includes frontier models from major providers:

- **Gemini family:** Gemini 2.5 Pro, Gemini 2.5 Flash, Gemini 2.0 Flash, Gemini 2.0 Flash Lite
- **Claude family:** Claude 4 Opus, Claude 4 Sonnet, Claude 3.7 Sonnet, Claude 3.5 Sonnet, Claude 3.5 Haiku, Claude 3 Haiku
- **OpenAI family:** GPT 5, o4 mini, o3 mini high, GPT 4.1, GPT 4.1 mini, GPT 4o, GPT 4o mini
- **Other families:** DeepSeek (R1, V3), Qwen (3 235B, 32B, 30B A3B, 2.5 72B, 2.5 Coder), Grok (4, 3 mini beta), Gemma (3 27B, 12B), Mistral Medium 3, MiniMax M1, Kimi K2

Table 4 in Appendix D.1 provides complete details including LMArena designations and Elo ratings from the September 18, 2025 leaderboard. The inclusion of both "thinking" models (e.g., Claude 4 Opus with thinking mode) and standard models allows the authors to examine whether extended reasoning improves instruction following.

**Benchmark Construction.** The Vibe Checker testbed augments two established benchmarks (Section 4.1):

| Benchmark | Source | Instances | Domain | Instructions per Instance |
|-----------|--------|-----------|--------|---------------------------|
| BigVibeBench | BigCodeBench | 1,140 | Real-world programming tasks | 5 |
| LiveVibeBench | LiveCodeBench v1–v6 | 1,055 | Algorithmic/contest problems | 5 |

This yields over 10,000 instruction-level evaluations total. The choice to augment existing benchmarks rather than create new ones is methodologically sound: it allows direct comparison with prior work and leverages established unit tests for functional correctness.

**Implementation Details.** Several configuration choices affect reproducibility (Section 4.1):

- **APIs:** Vertex AI and OpenRouter
- **Temperature:** 0.0 for benchmark augmentation; 0.0 for BigVibeBench evaluation; 0.2 for LiveCodeBench evaluation; 1.0 for Claude models with thinking mode (API requirement)
- **Thinking mode:** Enabled on all supporting models
- **Context length:** Model maximum, capped at 32,768 tokens

A critical data quality note: on LiveCodeBench, models show significantly higher failure rates due to OpenRouter provider errors or exceeding the token limit. Each task is attempted up to three times, and models with error rates exceeding 10% are excluded. This reduces the LiveCodeBench analysis from 31 to 24 LLMs (Appendix D.1). This exclusion criterion is appropriate for reliability but means LiveCodeBench results are not directly comparable to BigVibeBench in terms of model coverage.

### 5.2 Functional Correctness Under Instruction Constraints

The paper's first major finding is that adding non-functional instructions causes measurable functional regression. This is counterintuitive: the instructions do not target functionality, yet performance drops.

**Quantitative Results.** Table 2 reports regression rates for top-performing models. The regression rate $FR_k$ is calculated as:

$$FR_k = \frac{S_0 - S_k}{S_0}$$

where $S_0$ is the base pass@1 score and $S_k$ is the score after adding $k$ instructions.

Key numbers from Table 2 (BigVibeBench, multi-turn editing, 5 instructions):

| Model | Base pass@1 | Regression at 5 Inst |
|-------|-------------|---------------------|
| Gemini 2.5 Pro | 50.35% | 5.04% |
| Claude 4 Opus | 51.05% | 3.78% |
| GPT 5 | 46.49% | 5.46% |
| o4 mini | 52.28% | 8.05% |
| Kimi K2 | 47.19% | 6.12% |

The average regression across all models under five instructions is **5.85% on BigVibeBench** and **6.61% on LiveVibeBench** (Section 4.2). Some models show severe degradation: o4 mini exceeds 10% regression in more than half of LiveCodeBench test configurations.

**Single-Turn vs. Multi-Turn Comparison.** Figure 3a illustrates the divergence between interaction modes. On BigVibeBench:

- Single-turn average regression: 2.48% (1 instruction) → 5.76% (5 instructions)
- Multi-turn average regression: 3.18% (1 instruction) → 9.31% (5 instructions)

The multi-turn setting produces **~62% more regression** at five instructions (9.31% vs. 5.76%). This is a critical finding: iterative editing, while more natural for real-world workflows, accumulates functional errors across turns.

**Assessment of Evidence.** The functional regression results are robust:

1. **Consistency:** Every top model shows regression above 5% on BigVibeBench multi-turn except Gemini 2.5 Flash and Claude 4 Opus (Table 2). This universality suggests the effect is not model-specific.
2. **Monotonicity:** Regression increases monotonically with instruction count (Figure 3a), indicating a systematic relationship rather than noise.
3. **Cross-benchmark validation:** The effect appears on both BigVibeBench (real-world tasks) and LiveVibeBench (algorithmic problems), though LiveVibeBench shows larger variance.

One limitation: the paper does not analyze *why* functionality degrades. Is it attention dilution? Constraint conflict? Token budget pressure? The authors note the phenomenon but do not provide mechanistic analysis. This is acceptable for a benchmark paper but leaves room for follow-up work.

### 5.3 Instruction Following Performance

If functionality degrades under constraints, how well do models actually follow the instructions? The answer reveals a second challenge: instruction following itself is difficult, even for frontier models.

**Task-Level Success Rates.** Table 3 reports task-level IF scores, where success requires satisfying *all* constraints simultaneously. The formula is:

$$IF_{task} = \mathbb{1}\left[\sum_{j=1}^{k} I_j = k\right]$$

This is a strict metric: one failed instruction means task failure.

Key numbers from Table 3 (BigVibeBench, single-turn, 5 instructions):

| Model | 1 Inst | 2 Inst | 3 Inst | 4 Inst | 5 Inst |
|-------|--------|--------|--------|--------|--------|
| Claude 4 Opus | 88.77% | 76.32% | 64.21% | 52.98% | 46.75% |
| GPT 5 | 82.89% | 67.63% | 54.04% | 42.98% | 34.39% |
| o4 mini | 84.82% | 70.79% | 57.11% | 47.98% | 41.32% |
| Kimi K2 | 85.00% | 68.86% | 53.68% | 41.23% | 30.18% |

The best-performing model (Claude 4 Opus) reaches only **46.75%** task-level success at five instructions on BigVibeBench. On LiveVibeBench, the same model drops to **35.17%**. The paper notes: "with three or more instructions, most advanced models fall below 50 across both benchmarks" (Section 4.3).

**Exponential Decay Pattern.** The performance decay is not linear—it is multiplicative. The authors explain: "With each added instruction, the probability of satisfying all constraints decreases multiplicatively, yielding an exponential decay in task-level success" (Section 4.3). This is mathematically expected: if each instruction has independent probability $p$ of success, task-level success is $p^k$. However, the observed decay is steeper than independence would predict, suggesting instruction interactions compound difficulty.

**Single-Turn vs. Multi-Turn Comparison.** Figure 3b shows the opposite pattern from functionality:

- BigVibeBench: Multi-turn maintains 3% to 4.5% advantage in task-level IF
- LiveVibeBench: Multi-turn advantage reaches ~8%

This creates a **fundamental tension**: multi-turn editing improves instruction following but worsens functional correctness. Practitioners must choose based on their priority:
- If correctness is paramount: single-turn generation
- If constraint adherence is paramount: multi-turn editing

**Instruction-Level vs. Task-Level.** The paper also reports instruction-level IF scores (Appendix Tables 7 and 9), which allow partial credit. At five instructions on BigVibeBench single-turn:

- Claude 4 Opus instruction-level: 85.60%
- Claude 4 Opus task-level: 46.75%

The 38.85 percentage point gap reveals that models typically satisfy *most* instructions but rarely *all* of them. This has practical implications: users may tolerate partial adherence in some contexts but not others.

**Assessment of Evidence.** The instruction following results are convincing but raise questions:

1. **Difficulty calibration:** The instructions were filtered to ensure &lt;90% success rate (Section 2.2), but the observed rates (~40-50% at five instructions) suggest they may be too hard for practical use. Are these instructions representative of real user requests, or are they stress tests?
2. **Verifier reliability:** The verifiers are deterministic (linter-based), which is a strength. However, some instructions (e.g., documentation quality) may have edge cases where the verifier fails to capture intent. The paper does not report inter-rater reliability or false positive/negative rates.
3. **Cross-model consistency:** All models show similar decay patterns, suggesting the benchmark measures a real capability rather than artifact.

### 5.4 Instruction Position Effects

The paper investigates whether instruction position affects adherence—a known phenomenon in long-context generation called "lost-in-the-middle" (Liu et al., 2024).

**Position Bias Patterns.** Figure 4 shows instruction-level IF by position (1–5) for both interaction modes:

| Setting | Best Position | Worst Position | Pattern |
|---------|---------------|----------------|---------|
| Single-Turn (BigVibeBench) | Position 1 | Position 3 | U-shape (primacy bias) |
| Multi-Turn (BigVibeBench) | Position 5 | Position 3 | U-shape (recency bias) |

On BigVibeBench single-turn, Position 1 averages ~78% IF while Position 3 drops to ~74%. On multi-turn, Position 5 peaks at ~83% while Position 3 remains lowest. The authors note this occurs "despite our prompts being only a few hundred tokens long" (Section 4.4)—the effect appears even without extreme context lengths.

**LiveVibeBench Variation.** The U-shape does not generalize to LiveVibeBench (algorithmic tasks), but the underlying positional preferences remain: single-turn favors Position 1, multi-turn favors Position 5 (Section 4.4). This suggests task type modulates but does not eliminate position effects.

**Practical Implications.** For practitioners designing prompts:

- **Single-turn:** Place critical instructions first (primacy bias)
- **Multi-turn:** Place critical instructions last (recency bias)
- **Avoid:** Mid-position instructions in both modes

The paper does not test instruction reordering as a mitigation strategy, but the findings suggest simple prompt engineering could improve IF scores by 3-5 percentage points.

**Assessment of Evidence.** The position analysis is a valuable addition but limited in scope:

1. **Sample size:** Results are averaged across all models and instructions. Per-instruction or per-model variation is not reported.
2. **Mechanism:** The paper does not explain *why* position matters in short prompts. Is it attention allocation? Training data patterns? This remains an open question.
3. **Actionability:** The findings are immediately useful for prompt design, which is a strength.

### 5.5 Correlation with Human Preference

The paper's ultimate claim is that instruction following correlates with human preference. This is tested against LMArena coding Elo ratings (800K+ human votes).

**Correlation Methodology.** The authors compute correlations between LMArena Elo and a composite score:

$$\text{Composite} = \alpha \cdot IF + (1 - \alpha) \cdot Func$$

where $\alpha \in [0, 1]$ is the weight on instruction following. They test both Pearson and Spearman correlation coefficients.

**Optimal Blends.** Figure 5 shows the correlation curves across $\alpha$ values:

| Benchmark | Correlation Type | Optimal $\alpha$ | IF Weight |
|-----------|-----------------|------------------|-----------|
| BigVibeBench | Pearson | 0.4 | 40% |
| BigVibeBench | Spearman | 0.7 | 70% |
| LiveVibeBench | Pearson | 0.4 | 40% |
| LiveVibeBench | Spearman | 0.6 | 60% |

The key finding: **peak correlation occurs at intermediate $\alpha$**, not at $\alpha = 0$ (functionality only) or $\alpha = 1$ (IF only). This validates the composite metric approach.

**Task-Type Differences.** The paper notes an important nuance (Section 4.5):

- **BigVibeBench (real-world):** Pure IF ($\alpha = 1$) correlates 0.1 points higher than pure Func on Spearman correlation
- **LiveVibeBench (algorithmic):** Pure Func holds a clear advantage over pure IF

This suggests user priorities shift by context:
- Real-world development: style, conventions, and documentation matter
- Competitive programming: correctness dominates

**Assessment of Evidence.** The correlation analysis is the paper's strongest claim but has limitations:

1. **LMArena as ground truth:** LMArena Elo is itself an aggregate of human preferences, but it is noisy and may reflect factors beyond code quality (e.g., model brand, response style). The paper acknowledges this but does not control for confounding variables.
2. **IF metric selection:** The IF used for correlation is from single-turn with one instruction (Section 4.5). This isolates IF capability but may not represent multi-instruction scenarios that users actually encounter.
3. **Causality:** Correlation does not imply causation. The results show IF *predicts* preference but do not prove that improving IF *causes* higher preference.

Appendix E.2 reports additional correlation types and results with LMArena style control disabled, with conclusions remaining consistent. This robustness check strengthens the findings.

### 5.6 Critical Assessment: Do Experiments Support the Claims?

The paper makes four central claims. We evaluate each against the experimental evidence:

| Claim | Evidence Strength | Supporting Data | Limitations |
|-------|------------------|-----------------|-------------|
| Non-functional instructions cause functional regression | **Strong** | Table 2, Figure 3a: consistent 5-10% regression across 31 models | No mechanistic explanation for *why* regression occurs |
| Instruction following is challenging for LLMs | **Strong** | Table 3: best model at 46.75% task-level IF (5 instructions) | Instructions may be harder than typical user requests |
| Single-turn vs. multi-turn alters behavior | **Strong** | Figure 3: clear divergence in functionality vs. IF trade-off | Only two modes tested; no hybrid approaches |
| Composite score correlates best with human preference | **Moderate-Strong** | Figure 5: peak correlation at intermediate $\alpha$ | LMArena Elo is noisy; correlation ≠ causation |

**Ablation Studies.** The paper includes limited ablation analysis:

- **Selector model:** Both Gemini 2.5 Pro and Claude 4 Opus tested as instruction selectors; Claude 4 Opus chosen for lower invalid-parameter rate (0.96% vs. 2.47%) (Section 3.1)
- **Style control:** Correlation analysis repeated with LMArena style control enabled/disabled (Appendix E.2); conclusions consistent
- **Model exclusion:** LiveCodeBench excludes models with >10% error rate (Appendix D.1)

Missing ablations that would strengthen the paper:
- Instruction category ablation (which of the 5 categories drives regression most?)
- Parameter sensitivity (how do `line_length` or `max_branches` values affect difficulty?)
- Verifier error analysis (false positive/negative rates)

**Failure Cases.** The paper documents several failure modes:

1. **Position bias:** Mid-position instructions consistently underperformed (Figure 4)
2. **Exponential decay:** Task-level IF collapses rapidly with instruction count (Table 3)
3. **LiveCodeBench failures:** 7 of 31 models excluded due to >10% error rate (Appendix D.1)

These are not framed as bugs but as empirical findings, which is appropriate.

**Robustness Checks.** The paper demonstrates robustness through:

- **Cross-benchmark validation:** Results consistent across BigVibeBench and LiveVibeBench
- **Cross-model validation:** 31 models from 10 families show similar patterns
- **Correlation robustness:** Multiple correlation types (Pearson, Spearman) and style control settings (Appendix E.2)

One gap: the paper does not report confidence intervals or statistical significance tests for the correlation results. Given the large sample sizes (800K+ LMArena votes, 10K+ instruction evaluations), effects are likely significant, but explicit testing would strengthen claims.

### 5.7 Summary: Experimental Validity

The experiments are well-designed and largely support the paper's claims. The key strengths are:

1. **Scale:** 31 models, 10K+ evaluations, 800K+ human preference data points
2. **Verifiability:** Deterministic verifiers enable reproducible measurement
3. **Real-world grounding:** Benchmarks adapted from established suites with real unit tests

The key limitations are:

1. **Mechanistic gaps:** The paper shows *what* happens but not *why* (e.g., why does functionality degrade?)
2. **Instruction difficulty:** VeriCode instructions may be harder than typical user requests, limiting generalizability
3. **Correlation causality:** Human preference correlation is suggestive but not definitive

For practitioners, the experimental results provide actionable guidance: expect 5-10% functional regression when adding constraints, prioritize critical instructions at prompt boundaries, and optimize for both functionality and instruction following when targeting real-world deployment. For researchers, the Vibe Checker testbed offers a foundation for deeper investigation into the functionality-IF trade-off.

## 6. Limitations and Trade-offs

No evaluation framework is complete without acknowledging its boundaries. This section examines the assumptions underlying Vibe Checker, the scenarios it does not address, and the open questions that remain. Understanding these limitations is critical for interpreting the results appropriately and for guiding future research.

### 6.1 Scope and Generalizability Constraints

**Language Limitation.** The VeriCode taxonomy is instantiated exclusively for Python (Section 2.3). The authors state: "Our current instantiation focuses on Python, the dominant language in code evaluation, but the framework is language-agnostic and can be applied to other languages using standard linters." While this is a reasonable starting choice—Python dominates code evaluation benchmarks—the claim of language-agnosticism is unproven. Different languages have different:

- Linting ecosystems (e.g., ESLint for JavaScript, RuboCop for Ruby)
- Style conventions (e.g., brace placement, naming conventions)
- AST structures affecting verifier implementation

The paper provides no evidence that the 30 instructions transfer meaningfully to other languages. An instruction like "use Google-style docstrings" is Python-specific; equivalent documentation constraints in other languages would require different verifiers. Researchers applying Vibe Checker to non-Python code cannot assume the same instruction set or difficulty profiles.

**Instruction Representativeness.** The 30 instructions were filtered for difficulty—any instruction with success rate above 90% and no pass@1 degradation was removed (Section 2.2). This creates a **selection bias toward challenging constraints**. The authors acknowledge this is intentional ("ensuring that recent advanced LLMs exhibit imperfect adherence"), but it raises a validity question: do these instructions reflect what users actually request?

In real-world "vibe coding," users may issue simpler, more common requests like "add comments" or "make it cleaner" rather than "ensure each function has at most 2 branches." The taxonomy captures verifiable constraints well but may underrepresent vague, subjective preferences that still influence user acceptance. The gap between verifiable instructions and actual user language remains unmeasured.

**Task Domain Coverage.** Vibe Checker augments two benchmarks: BigCodeBench (real-world programming) and LiveCodeBench (algorithmic/contest problems). While this covers two important domains, significant gaps remain:

- **Repository-level tasks:** No evaluation of multi-file changes, import management across modules, or refactoring existing codebases
- **Domain-specific code:** No scientific computing, web development, data engineering, or embedded systems scenarios
- **Security-critical code:** No instructions targeting security properties (e.g., input validation, cryptographic best practices)

The Library & API Constraints category contains only 2 instructions (Section 2.3), which is surprisingly sparse given how often users request specific library usage. This limits the testbed's ability to measure adherence to project-specific dependency constraints.

### 6.2 Methodological Assumptions and Gaps

**Binary Verification Assumption.** All 30 instructions use binary pass/fail verifiers (Section 2.1). This enables scalable evaluation but loses nuance. Consider documentation quality: a docstring either follows Google format (pass) or it does not (fail). But a poorly written Google-style docstring still passes, while a well-written NumPy-style docstring fails. The verifier captures format compliance, not actual documentation usefulness.

The authors prioritize verifiability over fidelity, which is a defensible trade-off for benchmark construction. However, users performing "vibe checks" likely care about documentation quality, not just format. The paper does not measure this gap between verifier signals and actual user satisfaction.

**Instruction Independence Assumption.** The benchmark augmentation pipeline selects instructions that are "non-conflicting" (Section 3.1), but this is determined by an LLM selector, not formal verification. Some instruction combinations may have subtle interactions that the selector misses. For example, a line length constraint combined with a verbose docstring requirement could create implicit tension that affects both functionality and instruction following.

The exponential decay in task-level IF (Section 4.3) suggests instructions do interact—success probability drops faster than independent events would predict. The paper notes this phenomenon but does not analyze which instruction pairs cause the most conflict. This limits actionable insights for prompt design.

**Single-Instruction IF for Correlation.** The human preference correlation analysis uses IF from single-turn with one instruction only (Section 4.5). The authors state: "IF, taken from the single-turn setting under one instruction." This isolates instruction following capability from multi-instruction complexity, but it creates a mismatch:

- Real-world vibe coding involves multiple iterative constraints
- The correlation validates single-instruction IF as a preference signal
- Multi-instruction IF (more realistic) is not correlated with human preference

This is a significant gap. If multi-instruction IF behaves differently than single-instruction IF (which the results suggest it does, given the exponential decay), then the correlation findings may not generalize to realistic usage scenarios.

**LMArena as Ground Truth.** The human preference correlation relies on LMArena Elo ratings, which aggregate 800K+ human votes (Section 4.5). However, LMArena ratings are noisy and may reflect factors beyond code quality:

- Model brand recognition
- Response formatting and explanation style
- Voting population biases (e.g., competitive programmers vs. enterprise developers)

The paper acknowledges this implicitly by testing with and without style control (Appendix E.2), but does not control for other confounding variables. The correlation shows IF *predicts* preference but does not prove that improving IF *causes* higher preference. This limits prescriptive conclusions about model optimization.

### 6.3 Computational and Scalability Constraints

**Multi-Turn Evaluation Cost.** The multi-turn editing protocol requires 5 additional API calls per task (one per instruction round) beyond the initial generation (Section 3.2). For 10,000+ instruction-level evaluations across 31 models, this represents substantial computational cost:

- Single-turn: 1 API call per task
- Multi-turn: 6 API calls per task (initial + 5 rounds)

The paper does not report total API costs or inference time, but the 6x multiplier for multi-turn evaluation is significant. For researchers with limited budgets, this may constrain the number of models or tasks that can be evaluated. The trade-off between evaluation fidelity (multi-turn better reflects real usage) and cost is not discussed.

**Token Budget Limitations.** Context length is capped at 32,768 tokens (Section 4.1). On LiveCodeBench, this caused measurable failures: "models demonstrate a significantly higher rate of failure to generate complete responses... attributed to either OpenRouter provider errors or exceeding the 32,768-token limit" (Appendix D.1). Seven of 31 models were excluded from LiveCodeBench analysis due to >10% error rates.

This exclusion criterion ensures reliability but introduces selection bias. The excluded models may systematically differ from included models (e.g., smaller context windows, different tokenization). The paper does not analyze whether excluded models would have shown different IF or functionality patterns.

**Verifier Implementation Overhead.** Each of the 30 instructions requires a deterministic verifier implementation (Section 2.2). While 27 use Ruff linter rules, 3 require custom AST analysis or regular expressions. Extending the taxonomy to new instructions or languages requires:

- Domain expert review for relevance
- Verifier code implementation and testing
- Difficulty calibration on target models

This creates a maintenance burden. The authors plan to publicly release the taxonomy and verifiers (footnote 4, Section 2.3), but community extension depends on sustained maintenance. Without it, the taxonomy risks becoming stale as coding conventions evolve.

### 6.4 Unanswered Mechanistic Questions

**Why Does Functionality Degrade?** The paper robustly demonstrates that non-functional instructions cause functional regression (5.85% to 6.61% average pass@1 drop, Section 4.2), but does not explain why. Potential mechanisms include:

- **Attention dilution:** Model attention split between functionality and constraints
- **Constraint conflict:** Instructions implicitly contradict functional requirements
- **Token budget pressure:** Instruction text reduces space for reasoning about functionality
- **Training objective mismatch:** Models optimized for pass@k sacrifice IF during generation

Without mechanistic analysis, practitioners cannot design targeted interventions. Should they increase context length? Adjust training objectives? Use chain-of-thought prompting? The paper identifies the problem but does not diagnose causes.

**Why Does Position Matter in Short Prompts?** The "lost-in-the-middle" effect appears despite prompts being "only a few hundred tokens long" (Section 4.4). Prior work (Liu et al., 2024) documented this effect for long contexts (thousands of tokens), but the mechanism in short prompts is unexplained. Is it:

- Training data patterns (instructions more common at prompt boundaries)?
- Attention architecture biases?
- Instruction ordering in fine-tuning data?

The paper reports the phenomenon but does not investigate causes. This limits actionable guidance beyond "put important instructions first/last."

**What Instruction Categories Matter Most?** The five categories (Coding Style, Logic Patterns, Documentation, Error Handling, Library Constraints) show different distributions across benchmarks (Figure 12, Appendix C.1), but the paper does not analyze which categories drive functional regression or IF failure most. Are documentation requirements more costly than style requirements? Do library constraints cause more conflicts than logic patterns?

This ablation would inform prioritization: if one category drives most regression, practitioners could focus optimization there. The paper treats all 30 instructions as a unified set without category-level analysis.

### 6.5 Statistical and Reproducibility Considerations

**No Confidence Intervals.** The correlation results (Figure 5) report point estimates without confidence intervals or p-values. Given the large sample sizes (800K+ LMArena votes, 10K+ instruction evaluations), effects are likely statistically significant, but explicit testing would strengthen claims. The difference between $\alpha = 0.4$ and $\alpha = 0.7$ optima for Pearson vs. Spearman correlation may or may not be meaningful—without uncertainty quantification, this is unclear.

**Temperature Variation Across Models.** Claude models with thinking mode require temperature 1.0, while other models use 0.0 or 0.2 (Section 4.1). This introduces a confounding variable: differences in IF or functionality could reflect temperature effects rather than model capability. The paper does not control for this or analyze temperature sensitivity.

**API Provider Variability.** Models are queried via Vertex AI and OpenRouter APIs (Section 4.1). API providers may implement different preprocessing, postprocessing, or model versions than the original providers. The paper does not verify that API responses match direct provider outputs, which could affect reproducibility if API implementations change.

### 6.6 Summary: Key Trade-offs

| Trade-off | Choice Made | Consequence |
|-----------|-------------|-------------|
| Verifiability vs. Fidelity | Binary verifiers | Scalable evaluation but loses nuance in instruction quality |
| Difficulty vs. Representativeness | Filtered for &lt;90% success | Challenging benchmark but may not reflect typical user requests |
| Single vs. Multi-turn | Both evaluated | Captures different behaviors but 6x cost for multi-turn |
| Python vs. Multi-language | Python only | Deep coverage of one language, unproven generalization |
| Correlation vs. Causation | Observational analysis | Shows prediction but not causal mechanisms |
| Coverage vs. Maintenance | 30 instructions | Manageable scope but limited non-functional requirement coverage |

These limitations do not invalidate the paper's contributions but define the boundaries within which the findings apply. The Vibe Checker testbed is a significant step toward aligning code evaluation with human preference, but it is a starting point rather than a complete solution. Future work should address the mechanistic gaps, expand language and domain coverage, and investigate causal relationships between instruction following and user satisfaction.

## 7. Implications and Future Directions

This paper does more than introduce a new benchmark — it challenges the foundational assumptions of how the code LLM community measures success. This section examines how `Vibe Checker` reshapes evaluation priorities, what research paths it opens, and how practitioners can apply these findings in real-world development workflows.

### 7.1 Shifting the Code Evaluation Paradigm

**From Functionality-Only to Composite Metrics.** The most significant landscape change this work enables is moving beyond `pass@k` as the sole evaluation signal. Prior to this paper, the code evaluation community operated under an implicit assumption: if code passes unit tests, it is good enough. The correlation analysis in **Figure 5** disproves this. A composite score of functional correctness and instruction following correlates better with human preference than either metric alone:

- **BigVibeBench Pearson:** Optimal at $\alpha = 0.4$ (40% IF, 60% Func)
- **BigVibeBench Spearman:** Optimal at $\alpha = 0.7$ (70% IF, 30% Func)

This finding has direct implications for how leaderboards should be constructed. A model ranking based solely on `pass@1` may misrepresent real-world utility, especially for real-world programming tasks where instruction following emerges as "the primary differentiator among advanced models" (Section 4.5).

**Redefining What Counts as Verifiable.** Prior work treated non-functional requirements as too subjective for deterministic measurement. The `VeriCode` taxonomy demonstrates that 30 common non-functional constraints — spanning coding style, logic patterns, documentation, error handling, and library usage — can be paired with binary verifiers using standard linters and AST analysis (Section 2.2). This shifts the boundary of what the community considers measurable:

> "Each instruction is paired with an automated, deterministic verifier that returns a binary pass/fail signal, enabling objective and scalable evaluation." (Section 2.1)

This verifiability is critical for training. Unlike LLM-as-a-judge approaches, deterministic verifiers can serve as reward signals in RLVR training without introducing judge model bias or scaling costs.

**Acknowledging the Functionality-IF Trade-off.** Before this work, the community lacked quantitative evidence that non-functional constraints degrade functional performance. **Table 2** shows average `pass@1` drops of 5.85% to 6.61% under five instructions — a meaningful penalty that changes how we think about optimization. Model developers can no longer assume that improving instruction following comes "for free." The trade-off must be explicitly modeled during training and evaluation.

### 7.2 Research Directions Enabled by Vibe Checker

The paper opens several concrete research paths that were previously intractable due to lack of verifiable measurement.

**Mechanistic Analysis of Functional Regression.** The paper robustly demonstrates that functionality degrades under instruction constraints but does not explain why (Section 6.4). Follow-up work could investigate:

- **Attention allocation:** Do models attend less to functional requirements when processing multiple constraints? Intervention studies using attention visualization could test this.
- **Constraint conflict detection:** Which instruction pairs cause the most functional degradation? The taxonomy's category structure (5 categories, 30 instructions) enables systematic pairwise analysis.
- **Token budget effects:** Does instruction text reduce reasoning capacity for functionality? Experiments varying instruction verbosity while holding semantic content constant could isolate this factor.
- **Training objective analysis:** Are models optimized for `pass@k` during pretraining or fine-tuning implicitly learning to ignore non-functional constraints?

**Training Methods for Instruction Following.** The verifiable nature of `VeriCode` enables new training approaches:

- **RLVR with IF rewards:** Since each instruction has a deterministic verifier, instruction following can serve as a reward signal alongside functional correctness. The paper notes that `pass@k` "remains a dominant verifiable reward signal in RLVR training" (Section 1) — this work provides the infrastructure to change that.
- **Curriculum learning by instruction count:** **Table 3** shows exponential decay in task-level IF with increasing instruction count. Training could progress from 1 instruction to 5+ instructions, matching the difficulty curve.
- **Position-aware training:** **Figure 4** reveals position bias (primacy in single-turn, recency in multi-turn). Models could be trained with randomized instruction ordering to reduce position dependence.

**Taxonomy Expansion and Cross-Language Validation.** The current instantiation focuses on Python with 30 instructions (Section 2.3). Research directions include:

- **Language extension:** Applying the framework to JavaScript (ESLint), Ruby (RuboCop), or Go (golangci-lint) using language-specific linters. The paper claims the framework is "language-agnostic" but provides no evidence.
- **Category expansion:** The Library & API Constraints category contains only 2 instructions (Section 2.3), which is sparse given how often users request specific library usage. Expanding this category would improve coverage.
- **Difficulty calibration:** The instructions were filtered for &lt;90% success rate (Section 2.2), but observed rates (~40-50% at five instructions, **Table 3**) suggest they may be too hard. Research could develop difficulty tiers for different model capability levels.

**Human Preference Grounding.** The correlation with LMArena Elo is suggestive but not definitive (Section 5.5). Follow-up work could:

- **Collect targeted preference data:** Rather than relying on aggregate LMArena votes, collect preference judgments specifically on code varying in IF and functionality to establish causal relationships.
- **Analyze preference by user type:** Do enterprise developers weight instruction following differently than competitive programmers? The BigVibeBench vs. LiveVibeBench difference (Section 4.5) suggests task type matters, but user demographics remain unexamined.
- **Qualitative preference analysis:** What specific instruction violations cause users to reject code? Eye-tracking or think-aloud studies during vibe checking could reveal which constraints matter most.

### 7.3 Practical Applications for Developers and Model Builders

**For Model Developers: Optimization Priorities.** The findings provide concrete guidance for model training and evaluation:

| Deployment Context | Priority | Rationale |
|-------------------|----------|-----------|
| Real-world development tools (Copilot, Cursor) | Instruction Following | BigVibeBench Spearman: pure IF correlates 0.1 points higher than pure Func (Section 4.5) |
| Competitive programming assistants | Functional Correctness | LiveVibeBench: pure Func holds clear advantage over pure IF (Section 4.5) |
| General-purpose coding models | Composite Optimization | Peak correlation at intermediate $\alpha$ across all settings (Figure 5) |

Model teams should incorporate instruction following metrics into their evaluation suites, not just `pass@k`. The `Vibe Checker` testbed provides a ready-to-use framework for this.

**For Practitioners: Prompt Design Strategies.** The position bias findings (**Figure 4**, Section 4.4) enable immediate prompt engineering improvements:

- **Single-turn generation:** Place critical instructions first (primacy bias). Position 1 averages ~78% IF while Position 3 drops to ~74% on BigVibeBench.
- **Multi-turn editing:** Place critical instructions last (recency bias). Position 5 peaks at ~83% while Position 3 remains lowest.
- **Avoid mid-position instructions:** Both modes show U-shape patterns with worst performance at Position 3.

These simple adjustments could improve instruction following by 3-5 percentage points without model changes.

**For Evaluators: Choosing Interaction Modes.** The single-turn vs. multi-turn comparison (**Figure 3**, Section 4.2-4.3) reveals fundamentally different trade-offs:

| Metric | Single-Turn | Multi-Turn | When to Use |
|--------|-------------|------------|-------------|
| Functional Regression (5 inst, BigVibeBench) | 5.76% | 9.31% | Single-turn if correctness is paramount |
| Task-Level IF (5 inst, BigVibeBench) | ~30-40% | ~33-44% | Multi-turn if constraint adherence is paramount |
| API Cost | 1 call per task | 6 calls per task | Single-turn if budget-constrained |

Practitioners should match evaluation mode to deployment context. If a tool will be used iteratively (like Cursor), multi-turn evaluation is more predictive of real-world performance. If one-shot generation is the use case, single-turn metrics matter more.

**For Benchmark Designers: Augmentation Strategy.** The paper's benchmark augmentation approach (Section 3.1) provides a template for extending other evaluation suites:

1. Start with established benchmarks that have unit tests (ensures functional correctness measurement)
2. Use an LLM selector to choose relevant, non-conflicting instructions from a verifiable taxonomy
3. Validate selected parameters with rule-based checks (Claude 4 Opus achieved 0.96% invalid-parameter rate vs. 2.47% for Gemini 2.5 Pro)
4. Evaluate models on both functionality and instruction following axes

This approach preserves comparability with prior work while adding the instruction following dimension.

### 7.4 Integration and Reproducibility Guidance

**When to Adopt Vibe Checker.** The testbed is most valuable in these scenarios:

- **Model selection for deployment:** If choosing between models for a real-world coding tool, `Vibe Checker` metrics predict user preference better than `pass@k` alone (Section 4.5).
- **Training signal development:** If developing RLVR training for code models, the deterministic verifiers provide scalable reward signals without LLM judge costs.
- **Ablation studies:** If testing how architectural changes affect non-functional capabilities, the 30-instruction taxonomy enables fine-grained analysis by category.

**When Alternative Approaches May Suffice.** `Vibe Checker` is not always the right choice:

- **Pure functionality focus:** If the use case is competitive programming or algorithmic problem-solving where correctness dominates, standard benchmarks like LiveCodeBench without augmentation may be sufficient (Section 4.5 shows Func advantage on LiveVibeBench).
- **Resource-constrained evaluation:** Multi-turn evaluation requires 6x API calls per task (Section 6.3). If budget is limited, single-turn evaluation captures most signal at lower cost.
- **Non-Python languages:** The taxonomy is Python-specific (Section 2.3). For other languages, practitioners must implement language-specific verifiers before adopting the framework.

**Reproducibility Checklist.** Researchers replicating or extending this work should note:

- **API configuration:** Models were queried via Vertex AI and OpenRouter APIs with specific temperature settings (0.0 for BigVibeBench, 0.2 for LiveCodeBench, 1.0 for Claude thinking mode) (Section 4.1).
- **Context length:** Capped at 32,768 tokens, which caused measurable failures on LiveCodeBench (Appendix D.1).
- **Error handling:** Tasks were attempted up to three times; models with >10% error rate were excluded from LiveCodeBench analysis (reducing from 31 to 24 models).
- **Verifier implementation:** 27 of 30 verifiers use Ruff linter rules; 3 require custom AST analysis or regular expressions (Appendix B.1).

**Known Integration Challenges.** Practitioners should anticipate:

- **Instruction conflicts:** The LLM selector filters for non-conflicting instructions, but subtle interactions may remain. Monitor for unexpected functional regression spikes with specific instruction combinations.
- **Parameter sensitivity:** Instructions like `line_length` and `max_branches` have tunable parameters (Table 1). Default values may not match project conventions — adjust parameters to match target deployment context.
- **Verifier false positives/negatives:** The paper does not report verifier error rates. For high-stakes evaluation, manually sample verifier outputs to check for edge case failures.

**Path to Community Adoption.** The authors plan to "publicly release the taxonomy together with the corresponding verifiers" (footnote 4, Section 2.3). For sustainable community use:

1. **Start with the released verifiers** rather than re-implementing — this ensures consistency with paper results.
2. **Contribute language extensions** by following the taxonomy schema (category, description, prompts, parameters, verification code) from Section 2.3.
3. **Report category-level results** in addition to aggregate scores — this enables cross-study comparison of which non-functional dimensions matter most.

### 7.5 Summary: A Foundation for Human-Aligned Code Models

This work establishes instruction following as "an essential, yet overlooked, component of code evaluation" (Section 1). The implications extend beyond benchmarking:

- **For evaluation:** Composite metrics better predict human preference than functionality alone.
- **For training:** Verifiable non-functional constraints provide scalable reward signals for RLVR.
- **For deployment:** Interaction mode (single-turn vs. multi-turn) fundamentally alters model behavior and should match use case.

The `Vibe Checker` testbed is not a final solution but a foundation. As the authors state in the conclusion: "Our work calls for moving beyond pass@k and for optimizing both functional and non-functional qualities in future research for coding" (Section 6). The research directions outlined above — mechanistic analysis, training methods, taxonomy expansion, and preference grounding — provide a roadmap for building models that truly align with how humans write and evaluate code.
