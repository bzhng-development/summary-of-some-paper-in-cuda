## 1. Executive Summary

This paper introduces CUDA-L2, a system that combines Large Language Models (LLMs) and Reinforcement Learning (RL) to automatically optimize Half-precision General Matrix Multiply (HGEMM) CUDA kernels, solving the challenge of manual tuning across vast matrix dimension spaces. Evaluated on A100 GPUs across 1,000 configurations covering dimensions from 64 to 16384, the system systematically outperforms NVIDIA's highly optimized libraries, achieving an 11.4% speedup over `cuBLASLt-AutoTuning` in offline mode and 15.9% in server mode. These results demonstrate that even heavily optimized, performance-critical kernels can be improved through LLM-guided RL automation at scales impractical for human experts.

## 2. Context and Motivation

### The Fundamental Challenge: Matrix Multiplication at Scale

Matrix multiplication (matmul) is not merely *a* operation in deep learning—it is *the* operation. In Large Language Models (LLMs), matmul accounts for a very significant portion of computation time in both training and inference (Section 1). Every attention layer, every feed-forward network (FFN) layer, every linear transformation boils down to computing $C = AB$ where the matrices can have vastly different dimensions depending on the model architecture and sequence length.

The paper focuses specifically on **Half-precision General Matrix Multiplication (HGEMM)**, which computes $C = \alpha AB + \beta C$ using 16-bit arithmetic (Section 2.1). In practice, $\alpha = 1$ and $\beta = 0$, yielding $C = AB$. An $(M, N, K)$ triplet denotes the dimension configuration where matrix $A \in \mathbb{R}^{M \times K}$ and matrix $B \in \mathbb{R}^{K \times N}$.

### Why Manual Optimization Fails at Scale

Here lies the core problem: **different matrix dimension triplets require different optimization strategies**, and these strategies rarely transfer across GPU architectures (Section 1). Consider the following compounding challenges:

- **Vast configuration space**: The paper evaluates 1,000 configurations representing all $10^3$ combinations of $M, N, K$ values from $\{64, 128, 256, 512, 1024, 2048, 4096, 8192, 12288, 16384\}$ (Abstract). This already covers dimensions used in attention and FFN layers of open-sourced models like Qwen, Llama, and DeepSeek. Manually tuning kernels for each configuration is impractical.

- **Architecture-specific constraints**: Optimization strategies that work on one GPU architecture (e.g., Ampere A100) do not necessarily transfer to another (e.g., Hopper H100 or Blackwell B200) due to different hardware characteristics like memory hierarchy, tensor core dimensions, and cache sizes (Section 1).

- **Accumulator precision matters**: Even for identical matrix dimensions on the same GPU, the optimal kernel can differ depending on whether FP16 or FP32 accumulators are used. This affects register pressure and thus requires different optimization strategies (Section 1).

The significance of this problem is evidenced by real-world impact: the TensorRT-LLM team achieved a **13% speedup** from targeted Grouped GEMM optimizations for the DeepSeek-R1 model through narrowly focused human expertise on a single target (Section 1). If such gains are possible with focused effort on one configuration, what could systematic optimization across hundreds of configurations achieve?

### Prior Approaches and Their Limitations

#### Library-Based Baselines

Before CUDA-L2, practitioners relied on several tiers of optimized libraries:

| Baseline | Description | Key Limitation |
|----------|-------------|----------------|
| `torch.matmul` | PyTorch's default matmul, dispatches to cuBLAS internally | Additional overhead from PyTorch's tensor dispatch system and memory management (Section 2.2.1) |
| `cuBLAS` | NVIDIA's highly optimized library with black-box heuristics | Hides algorithm selection; users cannot enumerate or evaluate different algorithm variants (Section 2.2.2) |
| `cuBLASLt-heuristic` | Lower-level interface using NVIDIA's heuristic API | Selects only the top-ranked algorithm (index 0) from heuristic suggestions without exhaustive testing (Section 2.2.3) |
| `cuBLASLt-AutoTuning` | Exhaustive strategy testing up to 100 algorithm candidates | Still limited to NVIDIA's predefined algorithm space; requires substantial tuning effort and expertise (Section 2.2.3) |

The key insight: even `cuBLASLt-AutoTuning`, which represents the most competitive baseline by exhaustively testing up to 100 candidates per configuration, operates within a **fixed algorithm space** defined by NVIDIA. It cannot discover fundamentally new optimization strategies outside this space.

#### LLM-Based Kernel Generation

The past year has witnessed growing interest in harnessing LLMs, especially RL-augmented LLMs, for autonomous CUDA kernel generation (Section 1). Notable prior work includes:

- **Sakana AI's AI CUDA Engineer** [6]: Focuses on kernel discovery and optimization but primarily evaluates on benchmark tasks rather than production-relevant configurations.

- **CUDA-L1** [7]: A pipelined system for optimizing kernels in KernelBench [10], which comprises 250 diverse CUDA kernel tasks. CUDA-L1 consists of three components: (1) an SFT module fine-tuning on existing LLM-generated kernels, (2) a self-supervised learning module training on successful self-generated kernels, and (3) a contrastive RL module using execution speed as reward (Section 3.1).

However, these approaches have critical limitations for HGEMM optimization:

1. **Limited generalization**: CUDA-L1's SFT stage only fine-tunes on KernelBench kernels, limiting the model's ability to generalize to other kernel types like HGEMM (Section 3.1).

2. **Outdated knowledge**: The pre-trained LLM has limited access to recent developments critical to HGEMM optimization, such as newer CUTLASS versions, the CuTe library, and new GPU architectures (Section 3.1).

3. **Single-configuration evaluation**: Existing work primarily optimizes kernels from benchmarks where each operation is evaluated on a **single, fixed configuration** (e.g., one specific input dimension). It remains unclear how these benchmark-optimized kernels translate to real-world production environments with varying matrix dimensions (Section 1).

4. **Performance gap**: To the best of the authors' knowledge, no prior work has achieved performance comparable to manually-optimized matmul kernels, particularly when compared against NVIDIA's highly-optimized cuBLAS library (Section 1).

### How CUDA-L2 Positions Itself

CUDA-L2 directly addresses these gaps through four key technical enhancements over CUDA-L1 (Section 3.2):

1. **Continued pretraining with diverse CUDA code**: Extends beyond KernelBench to include code from PyTorch, ATen, CUTLASS, NVIDIA tutorials, and web sources, enabling more general-purpose CUDA optimization capabilities (Section 3.2.1).

2. **Multi-stage RL training**: Progresses from broad (general kernel types) to specialized domains (matmul-specific kernels), allowing the model to first learn general optimization principles before focusing on HGEMM-specific strategies (Section 3.2.2–3.2.3).

3. **Comprehensive NCU profiling metrics**: Incorporates NVIDIA Nsight Compute metrics (memory throughput, SM occupancy, cache efficiency) into the context, enabling the model to learn from detailed performance characteristics rather than relying solely on end-to-end execution time (Section 3.2.3).

4. **Retrieval-augmented context**: Accommodates new knowledge or architectural characteristics not covered in the foundation model by retrieving relevant documentation and code examples during training (Section 3.2.1).

The critical distinction: CUDA-L2 is the first system to demonstrate that LLM-guided RL can **systematically outperform** NVIDIA's production-grade libraries across a **comprehensive configuration space** (1,000 configurations) rather than isolated benchmark tasks. This positions CUDA-L2 not as a replacement for manual optimization, but as a scalable automation framework that explores configuration spaces at scales impractical for human experts.

### Why This Matters Beyond Academic Benchmarks

The real-world significance extends beyond raw speedup numbers. Consider the deployment scenario:

- **Offline mode**: Kernels executed consecutively without time intervals, measuring peak throughput (Section 2.4.2).
- **Server mode**: Kernels executed at random intervals simulating real-time inference, where GPU caches can cool down between requests (Section 2.4.2).

CUDA-L2 achieves **larger speedups in server mode** (+15.9% over cuBLASLt-AutoTuning) compared to offline mode (+11.4%), indicating that the discovered optimizations are particularly valuable for production inference workloads where request patterns are unpredictable (Abstract, Section 4.1). This is not merely an academic improvement—it translates directly to reduced inference latency and cost for LLM deployments at scale.

## 3. Technical Approach

### 3.1 Reader orientation (approachable technical breakdown)

CUDA-L2 is an automated system that uses a large language model trained with reinforcement learning to write and optimize CUDA code for matrix multiplication without human intervention. The system solves the problem that manually tuning matrix multiplication kernels for every possible matrix size is impractical by having an AI systematically explore thousands of configurations and learn which optimization strategies work best for each scenario.

### 3.2 Big-picture architecture (diagram in words)

The CUDA-L2 system consists of four major components working in sequence. First, the **Foundation Model** (based on DeepSeek 671B) serves as the code generation engine, initialized through continued pretraining on diverse CUDA code from web sources and established libraries like PyTorch, ATen, and CUTLASS. Second, the **General Kernel RL Module** trains the model on approximately 1,000 CUDA kernels covering various operation types (linear algebra, convolution, reduction, attention, etc.) using execution speedup as the reward signal. Third, the **HGEMM-Specific RL Module** narrows the focus to matrix multiplication kernels across all 1,000 (M, N, K) configurations, incorporating NVIDIA Nsight Compute (NCU) profiling metrics into the training context. Fourth, the **Evaluation and Validation Pipeline** compiles generated kernels as `.cu` files, verifies correctness against FP32 CPU reference implementations, and measures execution time using CUDA events with proper synchronization to prevent timing manipulation. Information flows from the foundation model through general RL training, then to specialized HGEMM training, with each generated kernel being compiled, validated, timed, and fed back as reward signals to update the model parameters.

### 3.3 Roadmap for the deep dive

- We begin with the **training pipeline structure** because understanding the three-stage progression (continued pretraining → general kernel RL → HGEMM RL) explains how the model builds optimization capabilities incrementally rather than learning everything at once.
- We then examine the **reward function design** because the mathematical formulation reveals how CUDA-L2 balances speed, correctness, and code conciseness—critical for understanding why the model produces valid kernels rather than just fast but broken code.
- Next, we detail the **NCU profiling integration** because this distinguishes CUDA-L2 from prior work by giving the model access to intermediate performance metrics (memory throughput, cache efficiency) rather than only final execution time.
- We then explain the **correctness validation mechanism** because verifying that generated kernels produce numerically accurate results is a prerequisite for any performance claims.
- Finally, we cover the **timing measurement safeguards** because the paper explicitly addresses how prior RL systems could "hack" timing measurements, and CUDA-L2 implements specific countermeasures.
- Throughout, we reference the specific hyperparameters and configuration values (1,000 configurations, 10-second warmup, 30-second minimum evaluation, etc.) to ground the explanation in concrete implementation details.

### 3.4 Detailed, sentence-based technical breakdown

This is a systems paper that demonstrates how reinforcement learning can automate CUDA kernel optimization at a scale impossible for human experts, with the core idea being that an LLM can learn to generate superior matrix multiplication kernels by receiving execution speed as a reward signal while being penalized for numerical errors and excessive code length.

**Training Pipeline Structure and Progression**

The training process follows a deliberate three-stage progression that builds capabilities from general to specialized. The first stage performs **continued pretraining** on diverse CUDA code collected from two sources: web sources that are cleaned and segmented using rule-based filtering combined with LLM-based cleaning, and implementations from established libraries including PyTorch, ATen, CUTLASS, NVIDIA tutorials, and examples (Section 3.2.1). For web-sourced code that lacks natural instruction descriptions, the system uses Claude Sonnet 4 to generate instruction descriptions for each CUDA code snippet, creating instruction-context-code triplets for training on DeepSeek 671B (Section 3.2.1). This stage addresses CUDA-L1's limitation of only fine-tuning on KernelBench kernels, which restricted the model's ability to generalize to other kernel types like HGEMM (Section 3.1).

The second stage conducts **General Kernel RL training** on approximately 1,000 CUDA kernels covering a wide range of operations including linear algebra, convolution operations, reduction operations, element-wise operations, attention mechanisms, sampling, and uncategorized operations like embedding lookups, loss functions, gradient clipping, and optimizer steps (Section 3.2.2). Each CUDA kernel is paired with an official or successful implementation from PyTorch, ATen, or CUTLASS as a reference, and the model is trained using a contrastive RL strategy where it performs comparative analysis of previously generated CUDA variants and their execution performances (Section 3.2.2). The system adopts GRPO (Group Relative Policy Optimization) for LLM parameter updates, with rewards smoothed and clipped to alleviate reward hacking during training as suggested in prior work (Section 3.2.2).

The third and final stage performs **HGEMM-Specific RL training** by limiting generated kernels to HGEMM operations with varying configurations of M, N, and K dimensions (Section 3.2.3). This stage incorporates NVIDIA Nsight Compute (NCU) profiling metrics into the training context, enabling the model to learn from detailed performance characteristics such as memory throughput, compute utilization, warp occupancy, and cache hit rates rather than relying solely on end-to-end execution time (Section 3.2.3). Generated HGEMM kernels are parsed into `.cu` files and compiled with `nvcc`, which means the model can use CUDA C/C++, CuTe, inline PTX assembly, CUDA intrinsics, and CUTLASS templates, but cannot use Python-based DSLs like Triton (Section 3.2.3).

**Reward Function Design and Mathematical Formulation**

The reward function balances three competing objectives: maximizing execution speedup, ensuring numerical correctness, and minimizing code length. Before presenting the equation, the intuition is that the model should receive high reward for fast kernels, but this reward should be reduced if the kernel produces incorrect results or if the code is unnecessarily verbose. The reward for a generated HGEMM kernel is computed by averaging speedup scores across all test iterations, with penalties applied for numerical deviation and code length (Section 3.2.3).

The mathematical formulation is:

$$r(\text{custom}) = \frac{1}{N} \sum_{i=1}^{N} \left[ \frac{t_i^{\text{ref}}}{t_i^{\text{custom}}} - \alpha \cdot \text{diff}_i \right] - \beta L(\text{custom})$$

where $N$ represents the number of test iterations, $t_i^{\text{ref}}$ denotes the execution time of the reference kernel on iteration $i$, $t_i^{\text{custom}}$ denotes the execution time of the custom generated kernel on iteration $i$, $\text{diff}_i = \max_j |\text{out}_i^{\text{FP32}}[j] - \text{out}_i^{\text{custom}}[j]|$ represents the maximum element-wise absolute difference between the FP32 CPU reference output and the custom kernel output, $L(\text{custom})$ denotes the length of the generated code, and $\alpha > 0$ and $\beta > 0$ are penalty coefficients that control the strength of correctness and code length penalties respectively (Section 3.2.3).

The speedup term $\frac{t_i^{\text{ref}}}{t_i^{\text{custom}}}$ produces values greater than 1 when the custom kernel is faster than the reference, equal to 1 when performance is identical, and less than 1 when the custom kernel is slower. The correctness penalty $\alpha \cdot \text{diff}_i$ reduces the reward proportionally to how much the kernel's output deviates from the FP32 CPU ground truth, ensuring that speed gains cannot come at the expense of numerical accuracy. The code length penalty $\beta L(\text{custom})$ encourages concise implementations, which indirectly favors efficient abstractions like CuTe over verbose raw CUDA intrinsics for complex optimizations (Section 5.1).

**NCU Profiling Integration for Intermediate Feedback**

A critical innovation in CUDA-L2 is the incorporation of NVIDIA Nsight Compute (NCU) profiling metrics into the training context, which provides the model with intermediate performance signals beyond just final execution time. The system includes metrics such as memory throughput, compute utilization, warp occupancy, and cache hit rates in the context provided to the model during HGEMM training (Section 3.2.3). This design choice addresses a key limitation of using only end-to-end execution time as the reward signal: execution time alone does not tell the model *why* a kernel is slow or which specific resource is the bottleneck.

For example, if a kernel has low memory throughput but high compute utilization, the model can learn that memory access patterns need optimization rather than computation scheduling. If warp occupancy is low, the model can infer that register usage or shared memory allocation may be constraining parallelism. By exposing these intermediate metrics during training, CUDA-L2 enables the model to develop a more nuanced understanding of GPU performance characteristics and make targeted optimization decisions rather than blindly searching the configuration space.

**Correctness Validation Mechanism**

A custom HGEMM kernel is considered successful only if it is both executable and correct, with executability meaning the kernel successfully compiles, launches, and executes to completion within a reasonable time limit (Section 2.3). The system uses `compute-sanitizer --tool memcheck` to check for memory access violations, ensuring that generated kernels do not produce undefined behavior from out-of-bounds accesses (Section 2.3.1).

Correctness validation employs two complementary criteria because exact equivalence between GPU and CPU floating-point computations is theoretically impossible due to the non-associative nature of floating-point arithmetic where $(a + b) + c \neq a + (b + c)$ (Section 2.3.2). The first criterion is **Exact Match with Binary Inputs**, where matrices A and B are randomly generated with elements being binary in $\{0, 1\}$ (Section 2.3.2). For matrix multiplication $C = A \times B$, each output element is:

$$c_{ij} = \sum_{k=1}^{K} a_{ik} \cdot b_{kj}$$

Since $a_{ik}, b_{kj} \in \{0, 1\}$, each product is either 0 or 1, and the sum is guaranteed to be a non-negative integer (Section 2.3.2). The system computes the reference output $C_{\text{ref}}$ using FP32 on CPU, which provides exact integer results, then computes the output $C_{\text{custom}}$ using the custom kernel (Section 2.3.2). For each position $(i, j)$ where $c_{ij}^{\text{ref}} < 2048$, the system requires $c_{ij}^{\text{test}} = c_{ij}^{\text{ref}}$ exactly, and ignores positions where $c_{ij}^{\text{ref}} \geq 2048$ (Section 2.3.2).

The threshold of 2048 is derived from half-precision floating-point properties: FP16 has 10 mantissa bits plus 1 implicit leading bit, yielding 11 bits of significand precision, which means all integers in $[0, 2048)$ are exactly representable in half-precision while integers $\geq 2048$ may not be (Section 2.3.2). Crucially, since each term $a_{ik} \cdot b_{kj} \in \{0, 1\}$, the partial sums are monotonically non-decreasing, so if the final element value is below 2048, all intermediate values must also be below 2048, ensuring exactness for all intermediate steps when computing $\sum_k a_{ik} \cdot b_{kj}$ (Section 2.3.2). The system repeats this process multiple times with different random inputs, and if any single iteration fails, the kernel is considered incorrect (Section 2.3.2).

The second criterion is **Baseline-Bounded Deviation**, where the system selects a set of highly reliable baseline kernels developed by NVIDIA including cuBLAS-NN, cuBLAS-TN, cuBLASLt-heuristic-NN, cuBLASLt-heuristic-TN, cuBLASLt-AutoTuning-NN, and cuBLASLt-AutoTuning-TN (Section 2.3.2). For each test input, the system computes the maximum elementwise difference among the outputs of these baseline kernels, which reflects the upper bound of variability for floating-point computations (Section 2.3.2). A custom kernel is considered incorrect if its maximum elementwise deviation from the reference output exceeds this baseline's value (Section 2.3.2). This approach acknowledges that some numerical variation is acceptable as long as it stays within the range of variation exhibited by NVIDIA's own optimized libraries.

**Timing Measurement Safeguards Against Reward Hacking**

The paper explicitly addresses how prior RL systems could manipulate timing measurements, with prior work identifying two strategies to hack evaluation during RL training: creating additional CUDA streams that execute asynchronously, and exploiting Python's lazy mode where calling the kernel function does not ensure the output is actually materialized or computed (Section 2.4.1). CUDA-L2 avoids both issues through specific implementation constraints.

The system disallows additional CUDA stream creation, ensuring that all kernel execution happens on the default stream with proper synchronization (Section 2.4.1). Additionally, CUDA-L2 generates HGEMM kernels only as CUDA code in `.cu` files, which naturally bypasses Python's lazy evaluation that could delay actual computation until the result is accessed (Section 2.4.1). The timing code follows the standard kernel timing strategy with explicit synchronization:

```cpp
torch.cuda.synchronize()
start_event = torch.cuda.Event(enable_timing=True)
end_event = torch.cuda.Event(enable_timing=True)
start_event.record()
kernel(a, b, b_col_major, out)
end_event.record()
torch.cuda.synchronize()
elapsed_time_ms = start_event.elapsed_time(end_event)
```

This ensures that the GPU completes all prior work before timing starts, the kernel executes fully before timing ends, and no asynchronous operations can hide execution time (Section 2.4.1, Listing 1).

**Evaluation Protocol and Configuration Space**

The evaluation runs for a minimum of 30 seconds after a 10-second warmup period to account for GPU performance variability (Section 2.4). In each iteration, the system randomizes execution order to eliminate ordering effects, and the final evaluation score is the mean speed score over all runs (Section 2.4). The single-run speedup $s(\text{custom})$ is defined as:

$$s(\text{custom}) = \frac{t_{\text{ref}}}{t_{\text{custom}}} - 1$$

where $t_{\text{ref}}$ and $t_{\text{custom}}$ denote the execution time of the reference and custom kernels respectively (Section 2.4).

The configuration space covers 1,000 configurations representing all $10^3$ combinations of M, N, K values from $\{64, 128, 256, 512, 1024, 2048, 4096, 8192, 12288, 16384\}$ (Abstract, Section 1). This already covers dimensions used in attention and FFN layers of open-sourced models like Qwen, Llama, and DeepSeek (Abstract). The system evaluates under two scenarios: offline mode where kernels are executed consecutively without time intervals, and server mode where kernels are executed at random intervals simulating real-time inference (Section 2.4.2). It is worth noting that the interval time is not included in the execution time used for the speedup comparison in server mode (Section 2.4.2).

The offline scenario measures peak throughput by keeping the GPU fully busy, while the server scenario reflects real-world deployment where requests arrive at unpredictable times (Section 2.4.2). Performance differs between them because back-to-back execution in offline mode keeps GPU caches warm with the GPU running at full speed, while in server mode, the GPU's caches can cool down and then start from a cold state (Section 2.4.2). The current version of CUDA-L2 is limited to A100 Ampere architectures, though the framework is designed for broad applicability with ongoing work to extend it to additional GPU architectures including Ampere (e.g., RTX 3090), Ada Lovelace (e.g., RTX 4090), Hopper (e.g., H100), and Blackwell (e.g., B200) (Section 1).

**Design Choices and Alternative Considerations**

The decision to use a three-stage training pipeline (continued pretraining → general kernel RL → HGEMM RL) rather than training directly on HGEMM kernels addresses the cold-start problem where an LLM with limited CUDA knowledge would struggle to generate valid kernels from scratch. By first exposing the model to diverse CUDA code, then training on general kernel types, and finally specializing to HGEMM, the system builds capabilities incrementally with each stage providing a stronger foundation for the next.

The choice to compile kernels as `.cu` files rather than using Python-based DSLs like Triton enables access to lower-level CUDA features including inline PTX assembly, CUDA intrinsics, and CUTLASS templates that may not be exposed through higher-level abstractions (Section 3.2.3). This trade-off sacrifices some development convenience for maximum optimization flexibility, which is necessary when competing against NVIDIA's hand-tuned libraries.

The inclusion of code length penalty in the reward function serves a dual purpose: it encourages concise implementations that are easier to maintain and debug, and it indirectly favors efficient abstractions like CuTe over verbose raw CUDA intrinsics for complex optimizations since CuTe's abstractions result in more concise code compared to raw CUDA intrinsics (Section 5.1). This design choice aligns the model's optimization objectives with practical software engineering concerns beyond raw performance.

The use of GRPO for parameter updates rather than standard PPO or other RL algorithms follows prior work in CUDA-L1, which found that contrastive RL strategies where the model performs comparative analysis of previously generated variants and their execution performances leads to more stable training (Section 3.2.2). Rewards are smoothed and clipped to alleviate the effect of reward hacking during RL training, preventing the model from exploiting edge cases in the timing measurement system (Section 3.2.2).

## 4. Key Insights and Innovations

This section isolates the fundamental contributions of CUDA-L2 that distinguish it from prior automated kernel optimization efforts. While previous work demonstrated that LLMs could generate *valid* CUDA kernels, CUDA-L2 proves they can generate *superior* kernels compared to vendor-optimized libraries across a vast configuration space. The innovations here are not merely incremental speedups but represent shifts in how we approach kernel automation: moving from parameter tuning within fixed algorithm spaces to structural discovery within open code spaces.

### 4.1 Surpassing the Exhaustive Search Ceiling

The most significant claim of this paper is not that CUDA-L2 beats `torch.matmul` or standard `cuBLAS`, but that it systematically outperforms `cuBLASLt-AutoTuning`. This distinction is critical because `cuBLASLt-AutoTuning` represents the "exhaustive search ceiling" of NVIDIA's own library.

*   **The Baseline Constraint:** As described in Section 2.2.3, `cuBLASLt-AutoTuning` retrieves up to 100 algorithm candidates for each $(M, N, K)$ configuration and empirically selects the fastest one. This is effectively a brute-force search within NVIDIA's predefined set of optimized algorithms.
*   **The Innovation:** CUDA-L2 exceeds this baseline by **11.4% in offline mode** and **15.9% in server mode** (Abstract, Section 4.1).
*   **Why This Matters:** This result implies that NVIDIA's predefined algorithm space is **incomplete**. Even after testing 100 variants, the library misses optimization strategies that an RL-guided LLM can discover. CUDA-L2 does not select from a list; it generates code semantically, allowing it to explore implementation strategies (e.g., specific instruction scheduling, memory layout transformations) that do not exist as selectable options in the cuBLASLt API. This shifts the paradigm from "selecting the best known algorithm" to "inventing a better algorithm."

### 4.2 Discovery of Non-Standard Kernel Structures

A common misconception about automated kernel optimization is that it primarily tunes hyperparameters (e.g., block sizes, thread counts) within a fixed template. CUDA-L2 demonstrates the capability to modify the **structural logic** of the kernel itself. The paper documents specific techniques discovered by the RL agent that contradict standard optimization heuristics.

*   **Padding for Non-Divisible Tile Sizes:** Standard CUDA optimization dictates that tile dimensions (e.g., $B_M$) should divide the matrix dimension (e.g., $M$) to avoid boundary checks and wasted computation. CUDA-L2 violates this rule. In Section 5.2, for a configuration with $M=8192$, the model selects a tile size $B_M = 160$, which does not divide 8192. It instead pads the matrix to 8320 (a **1.6% overhead**).
    *   **Reasoning:** The performance gain from using a more efficient $B_M=160$ (likely due to better tensor core utilization or memory coalescing) outweighs the computational cost of processing the padded zeros. This trade-off is non-obvious and rarely explored in manual tuning.
    *   **Impact:** This specific configuration yields a **+15.2% speedup** over `cuBLASLt-AutoTuning-TN`, whereas forcing a conventional $B_M=128$ drops the speedup to **+0.4%** (Section 5.2).

*   **Staggered Prefetch Scheduling:** Standard prefetching loads both input matrices (A and B) consecutively before computation. CUDA-L2 discovers a **staggered** approach where it prefetches matrix A, executes the matrix multiply-accumulate (MMA) operation, and *then* prefetches matrix B (Section 5.3.4, Listing 5).
    *   **Reasoning:** This interleaving increases instruction-level parallelism. Since the MMA operation uses data already in registers from the current iteration, it does not depend on the new prefetches. By separating the prefetches, the model fills execution slots that would otherwise be idle, better overlapping memory and compute pipelines.
    *   **Significance:** This is a micro-architectural scheduling decision that depends on the specific balance of memory bandwidth vs. compute throughput for a given $(M, N, K)$, demonstrating the model's ability to reason about hardware pipelines.

### 4.3 Robustness in Production-like Server Environments

Many kernel optimizations are brittle: they achieve peak throughput in controlled benchmarks but degrade in real-world deployment where request patterns are irregular. CUDA-L2 shows improved robustness in **server mode**, which simulates real-time inference with random intervals between kernel executions (Section 2.4.2).

*   **The Observation:** Speedups over baselines are consistently **higher in server mode** than in offline mode. For example, against `cuBLASLt-AutoTuning`, the gain rises from **+11.4% (offline)** to **+15.9% (server)** (Abstract, Section 4.1).
*   **The Mechanism:** Section 4.1 attributes this to **GPU thermal dynamics and cache behavior**. In offline mode, continuous execution keeps the GPU in a steady thermal state with warm caches. In server mode, idle periods allow caches to cool and clocks to throttle.
*   **Why This Matters:** This indicates that CUDA-L2's optimizations are not just maximizing peak arithmetic throughput but are also improving **memory locality and cache efficiency**. Kernels that rely heavily on warm caches would suffer larger penalties in server mode. The fact that CUDA-L2 gains *more* ground in this scenario suggests its generated kernels are better suited for the unpredictable latency requirements of production LLM inference, not just offline training benchmarks.

### 4.4 Diagnostic-Driven Reinforcement Learning

Prior RL-based kernel optimization systems typically use a scalar reward signal (e.g., execution time) to guide learning. CUDA-L2 introduces a methodological innovation by incorporating **NVIDIA Nsight Compute (NCU) profiling metrics** directly into the model's context during training (Section 3.2.3).

*   **The Change:** Instead of only knowing "Kernel A is faster than Kernel B," the model receives detailed metrics such as **memory throughput, SM occupancy, warp occupancy, and cache hit rates** alongside the code.
*   **The Benefit:** This transforms the learning problem from black-box optimization to **diagnostic-driven refinement**.
    *   If the reward is low and the context shows low **memory throughput**, the model learns to focus on memory access patterns (e.g., coalescing, shared memory usage).
    *   If **warp occupancy** is low, it learns to adjust register usage or block dimensions.
*   **Significance:** This mimics how human experts optimize kernels: by profiling to identify bottlenecks before rewriting code. By providing this "debugger view" to the RL agent, CUDA-L2 accelerates convergence on effective strategies and reduces the likelihood of the model getting stuck in local optima where execution time is flat but resource utilization is suboptimal. This approach bridges the gap between high-level code generation and low-level hardware performance analysis.

## 5. Experimental Analysis

This section examines how CUDA-L2 was evaluated, what the results actually show, and whether the experiments convincingly support the paper's central claims. We will walk through the evaluation methodology step-by-step, analyze the quantitative results with specific numbers, and critically assess the strengths and limitations of the experimental design.

### 5.1 Evaluation Methodology

#### Configuration Space and Coverage

The evaluation spans **1,000 distinct configurations** representing all $10^3$ combinations of matrix dimensions $M$, $N$, and $K$ drawn from the set $\{64, 128, 256, 512, 1024, 2048, 4096, 8192, 12288, 16384\}$ (Abstract, Section 1). This configuration space is not arbitrary—it deliberately covers the matrix dimensions used in attention and feed-forward network (FFN) layers of widely deployed open-source models including Qwen, Llama, and DeepSeek (Abstract).

The choice of 1,000 configurations is significant. Prior LLM-based kernel optimization work typically evaluates on **single, fixed configurations** per kernel type (Section 1). By contrast, CUDA-L2's comprehensive coverage allows the authors to claim systematic improvement across a production-relevant space rather than isolated benchmark wins.

#### Baseline Selection Strategy

The paper compares against four tiers of baselines, each representing a different level of optimization sophistication (Section 2.2):

| Baseline | Selection Logic | What It Represents |
|----------|-----------------|-------------------|
| `torch.matmul` | PyTorch's default dispatch | Standard practitioner workflow |
| `cuBLAS` | NVIDIA's optimized library with internal heuristics | Vendor-optimized default |
| `cuBLASLt-heuristic` | Top-ranked algorithm from heuristic API (index 0) | NVIDIA's recommended tuning approach |
| `cuBLASLt-AutoTuning` | Fastest among up to 100 candidates via exhaustive testing | The "exhaustive search ceiling" within NVIDIA's algorithm space |

For each baseline, the evaluation considers three layout configurations:
- **NN (normal-normal)**: Both input matrices in row-major layout
- **TN (transposed-normal)**: Mixed layout with transposed first matrix
- **max(NN, TN)**: The better of the two layouts for each configuration

This multi-layout approach is important because layout choice can significantly affect performance, and comparing only one layout would risk unfair comparisons (Section 2.2.2, 2.2.3).

#### Timing Protocol and Safeguards

The evaluation follows a rigorous timing protocol designed to prevent reward hacking—a known issue in prior RL-based kernel optimization systems (Section 2.4.1):

- **Warmup period**: 10 seconds before measurement begins
- **Minimum evaluation duration**: 30 seconds of timed execution
- **Execution order randomization**: Each iteration randomizes kernel execution order to eliminate ordering effects
- **Synchronization**: Explicit `torch.cuda.synchronize()` calls before and after kernel execution (Listing 1, Section 2.4.1)
- **No additional CUDA streams**: All execution happens on the default stream to prevent asynchronous hiding of execution time

The speedup metric is defined as:

$$s(\text{custom}) = \frac{t_{\text{ref}}}{t_{\text{custom}}} - 1$$

where $t_{\text{ref}}$ and $t_{\text{custom}}$ denote the execution time of the reference and custom kernels respectively (Section 2.4). A speedup of 0.114 (or 11.4%) means the custom kernel runs 11.4% faster than the reference.

#### Two Deployment Scenarios

The evaluation distinguishes between two execution modes that reflect different real-world deployment patterns (Section 2.4.2):

- **Offline mode**: Kernels executed consecutively without time intervals, measuring peak throughput with warm GPU caches
- **Server mode**: Kernels executed at random intervals simulating real-time inference, where GPU caches can cool between requests

Critically, the interval time in server mode is **not included** in the execution time used for speedup comparison—only the kernel execution time itself is measured (Section 2.4.2). This ensures fair comparison while still capturing the performance degradation that occurs when caches are cold.

#### Hardware and Architecture Constraints

All experiments were conducted on **A100 GPUs** (Ampere architecture) (Abstract, Section 1). The paper explicitly acknowledges this limitation and notes ongoing work to extend the framework to other architectures including Ada Lovelace (RTX 4090), Hopper (H100), and Blackwell (B200) (Section 1). This constraint is important to keep in mind when generalizing the results—optimization strategies discovered for Ampere may not transfer directly to other architectures due to different memory hierarchies and tensor core characteristics.

### 5.2 Main Quantitative Results

#### Overall Performance Against Baselines

Table 1 and Figure 1 present the core performance results across all 1,000 configurations (Section 4.1, Abstract):

**Offline Mode (Figure 1a):**
- **+22.0%** over `torch.matmul`
- **+19.2%** over `cuBLAS-max` (optimal NN or TN layout)
- **+16.8%** over `cuBLASLt-heuristic-max`
- **+11.4%** over `cuBLASLt-AutoTuning-max`

**Server Mode (Figure 1b):**
- **+28.7%** over `torch.matmul`
- **+26.0%** over `cuBLAS-max`
- **+22.4%** over `cuBLASLt-heuristic-max`
- **+15.9%** over `cuBLASLt-AutoTuning-max`

Several observations are worth highlighting:

1. **Server mode gains exceed offline mode gains** across all baselines. For example, against `cuBLASLt-AutoTuning-max`, the speedup increases from 11.4% (offline) to 15.9% (server). Section 4.1 attributes this to GPU thermal dynamics: idle periods in server mode allow caches to cool and clocks to throttle, making efficient memory access patterns more valuable.

2. **The performance hierarchy is consistent**: `torch.matmul` < `cuBLAS` < `cuBLASLt-heuristic` < `cuBLASLt-AutoTuning` < `CUDA-L2`. Each step represents a more sophisticated optimization approach, and CUDA-L2 exceeds even the most aggressive baseline.

3. **Win rates are high**: The paper reports win rates spanning **79.3% to 95.7%** across all baselines (Section 4.1). This confirms that improvements are systematic rather than driven by a few outlier configurations where CUDA-L2 happens to excel.

#### Real-World Scenario: max(CUDA-L2, baseline)

Table 2 explores a practical scenario where users have access to both CUDA-L2 and baseline kernels and can select whichever runs faster for each configuration (Section 4.2):

**Offline Mode:**
- `torch.matmul`: 22.0% → **23.1%** (additional +1.1%)
- `cuBLAS-max`: 19.2% → **20.2%** (additional +1.0%)
- `cuBLASLt-heuristic-max`: 16.8% → **17.0%** (additional +0.2%)
- `cuBLASLt-AutoTuning-max`: 11.4% → **13.2%** (additional +1.8%)

**Server Mode:**
- `torch.matmul`: 28.7% → **29.8%** (additional +1.1%)
- `cuBLAS-max`: 26.0% → **27.2%** (additional +1.2%)
- `cuBLASLt-heuristic-max`: 22.4% → **22.7%** (additional +0.3%)
- `cuBLASLt-AutoTuning-max`: 15.9% → **18.1%** (additional +2.2%)

The key insight: even when users can cherry-pick the best kernel per configuration, CUDA-L2 still provides meaningful additional gains. The largest additional gains come against `cuBLASLt-AutoTuning-max` (+1.8% offline, +2.2% server), suggesting that CUDA-L2 discovers optimizations in configurations where NVIDIA's exhaustive search fails to find the best algorithm.

#### Speedup vs. Problem Size (Figure 3)

Figure 3 reveals an important conditional pattern: **speedup decreases as problem size increases** (Section 4.3):

- **Small problems** (log₂(M×N×K) ≈ 18-20): ~1.4× speedup (approximately 40% faster)
- **Large problems** (log₂(M×N×K) > 38): ~1.0× speedup (converges to baseline performance)

This pattern is confirmed across three different x-axis representations:
- Figure 3a: Relative speedup vs. log₂(M×N×K)
- Figure 3b: Relative speedup vs. average dimension
- Figure 3c: Relative speedup vs. maximum dimension

**Why does this happen?** Section 4.3 provides a clear explanation:

> "For small matrix multiplications, the GPU is underutilized and there is significant room for optimization through better memory access patterns, tiling strategies, and kernel configurations. CUDA-L2 exploits these opportunities effectively. On the contrary, large matrices saturate the GPU's floating-point throughput, leaving less room for improvement."

This is a critical nuance. CUDA-L2's advantages are **not uniform** across all problem sizes. For very large matrices where the GPU is already compute-bound, there is simply less headroom for improvement regardless of optimization strategy. This does not invalidate the results—it clarifies the **conditions under which CUDA-L2 provides the most value**.

### 5.3 Optimization Technique Analysis (Section 6)

The paper goes beyond reporting speedup numbers to analyze **which optimization techniques are selected under which conditions**. This analysis helps explain why CUDA-L2 succeeds and provides guidance for practitioners.

#### Tile Size Selection (Figure 4a-d)

Figure 4 reveals correlations between problem dimensions and optimal tile sizes (Section 6.1):

- **BM (M-dimension tile)**: Strong positive correlation with M dimension (ρ = 0.652). As M increases from ≤256 to >4K, BM grows from approximately 60 to 160.
- **BN (N-dimension tile)**: Even stronger correlation with N dimension (ρ = 0.705).
- **BK (K-dimension tile)**: Weak correlation with K (ρ = 0.256), indicating BK selection depends on additional factors like register pressure and memory bandwidth constraints.
- **BM-BN relationship**: High correlation (ρ = 0.695), reflecting that tensor cores operate efficiently on square or near-square tiles.

This analysis validates that CUDA-L2 learns sensible optimization patterns rather than random configurations. The tile sizes scale appropriately with problem dimensions, and the BM-BN balance reflects hardware constraints.

#### Pipeline Stage Selection (Figure 4e)

Figure 4e shows how the number of pipeline stages should vary with the K dimension (Section 6.2):

- **Small K (≤128)**: 2-3 stages suffice for adequate latency hiding
- **Large K (>8K)**: 6+ stages required to maintain high throughput by keeping multiple data loads in flight simultaneously

This demonstrates that CUDA-L2 learns to trade off register usage (more stages require more registers) against latency hiding (more stages better hide memory access latency). The model automatically discovers this trade-off rather than using a fixed pipeline depth.

#### Block Swizzle Usage (Figure 4f)

BlockSwizzle reorders thread block execution to improve L2 cache utilization (Section 6.3). Figure 4f reveals a clear threshold-based pattern:

| Problem Size | Operations | BlockSwizzle Usage |
|--------------|------------|-------------------|
| Small | < 2²⁷ (~134M) | 44% (optional) |
| Medium | 2²⁷ to 2³³ | 73-80% |
| Very Large | > 2³⁶ (~68B) | 99% (nearly universal) |

The swizzle_stride parameter also scales with problem size:
- **Smaller problems using swizzling**: stride values of 8-128
- **Larger problems**: stride values of 512-16,384

The correlation between problem size and swizzle usage (ρ = 0.453) demonstrates that CUDA-L2 learns when the overhead of block reordering is justified by improved cache locality (Section 6.3).

### 5.4 Critical Assessment of Experimental Design

#### Strengths

**Comprehensive configuration coverage**: The 1,000 configuration evaluation is substantially more thorough than prior LLM-based kernel optimization work, which typically evaluates on single fixed configurations per kernel type (Section 1). This allows meaningful claims about systematic improvement rather than isolated wins.

**Strong baseline selection**: Comparing against `cuBLASLt-AutoTuning`—which exhaustively tests up to 100 algorithm candidates—is a high bar. Many kernel optimization papers compare only against naive implementations or basic library calls. Exceeding NVIDIA's own exhaustive search within their algorithm space is a significant claim.

**Two deployment scenarios**: Evaluating both offline and server modes captures different real-world usage patterns. The fact that CUDA-L2 performs better in server mode (+15.9% vs. +11.4% against `cuBLASLt-AutoTuning-max`) suggests the optimizations are robust to cache cooling and thermal throttling—critical for production inference workloads.

**Timing safeguards**: The explicit measures to prevent reward hacking (no additional CUDA streams, `.cu` file compilation to bypass Python lazy evaluation, explicit synchronization) address known vulnerabilities in prior RL-based kernel optimization systems (Section 2.4.1).

#### Limitations and Open Questions

**No ablation studies on RL components**: The paper describes four technical enhancements over CUDA-L1 (continued pretraining, multi-stage RL, NCU metrics, retrieval-augmented context) but does not provide ablation studies isolating the contribution of each component (Section 3.2). Readers cannot determine whether all four enhancements are necessary or whether some provide marginal benefit. This is a notable gap in the experimental design.

**Single architecture evaluation**: All results are on A100 (Ampere) GPUs. While the paper notes ongoing work to extend to other architectures, the current evaluation cannot support claims about cross-architecture generalization (Section 1). Optimization strategies discovered for Ampere may not transfer to Hopper or Blackwell due to different memory hierarchies and tensor core characteristics.

**Speedup diminishes for large problems**: Figure 3 clearly shows that speedup converges to ~1.0× for very large matrices (log₂(M×N×K) > 38). This is explained by compute saturation, but it means CUDA-L2's value is **concentrated on small-to-medium problem sizes**. For practitioners working primarily with very large matrices, the benefits may be limited.

**No wall-clock training cost reported**: The paper does not specify how much compute time was required to train CUDA-L2 or how many kernel generations were needed to converge. This information is important for assessing whether the approach is practical for organizations to replicate.

**Correctness threshold details**: While the paper describes the binary input correctness test with the 2048 threshold (Section 2.3.2), it does not report how many generated kernels failed correctness checks during training or what the final success rate was. This would help assess the reliability of the RL training process.

#### Do the Experiments Support the Claims?

**Claim 1: "CUDA-L2 systematically outperforms major matmul baselines"**

**Supported.** The 1,000 configuration evaluation with win rates of 79.3% to 95.7% across all baselines provides strong evidence of systematic rather than sporadic improvement (Section 4.1).

**Claim 2: "LLM-guided RL can discover superior implementations by scaling optimization across configuration spaces at scales impractical for manual tuning"**

**Partially supported.** The results demonstrate superior performance, and the 1,000 configuration space is indeed impractical for manual tuning. However, the paper does not directly compare against human-expert-tuned kernels (only against library baselines), so the claim about surpassing manual optimization is inferred rather than directly demonstrated.

**Claim 3: "The framework is designed for broad applicability" to other GPU architectures**

**Not yet supported.** While the framework architecture may be generalizable, all current results are on A100 only. The claim about broad applicability is aspirational pending future evaluation on other architectures (Section 1).

**Claim 4: "Server mode speedups further increase" indicating production relevance**

**Supported.** The consistent pattern of higher speedups in server mode across all baselines (+4-7 percentage points higher than offline mode) provides evidence that the optimizations are valuable for real-world inference workloads (Abstract, Section 4.1).

### 5.5 Summary of Experimental Findings

Table 5 summarizes the key experimental findings:

| Aspect | Finding | Evidence |
|--------|---------|----------|
| Overall speedup | +11.4% to +28.7% depending on baseline and mode | Table 1, Figure 1 |
| Win rate | 79.3% to 95.7% across baselines | Section 4.1 |
| Server vs. offline | Server mode shows larger gains | Figure 1a vs. 1b |
| Problem size effect | Speedup decreases for larger matrices | Figure 3 |

The experiments provide convincing evidence that CUDA-L2 achieves meaningful speedups over state-of-the-art baselines across a comprehensive configuration space. The main limitations are the single-architecture evaluation, lack of ablation studies on RL components, and diminishing returns for very large problem sizes. These limitations do not invalidate the core findings but clarify the conditions under which CUDA-L2 provides the most value.

## 6. Limitations and Trade-offs

While CUDA-L2 demonstrates compelling performance gains, a critical examination reveals several important limitations, assumptions, and trade-offs that practitioners should understand before adopting this approach. This section analyzes what the system does not address, the constraints under which it operates, and open questions that remain.

### 6.1 Architectural Constraints and Generalization Gap

**Single-Architecture Evaluation**

The most significant limitation is that **all experimental results are confined to A100 GPUs (Ampere architecture)** (Section 1). While the paper states the framework is "designed for broad applicability" with ongoing work to extend to Ada Lovelace, Hopper, and Blackwell, no empirical evidence supports cross-architecture transfer.

This matters because:

- **Memory hierarchy differences**: Hopper H100 introduces a new memory hierarchy with L2 cache improvements and Tensor Memory Accelerator (TMA) units that Ampere lacks. Optimization strategies tuned for Ampere's memory subsystem may not generalize.

- **Tensor core evolution**: Different architectures have different tensor core dimensions and instruction sets. For example, Hopper supports TF32 and FP8 tensor cores that Ampere does not. A kernel optimized for Ampere's FP16 tensor cores may not exploit these newer capabilities.

- **Block swizzle parameters**: Section 6.3 shows that `swizzle_stride` values are tuned based on problem size for A100. These optimal values are architecture-specific—what works for Ampere's L2 cache structure may not work for Hopper's.

The paper acknowledges this explicitly:

> "While the current version of CUDA-L2 only focuses on A100 GPUs, the framework is designed for broad applicability, with ongoing work to extend it to other GPU architectures" (Abstract, Section 1).

This is an honest limitation, but it means the claimed "broad applicability" remains **aspirational rather than demonstrated**. Practitioners deploying on non-Ampere hardware cannot assume the reported speedups will transfer.

### 6.2 Problem Size Dependence and Diminishing Returns

**Speedup Convergence for Large Matrices**

Figure 3 reveals a critical conditional pattern: **CUDA-L2's advantages diminish as problem size increases** (Section 4.3):

| Problem Size | log₂(M×N×K) | Approximate Speedup |
|--------------|-------------|---------------------|
| Small | 18-20 | ~1.4× (40% faster) |
| Medium | 27-33 | ~1.2× (20% faster) |
| Large | >38 | ~1.0× (no gain) |

The paper explains this as GPU compute saturation:

> "For small matrix multiplications, the GPU is underutilized and there is significant room for optimization through better memory access patterns, tiling strategies, and kernel configurations. CUDA-L2 exploits these opportunities effectively. On the contrary, large matrices saturate the GPU's floating-point throughput, leaving less room for improvement" (Section 4.3).

**Practical Implication**: CUDA-L2 provides the most value for **small-to-medium problem sizes** typical of attention layers and early FFN layers in LLMs. For practitioners working primarily with very large batch sizes or sequence lengths where matrices exceed log₂(M×N×K) ≈ 38, the benefits may be negligible. This does not invalidate the results, but it clarifies the **conditions under which CUDA-L2 provides the most value**.

### 6.3 Methodological Gaps in Experimental Design

**No Ablation Studies on RL Components**

Section 3.2 describes four technical enhancements over CUDA-L1:
1. Continued pretraining with diverse CUDA code
2. Multi-stage RL training (general → HGEMM-specific)
3. NCU profiling metrics in context
4. Retrieval-augmented context

However, **the paper provides no ablation studies isolating the contribution of each component**. Readers cannot determine:

- Whether all four enhancements are necessary, or whether some provide marginal benefit
- Which component contributes most to the 11.4% speedup over `cuBLASLt-AutoTuning`
- Whether the multi-stage training progression is essential, or if direct HGEMM training would suffice

This is a notable gap. Without ablation analysis, it is unclear whether the system's complexity is justified or whether a simpler pipeline could achieve similar results.

**Training Cost Not Reported**

The paper does not specify:

- How much compute time was required to train CUDA-L2
- How many kernel generations were needed to converge
- What the RL training sample efficiency was
- Whether the training cost is amortized across the 1,000 configurations or per-configuration

This information is critical for assessing **practical replicability**. If training requires thousands of GPU-hours, the approach may be feasible only for well-resourced organizations. If the training cost is modest, it becomes more accessible. The omission leaves this question unanswered.

**Correctness Failure Rate Undisclosed**

Section 2.3 describes rigorous correctness validation (binary input exact match, baseline-bounded deviation), but the paper does not report:

- How many generated kernels failed correctness checks during training
- What the final success rate was after RL convergence
- Whether certain configurations were more prone to generating incorrect kernels

This matters for understanding the **reliability of the RL training process**. A high failure rate would indicate the reward function or training procedure needs refinement. The absence of this data makes it difficult to assess training stability.

### 6.4 Configuration Space Coverage Limitations

**Discrete Dimension Set**

The 1,000 configurations cover all combinations of M, N, K from {64, 128, 256, 512, 1024, 2048, 4096, 8192, 12288, 16384} (Abstract, Section 1). While this covers dimensions used in Qwen, Llama, and DeepSeek, it is still a **discrete set with gaps**.

Real-world scenarios may involve:

- **Non-power-of-2 dimensions**: Some models use dimensions like 1536, 3072, or 6144 that fall between the evaluated points.
- **Dynamic sequence lengths**: Inference workloads often have variable sequence lengths that do not align with the pre-optimized configurations.
- **Batch size variations**: The evaluation does not explicitly address how batch size interacts with the M, N, K dimensions.

The paper does not discuss how CUDA-L2 handles configurations **outside the trained distribution**. Does it generalize to unseen dimensions, or does it require retraining? This is an open question.

### 6.5 Precision and Operation Scope

**Half-Precision Only**

CUDA-L2 focuses exclusively on **HGEMM (Half-precision General Matrix Multiply)** (Section 2.1). Other precisions commonly used in LLMs are not addressed:

- **BF16 (BFloat16)**: Widely used in training for better numerical stability than FP16
- **FP32**: Required for certain layers or when numerical precision is critical
- **FP8**: Emerging precision for inference optimization on Hopper and Blackwell
- **INT8/INT4**: Quantized inference kernels for latency-critical deployments

The paper does not discuss whether the optimization strategies discovered for FP16 would transfer to these other precisions. Given that different precisions have different numerical properties and hardware support, this is a meaningful limitation.

**Single Operation Type**

While the general kernel RL stage covers diverse operations (Section 3.2.2), the HGEMM-specific RL stage focuses only on matrix multiplication. Other critical LLM operations are not optimized:

- **Softmax**: Used in every attention layer
- **LayerNorm**: Used throughout transformer architectures
- **Activation functions** (GeLU, SiLU, etc.): Computationally significant in FFN layers
- **Attention mechanisms**: FlashAttention variants, grouped query attention

A production system would need to optimize the entire operator graph, not just matmul. CUDA-L2 demonstrates the approach works for HGEMM, but extending it to a complete operator suite remains future work.

### 6.6 Deployment and Integration Trade-offs

**Static Optimization vs. Dynamic Adaptation**

CUDA-L2 generates **statically optimized kernels** for specific (M, N, K) configurations. This contrasts with dynamic runtime adaptation approaches that adjust kernel parameters based on real-time conditions.

Limitations of static optimization:

- **Cold start overhead**: New configurations require kernel generation and compilation before execution
- **Memory footprint**: Storing 1,000+ optimized kernels consumes memory
- **Inflexibility**: Cannot adapt to runtime conditions like thermal throttling or concurrent workload pressure

The paper does not discuss the compilation overhead or deployment logistics of managing a large kernel library.

**Server Mode Measurement Caveat**

In server mode evaluation, **the interval time between kernel executions is not included in the execution time** used for speedup comparison (Section 2.4.2):

> "It's worth noting that the interval time is not included in the execution time used for the speedup comparison in server mode" (Section 2.4.2).

This is methodologically sound for isolating kernel performance, but it **masks potential latency issues** in real deployments. If CUDA-L2 kernels have higher launch latency or synchronization overhead, this would not be captured in the reported metrics. Practitioners concerned with end-to-end inference latency (not just kernel execution time) should verify that the speedups translate to actual latency reductions in their deployment context.

**CUDA C++ vs. Higher-Level DSLs**

Generated kernels are compiled as `.cu` files using CUDA C/C++, CuTe, inline PTX assembly, and CUTLASS templates—but **not Python-based DSLs like Triton** (Section 3.2.3).

Trade-offs:

| Aspect | CUDA C++ (.cu files) | Triton/Python DSL |
|--------|---------------------|-------------------|
| Optimization flexibility | Maximum (inline PTX, intrinsics) | Limited by DSL abstractions |
| Accessibility | Requires CUDA expertise | More accessible to Python developers |
| Portability | Architecture-specific | Some cross-architecture abstraction |
| Debugging complexity | Higher | Lower |

This design choice prioritizes **maximum optimization flexibility** over accessibility. Organizations without CUDA expertise may find it difficult to maintain, debug, or extend the generated kernels.

### 6.7 Reward Function Design Choices

**Undisclosed Penalty Coefficients**

The reward function includes penalty coefficients α (correctness) and β (code length):

$$r(\text{custom}) = \frac{1}{N} \sum_{i=1}^{N} \left[ \frac{t_i^{\text{ref}}}{t_i^{\text{custom}}} - \alpha \cdot \text{diff}_i \right] - \beta L(\text{custom})$$

However, **the specific values of α and β are not disclosed** (Section 3.2.3). This matters because:

- **Too high α**: May overly penalize minor numerical deviations, preventing the model from exploring aggressive optimizations
- **Too low α**: May allow numerically unstable kernels to receive high rewards
- **Too high β**: May favor overly concise code at the expense of performance
- **Too low β**: May produce verbose, hard-to-maintain kernels

Without knowing these values, practitioners cannot replicate the training setup or tune the reward function for their own use cases.

**Code Length as Proxy for Efficiency**

The code length penalty assumes that **shorter code correlates with better abstractions** (Section 5.1):

> "CuTe's abstractions result in more concise code compared to raw CUDA intrinsics. Since the RL reward favors shorter code, CUDA-L2 naturally leans towards CuTe for complex optimizations" (Section 5.1).

This is a reasonable heuristic, but it is not universally true. Some highly optimized kernels require verbose code for fine-grained control. The code length penalty may inadvertently discourage certain valid optimization strategies that happen to be verbose.

| Limitation | Severity | Evidence |
|------------|----------|----------|
| Single-architecture evaluation (A100 only) | High | Section 1 |
| Speedup diminishes for large problems (log₂ > 38) | Medium-High | Figure 3, Section 4.3 |
| No ablation studies on RL components | Medium | Section 3.2 |
| Training cost not reported | Medium | Not disclosed |
| Correctness failure rate undisclosed | Medium | Section 2.3 |

### 6.9 Open Questions for Future Work

Based on the limitations identified, several important questions remain:

1. **Cross-architecture transfer**: Can kernels optimized for Ampere be adapted to Hopper or Blackwell with minimal retraining, or does each architecture require training from scratch?

2. **Unseen configuration generalization**: How does CUDA-L2 perform on (M, N, K) configurations outside the trained distribution (e.g., non-power-of-2 dimensions)?

3. **Training efficiency**: What is the compute cost of training CUDA-L2, and can it be reduced through more sample-efficient RL algorithms?

4. **Multi-precision support**: Can the approach extend to BF16, FP8, and quantized precisions while maintaining similar speedups?

5. **Full operator graph optimization**: Can CUDA-L2 optimize entire operator sequences (e.g., matmul + softmax + layernorm) rather than individual kernels in isolation?

6. **Deployment logistics**: What is the overhead of managing and deploying 1,000+ optimized kernels in production, and how does this compare to the performance gains?

These questions do not diminish CUDA-L2's achievements but highlight the path forward for making the approach more broadly applicable and production-ready.

## 7. Implications and Future Directions

CUDA-L2 represents a inflection point in automated kernel optimization, shifting the field from parameter tuning within fixed algorithm spaces to structural code generation across open search spaces. This section outlines how this work changes the landscape, the research frontiers it opens, practical deployment scenarios, and guidance for integration.

### 7.1 Shifting the Paradigm of Kernel Optimization

The most profound implication of CUDA-L2 is the demonstration that **vendor-optimized libraries are not the performance ceiling**. Prior to this work, the prevailing assumption was that libraries like `cuBLASLt` represented the upper bound of performance because they exhaustively search a predefined space of optimized algorithms. CUDA-L2 disproves this by achieving an **11.4% speedup in offline mode** and **15.9% in server mode** over `cuBLASLt-AutoTuning` (Section 4.1).

This result forces a reevaluation of how kernel optimization is approached:

*   **From Selection to Generation**: Traditional auto-tuning (e.g., `cuBLASLt-AutoTuning`) selects the best algorithm from a fixed list (up to 100 candidates). CUDA-L2 generates the algorithm itself. This allows it to discover structural optimizations—such as **padding input matrices for non-divisible tile sizes** (Section 5.2) or **staggered prefetch scheduling** (Section 5.3.4)—that do not exist as selectable options in vendor APIs.
*   **Scaling Beyond Human Capacity**: The system optimizes **1,000 configurations** covering all combinations of $M, N, K \in \{64, \dots, 16384\}$ (Abstract). Manually tuning this space is impractical due to the non-transferability of strategies across dimensions and architectures (Section 1). CUDA-L2 demonstrates that LLM-guided RL can scale optimization to cover production-relevant configuration spaces systematically rather than relying on isolated benchmark wins.
*   **Diagnostic-Driven Learning**: By incorporating **NCU profiling metrics** (memory throughput, SM occupancy) into the RL context (Section 3.2.3), CUDA-L2 moves beyond black-box reward signals. This suggests a future where optimization agents understand *why* a kernel is slow (e.g., "low cache hit rate") rather than just *that* it is slow, enabling more targeted improvements.

### 7.2 Research Frontiers Enabled by CUDA-L2

This work opens several critical avenues for follow-up research, addressing both the limitations identified in Section 6 and new possibilities revealed by the results.

*   **Cross-Architecture Generalization**: The current evaluation is limited to **A100 (Ampere) GPUs** (Section 1). A primary research direction is extending the framework to **Ada Lovelace (RTX 4090), Hopper (H100), and Blackwell (B200)**. This is non-trivial because optimization strategies (e.g., `swizzle_stride` values in Figure 4f) are architecture-specific due to differences in memory hierarchy and tensor core instructions. Research should investigate whether the foundation model can transfer optimization principles across architectures with minimal fine-tuning.
*   **Multi-Precision and Operator Support**: CUDA-L2 focuses exclusively on **Half-precision (FP16) GEMM** (Section 2.1). Future work should extend this to **BF16** (critical for training stability), **FP8** (emerging on Hopper/Blackwell), and **quantized precisions (INT8/INT4)**. Additionally, while the General Kernel RL stage covers diverse operations (Section 3.2.2), the HGEMM stage is specialized. Extending the specialized RL stage to other critical operators (Softmax, LayerNorm, FlashAttention) would enable end-to-end model optimization rather than single-kernel improvements.
*   **Training Efficiency and Ablation**: The paper does not disclose the **compute cost of training** or the contribution of individual components (continued pretraining, NCU metrics, etc.) due to a lack of ablation studies (Section 6.3). Research should quantify the sample efficiency of this RL approach. Is the 11.4% speedup worth hundreds of GPU-hours of training? Can the training cost be amortized across thousands of configurations? Answering this is essential for democratizing the technology.
*   **Dynamic Adaptation**: CUDA-L2 generates **statically optimized kernels** for specific $(M, N, K)$ configurations (Section 3.2.3). Research into **dynamic runtime adaptation** could enable kernels that adjust parameters (e.g., pipeline stages) based on real-time thermal conditions or concurrent workload pressure, addressing the static optimization limitation noted in Section 6.6.

### 7.3 Practical Applications and Downstream Use Cases

For practitioners, CUDA-L2 offers immediate value in specific deployment scenarios, particularly where latency and throughput are critical.

*   **LLM Inference Servers**: The **server mode** evaluation simulates real-time inference with random intervals between requests (Section 2.4.2). CUDA-L2 achieves **larger speedups in server mode (+15.9% over cuBLASLt-AutoTuning)** compared to offline mode (+11.4%) (Section 4.1). This indicates the generated kernels are particularly effective at handling **cache cooling and thermal throttling**, making them ideal for production inference servers where request patterns are unpredictable.
*   **Small-to-Medium Matrix Dimensions**: Figure 3 shows that speedups are **highest for smaller problems** (~1.4× for log₂(M×N×K) ≈ 18-20) and converge to baseline for very large matrices (Section 4.3). This aligns perfectly with **attention layers and early FFN layers** in LLMs, which often process smaller matrices per token or per head. Practitioners should prioritize deploying CUDA-L2 kernels for these layers rather than large batched matmuls where GPU compute is already saturated.
*   **Static Kernel Libraries**: Organizations can build a **pre-optimized kernel library** for their specific model architectures (e.g., Llama, Qwen, DeepSeek dimensions listed in Abstract). Since the 1,000 configurations cover these models, teams can compile the generated `.cu` files into a static library linked against their inference engine, bypassing the need for runtime auto-tuning which incurs latency overhead.

### 7.4 Reproduction and Integration Guidance

For teams looking to adopt or reproduce this work, the following guidance clarifies when to prefer this method and how to integrate it safely.

*   **When to Prefer CUDA-L2**:
    *   **Hardware**: Currently only validated on **A100 GPUs** (Section 1). Do not deploy on Hopper or Blackwell without re-validation.
    *   **Problem Size**: Prioritize for **small-to-medium matrices** (log₂(M×N×K) < 30) where Figure 3 shows the highest gains (Section 4.3).
    *   **Precision**: Only for **FP16 HGEMM**. Do not use for BF16 or FP32 accumulators without correctness re-validation.
    *   **Latency Sensitivity**: Ideal for **server-mode inference** where the +15.9% speedup over `cuBLASLt-AutoTuning` translates directly to reduced tail latency (Section 4.1).

*   **Integration Steps**:
    1.  **Compilation**: Generated kernels are standard **CUDA C++ (`.cu` files)** compiled with `nvcc` (Section 3.2.3). They do not require Python runtimes, making them suitable for C++ production environments.
    2.  **Correctness Validation**: Before deployment, replicate the **two-tier correctness check** (Section 2.3):
        *   **Exact Match**: Use binary inputs $\{0, 1\}$ and verify exact equality for outputs $< 2048$ (Section 2.3.2).
        *   **Baseline-Bounded Deviation**: Ensure numerical deviation stays within the bounds of NVIDIA's own libraries (Section 2.3.2).
    3.  **Timing Safeguards**: When benchmarking, enforce **explicit synchronization** and disallow additional CUDA streams to prevent reward hacking (Section 2.4.1, Listing 1).
    4.  **Fallback Mechanism**: Implement a runtime fallback to `cuBLASLt` for configurations outside the trained 1,000 set or if correctness checks fail. The `max(CUDA-L2, baseline)` results in Table 2 show that combining both kernels yields additional gains (+1.8% offline over AutoTuning), suggesting a hybrid approach is optimal (Section 4.2).

*   **Critical Caveats**:
    *   **Undisclosed Hyperparameters**: The reward function penalty coefficients $\alpha$ (correctness) and $\beta$ (code length) are not disclosed (Section 3.2.3). Reproducers must tune these carefully to balance speed and numerical stability.
    *   **Compilation Overhead**: Managing 1,000+ specialized kernels introduces build complexity. Teams should assess whether the performance gain justifies the maintenance burden of a large kernel library.
    *   **Server Mode Measurement**: Note that server mode speedups exclude the interval time between requests (Section 2.4.2). While this isolates kernel performance, practitioners must verify end-to-end latency improvements in their specific deployment pipeline.

CUDA-L2 proves that automated kernel optimization can surpass human-engineered libraries at scale. By treating kernel generation as a learnable task rather than a manual engineering chore, it opens the door to a future where compilers and AI agents co-design hardware-aware software, systematically extracting performance that manual tuning leaves on the table.
