## 1. Executive Summary
To address the limitation of current benchmarks that evaluate language models on isolated, well-specified tasks, this paper introduces CodeClash, a benchmark where 8 frontier LMs (including Claude Sonnet 4.5 and GPT-5) compete in 1680 multi-round tournaments across 6 competitive code arenas to achieve high-level objectives without explicit guidance. The study reveals that while models exhibit diverse development styles over 25,200 total rounds, they share fundamental limitations in strategic reasoning and codebase maintenance, with top models losing every round against expert human programmers. This work advances the study of autonomous, goal-oriented code development by exposing the stark performance gap between current LMs and human experts in strategic, long-horizon tasks.

## 2. Context and Motivation

### The Gap Between Benchmark Tasks and Real-World Development

Current language model coding benchmarks operate under a fundamental assumption that does not reflect how professional software engineers actually work. These benchmarks present models with **concrete, well-specified tasks**: implement this algorithm, fix this specific bug, write a test for this target class. The problem statement is straightforward and fine-grained. Given explicit instructions, models are evaluated purely on their ability to execute them correctly.

However, real-world software development demands a much broader scope of agency. Engineers are rarely handed precise specifications. Instead, they are driven by **high-level goals** like improving user retention, increasing revenue, or reducing operational costs. Achieving these objectives requires fundamentally different capabilities:

- **Recursive decomposition**: Breaking abstract goals into actionable steps without explicit guidance
- **Strategic prioritization**: Deciding which solutions to pursue among many possible approaches
- **Continuous feedback loops**: Proposing changes, deploying them, analyzing real-world metrics or competitor behavior, and iterating

This gap matters because benchmarks shape what capabilities the research community prioritizes. If we only evaluate models on tasks with explicit instructions, we optimize for execution rather than strategic reasoning. The paper argues this creates a misleading picture of model readiness for autonomous software development.

### Why This Problem Matters

The importance of addressing this gap operates on two levels:

**Practical Impact**: As organizations increasingly deploy AI coding assistants, understanding whether these systems can handle open-ended, goal-oriented work becomes critical. A model that excels at fixing specified bugs but cannot determine *which* bugs to fix or *how* to improve a codebase strategically offers limited value for autonomous development workflows.

**Theoretical Significance**: The challenge touches on core questions about AI agency and long-horizon planning. Can language models maintain coherent strategies across multiple iterations? Can they extract meaningful insights from noisy feedback? Can they organize and maintain codebases over extended periods without human intervention? These questions extend beyond coding into broader questions about autonomous AI systems.

### Prior Approaches and Their Limitations

The paper situates CodeClash within several threads of existing work, each with distinct shortcomings:

**Traditional Coding Benchmarks** (e.g., HumanEval, MBPP, LiveCodeBench): These evaluate models on function-level or algorithm-level tasks with clear correctness criteria. The limitation is obvious: success is binary and immediate. Once a model passes the test, there is no signal for further improvement. This does not capture the iterative nature of real development.

**Repository-Level Benchmarks** (e.g., SWE-bench): These represent a significant advance, requiring models to resolve GitHub issues within existing codebases. However, the objective remains explicitly specified—the issue description tells the model exactly what to fix. The evaluation metric (does the patch pass tests?) remains binary. As the paper notes, "unit tests only provide binary pass/fail feedback, and once passed, they are no longer useful for further refinement."

**Code Optimization Benchmarks** (e.g., EffiBench, SWE-Perf): These evaluate models on improving code efficiency or reducing runtime. Like CodeClash, the methodology is self-prescribed—models decide how to optimize. However, these tasks lack the **adversarial component**. Models optimize independently; their codebases do not compete against evolving opponents. The objective is also narrow (runtime performance) rather than the diverse win conditions found in real competitive scenarios.

**Game-Playing Benchmarks** (e.g., AlphaGo, Dota 2 agents, GameArena): Games have long served as testbeds for AI, with recent work applying them to language models. However, these typically have the AI **directly play the game** through action selection. CodeClash introduces a critical distinction: models **iteratively refine code that competes as their proxy**. This shifts the evaluation from in-game decision-making to software engineering capability—writing, testing, and improving code that will perform well under competition conditions.

**Self-Improving Agent Research**: Recent work explores how language models can evolve their own scaffolds or workflows. However, these efforts typically rely on static benchmarks with fixed metrics. CodeClash's competitive setting with constantly evolving opponents provides a **perpetual learning signal that doesn't saturate**. Performance is graded relatively against opponents rather than against a fixed correctness threshold.

### How CodeClash Positions Itself

CodeClash occupies a distinct niche by combining several properties that no prior benchmark simultaneously offers:

| Property | Traditional Benchmarks | SWE-bench | Game-Playing AI | **CodeClash** |
|----------|----------------------|-----------|-----------------|---------------|
| Open-ended objectives | ✗ | ✗ | Partially | ✓ |
| Multi-round iteration | ✗ | ✗ | ✓ | ✓ |
| Competitive/adversarial | ✗ | ✗ | ✓ | ✓ |
| Code-as-proxy (not direct play) | N/A | N/A | ✗ | ✓ |
| Self-directed improvement strategy | ✗ | ✗ | ✗ | ✓ |
| Long-horizon codebase maintenance | ✗ | Partially | ✗ | ✓ |

The key innovation is the **tournament structure**. Multiple language model systems compete to build the best codebase for achieving a high-level objective over multiple rounds. Each round proceeds in two phases:

1. **Edit Phase**: Agents independently modify their codebases using whatever strategies they deem appropriate within a fixed budget
2. **Competition Phase**: All codebases are executed within a code arena, which determines winners based on objectives like score maximization, resource acquisition, or survival

Critically, models receive **no explicit guidance** beyond a brief environment description. They must proactively discover arena mechanics, analyze competition logs, write their own analysis tools, and decide how to improve. Information from previous rounds is not automatically retained—models must explicitly record insights in their codebase, making codebase organization itself a measurable capability.

This design directly tests the capabilities the paper identifies as missing from current benchmarks: strategic reasoning, adaptation to opponents, long-term planning, and autonomous codebase maintenance. By evaluating 8 frontier models across 6 diverse arenas (from BattleSnake survival to Poker strategy to RoboCode tank combat), CodeClash provides a multidimensional view of where current models succeed and where they fundamentally struggle.

## 3. Technical Approach

This is an empirical benchmarking paper that introduces a tournament-based evaluation framework for measuring language models' capabilities in autonomous, goal-oriented software engineering. The core idea is to place multiple language model agents in competitive multi-round tournaments where they iteratively improve codebases that compete as proxies in various game-like arenas, revealing their strategic reasoning and long-horizon planning capabilities without explicit task specifications.

### 3.1 Reader orientation (approachable technical breakdown)

CodeClash is a benchmarking system where language models act as software developers, editing their own codebases over multiple rounds while their code competes against opponent code in game arenas. The system solves the problem of evaluating open-ended software engineering capability by creating a competitive tournament structure where models must self-direct their improvement strategies, analyze competition feedback, and maintain organized codebases over 15-round tournaments without human intervention.

### 3.2 Big-picture architecture (diagram in words)

The CodeClash system consists of four major components that interact in a cyclical tournament loop. First, **Player Agents** (language models equipped with mini-SWE-agent interfaces) maintain individual codebases throughout a tournament and issue bash commands to edit files during edit phases. Second, **Code Arenas** (six distinct competitive environments like BattleSnake or Poker) execute multiple codebases against each other and produce outcome logs. Third, the **Tournament Orchestrator** manages the round structure, copying logs from arenas back into player codebases between rounds and enforcing turn/cost limits. Fourth, the **Evaluation Pipeline** aggregates round outcomes into tournament wins, computes Elo ratings using Bradley-Terry models, and performs statistical validation through bootstrapping. Information flows from players to arenas (code submissions), from arenas to players (competition logs), and from the orchestrator to the evaluation pipeline (win/loss records).

### 3.3 Roadmap for the deep dive

- **Tournament Structure and Round Mechanics**: We first explain how tournaments are organized (15 rounds, 2 phases per round) because this defines the temporal structure that all other components operate within.
- **Player Agent Implementation**: We then detail how language models interact with codebases via mini-SWE-agent, since this determines what capabilities models can actually exercise.
- **Code Arena Design**: We describe the six competitive environments and their win conditions, as these define the objectives models must pursue without explicit guidance.
- **Feedback and Memory Mechanisms**: We explain how competition logs are returned to players and why codebase-as-memory is a critical design choice for measuring long-horizon capability.
- **Evaluation Metrics and Statistical Methods**: We conclude with how performance is quantified (Elo ratings, win rates) and validated (bootstrapping), since this determines how to interpret all experimental results.

### 3.4 Detailed, sentence-based technical breakdown

**Tournament Structure and Round Mechanics**

CodeClash formalizes competitive coding as a multi-round tournament where two or more players compete in a code arena, with each tournament consisting of exactly 15 rounds for the main leaderboard experiments. Each round proceeds in two distinct phases that alternate throughout the tournament. In the **edit phase**, each player independently modifies their codebase using whatever strategies they deem appropriate within a fixed budget of 30 interactive turns and a $1 cost limit for token consumption. During the **competition phase**, all player codebases are compiled and executed within the code arena, where they interact and compete directly against each other to produce measurable outcomes. The arena determines a winner or declares a tie based on the codebases' performance according to arena-specific objectives like eliminating opponents, acquiring resources, or maximizing profit. After each competition phase concludes, the results and logs are copied into each player's codebase as the sole source of new information for the next round, creating a feedback loop where models must decide what insights to preserve.

The experimental design runs 10 tournaments per model pair per arena across 6 arenas with 8 models, yielding $\binom{8}{2} \times 6 \times 10 \times 15 = 25,200$ total rounds for the main leaderboard. Tournament runtime varies by arena, taking 75 minutes on average, totaling approximately 2.4 million hours of runtime when accounting for model latency, with parallelization across independent tournaments. For each round's competition phase, the arena runs the competition 1,000 times to account for non-determinism in most arenas, declaring the winner as whichever player wins the most out of the 1,000 simulations rather than requiring a specific win percentage threshold. A single round is won by a model if it achieves a higher score in the arena than its opponent or if its opponent makes an invalid submission. A tournament is won by the model that wins more rounds than its opponent, or if both models win equally many rounds, by the model that scores the last win.

**Player Agent Implementation**

To implement a player, CodeClash uses mini-SWE-agent, an Agent Computer Interface (ACI) that enables a language model to interact with a codebase by issuing bash actions to a terminal environment. Each turn, the language model generates a ReAct-style response containing a thought (in natural language explaining reasoning) and a bash action, then receives standard output from the terminal environment in return. The system intentionally avoids tool-heavy scaffolds such as SWE-agent or OpenHands, as they are often optimized for specific models and benchmarks, which would introduce confounding variables. By restricting interactions to bash commands only, mini-SWE-agent avoids imposing predefined assumptions via tools about how language models should approach codebase modifications or competitive play. This design decision leaves an opportunity for language models to synthesize their own tools across rounds, making tool creation itself a measurable capability.

The system prompt provided to each player at the start of every edit phase contains no guidance beyond high-level suggestions about how to enhance its codebase, with all decisions and changes necessarily autonomous. The prompt explicitly informs models that their working memory is not retained across rounds, encouraging them to use the codebase to maintain long-term information, tools, and general progress. Models are reminded of the number of turns and cost consumed after every response with a limit note tag, and once they reach the step or cost limit, they cannot continue working on the task. The cost limit is almost never reached in practice, but turn limits are exhausted frequently for specific models, with Claude Sonnet 4.5 typically using 23 of the allotted 30 editing turns per round while GPT-5 and o3 typically conclude after just 15 steps. Directory or environment variable changes are not persistent across actions since every action is executed in a new subshell, but models can prefix actions with environment variables or write and load environment variables from files.

**Code Arena Design**

CodeClash's initial release features a suite of 6 code arenas that vary significantly in codebase structure, interface mechanisms, and feedback types, as summarized in Table 2 of the paper. **BattleSnake** is a grid-based survival game where players program snakes in Python to collect food, avoid collisions, and outlast opponents on an 11×11 cell grid. **Core War** involves assembly-like programs called "warriors" written in RedCode that compete in shared virtual memory, with the goal of disabling all opposing programs by overwriting their code with invalid instructions. **Halite** is a resource collection and territory expansion game on a grid where bots can be written in C, C++, OCaml, or Rust, competing to control the largest share of territory by accumulating halite resources. **Poker** implements no-limit Texas Hold'em where Python bots make betting decisions across four stages (pre-flop, flop, turn, river) to maximize chip accumulation. **RoboCode** features tank combat in a 2D grid battlefield where Java programs control tank movement, scanning, and firing to outlast and outscore opposing tanks. **RobotRumble** is a turn-based grid battle game where JavaScript programs control teams of robots that spawn every 10 turns, with the player having more robots after 100 turns declared the winner.

Each arena provides starter codebases containing three essential assets: documentation in a docs/ folder explaining arena mechanics, arena executables and assets needed to run competitions, and a working submission that demonstrates core mechanics so players do not need to code valid submissions from scratch. The exact bash commands to run the arena are not disclosed to models, placing the burden on them to figure out how to use available assets through exploration. Arena configurations are specified via YAML files that define tournament parameters like number of rounds, simulations per round, and game-specific arguments (e.g., grid dimensions for BattleSnake). To eliminate positional advantages where one player might consistently win with identical codebases due to turn order or starting position, the system randomly shuffles the order of players with equal probability immediately after the codebase validation step in each round.

**Feedback and Memory Mechanisms**

CodeClash implements a codebase-as-memory design where players have no explicit memory of actions from previous rounds beyond whatever they chose to record in their codebase. This is a critical design choice because it makes codebase organization itself a measurable capability, forcing models to decide what insights to preserve and how to structure them for future retrieval. After each competition phase, competition logs are copied from the arena container back into the agent containers, specifically in a designated logs/ folder within each agent's codebase. These logs can range from several hundred to thousands of runs against opponents, detailing exact behaviors that led to losses or wins, but how models interpret these logs or act upon them is entirely self-driven. The initial system prompt mentions that analyzing logs might be helpful but provides no arena-specific advice on how exactly logs should be interpreted, creating a spectrum of approaches where some models directly read raw logs while others write scripts to solicit insights.

Log formats vary by arena to reflect the nature of each competition. BattleSnake logs are JSONL files where each line captures the complete game state at a given turn including snake positions, health, and food locations. Core War logs report high-level summaries of win/loss/tie outcomes, survival counts, and match duration rather than step-by-step instruction execution. Poker logs record each hand as a sequence of betting rounds with player actions, bet sizes, pot totals, and community cards. The system includes a viewer for CodeClash logs that provides visualizations of log content and automatically calculates game statistics like p-values to indicate if round winners are statistically significant. Models can write analysis scripts to parse these logs, maintain notes about past rounds or opponents in markdown files, or generate multiple candidate versions to test against one another, but all such tooling must be created by the models themselves.

**Evaluation Metrics and Statistical Methods**

Performance per model is calculated as an aggregation across all tournaments won across all arenas, with the win rate of a model defined as the fraction of tournaments it has won. To quantify overall model strength beyond simple win rates, CodeClash uses Elo scores with a base rating of $R_0 = 1200$ and a slope of $\beta = 400$, following chess rating conventions. Elo ratings are based on the Bradley-Terry model that models win probabilities between two players $i$ and $j$ with strengths $s_i$ and $s_j$ via logistic regression of the strength difference. The probability that model $i$ wins over model $j$ is given by:

$$P(\text{model } i \text{ wins over } j) = \frac{1}{1 + \exp(s_j - s_i)} = \sigma(s_i - s_j)$$

where $\sigma$ is the logistic function. Repetitions of independent games are Bernoulli-distributed, and optimal values of $s_i$ and $s_j$ are calculated using a maximum likelihood fit to the win numbers $w_{ij}$ (number of times $i$ won over $j$). The log-likelihood function is:

$$\log L = \sum_{i<j} \left[ w_{ij} \log \sigma(s_i - s_j) + w_{ji} \log \sigma(s_j - s_i) \right]$$

Instead of calculating Elo scores using sequential updates (which require a choice of step size and depend on update order), CodeClash performs a more rigorous maximum likelihood fit to the win rates. There is a gauge freedom in the strengths $s_i$ because all $s_i$ can be shifted by a constant factor $s_i \rightarrow s_i + S$ without changing the value of $L$, so the system chooses $\sum_i s_i = 0$ to fully constrain the fit. Player strengths are converted to Elo scores $R_i$ as:

$$R_i = R_0 + \frac{\beta}{\log 10} s_i$$

The covariance matrix $\Sigma$ of player strengths $s_i$ is given by the inverse of the Hessian matrix of $\log L$, with the Hessian calculated as:

$$H_{ij} = \frac{\partial^2 \log L}{\partial s_i \partial s_j} = -\sum_{i<j} n_{ij} p_{ij} (1 - p_{ij}) \begin{cases} 1 & i = j \\ -1 & i \neq j \end{cases}$$

where $p_{ij} = \sigma(s_i - s_j)$ and $n_{ij} = w_{ij} + w_{ji}$. Since this Hessian is singular due to shift-invariance, the system inverts $H$ in the constrained subspace where $\sum_i s_i = 0$.

Statistical validation is performed through both parametric and non-parametric bootstrapping experiments with 1,000 samples each to assess rank stability. Non-parametric bootstrapping samples with replacement from all tournaments to generate new win counts $w_{ij}$ for recalculating Elo rankings. Parametric bootstrapping generates bootstrap replicas from the fitted Bradley-Terry model by drawing $\tilde{w}_{ij} \sim \text{Binomial}(n_{ij}, p^*_{ij})$ where $p^*_{ij} = \sigma(\hat{s}_i - \hat{s}_j)$ using the fitted player strengths. The pairwise order agreement of the ranking is more than 98% in bootstrapping experiments, with Kendall's $\tau$ at 0.966 (non-parametric) and 0.956 (parametric), and Spearman's $\rho$ at 0.988 (non-parametric) and 0.984 (parametric). For multi-player tournaments with 3+ players where Elo and win rate are limited to one-on-one settings, the system uses the TrueSkill rating system instead.

**Design Choices and Alternatives Considered**

The decision to use mini-SWE-agent rather than tool-heavy scaffolds reflects an intentional choice to evaluate models rather than scaffolds or tools, as various workflows and tools can be unintentionally biased to favor particular models. The codebase-as-memory design (no persistent memory across rounds) is chosen because it makes information retention a measurable capability, forcing models to explicitly add insights to their codebase rather than relying on external memory systems. The 1,000 simulations per round (rather than a single run) accounts for non-determinism observed in most CodeClash arenas, where given the same codebases and arena, the outcome of a single simulation is indeterminate. The 30-turn limit per edit phase with a $1 cost limit forces agents to allocate their context budget carefully, encouraging concise reasoning and selective use of code rather than dumping entire files into context or flooding the interaction buffer with irrelevant information. The decision to keep opponent codebases opaque by default (though explored as an ablation in Section 4.1) reflects real-world settings where human players develop solutions independently with the option to keep codebases closed source. Docker containerization is used for both agent containers (where models develop codebases) and arena containers (where competitions run) to ensure portability and reproducibility, with no codebase edits or arena runs ever performed on the local device.

## 4. Key Insights and Innovations

CodeClash introduces several fundamental innovations that distinguish it from prior benchmarking approaches. Rather than incremental improvements to existing evaluation frameworks, these contributions represent conceptual shifts in how we measure language model capabilities for autonomous software engineering. Below, we identify the four most significant innovations and explain why each matters.

### 4.1 Competitive Tournaments as a Perpetual Learning Signal

**What makes this different:** Traditional coding benchmarks provide binary, saturating feedback. A model either passes a unit test or it does not. Once the test is passed, there is no signal for further improvement. SWE-bench and similar repository-level benchmarks share this limitation—the issue is either resolved or not. CodeClash replaces this static correctness metric with **relative performance against evolving opponents**.

**Why it's significant:** This design creates a **perpetual learning signal that does not saturate**. As Section 3.4 described, models compete across 15-round tournaments where opponents adapt between rounds. A strategy that wins in round 1 may fail by round 10 as opponents develop countermeasures. This mirrors real-world software development where competitive advantages are temporary and continuous adaptation is required.

The innovation extends beyond evaluation into training potential. As the paper notes in Section 6, "CodeClash's competitive setting with constantly evolving opponents provides a perpetual learning signal that doesn't saturate." This makes the benchmark suitable not just for evaluation but for **self-play and reinforcement learning** approaches where models can continuously improve against increasingly sophisticated opponents. Existing benchmarks cannot support this because their objectives are fixed and exhaustible.

**Fundamental vs. Incremental:** This is a **fundamental innovation** in evaluation design. It shifts the paradigm from "does the code work?" to "does the code win against adaptive competition?"—a qualitatively different question that tests capabilities no prior benchmark measures.

### 4.2 Codebase-as-Memory: Making Organization a Measurable Capability

**What makes this different:** Most agent systems provide external memory mechanisms—vector databases, conversation history, or explicit state tracking—that persist across interaction turns. CodeClash deliberately removes this crutch. As Section 3.4 explained, players have no explicit memory of actions from previous rounds; their information is limited to whatever they chose to record in their codebase.

**Why it's significant:** This design choice transforms **codebase organization from an implementation detail into a measurable capability**. The paper's analysis in Section 5.1 reveals stark differences in how models manage their repositories over time. Claude Sonnet 4.5 averages more than 30 files per tournament while o3 creates fewer than 5. More critically, Figure 6 shows that "the average number of agent-created files scales almost linearly with the number of rounds" rather than plateauing as human-managed codebases do.

This finding exposes a fundamental limitation: models favor "continual generation of new, often redundant scripts over the systematic refinement and reuse of existing code." The codebase-as-memory design makes this pathology visible and quantifiable. Figure 7 quantifies "throwaway files"—files created but never referenced again—with Claude 4.5 Sonnet generating 18 such files per tournament on average.

**Fundamental vs. Incremental:** This is a **fundamental innovation** in what we measure. Prior benchmarks assess code correctness but ignore codebase hygiene. CodeClash reveals that models can produce winning code while simultaneously creating unmaintainable repositories—a critical distinction for real-world deployment where long-term maintainability matters as much as immediate functionality.

### 4.3 Exposing Strategic Reasoning Limitations Through Log Analysis

**What makes this different:** Existing benchmarks rarely examine *how* models arrive at their solutions. They measure outcomes (pass/fail, win/loss) but not the reasoning process. CodeClash's multi-round structure with detailed competition logs enables analysis of whether models can **interpret feedback, diagnose failures, and validate changes**.

**Why it's significant:** The analysis in Section 5.2 reveals systematic failures that outcome-only metrics would miss. Figure 8(a) shows that for most models, edits are "ungrounded in more than 65% of all rounds"—meaning changes are not motivated by analysis of previous round logs or testing. Figure 8(b) quantifies hallucinated loss causality, where Claude Sonnet 4.5 "makes uncorroborated claims about the exact reason a game was lost in more than 17% of rounds on average," rising to 46% in BattleSnake specifically.

Perhaps most damning is Figure 8(c): "only Claude Sonnet 4.5 validates changes in a majority of rounds (56%), whereas Gemini 2.5 Pro and o3 perform meaningful validation in one out of five rounds." This reveals that models routinely deploy untested code despite having access to arena simulation tools and explicit prompts suggesting validation.

Figure 4 compounds this finding by showing models' inability to recover after losses. Even for Claude Sonnet 4.5, "losing a single round results in a comeback probability of less than one third—less than half of the overall round win rate of 71%." After five consecutive defeats, comeback rates fall below 15%. This suggests models cannot effectively reconsider strategies or adapt to opponents after initial approaches fail.

**Fundamental vs. Incremental:** This is a **fundamental innovation** in diagnostic capability. Prior benchmarks cannot distinguish between a model that wins through strategic reasoning versus one that wins through luck or opponent weakness. CodeClash's trajectory analysis reveals that even winning models often lack grounded reasoning processes—a critical insight for understanding where improvement efforts should focus.

### 4.4 Quantifying the Human Performance Gap in Strategic Tasks

**What makes this different:** Most coding benchmarks compare models against each other or against fixed test suites. Human performance baselines are rare, and when they exist, they typically measure task completion speed or accuracy on well-specified problems. CodeClash's ablation in Section 4.1 directly pits the top-performing model against an expert human-written bot in RobotRumble.

**Why it's significant:** The results are stark. "Claude Sonnet 4.5 is dominated by gigachad, winning exactly zero of the 150 rounds." Across 37,500 individual simulations (150 rounds × 250 simulations per round), the top model's code wins zero times against a static human-written solution. The human submission used was ranked fourth overall on RobotRumble's public leaderboard, with three closed-source submissions ranking higher still.

This finding contextualizes all other results. Models may show diverse development styles and achieve varying Elo ratings against each other, but "these limitations are stark: top models lose every round against expert human programmers." The gap is not marginal—it is categorical.

**Fundamental vs. Incremental:** This is a **fundamental innovation** in benchmarking rigor. By establishing a human expert ceiling, CodeClash prevents the field from mistaking relative model improvements for absolute capability. A model can improve its Elo rating against other models while remaining utterly incapable of competing with human strategic reasoning. This distinction is essential for setting realistic expectations about autonomous software engineering capabilities.

### 4.5 Synthesis: What These Innovations Reveal Together

Taken together, these four innovations paint a coherent picture that no single prior benchmark could produce:

| Capability | Model Performance | Human Baseline | Gap |
|------------|------------------|----------------|-----|
| Code correctness | Strong (85%+ bash success) | N/A | Minimal |
| Strategic adaptation | Weak (<30% comeback after loss) | Strong (100% win vs. top model) | Severe |
| Codebase maintenance | Poor (linear file growth) | Stable (plateaus after setup) | Severe |
| Feedback interpretation | Weak (65%+ ungrounded edits) | Strong (expert-level bots) | Severe |

The critical insight is that **technical proficiency does not imply strategic competence**. Models execute bash commands reliably and produce syntactically correct code, but they struggle with the higher-order capabilities that distinguish expert software engineers: interpreting noisy feedback, validating changes systematically, maintaining organized repositories over time, and adapting strategies based on competitive outcomes.

This distinction has profound implications for how the research community should prioritize improvement efforts. Optimizing for code generation accuracy alone will not close the gap revealed by CodeClash. The benchmark identifies specific capability deficits—log interpretation, change validation, long-horizon planning—that require targeted architectural or training interventions beyond current approaches.

## 5. Experimental Analysis

This section walks through how CodeClash evaluates models, what the results actually show, and whether the experiments support the paper's central claims. We will examine the evaluation methodology in detail, present specific quantitative results with proper context, analyze ablation studies that test key design choices, and critically assess where the evidence is strong versus where limitations remain.

### 5.1 Evaluation Methodology

**Models and Scale**

CodeClash evaluates 8 frontier language models selected based on their estimated performance on existing coding benchmarks. The model roster includes:

- **Anthropic family**: Claude Sonnet 4.5 and Claude Sonnet 4
- **OpenAI family**: GPT-5, GPT-5-mini, and o3
- **Google**: Gemini 2.5 Pro
- **x.ai**: Grok Code Fast 1
- **Alibaba**: Qwen3-Coder

The experimental scale is substantial. For the main leaderboard, models compete in one-on-one tournaments with the following configuration:

- **Tournaments per model pair per arena**: 10
- **Rounds per tournament**: 15
- **Total arenas**: 6
- **Total model pairs**: $\binom{8}{2} = 28$

This yields $\binom{8}{2} \times 6 \times 10 \times 15 = 25,200$ total rounds for the main leaderboard. Tournament runtime varies by arena, averaging 75 minutes per tournament, which totals approximately 2.4 million hours of runtime when accounting for model latency. The experiments are parallelized across independent tournaments.

**Why this scale matters:** Prior coding benchmarks typically evaluate models on hundreds or thousands of individual tasks. CodeClash's 25,200 rounds provide statistical power to detect meaningful differences between models while accounting for the inherent variance in competitive settings. The 15-round tournament structure also enables analysis of how models adapt over time, which single-shot benchmarks cannot capture.

**Metrics and Statistical Methods**

CodeClash uses two primary metrics to quantify performance:

**Win Rate**: The fraction of tournaments a model has won. A tournament is won by the model that wins more rounds than its opponent. If both models win equally many rounds, the model that scores the last win takes the tournament. This tiebreaker rule handles the rare case where draws occur.

**Elo Ratings**: To quantify absolute model strength beyond pairwise win rates, CodeClash uses Elo scores based on the Bradley-Terry model. The probability that model $i$ wins over model $j$ is modeled as:

$$P(\text{model } i \text{ wins over } j) = \frac{1}{1 + \exp(s_j - s_i)} = \sigma(s_i - s_j)$$

where $s_i$ and $s_j$ represent player strengths and $\sigma$ is the logistic function. Optimal strength values are calculated using maximum likelihood fit to win counts $w_{ij}$:

$$\log L = \sum_{i<j} \left[ w_{ij} \log \sigma(s_i - s_j) + w_{ji} \log \sigma(s_j - s_i)) \right]$$

Player strengths are converted to Elo scores using chess conventions with base rating $R_0 = 1200$ and slope $\beta = 400$:

$$R_i = R_0 + \frac{\beta}{\log 10} s_i$$

**Critical design choice:** Instead of sequential Elo updates (which depend on update order and step size), CodeClash performs a maximum likelihood fit to all win rates simultaneously. This produces more stable rankings. The paper validates rank stability through both parametric and non-parametric bootstrapping with 1,000 samples each, observing more than 98% pairwise order agreement (Table 4).

**Arena Diversity**

The 6 code arenas represent fundamentally different competitive domains, as summarized in Table 2:

| Arena | Language | Objective | Players |
|-------|----------|-----------|---------|
| BattleSnake | Python | Survival (last snake standing) | 2+ |
| Core War | Redcode | Elimination (disable opponent processes) | 2+ |
| Halite | C/C++/OCaml/Rust | Territory control | 2+ |
| Poker | Python | Chip maximization | 2+ |
| RoboCode | Java | Tank combat scoring | 2+ |
| RobotRumble | JavaScript | Unit count after 100 turns | 2 |

This diversity is intentional. As Table 1 shows, no single model dominates all arenas—Claude Sonnet 4.5 places just 4th in Poker despite leading overall. This prevents models from succeeding through domain-specific optimization and tests general adaptive capability.

### 5.2 Main Quantitative Results

**Overall Rankings**

Table 1 presents Elo ratings per model per arena. The overall rankings show clear stratification:

**Arena-Specific Performance**

The arena-by-arena breakdown in Table 1 reveals important nuances:

- **Claude Sonnet 4.5** dominates Core War (1641 Elo) and BattleSnake (1470 Elo) but underperforms in Poker (1248 Elo)

This variation supports the paper's claim that diverse arenas are necessary for comprehensive evaluation. A benchmark using only Poker would crown GPT-5 as the top model, while a Core War-only benchmark would favor Claude Sonnet 4.5 by a massive margin. The overall ranking emerges from aggregating across domains, which better reflects general adaptive capability.

**Temporal Dynamics**

Figure 3 tracks win rates across the 15 rounds of tournaments, revealing how models gain or lose momentum over time:

- **Claude Sonnet 4.5** shows steady improvement, starting around 60% win rate in early rounds and reaching above 75% by round 15
- **GPT-5** exhibits the opposite pattern, starting strong (above 70%) but declining to around 60% by round 15
- **o3** remains relatively flat throughout

This pattern suggests Claude Sonnet 4.5 is better at long-horizon adaptation—learning from competition logs and refining strategies across rounds. GPT-5's decline indicates it may exhaust its initial strategic ideas without developing effective countermeasures to opponent adaptations.

**Recovery After Losses**

Figure 4 presents one of the most revealing findings: models struggle dramatically to recover after losing rounds. The chart shows the probability of winning the next round after losing several consecutive rounds:

- **Claude Sonnet 4.5**: Overall win rate is 71%, but after a single loss, comeback probability drops to less than 33% (less than half the baseline). After five consecutive defeats, comeback rates fall below 15%.
- **o3**: Overall win rate is 65%, but drops to only 26% after a single loss.
- **All other models**: Comeback rates after five consecutive losses fall below 10%.

**Why this matters:** In real-world software development, early approaches often fail, and the ability to diagnose failures and pivot strategies is critical. Figure 4 demonstrates that even the strongest models cannot effectively reconsider their strategies after initial approaches fail. This is not a marginal weakness—it represents a fundamental limitation in adaptive reasoning.

### 5.3 Behavioral Analysis: How Models Work

Beyond win rates, CodeClash enables detailed analysis of how models interact with their codebases. These behavioral metrics reveal differences that outcome-only benchmarks would miss.

**Edit Patterns**

Section 5.1 quantifies how models modify their codebases:

- **Files edited per round**: o3 and Gemini 2.5 Pro typically edit only 2 files per round, while GPT-5 changes 5-6 files
- **Lines changed per round**: o3 adds/removes approximately 51 lines per round, which is 8× less than Qwen3-Coder or the Claude Sonnet family (both modify more than 400 lines)
- **Thought verbosity**: Gemini 2.5 Pro generates an average of 105 words per thought, more than double the average. o3 is most terse at under 19 words per response (though the paper notes o3's actual token count is likely underestimated due to hidden reasoning tokens)
- **Turn consumption**: Claude Sonnet 4.5 uses 23 of the allotted 30 editing turns per round, while GPT-5 and o3 typically conclude after just 15 steps

**Critical finding:** The paper explicitly states "we did not find any correlations between any of these behaviors and win rates." Both minimalists (o3) and high-activity editors (Claude Sonnet 4.5) succeed. This challenges assumptions about what coding behaviors indicate competence—more edits or longer reasoning traces do not necessarily produce better outcomes.

**Codebase Degradation Over Time**

Figure 6 shows that "the average number of agent-created files scales almost linearly with the number of rounds" rather than plateauing as human-managed codebases do. Specific numbers:

- **Claude Sonnet 4.5**: Averages more than 30 files per tournament
- **GPT-5**: Averages 21 files per tournament
- **o3**: Creates fewer than 5 files per tournament

Figure 7 quantifies "throwaway files"—files created but never referenced, reused, or modified in subsequent rounds:

- **Claude Sonnet 4.5**: 18 throwaway files per tournament
- **GPT-5**: 15 throwaway files per tournament
- **o3**: Near the bottom (minimal throwaway files)

Figure 51 tracks filename redundancy (files sharing name prefixes like `analyze_round_1.py`, `analyze_round_2.py`), which climbs across all models. Qwen3-Coder shows the highest redundancy at 59%, followed by Claude Sonnet models at approximately 35%.

**Interpretation:** Models favor "continual generation of new, often redundant scripts over the systematic refinement and reuse of existing code." This is a fundamental maintainability issue. Human developers typically establish a codebase structure early and then refine it. Models instead accumulate technical debt with every round, creating repositories that would become increasingly difficult to maintain in production settings.

**Strategic Reasoning Quality**

Section 5.2 uses an LM-as-a-judge approach to analyze whether models' edits are grounded in actual analysis. Figure 8 presents three critical metrics:

**Figure 8(a) - Groundedness of Edits**: Whether changes to solutions are motivated by analysis of previous rounds or testing.

- **Claude Sonnet models**: Most edits can be motivated by prior analysis
- **All other models**: Edits are ungrounded in more than 65% of all rounds
- **o3**: Particularly low, with ungrounded edits in almost 80% of rounds

**Figure 8(b) - Hallucinated Loss Causality**: Whether models make uncorroborated claims about why rounds were lost.

- **Claude Sonnet 4.5**: Makes uncorroborated claims in more than 17% of rounds on average
- **BattleSnake specifically**: Claude Sonnet 4 and 4.5 hallucinate about loss causality in 34% and 46% of rounds respectively
- **Most hallucinations**: Misinterpretations or over-interpretations of log files

**Figure 8(c) - Validation of Edits**: Whether changes are validated by arena simulations or unit tests.

- **Claude Sonnet 4.5**: Validates changes in 56% of rounds (the only model above 50%)
- **GPT-5**: Validates in 50% of rounds
- **Gemini 2.5 Pro and o3**: Perform meaningful validation in only one out of five rounds (approximately 20%)

**Why this matters:** The prompt explicitly suggests running arena simulations between different versions of code or writing unit tests to validate intended behavior. Despite this guidance and the availability of tools, most models deploy untested code. This reveals a gap between capability (models can write tests) and practice (models choose not to validate systematically).

### 5.4 Ablation Studies

The paper includes several ablation experiments that test key design choices and reveal additional insights.

**Human Baseline Comparison (Section 4.1)**

This is the most significant ablation. The authors identified the top open-source RobotRumble submission as of October 31, 2025—a bot called `gigachad` authored by `entropicdrifter`, ranked fourth overall on the public leaderboard with 1554 Elo. Three closed-source submissions ranked higher, with the top submission nearly 700 Elo points above `gigachad`.

The experiment pitted Claude Sonnet 4.5 (the #1 model on RobotRumble) against `gigachad` in 10 tournaments of 15 rounds each. The human-written bot remained static—no optimization between rounds.

**Results:**
- Claude Sonnet 4.5 won exactly **zero of the 150 rounds**
- Across 150 × 250 = **37,500 individual simulations**, Claude Sonnet 4.5's code won zero times

**Interpretation:** This is not a close competition. The paper states: "These limitations are stark: top models lose every round against expert human programmers." The gap is categorical, not marginal. Even after 15 rounds of adaptation against a static opponent, the model could not develop any effective countermeasures.

**Limitation to note:** The paper acknowledges this result is "drawn from a limited sample." Leaderboards for other arenas (Core War, RoboCode) do not exist or lack readily open-source ranked submissions. The RobotRumble result is compelling but represents one arena. Future work should expand human baselines across multiple domains.

**Transparent Codebases (Section 4.1)**

The default CodeClash setting keeps opponent codebases opaque, reflecting real-world closed-source development. The ablation lifts this restriction: before the edit phase of round $n$, each player receives a read-only copy of their opponent's code from round $n-1$.

The experiment ran 10 Core War tournaments of 15 rounds each among Claude Sonnet 4.5, GPT-5, and Gemini 2.5 Pro.

**Results:**
- **GPT-5**: 74.6% win rate (+7.8% from default setting)
- **Claude Sonnet 4.5**: 53.2% win rate (−1.8% from default)
- **Gemini 2.5 Pro**: 22.7% win rate (−5.5% from default)

**Access patterns:**
- **GPT-5**: Accesses opponent codebase in only 12.8% of rounds
- **Claude Sonnet 4.5**: Accesses in 99.3% of rounds
- **Gemini 2.5 Pro**: Accesses in 52.9% of rounds

**Interpretation:** Frequent inspection of opponent code does not necessarily translate to competitive advantage. Claude Sonnet 4.5 inspects opponent code almost every round but performs worse than in the opaque setting. GPT-5 rarely inspects opponent code but gains the most from transparency. This suggests that simply having access to opponent code is insufficient—models must also have the capability to extract actionable insights from it, which appears to vary significantly.

**Multi-Player Tournaments (Section 4.1)**

The main leaderboard uses one-on-one tournaments. The paper also ran 20 Core War tournaments with 6 players (excluding GPT-5-mini and Claude Sonnet 4) to test multi-player dynamics. Performance was quantified using the TrueSkill rating system since Elo is limited to pairwise comparisons.

**Results:**
- Rankings were similar to the 2-player Core War results in Table 1, with GPT-5 and Grok Code Fast (two models of similar Elo) switching positions
- **Lead changes** (round $n$ winner different from round $n-1$): 48.4% in 6-player tournaments vs. 18.2% in 2-player tournaments
- **Winner's share of total points**: 28.6% on average in 6-player settings vs. 78.0% in 2-player settings

**Interpretation:** Multi-player tournaments exhibit far more competitive volatility. The reduced winner's share indicates that no single model can dominate when facing multiple adaptive opponents simultaneously. This better reflects real-world competitive dynamics where multiple competitors evolve strategies in parallel. The paper notes this setting could serve as a testbed for studying coalition dynamics, positional play, and risk management in future work.

### 5.5 Critical Assessment: Do the Experiments Support the Claims?

**Claim 1: Models exhibit diverse development styles**

**Evidence:** Strong support. Figures 29-36 in Appendix D.1 document substantial variation in files edited, lines changed, turn consumption, and thought verbosity across models. Figure 5 shows solutions become increasingly dissimilar round over round, with code similarity dropping from approximately 0.63 (o3, round 1) to 0.41 (GPT-5, round 1) and continuing to decline.

**Assessment:** The data convincingly demonstrates diversity. The finding that these behavioral differences do not correlate with win rates is particularly valuable—it prevents over-interpreting specific coding styles as indicators of competence.

**Claim 2: Models share fundamental limitations in strategic reasoning**

**Evidence:** Strong support from multiple angles:
- Figure 4 shows comeback probabilities below 33% after single losses (vs. 65-71% baseline win rates)
- Figure 8(a) shows 65%+ of edits are ungrounded for most models
- Figure 8(b) quantifies hallucinated loss causality at 17-46% depending on model and arena
- Human baseline ablation shows 0/150 round wins against expert human code

**Assessment:** The evidence is compelling and multi-faceted. The human baseline comparison is particularly persuasive because it establishes an absolute ceiling rather than relative model rankings. The consistency of limitations across different models (despite varying architectures and training) suggests these are systemic issues rather than model-specific bugs.

**Claim 3: Models struggle with long-term codebase maintenance**

**Evidence:** Strong quantitative support:
- Figure 6 shows linear file growth across rounds (not plateauing as human codebases do)
- Figure 7 quantifies throwaway files (18 per tournament for Claude Sonnet 4.5)
- Figure 51 shows filename redundancy climbing to 59% for Qwen3-Coder
- Section 5.1 notes "most agent-created files are never referenced, reused, or modified in subsequent rounds"

**Assessment:** The evidence is clear and quantified. The linear growth pattern is particularly damning because it indicates models do not develop sustainable development practices over time. However, the paper does not explore whether this affects eventual performance—models might create messy codebases but still win tournaments. The maintainability issue is demonstrated but its impact on competitive outcomes is not fully established.

**Claim 4: Top models lose every round against expert human programmers**

**Evidence:** Strong but limited. The RobotRumble ablation shows 0/150 round wins for Claude Sonnet 4.5 against the `gigachad` bot across 37,500 simulations.

**Assessment:** The result is unambiguous within its scope. However, the paper acknowledges this is "drawn from a limited sample" since leaderboards for other arenas do not exist or lack open-source submissions. The claim is supported for RobotRumble but should not be overgeneralized to all competitive coding domains without additional human baseline experiments.

**Claim 5: CodeClash elicits substantial creativity from models**

**Evidence:** Moderate support. Figure 5 shows code similarity decreasing across rounds, indicating models produce diverse solutions even against the same opponent. The paper notes "models' solutions become increasingly dissimilar round over round, even when facing the same opponent in the same arena."

**Assessment:** The evidence shows diversity but does not fully establish that this diversity represents "creativity" in a meaningful sense. Diverse solutions could indicate exploration or could indicate instability and lack of convergent optimization. The paper acknowledges this interpretation requires further investigation.

### 5.6 Limitations and Open Questions

**Limited Human Baselines**

As noted above, human expert comparisons exist only for RobotRumble. The paper states: "Leaderboards for other arenas do not exist (Core War, RoboCode) or do not have readily open source, ranked submissions (Halite, BattleSnake, Poker)." This limits the generalizability of the human performance gap claim.

**Arena Complexity**

The paper acknowledges in Section 7 that "CodeClash's code arenas are relatively smaller and more self-contained than most real-world software systems." Real-world software development often involves larger codebases, multiple competing objectives, and more complex stakeholder dynamics. The arenas test strategic reasoning but may not capture the full scope of production software engineering.

**Text-Only Feedback**

Section 7 notes: "logs from the competition phase are entirely text-based. We don't explore Vision Language Models (VLMs) in this work." Some arenas could potentially benefit from visual feedback (e.g., visualizing BattleSnake board states). This limitation may affect models differently depending on their multimodal capabilities.

**Agent Scaffold Choice**

The paper intentionally uses mini-SWE-agent rather than tool-heavy scaffolds to avoid confounding model capabilities with scaffold optimizations. However, Section 7 acknowledges "a simple next step could be to swap out mini-SWE-agent with tool-based frameworks to maximize AI systems' performance." The current results reflect model capabilities within a specific scaffold—different scaffolds might produce different behavioral patterns.

**Cost and Turn Limits**

The 30-turn limit and $1 cost limit per round force agents to allocate their context budget carefully. The paper observes the cost limit is "almost never reached" but turn limits "are exhausted frequently for specific models." It is unclear whether these constraints meaningfully limit model performance or whether models would exhibit similar behaviors with more generous budgets.

### 5.7 Summary of Experimental Contributions

CodeClash's experimental design makes several distinctive contributions:

1. **Scale and Statistical Power**: 25,200 rounds across 8 models and 6 arenas provides robust statistical grounding for rankings and behavioral analysis.

2. **Multi-Dimensional Evaluation**: Beyond win rates, the benchmark quantifies codebase maintenance quality, strategic reasoning groundedness, and adaptation dynamics across rounds.

3. **Human Baseline**: The RobotRumble ablation establishes an absolute performance ceiling, preventing misinterpretation of relative model improvements as absolute capability gains.

4. **Ablation Rigor**: Transparent codebase and multi-player ablations test key design choices and reveal nuanced insights about how models leverage (or fail to leverage) additional information.

5. **Behavioral Diagnostics**: The LM-as-a-judge analysis of edit groundedness, hallucination rates, and validation practices exposes failure modes that outcome-only metrics would miss.

The experiments convincingly support the paper's central claims about model limitations in strategic reasoning and codebase maintenance. The primary limitation is the narrow scope of human baseline comparisons, which should be expanded in future work. Nevertheless, the evidence is sufficient to establish CodeClash as a meaningful benchmark for evaluating autonomous, goal-oriented software engineering capabilities.

## 6. Limitations and Trade-offs

CodeClash represents a significant advance in benchmarking autonomous software engineering, but the approach rests on several assumptions and faces practical constraints that shape what conclusions we can draw. This section examines the benchmark's limitations honestly—distinguishing between inherent trade-offs in the design choices and gaps that future work should address.

### 6.1 Assumptions Embedded in the Design

**Codebase-as-Memory Assumes Uniform Retrieval Capability**

The benchmark deliberately removes external memory mechanisms, forcing models to record insights in their codebase. This design assumes all models have comparable ability to retrieve and utilize information they previously wrote. However, this may disadvantage models with weaker long-context retrieval or file navigation capabilities independent of their strategic reasoning ability.

The paper acknowledges this implicitly by noting that "how to represent such knowledge is left entirely to the model's discretion" (Section 2.3). But if one model naturally organizes files with clear naming conventions while another creates redundant files like `analyze_round_13_v2.py` (Figure 53), the latter may struggle to retrieve its own prior insights—not because of poor strategic reasoning, but because of poor information architecture. This confounds codebase organization skill with strategic adaptation capability.

**Mini-SWE-Agent as a Neutral Scaffold**

The decision to use `mini-SWE-agent` rather than tool-heavy frameworks like OpenHands reflects an intentional choice to "evaluate models, not scaffolds or tools" (Appendix A). However, this assumes the scaffold itself does not meaningfully constrain model behavior.

Evidence suggests this assumption may not fully hold. The paper notes that "turn limits are exhausted frequently for specific models" with Claude Sonnet 4.5 typically using 23 of 30 turns while GPT-5 and o3 conclude after just 15 steps (Section 3). If the 30-turn limit constrains Claude Sonnet 4.5 more than other models, the scaffold introduces differential constraints that affect comparative rankings. The paper states the cost limit is "almost never reached," but turn limits clearly bind for some models more than others.

**Arena Non-Determinism Handled by Simulation Volume**

To account for non-determinism, each round runs 1,000 simulations (Appendix A). This assumes 1,000 samples sufficiently stabilize outcomes. For most arenas, this appears reasonable—Figure 24 shows relatively tight score distributions. However, the paper notes MIT Battlecode 2025 showed evidence of positional advantage requiring player order shuffling to fix. This suggests some arenas may have hidden sources of variance that 1,000 simulations do not fully eliminate.

**Strategic Opacity as Default**

The default setting keeps opponent codebases opaque, reflecting "real world settings, where human players develop their solutions independently" (Section 4.1). This assumes closed-source development is the more relevant benchmark condition. However, many real-world software teams work with open-source dependencies and can study competitor APIs or public repositories. The ablation showing GPT-5 gains +7.8% win rate with transparent codebases suggests opacity may artificially constrain some models more than others.

### 6.2 Scenarios and Problem Settings Not Addressed

**Limited Human Baseline Coverage**

The most significant gap is human performance comparison. Section 4.1 states: "Leaderboards for other arenas do not exist (Core War, RoboCode) or do not have readily open source, ranked submissions (Halite, BattleSnake, Poker)." The RobotRumble human baseline—while striking at 0/150 round wins for Claude Sonnet 4.5—represents only one of six arenas.

This limitation matters because the paper's claim that "top models lose every round against expert human programmers" cannot be generalized beyond RobotRumble without additional evidence. The human submission used (`gigachad` by `entropicdrifter`) was ranked fourth overall on RobotRumble's leaderboard with 1554 Elo, and "three additional, closed source submissions rank above" (Appendix D.3.4). If the top human submission is ~700 Elo points higher than `gigachad`, the actual human-model gap may be even larger—but this remains unquantified.

**Arena Complexity vs. Real-World Software**

Section 7 explicitly acknowledges: "CodeClash's code arenas are relatively smaller and more self-contained than most real-world software systems." The arenas range from single-file bots (BattleSnake `main.py`) to modest multi-file projects (RoboCode Java classes). None approach the scale of production codebases with:

- Hundreds or thousands of files
- Complex dependency graphs
- Multiple stakeholders with conflicting objectives
- Regulatory or compliance constraints
- Legacy code integration requirements

The paper expresses interest in "tougher settings, where SWE-agents manage larger codebases attempting to win multiple competitive objectives (e.g., city planning, disaster preparedness, cybersecurity)" (Section 7). Until such arenas exist, CodeClash measures strategic reasoning in simplified environments that may not capture the full complexity of autonomous software engineering.

**Text-Only Feedback Excludes Multimodal Capabilities**

Section 7 notes: "logs from the competition phase are entirely text-based. We don't explore Vision Language Models (VLMs) in this work." This is a meaningful limitation because:

- Some arenas (BattleSnake, RoboCode, RobotRumble) have natural visual representations
- Visual feedback might enable different strategic approaches (e.g., pattern recognition in opponent movement)
- Modern multimodal models cannot leverage their full capability set

The paper states "supporting multimodal feedback is on the road-map for future investigations," but current results apply only to text-only model interactions. This may disadvantage models with strong vision capabilities and advantage text-specialized models.

**Single-Objective Optimization Per Arena**

Each arena has a single primary objective (survival, territory control, chip maximization). Real-world software development often involves competing objectives: optimize for performance while maintaining readability, reduce costs while improving reliability, ship quickly while minimizing technical debt.

CodeClash does not test models' ability to balance trade-offs between conflicting goals. A model might excel at maximizing a single metric while producing unmaintainable code—but the benchmark does not penalize this if the model wins tournaments. Figure 6 shows linear file growth across rounds, yet this does not appear to significantly harm winning performance for top models. This suggests the benchmark may not adequately incentivize sustainable development practices.

### 6.3 Computational and Scalability Constraints

**Prohibitive Runtime for Replication**

The main leaderboard required approximately 2.4 million hours of runtime, "mostly due to model latency" (Section 3). Even with parallelization across independent tournaments, this represents an enormous computational burden:

- **25,200 total rounds** for the main leaderboard
- **75 minutes average** per tournament
- **1,000 simulations** per round to handle non-determinism

This scale creates significant barriers:

1. **Replication**: Few research groups can afford to reproduce these experiments, especially with closed-source API models where costs scale with token usage
2. **Iteration**: Rapid benchmark iteration (adding arenas, testing new models) is computationally expensive
3. **Accessibility**: The paper open-sources CodeClash, but practical use requires substantial compute resources

The paper mentions using Princeton Language & Intelligence (PLI) credits for closed-source API models (Acknowledgments), indicating non-trivial financial cost beyond compute time.

**Model API Dependency**

Six of eight evaluated models are closed-source API offerings (Claude family, GPT family, Gemini, Grok). This creates several constraints:

- **Reproducibility**: Model behavior may change between API versions without notice
- **Cost Sensitivity**: Results may shift if cost limits are adjusted (currently $1 per round)
- **Availability**: Some models may become unavailable or change pricing, affecting future evaluations

The paper notes the cost limit "is almost never reached," suggesting current budgets accommodate model behavior. However, as models improve and potentially use more tokens for analysis, cost constraints may become more binding.

**Limited Multi-Player Exploration**

The main leaderboard uses one-on-one tournaments. Section 4.1 explores 6-player Core War tournaments but notes this was limited: "we do not run comprehensive experiments due to both cost limitations and the analytical complexity introduced by multi-way competition."

The 6-player results show meaningful differences:
- **Lead changes**: 48.4% in 6-player vs. 18.2% in 2-player tournaments
- **Winner's share**: 28.6% average in 6-player vs. 78.0% in 2-player (Section 4.1)

These differences suggest multi-player dynamics reveal distinct capabilities (coalition formation, risk management, positional play) that 2-player tournaments cannot capture. However, computational constraints limit this exploration to one arena with six of eight models. The paper acknowledges "future work that can leverage CodeClash's multi-player tournaments as a testbed for understanding strategic behaviors" but does not provide comprehensive multi-arena, multi-player results.

### 6.4 Weaknesses and Open Questions

**Disconnect Between Codebase Quality and Winning**

A troubling finding is that poor codebase maintenance does not clearly penalize tournament performance. Claude Sonnet 4.5 creates the most files (>30 per tournament) and the most throwaway files (18 per tournament), yet achieves the highest overall Elo (1389). Meanwhile, o3 creates fewer than 5 files per tournament with minimal throwaway files but ranks third (1343 Elo).

The paper states "we did not find any correlations between any of these behaviors and win rates" (Section 5.1). This raises an open question: **does codebase organization matter for long-term success?** The 15-round tournament may be too short to observe degradation from technical debt. Real-world software projects span months or years—CodeClash cannot yet test whether messy codebases eventually impair adaptation.

**Unclear Whether Diversity Indicates Creativity or Instability**

Section 5.1 claims "CodeClash elicits substantial creativity from models" based on decreasing code similarity across rounds (Figure 5). However, diversity could indicate:

- **Healthy exploration**: Models testing different strategies to find optimal approaches
- **Unstable optimization**: Models failing to converge on effective solutions, constantly rewriting code without systematic improvement

The paper acknowledges this ambiguity: "Unlike existing code benchmarks where models quickly converge on canonical solutions, CodeClash elicits substantial creativity from models." But without measuring whether diverse solutions correlate with improvement (rather than just change), "creativity" remains an interpretation rather than a demonstrated capability.

**LM-as-Judge Validation Concerns**

Section 5.2 uses "GPT-5 with high reasoning as a judge" to analyze agent trajectories for groundedness, hallucination, and validation metrics. This introduces potential circularity:

- The judge model may share similar blind spots as evaluated models
- Structured output schemas (Appendix D.3.2) constrain what failure modes can be detected
- The judge cannot access model thoughts for o-series models where "OpenAI's API is configured to hide intermediate thinking tokens" (Appendix D.1)

The paper does not validate judge accuracy against human annotations. While LM-as-judge is standard practice, the specific metrics (e.g., "ungrounded edits") rely on judge interpretation of what constitutes adequate log analysis.

**Unknown Long-Horizon Behavior Beyond 15 Rounds**

All main results use 15-round tournaments. Figure 3 shows some models gain momentum (Claude Sonnet 4.5) while others lose it (GPT-5) across these 15 rounds. But whether these trends continue, plateau, or reverse beyond round 15 remains unknown.

The paper expresses interest in "long-running, autonomous software development systems" (Section 7), yet the benchmark does not test horizons long enough to observe:

- Whether codebase degradation eventually impairs performance
- Whether models develop sustainable development practices over time
- Whether strategic adaptation continues indefinitely or saturates

**Limited Arena Diversity in Key Dimensions**

While six arenas span different game types, they cluster in some dimensions:

- **All are competitive**: No cooperative or mixed-motive scenarios
- **All have clear win conditions**: No ambiguous success criteria like real-world business metrics
- **All are turn-based or discrete**: No real-time pressure beyond arena mechanics
- **All use synthetic environments**: No integration with real systems, APIs, or user feedback

The paper mentions potential expansion to "cybersecurity, healthcare, and city planning" (Appendix D.4), but current results apply only to game-like competitive environments. Whether findings generalize to other domains remains an open empirical question.

### 6.5 Summary of Trade-offs

| Design Choice | Benefit | Cost |
|--------------|---------|------|
| Codebase-as-memory | Makes organization measurable | May confound retrieval ability with reasoning |
| Mini-SWE-agent | Evaluates models, not scaffolds | Turn limits bind differentially across models |
| Opaque opponent code | Reflects closed-source development | May artificially constrain some models |
| 1,000 simulations/round | Stabilizes non-deterministic outcomes | Massive computational cost (2.4M hours) |
| 15-round tournaments | Enables adaptation analysis | Too short to observe long-term degradation |
| Text-only logs | Simplifies infrastructure | Excludes multimodal model capabilities |
| Game-like arenas | Clear win conditions, reproducible | Less complex than real-world software |

CodeClash makes deliberate trade-offs to enable evaluation of capabilities no prior benchmark measures. These trade-offs are reasonable for a first-generation benchmark but should be acknowledged when interpreting results. The benchmark's greatest value may be establishing a foundation for iterative improvement—adding human baselines across arenas, extending tournament length, incorporating multimodal feedback, and expanding to more complex software domains.

## 7. Implications and Future Directions

CodeClash does more than introduce a new benchmark—it reframes how we should evaluate language models for autonomous software engineering. This section examines how the work shifts the research landscape, what specific follow-up studies it enables, where practitioners should integrate these methods, and concrete guidance for adopting this evaluation paradigm.

### 7.1 Shifting the Evaluation Paradigm

**From Task Completion to Goal Pursuit**

The most significant conceptual shift CodeClash introduces is moving evaluation from **task completion** to **goal pursuit**. Traditional benchmarks ask: "Can the model fix this bug?" or "Can the model implement this function?" CodeClash asks: "Can the model figure out what to fix, how to improve, and whether changes actually help—without being told?"

This distinction matters because it exposes capabilities that binary benchmarks cannot measure. As Section 5.2 demonstrated, models achieve 85%+ bash command success rates and produce syntactically correct code, yet Figure 8 reveals that 65%+ of edits are ungrounded in actual analysis. A model can be technically proficient while strategically incompetent. CodeClash makes this gap visible and quantifiable.

**From Absolute Correctness to Relative Performance**

Prior benchmarks rely on absolute correctness metrics—does the code pass unit tests? Does the patch resolve the issue? CodeClash replaces this with **relative performance grading**. As Section 4.1 explains, "CodeClash's competitive setting with constantly evolving opponents provides a perpetual learning signal that doesn't saturate."

This has two implications:

1. **No Ceiling Effect**: Traditional benchmarks saturate once models achieve high pass rates. CodeClash's relative grading means improvement is always possible—there is always a better strategy to discover.

2. **Adaptation Matters**: A model cannot succeed through static optimization. Figure 3 shows Claude Sonnet 4.5 improving across rounds while GPT-5 declines, revealing adaptation capability that single-shot benchmarks miss.

**From Code Quality to Codebase Hygiene**

Perhaps the most novel contribution is making **codebase organization a measurable capability**. Section 5.1 quantifies what no prior benchmark captures:

- File creation rates scaling linearly with rounds (Figure 6)
- Throwaway file counts (18 per tournament for Claude Sonnet 4.5, Figure 7)
- Filename redundancy ratios (59% for Qwen3-Coder, Figure 51)

The paper notes that "in most human-managed codebases, the rate of file creation quickly plateaus once the overall structure has been established" (Section 5.1). Models do the opposite. This finding alone should reshape how we think about AI coding assistants—correctness without maintainability is insufficient for production deployment.

**From Outcome Metrics to Process Diagnostics**

CodeClash enables **trajectory analysis** that reveals how models arrive at solutions, not just whether they succeed. The LM-as-judge methodology in Section 5.2 produces three diagnostic metrics no prior benchmark offers:

| Metric | What It Measures | Why It Matters |
|--------|-----------------|----------------|
| Groundedness (Figure 8a) | Whether edits are motivated by analysis | Distinguishes strategic reasoning from guessing |
| Hallucination Rate (Figure 8b) | Whether models make uncorroborated claims | Reveals overconfidence and log misinterpretation |
| Validation Rate (Figure 8c) | Whether changes are tested before deployment | Shows engineering discipline and risk awareness |

These metrics enable targeted improvement efforts. Rather than broadly "improving coding capability," researchers can now focus on specific deficits like log interpretation or change validation.

### 7.2 Enabled Research Directions

CodeClash opens several concrete research avenues that were previously infeasible. Below we outline specific follow-up studies the benchmark enables, with methodological details drawn from the paper's own suggestions.

**1. Extended Human Baseline Studies**

The RobotRumble ablation (Section 4.1) shows Claude Sonnet 4.5 winning 0 of 150 rounds against expert human code. However, the paper acknowledges this is "drawn from a limited sample" since "leaderboards for other arenas do not exist (Core War, RoboCode) or do not have readily open source, ranked submissions (Halite, BattleSnake, Poker)" (Section 4.1).

**Follow-up study design:**
- Recruit expert developers for all 6 arenas
- Run 10 tournaments per arena (matching main leaderboard scale)
- Measure not just win rates but adaptation speed, codebase organization, and validation practices
- Quantify the human-model gap across domains to identify where models are closest to human parity

This would transform the claim from "models lose in RobotRumble" to a comprehensive map of human-AI capability gaps across competitive coding domains.

**2. Long-Horizon Tournament Extensions**

All main results use 15-round tournaments. Figure 3 shows momentum shifts within these 15 rounds, but Section 6.4 notes it remains "unknown whether these trends continue, plateau, or reverse beyond round 15."

**Follow-up study design:**
- Run 50-round and 100-round tournaments with top models
- Track codebase degradation metrics (file count, redundancy, throwaway ratio) across extended horizons
- Measure whether messy codebases eventually impair performance
- Identify if models develop sustainable practices over time or continue accumulating technical debt

This addresses a core limitation: the 15-round tournament may be too short to observe whether poor codebase hygiene eventually causes failure.

**3. Multimodal Feedback Integration**

Section 7 explicitly states: "logs from the competition phase are entirely text-based. We don't explore Vision Language Models (VLMs) in this work." Some arenas (BattleSnake, RoboCode, RobotRumble) have natural visual representations that text logs cannot fully capture.

**Follow-up study design:**
- Extend arena logging to include board state snapshots, replay videos, or visual heatmaps
- Evaluate multimodal models (GPT-4V, Gemini Pro Vision, Claude with vision) against text-only baselines
- Measure whether visual feedback improves strategic reasoning (e.g., pattern recognition in opponent movement)
- Quantify the value-add of multimodal capabilities for competitive coding tasks

This would determine whether current text-only evaluation systematically underestimates multimodal model capabilities.

**4. Training on CodeClash Traces**

Section 6 suggests CodeClash could serve as "a reliable, extensible training ground for evaluating and building the next generation of long-running, autonomous software development systems." The paper specifically mentions "pre-training on traces of models' edits or post-training with techniques like self-play and reinforcement learning" (Section 7).

**Follow-up study design:**
- Collect full trajectories (thoughts, actions, outputs, logs) from CodeClash tournaments
- Fine-tune models on successful vs. unsuccessful trajectories
- Implement self-play where models compete against their own previous versions
- Use reinforcement learning with tournament win rate as reward signal
- Measure whether training on CodeClash traces improves strategic reasoning on held-out arenas

This transforms CodeClash from purely evaluative to a training infrastructure, potentially closing the capability gaps the benchmark exposes.

**5. Scaffold Optimization Studies**

The paper intentionally uses `mini-SWE-agent` to "evaluate models, not scaffolds or tools" (Appendix A). However, Section 7 acknowledges "a simple next step could be to swap out mini-SWE-agent with tool-based frameworks (Yang et al., 2024a; Wang et al., 2025b) to maximize AI systems' performance."

**Follow-up study design:**
- Run identical tournaments with different agent scaffolds (mini-SWE-agent, OpenHands, SWE-agent, custom toolkits)
- Measure whether scaffold choice affects strategic reasoning quality or just execution efficiency
- Test whether providing pre-built analysis tools improves log interpretation vs. requiring models to synthesize tools
- Quantify the scaffold contribution to overall performance

This would disentangle model capability from scaffold capability—critical for understanding where improvement efforts should focus.

**6. Multi-Player Strategic Dynamics**

Section 4.1 explores 6-player Core War tournaments but notes comprehensive multi-player experiments were limited "due to both cost limitations and the analytical complexity introduced by multi-way competition." The results show meaningful differences: lead changes occur 48.4% of the time in 6-player vs. 18.2% in 2-player tournaments.

**Follow-up study design:**
- Run full 6-arena, 8-model multi-player tournaments
- Develop metrics for coalition formation, risk management, and positional play
- Analyze whether models exhibit game-theoretic behaviors (tit-for-tat, bluffing, cooperation)
- Measure if multi-player performance correlates with 2-player performance

This would extend CodeClash beyond dyadic competition to capture the complexity of multi-agent software ecosystems.

### 7.3 Practical Applications and Use Cases

**Evaluating Autonomous Coding Agents Before Deployment**

Organizations deploying AI coding assistants should use CodeClash to assess **strategic reasoning capability** before production use. Traditional benchmarks (HumanEval, SWE-bench) measure whether models can complete specified tasks. CodeClash measures whether models can **determine which tasks matter and how to prioritize them**.

**When to prefer CodeClash:**
- Evaluating agents for autonomous development workflows (not just code completion)
- Assessing long-horizon planning capability
- Testing adaptation to changing requirements or competitive conditions
- Measuring codebase maintainability over time

**When traditional benchmarks suffice:**
- Evaluating code completion or suggestion features
- Testing specific bug-fixing capabilities
- Measuring syntax correctness or API usage

**Generating Training Data for Strategic Reasoning**

CodeClash trajectories provide rich training data for improving model strategic reasoning. The 25,200 rounds generate diverse examples of:
- Successful vs. unsuccessful adaptation strategies
- Grounded vs. ungrounded edit decisions
- Effective vs. ineffective log analysis approaches

**Integration approach:**
- Extract trajectories where models successfully recovered from losses
- Use LM-as-judge annotations (Section 5.2) to label grounded vs. ungrounded edits
- Fine-tune models on high-quality trajectories
- Validate improvement on held-out CodeClash tournaments

**Identifying Specific Capability Gaps in AI Assistants**

CodeClash's diagnostic metrics enable targeted capability assessment. Rather than asking "is this model good at coding?", practitioners can ask:

- Does this model validate changes before deployment? (Figure 8c metric)
- Does this model hallucinate failure explanations? (Figure 8b metric)
- Does this model maintain organized codebases over time? (Figure 6, 7 metrics)
- Does this model recover after setbacks? (Figure 4 metric)

**Assessment workflow:**
1. Run 10 tournaments (150 rounds) with candidate model
2. Compute diagnostic metrics using LM-as-judge pipeline
3. Compare against baseline models (Claude Sonnet 4.5, GPT-5, o3)
4. Identify specific weakness areas for targeted improvement

**Benchmarking Continuous Improvement Systems**

Organizations building self-improving AI systems need benchmarks that don't saturate. CodeClash's relative grading and multi-round structure make it suitable for tracking improvement over time.

**Tracking protocol:**
- Establish baseline Elo ratings across 6 arenas
- Run monthly evaluation tournaments with updated models
- Track both overall Elo and arena-specific performance
- Monitor codebase hygiene metrics to ensure improvement doesn't come at maintainability cost

### 7.4 Reproducibility and Integration Guidance

**Accessing CodeClash**

The paper states: "We release CodeClash as an open source toolkit, including the code, arena logs, and a leaderboard" (Section 1). Specific resources:

- **Code repository**: `https://github.com/CodeClash-ai/CodeClash`
- **Trajectory viewer and leaderboard**: `https://codeclash.ai`
- **Arena documentation**: Appendix B provides detailed arena cards for all 6 environments

**Computational Requirements**

The full main leaderboard required approximately **2.4 million hours of runtime**, "mostly due to model latency" (Section 3). Practitioners should understand the resource implications:

| Evaluation Scale | Rounds | Estimated Runtime | Use Case |
|-----------------|--------|-------------------|----------|
| Quick sanity check | 150 (10 tournaments) | ~1,000 hours | Initial model screening |
| Arena-specific evaluation | 600 (40 tournaments, 1 arena) | ~4,000 hours | Domain-specific assessment |
| Full leaderboard replication | 25,200 | ~2.4M hours | Comprehensive benchmarking |

**Cost considerations:**
- The paper used Princeton Language & Intelligence (PLI) credits for closed-source API models (Acknowledgments)
- Cost limit is $1 per round, but paper notes this is "almost never reached" (Appendix C.1)
- Open-weight models (Qwen3-Coder, Grok Code Fast) reduce API costs but require local inference infrastructure

**When to Use CodeClash vs. Alternatives**

| Scenario | Recommended Benchmark | Rationale |
|----------|----------------------|-----------|
| Evaluating code completion features | HumanEval, MBPP | Faster, task-specific, well-established |
| Testing bug-fixing capability | SWE-bench | Real GitHub issues, clear pass/fail |
| Assessing autonomous development | **CodeClash** | Measures strategic reasoning, adaptation |
| Evaluating code optimization | EffiBench, SWE-Perf | Focused on performance metrics |
| Testing game-playing AI | GameArena, VideoGameBench | Direct play, not code-as-proxy |
| Long-horizon planning assessment | **CodeClash** | Multi-round tournaments reveal adaptation |
| Codebase maintainability evaluation | **CodeClash** | Only benchmark measuring organization over time |

**Integration Checklist**

For teams adopting CodeClash:

- [ ] **Define evaluation scope**: Which arenas match your use case? (BattleSnake for survival logic, Poker for decision-making under uncertainty, etc.)
- [ ] **Select models**: Include both closed-source API models and open-weight models for comparison
- [ ] **Configure tournament parameters**: 15 rounds is standard; consider 30+ for long-horizon assessment
- [ ] **Set up infrastructure**: Docker containers for agent and arena environments (Appendix A)
- [ ] **Plan compute budget**: Estimate runtime based on model latency and tournament scale
- [ ] **Implement diagnostic pipeline**: Use LM-as-judge methodology for groundedness, hallucination, validation metrics
- [ ] **Establish baselines**: Compare against published Elo ratings (Table 1) for context

**Common Pitfalls to Avoid**

1. **Insufficient simulation volume**: Each round should run 1,000 simulations to account for non-determinism (Appendix A). Fewer simulations introduce variance that obscures true capability differences.

2. **Ignoring codebase metrics**: Win rate alone misses critical maintainability signals. Track file creation, redundancy, and throwaway ratios alongside tournament outcomes.

3. **Overgeneralizing from single arenas**: Table 1 shows no model dominates all arenas. Claude Sonnet 4.5 leads overall but places 4th in Poker. Multi-arena evaluation is essential.

4. **Neglecting human baselines**: Where possible, compare against expert human implementations. The RobotRumble ablation (0/150 wins) provides crucial context for model rankings.

5. **Assuming 15 rounds is sufficient**: For long-horizon capability assessment, consider extended tournaments. Figure 3 shows some models improve while others decline across 15 rounds—trends may continue or reverse beyond this horizon.

### 7.5 Summary: What CodeClash Changes

CodeClash shifts the field's focus in four fundamental ways:

| Dimension | Before CodeClash | After CodeClash |
|-----------|-----------------|-----------------|
| **Evaluation objective** | Task completion | Goal pursuit |
| **Success metric** | Binary correctness | Relative performance |
| **Measured capability** | Code generation | Strategic reasoning + codebase maintenance |
| **Learning signal** | Saturating (pass/fail) | Perpetual (adaptive competition) |

The benchmark reveals that **technical proficiency does not imply strategic competence**. Models execute commands reliably and produce correct code, but struggle with interpreting feedback, validating changes, maintaining organized repositories, and adapting strategies based on outcomes.

For researchers, CodeClash provides a testbed for studying autonomous software engineering capabilities that no prior benchmark measures. For practitioners, it offers diagnostic tools to assess whether AI coding assistants are ready for autonomous deployment. For the field broadly, it establishes that evaluating AI for software engineering requires moving beyond task completion to measure the higher-order capabilities that distinguish expert developers from competent coders.

The work's greatest contribution may be establishing a foundation for iterative improvement. As the paper concludes: "We hope CodeClash will serve as a reliable, extensible training ground for evaluating and building the next generation of long-running, autonomous software development systems" (Section 7). The benchmark is not an endpoint—it is infrastructure for the next phase of AI software engineering research.
