## 1. Executive Summary

This survey addresses the limitation of large language models (LLMs) in open-ended environments by formalizing the paradigm of Agentic Reasoning, which reframes LLMs as autonomous agents that plan, act, and learn through continual interaction rather than static inference. It provides a systematic roadmap organized along three complementary dimensions—foundational, self-evolving, and collective multi-agent reasoning—while distinguishing between two optimization modes: in-context orchestration and post-training methods like reinforcement learning. This work matters because it synthesizes diverse methods into a unified framework, offering actionable guidance for developing adaptive systems across domains such as science, robotics, and healthcare, and identifies key future challenges like personalization and governance.

## 2. Context and Motivation

### The Core Problem: Static Reasoning in Dynamic Worlds

The fundamental gap this survey addresses is a **mismatch between how large language models reason and how real-world problems unfold**. Current LLMs excel at closed-world reasoning tasks—solving math problems from GSM8K [455], generating code from fixed specifications, or answering questions from static contexts. However, these capabilities rest on a critical assumption: that all necessary information is present in the input prompt and that the problem can be solved in a single forward pass or a few reasoning steps.

> "Conventional LLMs lack mechanisms to act, adapt, or improve in open-ended environments where information evolves over time." (Section 1)

This limitation becomes stark when agents face **dynamic, partially observable environments** where:
- Information must be actively gathered through interaction (e.g., browsing the web, querying databases)
- Plans must be revised when execution fails or new evidence emerges
- Long-horizon tasks require maintaining state across many interaction turns
- Tools and external systems must be invoked to extend the model's native capabilities

The paper identifies this as a **paradigm-level problem**, not merely a capability gap. Traditional LLM reasoning treats inference as passive sequence generation, while agentic reasoning reframes it as **sequential decision-making under uncertainty** [8, 9, 10].

### Why This Problem Matters

The importance of bridging this gap operates at two levels:

**Theoretical Significance.** The transition from static to agentic reasoning represents a conceptual shift in how we understand machine intelligence. Table 1 in Section 2 contrasts the two paradigms across six dimensions:

| Dimension | LLM Reasoning | Agentic Reasoning |
|-----------|---------------|-------------------|
| Paradigm | passive | interactive |
| Input | static | dynamic context |
| Computation | single pass | multi-step |
| Statefulness | context window | external memory |
| Learning | offline pretraining | continual improvement |
| Goal Orientation | prompt-based | explicit planning |

This reframing positions reasoning not as an internal computation trace but as **the organizing principle linking perception, planning, decision, and verification**. The theoretical contribution lies in unifying these previously fragmented capabilities under a single control-theoretic framework (Section 2.2), where the agent's policy factors into internal thought ($\pi_{\text{reason}}$) and external action ($\pi_{\text{exec}}$):

$$\pi_\theta(z_t, a_t \mid h_t) = \pi_{\text{reason}}(z_t \mid h_t) \cdot \pi_{\text{exec}}(a_t \mid h_t, z_t)$$

This decomposition makes explicit the "think–act" structure that agentic systems require.

**Real-World Impact.** The practical stakes are equally high. Domains where agentic reasoning is essential include:

- **Scientific Discovery:** Agents must formulate hypotheses, run experiments, interpret results, and revise theories across iterative cycles (Section 6.2)
- **Healthcare:** Clinical decision-making requires integrating patient history, querying medical databases, and adapting to evolving symptoms under strict safety constraints (Section 6.4)
- **Robotics:** Embodied agents must ground language in physical perception, execute multi-step manipulation tasks, and recover from execution failures (Section 6.3)
- **Autonomous Research:** End-to-end research workflows demand literature search, experiment design, code execution, and manuscript writing coordinated across long horizons (Section 6.5)

In each case, static reasoning fails because the environment provides feedback that must shape subsequent decisions. A medical agent cannot diagnose correctly without querying patient records; a robotics agent cannot complete a task without verifying whether actions succeeded.

### Limitations of Prior Approaches

The survey explicitly positions itself against two bodies of existing work, each with distinct shortcomings:

**LLM Reasoning Surveys** [51, 52, 53, 54] focus on eliciting or enhancing reasoning **within a model's internal computation process**. These works summarize techniques like:
- Chain-of-Thought prompting [1]
- Reinforcement post-training for reasoning [60]
- Long-context reasoning methods

The critical limitation: these approaches **remain within static inference settings**, where reasoning unfolds in a single forward pass without external interaction. They improve the internal expressiveness of reasoning traces but do not address how agents should act when information is incomplete or environments change.

**AI Agent Surveys** [56, 57, 58, 59] examine LLM-based agents from **architectural or system perspectives**. These works analyze:
- Reinforcement learning for agentic search and decision-making [56, 57]
- Self-evolving and lifelong agentic systems [58, 59]
- Tool-use modules and planning architectures

The critical limitation: these surveys treat reasoning as an **implicit by-product of architectural design** rather than the central mechanism being studied. They describe what components exist but do not systematically explain how reasoning drives adaptation, feedback integration, and collaboration.

### How This Survey Positions Itself

This paper occupies a distinct niche at the **intersection of model-centric reasoning and system-level intelligence**. Its positioning has three distinguishing features:

**1. Reasoning as the Unifying Mechanism.** Rather than cataloging agent architectures or reasoning techniques separately, the survey treats reasoning as **the process that links single-agent reinforcement, multi-agent coordination, and self-evolving intelligence**. This reasoning-centric lens explains why certain architectural choices matter: memory structures enable long-horizon reasoning, feedback mechanisms support reasoning refinement, and communication protocols extend reasoning across agents.

**2. Three-Layer Taxonomy Across Environmental Dynamics.** The survey organizes agentic reasoning along three complementary scopes (Section 1, Figure 1):
- **Foundational Agentic Reasoning:** Core single-agent capabilities (planning, tool use, search) operating in stable environments
- **Self-Evolving Agentic Reasoning:** Mechanisms for feedback, memory, and adaptation in evolving settings
- **Collective Multi-Agent Reasoning:** Coordination, role assignment, and shared knowledge in collaborative scenarios

This taxonomy reflects how environmental complexity scales—from individual agents solving fixed tasks to populations co-evolving through interaction.

**3. Dual Optimization Modes.** Across all three layers, the survey distinguishes between (Section 1, "Survey Scope" box):
- **In-Context Reasoning:** Scaling inference-time interaction through structured orchestration without parameter updates
- **Post-Training Reasoning:** Internalizing reasoning strategies into model weights via reinforcement learning and fine-tuning

This distinction matters because it clarifies **when to modify the model versus when to modify the workflow**—a practical design decision that prior surveys conflate.

### What This Survey Does Not Claim

The authors are explicit about scope boundaries (Section 2.1):
- This is **not** a comprehensive catalog of all LLM reasoning techniques (that is covered by [51, 52, 53, 54])
- This is **not** purely an architectural survey of agent systems (that is covered by [56, 57, 58, 59])
- This **does** focus on how reasoning transforms static inference into adaptive, interactive, and continually improving intelligence

The survey summarizes progress up to 2025 and provides a GitHub repository for ongoing updates (§ Github: https://github.com/weitianxin/Awesome-Agentic-Reasoning).

### The Gap This Survey Fills

In summary, the problem this paper addresses is the **absence of a unified framework** that explains how reasoning mechanisms enable agents to bridge thought and action across varying environmental dynamics. Prior work either:
- Studies reasoning in isolation from action (LLM reasoning surveys)
- Studies architectures without centering reasoning as the explanatory mechanism (agent surveys)

This survey fills that gap by providing a **reasoning-centric roadmap** that connects foundational capabilities to self-evolution and collective intelligence, while offering practical guidance on optimization modes (in-context vs. post-training) and application domains. The result is a framework that helps practitioners understand not just what agentic systems exist, but **why certain reasoning mechanisms work** and **how to design agents for specific environmental constraints**.

## 3. Technical Approach

This survey paper does not propose a single new algorithm but instead formalizes a **unified technical framework** for understanding how large language models transition from passive text generators to active reasoning agents. The core idea is that agentic reasoning emerges from systematically combining three capability layers—foundational planning and tool use, self-evolving feedback and memory, and collective multi-agent coordination—each realizable through either inference-time orchestration or parameter-level optimization via reinforcement learning.

### 3.1 Reader orientation (approachable technical breakdown)

The system being described is a **reasoning architecture for LLM-based agents** that transforms static language models into interactive problem-solvers capable of planning, acting, learning from feedback, and collaborating with other agents. It solves the problem of LLMs failing in dynamic, open-ended environments by providing a structured framework that specifies how agents should think before acting, how they should store and retrieve experience, and how multiple agents should coordinate their reasoning processes.

### 3.2 Big-picture architecture (diagram in words)

The agentic reasoning framework consists of **five interconnected technical components** that operate across three capability layers:

1. **Policy Factorization Module** (Section 2.2): Decomposes the agent's decision-making into internal reasoning traces ($z_t$) and external actions ($a_t$), making the think-act structure explicit rather than implicit.

2. **Foundational Capability Engine** (Section 3): Implements core single-agent abilities including planning (goal decomposition and sequencing), tool use (API invocation and execution), and search (retrieval-augmented generation with dynamic control).

3. **Self-Evolution Loop** (Section 4): Enables continual improvement through feedback mechanisms (reflective critique, parametric adaptation, validator-driven retry) and memory systems (flat, structured, and post-training controlled).

4. **Multi-Agent Coordination Layer** (Section 5): Manages role assignment (leader, worker, critic, memory keeper), collaboration topology (hierarchical, decentralized, hybrid), and collective evolution across agent populations.

5. **Optimization Mode Selector** (Sections 3-5): Determines whether capabilities are realized through in-context orchestration (frozen parameters, structured prompting) or post-training optimization (reinforcement learning, supervised fine-tuning).

Information flows through these components as follows: the policy factorization module receives observations and produces reasoning-action pairs; the foundational engine executes these pairs using tools and search; the self-evolution loop captures outcomes as feedback and memory entries; the multi-agent layer distributes tasks across specialized agents; and the optimization mode selector determines whether improvements happen at inference time or through parameter updates.

#### Formal Framework: Agentic Reasoning as Sequential Decision-Making

The paper formalizes agentic reasoning using a **Partially Observable Markov Decision Process (POMDP)** framework, which is a mathematical model for decision-making when the agent cannot directly observe the full state of the environment. The formalization introduces a tuple $\langle \mathcal{X}, \mathcal{O}, \mathcal{A}, \mathcal{Z}, \mathcal{M}, \mathcal{T}, \Omega, R, \gamma \rangle$ where each symbol represents a specific component of the agent-environment interaction. $\mathcal{X}$ denotes the latent environment state space that remains unobservable to the agent, such as the true configuration of a database or the actual health status of a patient. $\mathcal{O}$ represents the observation space containing what the agent can perceive, including user queries, API returns, and sensor readings. $\mathcal{A}$ is the external action space consisting of tool invocations, API calls, and final answers that affect the environment. $\mathcal{Z}$ defines the reasoning trace space where internal thoughts, plans, and chain-of-thought sequences live, optionally verbalized as natural language. $\mathcal{M}$ represents the agent's internal memory or context space, which serves as a sufficient statistic of the interaction history. $\mathcal{T}$ and $\Omega$ denote the transition and observation kernels that govern how the environment changes and what the agent observes. $R$ is the reward function that evaluates outcomes, and $\gamma \in (0, 1)$ is the discount factor that determines how much future rewards matter relative to immediate ones.

At each timestep $t$, the agent conditions on a history $h_t = (o_{\leq t}, z_{<t}, a_{<t})$, meaning the agent observes $o_t$ before generating reasoning $z_t$ and then action $a_t$. The history can equivalently be summarized by an internal memory state $m_t \in \mathcal{M}$. The critical technical innovation is the **policy factorization** that separates internal thought from external action:

$$\pi_\theta(z_t, a_t \mid h_t) = \pi_{\text{reason}}(z_t \mid h_t) \cdot \pi_{\text{exec}}(a_t \mid h_t, z_t)$$

This equation states that the probability of producing a reasoning-action pair equals the probability of generating internal thought multiplied by the probability of executing an action given that thought. The decomposition highlights the core shift in agentic systems: computation happens in $\mathcal{Z}$ (thinking) before committing to $\mathcal{A}$ (acting). The objective remains maximizing expected return $J(\theta) = \mathbb{E}_\tau\left[\sum_{t \geq 0} \gamma^t r_t\right]$, where $\tau$ represents a trajectory of interactions.

This formalization matters because it makes explicit what traditional LLM reasoning leaves implicit: the distinction between deliberation and execution, the role of memory in summarizing history, and the sequential nature of agent-environment interaction. Alternative formulations that conflate context with environment state cannot capture the think-act structure that agentic policies require.

#### Foundational Capability Engine: Planning, Tool Use, and Search

The foundational layer establishes three core capabilities that transform a static LLM into an autonomous problem-solver, and each capability can be implemented through either in-context or post-training methods.

**Planning Reasoning** enables agents to decompose problems, sequence decisions, and navigate complex environments with foresight. In-context planning methods structure the overall planning process into distinct stages such as perception, reasoning, execution, and verification, which are either explicitly scaffolded or learned implicitly. Workflow-based approaches like ReWOO [71] decouple reasoning from observations for efficiency, while methods like ToT (Tree of Thoughts) [4] apply breadth-first or depth-first strategies to explore structured thought trees. Tree-search strategies including BFS, DFS, A*, MCTS (Monte Carlo Tree Search), and beam search have become prominent as interpretable and effective planning scaffolds, with over 12 distinct MCTS variations cited in Table 2 [112-123]. Process formalization approaches encode plans as code-like artifacts or PDDL (Planning Domain Definition Language) programs, enabling downstream tool execution and cleaner interfaces with classical planners [138-142]. Post-training planning methods design suitable reward structures and solve for optimal behavior using reinforcement learning or control-theoretic tools, with works like Reflexion [14] incorporating utility-based learning to guide planning behavior.

**Tool-Use Optimization** addresses the agent's capacity to augment intrinsic capabilities by intelligently invoking external modules, overcoming limitations like outdated knowledge or inability to perform precise calculations. The core challenge lies in reasoning about when to use a tool, which tool to select from a library, and how to generate a valid call. In-context tool-integration leverages the model's existing in-context learning ability by providing carefully crafted instructions and examples directly in the prompt. ReAct [5] introduced the "Reasoning+Acting" synergy where reasoning creates and tracks action plans while actions interface with external environments. ART [199] maintains a library of successful task demonstrations and retrieves relevant multi-step exemplars as few-shot prompts. Post-training tool-integration uses supervised fine-tuning over curated tool-use reasoning steps, with Toolformer [6] introducing a self-supervised framework where models generate, validate, and retain useful API calls within unlabeled text. ToolLLM [203] scales SFT training to over 16,000 real-world APIs, applying supervised fine-tuning on massive curated demonstrations. Recent studies leverage reinforcement learning during post-training to achieve mastery beyond imitation, with methods like ToolRL [207] introducing principled reward designs for stable and scalable multi-tool learning. Orchestration-based tool-integration extends beyond single-model, single-tool settings to require coordination among multiple tools, with HuggingGPT [7] employing a centralized agent that plans which tools to invoke and when.

**Agentic Search** embeds reasoning and control into a centralized agent that governs the entire retrieval-generation loop, differing from traditional RAG (Retrieval-Augmented Generation) pipelines that perform fixed, one-shot retrieval before generation. In-context agentic RAG systems embed retrieval behavior directly into inference through carefully designed prompting strategies, with ReAct [5] interleaving Chain-of-Thought reasoning with tool-use commands like `<Search>` to dynamically invoke external APIs. Extensions like Self-Ask [254] and IRCoT [213] prompt the model to recursively decompose questions and retrieve sub-evidence accordingly. Structure-enhanced agentic RAG systems enable reasoning over symbolic knowledge sources like knowledge graphs through dynamic querying, with Agent-G [262] integrating unstructured document retrieval with structured graph reasoning using feedback loops. Post-training agentic RAG methods fine-tune models via supervised fine-tuning or reinforcement learning to determine when retrieval is necessary, how to formulate queries, and how to incorporate retrieved evidence. WebGPT [258] introduces reward modeling to supervise search-augmented chains aligned with human judgment, while Search-R1 [27] and Deep-Researcher [260] train agents to dynamically issue retrieval actions and operate in open-ended environments like the live web.

#### Self-Evolution Loop: Feedback Mechanisms and Memory Systems

The self-evolving layer enables agents to improve their own reasoning processes through experience, with feedback and memory serving as the two fundamental mechanisms that transform reasoning from static to dynamic and adaptive.

**Agentic Feedback Mechanisms** operate through three distinct regimes that define a continuum between dynamic inference-time adaptability, durable learning through parameter updates, and efficient correction through external signals. Reflective feedback methods improve model reliability by modifying the reasoning process during inference without updating model parameters, exposing intermediate reasoning outputs and introducing additional assessment steps that directly influence how the model continues generation. Early self-critique methods implement reflection through an explicit generate-critique-revise loop where a model first produces an answer with reasoning, then analyzes this output to identify logical errors or missing steps, and appends the critique as context for revised generation [14, 270]. Search-based reasoning strategies improve reliability by generating and comparing multiple candidate reasoning paths, then selecting or aggregating outputs using voting schemes or heuristic scores [4, 74, 272]. Parametric adaptation incorporates feedback into model parameters through additional training, producing persistent behavioral changes that generalize beyond individual inference episodes. Trajectory-level supervised fine-tuning approaches attach feedback to intermediate reasoning traces rather than only final answers, with models generating multi-step trajectories that are reviewed by humans, auxiliary models, or automated verifiers [274, 103]. Preference-alignment approaches incorporate feedback as comparative judgments distinguishing preferred from dispreferred outputs, with training objectives like reward modeling or direct preference optimization adjusting parameters so preferred behaviors become more likely [276-278]. Validator-driven feedback improves outputs using external success or failure signals without modifying the model's reasoning process or parameters, with retry-based systems repeatedly sampling candidate outputs until one passes validation [281, 282]. This strategy is particularly effective in domains with reliable and inexpensive validation like program synthesis, where generated code can be executed against unit tests providing unambiguous correctness signals [283-285].

**Agentic Memory** shifts focus from static memory storage to dynamic, interactive mechanisms that directly support reasoning, treating memory as an integral component of the reasoning loop rather than merely extending the context window. Formally, an agent maintains a memory module where each entry may represent a raw observation, summarized trajectory, subgoal, tool invocation trace, or other structured element depending on system design. The agent's reasoning process then operates not only on immediate context but also on this persistent memory, enabling reflection, generalization, and long-term goal tracking. In-context use of flat memory captures textual and semantic information from prior interactions, with traditional systems treating memory as a passive buffer mainly storing dialogue histories or recent observations [252, 293, 296, 297]. Emerging agentic memory treats memory as part of the reasoning loop, with Amem [24] enabling LLM agents to autonomously generate contextual memory descriptions and build dynamic links between related experiences. Experience memory tracks procedural traces to enable plan recovery and consistent reasoning, with Workflow Memory [298] supporting plan recovery and long-term consistency through explicit procedural trace tracking. Structured memory representations use semantic graphs, workflows, and hierarchical trees to better capture dependencies and contextual relationships, with GraphRAG [305] augmenting retrieval with graph-structured reasoning for more contextually coherent multi-hop information integration. Post-training memory control enables agents to evolve, update, and retrieve memory through learned reward-based mechanisms, with MemAgent [314] formulating memory overwrite as a reinforcement learning problem where the agent is rewarded for preserving useful information and discarding irrelevant content. Memory-R1 [15] introduces a dual-agent design with a Memory Manager that dynamically decides when to add, update, or delete entries and an Answer Agent that distills relevant retrieved memories to guide response generation.

#### Multi-Agent Coordination Layer: Roles, Collaboration, and Collective Evolution

The collective multi-agent layer extends foundational and self-evolving capabilities to collaborative settings where multiple reasoning agents interact to jointly solve complex tasks, introducing new challenges in role differentiation, communication, and collective memory.

**Role Taxonomy** defines generic roles that appear across multi-agent systems, with each role serving a specific function in the collaborative reasoning process. The Leader/Coordinator maintains high-level coherence by setting global objectives, decomposing tasks into manageable subgoals, and assigning them to appropriate agents while arbitrating conflicts between agents with overlapping outputs. The Worker/Executor serves as the operational backbone, engaging in concrete actions like invoking external tools, writing code, retrieving documents, or interfacing with the environment, often under leader directives but with adaptive autonomy in well-designed systems. The Critic/Evaluator centers on quality assurance by verifying correctness, testing hypotheses, red-teaming responses, and surfacing potential risks, often corresponding to LLM-as-a-judge setups in LLM-based systems. The Memory Keeper curates and maintains long-term knowledge structures like episodic logs, semantic embeddings, retrieval indices, or knowledge graphs, balancing short-term reactivity with long-term continuity. The Communication Facilitator governs protocols for inter-agent exchange including message schemas, communication bandwidth management, gating mechanisms, and consensus-building orchestration. Domain-specific roles reflect professional practices in particular industries, with software engineering mapping to architects, developers, code reviewers, CI orchestrators, and release managers [17, 350], while finance decomposes into analysts, risk managers, traders/execution agents, and compliance officers [353, 354].

**Collaboration and Division of Labor** organizes how responsibilities are decomposed and coordinated across specialized agents, with existing approaches organized along two dimensions based on optimization timing. In-context collaboration specifies or induces coordination strategies at inference time without additional training, with manually crafted pipelines relying on predefined hierarchies or fixed collaboration workflows where agent roles, execution order, and communication rules are determined before execution [16, 17, 374-381]. LLM-driven pipelines leverage LLMs as orchestrators that decompose high-level goals into subgoals, route them to role-specialized agents, and iteratively refine workflows based on intermediate feedback [382-385]. Agent routing explicitly models agent selection as a decision layer that selects appropriate specialists for each query or subtask, with AgentRouter [386] proposing a knowledge-graph-guided router and Talk to Right Specialists [387] framing routing and planning as unified inference-time processes. Post-training collaboration optimizes agent roles, interaction structures, or routing policies through learning or search, with prompt optimization focusing on how agent roles, workflows, and feedback are encoded in prompts [16, 18, 394-396]. Topology optimization treats multi-agent systems as graphs where agents are nodes and inter-agent communication forms edges, with graph generation methods constructing communication topologies from scratch [397-399], graph pruning methods starting from dense graphs and pruning to compact pipelines [400-402], and topology search methods exploring graph space to identify effective pipelines [332, 395, 403, 404, 68].

**Multi-Agent Evolution** extends self-evolution from single-agent to multi-agent co-evolution where adaptation emerges across distributed agents, targeting shared memory, communication mechanisms, and collaboration protocols beyond individual model parameters. Intra-test-time evolution refers to agents adapting and improving during task execution, enabling correction of failures and strategy refinement when facing unseen states or unexpected feedback, with methods leveraging natural-language self-critique [14, 270] and runtime adaptive planning [342, 431] to generate corrective signals without external supervision. Inter-test-time evolution extends self-improvement to across-task learning where adaptations from one task consolidate and transfer to future tasks, enabling accumulation of persistent, generalizable capabilities over lifelong interaction streams through offline self-distillation [337, 435, 436] and online reinforcement learning frameworks [344, 345]. Multi-agent memory management must jointly evolve along architecture, topology, content, and management dimensions, with hierarchical designs optimizing for global coherence and cross-episode learning while heterogeneous designs optimize for role fidelity and computational efficiency [425, 442, 443]. Storage topology balances scalability, privacy, and coherence through centralized approaches with verification gates [426], distributed approaches with private versus shared memory fragments [427], or uncontrolled pooling where all agents freely exchange experiences [444]. Training multi-agent systems to evolve represents a critical step toward adaptive, long-horizon intelligence beyond static coordination, with co-evolution via interaction and intrinsic feedback forming self-improving curricula that enable collective skill growth without external supervision [446, 451]. Multi-agent reinforcement fine-tuning frameworks address key mismatches between classical MARL assumptions and LLM-based agent organizations like role heterogeneity, dynamic coordination, and long-horizon dialogue [447, 448, 413].

#### Optimization Mode Distinction: In-Context Versus Post-Training

The optimization mode selector determines whether capabilities are realized through inference-time orchestration or parameter-level optimization, and this choice affects implementation across all previous layers.

**In-Context Reasoning** focuses on scaling inference-time compute through structured orchestration, search-based planning, and adaptive workflow design, enabling agents to navigate complex problem spaces dynamically without modifying model parameters. Methods like ReAct [5] perform greedy decoding over alternating thoughts $z$ and actions $a$, while Tree-of-Thoughts [4] and related MCTS-style approaches treat partial thoughts as nodes $u \in \mathcal{U}$ and search for an optimal path:

$$\tau^\star \in \arg\max_\tau \sum_t \hat{v}_\phi(u_t)$$

where $\hat{v}_\phi$ is a heuristic evaluator or verifier. This corresponds to planning in $\mathcal{Z}$ without updating policy parameters, making it highly flexible and deployable but ultimately bounded by the inherent capabilities of the frozen LLM and context window length.

**Post-Training Reasoning** targets capability internalization by consolidating successful reasoning patterns or tool-use strategies into the model's weights via reinforcement learning and fine-tuning. While PPO (Proximal Policy Optimization) [62] is standard, GRPO (Group Relative Policy Optimization) [63]-based methods are widely used for reasoning tasks because GRPO eliminates the value network by constructing advantages from group-relative rewards. For a group of $G$ sampled outputs $\{y_i\}_{i=1}^G$ from the same prompt $q$, a common GRPO objective is:

$$\mathcal{L}_{\text{GRPO}}(\theta) = \mathbb{E}_{q \sim P(Q)}\left[\frac{1}{G}\sum_{i=1}^G\left(\min\left(\rho_i \hat{A}_i, \text{clip}(\rho_i, 1-\epsilon, 1+\epsilon)\hat{A}_i\right) - \beta D_{\text{KL}}(\pi_\theta \parallel \pi_{\text{ref}})\right)\right]$$

where $\rho_i = \frac{\pi_\theta(y_i|q)}{\pi_{\theta_{\text{old}}}(y_i|q)}$ and the group-normalized advantage is:

$$\hat{A}_i = \frac{r_i - \mu}{\sigma + \delta}, \quad \mu = \frac{1}{G}\sum_{j=1}^G r_j, \quad \sigma = \sqrt{\frac{1}{G}\sum_{j=1}^G (r_j - \mu)^2}$$

with $\delta > 0$ a small constant for numerical stability. Advanced methods like ARPO [64] and DAPO [65] extend this framework to handle sparse rewards and improve stability in complex tool-use environments via replay/rollout strategies and decoupled clipping. This paradigm optimizes $\theta$ to align the policy with long-horizon rewards $r_t$ including correctness and safety, yielding more robust, adaptive, and generalizable policies than in-context methods alone, often transferring effectively to out-of-domain tasks [240-244].

The distinction between these modes matters practically because it clarifies when to modify the model versus when to modify the workflow—a design decision that prior surveys conflate. In-context methods offer flexibility and rapid deployment without training costs, while post-training methods provide durable improvements that generalize beyond individual episodes at the cost of additional training and reduced flexibility.

## 4. Key Insights and Innovations

This survey makes several conceptual and practical contributions that distinguish it from prior work on LLM reasoning and agent architectures. Below, I identify the most significant innovations, explain what makes them fundamentally different from existing approaches, and clarify why they matter for both research and deployment.

### 4.1 Three-Layer Taxonomy Organized by Environmental Dynamics

**What It Is.** The survey introduces a taxonomy that organizes agentic reasoning along three complementary layers—**Foundational**, **Self-Evolving**, and **Collective Multi-Agent**—structured by how environmental complexity scales rather than by capability type alone (Section 1, Figure 1).

**What Makes It Different.** Prior surveys organize agent capabilities horizontally (e.g., "planning methods," "memory methods," "tool-use methods") without accounting for how environmental demands change across settings. This taxonomy instead recognizes that:

- **Foundational reasoning** operates in stable environments where the agent solves fixed tasks with known toolsets
- **Self-evolving reasoning** addresses evolving environments where the agent must adapt its own capabilities through feedback and memory
- **Collective reasoning** handles collaborative environments where multiple agents coordinate roles and share knowledge

This is a **fundamental innovation** rather than an incremental improvement because it shifts the design question from "what capabilities should an agent have?" to "what environmental dynamics must the agent handle?" This reframing has direct implications for system architecture: a healthcare agent requires self-evolving memory for longitudinal patient tracking, while a math solver may only need foundational planning.

**Why It Matters.** The taxonomy provides **actionable design guidance** that prior surveys lack. Practitioners can now match their environmental constraints to the appropriate reasoning layer:
- Static benchmark tasks → Foundational capabilities suffice
- Long-horizon interaction with feedback → Self-evolving mechanisms required
- Multi-stakeholder or distributed tasks → Collective coordination necessary

This prevents over-engineering (adding multi-agent complexity to single-user tasks) and under-engineering (deploying static agents in dynamic environments).

### 4.2 Dual Optimization Mode Framework (In-Context vs. Post-Training)

**What It Is.** Across all three reasoning layers, the survey systematically distinguishes between **in-context orchestration** (scaling inference-time interaction without parameter updates) and **post-training optimization** (internalizing reasoning strategies via reinforcement learning and fine-tuning) (Section 1, "Survey Scope" box; Section 2.2).

**What Makes It Different.** Existing surveys conflate these modes or treat them as implementation details. This framework elevates the distinction to a **first-class design decision** with clear trade-offs:

| Aspect | In-Context | Post-Training |
|--------|------------|---------------|
| Flexibility | High (prompt changes suffice) | Low (retraining required) |
| Deployment Cost | Minimal | Significant (compute, data) |
| Generalization | Bounded by frozen model | Durable across episodes |
| Debugging | Transparent (inspect prompts) | Opaque (inspect weights) |

This is a **fundamental innovation** because it clarifies a decision that practitioners currently make ad hoc. The survey shows that this choice cascades through all capability layers: planning can use tree-search (in-context) or RL-trained policies (post-training); memory can use retrieval augmentation (in-context) or learned memory control (post-training); multi-agent collaboration can use fixed role prompts (in-context) or co-evolved policies (post-training).

**Why It Matters.** The framework enables **resource-aware system design**. Organizations with limited training infrastructure can prioritize in-context methods while accepting their limitations. Teams with substantial compute can invest in post-training for durable improvements. Critically, the survey identifies where each mode excels: in-context methods dominate when tasks vary frequently (e.g., research agents adapting to new domains), while post-training methods excel when consistency matters (e.g., clinical agents requiring reliable behavior).

### 4.3 Policy Factorization as a Unifying Formalism

**What It Is.** Section 2.2 introduces a mathematical formalization that decomposes the agent's policy into internal reasoning ($\pi_{\text{reason}}$) and external execution ($\pi_{\text{exec}}$):

$$\pi_\theta(z_t, a_t \mid h_t) = \pi_{\text{reason}}(z_t \mid h_t) \cdot \pi_{\text{exec}}(a_t \mid h_t, z_t)$$

**What Makes It Different.** Prior agent frameworks describe architectures informally (e.g., "the agent thinks then acts") without formalizing the think-act structure. This factorization is a **theoretical innovation** because it:

1. **Makes deliberation explicit** as a distinct computational step in $\mathcal{Z}$ (reasoning trace space) before committing to $\mathcal{A}$ (action space)
2. **Enables modular analysis** where reasoning quality and execution reliability can be evaluated separately
3. **Supports hybrid optimization** where $\pi_{\text{reason}}$ might use in-context methods while $\pi_{\text{exec}}$ uses post-training (or vice versa)

Alternative formulations that conflate context with environment state cannot capture this structure, limiting their ability to analyze where failures originate (reasoning errors vs. execution errors).

**Why It Matters.** The formalism provides a **common language for comparing disparate methods**. When evaluating ReAct [5] versus Reflexion [14], researchers can now ask: do these methods modify $\pi_{\text{reason}}$, $\pi_{\text{exec}}$, or both? When designing new systems, engineers can ask: should we optimize the reasoning component or the execution component? This precision accelerates cumulative progress by enabling apples-to-apples comparisons that prior informal descriptions prevented.

### 4.4 Cross-Domain Instantiation Analysis

**What It Is.** Sections 6.1–6.5 analyze how the same reasoning mechanisms (planning, tool use, search, feedback, memory, collaboration) instantiate differently across five domains: math/code, scientific discovery, embodied robotics, healthcare, and autonomous research (Figure 11).

**What Makes It Different.** Domain-specific surveys exist for each area separately, but this work is the **first to systematically compare** how agentic reasoning adapts to different constraints:

- **Math/Code agents** emphasize symbolic verification and execution feedback (unit tests, compilers)
- **Scientific agents** prioritize hypothesis generation and experimental iteration loops
- **Embodied agents** require multimodal grounding and physical safety constraints
- **Healthcare agents** demand strict auditability and regulatory compliance
- **Research agents** coordinate long-horizon workflows across literature, code, and writing

This is a **fundamental innovation** because it reveals that agentic reasoning is not domain-agnostic: the same foundational capability (e.g., tool use) requires different implementations depending on whether tools are APIs (code), laboratory instruments (science), or robotic actuators (embodied).

**Why It Matters.** The analysis prevents **naive transfer** of methods across domains. A tool-use strategy that works for code generation (execute and check for errors) may fail in healthcare (where execution errors have safety implications). Practitioners can now identify which aspects of their system require domain-specific adaptation versus which can use general-purpose methods. This accelerates deployment by clarifying where customization is necessary.

### 4.5 Open Problems Framework with Specificity

**What It Is.** Section 8 identifies six open challenges—personalization, long-horizon interaction, world models, multi-agent training, latent reasoning, and governance—with concrete technical barriers rather than vague aspirations (Section 8.1–8.6).

**What Makes It Different.** Many surveys conclude with generic future directions (e.g., "improve efficiency," "enhance safety"). This framework is **actionably specific**:

- **Long-horizon credit assignment** is framed as assigning credit across tokens, tool calls, skills, and memory updates—not just "improve long-term reasoning"
- **World models** are identified as requiring joint training with agents over long time scales, not just "add simulation"
- **Governance** is specified as addressing planning-time failures and multi-agent dynamics, not just "ensure safety"

This is an **incremental but significant innovation** because it transforms open problems from research slogans into technical specifications that can guide funding decisions and research agendas.

**Why It Matters.** The specificity enables **targeted research investment**. Rather than pursuing "better agents" broadly, teams can focus on identified bottlenecks: How do we assign credit across 100+ interaction turns? How do we audit latent reasoning when intermediate steps are unobservable? How do we govern multi-agent ecosystems where failures emerge from interactions? This precision accelerates progress by directing effort toward solvable subproblems rather than ill-defined grand challenges.

### 5.1 Evaluation Methodology Landscape

The survey organizes agentic reasoning evaluation along two complementary perspectives (Section 7 introduction):

1. **Mechanism-Centric Benchmarks** (Section 7.1): Isolate core capabilities like tool use, search, memory, planning, and multi-agent coordination in controlled settings

### 5.4 Critical Assessment of Evaluation Practices

The survey reveals several **significant limitations** in current agentic reasoning evaluation:

#### 5.4.1 Lack of Standardized Metrics

> "Existing suites often differ in what they treat as the core capability, such as tool invocation accuracy, memory retention under long contexts, or coordination quality in multi-agent settings." (Section 7 introduction)

This fragmentation makes **cross-paper comparison difficult**. For example:
- Tool-use benchmarks range from binary success/failure (SWE-bench [285]) to six-dimensional assessment (UltraTool [695])
- Memory benchmarks use accuracy, recall, capacity, retrieval speed, F1, ROUGE, and GPT-4 scoring without standardization
- Multi-agent benchmarks measure win rates, social welfare, communication quality, and coordination accuracy with no unified framework

#### 5.4.2 Limited Long-Horizon Evaluation

The survey explicitly identifies **long-horizon credit assignment** as an open problem (Section 8.2):

> "While methods such as ReAct and Tree of Thought improve short-horizon reasoning [5, 4], errors still compound rapidly in long tasks, as illustrated by embodied agents like Voyager [36]. RL-trained agents such as WebRL and Agent-R1 improve performance in realistic environments but rely on heavily engineered, domain-specific rewards and largely treat episodes independently [437, 28]."

Most benchmarks focus on **single-episode or short-horizon tasks**, with limited evaluation of:
- Credit assignment across 100+ interaction turns
- Cross-episode learning and capability accumulation
- Memory retention over extended interaction streams (though LOCOMO [322] and LONGMEMEVAL [323] begin addressing this)

#### 5.4.3 Domain-Specific vs. General-Purpose Tension

The cross-domain analysis (Section 6) reveals that **evaluation is highly domain-specific**:

- Math/code agents emphasize symbolic verification (unit tests, compilers)
- Scientific agents prioritize hypothesis-experiment iteration
- Healthcare agents demand auditability and regulatory compliance
- Embodied agents require physical safety constraints

This creates a **transferability problem**: methods validated on one domain (e.g., code generation with ToolFormer [6]) may not generalize to others (e.g., clinical decision-making with MedAgent-Pro [368]).

#### 5.4.4 Governance and Safety Evaluation Gaps

Section 8.6 explicitly notes:

> "Existing benchmarks and guardrails mainly focus on short-horizon behaviors [773, 781], leaving planning-time failures and multi-agent dynamics underexplored."

Safety benchmarks like GuardAgent [773] and R-Judge [781] address **immediate risk detection** but do not evaluate:
- Planning-time failures that emerge over long horizons
- Multi-agent interaction risks (e.g., emergent collusion, coordination failures)
- Real-world deployment governance under dynamic conditions

---

### 5.5 Do the Benchmarks Support the Paper's Claims?

**Strengths:**

1. **Comprehensive Coverage:** The survey catalogs **70+ distinct benchmarks** across mechanism and application levels, providing unprecedented visibility into the evaluation landscape.

2. **Structured Organization:** The two-perspective framework (mechanism-centric vs. application-level) clarifies **what each benchmark type measures** and when each is appropriate.

3. **Gap Identification:** The survey explicitly identifies evaluation gaps (long-horizon, governance, standardization) that align with the open problems in Section 8.

**Limitations:**

1. **No Quantitative Synthesis:** The survey **does not provide meta-analysis** of benchmark results across papers. For example, it lists ToolLLM-ToolBench [203] with 16,464 APIs but does not report how different methods perform on this benchmark.

2. **Limited Ablation Evidence:** The survey reviews methods but **does not synthesize ablation studies** showing which components (planning vs. memory vs. tool use) contribute most to performance.

3. **Benchmark Quality Variation:** Some benchmarks are **preliminary or under review** (e.g., WebRAGent [654] noted as "under review" in references), raising questions about evaluation maturity.

4. **Reproducibility Concerns:** Many benchmarks rely on **LLM-as-judge** scoring (e.g., ELITR-Bench [713], MultiAgentBench [21]), which introduces evaluation bias and reproducibility challenges.

## 6. Limitations and Trade-offs

This survey formalizes agentic reasoning as a unified framework, but the paradigm itself rests on assumptions and trade-offs that practitioners must understand before deployment. This section critically examines what the agentic reasoning framework assumes, what scenarios it does not address, what computational and scalability constraints exist, and what weaknesses remain open. These limitations are not failures of the survey itself but rather inherent constraints in the current state of agentic reasoning research as synthesized by the authors.

### 6.1 Core Assumptions Underlying the Framework

The agentic reasoning framework rests on several technical and practical assumptions that may not hold in all deployment contexts.

#### 6.1.1 Assumption: Reasoning Can Be Factorized from Execution

The formal framework in Section 2.2 assumes that the agent's policy can be cleanly decomposed into internal reasoning ($\pi_{\text{reason}}$) and external execution ($\pi_{\text{exec}}$):

$$\pi_\theta(z_t, a_t \mid h_t) = \pi_{\text{reason}}(z_t \mid h_t) \cdot \pi_{\text{exec}}(a_t \mid h_t, z_t)$$

**Why This May Not Hold.** This factorization assumes that reasoning traces ($z_t$) are separable from actions ($a_t$) and that errors can be attributed to one component or the other. However, in practice:

- **Tightly Coupled Domains:** In embodied robotics (Section 6.3), perception, reasoning, and action are often interleaved at millisecond timescales. A robot navigating a dynamic environment cannot afford to complete a full reasoning trace before executing—the "think-act" structure may collapse into reactive policies where $\pi_{\text{reason}}$ and $\pi_{\text{exec}}$ are entangled.

- **Latent Reasoning:** Section 8.5 explicitly notes that latent agentic reasoning performs planning in internal latent spaces rather than explicit natural language traces. When reasoning is not verbalized, the factorization becomes unobservable, making it impossible to diagnose whether failures originate in $\pi_{\text{reason}}$ or $\pi_{\text{exec}}$.

- **Tool-Use Feedback Loops:** In tool-use scenarios (Section 3.2), the output of a tool call may immediately reshape the reasoning trace. For example, a code execution error may trigger immediate backtracking without a distinct reasoning step, blurring the boundary between thought and action.

**Practical Implication.** Practitioners deploying agentic systems in latency-sensitive or tightly-coupled environments should not assume the think-act decomposition will cleanly separate failure modes. Alternative formalisms that model reasoning and execution as jointly distributed may be necessary.

#### 6.1.2 Assumption: Feedback Signals Are Available and Reliable

The self-evolving layer (Section 4) assumes that agents can access feedback signals to improve their reasoning. However, the survey reveals significant variation in feedback availability:

| Feedback Type | Availability | Reliability Concerns |
|---------------|--------------|----------------------|
| Validator-driven (unit tests) | High in code domains [283-285] | Binary signals are non-diagnostic [Section 4.1.3] |
| Reflective (self-critique) | Always available | May propagate model's own biases [14, 270] |
| Human preference | Expensive to collect [416, 417] | Subjective and inconsistent across annotators |
| Environment reward | Domain-specific [36, 437] | Often sparse or engineered [Section 8.2] |

**Why This Matters.** The survey explicitly notes in Section 4.1.3 that validator-driven feedback "is non-diagnostic, correcting individual outputs without explaining failures or altering the model's reasoning behavior." This means that even when feedback is available, it may not support the kind of granular learning that self-evolving agents require.

**Practical Implication.** Domains without reliable validators (e.g., creative writing, strategic planning) cannot leverage validator-driven feedback and must rely on weaker signals like self-critique or human preference, which introduce their own biases and costs.

#### 6.1.3 Assumption: Memory Can Be Persistently Maintained

The agentic memory framework (Section 4.2) assumes that agents can maintain persistent memory across interactions. However, the survey identifies several constraints:

- **Context Window Limits:** Even with external memory, the agent must retrieve relevant entries into the context window for reasoning. Section 4.2.1 notes that traditional memory systems "rely on fixed heuristics and unstructured token lists [297], limiting adaptability for tasks involving goal decomposition [324, 143], long-term planning [150], or iterative self-improvement [325]."

- **Memory Management Trade-offs:** Section 4.2.3 describes three management strategies with conflicting priorities:
  - **Forgetting-based** (e.g., Lyfe Agents [429]): Prioritizes storage efficiency but "risks losing nuanced details for edge cases"
  - **Verification-based** (e.g., SEDM [426]): Prioritizes reliability but requires "verifiable write admission" infrastructure
  - **Learning-based** (e.g., MemAgent [314]): Prioritizes adaptability but requires reinforcement learning training

- **Multi-Agent Memory Coherence:** Section 5.3.2 notes that multi-agent memory must "jointly evolve along architecture, topology, content, and management dimensions," introducing coordination overhead that single-agent systems do not face.

**Practical Implication.** Practitioners cannot assume memory will "just work." Memory design requires explicit trade-off decisions between efficiency, reliability, and adaptability, and multi-agent systems face additional coherence challenges.

### 6.2 Scenarios and Edge Cases Not Addressed

The survey explicitly identifies several problem settings that remain underexplored or unaddressed by current agentic reasoning methods.

#### 6.2.1 Long-Horizon Credit Assignment

Section 8.2 identifies this as a "central open challenge":

> "While methods such as ReAct and Tree of Thought improve short-horizon reasoning [5, 4], errors still compound rapidly in long tasks, as illustrated by embodied agents like Voyager [36]. RL-trained agents such as WebRL and Agent-R1 improve performance in realistic environments but rely on heavily engineered, domain-specific rewards and largely treat episodes independently [437, 28]."

**What This Means.** Current methods work well for tasks requiring 5-10 interaction turns but degrade significantly beyond 50-100 turns. The survey does not provide systematic evaluation of agents on tasks requiring 1000+ turns, which are common in:
- Longitudinal healthcare monitoring (patient tracking over months)
- Scientific research workflows (hypothesis → experiment → revision cycles over weeks)
- Enterprise automation (multi-day task coordination)

**Why It Is Not Addressed.** The fundamental technical barrier is credit assignment: when a task succeeds or fails after 100 turns, attributing responsibility to specific decisions, tool calls, or memory updates is computationally intractable with current methods. Section 8.2 notes that "a core open problem is how to assign credit across tokens, tool calls, skills, and memory updates, and to generalize such learning across a long sequence of episodes and tasks."

#### 6.2.2 Adversarial and Security-Critical Settings

The survey mentions security briefly in Section 5.2.2 (G-Safeguard [402] prunes risky communication edges) and Section 8.6 (governance), but does not systematically address:

- **Adversarial Prompting:** How agentic systems respond to malicious users attempting to manipulate reasoning traces or tool calls
- **Tool Poisoning:** What happens when external tools return adversarial outputs (e.g., a compromised API returning false data)
- **Multi-Agent Collusion:** How to prevent agents from learning to coordinate in ways that circumvent safety constraints

Section 8.6 explicitly notes: "Existing benchmarks and guardrails mainly focus on short-horizon behaviors [773, 781], leaving planning-time failures and multi-agent dynamics underexplored."

**Practical Implication.** Deploying agentic systems in security-critical domains (finance, healthcare, autonomous vehicles) requires additional safeguards not covered by the surveyed methods.

#### 6.2.3 Resource-Constrained Deployment

The survey focuses primarily on methods assuming access to large language models with substantial context windows and compute. It does not systematically address:

- **Edge Deployment:** Running agentic systems on devices with limited memory, compute, or connectivity (e.g., mobile phones, IoT devices)
- **Cost Constraints:** The survey mentions that post-training methods require "significant (compute, data)" investment (Section 4.2) but does not provide cost-benefit analysis
- **Latency Requirements:** Real-time applications (e.g., robotics, trading) may not tolerate the multi-turn deliberation that agentic reasoning requires

Section 3.2.1 notes that GEAR [201] "delegates the tool selection process to a small language model while reserving the more powerful LLM for the final reasoning step to reduce costs," but this is one of few mentions of cost optimization.

#### 6.2.4 Non-Stationary User Preferences

Section 8.1 identifies user-centric personalization as an open problem:

> "User-centric agentic reasoning involves balancing short-term task rewards with long-term user experience, satisfaction, and trust, which introduces non-stationary objectives and long-horizon credit assignment challenges beyond conventional agentic reasoning settings."

**What This Means.** Current agentic systems assume fixed objectives (e.g., solve the math problem, complete the task). However, real users change their preferences over time, and agents must adapt without explicit retraining. The survey does not cover methods for:
- Inferring implicit preference changes from interaction patterns
- Balancing exploration (learning user preferences) vs. exploitation (completing tasks efficiently)
- Handling conflicting preferences across multiple stakeholders

### 6.3 Computational, Data, and Scalability Constraints

The agentic reasoning framework introduces significant resource requirements that may limit deployment.

#### 6.3.1 Inference-Time Compute Overhead

In-context methods (Section 3-5) scale inference-time compute through structured orchestration, but this comes at a cost:

| Method Type | Additional Tokens | Latency Impact |
|-------------|-------------------|----------------|
| Chain-of-Thought [1] | 2-5× output length | Moderate |
| Tree-of-Thoughts [4] | 10-50× (multiple branches) | High |
| ReAct [5] | 3-10× (reasoning + action turns) | Moderate-High |
| Multi-Agent Collaboration [17, 66] | N× per agent (N = agent count) | Very High |

**Why This Matters.** Section 3.1.1 notes that workflow-based approaches "can accumulate errors over long horizons," and Section 3.3.1 notes that in-context search methods "often rely on prompt engineering and may struggle with stability across diverse domains." The compute overhead compounds with task complexity.

**Scalability Limit.** For a 10-agent collaborative system (Section 5.1) with 5 reasoning turns per agent, the token cost is approximately 50× a single-pass inference. This makes in-context multi-agent systems prohibitively expensive for high-volume applications.

#### 6.3.2 Post-Training Data Requirements

Post-training methods (Section 3-5) require substantial curated data:

- **ToolLLM [203]:** "Scales SFT training to over 16,000 real-world APIs" with "massive curated demonstrations"
- **ToolAlpaca [204]:** "Automatically constructing a diverse toolset and generating multi-turn tool-use dialogues via multi-agent simulation"
- **WebRL [437]:** "Self-evolving online curriculum that generates new tasks from unsuccessful attempts"

**Data Quality Concerns.** Section 3.2.2 notes that SFT-based methods "suffer from overfitting to the specific patterns in the training data [229, 230, 231], leading to brittle tool-selection strategies and limited adaptability in unseen downstream application scenarios [232, 207, 233]."

**Practical Implication.** Organizations without access to large-scale curated datasets or simulation infrastructure cannot leverage post-training methods effectively. This creates a barrier to entry favoring well-resourced organizations.

#### 6.3.3 Multi-Agent Coordination Overhead

Section 5 identifies several scalability challenges in multi-agent systems:

- **Communication Bottlenecks:** Section 5.1 notes that "communication overhead can easily undermine MAS efficiency," requiring dedicated Communication Facilitator roles
- **Topology Optimization:** Section 5.2.2 describes graph pruning methods (AgentPrune [400], AGP [401]) specifically designed to reduce token and compute costs from dense collaboration graphs
- **Credit Assignment:** Section 8.4 notes that "credit assignment at the group level remains poorly understood" and "scaling to larger agent populations further introduces challenges in topology adaptation, coordination overhead, and safety [793, 794, 766]"

**Scalability Limit.** The survey does not provide empirical analysis of how performance scales with agent count, but Section 5.3.3's discussion of multi-agent reinforcement fine-tuning [447, 448, 413] implies that training complexity grows superlinearly with agent population.

### 6.4 Weaknesses and Open Questions

The survey explicitly identifies several weaknesses in current agentic reasoning methods that remain unresolved.

#### 6.4.1 Evaluation Fragmentation

Section 7 reveals that "existing suites often differ in what they treat as the core capability, such as tool invocation accuracy, memory retention under long contexts, or coordination quality in multi-agent settings." This fragmentation creates several problems:

- **No Unified Metric:** There is no equivalent to "accuracy" or "F1 score" for agentic reasoning. Benchmarks use win rates, social welfare, communication quality, GPT-4 scoring, and task success with no standardization.
- **Cross-Paper Comparison Difficulty:** Section 5.4.1 notes that "this fragmentation makes cross-paper comparison difficult." A method validated on ToolLLM-ToolBench [203] cannot be directly compared to one validated on MTU-Bench [696].
- **LLM-as-Judge Bias:** Many benchmarks (ELITR-Bench [713], MultiAgentBench [21]) rely on LLM scoring, which "introduces evaluation bias and reproducibility challenges" (Section 5.4).

**Open Question.** How can the community develop standardized, reproducible evaluation protocols that capture the multi-dimensional nature of agentic reasoning (planning, tool use, memory, collaboration)?

#### 6.4.2 Domain Transfer Limitations

Section 6's cross-domain analysis reveals that methods do not transfer naively:

> "A tool-use strategy that works for code generation (execute and check for errors) may fail in healthcare (where execution errors have safety implications)." (Section 4.4)

**Evidence from Survey:**
- **Math/Code:** Emphasize symbolic verification (unit tests, compilers) [Section 6.1]
- **Scientific:** Prioritize hypothesis-experiment iteration [Section 6.2]
- **Healthcare:** Demand auditability and regulatory compliance [Section 6.4]
- **Embodied:** Require physical safety constraints [Section 6.3]

**Open Question.** What components of agentic reasoning are domain-agnostic versus domain-specific? The survey does not provide a systematic ablation analysis answering this question.

#### 6.4.3 Governance and Safety Gaps

Section 8.6 explicitly identifies governance as a "cross-cutting challenge":

> "Beyond standard LLM safety issues, agentic systems introduce new risks due to long-horizon planning, persistent memory, and real-world action execution [797]. Failures may arise from interactions across time and components, making attribution and auditing difficult."

**Specific Gaps:**
- **Planning-Time Failures:** Current guardrails (GuardAgent [773], R-Judge [781]) focus on output safety, not unsafe reasoning trajectories that may lead to harmful actions
- **Multi-Agent Dynamics:** Section 8.6 notes that "multi-agent dynamics" are "underexplored," including emergent collusion, coordination failures, and distributed responsibility
- **Auditability:** When an agent makes a harmful decision after 50 interaction turns, attributing responsibility to specific reasoning steps, tool calls, or memory retrievals is technically challenging

**Open Question.** How can governance frameworks address planning-time failures and multi-agent interactions, not just output safety?

#### 6.4.4 Latent Reasoning Interpretability

Section 8.5 identifies a fundamental tension:

> "Latent agentic reasoning explores performing planning, decision-making and collaboration in internal latent spaces rather than explicit natural language or symbolic traces. Recent work suggests that latent reasoning can improve efficiency and scalability, but at the cost of reduced interpretability and controllability."

**Trade-off:**
| Aspect | Explicit Reasoning | Latent Reasoning |
|--------|-------------------|------------------|
| Interpretability | High (inspect traces) | Low (unobservable) |
| Efficiency | Low (verbose tokens) | High (compressed) |
| Controllability | High (prompt engineering) | Low (weight-level) |
| Debugging | Transparent | Opaque |

**Open Question.** How can latent reasoning be made auditable without sacrificing its efficiency benefits? The survey does not cover methods for probing or explaining latent reasoning processes.

### 6.5 Summary: Key Trade-offs

The table below summarizes the primary trade-offs practitioners must navigate when designing agentic reasoning systems:

| Trade-off | Option A | Option B | When to Choose A | When to Choose B |
|-----------|----------|----------|------------------|------------------|
| **Optimization Mode** | In-Context | Post-Training | Rapid deployment, task variability | Durable improvements, consistency |
| **Memory Management** | Forgetting-based | Verification-based | Storage-constrained environments | Safety-critical applications |
| **Reasoning Representation** | Explicit (natural language) | Latent | Interpretability required | Efficiency prioritized |
| **Agent Architecture** | Single-Agent | Multi-Agent | Simple tasks, low latency | Complex tasks, division of labor |
| **Feedback Type** | Validator-driven | Reflective | Reliable validators available | No external validators |
| **Planning Horizon** | Short (5-10 turns) | Long (50+ turns) | Well-defined tasks | Open-ended exploration |

**Critical Takeaway.** There is no universally optimal agentic reasoning system. The survey's three-layer taxonomy (Foundational, Self-Evolving, Collective) and dual optimization modes (In-Context, Post-Training) provide a design space, not a prescription. Practitioners must explicitly trade off flexibility vs. durability, interpretability vs. efficiency, and capability vs. cost based on their deployment constraints.

### 6.6 What This Means for Practitioners

Based on the limitations identified above, practitioners should:

1. **Validate Assumptions:** Before deploying, verify that reasoning-execution factorization holds in your domain. For latency-sensitive or tightly-coupled tasks, consider reactive policies over deliberative agents.

2. **Plan for Feedback:** Ensure reliable feedback signals exist for your use case. If validators are unavailable, budget for human preference collection or accept weaker self-critique signals.

3. **Design Memory Explicitly:** Do not treat memory as an afterthought. Choose management strategies (forgetting, verification, learning) based on your priorities for efficiency, reliability, or adaptability.

4. **Budget for Compute:** In-context multi-agent systems can incur 50×+ token costs. Post-training methods require substantial curated datasets. Plan resources accordingly.

5. **Accept Domain Specificity:** Do not assume methods transfer across domains without adaptation. Healthcare agents require different safeguards than code agents.

6. **Monitor Long-Horizon Behavior:** Current methods degrade beyond 50-100 turns. For longer tasks, implement explicit checkpointing, human oversight, or episode segmentation.

7. **Plan for Governance:** Output safety is insufficient. Implement audit trails for reasoning traces, tool calls, and memory operations to enable post-hoc attribution.

The survey provides a comprehensive roadmap for agentic reasoning, but practitioners must navigate these limitations and trade-offs to deploy systems that are not only capable but also reliable, efficient, and safe.

## 7. Implications and Future Directions

This survey does more than catalog existing methods—it establishes agentic reasoning as a **coherent research paradigm** with clear design principles, comparison frameworks, and open directions. The implications extend across theoretical understanding, research agendas, and practical deployment. Below, I articulate how this work reshapes the field, what follow-up research it enables, and how practitioners should navigate the design space it reveals.

### 7.1 How This Work Changes the Landscape

The survey's primary contribution is **paradigm unification**. Prior to this work, agentic reasoning research was fragmented across three silos:

1. **LLM reasoning papers** focused on internal computation (Chain-of-Thought, Tree-of-Thoughts) without addressing external interaction
2. **Agent architecture papers** described system components without centering reasoning as the explanatory mechanism
3. **Domain-specific works** (robotics, healthcare, science) developed isolated solutions without cross-domain generalization

This survey collapses these silos by providing a **common formal language** (the policy factorization $\pi_\theta(z_t, a_t \mid h_t) = \pi_{\text{reason}}(z_t \mid h_t) \cdot \pi_{\text{exec}}(a_t \mid h_t, z_t)$ from Section 2.2) and a **unified taxonomy** (Foundational → Self-Evolving → Collective from Section 1, Figure 1). The landscape shift operates at three levels:

#### 7.1.1 Theoretical Reframing: From Capabilities to Environmental Dynamics

The three-layer taxonomy reframes the fundamental design question. Instead of asking "what capabilities should an agent have?", researchers now ask "**what environmental dynamics must the agent handle?**"

| Environmental Dynamic | Required Reasoning Layer | Example Domains |
|----------------------|-------------------------|-----------------|
| Stable, fixed tasks | Foundational | Math benchmarks, code generation |
| Evolving with feedback | Self-Evolving | Healthcare monitoring, scientific discovery |
| Collaborative, distributed | Collective | Multi-robot teams, enterprise workflows |

This reframing has immediate consequences:
- **Prevents over-engineering**: A math solver does not need collective multi-agent coordination
- **Prevents under-engineering**: A clinical agent cannot rely solely on foundational planning without self-evolving memory for longitudinal tracking
- **Enables targeted research**: Teams can now identify which layer their work advances rather than claiming generic "agentic improvements"

#### 7.1.2 Methodological Clarification: In-Context vs. Post-Training as First-Class Decisions

The dual optimization mode framework (Section 1, "Survey Scope" box; Section 3-5) elevates a decision that practitioners previously made ad hoc. The survey shows this choice cascades through all capability layers:

| Capability | In-Context Implementation | Post-Training Implementation |
|------------|--------------------------|-----------------------------|
| Planning | Tree-of-Thoughts [4], MCTS [112-123] | RL-trained policies [14, 342] |
| Memory | Retrieval augmentation [252, 293] | Learned memory control [314, 315] |
| Multi-Agent | Fixed role prompts [17, 374] | Co-evolved policies [446, 447] |

This clarification enables **resource-aware system design**. Organizations with limited training infrastructure can prioritize in-context methods while accepting their limitations (bounded by frozen model capabilities, Section 3.3.1). Teams with substantial compute can invest in post-training for durable improvements that generalize across episodes (Section 3.2.2).

#### 7.1.3 Evaluation Standardization: From Fragmented to Structured

Section 7's two-perspective framework (mechanism-centric vs. application-level benchmarks) provides the first **structured map of evaluation practices** across 70+ distinct benchmarks. This enables:

- **Attribution of improvements**: Mechanism-centric benchmarks (Section 7.1) isolate whether gains come from better planning, memory, or tool use
- **Realistic validation**: Application-level benchmarks (Section 7.2) capture combined effects under domain constraints
- **Gap identification**: The survey explicitly reveals underexplored areas (long-horizon credit assignment, governance, Section 5.4)

Prior to this work, comparing a method validated on ToolLLM-ToolBench [203] to one validated on MTU-Bench [696] was apples-to-oranges. The survey provides the vocabulary to articulate what each benchmark measures and when each is appropriate.

### 7.2 Follow-Up Research Directions

The survey's framework enables several concrete research agendas. I organize these by the six open problems identified in Section 8, augmented with specific technical pathways the survey suggests.

#### 7.2.1 Long-Horizon Credit Assignment (Section 8.2)

**The Problem.** Current methods degrade beyond 50-100 interaction turns because "errors compound rapidly in long tasks" (Section 8.2). RL-trained agents like WebRL [437] and Agent-R1 [28] "rely on heavily engineered, domain-specific rewards and largely treat episodes independently."

**Research Pathways Enabled by This Survey:**

1. **Hierarchical Credit Assignment**: The policy factorization formalism suggests decomposing credit across $\pi_{\text{reason}}$ and $\pi_{\text{exec}}$ separately. Follow-up work could develop:
   - Reasoning-level rewards (e.g., penalizing logically inconsistent traces)
   - Execution-level rewards (e.g., tool call success rates)
   - Cross-component attribution (e.g., did execution fail due to bad reasoning or bad tool output?)

2. **Memory-Augmented Credit**: Section 4.2's memory taxonomy suggests using memory structures to propagate credit across episodes:
   - Workflow Memory [298] could store not just procedural traces but also reward signals
   - Graph-based memory [305, 306] could enable credit propagation along semantic relationships
   - Post-training memory control [314, 315] could learn to preserve high-reward memories

3. **Process Supervision**: Section 4.1.2 notes that "trajectory-level supervised fine-tuning approaches attach feedback to intermediate reasoning traces rather than only final answers" [274, 103]. Extending this to long horizons requires:
   - Automated process reward models (PRMs) that scale to 100+ turns
   - Sparse reward densification techniques for multi-turn settings
   - Cross-episode credit accumulation mechanisms

**Concrete Milestone.** A benchmark evaluating credit assignment across 100+ interaction turns with standardized metrics (currently absent from Section 7.1.3).

#### 7.2.2 World Models for Agentic Reasoning (Section 8.3)

**The Problem.** World-model-based agents "rely on ad hoc representations and are typically trained on short-horizon or environment-specific data, raising concerns about calibration and generalization" (Section 8.3). Only "a few works explore co-evolving world models and agents over long time scales" [610, 791].

**Research Pathways Enabled by This Survey:**

1. **Integration with Policy Factorization**: The formalism $\pi_\theta(z_t, a_t \mid h_t)$ suggests world models could inform $\pi_{\text{reason}}$ by simulating outcomes before committing to $a_t$:
   - Latent world models that predict observation transitions $\mathcal{T}$ and rewards $R$ (Section 2.2 tuple)
   - Imagination-augmented reasoning where $z_t$ includes simulated trajectories
   - World model uncertainty as a signal for when to retrieve vs. simulate

2. **Domain-Specific World Models**: Section 6's cross-domain analysis reveals different world model requirements:
   - **Embodied agents**: Physical dynamics models (e.g., Cosmos-Reason1 [547], CoT-VLA [548])
   - **Web agents**: Page state transition models (e.g., WebEvolver [610])
   - **Scientific agents**: Simulation environments (e.g., DISCOVERYWORLD [757], ScienceWorld [758])
   - **Healthcare agents**: Patient state evolution models (e.g., MedAgentSim [595])

3. **Co-Evolution with Agents**: Section 5.3's multi-agent evolution framework suggests world models could co-evolve with agent policies:
   - Agents improve world models through interaction data
   - World models improve agents through better simulation
   - Multi-agent settings could enable distributed world model learning

**Concrete Milestone.** A world model benchmark evaluating calibration, generalization, and long-horizon prediction accuracy across multiple domains (currently absent from Section 7).

#### 7.2.3 Multi-Agent Collaborative Training (Section 8.4)

**The Problem.** "Most collaboration structures are still manually designed" (Section 8.4). While "recent multi-agent RL approaches begin to treat collaboration itself as a trainable skill" [409, 413, 26], "credit assignment at the group level remains poorly understood."

**Research Pathways Enabled by This Survey:**

1. **Topology Learning**: Section 5.2.2's topology optimization framework provides a foundation:
   - Graph generation methods [397-399] could learn communication structures end-to-end
   - Graph pruning methods [400-402] could adapt topologies dynamically during execution
   - Topology search [332, 395, 403, 404, 68] could discover effective collaboration patterns

2. **Role Specialization**: Section 5.1's role taxonomy suggests learning role assignments rather than fixing them:
   - Dynamic role allocation based on task requirements
   - Role emergence through self-organization (e.g., EvoMAC [351], SEW [502])
   - Cross-role credit assignment for heterogeneous agent populations

3. **Collective Memory Evolution**: Section 5.3.2's multi-agent memory framework enables:
   - Shared memory structures that evolve through interaction
   - Memory governance mechanisms for privacy and coherence [426, 427]
   - Cross-agent memory retrieval optimization

**Concrete Milestone.** A multi-agent training benchmark evaluating scalability (performance vs. agent count), adaptability (task transfer), and safety (collusion prevention) — currently fragmented across Section 7.1.4's game-based, simulation-centric, and language-based benchmarks.

#### 7.2.4 Latent Agentic Reasoning (Section 8.5)

**The Problem.** "Latent reasoning can improve efficiency and scalability, but at the cost of reduced interpretability and controllability" (Section 8.5). "Diagnosing failures becomes particularly difficult when intermediate reasoning steps are not externally observable."

**Research Pathways Enabled by This Survey:**

1. **Probing Methods**: The policy factorization suggests probing $\pi_{\text{reason}}$ even when $z_t$ is latent:
   - Attention-based probing to identify reasoning-relevant neurons
   - Causal intervention to test reasoning hypotheses
   - Counterfactual reasoning to validate latent decision boundaries

2. **Hybrid Representations**: Section 3.1's planning methods suggest combining explicit and latent reasoning:
   - High-level plans in natural language, low-level execution in latent space
   - Explicit reasoning for safety-critical decisions, latent for efficiency
   - Latent-to-explicit translation for auditability

3. **Evaluation Frameworks**: Section 7's benchmark landscape lacks latent reasoning evaluation:
   - Efficiency metrics (token reduction, latency improvement)
   - Interpretability proxies (probing accuracy, intervention success)
   - Controllability measures (prompt engineering effectiveness)

**Concrete Milestone.** A latent reasoning benchmark comparing explicit vs. latent methods on efficiency, accuracy, and interpretability trade-offs.

#### 7.2.5 Governance of Agentic Systems (Section 8.6)

**The Problem.** "Beyond standard LLM safety issues, agentic systems introduce new risks due to long-horizon planning, persistent memory, and real-world action execution" (Section 8.6). "Existing benchmarks and guardrails mainly focus on short-horizon behaviors [773, 781], leaving planning-time failures and multi-agent dynamics underexplored."

**Research Pathways Enabled by This Survey:**

1. **Planning-Time Safety**: Section 4.1's feedback mechanisms suggest safety checks during reasoning:
   - Constraint verification before action execution
   - Adversarial testing of reasoning traces
   - Uncertainty estimation for risky decisions

2. **Memory Governance**: Section 4.2's memory taxonomy raises governance questions:
   - What memories should be retained vs. forgotten for privacy?
   - How to audit memory retrieval decisions?
   - Multi-agent memory access control [427]

3. **Multi-Agent Accountability**: Section 5.3's collective evolution framework creates attribution challenges:
   - Distributed responsibility for harmful outcomes
   - Emergent collusion detection
   - Cross-agent audit trails

**Concrete Milestone.** A governance benchmark evaluating planning-time safety, memory auditability, and multi-agent accountability — currently absent from Section 7's safety benchmarks (GuardAgent [773], R-Judge [781]).

#### 7.2.6 User-Centric Personalization (Section 8.1)

**The Problem.** "User-centric agentic reasoning involves balancing short-term task rewards with long-term user experience, satisfaction, and trust, which introduces non-stationary objectives and long-horizon credit assignment challenges" (Section 8.1).

**Research Pathways Enabled by This Survey:**

1. **Preference Learning**: Section 4.1.2's parametric adaptation framework extends to user preferences:
   - Preference-aligned fine-tuning for individual users
   - Online preference learning from interaction feedback
   - Multi-user preference aggregation for shared agents

2. **Longitudinal Memory**: Section 4.2's memory systems enable user modeling:
   - User-specific memory entries for personalization
   - Preference evolution tracking over time
   - Cross-session consistency maintenance

3. **Trust Calibration**: Section 6.4's healthcare agents highlight trust requirements:
   - Uncertainty communication to users
   - Explanation generation for decisions
   - User control over agent autonomy levels

**Concrete Milestone.** A personalization benchmark evaluating preference learning accuracy, longitudinal consistency, and user trust metrics.

### 7.3 Practical Applications and Downstream Use Cases

The survey's framework enables practitioners to match environmental constraints to appropriate reasoning layers and optimization modes. Below, I provide concrete guidance for key application domains.

#### 7.3.1 When to Use Foundational vs. Self-Evolving vs. Collective Reasoning

| Application Scenario | Recommended Layer | Rationale |
|---------------------|-------------------|-----------|
| **Math problem solving** | Foundational | Stable environment, fixed task structure, symbolic verification available (Section 6.1) |
| **Code generation with testing** | Foundational + Self-Evolving | Validator-driven feedback enables iterative improvement (Section 4.1.3, 6.1.2) |
| **Clinical decision support** | Self-Evolving | Longitudinal patient tracking requires persistent memory (Section 6.4.2) |
| **Scientific discovery** | Self-Evolving + Collective | Hypothesis-experiment iteration + specialist collaboration (Section 6.2) |
| **Enterprise workflow automation** | Collective | Multi-stakeholder coordination, role specialization (Section 5.1.2) |
| **Multi-robot coordination** | Collective + Self-Evolving | Distributed execution + adaptive collaboration (Section 6.3.3) |

**Key Principle.** Start with the minimal layer required for your environmental dynamics. Adding unnecessary complexity (e.g., multi-agent for single-user tasks) increases cost without benefit (Section 6.3.1).

#### 7.3.2 When to Use In-Context vs. Post-Training Optimization

| Decision Factor | Prefer In-Context | Prefer Post-Training |
|-----------------|-------------------|---------------------|
| **Training infrastructure** | Limited compute/data access | Substantial RL/SFT resources |
| **Task variability** | High (frequent domain changes) | Low (stable task distribution) |
| **Deployment speed** | Rapid iteration required | Can tolerate training time |
| **Consistency requirements** | Moderate | High (e.g., healthcare, finance) |
| **Debugging needs** | Transparent inspection needed | Can accept opaque weights |
| **Long-horizon tasks** | Short episodes (<50 turns) | Long episodes (50+ turns) |

**Key Principle.** In-context methods offer flexibility; post-training methods offer durability. The survey's dual-mode framework (Section 3-5) enables hybrid approaches: use in-context for rapid prototyping, then post-train successful patterns for deployment.

#### 7.3.3 Domain-Specific Integration Guidance

**Healthcare Agents (Section 6.4):**
- **Priority**: Safety, auditability, regulatory compliance
- **Recommended**: Self-evolving memory for patient history [581, 595], validator-driven feedback for clinical calculations [586], multi-agent specialist consultation [364, 600]
- **Avoid**: Pure in-context methods without audit trails; latent reasoning without interpretability
- **Benchmark**: MedAgentBench [770], MedAgentsBench [771], AgentClinic [769]

**Scientific Discovery Agents (Section 6.2):**
- **Priority**: Hypothesis generation, experimental iteration, literature grounding
- **Recommended**: Self-evolving planning for hypothesis refinement [525, 536], tool-use for laboratory automation [522, 524], collective multi-agent for specialist collaboration [507, 523]
- **Avoid**: Static planning without revision loops; single-agent for multi-modal tasks
- **Benchmark**: ScienceAgentBench [759], LAB-Bench [760], DISCOVERYWORLD [757]

**Embodied Agents (Section 6.3):**
- **Priority**: Physical safety, multimodal grounding, latency constraints
- **Recommended**: Foundational planning with execution feedback [136, 541], self-evolving skill libraries [36, 569], collective multi-robot coordination [552, 577]
- **Avoid**: Long reasoning traces in latency-sensitive settings; in-context methods without physical verification
- **Benchmark**: ALFWorld [48], OSWorld [756], BALROG [751]

**Autonomous Research Agents (Section 6.5):**
- **Priority**: Long-horizon coordination, literature synthesis, code execution
- **Recommended**: Self-evolving memory for experiment tracking [634, 651], collective multi-agent for author-reviewer-editor workflows [685, 686], post-training for consistent research quality [260, 677]
- **Avoid**: Single-pass reasoning for multi-week workflows; in-context methods without persistent experiment logs
- **Benchmark**: The AI Scientist [651], Agent Laboratory [634], WorkArena [762]

### 7.4 Reproducibility and Integration Guidance

The survey's GitHub repository (§ Github: https://github.com/weitianxin/Awesome-Agentic-Reasoning) provides ongoing updates, but practitioners should note several integration considerations:

#### 7.4.1 Method Selection Checklist

Before implementing an agentic reasoning system, verify:

1. **Environmental Dynamics**: Does your task require foundational only, self-evolving, or collective reasoning? (Section 1, Figure 1)
2. **Feedback Availability**: Do you have reliable validators, or must you rely on self-critique? (Section 4.1)
3. **Memory Requirements**: Do you need flat, structured, or post-training controlled memory? (Section 4.2, Figure 6)
4. **Optimization Mode**: Can you afford post-training, or must you use in-context methods? (Section 3-5)

#### 7.4.2 Common Integration Pitfalls

Based on the survey's analysis, avoid these mistakes:

| Pitfall | Consequence | Mitigation |

| **Deploying latent reasoning without auditability** | Unexplainable failures (Section 8.5) | Implement probing or hybrid explicit-latent representations |
