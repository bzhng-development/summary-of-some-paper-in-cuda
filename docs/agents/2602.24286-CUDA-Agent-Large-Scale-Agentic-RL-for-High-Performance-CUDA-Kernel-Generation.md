## 1. Executive Summary

This paper introduces CUDA Agent, a large-scale agentic reinforcement learning system that addresses the inability of standard large language models to compete with compiler-based systems like `torch.compile` for high-performance CUDA kernel generation. Through a scalable data synthesis pipeline and stable RL training techniques, CUDA Agent achieves state-of-the-art performance on KernelBench, delivering 100%, 100%, and 92% faster rates over `torch.compile` on Level-1, Level-2, and Level-3 splits while outperforming proprietary models like Claude Opus 4.5 by approximately 40% on the hardest settings. This work establishes LLM-based kernel generation as a superior alternative to traditional compiler-driven optimization, enabling automated performance-critical software development in GPU computing.

## 2. Context and Motivation

### The Core Problem: LLMs Cannot Compete with Compilers for CUDA

Despite remarkable progress in general software development, large language models face a critical limitation: they cannot generate CUDA kernels that match the performance of compiler-based optimization systems like `torch.compile`. This gap is not marginal—existing LLM approaches produce kernels that are often **slower** than automatically compiled code, even when the generated code is functionally correct.

The paper identifies a fundamental distinction that explains this failure:

> "Existing CUDA code generation approaches either rely on training-free refinement or fine-tune models within fixed multi-turn execution-feedback loops, while both paradigms fail to fundamentally improve the model's intrinsic CUDA optimization ability."

This statement captures the core gap. Prior methods treat CUDA generation as a **test-time scaling problem** (using more inference compute to search for better solutions) or a **supervised fine-tuning problem** (teaching the model to mimic existing kernels). Neither approach teaches the model to *reason about GPU hardware* during training. The model never learns to internalize optimization strategies—it only learns to produce text that looks like CUDA code.

### Why CUDA Kernel Optimization Matters

This problem carries both practical and theoretical significance:

**Real-World Impact:** GPU kernels are the foundational building blocks of modern deep learning infrastructure. Every neural network operation—matrix multiplication, convolution, attention, normalization—ultimately executes as CUDA kernels on NVIDIA GPUs, which dominate the AI hardware ecosystem. Even modest per-kernel improvements compound across large models:

- A 2× speedup on a frequently-used operator translates directly to reduced training time and inference latency
- Manual kernel optimization requires **deep hardware expertise** that is scarce and expensive
- The optimization process demands sophisticated profiling toolkits and understanding of GPU microarchitectural features (shared memory, register allocation, occupancy, memory coalescing)

**Theoretical Significance:** This work addresses a broader question: can foundation models learn **hardware-aware reasoning** rather than just syntactic code generation? Success here would demonstrate that LLMs can move from passive code generators to active systems optimizers—a capability that extends beyond CUDA to any performance-critical domain.

### Prior Approaches and Their Limitations

The paper categorizes existing work into two distinct paradigms, each with characteristic failure modes:

#### Training-Free Systems (Test-Time Search)

Several works design explicit search procedures to navigate the irregular optimization landscape:

| System | Approach | Key Limitation |
|--------|----------|----------------|
| STARK [5] | Multi-agent team with planning, coding, debugging roles exploring tree-structured search | Heavily relies on base model's CUDA capability; performance capped by intrinsic model ability |
| ReGraphT [6] | Retrieval-augmented framework distilling optimization trajectories into reasoning graphs | Focuses on capability distillation to smaller models, not maximizing absolute performance |
| EvoEngineer [8] | LLM-driven evolutionary loop iteratively editing and validating kernels | Evaluated on only 91 of 250 KernelBench tasks (selection bias) |
| CudaForge [26] | Two-agent system where Judge uses Nsight profiling to diagnose bottlenecks | Fixed interaction protocol limits agent autonomy |

**Why these fail:** These methods apply search and refinement *at inference time* without improving the underlying model. As the paper notes:

> "These methods do not remedy the fundamental lack of CUDA-coding abilities in the base models, causing performance gains to be significantly capped by the model's intrinsic capabilities."

Think of this as giving a student who doesn't understand calculus a graphing calculator and unlimited time—they might eventually find the right answer through trial and error, but they haven't learned the underlying mathematics. When faced with a novel problem outside the search space, performance collapses.

#### Fine-Tuning Approaches (Supervised or RL)

A parallel line of research attempts to train models directly:

| System | Approach | Key Limitation |
|--------|----------|----------------|
| Kevin [4] | Multi-turn RL framework modeling iterative developer workflow | Trains on KernelBench subsets—data leakage between train and test |
| CUDA-L1 [14] | Contrastive RL evaluating multiple kernel variants | Constructs training data directly from KernelBench references (severe data leakage) |
| ConCuR [11] | Synthesizes kernels with reasoning traces, fine-tunes QwQ-32B | Uses Kevin-32B (trained on KernelBench) to generate training data—contamination propagates |

**Why these fail:** These approaches face three fundamental constraints:

1. **Data scarcity:** High-performance CUDA kernels are prohibitively expensive to create manually. Expert-level reference code requires significant human effort.

2. **Limited training scale:** Existing methods train on small datasets without systematic scaling, capping achievable performance.

3. **Hand-designed optimization loops:** Fixed multi-turn refinement patterns constrain the agent's autonomy to learn debugging, search, and profiling strategies.

The data leakage issue is particularly severe. When a model trains on the same benchmark it will be evaluated on, reported gains may reflect **benchmark-specific adaptation** rather than generalizable kernel generation capability. This makes fair comparison impossible.

### How CUDA Agent Positions Itself

CUDA Agent distinguishes itself through three complementary dimensions that directly address the limitations above:

**1. Scalable Data Synthesis (Solving Data Scarcity)**

Rather than relying on manually curated kernels, the paper introduces a three-stage pipeline (Figure 1):

```
Seed Problem Crawling → Combinatorial Problem Synthesis → Rubric-based Filtering
```

The key insight is that **composing multiple operators into fused tasks** creates valuable training problems that are not equivalent to optimizing each operator in isolation. Fusion reshapes the optimization landscape by:
- Avoiding intermediate global-memory materialization
- Coupling stages through shared register/SMEM/occupancy constraints
- Requiring unified parallel mapping and data layout

This produces 6,000 curated training samples (`CUDA-Agent-Ops-6K`) without contaminating the evaluation benchmark.

**2. Skill-Integrated Agent Environment (Solving Fixed Loop Constraints)**

Instead of hardcoding a refinement procedure, CUDA Agent equips the model with structured **Agent Skills** that formalize the standard CUDA development workflow:

1. Profile the native PyTorch implementation to identify bottlenecks
2. Implement custom CUDA operators targeting performance-critical paths
3. Compile and evaluate in a sandbox environment with correctness checks
4. Iterate until achieving ≥5% speedup over `torch.compile`

The agent autonomously decides when to invoke tools, interpret feedback, and revise kernels—learning optimization strategies rather than following scripted patterns.

**3. Stable RL Training (Solving Training Instability)**

The paper identifies a critical failure mode: initial RL trials collapsed after only 17 training steps due to **domain distribution mismatch**. CUDA code represents less than 0.01% of typical pretraining data, so the base model produces many low-probability tokens. When training and inference engines use different numerical precisions (e.g., BF16 vs FP16), small errors in computing token probabilities near the precision floor cause importance sampling ratios to explode.

The solution is a **multi-stage warm-up strategy** (Figure 3):
- Single-turn RL warm-up on the base model
- Rejection Fine-Tuning (RFT) to initialize the actor with high-quality trajectories
- Value Pretraining to initialize the critic with meaningful advantage estimates

This enables stable training for 150+ steps with consistent reward growth.

### Summary of Positioning

CUDA Agent occupies a distinct niche in the research landscape:

| Dimension | Training-Free Systems | Fine-Tuning Approaches | CUDA Agent |
|-----------|----------------------|------------------------|------------|
| **Improves base model?** | No | Partially | Yes (systematically) |
| **Training data source** | N/A | Often benchmark-contaminated | Synthesized, decontaminated |
| **Agent autonomy** | Fixed search procedures | Fixed multi-turn loops | Learned optimization strategies |
| **Training stability** | N/A | Often unstable | Stable via warm-up |
| **Evaluation fairness** | Varies | Compromised by leakage | Strict train/test separation |

The paper's contribution is not incremental improvement but a **paradigm shift**: from treating CUDA generation as a code completion task to treating it as a **hardware-aware optimization problem** that requires learned reasoning, not just pattern matching.

## 3. Technical Approach

This paper presents a systems-building contribution that constructs a complete reinforcement learning pipeline for training language models to generate optimized CUDA kernels through agentic interaction with a GPU execution environment.

### 3.1 Reader orientation (approachable technical breakdown)

CUDA Agent is a training system that teaches a large language model to write, debug, and optimize CUDA kernel code by interacting with a sandboxed GPU environment over multiple turns, receiving rewards based on correctness and speedup measurements. The system solves the problem of LLMs producing CUDA code that is slower than compiler-optimized baselines by replacing static code generation with an iterative agent loop backed by stable reinforcement learning training, scalable synthetic data, and carefully designed reward signals that prevent the model from learning shortcuts or hacks.

### 3.2 Big-picture architecture (diagram in words)

The CUDA Agent system consists of three interconnected components that form a complete training pipeline. First, the **Scalable Data Synthesis Pipeline** (Section 3.1, Figure 1) crawls seed operators from PyTorch libraries, uses an LLM to compose them into fused multi-operator tasks, and filters them through execution-based criteria to produce 6,000 training problems. Second, the **Skill-Integrated Agent Loop** (Section 3.2, Figure 2) provides the model with structured tools (Bash, file editing, profiling scripts) and a formal workflow specification (`SKILL.md`) that guides iterative kernel development, compilation, correctness verification, and performance measurement in an isolated GPU sandbox. Third, the **RL Algorithmic Infrastructure** (Section 3.3, Figure 3) implements a multi-stage warm-up strategy combining single-turn RL, Rejection Fine-Tuning for actor initialization, Value Pretraining for critic initialization, and PPO optimization to enable stable training over 150 steps with context lengths up to 128k tokens. Information flows from the data pipeline into the agent environment, where trajectories are collected and filtered, then fed into the RL training loop which updates the policy and value networks before the next round of agent interaction.

### 3.3 Roadmap for the deep dive

- We begin with the **data synthesis pipeline** because reinforcement learning requires tasks to optimize, and understanding how the 6,000 training problems are constructed explains what the model actually learns to solve.
- We then examine the **agent loop and environment** because the interaction protocol determines what actions the model can take, what feedback it receives, and how reward signals are computed from execution.
- We proceed to the **reward design** because the reward function directly shapes what behaviors the RL algorithm reinforces, and the paper's discrete milestone-based reward differs substantially from prior continuous speedup-based approaches.
- We analyze the **RL training stability mechanisms** because the paper identifies training collapse as a critical failure mode and proposes specific warm-up techniques to prevent it.
- We conclude with **hyperparameters and infrastructure** because the scale of training (128k context, 128 GPUs, 150 steps) is essential to understanding the resource requirements and reproducibility constraints.

### 3.4 Detailed, sentence-based technical breakdown

#### 3.4.1 Scalable Training Data Synthesis Pipeline

The data synthesis pipeline addresses the fundamental bottleneck that manually creating expert-level CUDA kernel reference code is prohibitively expensive, which would limit supervised fine-tuning to small datasets. The pipeline operates in three sequential stages as illustrated in Figure 1, with each stage transforming the problem space to increase diversity and difficulty while maintaining executability.

**Stage 1: Seed Problem Crawling** extracts reference operators implemented in PyTorch from the `torch` and `transformers` libraries to establish a comprehensive seed problem set. Each operator is represented as a Python class with initialization (`__init__`) and forward (`forward`) methods, ensuring that the operators are widely used and well-maintained rather than individually maintained repositories that lack sufficient code quality. This crawling produces a repository of fundamental computational primitives such as `matmul`, `relu`, and `conv2d` that serve as building blocks for more complex tasks.

**Stage 2: Combinatorial Problem Construction** expands the dataset and introduces higher complexity by utilizing an LLM to synthesize aggregated operators from the seed primitives. The LLM is prompted to sample no more than 5 operator classes from the `torch` library and compose them sequentially by stacking them into a single computational layer. The paper explicitly excludes sampling operator classes from the `transformers` library during this composition stage because these operators are typically higher-level modules that already encapsulate multiple primitive operations, which would make the optimization task ill-defined. The key observation motivating this design is that composing multiple operators into fused tasks yields valuable new training problems because the combined problem is often not equivalent to trivially optimizing each operator in isolation and then chaining them. Fusion reshapes the optimization landscape by avoiding intermediate global-memory materialization, coupling stages through shared register/SMEM/occupancy constraints, and requiring a unified parallel mapping and data layout that may favor downstream consumption.

**Stage 3: Problems Filtering** implements a rigorous data selection process to filter out problems that are either too easy or too difficult based on execution feedback. Each operator is validated against four explicit criteria: (1) the operator must execute successfully in both Eager and Compile modes to ensure baseline executability, (2) operators with inherent stochasticity are excluded to ensure reproducibility of evaluation, (3) for anti-hacking purposes, outputs for different inputs are verified to be neither constant values nor numerically indistinguishable to prevent trivial solutions, and (4) execution time in eager mode is restricted to the range of 1 ms to 100 ms to filter out trivial or excessively heavy tasks. Additionally, operators that exhibit high similarity to KernelBench test cases are excluded to prevent training-test contamination, with the similarity distribution provided in Appendix A Figure 7 showing that no training sample exceeds the decontamination threshold of 0.9 AST similarity after filtering.

The final filtered synthesized training dataset contains exactly 6,000 samples, forming `CUDA-Agent-Ops-6K`, a curated operator-level dataset for training CUDA-capable agents. Table 3 in Appendix A shows the composition: 83.77% of samples are compositions of 2 torch operators, 7.62% are 3-operator compositions, 2.80% are 4-operator compositions, 1.23% are 5-operator compositions, 3.40% are single torch operators, and 1.18% are transformers operators used as standalone tasks. This distribution reflects a deliberate balance between simple operators, moderately complex compositions, and higher-level transformers modules.

#### 3.4.2 Skill-Integrated Agent Loop

The agent loop is designed to align with the widely adopted OpenHands framework to ensure generalizability beyond this specific task, while incorporating CUDA-specific tools and workflows that formalize the standard process for writing, validating, and optimizing CUDA kernels. The LLM is provided with a standard suite of shell utilities including `BashTool`, `GlobTool`, `MultiEditTool`, and `TodoWriteTool` to fully support CUDA coding development, with the complete tool list detailed in Appendix B.1.

The agent loop follows the ReAct-style paradigm, interleaving reasoning, action execution, and observation to enable iterative coding, debugging, and performance optimization as visualized in Figure 2. On top of the general tools, the system deliberately puts CUDA coding specific instructions and tools (e.g., the profiling tool to compare the performance of generated kernel against `torch.compile`) in an Agent Skills format inspired by prior work on agent capabilities. A CUDA kernel coding specific instruction file called `SKILL.md` formulates the standard process to optimize the CUDA kernel with four explicit steps: (1) analyze the performance of the native PyTorch implementation using the provided `profile.py` script to identify performance bottlenecks and optimization opportunities such as excessive kernel launches and suboptimal memory access patterns, (2) implement custom CUDA operators by rewriting the model in `model_new.py` and developing the corresponding CUDA kernel source files and binding code targeting the performance-critical operators identified in the analysis stage, (3) compile and evaluate the optimized model in the provided GPU sandbox environment and iteratively refine the CUDA kernel implementations until both numerical correctness and performance requirements are met, and (4) repeat the optimization process starting from Step 2 until the final implementation achieves at least a 5% speedup over the `torch.compile` baseline while passing all numerical correctness checks.

The workspace structure enforces separation of concerns to prevent the agent from modifying infrastructure files. The agent can only modify files in the `kernels/` directory (for CUDA kernel implementations), `model_new.py` (for the optimized PyTorch model using custom ops), and binding files within the kernels directory. Critical infrastructure files including `binding.cpp`, `binding_registry.h`, the `utils/` directory containing verification and profiling tools, and the original `model.py` are protected and cannot be modified. This design choice prevents reward hacking by ensuring that performance gains originate solely from the generated CUDA kernels rather than from modifying the evaluation logic or invoking fallback implementations.

The sandbox environment is architected as a CPU–GPU resource–decoupled system to ensure accurate speedup measurement for reliable reward signals and efficient GPU utilization. A Docker-based terminal sandbox handles CPU-centric tasks such as kernel compilation, while the agent leverages pre-defined CUDA Agent skill scripts to dispatch verification and profiling jobs to a dedicated GPU sandbox pool consisting of 128 NVIDIA H20 GPUs. This process-level isolation and exclusive resource allocation eliminate inter-process interference, ensuring stable latency measurements and guaranteed HBM capacity. The profiling pipeline is carefully engineered with proper device synchronization, warm-up iterations, and repeated measurements with averaging to substantially reduce measurement noise and metric fluctuation.

To further avoid reward hacking, the system enforces five specific constraints: (1) the provided Python scripts for correctness verification and performance profiling are protected via file permission controls, preventing the agent from modifying or interfering with the evaluation logic, (2) execution-time constraints are enforced using context managers that explicitly forbid invoking fallback implementations from `torch.nn.functional`, (3) for each problem, kernel outputs are validated against five randomly sampled inputs strictly following the KernelBench evaluation protocol to ensure functional correctness, (4) the agent is not provided with any web search or external information retrieval tools, ensuring that all solutions are derived purely from the local execution environment, and (5) the workspace structure prevents modification of infrastructure files as described above.

#### 3.4.3 Robust Reward Design

The reward design addresses a critical observation that existing RL approaches for CUDA generation use speedup over baselines as the reward signal, but this approach suffers from outliers and bias toward easy kernels because operators vary substantially in optimization difficulty, making raw speedup an unreliable proxy for code quality. The paper proposes a normalized, robust reward scheme to remedy this and jointly optimize correctness and execution latency by assigning discrete reward values based on achieving specific performance milestones rather than directly regressing on noisy runtime ratios.

The reward score $r$ is assigned from the set $\{-1, 1, 2, 3\}$ based on correctness and performance according to the following piecewise function:

$$
r =
\begin{cases}
-1 & \text{if correctness check fails} \\
3 & \text{if } b(t, t_{\text{eager}}) \land b(t, t_{\text{compile}}) \\
2 & \text{if } b(t, t_{\text{eager}}) \\
1 & \text{otherwise}
\end{cases}
$$

where $t$ is the generated kernel's runtime, $t_{\text{eager}}$ and $t_{\text{compile}}$ are the runtimes of PyTorch's eager implementation and `torch.compile` version respectively, and the function $b(t, t_0)$ indicates a significant speedup over baseline $t_0$ defined as:

$$
b(t, t_0) = \mathbb{I}\left[\frac{t_0 - t}{t_0} > 5\%\right]
$$

where $\mathbb{I}[\cdot]$ is the indicator function that returns 1 if the condition is true and 0 otherwise. This design means that a reward of 3 is given only when the generated kernel is correct AND achieves more than 5% speedup over BOTH the eager and compile baselines, a reward of 2 is given when correct AND faster than eager only, a reward of 1 is given when correct but not faster than either baseline, and a reward of -1 is given when the correctness check fails regardless of speed.

The paper validates this design in Section 4.3.2, showing that replacing the robust reward schedule with a raw speedup reward (defined as $r_s = t_{\text{compile}} / t_{\text{gen}}$ for correct solutions and $-1$ for incorrect ones) yields comparable functional correctness but substantially weaker optimization performance. The ablation study in Table 2 shows that without the robust reward, the Faster Rate vs. Compile drops from 96.8% to 60.4%, and the Geometric Mean Speed-up vs. Compile drops from 2.11× to 1.25×. These results indicate that a normalized, milestone-based reward is better aligned with the goal of producing consistently faster kernels because it assigns credit to clear performance targets rather than directly optimizing noisy continuous ratios.

#### 3.4.4 Algorithmic Improvements for Stable RL Training

The paper identifies a critical failure mode in initial RL trials: training could only proceed stably for 17 steps before the model's performance collapsed. The root cause is identified as a severe domain distribution mismatch, where the base model's learned prior deviates significantly from the data distribution required for CUDA kernel coding because CUDA coding data accounts for less than 0.01% of pretraining data. This distribution gap results in numerous sampled low-probability CUDA kernel code tokens, and when training and inference engines use different numerical precisions (e.g., BF16 vs FP16), those low-probability tokens result in large importance sampling ratio variance because small numerical errors in computing token probabilities $\pi_\theta(y_t)$ near the precision floor (e.g., $\pi_\theta(a_t | s_t) \approx 10^{-9}$) cause the importance ratio $\rho_t(\theta)$ to fluctuate wildly or explode.

The importance sampling ratio is defined as:

$$
\rho_t(\theta) = \frac{\pi_\theta(a_t | s_t)}{\pi_{\theta_{\text{old}}}(a_t | s_t)}
$$

where $\pi_\theta(a_t | s_t)$ is the probability of taking action $a_t$ (i.e., generating token $t$) in state $s_t$ under the current policy parameters $\theta$, and $\pi_{\theta_{\text{old}}}(a_t | s_t)$ is the probability under the old policy parameters used for trajectory sampling. When both numerator and denominator are near the precision floor, small numerical errors cause the ratio to become unstable, similar to issues discussed in prior work on RL collapse from training-inference mismatch.

To achieve stable reinforcement learning, the paper proposes a multi-stage warm-up strategy illustrated in Figure 3 that initializes both the actor and critic models using agent trajectories generated by the base model after single-turn RL. This strategy consists of four sequential phases.

**Phase 1: Single-Turn Warm-up** performs single-turn RL on the base model to enhance its capability in CUDA kernel generation before introducing the complexity of multi-turn agent interaction. The paper uses PPO for optimization, where the base model serves as both the policy and value network during this phase. The context window length is 32,768 tokens for single-turn RL, which is smaller than the 131,072 tokens used for agentic RL to reduce computational cost during this preliminary phase.

**Phase 2: Actor Initialization via Rejection Fine-Tuning (RFT)** uses the model from single-turn RL to collect CUDA agent trajectories by running the agent in the agent loop from Section 3.2, then applies rejection sampling to retain only high-quality rollouts according to two rubrics. First, outcome filtering keeps only trajectories that achieve a positive reward ($R > 0$). Second, pattern filtering discards trajectories that exhibit inefficient or invalid behaviors, such as redundant multi-turn loops or hallucinations that violate the predefined tool-call schema. The filtered trajectories are then used to optimize the actor via standard supervised fine-tuning with the following objective:

$$
\mathcal{L}_{\text{RFT}}(\theta) = -\mathbb{E}_{\tau \sim \mathcal{D}'}\left[\sum_{t=1}^{T} \log \pi_\theta(a_t | s_t, a_{&lt;t})\right]
$$

where $\tau = (s_0, s_1, \ldots, s_{T-1})$ denotes a filtered CUDA agent trajectory, $\pi_\theta$ is the policy parameterized by $\theta$, and $\mathcal{D}'$ represents the dataset after rejection sampling. The paper ablates this stage in Section 4.3.3, showing that removing RFT causes training reward to collapse as shown in Figure 4a, with a concurrent increase in actor entropy (Figure 4b) indicating that the policy distribution becomes increasingly diffuse and produces incoherent outputs. RFT provides a critical prior that constrains entropy growth during reinforcement learning and keeps the optimization trajectory within a well-structured output distribution.

**Phase 3: Critic Initialization via Value Pretraining** utilizes the sampled trajectory data comprising state sequences and their corresponding outcome rewards to pretrain the critic network before agentic RL begins. Let $\tau = (s_0, s_1, \ldots, s_{T-1})$ denote a trajectory where $s_t$ represents the state at token position $t$, and let $r$ denote the outcome reward assigned at the final token (i.e., $r_t = 0$ for $t < T-1$ and $r_{T-1} = r$). Target values are computed using Generalized Advantage Estimation:

$$
V^{\text{targ}}_t = V_\phi(s_t) + \hat{A}_t
$$

where the advantage estimate $\hat{A}_t$ is computed as:

$$
\hat{A}_t = \sum_{l=0}^{T-1-t} (\gamma \lambda)^l \delta_{t+l}
$$

and $\delta_t = r_t + \gamma V_\phi(s_{t+1}) - V_\phi(s_t)$ is the temporal difference error with $V_\phi(s_T) = 0$. The paper sets $\gamma = 1$ and $\lambda = 0.95$ in experiments. The critic parameters $\phi$ are optimized by minimizing the mean squared error:

$$
\mathcal{L}_{\text{VP}}(\phi) = \frac{1}{2} \mathbb{E}_{\tau \sim \mathcal{D}}\left[\frac{1}{T} \sum_{t=0}^{T-1} \left(V_\phi(s_t) - V^{\text{targ}}_t\right)^2\right]
$$

where $\mathcal{D}$ denotes the collection of agent trajectories. The ablation study in Section 4.3.3 shows that without Value Pretraining, the critic fails to learn a meaningful value function as reflected by low explained variance (Figure 5a), which leads to an explosion in trajectory length (Figure 5b) because the uninitialized critic fails to penalize fruitless or redundant search paths. Value Pretraining ensures that the critic can immediately provide accurate feedback, guiding the agent toward efficient optimization paths and avoiding the computational instability of near-infinite interaction loops.

**Phase 4: Agentic RL with PPO** employs Proximal Policy Optimization to optimize the actor model $\pi_\theta$ using the clipped surrogate objective. Let $\pi_\theta$ denote the policy to be optimized, and $\pi_{\theta_{\text{old}}}$ denote the policy used for trajectory sampling. The objective maximizes the expected return:

$$
\mathcal{L}_{\text{CLIP}}(\theta) = \mathbb{E}_{\tau \sim \mathcal{D}}\left[\frac{1}{T} \sum_{t=0}^{T-1} \min\left(\rho_t(\theta) \hat{A}_t, \text{clip}(\rho_t(\theta), 1 - \epsilon_{\text{lower}}, 1 + \epsilon_{\text{higher}}) \hat{A}_t\right)\right]
$$

where $\rho_t(\theta) = \frac{\pi_\theta(a_t | s_t)}{\pi_{\theta_{\text{old}}}(a_t | s_t)}$ is the importance sampling ratio between the current and old policies, $a_t$ denotes the action (i.e., token) taken at position $t$, $\epsilon_{\text{lower}} = 0.2$, and $\epsilon_{\text{higher}} = 0.28$ following prior work on large-scale RL systems. With this modification, the RL trial can stably train for 150 steps (extended to 200 agent turns during evaluation) with consistent reward growth, compared to the 17-step collapse without warm-up.

#### 3.4.5 Training Configurations and Infrastructure

The training system leverages Seed1.6 as the base model, which is a Mixture-of-Experts (MoE) model with 23 billion active parameters and 230 billion total parameters. The global batch size is set to 1,024, matching the mini-batch size for online PPO updates. The learning rate for the actor is $3 \times 10^{-6}$ and for the critic is $6 \times 10^{-6}$, with these hyperparameters shared between single-turn RL and agentic RL phases. The context window length is 32,768 tokens for single-turn RL and 131,072 tokens for agentic RL, enabling the model to handle long interaction trajectories. A maximum of 150 agent turns is imposed during training rollouts, which is relaxed to 200 agent turns during evaluation to allow more exploration. The model is trained over 150 training steps total.

The GPU infrastructure consists of a dedicated pool of 128 NVIDIA H20 GPUs for the sandbox environment, with process-level isolation and exclusive resource allocation to eliminate inter-process interference. This reliance on massive GPU resources is noted as a limitation in Appendix E, as it incurs considerable computational and engineering cost that may limit accessibility for the broader research community. The paper does not compare against more sophisticated compiler frameworks such as TVM because these systems are difficult to integrate into a large-scale RL training loop with thousands of rollouts due to their substantial tuning overhead and complex deployment requirements, so the focus remains on `torch.compile` as a widely adopted, training-friendly baseline.

## 4. Key Insights and Innovations

The contributions of CUDA Agent extend beyond engineering scale; they represent conceptual shifts in how we apply reinforcement learning (RL) to hardware-aware code generation. Below, we distill the four most significant innovations, distinguishing between incremental improvements and fundamental changes to the training paradigm.

### 4.1 Combinatorial Fusion as a Curriculum for Hardware Constraints

**The Innovation:** Rather than treating data synthesis as merely "creating more problems," CUDA Agent introduces **combinatorial fusion** as a deliberate curriculum to teach hardware resource contention.

**Why This Differs from Prior Work:**
Existing datasets for kernel generation typically consist of isolated operators (e.g., "optimize this matrix multiplication"). While useful for syntax, single operators rarely expose the complex constraints that define high-performance CUDA. As noted in Section 3.1, the paper observes that optimizing a fused task is **not equivalent** to optimizing each operator in isolation and chaining them.

**The Mechanism:**
By composing 2–5 operators into a single computational layer (Table 3 in Appendix A shows 83.77% are 2-operator compositions), the data synthesis pipeline forces the model to learn **interaction effects**:
*   **Shared Memory Contention:** Multiple operators competing for the same limited shared memory (SMEM).
*   **Register Pressure:** Combined register usage affecting occupancy across the fused kernel.
*   **Memory Layout Coupling:** Data layouts that favor the first operator might penalize the second, requiring a unified strategy.

**Significance:**
This transforms the learning objective from "syntax correctness" to "resource management." A model trained only on single operators might learn to optimize memory coalescing for one kernel but fail to recognize when fusing two kernels eliminates a global memory write entirely. This curriculum design is a fundamental innovation because it generates training signals for **optimization strategies** (like fusion) that are invisible in single-operator datasets.

### 4.2 Discrete Milestone Rewards for Noisy Hardware Signals

**The Innovation:** Replacing continuous speedup ratios with **discrete milestone-based rewards** to mitigate hardware measurement noise.

**Why This Differs from Prior Work:**
Prior RL approaches for code optimization (e.g., Kevin [4], CUDA-L1 [14]) typically use a continuous reward signal like $r = t_{\text{baseline}} / t_{\text{generated}}$. The paper identifies a critical flaw: hardware execution times are inherently noisy due to system interference, clock scaling, and measurement variance. Optimizing directly on a noisy continuous signal causes the policy to chase statistical outliers rather than genuine improvements.

**The Mechanism:**
As defined in Equation 1 (Section 3.2), the reward function converts performance into discrete categories:
*   $r=3$: Correct + >5% faster than **both** eager and compile baselines.
*   $r=2$: Correct + >5% faster than **eager** only.
*   $r=1$: Correct but not faster.
*   $r=-1$: Incorrect.

This design acts as a **low-pass filter** on the reward signal. A kernel must clear a specific threshold (5% speedup) to receive credit, preventing the model from overfitting to micro-variations in runtime measurement.

**Significance:**
The ablation study in Table 2 provides empirical proof of this insight. Removing the robust reward (`w/o Robust Reward`) maintains high correctness (96.8% Pass Rate) but collapses optimization performance (Faster Rate vs. Compile drops from 96.8% to 60.4%). This demonstrates that **reward shaping is more critical than reward magnitude** in hardware RL. The innovation lies in recognizing that for hardware metrics, "better" is a categorical state (did you beat the baseline significantly?) rather than a continuous gradient.

### 4.3 Precision-Aware Warm-up for Stable Code RL

**The Innovation:** Identifying and solving a specific **numerical precision mismatch** failure mode that causes RL collapse in code generation tasks.

**Why This Differs from Prior Work:**
Standard RL instability is often attributed to hyperparameter sensitivity or distribution shift. CUDA Agent diagnoses a deeper systems-level issue: the mismatch between training precision (e.g., BF16) and inference precision (e.g., FP16) when computing token probabilities for rare tokens.

**The Mechanism:**
As detailed in Section 3.3, CUDA tokens are low-probability events in general LLMs (&lt;0.01% of pretraining data). When probabilities approach the precision floor (e.g., $\pi_\theta(a_t | s_t) \approx 10^{-9}$), small numerical errors cause the importance sampling ratio $\rho_t(\theta)$ to explode:

$$
\rho_t(\theta) = \frac{\pi_\theta(a_t | s_t)}{\pi_{\theta_{\text{old}}}(a_t | s_t)}
$$

The proposed **multi-stage warm-up** (Figure 3) is not just a training trick; it is a corrective measure for this precision gap. By initializing the actor with Rejection Fine-Tuning (RFT) and the critic with Value Pretraining, the system ensures that the policy operates in a probability range where numerical precision is stable before full RL begins.

**Significance:**
Figure 4 and Figure 5 illustrate the catastrophic failure without this insight:
*   **Without RFT:** Actor entropy spikes (Figure 4b), indicating the policy becomes diffuse and incoherent.
*   **Without Value Pretraining:** The critic fails to estimate value (Figure 5a), leading to infinite interaction loops (Figure 5b).

This is a fundamental contribution to **RL for Code** broadly. It suggests that for domains with long-tail token distributions (like specialized programming languages), standard PPO implementations may fail due to numerical precision limits, not just algorithmic flaws.

### 4.4 Autonomous Strategy Learning vs. Fixed Workflows

**The Innovation:** Shifting the learning objective from **generating code** to **managing the optimization process**.

**Why This Differs from Prior Work:**
Most agentic systems (e.g., STARK [5], CudaForge [26]) rely on fixed, hand-designed workflows (e.g., "Plan → Code → Debug → Repeat"). These systems constrain the agent to a predefined number of turns or a rigid sequence of actions. CUDA Agent allows up to 200 interaction turns (Section 4.1) where the model decides **when** to profile, **when** to edit, and **when** to stop.

**The Mechanism:**
The agent learns a **policy over actions**, not just tokens. It must infer that:
*   Profiling early is useful to identify bottlenecks.
*   Profiling repeatedly without code changes is wasteful.
*   Stopping after achieving a 5% speedup is sufficient (or continuing to push for more).

**Significance:**
Table 2 (`w/o Agent Loop`) shows that removing the multi-turn interaction capability drops the Faster Rate vs. Compile from 96.8% to 14.1%. This massive gap indicates that **optimization is an iterative search process**, not a single-step generation task. The innovation is treating the **debugging loop itself as learnable**. Prior work hardcoded the loop; CUDA Agent learns the strategy. This enables the model to adapt its effort based on problem difficulty—spending more turns on complex Level-3 tasks (ResNet blocks) and fewer on simple Level-1 tasks (element-wise ops).

### 4.5 Critical Assessment of Trade-offs

While these innovations drive state-of-the-art performance, they introduce significant trade-offs that future work must address:

*   **Compute Cost:** The stability mechanisms (RFT, Value Pretraining, 150 RL steps) and infrastructure (128 NVIDIA H20 GPUs, Section 4.1) make this approach prohibitively expensive for most research groups. This is a **systems-heavy** solution rather than an algorithmic shortcut.
*   **Baseline Dependency:** The reward function relies on `torch.compile` as a baseline. If the compiler improves significantly, the reward landscape shifts. The model learns to **beat the compiler**, not necessarily to find the absolute theoretical hardware limit.
*   **Generalizability:** The data synthesis pipeline is specific to PyTorch/CUDA. Applying this to other hardware (e.g., TPUs, NPUs) would require rebuilding the sandbox, profiling tools, and data synthesis logic from scratch.

These limitations highlight that CUDA Agent is a **proof of scalability** for agentic RL in systems programming. The core insights—combinatorial data, discrete rewards, and precision-aware stability—are transferable, but the infrastructure cost remains a barrier to entry.

## 5. Experimental Analysis

This section examines how the authors validate CUDA Agent's claims through rigorous experimentation. We will walk through the evaluation methodology, analyze the quantitative results with specific numbers, assess whether the experiments convincingly support the paper's claims, and examine the ablation studies that isolate each component's contribution.

### 5.1 Evaluation Methodology

#### Benchmark Dataset: KernelBench

The primary evaluation benchmark is **KernelBench** [17], a standardized test suite for CUDA kernel generation. The benchmark consists of 250 distinct operator tasks distributed across three difficulty levels:

| Level | Task Count | Description |
|-------|------------|-------------|
| Level 1 | 100 tasks | Simple operators (e.g., element-wise operations, basic matrix operations) |
| Level 2 | 100 tasks | Operator sequences requiring fusion (e.g., multiple operations chained together) |
| Level 3 | 50 tasks | Complex neural network blocks (e.g., ResNet BasicBlock, attention mechanisms) |

**Critical Design Choice:** The authors adapt KernelBench tasks from their original single-file format into the multi-file development environment described in Section 3.2. This ensures fair comparison—baseline models are evaluated under the same agent loop conditions as CUDA Agent, not in a disadvantaged single-turn setting.

**Train/Test Separation:** To prevent data leakage (a critical flaw in prior work like Kevin [4] and CUDA-L1 [14]), the authors perform explicit decontamination using AST-based code similarity. As shown in Appendix A Figure 7, no training sample exceeds the 0.9 AST similarity threshold with any evaluation sample after filtering. This strict separation ensures that performance gains reflect genuine generalization, not benchmark memorization.

#### Evaluation Metrics

The paper employs three complementary metrics to capture different aspects of kernel generation quality:

1. **Pass Rate:** The percentage of tasks where the generated kernel successfully compiles AND passes functional correctness checks. This measures basic competency—can the model produce working code?

2. **Faster Rate:** The percentage of correct kernels that achieve faster execution time than the baseline. Reported separately for:
   - vs. Eager: Comparison against PyTorch's default eager execution
   - vs. Compile: Comparison against `torch.compile` (the more challenging baseline)

3. **Speed-up (Geometric Mean):** The geometric mean of execution speed-up ratios, computed exclusively for correct solutions. This captures the magnitude of improvement, not just whether improvement occurred.

**Why Three Metrics?** Each metric addresses a different failure mode. A model could achieve 100% Pass Rate but produce slow kernels (high correctness, low optimization). Conversely, a model could produce fast kernels that are numerically incorrect (high speed-up, low correctness). The combination ensures comprehensive evaluation.

#### Baseline Models

The authors compare against four categories of baselines, all evaluated under the same agent loop conditions:

| Category | Models | Purpose |
|----------|--------|---------|
| Proprietary LLMs | Claude Opus 4.5 [3], Gemini 3 Pro [7] | Test against state-of-the-art general-purpose coding models |
| Open-Source LLMs | GLM 4.6 [27], Kimi K2 [21] | Test against leading open-weight coding models |
| Base Model | Seed1.6 [20] | Measure improvement from RL training |
| Compiler | torch.compile (Eager and Compile modes) | Test against automatic optimization systems |

**Important Note:** The authors report that ChatGPT-5 series models (5, 5.1, 5.2) were not amenable to evaluation because they "consistently declined to respond to CUDA-related prompts" (footnote 1, page 2). This is a notable limitation—the strongest proprietary models may be inaccessible for this task due to safety filtering.

#### Experimental Setup

**Hardware Infrastructure:**
- GPU Pool: 128 NVIDIA H20 GPUs for sandbox environment
- Architecture: CPU-GPU resource-decoupled sandbox (Docker for CPU tasks, dedicated GPU pool for profiling)
- Purpose: Process-level isolation eliminates inter-process interference, ensuring stable latency measurements

**Training Configuration:**
- Base Model: Seed1.6 (MoE architecture, 23B active parameters, 230B total parameters)
- Batch Size: 1,024 (global and mini-batch for online PPO)
- Learning Rates: Actor $3 \times 10^{-6}$, Critic $6 \times 10^{-6}$
- Context Length: 32,768 tokens (single-turn RL), 131,072 tokens (agentic RL)
- Training Steps: 150 total steps
- Agent Turns: Maximum 150 during training, relaxed to 200 during evaluation

**Evaluation Protocol:**
- For each task, the best-performing solution from the agent's interaction trajectory is extracted (the one achieving maximum speed-up vs. `torch.compile`)
- Correctness validated against five randomly sampled inputs per problem (following KernelBench protocol)
- Profiling includes device synchronization, warm-up iterations, and repeated measurements with averaging

### 5.2 Main Quantitative Results

#### Overall Performance (Table 1)

Table 1 presents the primary results comparing CUDA Agent against all baselines. The numbers reveal three critical findings:

**Finding 1: CUDA Agent Dominates Proprietary Models**

| Model | Pass Rate | Faster vs. Compile | Speed-up vs. Compile |
|-------|-----------|-------------------|---------------------|
| Claude Opus 4.5 | 95.2% | 66.4% | 1.46× |
| Gemini 3 Pro | 91.2% | 69.6% | 1.42× |
| **CUDA Agent** | **98.8%** | **96.8%** | **2.11×** |

The gap is substantial: CUDA Agent achieves **30.4 percentage points higher Faster Rate** than Claude Opus 4.5 (96.8% vs. 66.4%) and **44.5% higher geometric mean speed-up** (2.11× vs. 1.46×). This is not a marginal improvement—it represents a fundamental capability gap.

**Why This Matters:** Proprietary models like Claude Opus 4.5 represent the peak of general-purpose coding capability, trained on massive datasets with enormous compute. CUDA Agent's superior performance demonstrates that **specialized RL training on a focused domain can outperform general-purpose scaling**. The model learns hardware-aware optimization strategies that general LLMs do not internalize.

**Finding 2: CUDA Agent Consistently Beats torch.compile**

The Abstract claims "100%, 100%, and 92% faster rate over torch.compile on Level-1, Level-2, and Level-3 splits." Table 1 confirms:

| Level | Faster Rate vs. Compile | Speed-up vs. Compile |
|-------|------------------------|---------------------|
| Level 1 | 97.0% | 1.87× |
| Level 2 | 100.0% | 2.80× |
| Level 3 | 90.0% | 1.52× |

**Critical Observation:** Level 2 shows the largest speed-up (2.80×). As the authors explain in Section 4.2, Level 2 tasks involve operator sequences where fusion provides maximum benefit. Traditional compilers rely on predefined, rule-based patterns for kernel fusion which struggle with non-trivial operator combinations. CUDA Agent explores a larger design space through its iterative agent loop, discovering hardware-specific memory access patterns and tiling strategies inaccessible to static backends.

**Finding 3: Performance Degrades with Difficulty (But Remains Strong)**

| Level | Pass Rate | Faster vs. Compile |
|-------|-----------|-------------------|
| Level 1 | 100.0% | 97.0% |
| Level 2 | 100.0% | 100.0% |
| Level 3 | 94.0% | 90.0% |

Level 3 (complex neural network blocks) shows the largest drop—6 percentage points in Pass Rate and 10 percentage points in Faster Rate. This is expected: ResNet blocks involve convolutions, batch normalization, residual connections, and activations, creating a much larger optimization space. However, 90% Faster Rate on Level 3 still represents a **40 percentage point advantage over Claude Opus 4.5** (90.0% vs. 50.0%).

#### Base Model Improvement

Comparing CUDA Agent to its base model (Seed1.6) isolates the contribution of RL training:

| Model | Pass Rate | Faster vs. Compile | Speed-up vs. Compile |
|-------|-----------|-------------------|---------------------|
| Seed1.6 (base) | 74.0% | 27.2% | 0.69× |
| CUDA Agent | 98.8% | 96.8% | 2.11× |

**Interpretation:** The base model achieves only 27.2% Faster Rate—meaning 72.8% of its correct kernels are **slower** than `torch.compile`. After RL training, this flips dramatically to 96.8%. The speed-up improves from 0.69× (31% slower than compile) to 2.11× (111% faster than compile).

This is the clearest evidence that RL training fundamentally improves the model's intrinsic CUDA optimization ability, not just its ability to search at inference time.

### 5.3 Ablation Studies: Isolating Component Contributions

Table 2 presents ablation studies where each component is removed individually. This is critical for validating that all three pillars (data, environment, RL algorithm) contribute meaningfully.

#### 5.3.1 Impact of Agent Loop (w/o Agent Loop)

| Metric | Full Model | w/o Agent Loop | Drop |
|--------|-----------|----------------|------|
| Pass Rate | 98.8% | 77.1% | -21.7 pp |
| Faster vs. Compile | 96.8% | 14.1% | -82.7 pp |
| Speed-up vs. Compile | 2.11× | 0.69× | -67.3% |

**Analysis:** Removing the agent loop causes catastrophic degradation, particularly in optimization quality (82.7 percentage point drop in Faster Rate). The "w/o Agent Loop" variant is trained as a single-turn model without execution feedback—it predicts the final kernel in one pass.

**Why This Happens:** Optimization is an iterative search process, not a single-step generation task. Without the ability to:
- Profile the native implementation to identify bottlenecks
- Compile and test intermediate versions
- Debug correctness failures across multiple turns
- Refine based on performance feedback

The model cannot learn optimization strategies. It can only produce syntactically correct code. This validates Insight 4.4 from the previous section: **the debugging loop itself is learnable**, and removing it collapses performance.

#### 5.3.2 Impact of Reward Design (w/o Robust Reward)

| Metric | Full Model | w/o Robust Reward | Drop |
|--------|-----------|-------------------|------|
| Pass Rate | 98.8% | 96.8% | -2.0 pp |
| Faster vs. Compile | 96.8% | 60.4% | -36.4 pp |
| Speed-up vs. Compile | 2.11× | 1.25× | -40.8% |

**Analysis:** Correctness remains high (96.8% Pass Rate), but optimization quality collapses. This variant uses a continuous speedup reward ($r_s = t_{\text{compile}} / t_{\text{gen}}$) instead of the discrete milestone-based reward.

**Why This Happens:** As explained in Section 3.2, hardware execution times are inherently noisy. A continuous reward signal causes the policy to chase statistical outliers rather than genuine improvements. The discrete milestone reward acts as a low-pass filter—the kernel must clear a 5% speedup threshold to receive credit.

**Key Insight:** Reward shaping is more critical than reward magnitude for hardware RL. The model can learn correctness easily, but learning optimization requires careful reward design that accounts for measurement noise.

#### 5.3.3 Impact of RFT (w/o RFT)

| Metric | Full Model | w/o RFT | Drop |
|--------|-----------|---------|------|
| Pass Rate | 98.8% | 95.6% | -3.2 pp |
| Faster vs. Compile | 96.8% | 49.8% | -47.0 pp |
| Speed-up vs. Compile | 2.11× | 1.05× | -50.2% |

**Analysis:** Removing Rejection Fine-Tuning causes severe optimization degradation (47 percentage point drop) and training instability. Figure 4a shows training reward collapsing around step 20 without RFT, while Figure 4b shows actor entropy spiking concurrently.

**Why This Happens:** RFT provides a strong behavioral prior before full RL begins. Without it, the policy distribution becomes diffuse (high entropy), producing incoherent outputs. The importance sampling ratios become unstable when token probabilities approach the precision floor, causing the optimization to diverge.

**Critical Detail:** The ablation reports results from "the final validation step before training collapse"—meaning the model literally cannot complete training without RFT. This validates Insight 4.3: precision-aware warm-up is not optional for stable code RL.

#### 5.3.4 Impact of Value Pretraining (w/o Value Pretraining)

| Metric | Full Model | w/o Value Pretraining | Drop |
|--------|-----------|----------------------|------|
| Pass Rate | 98.8% | 98.6% | -0.2 pp |
| Faster vs. Compile | 96.8% | 50.9% | -45.9 pp |
| Speed-up vs. Compile | 2.11× | 1.00× | -52.6% |

**Analysis:** Correctness is nearly preserved (98.6% vs. 98.8%), but optimization quality halves. Figure 5a shows the critic's explained variance remains near zero without Value Pretraining, and Figure 5b shows response length exploding (high clipped ratio indicates the critic fails to penalize redundant search paths).

**Why This Happens:** An uninitialized critic cannot distinguish valuable states from wasteful ones. The agent enters near-infinite interaction loops, exploring fruitless search paths without penalty. Value Pretraining ensures the critic can immediately provide accurate feedback, guiding the agent toward efficient optimization paths.

**Key Insight:** The critic is as important as the actor in agentic RL. Without a meaningful value function, the agent cannot learn when to stop searching.

### 5.4 Case Studies: Understanding Optimization Patterns

Appendix D provides detailed case studies across all three difficulty levels. These illustrate what CUDA Agent actually learns to do.

#### Level 1: Diagonal Matrix Multiplication (73.31× Speed-up)

**Task:** Compute the product of a diagonal matrix (constructed from a vector) with a dense matrix.

**Naive Approach:** Explicitly materialize the diagonal matrix, then invoke general matrix multiplication (GEMM). Time complexity: $O(N^2M)$.

**CUDA Agent's Optimization:** Recognize that left-multiplying by a diagonal matrix is equivalent to row-wise scaling. Implement a custom kernel that directly performs element-wise broadcast multiplication without constructing the intermediate diagonal matrix. Time complexity: $O(NM)$.

**Significance:** This demonstrates **algebraic simplification**—the model learns to recognize implicit mathematical structure and replace generic operators with specialized kernels. This is not pattern matching; it requires understanding the underlying computation.

#### Level 2: Matrix Multiplication + Division + Summation + Scaling (24.04× Speed-up)

**Task:** A sequence of operations including matrix multiplication, division, summation, and scaling.

**CUDA Agent's Optimization:**
1. Algebraic rearrangement: Transform matrix-matrix multiplication followed by reduction into column-wise reduction of weights followed by dot product
2. Kernel fusion: Collapse multiple arithmetic stages into a single kernel
3. Memory optimization: Use vectorized loads (float4) and shared-memory tree reduction

**Significance:** This demonstrates **kernel fusion**—the model learns to eliminate intermediate tensor materialization and reduce kernel launch overhead. The 24× speed-up comes from avoiding global memory writes between operations.

#### Level 3: ResNet BasicBlock (3.59× Speed-up)

**Task:** A realistic deep learning workload with convolutions, batch normalization, nonlinearities, and residual connections.

**CUDA Agent's Optimization:**
1. Fold BatchNorm parameters into convolution weights (eliminating the BatchNorm operator entirely at inference)
2. Use `cudnnConvolutionBiasActivationForward` to execute convolution, bias, and ReLU in a single cuDNN kernel
3. Enable TF32 computation to leverage Tensor Cores on Hopper GPUs
4. Fuse residual addition and final ReLU into a single custom kernel

**Significance:** This demonstrates **library-aware optimization**—the model learns when to use vendor-optimized libraries (cuDNN) vs. custom kernels. It also shows **hardware-aware optimization** (TF32 for Tensor Cores) and **graph transformation** (folding BatchNorm).

**Why Speed-up is Lower:** Level 3 shows smaller speed-ups (3.59× vs. 73× for Level 1) because:
- Complex workloads have less optimization headroom (more operations are already compute-bound)
- Library calls (cuDNN) are highly optimized, leaving less room for improvement
- Layout conversions (NCHW vs. NHWC) introduce overhead that can offset gains

This is expected and does not indicate failure—it reflects the inherent difficulty of the task.

### 5.5 Critical Assessment: Do the Experiments Support the Claims?

#### Claims vs. Evidence

| Claim | Evidence | Assessment |
|-------|----------|------------|
| "State-of-the-art on KernelBench" | Table 1: 96.8% Faster vs. Compile (best among all models) | **Strongly Supported** |
| "100%, 100%, 92% faster over torch.compile on Level 1-3" | Table 1: 97%, 100%, 90% (slight discrepancy in Level 3) | **Mostly Supported** (90% vs. claimed 92%) |
| "Outperforms proprietary models by ~40% on Level 3" | Table 1: 90% vs. 50-52% for Claude/Gemini = ~38-40 pp | **Supported** |
| "Stable training for 150 steps" | Figure 4-5: Training collapses without warm-up, stable with it | **Supported** |
| "RL improves intrinsic ability, not just search" | Seed1.6 base: 27.2% Faster vs. Compile → CUDA Agent: 96.8% | **Strongly Supported** |

#### Strengths of the Experimental Design

1. **Strict Train/Test Separation:** The AST-based decontamination (Appendix A Figure 7) ensures no training sample exceeds 0.9 similarity with evaluation samples. This addresses the data leakage problem in prior work.

2. **Fair Baseline Evaluation:** All baseline models are evaluated under the same agent loop conditions (Section 4.1). This prevents CUDA Agent from having an unfair advantage due to environment differences.

3. **Comprehensive Ablations:** Table 2 isolates each component's contribution, showing that all three pillars (agent loop, reward design, RL stability) are necessary.

4. **Multiple Difficulty Levels:** Evaluating across Level 1-3 shows how performance scales with task complexity, not just aggregate performance.

#### Limitations and Caveats

1. **No Comparison to TVM:** Appendix E acknowledges that more sophisticated compiler frameworks (TVM, MLIR) are not evaluated. The authors cite integration difficulty with large-scale RL training, but this leaves open whether CUDA Agent would still outperform production-grade compilers.

2. **GPU Resource Requirements:** The 128 NVIDIA H20 GPU pool is prohibitively expensive for most research groups. This limits reproducibility and accessibility.

3. **Proprietary Model Availability:** ChatGPT-5 series models could not be evaluated due to safety filtering (footnote 1). The strongest current models may not be testable on this benchmark.

4. **Level 3 Discrepancy:** The Abstract claims 92% Faster Rate on Level 3, but Table 1 shows 90%. This minor inconsistency (2 percentage points) suggests possible rounding or different evaluation runs.

5. **Single Hardware Configuration:** All evaluations use NVIDIA H20 GPUs. Performance may differ on other GPU architectures (A100, H100, consumer GPUs), limiting generalizability.

#### Mixed Results and Trade-offs

**Speed-up Varies by Task Type:**
- Level 1 (simple ops): Up to 73× speed-up (diagonal matmul)
- Level 2 (fusion): Up to 24× speed-up (fused arithmetic)
- Level 3 (neural blocks): ~3.6× speed-up (ResNet)

This is not a weakness—it reflects the inherent optimization headroom available. Simple operations with obvious algebraic simplifications have more room for improvement than complex, already-optimized neural network blocks.

**Correctness vs. Optimization Trade-off:**
The ablation studies show that correctness is easier to achieve than optimization:
- w/o Robust Reward: 96.8% Pass Rate, but only 60.4% Faster Rate
- w/o Value Pretraining: 98.6% Pass Rate, but only 50.9% Faster Rate

This suggests the model can learn to produce correct code relatively easily, but learning to optimize requires all components working together.

### 5.6 Summary of Experimental Findings

The experiments provide strong evidence for CUDA Agent's core claims:

1. **Specialized RL training outperforms general-purpose LLMs** by 30+ percentage points on optimization metrics, demonstrating that domain-specific training matters more than model scale alone.

2. **All three components are necessary:** Removing any pillar (agent loop, reward design, RL stability) causes severe degradation. This validates the systems-building approach rather than incremental improvements.

3. **Optimization is learnable:** The base model achieves only 27.2% Faster Rate, but after RL training reaches 96.8%. This proves the model learns intrinsic optimization ability, not just inference-time search.

4. **Difficulty matters:** Performance degrades gracefully with task complexity (Level 1 → Level 3), but remains substantially above baselines at all levels.

5. **Limitations are acknowledged:** The authors transparently report resource requirements, baseline limitations, and hardware specificity in Appendix E.

The experiments are thorough, well-controlled, and support the paper's conclusions. The primary limitation is reproducibility cost—the 128 GPU infrastructure is not accessible to most researchers. However, the conceptual insights (combinatorial data, discrete rewards, precision-aware warm-up) are transferable to resource-constrained settings.

## 6. Limitations and Trade-offs

While CUDA Agent demonstrates state-of-the-art performance on CUDA kernel generation, the approach carries significant constraints that affect its reproducibility, generalizability, and practical deployment. This section critically examines the assumptions underlying the system, the scenarios it does not address, and the open questions that remain.

### 6.1 Computational and Infrastructure Constraints

#### Massive GPU Resource Requirements

The most prominent limitation is the **extraordinary computational cost** required to train and evaluate CUDA Agent. As stated in Appendix E:

> "Our training pipeline relies on a large GPU pool with process-level isolation, which incurs considerable computational and engineering cost. This reliance on massive GPU resources may limit accessibility for broader research community."

The infrastructure requirements are specific and substantial:

| Resource | Specification | Implication |
|----------|---------------|-------------|
| GPU Pool | 128 NVIDIA H20 GPUs | Dedicated hardware unavailable to most research groups |
| Process Isolation | CPU-GPU decoupled sandbox architecture | Complex engineering overhead for deployment |
| Training Steps | 150 RL steps with 1,024 batch size | Extended training time even with massive parallelism |
| Context Length | 131,072 tokens for agentic RL | High memory requirements per rollout |

**Why This Matters:** This is not a minor engineering detail—it fundamentally limits who can reproduce or extend this work. A research group without access to a 128-GPU cluster cannot validate the claims or adapt the approach to new domains. The paper acknowledges this explicitly in Appendix E, leaving "exploring more resource-efficient training strategies as important directions for future work."

**Trade-off:** The stability mechanisms (RFT, Value Pretraining, 150 RL steps) that enable training are computationally expensive. Figure 4 and Figure 5 show that removing these components causes training collapse—but they also multiply the compute cost. This creates a **barrier to entry** that contradicts the open-source ethos common in the LLM research community.

#### Inference-Time Computational Cost

During evaluation, the agent is allowed up to **200 interaction turns** (Section 4.1), compared to 150 during training. Each turn involves:

1. Model inference (131k context window)
2. Tool execution (compilation, profiling, verification)
3. Feedback processing

**Implication:** Generating a single optimized kernel may require minutes to hours of wall-clock time, depending on compilation and profiling overhead. This makes CUDA Agent unsuitable for **real-time code generation** scenarios where immediate feedback is required. The system is designed for **offline optimization** of critical kernels, not interactive development.

### 6.2 Baseline and Evaluation Limitations

#### torch.compile as the Primary Baseline

The reward function and evaluation metrics are anchored to `torch.compile` performance. As defined in Equation 1 (Section 3.2), the reward depends on whether the generated kernel beats `torch.compile`:

$$
b(t, t_0) = \mathbb{I}\left[\frac{t_0 - t}{t_0} > 5\%\right]
$$

where $t_0$ includes $t_{\text{compile}}$.

**Critical Assumption:** This design assumes `torch.compile` represents a meaningful optimization ceiling. However, Appendix E acknowledges:

> "We do not compare CUDA Agent against more sophisticated compiler frameworks such as TVM. While these systems can potentially provide stronger baselines, they are difficult to integrate into a large-scale RL training loop with thousands of rollouts due to their substantial tuning overhead and complex deployment requirements."

**Why This Is Problematic:**

| Compiler | Integration Difficulty | Potential Performance |
|----------|----------------------|----------------------|
| torch.compile | Low (native PyTorch integration) | Moderate |
| TVM | High (complex tuning, deployment) | Potentially Higher |
| MLIR-based compilers | High (infrastructure complexity) | Potentially Higher |

**Implication:** CUDA Agent learns to **beat torch.compile**, not necessarily to find the **absolute theoretical hardware limit**. If `torch.compile` improves in future PyTorch versions, the reward landscape shifts, and the model may require retraining. More critically, if TVM or other compilers can produce kernels 3× faster than `torch.compile` on certain operators, CUDA Agent would not discover these optimizations because its reward signal saturates at beating `torch.compile`.

**Open Question:** Would CUDA Agent still outperform production-grade compilers like TVM on complex operators? The paper does not answer this, leaving uncertainty about whether the approach truly advances the state of the art or simply exceeds a convenient baseline.

#### Proprietary Model Evaluation Gaps

Footnote 1 on page 2 reveals a significant evaluation gap:

> "We found that models in the ChatGPT-5 series (5, 5.1, and 5.2) were not amenable to evaluation on this task, as they consistently declined to respond to CUDA-related prompts."

**Implication:** The strongest proprietary models available at the time of writing could not be tested due to safety filtering. This creates two problems:

1. **Incomplete Comparison:** CUDA Agent's advantage over "state-of-the-art proprietary models" excludes potentially stronger systems that refused to engage with the task.

2. **Safety Policy Dependency:** If future models relax CUDA-related restrictions, the competitive landscape could shift. The paper's claims are contingent on current safety policies, not fundamental capability differences.

**Fair Assessment:** This is not the authors' fault—they cannot force proprietary APIs to respond. However, readers should interpret the "outperforms proprietary models" claim with the understanding that the strongest models were not evaluated.

#### Minor Discrepancy in Level 3 Results

The Abstract claims:

> "CUDA Agent achieves speedups of 100%, 100%, and 92% over torch.compile on the Level-1, Level-2, and Level-3 splits of KernelBench"

However, Table 1 shows:

| Level | Abstract Claim | Table 1 Value |
|-------|---------------|---------------|
| Level 1 | 100% | 97.0% (Faster Rate vs. Compile) |
| Level 2 | 100% | 100.0% |
| Level 3 | 92% | 90.0% |

**Assessment:** This is a minor inconsistency (2 percentage points on Level 3), possibly due to rounding or different evaluation runs. However, it highlights that **exact numbers should be verified against Table 1**, not the Abstract. For a paper emphasizing precise measurement, such discrepancies warrant attention.

### 6.3 Generalizability Concerns

#### PyTorch/CUDA Specificity

The entire system is built around PyTorch and NVIDIA CUDA:

- Data synthesis crawls `torch` and `transformers` libraries (Section 3.1)
- Agent tools assume CUDA compilation workflow (Section 3.2)
- Reward signals depend on PyTorch eager and compile modes (Section 3.2)
- GPU sandbox uses NVIDIA H20 GPUs (Section 4.1)

**Implication:** Applying CUDA Agent to other hardware platforms would require **rebuilding the entire infrastructure**:

| Component | PyTorch/CUDA | TPU/NPX/Other | Effort Required |
|-----------|--------------|---------------|-----------------|
| Data Synthesis | PyTorch operators | Platform-specific operators | High |
| Agent Tools | CUDA compilation | Platform-specific toolchain | High |
| Profiling | NVIDIA profiling tools | Platform-specific profilers | High |
| Reward Function | torch.compile baseline | Platform-specific baseline | Medium |

**Open Question:** Can the conceptual insights (combinatorial data, discrete rewards, precision-aware warm-up) transfer to other hardware without the infrastructure cost? The paper does not address this, leaving uncertainty about whether CUDA Agent is a **general framework for hardware-aware code generation** or a **CUDA-specific solution**.

#### Operator Coverage Bias

Table 3 in Appendix A shows the training data composition:

| Operator Category | Proportion |
|-------------------|------------|
| torch operators ×2 | 83.77% |
| torch operators ×3 | 7.62% |
| torch operators ×4 | 2.80% |
| torch operators ×5 | 1.23% |
| torch operators ×1 | 3.40% |
| transformers operator | 1.18% |

**Observation:** 83.77% of training samples are 2-operator compositions. Only 4.03% are 4+ operator compositions.

**Implication:** The model may be biased toward optimizing **moderately complex fused kernels** rather than extremely complex operators or entire computation graphs. Level 3 tasks (ResNet blocks) show lower speed-ups (3.59× in Case D.4) compared to Level 1 (73.31× in Case D.2), partly because complex workloads have less optimization headroom—but also because the training data underrepresents such complexity.

**Open Question:** Would performance on Level 3 improve with more complex training samples? The paper does not ablate training data composition, leaving this unexplored.

### 6.4 Training Stability Dependencies

#### Warm-up Stages Are Non-Optional

Section 4.3.3 and Figures 4-5 demonstrate that removing RFT or Value Pretraining causes **training collapse**, not just degradation:

- **Without RFT:** Training reward collapses around step 20 (Figure 4a), actor entropy spikes (Figure 4b)
- **Without Value Pretraining:** Critic explained variance remains near zero (Figure 5a), trajectory length explodes (Figure 5b)

**Implication:** These are not optional improvements—they are **prerequisites for training**. This adds complexity to the training pipeline:

1. Single-turn RL warm-up
2. Trajectory collection
3. Rejection Fine-Tuning (actor initialization)
4. Value Pretraining (critic initialization)
5. Full agentic RL

**Trade-off:** The multi-stage pipeline improves stability but increases engineering complexity and compute cost. A research group attempting to replicate this work must implement all five stages correctly; skipping any causes failure.

#### Precision Mismatch Sensitivity

Section 3.3 identifies a specific failure mode:

> "When training and inference engines use different numerical precisions (e.g., BF16 vs FP16), those low-probability tokens result in large importance sampling ratio variance because small numerical errors in computing token probabilities... cause the importance ratio to fluctuate wildly or explode."

**Implication:** The system is sensitive to **numerical precision configuration**. If the training infrastructure changes (e.g., switching from BF16 to FP16), the warm-up strategy may need adjustment. This creates a **fragility** in the training pipeline that is not typical of standard PPO implementations.

**Open Question:** Is this precision sensitivity specific to CUDA code generation (where tokens are low-probability), or does it affect all specialized code domains? The paper does not generalize this finding, leaving uncertainty about broader applicability.

### 6.5 Optimization Scope Limitations

#### Learns to Beat the Compiler, Not the Hardware Limit

The reward function incentivizes beating `torch.compile`, not achieving absolute optimal performance:

$$
r = 3 \text{ if } b(t, t_{\text{eager}}) \land b(t, t_{\text{compile}})
$$

**Implication:** Once the model achieves >5% speedup over `torch.compile`, the reward saturates at 3. There is no additional incentive to push from 2× to 3× speedup if both beat the baseline.

**Evidence:** Section 3.2 states the iteration requirement:

> "MINIMUM: Must be at least 5% faster than torch.compile baseline"
> "GOAL: Achieve the best possible performance - every microsecond counts!"

However, the reward function does not differentiate between 5% and 50% speedup—both receive $r=3$.

**Trade-off:** This design choice stabilizes training (by reducing reward variance) but may **cap optimization quality**. The model learns to clear the threshold, not necessarily to maximize performance.

**Open Question:** Would a continuous reward component (e.g., bonus for exceeding threshold by larger margins) improve final performance, or would it reintroduce the noise problems the discrete reward was designed to solve?

#### No Cross-Kernel Optimization

CUDA Agent optimizes kernels **individually**, not as part of a larger computation graph. Each training problem is a single operator or fused operator sequence (Section 3.1).

**Implication:** The system does not learn **inter-kernel optimizations** such as:

- Kernel fusion across operator boundaries in a full model
- Memory layout optimization across multiple kernels
- Shared memory reuse between consecutive kernels

**Why This Matters:** Real-world deep learning workloads involve hundreds of kernels executing in sequence. Optimizing individual kernels in isolation may miss system-level optimization opportunities.

**Open Question:** Can the agentic RL approach scale to full-model optimization, or does the complexity explode beyond tractable training?

### 6.6 Summary of Limitations

| Category | Limitation | Severity | Mitigation |
|----------|------------|----------|------------|
| **Infrastructure** | 128 GPU requirement | High | Resource-efficient training strategies (future work) |
| **Baseline** | No TVM comparison | Medium | Acknowledged in Appendix E |
| **Evaluation** | ChatGPT-5 unavailable | Medium | Outside authors' control |
| **Generalizability** | PyTorch/CUDA specific | High | Requires rebuilding for other platforms |
| **Training** | Multi-stage warm-up required | High | Non-optional for stability |
| **Reward Design** | Saturates at threshold | Medium | Trade-off for stability |
| **Scope** | Individual kernel optimization | Medium | Does not address full-model optimization |

**Fair Assessment:** CUDA Agent is a **proof of scalability** for agentic RL in systems programming, not a turnkey solution. The core insights—combinatorial data synthesis, discrete milestone rewards, and precision-aware warm-up—are transferable concepts. However, the infrastructure cost, platform specificity, and training complexity create significant barriers to adoption.

The authors are transparent about these limitations in Appendix E, which strengthens the paper's credibility. Future work should address:

1. **Resource efficiency:** Can similar performance be achieved with fewer GPUs or shorter training?
2. **Cross-platform generalization:** Can the framework adapt to TPUs, NPUs, or non-NVIDIA GPUs?
3. **Stronger baselines:** How does CUDA Agent compare to TVM or MLIR-based compilers?
4. **Full-model optimization:** Can the approach scale beyond individual kernels to entire computation graphs?

Until these questions are answered, CUDA Agent remains a compelling demonstration of what is possible with sufficient resources—not a universally accessible tool for CUDA optimization.

## 7. Implications and Future Directions

CUDA Agent represents more than a performance benchmark; it signals a structural shift in how we apply foundation models to systems programming. By demonstrating that reinforcement learning (RL) can teach hardware-aware optimization strategies rather than just syntactic code generation, this work opens new avenues for automating performance-critical software development. This section explores how this contribution changes the research landscape, identifies concrete follow-up directions, outlines practical use cases, and provides guidance for researchers aiming to reproduce or integrate these techniques.

### 7.1 Shifting the Paradigm: From Code Generation to Systems Optimization

The most significant implication of CUDA Agent is the redefinition of the LLM's role in software engineering. Prior agentic systems treated code generation as a **completion task**—the model predicts the next token until a solution is formed. CUDA Agent reframes this as an **optimization task**—the model learns a policy over actions (profile, edit, compile, test) to navigate a hardware constraint space.

This shift has three downstream consequences for the field:

*   **Learning Optimization Strategies:** The model does not merely memorize kernel patterns; it learns *when* to fuse operators, *how* to manage shared memory, and *where* to apply algebraic simplifications. As shown in the Level 2 ablation (Table 2), removing the agent loop collapses the Faster Rate from 96.8% to 14.1%, proving that the **iterative process itself is the learned capability**. Future systems should prioritize training models to manage development workflows, not just write code.
*   **Hardware as a Reward Signal:** By successfully using execution latency as a reward signal (Equation 1, Section 3.2), this work validates that **physical metrics can drive RL training** despite noise. This encourages similar approaches for other hardware domains (e.g., optimizing SQL queries for latency, compiling for energy efficiency, or tuning FPGA logic).
*   **Stability as a First-Class Constraint:** The identification of precision-induced RL collapse (Section 3.3) highlights that training stability in code domains is not just about hyperparameters—it is about **numerical compatibility between training and inference engines**. This insight extends beyond CUDA to any domain where target tokens are low-probability (e.g., formal verification, specialized DSLs).

### 7.2 Promising Research Directions

The limitations identified in Section 6 directly map to high-value research opportunities. Addressing these constraints could democratize access to agentic RL for systems programming.

#### 7.2.1 Resource-Efficient Training Strategies
The current pipeline requires 128 NVIDIA H20 GPUs (Section 4.1), creating a barrier to entry. Future work should investigate:
*   **Trajectory Reuse:** Can agent trajectories collected on one problem set be reused to initialize policies for new operators, reducing the need for fresh rollouts?
*   **Sparse Rewards:** The current reward function (Equation 1) is dense enough to guide learning but discrete. Research into **curriculum learning** that starts with easier operators (Level 1) and progressively introduces complexity (Level 3) might reduce the number of required RL steps from 150 to fewer steps.
*   **Critic Distillation:** Since Value Pretraining is critical for stability (Figure 5), can a smaller critic model be distilled from the large critic to reduce inference overhead during training?

#### 7.2.2 Cross-Platform Generalization
CUDA Agent is currently specific to PyTorch and NVIDIA GPUs (Appendix E). Extending this framework requires:
*   **Unified Agent Interfaces:** Developing a hardware-abstracted agent loop where tools like `profile.py` and `verify.py` are swapped based on the target backend (e.g., replacing `torch.compile` with TVM or MLIR-based compilers).
*   **Transfer Learning:** Investigating whether optimization strategies learned on CUDA (e.g., memory coalescing, tiling) transfer to other architectures like TPUs or NPUs, where memory hierarchies differ but optimization principles remain similar.
*   **Data Synthesis Portability:** The combinatorial data synthesis pipeline (Figure 1) is framework-agnostic. Applying this to generate training tasks for other compilers could bootstrap agents without manual data collection.

#### 7.2.3 Stronger Baselines and Full-Model Optimization
The current evaluation anchors performance to `torch.compile` (Section 5.1). Future work should:
*   **Integrate Production Compilers:** Incorporating TVM or MLIR into the sandbox environment would test whether CUDA Agent can exceed **production-grade optimization ceilings**, not just default compiler flags.
*   **Graph-Level Optimization:** Currently, the agent optimizes individual kernels or fused operator sequences (Section 3.1). Extending the agent to optimize **entire computation graphs** (e.g., full ResNet or Transformer blocks) would capture inter-kernel optimization opportunities like memory layout persistence across layers.
*   **Dynamic Batch Sizes:** The current evaluation uses fixed input shapes. Training agents to generate kernels that perform well across a **distribution of input sizes** would increase practical utility for variable-length workloads (e.g., NLP inference).

#### 7.2.4 Generalizing RL Stability Mechanisms
The precision-aware warm-up strategy (Section 3.3) addresses a fundamental instability in code RL. Follow-up research should:
*   **Characterize Precision Floors:** Systematically measure the token probability thresholds where importance sampling ratios $\rho_t(\theta)$ become unstable across different model architectures (MoE vs. Dense) and precisions (BF16 vs. FP16).
*   **Automated Warm-up:** Develop automated diagnostics that detect when a policy is approaching the precision floor and trigger RFT or Value Pretraining dynamically, rather than using a fixed pipeline.

### 7.3 Practical Applications and Downstream Use Cases

While the infrastructure cost is high, the performance gains justify deployment in specific high-value scenarios.

#### 7.3.1 Automated Kernel Library Generation
Cloud providers and hardware vendors maintain large libraries of optimized kernels (e.g., cuDNN, oneDNN). CUDA Agent offers a pathway to:
*   **Rapid Porting:** Automatically generate optimized kernels for new hardware architectures by adapting the agent's toolset to the new backend.
*   **Custom Operator Support:** When users define custom operators (e.g., novel attention mechanisms), CUDA Agent can generate optimized implementations faster than manual engineering, reducing time-to-market for new model architectures.

#### 7.3.2 Compiler Assistance and Hybrid Systems
CUDA Agent does not need to replace compilers; it can augment them.
*   **Fallback Optimization:** Integrate CUDA Agent into compilers like `torch.compile` as a fallback when standard fusion passes fail to meet performance targets.
*   **Search Space Expansion:** Use the agent to propose kernel variants that the compiler's heuristic search might miss, then let the compiler handle low-level instruction scheduling.

#### 7.3.3 Education and Tooling for Hardware Engineers
The agent's trajectory data (Section 3.3) contains rich information about optimization strategies.
*   **Training Data for Engineers:** The filtered trajectories from Rejection Fine-Tuning (RFT) can serve as educational examples for junior engineers learning CUDA optimization.
*   **Debugging Assistants:** The agent's ability to diagnose bottlenecks (Section 3.2, `SKILL.md` Step 1) can be extracted into a standalone tool that profiles existing kernels and suggests specific optimizations (e.g., "enable TF32", "fuse residual addition").

### 7.4 Reproduction and Integration Guidance

For researchers and practitioners considering adopting or extending CUDA Agent, the following guidance clarifies when this approach is appropriate and what resources are required.

#### 7.4.1 When to Prefer This Method
CUDA Agent is not a universal solution for all code generation tasks. It is best suited for:
*   **Performance-Critical Kernels:** Use when latency reduction directly impacts cost or user experience (e.g., inference serving, training loops). For non-critical code, `torch.compile` or manual implementation is more cost-effective.
*   **High-Volume Operators:** The infrastructure cost is justified when the optimized kernel will be executed millions of times. Optimizing a rarely-used operator does not repay the training compute debt.
*   **Complex Fusion Scenarios:** As shown in Level 2 results (Table 1), the agent excels at operator fusion where compilers struggle. Prefer this method for sequences of operations that standard compilers fail to fuse efficiently.

#### 7.4.2 Infrastructure Requirements
Reproducing this work requires significant investment. Based on Section 4.1 and Appendix E:
*   **GPU Cluster:** A minimum of 128 GPUs (e.g., NVIDIA H20) is required for the sandbox environment to ensure process-level isolation and stable latency measurements. Smaller clusters may suffer from measurement noise that destabilizes the reward signal.
*   **Storage:** The 131,072 token context window (Section 4.1) requires high-memory GPUs for inference during rollouts.
*   **Engineering Overhead:** Building the CPU-GPU decoupled sandbox (Section 3.2) requires substantial systems engineering to prevent reward hacking and ensure safety.

#### 7.4.3 Integration Checklist
If integrating CUDA Agent techniques into an existing pipeline:
1.  **Start with Data Synthesis:** The combinatorial data pipeline (Section 3.1) is the lowest-cost component to adopt. Generate fused operator tasks to augment existing training data.
2.  **Implement Discrete Rewards:** Replace continuous speedup rewards with the milestone-based reward (Equation 1) to reduce sensitivity to measurement noise.
3.  **Adopt Warm-up Stages:** Do not skip Rejection Fine-Tuning (RFT) or Value Pretraining (Section 3.3). These are prerequisites for stability, not optional improvements.
4.  **Validate Decontamination:** Use AST-based similarity checks (Appendix A) to ensure training data does not overlap with evaluation benchmarks, preventing inflated performance claims.

#### 7.4.4 Expected Outcomes
Based on Table 1, practitioners should expect:
*   **Speed-up:** 1.5× to 2.8× over `torch.compile` depending on task complexity (Level 1–3).
*   **Correctness:** >98% Pass Rate if the agent loop and reward design are preserved.
*   **Training Time:** 150 RL steps with batch size 1,024 (Section 4.1). Expect significant wall-clock time even with parallelism.

### 7.5 Summary of Impact

CUDA Agent demonstrates that **agentic reinforcement learning can bridge the gap between general-purpose language models and specialized hardware optimization**. By systematically addressing data scarcity, reward noise, and training instability, this work provides a blueprint for automating systems programming tasks that were previously reserved for human experts.

The path forward involves reducing the computational barrier, expanding to diverse hardware platforms, and integrating these agents into broader compiler ecosystems. While the current infrastructure cost is prohibitive for most, the conceptual innovations—combinatorial data synthesis, discrete milestone rewards, and precision-aware warm-up—are immediately applicable to lower-resource settings. As hardware complexity continues to outpace manual optimization capacity, systems like CUDA Agent will transition from research curiosities to essential infrastructure for deep learning deployment.
