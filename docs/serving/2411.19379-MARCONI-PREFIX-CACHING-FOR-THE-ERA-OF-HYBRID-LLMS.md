## 1. Executive Summary
This paper introduces **Marconi**, the first prefix caching system designed specifically for **Hybrid LLMs** (models combining Attention and State Space Model layers like **Mamba** or **Jamba**), which previously suffered from severe cache thrashing due to the "all-or-nothing" nature and large fixed size of recurrent SSM states. By replacing standard recency-based policies with novel **judicious admission** (using speculative insertion on a radix tree) and **FLOP-aware eviction** strategies, Marconi optimizes the trade-off between memory footprint and compute savings. The system achieves up to **34.4× higher token hit rates** and reduces P95 Time-To-First-Token (TTFT) by **71.1% (617 ms)** compared to extended state-of-the-art baselines like **vLLM** and **SGLang** across diverse workloads including **LMSys**, **ShareGPT**, and **SWE-Bench**.

## 2. Context and Motivation

### The Shift to Hybrid Architectures
The demand for Large Language Models (LLMs) to process increasingly long contexts—driven by applications like complex reasoning, detailed document analysis, and multi-turn coding assistants—has exposed a fundamental bottleneck in traditional Transformer architectures. In standard Transformers, the **Attention mechanism** computes relationships between every token in a sequence and every other token. This results in **quadratic computational complexity** ($O(L^2)$) and a **linear memory footprint** ($O(L)$) for the Key-Value (KV) cache, where $L$ is the sequence length. As context windows expand to millions of tokens (e.g., Gemini 1.5, Claude 3), the cost of storing and computing over these KV states becomes prohibitive.

To address this, the field has shifted toward **Hybrid LLMs**. These models interleave standard Attention layers with **State Space Models (SSMs)** (e.g., Mamba, Jamba).
*   **Attention Layers:** Provide high recall and strong in-context learning capabilities but are computationally expensive for long sequences.
*   **SSM Layers:** Compress the entire history of tokens into a fixed-size recurrent state. This reduces computational complexity to **linear** ($O(L)$) and, crucially, keeps the memory footprint for the state **constant** ($O(1)$) regardless of sequence length.

Typical Hybrid architectures (Figure 1a) might include one Attention layer for every 6 to 10 SSM layers. While this design drastically improves *per-request* efficiency, it introduces a unique and severe challenge for *cross-request* optimization techniques, specifically **prefix caching**.

### The Specific Problem: The "All-or-Nothing" Cache Dilemma
**Prefix caching** is a standard optimization in LLM serving where the system caches the internal model states (KV cache for Attention, recurrent states for SSMs) of common prefixes shared across different requests. For example, if multiple users start a chat with the same system prompt, the system computes the states for that prompt once, caches them, and reuses them for subsequent requests, skipping redundant computation.

In pure Transformer models, this works seamlessly because KV states can be sliced. If a sequence of 1,000 tokens is cached, the system can easily extract and reuse the states for the first 500 tokens for a new request.

However, in Hybrid models, the properties of SSM states break this mechanism:
1.  **In-Place Updates:** SSMs update their state recurrently. The state at token $t$ overwrites the state at token $t-1$. There is no inherent way to "roll back" the state to represent an earlier prefix without recomputing it from scratch (Figure 1b).
2.  **Exact Match Requirement:** Because states cannot be rolled back, a cache hit for an SSM layer requires an **exact match** of the entire prefix sequence. You cannot reuse the state of a 1,000-token sequence for a request that only needs the first 500 tokens.
3.  **Massive State Size:** While an SSM state is constant in size relative to sequence length, it is orders of magnitude larger than the KV state of a *single* token. For a typical 7B Hybrid model, an SSM state block can be **10–100× larger** than the KV block of a single token.

This creates a **dilemma** for system designers (Section 3):
*   **Option A: Fine-Grained Checkpointing.** To maximize reuse opportunities (allowing hits on arbitrary prefixes), the system must save a snapshot of the SSM state frequently (e.g., every 32 tokens).
    *   *Consequence:* This generates a deluge of massive cache entries. Figure 3b illustrates that for a single 10k token sequence with a block size of 16, the cache size balloons to **17.4 GB**, which is **3.3× larger** than a comparable Transformer. Most of these entries are "sparsely-hit" (used rarely), leading to rapid cache thrashing where useful data is evicted to make room for low-utility snapshots.
*   **Option B: Coarse Checkpointing.** To save memory, the system checkpoints less frequently.
    *   *Consequence:* Reuse opportunities plummet because the probability of a new request matching a long, exact prefix is low. Figure 3a shows that with a block size of 32, while **25.0%** of KV blocks are reused, only **0.4%** of SSM states are reused—a **65.3× disparity**.

The core gap this paper addresses is that **existing prefix caching systems (like vLLM and SGLang) are fundamentally incompatible with the structural constraints of Hybrid LLMs.** They treat all layers uniformly, leading to either memory exhaustion or negligible compute savings when applied naively to Hybrid models.

### Limitations of Prior Approaches
Prior work in LLM serving optimization falls short in this specific domain for three key reasons:

1.  **Assumption of Sliceable States:** Systems like **vLLM** (PagedAttention) and **SGLang** (RadixAttention) rely on the ability to manage memory in fixed-size blocks that can be arbitrarily composed. They assume that if you have the cache for tokens $1 \dots N$, you automatically have the cache for $1 \dots M$ (where $M < N$). This holds for Attention KVs but fails completely for SSM states, which are monolithic and non-sliceable.
2.  **Recency-Only Eviction Policies:** Standard caching policies (e.g., LRU - Least Recently Used) evict entries based solely on how recently they were accessed. In Hybrid models, this is suboptimal because it ignores the **compute-to-memory ratio**. A short sequence might be accessed recently but offers minimal FLOP savings relative to its large SSM state size, whereas a long sequence accessed slightly less often might save massive amounts of computation. Existing systems lack a metric to distinguish these cases.
3.  **Indiscriminate Admission:** Current systems admit *every* token block into the cache upon completion of a request. Given the massive size of SSM states, admitting every intermediate state (most of which will never be hit due to the exact-match requirement) wastes precious GPU High Bandwidth Memory (HBM) on low-utility data.

### Positioning of Marconi
**Marconi** positions itself not as a replacement for the underlying model architecture, but as the **first system-level solution specifically redesigned for the hybrid nature of modern LLMs**.

Unlike prior works that attempt to force Hybrid models into existing caching frameworks (resulting in the "fine-grained checkpointing" failure mode), Marconi fundamentally changes the caching logic:
*   **From "Cache Everything" to "Judicious Admission":** Instead of blindly caching every block, Marconi uses a **taxonomy of reuse scenarios** (purely input vs. input+output) and a **speculative insertion** mechanism on a radix tree to predict which SSM states are actually likely to be reused. It only admits high-value states, drastically reducing memory pressure.
*   **From "Recency" to "FLOP-Awareness":** Marconi introduces a novel eviction metric called **FLOP efficiency** (Equation 1), defined as the total floating-point operations saved divided by the memory consumed. This allows the system to prioritize caching long sequences (which have high FLOP efficiency in SSMs) over short ones, even if the short ones were accessed more recently.

By acknowledging that SSM states and KV states have different memory/compute trade-offs, Marconi bridges the gap between the theoretical efficiency of Hybrid models and their practical deployment, enabling the long-context capabilities of models like Jamba and Mamba to be realized in production environments without cache thrashing.

## 3. Technical Approach

This section details the architectural design and algorithmic innovations of Marconi, the first prefix caching system engineered specifically for the constraints of Hybrid LLMs. The core idea is to replace the "cache everything, evict by recency" paradigm of traditional systems with a **value-aware strategy** that explicitly models the trade-off between the massive, fixed memory cost of SSM states and the variable compute savings they provide. By treating SSM states and KV states as a unified but distinct entity within a radix tree, Marconi selectively admits only high-probability reuse states and evicts entries based on their "FLOP efficiency" rather than just their age.

### 3.1 Reader orientation (approachable technical breakdown)
Marconi is a memory management system for AI servers that intelligently decides which parts of a neural network's internal "memory" to keep on the GPU and which to discard when serving Hybrid models (models mixing Attention and recurrent layers). It solves the problem of cache thrashing caused by the huge, non-sliceable nature of recurrent states by predicting which states are actually useful before storing them and by prioritizing the retention of states that save the most computation per byte of memory used.

### 3.2 Big-picture architecture (diagram in words)
The Marconi system operates as an intermediary layer between the incoming request stream and the GPU inference engine, orchestrated around a central **Radix Tree Metadata Store**.
*   **Request Parser & Speculative Insertion Module:** This component intercepts incoming prompts before computation begins. It traverses the Radix Tree to identify existing common prefixes and performs a "dry-run" insertion to detect if the new prompt creates a branching point (indicating a shared "purely input" prefix).
*   **Judicious Admission Controller:** Based on the output of the speculative insertion, this logic gate decides whether to materialize and store the SSM states for specific checkpoints. It admits states only if they represent a branching point (shared system prompt) or the final token of a sequence (conversation history), rejecting intermediate states that are unlikely to be reused.
*   **Hybrid State Manager:** This component handles the physical storage of model states. Unlike prior systems that store KV and SSM states separately, it binds them together at each node of the Radix Tree, ensuring that a cache hit retrieves both the Attention KVs and the exact matching SSM state required for valid inference.
*   **FLOP-Aware Eviction Engine:** When the cache reaches capacity, this module calculates a utility score for every node in the tree. It combines recency with a novel "FLOP efficiency" metric to determine which nodes to evict, preferentially removing short sequences with low compute savings to make room for long, high-value sequences.

### 3.3 Roadmap for the deep dive
To fully understand how Marconi overcomes the limitations of prior work, we will dissect its mechanisms in the following logical order:
1.  **The Data Structure Foundation:** We first explain how Marconi adapts the **Radix Tree** to jointly manage non-sliceable SSM states and sliceable KV states, establishing the structural basis for all subsequent decisions.
2.  **Judicious Admission via Speculative Insertion:** We detail the "filter" mechanism that prevents cache pollution by explaining how the system predicts reuse likelihood *before* performing expensive state checkpointing.
3.  **State Materialization Mechanics:** We describe the specific computational techniques (chunked passing vs. two-pass prefill) used to extract the exact SSM states required for caching, addressing the "in-place update" constraint.
4.  **FLOP-Aware Eviction Policy:** We derive the mathematical formulation of **FLOP efficiency**, explaining why this metric is critical for Hybrid models and how it replaces standard Least Recently Used (LRU) policies.
5.  **Dynamic Hyperparameter Tuning:** We conclude with the runtime mechanism Marconi uses to automatically balance recency and compute savings, ensuring optimal performance across varying workloads without manual configuration.

### 3.4 Detailed, sentence-based technical breakdown

#### The Unified Radix Tree Structure
Marconi utilizes a **radix tree** (a space-efficient prefix tree where edges represent sequences of tokens) as its central metadata structure, but it fundamentally alters what each node represents compared to prior systems like SGLang.
*   In traditional systems, nodes map strictly to KV blocks; in Marconi, each edge in the tree is associated with both the **KV tensors** for the tokens it represents and the **SSM recurrent states** that represent the cumulative history of all tokens prior to the last token on that edge.
*   This coupling is mandatory because a valid cache hit in a Hybrid model requires *both* components: the Attention layers need the specific KV slice, and the SSM layers need the exact state corresponding to the end of that prefix.
*   Nodes in the tree represent **branch points**; a node with multiple children indicates a prefix shared by multiple distinct requests (e.g., a system prompt), while a node with a single child typically represents a linear continuation of a conversation.
*   Marconi enforces a strict invariant: if an SSM state is cached at a node, the system guarantees that all preceding states required to reach that node are also logically present, either as cached entries or as part of the path from the root.

#### Judicious Admission via Speculative Insertion
The first major innovation is the **Judicious Admission** policy, which prevents the cache from being flooded with low-utility SSM states by admitting them only when a specific "reuse scenario" is detected.
*   Marconi classifies prefix reuse into two categories: **"Purely Input"** (e.g., system prompts shared across many users) and **"Input + Output"** (e.g., conversation history where a user resumes a chat).
*   To detect "Purely Input" opportunities, Marconi performs a **speculative insertion** step *before* the actual prefill computation begins.
*   During speculative insertion, the system simulates traversing the radix tree with the incoming input tokens to determine if inserting this sequence would create a new **intermediate node** (a branch point).
*   If the simulation reveals that the incoming sequence branches off an existing path (meaning this prefix has been seen before in a different context), Marconi flags this location as a high-value candidate for caching.
*   Conversely, if the sequence simply extends an existing leaf without branching, it is treated as a unique continuation, and intermediate SSM states along this path are **not** admitted, avoiding the storage of sparsely-hit data.
*   For "Input + Output" scenarios, the policy is simpler: Marconi unconditionally admits the SSM state corresponding to the **last decoded token** of any completed sequence, as this is the specific state required to resume a conversation.
*   This approach results in a maximum of **two SSM state checkpoints per sequence**: one at the identified branch point (if it exists) and one at the very end.
*   By rejecting all intermediate checkpoints between the branch point and the end of the sequence, Marconi reduces the number of cached SSM states by orders of magnitude compared to fine-grained checkpointing, directly addressing the memory explosion shown in Figure 3b.

#### State Materialization Mechanics
Once the admission controller decides to cache a state, Marconi must overcome the technical hurdle that SSM states are updated **in-place**, meaning the state at token $t$ overwrites the state at $t-1$, making it impossible to simply "slice" out a past state.
*   Marconi employs two distinct strategies to materialize the required SSM states depending on the underlying model architecture's support for **chunked state passing**.
*   **Strategy A (Chunked Passing):** For models that support processing inputs in chunks (e.g., certain Mamba variants), Marconi leverages the intra-chunk state computation.
    *   If the system needs to cache the state at token $T$, and the model processes chunks of size $C$, Marconi materializes and stores the state at the end of the chunk immediately preceding $T$ (e.g., at token $T-C$).
    *   While this may miss the exact state at token $T$ by a few tokens, the paper notes this introduces minimal runtime overhead and captures the vast majority of the compute savings.
    *   Optionally, custom kernels can be invoked to "roll forward" the state from the chunk boundary to the exact token location, though this is an optimization rather than a requirement.
*   **Strategy B (Two-Pass Prefill):** For models that do not support chunked state passing and strictly update states in-place, Marconi executes a **two-pass prefill**.
    *   In the first pass, the system runs the prefill computation *only* up to the target prefix length (e.g., the first 80 tokens) to generate the precise SSM state at that boundary.
    *   This state is immediately captured and stored in the cache.
    *   In the second pass, the system resumes computation from the cached state to process the remaining tokens (e.g., tokens 81 to 100) and generate the final output.
    *   Although this doubles the computation for that specific request, it is a one-time cost incurred only when a high-value prefix is identified, paying off in all future requests that hit this cache entry.

#### FLOP-Aware Eviction Policy
The second major innovation is the **FLOP-Aware Eviction** policy, which replaces standard recency-based eviction (like LRU) with a metric that accounts for the unique compute-to-memory ratio of Hybrid models.
*   The core insight is that in Hybrid models, the memory cost of an SSM state is **fixed** regardless of sequence length, but the compute savings (FLOPs) gained by reusing that state scales **linearly** (for SSMs) and **quadratically** (for Attention) with sequence length.
*   To quantify this, Marconi defines a new metric called **FLOP efficiency**, which measures the amount of computation saved per byte of memory consumed by a cache entry.
*   The mathematical definition of FLOP efficiency for a given cache entry is:
    $$ \text{flop efficiency} = \frac{\text{Total FLOPs across layers}}{\text{Memory consumption of all states}} $$
*   Here, the numerator represents the sum of redundant floating-point operations avoided across all layer types (Attention, SSM, and MLP) if the prefix is reused.
*   The denominator represents the total memory footprint of the cached states, including both the KV tensors and the SSM recurrent states.
*   As illustrated in Figure 5, the FLOP efficiency of SSM-heavy models increases steeply with sequence length, meaning long sequences are vastly more "valuable" per byte than short ones.
*   Traditional LRU policies fail here because they might evict a long, highly efficient sequence (accessed slightly less recently) to keep a short, inefficient sequence (accessed very recently), leading to a net loss in system throughput.
*   Marconi calculates a **Utility Score** $S(n)$ for every node $n$ in the radix tree to guide eviction decisions:
    $$ S(n) = \text{recency}(n) + \alpha \cdot \text{flop efficiency}(n) $$
*   In this equation, $\text{recency}(n)$ and $\text{flop efficiency}(n)$ are both normalized to the range $(0, 1)$ based on the current population of the cache.
*   The hyperparameter $\alpha$ controls the trade-off: a higher $\alpha$ prioritizes compute savings (favoring long sequences), while $\alpha = 0$ reverts to pure LRU.
*   During eviction, Marconi iteratively removes the node with the **lowest utility score** until sufficient space is freed for the new incoming request.
*   Crucially, when calculating the FLOP savings for a child node, Marconi considers the *marginal* savings relative to its parent, ensuring that the value of shared prefixes is correctly attributed.
*   This policy inherently trades the hit rate of shorter sequences for that of longer sequences, a desirable outcome given that Hybrid models derive their primary efficiency benefits from long-context processing.

#### Implementation Nuances and Dynamic Tuning
Beyond the core algorithms, Marconi includes specific implementation details and a dynamic tuning mechanism to ensure robustness in production environments.
*   **Eviction Scope:** Unlike standard systems that only evict leaf nodes, Marconi allows the eviction of **intermediate nodes with a single child**.
    *   The reasoning is that a node with only one child represents a linear path that is unlikely to be a shared "branch point" for future diverse requests; thus, its SSM state is low-utility.
    *   When such a node is evicted, its SSM state is released, and its KV data is logically absorbed by its child node to maintain the integrity of the remaining path.
*   **Timestamp Updates:** On a cache hit, Marconi updates the timestamp *only* for the accessed node, not for all its ancestors.
    *   This is valid because the SSM states of ancestors are not reused (due to the exact-match requirement), and the KV data of ancestors is subsumed by the child's data if the child is retained.
*   **Dynamic $\alpha$ Tuning:** Recognizing that the optimal balance between recency and FLOP efficiency depends on the specific workload, Marconi implements an online **grid search** to tune $\alpha$.
    *   Upon startup, the system operates in LRU mode ($\alpha = 0$) until the first eviction occurs.
    *   It then enters a **bootstrap period**, collecting a sample of $5\times$ to $15\times$ the number of requests seen before the first eviction.
    *   Using this snapshot of the workload, Marconi asynchronously replays the request trace on CPU cores, testing a range of $\alpha$ values to find the one that maximizes the token hit rate.
    *   This tuning process typically completes in seconds (faster than a single request's prefill time), allowing the system to adapt to changing traffic patterns without manual intervention.
*   **Handling Model Variants:** The system is designed to be agnostic to the specific ratio of Attention to SSM layers.
    *   As the ratio of SSM layers increases (e.g., from 1:2 to 1:8), the memory pressure from SSM states grows, making the judicious admission and FLOP-aware eviction increasingly critical, as demonstrated in the microbenchmarks in Figure 12a.
    *   Similarly, as the state dimension $N$ of the SSM increases (a trend in newer models for better accuracy), the fixed cost per state rises, further amplifying the benefits of Marconi's selective caching over naive approaches.

By integrating these components, Marconi transforms the prefix caching problem from a simple memory management task into a nuanced optimization problem that respects the distinct physical properties of Hybrid LLM layers. The system ensures that every byte of GPU HBM allocated to the cache contributes maximally to reducing inference latency, effectively unlocking the theoretical efficiency of Hybrid architectures for real-world serving.

## 4. Key Insights and Innovations

Marconi's contributions extend beyond simple engineering optimizations; they represent a fundamental rethinking of how caching systems must evolve to accommodate non-Transformer architectures. The following insights distinguish Marconi from prior work, moving from incremental adjustments to paradigm shifts in system design.

### 4.1 The Decoupling of Memory Cost and Compute Value
**Innovation Type:** Fundamental Theoretical Advance

Prior prefix caching systems (e.g., vLLM, SGLang) operate on an implicit assumption inherited from Transformer architectures: **memory cost scales linearly with compute savings.** In a standard Attention model, caching a prefix of length $L$ requires $O(L)$ memory (for KVs) and saves $O(L^2)$ computation. Because both metrics grow with sequence length, a "recency-based" eviction policy (like LRU) serves as a reasonable heuristic; recently used long sequences are naturally valuable because they occupy space proportional to their utility.

Marconi identifies and exploits a critical break in this correlation within Hybrid models. As detailed in **Section 4.2** and **Figure 5**, SSM states have a **fixed memory footprint** ($O(1)$) regardless of the sequence length they represent, yet the compute savings they offer still scales linearly ($O(L)$) with the number of tokens skipped.
*   **Why this is novel:** This creates a scenario where a cache entry representing 10,000 tokens consumes the *same* memory as one representing 10 tokens, but delivers **1,000× more compute savings**.
*   **Significance:** This insight renders traditional size-aware or recency-only eviction policies mathematically suboptimal. It necessitates the introduction of **FLOP efficiency** (Equation 1) as a first-class citizen in cache management. By quantifying value as $\text{FLOPs} / \text{Bytes}$, Marconi reveals that short sequences are essentially "memory bloat" in Hybrid models, while long sequences are "high-density compute savings." This theoretical shift allows Marconi to intentionally sacrifice hit rates on short sequences to drastically boost throughput on long ones, a trade-off that would be catastrophic for pure Transformers but is optimal for Hybrid architectures.

### 4.2 Predictive Admission via Speculative Insertion
**Innovation Type:** New Capability (Proactive vs. Reactive)

Existing systems employ a **reactive admission strategy**: they compute the full sequence, generate states for every token block, and *then* admit them into the cache. In the context of Hybrid models, this approach is fatal because the "cost of being wrong" (admitting a low-utility SSM state) is extremely high in terms of memory pressure (Section 3).

Marconi introduces **Speculative Insertion**, a **proactive admission mechanism** that determines the value of a cache entry *before* any heavy state materialization occurs.
*   **How it differs:** Instead of blindly checkpointing every block, Marconi performs a lightweight "dry-run" traversal of the radix tree using the incoming prompt tokens. It checks if the new sequence creates a **branch point** (indicating a shared "purely input" prefix like a system prompt) or if it represents the end of a conversation turn.
*   **Significance:** This acts as a high-precision filter. If the speculative insertion shows the sequence is merely a unique extension of an existing path with no branching, Marconi **rejects** all intermediate SSM states, admitting only the final state. This prevents the "deluge" of sparsely-hit entries described in **Figure 2** and **Figure 3**.
*   **Impact:** This shifts the system from a "store everything, evict later" mindset to a "store only what matters" philosophy. It effectively solves the **cache thrashing** problem at the source rather than trying to mitigate it via eviction. The result is a cache population composed almost entirely of high-utility entries, enabling the **34.4× higher token hit rates** observed in **Figure 7** compared to baselines that pollute the cache with low-value data.

### 4.3 Holistic State Coupling in the Radix Tree
**Innovation Type:** Architectural Redesign

Prior systems like SGLang utilize radix trees to manage KV caches, treating memory as a composable resource where any prefix can be sliced and diced. This works because Attention states are **sliceable**. However, applying this same structure to Hybrid models without modification fails because SSM states are **monolithic** and **non-sliceable** (Section 3).

Marconi's innovation lies in **holistically coupling** disparate state types within a single logical unit.
*   **The Design Change:** In Marconi, a node in the radix tree does not just point to a memory address for KVs; it represents a **synchronization point** where *both* the exact KV slice for Attention layers AND the exact recurrent state for SSM layers must co-exist to be valid.
*   **Why this matters:** This enforces a strict **atomicity constraint** on cache hits. You cannot have a "partial hit" where you reuse KVs but recompute SSM states, because the SSM recurrence depends on the exact history up to that point.
*   **Significance:** This design choice forces the system to respect the "all-or-nothing" nature of Hybrid inference. It prevents the subtle correctness bugs that would arise from naively mixing sliceable and non-sliceable states. Furthermore, it enables the **eviction of intermediate single-child nodes** (Section 4.3), a maneuver impossible in standard systems. By recognizing that a linear chain of nodes offers no branching utility for SSMs, Marconi can collapse or evict these segments to free up massive amounts of memory, a capability unique to its coupled architecture.

### 4.4 Workload-Adaptive Hyperparameter Tuning
**Innovation Type:** Practical System Robustness

While the concept of weighting eviction by value (like Greedy-Dual-Size) exists in general computer science, applying it to LLM serving introduces a new variable: the optimal balance between **recency** and **FLOP efficiency** ($\alpha$ in Equation 2) is highly dependent on the specific workload's sequence length distribution and arrival patterns. A static policy would fail across diverse scenarios (e.g., chat vs. document processing).

Marconi introduces an **online, asynchronous grid search** to dynamically tune this balance.
*   **Mechanism:** Upon detecting cache contention, the system snapshots the current radix tree and replays a bootstrap sample of recent requests across multiple CPU cores, testing various $\alpha$ values in parallel to find the one that maximizes hit rate.
*   **Differentiation:** Unlike offline profiling or manual tuning required by other research prototypes, this process happens **at runtime** and completes in seconds—faster than the time it takes to prefill a single long request.
*   **Significance:** This ensures Marconi remains optimal even as workload characteristics shift (e.g., a sudden influx of long-context coding agents vs. short chat queries). It transforms the FLOP-aware policy from a theoretical construct into a self-healing system component, explaining why Marconi maintains consistent wins across the diverse datasets of **LMSys**, **ShareGPT**, and **SWE-Bench** (Figure 8) without manual intervention.

### Summary of Impact
These innovations collectively address the root cause of Hybrid LLM serving inefficiencies. While prior work attempted to force Hybrid models into Transformer-shaped caching boxes (leading to the memory explosions in **Figure 3b**), Marconi redesigns the box itself. By decoupling memory from compute value, predicting admission needs, coupling state types atomically, and adapting to workload dynamics, Marconi achieves what was previously thought impossible: efficient, scalable prefix caching for recurrent-transformer hybrids. The result is not just an incremental speedup, but the enabling of **long-context Hybrid models** (like Jamba-1.5) in production environments where they would otherwise be prohibitively expensive to serve.

## 5. Experimental Analysis

This section dissects the empirical evaluation of Marconi, moving beyond the headline metrics to understand *how* and *why* the system achieves its gains. The authors construct a rigorous experimental framework designed to stress-test the specific hypotheses regarding SSM state management: that judicious admission prevents thrashing and that FLOP-aware eviction optimizes for long-context efficiency.

### 5.1 Evaluation Methodology: Constructing a Fair Baseline

A critical challenge in evaluating Marconi is the lack of existing systems that natively support Hybrid LLMs. To ensure a fair comparison, the authors do not compare against vanilla versions of vLLM or SGLang (which would simply crash or produce incorrect outputs on SSM layers). Instead, they construct **extended baselines** by modifying these state-of-the-art systems to handle Hybrid architectures, albeit with their original, suboptimal caching logic.

**Baselines:**
*   **Vanilla Inference:** A control group with no prefix caching, representing the raw compute cost of the model.
*   **vLLM+ (Fine-Grained Checkpointing):** The authors extend vLLM to support Hybrid models by implementing fine-grained checkpointing. They set the token block size to **32**, the maximum supported by vLLM. This baseline represents the "naive" approach: it caches an SSM state for *every* 32-token block. As predicted in Section 3, this floods the cache with massive, low-utility entries.
*   **SGLang+ (Radix Tree + LRU):** The authors extend SGLang to support Hybrid models. Crucially, while SGLang uses a radix tree (like Marconi), it retains its original **LRU (Least Recently Used)** eviction policy and admits states for all tokens. This baseline isolates the impact of Marconi's *eviction* and *admission* policies. If Marconi outperforms SGLang+, the gain is attributable to the novel FLOP-aware logic, not just the data structure.

**Workloads and Datasets:**
The evaluation spans three distinct datasets (Figure 6) to test robustness across different sequence length distributions and reuse patterns:
1.  **LMSys:** A multi-turn conversational dataset with relatively long output sequences (often thousands of tokens).
2.  **ShareGPT:** Another conversational dataset but with succinct outputs (tens to hundreds of tokens), testing performance on shorter contexts.
3.  **SWE-Bench (via SWE-Agent):** An agentic workload where an LLM interacts with a coding environment. This dataset features the widest variance in sequence length, ranging from hundreds to tens of thousands of tokens, providing the ideal stress test for FLOP-aware eviction.

**Metrics:**
*   **Token Hit Rate (%):** The primary metric, defined as the ratio of tokens that skipped prefill computation to the total input tokens. This directly proxies compute savings (FLOPs).
*   **Time-To-First-Token (TTFT):** Measured at P5, P50, and P95 percentiles. This captures the latency benefit perceived by the user.
*   **FLOP Saved:** Used in microbenchmarks to quantify the actual computational work avoided.

**Hardware Setup:**
Experiments were conducted on an AWS `p4d.24xlarge` instance equipped with **eight NVIDIA A100-40GB GPUs**. The primary model used is a **7B Hybrid model** with a layer composition of **4 Attention, 24 SSM, and 28 MLP layers** (a 1:6 ratio). For specific latency benchmarks, the authors also utilize **Jamba-1.5-Mini** (12B active parameters), serving it with a state dimension of 128.

### 5.2 End-to-End Performance: The Cost of Naive Caching

The end-to-end results provide a stark demonstration of why standard caching strategies fail on Hybrid models and how Marconi's specific interventions resolve these failures.

**Marconi vs. vLLM+ (The Admission Win):**
Figure 7 illustrates the catastrophic failure of fine-grained checkpointing (vLLM+). Because vLLM+ admits an SSM state for every block, the cache is rapidly filled with "sparsely-hit" entries (as predicted in Figure 3a).
*   **Token Hit Rate:** Marconi improves the token hit rate by an average of **4.5×** on LMSys, **7.3×** on ShareGPT, and a staggering **34.4×** on SWE-Bench compared to vLLM+.
*   **Latency Impact:** These hit rate gains translate directly to latency reductions. Compared to vanilla inference, Marconi reduces P95 TTFT by up to **71.1% (617.0 ms)** on SWE-Bench. In contrast, vLLM+ achieves minimal gains because its cache is constantly thrashing—evicting useful data to make room for the deluge of low-utility SSM states.
*   **Interpretation:** The massive gap on SWE-Bench (34.4×) confirms that workloads with diverse, long sequences suffer most from indiscriminate admission. Marconi's "judicious admission" acts as a filter, ensuring that the limited GPU HBM is reserved only for prefixes with high reuse potential (branch points and conversation ends).

**Marconi vs. SGLang+ (The Eviction Win):**
Figure 8 isolates the benefit of **FLOP-aware eviction** by comparing Marconi against SGLang+ (which uses the same radix tree but relies on LRU).
*   **Token Hit Rate Win:** Marconi outperforms SGLang+ by **19.0%** on ShareGPT, **45.6%** on LMSys, and **219.7%** on SWE-Bench (P95).
*   **Why the Variance?** The performance gap correlates directly with sequence length variance. ShareGPT consists mostly of short sequences (&lt;2K tokens), where the difference in FLOP efficiency between short and long entries is less pronounced. SWE-Bench, however, contains sequences up to tens of thousands of tokens. In this regime, the FLOP efficiency of long sequences is vastly superior (Figure 5). LRU (SGLang+) blindly evicts these valuable long sequences if they haven't been accessed *very* recently, whereas Marconi retains them because their compute-to-memory ratio is too high to discard.
*   **Latency Distribution:** Figure 9 shows that Marconi's P95 TTFT is significantly lower than SGLang+. On SWE-Bench, Marconi reduces P95 TTFT by **46.8% (617.0 ms)** relative to vanilla, while SGLang+ achieves a smaller reduction. This confirms that optimizing for FLOP efficiency specifically targets the tail latency caused by long-context prefill bottlenecks.

### 5.3 Fine-Grained Analysis: The Strategic Trade-off

Perhaps the most insightful result is found in **Figure 10**, which dissects the behavior of the FLOP-aware eviction policy. This analysis reveals that Marconi's superiority is not universal across *all* sequence lengths; rather, it is the result of a deliberate, optimal trade-off.

**The "Short Sequence Sacrifice":**
Figure 10a plots the difference in hit rate between Marconi and SGLang+ binned by sequence length.
*   **Short Sequences (&lt;7K tokens):** Marconi actually performs *worse* than SGLang+, with a hit rate deficit of up to **-3.0%**.
*   **Long Sequences (>7K tokens):** Marconi dominates, achieving a hit rate improvement of up to **25.5%**.

**Why is this desirable?**
This is not a bug; it is a feature derived from the physics of Hybrid models.
1.  **Compute Cost:** Prefilling a short sequence (e.g., 500 tokens) on a Hybrid model is extremely fast due to the linear complexity of SSM layers. Missing a cache hit here adds negligible latency (Figure 10b shows the P5 TTFT degradation is only **2.1 ms**).
2.  **Memory Opportunity Cost:** The SSM state for that short sequence occupies the *same* fixed memory footprint as a state for a 10,000-token sequence. If the cache keeps the short sequence (due to recency), it displaces the long sequence.
3.  **Net Gain:** Missing a hit on the long sequence would cost hundreds of milliseconds of compute. Therefore, Marconi intentionally evicts short, "recency-heavy" entries to protect long, "FLOP-heavy" entries. The net result is a **90.3% increase in total FLOPs saved** compared to SGLang+, despite the slight dip in short-sequence hit rates.

This analysis convincingly supports the paper's claim that **recency is a flawed proxy for value** in Hybrid serving. The system successfully identifies that saving 10ms on a short prompt is not worth sacrificing 500ms on a long document.

### 5.4 Microbenchmarks and Robustness

The authors conduct extensive ablation studies to verify that Marconi's benefits scale with the trends of modern model development.

**Impact of Cache Contention (Figure 11):**
Marconi's advantages are most pronounced under **moderate cache contention**.
*   **High Contention (60 GB):** Both systems struggle, with hit rates dropping below 10% as the cache is too small to hold meaningful prefixes.
*   **Low Contention (140 GB):** The cache is large enough to hold almost everything, so eviction policy matters less; Marconi's win shrinks to **10.0%**.
*   **Moderate Contention:** This is the "sweet spot" where eviction decisions are critical. Here, Marconi's FLOP-aware policy yields a **51.5% to 68.3%** improvement over SGLang+. This confirms that Marconi is specifically designed for the resource-constrained environments typical of production serving.

**Scaling with Model Architecture (Figure 12):**
As Hybrid models evolve, they tend to include more SSM layers and larger state dimensions to improve accuracy. Marconi's performance *improves* under these conditions, whereas baselines degrade.
*   **SSM Layer Ratio:** As the ratio of SSM layers increases from 1:2 to 1:8, Marconi's token hit rate improvement over vLLM+ grows from **13.5%** to **2.6×**. With more SSM layers, the memory pressure from fixed-size states increases, making the "judicious admission" filter increasingly vital.
*   **State Dimension:** Increasing the SSM state dimension from 16 (Mamba1) to 128 (Mamba2) exacerbates the memory cost per entry. Marconi's win over vLLM+ jumps from **5.7×** to **35.4×**. This is a crucial finding: **Marconi future-proofs serving infrastructure** against the trend of larger, more capable recurrent states.

**Request Arrival Patterns (Figure 13):**
The authors test robustness against varying traffic patterns.
*   **High Session Arrival Rate:** As the number of concurrent sessions increases, overall hit rates drop for all systems due to capacity limits. However, Marconi's *relative* advantage grows (from **1.4×** to **1.6×**), indicating that its efficient space utilization allows it to withstand higher contention better than LRU-based systems.
*   **Long Inter-Request Delays:** Even when users take longer to respond (simulating human thinking time or IDE interaction), Marconi maintains its lead, proving that its retention of high-value long sequences persists even when recency signals decay.

### 5.5 Critical Assessment

**Do the experiments support the claims?**
Yes, the experimental evidence is robust and multi-faceted.
1.  **Causality:** By constructing specific baselines (vLLM+ for admission, SGLang+ for eviction), the authors successfully isolate the contribution of each novel component. The massive gap with vLLM+ proves the necessity of judicious admission; the significant gap with SGLang+ on long-tail workloads proves the necessity of FLOP-aware eviction.
2.  **Realism:** The use of real-world traces (LMSys, SWE-Bench) rather than synthetic data adds credibility. The observation that Marconi trades short-sequence hits for long-sequence hits (Figure 10) demonstrates a deep understanding of the workload characteristics, moving beyond simple "higher is better" metrics to a nuanced analysis of *value*.
3.  **Scalability:** The microbenchmarks (Figure 12) convincingly argue that Marconi is not just a fix for current models but a necessary infrastructure upgrade for the *future* of Hybrid LLMs, which will inevitably feature larger state dimensions.

**Limitations and Conditions:**
*   **Workload Dependency:** The benefits of Marconi are conditional on the presence of **long sequences** and **cache contention**. In workloads dominated entirely by short prompts (e.g., simple classification tasks) or in scenarios with infinite cache memory, the complex logic of FLOP-aware eviction yields diminishing returns (as seen in the low-contention regime of Figure 11). However, given the industry trend toward long-context reasoning and coding agents, these conditions are increasingly the norm rather than the exception.
*   **Bootstrap Overhead:** The dynamic tuning of $\alpha$ requires a bootstrap period of $5\times$ to $15\times$ the requests seen before the first eviction. While the paper notes this takes only seconds, there is a brief window at system startup where the policy is suboptimal (pure LRU). For extremely short-lived serving instances, this warm-up period might slightly blunt the initial performance, though the asynchronous nature of the tuning minimizes this impact.

In summary, the evaluation convincingly demonstrates that Marconi is not merely an incremental optimization but a fundamental correction to how caching systems interact with recurrent architectures. By aligning cache management policies with the unique memory/compute physics of Hybrid models, Marconi unlocks the theoretical efficiency promises of architectures like Mamba and Jamba, transforming them from research curiosities into viable production solutions for long-context AI.

## 6. Limitations and Trade-offs

While Marconi demonstrates significant performance gains for Hybrid LLM serving, its design involves specific trade-offs and relies on assumptions that constrain its applicability in certain edge cases. Understanding these limitations is crucial for deploying the system effectively in production environments.

### 6.1 The "Third-Occurrence" Latency Penalty
A direct consequence of Marconi's **judicious admission** strategy is a deliberate delay in caching "purely input" prefixes (e.g., system prompts or few-shot examples).
*   **The Mechanism:** As detailed in **Section 4.1**, Marconi uses a **speculative insertion** step to identify branch points in the radix tree. It only admits the SSM state for a prefix if the simulation confirms that the incoming sequence creates a new intermediate node (a branch).
*   **The Trade-off:** This means a shared prefix is **not cached on its second occurrence**. The system uses the second request to *identify* the pattern and checkpoint the state, but the actual cache hit only occurs on the **third occurrence** of that prefix.
*   **Impact:** The paper acknowledges this explicitly: *"Marconi's judicious admission... sacrifices the benefit of reusing a 'purely input' prefix on its second occurrence."*
*   **Mitigation:** The authors argue this is negligible because purely input prefixes (like system prompts) are typically shared across *many* requests, not just two. However, in workloads with low reuse frequency (e.g., a prompt shared by exactly two users and then abandoned), Marconi incurs a full prefill cost twice, whereas a naive fine-grained caching system (like vLLM+) would have cached it immediately after the first request. In such sparse reuse scenarios, Marconi's latency could theoretically be higher than a "cache everything" approach, though the latter would likely thrash the cache globally.

### 6.2 Dependency on Cache Contention and Sequence Length
Marconi's advantages are not universal; they are highly dependent on the specific pressure conditions of the serving environment.
*   **The "Sweet Spot" Constraint:** The benefits of **FLOP-aware eviction** are most pronounced under **moderate cache contention**.
    *   **High Contention:** As shown in **Figure 11**, when cache size is extremely limited (e.g., 60 GB), the hit rate for *all* systems drops below 10%. In this regime, even Marconi cannot find enough high-value entries to store, and the relative gain shrinks.
    *   **Low Contention:** Conversely, when memory is abundant (e.g., 140 GB), the cache can hold almost all prefixes regardless of their efficiency. Here, the sophisticated eviction logic provides diminishing returns, with Marconi's win over SGLang+ dropping to just **10.0%**.
*   **Sequence Length Bias:** The system is explicitly optimized for long contexts. **Figure 10a** reveals that Marconi intentionally **sacrifices hit rates for short sequences** (&lt;7K tokens) to protect long ones.
    *   In workloads dominated entirely by short prompts (e.g., simple classification, short Q&A, or summarization tasks where inputs are &lt;500 tokens), Marconi's FLOP-aware policy offers little advantage and may even underperform standard LRU due to the overhead of computing FLOP efficiency metrics for entries that all have similarly low compute savings. The system's core value proposition—that long sequences offer vastly superior FLOP/byte ratios—collapses when no long sequences exist.

### 6.3 Computational Overhead of State Materialization
To overcome the "in-place update" constraint of SSMs, Marconi must employ specific techniques to extract states for caching, which introduce computational overheads not present in Transformer serving.
*   **Two-Pass Prefill Cost:** For models that do not support chunked state passing, Marconi must execute a **two-pass prefill** (Section 4.1).
    *   **Mechanism:** The system runs the model up to the checkpoint token to generate the state, saves it, and then restarts from that state to finish the sequence.
    *   **Constraint:** This effectively doubles the compute cost for the *prefix portion* of that specific request. While the paper argues this is a one-time cost amortized over future hits, it introduces a latency spike for the *first* few requests that trigger a new cache entry. In scenarios with high churn (many unique prefixes that are never reused), this overhead becomes pure waste, degrading throughput compared to a single-pass vanilla inference.
*   **Chunked Approximation Error:** For models using chunked state passing, Marconi may cache the state at a chunk boundary (e.g., token 64) rather than the exact branch point (e.g., token 80).
    *   **Limitation:** This results in a **partial cache hit**. The system can reuse the state up to token 64 but must recompute tokens 65–80. While the paper states this introduces "minimal runtime overhead," it means Marconi does not achieve 100% compute elimination even on a "hit," unlike perfect prefix caching in Transformers. The magnitude of this inefficiency depends on the chunk size, a hyperparameter not deeply explored in the ablation studies.

### 6.4 Bootstrap Latency and Dynamic Tuning
Marconi relies on an online **grid search** to determine the optimal balance parameter $\alpha$ (Equation 2) between recency and FLOP efficiency.
*   **The Warm-up Problem:** The system starts in pure LRU mode ($\alpha=0$) and requires a **bootstrap period** to collect enough request samples ($5\times$ to $15\times$ the number of requests seen before the first eviction) to tune $\alpha$.
*   **Edge Case:** For short-lived serving instances (e.g., serverless functions that spin up for a few seconds to handle a burst and then terminate), the system may never exit the bootstrap phase or may terminate before the optimal $\alpha$ is found. In these transient scenarios, Marconi operates sub-optimally as a standard LRU cache, failing to leverage its primary innovation.
*   **Workload Stationarity Assumption:** The tuning mechanism assumes that the workload characteristics (sequence length distribution, arrival patterns) remain relatively stable during the bootstrap window. If the traffic pattern shifts drastically immediately after the tuning completes (e.g., a sudden switch from chat traffic to long-document processing), the static $\alpha$ value may become suboptimal until the next re-tuning cycle, potentially leading to temporary performance degradation.

### 6.5 Hardware and Model Architecture Constraints
While Marconi is designed to be model-agnostic, its effectiveness is tied to specific architectural trends.
*   **SSM State Size Sensitivity:** The system's gains are predicated on SSM states being significantly larger than single-token KVs (the "fixed cost" problem). If future Hybrid architectures evolve to have extremely small SSM state dimensions (approaching the size of a single KV block), the memory pressure that drives the need for judicious admission diminishes. In such a hypothetical scenario, the complexity of Marconi's admission logic might outweigh its benefits.
*   **Implementation Complexity:** The requirement to manage **holistic state coupling** (binding KVs and SSM states atomically) and perform **speculative insertions** adds significant engineering complexity compared to standard paged attention systems.
    *   **Missing Support:** The paper notes that custom kernels may be needed to "roll forward" states for exact checkpointing (Section 4.1). This implies that Marconi cannot be deployed on *any* off-the-shelf Hybrid model without potential kernel development or acceptance of the chunked approximation error. This creates a barrier to entry for newer, non-standard recurrent architectures that do not yet have optimized serving kernels.

### 6.6 Unaddressed Scenarios
*   **Distributed Serving Across Nodes:** The evaluation in **Section 5** is conducted on a single node (8x A100s). The paper does not address how Marconi's radix tree and state management scale to a **multi-node cluster** where prefixes might be sharded across different GPUs. The "atomicity" requirement of coupling KV and SSM states complicates distributed caching strategies that rely on disaggregating state types.
*   **Dynamic Context Windowing:** The experiments assume fixed or naturally occurring sequence lengths. The system does not explicitly address scenarios where the context window is dynamically truncated or manipulated by the application layer in ways that break the radix tree's prefix assumptions (e.g., sliding window attention mechanisms applied to the SSM layers, though this is less common in pure SSMs).

In summary, Marconi is a highly specialized solution optimized for **long-context, moderate-contention, multi-turn workloads** on Hybrid models with large state dimensions. Its trade-offs—specifically the delayed caching of new prefixes and the intentional sacrifice of short-sequence hits—make it less suitable for short-prompt, low-reuse, or highly transient serving environments.

## 7. Implications and Future Directions

Marconi represents a pivotal shift in LLM serving infrastructure, moving the field from a "one-size-fits-all" caching paradigm designed for Transformers to a **heterogeneity-aware architecture** that respects the distinct physical properties of emerging hybrid models. Its success signals that as model architectures diverge from pure Attention, system software must evolve from generic memory managers into specialized optimizers that understand the compute-to-memory physics of each layer type.

### 7.1 Reshaping the Serving Landscape
The primary implication of Marconi is the **democratization of long-context Hybrid models**. Prior to this work, the theoretical efficiency of architectures like Mamba or Jamba was severely undermined in production by the "cache thrashing" phenomenon described in Section 3. Systems attempting to serve these models with standard prefix caching (e.g., vLLM+) faced a binary choice: exhaust GPU memory with fine-grained checkpoints or suffer negligible hit rates with coarse checkpoints.

Marconi breaks this deadlock, proving that **Hybrid models can outperform Transformers in serving scenarios only if the caching layer is co-designed with the model architecture**.
*   **From Recency to Value:** The field must abandon the assumption that "recently used" equals "valuable." Marconi's **FLOP-aware eviction** (Section 4.2) establishes a new standard where cache value is defined by $\text{Compute Saved} / \text{Memory Consumed}$. This metric is likely to become essential for any future architecture featuring fixed-size states with variable compute savings (e.g., certain retrieval-augmented generation states or compressed context representations).
*   **Proactive vs. Reactive Management:** The success of **speculative insertion** (Section 4.1) suggests a broader trend toward **predictive resource management**. Instead of reacting to cache misses by evicting data, next-generation systems will increasingly simulate future access patterns to prevent low-utility data from entering the cache in the first place.

### 7.2 Enabling Follow-Up Research
Marconi opens several critical avenues for future research, moving beyond the specific implementation details to broader system-design questions:

*   **Generalized Heterogeneous Caching:** Marconi handles two state types (KV and SSM). Future research should explore a **unified caching framework** capable of managing diverse state types simultaneously, including:
    *   **Retrieval States:** Compressed vectors from RAG pipelines.
    *   **LoRA Adapters:** Low-rank adaptation weights that may need to be swapped in/out alongside context states.
    *   **Multi-Modal Embeddings:** Visual or audio tokens with different memory/compute profiles.
    *   *Research Question:* Can a single "universal utility metric" replace the specific FLOP efficiency metric to manage all these heterogeneous states atomically?

*   **Distributed Prefix Caching:** The current evaluation is limited to a single node (8x A100s). Scaling Marconi to a **multi-node cluster** introduces complex challenges:
    *   How does one maintain the **atomicity** of coupled KV/SSM states when prefixes are sharded across GPUs?
    *   Does the "speculative insertion" logic need to be distributed, and what is the communication overhead of synchronizing radix tree metadata across nodes?
    *   *Research Direction:* Developing a distributed radix tree protocol that preserves Marconi's admission/eviction guarantees without becoming a bottleneck.

*   **Dynamic Architecture Adaptation:** Marconi assumes a static model architecture during serving. Future work could explore **runtime-adaptive caching** where the system dynamically adjusts its strategy based on the active layers. For instance, if a model dynamically switches between "high-recall" (more Attention) and "high-speed" (more SSM) modes, the caching policy could shift its $\alpha$ parameter (Equation 2) in real-time to match the dominant compute bottleneck.

*   **Lossy Compression for SSM States:** Marconi treats SSM states as immutable blobs. Given their large size, research into **lossy compression techniques** specific to recurrent states (e.g., quantization or low-rank approximation of the state matrix) could further reduce the memory footprint, potentially allowing even finer-grained checkpointing without thrashing.

### 7.3 Practical Applications and Downstream Use Cases
Marconi directly enables high-performance deployment for several key application domains that rely on long contexts and multi-turn interactions:

*   **Autonomous Coding Agents (e.g., SWE-Agent):**
    *   *Context:* These agents maintain a growing trajectory of code edits, terminal outputs, and reasoning steps, often exceeding 10k–50k tokens.
    *   *Impact:* As shown in the **SWE-Bench** results (Figure 8), Marconi reduces P95 TTFT by **~617 ms** compared to baselines. This latency reduction is critical for agent loop speed, allowing developers to iterate faster and reducing the cost of long-running debugging sessions.

*   **Enterprise RAG and Document Analysis:**
    *   *Context:* Applications ingesting entire legal contracts, technical manuals, or medical records where the "system prompt" includes thousands of tokens of context.
    *   *Impact:* The **judicious admission** policy ensures that these massive, shared document prefixes are cached efficiently after the second occurrence, enabling rapid querying across thousands of concurrent users without exhausting GPU memory.

*   **Long-Form Creative Writing and Roleplay:**
    *   *Context:* Chatbots that maintain coherent personas over hundreds of turns, requiring the full conversation history to be re-processed or cached.
    *   *Impact:* By prioritizing long sequences via FLOP-aware eviction, Marconi ensures that active, long-running conversations remain in the cache, preventing the "context collapse" where the model forgets early instructions due to cache evictions.

### 7.4 Reproduction and Integration Guidance

For practitioners and researchers looking to adopt or build upon Marconi, the following guidance outlines when and how to integrate these techniques:

**When to Prefer Marconi:**
*   **Model Architecture:** You are serving **Hybrid models** (e.g., Jamba, Zamba, Mamba-Transformer mixes) or pure SSMs. *Do not use Marconi for pure Transformers;* standard PagedAttention (vLLM) or RadixAttention (SGLang) remains optimal there as KV states are sliceable and uniform.
*   **Workload Characteristics:** Your workload features **long sequences** (>2k tokens) and **moderate-to-high cache contention**. If your cache is large enough to hold all active contexts, the overhead of FLOP-aware calculation may not yield significant gains.
*   **Latency Sensitivity:** You are optimizing for **Tail Latency (P95 TTFT)**. Marconi's strategy of sacrificing short-sequence hits to protect long-sequence hits specifically targets the long tail of latency distribution.

**Integration Considerations:**
*   **State Materialization Overhead:** Be prepared to implement **two-pass prefill** or **chunked state passing** kernels. As noted in Section 4.1, this adds a one-time compute cost for new prefixes. Ensure your inference engine supports custom kernel injection to minimize this overhead.
*   **Bootstrap Period:** When deploying Marconi, anticipate a brief **warm-up period** (seconds to minutes depending on traffic) where the system operates in LRU mode while collecting statistics to tune $\alpha$. For short-lived serverless instances, consider pre-warming the cache or using a static $\alpha$ derived from offline profiling of similar workloads.
*   **Memory Budgeting:** While Marconi reduces thrashing, SSM states are inherently large. Ensure your GPU memory budget accounts for the fixed size of SSM states ($D \times N \times 2$ bytes per layer) rather than just the sequence length.

**Reproducibility:**
The authors have open-sourced Marconi at `github.com/ruipeterpan/marconi`. The artifact includes:
*   Pre-processed traces for **LMSys**, **ShareGPT**, and **SWE-Bench**.
*   Scripts to reproduce the **FLOP-aware eviction** grid search and the **speculative insertion** logic.
*   Configurations for the **7B Hybrid model** and **Jamba-1.5-Mini** used in the evaluation.
*   *Note:* To fully replicate the results, ensure you have access to A100-class GPUs, as the memory bandwidth and capacity are critical factors in the observed contention scenarios (Section 5.4).

In conclusion, Marconi does not merely optimize an existing pipeline; it validates a new design philosophy for the post-Transformer era. By aligning system policies with the mathematical realities of hybrid architectures, it paves the way for the next generation of efficient, long-context AI applications.
