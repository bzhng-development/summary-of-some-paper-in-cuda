## 1. Executive Summary

SARATHI addresses the critical inefficiency in Large Language Model (LLM) inference caused by the memory-bound nature of autoregressive decoding and the pipeline bubbles resulting from imbalanced prefill and decode compute times. By introducing **chunked-prefills** to split prompts into uniform compute units and **decode-maximal batching** to "piggyback" decode requests onto these compute-saturating chunks, the system converts inefficient decodes into compute-bound operations while balancing micro-batch workloads. This approach yields substantial performance gains, improving decode throughput by up to **10×** for **LLaMA-13B** on an **A6000 GPU** and reducing pipeline bubbles by **6.29×** in **GPT-3** deployments, ultimately achieving up to **1.91×** higher end-to-end throughput.

## 2. Context and Motivation

To understand why SARATHI is necessary, we must first dissect the fundamental anatomy of Large Language Model (LLM) inference and identify where current hardware utilization breaks down. The inefficiency is not a minor optimization issue; it is a structural mismatch between how transformers compute and how modern GPUs are designed to operate.

### The Two-Phase Nature of LLM Inference

Every request to an LLM undergoes two distinct computational phases, each with radically different resource profiles:

1.  **The Prefill Phase:** This stage processes the entire input prompt (the user's question or context) in parallel. If a prompt contains 512 tokens, the GPU computes representations for all 512 tokens simultaneously. Mathematically, this involves **matrix-matrix multiplications** ($[B, L, H] \times [H, H]$), where $B$ is batch size, $L$ is sequence length, and $H$ is hidden dimension. Because $L$ is large, the arithmetic intensity (the ratio of floating-point operations to memory accesses) is high. As shown in **Figure 3** and **Figure 4b**, even a single request ($B=1$) with a sequence length of 512 is sufficient to saturate the GPU's compute units. The GPU is "compute-bound," meaning it is working as hard as it physically can.
2.  **The Decode Phase:** Once the prompt is processed, the model generates output tokens one by one (autoregressively). To generate the next token, the model only processes the *single* most recent token. Mathematically, this degrades to **vector-matrix multiplications** ($[B, 1, H] \times [H, H]$). Here, the arithmetic intensity plummets because the model must fetch the entire set of weights from global memory to process just one token per request.

**The Critical Gap:** The paper identifies a massive disparity in efficiency between these phases. **Figure 3** reveals that for the **LLaMA-13B** model on an **A6000 GPU**, the cost to process a single token during the decode phase can be **~200 times higher** than during the prefill phase when the batch size is small. While prefill saturates the GPU immediately, decode remains **memory-bound**; the GPU spends most of its time waiting for data to move from memory to the compute cores, leaving the actual calculation units idle.

Since a typical inference request involves one short prefill phase followed by many decode steps (one per generated token), the overall system efficiency is dragged down by this long tail of inefficient decoding.

### The Pipeline Parallelism Bottleneck

As models grow larger (e.g., GPT-3, LLaMA-33B), they no longer fit on a single GPU. Engineers use **Pipeline Parallelism (PP)** to split the model layers across multiple GPUs. In PP, the model is divided into stages (e.g., layers 1–10 on GPU 1, layers 11–20 on GPU 2). To keep all GPUs busy, the workload is divided into **micro-batches** that flow through the pipeline like an assembly line.

However, PP relies on a critical assumption: **uniform execution time**. If every micro-batch takes the exact same amount of time to process, the pipeline flows smoothly. If one micro-batch is slow, subsequent stages must wait, creating idle periods known as **pipeline bubbles**.

**Figure 1(a)** and **Figure 5** illustrate why standard LLM inference creates catastrophic bubbles:
*   **Varying Prompt Sizes:** Different requests have different input lengths, causing prefill micro-batches to have varying compute times.
*   **Phase Mismatch:** A micro-batch containing only prefill tokens might take significantly longer (or shorter) than a micro-batch containing only decode tokens.
*   **Context Growth:** As decoding proceeds, the Key-Value (KV) cache grows, slightly increasing the compute time for attention operations in later decode steps.

Prior systems like **Orca** [48] attempted to solve this with "iteration-level scheduling," dynamically adding or removing requests from a batch. However, Orca still treats the prefill as a single, monolithic block. If a large prompt enters the system, it creates a massive spike in compute time that disrupts the rhythm of the pipeline, leading to the bubbles shown in **Figure 1(a)**. The paper notes that even with iteration-level scheduling, the mixture of varying-length prefills and decodes results in significant wasted GPU cycles.

### Limitations of Prior Approaches

The paper positions existing solutions as insufficient for two primary reasons:

1.  **Inability to Fix Memory-Bound Decoding:** Traditional batching strategies try to increase the batch size ($B$) to make decoding compute-bound. However, **Figure 4a** shows that decoding only saturates the GPU at extremely large batch sizes (e.g., $B=256$ for a single layer). For a full 40-layer model like LLaMA-13B, memory constraints (specifically the KV cache footprint) limit the practical batch size to around 18. Thus, in real-world deployments, decoding remains stubbornly memory-bound.
2.  **Rigid Micro-Batch Construction:** Systems like **FasterTransformer** [6] use request-level scheduling (waiting for all requests in a batch to finish), which causes padding waste. Newer systems like **Orca** [48] and **vLLM** [20] use iteration-level scheduling but still submit the *entire* input sequence as a single prefill unit. This prevents them from constructing micro-batches with uniform compute properties. As noted in **Section 3.2**, Orca's approach can still lead to bubbles because it cannot balance the compute load when a massive prefill request arrives alongside small decode requests.

### SARATHI's Strategic Positioning

SARATHI positions itself not as a replacement for model parallelism or memory management techniques (like vLLM's paging), but as a fundamental rethinking of **how work units are constructed** before they enter the pipeline.

The paper argues that the solution lies in **hybrid batching**: mixing prefill and decode tokens within the *same* micro-batch to create a unit of work that is both **compute-saturating** and **uniform**.
*   **Against Monolithic Prefills:** Unlike Orca, which processes a 1024-token prompt as one heavy block, SARATHI splits it into smaller, equal-sized **chunks** (e.g., four chunks of 256 tokens).
*   **Against Pure Decode Batches:** Instead of running a batch of only inefficient decodes, SARATHI fills the remaining slots of a prefill chunk with decode requests.

This approach leverages the fact that the linear operations (matrix multiplications) for prefill and decode use the same weights. By fusing them, the weights are fetched once for the prefill chunk and immediately reused for the piggybacking decodes, effectively converting the decode phase from memory-bound to compute-bound without requiring impossible batch sizes.

In summary, SARATHI addresses the gap between the **theoretical peak performance** of GPUs and the **actual low utilization** caused by the erratic nature of LLM workloads. It transforms the inference problem from one of managing variable-length requests to one of assembling uniform, compute-dense tiles, thereby solving both the single-GPU efficiency problem and the multi-GPU pipeline bubble problem simultaneously.

## 3. Technical Approach

This section details the architectural design and algorithmic mechanisms of SARATHI, a system that fundamentally restructures how Large Language Model (LLM) inference requests are scheduled and executed to maximize GPU utilization. The core idea is to abandon the traditional separation of "prefill-only" and "decode-only" batches in favor of **hybrid micro-batches** that combine a fragment of a new prompt with multiple ongoing generation requests, thereby converting memory-bound decoding into compute-bound operations while enforcing uniform execution times across pipeline stages.

### 3.1 Reader orientation (approachable technical breakdown)
SARATHI is a scheduling engine that acts as a traffic controller for GPU compute, breaking large input prompts into small, standardized "chunks" and packing them alongside single-token generation requests to ensure the GPU is always working at full capacity. It solves the dual problems of wasted GPU cycles during slow token generation and stalled pipelines caused by uneven workloads by forcing all computational units to have the exact same size and density, regardless of whether they are processing new inputs or generating outputs.

### 3.2 Big-picture architecture (diagram in words)
The SARATHI architecture can be visualized as a three-stage pipeline transforming raw user requests into optimized GPU kernels:
1.  **Chunked-Prefill Splitter:** This component intercepts incoming long prompts (e.g., 1024 tokens) and slices them into fixed-size compute units (e.g., 256 tokens), outputting a stream of standardized prefill chunks rather than monolithic requests.
2.  **Decode-Maximal Batcher:** This scheduler maintains a queue of active decoding requests; for every incoming prefill chunk, it greedily fills the remaining available GPU memory slots with as many single-token decode requests as possible, constructing a "hybrid batch."
3.  **Fused Kernel Executor:** This low-level execution engine takes the hybrid batch and performs a single, fused matrix multiplication for the linear layers (feeding both chunk tokens and decode tokens simultaneously), while handling attention mechanisms separately to respect causal dependencies, outputting the next set of hidden states and logits for all requests in the batch.

### 3.3 Roadmap for the deep dive
*   **Chunked-Prefills Mechanism:** We first explain how splitting prompts works mathematically and how the system preserves attention correctness across splits, as this is the foundational unit of work.
*   **Decode-Maximal Batching Strategy:** We then detail how the system constructs hybrid batches, explaining the memory constraints and the "piggybacking" logic that boosts decode efficiency.
*   **Kernel Fusion and Arithmetic Intensity:** We analyze the specific linear algebra operations that allow decodes to become compute-bound, contrasting the fused approach against baseline separate executions.
*   **Optimal Chunk Size Selection:** We discuss the critical trade-off between prefill efficiency and decode coverage, including the impact of GPU hardware tile sizes on performance.
*   **Pipeline Parallelism Integration:** Finally, we explain how these uniform micro-batches eliminate pipeline bubbles in multi-GPU deployments, completing the end-to-end performance picture.

### 3.4 Detailed, sentence-based technical breakdown

#### The Core Mechanism: Chunked-Prefills
The primary innovation of SARATHI is the **chunked-prefill** technique, which decomposes a large input sequence into smaller, equal-sized segments to create uniform units of compute. In standard inference, a prompt of length $L$ is processed in a single pass, creating a variable-time workload; SARATHI instead divides this prompt into $N$ chunks, where each chunk has a fixed sequence length $C$ (e.g., $C=256$ tokens).

To ensure mathematical equivalence with the standard full-prefill, the system must carefully manage the **attention mask**, which dictates which tokens a given token is allowed to "see" during the self-attention calculation.
*   Let the input prompt be represented as a sequence of tokens $q_0, q_1, \dots, q_{L-1}$.
*   When processing the first chunk (tokens $q_0$ to $q_{C-1}$), the attention mask allows each token $q_i$ to attend to all keys $k_j$ where $j \le i$, exactly as in a standard transformer.
*   When processing the second chunk (tokens $q_C$ to $q_{2C-1}$), the system must ensure that these new query tokens can attend to *all* preceding keys from the first chunk ($k_0$ to $k_{C-1}$) as well as their own preceding keys within the second chunk.
*   **Figure 6** illustrates this progressive masking: for the $n$-th chunk, the attention mask is constructed such that every query token in that chunk has visibility into the Key-Value (KV) pairs of all prior chunks, which are retrieved from the GPU memory cache.

This approach introduces a specific overhead: the KV cache for earlier chunks must be re-read from global memory for every subsequent chunk processing step. If a prompt is split into $N$ chunks, the KV pairs of the first chunk are loaded $N$ times, the second chunk $N-1$ times, and so on. However, the paper argues this overhead is negligible because the attention computation itself constitutes a small fraction of the total runtime compared to the Feed-Forward Network (FFN) linear operations, as evidenced by the operation breakdown in **Table 2**.

#### Constructing Hybrid Batches: Decode-Maximal Batching
Once the prefill requests are fragmented into uniform chunks, SARATHI employs **decode-maximal batching** to construct the actual GPU workloads. The goal is to create a batch that saturates the GPU's compute units (which the prefill chunk naturally does) while filling any remaining capacity with decode requests that would otherwise be inefficient.

The construction of a hybrid batch is governed by the GPU's memory constraints. The maximum batch size $B$ is determined by the available GPU memory $M_G$, the model parameter memory $M_S$, the maximum sequence length $L$, and the memory required per KV pair per token $m_{kv}$. The theoretical maximum batch size is calculated as:

$$
B = \left\lfloor \frac{M_G - M_S}{L \times m_{kv}} \right\rfloor
$$

In a baseline system, a batch could contain up to $B$ decode requests. In SARATHI, the scheduler reserves space for exactly **one** prefill chunk and fills the remaining $B-1$ slots with decode requests.
*   The "piggybacking" concept relies on the observation that both prefill and decode phases utilize the exact same weight matrices for linear transformations (the `preproj`, `postproj`, and `ffn` layers described in **Section 2.1**).
*   By placing the prefill chunk and the $B-1$ decode tokens into the same batch tensor, the system performs a single **matrix-matrix multiplication** for the linear layers.
*   Mathematically, instead of performing a vector-matrix multiplication for each decode request ($[1, H] \times [H, H']$), the system performs a larger matrix-matrix multiplication where the input dimension is effectively $[B, H]$.
*   This transformation drastically increases the **arithmetic intensity** (the ratio of FLOPs to memory bytes fetched). Since the model weights are fetched from global memory once for the entire batch, the cost of fetching weights is amortized across the prefill chunk and all piggybacking decodes.

**Table 2** provides concrete evidence of this efficiency gain. For a **LLaMA-13B** model on an **A6000 GPU**:
*   A baseline decode-only batch (size 4) incurs a per-token time of **12.49 ms**, dominated by memory latency.
*   A SARATHI decode-maximal batch (1 prefill chunk of 1021 tokens + 3 decodes) reduces the marginal per-token decode time to just **1.2 ms**.
*   This represents an order-of-magnitude improvement ($\approx 10\times$), effectively converting the decode phase from memory-bound to compute-bound.

#### Handling Attention in Hybrid Batches
While the linear layers are fused, the attention mechanism requires special handling due to the differing nature of prefill and decode data.
*   For the **prefill chunk**, the attention operation is a standard batched matrix multiplication over the sequence length $C$, computing interactions between all tokens in the chunk and their history.
*   For the **decode requests**, the attention operation is distinct: each decode token only needs to attend to its own historical KV cache and the current context.
*   SARATHI implements this by running the attention kernels separately for the prefill portion and the decode portion of the batch, while keeping the linear layers fused. The paper notes that since attention is a smaller fraction of the total runtime (especially for large hidden dimensions), the inability to fully fuse attention does not significantly detract from the overall gains achieved by fusing the linear layers.

#### Selecting the Ideal Chunk Size
A critical design choice in SARATHI is determining the optimal chunk size $C$. This is not simply "the smallest size that saturates compute," but a trade-off between prefill efficiency and decode coverage.
*   **Smaller Chunks:** Create more chunks per prompt, providing more opportunities to piggyback decode requests. This is beneficial when the ratio of prefill tokens to decode tokens (P:D ratio) is low. However, smaller chunks reduce the arithmetic intensity of the prefill itself and increase the KV cache reload overhead.
*   **Larger Chunks:** Maintain high prefill efficiency but generate fewer chunks, potentially leaving decode requests waiting in the queue without a prefill vehicle to piggyback on.

The paper defines the **P:D ratio** as the ratio of total prefill tokens to total decode tokens in a workload. The optimal operating point occurs when the number of prefill chunks exactly matches the demand for decode slots. If $P$ is the total prefill length and $D$ is the total decode length, perfect piggybacking occurs when:
$$
\frac{P}{C} = \frac{D}{B-1} \implies \frac{P}{D} = \frac{C}{B-1}
$$
Thus, the ideal chunk size $C$ depends on the expected workload P:D ratio and the maximum batch size $B$.

Furthermore, the paper identifies a subtle but crucial hardware constraint: **tile quantization**. GPUs execute matrix multiplications by dividing matrices into fixed-size tiles (e.g., $128 \times 128$). If the matrix dimensions are not multiples of the tile size, the GPU must process partial tiles, leading to wasted compute cycles.
*   **Figure 7** demonstrates that increasing the sequence length from 128 to 129 tokens can cause a disproportionate spike in runtime (e.g., a 32% increase for a single token) because it forces the allocation of an additional tile row/column.
*   Therefore, SARATHI selects a chunk size $C$ such that the total effective sequence length of the fused batch aligns with the GPU's tile size. For example, if the tile size is 128 and the batch size is $B$, the system may adjust the prefill chunk size to $C' = C - (B-1)$ to ensure the total dimension remains a multiple of 128, maximizing hardware utilization.

#### Integration with Pipeline Parallelism
The uniformity of SARATHI's micro-batches directly addresses the pipeline bubble problem in multi-GPU deployments.
*   In standard Pipeline Parallelism (PP), bubbles arise because micro-batches have variable execution times (e.g., a large prefill followed by a small decode).
*   Because SARATHI enforces that every micro-batch consists of exactly one prefill chunk (of fixed size $C$) and $B-1$ decode tokens, the compute load for every micro-batch is nearly identical.
*   This uniformity ensures that all stages in the pipeline finish their work at roughly the same time, allowing the next micro-batch to flow immediately without waiting.
*   **Figure 1(b)** contrasts this with **Figure 1(a)**: where the baseline shows significant gaps (bubbles) between pipeline stages, the SARATHI schedule shows a tight, continuous flow of computation.
*   The paper quantifies this in **Section 5.3**, noting that for **GPT-3** on a simulated 64-GPU cluster, SARATHI reduces the median bubble time per request by **6.29×**, transforming pipeline parallelism from a bubble-prone architecture into a highly efficient serving strategy.

#### Implementation Specifics
The authors implemented SARATHI on top of the **nanoGPT** codebase, modifying the scheduling logic to support dynamic chunking and hybrid batch formation.
*   For attention computations, they utilized the **xformers** library [21], which provided better performance in their specific setup compared to PyTorch 2.0's native Flash Attention or memory-efficient attention kernels.
*   To minimize memory allocation overhead, the system pre-allocates the KV cache for the maximum possible sequence length at the start of execution and updates the cache entries in-place during each decode iteration.
*   The system supports various model configurations, including **LLaMA-13B** (40 layers, hidden size 5120), **LLaMA-33B** (60 layers, hidden size 6656), and **GPT-3** (96 layers, hidden size 12288), demonstrating the generality of the approach across different model scales and hidden dimension sizes.

In summary, SARATHI's technical approach is a cohesive strategy that aligns the *logical* structure of LLM inference (variable-length prompts and autoregressive decoding) with the *physical* constraints of GPU hardware (compute saturation, memory bandwidth, and tile alignment). By fragmenting prefills and fusing them with decodes, it creates a "synthetic" workload that behaves like a perfect, compute-bound matrix multiplication, unlocking throughput gains that were previously inaccessible to standard scheduling policies.

## 4. Key Insights and Innovations

SARATHI introduces a paradigm shift in LLM inference scheduling, moving away from treating requests as monolithic units toward constructing synthetic, hardware-aligned work units. The following innovations distinguish this work from prior systems like Orca [48] or vLLM [20], which primarily focus on memory management or dynamic request admission without fundamentally altering the compute structure of the batches.

### 4.1 The "Piggybacking" Paradigm: Converting Memory-Bound to Compute-Bound
The most fundamental theoretical contribution of SARATHI is the insight that **decode efficiency can be decoupled from batch size** by fusing decode tokens with prefill chunks.

*   **Differentiation from Prior Work:** Traditional optimization strategies attempt to make decoding efficient by increasing the batch size ($B$) until the arithmetic intensity is high enough to saturate the GPU. As shown in **Figure 4a**, this requires impractically large batches (e.g., $B=256$) that exceed KV cache memory limits for full models. Prior schedulers accept low utilization for small-batch decodes or rely on request arrival randomness to occasionally mix prefills and decodes (as in Orca's "best-case" scenario).
*   **The Innovation:** SARATHI actively engineers a **hybrid batch** where a single compute-saturating prefill chunk acts as a "vehicle" for $B-1$ decode requests. By fusing the linear operations (matrix-matrix multiplications) of the prefill and decode tokens, the system forces the decode tokens to participate in a high-arithmetic-intensity operation.
*   **Significance:** This transforms the decode phase from a **memory-bound** bottleneck (waiting for weights) into a **compute-bound** operation (utilizing FLOPs) without requiring massive batch sizes. **Table 2** quantifies this breakthrough: the marginal cost of a decode token drops from **12.49 ms** in a baseline system to **1.2 ms** when piggybacked. This is not an incremental tuning of kernel parameters; it is a structural change that allows small-batch decodes to achieve near-peak hardware utilization, yielding up to **10×** higher decode throughput (**Figure 8**).

### 4.2 Chunked-Prefills as a Mechanism for Workload Uniformity
While splitting sequences is a known concept in training (micro-batching), applying it to inference prefill phases to solve **pipeline imbalance** is a novel application specific to the heterogeneity of LLM serving.

*   **Differentiation from Prior Work:** Existing inference systems treat the prefill phase as an atomic, indivisible block. If a user submits a 4,000-token prompt, the GPU executes it as one long, variable-duration task. In pipeline parallelism, this creates **Pipeline Bubbles (PB1 and PB2 in Figure 5)** because subsequent stages must wait for this irregular block to clear. Orca's iteration-level scheduling mitigates this slightly by allowing requests to exit early, but it cannot break a single large prefill into smaller, uniform pieces.
*   **The Innovation:** SARATHI introduces **chunked-prefills** specifically to enforce **uniform compute density** across micro-batches. By slicing a large prompt into fixed-size chunks (e.g., 256 tokens), the system guarantees that every micro-batch entering the pipeline has nearly identical execution time, regardless of the original prompt length.
*   **Significance:** This directly targets the root cause of pipeline inefficiency. By ensuring that every micro-batch is a "perfect tile" of compute, SARATHI eliminates the variance that causes bubbles. The result is a **6.29× reduction in pipeline bubble time** for GPT-3 (**Figure 12a**). This innovation makes Pipeline Parallelism (PP) viable for inference serving, allowing it to outperform Tensor Parallelism (TP) in cross-node settings where bandwidth is limited, achieving a **1.91×** end-to-end speedup over baseline PP.

### 4.3 Hardware-Aware Tile Quantization for Batch Construction
A subtle but critical innovation in SARATHI is the integration of **GPU tile quantization effects** into the high-level scheduling logic.

*   **Differentiation from Prior Work:** Most scheduling algorithms operate at the level of tokens or requests, assuming that compute time scales linearly with sequence length. They ignore the low-level hardware reality that GPUs execute matrix multiplications in fixed-size tiles (e.g., $128 \times 128$). Prior work treats a sequence length of 128 and 129 as nearly equivalent in cost, whereas hardware behavior shows a discontinuous jump.
*   **The Innovation:** SARATHI explicitly models the **tile quantization overhead** during batch construction. As illustrated in **Figure 7**, adding a single token beyond a tile boundary (e.g., going from 128 to 129) can increase runtime by over 30% due to wasted thread block cycles. SARATHI's scheduler adjusts the prefill chunk size $C$ dynamically so that the total dimension of the fused batch ($C + \text{decode tokens}$) remains a multiple of the hardware tile size.
*   **Significance:** This prevents the system from inadvertently selecting a "bad" chunk size that looks optimal theoretically but performs poorly on silicon. **Figure 13c** shows that chunk sizes aligned with tile boundaries (e.g., 256) significantly outperform non-aligned sizes (e.g., 320), even if the latter allow for slightly more decode piggybacking. This demonstrates that optimal inference scheduling requires co-design between the algorithmic layer and the underlying GPU architecture.

### 4.4 Dynamic Trade-off Management via P:D Ratio Tuning
SARATHI provides a rigorous framework for navigating the trade-off between **prefill efficiency** and **decode coverage**, formalized through the **Prefill-to-Decode (P:D) ratio**.

*   **Differentiation from Prior Work:** Previous systems generally optimize for a single metric (e.g., maximize prefill throughput or minimize decode latency) in isolation. They lack a mechanism to adapt the compute unit size based on the workload mix.
*   **The Innovation:** The paper derives a closed-form condition for optimal performance: peak efficiency occurs when the number of prefill chunks exactly matches the decode demand, i.e., when $\text{P:D} \approx C/(B-1)$. SARATHI allows operators to tune the chunk size $C$ based on the expected workload characteristics.
*   **Significance:** This insight reveals that there is no single "best" chunk size. As shown in **Figure 9**, a chunk size of 256 is optimal for a P:D ratio of ~14, while a size of 512 is better for a ratio of ~28. This flexibility allows SARATHI to maintain robust performance (**>10% gain**) across a wide spectrum of application scenarios, whereas rigid systems degrade significantly when the workload shifts away from their fixed assumptions.

In summary, SARATHI's contributions are not merely incremental optimizations of existing kernels. They represent a **fundamental re-architecting of the inference workload**: transforming variable, memory-bound tasks into uniform, compute-bound tiles that align perfectly with both the mathematical requirements of the transformer and the physical constraints of the GPU.

## 5. Experimental Analysis

The authors conduct a rigorous evaluation of SARATHI across three distinct dimensions: single-GPU throughput optimization, comparison against state-of-the-art schedulers, and multi-GPU pipeline parallelism efficiency. The experimental design is carefully constructed to isolate the benefits of **chunked-prefills** and **decode-maximal batching** under varying workload characteristics.

### 5.1 Evaluation Methodology and Setup

To ensure the results are representative of real-world deployments, the evaluation spans multiple model scales, hardware configurations, and execution modes, as detailed in **Table 3**:

*   **Models & Hardware:**
    *   **LLaMA-13B** deployed on a single **NVIDIA A6000 (48GB)** GPU.
    *   **LLaMA-33B** deployed on a single **NVIDIA A100 (80GB)** GPU.
    *   **GPT-3** (simulated) deployed across a cluster of **64 A100 GPUs** using 8-way Tensor Parallelism (TP) and 8-way Pipeline Parallelism (PP).
*   **Baselines:**
    *   **Prefill/Decode-Separate Baseline:** A standard system that processes batches containing *only* prefill requests or *only* decode requests. This represents the behavior of frameworks like FasterTransformer [6].
    *   **Orca [48]:** The state-of-the-art iteration-level scheduler. The authors evaluate Orca in two modes: "Best-case" (where prefill and decode requests serendipitously overlap) and "Worst-case" (where they do not overlap, behaving like the separate baseline).
*   **Metrics:**
    *   **Decode Throughput:** Measured in tokens per millisecond or as a speedup factor relative to the baseline.
    *   **End-to-End Throughput:** The total number of tokens (prefill + decode) processed per unit time.
    *   **Pipeline Bubble Time:** The cumulative idle time of GPU stages in a pipeline-parallel setup.
    *   **P:D Ratio:** The ratio of total prefill tokens to total decode tokens in a workload, used to analyze performance across different application types (e.g., chat vs. summarization).

The authors utilize the **nanoGPT** codebase for implementation, replacing the native attention kernels with **xformers** [21] due to superior performance in their specific setup. For the large-scale GPT-3 experiments, they employ a **profile-driven simulation** calibrated against empirical measurements on an 8-GPU DGX node, ensuring prediction errors remain within 5%.

### 5.2 Single-GPU Performance: Decode Efficiency and Throughput

The primary claim of SARATHI is its ability to convert memory-bound decoding into compute-bound operations. The results strongly support this.

**Decode Speedup:**
**Figure 8** illustrates the decode-only speedup for LLaMA-13B on an A6000 GPU with a chunk size of 256.
*   At small batch sizes, SARATHI achieves a speedup of nearly **10×** compared to the baseline.
*   Even as batch sizes increase to the maximum fit (e.g., batch size 18), the speedup remains significant, ranging from **2.8× to 5.5×** depending on sequence length.
*   The speedup diminishes slightly at larger batch sizes because the baseline decode efficiency naturally improves as arithmetic intensity increases, but SARATHI maintains a distinct advantage.

**Peak End-to-End Throughput:**
**Table 4** summarizes the peak throughput gains across different models and sequence lengths.
*   **LLaMA-13B (A6000):** Achieves up to **1.33×** end-to-end throughput improvement with a sequence length of 1K and a P:D ratio of 50:1. The decode throughput specifically sees a **5.45×** boost.
*   **LLaMA-33B (A100):** Achieves up to **1.25×** end-to-end improvement. Notably, the decode throughput speedup reaches **4.25×** at a sequence length of 2K.
*   **Hardware Sensitivity:** The gains are slightly higher on the A6000 than the A100. The authors attribute this to the A100's higher FLOPs-to-Memory-Bandwidth ratio (~156 vs. ~53), which makes the A100 naturally more efficient at compute-bound tasks, thereby reducing the relative margin for improvement. However, SARATHI still provides consistent gains (>1.14×) on the faster hardware.

**Crucial Observation:** While decode throughput improves by an order of magnitude, the *end-to-end* speedup is capped around **1.25×–1.33×**. The authors explain in **Section 5.1.2** that this is expected: the technique optimizes the decode phase, but the prefill phase (which constitutes a portion of the total time) remains largely unchanged in terms of raw throughput. Since decodes often dominate the total runtime in long-generation tasks, the overall system benefit is substantial but bounded by the unoptimized prefill fraction.

### 5.3 Sensitivity to Workload Characteristics (P:D Ratio)

A critical aspect of SARATHI is its dependence on the **Prefill-to-Decode (P:D) ratio**. The system performs best when the supply of prefill chunks matches the demand for decode slots.

**Figure 9** plots normalized throughput against varying P:D ratios for LLaMA-13B with different chunk sizes (128, 256, 512).
*   **The "Sweet Spot":** Performance peaks when the P:D ratio aligns with the theoretical optimum derived in **Section 4.4**: $\text{P:D} \approx C/(B-1)$.
    *   For a chunk size $C=256$ and batch size $B=18$, the peak occurs at $\text{P:D} \approx 14$ ($256/17$). **Figure 9a** confirms the peak throughput gain of **1.27×** occurs exactly here.
    *   For a larger chunk size $C=512$, the peak shifts right to $\text{P:D} \approx 28$, achieving a gain of **1.23×**.
*   **Trade-offs:**
    *   **Low P:D (Decode-heavy):** If the workload has very few prefill tokens relative to decodes, SARATHI runs out of prefill chunks to piggyback decodes onto. In this regime, it falls back to less efficient decode-only batches.
    *   **High P:D (Prefill-heavy):** If the workload is mostly prefills, there are insufficient decode requests to fill the hybrid batches. The system processes extra prefill chunks without maximizing the "piggyback" benefit.
*   **Robustness:** Despite these peaks and valleys, SARATHI maintains a throughput improvement of at least **10%** across a wide range of P:D ratios (from ~20 to ~100), demonstrating robustness to workload variations.

**Ablation on Chunk Size (Figure 13):**
The authors perform a detailed ablation study on chunk sizes in **Section 5.4** and **Figure 13**.
*   **Overhead:** Very small chunks (e.g., 64) incur high overheads due to repeated KV cache reads and lower arithmetic intensity, slowing down the prefill phase by up to **5×** (**Figure 13b**).
*   **Compensation:** However, **Figure 13c** shows that even with a 5× slower prefill, a chunk size of 64 can match the baseline end-to-end performance because it allows *so many* decodes to piggyback that the aggregate gain offsets the prefill loss.
*   **Tile Quantization:** The study confirms the "tile quantization" effect. Chunk sizes that are multiples of the GPU tile size (128) perform significantly better. For instance, chunk size **256** outperforms **320**, even though 320 is larger, because 320 causes misaligned matrix dimensions leading to wasted compute cycles.

### 5.4 Comparison with Iteration-Level Scheduling (Orca)

The paper rigorously compares SARATHI against **Orca** [48], the leading iteration-level scheduler.

**Figure 11a** compares throughput across sequence lengths (1K, 2K, 3K):
*   **Orca Best-Case:** At 1K sequence length, Orca achieves a **1.11×** speedup over the baseline due to incidental overlapping of prefills and decodes. However, as sequence length increases to 2K and 3K, Orca's performance degrades toward the baseline. This happens because Orca treats the prefill as a monolithic block; as sequences get longer, the single large prefill dominates the schedule, leaving fewer opportunities for overlap.
*   **SARATHI:** Consistently outperforms Orca in all scenarios, achieving **1.27×**, **1.25×**, and **1.23×** speedups for 1K, 2K, and 3K sequences respectively. By chunking the prefill, SARATHI sustains the overlap opportunity throughout the entire request lifecycle.

**Figure 11b** analyzes performance across P:D ratios:
*   Orca's best-case performance is essentially a special case of SARATHI where the chunk size equals the maximum sequence length. Consequently, Orca peaks at very high P:D ratios and offers flat, modest gains elsewhere.
*   SARATHI with tuned chunk sizes (256 or 512) dominates Orca across the entire spectrum, particularly in the lower-to-mid P:D regimes where dynamic generation is most common.

### 5.5 Multi-GPU Pipeline Parallelism: Eliminating Bubbles

The most dramatic validation of SARATHI's "uniform compute" design comes from the simulated GPT-3 experiments on 64 GPUs.

**Pipeline Bubble Reduction:**
**Figure 12a** presents the Cumulative Distribution Function (CDF) of pipeline bubble time per request.
*   **Baseline (TP+PP with Orca):** Exhibits significant bubbles due to the variance in micro-batch execution times (mixed large prefills and small decodes).
*   **SARATHI:** Shifts the curve drastically to the left, reducing the **median bubble time by 6.29×**. This confirms that constructing uniform micro-batches (1 chunk + $B-1$ decodes) successfully synchronizes the pipeline stages.

**End-to-End Request Completion:**
**Figure 12b** plots the time to complete 10,000 requests under three strategies:
1.  **TP-Only (8 replicas):** Fast but memory-inefficient (max batch size 11).
2.  **TP+PP Baseline (Orca):** Supports larger batches (size 27) but suffers from bubbles, making it **1.28× slower** than TP-Only despite the batch size advantage.
3.  **TP+PP with SARATHI:** Leverages the large batch size (27) *and* eliminates bubbles.
    *   Result: SARATHI accelerates the TP+PP setup by **1.91×** compared to the baseline Orca schedule.
    *   Crucially, it also outperforms the **TP-Only** strategy by **1.48×**.

This result is pivotal: it demonstrates that SARATHI makes **Pipeline Parallelism** a superior choice for inference serving even in environments where high-bandwidth interconnects (required for pure Tensor Parallelism) are available. By solving the bubble problem, SARATHI unlocks the memory capacity benefits of PP without the traditional latency penalty.

### 5.6 Critical Assessment of Results

The experiments convincingly support the paper's core claims:
1.  **Decode Efficiency:** The **10×** decode speedup (Figure 8) and **1.2 ms** marginal decode cost (Table 2) provide irrefutable evidence that piggybacking converts memory-bound decodes into compute-bound operations.
2.  **Pipeline Balance:** The **6.29×** reduction in bubbles (Figure 12a) validates the hypothesis that chunked-prefills create the necessary uniformity for efficient pipeline scheduling.
3.  **Generality:** The consistency of results across LLaMA-13B, LLaMA-33B, and GPT-3 suggests the approach is robust to model scale and architecture.

**Limitations and Conditions:**
*   **P:D Dependency:** The benefits are conditional on the P:D ratio. While robust, the system does not offer uniform gains if the workload is extremely skewed (e.g., almost exclusively decoding with no new prompts). In such edge cases, the system reverts to baseline behavior.
*   **Prefill Overhead:** The ablation study (Figure 13) admits that chunking introduces overheads (KV cache reloads). The system relies on the decode gains to *overcompensate* for this prefill penalty. If a workload has zero decodes (pure prefill benchmark), SARATHI would theoretically be slower than a monolithic prefill due to these overheads, though such workloads are rare in interactive serving.
*   **Simulation Reliance:** The GPT-3 results rely on simulation. While the authors calibrated the simulator to within 5% of empirical data, real-world network jitter and contention in a 64-GPU cluster could introduce variances not captured in the model.

In conclusion, the experimental analysis demonstrates that SARATHI is not merely an incremental tweak but a structural optimization that aligns LLM inference patterns with GPU hardware constraints. The data clearly shows that by sacrificing a small amount of prefill efficiency (via chunking), the system gains massive improvements in decode throughput and pipeline utilization, resulting in superior end-to-end performance.

## 6. Limitations and Trade-offs

While SARATHI demonstrates significant throughput gains, its design relies on specific assumptions about workload characteristics and hardware behavior. Understanding these constraints is critical for determining where the system excels and where it may falter. The paper explicitly acknowledges several limitations in **Section 6 (Discussion)** and implicitly reveals others through its ablation studies and experimental setup.

### 6.1 Dependence on Workload Composition (The P:D Ratio Constraint)
The most fundamental limitation of SARATHI is its reliance on a balanced mix of prefill and decode tokens. The core mechanism—"piggybacking" decodes onto prefill chunks—requires a steady stream of new prompts to act as the "vehicle" for inefficient decodes.

*   **The "Decode-Only" Cliff:** If the system encounters a workload with an extremely low Prefill-to-Decode (P:D) ratio (e.g., a scenario where users submit one short prompt and request thousands of generated tokens, or a period of inactivity where no new prompts arrive), SARATHI runs out of prefill chunks. In this state, the system must revert to processing decode-only batches, which suffer from the same memory-bound inefficiencies as the baseline. As noted in **Section 5.1.3**, when the P:D ratio is too low, "SARATHI either runs out of prefill tokens... [and] can switch to... standard baseline processing." The performance gains vanish in these extremes.
*   **The "Prefill-Only" Inefficiency:** Conversely, in workloads dominated by massive prefills with minimal generation (high P:D ratio), the system generates more prefill chunks than there are decode requests to fill the hybrid batches. While the prefill chunks themselves are compute-efficient, the opportunity to amortize the cost of weight fetching across decodes is lost. **Figure 9** shows that while performance remains robust (>10% gain) across a wide range, the peak efficiency is strictly tied to the condition $\text{P:D} \approx C/(B-1)$.

### 6.2 The Prefill Overhead Trade-off
SARATHI intentionally sacrifices raw prefill efficiency to gain decode efficiency. This is a calculated trade-off, but it introduces a tangible penalty that must be managed.

*   **KV Cache Reload Penalty:** As detailed in **Section 4.2** and **Section 5.4**, splitting a prompt into $N$ chunks forces the system to re-read the Key-Value (KV) cache of earlier tokens from global memory for every subsequent chunk. For a prompt split into $N$ chunks, the KV pairs of the first chunk are loaded $N$ times.
*   **Quantifiable Loss:** **Figure 13b** shows that for very small chunk sizes (e.g., 64 tokens), the prefill runtime can increase by up to **5×** compared to a monolithic prefill. Even with optimized chunk sizes (256 or 512), the paper admits to a prefill efficiency loss of **10%–20%**.
*   **Reliance on Decode Dominance:** The system's net positive gain depends entirely on the decode phase being the dominant bottleneck. If a specific application has very short generation lengths (e.g., classification tasks or single-token completion), the overhead of chunked-prefills might outweigh the benefits of piggybacking, potentially making SARATHI slower than a baseline system that processes prefills monolithically. The authors note in **Section 5.1.2** that end-to-end speedups are capped at ~1.33× precisely because the prefill phase itself is not accelerated, only the decode phase.

### 6.3 Hardware Sensitivity and Tile Quantization
SARATHI's performance is tightly coupled to low-level GPU architectural details, specifically matrix multiplication tile sizes. This creates a portability challenge.

*   **Non-Linear Performance Cliffs:** As illustrated in **Figure 7** and discussed in **Section 4.4**, GPU performance does not scale linearly with sequence length due to tile quantization. Adding a single token beyond a tile boundary (e.g., going from 128 to 129) can cause a **32% spike** in runtime.
*   **Configuration Complexity:** To avoid these cliffs, SARATHI must dynamically adjust chunk sizes so that the total dimension of the fused batch aligns with the hardware tile size (e.g., multiples of 128). This implies that the "optimal" chunk size is not universal; it varies by GPU architecture (A6000 vs. A100), model hidden dimension, and even the current batch size. **Section 6** highlights this as an open challenge: "we leave it to future work to explore how to pick an optimal chunk size as it depends on several factors like the hardware, model characteristics... especially in scenarios where the P:D ratio may not be known ahead of time." A static configuration could lead to suboptimal performance on different hardware.

### 6.4 Simplifying Assumptions in Evaluation
The experimental validation, while rigorous, relies on several simplifications that may not hold in complex production environments.

*   **Uniform Sequence Lengths:** With the exception of the GPT-3 simulation, the single-GPU experiments in **Section 5.1** assume that "each request in a batch has the same number of prefill and decode tokens" (**Section 6**). In real-world serving, request lengths follow a heavy-tailed distribution. While SARATHI's chunking helps mitigate variance, the interaction between highly variable sequence lengths and the fixed-size chunk scheduler requires further validation. The paper admits: "in the real world, the sequence lengths can vary significantly across different LLM inference requests."
*   **Simulation vs. Reality for Scale:** The most impressive results (1.91× speedup) come from a **simulated** 64-GPU cluster for GPT-3 (**Section 5.3**). While the authors calibrated their simulator to within 5% of empirical 8-GPU data, large-scale distributed systems introduce complexities like network jitter, contention, and synchronization delays that are difficult to model perfectly. The claim that SARATHI makes Pipeline Parallelism superior to Tensor Parallelism relies on this simulation holding true at scale.
*   **Limited Context Window Scope:** The evaluation focuses on sequence lengths up to **3K tokens** and P:D ratios up to 200. The authors explicitly flag the emerging trend of "very long sequences (e.g., 10s-100s of thousands)" as a potential weakness (**Section 6**). Since attention complexity grows quadratically ($O(L^2)$), the cost of the attention mechanism in chunked-prefills (which requires repeated reads of growing KV caches) could become prohibitive at extreme context lengths, potentially eroding the gains from linear layer fusion.

### 6.5 Scheduling Complexity and Latency Trade-offs
Finally, SARATHI optimizes for **throughput** (tokens per second) rather than **latency** (time to first token or time per request).

*   **Increased Time-to-First-Token (TTFT):** By chunking a prefill, the system delays the completion of the full prompt processing. A user with a 1024-token prompt must wait for four sequential chunk iterations before the model can begin generating coherent output based on the full context. While the paper focuses on throughput, this fragmentation inherently increases the latency for the initial response compared to a monolithic prefill that processes the whole prompt in one go.
*   **Scheduling Overhead:** The logic required to dynamically split prompts, track chunk progress, align tile sizes, and construct hybrid batches adds computational overhead to the CPU scheduler. While not quantified in the paper, this complexity increases the risk of scheduling bottlenecks in high-frequency trading or ultra-low-latency applications where microsecond-level precision is required.

In summary, SARATHI is not a silver bullet for all inference scenarios. It is a specialized optimization for **throughput-oriented serving** of models with **moderate context lengths** and **balanced prefill/decode workloads**. Its effectiveness diminishes in decode-heavy starvation scenarios, prefill-only bursts, or extreme long-context settings, and it introduces configuration complexity due to its sensitivity to hardware tile alignment.

## 7. Implications and Future Directions

SARATHI represents a fundamental shift in how the systems community approaches LLM inference optimization. By moving beyond memory management tweaks and kernel-level fusion, it demonstrates that **structural re-architecting of the workload itself**—specifically, fragmenting monolithic prefills to create uniform, hybrid compute units—can unlock order-of-magnitude efficiency gains. This section explores how this work reshapes the landscape of distributed serving, outlines critical avenues for future research, and provides practical guidance for integrating these techniques into real-world infrastructure.

### 7.1 Reshaping the Inference Landscape: The Viability of Pipeline Parallelism
The most profound implication of SARATHI is that it rehabilitates **Pipeline Parallelism (PP)** as a first-class citizen for inference serving, challenging the dominance of Tensor Parallelism (TP) in high-performance clusters.

*   **Breaking the Bubble Barrier:** Historically, PP has been avoided for inference due to **pipeline bubbles**—idle GPU cycles caused by the variable execution times of prefills and decodes. As shown in **Figure 12b**, standard PP with iteration-level scheduling (Orca) was actually *slower* than TP-only deployments despite supporting larger batch sizes, purely because of these bubbles. SARATHI's **chunked-prefills** eliminate this variance, reducing bubble time by **6.29×**.
*   **Enabling Cross-Node Scaling:** By making PP efficient, SARATHI allows models to be scaled across nodes connected by standard interconnects (like InfiniBand) rather than requiring expensive, high-bandwidth NVLink fabrics necessary for TP. The simulation results indicate that SARATHI-enabled PP can outperform TP-only setups by **1.48×** in end-to-end throughput. This suggests a future where massive LLMs can be served cost-effectively on commodity clusters, significantly lowering the barrier to entry for large-scale deployment.
*   **From "Memory-Bound" to "Compute-Bound":** The paper fundamentally alters the understanding of the decode phase. Prior wisdom held that decoding was inevitably memory-bound unless batch sizes were impractically large. SARATHI proves that by **piggybacking** decodes onto compute-saturating prefill chunks, the decode phase can be transformed into a compute-bound operation without increasing the physical batch size. This decouples efficiency from memory capacity, allowing smaller GPUs to achieve throughput previously thought possible only on larger devices.

### 7.2 Avenues for Future Research
SARATHI opens several critical research directions, particularly in adapting its static design choices to dynamic, real-world environments.

*   **Dynamic Chunk Size Adaptation:** Currently, SARATHI relies on a fixed chunk size ($C$) chosen based on an estimated Prefill-to-Decode (P:D) ratio. However, real-world workloads are non-stationary; a chatbot might experience bursts of short prompts followed by long summarization tasks. Future work must develop **online controllers** that dynamically adjust $C$ in real-time based on the observed queue depth of prefills vs. decodes. The goal is to maintain the optimal condition $\text{P:D} \approx C/(B-1)$ continuously, rather than statically.
*   **Handling Extreme Context Lengths:** The evaluation in SARATHI is limited to sequence lengths up to **3K tokens**. With the industry shifting toward context windows of **100K+ tokens** (e.g., for document analysis), the overhead of **chunked-prefills** becomes a major concern. As noted in **Section 6**, the repeated reading of KV caches for earlier chunks scales poorly with sequence length due to the quadratic nature of attention. Future research must integrate SARATHI with **linear attention mechanisms** or **sparse attention** patterns to ensure that the KV cache reload overhead does not negate the benefits of piggybacking at extreme scales.
*   **Latency-Aware Scheduling:** SARATHI optimizes strictly for **throughput** (tokens/sec). However, chunking a prefill inherently increases the **Time-To-First-Token (TTFT)** latency, as the full prompt must be processed in sequential chunks before generation can begin coherently. A critical follow-up direction is developing **hybrid schedulers** that can toggle between "monolithic prefill" mode (for low-latency interactive users) and "chunked prefill" mode (for high-throughput background batch processing), effectively creating a quality-of-service (QoS) hierarchy.
*   **Co-Design with Memory Management Systems:** SARATHI assumes a pre-allocated KV cache. Integrating its chunking logic with paged memory managers like **vLLM** [20] is a natural next step. Research is needed to determine how chunk boundaries interact with memory paging: does splitting a prompt disrupt memory locality or complicate the page table management? Optimizing the interaction between *logical* chunking and *physical* memory paging could yield further gains.

### 7.3 Practical Applications and Downstream Use Cases
The specific strengths of SARATHI make it uniquely suited for several high-value application domains:

*   **High-Throughput Batch Inference:** For offline tasks such as dataset generation, model distillation, or bulk content moderation, where latency is secondary to total completion time, SARATHI's ability to maximize GPU utilization makes it the ideal scheduler. The **1.91×** speedup in pipeline settings translates directly to reduced cloud compute costs for these massive jobs.
*   **Commodity Cluster Deployments:** Organizations that cannot afford NVIDIA DGX superpods with NVLink can now deploy large models (e.g., LLaMA-33B, GPT-3 scale) across standard server clusters using Pipeline Parallelism. SARATHI ensures that the lack of high-bandwidth interconnects does not result in wasted GPU cycles, making enterprise-grade LLM serving more accessible.
*   **Mixed-Workload Serving Platforms:** Cloud providers offering LLM APIs often face a mix of short queries (chat) and long completions (code generation). SARATHI's robustness across a wide range of P:D ratios (**Figure 9**) suggests it can handle these heterogeneous workloads more gracefully than rigid schedulers, maintaining high utilization even as the mix of user requests fluctuates throughout the day.

### 7.4 Reproducibility and Integration Guidance
For practitioners and researchers looking to adopt or build upon SARATHI, the following guidelines clarify when and how to apply these techniques:

*   **When to Prefer SARATHI:**
    *   **Scenario:** You are operating a serving infrastructure where **throughput** is the primary metric (e.g., batch processing, high-volume API endpoints) and the workload contains a mix of new prompts and ongoing generations.
    *   **Hardware:** You are using **Pipeline Parallelism** across multiple GPUs or nodes, especially if interconnect bandwidth is limited (preventing efficient Tensor Parallelism).
    *   **Constraint:** Your GPU memory is tight, preventing you from scaling batch sizes to the point where decoding becomes naturally compute-bound (i.e., you cannot reach $B \ge 256$).

*   **When to Stick with Baselines (Orca/vLLM):**
    *   **Latency-Critical Apps:** If your Service Level Agreement (SLA) demands minimal **Time-To-First-Token (TTFT)**, the sequential nature of chunked-prefills may introduce unacceptable latency. In these cases, monolithic prefills (standard Orca/vLLM) are preferable.
    *   **Decode-Only Workloads:** If your application involves continuing existing conversations with almost no new system prompts (extremely low P:D ratio), SARATHI loses its "vehicle" for piggybacking and reverts to baseline performance, potentially adding scheduling overhead without benefit.
    *   **Extreme Context (>10K):** Until the KV cache reload overhead is optimized for long sequences, standard attention implementations may be more stable for very long-context tasks.

*   **Integration Checklist:**
    1.  **Profile Tile Sizes:** Before deploying, profile your specific GPU architecture (A100, H100, A6000) to determine the optimal **matrix tile size** (often 128 or 256). As shown in **Figure 7**, misalignment here causes massive performance cliffs.
    2.  **Estimate P:D Ratio:** Analyze your historical logs to estimate the average ratio of input tokens to output tokens. Use the formula $C \approx \text{P:D} \times (B-1)$ to select an initial chunk size.
    3.  **Kernel Fusion:** Ensure your inference engine supports fusing linear layers (`preproj`, `ffn`) across heterogeneous batch elements (prefill tokens + decode tokens). SARATHI cannot be implemented with standard libraries that enforce uniform sequence lengths within a batch without modification to the kernel launch logic.
    4.  **Attention Masking:** Implement the progressive attention masking strategy described in **Figure 6** carefully. Incorrect masking will break the causal chain, leading to hallucinated or nonsensical outputs.

In conclusion, SARATHI provides a blueprint for the next generation of inference engines: systems that do not just passively schedule requests, but actively **reshape** them to match the physical realities of the underlying hardware. By treating the GPU not as a black box but as a tiled, bandwidth-constrained architecture, SARATHI turns the inherent irregularity of LLM workloads into a structured, high-efficiency stream.
