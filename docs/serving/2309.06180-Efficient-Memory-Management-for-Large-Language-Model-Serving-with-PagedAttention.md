## 1. Executive Summary

This paper addresses the critical memory bottleneck in Large Language Model (LLM) serving caused by inefficient Key-Value (KV) cache management in existing systems like Orca and FasterTransformer, where fragmentation leads to only 20.4%–38.2% effective memory usage. The authors introduce `PagedAttention`, an attention algorithm inspired by operating system paging that allows non-contiguous KV cache storage, and implement it in `vLLM`, a serving engine that achieves near-zero memory waste and enables flexible memory sharing across sequences. Evaluations on OPT (13B–175B) and LLaMA models using ShareGPT and Alpaca datasets demonstrate that `vLLM` improves serving throughput by 2-4× compared to state-of-the-art systems while maintaining equivalent latency.

## 2. Context and Motivation

### 2.1 The LLM Serving Challenge

Large Language Models (LLMs) like GPT and PaLM have enabled transformative applications including programming assistants and universal chatbots. However, running these applications is extraordinarily expensive. The paper notes that "processing an LLM request can be 10× more expensive than a traditional keyword query" (Section 1). This cost disparity stems from the fundamental architecture of Transformer-based language models.

At the core of LLMs lies an **autoregressive Transformer model** (Section 2.1). This model generates tokens one at a time, where each new token depends on all previously generated tokens. Mathematically, the joint probability over a sequence $(x_1, ..., x_n)$ is factorized as:

$$P(x) = P(x_1) \cdot P(x_2 | x_1) \cdots P(x_n | x_1, ..., x_{n-1})$$

This sequential dependency creates a critical bottleneck: the **Key-Value cache (KV cache)**. During generation, the model must store the key and value vectors from all previous tokens to compute attention for the next token. For a self-attention layer, the attention output $o_i$ at position $i$ is computed as:

$$o_i = \sum_{j=1}^{i} a_{ij} v_j$$

where $a_{ij}$ represents attention scores between query $q_i$ and all previous keys $k_j$. Since $v_j$ (value vectors) from all prior positions must be accessible, they are cached in GPU memory.

### 2.2 Why Memory Management Matters

The memory footprint of LLM serving is dominated by two components (Figure 1, left):

- **Model parameters**: Approximately 65% of GPU memory (e.g., 26GB for a 13B parameter model on NVIDIA A100 40GB). These remain static throughout serving.
- **KV cache**: Over 30% of GPU memory, dynamically allocated per request. This is the critical variable component.
- **Activations**: A small ephemeral fraction used during computation.

Since parameters are fixed and activations are minimal, "the way the KV cache is managed is critical in determining the maximum batch size" (Section 1). When KV cache memory is managed inefficiently, it directly limits how many requests can be processed simultaneously, which caps throughput.

The problem is compounded by hardware trends. The paper observes that "GPU's computation speed grows faster than the memory capacity" (Section 3). For example, from NVIDIA A100 to H100, FLOPS increases by more than 2×, but GPU memory stays at 80GB maximum. This widening gap means memory will become "an increasingly significant bottleneck" for LLM serving.

### 2.3 The Memory Fragmentation Problem

Existing LLM serving systems face three specific types of memory waste when managing KV cache (Figure 3):

1. **Reserved slots for future tokens**: Systems pre-allocate memory for the maximum possible sequence length (e.g., 2048 tokens), even though actual outputs may be much shorter. This space is occupied for the entire request lifetime.

2. **Internal fragmentation**: When a request's actual length is much shorter than its maximum length, the pre-allocated contiguous chunk contains unused slots. For example, Figure 3 shows Request A with 2038 slots never used due to internal fragmentation.

3. **External fragmentation**: Different requests have different pre-allocated sizes, leading to gaps in memory that cannot be utilized by other requests (similar to the buddy allocator problem in operating systems).

The cumulative impact is severe. Figure 2 quantifies memory waste across different systems:

| System | KV Cache Usage (%) |
|--------|-------------------|
| Orca (Max) | 20.4% |
| Orca (Pow2) | 13.3% - 38.2% |
| Orca (Oracle) | 57.3% |
| vLLM | 96.3% |

> "Our profiling results in Fig. 2 show that only 20.4% - 38.2% of the KV cache memory is used to store the actual token states in the existing systems" (Section 1).

This means 60-80% of allocated KV cache memory is pure waste, directly limiting batch size and throughput.

### 2.4 Prior Approaches and Their Limitations

**FasterTransformer** (Section 6.1): A distributed inference engine highly optimized for latency. However, it lacks fine-grained scheduling and manages memory inefficiently similar to Orca (Max). The paper notes FasterTransformer "does not utilize a fine-grained scheduling mechanism and inefficiently manages the memory" (Section 6.2).

**Orca** (Section 6.1): A state-of-the-art LLM serving system optimized for throughput using iteration-level scheduling. Orca introduces fine-grained batching where "after each iteration, completed requests are removed from the batch, and new ones are added" (Section 2.3). However, Orca still stores KV cache in contiguous memory space, requiring pre-allocation based on maximum sequence length. The authors implemented three Orca variants for evaluation:

- **Orca (Oracle)**: Assumes knowledge of actual output lengths (upper-bound, infeasible in practice)
- **Orca (Pow2)**: Over-reserves space by at most 2× (e.g., 25 tokens → 32 positions reserved)
- **Orca (Max)**: Always reserves space up to maximum sequence length (2048 tokens)

Even Orca (Oracle), which has perfect knowledge of output lengths, achieves only 57.3% KV cache usage (Figure 2), indicating fundamental limitations in contiguous memory allocation.

**General Model Serving Systems** (Section 9): Systems like Clipper, TensorFlow Serving, Nexus, InferLine, and Clockwork study batching, caching, and scheduling for general deep learning models. However, "these general systems fail to take into account the autoregressive property and token state of LLM inference, resulting in missed opportunities for optimization" (Section 9).

**Memory Optimization Techniques** (Section 9): Swapping, recomputation, and their combination have been used to reduce peak memory in training. FlexGen studies swapping for LLM inference but "does not target the online serving settings." FlashAttention applies tiling to reduce attention computation memory but does not address block-level memory management for serving.

### 2.5 The Memory Sharing Opportunity

Beyond fragmentation, existing systems cannot exploit **memory sharing opportunities** that arise in complex decoding algorithms (Section 3):

- **Parallel sampling**: When generating multiple outputs from a single prompt, the prompt's KV cache can be shared across all samples. The paper notes this accounts for "12% of the total KV cache memory in our experiment" (Section 3).

- **Beam search**: Different beam candidates can share larger portions of KV cache (up to 55% memory saving, Section 6.3), and sharing patterns evolve dynamically as decoding advances.

- **Shared prefix**: Many applications use system prompts or few-shot examples that are common across requests. Figure 10 illustrates how translation tasks share instruction prefixes.

Existing systems cannot exploit these opportunities because "the KV cache of the sequences is stored in separate contiguous spaces" (Section 1). Memory sharing requires the ability to map multiple logical sequences to the same physical memory blocks, which contiguous allocation prevents.

### 2.6 Positioning This Work

This paper positions itself at the intersection of three research areas:

1. **Operating Systems**: The authors explicitly draw inspiration from "virtual memory and paging techniques in operating systems" (Abstract). The key insight is treating KV cache blocks like OS pages, logical blocks like virtual addresses, and physical blocks like physical memory pages.

2. **LLM Serving Systems**: Unlike Orca which focuses on scheduling, vLLM focuses on memory management. The paper states: "While both systems aim to increase the GPU utilization and hence the throughput of LLM serving, Orca achieves it by scheduling and interleaving the requests so that more requests can be processed in parallel, while vLLM is doing so by increasing memory utilization so that the working sets of more requests fit into memory" (Section 9).

3. **Attention Algorithms**: PagedAttention is presented as "an attention algorithm that operates on KV cache stored in non-contiguous paged memory" (Section 1). This distinguishes it from FlashAttention, which optimizes attention computation I/O but does not change memory management semantics.

The paper's core contribution is recognizing that LLM serving has unique properties making it particularly suited for OS-style virtual memory:

- Dynamic memory allocation (output length unknown a priori)
- Performance bound by GPU memory capacity (not compute)
- All blocks of a sequence accessed together (enables all-or-nothing eviction)
- Application-specific semantics enable optimizations not feasible in general OS (e.g., recomputation for recovery)

> "The idea of virtual memory and paging is effective for managing the KV cache in LLM serving because the workload requires dynamic memory allocation (since the output length is not known a priori) and its performance is bound by the GPU memory capacity" (Section 8).

This positioning clarifies why prior approaches fall short: they either ignore the autoregressive nature of LLMs (general serving systems), optimize scheduling without addressing memory fragmentation (Orca), or focus on training/inference rather than online serving (FlexGen, FlashAttention).

## 3. Technical Approach

This is a systems paper that proposes a novel memory management architecture for LLM serving, with the core idea being that KV cache can be stored in non-contiguous memory blocks (like OS pages) rather than contiguous tensors, enabling dynamic allocation and memory sharing without modifying the model's mathematical correctness.

### 3.1 Reader orientation (approachable technical breakdown)

vLLM is a serving engine that manages GPU memory for Large Language Model inference requests using a paging system inspired by operating system virtual memory. It solves the problem of memory fragmentation and waste in existing LLM serving systems by allowing the KV cache to be stored in non-contiguous blocks that can be allocated on-demand and shared across multiple sequences, rather than pre-allocating large contiguous chunks for each request.

### 3.2 Big-picture architecture (diagram in words)

The vLLM system consists of five major components that work together in a centralized control and distributed execution pattern (Figure 4):

- **Centralized Scheduler**: Receives incoming requests, maintains the queue, and makes batching decisions at each iteration; it broadcasts control messages containing token IDs and block tables to all GPU workers.
- **KV Cache Manager**: Maintains the mapping between logical KV blocks (what each sequence thinks it has) and physical KV blocks (where data actually lives in GPU memory); it tracks reference counts for shared blocks and handles allocation/deallocation.
- **GPU Block Allocator**: Manages a contiguous chunk of GPU DRAM divided into fixed-size physical KV blocks; it allocates and frees blocks based on KV Cache Manager instructions.
- **CPU Block Allocator**: Manages CPU RAM as swap space for evicted KV blocks when GPU memory is exhausted; it handles block transfers between CPU and GPU during preemption.
- **GPU Workers (Model Shards + Cache Engines)**: Multiple workers execute model layers in parallel using tensor model parallelism; each worker stores a portion of the KV cache for its assigned attention heads and executes the PagedAttention kernel using the block table provided by the scheduler.

The information flow proceeds as follows: First, the scheduler receives requests and the KV Cache Manager allocates physical blocks for the prompt phase. Second, at each decoding iteration, the scheduler selects which sequences to batch, updates block tables for newly generated tokens, and broadcasts token IDs plus block tables to all workers. Third, GPU workers execute the model layers, reading KV cache from physical blocks according to the block table, computing attention via PagedAttention, and writing new KV cache to newly allocated physical blocks. Fourth, workers send sampled tokens back to the scheduler, which updates sequence states and repeats until all sequences complete.

### 3.3 Roadmap for the deep dive

- **PagedAttention algorithm**: We start with the core algorithmic innovation because everything else builds on the ability to compute attention over non-contiguous KV blocks.
- **KV Cache Manager and block tables**: Next we explain how logical-to-physical block mapping is maintained, as this is the data structure that enables PagedAttention to work.
- **Memory allocation and dynamic growth**: We then describe how blocks are allocated on-demand during decoding, which eliminates pre-allocation waste.
- **Memory sharing mechanisms**: We cover copy-on-write and reference counting, which enable parallel sampling and beam search without duplication.
- **Scheduling and preemption policies**: We explain how the system handles memory pressure through swapping and recomputation.
- **Distributed execution**: Finally we describe how the architecture scales across multiple GPUs while maintaining a unified memory view.

### 3.4 Detailed, sentence-based technical breakdown

#### 3.4.1 The PagedAttention Algorithm

PagedAttention is an attention algorithm that allows key and value vectors to be stored in non-contiguous memory blocks while producing mathematically identical output to standard attention. In traditional self-attention, the query vector at position $i$ attends to all previous key-value pairs stored in a contiguous tensor, but PagedAttention partitions this tensor into fixed-size blocks that can reside anywhere in physical memory.

The key mathematical transformation is from position-wise attention to block-wise attention. In standard attention, the attention score $a_{ij}$ between query at position $i$ and key at position $j$ is computed as:

$$a_{ij} = \frac{\exp(q_i^\top k_j / \sqrt{d})}{\sum_{t=1}^{i} \exp(q_i^\top k_t / \sqrt{d})}$$

where $q_i$ is the query vector, $k_j$ is the key vector, and $d$ is the hidden dimension. The output $o_i$ is then the weighted sum of value vectors:

$$o_i = \sum_{j=1}^{i} a_{ij} v_j$$

PagedAttention transforms this by grouping tokens into blocks of size $B$. If we denote the $j$-th key block as $K_j = (k_{(j-1)B+1}, ..., k_{jB})$ and value block as $V_j = (v_{(j-1)B+1}, ..., v_{jB})$, then the block-wise attention score $A_{ij}$ (a row vector of scores for all tokens in block $j$) is:

$$A_{ij} = \frac{\exp(q_i^\top K_j / \sqrt{d})}{\sum_{t=1}^{\lceil i/B \rceil} \exp(q_i^\top K_t 1 / \sqrt{d})}$$

where $1$ represents a vector of ones for proper dimension matching. The output is then computed as:

$$o_i = \sum_{j=1}^{\lceil i/B \rceil} V_j A_{ij}^\top$$

This formulation is mathematically equivalent to standard attention but allows the kernel to fetch KV blocks from arbitrary physical memory locations rather than requiring contiguous storage.

The PagedAttention kernel operates by first reading the block table to identify which physical blocks contain the KV cache for the current sequence. For each block, the kernel fetches the key and value vectors, computes attention scores with the current query, and accumulates the weighted values. The kernel must handle variable sequence lengths within a batch because different sequences may have different numbers of filled blocks. This is achieved by tracking the number of filled positions in each logical block within the block table.

The design choice to use block size $B > 1$ (rather than one token per block) is critical for performance. If each block contained only one token, the kernel would need to perform one memory fetch per token, which would severely underutilize GPU memory bandwidth. By grouping multiple tokens into a block, the kernel can process more positions in parallel within each memory transaction. However, larger block sizes increase internal fragmentation because the last block of a sequence may not be fully filled. The paper evaluates this tradeoff in Section 7.2 and finds that block size 16 provides the best balance: it is "large enough to efficiently utilize the GPU and small enough to avoid significant internal fragmentation in most workloads." Consequently, vLLM sets its default block size to 16 tokens.

#### 3.4.2 KV Cache Manager and Block Tables

The KV Cache Manager is the central component that maintains the mapping between logical KV blocks (what each sequence believes it owns) and physical KV blocks (where data actually resides in GPU memory). This separation of logical and physical addresses is directly inspired by operating system virtual memory, where a process's virtual address space can map to non-contiguous physical RAM pages.

Each sequence maintains a block table that records two pieces of information for each logical block: (1) the physical block ID where that logical block's data is stored, and (2) the number of filled positions within that block. For example, if a sequence has generated 7 tokens and the block size is 4, then the block table would show 2 logical blocks: logical block 0 with 4 filled positions and logical block 1 with 3 filled positions. The physical block IDs might be 7 and 1 respectively, indicating that these logical blocks are stored in non-contiguous physical memory locations (Figure 6).

The block table is updated dynamically as new tokens are generated. During the prompt phase, vLLM allocates enough physical blocks to store the KV cache for all prompt tokens plus one additional slot for the first generated token. For a prompt with 7 tokens and block size 4, vLLM allocates 2 physical blocks (since $\lceil 7/4 \rceil = 2$ logical blocks are needed). The block table entries are populated with the assigned physical block IDs, and the filled position counts are set based on how many tokens are stored in each block.

During autoregressive decoding, the process proceeds iteratively. At each iteration, vLLM first checks whether the last logical block has available slots. If the last block is not full (e.g., 3 out of 4 positions filled), the newly generated token's KV cache is written to the next available slot in the existing physical block, and the filled position count is incremented. If the last block is full, vLLM allocates a new physical block from the GPU Block Allocator, adds a new entry to the block table mapping the new logical block to this physical block, and writes the KV cache to the first position of the new block.

This on-demand allocation strategy eliminates the pre-allocation waste present in existing systems. In Orca and FasterTransformer, memory for the maximum sequence length (e.g., 2048 tokens) is reserved at request arrival time, regardless of actual output length. In vLLM, "vLLM does not require reserving the memory for the maximum possible generated sequence length initially" (Section 4.3). Instead, physical blocks are allocated only when needed, which means memory waste is limited to at most one partially-filled block per sequence.

The block table is broadcast to all GPU workers at each decoding iteration along with the token IDs. This allows workers to know exactly which physical blocks to read for attention computation. Although each worker receives the same block table, each worker only stores a portion of the KV cache corresponding to its assigned attention heads (in tensor model parallelism). The KV Cache Manager maintains a single global view of block allocation across all workers, ensuring consistency.

#### 3.4.3 Memory Allocation and Dynamic Growth

The GPU Block Allocator manages a contiguous chunk of GPU DRAM that is divided into fixed-size physical KV blocks. At system initialization, the allocator reserves this chunk and divides it into blocks, each capable of storing the KV cache for $B$ tokens across all layers and attention heads. For a 13B parameter OPT model, the paper calculates that "the KV cache of a single token demands 800 KB of space, calculated as 2 (key and value vectors) × 5120 (hidden state size) × 40 (number of layers) × 2 (bytes per FP16)" (Section 3). With block size 16, each physical block therefore requires approximately 12.8 MB of GPU memory.

When a new request arrives, the KV Cache Manager first determines how many logical blocks are needed for the prompt phase. This is computed as $\lceil \text{prompt\_length} / B \rceil$. The manager then requests this many physical blocks from the GPU Block Allocator. The allocator returns available physical block IDs from its free list, and the KV Cache Manager creates the initial block table entries.

As decoding proceeds and sequences grow, new physical blocks are allocated on-demand. The allocator maintains a free list of available physical block IDs. When a sequence needs a new block, the allocator removes an ID from the free list and assigns it to that sequence. When a sequence completes (either by generating an end-of-sequence token or reaching maximum length), all its physical blocks are returned to the free list for reuse by other sequences.

This dynamic allocation strategy enables vLLM to achieve near-zero memory waste. Figure 2 shows that vLLM achieves 96.3% KV cache usage compared to 20.4%-57.3% for existing systems. The remaining 3.7% waste comes from two sources: (1) the partially-filled last block of each sequence (internal fragmentation), and (2) small overhead for block table metadata. Since all blocks have the same size, there is no external fragmentation, unlike the buddy allocator used in existing systems.

The CPU Block Allocator operates similarly but manages CPU RAM instead of GPU DRAM. It is used when GPU memory is exhausted and sequences need to be preempted. The CPU allocator maintains its own free list of CPU memory blocks that can hold evicted KV cache. When a sequence is swapped out, its physical blocks are copied from GPU to CPU, and the GPU blocks are returned to the GPU free list. When the sequence is swapped back in, new GPU blocks are allocated and the data is copied from CPU to GPU.

#### 3.4.4 Memory Sharing Mechanisms

One of the most powerful features enabled by PagedAttention is the ability to share physical KV blocks across multiple logical sequences. This is critical for complex decoding algorithms like parallel sampling and beam search, where multiple output sequences share common prefixes.

The mechanism for sharing is reference counting. Each physical block maintains a reference count that tracks how many logical blocks (across all sequences) map to it. When a physical block is first allocated, its reference count is 1. When a new sequence is created that should share an existing block (e.g., during parallel sampling where multiple outputs share the same prompt), the new sequence's block table is configured to point to the same physical block ID, and the reference count is incremented.

For example, in parallel sampling with 2 output sequences from the same prompt (Figure 8), both sequences share the physical blocks containing the prompt's KV cache. If the prompt requires 2 logical blocks, both sequences' block tables map logical blocks 0 and 1 to the same physical blocks (e.g., physical blocks 7 and 1). The reference counts for these physical blocks are both 2, indicating they are shared by two sequences.

The copy-on-write mechanism handles the case where a shared block needs to be modified. When a sequence attempts to write to a physical block whose reference count is greater than 1, vLLM recognizes that the block is shared and cannot be modified in-place. Instead, vLLM allocates a new physical block, copies the data from the shared block to the new block, updates the sequence's block table to point to the new block, and decrements the reference count of the original shared block. The sequence then writes its new KV cache to the newly allocated block.

This copy-on-write approach minimizes memory copying overhead. In the parallel sampling example, when sequence A1 needs to generate its first output token, it writes to the last logical block (which is shared). vLLM detects the reference count of 2, allocates a new physical block (e.g., physical block 3), copies the data from physical block 1 to physical block 3, decrements the reference count of physical block 1 to 1, and updates A1's block table to map logical block 1 to physical block 3. Sequence A2 can then continue using physical block 1 directly since its reference count is now 1.

Beam search involves more complex sharing patterns because different beam candidates may share different prefixes at different iterations (Figure 9). At each iteration, beam search retains the top-$k$ most probable sequences out of $k \times |V|$ candidates (where $|V|$ is vocabulary size). Some candidates may share long prefixes while others diverge early. vLLM handles this by dynamically updating block table mappings and reference counts as candidates are pruned or extended.

When a beam candidate is pruned (no longer in the top-$k$), its logical blocks are freed, and the reference counts of the corresponding physical blocks are decremented. If a physical block's reference count reaches 0, it is returned to the free list. When new candidates are created (by extending existing candidates), new block table entries are created, potentially sharing physical blocks with parent candidates. The paper notes that in beam search with width 6, vLLM achieves 55.2% memory saving on the Alpaca dataset (Figure 15b) compared to not sharing, demonstrating the effectiveness of this mechanism.

Shared prefix optimization extends this sharing across different requests, not just within a single request. For applications where many requests share a common system prompt or few-shot examples (Figure 10), vLLM can pre-cache the KV cache for the shared prefix in physical blocks. When a new request arrives with the same prefix, its block table is configured to map the prefix logical blocks to the pre-cached physical blocks (with the last block marked for copy-on-write). This avoids recomputing the KV cache for the shared prefix, reducing both memory usage and computation time. The paper shows in Section 6.4 that with a 5-shot prefix (341 tokens), vLLM achieves 3.58× higher throughput than Orca (Oracle).

#### 3.4.5 Scheduling and Preemption Policies

When request traffic exceeds system capacity, vLLM must decide which requests to serve and which to preempt. The system adopts a First-Come-First-Serve (FCFS) scheduling policy to ensure fairness and prevent starvation. This means earlier-arrived requests are served first, and later-arrived requests are preempted first when memory pressure occurs.

The preemption decision is triggered when the GPU Block Allocator exhausts its free list and cannot allocate new physical blocks for newly generated tokens. At this point, vLLM must evict some sequences to free up GPU memory. The paper implements an "all-or-nothing" eviction policy: either all blocks of a sequence are evicted, or none are. This policy exploits the fact that "all blocks of a sequence are accessed together" during attention computation (Section 8). Evicting only part of a sequence's blocks would require complex dependency tracking and provide minimal memory relief.

Furthermore, multiple sequences within one request (e.g., beam candidates in beam search) are "gang-scheduled" as a sequence group. This means all sequences in the group are preempted or rescheduled together. This is necessary because of memory sharing across sequences within the group: evicting one sequence's blocks while keeping others would break the sharing relationships and corrupt the reference counts.

vLLM supports two techniques for recovering evicted blocks when preempted sequences are rescheduled:

**Swapping** copies evicted blocks from GPU to CPU RAM, similar to OS virtual memory paging to disk. When GPU memory is exhausted, vLLM selects sequences to evict based on the FCFS policy (latest arrivals first). The KV cache for these sequences is copied to CPU RAM managed by the CPU Block Allocator. The GPU physical blocks are then freed and can be allocated to other sequences. When a request completes and frees its GPU blocks, preempted sequences are swapped back in: new GPU blocks are allocated, and the KV cache is copied from CPU to GPU. The paper notes that "the number of blocks swapped to the CPU RAM never exceeds the number of total physical blocks in the GPU RAM, so the swap space on the CPU RAM is bounded by the GPU memory allocated for the KV cache" (Section 4.5).

**Recomputation** regenerates the KV cache by re-running the prompt phase computation. Instead of copying evicted blocks from CPU, vLLM simply recomputes the KV cache when the sequence is rescheduled. This is feasible because "the tokens generated at decoding can be concatenated with the original user prompt as a new prompt—their KV cache at all positions can be generated in one prompt phase iteration" (Section 4.5). Recomputation can be faster than swapping for small block sizes because it avoids PCIe bandwidth limitations.

The choice between swapping and recomputation depends on block size and hardware characteristics. Figure 19a shows that swapping incurs excessive overhead with small block sizes (e.g., 1-8 tokens per block) because "small block sizes often result in numerous small data transfers between CPU and GPU, which limits the effective PCIe bandwidth." Recomputation overhead remains constant across block sizes since it does not utilize KV blocks. For medium block sizes (16-64), the two methods exhibit comparable end-to-end performance (Figure 19b). The paper notes that "recomputation overhead is never higher than 20% of swapping's latency" (Section 7.3), making recomputation a viable alternative even when swapping is available.

#### 3.4.6 Distributed Execution

Many LLMs exceed the memory capacity of a single GPU, requiring model parallelism across multiple GPUs. vLLM supports Megatron-LM style tensor model parallelism, where the model's linear layers are partitioned across GPUs and attention heads are split among workers.

In this architecture, each GPU worker stores a portion of the model parameters and a portion of the KV cache. Specifically, "the attention operator is split on the attention head dimension, each SPMD process takes care of a subset of attention heads in multi-head attention" (Section 4.6). Despite this partitioning, all workers process the same input tokens and require KV cache for the same positions. This means the block table mapping is identical across all workers, though each worker only stores the KV cache for its assigned attention heads.

The KV Cache Manager remains centralized in the scheduler, maintaining a single global view of block allocation. At each decoding iteration, the scheduler prepares a control message containing: (1) input token IDs for each request in the batch, and (2) the block table for each request. This control message is broadcast to all GPU workers. Each worker then executes its portion of the model layers, reading KV cache from its local GPU memory according to the block table.

During attention computation, workers synchronize intermediate results using the all-reduce communication primitive (via NCCL). This synchronization happens without scheduler coordination, as in standard tensor parallelism. After completing the iteration, workers send the sampled tokens back to the scheduler, which updates sequence states and prepares the next iteration's control message.

The key insight is that "GPU workers do not need to synchronize on memory management as they only need to receive all the memory management information at the beginning of each decoding iteration along with the step inputs" (Section 4.6). This decouples memory management (handled centrally) from model execution (handled distributively), simplifying the distributed architecture while maintaining efficient memory utilization.

Table 1 shows the model sizes and server configurations used in evaluation. For the 175B parameter model, 8×A100-80GB GPUs are used with 640 GB total GPU memory. Of this, 346 GB is used for parameters, leaving 264 GB for KV cache, which can hold up to 60.1K KV cache slots. This demonstrates that vLLM can scale to very large models while maintaining efficient memory management across distributed GPUs.

#### 3.4.7 Kernel-Level Optimizations

PagedAttention introduces memory access patterns that are not efficiently supported by existing GPU kernels, requiring custom CUDA kernel development. The vLLM engine includes 2K lines of C++/CUDA code specifically for these optimized kernels (Section 5).

Three key kernel optimizations are implemented:

**Fused reshape and block write** combines three operations into a single kernel launch: splitting new KV cache into blocks, reshaping to a memory layout optimized for block read, and saving at positions specified by the block table. In traditional implementations, these would be separate kernel launches, each incurring launch overhead. By fusing them, vLLM minimizes kernel launch overhead and improves memory access coalescing.

**Fusing block read and attention** adapts the attention kernel from FasterTransformer to read KV cache according to the block table and perform attention operations on the fly. To ensure coalesced memory access (where consecutive threads access consecutive memory addresses), "we assign a GPU warp to read each block" (Section 5.1). A GPU warp consists of 32 threads that execute in lockstep, so assigning one warp per block ensures that all threads in the warp access the same block's data together. The kernel also supports variable sequence lengths within a batch, which is necessary because different sequences may have different numbers of filled blocks.

**Fused block copy** handles the copy-on-write mechanism efficiently. Block copy operations may operate on discontinuous blocks (since shared blocks can be scattered in physical memory). Using the standard `cudaMemcpyAsync` API would require numerous invocations for small data movements, incurring significant overhead. Instead, vLLM implements a kernel that "batches the copy operations for different blocks into a single kernel launch" (Section 5.1), reducing launch overhead and improving throughput.

These optimizations do introduce some overhead compared to highly-optimized baselines. Figure 18a shows that PagedAttention attention kernels have 20-26% higher latency than FasterTransformer due to "extra overheads of accessing the block table, executing extra branches, and handling variable sequence lengths" (Section 7.1). However, this overhead is acceptable because it "only affects the attention operator but not the other operators in the model, such as Linear." The end-to-end throughput improvement from better memory utilization far outweighs the per-kernel latency increase, as demonstrated in Section 6.

## 4. Key Insights and Innovations

The contributions of this paper extend beyond incremental system optimizations; they represent a fundamental rethinking of how GPU memory is managed for autoregressive workloads. While Section 3 detailed the mechanical implementation of `vLLM`, this section isolates the conceptual innovations that distinguish it from prior art. These insights challenge established assumptions in deep learning systems and unlock capabilities previously deemed too expensive for production serving.

### 4.1 Breaking the Contiguous Memory Barrier

The most profound innovation in this work is the decoupling of **logical sequence continuity** from **physical memory continuity**. Prior to this work, the prevailing assumption in deep learning frameworks (e.g., PyTorch, TensorFlow) was that tensors—including the KV cache—must reside in contiguous memory regions to be processed efficiently by GPU kernels.

*   **Distinction from Prior Work:** Systems like FlashAttention [13] optimized memory *I/O* via tiling but still required the underlying KV cache tensor to be contiguous in physical memory. Similarly, Orca [60] and FasterTransformer [31] adhered to contiguous allocation, forcing them to pre-allocate maximum sequence lengths to avoid reallocation overhead.
*   **Why It Matters:** By proving that attention can be computed over non-contiguous blocks without prohibitive kernel overhead (Section 7.1 notes only 20–26% kernel latency increase), `vLLM` removes the structural cause of memory fragmentation. This shifts the bottleneck from memory capacity to compute throughput.
*   **Significance:** This innovation directly enables the near-zero waste observed in evaluations. As shown in **Figure 2**, existing systems waste 60–80% of KV cache memory on fragmentation and reservation, whereas `vLLM` achieves **96.3% utilization**. This is not merely an efficiency gain; it fundamentally changes the economics of serving by allowing 2–4× larger batch sizes on the same hardware (Section 6.2).

### 4.2 Memory Sharing as a First-Class Primitive

`vLLM` treats memory sharing not as an edge-case optimization, but as a core system primitive enabled by the paged memory architecture. In prior systems, sharing KV cache across sequences (e.g., for beam search or parallel sampling) was effectively impossible because each sequence required its own contiguous memory chunk.

*   **Distinction from Prior Work:** Traditional serving systems handle multiple sequences from the same prompt by duplicating the KV cache for each sequence. This leads to linear memory growth with respect to the number of samples. `vLLM` introduces reference counting and copy-on-write (CoW) at the block level (Section 4.4), allowing multiple logical sequences to map to the same physical blocks.
*   **Why It Matters:** This capability makes complex decoding algorithms feasible at scale. For example, in beam search, candidates share significant portions of their history. Without sharing, the memory cost grows exponentially with beam width. With `vLLM`, the cost grows only with the *divergence* points.
*   **Significance:** The impact is quantified in **Figure 15**, which shows memory savings of **37.6%–55.2% for beam search** and **6.1%–9.8% for parallel sampling** on the Alpaca dataset. More importantly, this enables throughput improvements of up to **2.3× for beam search** compared to Orca (Oracle) (Section 6.3). This innovation expands the design space for LLM applications, allowing providers to offer higher-quality decoding strategies (like wide beam search) without prohibitive hardware costs.

### 4.3 LLM-Specific Virtual Memory Policies

While `vLLM` borrows the concept of virtual memory from operating systems, it critically adapts eviction and recovery policies to the semantics of LLM inference. General OS paging policies (e.g., LRU page eviction) are suboptimal for LLMs because the access patterns and recovery costs differ fundamentally.

*   **Distinction from Prior Work:** Standard virtual memory systems evict individual pages based on access frequency. `vLLM` implements an **"all-or-nothing" eviction policy** (Section 4.5), where either all blocks of a sequence are evicted or none are. This exploits the autoregressive property: attention computation requires access to *all* previous tokens, so partial eviction provides no memory relief while breaking the sequence.
*   **Why It Matters:** This policy simplifies memory management and enables **recomputation** as a viable recovery strategy. In general OS contexts, recomputing a page's content is often impossible or more expensive than swapping. In LLM serving, evicted KV cache can be regenerated by re-running the prompt phase on the known tokens.
*   **Significance:** The paper demonstrates that recomputation is competitive with swapping, particularly for smaller block sizes where PCIe bandwidth limits swap performance. **Figure 19b** shows that for medium block sizes (16–64), recomputation and swapping exhibit comparable end-to-end performance. Furthermore, Section 7.3 notes that **"recomputation overhead is never higher than 20% of swapping's latency,"** making it a robust fallback mechanism that avoids the complexity of managing CPU swap space. This insight highlights that applying OS concepts to accelerators requires domain-specific adaptation, not just direct translation.

### 4.4 Decoupling Memory Management from Model Execution

A subtle but critical architectural innovation is the separation of the **centralized KV Cache Manager** from the **distributed GPU Workers** (Section 4.6). In many distributed inference systems, memory management is either fully decentralized (leading to fragmentation across devices) or tightly coupled with model execution (leading to synchronization overhead).

*   **Distinction from Prior Work:** Systems using tensor model parallelism often require complex coordination to manage memory shards across devices. `vLLM` maintains a single global view of block allocation in the scheduler, while workers remain stateless regarding memory management.
*   **Why It Matters:** This design ensures that memory sharing (e.g., shared prefixes across requests) works seamlessly even when the model is split across multiple GPUs. Workers simply receive block tables and execute; they do not need to negotiate memory ownership.
*   **Significance:** This allows `vLLM` to scale to large models (e.g., 175B parameters on 8×A100s, Table 1) without sacrificing the memory efficiency gains of paging. It ensures that the benefits of PagedAttention are preserved in distributed settings, where memory fragmentation would otherwise be compounded across devices.

## 5. Experimental Analysis

This section validates the claims made in Sections 3 and 4 regarding `vLLM`'s memory efficiency and throughput improvements. The evaluation is designed to answer three core questions: (1) Does `vLLM` achieve higher throughput than state-of-the-art systems under realistic workloads? (2) Do the memory sharing mechanisms (copy-on-write, reference counting) provide tangible benefits for complex decoding? (3) What are the trade-offs introduced by the paging mechanism (e.g., kernel overhead, block size selection)?

### 5.1 Evaluation Methodology

To ensure a fair comparison, the authors construct a rigorous experimental setup that controls for hardware, model sizes, and workload characteristics.

**Hardware and Model Configurations:**
Experiments are conducted on Google Cloud Platform A2 instances using NVIDIA A100 GPUs. The paper evaluates three model scales to test scalability, detailed in **Table 1**:
*   **OPT-13B:** Served on 1×A100 (40GB). Parameters occupy 26GB, leaving 12GB for KV cache.
*   **OPT-66B:** Served on 4×A100 (160GB total). Parameters occupy 132GB, leaving 21GB for KV cache.
*   **OPT-175B:** Served on 8×A100-80GB (640GB total). Parameters occupy 346GB, leaving 264GB for KV cache.

Note the diminishing memory headroom for larger models. For the 13B model, KV cache memory is only 12GB, making memory management critical. For the 175B model, 264GB is available for KV cache, which can accommodate more requests before hitting memory limits.

**Workloads and Datasets:**
The authors synthesize request traces using two real-world datasets to capture different sequence length distributions (**Figure 11**):
*   **ShareGPT:** Contains user conversations with ChatGPT. It features long inputs (mean 161 tokens) and long outputs (mean 338 tokens). This workload is heavily **memory-bound**.
*   **Alpaca:** An instruction-following dataset. It features short inputs (mean 19 tokens) and short outputs (mean 58 tokens). This workload is more likely to be **compute-bound**.

Request arrival times are generated using a Poisson distribution to simulate realistic traffic spikes.

**Baselines:**
`vLLM` is compared against two primary baselines:
1.  **FasterTransformer:** A highly optimized inference engine. The authors implement a custom scheduler for it since it lacks one.
2.  **Orca:** A state-of-the-art serving system optimized for throughput. Since Orca is not publicly available, the authors implement their own version. To account for uncertainty in Orca's memory allocation strategy, they evaluate three variants:
    *   **Orca (Oracle):** Assumes perfect knowledge of output lengths (theoretical upper bound).
    *   **Orca (Pow2):** Over-reserves memory by at most 2× (e.g., 25 tokens → 32 slots).
    *   **Orca (Max):** Reserves memory for the maximum sequence length (2048 tokens).

**Metrics:**
The primary metric is **normalized latency**, defined as the mean end-to-end latency of a request divided by its output length (seconds per token). A high-throughput system should maintain low normalized latency even as the request rate increases. The "capacity" of a system is defined as the maximum request rate it can sustain before latency explodes.

### 5.2 Throughput Improvements in Basic Sampling

The most fundamental test is basic sampling (one output sequence per request). The results are presented in **Figure 12**, which plots normalized latency against request rate.

**Interpreting the Latency Curves:**
Notice the shape of the curves in **Figure 12**. Initially, latency increases gradually as the system loads increase. However, at a certain threshold, the latency curve spikes vertically. This "cliff" indicates that the system has reached saturation: the queue length grows infinitely because requests arrive faster than they can be processed. The position of this cliff on the x-axis represents the system's maximum throughput capacity.

**Quantitative Results:**
*   **ShareGPT (Memory-Bound):** On the OPT-13B model with ShareGPT traces (**Figure 12a**), `vLLM` sustains **1.7×–2.7× higher request rates** compared to Orca (Oracle), and **2.7×–8× higher** compared to Orca (Max).
    *   *Why?* ShareGPT sequences are long. Orca's contiguous pre-allocation wastes significant memory on unused slots (internal fragmentation). `vLLM`'s paging allows it to fit more concurrent requests into the same 12GB KV cache budget.
    *   **Figure 13a** confirms this: at a fixed request rate (2 req/s), `vLLM` batches **30.42 requests** on average, whereas Orca (Oracle) batches only **13.62 requests**. This 2.2× larger batch size directly translates to higher throughput.
*   **Alpaca (Compute-Bound):** On the OPT-175B model with Alpaca traces (**Figure 12f**), the performance gap narrows.
    *   *Why?* The 175B configuration has 264GB for KV cache (**Table 1**), and Alpaca sequences are short. Even Orca's inefficient allocation is sufficient to batch many requests. The bottleneck shifts from memory capacity to GPU computation power. In this regime, `vLLM`'s memory advantages matter less because memory is not the limiting factor.

**Comparison to FasterTransformer:**
`vLLM` sustains up to **22× higher request rates** than FasterTransformer on OPT-13B. This massive gap is due to two factors: (1) FasterTransformer lacks fine-grained iteration-level scheduling (it waits for entire batches to complete), and (2) it suffers from the same contiguous memory fragmentation as Orca (Max).

### 5.3 Benefits of Memory Sharing Mechanisms

Section 4.4 described how `vLLM` enables memory sharing via reference counting and copy-on-write. The experiments quantify the impact of these mechanisms on complex decoding algorithms.

**Parallel Sampling:**
In parallel sampling, multiple output sequences are generated from a single prompt. The prompt's KV cache should be shared.
*   **Figure 14 (a-c)** shows results for parallel sizes 2, 4, and 6. As the parallel size increases, `vLLM`'s advantage over Orca grows.
*   **Figure 15a** quantifies the memory savings: `vLLM` saves **6.1%–9.8%** of memory on Alpaca and **16.2%–30.5%** on ShareGPT. The savings are higher on ShareGPT because the prompts are longer, meaning a larger portion of the total KV cache is shared.

**Beam Search:**
Beam search is more complex because candidates share prefixes dynamically.
*   **Figure 14 (d-f)** shows that `vLLM` maintains low latency at much higher request rates than Orca during beam search.
*   **Figure 15b** shows significant memory savings: **37.6%–55.2%** on Alpaca and **44.3%–66.3%** on ShareGPT.
*   *Throughput Impact:* The improvement of `vLLM` over Orca (Oracle) on OPT-13B increases from **1.3× in basic sampling** to **2.3× in beam search** (width 6). This confirms that `vLLM`'s block-level sharing is critical for advanced decoding strategies that were previously too memory-expensive to deploy at scale.

**Shared Prefix (Few-Shot Learning):**
Many applications use a common system prompt or few-shot examples (**Figure 10**).
*   **Figure 16** evaluates a translation workload with shared prefixes.
*   **1-shot prefix (80 tokens):** `vLLM` achieves **1.67×** higher throughput than Orca (Oracle).
*   **5-shot prefix (341 tokens):** `vLLM` achieves **3.58×** higher throughput.
*   *Reasoning:* As the shared prefix grows, the memory saving from caching the prefix KV cache becomes dominant. Orca must recompute or duplicate this prefix for every request, whereas `vLLM` maps multiple logical requests to the same physical prefix blocks.

**Chatbot Workload:**
Chatbots maintain conversation history, leading to long prompts (previous turns + new query).
*   **Figure 17** shows `vLLM` sustains **2× higher request rates** than Orca baselines.
*   *Critical Detail:* The experiment limits prompts to 1024 tokens. Orca baselines behave similarly across variants (Max, Pow2, Oracle) because the buddy allocation algorithm reserves space for 1024 output tokens regardless of prediction. `vLLM` avoids this reservation waste, handling long prompts efficiently without pre-allocating for maximum output length.

### 5.4 Ablation Studies and Design Trade-offs

The authors conduct ablation studies to justify specific design choices, particularly regarding kernel overhead and block size.

**Kernel Overhead (**Figure 18a**):**
PagedAttention requires indirect memory access (reading block tables) compared to contiguous attention.
*   **Result:** The `vLLM` attention kernel has **20–26% higher latency** than the highly-optimized FasterTransformer kernel.
*   **Analysis:** While this overhead sounds significant, it affects *only* the attention operator, not the feed-forward or embedding layers. More importantly, the ability to batch **2–4× more requests** (due to memory efficiency) amortizes this per-token overhead. The net system throughput increases despite the slower individual kernel execution.

**Block Size Selection (**Figure 18b**):**
The block size $B$ determines the granularity of paging.
*   **Too Small:** If $B$ is too small (e.g., 1), the GPU cannot parallelize memory reads effectively, underutilizing bandwidth.
*   **Too Large:** If $B$ is too large (e.g., 256), internal fragmentation increases (the last block of a sequence may be mostly empty), and sharing granularity decreases.
*   **Result:** On ShareGPT, block sizes **16–128** perform best. On Alpaca (shorter sequences), larger blocks degrade performance because sequences are shorter than the block size, leading to wasted space.
*   **Decision:** `vLLM` sets the default block size to **16**, balancing GPU utilization and fragmentation across most workloads.

**Swapping vs. Recomputation (**Figure 19**):**
When GPU memory is exhausted, `vLLM` can either swap blocks to CPU RAM or recompute them.
*   **Microbenchmark (**Figure 19a**):** Swapping incurs high overhead for small block sizes due to PCIe bandwidth limits (many small transfers). Recomputation overhead is constant across block sizes.
*   **End-to-End (**Figure 19b**):** For medium block sizes (16–64), both methods perform comparably.
*   **Key Finding:** "Recomputation overhead is never higher than 20% of swapping's latency" (Section 7.3). This suggests recomputation is a robust, simpler alternative to managing CPU swap space, especially for smaller block sizes.

### 5.5 Critical Assessment of Experimental Validity

The experiments convincingly support the paper's core claims, but there are nuances to consider.

**Strengths:**
*   **Memory Efficiency Proof:** **Figure 2** provides direct evidence of the fragmentation problem (20.4% usage in Orca vs. 96.3% in `vLLM`). This is the foundational metric validating the need for PagedAttention.
*   **Workload Diversity:** By testing both long (ShareGPT) and short (Alpaca) sequences, the authors demonstrate where `vLLM` shines (memory-bound) and where gains are limited (compute-bound).
*   **Decoding Variety:** Evaluating parallel sampling, beam search, and shared prefixes confirms that the memory sharing mechanisms work across different access patterns, not just simple generation.

**Limitations and Caveats:**
*   **Orca Implementation:** Since Orca is not open-source, the authors implemented their own version. While they followed the paper's description, subtle differences in scheduling or kernel optimization could affect the baseline performance. However, the **Orca (Oracle)** variant serves as a strong theoretical upper bound for contiguous allocation, and `vLLM` still beats it, suggesting the advantage comes from the memory architecture, not just implementation quality.
*   **Hardware Specificity:** Results are tied to NVIDIA A100 GPUs. The trade-off between swapping and recomputation (Figure 19) depends heavily on PCIe bandwidth and GPU compute power. On systems with slower CPU-GPU interconnects, recomputation might be even more favorable.
*   **Preemption Overhead:** The experiments show swapping/recomputation costs, but they assume preemption is relatively rare (handled when memory is exhausted). In highly overloaded systems where preemption is frequent, the "all-or-nothing" eviction policy might lead to throughput oscillations not fully captured in the 1-hour traces.

**Conclusion of Analysis:**
The experimental data robustly validates the hypothesis that KV cache memory management is the primary bottleneck in LLM serving. By achieving near-zero waste (**Figure 2**) and enabling block-level sharing (**Figure 15**), `vLLM` converts memory capacity into throughput. The 2–4× improvement is not merely an optimization; it represents a fundamental shift in how many requests can be served per GPU, directly addressing the cost concerns raised in the Introduction.

## 6. Limitations and Trade-offs

While `vLLM` demonstrates substantial throughput improvements, its design involves specific trade-offs and relies on assumptions that constrain its applicability. Understanding these limitations is crucial for deploying the system effectively and recognizing where alternative approaches might be preferable.

### 6.1 Inherent Kernel Overhead

The core innovation of `PagedAttention`—accessing KV cache via indirect block tables—introduces a non-negligible computational cost. Unlike standard attention kernels that operate on contiguous memory tensors, `PagedAttention` must perform additional memory lookups and handle variable sequence lengths within a batch.

*   **Latency Tax:** Section 7.1 quantifies this overhead, noting that `vLLM`'s attention kernels exhibit **20–26% higher latency** compared to the highly-optimized FasterTransformer implementation. This is due to "extra overheads of accessing the block table, executing extra branches, and handling variable sequence lengths."
*   **The Trade-off:** This overhead is acceptable only when the system is **memory-bound**. The increased batch size enabled by efficient memory management amortizes the per-token kernel cost. However, in regimes where the GPU is already compute-saturated, this overhead becomes a pure penalty without the compensating benefit of larger batches.
*   **Implication:** Users cannot expect `vLLM` to reduce *per-request* latency in isolation. Its benefit is strictly in *throughput* (requests per second) under high concurrency. For low-latency, single-request scenarios, the indirect memory access may actually degrade performance compared to contiguous baselines.

### 6.2 Sensitivity to Workload Characteristics

The effectiveness of `vLLM` is highly dependent on the sequence length distribution of the workload. The paging mechanism introduces a granularity parameter—the block size $B$—that must be tuned to match the data.

*   **Block Size Tuning:** Section 7.2 reveals that performance is sensitive to block size selection. While the default is set to **16 tokens**, this is not universal.
    *   **Short Sequences:** On the Alpaca dataset (mean output 58 tokens), larger block sizes (e.g., 128 or 256) significantly degrade performance because sequences are shorter than the block size, leading to high internal fragmentation.
    *   **Long Sequences:** On ShareGPT (mean output 338 tokens), larger blocks perform better by utilizing GPU parallelism.
*   **Operational Burden:** This implies that `vLLM` is not a "set-and-forget" system. Operators must profile their specific workload distributions (input/output lengths) to select an optimal block size. A mismatch here can erode the throughput gains, as seen in **Figure 18b** where normalized latency varies by over **2×** depending on block size choice for the same workload.
*   **Compute-Bound Regime:** The benefits diminish when memory is not the bottleneck. In **Figure 12f** (OPT-175B on Alpaca), the performance gap between `vLLM` and Orca (Oracle) narrows significantly. Because the 175B configuration has **264 GB** available for KV cache (**Table 1**) and Alpaca sequences are short, even inefficient contiguous allocation suffices to batch many requests. In this compute-bound regime, `vLLM`'s memory optimizations provide marginal utility while still incurring the kernel overhead described in Section 6.1.

### 6.3 Preemption and Recovery Constraints

When GPU memory is exhausted, `vLLM` must preempt requests. The strategies employed (swapping and recomputation) introduce specific constraints on system behavior and availability.

*   **All-or-Nothing Eviction:** Section 4.5 implements an "all-or-nothing" eviction policy, where either all blocks of a sequence are evicted or none are. This simplifies management but lacks granularity.
    *   **Shared Prefix Risk:** While reference counting allows multiple requests to share a prefix, the eviction policy operates at the sequence level. If memory pressure forces eviction, the system may evict a sequence that is sharing a prefix with many other active sequences. While the reference count prevents the physical block from being freed until all users are done, the *eviction logic* might still target the sequence, potentially complicating the swap-in process if the shared block is simultaneously needed by others.
*   **Service Stall During Preemption:** A critical operational limitation is found in Section 4.5: "Once it preempts a sequence and evicts its blocks, vLLM stops accepting new requests until all preempted sequences are completed."
    *   **Impact:** This creates a potential availability hole. During periods of extreme overload where preemption is frequent, the system may enter a cycle of evicting and recomputing/swapping without admitting new work. This protects stability but reduces responsiveness during traffic spikes.
*   **Swapping Bandwidth Limits:** Section 7.3 shows that swapping is inefficient for small block sizes due to PCIe bandwidth limits. For block sizes below 16, recomputation is strictly preferred. This limits the utility of swapping as a universal fallback; it is only viable when the block size is sufficiently large to amortize transfer latency.

### 6.4 Scope and Applicability Boundaries

The authors explicitly delineate where their approach does *not* apply. `vLLM` is specialized for autoregressive LLM serving and does not generalize to all deep learning workloads.

*   **Static Shape Workloads:** Section 8 notes that in DNN training or non-autoregressive inference, "tensor shapes are typically static, and thus memory allocation can be optimized ahead of time." In these scenarios, the indirection overhead of `PagedAttention` provides no benefit and would only degrade performance.
*   **Compute-Bound Serving:** For serving DNNs that are not LLMs (e.g., vision models), performance is often compute-bound rather than memory-capacity-bound. Section 8 states that "introducing the vLLM's techniques may rather degrade the performance due to the extra overhead of memory indirection."
*   **Model Architecture Constraints:** `vLLM` currently supports Transformer-based models (GPT, OPT, LLaMA) using standard self-attention. It relies on the property that "all blocks of a sequence are accessed together" (Section 8). Architectures with complex, non-sequential attention patterns or dynamic compute graphs that deviate from standard autoregressive generation may not fit the paging model without significant modification.

### 6.5 Implementation and Maintenance Complexity

Adopting `vLLM` requires accepting higher engineering complexity compared to standard inference pipelines.

*   **Custom Kernels:** The system relies on **2K lines of C++/CUDA code** for custom kernels (Section 5). This includes fused operations for block reads, writes, and copies.
    *   **Maintenance Burden:** Unlike systems that rely solely on standard libraries (e.g., PyTorch, CUDA graphs), `vLLM` requires specialized expertise to maintain and port to new GPU architectures. As new GPU generations emerge (e.g., H100 to B100), these custom kernels must be re-optimized to maintain the performance edge.
*   **Scheduler Coupling:** The centralized scheduler manages both batching and memory allocation. While this enables global optimization, it creates a potential single point of contention. Section 4.6 notes that workers receive control messages from the scheduler at every iteration. In extremely large-scale distributed settings (e.g., hundreds of GPUs), the broadcast overhead of block tables and token IDs could become a bottleneck, though the paper does not quantify this limit.

### 6.6 Open Questions and Future Challenges

Several aspects of the design remain open for optimization or present unresolved challenges:

*   **Scheduling Policies:** The current implementation uses First-Come-First-Serve (FCFS) scheduling (Section 4.5). While fair, this may not optimize for Service Level Objectives (SLOs) such as tail latency. A priority-based scheduler that preempts low-priority requests to protect high-priority SLAs is not implemented.
*   **Multi-Tenancy Isolation:** The paper does not address memory isolation between different tenants or models running on the same cluster. Since physical blocks are shared via reference counting, a noisy neighbor could theoretically impact the eviction rate of shared blocks, though the FCFS policy mitigates starvation.
*   **Prefix Cache Eviction:** While shared prefixes are cached (Section 4.4), the paper does not detail a policy for evicting *unused* shared prefixes to free up space for new models or tenants. If the library of shared prefixes grows larger than GPU memory, the system lacks a defined strategy for managing this cache beyond the sequence-level eviction policy.
*   **Dynamic Model Loading:** The evaluation assumes static model loading (Table 1). The overhead of swapping entire model weights (not just KV cache) to support serving many different models on limited hardware is not addressed. `vLLM` optimizes KV cache, but model parameters still occupy **65%** of memory (Figure 1, left), which remains a fixed cost per model.

In summary, `vLLM` trades kernel simplicity and contiguous memory performance for massive gains in memory utilization. This trade-off is overwhelmingly positive for memory-bound LLM serving but introduces tuning requirements, operational constraints during preemption, and engineering complexity that must be weighed against the throughput benefits.

## 7. Implications and Future Directions

The introduction of `PagedAttention` and `vLLM` represents a paradigm shift in how we architect systems for autoregressive generative models. By demonstrating that GPU memory management—not just compute optimization—is the primary lever for throughput, this work redefines the performance ceiling for LLM serving. This section explores how this work changes the field, what research questions it opens, and how practitioners should approach deployment.

### 7.1 Shifting the Serving Bottleneck

Historically, optimization efforts for LLM inference focused on reducing computational latency through kernel fusion (e.g., combining matrix multiplications) or quantization. `vLLM` demonstrates that for high-concurrency serving, the bottleneck is often memory capacity, not compute speed.

*   **Economic Impact:** The paper reports throughput improvements of **2–4×** compared to state-of-the-art systems like Orca and FasterTransformer (Abstract). In practical terms, this means a service provider can handle the same traffic with **50–75% fewer GPUs**, or alternatively, serve 4× more users on the same hardware. Given that "processing an LLM request can be 10× more expensive than a traditional keyword query" (Section 1), this efficiency gain directly addresses the cost barrier preventing widespread LLM adoption.
*   **Algorithmic Feasibility:** Prior to this work, complex decoding algorithms like beam search were often deemed too expensive for production due to their memory footprint. `vLLM` changes this by enabling memory sharing across sequences. Section 6.3 shows that `vLLM` achieves **37.6%–55.2% memory saving** for beam search on the Alpaca dataset. This makes high-quality decoding strategies viable for real-time applications, potentially improving output quality without prohibitive hardware costs.
*   **Memory as a First-Class Resource:** The work elevates memory management to the same level of importance as compute scheduling. Figure 2 illustrates that existing systems waste **60–80%** of KV cache memory on fragmentation. By treating KV cache like OS virtual memory, `vLLM` achieves **96.3% utilization**. This suggests that future system designs must prioritize memory elasticity alongside computational parallelism.

### 7.2 Promising Avenues for Future Research

While `vLLM` solves the fragmentation problem, it opens several new research questions regarding how to further optimize paged memory systems for LLMs.

*   **Combining Attention Optimizations:** The paper notes in Section 9 that `FlashAttention` optimizes attention I/O via tiling but does not address block-level memory management for serving. A natural follow-up is to fuse `PagedAttention` with `FlashAttention` techniques. Since `PagedAttention` already introduces non-contiguous access patterns (Section 4.1), integrating FlashAttention's I/O-aware tiling could reduce the **20–26% kernel overhead** observed in Figure 18a. This would retain the memory efficiency of paging while recovering the raw kernel speed of contiguous attention.
*   **SLO-Aware Scheduling:** Currently, `vLLM` uses a First-Come-First-Serve (FCFS) policy (Section 4.5). Section 6.6 identifies this as a limitation, noting that it does not optimize for Service Level Objectives (SLOs) like tail latency. Future work could develop priority-based schedulers that preempt low-priority requests (using the recomputation/swapping mechanisms in Section 4.5) to protect high-priority SLAs. The "all-or-nothing" eviction policy provides a clean abstraction for building such preemptive schedulers.
*   **Multi-Model Serving:** The evaluation assumes static model loading (Table 1), where model parameters occupy **65%** of memory (Figure 1, left). Section 6.6 highlights that dynamic model loading is not addressed. Future systems could extend the paging concept to model weights themselves, swapping entire model shards between CPU and GPU to serve hundreds of different models on a single cluster, similar to how `FlexGen` approaches single-GPU inference (Section 9).
*   **Hardware-Specific Tuning:** The trade-off between swapping and recomputation (Section 7.3) depends on PCIe bandwidth and GPU compute power. As hardware evolves (e.g., from A100 to H100, where FLOPS increase **2×** but memory stays at 80GB maximum, Section 3), the optimal block size and preemption strategy may shift. Research into automated tuning systems that adapt block size $B$ and preemption policies to specific hardware configurations would make these systems more robust.

### 7.3 Practical Applications and Deployment Scenarios

The architectural choices in `vLLM` make it particularly well-suited for specific classes of applications. Practitioners should evaluate their workload characteristics against the system's strengths.

*   **High-Throughput APIs:** For services handling many concurrent requests (e.g., chatbots, coding assistants), `vLLM` is the preferred choice. The memory efficiency allows larger batch sizes, directly increasing throughput. Section 6.2 shows that on the ShareGPT dataset (long sequences), `vLLM` sustains **2.7×–8× higher request rates** than Orca (Max).
*   **Complex Decoding Tasks:** Applications requiring high-quality output, such as machine translation or complex reasoning, often use beam search or parallel sampling. Section 6.3 demonstrates that `vLLM`'s memory sharing makes these algorithms feasible at scale. For example, in parallel sampling with size 6, `vLLM` maintains low latency where baselines saturate (Figure 14c).
*   **Shared Context Workloads:** Applications where many requests share a common prefix (e.g., system instructions, few-shot examples, or retrieval-augmented generation contexts) benefit significantly. Section 6.4 shows that with a **5-shot prefix (341 tokens)**, `vLLM` achieves **3.58× higher throughput** than Orca (Oracle). This is because the KV cache for the prefix is computed once and shared across requests via reference counting (Section 4.4).
*   **When Not to Use:** `vLLM` is less advantageous in low-concurrency, latency-critical scenarios. If serving a single request at a time, the **20–26% kernel overhead** (Section 7.1) provides no benefit because memory capacity is not the bottleneck. Similarly, for very short sequences (like Alpaca) on large models (like OPT-175B), the system becomes compute-bound, and the performance gap narrows (Figure 12f).

### 7.4 Reproduction and Integration Guidance

For researchers and engineers looking to adopt or build upon this work, the following practical guidance is derived from the paper's implementation details.

*   **Code Availability:** The authors state that "`vLLM`'s source code is publicly available at https://github.com/vllm-project/vllm" (Abstract). This allows for immediate integration into existing serving pipelines.
*   **Block Size Configuration:** The block size $B$ is a critical hyperparameter. Section 7.2 recommends a default block size of **16 tokens**. This value balances GPU utilization (which favors larger blocks) against internal fragmentation (which favors smaller blocks).
    *   **Guidance:** If your workload consists of very long sequences (mean > 300 tokens, like ShareGPT), you may experiment with larger block sizes (up to 128). If your workload consists of short instructions (mean < 100 tokens, like Alpaca), stick to the default **16** or smaller to avoid wasting space in partially filled blocks (Figure 18b).
*   **Memory Estimation:** To plan capacity, use the paper's formula for KV cache size. For a 13B parameter OPT model, "the KV cache of a single token demands **800 KB** of space, calculated as 2 (key and value vectors) × 5120 (hidden state size) × 40 (number of layers) × 2 (bytes per FP16)" (Section 3).
    *   **Calculation:** For a batch of $N$ requests with average sequence length $L$, the required KV cache memory is approximately $N \times L \times 800 \text{ KB}$. Ensure this fits within the available GPU memory after subtracting model weights (e.g., **12 GB** available for KV cache on a 40GB A100 with a 13B model, Table 1).
*   **Preemption Strategy:** When configuring preemption, consider the block size. Section 7.3 indicates that for block sizes **16–64**, recomputation and swapping have comparable performance. However, for smaller blocks, recomputation is preferred because swapping incurs excessive PCIe overhead. If your deployment uses small block sizes, enable recomputation to avoid bandwidth bottlenecks.
*   **Distributed Setup:** For models exceeding single-GPU memory (e.g., 175B parameters), `vLLM` supports tensor model parallelism (Section 4.6). Ensure that the network interconnect between GPUs supports high-bandwidth all-reduce operations (e.g., NCCL), as workers synchronize intermediate results without scheduler coordination. The KV Cache Manager remains centralized, so the scheduler must have low-latency access to all workers to broadcast block tables efficiently.

By understanding these implications and guidelines, practitioners can effectively leverage `vLLM` to reduce costs and enable new capabilities, while researchers can identify the next frontiers in efficient generative serving.
