## 1. Executive Summary

Mooncake introduces a KVCache-centric disaggregated architecture for LLM serving that physically separates prefill and decoding clusters while leveraging underutilized CPU, DRAM, and SSD resources to create a distributed cache pool, specifically optimizing for long-context workloads like those served by Kimi. By implementing a prediction-based early rejection policy to manage overload scenarios and a chunked pipeline parallelism mechanism to reduce Time to First Token (TTFT), the system achieves up to a 525% throughput increase over baseline methods in simulated long-context scenarios while strictly adhering to latency Service Level Objectives (SLOs). In real-world deployments using traces from Kimi, this architecture enables the platform to handle 75% more requests compared to traditional coupled systems without violating Time Between Tokens (TBT) constraints.

## 2. Context and Motivation

### The Fundamental Mismatch in LLM Serving
To understand the necessity of Mooncake, one must first recognize a fundamental architectural mismatch in how Large Language Models (LLMs) are currently served. Modern LLM inference consists of two distinct phases with radically different resource requirements:

1.  **The Prefill Stage:** The model processes the entire input prompt (which can be tens of thousands of tokens) in parallel. This phase is **computation-bound**. As shown in **Figure 2** (left), the latency for this stage scales superlinearly with sequence length because the attention mechanism's complexity is quadratic ($O(N^2)$). The primary metric here is **Time to First Token (TTFT)**—how long a user waits before seeing any output.
2.  **The Decoding Stage:** The model generates tokens one by one in an autoregressive manner. This phase is **memory-bound**. As shown in **Figure 2** (right), throughput scales sublinearly with batch size because the system is limited by the bandwidth required to fetch the Key-Value Cache (KVCache) from memory, not by compute power. The primary metric here is **Time Between Tokens (TBT)**—the smoothness of the text generation.

**The Problem:** Traditional serving systems (like standard deployments of vLLM or TensorRT-LLM) couple these two stages on the same GPU instances. This creates a "head-of-line" blocking problem. When a long-context request enters the prefill stage, it hogs the GPU's compute resources, stalling the decoding of other requests and violating their TBT Service Level Objectives (SLOs). Conversely, if the system is optimized for decoding (large batches), long prefill requests suffer excessive TTFT.

### The Gap: From Resource Utilization to Overload Management
Existing research largely operates under an assumption that is increasingly false in production environments: **that the cluster has sufficient resources to process all incoming requests.** Papers like *Orca* [12] and *vLLM* [13] focus on maximizing throughput or minimizing latency assuming the system is not saturated.

However, Moonshot AI faces a different reality with their service, Kimi. Due to rapid user growth and limited GPU supply, the system frequently operates in **severe overload scenarios**. In this regime, the optimization goal shifts. It is no longer just about speed; it is about **Goodput**—maximizing the number of *successfully completed* requests that meet their SLOs.

This introduces a critical gap in prior work:
*   **Wasted Computation:** In a coupled architecture, if a request is accepted, completes its expensive prefill stage, but is then rejected at the decoding stage because the queue is too full, the computational resources spent on prefill are entirely wasted.
*   **Lack of Early Rejection Logic:** Existing schedulers do not effectively predict future load to reject requests *before* they consume resources. A naive "reject if busy" policy leads to instability, as discussed later in the paper.

### Limitations of Prior Disaggregated Approaches
The idea of separating prefill and decoding is not entirely new. Works like *Splitwise* [7], *DistServe* [8], and *TetriInfer* [9] propose disaggregated architectures. However, Mooncake identifies specific shortcomings in these approaches when applied to extreme long-context and overload scenarios:

1.  **Inefficient Parallelism for Long Contexts:**
    *   Prior approaches often rely on **Sequence Parallelism (SP)** to split long prompts across multiple nodes. While SP reduces memory pressure, it requires frequent cross-node communication (e.g., Ring Attention) for every transformer layer. This consumes network bandwidth that is also needed for transferring KVCache, reducing the overall Model FLOPs Utilization (MFU).
    *   Other approaches suggest static partitioning of nodes into "prefill-only" and "decode-only" groups. This lacks elasticity; if the workload shifts, one group sits idle while the other is overloaded. Dynamic resizing (Elastic SP) adds significant engineering complexity and latency overhead.

2.  **KVCache as an Afterthought:**
    *   Most systems treat KVCache as local state attached to a GPU. When requests move or need reuse, they rely on simple eviction or local caching.
    *   In long-context scenarios (e.g., 128k tokens), the KVCache is massive. Without a centralized, tiered storage system leveraging CPU DRAM and SSDs, cache reuse is limited, forcing the system to recompute prefixes unnecessarily.

3.  **Instability in Overload:**
    *   As detailed in **Section 7.3**, simply implementing an "Early Rejection" policy (checking decoding load before starting prefill) causes **anti-phase load fluctuations**. If the system sees low decoding load, it accepts a flood of requests. By the time those requests finish prefill and hit the decoder, the load spikes, causing the system to reject new inputs. This creates a sawtooth pattern of utilization (**Figure 9**), leading to poor resource efficiency. Prior work has not adequately addressed this feedback loop in disaggregated settings.

### Mooncake's Positioning
Mooncake positions itself not just as a faster inference engine, but as a **production-grade orchestration layer** designed specifically for the constraints of a high-growth MaaS (Model as a Service) provider. It differentiates itself through three core shifts in design philosophy:

*   **KVCache-Centric Disaggregation:** Instead of just separating compute, Mooncake separates the *state*. It constructs a distributed KVCache pool using underutilized CPU, DRAM, and SSD resources across the cluster (**Figure 1**). This allows the system to treat KVCache as a first-class citizen that can be migrated, replicated, and swapped independently of the GPU compute instances.
*   **Chunked Pipeline Parallelism (CPP):** To solve the long-context bottleneck without the communication overhead of Sequence Parallelism, Mooncake introduces CPP (**Section 5.1**). This splits a long prompt into chunks processed by different nodes in a pipeline, requiring communication only at stage boundaries. This preserves high MFU while scaling to massive context lengths.
*   **Prediction-Based Overload Control:** Addressing the instability of naive rejection, Mooncake implements a **prediction-based early rejection policy** (**Section 7.4**). By forecasting the decoding load at the *future* time when a current prefill request will arrive, the scheduler can make stable accept/reject decisions, smoothing out load fluctuations and maximizing goodput.

In essence, while prior work optimizes for "how fast can we run a request?", Mooncake optimizes for "how many requests can we successfully complete per dollar of hardware under extreme load?"

## 3. Technical Approach

This section provides a comprehensive technical breakdown of Mooncake, detailing how its disaggregated architecture, specialized parallelism strategies, and predictive scheduling algorithms function together to solve the challenges of long-context LLM serving under overload.

### 3.1 Reader orientation (approachable technical breakdown)
Mooncake is a distributed serving platform that physically separates the "reading" (prefill) and "writing" (decoding) phases of LLM inference into distinct server pools, connected by a shared, tiered memory system that acts as a global cache for intermediate computation states. It solves the problem of long-context requests stalling real-time generation and wasting resources during traffic spikes by treating the Key-Value Cache (KVCache) as a movable, first-class resource that can be streamed, replicated, and managed independently of the GPU compute units.

### 3.2 Big-picture architecture (diagram in words)
The system is organized around a central controller called the **Conductor**, which orchestrates two distinct resource pools: a **Prefill Pool** optimized for heavy computation and a **Decoding Pool** optimized for high-bandwidth memory access.
*   **The Conductor:** A global scheduler that receives incoming requests, calculates hash signatures for input tokens to check for cache hits, and selects the optimal pair of prefill and decoding instances based on current load, cache distribution, and predicted latency.
*   **The Prefill Pool:** A cluster of GPU nodes that ingest raw text prompts, load reusable prefix caches from the distributed pool, compute new KVCache blocks, and stream these blocks out layer-by-layer.
*   **The Distributed KVCache Pool:** A logical storage layer formed by aggregating the underutilized CPU DRAM and SSDs across all nodes in the cluster, managed by a component called **Messenger** which handles high-speed RDMA transfers between CPU memory and GPU VRAM.
*   **The Decoding Pool:** A cluster of GPU nodes that receive streamed KVCache blocks into their local CPU memory, load them into VRAM asynchronously, and perform autoregressive token generation using continuous batching.

### 3.3 Roadmap for the deep dive
To fully understand Mooncake's operation, we will traverse the system in the following logical order:
1.  **Data Representation & Caching Logic:** We first explain how input text is converted into hashable blocks and stored in the tiered CPU/SSD pool, as this state management is the foundation of all subsequent scheduling decisions.
2.  **The Request Workflow:** We walk through the four-step lifecycle of a single request (Reuse, Incremental Prefill, Transfer, Decoding) to establish the baseline data flow.
3.  **Prefill Optimization (CPP & Layer-wise):** We detail the specific computational techniques (Chunked Pipeline Parallelism and Layer-wise streaming) used to handle massive context lengths without saturating network or VRAM.
4.  **KVCache-Centric Scheduling:** We analyze the Conductor's algorithm for selecting instances, balancing cache reuse against queue times to meet Time to First Token (TTFT) targets.
5.  **Overload Management:** Finally, we explain the prediction-based early rejection mechanism that prevents resource waste and stabilizes the system when demand exceeds capacity.

### 3.4 Detailed, sentence-based technical breakdown

#### Foundation: Tiered KVCache Storage and Hashing
The core innovation of Mooncake lies in decoupling the storage of the Key-Value Cache (KVCache) from the GPU compute units, allowing the system to utilize the abundant but slower CPU DRAM and SSDs as a massive extension of GPU VRAM.
*   **Block-wise Hashing for Deduplication:** Instead of storing raw text or treating the entire prompt as a single unit, Mooncake divides input tokens into fixed-size blocks (typically 512 tokens per block) and computes a cumulative hash for each block to enable prefix caching. As illustrated in **Figure 3**, the hash for a block $i$ is computed as $H_i = \text{Hash}(H_{i-1} + \text{tokens}_i)$, where $H_{i-1}$ is the hash of the preceding block chain; this ensures that a block is only considered a "match" if its entire prefix context is identical, preventing semantic errors from partial matches.
*   **Tiered Storage Hierarchy:** The system organizes these hashed blocks into a three-tier hierarchy: hot blocks reside in GPU VRAM for immediate access, warm blocks are stored in the aggregated CPU DRAM of the cluster (the "Distributed KVCache Pool"), and cold blocks are evicted to local SSDs.
*   **The Messenger Component:** Data movement between these tiers is managed by a dedicated process called `Messenger`, which runs on every node and utilizes GPUDirect RDMA to transfer KVCache blocks directly between the CPU DRAM of one node and the GPU VRAM of another, bypassing the CPU of the destination node to minimize latency.
*   **Cache Eviction Policies:** The system supports standard eviction algorithms like LRU (Least Recently Used) and LFU (Least Frequently Used), but also implements a `LengthAwareCache` strategy that prioritizes blocks appearing later in long sequences, acknowledging that in long-context tasks, the most recent context is often the most critical for immediate generation.

#### The Four-Step Request Workflow
When a request arrives, it does not simply enter a queue; it undergoes a orchestrated four-step journey designed to maximize cache reuse and overlap communication with computation.
1.  **KVCache Reuse:** Upon tokenization, the Conductor identifies which prefix blocks of the new request already exist in the Distributed KVCache Pool. It then selects a prefill instance and instructs the `Messenger` to load these specific prefix blocks from remote CPU memory into the prefill instance's GPU VRAM, effectively "bootstrapping" the request so the model does not recompute known context.
2.  **Incremental Prefill with Chunking:** The prefill instance processes only the *new* tokens (the incremental part) that were not found in the cache. If the number of uncached tokens exceeds a configurable threshold called `prefill_chunk` (typically set to $>1000$ tokens to maximize GPU utilization), the prefill stage is split into multiple chunks and processed in a pipeline fashion rather than all at once.
3.  **Asynchronous KVCache Transfer (Streaming):** Crucially, the system does not wait for the entire prefill stage to finish before moving data. As shown in **Figure 4**, as soon as the prefill instance finishes computing the KVCache for a specific transformer layer, it immediately streams that layer's output to the destination decoding instance's CPU memory via RDMA. This "layer-wise" streaming overlaps the network transfer time with the remaining prefill computation, hiding the latency of data movement.
4.  **Decoding with Async Load:** Once the decoding instance receives the full KVCache in its CPU DRAM, it begins loading the blocks into its GPU VRAM asynchronously while simultaneously starting the token generation process. This double-buffering approach ensures the GPU never sits idle waiting for memory transfers, provided the bandwidth is sufficient to keep up with the generation rate.

#### Scaling Prefill: Chunked Pipeline Parallelism (CPP)
Handling context lengths ranging from 8k to 128k tokens requires parallelizing the prefill stage across multiple GPU nodes, but traditional methods introduce excessive communication overhead.
*   **Limitations of Sequence Parallelism (SP):** Prior approaches like Ring Attention or Striped Attention use Sequence Parallelism, which requires nodes to communicate intermediate attention results (keys, values, and gradients) at *every* transformer layer. For a model with many layers, this frequent synchronization consumes significant network bandwidth and reduces the Model FLOPs Utilization (MFU) because GPUs spend time waiting for network packets.
*   **The CPP Mechanism:** Mooncake introduces Chunked Pipeline Parallelism (CPP), where a long input sequence is divided into large chunks, and each chunk is assigned to a different node in a pipeline group. Unlike SP, CPP only requires communication at the boundaries of the pipeline stages (when passing the hidden states from one chunk's processing to the next), drastically reducing the frequency of cross-node messages.
*   **Elasticity and Efficiency:** This design allows Mooncake to group $X$ nodes into a pipelined prefill group dynamically. Short requests can be processed by a single node (no parallelism overhead), while long requests automatically span multiple nodes. Because communication is infrequent, it can be easily overlapped with computation, maintaining high MFU even when scaling to dozens of nodes for million-token contexts.

#### Memory Optimization: Layer-wise Prefill
Beyond compute scaling, Mooncake addresses the constraint of limited GPU VRAM, which is often the bottleneck for storing the massive KVCache of long sequences.
*   **Reducing Occupation Cost:** The "occupation cost" of a request is defined as the product of its KVCache size $S$ and the time $T$ it occupies VRAM ($Cost = S \times T$). In traditional chunked prefill, the entire KVCache might need to be resident or managed in a way that extends $T$, increasing the cost.
*   **Asynchronous Layer Operations:** Mooncake implements a strict layer-wise execution flow where, before computing layer $L$, the system asynchronously loads the KVCache for layer $L$ from CPU to GPU. Immediately after computing layer $L$, it asynchronously dumps the resulting KVCache back to CPU DRAM.
*   **VRAM Independence:** As demonstrated in **Figure 7**, this overlapping of load/store operations with computation reduces the effective latency of memory management to near zero for long sequences. The practical result is that the prefill scheduler can ignore the available VRAM capacity of a node when making placement decisions, requiring only enough VRAM to hold a *single* request's active layer, rather than the entire context history.

#### The Conductor: KVCache-Centric Scheduling Algorithm
The global scheduler, or Conductor, is the brain of Mooncake, solving a complex optimization problem: selecting the best prefill instance $p$ and decoding instance $d$ to minimize latency while maximizing cache reuse.
*   **Input Variables:** For a new request $R$, the Conductor computes the set of block keys $block\_keys$ from the prompt. It then queries every available prefill instance $i \in P$ to determine $prefix\_len_i$, the length of the matching prefix cache already present on that instance.
*   **TTFT Estimation:** The scheduler estimates the Time to First Token ($TTFT$) for each candidate instance using the formula:
    $$TTFT_i = T_{queue,i} + T_{prefill}(len(R), prefix\_len_i) + T_{transfer}$$
    Where $T_{queue,i}$ is the estimated wait time based on the current queue depth, $T_{prefill}$ is a predictive model derived from offline profiling that estimates compute time based on input length and cache hit ratio, and $T_{transfer}$ is the time required to fetch missing cache blocks from other nodes if the instance with the best cache is not selected.
*   **Cache-Aware Load Balancing:** A naive strategy would always pick the instance with the longest prefix match. However, if that instance is heavily loaded, the queue time $T_{queue}$ might outweigh the compute savings. Mooncake uses a threshold-based heuristic: if the prefix length on the best-matched instance is significantly larger than the local prefix length of a less-loaded instance (specifically, if $\frac{best\_prefix\_len}{local\_prefix\_len} > kvcache\_balancing\_threshold$), the scheduler may still choose the less-loaded instance and trigger a background transfer of the hot cache blocks to it.
*   **Hot-Spot Migration:** This decision process naturally drives an automated "hot-spot migration" policy. Frequently accessed blocks (like system prompts) are implicitly replicated across multiple prefill nodes because the scheduler frequently routes requests to less-loaded nodes and triggers transfers, populating their local caches without requiring explicit, centralized prediction of future access patterns.

#### Overload-Oriented Scheduling and Early Rejection
In production environments like Kimi, demand often exceeds supply, requiring the system to reject requests to protect Service Level Objectives (SLOs) for accepted users.
*   **The Waste Problem:** In a disaggregated system, rejecting a request *after* it has completed the expensive prefill stage (because the decoding queue is full) wastes the compute resources spent on prefill. This reduces "goodput" (successfully completed requests per dollar).
*   **Naive Early Rejection Instability:** A simple "Early Rejection" policy checks the decoding load *before* starting prefill. However, as shown in **Figure 9** and **Figure 10a**, this causes severe anti-phase oscillations. When decoding load is low, the system accepts a flood of requests; by the time these requests finish prefill (seconds later), they all hit the decoder simultaneously, spiking the load and causing the system to reject new inputs. This leads to a cycle of "flood and drought" where resources are underutilized.
*   **Prediction-Based Early Rejection:** To stabilize this, Mooncake implements a **Prediction-Based Early Rejection** mechanism (**Section 7.4**). Instead of checking the *current* decoding load, the Conductor predicts the decoding load at time $t + \Delta t$, where $\Delta t$ is the estimated duration of the prefill stage.
*   **System-Level Prediction Model:** Since predicting the exact output length of every request is difficult and costly, Mooncake uses a system-level approximation. It assumes a uniform decoding time $t_d$ per token. The predictor simulates the future state of the decoding pool by:
    1.  Adding all requests currently in prefill that will finish by time $t + \Delta t$.
    2.  Removing all requests currently in decoding that will complete (exit the system) before $t + \Delta t$.
    3.  Calculating the projected Time Between Tokens (TBT) ratio for this future state.
*   **Decision Logic:** If the *predicted* future TBT violates the SLO (e.g., $TBT_{predicted} > TBT_{SLO}$), the request is rejected immediately upon arrival. This smooths the intake rate, ensuring that the decoding pool never becomes overwhelmed by the "delayed wave" of finishing prefill requests, thereby eliminating the oscillation seen in naive approaches.

#### Configuration and Hyperparameters
The effectiveness of Mooncake relies on several key configuration parameters derived from the paper's implementation details:
*   **Block Size:** The KVCache is managed in blocks of **512 tokens**, balancing the granularity of cache reuse with the overhead of hash management.
*   **Prefill Chunk Threshold:** The `prefill_chunk` size is typically set to **>1000 tokens**. This threshold determines when a request is split for pipeline parallelism; it is tuned to ensure that each chunk is large enough to saturate the GPU's compute units, avoiding the overhead of launching too many small kernels.
*   **SLO Targets:** In the reported experiments, the system enforces a **TTFT P90 = 10$\times$** and a **TBT P90 = 5$\times$** relative to the baseline latency of a single request running in isolation. In real deployments, these are fixed absolute values (e.g., TTFT < 30s, TBT < 0.1s/token).
*   **Cache Balancing Threshold:** The ratio used to decide between "best cache match" and "load balancing" is currently manually adjusted but acts as a lever to trade off between compute efficiency (reuse) and latency consistency (queue time).

By integrating these components—tiered storage, streaming transfers, chunked pipeline parallelism, and predictive scheduling—Mooncake transforms the KVCache from a static memory burden into a dynamic, schedulable resource, enabling it to handle workloads that would cause traditional coupled systems to collapse under latency violations or resource waste.

## 4. Key Insights and Innovations

Mooncake's contributions extend beyond standard engineering optimizations; they represent fundamental shifts in how we conceptualize resource management for LLM serving under extreme scale. While prior work focused on accelerating individual stages or maximizing throughput in ideal conditions, Mooncake introduces architectural paradigms specifically designed for **overload resilience** and **state mobility**. The following insights distinguish Mooncake from incremental improvements found in systems like vLLM or Splitwise.

### 4.1 The Paradigm Shift: From Throughput Maximization to "Goodput" via Overload Control
**Innovation Type:** Fundamental Conceptual Shift

Most existing literature on LLM serving (e.g., *Orca*, *vLLM*, *DistServe*) operates under the assumption that the cluster has sufficient capacity to process all incoming requests, optimizing primarily for **throughput** (tokens processed per second) or **latency** (TTFT/TBT). Mooncake challenges this assumption by recognizing that in high-growth MaaS (Model as a Service) environments, **overload is the default state**, not an edge case.

*   **The Distinction:** Traditional schedulers aim to keep GPUs busy at 100% utilization. Mooncake argues that in an overloaded system, 100% utilization of *compute* can lead to 0% **goodput** (successfully completed requests meeting SLOs) if the system accepts more requests than it can finish. If a request completes its expensive prefill stage only to be dropped at the decoding stage due to queue overflow, the computational resources spent on prefill are entirely wasted.
*   **Why It Matters:** This insight drives the development of the **Prediction-Based Early Rejection** policy (Section 7.4). Unlike naive rejection policies that cause unstable "sawtooth" load fluctuations (Figure 9), Mooncake's approach treats **rejection as a resource optimization tool**. By predicting the *future* state of the decoding pool at the moment a prefill request would arrive, the system proactively throttles intake.
*   **Impact:** This shifts the optimization objective from "process as many tokens as possible" to "maximize the number of *completed* requests per dollar of hardware." In experiments, this allowed Mooncake to handle **75% more real-world requests** than baseline systems while strictly adhering to SLOs, a feat impossible for systems that blindly accept traffic until collapse.

### 4.2 Chunked Pipeline Parallelism (CPP): Breaking the Communication Barrier for Long Contexts
**Innovation Type:** Architectural Innovation

Handling context lengths scaling from 8k to 128k+ tokens requires distributing the prefill stage across multiple nodes. Prior disaggregated approaches relied heavily on **Sequence Parallelism (SP)** (e.g., Ring Attention), which partitions sequences across nodes but requires frequent cross-node communication (all-reduce operations) at *every* transformer layer.

*   **The Distinction:** Mooncake identifies that SP creates a bottleneck where network bandwidth competes between **model synchronization** (required by SP) and **KVCache transfer** (required by disaggregation). Mooncake replaces this with **Chunked Pipeline Parallelism (CPP)** (Section 5.1). Instead of synchronizing every layer, CPP processes large chunks of tokens in a pipeline fashion, requiring communication only at pipeline stage boundaries.
*   **Why It Matters:** This design decouples the communication frequency from the model depth.
    1.  **Higher MFU:** By eliminating per-layer synchronization, GPUs spend more time computing and less time waiting for network packets, significantly improving Model FLOPs Utilization (MFU).
    2.  **Network Preservation:** It frees up critical RDMA bandwidth for the concurrent transfer of KVCache blocks to decoding nodes, a task that would otherwise be starved by SP traffic.
    3.  **Elasticity:** Unlike static SP configurations, CPP naturally handles variable context lengths without requiring complex dynamic reconfiguration of communication groups.
*   **Impact:** This enables Mooncake to scale prefill processing to massive context lengths with linear efficiency gains, achieving up to a **525% increase in throughput** in simulated long-context scenarios compared to baselines that struggle with SP overheads.

### 4.3 KVCache as a First-Class, Disaggregated Resource
**Innovation Type:** System Design Innovation

In traditional and even many modern serving systems, the KVCache is treated as local state tightly coupled to the GPU instance executing the request. If a request moves, the cache must move with it, often incurring high latency or forcing recomputation.

*   **The Distinction:** Mooncake elevates the KVCache to a **first-class citizen** independent of compute. It constructs a **Distributed KVCache Pool** by aggregating underutilized CPU DRAM and SSDs across the entire cluster (Section 3). This transforms the cache from a local constraint into a globally addressable, tiered storage system.
*   **Why It Matters:** This architecture enables two critical capabilities that local caching cannot support:
    1.  **Decoupled Scheduling:** The Conductor can schedule a prefill job on Node A and a decoding job on Node B without worrying about physical proximity, because the KVCache is streamed asynchronously via RDMA (Section 3.4).
    2.  **Automated Hot-Spot Migration:** Instead of relying on complex global predictors to guess future cache usage, Mooncake uses a heuristic where scheduling decisions *drive* cache replication. When a request is routed to a less-loaded node that lacks the optimal prefix cache, the system automatically pulls that cache, effectively replicating "hot" blocks (like system prompts) to where the load is. This creates a self-balancing cache distribution without explicit migration commands.
*   **Impact:** This approach maximizes cache reuse (reducing redundant computation) while preventing network congestion from "thundering herd" scenarios where all nodes try to fetch the same popular block from a single source. It allows the system to achieve high cache hit rates even with dynamic, unpredictable workloads.

### 4.4 Layer-Wise Streaming: Hiding Memory Latency via Fine-Grained Overlap
**Innovation Type:** Implementation Innovation

A common bottleneck in disaggregated systems is the latency of transferring massive KVCache blocks between CPU memory and GPU VRAM. Standard approaches often wait for the full prefill to complete before transferring the entire cache, leading to significant idle time.

*   **The Distinction:** Mooncake implements **Layer-Wise Prefill** (Section 5.2), a fine-grained streaming mechanism where KVCache blocks are transferred layer-by-layer, strictly overlapping with the computation of subsequent layers. As soon as Layer $L$ finishes computation, its output is streamed to the decoder's CPU memory while Layer $L+1$ begins computation.
*   **Why It Matters:** This technique effectively hides the data transfer latency behind computation time. Crucially, it reduces the **VRAM occupation cost** ($Size \times Time$) of the prefill instance. Because the cache is dumped to CPU DRAM immediately after use, the prefill node only needs enough VRAM to hold the active layer's state, not the entire sequence history.
*   **Impact:** This relaxation of VRAM constraints allows the scheduler to ignore GPU memory capacity as a limiting factor for prefill placement. It enables the system to process extremely long contexts on nodes that would otherwise run out of memory, further enhancing the flexibility of the disaggregated architecture.

These innovations collectively transform Mooncake from a mere inference engine into a robust, production-grade orchestration platform capable of sustaining exponential workload growth while maintaining strict latency guarantees.

## 5. Experimental Analysis

This section critically evaluates Mooncake's performance claims through the lens of the paper's experimental methodology. The authors move beyond synthetic benchmarks to validate their architecture against real-world production traces from Kimi, focusing specifically on the twin challenges of **long-context efficiency** and **overload stability**.

### 5.1 Evaluation Methodology

To ensure reproducibility while protecting user privacy, the authors constructed a rigorous testing environment that balances public benchmarks with proprietary production data.

**Datasets and Workloads**
The evaluation utilizes four distinct datasets, detailed in **Table 2**:
*   **ArXiv Summarization:** A public dataset featuring long inputs (avg. 8,088 tokens) and moderate outputs (avg. 229 tokens) with negligible cache reuse (~0%). This tests raw compute throughput.
*   **L-Eval:** A long-context benchmark with very long inputs (avg. 19,019 tokens) and high prefix cache reuse (>80%). This specifically targets Mooncake's cache-centric optimizations.
*   **Simulated Data:** Synthetic workloads with controlled input lengths (16k, 32k, 64k, 128k) and a fixed 50% cache ratio. This isolates the impact of extreme context lengths.
*   **Real Data:** A sampled trace of 23,608 requests from Kimi's production environment over one hour. This trace preserves actual arrival timestamps, input/output lengths (avg. 7,955 input / 194 output), and remapped block hashes for cache analysis. As noted in **Section 4**, this is the first open-source dataset enabling real-world reuse analysis.

**Experimental Setup and Baselines**
*   **Hardware:** Experiments were conducted on a cluster of nodes, each equipped with **8 NVIDIA A800-SXM4-80GB GPUs** connected via NVLink, with **800 Gbps RDMA** interconnects between nodes.
*   **Model:** To protect proprietary weights, all results use a "dummy" model matching the architecture of **LLaMA2-70B**.
*   **Baseline:** The primary baseline is **vLLM**, a state-of-the-art open-source serving system using PagedAttention and continuous batching. The authors note that vLLM couples prefill and decoding, making it susceptible to head-of-line blocking.
*   **Configurations:**
    *   **vLLM-[4M]:** 4 coupled instances (standard deployment).
    *   **Mooncake-[3P+1D]:** 3 Prefill instances + 1 Decoding instance (compute-heavy bias).
    *   **Mooncake-[2P+2D]:** 2 Prefill instances + 2 Decoding instances (balanced).
    *   **Mooncake-[10P+10D] vs vLLM-[20M]:** Used for real workload replay to match total GPU count (20 GPUs).

**Metrics and SLOs**
The core metric is **Goodput**: the throughput of requests that *successfully complete* while meeting Service Level Objectives (SLOs).
*   **SLO Definitions:** The experiments define strict latency constraints:
    *   **TTFT (Time to First Token):** P90 latency must be $\le 10\times$ the baseline single-request latency.
    *   **TBT (Time Between Tokens):** P90 latency must be $\le 5\times$ the baseline.
*   **Failure Condition:** Any request exceeding these thresholds is considered a failure, and its consumed resources are counted as wasted. The graphs normalize latencies against these limits, where a value $>1.0$ indicates an SLO violation.

### 5.2 Quantitative Results

The experiments provide compelling evidence that Mooncake's disaggregated architecture significantly outperforms coupled baselines, particularly as context length and cache reuse increase.

**Performance on Public Datasets (Figure 11)**
On standard benchmarks, Mooncake demonstrates superior throughput while maintaining SLO compliance:
*   **ArXiv Summarization:** Mooncake-[3P+1D] achieves a **20% higher throughput** than vLLM-[4M] before hitting the TTFT SLO limit.
*   **L-Eval:** The advantage widens to **40% higher throughput**. The authors attribute this gain to Mooncake's efficient prefix caching, which drastically reduces the recomputation required for the highly repetitive inputs in this dataset.
*   **Configuration Sensitivity:** Interestingly, the balanced Mooncake-[2P+2D] configuration showed lower TTFT performance than the prefill-heavy [3P+1D] setup on these specific datasets. The authors explain this as a temporary load imbalance; in production, the ratio of prefill to decoding nodes can be statically tuned to match workload characteristics.

**Dominance in Long-Context Scenarios (Figure 12)**
The most dramatic results appear in the simulated long-context experiments, where traditional architectures struggle:
*   **Throughput Gains:** As input lengths scale from 16k to 128k, Mooncake's throughput advantage over vLLM grows exponentially. The paper reports throughput increases ranging from **50% to 525%** depending on the sequence length.
*   **SLO Stability:** While vLLM's TBT latency spikes uncontrollably as long prefill requests block the decoding batch (violating SLOs), Mooncake maintains stable TBT metrics across all request rates. This confirms that physically separating the compute-bound prefill stage prevents it from starving the memory-bound decoding stage.
*   **Batching Efficiency:** Notably, the baseline vLLM was forced to process requests *individually* (no batching) in these tests to prevent total system collapse, whereas Mooncake successfully utilized continuous batching, further amplifying its throughput lead.

**Real-World Workload Validation (Figure 13)**
Replaying the Kimi production trace provides the strongest validation of Mooncake's production readiness:
*   **Request Capacity:** Mooncake-[10P+10D] successfully handled **75% more requests** than the equivalent 20-GPU vLLM-[20M] cluster while adhering to SLOs.
*   **TBT Compliance:** The Cumulative Distribution Function (CDF) plots reveal a stark contrast in latency stability.
    *   **TTFT:** Both systems performed similarly, with nearly 100% of requests meeting the 30-second TTFT cap.
    *   **TBT:** Mooncake achieved **~100% compliance** with the 0.1s/token TBT SLO. In contrast, vLLM failed to meet the TBT SLO for **43% of requests** (only 57% compliance), with some requests experiencing extreme latency spikes. This empirically proves that coupled architectures cannot guarantee smooth generation under mixed, real-world loads.

**Overload Management and Early Rejection (Table 3)**
The paper also evaluates the "Prediction-Based Early Rejection" policy designed to stabilize the system during overload (simulated by replaying traces at 2x speed):
*   **Baseline Strategy:** A naive approach that rejects based on current load resulted in **4,183 rejected requests**. Crucially, many of these rejections likely occurred *after* wasted prefill computation.
*   **Early Rejection:** Simply checking decoding load before prefill reduced rejections to **3,771**, saving compute resources.
*   **Prediction-Based Early Rejection:** By forecasting future load, Mooncake further reduced rejections to **3,589**.
*   **Significance:** While the raw reduction in rejected requests (~14% vs baseline) might seem modest, the critical win is **stability**. As shown in **Figure 9** and **Figure 10**, naive early rejection causes severe anti-phase oscillations (feast-or-famine cycles) between prefill and decoding pools. The prediction-based approach smooths these fluctuations, ensuring consistent resource utilization and preventing the "thundering herd" problem where a batch of finished prefill requests simultaneously overwhelms the decoder.

### 5.3 Critical Assessment

**Strengths of the Evaluation**
*   **Realism:** The inclusion of the Kimi production trace is a major strength. Many serving papers rely solely on synthetic Poisson arrivals, which fail to capture the burstiness and cache locality of real user sessions.
*   **SLO-Centric Metrics:** By defining success as "requests completed within SLOs" rather than raw throughput, the evaluation aligns with actual business goals (revenue/user experience) rather than academic vanity metrics.
*   **Ablation of Scheduling Policies:** The comparison between random, load-balancing, cache-aware, and full KVCache-centric scheduling (**Figure 8**) clearly isolates the value of each component. The full KVCache-centric approach reduced average TTFT from **92.07s** (random) to **6.26s**, a **14.7x improvement**.

**Limitations and Trade-offs**
*   **Static Node Partitioning:** The experiments highlight a sensitivity to the ratio of prefill vs. decoding nodes. In **Figure 11**, the [2P+2D] configuration underperformed [3P+1D] on certain metrics due to load imbalance. The paper acknowledges that while static tuning works for stable periods, dynamic conversion between node roles remains future work. This implies operational overhead for cluster managers to manually adjust ratios as workload patterns shift.
*   **Dummy Model Constraints:** While the architecture matches LLaMA2-70B, the use of a dummy model means the results do not account for potential nuances in real weight loading or specific kernel optimizations that might vary with proprietary models. However, given that the bottlenecks are memory bandwidth and network transfer (architecture-bound), this limitation is likely minor.
*   **Prediction Accuracy:** The overload management relies on a system-level prediction assuming uniform decoding time ($t_d$). The paper admits that request-level output length prediction is left for future work. In scenarios where output lengths vary wildly (e.g., some users stop after 10 tokens, others generate 2,000), the uniform assumption could lead to suboptimal rejection decisions, though the current results suggest it is sufficient for stabilization.

**Conclusion on Experimental Validity**
The experiments convincingly support the paper's central thesis: **disaggregation is essential for long-context, overloaded LLM serving.** The data clearly shows that coupled systems like vLLM hit a hard wall in TBT stability as context lengths grow, whereas Mooncake scales linearly. The 75% increase in handled requests on real traces is a definitive proof point that the architectural complexity of Mooncake yields tangible production benefits. The only conditional aspect is the need for careful tuning of the prefill/decoding node ratio, a trade-off the authors openly discuss and propose to address in future iterations.

## 6. Limitations and Trade-offs

While Mooncake demonstrates significant improvements in throughput and stability for long-context, overloaded LLM serving, its architectural choices introduce specific trade-offs and rely on assumptions that may not hold in all environments. A critical analysis of the paper reveals limitations in dynamic resource elasticity, prediction granularity, and operational complexity.

### 6.1 Static Resource Partitioning and Elasticity Constraints
The most prominent operational limitation identified in the evaluation is the **static partitioning of prefill and decoding nodes**. Mooncake's architecture requires the cluster to be physically or logically divided into a Prefill Pool and a Decoding Pool prior to workload execution.

*   **Evidence of Imbalance:** In the end-to-end experiments on public datasets (**Figure 11**), the balanced configuration (`Mooncake-[2P+2D]`) underperformed the prefill-heavy configuration (`Mooncake-[3P+1D]`) on TTFT metrics. The authors attribute this to a temporary mismatch between the fixed node ratio and the specific workload characteristics of the ArXiv and L-Eval datasets.
*   **The Trade-off:** This design forces operators to make a priori assumptions about the ratio of compute-bound (prefill) to memory-bound (decoding) work. While the paper notes that "demand... generally remains stable over certain periods" (**Section 8.1.1**), real-world traffic often exhibits rapid shifts (e.g., a sudden surge in chat interactions vs. document summarization).
*   **Unaddressed Dynamic Scaling:** The paper explicitly acknowledges that "dynamic conversion between node roles" is left for **future work** (**Section 8.1.1**). Unlike coupled systems where a single GPU can seamlessly switch between prefill and decoding tasks based on immediate queue depth, Mooncake currently lacks a mechanism to instantly repurpose a decoding node as a prefill node (or vice versa) without restarting services or reconfiguring the cluster. This rigidity can lead to underutilization if the workload distribution shifts away from the预设 (preset) ratio.

### 6.2 Coarse-Grained Load Prediction
The stability of Mooncake's overload management relies heavily on its **Prediction-Based Early Rejection** policy (**Section 7.4**). However, the current implementation uses a simplified, system-level prediction model that introduces potential inaccuracies.

*   **Uniform Decoding Assumption:** The prediction algorithm assumes that every request's decoding stage takes a **uniform time ($t_d$)** per token. It does not attempt to predict the specific output length of individual requests. The authors state, "The exploration of request-level prediction is left for future work" (**Section 7.4**).
*   **Scenario Vulnerability:** This assumption works well for aggregate load smoothing but fails in scenarios with high variance in generation length. For example, if a batch of accepted requests unexpectedly generates very long outputs (e.g., code generation or creative writing) while the model assumed short answers, the predicted decoding load will be significantly lower than the actual future load.
*   **Consequence:** In such edge cases, the system might accept more requests than it can actually handle within the TBT SLO, leading to localized SLO violations that the coarse predictor failed to anticipate. Conversely, if the workload consists mostly of short answers, the system might be overly conservative, rejecting valid requests and sacrificing goodput unnecessarily.

### 6.3 Network Dependency and Infrastructure Costs
Mooncake's performance is inextricably linked to the availability of high-bandwidth, low-latency networking, specifically **RDMA (Remote Direct Memory Access)**.

*   **Architectural Dependency:** The "Distributed KVCache Pool" and the "Layer-wise Prefill" streaming mechanism (**Section 5.2**) rely on the `Messenger` component to transfer massive amounts of data between CPU DRAM and GPU VRAM across nodes. The experiments were conducted on a cluster with **800 Gbps RDMA** interconnects (**Section 8.1**).
*   **Scalability Constraint:** In cloud environments or clusters where high-speed interconnects (like InfiniBand or RoCE) are unavailable or cost-prohibitive, the latency of transferring KVCache blocks could become a bottleneck. If the network transfer time ($T_{transfer}$) exceeds the computation time, the benefits of disaggregation and layer-wise streaming diminish, potentially making the system slower than a coupled architecture where data stays local.
*   **Congestion Risks:** While the paper mentions that Chunked Pipeline Parallelism (CPP) reduces network contention compared to Sequence Parallelism (**Section 5.1**), the system still relies on the network for *all* cache reuse and state transfer. In extreme "thundering herd" scenarios where many requests simultaneously require the same hot cache blocks, network congestion could still occur, despite the heuristic-based hot-spot migration.

### 6.4 Operational Complexity and Tuning
The shift from a coupled to a disaggregated, KVCache-centric architecture increases the system's configuration surface area, requiring careful manual tuning.

*   **Hyperparameter Sensitivity:** The effectiveness of the cache-aware scheduling depends on the `kvcache_balancing_threshold` (**Algorithm 1**, **Section 6.2**). The paper admits this threshold is "currently adjusted manually" and only suggests it *could* be adaptively adjusted in the future. Incorrect tuning of this parameter could lead to suboptimal decisions: setting it too low might cause excessive data transfer (network bound), while setting it too high might lead to queue congestion (compute bound).
*   **Chunk Size Configuration:** The `prefill_chunk` size (typically $>1000$ tokens) must be tuned to balance GPU utilization against pipeline overhead (**Section 3**). This adds another dimension of complexity for cluster administrators who must profile their specific hardware and model architecture to find the optimal setting.
*   **Debugging Difficulty:** In a coupled system, debugging a latency spike involves looking at a single node. In Mooncake, a latency issue could stem from the prefill queue, the RDMA transfer bandwidth, the CPU memory pressure in the cache pool, or the decoding queue. Tracing the root cause across these disaggregated components is inherently more complex.

### 6.5 Unaddressed Edge Cases
Several specific scenarios remain unexplored or explicitly excluded from the current scope:

*   **Heterogeneous Hardware:** The experiments assume a homogeneous cluster of NVIDIA A800 GPUs. The architecture's behavior when mixing different GPU generations (e.g., A100s with H100s) or integrating specialized inference accelerators is not evaluated. The varying compute speeds and memory bandwidths would complicate the $T_{prefill}$ and $T_{transfer}$ estimation models significantly.
*   **Multi-Tenant Isolation:** While the paper discusses priority-based scheduling as future work (**Section 10**), the current evaluation does not demonstrate strong isolation guarantees between different tenants or API keys. In a shared cluster, a "noisy neighbor" generating massive cache traffic could theoretically saturate the RDMA network, impacting the TTFT of other users, a problem the current heuristic balancing may not fully mitigate.
*   **Model Architecture Specificity:** The evaluation uses a dummy model matching **LLaMA2-70B**. It does not address how the system handles models with different architectural quirks, such as Mixture-of-Experts (MoE) models (like DeepSeek-V2 mentioned in **Section 10**) where the memory access patterns and compute requirements differ significantly from dense transformers. The layer-wise streaming logic might need substantial modification to handle the sparse activation patterns of MoE layers efficiently.

In summary, while Mooncake solves the critical problems of long-context latency and overload instability, it trades off **operational simplicity** and **dynamic elasticity** for these gains. Its success is contingent on high-speed networking, accurate (albeit coarse) load predictions, and careful manual tuning of the prefill/decoding node ratio, leaving open questions about its adaptability to highly volatile, heterogeneous, or multi-tenant production environments.

## 7. Implications and Future Directions

Mooncake represents a pivotal shift in the architecture of Large Language Model (LLM) serving systems, moving the field from a paradigm of **resource utilization maximization** to one of **overload-aware goodput optimization**. By successfully decoupling compute from state and treating the KVCache as a first-class, globally schedulable resource, Mooncake provides a blueprint for sustaining exponential workload growth in production environments where hardware supply is constrained. The implications of this work extend beyond immediate performance gains, reshaping how researchers and engineers approach system design for long-context inference.

### 7.1 Reshaping the Landscape: From Throughput to Goodput
The most profound impact of Mooncake is its redefinition of the primary optimization objective for LLM serving. Prior seminal works like *vLLM* [13] and *Orca* [12] fundamentally optimized for **throughput** (tokens per second) under the implicit assumption that the cluster could eventually process all incoming requests. Mooncake challenges this assumption, demonstrating that in high-growth Model-as-a-Service (MaaS) scenarios, **overload is the steady state, not the exception**.

*   **The Goodput Paradigm:** Mooncake establishes that maximizing raw throughput is counterproductive if it leads to SLO violations and wasted computation on requests that must eventually be dropped. By introducing **Prediction-Based Early Rejection** (Section 7.4), the paper argues that the true metric of success is **goodput**: the rate of *successfully completed* requests that meet latency constraints. This shifts the research focus from "how fast can we compute?" to "how many requests can we guarantee to finish?"
*   **Disaggregation as a Necessity, Not an Option:** While previous disaggregated approaches (e.g., *Splitwise* [7], *DistServe* [8]) treated separation as an optimization for specific workloads, Mooncake proves it is a structural requirement for handling **extreme long contexts** (128k+ tokens). The introduction of **Chunked Pipeline Parallelism (CPP)** demonstrates that traditional Sequence Parallelism (SP) creates untenable network contention when combined with the massive data transfers required for KVCache mobility. This suggests that future serving architectures for long-context models must inherently separate prefill and decoding pools to avoid head-of-line blocking.

### 7.2 Enabling Follow-Up Research
Mooncake's architecture opens several critical avenues for future investigation, particularly in refining the intelligence of the scheduler and expanding the hardware horizon.

*   **Request-Level Load Prediction:** Currently, Mooncake relies on a system-level prediction model assuming a uniform decoding time per token (Section 7.4). This coarse-grained approach leaves room for significant improvement. Future research could integrate **LLM-based output length predictors** to estimate the specific generation length of individual requests upon arrival. Accurate request-level prediction would allow the scheduler to make granular accept/reject decisions, further smoothing load fluctuations and increasing goodput beyond the current ~14% improvement over naive rejection.
*   **Dynamic Elasticity and Node Role Conversion:** A current limitation identified in Section 6.1 is the static partitioning of nodes into prefill or decoding roles. Follow-up work should explore **elastic role conversion**, where nodes can dynamically switch functions based on real-time demand ratios without service interruption. This would require developing lightweight state migration protocols to transfer a decoding instance's active KVCache to the global pool instantly, allowing the GPU to pivot to prefill tasks.
*   **Heterogeneous Accelerator Architectures:** The paper explicitly points toward a future where serving systems leverage **heterogeneous hardware** (Section 10). Mooncake's disaggregated design is uniquely positioned to exploit this:
    *   **Compute-Oriented Nodes:** High-end GPUs (e.g., H100s) dedicated solely to the compute-bound prefill stage.
    *   **Bandwidth-Oriented Nodes:** Cost-effective accelerators with massive memory bandwidth (e.g., using GDDR6, LPDDR, or emerging Process-in-Memory technologies) optimized for the memory-bound decoding stage.
    *   **Research Opportunity:** Developing scheduling algorithms that can map prefill and decoding tasks to these distinct hardware classes while managing the varying transfer speeds between them is a rich area for exploration.
*   **Advanced KVCache Compression and Selection:** Since Mooncake treats KVCache as a mobile resource, any reduction in cache size directly improves transfer speeds and storage capacity. Future work can integrate **KVCache compression techniques** (e.g., quantization, pruning, or layer sharing [48-63]) directly into the `Messenger` transfer pipeline. Furthermore, research into **smart eviction policies** that understand semantic importance (beyond simple LRU/LFU) could further boost the cache hit ratio in the distributed pool.

### 7.3 Practical Applications and Downstream Use Cases
The architectural choices in Mooncake make it particularly well-suited for specific classes of applications that struggle on traditional coupled systems.

*   **Long-Context RAG and Document Analysis:** Applications involving Retrieval-Augmented Generation (RAG) over massive corpora or single-document analysis (e.g., legal contract review, scientific paper summarization) often involve input lengths of 50k–100k tokens. Mooncake's CPP mechanism and tiered cache pool allow these systems to maintain low Time to First Token (TTFT) even when multiple users query different sections of the same large document, leveraging shared prefix caches efficiently.
*   **Multi-Turn Conversational Agents with Memory:** For chatbots that maintain long conversation histories (e.g., therapeutic agents, coding assistants with full project context), Mooncake's ability to store and replicate "hot" conversation prefixes across the cluster ensures that returning users experience low latency without recomputing their entire history. The **automated hot-spot migration** ensures that popular context blocks (like system prompts or common persona definitions) are naturally replicated where needed.
*   **Batch Processing and Asynchronous Jobs:** The decoupling of VRAM constraints via **Layer-Wise Prefill** (Section 5.2) enables new service tiers. As noted in Section 5.2, Mooncake could offer a "batch API" where non-urgent, massive-context jobs are processed by inlining decoding into the prefill stream, utilizing idle VRAM space that would otherwise be reserved. This allows providers to monetize spare capacity with lower-cost, higher-latency services.

### 7.4 Reproducibility and Integration Guidance
For practitioners and researchers looking to adopt or build upon Mooncake's principles, the following guidance outlines when and how to apply these techniques.

*   **When to Prefer Mooncake's Architecture:**
    *   **Workload Characteristic:** Adopt this disaggregated approach if your workload features **high variance in context length** (mix of short chats and 100k+ token documents) or **high prefix reuse** (multi-turn conversations, shared system prompts).
    *   **Operational State:** This architecture is critical when operating in **overloaded regimes** where request arrival rates frequently exceed the cluster's maximum sustainable throughput. If your system rarely queues requests, the complexity of disaggregation may yield diminishing returns compared to simpler coupled systems like vLLM.
    *   **Infrastructure Requirement:** Successful implementation requires a **high-bandwidth, low-latency network** (preferably RDMA/InfiniBand). The benefits of the Distributed KVCache Pool and Layer-Wise Streaming are contingent on the ability to move gigabytes of cache data between nodes faster than the compute time. On clusters limited to standard Ethernet, the transfer latency may negate the gains from disaggregation.

*   **Integration Strategy:**
    *   **Start with Cache Disaggregation:** If a full prefill/decoding split is too disruptive, practitioners can first implement Mooncake's **Distributed KVCache Pool** (Section 3) on top of existing coupled nodes. Utilizing underutilized CPU DRAM and SSDs for tiered caching can immediately improve cache hit rates and reduce recomputation costs.
    *   **Adopt Prediction-Based Rejection:** The **Early Rejection based on Prediction** logic (Section 7.4) is modular and can be integrated into existing schedulers. Even a simple system-level predictor assuming uniform decoding time can significantly stabilize latency and prevent resource waste compared to reactive rejection policies.
    *   **Leverage Open Source Traces:** To benchmark new scheduling algorithms, researchers should utilize the **open-sourced Kimi request trace** provided by the authors (Section 4). This dataset, containing 23,608 real-world requests with hash IDs for cache analysis, is the first of its kind and offers a realistic baseline for evaluating cache reuse policies and overload handling, surpassing synthetic Poisson distributions.

In conclusion, Mooncake does not merely offer an incremental speedup; it provides a robust architectural framework for the next generation of LLM services. By acknowledging the realities of hardware scarcity and overload, it shifts the industry toward systems that are not only fast but also **resilient, predictable, and economically efficient** at scale.
