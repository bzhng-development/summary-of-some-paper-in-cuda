## 1. Executive Summary

This paper introduces `TetriInfer`, a cloud-scale LLM inference system that eliminates performance degradation caused by mixing computation-heavy prefill and memory-intensive decode phases through three key mechanisms: partitioning prompts into fixed-size 512-token chunks, disaggregating prefill and decode into independent instances, and employing a two-level scheduler augmented by an `OPT-125M` length prediction model. By addressing these interference issues on mixed workloads derived from the ShareGPT dataset, `TetriInfer` significantly outperforms the baseline `vLLM` system, reducing average Time-to-First-Token (TTFT) by up to 97% and Job Completion Time (JCT) by 47% while using 38% fewer hardware resources. These improvements translate to a 2.4x increase in performance-per-dollar for common light-prefill/heavy-decode tasks, demonstrating that careful request scheduling and phase separation can drastically optimize LLM serving efficiency without requiring new hardware.

## 2. Context and Motivation

### The Hidden Cost of Mixed Workloads
Large Language Model (LLM) inference is not a monolithic operation; it consists of two distinct phases with fundamentally different resource profiles. The **prefill phase** processes the user's input prompt to generate the first token and the initial Key-Value (KV) cache. This phase is **computation-bound**, meaning its performance is limited by the GPU's ability to perform matrix multiplications. Specifically, the computational cost scales quadratically ($O(N^2)$) with the input prompt length. In contrast, the **decode phase** generates subsequent tokens one by one in an auto-regressive manner. This phase is **memory-bound**, limited by the bandwidth required to fetch model weights and the KV cache from memory for each new token. Its resource usage scales sublinearly with the number of generated tokens.

The core problem addressed by this paper is **interference**. In real-world cloud environments, LLM services do not handle uniform requests. As illustrated in **Figure 1**, downstream tasks vary wildly:
*   **Summarization tasks** involve long input prompts (heavy prefill) but short outputs (light decode).
*   **Content creation tasks** involve short prompts (light prefill) but long outputs (heavy decode).
*   Token lengths across these tasks can differ by more than two orders of magnitude.

Current deployment practices, such as those used in vanilla `vLLM`, typically co-locate these phases on the same hardware instances using continuous batching. The authors argue that this "one-size-fits-all" approach ignores the distinct characteristics of the two phases, leading to severe performance degradation when mixed workloads arrive simultaneously.

### Empirical Evidence of Interference
To quantify the severity of this issue, the authors conducted extensive experiments mixing requests of varying lengths. The results, detailed in **Section 2.2**, reveal three specific types of interference that cripple efficiency:

1.  **Prefill vs. Prefill Interference:** When multiple prefill requests are batched together, the total number of tokens can exceed the accelerator's saturation point. **Figure 3** shows that adding concurrent light prefill requests to a batch can increase latency by **2x to 8x**. If a light prefill request is mixed with heavy prefill requests, the slowdown exceeds **10x**. This occurs because the hardware is already computation-saturated; adding more jobs simply extends the queue without increasing throughput.
2.  **Prefill vs. Decode Interference:** This is the most damaging scenario. Mixing a computation-heavy prefill job with a latency-critical decode job creates a conflict. **Figure 4** demonstrates that a single heavy prefill request co-running with a light decode request can increase the decode iteration time by **5x**. Conversely, running decode requests alongside prefill requests can slow down the prefill phase by up to **2.5x**. The system fails because it attempts to run a batch job (prefill) and a latency-sensitive job (decode) on the same saturated resources.
3.  **Decode vs. Decode Interference:** Even within the decode phase, mixing requests of different lengths causes issues. **Figure 5** indicates that mixing heavy decode requests (long generation) with light decode requests leads to a **16% drop in throughput** and a **23% increase in latency** compared to batching only light requests. This is caused by memory bandwidth contention and "head-of-line blocking," where short jobs are stuck waiting behind long jobs that hold memory resources.

### Limitations of Prior Approaches
Before `TetriInfer`, the standard approach to LLM serving relied on systems like `vLLM` (referenced throughout the paper), which optimize memory management via paging but still couple the prefill and decode phases on the same instances.
*   **Static Provisioning:** A naive solution to avoid interference would be to provision separate hardware clusters for every type of downstream task (e.g., one cluster for summarization, another for chat). However, the paper notes this is **impractical** due to the high cost of LLM infrastructure and the dynamic nature of traffic. Static provisioning leads to massive resource waste when traffic patterns shift.
*   **Existing Optimizations:** Prior works focused on optimizing batching strategies or memory paging but did not fundamentally address the architectural flaw of running conflicting phases (computation-heavy vs. memory-intensive) on the same hardware simultaneously. As noted in **Table 1**, systems like `Sarathi` introduced chunked prefill but still ran mixed prefill-decode chunks, failing to eliminate the interference between the two phases entirely.

### Positioning of TetriInfer
`TetriInfer` positions itself as a **disaggregated architecture** specifically designed to battle these interference patterns. Unlike prior systems that treat the LLM instance as a black box handling both phases, `TetriInfer` decomposes the serving pipeline:
*   It treats the prefill phase as a **batch processing job** that can be chunked and scheduled independently to maximize compute utilization.
*   It treats the decode phase as a **latency-critical service** that requires dedicated memory bandwidth and careful scheduling to avoid hotspots.
*   It introduces a **predictive scheduling layer** using a small, fine-tuned LLM (`OPT-125M`) to estimate the length of generated tokens, allowing the system to proactively balance loads rather than reacting to contention after it occurs.

By shifting from a coupled model to a disaggregated, schedule-aware architecture, `TetriInfer` aims to achieve the theoretical efficiency limits of the hardware, ensuring that computation-heavy and memory-intensive tasks never compete for the same resources at the same time.

## 3. Technical Approach

### 3.1 Reader orientation (approachable technical breakdown)
`TetriInfer` is a distributed LLM serving system that physically separates the computation-heavy "prefill" phase from the memory-intensive "decode" phase onto different server instances to prevent them from slowing each other down. It solves the problem of performance interference in mixed workloads by breaking large prompts into fixed-size chunks, predicting how long responses will be using a small auxiliary model, and intelligently routing requests to keep all hardware fully utilized without contention.

### 3.2 Big-picture architecture (diagram in words)
The system operates as a coordinated pipeline involving four primary logical components: a **Centralized Control Plane**, **Prefill Instances**, **Decode Instances**, and a **Length Prediction Model**.
*   The **Centralized Control Plane** acts as the traffic controller, receiving incoming user requests, monitoring the load of all instances, and dynamically scaling or flipping the role of instances (from prefill to decode or vice versa) based on demand.
*   **Prefill Instances** are dedicated workers that accept raw prompts, sort them using specific scheduling policies, break them into fixed-size computation chunks, run the main LLM to generate the initial Key-Value (KV) cache, and predict the output length.
*   The **Length Prediction Model** is a lightweight, fine-tuned classifier (e.g., `OPT-125M`) running alongside the main LLM on prefill instances to estimate the number of tokens a request will generate, providing crucial metadata for downstream scheduling.
*   **Decode Instances** are dedicated workers that receive the prefilled KV cache and metadata from prefill instances, managing the auto-regressive generation of tokens using memory-aware scheduling policies to prevent thrashing.
*   **Data Flow:** A request enters the Control Plane, is routed to a Prefill Instance, processed into chunks while predicting length, has its resulting KV cache transferred over the network to a selected Decode Instance, and finally streams the generated tokens back to the user via the Control Plane.

### 3.3 Roadmap for the deep dive
*   **Control Plane & Instance Lifecycle:** We first explain the centralized coordinator that manages request intake and the unique ability of `TetriInfer` to dynamically "flip" hardware roles between prefill and decode tasks.
*   **Prefill Optimization (Chunking & Scheduling):** We detail how input prompts are sliced into fixed-size chunks to saturate compute units and how scheduling policies (FCFS, SJF, LJF) organize these chunks to minimize latency.
*   **Length Prediction Mechanism:** We describe the auxiliary small-model architecture used to predict output lengths, including the training process, granularity choices, and the parallel execution mode that minimizes overhead.
*   **KV Cache Transfer & Dispatching:** We analyze the network abstraction layer that moves KV caches between instances and the decentralized "power-of-two" algorithm used to select the optimal decode instance.
*   **Decode Instance Scheduling:** We conclude with the memory-aware scheduling policies (`reserve-static` and `reserve-dynamic`) that use predicted lengths to prevent memory thrashing during token generation.

### 3.4 Detailed, sentence-based technical breakdown

This paper presents a system design solution that re-architects LLM inference serving by disaggregating phases and introducing predictive scheduling to eliminate resource contention. The core idea is that by treating prefill as a batch compute job and decode as a latency-sensitive memory job on separate hardware, and by using a small model to predict future resource needs, the system can schedule requests to avoid the interference patterns identified in the motivation section.

**The Centralized Control Plane and Dynamic Instance Flipping**
The system relies on a **Centralized Control Plane** composed of a **Global Scheduler** and a **Cluster Monitor** to manage the cluster without creating a single point of failure.
*   The **Cluster Monitor** collects load statistics (such as queue depth and utilization) from every prefill and decode instance at regular intervals, specifically every **100 ms**, and broadcasts this aggregated view to all prefill instances to enable decentralized decision-making.
*   The **Global Scheduler** maintains a request status table tracking arrival times, current phases, and SLA requirements, and it routes incoming requests to the prefill instance with the currently lowest load.
*   A unique feature of this architecture is the **Instance Flip** mechanism, which allows the system to dynamically adjust the ratio of prefill-to-decode hardware based on real-time workload shifts rather than static provisioning.
*   When the **Transition Watcher** module in the control plane detects that an instance type is underutilized (e.g., load below **10%** for the past minute), it initiates a role flip.
*   To flip a **Prefill Instance** to a **Decode Instance**, the global scheduler stops sending new requests to it, waits for the instance to drain its current queue, and then signals the instance to switch its internal mode, a process that takes approximately **5 to 7 ms** excluding drain time.
*   Flipping a **Decode Instance** to a **Prefill Instance** is slightly more complex because the global scheduler must first notify all active prefill instances to stop forwarding new KV caches to the target decode instance before the role change occurs.
*   This flipping is purely a software state change involving internal variable updates; it does not require restarting processes or reloading the massive LLM weights, ensuring rapid adaptation to workload changes.

**Prefill Instance: Chunked Execution and Scheduling Policies**
The **Prefill Instance** is designed to maximize computational throughput by eliminating the variability of prompt lengths through a technique called **Chunked Prefill**.
*   Instead of processing entire prompts in a single batch (which causes saturation issues when prompts vary wildly in size), the system partitions every input prompt into fixed-size chunks of **512 tokens** (denoted as `ChunkSize` for the `OPT-13B` model on their testbed).
*   If the final chunk of a prompt is smaller than `ChunkSize`, the system pads it with zeros to maintain the fixed dimension, ensuring the accelerator always operates at its computation-saturated limit without incurring the latency penalties of oversized batches.
*   Before chunking, requests pass through a **Prefill Scheduler** that sorts the raw request queue into a scheduled queue using one of three implemented policies: **First-Come-First-Serve (FCFS)**, **Shortest-Job-First (SJF)**, or **Longest-Job-First (LJF)**.
*   To prevent starvation (where short jobs wait indefinitely behind long ones in LJF, or vice versa in SJF), the scheduler employs a `PrefillSchedBatch` parameter that limits the number of requests sorted in a single scheduling round; for example, if the batch size is set to **16**, the scheduler sorts only the top 16 requests according to the policy, processes them, and then re-evaluates the queue.
*   Experimental results indicate that using **SJF** with a batch size of **16** reduces average prefill waiting time by **7.8%** compared to FCFS, and increasing the batch size to **128** can reduce Time-to-First-Token (TTFT) by **46.5%** due to better optimization opportunities.
*   The main LLM engine executes these chunks sequentially, maintaining a pointer for each request to track the last prefilled token position, allowing the system to interleave chunks from different requests efficiently.

**Length Prediction Model: Architecture and Execution**
To enable intelligent scheduling for the decode phase, every prefill instance runs a **Length Prediction Model**, which is a small, fine-tuned LLM classifier designed to estimate the number of tokens a request will generate.
*   The system uses a much smaller model (e.g., **`OPT-125M`**) to predict the behavior of the large target model (e.g., **`OPT-13B`**), leveraging the fact that the small model is roughly **10 times faster** and consumes negligible resources compared to the main inference task.
*   Instead of predicting an exact token count (which is statistically difficult due to sampling parameters like temperature and top-p), the model classifies the output length into fixed-range buckets with a specific **granularity**.
*   The authors found a granularity of **200 tokens** to be the optimal trade-off, achieving a prediction accuracy of **74.9%**, whereas a granularity of **100** yielded **58.9%** accuracy and **400** yielded **85%** accuracy.
*   The training process involves an offline step where prompts from a dataset (like ShareGPT) are fed to the target `OPT-13B` model to generate ground-truth responses, which are then labeled with bucket IDs and used to fine-tune the `OPT-125M` classifier using HuggingFace Trainer APIs.
*   During runtime, the system employs a **Parallel Mode** where the prediction model and the main LLM run simultaneously on the same hardware; this adds no latency to the critical path for over **80%** of requests, though it causes a minor **10%** throughput drop under extreme stress compared to running the large model alone.
*   The output of this predictor is a length range (lower and upper bounds), which provides the necessary metadata for the dispatcher to estimate memory requirements without needing precise token counts.

**Dispatcher and KV Cache Transfer Mechanisms**
Once a request's chunks are prefilled, the **Dispatcher** module within the prefill instance is responsible for selecting a destination decode instance and transferring the generated **Key-Value (KV) Cache**.
*   The dispatcher utilizes a decentralized **Power-of-Two** load balancing algorithm: it first identifies the set of decode instances ($\alpha$) that have sufficient predicted resources to handle the new request, randomly selects two candidates from this set, and assigns the request to the one that minimizes the ratio of heavy-to-light decode jobs to prevent hotspots.
*   The transfer of the KV cache is abstracted through a unified network stack that supports three physical link types: **Direct** (e.g., NVLink with ~900 GB/s bandwidth), **Direct-NIC** (e.g., RoCE with ~200 Gbps bandwidth), and **Indirect** (bouncing through CPU DRAM).
*   The system supports both **One-Sided** (sender initiates memory write on receiver, like RDMA) and **Two-Sided** (standard send/receive) communication primitives to maximize compatibility across different hardware clusters.
*   Although **Chunk-Level** transfer (sending KV cache as each chunk is finished) would allow for better parallelization between computation and networking, the current implementation uses **Request-Level** transfer (sending the aggregated KV cache after all chunks are prefilled) to simplify the engineering complexity.
*   In the evaluation testbed, due to hardware limitations, the authors implemented a **mock mechanism** that emulates network latency by calculating the theoretical transfer time based on the KV cache size and a configured bandwidth (e.g., **200 Gbps** or **300 GB/s**) and forcing the decode instance to wait for that duration before starting generation.

**Decode Instance: Memory-Aware Scheduling**
The **Decode Instance** receives the KV cache and metadata, and its primary challenge is to schedule auto-regressive generation without running out of GPU memory (thrashing) or causing head-of-line blocking.
*   Unlike standard systems like `vLLM` that use a greedy policy (adding requests as long as current memory allows), `TetriInfer` introduces **Working-Set-Aware** scheduling policies that utilize the predicted length ranges.
*   The **Reserve-Static** policy admits a new request into the batch only if its predicted maximum memory usage fits within the currently available accelerator memory, preventing immediate overflow.
*   The **Reserve-Dynamic** policy is more aggressive and efficient: it admits a new request if there will be sufficient memory available when the *shortest remaining job* in the current batch completes, effectively looking ahead into the near future to utilize memory that is about to be freed.
*   These policies rely on the lower bound of the predicted length range to estimate resource usage conservatively; experiments show that with the achieved **74.9%** prediction accuracy, `reserve-dynamic` matches the Job Completion Time (JCT) of the greedy baseline, but with perfect prediction, it could improve JCT by roughly **12%**.
*   The decode instance also runs a local scheduler that groups requests into dynamic batches using continuous batching, but strictly adheres to the memory reservations calculated by the predictive policies to ensure smooth, interference-free execution.

## 4. Key Insights and Innovations

The contributions of `TetriInfer` extend beyond standard engineering optimizations; they represent a fundamental re-architecting of how LLM inference resources are managed in mixed-workload environments. While prior works like `vLLM` focused on memory paging and `Sarathi` on chunked execution within a coupled model, `TetriInfer` introduces three distinct innovations that shift the paradigm from reactive batching to proactive, phase-aware disaggregation.

### 1. Fundamental Phase Disaggregation with Dynamic Role Flipping
The most significant architectural innovation is the **strict physical separation** of the prefill and decode phases into dedicated instances, coupled with a mechanism to dynamically "flip" hardware roles based on real-time demand.

*   **Differentiation from Prior Work:** Previous systems, including `Sarathi` (which introduced chunked prefill), still operate on a **coupled model** where prefill and decode tasks compete for the same GPU resources within a single instance. Even when using continuous batching, these systems inevitably suffer from the interference patterns described in Section 2.2, where a heavy prefill job stalls latency-critical decode iterations. `TetriInfer` is among the first (concurrent with `Splitwise`) to treat prefill and decode as entirely distinct services with separate resource pools.
*   **Why It Matters:** This disaggregation eliminates the root cause of the **5x decode slowdown** and **10x prefill latency spikes** observed in mixed batches. By ensuring computation-heavy prefill jobs never share an accelerator with memory-bound decode jobs, the system guarantees that decode latency remains stable regardless of prefill queue depth.
*   **The Innovation of "Flipping":** A naive disaggregated system would require static provisioning (e.g., 50% prefill nodes, 50% decode nodes), which is inefficient given the highly variable nature of LLM workloads (e.g., summarization vs. creative writing). `TetriInfer`'s **Instance Flip** mechanism (Section 3.5) solves this by allowing the cluster to morph its topology in milliseconds. The ability to switch an instance's role via a simple internal variable update (~5–7 ms) without reloading model weights or restarting processes is a critical systems innovation. It enables the cluster to behave like a fluid resource pool that automatically balances the ratio of compute-to-memory bandwidth based on the incoming traffic mix, a capability absent in static or coupled architectures.

### 2. Auxiliary Model-Based Predictive Scheduling
`TetriInfer` introduces a novel scheduling layer that relies on **speculative execution metadata** generated by a separate, lightweight LLM, rather than reacting to resource contention after it occurs.

*   **Differentiation from Prior Work:** Traditional schedulers (like `vLLM`'s greedy policy) are **oblivious to the future**; they admit requests based solely on current memory availability. This leads to "thrashing," where a batch fills up memory, only to be unable to complete the generation of long sequences, forcing swaps or stalling. `FastServe` uses priority queues but lacks knowledge of the *volume* of work remaining. `TetriInfer` deploys a dedicated `OPT-125M` classifier (Section 3.3.2) specifically trained to predict the output length bucket of the target `OPT-13B` model.
*   **Why It Matters:** This transforms the scheduler from reactive to **proactive**. By knowing a request is likely to generate 500+ tokens (heavy decode) versus 50 tokens (light decode) *before* it enters the decode queue, the system can implement **Working-Set-Aware** policies (`reserve-static` and `reserve-dynamic`).
*   **Significance:** The `reserve-dynamic` policy (Section 3.4) is a theoretical advance in batch scheduling: it admits a new request not based on *current* free memory, but on the *future* free memory expected when the shortest job in the batch finishes. This lookahead capability, enabled entirely by the auxiliary predictor, allows the system to pack batches tighter without risking out-of-memory errors, directly contributing to the **12% JCT improvement** seen in ideal prediction scenarios and preventing the throughput drops caused by head-of-line blocking.

### 3. Compute-Saturated Chunking with Decentralized Dispatch
While chunked prefill exists in prior literature, `TetriInfer` innovates by combining **fixed-size compute saturation** with a **decentralized, prediction-aware dispatch algorithm**.

*   **Differentiation from Prior Work:** `Sarathi` utilizes chunked prefill but mixes prefill and decode chunks in the same batch, meaning the "chunks" are still subject to interference from decode memory bandwidth contention. `TetriInfer` enforces chunks strictly within **prefill-only instances**, ensuring that every chunk operates at the accelerator's **computation-saturated limit** (e.g., 512 tokens for OPT-13B) without any memory-bandwidth noise from decoding.
*   **The Decentralized Twist:** Furthermore, the dispatch logic (Section 3.3.4) moves away from a centralized bottleneck. Instead of a global controller assigning every decode task, `TetriInfer` empowers each prefill instance to make routing decisions using a **Power-of-Two** algorithm augmented with the predicted length range.
*   **Why It Matters:** This design specifically targets the **Decode-vs-Decode interference** (Section 2.2.3). By actively spreading "heavy decode" requests across different instances to maintain a balanced heavy-to-light ratio, the system prevents specific nodes from becoming memory-bandwidth bottlenecks. The result is a **16% throughput recovery** in mixed decode batches compared to naive random assignment. This demonstrates that intelligent, localized load balancing based on predicted workload characteristics is more effective than global, blind distribution.

### Summary of Impact
These innovations are not merely incremental tweaks to batching logic; they address the **structural mismatches** in current LLM serving stacks.
1.  **Disaggregation + Flipping** solves the **hardware resource conflict** (Compute vs. Memory).
2.  **Predictive Scheduling** solves the **temporal uncertainty** (Unknown sequence lengths).
3.  **Decentralized Dispatch** solves the **spatial imbalance** (Hotspots in distributed clusters).

Together, they enable `TetriInfer` to achieve what static or coupled systems cannot: high efficiency across wildly diverse workloads (from chat to summarization) simultaneously, delivering up to **2.4x better performance-per-dollar** by ensuring that no hardware cycle is wasted on interference management.

## 5. Experimental Analysis

### 5.1 Evaluation Methodology and Setup
To validate the efficacy of `TetriInfer`, the authors constructed a rigorous evaluation framework comparing their system against the state-of-the-art baseline, **vanilla `vLLM`**. The experiments are designed to isolate the impact of disaggregation, chunking, and predictive scheduling under realistic, mixed-workload conditions.

**Hardware and Software Environment**
The testbed consists of a single server equipped with **four NVIDIA V100 GPUs** (32GB HBM each), an **Intel Xeon Gold 5218R CPU**, and **256GB DRAM**.
*   **Target Model:** The primary inference workload runs on **OPT-13B**, a large language model with billions of parameters.
*   **Prediction Model:** The auxiliary length predictor uses **OPT-125M**, a significantly smaller model (~10x faster) fine-tuned for classification.
*   **Network Emulation:** Due to limited access to high-end interconnects (like NVLink or RoCE) across multiple physical servers, the authors implemented a **mock mechanism** (described in Section 4) to emulate network latency. This mechanism pre-computes KV caches offline and forces the decode instance to wait for a calculated duration based on target bandwidths:
    *   **TS-RoCE:** Emulates **200 Gbps** bandwidth (representing Direct-NIC links like RoCE).
    *   **TS-NVLink:** Emulates **300 GB/s** bandwidth (representing Direct links like NVLink).
*   **Baseline Configuration:** The vanilla `vLLM` baseline uses its default continuous batching with a fixed batch size of **16** for prefill. `TetriInfer` configurations use a `ChunkSize` of **512 tokens** (the accelerator-saturation threshold for OPT-13B on this hardware) and a `PrefillSchedBatch` of **16**.

**Workloads and Metrics**
The evaluation utilizes the **ShareGPT dataset** [35] to generate five distinct workload profiles, reflecting the diversity of real-world downstream tasks identified in Figure 1:
1.  **LPLD (Light Prefill, Light Decode):** Typical chat interactions.
2.  **LPHD (Light Prefill, Heavy Decode):** Content creation or coding assistance.
3.  **HPLD (Heavy Prefill, Light Decode):** Document summarization.
4.  **HPHD (Heavy Prefill, Heavy Decode):** Complex reasoning or long-context analysis.
5.  **Mixed:** A random sampling of all the above types, simulating a production cluster.

Requests are categorized as "Heavy" if prompt tokens > **512** (prefill) or generated tokens > **128** (decode); otherwise, they are "Light."

The system is evaluated on three critical metrics:
*   **Time-to-First-Token (TTFT):** Measures user-perceived latency for the start of generation.
*   **Job Completion Time (JCT):** The total time from request arrival to the generation of the final token.
*   **Performance per Dollar (perf/$):** A efficiency metric derived from **resource usage time** (aggregate wall-clock time of all active instances). Lower resource usage time implies lower cost.

### 5.2 End-to-End Performance Results
The core claim of the paper is that `TetriInfer` eliminates interference to improve latency and efficiency simultaneously. The results across different workloads confirm this, though with notable nuances.

**Light Prefill, Heavy Decode (LPHD): The "Sweet Spot"**
This workload represents content creation, where short prompts trigger long generations. This is the scenario where `vLLM` suffers most from prefill-decode interference.
*   **TTFT:** `TetriInfer` reduces average TTFT by **97%** compared to `vLLM` (Figure 12a). This massive gain occurs because `vLLM` forces short prefill requests to wait behind or compete with long decode iterations, whereas `TetriInfer` processes prefill instantly on dedicated instances.
*   **JCT:** Average JCT improves by **47%** (Figure 12b). The dedicated decode instances, aided by the `reserve-dynamic` scheduler, avoid the memory thrashing that plagues the coupled baseline.
*   **Efficiency:** Surprisingly, despite using physically separate instances (which intuitively sounds like doubling hardware), `TetriInfer` uses **38% less total hardware resources** than `vLLM`. This is because the baseline runs so inefficiently due to interference that it holds resources for much longer. Consequently, `TetriInfer` achieves a **2.4x improvement in perf/$** (Figure 12c).

**Mixed Workloads: Real-World Robustness**
In the **Mixed** workload (Figure 15), which combines all request types, `TetriInfer` demonstrates consistent superiority:
*   **TTFT:** Reduced by **85%**.
*   **JCT:** Reduced by **50%**.
*   **Resource Usage:** Reduced by **21%**.
*   **Overall Efficiency:** The system delivers a **1.9x improvement in perf/$**. These results validate that the disaggregated architecture effectively handles the stochastic nature of real traffic without requiring manual tuning.

**Light Prefill, Light Decode (LPLD): Chat Scenarios**
Even for standard chat workloads where interference is less extreme, `TetriInfer` maintains an edge:
*   **TTFT & JCT:** Improvements of **44%** and **40%**, respectively (Figure 11).
*   **Efficiency:** While the resource usage time is comparable to `vLLM` (since interference is minimal in the baseline for small jobs), the faster completion times allow `TetriInfer` to recycle instances sooner, yielding a **1.4x perf/$** gain.

**Heavy Prefill Cases (HPLD & HPHD): The Trade-off**
The results for workloads with long prompts (Summarization/Complex Reasoning) reveal the limits of the current design.
*   **HPLD (Heavy Prefill, Light Decode):** `TetriInfer` improves TTFT by **9%** and JCT by **23%**. However, this comes at a cost: resource usage increases by **43%**. The overhead of transferring large KV caches (generated from long prompts) and running the prediction model outweighs the benefits of disaggregation when the decode phase is short. Consequently, `vLLM` outperforms `TetriInfer` in **perf/$ by 14%** for this specific case (Figure 13).
*   **HPHD (Heavy Prefill, Heavy Decode):** Here, the long decode phase allows `TetriInfer` to recover some efficiency. TTFT improves significantly (due to isolated prefill), and JCT drops by **19%**. Resource usage increases by only **7%**, leading to a marginal **1.1x perf/$** improvement (Figure 14).
*   **Analysis:** The authors explicitly note in Section 5.1 that `TetriInfer`'s design is "not ideal" for heavy prefill/light decode scenarios. The fixed overhead of network transfer and the prediction model cannot be amortized when the decode phase is trivial.

### 5.3 Microbenchmark Analysis and Ablation Studies
To understand *why* the system performs as it does, the authors conducted detailed microbenchmarks isolating each component.

**Prefill Scheduling and Chunking (Figure 16)**
This study isolates the impact of **Chunked Prefill** and scheduling policies (FCFS, SJF, LJF).
*   **Chunking Impact:** Simply switching from `vLLM`'s fixed-batch prefill to `TetriInfer`'s **Chunked Prefill** (even with naive FCFS scheduling) reduces average prefill latency by **86.4%**. This confirms the hypothesis that keeping the accelerator at its saturation point (512 tokens) avoids the quadratic latency explosion of oversized batches.
*   **Scheduling Policies:** Adding **Shortest-Job-First (SJF)** further reduces average waiting time by **7.8%** compared to FCFS.
*   **Batch Size Sensitivity:** Increasing the `PrefillSchedBatch` size from 16 to 128 allows SJF to optimize over a larger window, reducing TTFT by an additional **46.5%**. This suggests that larger scheduling windows yield diminishing but still significant returns.

**Length Prediction Accuracy and Overhead (Figure 17)**
The system relies on the `OPT-125M` predictor. The authors evaluated its accuracy and runtime cost.
*   **Accuracy vs. Granularity:** Predicting exact token counts is infeasible. Instead, the model predicts buckets.
    *   Granularity 100 tokens: **58.9%** accuracy.
    *   Granularity 200 tokens: **74.9%** accuracy (Selected default).
    *   Granularity 400 tokens: **85.0%** accuracy.
    The choice of 200 tokens balances sufficient resolution for scheduling with high enough accuracy to be useful.
*   **Runtime Overhead:** Running the predictor in **Parallel Mode** alongside the main LLM adds negligible latency for **80%** of requests. Under extreme stress, the main LLM's throughput drops by only **12%**, and average prefill latency increases by **10%**. This validates the design choice of using a small, parallel model rather than a sequential, heavy one.

**Decode Scheduling Policies (Figure 18)**
This ablation compares `vLLM`'s greedy policy against `TetriInfer`'s `reserve-static` (RS) and `reserve-dynamic` (RD) policies.
*   **Current Reality:** With the achieved **74.9%** prediction accuracy, `reserve-dynamic` performs on par with `vLLM`'s greedy approach in terms of JCT. It does not degrade performance despite the imperfect predictions.
*   **Potential:** In an "Ideal" simulation with 100% prediction accuracy, `reserve-dynamic` improves average JCT by **12%**, and `reserve-static` by **10%**. This highlights that future improvements in length prediction models could directly translate to better system performance without architectural changes.

**Inter-Decode Load Balancing (Figure 19)**
The authors tested the decentralized **Power-of-Two** dispatcher against Random and Imbalanced strategies.
*   **Result:** The proposed algorithm achieves the **lowest total decoding time** across all instance counts (2 to 8 instances).
*   **Mechanism:** The right panel of Figure 19 shows that `TetriInfer` successfully maintains a balanced ratio of heavy-to-light requests across all instances. In contrast, the "Imbalance" strategy creates hotspots where specific instances are overwhelmed by heavy decode jobs, causing severe latency spikes for co-located light jobs.

### 5.4 Critical Assessment of Claims
Do the experiments convincingly support the paper's claims?

**Strengths:**
1.  **Quantitative Validation of Interference:** The microbenchmarks (Figures 3, 4, 5) provide irrefutable evidence that mixing phases causes severe degradation (up to 10x slowdown). This strongly motivates the disaggregation approach.
2.  **Efficiency Paradox Resolved:** The most compelling result is the **LPHD** workload (Figure 12). The fact that a disaggregated system uses *fewer* resources than a coupled one is counter-intuitive but rigorously proven: the cost of interference in the baseline is higher than the cost of network transfer in `TetriInfer`.
3.  **Honesty on Limitations:** The authors do not hide the failure case. They explicitly show in Figure 13 that for **HPLD** workloads, `TetriInfer` is *less* efficient than `vLLM` in terms of cost. This transparency adds credibility to the overall evaluation.

**Weaknesses and Constraints:**
1.  **Network Emulation:** The reliance on a **mock mechanism** for network transfer (Section 4) is a significant limitation. While the logic is sound, real-world network jitter, congestion, and protocol overheads (especially for RDMA/One-Sided operations) are not fully captured. The "Indirect" CPU-bounce path used in the actual implementation (due to hardware limits) is likely slower than the emulated Direct-NIC speeds, meaning real-world gains might be slightly lower than reported.
2.  **Prediction Dependency:** The system's decode efficiency is tightly coupled to the prediction model's accuracy. While the current 74.9% accuracy is "good enough" to match the baseline, the promised 12% JCT improvement remains theoretical until better predictors are developed.
3.  **Scale:** The evaluation is limited to a 4-GPU server. While the architecture is designed for cloud scale, the behavior of the centralized control plane and the "Instance Flip" mechanism under thousands of concurrent nodes is not empirically verified in this work.

**Conclusion on Results:**
The experiments successfully demonstrate that **phase disaggregation** is a viable and often superior strategy for LLM serving, particularly for mixed and decode-heavy workloads. The data supports the claim that `TetriInfer` can improve performance-per-dollar by up to **2.4x** in common scenarios. However, the results also clarify that this is not a universal silver bullet; for workloads dominated by heavy prefill and light decode, the overhead of disaggregation currently outweighs the benefits, suggesting that hybrid strategies (switching between coupled and disaggregated modes) might be necessary for future systems.

## 6. Limitations and Trade-offs

While `TetriInfer` demonstrates significant performance gains for mixed and decode-heavy workloads, its architectural choices introduce specific trade-offs and rely on assumptions that limit its universality. The system is not a "one-size-fits-all" solution; rather, it optimizes for a specific subset of LLM serving patterns at the expense of others.

### 6.1 The Overhead of Disaggregation in Prefill-Dominated Workloads
The most critical limitation identified by the authors is that **disaggregation incurs a net penalty when the decode phase is trivial**. The core value proposition of `TetriInfer`—isolating prefill from decode to prevent interference—only pays off if the decode phase is long enough to benefit from dedicated, interference-free resources.

*   **The HPLD Failure Case:** In **Heavy Prefill, Light Decode (HPLD)** workloads (e.g., document summarization where inputs are long but outputs are short), `TetriInfer` underperforms the coupled baseline (`vLLM`) in terms of efficiency. As shown in **Figure 13**, while `TetriInfer` improves Time-to-First-Token (TTFT) by 9% and Job Completion Time (JCT) by 23%, it increases **resource usage time by 43%**.
*   **Root Cause:** The overhead of transferring large KV caches (generated from heavy prompts) across the network, combined with the computational cost of running the auxiliary prediction model, cannot be amortized over a short decode phase. In these scenarios, the "interference" avoided is minimal because the decode job finishes quickly, making the network transfer and phase separation purely additive costs. Consequently, `vLLM` achieves a **14% better performance-per-dollar** for this specific workload type.
*   **Implication:** This suggests that a production system based on `TetriInfer` would need a hybrid strategy: dynamically switching between disaggregated mode (for chat/creation) and coupled mode (for summarization) based on real-time workload characterization. The current implementation lacks this automatic mode-switching logic.

### 6.2 Dependency on Prediction Accuracy and Granularity
The efficacy of `TetriInfer`'s decode scheduling policies (`reserve-static` and `reserve-dynamic`) is strictly bounded by the accuracy of its auxiliary length prediction model.

*   **The Accuracy Ceiling:** The system uses an `OPT-125M` classifier to predict output length buckets. With a chosen granularity of **200 tokens**, the model achieves **74.9% accuracy** (**Section 5.2.2**).
*   **Performance Gap:** In microbenchmarks (**Figure 18**), the `reserve-dynamic` policy with current accuracy performs **on par** with `vLLM`'s greedy policy, but does not surpass it. The theoretical benefit—a **12% reduction in JCT**—is only realized under an "Ideal" simulation with 100% prediction accuracy.
*   **Trade-off in Granularity:** There is an inherent tension in choosing the prediction bucket size.
    *   **Fine Granularity (e.g., 100 tokens):** Yields lower accuracy (**58.9%**) because exact token counts are stochastic and sensitive to sampling parameters (temperature, top-p).
    *   **Coarse Granularity (e.g., 400 tokens):** Increases accuracy (**85%**) but reduces the scheduler's resolution, making it harder to distinguish between moderately long and very long requests, potentially leading to suboptimal packing.
*   **Open Question:** The system's future performance gains are contingent on advances in lightweight length prediction. If prediction accuracy plateaus or degrades on out-of-distribution prompts, `TetriInfer` loses its scheduling advantage over simpler greedy baselines.

### 6.3 Network Bandwidth and Hardware Constraints
The architecture assumes the availability of high-bandwidth, low-latency interconnects to make KV cache transfer viable. However, the evaluation reveals significant constraints regarding network infrastructure.

*   **Emulation vs. Reality:** Due to hardware limitations, the authors could not test on a multi-server cluster with real NVLink or RoCE connections. Instead, they used a **mock mechanism** (**Section 4**) that calculates theoretical latency and forces the decode instance to wait.
    *   **Missing Factors:** This emulation does not account for real-world network phenomena such as **packet loss, congestion, jitter, or TCP/IP stack overheads**. In a real cloud environment, network contention could introduce variable latency spikes that the deterministic mock mechanism misses.
*   **Implementation Limitation:** The actual implemented network stack in the testbed uses the **Indirect** path (bouncing data through CPU DRAM via sockets) rather than **Direct** (NVLink) or **Direct-NIC** (RDMA) paths (**Figure 9**).
    *   **Impact:** The Indirect path involves extra memory copies (GPU $\to$ CPU $\to$ Network $\to$ CPU $\to$ GPU), which consumes CPU cycles and adds latency. While the emulation assumes high bandwidth (200 Gbps+), the actual software implementation on the testbed likely suffers from higher overheads than reported, suggesting real-world gains might be slightly lower than the idealized curves in **Figures 11–15**.
*   **Scalability of Control Plane:** The **Centralized Control Plane** manages instance flipping and global routing. While the authors claim it is distributed to avoid single points of failure, the paper does not empirically validate its scalability to **thousands of nodes**. The "Instance Flip" mechanism relies on the Cluster Monitor broadcasting load stats every **100 ms**; at massive scale, this broadcast traffic could become a bottleneck, a risk not addressed in the 4-GPU evaluation.

### 6.4 Unexplored Optimization Opportunities
The paper explicitly identifies several design choices made for simplicity that leave performance on the table, representing open questions for future work.

*   **Request-Level vs. Chunk-Level Transfer:** Currently, `TetriInfer` waits until **all** chunks of a prompt are prefilled before transferring the aggregated KV cache (**Section 3.3.4**).
    *   **Missed Parallelism:** The authors acknowledge that **Chunk-Level transfer** (sending KV cache immediately after each chunk is processed) would allow network transfer to overlap with the computation of subsequent chunks. This would hide network latency more effectively, especially for heavy prefill requests. The current request-level approach serializes computation and networking, contributing to the overhead seen in HPLD workloads.
*   **One-Sided vs. Two-Sided Communication:** The system implements a unified abstraction but currently relies on **Two-Sided** communication (send/receive) involving the receiver's CPU.
    *   **Potential Gain:** The authors note that **One-Sided** operations (e.g., RDMA Write, where the sender writes directly to the receiver's GPU memory without CPU involvement) could further reduce latency and CPU overhead. However, this requires specific hardware support and complex memory registration logic that was not implemented due to hardware access constraints.
*   **Preemptive Prefill Scheduling:** The current prefill scheduler uses non-preemptive policies (FCFS, SJF, LJF) within a batch (**Section 3.3.1**). While `Chunked Prefill` technically enables preemptive scheduling (e.g., Shortest-Remaining-Time-First), the authors left this for future work. Implementing preemption could further reduce tail latency for short requests stuck behind long ones, a common issue in FCFS queues.

### 6.5 Summary of Trade-offs
| Feature | Benefit | Trade-off / Limitation |
| :--- | :--- | :--- |
| **Phase Disaggregation** | Eliminates Pre/Decode interference; stabilizes latency. | **High overhead for HPLD workloads**; network transfer cost dominates short decodes. |
| **Length Prediction** | Enables proactive memory reservation; prevents thrashing. | **Gains capped by accuracy** (74.9%); requires maintaining/training an auxiliary model. |
| **Chunked Prefill** | Saturates compute; prevents quadratic latency spikes. | Requires padding (wasted compute) for non-multiple lengths; adds scheduling complexity. |
| **Instance Flipping** | Dynamic resource allocation without reloading weights. | Relies on centralized monitoring; scalability to thousands of nodes unproven. |
| **Decentralized Dispatch** | Avoids global scheduler bottleneck; balances load. | Depends on accurate global load views (100ms lag); "Power-of-Two" is heuristic, not optimal. |

In conclusion, `TetriInfer` represents a powerful architectural shift for **decode-heavy and mixed workloads**, where the cost of interference far exceeds the cost of disaggregation. However, it is not universally superior. Its effectiveness diminishes in **prefill-heavy, decode-light** scenarios, and its full potential is currently gated by the accuracy of length predictors and the availability of high-performance network interconnects that were emulated but not physically tested in this study.

## 7. Implications and Future Directions

The introduction of `TetriInfer` marks a pivotal shift in LLM serving architecture, moving the field from **coupled, reactive batching** toward **disaggregated, predictive orchestration**. By empirically proving that the interference between computation-heavy prefill and memory-intensive decode phases is the primary bottleneck in mixed workloads, this work challenges the prevailing assumption that a single instance type should handle the entire inference lifecycle. The implications extend beyond immediate performance gains, reshaping how cloud providers design infrastructure, how researchers approach scheduling problems, and how developers deploy large-scale AI services.

### 7.1 Reshaping the Landscape of LLM Serving
`TetriInfer` fundamentally alters the design space for inference systems in three critical ways:

*   **From Static to Fluid Resource Pools:** Traditional clusters are statically provisioned with a fixed ratio of compute-to-memory resources, often leading to stranded capacity when workload patterns shift (e.g., a surge in summarization tasks vs. chat bots). `TetriInfer`'s **Instance Flip** mechanism (Section 3.5) demonstrates that hardware roles need not be static. By enabling sub-10ms role switching without model reloading, it introduces the concept of a **fluid compute fabric** where the cluster topology morphs dynamically to match the instantaneous demand curve. This suggests a future where "prefill nodes" and "decode nodes" are not distinct physical classes of hardware, but transient software states.
*   **The Rise of Auxiliary Models in Systems Design:** Historically, auxiliary models were used solely for accuracy improvements (e.g., reranking, retrieval). `TetriInfer` pioneers the use of a small, fine-tuned LLM (`OPT-125M`) as a **first-class system component** for resource management. By offloading the uncertainty of sequence length to a cheap predictor, the system transforms an intractable stochastic scheduling problem into a deterministic bin-packing problem. This validates a new paradigm: **using model intelligence to solve system constraints**, paving the way for future systems that might use predictors for memory footprint, compute intensity, or even network routing decisions.
*   **Disaggregation as the Default for Mixed Workloads:** Prior to this work, disaggregation was often viewed as too costly due to network overhead. `TetriInfer` flips this narrative by showing that for common workloads (Light Prefill/Heavy Decode), the **cost of interference in coupled systems exceeds the cost of network transfer**. The 2.4x improvement in performance-per-dollar (Figure 12) provides a compelling economic argument that disaggregation should be the default architecture for general-purpose LLM clouds, reserving coupled architectures only for niche, prefill-dominated tasks.

### 7.2 avenues for Follow-Up Research
The limitations and design choices identified in `TetriInfer` open several rich veins for future investigation:

*   **Adaptive Hybrid Architectures:** Since `TetriInfer` underperforms on Heavy Prefill/Light Decode (HPLD) workloads due to transfer overhead (Section 6.1), a logical next step is a **hybrid scheduler** that dynamically chooses between coupled and disaggregated modes. Research could focus on a meta-controller that analyzes incoming request characteristics in real-time: routing HPLD tasks to coupled instances to save bandwidth, while sending LPHD tasks to the disaggregated pool to avoid interference.
*   **Fine-Grained KV Cache Streaming:** The current implementation uses **Request-Level transfer**, waiting for all chunks to finish before sending data (Section 3.3.4). Future work should implement **Chunk-Level transfer**, where the KV cache is streamed to the decode instance immediately after each chunk is processed. This would overlap network transmission with prefill computation, effectively hiding the network latency that currently penalizes heavy prefill tasks. Combining this with the layer-wise transfer proposed in concurrent work like `Splitwise` could further optimize pipeline parallelism.
*   **Advanced Prediction Mechanisms:** The system's decode efficiency is capped by the 74.9% accuracy of its bucket-based predictor (Section 5.2.2). Research into **multi-modal predictors** (incorporating prompt semantics, user history, or task type) could significantly boost accuracy. Furthermore, exploring **online fine-tuning** (currently excluded in Figure 8) could allow the predictor to adapt to shifting data distributions (e.g., a sudden trend in long-form coding answers) without manual retraining.
*   **One-Sided Memory Access at Scale:** The evaluation relied on emulated networks and two-sided communication due to hardware constraints (Section 4). Implementing true **One-Sided RDMA** (where the prefill instance writes directly to the decode instance's GPU memory) would eliminate CPU involvement and context switches. Research is needed to manage the complexity of remote memory registration and safety in a multi-tenant cloud environment, potentially leveraging emerging hardware features like NVIDIA's GPUDirect RDMA or CXL memory pooling.

### 7.3 Practical Applications and Downstream Use Cases
The architectural advantages of `TetriInfer` map directly to specific high-value industry scenarios:

*   **General-Purpose AI Clouds:** Cloud providers offering "serverless LLM" endpoints face highly variable traffic mixes (chat, coding, summarization). `TetriInfer` is ideally suited for this environment, as its dynamic flipping and interference avoidance ensure consistent SLAs regardless of the incoming request mix. The 50% reduction in Job Completion Time (JCT) for mixed workloads (Figure 15) translates directly to higher throughput and lower operational costs.
*   **Interactive Content Creation & Coding Assistants:** Applications like GitHub Copilot or creative writing tools typically involve short prompts and long generations (LPHD). This is `TetriInfer`'s "sweet spot," where it delivers a **97% reduction in TTFT** (Figure 12). For interactive tools, minimizing the wait time for the first token is critical for user experience; `TetriInfer` makes near-instantaneous response feasible even under heavy load.
*   **Multi-Tenant Enterprise Hubs:** Large enterprises often run a single cluster for diverse departments: HR (summarizing resumes - HPLD), Marketing (writing blogs - LPHD), and Support (chat bots - LPLD). A coupled system would suffer severe interference when these departments peak simultaneously. `TetriInfer`'s ability to isolate these phases ensures that a marketing team generating long articles does not degrade the latency of HR's summarization tasks.

### 7.4 Reproduction and Integration Guidance
For practitioners considering adopting or extending `TetriInfer`, the following guidance clarifies when and how to deploy this architecture:

*   **When to Prefer `TetriInfer`:**
    *   **Workload Mix:** Deploy if your traffic includes a significant portion of **Long-Generation** tasks (e.g., >128 tokens). If your workload is 90% short Q&A, the overhead of disaggregation may not be justified.
    *   **Network Infrastructure:** The architecture relies heavily on high-bandwidth interconnects. It is most effective in environments with **NVLink, InfiniBand, or high-speed RoCE (≥100 Gbps)**. On clusters limited to standard Ethernet (10/25 Gbps), the KV cache transfer latency may negate the benefits of interference avoidance, particularly for larger models.
    *   **Latency Sensitivity:** Choose this approach if **Time-to-First-Token (TTFT)** is a critical SLA. The decoupling of prefill ensures that decode requests never wait for compute-heavy prompt processing.

*   **When to Stick with Coupled Systems (e.g., vanilla `vLLM`):**
    *   **Prefill-Dominated Tasks:** If your primary use case is **document summarization** or **RAG retrieval** where inputs are massive (thousands of tokens) but outputs are short, the coupled approach remains more efficient due to the lack of network transfer overhead.
    *   **Resource-Constrained Clusters:** If you lack the hardware to support a unified network stack or cannot spare the GPU memory for an auxiliary prediction model, the complexity of `TetriInfer` may outweigh its benefits.

*   **Integration Steps:**
    1.  **Profile Your Workload:** Before migration, analyze your request distribution (Prompt Length vs. Generation Length). If the distribution skews heavily toward Light Prefill/Heavy Decode, the ROI for `TetriInfer` is highest.
    2.  **Calibrate ChunkSize:** The optimal `ChunkSize` (512 in the paper) is hardware-dependent. Practitioners must profile their specific GPU/model combination to find the **accelerator-saturation threshold** where throughput plateaus.
    3.  **Train Domain-Specific Predictors:** Do not rely on the generic `OPT-125M` model provided. Fine-tune the predictor on your specific domain data (e.g., code, legal docs, medical notes) to improve the granularity accuracy, which directly unlocks the benefits of the `reserve-dynamic` scheduler.
    4.  **Implement Hybrid Fallbacks:** Given the HPLD limitation, consider implementing a simple router that directs known "short-output" tasks to a small pool of coupled instances while sending the rest to the disaggregated `TetriInfer` cluster.

In summary, `TetriInfer` provides a robust blueprint for the next generation of LLM serving systems. By treating interference as a first-class citizen and leveraging predictive intelligence to manage resources, it offers a path toward significantly more efficient, scalable, and responsive AI infrastructure. While not a universal panacea, its principles of disaggregation and dynamic adaptation are likely to become standard features in future cloud-native AI platforms.
