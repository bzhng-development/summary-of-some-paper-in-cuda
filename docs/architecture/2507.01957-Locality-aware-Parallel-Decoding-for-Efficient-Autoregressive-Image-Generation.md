## 1. Executive Summary
This paper introduces **Locality-aware Parallel Decoding (LPD)**, a framework that accelerates autoregressive image generation by reducing the sequential decoding steps from **256 to 20** for $256 \times 256$ images and from **1024 to 48** for $512 \times 512$ images on the **ImageNet** dataset, achieving at least **3.4$\times$ lower latency** than previous parallelized methods like **ARPG** and **RandAR**. By combining a **Flexible Parallelized Autoregressive Modeling** architecture that enables mutual visibility among concurrently generated tokens with a **Locality-aware Generation Ordering** schedule that optimizes spatial dependencies, LPD maintains high generation quality (e.g., **FID of 2.10** for the 752M parameter model) while overcoming the memory-bound bottlenecks of traditional next-patch prediction. This advancement is significant because it delivers efficient, flat-token autoregressive generation that remains fully compatible with unified multimodal foundation models, unlike multi-scale approaches that sacrifice interoperability for speed.

## 2. Context and Motivation

### The Fundamental Bottleneck: Memory-Bound Sequential Generation
To understand the necessity of this work, one must first grasp the operational mechanics of standard autoregressive (AR) image generation. In this paradigm, an image is discretized into a sequence of $N$ tokens (e.g., $256$ tokens for a $256 \times 256$ image using a $16\times$ downsampling factor). The model generates these tokens one by one, strictly following a predefined order, typically a raster scan (left-to-right, top-to-bottom).

Mathematically, the joint probability of the image tokens $x_1, \dots, x_N$ given a condition $c$ is factorized as:
$$ p(x_1, \dots, x_N; c) = \prod_{n=1}^{N} p(x_n | x_{&lt;n}; c) $$
While this approach yields high-quality results, it suffers from a critical systems-level inefficiency: it is **memory-bound**. As noted in the Introduction (Section 1), generating a single token per step requires loading the entire model's parameters from GPU memory into registers for every single step. Since modern GPUs have immense computational power but limited memory bandwidth, the time spent waiting for data transfer (latency) far exceeds the time spent computing the token. Consequently, the total generation time scales linearly with the number of tokens $N$. For high-resolution images (e.g., $512 \times 512$, requiring $1024$ steps), this latency becomes prohibitive for real-time applications.

### The Compatibility Trap: Flat vs. Multi-Scale Representations
Researchers have attempted to bypass this sequential bottleneck, but existing solutions introduce a new dilemma regarding **token representation compatibility**.

1.  **Multi-Scale Approaches (e.g., VAR):** Methods like Visual Autoregressive Modeling (VAR) generate images coarse-to-fine, predicting entire "scales" or resolutions at once rather than individual patches. This drastically reduces the number of generation steps (e.g., from 256 to 10). However, as highlighted in Section 1, these methods rely on **multi-scale token representations**. This fundamentally breaks compatibility with the vast ecosystem of **flat-token vision foundation models** (such as CLIP, DINO, or unified multimodal LLMs) which expect a uniform sequence of tokens representing the finest resolution. Using multi-scale tokens isolates the generative model from perception backbones, hindering the development of unified multimodal systems.
2.  **Non-Autoregressive Masked Models (e.g., MASKGIT):** These models predict multiple tokens simultaneously by masking parts of the image and iteratively refining them. While they enable parallelism, they require **bidirectional attention** (seeing the future context) during training and inference, which prevents the use of **KV caching**. KV caching is a standard optimization in AR models where previously computed key-value pairs are stored to avoid re-computation. Without it, masked models often incur higher computational costs per step compared to optimized AR inference.

The specific gap this paper addresses is the lack of a method that achieves **high parallelization** (fewer steps) while maintaining **flat token representations** and leveraging **efficient AR inference mechanisms** (KV caching).

### Limitations of Prior Parallel Autoregressive Works
Recent works have attempted to parallelize standard next-patch prediction by predicting groups of tokens simultaneously. However, the paper identifies two distinct architectural failures in these approaches (Section 2.2, Figure 6):

*   **Encoder-Decoder Architectures (e.g., SAR, ARPG):** These models use query tokens to attend to an encoder's context. The critical flaw, illustrated in **Figure 6(a)**, is that the target positions being generated do not contribute key-value pairs to each other within the same step. Consequently, tokens generated in parallel are **mutually independent**. If two adjacent pixels are generated simultaneously, neither "sees" the other, leading to structural inconsistencies and artifacts, especially as the degree of parallelization increases.
*   **Decoder-Only with Causal Masks (e.g., RandAR):** These models attempt to maintain a decoder-only structure by inserting "positional instruction tokens" to indicate where to generate. However, they typically retain the standard **causal attention mask** used in language modeling. As shown in **Figure 6(b)**, this forces the model to treat parallel generation as merely "batched next-token prediction" rather than true joint prediction. Furthermore, these instruction tokens must be stored in the KV cache, effectively **doubling the memory consumption** during inference.

Both approaches struggle to balance **generation quality** with **high degrees of parallelization**. When the number of tokens predicted per step increases, the lack of mutual visibility or the rigidity of the causal mask causes performance to degrade significantly.

### Positioning of Locality-aware Parallel Decoding (LPD)
LPD positions itself as the solution that reconciles efficiency, quality, and compatibility. It does not abandon the autoregressive framework but rather **re-architects** it to support flexible parallelism.

1.  **Architectural Innovation:** Unlike prior works that force parallel tokens to be independent or burden the cache with instruction tokens, LPD introduces **Flexible Parallelized Autoregressive Modeling**. This uses learnable **position query tokens** that guide generation at specific target locations while employing a specialized attention mask (Section 2.2, **Figure 4**) that ensures **mutual visibility** among concurrently generated tokens. This allows the model to jointly predict a group of tokens as a coherent unit, preserving spatial consistency.
2.  **Data-Driven Scheduling:** Recognizing that *how* tokens are grouped matters as much as *how* they are predicted, LPD moves beyond fixed raster orders or random shuffling. By analyzing attention maps in existing models (Section 2.3, **Figure 2** and **Figure 7**), the authors observe strong **spatial locality**: tokens primarily attend to their immediate neighbors.
    *   **Misconception Alert:** One might assume that spreading generated tokens uniformly across the image (low discrepancy) is optimal for parallelism. LPD argues the opposite based on empirical evidence.
    *   **LPD's Strategy:** The **Locality-aware Generation Ordering** schedule (Algorithm 1) explicitly maximizes proximity to *already generated* context (to ensure strong conditioning) while minimizing proximity *among concurrently generated* tokens (to reduce mutual dependency).

In summary, LPD distinguishes itself by being the first framework to simultaneously achieve:
*   **Flat token compatibility** (unlike multi-scale methods).
*   **True joint parallel prediction** with mutual visibility (unlike independent encoder-decoder methods).
*   **Efficient KV caching** without instruction token overhead (unlike causal-mask decoder methods).
*   **Optimized generation ordering** derived from spatial locality principles (unlike random or fixed orders).

This positioning allows LPD to reduce generation steps by over **12$\times$** (256 to 20) on ImageNet $256 \times 256$ while matching or exceeding the quality of much slower baselines, as demonstrated in **Table 1** and **Figure 1**.

## 3. Technical Approach

### 3.1 Reader orientation (approachable technical breakdown)
This system is a re-engineered autoregressive transformer that replaces the rigid "one-token-at-a-time" generation process with a flexible mechanism capable of predicting multiple image patches simultaneously while maintaining their spatial consistency. It solves the latency bottleneck of standard image generation by decoupling the roles of "providing context" and "requesting generation," allowing the model to look at already-generated neighbors and jointly predict a new group of distant-but-related tokens in a single step.

### 3.2 Big-picture architecture (diagram in words)
The LPD framework consists of three primary logical components that interact during the inference loop:
1.  **The Context Encoder (KV Cache Manager):** This component takes previously generated image tokens, processes them through the transformer layers, and stores their Key-Value (KV) pairs in a cache. Its sole responsibility is to build a dense, accessible representation of the image history without performing new predictions.
2.  **The Position Query Generator:** Instead of waiting for a token to naturally appear in a sequence, this component synthesizes special "learnable position query tokens." These are vector embeddings constructed by adding a shared learnable parameter to the positional embedding of a specific target coordinate (e.g., pixel row $i$, column $j$). Their responsibility is to act as "placeholders" that tell the model exactly which missing parts of the image to fill next.
3.  **The Parallel Decoder with Specialized Attention:** This is the core transformer engine that processes both the cached context and the new position queries simultaneously. It employs a custom attention mask that allows the position queries to see the full context *and* each other (mutual visibility), ensuring that the batch of tokens being generated now are consistent with one another. It outputs the logits (probability distributions) for all target positions in parallel, which are then sampled to become the new image tokens.

### 3.3 Roadmap for the deep dive
To fully understand how LPD achieves efficient parallelism without sacrificing quality, we will proceed in the following logical order:
*   **First**, we define the mathematical formulation of "Flexible Parallelized Autoregressive Modeling," explaining how the joint probability distribution is factorized to allow group-wise prediction rather than strict sequential prediction.
*   **Second**, we dissect the training mechanism, specifically the construction of "Position Query Tokens" and the design of the dual-pattern attention mask that enables mutual visibility among concurrent predictions.
*   **Third**, we detail the inference pipeline, describing how the encoding of past tokens and the decoding of new queries are fused into a single forward pass to minimize latency while leveraging KV caching.
*   **Fourth**, we analyze the "Locality-aware Generation Order Schedule," explaining the empirical observations of spatial attention that motivated the algorithm and walking through the step-by-step logic of selecting token groups based on proximity and repulsion thresholds.
*   **Finally**, we contrast this approach with prior methods (SAR, RandAR) to highlight why LPD's specific combination of mutual visibility and cache efficiency yields superior scaling properties.

### 3.4 Detailed, sentence-based technical breakdown

**Core Concept and Mathematical Formulation**
This paper presents a methodological shift from strict next-token prediction to group-wise joint prediction within an autoregressive framework. Standard autoregressive models factorize the probability of an image with $N$ tokens ($x_1, \dots, x_N$) given a condition $c$ into a product of $N$ conditional probabilities, forcing $N$ sequential steps:
$$ p(x_1, \dots, x_N; c) = \prod_{n=1}^{N} p(x_n | x_{&lt;n}; c) $$
LPD modifies this by partitioning the $N$ tokens into $G$ disjoint groups $\{X_1, \dots, X_G\}$, where each group $X_g$ contains $m$ tokens predicted jointly. The new factorization becomes:
$$ p(x_1, \dots, x_N; c) = \prod_{g=1}^{G} p(X_g | X_{&lt;g}; c) $$
Here, $X_{&lt;g}$ represents all tokens generated in previous steps, providing the context, while $X_g$ is the set of tokens generated in the current parallel step. The critical challenge addressed by LPD is modeling $p(X_g | X_{&lt;g}; c)$ effectively; if the tokens within $X_g$ are treated as independent (as in some prior works), the quality degrades rapidly as the group size increases. LPD ensures that the model learns the *joint* distribution of the group, capturing dependencies between tokens generated at the same time.

**Flexible Parallelized Autoregressive Modeling: Decoupling Context and Generation**
The fundamental architectural innovation of LPD is the decoupling of the two roles a token plays in standard transformers: providing context (via its hidden state) and requesting generation (via its output logits). In standard models, a token must exist in the input sequence to provide context, but its position in the sequence dictates when it is generated. LPD breaks this coupling by introducing **Learnable Position Query Tokens**.
*   **Construction of Query Tokens:** For every target position $(i, j)$ in the image grid that needs to be generated in the current step, the model constructs a query token $q_{i,j}$. This token is formed by summing a shared learnable embedding $E_{query}$ with the standard positional embedding $P_{i,j}$ corresponding to that specific 2D coordinate:
    $$ q_{i,j} = E_{query} + P_{i,j} $$
    This design allows the model to explicitly "request" generation at arbitrary spatial locations, regardless of the order in which previous tokens were generated.
*   **Training Formulation:** During training, the model receives a sequence containing both ground-truth image tokens (for context) and these position query tokens (for targets). The objective is to transform each position query token into the corresponding ground-truth image token. To enable this without leaking future information, LPD employs a specialized **Training Attention Mask** (illustrated in **Figure 4**) that defines two distinct attention patterns:
    1.  **Context Attention:** This pattern allows any subsequent token (whether an image token or a query token) to attend causally to all preceding ground-truth image tokens. This ensures that the generation is always conditioned on the full history of known pixels.
    2.  **Query Attention:** This is the crucial mechanism for parallelism. It allows all position query tokens *within the same generation step* to attend to each other (mutual visibility). Simultaneously, it prevents any tokens from attending to query tokens from *future* steps.
    By enabling mutual visibility, the transformer can compute attention scores between concurrently generated tokens, allowing the prediction of token $A$ to depend on the features of token $B$ even though both are being generated in the same step. This avoids the inconsistency artifacts seen in encoder-decoder models where parallel tokens are isolated.

**Inference Pipeline: Fused Encoding and Decoding**
At inference time, the goal is to minimize the number of forward passes through the GPU. A naive implementation might require two steps: one to encode the newly generated image tokens into the KV cache, and a second to decode the new position queries. LPD fuses these operations into a single step using a specialized **Inference Attention Mask** (illustrated in **Figure 5**).
*   **Step 1 (Initialization):** The process begins with a class token and potentially a few initial seed tokens.
*   **Step 2 (Fused Forward Pass):** In each subsequent iteration $k$, the input sequence consists of the previously generated image tokens $\{x_{prev}\}$ and the new set of position query tokens $\{q_{target}\}$ for the current group.
    *   The transformer processes the image tokens first to update the **KV Cache**. These keys and values are stored in GPU memory to represent the accumulated context.
    *   Crucially, the position query tokens are allowed to attend to this updated KV cache *and* to each other within the same forward pass.
    *   The model outputs logits only for the position query tokens. The image tokens in the input serve only to update the cache and do not produce output logits for generation.
*   **Step 3 (Sampling and Update):** The logits for the query tokens are sampled (e.g., using temperature sampling or classifier-free guidance) to produce the actual image tokens. These new tokens are then added to the sequence for the next iteration.
*   **Efficiency Gain:** Because the position query tokens are not stored in the KV cache (only the resulting sampled image tokens are stored in the next step), the memory footprint remains identical to standard autoregressive decoding. This contrasts with methods like RandAR, which must store "instruction tokens" in the cache, doubling the memory bandwidth requirement.

**Locality-aware Generation Order Schedule**
While the architecture enables parallel generation, the *order* in which groups of tokens are selected determines the quality of the output. LPD derives its scheduling strategy from an empirical analysis of attention maps in the LLAMAGEN model (Section 2.3, **Figure 2** and **Figure 7**).
*   **Empirical Observation:** The authors quantify **Per-Token Attention (PTA)** to neighborhoods of radius $s$. The results show that attention weights decay sharply with Euclidean distance; tokens predominantly attend to their immediate spatial neighbors. This implies two competing requirements for a generation schedule:
    1.  **High Proximity to Context:** New tokens should be generated close to already existing tokens to maximize the strength of the conditioning signal (since attention is local).
    2.  **Low Proximity within Group:** Tokens generated *simultaneously* should be spatially distant from each other to minimize their mutual dependency, making the joint prediction task easier.
*   **Algorithm 1 Implementation:** The schedule is precomputed before inference using the following logic for $K$ decoding steps with group sizes $O = [o_1, \dots, o_K]$ (typically increasing via a cosine schedule):
    1.  **Proximity Measurement:** For the current step $k$, calculate the Euclidean distance between every unselected grid position and the set of already selected positions $S$. Positions closer to $S$ receive a higher proximity score.
    2.  **Candidate Splitting:** Unselected tokens are sorted by proximity and split into two sets: $c_1$ (tokens with proximity greater than a threshold $\tau$) and $c_2$ (the rest).
    3.  **High-Proximity Selection with Repulsion:** The algorithm iteratively selects tokens from $c_1$. However, upon selecting a token, it applies a **repulsion filter**: any other candidate in $c_1$ that is within a distance $\rho$ of the newly selected token is moved to set $c_2$. This ensures that tokens chosen for the *current* parallel batch are spread out (low intra-group dependency).
    4.  **Low-Dependency Completion:** If the target group size $o_k$ is not met after exhausting valid candidates in $c_1$, the algorithm fills the remaining slots by selecting from $c_2$ using **Farthest Point Sampling (FPS)**. FPS explicitly maximizes the minimum distance between selected points, further enforcing spatial separation.
*   **Dynamic Group Sizes:** The number of tokens generated per step is not constant. It follows a cosine schedule, starting with very few tokens (e.g., 1 or 2) when the context is sparse, and gradually increasing to larger groups (e.g., 20+ tokens) as the image fills in and the context becomes richer. For a $256 \times 256$ image (576 tokens total with $16\times$ downsampling), the schedule reduces the steps from 576 (raster) to just 20 or 32.

**Design Choices and Comparison to Alternatives**
The specific design of LPD addresses precise failure modes in prior art.
*   **Vs. Encoder-Decoder (SAR, ARPG):** In these models, parallel tokens are generated via cross-attention to an encoder. As shown in **Figure 6(a)**, the query tokens do not generate Key-Value pairs for each other. Consequently, if the model tries to generate two adjacent pixels simultaneously, neither sees the other, leading to checkerboard artifacts or inconsistent edges. LPD's self-attention mechanism with mutual visibility (**Figure 6(c)**) solves this by allowing the batch of queries to interact directly.
*   **Vs. Decoder-Only with Causal Masks (RandAR):** RandAR inserts instruction tokens into the sequence but uses a standard causal mask. This means a query token at position $t$ cannot see a query token at position $t+1$ even if they are in the same batch, effectively serializing the "parallel" generation into a batched single-token prediction. Furthermore, RandAR must cache the instruction tokens, wasting memory. LPD's custom mask allows full visibility within the batch while excluding query tokens from the persistent KV cache, optimizing both consistency and memory bandwidth.
*   **Vs. Fixed Orders (PAR):** Previous parallel methods often used fixed raster blocks. LPD's locality-aware schedule is dynamic and data-agnostic but topology-aware, ensuring that the "frontier" of generation always has strong local support, which is critical for high-frequency detail synthesis.

**Configuration and Hyperparameters**
The effectiveness of LPD relies on specific configurations detailed in the experiments (Section 3 and Appendix A).
*   **Model Scale:** The authors validate the approach on three model sizes: **LPD-L** (337M parameters, 12 layers), **LPD-XL** (752M parameters, 36 layers), and **LPD-XXL** (1.4B parameters, 48 layers). All use a standard decoder-only transformer architecture with the LLAMAGEN tokenizer (codebook size 16,384, downsample factor 16).
*   **Training Schedule:** Models are trained on ImageNet $256 \times 256$ for 450 epochs with a batch size of 2048 (for LPD-L) using AdamW optimizer ($\beta_1=0.9, \beta_2=0.95$) and a peak learning rate of $1 \times 10^{-4}$ (effective). Crucially, during training, the number of decoding steps is randomly sampled from the set $\{8, 12, 16, 20, 24, 32, 64, 128, 256\}$, forcing the model to learn robust generation dynamics at varying degrees of parallelism.
*   **Inference Parameters:** For $256 \times 256$ generation, the optimal performance is achieved with **20 steps** (reducing steps by $12.8\times$ compared to raster) or **32 steps** for higher fidelity. For $512 \times 512$, the step count is **48** (reducing from 1024). The proximity threshold $\tau$ and repulsion threshold $\rho$ in Algorithm 1 are tuned to balance the trade-off between context strength and intra-group independence, though exact scalar values for $\tau$ and $\rho$ are implementation details managed within the precomputation script provided in Appendix C.
*   **Latency Measurement:** Efficiency is profiled on a single NVIDIA A100 GPU with BF16 precision. The reported **3.4$\times$ to 4.2$\times$ speedup** over ARPG and RandAR is measured at batch size 1, highlighting the reduction in memory-bound latency rather than just throughput gains.

## 4. Key Insights and Innovations

The success of Locality-aware Parallel Decoding (LPD) stems not merely from engineering optimizations, but from fundamental re-conceptualizations of how autoregressive models handle spatial dependencies and parallelism. Below are the four most critical innovations that distinguish this work from prior art.

### 4.1 Decoupling Context Provision from Generation Request via Position Queries
**The Innovation:** LPD fundamentally breaks the rigid coupling in standard transformers where a single token embedding must simultaneously serve as "context for the future" and "the target to be predicted." By introducing **Learnable Position Query Tokens** (Section 2.2), the authors create a distinct architectural pathway: existing image tokens solely provide context (updating the KV cache), while synthetic query tokens solely request generation at specific coordinates.

**Differentiation from Prior Work:**
*   **Vs. Standard AR:** In raster-scan AR, the model cannot generate token $x_{t+5}$ until $x_{t+4}$ exists, because $x_{t+4}$ is required to provide the hidden state context. LPD removes this temporal lock; the query token $q_{t+5}$ can attend to *any* previously generated context regardless of sequence order.
*   **Vs. RandAR:** While RandAR also uses instruction tokens, it treats them as part of the causal sequence, forcing them into the KV cache and limiting their visibility via standard causal masks. LPD's queries are ephemeral—they guide the attention mechanism but do not bloat the persistent memory state.

**Significance:** This decoupling is the **enabling condition** for arbitrary generation orders. It transforms the generation process from a strictly linear chain into a flexible graph traversal, allowing the model to prioritize spatially informative regions (the "frontier") rather than adhering to a blind raster scan. This is a **fundamental architectural shift** rather than an incremental tweak, as it changes the definition of the input sequence itself.

### 4.2 Enforcing Mutual Visibility for True Joint Prediction
**The Innovation:** The paper identifies a subtle but catastrophic failure mode in previous parallel methods: **independent parallel prediction**. In encoder-decoder approaches (like SAR or ARPG), tokens generated in the same step cannot see each other (Figure 6a). LPD solves this with a specialized **Training Attention Mask** (Figure 4) that grants **mutual visibility** among all position query tokens within the current generation step.

**Differentiation from Prior Work:**
*   **The "Independence Trap":** Prior parallel methods effectively treat a group of $k$ tokens as $k$ independent single-token predictions running in parallel. If two adjacent pixels are generated simultaneously, neither conditions on the other, leading to high-frequency inconsistencies (e.g., broken edges, checkerboard artifacts).
*   **LPD's Solution:** By allowing query tokens to attend to one another, LPD models the **joint conditional distribution** $p(X_g | X_{&lt;g})$ rather than the product of marginals $\prod p(x_i | X_{&lt;g})$. This ensures that the generated batch is internally consistent before it is even committed to the image.

**Significance:** This insight explains *why* LPD can sustain high degrees of parallelization (predicting 20+ tokens per step) without quality collapse. It transforms parallel decoding from a "guess and hope" strategy into a coherent **joint inference** step. This is a **theoretical advance** in how we formulate autoregressive likelihoods for visual data, directly enabling the $12.8\times$ reduction in generation steps reported in Table 1.

### 4.3 The Dual-Principle Locality Schedule: Proximity vs. Repulsion
**The Innovation:** Perhaps the most counter-intuitive insight is the derivation of the **Locality-aware Generation Order** (Section 2.3). Based on empirical attention analysis (Figure 7), LPD posits two competing principles for scheduling:
1.  **Maximize Proximity to Context:** New tokens should be close to *already generated* tokens to leverage strong local attention weights.
2.  **Minimize Proximity within Group:** New tokens should be far from *each other* (within the same parallel batch) to minimize mutual dependency.

**Differentiation from Prior Work:**
*   **Vs. Random/Halton Scheduling:** Methods like RandAR (random) or Halton schedulers focus primarily on spreading tokens out (low discrepancy) to reduce intra-group dependency. They neglect the first principle: that generation quality relies heavily on having dense, local context.
*   **Vs. Raster Scan:** Raster scanning maximizes context proximity but forces high intra-group dependency if parallelized (since adjacent pixels are generated together).

**Significance:** LPD is the first to explicitly optimize this **trade-off surface**. The algorithm (Algorithm 1) dynamically constructs batches that hug the "generation frontier" (satisfying Principle 1) while picking points sparsely along that frontier (satisfying Principle 2). The ablation study (Figure 9c) proves this is not incremental: applying both principles yields an FID of **1.92**, whereas using either principle alone results in significantly worse scores (2.00 and 2.06). This demonstrates that **spatial topology awareness** is as critical to efficiency as the model architecture itself.

### 4.4 Cache-Efficient Fusion of Encoding and Decoding
**The Innovation:** LPD achieves its latency gains not just by reducing steps, but by ensuring each step is memory-efficient. The **Inference Attention Mask** (Figure 5) allows the model to fuse the "encoding of past tokens" and "decoding of new queries" into a **single forward pass**, while strictly excluding query tokens from the KV cache.

**Differentiation from Prior Work:**
*   **Vs. Instruction-Token Methods:** Approaches that insert learnable instructions into the sequence (e.g., RandAR) must store these instructions in the KV cache for future steps, effectively doubling the memory bandwidth required for attention lookups as the sequence grows.
*   **LPD's Efficiency:** Since LPD's query tokens are transient and only the resulting *sampled image tokens* are cached, the memory footprint per step remains identical to standard autoregressive decoding.

**Significance:** In memory-bound workloads (which image generation is, as stated in Section 1), reducing computational FLOPs is less important than reducing memory traffic. By avoiding the caching of non-image tokens, LPD ensures that the theoretical speedup from fewer steps translates directly to **real-world latency reduction** (3.4$\times$ to 4.2$\times$ faster, Figure 1). This is a crucial **systems-level innovation** that ensures the architectural benefits are not lost to memory bandwidth bottlenecks.

### Summary of Impact
These innovations collectively move the field beyond the "parallelism vs. quality" trade-off.
*   **Position Queries** enable flexibility.
*   **Mutual Visibility** ensures consistency.
*   **Locality Scheduling** optimizes the data flow.
*   **Cache Fusion** maximizes hardware efficiency.

Unlike incremental improvements that might shave off 10-20% latency by tuning hyperparameters, LPD represents a **paradigm shift** in autoregressive image generation, proving that flat-token models can be as efficient as multi-scale methods while retaining full compatibility with unified multimodal ecosystems.

## 5. Experimental Analysis

### 5.1 Evaluation Methodology and Setup
To rigorously validate the claims of efficiency and quality, the authors conduct a comprehensive evaluation on the **ImageNet** class-conditional generation benchmark. The experimental design is structured to isolate the benefits of LPD against three distinct categories of baselines: standard autoregressive models, parallelized autoregressive variants, and non-autoregressive/diffusion approaches.

**Datasets and Resolutions:**
Experiments are performed on two resolutions to test scalability:
*   **ImageNet $256 \times 256$:** The primary benchmark, corresponding to $16 \times 16 = 256$ tokens (using a $16\times$ downsampling tokenizer).
*   **ImageNet $512 \times 512$:** A high-resolution stress test, corresponding to $32 \times 32 = 1024$ tokens.

**Metrics:**
The paper employs a standard suite of generative metrics to assess quality and diversity:
*   **Fréchet Inception Distance (FID):** The primary metric for distribution similarity (lower is better).
*   **Inception Score (IS):** Measures class distinctiveness and diversity (higher is better).
*   **Precision and Recall:** Quantifies the trade-off between sample fidelity (Precision) and coverage of the data manifold (Recall).
*   **Latency (seconds) and Throughput (images/sec):** Measured on a single **NVIDIA A100 GPU** with **BF16** precision. Latency is reported for batch size 1 (memory-bound regime), while throughput is measured at batch size 64.

**Baselines:**
The comparison set is extensive, covering:
*   **Diffusion:** ADM-G, DiT-XL/2, SiT-XL/2.
*   **Masked Non-Autoregressive:** MaskGIT, MAGVIT-v2, MaskBit.
*   **Standard Autoregressive:** LlamaGen (various sizes), RAR.
*   **Parallelized Autoregressive (Direct Competitors):** PAR, RandAR, ARPG, NAR.
*   **Multi-Scale Autoregressive:** VAR (included for efficiency context, though token-incompatible).

**Model Configurations:**
The authors train three LPD variants to ensure fair parameter-matched comparisons:
*   **LPD-L:** 337M parameters (12 layers).
*   **LPD-XL:** 752M parameters (36 layers).
*   **LPD-XXL:** 1.4B parameters (48 layers).
All models use the **LLAMAGEN** tokenizer (codebook size 16,384) and are trained for 450 epochs on $256 \times 256$, with an additional 50 epochs of fine-tuning for $512 \times 512$ models using interpolated positional embeddings.

### 5.2 Main Quantitative Results: Efficiency vs. Quality Trade-off
The core claim of LPD is that it can drastically reduce generation steps without compromising image quality. The results in **Table 1** ($256 \times 256$) and **Table 2** ($512 \times 512$) provide compelling evidence for this.

**Reduction in Generation Steps:**
LPD achieves a massive reduction in sequential steps compared to standard raster-scan autoregression:
*   **$256 \times 256$:** Steps reduced from **256** (standard) to **20** (LPD), a **$12.8\times$ reduction**.
*   **$512 \times 512$:** Steps reduced from **1024** (standard) to **48** (LPD), a **$21.3\times$ reduction**.

**Quality Preservation (FID):**
Critically, this speedup does not come at the cost of quality.
*   **LPD-XL (752M)** achieves an **FID of 2.10** with only **20 steps**.
*   This matches or exceeds the quality of **ARPG-XXL** (1.3B params, FID 1.94) which requires **64 steps**, and significantly outperforms **RandAR-XXL** (1.4B params, FID 2.15) which requires **88 steps**.
*   When slightly increasing steps to **32**, LPD-XL achieves an **FID of 1.92**, surpassing even the largest parallelized baselines while maintaining a massive latency advantage.

**Latency and Throughput Gains:**
The system-level profiling confirms that the reduction in steps translates directly to real-world speedups, overcoming the memory-bound bottleneck.
*   **Latency:** On $256 \times 256$, LPD-XL achieves a latency of **0.41s** (20 steps). In comparison:
    *   **ARPG-XL** (64 steps): **1.71s**.
    *   **RandAR-XL** (88 steps): **2.78s**.
    *   **Standard LlamaGen-XL** (576 steps): **18.51s**.
    *   This represents a **$4.2\times$ speedup** over ARPG and a **$6.8\times$ speedup** over RandAR.
*   **Throughput:** At batch size 64, LPD-XL reaches **75.20 images/sec**, vastly outperforming ARPG-XL (**36.53 img/s**) and RandAR-XL (**17.06 img/s**).

**High-Resolution Scalability:**
The advantages persist at higher resolutions (**Table 2**). For $512 \times 512$ generation:
*   **LPD-XL** generates images in **1.01s** (48 steps) with an **FID of 2.10**.
*   Standard raster autoregression (Raster Counterpart-XL) takes **20.93s** (1024 steps) for a similar FID (2.09).
*   This demonstrates that LPD scales efficiently, avoiding the linear latency explosion inherent in raster-scan methods as resolution increases.

**Visual Comparison:**
**Figure 1** visualizes the Pareto frontier of Latency vs. FID. LPD models (blue dots) occupy the bottom-left region of the chart, indicating simultaneously lower latency and lower (better) FID compared to all other parallelized autoregressive models (PAR, RandAR, ARPG).

### 5.3 Ablation Studies: Validating Design Choices
The authors perform targeted ablation studies (**Figure 9**) to verify that each component of LPD is necessary for its performance. These experiments are conducted on the XL model ($256 \times 256$).

**1. Effectiveness of Flexible Parallelized Modeling (Mutual Visibility):**
*   **Experiment:** The authors compare LPD against RandAR and ARPG using a *random* generation order for all methods to isolate the architectural impact.
*   **Result:** As the number of steps decreases (parallelism increases), RandAR and ARPG suffer significant FID degradation. For instance, at **32 steps**, ARPG and RandAR show a sharp rise in FID, whereas LPD maintains performance nearly identical to its 256-step baseline.
*   **Conclusion:** This confirms that **mutual visibility** (Section 4.2) is critical. Without the ability for concurrent tokens to attend to each other, high-degree parallelism leads to inconsistent predictions and quality collapse.

**2. Effectiveness of Locality-aware Scheduling:**
*   **Experiment:** LPD's schedule is compared against **Random** ordering and **Halton** ordering (a low-discrepancy sequence used in prior work).
*   **Result:** **Figure 9(b)** shows that LPD consistently achieves lower FID across all step counts (16, 20, 24, 32, 64).
    *   At 32 steps, LPD achieves an FID of **~1.92**, while Random and Halton orders yield FIDs of **~2.10** and **~2.05** respectively.
*   **Conclusion:** Simply spreading tokens out (Halton) or shuffling them (Random) is insufficient. The specific strategy of hugging the generated frontier (high proximity to context) while spacing out concurrent tokens (low intra-group dependency) provides a measurable quality boost.

**3. Synergy of Locality Principles:**
*   **Experiment:** The two principles of the scheduler are ablated individually:
    *   *Principle 1 Only:* Maximize proximity to previous tokens (ignore intra-group distance).
    *   *Principle 2 Only:* Maximize distance within the group (ignore proximity to context).
*   **Result (**Figure 9(c)**):**
    *   Random Baseline: FID **2.11**.
    *   Principle 1 Only: FID **2.00**.
    *   Principle 2 Only: FID **2.06**.
    *   **LPD (Both Principles):** FID **1.92**.
*   **Conclusion:** Neither principle alone is optimal. Maximizing context proximity helps, but without repulsion, concurrent tokens interfere with each other. Conversely, spreading tokens out helps consistency but weakens conditioning if they are too far from existing context. The **synergy** of both is required for state-of-the-art results.

### 5.4 Zero-Shot Generalization and Robustness
Beyond standard generation, the paper evaluates LPD's ability to perform **zero-shot editing** tasks, leveraging its flexible generation order (Section 3.3, **Figure 10**).

*   **Inpainting & Outpainting:** By pre-filling the KV cache with tokens from unmasked regions and generating the missing regions in a random order, LPD successfully completes images without retraining. The seamless integration of generated tokens with the context demonstrates that the model has learned robust spatial dependencies that are not tied to a specific raster order.
*   **Class-Conditional Editing:** The model can edit specific regions by swapping the class embedding and regenerating only the target area.
*   **Significance:** This capability is a direct consequence of the **decoupled position query** design. Standard raster-scan models cannot easily inpaint arbitrary shapes without complex masking strategies during training, and multi-scale models often struggle with localized edits due to their coarse-to-fine dependency. LPD's flexibility here serves as a strong qualitative validation of its architectural design.

### 5.5 Critical Assessment
Do the experiments convincingly support the claims? **Yes.**

*   **Strength of Evidence:** The comparison against a wide range of baselines (Table 1 & 2) is rigorous. The authors do not just compare against old methods; they include very recent parallelized works (ARPG, RandAR, NAR) and show clear margins of improvement in both FID and latency.
*   **Causality:** The ablation studies (Figure 9) effectively rule out the possibility that the gains are merely due to larger model size or more training data. They pinpoint the *mechanism* of improvement: mutual visibility prevents quality collapse at high parallelism, and locality scheduling optimizes the conditioning signal.
*   **Real-World Relevance:** By reporting latency on a single A100 with BF16, the paper addresses the actual deployment bottleneck (memory bandwidth) rather than just theoretical FLOPs. The $4.2\times$ speedup is a tangible system-level win.

**Limitations and Trade-offs:**
*   **Precomputation Overhead:** The locality-aware schedule requires precomputing the generation order (Algorithm 1). While the paper notes this incurs "no additional latency" during inference (as it is done once and stored), it does add a minor setup cost and memory requirement to store the schedule indices. However, this is negligible compared to the per-image latency savings.
*   **Hyperparameter Sensitivity:** The schedule relies on thresholds $\tau$ (proximity) and $\rho$ (repulsion). While the paper provides the logic, it does not extensively sweep these values in the main text, implying they may require tuning for different tokenizers or resolutions.
*   **Comparison to Diffusion:** While LPD is faster than many diffusion models (e.g., DiT-XL/2 takes 4.34s vs LPD-XL's 0.41s), the very fastest masked models (like MaskBit at 1.03s) are competitive in latency, though LPD still holds a quality advantage (FID 2.10 vs 1.62 for MaskBit, though MaskBit is a different paradigm). The primary victory is within the *autoregressive* family, where LPD is now dominant.

In summary, the experimental section provides a robust, multi-faceted validation of LPD. It demonstrates that by rethinking the attention mechanism and generation schedule, autoregressive models can break the sequential bottleneck without sacrificing the high fidelity that makes them attractive for unified multimodal systems.

## 6. Limitations and Trade-offs

While Locality-aware Parallel Decoding (LPD) represents a significant leap in autoregressive efficiency, a critical examination reveals specific assumptions, operational constraints, and unresolved challenges. Understanding these limitations is essential for determining where LPD is applicable and where future research must focus.

### 6.1 Reliance on Precomputed Static Schedules
A fundamental assumption of LPD is that an optimal generation order can be determined **a priori** and remains static across all samples.
*   **The Constraint:** The **Locality-aware Generation Order Schedule** (Algorithm 1) is precomputed based on grid topology and stored as a fixed sequence of indices before inference begins. The paper explicitly states, "the generation order can be precomputed and stored for direct use during inference, incurring no additional latency" (Section 2.3).
*   **The Trade-off:** This design assumes that the optimal spatial dependency structure is **image-agnostic**. It presumes that the "best" next tokens to generate depend only on the geometry of the already-filled region, not on the *content* of those tokens.
*   **The Weakness:** In complex images with heterogeneous structures (e.g., a sharp object boundary on the left and a smooth sky on the right), a static schedule cannot adapt. A dynamic scheduler might prioritize generating the high-frequency edge first to establish a strong context, while LPD blindly follows the precomputed path. This rigidity potentially leaves performance on the table compared to a hypothetical content-adaptive scheduler, though such a scheduler would introduce its own latency overhead.

### 6.2 Sensitivity to Hyperparameters ($\tau$ and $\rho$)
The efficacy of the generation schedule hinges on two critical hyperparameters: the **proximity threshold** ($\tau$) and the **repulsion threshold** ($\rho$).
*   **The Assumption:** The method assumes there exists a "sweet spot" where tokens are close enough to the context to be well-conditioned but far enough from each other to be independent.
*   **The Gap:** The paper provides the algorithmic logic and the PyTorch implementation (Appendix C) but **does not report a sensitivity analysis** for these thresholds in the main text. It is unclear how robust the FID scores are if $\tau$ or $\rho$ are suboptimal.
*   **The Risk:** If $\tau$ is set too high, the candidate pool $c_1$ becomes empty, forcing the model to rely entirely on Farthest Point Sampling (FPS) from the distant set $c_2$, which violates the "high proximity to context" principle and likely degrades quality. Conversely, if $\rho$ is too small, the repulsion filter becomes ineffective, allowing spatially adjacent tokens to be generated in the same step, re-introducing the inconsistency artifacts LPD aims to solve. The lack of explicit tuning guidelines suggests these values may require careful re-calibration for different tokenizers or resolutions.

### 6.3 Dependency on Flat Tokenizers and Spatial Locality
LPD's theoretical foundation is deeply rooted in the specific properties of the **LLAMAGEN** tokenizer and the observed spatial locality of its attention maps.
*   **The Assumption:** The entire "Locality-aware" strategy relies on the empirical finding that "attention diminishes rapidly over distance" (Figure 7). This strong spatial locality justifies the strategy of growing the generation frontier from existing tokens.
*   **The Limitation:** This approach may not generalize seamlessly to tokenizers that do not exhibit such strong spatial locality. For instance, semantic tokenizers or those using global pooling mechanisms might distribute attention more uniformly across the image. If the attention map is diffuse rather than local, the "proximity to context" principle (Principle 1) loses its predictive power, and the carefully crafted schedule could become suboptimal or even detrimental.
*   **Unaddressed Scenario:** The paper validates LPD exclusively on ImageNet using the LLAMAGEN tokenizer (codebook size 16,384, downsample factor 16). It does not demonstrate performance on datasets with vastly different spatial statistics (e.g., natural landscapes vs. synthetic diagrams) or with different tokenization schemes (e.g., VQVAE-2 or TiTok), leaving the generalizability of the locality principles an open question.

### 6.4 Scalability of the "First Token" Bottleneck
While LPD reduces the total steps from 256 to 20, it does not eliminate the sequential nature of the *groups*.
*   **The Constraint:** The schedule typically starts with very small group sizes (e.g., 1 or 2 tokens) because the context is initially sparse (Section 3.4, "Dynamic Group Sizes"). The group size grows via a cosine schedule as the image fills.
*   **The Weakness:** The initial steps remain strictly sequential and memory-bound. In the extreme case of the very first token, the parallelism factor is 1. While this is a minor fraction of the total latency in a 20-step process, it represents a hard lower bound on latency that cannot be parallelized away without changing the fundamental autoregressive formulation.
*   **Comparison:** This contrasts with fully non-autoregressive models (like MaskGIT) which can theoretically predict all tokens in the first step (albeit with iterative refinement). LPD trades the massive step count of raster AR for a smaller count of dependent groups, but the dependency chain still exists.

### 6.5 Memory Overhead of Schedule Storage
Although the paper claims the precomputation incurs "no additional latency," it implicitly introduces a memory storage cost.
*   **The Detail:** For every resolution and every desired step count (e.g., 20 steps, 32 steps), a specific permutation of token indices must be stored.
*   **The Constraint:** For high-resolution generation (e.g., $1024 \times 1024$), the number of tokens grows quadratically ($4096$ tokens). Storing multiple precomputed schedules for various step counts and resolutions consumes GPU host memory. While negligible compared to model weights for $256 \times 256$, this could become a non-trivial overhead in memory-constrained edge deployment scenarios where multiple resolution schedules are required.

### 6.6 Unaddressed Text-to-Image Conditioning Nuances
The experiments focus exclusively on **class-conditional** generation (ImageNet).
*   **The Gap:** The paper mentions compatibility with unified multimodal models but does not provide empirical results for **text-to-image** generation with complex, long prompts.
*   **The Open Question:** In text-to-image tasks, the alignment between specific text tokens and image regions is often non-local and semantic. It is unclear if the spatially-driven locality schedule (which prioritizes geometric proximity) is optimal for semantic grounding. A text prompt like "a red bird on the left and a blue car on the right" might benefit from generating both sides simultaneously to ensure color consistency with the text, whereas LPD's schedule might generate one side fully before starting the other, potentially leading to drift in the second region's adherence to the prompt.

### 6.7 Comparison to Fastest Non-Autoregressive Baselines
While LPD dominates *autoregressive* baselines, its advantage over the fastest *non-autoregressive* (NAR) masked models is less absolute.
*   **The Evidence:** In Table 1, **MaskBit** (a masked NAR model) achieves an FID of **1.62** with **64 steps** and a latency of **1.03s**. LPD-XXL achieves an FID of **2.00** with **20 steps** and **0.55s**.
*   **The Trade-off:** LPD is faster (0.55s vs 1.03s), but MaskBit achieves significantly better quality (FID 1.62 vs 2.00). This suggests that while LPD solves the AR efficiency problem, it has not yet closed the **quality gap** with state-of-the-art masked modeling approaches that can leverage bidirectional context more freely. LPD offers a better speed/quality trade-off *within* the AR family, but AR as a paradigm still trails the highest-fidelity NAR methods in this specific benchmark.

In summary, LPD is a highly optimized solution for **flat-token, class-conditional autoregressive generation** where memory bandwidth is the bottleneck. Its limitations lie in its reliance on static, geometry-based scheduling, its unproven generalizability to non-spatially-local tokenizers, and the remaining quality gap compared to the most advanced non-autoregressive paradigms.

## 7. Implications and Future Directions

### 7.1 Reshaping the Autoregressive Landscape: Breaking the Speed-Quality-Compatibility Trilemma
Locality-aware Parallel Decoding (LPD) fundamentally alters the trajectory of autoregressive (AR) visual generation by dismantling a long-standing trilemma. Prior to this work, researchers were forced to choose two of the following three attributes:
1.  **High Efficiency:** Achieved by multi-scale methods (e.g., VAR) or non-autoregressive masked models (e.g., MaskGIT), but often at the cost of token compatibility or bidirectional attention overhead.
2.  **High Quality & Consistency:** Achieved by standard raster-scan AR models, but crippled by prohibitive latency due to memory-bound sequential decoding.
3.  **Flat-Token Compatibility:** Essential for unified multimodal systems (integrating with CLIP, DINO, or LLMs), but historically restricted to slow, sequential AR models.

LPD demonstrates that **flat-token autoregressive models can be as efficient as multi-scale methods** without sacrificing their universal interoperability. By proving that generation steps can be reduced by over **12$\times$** (from 256 to 20) while maintaining state-of-the-art FID scores, LPD removes the primary systems-level argument against using AR models for real-time applications. This shifts the field's focus from "how to bypass autoregression" to "how to optimize autoregressive parallelism," validating the AR paradigm as a viable, high-speed backbone for next-generation unified multimodal foundation models.

### 7.2 Enabling New Research Frontiers
The architectural innovations in LPD open several concrete avenues for follow-up research:

*   **Content-Adaptive Dynamic Scheduling:**
    Currently, LPD relies on **static, precomputed schedules** based on grid topology (Section 2.3). A natural evolution is **dynamic scheduling**, where the generation order adapts to the image content in real-time. For instance, a model could detect high-frequency edges or semantic boundaries in the early steps and prioritize generating tokens along those structures to maximize contextual support, rather than following a fixed geometric pattern. This would require lightweight mechanisms to predict "uncertainty maps" or "information density" on the fly to guide the next group selection.

*   **Generalizing Locality Principles to Other Modalities:**
    The core insight—that attention exhibits strong **spatial locality** (Figure 7)—drives LPD's success. This principle invites investigation into other domains:
    *   **Video Generation:** Does spatiotemporal locality (attending to nearby frames and pixels) allow for similar parallel decoding schedules in video autoregression?
    *   **3D Point Clouds:** Can locality-aware ordering accelerate AR generation for 3D structures, where Euclidean distance is also a strong predictor of dependency?
    *   **Audio/Speech:** While 1D, audio signals exhibit local temporal dependencies. Adapting LPD's repulsion/proximity logic to 1D sequences could accelerate speech synthesis.

*   **Hybrid AR-NAR Architectures:**
    LPD bridges the gap between AR and Non-Autoregressive (NAR) models. Future work could explore **hybrid training objectives** where the model is trained to switch between strict AR mode (for difficult regions) and high-parallelism LPD mode (for smooth regions) within a single generation pass, dynamically adjusting the degree of parallelism based on local image complexity.

*   **Tokenizer-Aware Scheduling:**
    The current schedule assumes a generic grid topology. Future research could tailor the generation order to the specific **inductive biases of different tokenizers**. For example, if a tokenizer produces tokens with varying semantic granularity (e.g., some tokens represent objects, others textures), the schedule could prioritize "semantic anchor" tokens first to establish global structure before filling in details.

### 7.3 Practical Applications and Downstream Use Cases
The reduction in latency from seconds to sub-second ranges unlocks several practical applications that were previously infeasible for flat-token AR models:

*   **Real-Time Interactive Image Editing:**
    With latency dropping to **~0.4s** for $256 \times 256$ images (Table 1), LPD enables **interactive inpainting and outpainting** tools where the user sees results almost instantly. The zero-shot editing capability (Section 3.3) allows for seamless integration into design software, enabling features like "smart fill" or real-time style transfer without the lag associated with diffusion models or standard AR.

*   **Edge Deployment and Mobile Generative AI:**
    The memory-efficient design (avoiding instruction token caching) makes LPD suitable for resource-constrained environments. The ability to generate high-quality images with fewer memory accesses reduces power consumption, paving the way for deploying sophisticated generative models on **mobile devices or edge GPUs** where memory bandwidth is the primary bottleneck.

*   **Unified Multimodal Agents:**
    As the paper highlights, flat tokens are critical for **unified multimodal systems** (Section 1). LPD allows large language models (LLMs) to generate visual outputs as part of a continuous stream of tokens without stalling the inference pipeline. This is crucial for **embodied AI agents** that need to "imagine" future states or generate visual plans in real-time while processing language instructions.

*   **High-Throughput Data Synthesis:**
    The **4.2$\times$ throughput increase** (up to 75 images/sec on an A100) makes LPD an ideal engine for **synthetic data generation**. Researchers can rapidly generate massive datasets of class-conditioned images for training downstream discriminative models, significantly reducing the cost and time of data curation.

### 7.4 Reproducibility and Integration Guidance
For practitioners considering adopting LPD, the following guidelines clarify when and how to integrate this method:

*   **When to Prefer LPD:**
    *   **Requirement for Flat Tokens:** Choose LPD if your pipeline requires compatibility with existing vision encoders (CLIP, DINO) or unified multimodal LLMs that expect a fixed sequence of flat tokens. Do *not* use multi-scale methods (VAR) if this compatibility is needed.
    *   **Latency Sensitivity:** If your application is latency-bound (e.g., interactive tools, real-time agents), LPD is superior to standard raster-scan AR (LlamaGen) and competitive with or faster than masked NAR models (MaskGIT) while offering a simpler AR inference stack (KV caching).
    *   **Memory Constraints:** LPD is preferable to instruction-token-based methods (RandAR) in memory-constrained settings, as it avoids doubling the KV cache size.

*   **Integration Steps:**
    1.  **Tokenizer Alignment:** Ensure your tokenizer produces a 2D grid of tokens (e.g., $16 \times 16$ or $32 \times 32$). The code provided in **Appendix C** assumes a square grid; rectangular grids may require minor modifications to the Euclidean distance calculations.
    2.  **Schedule Precomputation:** Before deployment, precompute the generation order schedules for your target resolutions and desired step counts (e.g., 20, 32, 48 steps). Use the provided PyTorch implementation (Algorithm 1) with default thresholds ($\tau=1, \rho=1$) as a starting point, but be prepared to tune $\rho$ (repulsion) if you observe spatial artifacts in generated batches.
    3.  **Model Adaptation:** Integrating LPD requires modifying the attention mask logic in the transformer. You must implement the **dual-pattern attention mask** (Context + Query Attention) described in Section 2.2. This is not a drop-in replacement for standard causal masks; it requires changes to the underlying attention kernel to allow mutual visibility among query tokens while maintaining causality with respect to the image history.
    4.  **Training Strategy:** To fully leverage LPD, models should be trained with **variable step counts** (randomly sampling from a set of step sizes, as done in Section 3.1). Training only on a fixed step count (e.g., always 20 steps) may reduce the model's robustness when inference conditions change.

*   **Potential Pitfalls:**
    *   **Static Schedule Rigidity:** Be aware that the static schedule may not be optimal for every image. If your domain involves highly irregular structures, monitor for artifacts in early generation steps where the static order might conflict with the image's semantic layout.
    *   **Hyperparameter Tuning:** The repulsion threshold $\rho$ is critical. If set too low, you risk generating adjacent tokens simultaneously, leading to inconsistencies. If set too high, you may fail to fill the batch size, reducing parallelism. Validate the schedule visually on a small subset before full-scale deployment.

In conclusion, LPD provides a robust, efficient, and compatible framework that re-establishes autoregressive modeling as a leading candidate for real-time, unified multimodal generation. Its blend of architectural elegance and systems-level efficiency offers a clear path forward for both research and practical deployment.
