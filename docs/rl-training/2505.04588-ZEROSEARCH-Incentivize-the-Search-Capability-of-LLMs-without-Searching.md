## 1. Executive Summary

ZEROSEARCH solves the prohibitive API costs (involving hundreds of thousands of search requests) and uncontrolled document quality associated with training search-augmented LLMs by replacing real search engines with a simulated retrieval module during reinforcement learning. Using a curriculum-based rollout strategy with fine-tuned simulators (e.g., 7B or 14B LLMs), the framework achieves performance comparable to or exceeding real search engine-based methods like Search-R1 on benchmarks such as NQ and HotpotQA while incurring zero API costs. This approach enables scalable training of search capabilities across various model sizes, including 3B, 7B, and 14B configurations, without relying on external search APIs.

## 2. Context and Motivation

### The Core Problem: Training Search-Capable LLMs at Scale

Large Language Models (LLMs) face a fundamental limitation: their knowledge is **static**, frozen at the time of pretraining. As stated in the Introduction (Section 1), this makes LLMs "prone to generating hallucinated content or outdated information," which undermines their reliability in practical applications. To address this, researchers have developed methods that enable LLMs to access external information sources during reasoning and generation.

The specific problem ZEROSEARCH addresses is **how to train LLMs to effectively use search engines through reinforcement learning (RL) without incurring prohibitive costs or suffering from training instability**. This problem has two concrete dimensions:

1. **Uncontrolled Document Quality**: When training with real search engines, the quality of retrieved documents is unpredictable. Some queries return highly relevant information; others return noise. This variability introduces instability into the RL training process, making it difficult for the policy model to learn consistent search behaviors.

2. **Prohibitively High API Costs**: RL training requires frequent **rollouts** (multiple sampling iterations per training step to estimate gradients). As quantified in Appendix D (Table 8), a typical training run involves approximately **64,000 search requests**. Using Google Web Search via SerpAPI, this incurs **$586.7 in API costs** for a single training run. This cost scales linearly with batch size and training steps, severely constraining scalability for research and development.

### Why This Problem Matters

The importance of this problem extends across both practical and theoretical dimensions:

**Real-World Impact:**
- **Cost Barrier to Entry**: At nearly $600 per training run (and this is for a relatively modest 200 training steps with batch size 64), only well-funded organizations can afford to develop search-augmented LLMs using real search engines. This limits innovation and reproducibility.
- **Deployment Scalability**: If training is cost-prohibitive, then iterating on model architectures, hyperparameters, or reward designs becomes impractical. This slows down progress in the field.
- **Training Stability**: Uncontrolled document quality means that two training runs with the same hyperparameters can produce different results simply due to variations in what the search engine returns. This undermines scientific rigor.

**Theoretical Significance:**
- **RL in Grounded Environments**: Training agents to interact with external tools is a core challenge in building autonomous AI systems. Search is a canonical example of tool use—understanding how to train this capability efficiently has implications beyond information retrieval.
- **Simulation vs. Reality**: ZEROSEARCH explores whether a learned simulator (an LLM trained to mimic a search engine) can substitute for real environments during training. This question is relevant to robotics, game playing, and any domain where real-world interaction is expensive or dangerous.

### Prior Approaches and Their Limitations

To understand ZEROSEARCH's contribution, we must examine the landscape of existing methods for enabling LLMs to use external information.

#### Retrieval-Augmented Generation (RAG)

Early work focused on **Retrieval-Augmented Generation (RAG)**, which incorporates external knowledge into the generation pipeline (Section 2.1). Initial approaches used **prompt-based strategies** that guide LLMs through:
- Query generation
- Query decomposition
- Multi-turn information retrieval

While effective, these methods "require meticulous prompt engineering and place high demands on the model's reasoning capabilities" (Section 1). They also do not *learn* search behavior—they rely on the model's pre-existing abilities.

Subsequent work applied **Supervised Fine-Tuning (SFT)** to enhance smaller LLMs (e.g., Self-RAG, RetroLLM). However, SFT requires labeled trajectories showing correct search behavior, which are expensive to collect and may not generalize to unseen queries.

**Test-time scaling techniques** like Monte Carlo Tree Search (MCTS) dynamically expand the search space during inference (e.g., RAG-star, AirRAG). While promising, these methods "incur significant computational overhead, posing challenges for practical deployment" (Section 1).

#### Reinforcement Learning for Search

More recently, RL has emerged as a strategy for training LLMs to search more effectively. Representative works include:

| Method | Key Characteristic | Limitation |
|--------|-------------------|------------|
| Search-R1 (Section 1, 4.2) | RL-trained to generate multiple search queries during reasoning | Uses real search engine (high API cost) |
| R1-Searcher (Section 1, 2.2) | Two-stage, outcome-based RL for search capabilities | Typically uses static corpora (e.g., Wikipedia) |
| ReSearch (Section 1, 2.2) | RL to reason through searches without intermediate supervision | Static corpora, not real-world search |
| DeepResearcher (Section 1, 2.2) | Live interaction with commercial search engines (Google) | High API costs, uncontrolled document quality |
| WebThinker (Section 1, 2.2) | Direct interaction with real-world web search | Same cost and quality issues |

The critical distinction is between methods that use **static, local textual corpora** (like Wikipedia) versus those that interact with **live commercial search engines**. Static corpora are cost-effective but "fail to capture the complexities of real-world interaction" (Section 2.2). Live search engines capture real-world complexity but introduce the cost and quality problems described above.

### ZEROSEARCH's Positioning

ZEROSEARCH positions itself as a **middle ground** between static corpora and live search engines. The key insight (Section 1) is:

> "LLMs have acquired extensive world knowledge during large-scale pretraining and are capable of generating relevant documents given a search query."

This means an LLM can **simulate** a search engine. The primary difference between a real search engine and a simulation LLM "lies in the textual style of the returned content." With lightweight supervised fine-tuning, "even relatively small LLMs can effectively simulate the behavior of real search engines" (Section 1).

ZEROSEARCH differentiates itself from prior work in three ways:

1. **No Real Search Engine During Training**: Unlike Search-R1, DeepResearcher, and WebThinker, ZEROSEARCH does not call external APIs during RL training. The simulation LLM generates all documents. This eliminates API costs entirely.

2. **Controllable Document Quality**: Unlike real search engines, the simulation LLM can be prompted to generate either **useful** or **noisy** documents by "adjusting a few words in the prompt" (Section 1). This enables curriculum learning—starting with easy (high-quality) documents and progressively introducing harder (noisier) scenarios.

3. **Compatibility and Generalizability**: ZEROSEARCH works with "both base and instruction-tuned models of various parameter sizes" and integrates with multiple RL algorithms (REINFORCE, PPO, GRPO) (Section 1). This contrasts with methods that require specific model types or custom RL implementations.

The empirical claim is striking: a **7B simulation LLM achieves performance comparable to real search engines**, and a **14B simulator surpasses real search engine performance** (Section 1, Section 5.2). This suggests that the simulation approach is not merely a cost-saving approximation—it may actually provide *better* training signals than real search engines due to the controllable curriculum.

### Summary of the Gap

In summary, the research gap ZEROSEARCH addresses is:

| Aspect | Static Corpora Methods | Real Search Engine Methods | ZEROSEARCH |
|--------|----------------------|---------------------------|------------|
| API Cost | None | High (~$586 per run) | None (GPU cost only: ~$18–$71) |
| Document Quality Control | Fixed | Uncontrolled | Controllable via curriculum |
| Real-World Fidelity | Low | High | High (with 7B+ simulators) |
| Training Stability | High | Low (quality variance) | High (curriculum-based) |
| Scalability | High | Low (cost-limited) | High (GPU-parallelizable) |

ZEROSEARCH does not claim that simulation is perfect—it acknowledges that "deploying the simulated search LLM requires access to GPU servers" which "introduces additional infrastructure costs" (Section 6). However, as Table 8 shows, even the highest-cost simulator (14B on 4 GPUs at $70.8) is **8× cheaper** than real search API costs ($586.7), while offering superior training control.

This positions ZEROSEARCH as a practical enabler for research and development of search-augmented LLMs, removing the cost and stability barriers that have limited prior work.

## 3. Technical Approach

This is a reinforcement learning systems paper that proposes replacing expensive, unpredictable real search engines with a trainable LLM simulator during policy optimization. The core idea is that instead of calling Google Search APIs during RL training, we use a fine-tuned LLM to generate search results, allowing us to control document quality through a curriculum while eliminating API costs entirely.

### 3.1 Reader orientation (approachable technical breakdown)

ZEROSEARCH builds a two-model system where one LLM (the policy) learns to search and reason, while another LLM (the simulator) pretends to be a search engine by generating documents in response to queries. This solves the dual problem of prohibitive API costs and unstable training caused by unpredictable search result quality by substituting controllable, cheap simulation for expensive, noisy reality.

### 3.2 Big-picture architecture (diagram in words)

The system consists of four major components that interact during training:

1. **Policy Model (`πθ`)**: The LLM being trained to perform reasoning and issue search queries. It receives questions as input and outputs reasoning traces, search queries, and final answers.

2. **Simulation LLM (`πψ`)**: A separate, frozen LLM that acts as a search engine surrogate. It receives search queries from the policy model and returns generated documents (either useful or noisy, controlled by prompt).

3. **Reward Model (`rϕ`)**: A rule-based function that evaluates the policy model's final answer against ground truth, producing an F1 score that serves as the training signal.

4. **Rollout Module**: The orchestration layer that manages multi-turn interactions between the policy and simulator, applies the curriculum noise schedule, and handles loss masking during gradient computation.

Information flows as follows: A question enters the policy model → the policy reasons and optionally issues a search query → the simulation LLM generates documents → the policy receives documents and continues reasoning → this loop repeats until the policy produces a final answer → the reward model scores the answer → gradients update only the policy model (not the simulator).

### 3.3 Roadmap for the deep dive

- **RL objective formulation**: We start with the mathematical foundation to establish what is being optimized and why the simulator parameters remain frozen.
- **Interaction template design**: Next, we explain how the policy model is structured to produce searchable outputs, since the format determines what the simulator receives.
- **Search simulation tuning**: We then detail how the simulator LLM is trained to generate realistic documents, as this component replaces the real search engine.
- **Curriculum rollout mechanism**: We explain the progressive noise injection strategy, which is the key innovation enabling stable training without real search results.
- **Reward design and loss masking**: Finally, we cover how training signals are computed and why document tokens are excluded from gradient computation.

### 3.4 Detailed, sentence-based technical breakdown

#### Reinforcement Learning Objective Without Search Engine

The paper formulates search-augmented training as a standard reinforcement learning problem with a KL-divergence regularization term, but with a critical modification: the environment (search engine) is replaced by a learned simulator. The optimization objective is:

$$\max_{\pi_\theta} \mathbb{E}_{x \sim \mathcal{D}, y \sim \pi_\theta(\cdot|x; \pi_\psi)} [r_\phi(x, y)] - \beta D_{KL}[\pi_\theta(y | x; \pi_\psi) \| \pi_{ref}(y | x; \pi_\psi)]$$

In plain language, this equation states that we want to maximize the expected reward (answer accuracy) while keeping the policy model's output distribution close to a reference model to prevent degenerate behavior. Here, `πθ` represents the policy model being optimized (the searcher), `πref` is a fixed reference model (typically the initial policy before RL), `rϕ` is the reward function that scores answers, and `πψ` is the simulation LLM that generates search documents. The key observation is that `πψ` appears in the conditional distribution but its parameters remain fixed throughout training—only `πθ` receives gradient updates. This design choice prevents the simulator from adapting to the policy's weaknesses, which would create a feedback loop where both models co-adapt to generate easy scenarios rather than learning robust search behavior. The KL-divergence term (weighted by `β`) serves as a regularizer to prevent the policy from drifting too far from its initial behavior, which is standard practice in RLHF (Reinforcement Learning from Human Feedback) to maintain output quality.

#### Training Template and Interaction Structure

The policy model is constrained to produce outputs in a structured format that explicitly separates reasoning, searching, and answering into distinct phases. As shown in Table 1, the template requires the model to first articulate its internal reasoning within `` and `</think>` tags, then optionally issue a search query within `&lt;search>` and `</search>` tags, and finally provide the answer within `&lt;answer>` and `</answer>` tags. This explicit separation enforces a structured decision-making process where the model cannot produce an answer without first demonstrating its reasoning chain. The template states "You can search as many times as you want," which enables multi-turn interaction—the policy can issue multiple sequential queries, each followed by new reasoning based on the returned documents. This design mirrors how humans search: we reason about what we know, identify gaps, query for missing information, then reason again with new evidence. The template is appended to every training question, and the model must learn to follow this format through RL rewards rather than supervised fine-tuning on the format itself. The paper notes that "the model consistently produces well-formed responses without explicit supervision" on output format (Section 3.5), meaning the reward signal alone is sufficient to teach proper tag usage. This is a non-trivial design choice: many RL approaches add format penalties to the reward function, but ZEROSEARCH avoids this complexity by relying on the base model's instruction-following capability.

#### Search Simulation Tuning

The simulation LLM must be transformed from a general-purpose language model into a retrieval module that can generate documents resembling real search engine results. This transformation occurs through a lightweight supervised fine-tuning procedure that precedes RL training. The process begins by collecting interaction trajectories where an LLM engages with a real search engine (Google Web Search via SerpAPI) in a multi-turn manner until reaching a final answer. From these trajectories, the system extracts query-document pairs—each search query issued by the model is paired with the actual documents returned by the search engine. The critical innovation is that these documents are then labeled as either **useful** or **noisy** using the LLM itself as a judge. The LLM independently assesses whether each document contains sufficient information to answer the corresponding query, and if affirmative, the document is labeled as useful; otherwise, it is labeled as noisy. This labeling step is essential because it enables controllable document generation during RL training. After labeling, the simulation LLM undergoes supervised fine-tuning on these query-document pairs with the labels embedded in the prompt. As shown in Table 2, the distinction between useful and noisy outputs is controlled by adjusting keywords in the prompt: the template includes "Given a query, you need to generate five [useful / noisy] documents for the query." By simply changing the bracketed keyword from "useful" to "noisy," the same fine-tuned model can generate either high-quality documents containing answer-relevant information or low-quality documents that lack sufficient information. The prompt also includes the input question and its ground-truth answer ("The user is trying to answer the question: [question] whose answer is [ground truth]"), which broadens the simulation LLM's knowledge boundary and helps it generate documents that are contextually appropriate for the specific query. Each generated document should contain about 30 words according to the template, which keeps the simulation lightweight while providing enough content for the policy model to extract relevant information. This lightweight SFT is a critical design choice: rather than training the simulator from scratch or using prompting alone, the paper finds that "fine-tuned simulation engines significantly outperform prompt-based simulation engines" (Section 5.2). The distribution gap between prompted outputs and real search results is substantial enough that fine-tuning is necessary for the simulator to produce realistic document styles.

#### Curriculum-Based Rollout Strategy

The curriculum rollout mechanism is the core innovation that enables stable RL training without real search engines. During rollout, the policy model performs interactive reasoning and generates search queries, which are fed into the simulation LLM to produce corresponding documents. However, rather than always generating useful documents, the system progressively degrades document quality over time according to a probability function. The noise probability at training step `i` is calculated as:

$$p_i = p_s + \frac{i/m - 1}{b - 1}(p_e - p_s)$$

This equation governs the likelihood of generating noisy documents at each training step. Here, `ps` represents the initial noise probability (at the start of training), `pe` represents the final noise probability (at the end of training), `i` denotes the current training step, `m` denotes the total number of training steps, and `b` is the exponential base with a default value of 4. As training progresses, the ratio `i/m` increases from 0 to 1, leading to a higher `pi` value—meaning a greater chance of producing noisy documents. The paper uses different noise schedules for different model sizes: for Qwen-2.5-7B (Base/Instruct), `ps = 0` and `pe = 0.75`; for Qwen-2.5-3B (Base/Instruct), `ps = 0` and `pe = 0.25`; for LLaMA-3.2-3B (Base/Instruct), `ps = 0.25` and `pe = 0.5` (Appendix F). These hyperparameters reflect the insight that smaller or less capable models need gentler curricula—starting with too much noise would prevent them from learning basic search behaviors. The curriculum serves two purposes: first, it allows the policy model to "learn basic output structures and task requirements" before facing challenging scenarios (Section 3.4). Early in training, when `pi` is low, the policy receives mostly useful documents, making it easier to learn the correct format and understand how to use search results. Second, as `pi` increases, the policy must "gradually adapt to more challenging and noisy retrieval scenarios," which forces it to develop more robust reasoning capabilities rather than relying on perfect search results. This is analogous to a student first learning with clear textbooks, then progressively working with more ambiguous sources. The ablation study in Table 6 confirms the curriculum's effectiveness: replacing the curriculum with a random rollout (fixed noise probability of 0.5 throughout training) reduces average performance by approximately 2 percentage points across both Qwen-2.5-3B-Base and LLaMA-3.2-3B-Base models. The curriculum's advantage is most pronounced on harder datasets like HotpotQA and Musique, where reasoning under noisy conditions is most critical.

#### Reward Design

The reward signal serves as the primary supervision in the reinforcement learning process, and ZEROSEARCH adopts a deliberately simple rule-based reward function that focuses solely on answer accuracy. During preliminary experiments, the authors observed that using exact match (EM) as the reward metric led to **reward hacking**: the policy model tended to produce excessively long answers to increase the chance of including the correct answer substring. To mitigate this issue, the system adopts an F1 score-based reward, which balances precision and recall. The reward is calculated as:

$$r_\phi(x, y) = \frac{2 \times IN}{PN + RN}$$

Here, `IN` denotes the number of overlapping words between the prediction and the ground truth, `PN` is the number of words in the prediction, and `RN` is the number of words in the ground truth. This formulation penalizes both overly long answers (which would have low precision) and incomplete answers (which would have low recall), encouraging the model to produce concise, accurate responses. The paper explicitly states "We do not incorporate an additional reward for output format, as we observe that the model consistently produces well-formed responses without explicit supervision" (Section 3.5). This is a notable design choice: many RL systems add format compliance penalties to ensure the model follows the template, but ZEROSEARCH relies on the base model's instruction-following capability and the structure of the interaction template itself to maintain format compliance. The reward is computed only on the final answer within the `<answer>` tags, not on intermediate reasoning or search queries. This outcome-based reward design aligns with recent work like DeepSeek-R1 and OpenAI-o1, which demonstrate that "reward-driven learning, without relying on explicit step-by-step supervision" can produce strong reasoning capabilities (Section 1).

#### Training Algorithm and Loss Masking

ZEROSEARCH is designed to be compatible with a wide range of reinforcement learning algorithms, including REINFORCE, Proximal Policy Optimization (PPO), and Group Relative Policy Optimization (GRPO). Each algorithm offers distinct advantages: REINFORCE provides training stability, PPO offers sample efficiency with value function estimation, and GRPO enables group-based relative optimization without a value model. However, a critical technical challenge arises because the rollout sequence comprises both tokens generated by the policy model and document tokens returned by the simulation LLM. Applying the same optimization procedure uniformly across both types of tokens can lead to training instability, as the document tokens are externally generated and not directly controlled by the policy model. To mitigate this, the system introduces a **loss masking mechanism** for document tokens, ensuring that gradients are only computed with respect to the model's own outputs. This strategy stabilizes the RL training process while preserving the effectiveness of retrieval-augmented generation. The ablation study in Figure 4c and Table 7 demonstrates the importance of this masking: removing document token loss masking leads to a substantial drop in reward during training and reduces average benchmark performance from 36.07 to 34.53 (approximately 1.5 percentage points) on LLaMA-3.2-3B-Base. Without masking, the policy model receives gradient signals for tokens it did not generate, which creates conflicting optimization pressures—the model is penalized or rewarded for content it had no control over. This is analogous to grading a student on text written by their teacher rather than their own essay. The masking ensures that only the policy's reasoning, queries, and answers contribute to the loss, while the simulator's documents are treated as fixed context. For implementation, the paper uses REINFORCE as the default algorithm across experiments. For both REINFORCE and GRPO, the policy LLM is trained with a learning rate of 1e-6, and five responses are sampled per prompt. In the PPO setting, the policy LLM is trained with a learning rate of 1e-6, while the value model is trained with a separate learning rate of 1e-5. The system applies Generalized Advantage Estimation (GAE) with hyperparameters `λ = 1` and `γ = 1` for PPO (Appendix F). These hyperparameters indicate that the system uses full-return estimates (γ = 1 means no discounting) and does not blend multi-step advantages (λ = 1 means using complete returns rather than truncated traces).

#### Hardware Configuration and Training Scale

The training infrastructure is split between two separate GPU clusters to enable parallel simulation and policy updates. The simulation server is deployed on 4 H20 GPUs, while RL training occurs on another 4 H20 GPUs (Section 4.3, Appendix F). This separation allows the simulation LLM to generate documents in parallel while the policy model performs gradient updates, maximizing hardware utilization. The default simulated search engine across all experiments is the fine-tuned Qwen-2.5-14B-Instruct model, which Section 5.2 shows surpasses real search engine performance. For SFT of the simulation LLM, the learning rate is set to 1e-6. The RL training configuration uses a batch size of 64, a rollout repetition of 5, and a total of 200 training steps, resulting in approximately 12 hours of training time and generating approximately 64,000 search requests (Appendix D). This scale is significant: 64,000 search requests via a real API would cost $586.7, but ZEROSEARCH incurs only GPU costs ($17.7 to $70.8 depending on simulator size). The training dataset merges the training sets of NQ and HotpotQA to create a unified dataset for all fine-tuning-based approaches, and evaluation is conducted on seven datasets to assess both in-domain and out-of-domain performance (Section 4.3). During inference, all models interact with the real web environment via Google Web Search to ensure fair comparison—ZEROSEARCH only replaces the search engine during training, not during evaluation. This design ensures that the learned search capabilities transfer to real-world search engines rather than overfitting to the simulator's output distribution.

## 4. Key Insights and Innovations

ZEROSEARCH introduces several conceptual breakthroughs that extend beyond the technical mechanics described in Section 3. This section identifies the fundamental innovations that distinguish this work from prior search-augmented RL approaches, separating true novelties from incremental improvements.

### 4.1 Fundamental Innovation: Decoupling Training from Real Search Infrastructure

The most significant contribution is the conceptual shift from **training with real environments** to **training with learned simulators** for search-augmented reasoning. Prior RL-based search methods (Search-R1, DeepResearcher, WebThinker) all assume that realistic training requires interaction with actual search engines. ZEROSEARCH challenges this assumption by demonstrating that an LLM can substitute for the search environment during training without sacrificing—and potentially improving—final performance.

This is a **fundamental innovation**, not an incremental improvement, for three reasons:

**First, it reframes the search problem as a simulation learning problem.** Rather than treating the search engine as a fixed external environment, ZEROSEARCH treats it as a learnable component that can be optimized for training effectiveness. This parallels advances in robotics where simulated environments enable faster, safer training before deployment in the real world. The key insight (Section 1) is that "the primary difference between a real search engine and a simulation LLM lies in the textual style of the returned content"—a gap that lightweight SFT can bridge.

**Second, it eliminates a hard scalability constraint.** As quantified in Appendix D (Table 8), real search API costs scale linearly with training steps and batch size. This creates a hard ceiling on how much search-augmented RL research can be conducted. ZEROSEARCH removes this ceiling by replacing API costs with GPU costs, which are more predictable and can be parallelized. The cost difference is substantial: $586.7 for real search versus $17.7–$70.8 for simulation (Table 8).

**Third, it enables reproducibility.** Real search engine results vary over time—the same query issued today versus next month may return different documents. This variability makes exact reproduction of training runs impossible. ZEROSEARCH's simulated search produces deterministic outputs given the same random seed, enabling scientific reproducibility that real search cannot provide.

The empirical validation is critical: Section 5.2 (Table 4) shows that a 7B simulator matches Google Search performance, and a 14B simulator surpasses it. This validates that simulation is not merely a cost-saving approximation—it can provide equivalent or superior training signals.

### 4.2 Counterintuitive Finding: Controlled Noise Improves Learning More Than Real Results

A surprising insight emerges from the curriculum rollout mechanism: **progressively degraded document quality produces better policy models than consistently high-quality real search results**. This finding challenges the intuitive assumption that training should always use the best available information.

Section 5.1 (Figure 2) shows that ZEROSEARCH initially lags behind Search-R1 but eventually surpasses it with less fluctuation. The explanation (Appendix A) is that real search engine quality is "uncontrollable during rollout"—early training may receive low-quality documents that prevent the policy from learning correct task understanding, while later training may receive documents that are too high-quality, failing to challenge the policy's reasoning capability.

This is a **fundamental innovation** in how we think about training data quality for RL:

| Training Phase | Real Search Engine | ZEROSEARCH Curriculum |
|---------------|-------------------|----------------------|
| Early | Unpredictable quality | High-quality (useful) documents |
| Mid | Unpredictable quality | Mixed quality |
| Late | Unpredictable quality | Challenging (noisy) documents |

The curriculum forces the policy to develop robust reasoning that works even when search results are imperfect—a capability that matters for real-world deployment where search results are never guaranteed to be perfect. Table 6 confirms this: replacing the curriculum with random noise (fixed 0.5 probability) reduces average performance by ~2 percentage points across both Qwen-2.5-3B-Base and LLaMA-3.2-3B-Base models.

This insight has implications beyond search-augmented LLMs. Any RL task where environment quality varies (robotics with sensor noise, game playing with opponent variability) could potentially benefit from curriculum-based difficulty adjustment rather than training on fixed-distribution environments.

### 4.3 New Capability: Explicit Control Over Training Difficulty via Prompt Keywords

The ability to control document quality by "adjusting a few words in the prompt" (Section 1) represents a **new capability** that prior search-augmented RL methods simply cannot access. Table 2 shows the template: changing `[useful / noisy]` switches the simulation LLM between generating answer-relevant documents and generating irrelevant documents.

This is distinct from prior work in two ways:

**Prior methods cannot control search result quality.** Search-R1, DeepResearcher, and WebThinker all accept whatever the search engine returns. If a query returns poor results, the policy must learn from them—or the training step is wasted. There is no mechanism to say "give me easier results" or "give me harder results."

**Prior methods cannot systematically vary difficulty.** Some work uses static corpora (Wikipedia) where document quality is fixed. Others use real search where quality is random. Neither allows the deliberate progression from easy to hard that curriculum learning requires.

The significance extends beyond search training. This demonstrates that **simulated environments can expose control interfaces that real environments cannot**. In robotics, you cannot ask the physical world to "make this task 10% harder." But in a learned simulator, you can parameterize difficulty. ZEROSEARCH shows this principle applies to information retrieval—a domain where the "environment" was previously assumed to be fixed (the actual web).

Section 5.3 (Figure 2c) illustrates the training dynamics: interaction turns drop sharply early (learning format), then both turns and reward increase together (learning to search effectively), then stabilize despite increasing curriculum difficulty (maintaining performance under harder conditions). This trajectory would be impossible to achieve without explicit difficulty control.

### 4.4 Framework Robustness: Compatibility Across Model Types and RL Algorithms

While less conceptually novel than the above innovations, ZEROSEARCH's **broad compatibility** represents a significant practical contribution. Section 1 states the framework works with "both base and instruction-tuned models of various parameter sizes" and integrates with "REINFORCE, PPO, and GRPO."

Table 3 demonstrates this across three model families (Qwen-2.5-3B, Qwen-2.5-7B, LLaMA-3.2-3B) and both base and instruct variants. Table 5 shows consistent performance across three RL algorithms, with REINFORCE achieving the best results (34.47 average) followed by GRPO (33.17) and PPO (32.67).

This compatibility matters for two reasons:

**First, it removes warm-up requirements.** Appendix B (Figure 4a-b) shows that base models "can effectively acquire search capabilities through reinforcement learning without the need for supervised fine-tuning as a warm-up." Prior methods often require SFT before RL to teach basic format compliance. ZEROSEARCH's template design and reward structure eliminate this prerequisite, simplifying the training pipeline.

**Second, it future-proofs the approach.** RL algorithm research moves quickly—today's best algorithm may be tomorrow's baseline. By designing ZEROSEARCH to be algorithm-agnostic (the simulator is separate from the policy optimization), the framework remains usable as new RL methods emerge. This contrasts with methods tightly coupled to specific algorithms.

The loss masking mechanism (Section 3.6, Figure 4c, Table 7) is essential to this compatibility. Without masking document tokens, different RL algorithms would handle the mixed policy/simulator output differently, creating algorithm-specific instability. Masking standardizes the gradient computation across all algorithms.

### 4.5 Distinguishing Incremental from Fundamental Contributions

To clarify the novelty landscape:

| Contribution | Type | Why |
|-------------|------|-----|
| LLM as search simulator | **Fundamental** | First work to replace real search with learned simulation during RL training |
| Controllable document quality via prompts | **Fundamental** | Enables curriculum learning impossible with real search |
| Curriculum noise schedule | **Fundamental** | First application of progressive difficulty to search-augmented RL |
| F1-based reward (vs. exact match) | Incremental | Addresses known reward hacking; similar fixes exist in other RL work |
| Loss masking for document tokens | Incremental | Standard technique adapted to this setting; important but not novel |
| Multi-algorithm compatibility | **Significant** | Not conceptually novel, but practically important for adoption |
| Better-than-real performance | **Fundamental finding** | Counterintuitive empirical result that validates the simulation approach |

The fundamental innovations (simulation, controllability, curriculum) work together: simulation enables controllability, which enables curriculum, which produces better-than-real performance. This interdependence means the contributions reinforce each other rather than standing alone.

### 4.6 Implications Beyond This Paper

These innovations suggest broader directions for RL research:

1. **Learned simulators for other tools**: If LLMs can simulate search engines, can they simulate code executors, calculators, or database queries? This could reduce API costs across all tool-augmented LLM training.

2. **Curriculum design as a research direction**: ZEROSEARCH uses a simple exponential schedule. More sophisticated curricula (adapting to policy performance, varying noise types) could further improve training.

3. **Simulation-to-reality transfer**: The paper shows training on simulation transfers to real search at inference. Understanding when and why this transfer succeeds could inform other domains where simulation is cheaper than reality.

These implications position ZEROSEARCH not just as a method for search-augmented LLMs, but as a case study in how learned simulators can enable scalable RL training for tool-using agents.

## 5. Experimental Analysis

This section examines how ZEROSEARCH was evaluated, what the results demonstrate, and whether the evidence convincingly supports the paper's central claims. We walk through the evaluation methodology, dissect the quantitative results with specific numbers, analyze ablation studies, and identify conditions where performance varies.

### 5.1 Evaluation Methodology

#### Datasets and Task Coverage

The evaluation spans **seven question answering benchmarks** designed to test both single-hop and multi-hop reasoning capabilities (Section 4.1):

| Category | Datasets | What They Test |
|----------|----------|----------------|
| Single-Hop QA | NQ, TriviaQA, PopQA | Direct fact retrieval (one lookup sufficient) |
| Multi-Hop QA | HotpotQA, 2WikiMultiHopQA, Musique, Bamboogle | Multi-step reasoning (requires chaining multiple facts) |

This dataset selection is deliberate: single-hop tasks verify that ZEROSEARCH does not degrade simple retrieval performance, while multi-hop tasks stress-test the model's ability to conduct sequential searches and synthesize information across multiple documents. The inclusion of **Bamboogle** is particularly notable—it contains questions that are specifically designed to be unanswerable by standard search engines, testing whether the model can recognize when search fails.

#### Evaluation Metric

The primary metric is **Exact Match (EM)** (Section 4.1):

> "A prediction is deemed correct if its normalized form exactly matches any of the normalized ground-truth answers."

Normalization typically involves lowercasing, removing punctuation, and stripping articles. This is a strict metric—a prediction of "London, England" would not match "London" without normalization. The paper chose EM over F1 for evaluation (though F1 is used for training rewards, Section 3.5) to enable direct comparison with prior work like Search-R1.

**Important distinction:** The reward function during RL training uses F1 score to prevent reward hacking (models producing excessively long answers), but final evaluation uses EM. This separation ensures training stability without compromising evaluation rigor.

#### Baselines

The comparison spans three categories of methods (Section 4.2):

**Vanilla Prompting Methods:**
- Direct Answer (no search, no reasoning)
- Chain-of-Thought (CoT) (reasoning without search)
- Standard RAG (single retrieval step)

**Advanced RAG Methods:**
- RA-Agent (iterative search with agent framework)
- Search-o1 (agentic search-enhanced reasoning)

**RL Tuning Methods:**
- R1-base / R1-instruct (RL for reasoning without search)
- Search-R1-base / Search-R1-inst (RL for reasoning **with real search engine**)

The critical baseline is **Search-R1**, which uses the same RL framework but interacts with a real search engine during training. This is the fairest comparison because it isolates the effect of simulation vs. real search while holding the RL algorithm constant. As Section 4.2 states:

> "Among RL-based search baselines, we compare only with Search-R1, as it avoids complex reward design, data selection, or elaborate training pipelines. This setting allows for a direct and equitable comparison between the real search engine and our simulated search engine."

#### Experimental Setup

The experimental configuration is carefully controlled to ensure fair comparison (Section 4.3, Appendix F):

| Component | Configuration |
|-----------|---------------|
| Policy Models | Qwen-2.5-7B (Base/Instruct), Qwen-2.5-3B (Base/Instruct), LLaMA-3.2-3B (Base/Instruct) |
| Simulation LLM (default) | Qwen-2.5-14B-Instruct (fine-tuned) |
| Search Engine (evaluation) | Google Web Search via SerpAPI (all methods) |
| Retrieved Documents | Fixed at 5 across all methods |
| RL Algorithm (default) | REINFORCE |
| Training Steps | 200 |
| Batch Size | 64 |
| Rollout Repetition | 5 responses per prompt |
| Hardware | 4 H20 GPUs (simulation) + 4 H20 GPUs (RL training) |
| Training Data | Merged NQ + HotpotQA training sets |

**Critical design choice:** All methods use Google Web Search **during inference**, even ZEROSEARCH. This ensures that performance differences reflect training quality, not inference-time advantages. ZEROSEARCH only replaces the search engine during training—not evaluation. This is essential for validating that learned search capabilities transfer to real-world search engines.

### 5.4 Training Dynamics Analysis

#### Reward Curves (Figures 2, 3)

Figures 2a-b and 3a-d compare reward curves between ZEROSEARCH and Search-R1 across different model configurations. Key patterns:

1. **Initial lag, eventual surpass:** ZEROSEARCH typically starts below Search-R1 but overtakes it by step 100-150. For example, Figure 2b (Qwen-2.5-3B-Inst) shows ZEROSEARCH at ~0.25 reward at step 50 vs Search-R1 at ~0.30, but ZEROSEARCH reaches ~0.50 by step 200 while Search-R1 plateaus at ~0.45.

2. **Smoother curves:** ZEROSEARCH exhibits less fluctuation than Search-R1. Appendix A explains: "the quality of documents returned by the real search engine is uncontrollable during rollout," causing variance in training signals. ZEROSEARCH's curriculum provides consistent difficulty progression.

3. **Convergence:** Both methods converge to similar reward ranges (~0.4-0.5), but ZEROSEARCH reaches this with lower variance, indicating more stable training.

#### Interaction Turns (Figure 2c)

Figure 2c tracks the number of search interactions during training for LLaMA-3.2-3B-Base:

- **Early phase (steps 0-50):** Interaction turns drop sharply from ~2.2 to ~1.6. The policy learns to eliminate redundant searches.
- **Mid phase (steps 50-150):** Both turns and reward increase together. The policy learns to search more effectively when needed.
- **Late phase (steps 150-200):** Turns stabilize while reward remains high despite increasing curriculum difficulty.

This trajectory confirms that the policy is not simply learning to search more—it is learning to search **strategically**. If the policy were just increasing search frequency, turns would continue rising. Instead, turns stabilize while performance improves, indicating better search quality per query.

### 5.6 Critical Assessment of Claims

#### Claim 1: "ZEROSEARCH outperforms real search engine-based models"

**Supported.** Table 3 shows ZEROSEARCH-base exceeds Search-R1-base on average across all three model families (+1.42 to +1.86 points). Table 4 shows SFT-14B (34.47) exceeds Google Search (32.81) by +1.66 points.

**Condition:** The advantage is most pronounced on multi-hop datasets (HotpotQA, Musique). On single-hop tasks like NQ, the gap is smaller (e.g., Qwen-7B: 42.40 vs 43.40, Search-R1 actually wins by 1 point). This suggests the curriculum particularly benefits complex reasoning scenarios.

#### Claim 2: "7B simulator achieves comparable performance to real search engine"

**Supported.** Table 4 shows SFT-7B (33.53) vs Google (32.81), a difference of +0.72 points—within typical experimental variance.

**Condition:** This holds for the Qwen-2.5-3B-Base policy model. The paper does not report 7B simulator results with 7B or 14B policy models, so generalizability across policy sizes is not fully established.

#### Claim 3: "Generalizes well across base and instruction-tuned models"

**Supported.** Table 3 shows ZEROSEARCH works with both base and instruct variants across all three model families. Appendix B (Figure 4a-b) shows both model types achieve similar final rewards.

**Nuance:** Base models show larger improvement from RL (starting lower, ending comparable), while instruct models start higher but improve less. This suggests instruct models benefit from pre-existing format compliance, while base models learn format through RL.

#### Claim 4: "Compatible with different RL algorithms"

**Supported.** Table 5 shows REINFORCE, GRPO, and PPO all achieve >32 EM, with REINFORCE performing best (34.47).

**Caveat:** The absolute performance difference between algorithms (34.47 vs 32.67, ~1.8 points) is smaller than the difference between ZEROSEARCH and non-RL baselines (~15 points). Algorithm choice matters less than the simulation framework itself.

#### Claim 5: "Zero API cost"

**Supported.** Table 8 confirms $0 API cost for ZEROSEARCH vs $586.7 for Google Search.

**Caveat:** GPU costs ($17.7-$70.8) are not zero. The claim should be understood as "zero external API cost" rather than "zero infrastructure cost." For researchers without GPU access, this may still be a barrier.

### 5.7 Limitations and Open Questions

#### Unanswered Questions

1. **Simulator overfitting:** Does training on a 14B simulator cause the policy to overfit to that simulator's output distribution? The paper shows transfer to real search at inference, but does not measure degradation if the simulator differs significantly from real search.

2. **Curriculum hyperparameters:** The noise schedule parameters (ps, pe, b) are tuned per model size (Appendix F), but the paper does not explore how sensitive performance is to these choices. A sensitivity analysis would strengthen the curriculum claim.

3. **Domain generalization:** All training uses NQ + HotpotQA. While evaluation includes out-of-domain datasets (TriviaQA, PopQA, etc.), the paper does not test entirely different domains (e.g., scientific QA, legal QA).

4. **Long-horizon search:** The maximum interaction turns observed is ~2.2 (Figure 2c). It is unclear whether ZEROSEARCH scales to tasks requiring 5+ search iterations.

#### Potential Failure Modes

1. **Simulator capacity bottleneck:** If the policy model exceeds the simulator's knowledge (e.g., 72B policy with 14B simulator), the simulator may not generate sufficiently challenging documents. The paper does not test this regime.

2. **Noisy document quality:** The "noisy" documents are generated by prompting the simulator to produce irrelevant content. These may not match the distribution of real-world noisy search results (e.g., SEO spam, outdated pages).

3. **Multi-lingual support:** All experiments use English datasets. The simulator's ability to generate useful/noisy documents in other languages is untested.

The primary limitation is the lack of exploration beyond English QA tasks and the absence of sensitivity analysis for curriculum hyperparameters.

## 6. Limitations and Trade-offs

While ZEROSEARCH demonstrates compelling advantages over real search engine-based training, the approach rests on several assumptions and introduces trade-offs that warrant careful consideration. This section examines the constraints, unaddressed scenarios, and open questions that define the boundaries of the method's applicability.

### 6.1 Infrastructure Assumptions and Hidden Costs

The paper's claim of "zero API cost" (Section 1) requires qualification. While ZEROSEARCH eliminates external API expenses, it **shifts costs from API fees to GPU infrastructure**. Appendix D (Table 8) quantifies this trade-off:

| Simulator Size | GPU Cost | API Cost | Total |
|----------------|----------|----------|-------|
| 3B | $17.7 | $0.0 | $17.7 |
| 7B | $35.4 | $0.0 | $35.4 |
| 14B | $70.8 | $0.0 | $70.8 |
| Google Search | $0.0 | $586.7 | $586.7 |

The 8.3× cost reduction is substantial, but the assumption that researchers have **access to GPU servers** is non-trivial. For academic labs or individual researchers without cloud GPU credits, the $17.70–$70.80 per training run may still represent a barrier—just a different one than API costs.

**GPU Utilization Inefficiency:** Appendix D explicitly acknowledges a significant operational limitation:

> "GPU utilization of the simulation server is inconsistent—high during the rollout and low during policy update—resulting in periods of underutilization and resource waste."

This inefficiency arises from the two-phase training cycle: the simulation server generates documents during rollout (GPU-intensive), then idles while the policy model performs gradient updates (compute on separate GPUs). The paper suggests sharing "a single simulation server across multiple RL training tasks" to mitigate this, but this solution **adds orchestration complexity** that is not detailed. Researchers implementing ZEROSEARCH must build or adopt infrastructure to manage this resource sharing, which introduces engineering overhead not present in simple API-based approaches.

### 6.2 Simulator Capacity Constraints

A fundamental assumption underlying ZEROSEARCH is that the simulation LLM possesses **sufficient knowledge to generate appropriate documents** for the policy model's queries. This creates a potential capacity mismatch:

**Policy-Simulator Size Mismatch:** The paper tests policy models up to 7B (Qwen-2.5-7B) with a 14B simulator. However, it does not address what happens when the **policy model exceeds the simulator's knowledge capacity**. For example:
- A 72B policy model trained with a 14B simulator may issue queries about topics the simulator cannot adequately simulate.
- The simulator's 14B parameter limit constrains the **ceiling of training signal quality**—the policy cannot learn to handle search scenarios that the simulator cannot generate.

This limitation is particularly relevant for organizations training large-scale models. If the goal is to train a 70B+ policy model, the simulator must be comparably large to avoid becoming a bottleneck. The paper does not report experiments with simulators larger than 14B, leaving this scalability question unanswered.

**Knowledge Boundary:** The simulation LLM's knowledge is frozen at its pretraining cutoff, just like the policy model. Section 1 acknowledges that LLMs are "prone to generating hallucinated content or outdated information." When the simulator generates documents, it may **hallucinate plausible-sounding but incorrect information** even in "useful" mode. The paper does not measure the rate of simulator hallucination or its impact on policy learning. In contrast, real search engines (while noisy) at least return documents that exist in the real world.

### 6.3 Curriculum Hyperparameter Sensitivity

The curriculum rollout mechanism (Section 3.4) is central to ZEROSEARCH's success, but its effectiveness depends on carefully tuned hyperparameters. Appendix F reveals different noise schedules for different model sizes:

| Model | Initial Noise (`ps`) | Final Noise (`pe`) |
|-------|---------------------|-------------------|
| Qwen-2.5-7B | 0 | 0.75 |
| Qwen-2.5-3B | 0 | 0.25 |
| LLaMA-3.2-3B | 0.25 | 0.5 |

**No Sensitivity Analysis:** The paper does not provide a sensitivity analysis showing how performance varies with different `ps`, `pe`, or exponential base `b` values. This leaves practitioners without guidance on:
- How to set curriculum parameters for unseen model sizes (e.g., 13B, 33B).
- How robust the method is to suboptimal curriculum settings.
- Whether the exponential schedule (Equation 1) is optimal compared to linear or adaptive schedules.

Table 6 shows that curriculum outperforms random noise by +1.23 to +1.88 points, but this comparison uses a **fixed 0.5 noise probability** for the random baseline. A more informative ablation would test different fixed noise levels (0.2, 0.5, 0.8) to establish whether the curriculum's advantage comes from progressive difficulty or simply from avoiding a poorly-chosen fixed value.

**Domain-Specific Curricula:** The curriculum parameters are tuned on merged NQ + HotpotQA training data. It is unclear whether these same parameters generalize to other domains (e.g., scientific QA, legal reasoning, medical questions) without re-tuning. The paper claims out-of-domain generalization on evaluation datasets (Section 4.4), but all evaluation tasks are still **English question answering**—a relatively narrow domain.

### 6.4 Noisy Document Distribution Mismatch

The "noisy" documents generated by the simulator may not accurately reflect the **distribution of real-world noisy search results**. Section 3.3 describes how noisy documents are labeled: the LLM judges whether each document "contains sufficient information to answer the corresponding query." If not, it is labeled noisy.

**Nature of Noise:** Real search engine noise takes specific forms:
- SEO-optimized content with keyword stuffing
- Outdated information (e.g., news from 5 years ago)
- Partially relevant documents (correct topic, wrong specifics)
- Spam or low-quality sources

The simulator's noisy documents are generated by prompting with the keyword "noisy" (Table 2). While Table 10 shows examples where noisy outputs lack correct answers, the paper does not analyze whether the **style and structure** of simulated noise matches real search noise. If the simulator generates uniformly random irrelevant text, but real search returns plausibly relevant but incorrect information, the policy may learn to handle the wrong type of noise.

**Evidence from Table 10:** The noisy output examples show documents that are topically related but lack specific answers (e.g., discussing Tour de France prize money instead of second place finisher). This suggests the simulator does generate realistic noise. However, without quantitative analysis of noise characteristics (e.g., lexical overlap with real search results, semantic similarity distributions), this remains an assumption.

### 6.5 Limited Search Depth and Complexity

The interaction turn analysis (Figure 2c, Section 5.3) reveals a constraint on search complexity:

> "During the early phase of training, the number of interaction turns drops sharply... As training progresses, both the number of interaction turns and the reward curve increase sharply and then stabilize."

The maximum observed interaction turns is approximately **2.2** (Figure 2c). This raises questions about scalability to more complex tasks:

**Multi-Hop Limitations:** While the paper evaluates on multi-hop datasets (HotpotQA, Musique, 2WikiMultiHopQA), these typically require 2-3 reasoning steps. Tasks requiring **5+ sequential searches** (e.g., complex research queries, investigative journalism workflows) are not tested. The curriculum may not adequately prepare the policy for such long-horizon search behavior.

**Search Strategy Learning:** The paper does not analyze whether the policy learns sophisticated search strategies (e.g., query refinement, source evaluation, cross-referencing) or simply learns to issue more queries until finding an answer. The case studies in Table 9 show multi-turn search, but the queries are straightforward refinements (e.g., "place of birth of Clara Novello's father" → "place of birth of Vincent Novello"). More complex strategies like **query decomposition** or **contradiction resolution** are not demonstrated.

### 6.6 Algorithm Performance Variance

Table 5 (Section 5.4) shows notable performance differences across RL algorithms:

| Algorithm | Average EM |
|-----------|------------|
| REINFORCE | 34.47 |
| GRPO | 33.17 |
| PPO | 32.67 |

The **1.80-point gap** between REINFORCE and PPO suggests that algorithm choice matters. While the paper frames multi-algorithm compatibility as a strength, this variance implies:
- Practitioners cannot arbitrarily choose an RL algorithm without performance implications.
- REINFORCE's advantage may not hold for all policy model sizes or domains.
- The loss masking mechanism (Section 3.6) may interact differently with each algorithm's gradient estimation.

The paper does not explain **why REINFORCE outperforms PPO** in this setting. PPO typically offers better sample efficiency due to its value function, but the mixed policy/simulator token distribution may create instability that PPO's clipping mechanism cannot fully address. This is an open question for future work.

### 6.7 Base vs. Instruct Model Trade-offs

Appendix B (Figure 4a-b) reveals a nuanced trade-off between base and instruction-tuned models:

- **Instruct models** achieve higher initial rewards due to better instruction-following capabilities.
- **Base models** show greater improvement from RL and reach comparable final performance.

This creates a practical decision point:
- Use **instruct models** for faster convergence and simpler training (less RL steps needed).
- Use **base models** for maximum final performance, accepting longer training time.

The paper states ZEROSEARCH "removes the need for separate supervised warm-up stages" (Section 1), which is true for base models. However, instruct models still benefit from their pre-existing instruction-following capability, suggesting that **SFT is not fully replaced**—it is merely shifted from explicit warm-up to the instruction-tuning phase that already occurred.

### 6.8 Single-Hop vs. Multi-Hop Performance Disparity

Table 3 reveals that ZEROSEARCH's advantage over Search-R1 is **not uniform across task types**:

| Dataset | Task Type | ZEROSEARCH-base | Search-R1-base | Difference |
|---------|-----------|-----------------|----------------|------------|
| NQ | Single-Hop | 42.40 | 43.40 | **-1.00** |
| PopQA | Single-Hop | 60.40 | 54.60 | **+5.80** |
| HotpotQA | Multi-Hop | 32.00 | 31.20 | **+0.80** |
| Bamboogle | Multi-Hop | 33.33 | 30.56 | **+2.77** |

On NQ (a single-hop dataset), Search-R1 **outperforms** ZEROSEARCH by 1 point. This suggests that for straightforward fact-retrieval tasks, real search engines may still provide superior training signals. The curriculum's progressive difficulty may be **overkill for simple tasks**, where consistent high-quality documents (as real search sometimes provides) are more beneficial than gradually increasing noise.

This disparity implies that ZEROSEARCH is best suited for **complex reasoning tasks** where robust search behavior under noisy conditions matters. For simple QA, the added complexity of simulation may not be justified.

### 6.9 Unaddressed Domains and Languages

The evaluation is constrained to **English question answering** across seven datasets (Section 4.1). Several important domains are not tested:

- **Multi-lingual QA:** The simulator's ability to generate useful/noisy documents in languages other than English is untested. Given that LLMs often have weaker performance in low-resource languages, the simulation quality may degrade significantly.
- **Domain-Specific QA:** Scientific, legal, medical, and technical domains have different search patterns and document structures. The simulator trained on NQ + HotpotQA may not generalize to these domains without additional fine-tuning.
- **Open-Ended Tasks:** All evaluation tasks have ground-truth answers. Open-ended generation tasks (e.g., writing reports, synthesizing research) where answer quality is subjective are not addressed.

### 6.10 Simulator Overfitting Risk

A subtle but important limitation is the risk of **policy overfitting to the simulator's output distribution**. While the paper demonstrates transfer to real search at inference (all methods use Google Web Search during evaluation, Section 4.3), it does not measure:
- Performance degradation when the simulator differs significantly from real search.
- Whether the policy learns simulator-specific artifacts (e.g., document length patterns, phrasing styles).
- How robust the policy is to simulator updates (e.g., if the 14B simulator is replaced with a newer version).

The reward curves in Figures 2-3 show stable training, but stability does not guarantee generalization. A policy could learn to exploit quirks of the simulator's document generation while still achieving high rewards, only to fail when encountering real search results with different characteristics.

### 6.11 Summary of Trade-offs

| Trade-off | Benefit | Cost |
|-----------|---------|------|
| Simulation vs. Real Search | 8.3× cost reduction, controllable curriculum | GPU infrastructure required, potential simulator bias |
| Curriculum Learning | Better multi-hop reasoning (+1.88 pts, Table 6) | Hyperparameter tuning per model size |
| Base vs. Instruct Models | No SFT warm-up needed for base | Instruct models converge faster |
| REINFORCE vs. PPO/GRPO | Best performance (34.47 EM, Table 5) | May not generalize to all settings |
| 14B vs. 7B Simulator | Surpasses real search (+1.66 pts, Table 4) | 2× GPU cost ($70.8 vs $35.4) |

These trade-offs do not invalidate ZEROSEARCH's contributions—they define the **operating envelope** within which the method excels. For researchers with GPU access working on complex multi-hop QA tasks, ZEROSEARCH offers a compelling alternative to real search-based training. For simple single-hop tasks, limited GPU resources, or non-English domains, the benefits may be less pronounced.

## 7. Implications and Future Directions

ZEROSEARCH represents a paradigm shift in how we approach tool-augmented reinforcement learning (RL). By decoupling training from real-world infrastructure, it changes the economic and technical constraints of developing search-capable models. This section explores how this work reshapes the research landscape, identifies concrete follow-up directions, outlines practical applications, and provides guidance for integrating this method into existing workflows.

### 7.1 Shifting the Research Landscape

The most immediate impact of ZEROSEARCH is the removal of the **cost barrier** that has historically limited search-augmented RL research. Prior methods like Search-R1 or DeepResearcher require live API interactions during training, incurring approximately **$586.7 per training run** for a modest 200-step schedule (Table 8). This cost scales linearly with batch size and steps, effectively restricting this research to well-funded organizations. ZEROSEARCH reduces this cost to **$17.70–$70.80** in GPU expenses (Table 8), an **8.3× reduction**. This democratization allows academic labs and independent researchers to experiment with search-augmented RL without prohibitive budgets.

Beyond cost, ZEROSEARCH introduces **deterministic reproducibility** to a domain previously characterized by stochasticity. Real search engine results vary over time—the same query issued today versus next month may return different documents due to index updates or ranking changes. This variability makes exact reproduction of training runs impossible. In contrast, the simulation LLM produces deterministic outputs given the same random seed. This stability enables rigorous scientific comparison of hyperparameters and algorithms, a foundational requirement for cumulative research progress.

Finally, the work challenges the assumption that **real-world interaction is necessary for robust tool use**. Section 5.2 (Table 4) demonstrates that a 14B simulator can train a policy model that outperforms one trained on real search results (+1.66 percentage points on average). This suggests that for RL training, **controlled curriculum quality** (Section 3.4) may be more valuable than **environmental fidelity**. This insight extends beyond search to any domain where real-world interaction is expensive or dangerous (e.g., robotics, healthcare), suggesting that learned simulators could substitute for reality during the training phase.

### 7.2 Enabled Research Directions

ZEROSEARCH opens several concrete avenues for follow-up research. The paper establishes the feasibility of simulation-based training, but many optimization questions remain unanswered.

**1. Adaptive Curriculum Schedules**
The current curriculum uses a fixed exponential noise schedule defined by Equation 1 in Section 3.4:
$$p_i = p_s + \frac{i/m - 1}{b - 1}(p_e - p_s)$$
where `ps` and `pe` are fixed hyperparameters tuned per model size (Appendix F). Future work could explore **adaptive curricula** that adjust noise based on the policy's current performance rather than training steps. For instance, if the policy's reward plateaus, the system could automatically reduce noise to stabilize learning. This would eliminate the need for manual hyperparameter tuning across different model sizes.

**2. Simulating Other Tools**
The paper focuses on search engines, but the framework is tool-agnostic. The simulation LLM could be fine-tuned to mimic:
- **Code Executors:** Generating execution outputs (including errors) for code generation tasks.
- **Calculators:** Producing numerical results with varying degrees of precision noise.
- **Database Queries:** Returning SQL query results with realistic schema constraints.
This would extend the cost-saving benefits of ZEROSEARCH to the broader field of **agentic RL**, where models learn to use multiple tools simultaneously.

**3. Simulator Scaling and Distillation**
Section 5.2 shows performance improves with simulator size (3B → 7B → 14B). A key open question is whether **distilling a larger simulator into a smaller one** could retain performance while reducing GPU costs. For example, could a 14B simulator train a 3B simulator, which then trains the policy? This would create a cascading training pipeline that minimizes infrastructure requirements while maintaining curriculum quality.

**4. Multi-Lingual and Domain-Specific Simulation**
All experiments in ZEROSEARCH use English question answering datasets (Section 4.1). The simulator's ability to generate useful/noisy documents in low-resource languages or specialized domains (e.g., legal, medical) is untested. Future work should evaluate whether the lightweight SFT procedure (Section 3.3) generalizes across languages or requires domain-specific trajectory collection. Given that LLMs often hallucinate more in low-resource languages, the simulator's reliability may vary significantly.

**5. Safety and Alignment via Simulation**
Because the simulator controls document quality, it can be used to **filter harmful content during training**. Real search engines may return toxic or misleading information that the policy model could learn to trust. ZEROSEARCH allows researchers to label such documents as "noisy" or exclude them entirely during the simulation phase. This creates a safer training environment without relying on post-hoc filtering of real search results.

### 7.3 Practical Applications and Downstream Use Cases

ZEROSEARCH is not merely a research tool; it has direct applicability in industrial and enterprise settings.

**Enterprise Search with Private Data**
In enterprise environments, employees often need to search internal knowledge bases (e.g., documentation, tickets) that are not accessible via public search engines. Building a real search API for RL training may be impractical. ZEROSEARCH allows organizations to fine-tune a simulator on their **internal document corpus** and use it to train policy models offline. The trained policy can then interact with the real internal search system at inference time. This enables custom search-augmented models without exposing sensitive data to external APIs during training.

**Cost-Constrained Model Development**
For startups or teams with limited budgets, ZEROSEARCH provides a path to develop competitive search-augmented models. The **$70.80 training cost** (14B simulator, Table 8) is accessible compared to the **$586.70 API cost** of real search. This is particularly relevant for iterating on model architectures—if a team needs to run 10 ablation studies, the savings compound to nearly **$5,000**.

**Robust Reasoning for Noisy Environments**
Applications that operate in environments with unreliable information retrieval (e.g., legacy systems, fragmented databases) can benefit from ZEROSEARCH's curriculum training. By exposing the policy to progressively noisier documents during training (Section 3.4), the model learns to reason even when search results are imperfect. This robustness is critical for real-world deployment where search quality is never guaranteed.

### 7.4 Reproduction and Integration Guidance

For practitioners looking to adopt ZEROSEARCH, the following guidance synthesizes the paper's findings into actionable recommendations.

**When to Use ZEROSEARCH:**
- **Multi-Hop Reasoning Tasks:** The curriculum benefit is most pronounced on datasets like HotpotQA and Musique (Table 3), where robust search behavior under noisy conditions is critical.
- **Limited API Budget:** If your project cannot sustain hundreds of dollars in API costs per training run, ZEROSEARCH is the preferred alternative.
- **Need for Reproducibility:** If you require exact reproduction of training runs for scientific or regulatory purposes, the deterministic simulator is essential.
- **Base Model Training:** If you prefer using base models without instruction-tuning warm-up, ZEROSEARCH supports this directly (Appendix B), simplifying the pipeline.

**When to Avoid ZEROSEARCH:**
- **Simple Single-Hop Tasks:** For straightforward fact retrieval (e.g., NQ), real search engines may still provide superior training signals (Table 3 shows Search-R1 beating ZEROSEARCH on NQ by 1 point). The curriculum complexity may be unnecessary.
- **No GPU Access:** While API costs are eliminated, GPU infrastructure is required. If you lack access to GPUs (e.g., 4× H20 or A100 equivalents), the infrastructure barrier may outweigh the API savings.
- **Non-English Domains:** The method is untested outside English QA. Until multi-lingual performance is validated, caution is advised for other languages.

**Key Configuration Recommendations:**
- **RL Algorithm:** Use **REINFORCE** as the default. It achieved the best performance (34.47 avg EM) and stability in experiments (Table 5). PPO and GRPO are compatible but showed slightly lower performance.
- **Simulator Size:** A **7B simulator** is the sweet spot for cost-performance balance, matching real search performance (Table 4). Use **14B** only if maximum performance is required and GPU budget allows.
- **Noise Schedule:** Follow the hyperparameters in Appendix F. For 7B models, set initial noise `ps = 0` and final noise `pe = 0.75`. For 3B models, use `ps = 0` and `pe = 0.25`. Smaller models need gentler curricula.
- **Loss Masking:** **Always enable document token loss masking** (Section 3.6). Disabling it reduces performance by ~1.5 points (Table 7) and causes training instability (Figure 4c).
- **Inference Deployment:** Remember that ZEROSEARCH only replaces the search engine **during training**. At inference, deploy the policy model with a **real search engine** (e.g., Google Web Search) to ensure access to up-to-date information (Section 4.3).

**Integration Checklist:**
1.  **Collect Trajectories:** Interact with a real search engine to collect query-document pairs for simulator SFT (Section 3.3).
2.  **Fine-Tune Simulator:** Apply lightweight SFT to enable useful/noisy document generation (Table 2).
3.  **Configure Rollout:** Set up the curriculum noise schedule based on your policy model size (Appendix F).
4.  **Enable Masking:** Ensure your RL implementation masks loss on simulator-generated tokens.
5.  **Validate Transfer:** Evaluate the trained policy on real search engines to confirm simulation-to-reality transfer.

By following these guidelines, practitioners can leverage ZEROSEARCH to train robust search-augmented models efficiently, balancing cost, performance, and infrastructure constraints.
