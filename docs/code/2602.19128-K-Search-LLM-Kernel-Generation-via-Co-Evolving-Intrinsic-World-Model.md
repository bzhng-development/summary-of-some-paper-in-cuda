## 1. Executive Summary

This paper presents K-Search, a framework that addresses the challenge of automated GPU kernel optimization by replacing static search heuristics with a co-evolving world model instantiated from Large Language Models (LLMs). Unlike prior evolutionary approaches such as OpenEvolve and ShinkaEvolve that treat LLMs as stochastic code generators, K-Search decouples high-level algorithmic planning from low-level program instantiation to navigate non-monotonic optimization paths. The method achieves an average 2.10× improvement over OpenEvolve on diverse FlashInfer kernels (including GQA, MLA, and MoE), with gains up to 14.3× on complex MoE kernels, and delivers state-of-the-art performance on the GPUMode TriMul task with 1030 µs latency on NVIDIA H100 GPUs.

## 2. Context and Motivation

### The Core Problem: Automated GPU Kernel Optimization Under Budget Constraints

This paper addresses a fundamental bottleneck in modern machine learning infrastructure: **automatically generating high-performance GPU kernels** within a limited evaluation budget. The problem is formally defined in Section 3.1 as finding an optimal program $x^\star = \arg\max_{x \in \mathcal{X}} J(x)$ using a fixed budget of $B$ evaluations, where $J(x)$ measures speedup relative to a state-of-the-art baseline.

Three interconnected challenges make this problem exceptionally difficult:

1. **Vast Design Space**: Achieving near-peak performance requires navigating complex interactions between tiling strategies, memory layouts, synchronization patterns, and architecture-specific primitives. As noted in the Introduction, modern GPUs like NVIDIA Hopper and Blackwell introduce new instructions that "fundamentally alter performance trade-offs, rendering previously optimized kernels sub-optimal."

2. **Non-Monotonic Optimization Paths**: High-performance kernels often require coordinated, multi-step structural transformations. For example, refactoring memory layout before applying vectorization may temporarily degrade performance before yielding gains. The paper explicitly states that "intermediate steps may not yield immediate performance gains," which creates a critical challenge for search algorithms.

3. **Expensive Evaluation**: Compiling and profiling generated kernels is computationally expensive, "normally mandating strictly limited testing budgets" (Introduction). This constraint makes every evaluation count and penalizes approaches that waste budget on invalid or low-quality candidates.

### Why This Problem Matters: Real-World Impact

The stakes are substantial. High-performance GPU kernels are "fundamental to modern machine learning systems for training and serving large models" (Introduction). The paper cites several widely-deployed optimized libraries that depend on expert-tuned kernels:

- **FlashInfer**: Enables high-throughput LLM serving with paged KV-cache and dynamic batching
- **FlashAttention**: Provides memory-efficient attention mechanisms
- **FlashLinearAttention**: Supports emerging model architecture variants like Mamba and RWKV

As model architectures diversify and introduce new attention mechanisms, "manually developing and maintaining specialized kernels becomes increasingly costly" (Section 2, Related Work). Automated methods that can "rapidly adapt to new workloads and hardware" become essential for sustaining ML infrastructure development.

### Prior Approaches and Their Limitations

The paper identifies two major categories of prior work, each with distinct shortcomings:

#### Compiler-Based Autotuning Systems

Traditional approaches like **TVM/Ansor** (Chen et al., 2018; Zheng et al., 2020) optimize tensor programs using learned cost models to search large scheduling spaces. Domain-specific frameworks like **Triton** (Tillet et al., 2019) and **CuTe** (NVIDIA, 2023) enable higher-level expression of GPU kernels. While powerful, these systems "illustrate the immense complexity of GPU kernel optimization" and still require significant expertise to coordinate tiling, memory layouts, and specialized instructions (Section 2).

#### LLM-Based Evolutionary Search

Recent work couples LLMs with evolutionary strategies:

- **OpenEvolve** (Superintelligence, 2025): Implements archive-based evolution with quality-diversity mechanisms like MAP-Elites
- **ShinkaEvolve** (Lange et al., 2025a): Combines performance-driven selection with novelty-aware rejection
- **FunSearch** (Romera-Paredes et al., 2024): Pairs LLMs with evaluators in evolutionary loops for mathematical domains

The critical limitation, as the paper emphasizes repeatedly, is that these methods "fundamentally treat the LLM merely as a stochastic code generator" (Section 2). They search directly in program space using heuristic mechanisms, which creates several failure modes:

> "By treating the LLM merely as a code generator without an explicit planning mechanism, existing evolutionary methods typically cannot plan multi-step optimization sequences in which intermediate edits fail to improve the objective, and they often prematurely discard theoretically sound strategies due to temporary compilation errors." (Introduction)

The empirical evidence supports this critique. Section 4.4 notes that "ShinkaEvolve suffers from a low yield of correct programs: in our GQA logs, the vast majority of generations receive score zero (incorrect or failed programs)." Similarly, "OpenEvolve's search struggles to escape the low-accuracy regime: mean final score stays near 3 versus K-Search's 44" on MoE kernels.

### How K-Search Positions Itself

K-Search positions itself as a **paradigm shift** from heuristic-guided search to model-based planning. The key conceptual innovation is treating the LLM not as a code generator but as an **intrinsic world model** that maintains and evolves understanding of the search space.

Table 1 in Section 3.1 formalizes this distinction through notation: instead of directly sampling programs $x_{t+1} \sim \pi_{\text{LLM}}(x \mid \{(x_k, o_k)\})$, K-Search operates on a search state $S_t$ with a frontier of actions $A(S_t)$ and priority scores $V(a \mid S_t) \in [0, 1]$.

The positioning relative to prior work can be summarized along three axes:

| Dimension | Prior Evolutionary Methods | K-Search |
|-----------|---------------------------|----------|
| **Search Space** | Direct program space | Structured search tree with actions |
| **LLM Role** | Stochastic code generator | World model for planning |
| **Error Handling** | Discard on compilation failure | Local refinement with stagnation tolerance |
| **Memory** | Archive of programs | Co-evolving search state with priority scores |

Section 3.2 explicitly contrasts the two approaches. The baseline heuristic search selects context subsets using evolutionary heuristics and generates candidates conditioned on raw program text and execution feedback. K-Search instead "structures code generation as a search process guided by an LLM repurposed as a world model with an intrinsic understanding of the design space."

This positioning draws inspiration from recent work on **LLMs as World Models** (Section 2), including RAP (Hao et al., 2023) which frames reasoning as planning, and works by Fang et al. (2025) and Gu et al. (2024) demonstrating LLMs can guide agentic planning by simulating action outcomes. K-Search extends this paradigm to the GPU kernel optimization domain, where the "world" is the space of possible kernel transformations and their performance consequences.

The paper's claim is not that LLMs lack code generation capability, but that their "latent planning capabilities" (Conclusion) are underutilized when confined to direct program generation. By decoupling "high-level algorithmic planning from low-level program instantiation," K-Search enables navigation of "complex, non-monotonic optimization paths while remaining resilient to temporary implementation defects" (Introduction).

## 3. Technical Approach

This is a **systems research paper** that proposes a new search framework for automated GPU kernel optimization. The core idea is to replace static evolutionary heuristics with a **co-evolving world model** instantiated from an LLM, which maintains an explicit search tree and learns from execution feedback to guide future optimization decisions.

### 3.1 Reader orientation (approachable technical breakdown)

K-Search is a search framework that treats an LLM as a planning engine rather than a code generator, maintaining an explicit tree of optimization hypotheses that evolves based on compilation and profiling feedback. It solves the problem of navigating complex, non-monotonic optimization paths by decoupling high-level algorithmic intent (what transformation to try) from low-level program instantiation (how to write the code), allowing the system to persist with promising strategies even when intermediate implementations fail.

### 3.2 Big-picture architecture (diagram in words)

The system consists of three interconnected components that operate in a loop:

1. **Search State Manager**: Maintains a tree-structured representation of the optimization process, partitioned into Closed nodes (visited actions with attached programs) and Open nodes (pending hypotheses with priority scores). This component tracks the frontier of available actions $A(S_t)$ and their estimated values $V(a \mid S_t)$.

2. **Local Refinement Engine**: Upon selecting an action from the frontier, this component repeatedly samples concrete CUDA implementations using the LLM as a stochastic policy $\pi_{\text{code}}$, evaluates each candidate, and continues until a stagnation condition is met (no improvement after $K$ consecutive attempts).

3. **World Model Updater**: After local refinement concludes, the LLM analyzes the execution trajectory and performs tree edit operations (Insert, Update, Prune) to modify the search state, effectively learning which optimization strategies are promising based on accumulated evidence.

Information flows as follows: The Search State Manager selects the highest-priority action → The Local Refinement Engine generates and evaluates code variants → The World Model Updater consumes the results and modifies the search tree → The cycle repeats until the evaluation budget is exhausted.

### 3.4 Detailed, sentence-based technical breakdown

#### Problem Formulation and Objective Function

K-Search formulates GPU kernel synthesis as an optimization problem under a fixed evaluation budget, which is critical because compiling and profiling kernels is computationally expensive. Table 1 in Section 3.1 defines the core notation: $x \in \mathcal{X}$ represents a kernel program implementation, $o \in \mathcal{O}$ is an observation tuple from execution, $\mathcal{E}: \mathcal{X} \rightarrow \mathcal{O}$ is the evaluator function, and $J: \mathcal{X} \rightarrow \mathbb{R}$ is the scalar objective score.

The evaluator returns a structured observation tuple for any candidate program $x$:

$$o = (s, p, m) = \mathcal{E}(x)$$

where $s \in \{0, 1\}$ indicates correctness (whether the kernel produces correct outputs), $p \in \mathbb{R}^+$ is the performance metric (specifically latency in microseconds), and $m$ contains metadata such as compiler logs and profiler output. This structured feedback is richer than the scalar scores used by prior evolutionary methods, enabling the world model to distinguish between compilation failures, correctness bugs, and genuine performance regressions.

The optimization objective $J(x)$ measures speedup relative to a reference state-of-the-art baseline with latency $p_{\text{ref}}$:

$$J(x) = s \cdot \frac{p_{\text{ref}}}{p} \cdot 100$$

This formulation ensures that incorrect programs ($s=0$) receive zero score regardless of their theoretical performance, while correct programs are scored proportionally to their speedup. The optimization goal is to identify $x^\star = \arg\max_{x \in \mathcal{X}} J(x)$ utilizing a fixed budget of $B$ evaluations, where the paper uses $B = 120$ for FlashInfer kernels and $B = 300$ for the GPUMODE TriMul task.

#### Search State Structure: Tree-Structured Hypothesis Space

The fundamental architectural innovation in K-Search is representing the search process as an explicit tree structure rather than a flat archive of programs. Figure 1 in Section 3.2 visualizes this search state $S_t$, which is partitioned into two types of nodes: **Closed nodes** (shown as blue boxes, e.g., $x_{12}$) represent visited actions where the local refinement process has concluded and the best-found programs are attached, while **Open nodes** (shown as orange dashed boxes, e.g., $u_{13}$) form the frontier $A(S_t)$ of pending actions waiting to be realized.

Each Open node encapsulates a **Proposed Optimization** tuple $(x_{\text{parent}}, \delta)$, which links a parent program to a specific natural language intent $\delta$ (for example, "Resolve bank conflicts via padding" or "fuse head"). This decoupling of intent from implementation is critical: the system can commit to exploring a high-level optimization strategy without discarding it due to transient syntax errors in the code generation phase.

Each Open node also maintains a **Priority Score** $V \in [0, 1]$, which is dynamically updated by the world model to guide action selection. Figure 1 shows an example where node $u_{11}$ has its priority score drop from 0.9 to 0.6 based on new evidence, demonstrating how the world model calibrates its beliefs during search. This scoring mechanism replaces the heuristic selection rules used by prior methods like MAP-Elites, allowing the LLM's domain knowledge to directly influence which branches of the search tree receive attention.

The search state $S_t$ encapsulates the current snapshot of the model's understanding of the search process, including the history of explored actions and their performance, the frontier $A(S_t)$ of available actions, and the estimated priority scores $V$ over those frontier actions. Crucially, this model is **co-evolving** with the search process: it continuously refines its understanding and beliefs based on past trials and accumulated experience via in-context learning, rather than relying on static heuristics.

#### Three-Phase Workflow: Action Selection, Instantiation, and Update

Algorithm 1 in Section 3.3 presents the complete K-Search workflow, which iterates through three phases until the evaluation budget $B$ is exhausted. I will explain each phase in execution order.

**Phase 1: Action Selection** retrieves the most promising action from the search frontier based on the world model's priority estimates. Formally, the system selects:

$$a_t = \arg\max_{a \in A(S_t)} V(a \mid S_t)$$

Here, an action $a_t = (x_{\text{parent}}, \delta)$ represents a specific intent $\delta$ applied to a specific parent program $x_{\text{parent}}$. The priority score $V$ is a scalar value predicted by the world model when the action is created or updated, representing the model's intrinsic assessment of the action's potential. This selection mechanism ensures that search resources concentrate on the most promising optimization directions according to the world model's current understanding.

**Phase 2: Program Instantiation** synthesizes a concrete program by applying the selected plan to the parent implementation found in $S_t$. The system employs the LLM as a stochastic policy $\pi_{\text{code}}$ and repeatedly samples implementations:

$$x \sim \pi_{\text{code}}(\cdot \mid a_t)$$

$$o = \mathcal{E}(x)$$

This sampling continues until a stagnation condition is met, specifically $K$ consecutive attempts without improvement. For FlashInfer kernels, the paper sets $K = 7$, while for the GPUMODE TriMul task (which uses Triton rather than CUDA), $K = 5$ because "Triton's implementation is simpler than CUDA" (Section 4.7). This local refinement loop ensures that valid actions are not discarded due to transient syntax errors or minor bugs—a critical improvement over prior methods that discard candidates immediately upon compilation failure.

The algorithm maintains a stagnation counter $n$ that resets to zero whenever an improvement is found ($J(x) > J(x_{\text{best}})$) and increments on failure. This mechanism allows the system to persist with a promising optimization intent even when the first few code generation attempts fail, addressing the non-monotonic optimization path problem identified in the Introduction.

**Phase 3: World Model Update** performs the search state transition to obtain $S_{t+1}$ conditioned on the newly accumulated experience. Formally:

$$S_{t+1} \sim P_{\text{model}}(S \mid S_t, a_t; x_t, o_t)$$

This co-evolution allows the world model to update its prior assumptions and beliefs effectively, performing the state transition to progressively sharpen the search policy. The update is performed via in-context learning from past observations, meaning the LLM reasons over the execution trajectory to determine how to modify the search tree structure.

#### Tree Edit Operations: Insert, Update, and Prune

The World Model Update phase modifies the search tree through three types of **Tree Edit Operations**, as illustrated in Step 3 of Figure 1. These operations enable the search state to evolve based on accumulated evidence rather than following a fixed expansion strategy.

**Insert** proposes new child nodes extending the current state. For example, after successfully refining the `fused_multi_head` action in the MLA Paged Decode case study (Figure 2, Section 3.4), the model Inserts refinement actions such as `register_resident_rescaling` and `occupancy_tuned_chunk32` to deepen the successful branch. This operation allows the system to propose new optimization hypotheses based on what has worked so far, effectively learning which transformations compose well together.

**Update** re-evaluates the priority of existing frontier nodes based on new evidence. Figure 1 shows node $u_{11}$ dropping from $V = 0.9$ to $V = 0.6$ after execution feedback indicates lower-than-expected performance. In the MLA Paged Decode trace (Figure 2, rounds r14-r34), the model Updates the belief state by downgrading the sibling `independent_heads` action, reasoning that "the proven efficacy of head fusion renders independent processing less promising." This dynamic recalibration prevents the search from wasting budget on strategies that have been superseded by better alternatives.

**Prune** identifies and permanently removes infeasible or redundant branches, concentrating search resources on promising directions. Figure 1 shows node $u_{10}$ being pruned, and Figure 2 (round 34) shows the `independent_heads` branch being permanently Pruned as evidence accumulates. This operation is essential for managing the search budget efficiently: without pruning, the frontier would grow unbounded and dilute attention across too many hypotheses.

Notably, the paper states that "in the current version of K-Search, the world model evolution is merely performed by in-context learning of past observations" (Section 3.3). This means the LLM is not fine-tuned during search; instead, it reasons over the accumulated trajectory within its context window to determine appropriate tree edits. This design choice reduces computational overhead compared to training-based approaches while still enabling adaptive search behavior.

#### Case Study: MLA Paged Decode Search Trace

Figure 2 in Section 3.4 provides a detailed visualization of how the search state evolves across 102 rounds on the MLA Paged Decode kernel, where each round corresponds to one invocation of the evaluator $\mathcal{E}(x)$ (one compiled and benchmarked kernel, consuming one unit of the budget).

At **round 1 (r1)**, the search initializes $S_0$ with three high-level actions in the frontier: `fused_multi_head`, `split_k_decoding`, and `independent_heads`. The World Model assigns the highest value ($V = 0.9$) to `fused_multi_head`, hypothesizing that "processing shared CKV heads together will reduce global memory traffic by 16× compared to independent processing," and selects it for the first round of program instantiation. This demonstrates how the LLM's prior domain knowledge informs initial action selection without requiring any search-specific training.

Between **rounds 14-34**, following the local refinement of `fused_multi_head` (achieving score $J = 34$), the model evolves the topology of $S_t$ to reflect this feedback. It first Inserts refinement actions to deepen the successful branch, simultaneously Updates beliefs by downgrading `independent_heads` (from $V = 0.7$ to $V = 0.1$, a drop of 0.2), and by round 34, permanently Prunes the `independent_heads` branch, reallocating search resources entirely to the `register_resident` subtree. This sequence illustrates how the co-evolving world model concentrates effort on proven strategies while deprioritizing alternatives.

Between **rounds 42-102**, the World Model exhibits a structural shift in reasoning. At round 42, it deletes the initial root-level `split_k` action but re-Inserts a targeted variant, `low_overhead_split_k`, deep within the `register_resident` branch. This edit reflects a learned insight: "split-K is ineffective as an isolated baseline but highly effective as a composable optimization atop a strong fusion kernel." After `chunk32_vectorized` succeeds, the model proposes `chunk32_prescale_vectorized` to apply `sm_scale` immediately upon loading Q, which eventually yields the global optimum (marked with a star) at round 102.

This trace highlights three key efficiencies of K-Search: (1) starting with high-level intent rather than raw code avoids enumerating a massive sparse search space, (2) local refinement filters out transient coding noise without discarding valid strategies, and (3) the World Model's understanding co-evolves with the kernel's optimization progress, enabling dynamic repositioning of strategies guided by intrinsic reasoning over accumulated experience.

#### Key Hyperparameters and Configurations

The paper specifies several critical hyperparameters that control search behavior, and these values vary depending on the target workload:

- **Evaluation Budget $B$**: Set to 120 iterations for FlashInfer kernels (Section 4.1), where one iteration equals one evaluation of a single candidate kernel on the benchmark workloads. For the GPUMODE TriMul task, the budget is 300 iterations: 150 steps with GPT-5.2, followed by 150 continuation steps with Gemini-3-Pro starting from the best solution found by GPT-5.2 (Section 4.7).

- **Stagnation Limit $K$**: Set to 7 for FlashInfer CUDA tasks (Section 4.2), meaning the local refinement loop continues sampling implementations until 7 consecutive attempts fail to improve the best score. For the GPUMODE TriMul task using Triton, $K = 5$ because "Triton's implementation is simpler than CUDA" (Section 4.7).

- **Priority Score Range**: The priority score $V(a \mid S_t)$ is constrained to $[0, 1]$, representing a normalized confidence measure that the world model assigns to each pending action.

- **Model Configuration**: The paper uses `gemini-3-pro-preview` for FlashInfer experiments (Section 4.2). For GPUMODE TriMul, the search uses GPT-5.2 for the first 150 iterations, then switches to Gemini-3-Pro for the remaining 150 iterations without seed Triton programs provided.

- **Hardware and Software Stack**: Experiments are conducted on NVIDIA H100 and B200 GPUs using CUDA 12.8, FlashInfer 0.5.3, and PyTorch 2.8.0 (Section 4.1). All candidate implementations must be written in CUDA and pass functional correctness tests before receiving a non-zero score.

#### Design Choices and Rationale

The decision to **decouple high-level planning from low-level instantiation** addresses a fundamental limitation of prior evolutionary methods. As Section 3.2 explains, existing approaches "inherently couple high-level optimization intent with low-level implementation," meaning "a theoretically sound strategy may be discarded simply because of a transient syntax error in $x_{t+1}$." By separating the action (intent $\delta$) from the implementation (code $x$), K-Search can persist with promising optimization directions even when initial code generation attempts fail.

The choice to use **in-context learning rather than fine-tuning** for world model evolution reduces computational overhead while still enabling adaptive behavior. The paper explicitly notes that "the world model evolution is merely performed by in-context learning of past observations" (Section 3.3), meaning the LLM reasons over the accumulated trajectory within its context window rather than updating weights. This design allows the system to adapt to new kernel types without retraining, though it may limit the depth of learning compared to gradient-based approaches.

The **tree-structured search state** enables explicit tracking of optimization hypotheses and their relationships, which is essential for navigating non-monotonic paths. Unlike flat archives that store programs without structural relationships, the tree representation allows the world model to reason about which transformations compose well together (as seen in the `split_k` re-Insertion deep within the `register_resident` branch in Figure 2). This structure also enables targeted pruning of entire subtrees when evidence indicates they are unpromising.

The **stagnation-based local refinement** mechanism directly addresses the problem of premature discarding due to implementation defects. Section 3.3 states this ensures "that valid actions are not discarded due to transient syntax errors or minor bugs." By allowing up to $K$ failed attempts before abandoning an action, the system tolerates the stochastic nature of LLM code generation while still making progress on promising optimization directions.

## 4. Key Insights and Innovations

This section distills the fundamental conceptual advances that distinguish K-Search from prior automated kernel optimization methods. Rather than cataloging technical components already described in Section 3, I identify the deeper insights that enable K-Search's performance gains and explain why each represents a paradigm shift rather than an incremental improvement.

### 4.1 Insight 1: LLMs as Planning Engines, Not Code Generators

**The Innovation:** K-Search reframes the LLM's role from a stochastic code generator to an **intrinsic world model** that maintains explicit beliefs about the optimization landscape. This is a fundamental architectural change, not merely a prompt engineering tweak.

**Why Prior Work Failed:** As Section 2 documents, methods like OpenEvolve and ShinkaEvolve condition the LLM on raw program text and execution feedback, then sample new programs directly:

$$x_{t+1} \sim \pi_{\text{LLM}}\left(x \mid \{(x_k, o_k)\}_{(x_k, o_k) \in C_t}\right)$$

This formulation couples optimization intent with implementation, meaning "a theoretically sound strategy may be discarded simply because of a transient syntax error in $x_{t+1}$" (Section 3.2). The LLM never develops a structured understanding of *which transformations work*—it only learns which specific programs scored well.

**What K-Search Changes:** The world model maintains a search state $S_t$ with explicit priority scores $V(a \mid S_t)$ over pending actions. When execution feedback arrives, the model doesn't just archive a program—it updates its beliefs about the *action space*. Figure 2 demonstrates this: the `independent_heads` action drops from $V = 0.7$ to $V = 0.1$ not because one implementation failed, but because the model *reasons* that head fusion renders independent processing less promising.

**Significance:** This shift enables **transferable optimization knowledge**. The world model learns that "split-K is ineffective as an isolated baseline but highly effective as a composable optimization atop a strong fusion kernel" (Section 3.4, round 42). Prior methods cannot capture this compositional insight because they track programs, not transformation relationships. The 14.3× improvement on MoE kernels (Section 4.4) stems directly from this capability—OpenEvolve's mean score stays near 3 because it cannot escape the low-accuracy regime without understanding *why* certain strategies fail.

**Incremental vs. Fundamental:** This is a **fundamental innovation**. Prior work on LLM-guided search (FunSearch, AlphaEvolve) uses LLMs identically—as program samplers. K-Search is the first to externalize the LLM's reasoning into an explicit, evolving search state that persists across iterations.

### 4.2 Insight 2: Decoupling Intent from Implementation Enables Non-Monotonic Search

**The Innovation:** K-Search explicitly separates high-level optimization intent ($\delta$, e.g., "fuse head") from low-level program instantiation ($x$, the actual CUDA code). This decoupling allows the system to navigate optimization paths where intermediate steps temporarily degrade performance.

**Why This Matters for GPU Kernels:** As the Introduction states, "high-performance kernels often require coordinated, multi-step structural transformations, such as refactoring memory layout before applying vectorization, where intermediate steps may not yield immediate performance gains." A refactored memory layout might initially *slow down* a kernel due to added complexity, but enables subsequent vectorization that yields 5× speedup. Prior evolutionary methods, which select candidates based on immediate performance, cannot traverse such paths.

**What K-Search Changes:** The action selection phase chooses an intent $a_t = (x_{\text{parent}}, \delta)$ based on priority score $V$, not observed performance. The local refinement phase (Section 3.3, Phase 2) then samples implementations $x \sim \pi_{\text{code}}(\cdot \mid a_t)$ until stagnation ($K$ consecutive failures). Critically, the *action* remains in the frontier even if early implementations fail—the system distinguishes between "this optimization strategy is bad" and "the LLM hasn't written correct code for this strategy yet."

**Evidence from Results:** Section 4.4 notes that ShinkaEvolve "suffers from a low yield of correct programs: in our GQA logs, the vast majority of generations receive score zero." K-Search avoids this waste by not discarding actions due to implementation noise. The MLA Paged Decode trace (Figure 2) shows the system persisting with the `fused_multi_head` branch through rounds 1-34, progressively refining it despite potential early implementation hiccups.

**Significance:** This decoupling transforms the search from **greedy hill-climbing** to **strategic exploration**. The system can commit to deep structural optimizations that require multiple refinement steps, which is essential for achieving state-of-the-art performance on complex kernels like MoE and MLA.

**Incremental vs. Fundamental:** This is a **fundamental innovation**. Prior LLM-based kernel optimizers (EvoEngineer, KernelEvolve) use iterative refinement but still select candidates based on immediate execution scores. K-Search is the first to maintain a persistent commitment to optimization intents independent of transient implementation quality.

### 4.3 Insight 3: Co-Evolving Search Topology Through Evidence-Based Tree Edits

**The Innovation:** The search tree structure itself evolves through three edit operations (Insert, Update, Prune) driven by the world model's reasoning over accumulated evidence. The topology is not fixed—branches are dynamically repositioned based on learned compositional relationships.

**What Makes This Novel:** Traditional tree search methods (MCTS, beam search) expand nodes based on value estimates but maintain a fixed expansion schema. K-Search's world model can **restructure** the tree: deleting a root-level action and re-inserting it as a child of a different branch when evidence suggests better composition.

**Concrete Example from Section 3.4:** At round 42 of the MLA Paged Decode trace, the model deletes the initial root-level `split_k` action but re-Inserts `low_overhead_split_k` deep within the `register_resident` branch. This edit reflects a learned insight about *where* split-K belongs in the optimization sequence. No prior evolutionary method can perform this operation—they can only add new candidates to an archive, not restructure relationships between existing hypotheses.

**Why This Enables Better Search:** The tree topology encodes the world model's understanding of which transformations compose well together. By pruning entire subtrees (e.g., `independent_heads` at round 34) and repositioning strategies (e.g., `split_k` at round 42), the system concentrates budget on promising regions while avoiding redundant exploration. Section 4.4's Fastp analysis (Figure 3c) shows K-Search achieving speedup ≥0.50 on 87.5% of GQA workloads versus 50.0% for OpenEvolve—this consistency stems from focused search rather than scattered sampling.

**Significance:** This capability allows K-Search to **learn optimization recipes** rather than just individual kernels. The tree structure captures which sequences of transformations work, enabling more efficient search on future kernels with similar characteristics.

**Incremental vs. Fundamental:** This is a **fundamental innovation**. Prior quality-diversity methods (MAP-Elites, ShinkaEvolve) maintain flat archives with no structural relationships between candidates. K-Search introduces hierarchical reasoning about optimization strategies.

### 4.4 Insight 4: Stagnation-Tolerant Local Refinement Protects Promising Strategies

**The Innovation:** The local refinement loop allows $K$ consecutive failed attempts before abandoning an action, protecting promising optimization intents from the stochastic nature of LLM code generation.

**Why This Is Necessary:** LLMs are imperfect code generators. Even with a sound optimization strategy, the first few implementation attempts may contain syntax errors, incorrect memory accesses, or suboptimal instruction choices. Prior methods that discard candidates immediately upon compilation failure waste budget on what are essentially transient defects rather than flawed strategies.

**What K-Search Changes:** Algorithm 1 (Section 3.3) maintains a stagnation counter $n$ that resets on improvement and increments on failure. The loop continues while $n < K$, allowing up to 7 failed attempts for CUDA kernels ($K=7$) or 5 for Triton ($K=5$). This tolerance is calibrated to the implementation complexity—Triton's higher-level abstractions require fewer refinement attempts.

**Evidence from Section 4.4:** The paper notes that OpenEvolve exhibits "high per-iteration variance: Many iterations yield programs that underperform the currently best program." K-Search's stagnation tolerance reduces this variance by ensuring each action receives sufficient refinement before evaluation. The result is more consistent progress: K-Search's score curves in Figure 3a show steady improvement, while baselines exhibit plateaus and regressions.

**Significance:** This mechanism transforms LLM code generation from a **one-shot gamble** into an **iterative refinement process** with explicit termination criteria. It acknowledges that implementation quality is separable from strategy quality—a distinction prior methods conflate.

**Incremental vs. Fundamental:** This is an **incremental but important improvement**. Iterative refinement exists in prior work (e.g., self-repair loops), but K-Search integrates it into a structured search framework with explicit stagnation detection. The innovation lies in how refinement is bounded and connected to the world model's action selection.

### 4.5 Synthesis: Why These Innovations Matter Together

The four insights above are not independent—they form a **coherent system** where each component enables the others:

| Insight | Enables | Requires |
|---------|---------|----------|
| LLM as World Model | Explicit belief tracking | Decoupled intent/implementation |
| Decoupled Intent | Non-monotonic search | Stagnation-tolerant refinement |
| Co-Evolving Topology | Compositional learning | World model reasoning |
| Stagnation Tolerance | Strategy persistence | Intent/implementation separation |

**The Compound Effect:** No single innovation explains the 2.10× average improvement over OpenEvolve. Rather, the gains emerge from their interaction:
- The world model's planning capability (Insight 1) is useless without decoupled intent (Insight 2) to act upon.
- Decoupled intent cannot navigate non-monotonic paths without stagnation tolerance (Insight 4) to survive implementation noise.
- Co-evolving topology (Insight 3) requires the world model to reason about relationships, which only matters if strategies persist long enough to demonstrate their value.

**Practical Impact:** The GPUMODE TriMul result (1030 µs, Section 4.7) demonstrates this compound effect. K-Search surpassed both prior automated methods *and* human-designed solutions using only 300 iterations—far fewer than the 25,600 iterations used by TTT-Discover's RL-based approach (Table 3). This sample efficiency stems directly from the world model's ability to plan strategically rather than sample blindly.

**Limitations and Open Questions:** The paper acknowledges that "world model evolution is merely performed by in-context learning of past observations" (Section 3.3). This design choice reduces computational overhead but may limit learning depth compared to gradient-based fine-tuning. Additionally, the tree structure assumes optimization strategies compose hierarchically, which may not hold for all kernel types. These are active areas for future work.

**Bottom Line:** K-Search demonstrates that LLMs possess "latent planning capabilities" (Conclusion) that remain untapped when confined to direct program generation. By externalizing this planning into an explicit, co-evolving search state, K-Search achieves state-of-the-art performance on complex GPU kernels with dramatically improved sample efficiency. This represents a paradigm shift from heuristic-guided search to model-based planning in automated system optimization.

## 5. Experimental Analysis

This section dissects the empirical evaluation of K-Search. We move beyond the architectural claims to examine **whether the data supports the hypothesis** that co-evolving world models outperform static evolutionary heuristics. We analyze the benchmark suite, scrutinize the quantitative gains, investigate the technical drivers behind performance differences, and critically assess the limitations revealed by the results.

### 5.1 Evaluation Methodology

To validate K-Search, the authors design a rigorous evaluation framework focused on **sample efficiency** and **final performance quality** under strict budget constraints.

#### Benchmark Suite and Workloads
The primary evaluation uses **FlashInfer-Bench** (Xing et al., 2026), a suite of highly optimized kernels used in production LLM serving. Table 2 in Section 4.3 details the four representative kernels selected for evaluation:

*   **MLA Paged Prefill & Decode:** Targets DeepSeek-V3 architecture features like multi-level attention (MLA) with paged KV cache. These are memory-bound and require specialized Hopper instructions.
*   **GQA Paged Decode:** Targets Qwen3 models. Optimizations focus on fusing query heads per KV head and vectorized memory access.
*   **FP8 MoE:** Targets DeepSeek-V3 Mixture-of-Experts on Blackwell architecture. This is the most complex kernel, involving irregular data-dependent routing and FP8 packing overhead.

Each kernel includes a fixed set of test traces captured from real traffic to ensure realistic workload distribution. The authors emphasize that "FlashInfer kernels are highly-optimized by experienced human engineers," making them a strong baseline (Section 4.1).

#### Metrics and Objective Function
The core metric is the objective score $J(x)$, defined in Section 3.1 as:

$$J(x) = s \cdot \frac{p_{\text{ref}}}{p} \cdot 100$$

where $s \in \{0, 1\}$ is correctness, $p$ is latency, and $p_{\text{ref}}$ is the latency of the reference FlashInfer baseline. This metric normalizes performance as a **percentage speedup**. A score of 100 means matching the baseline; 200 means 2× speedup. Crucially, incorrect kernels ($s=0$) receive a score of 0, penalizing methods that generate invalid code.

#### Baselines and Configuration
K-Search is compared against two state-of-the-art evolutionary methods:
*   **OpenEvolve** (Superintelligence, 2025): Uses quality-diversity search (MAP-Elites).
*   **ShinkaEvolve** (Lange et al., 2025a): Uses novelty-aware rejection sampling.

To ensure fairness, Section 4.2 specifies that all methods use:
*   **Identical Input Workloads:** Same test traces for every method.
*   **Identical Toolchain:** CUDA 12.8, FlashInfer 0.5.3, PyTorch 2.8.0.
*   **Identical Budget:** $B = 120$ iterations for FlashInfer tasks (Section 4.1). One iteration equals one compilation and benchmark run.
*   **Model Backend:** `gemini-3-pro-preview` for K-Search and baselines on FlashInfer tasks.

This controlled setup isolates the **search strategy** as the primary variable, ensuring that performance differences stem from the co-evolving world model rather than model capacity or hardware advantages.

### 5.2 Main Quantitative Results

The results demonstrate that K-Search consistently outperforms baselines across diverse kernel types, with gains scaling with kernel complexity.

#### Overall Performance Gains
Figure 3(a) in Section 4.4 plots the best-so-far score over 120 iterations. The aggregate results show significant improvements:

*   **Average Score:** K-Search achieves an overall average final score of **56.13**.
*   **vs. OpenEvolve:** This represents a **2.10× improvement** over OpenEvolve (score 26.68).
*   **vs. ShinkaEvolve:** This represents a **2.21× improvement** over ShinkaEvolve (score 25.37).

These averages mask important variations across kernels. The gains are not uniform; they correlate with the structural complexity of the optimization problem.

#### Performance by Kernel Type
Table 4.4 data breaks down the performance per kernel, revealing where the world model approach matters most:

| Kernel | K-Search Score | OpenEvolve Score | Improvement | ShinkaEvolve Score | Improvement |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **FP8 MoE** | 44.1 | 3.09 | **14.3×** | 27.9 | 1.58× |
| **MLA Prefill** | 57.4 | 19.5 | **2.95×** | 11.3 | 5.10× |
| **GQA Decode** | 76.0 | 44.2 | **1.72×** | 27.7 | 2.74× |
| **MLA Decode** | 47.1 | 39.9 | **18%** | 34.7 | **36%** |

**Key Observation:** The **14.3× gain on MoE kernels** is the most striking result. Section 4.4 notes that OpenEvolve's mean final score stays near 3, indicating it "struggles to escape the low-accuracy regime." In contrast, K-Search reaches 44.1. This supports the paper's core claim: complex kernels requiring coordinated, multi-step transformations (like routing + FFN + FP8 packing) benefit most from explicit planning. Simpler kernels (like MLA Decode) show smaller margins (18-36%), suggesting that for less complex search spaces, evolutionary heuristics are more competitive.

#### Distribution of Speedups (Fastp Analysis)
Figure 3(c) analyzes the fraction of workloads achieving specific speedup thresholds. This metric measures **consistency** across different input shapes.

*   **GQA Decode:** At a speedup threshold of $\ge 0.50$ (50% faster than baseline), K-Search succeeds on **87.5%** of workloads. OpenEvolve and ShinkaEvolve succeed on only **50.0%** and **39.6%** respectively.
*   **MLA Prefill:** At a speedup threshold of $\ge 0.40$, K-Search reaches this on **57.9%** of workloads. **None of the baseline solutions achieve speedup $\ge 0.40$** on this kernel.

This distributional evidence confirms that K-Search does not just find a single lucky optimization; it discovers robust strategies that generalize across workload variations.

### 5.3 Deep Dive: Why K-Search Wins (Kernel Analysis)

The paper provides detailed technical post-mortems in Sections 4.5 and 4.6 to explain *how* K-Search achieved these numbers. The differences are not accidental; they stem from specific architectural choices enabled by the search method.

#### FP8 MoE: Warp-Level Cooperation vs. Atomics
For the MoE kernel, the bottleneck is routing tokens to experts efficiently.
*   **K-Search Strategy:** Dedicates one GPU thread block per token (256 threads) and uses **warp-level cooperation** (`__shfl_down_sync`) to find global top-8 experts. This keeps work parallel and avoids serialization. For computation, it uses **tensor cores (WMMA)** on small $16 \times 16$ blocks with double-buffering.
*   **Baseline Strategy:** OpenEvolve uses a **persistent kernel** with a global counter (`atomicAdd`) to assign tiles. Section 4.5 states this "incurs prohibitive overhead as it requires an atomic operation to process the next tile inside a while loop." ShinkaEvolve uses plain for-loops without tensor cores.

**Why K-Search Found This:** The world model's ability to plan structural changes (Intent: "parallelize routing") allowed it to move away from the standard persistent kernel pattern that baselines got stuck in.

#### GQA Decode: Split-K Parallelism
For decode, the cost is sweeping over the key-value sequence.
*   **K-Search Strategy:** Splits the sequence across **multiple blocks**. Each block computes partial attention, and a final block merges results. It also uses a **deeper prefetch pipeline** (loading chunk $i+2$ while computing $i$).
*   **Baseline Strategy:** Uses a **single block per batch** that loops over the entire sequence. Memory load and compute are largely serialized.

**Why K-Search Found This:** The search tree allowed the model to hypothesize "split-K" as a high-level action. Even if initial implementations had merge bugs, the local refinement loop (Section 3.3) allowed it to fix them without discarding the strategy. Baselines, treating code as atomic candidates, likely discarded early split-K attempts due to merge errors.

### 5.4 Failure Cases and Trade-offs

A critical part of experimental analysis is understanding where the method *fails*. K-Search is not universally superior; its advantages come with specific trade-offs.

#### The Small Batch Size Penalty
Section 4.4 explicitly identifies a failure mode:
> "Interestingly, on some workloads for GQA decode, K-Search underperforms OpenEvolve and ShinkaEvolve, specifically those with small batch sizes: 16 of which have batch_size=1, and 4 with batch_size=16."

**Root Cause:** K-Search's optimal kernel employs **split-K parallelism** to maximize GPU utilization for large batches. This introduces synchronization costs for merging partial results.
*   **Large Batches:** Coordination overhead is amortized across many tokens; split-K wins.
*   **Small Batches:** Coordination overhead dominates; a simple single-block design (used by baselines) is faster.

**Implication:** The world model optimized for the *average* or *peak* performance case (large batches), sacrificing performance on edge cases (tiny batches). This is a valid engineering trade-off but highlights that K-Search's search objective ($J(x)$ averaged over workloads) can mask performance cliffs on specific sub-distributions.

#### Sample Efficiency vs. Absolute Peak
While K-Search is sample-efficient (120 iterations), it is worth noting that it does not always exceed human expert performance. Figure 3(c) shows that "generated kernels rarely exceed the expert-optimized FlashInfer kernel." K-Search closes the gap significantly compared to baselines, but it does not consistently surpass human engineers within the 120-iteration budget. This suggests that while the *search strategy* is superior, the LLM's *code generation ceiling* still limits absolute peak performance.

### 5.5 External Validation: GPUMODE TriMul

To test generalization beyond FlashInfer, the authors evaluated K-Search on the **GPUMODE TriMul** task (Section 4.7), a public kernel optimization competition.

*   **Task:** Triangle Multiplicative Update (AlphaFold3 core module), involving $O(N^3)$ complexity.
*   **Configuration:** Budget $B=300$ iterations. Model switch: GPT-5.2 (150 steps) $\rightarrow$ Gemini-3-Pro (150 steps).
*   **Result:** K-Search achieved **1030 µs** latency on H100.

Table 3 compares this to other leaderboard submissions:
*   **K-Search:** 1030 µs (300 iterations)
*   **shiyegao (Human/CUDA):** 1074 µs
*   **TTT (RL + Evolution):** 1161 µs (25,600 iterations)

**Significance:** K-Search surpasses both human-designed solutions and RL-based automated methods. Crucially, it does so with **85× fewer iterations** than TTT (300 vs 25,600). This validates the claim that model-based planning (K-Search) is more sample-efficient than pure reinforcement learning or evolutionary search in this domain.

### 5.6 Critical Assessment

Do the experiments convincingly support the paper's claims?

**Strengths of the Evidence:**
1.  **Controlled Baselines:** Using identical budgets, hardware, and model backends isolates the search algorithm as the variable.
2.  **Complexity Correlation:** The fact that gains scale with complexity (14.3× on MoE vs 18% on MLA Decode) strongly supports the hypothesis that planning matters most when the search space is rugged and non-monotonic.
3.  **Mechanism Verification:** The kernel analysis (Sections 4.5-4.6) confirms that K-Search finds qualitatively different (and superior) algorithmic structures, not just micro-optimizations.

**Limitations and Open Questions:**
1.  **Budget Sensitivity:** The experiments fix the budget at 120/300. It is unclear how K-Search scales if the budget is increased to 1,000+ iterations. Does the world model saturate, or does it continue to learn?
2.  **Workload Specificity:** The small batch size failure case suggests the method optimizes for the provided trace distribution. If the deployment distribution shifts (e.g., more small batches), the kernel might be sub-optimal.
3.  **Model Dependence:** The GPUMODE result relies on switching models (GPT-5.2 $\rightarrow$ Gemini-3-Pro). The contribution of model switching vs. search strategy is not ablated.

**Verdict:** The experiments provide **strong evidence** for the core claim: decoupling planning from implementation via a co-evolving world model enables more efficient search in complex kernel spaces. The 14.3× gain on MoE and the TriMul leaderboard result are difficult to explain without accepting the efficacy of the proposed search mechanism. However, the small-batch regression serves as a important reminder that automated search optimizes for the specified objective function, which may not capture all deployment constraints.

## 6. Limitations and Trade-offs

While K-Search demonstrates substantial improvements over prior evolutionary methods, the paper reveals several important limitations and design trade-offs that constrain its applicability. Understanding these boundaries is critical for practitioners considering adoption and for researchers identifying open problems.

### 6.1 Assumption: Hierarchical Composability of Optimization Strategies

K-Search's tree-structured search state assumes that kernel optimizations compose **hierarchically**—that is, transformations can be meaningfully organized as parent-child relationships where child nodes refine or extend parent strategies. This assumption enables the world model to reason about which transformations work well together (e.g., `split_k` re-Inserted under `register_resident` in Figure 2, Section 3.4).

**Why This Matters:** Not all optimization strategies admit clean hierarchical decomposition. Some transformations may be:
- **Orthogonal**: Independent optimizations that don't interact (e.g., register tiling vs. memory coalescing)
- **Conflicting**: Strategies that cannot coexist (e.g., two different memory layout choices)
- **Cyclically Dependent**: Where A enables B, but B also enables further refinement of A

The paper does not address how the search tree handles such cases. If the world model incorrectly assumes hierarchy where none exists, it may waste budget exploring invalid branches or miss beneficial combinations that span different subtrees.

**Evidence:** The MLA Paged Decode trace (Figure 2) shows successful hierarchical reasoning, but this kernel may be unusually amenable to tree-structured decomposition. The paper provides no analysis of kernels where optimization strategies are more entangled.

### 6.2 Limitation: In-Context Learning Only, No Weight Updates

Section 3.3 explicitly states: *"in the current version of K-Search, the world model evolution is merely performed by in-context learning of past observations."* The LLM is not fine-tuned during search; it reasons over accumulated trajectory within its context window.

**Trade-offs:**

| Aspect | In-Context Learning (Current) | Fine-Tuning (Alternative) |
|--------|------------------------------|---------------------------|
| **Computational Cost** | Low (no gradient updates) | High (requires training infrastructure) |
| **Learning Depth** | Limited by context window | Can capture deeper patterns |
| **Adaptability** | Adapts per-search without retraining | Requires retraining for new domains |
| **Memory** | Trajectory must fit in context | Learned weights persist indefinitely |

**Implications:**
- **Context Window Saturation**: As search progresses beyond ~100-200 iterations, the accumulated trajectory may exceed the LLM's context capacity, forcing truncation of older observations. The paper does not specify how K-Search handles this.
- **No Cross-Task Transfer**: Knowledge gained optimizing one kernel (e.g., GQA) is not retained for future kernels. Each search starts with the same base LLM priors, missing opportunities for meta-learning.
- **Shallow Pattern Recognition**: In-context learning can identify surface-level correlations but may miss deeper structural patterns that gradient-based fine-tuning could capture.

**Open Question:** Would fine-tuning the world model on successful search trajectories from multiple kernels improve sample efficiency on new kernels? The paper does not explore this.

### 6.3 Dependency: Initial Program Requirement

Section 4.3 notes: *"We provide an initial CUDA program for MLA decode kernel generation, as baselines struggle to write working kernels without an initial program."* While this is stated to help baselines, it reveals a dependency that affects K-Search as well.

**Why This Is a Limitation:**
- The search tree requires a root node (parent program $x_{\text{parent}}$) to attach actions to. Without a working initial program, the system cannot begin refinement.
- For entirely novel kernel types without existing implementations, K-Search would need a separate "bootstrap" phase to generate a correct (if slow) baseline before optimization can begin.
- The quality of the initial program constrains the search space. If the initial program has fundamental architectural flaws (e.g., wrong memory access pattern), the world model may struggle to propose transformations that require structural refactoring.

**Evidence:** The paper does not report results on kernels where no initial program was provided. The GPUMODE TriMul task (Section 4.7) was run *"without seed Triton programs provided,"* but this used a different configuration (300 iterations, model switching) and the paper does not clarify whether K-Search generated a working baseline from scratch or used a reference implementation.

### 6.4 Computational Overhead: Tree Maintenance and LLM Calls

K-Search introduces overhead beyond simple program evaluation:

1. **Search Tree Operations**: Each iteration requires Insert, Update, and Prune operations on the tree structure. While not quantified in the paper, these operations involve LLM calls to reason about the search state.

2. **Local Refinement Loop**: The stagnation-tolerant refinement (Algorithm 1, Section 3.3) may invoke the LLM up to $K$ times per action selection. With $K=7$ for CUDA kernels, a single action can consume 7× more LLM calls than a baseline method that samples once per iteration.

3. **Context Growth**: As the search tree grows, the world model must process increasingly large context windows to reason about the full trajectory, increasing latency per iteration.

**Missing Analysis:** The paper reports wall-clock time for final kernel performance (e.g., 1030 µs for TriMul) but does not report:
- Total search time (hours/days to complete 120 iterations)
- LLM API costs or compute hours consumed
- Comparison of search overhead vs. baselines

For practitioners, search time may be as important as kernel performance. A method that finds a 2× faster kernel but takes 10× longer to search may not be practical for rapid iteration.

### 6.5 Hardware and Architecture Specificity

All FlashInfer experiments are conducted on **NVIDIA H100 and B200 GPUs** with **CUDA 12.8** (Section 4.1). The GPUMODE TriMul task uses H100. This raises questions about portability:

- **Architecture-Specific Primitives**: K-Search's world model learns optimization strategies tied to specific GPU architectures (e.g., Hopper's TMA instructions, Blackwell's FP8 support). These strategies may not transfer to other architectures (AMD MI300, Intel Gaudi) or even older NVIDIA GPUs (A100, V100).

- **No Cross-Architecture Evaluation**: The paper does not test whether kernels optimized for H100 perform well on B200, or vice versa. Given that "new architectures introduce new instructions and architectural characteristics that fundamentally alter performance trade-offs" (Introduction), this is a significant gap.

- **Compiler Toolchain Dependence**: Results depend on CUDA 12.8 and FlashInfer 0.5.3. Different compiler versions may produce different performance characteristics, potentially invalidating the world model's learned priors.

**Implication:** K-Search may require re-search for each target architecture, limiting its utility for teams deploying across heterogeneous hardware.

### 6.6 Objective Function Limitations: Averaged Workloads Mask Edge Cases

The objective function $J(x)$ is computed as speedup relative to baseline, averaged over workloads (Section 3.1). Section 4.4 reveals a critical consequence:

> *"K-Search's kernel employs a split-K parallelism strategy that divides the key-value sequence across multiple thread blocks to maximize GPU utilization for large batches... While this approach excels when there is sufficient batch-level parallelism to amortize the coordination overhead, it introduces unnecessary synchronization costs for small batches."*

**The Trade-off:** K-Search optimized for **average performance** across the workload distribution, sacrificing performance on edge cases (batch_size=1 and batch_size=16). This is a valid engineering decision but reveals that:

1. **The objective function encodes deployment assumptions**. If the production workload includes many small batches, K-Search's kernel may be sub-optimal despite high average score.

2. **No multi-objective optimization**. The system does not support Pareto-front exploration (e.g., "find kernels that perform well on both small and large batches").

3. **Workload distribution sensitivity**. If the evaluation traces do not match production distribution, the optimized kernel may underperform in deployment.

**Open Question:** Can K-Search be extended to support multi-objective optimization or worst-case guarantees rather than average-case performance?

### 6.7 Scalability: Unknown Behavior Beyond Tested Budgets

All FlashInfer experiments use a fixed budget of $B=120$ iterations (Section 4.1). The GPUMODE task uses $B=300$. The paper does not address:

- **Diminishing Returns**: Does performance continue improving beyond 120 iterations, or does the world model saturate? Figure 3(a) shows K-Search still improving at iteration 120 for some kernels, suggesting untapped potential—but also raising questions about convergence.

- **Budget Sensitivity**: How does performance scale with budget? Is the 2.10× improvement over OpenEvolve maintained at $B=50$, $B=500$, or $B=1000$? Without this analysis, it's unclear whether K-Search's advantages are most pronounced at low budgets (where planning matters most) or persist at high budgets.

- **Tree Size Growth**: As iterations increase, the search tree grows. At what point does tree management become a bottleneck? Does pruning keep the frontier manageable, or does the tree grow unbounded?

**Evidence:** Figure 3(a) shows K-Search's score curves still rising at iteration 120 for MLA Prefill and MoE, suggesting the search has not converged. This is encouraging but leaves the scalability question unanswered.

### 6.8 Model Dependence and Switching Strategy

The GPUMODE TriMul result uses a **model switching strategy**: 150 iterations with GPT-5.2, then 150 iterations with Gemini-3-Pro (Section 4.7). The paper does not ablate this choice:

- **Why Switch Models?** Is GPT-5.2 better for early exploration and Gemini-3-Pro better for late-stage refinement? Or is this simply leveraging available model access?

- **Contribution of Model vs. Search**: How much of the 1030 µs result stems from K-Search's search strategy vs. the specific LLM capabilities? A comparison using a single model throughout would clarify this.

- **Reproducibility**: GPT-5.2 and Gemini-3-Pro are not publicly available models (as of the paper's writing). Practitioners cannot replicate the GPUMODE result without access to these specific models.

**FlashInfer Experiments:** All FlashInfer tasks use `gemini-3-pro-preview` (Section 4.2), but the paper does not test whether weaker models (e.g., 7B or 13B parameter LLMs) can effectively serve as world models. If K-Search requires large, proprietary models, its accessibility is limited.

### 6.9 Generalization Beyond Attention and MoE Kernels

The evaluation focuses exclusively on **attention variants** (MLA, GQA) and **Mixture-of-Experts** kernels from FlashInfer (Table 2, Section 4.3). These kernels share common characteristics:
- Regular memory access patterns (paged KV cache)
- Parallelism across tokens, heads, or experts
- Well-understood optimization primitives (tiling, vectorization, fusion)

**Unaddressed Kernel Types:**
- **Convolution kernels** (common in vision models)
- **Reduction kernels** (e.g., LayerNorm, softmax across large dimensions)
- **Sparse or irregular kernels** (e.g., graph neural networks)
- **Custom operators** with domain-specific constraints

The world model's priors are shaped by the training data and the kernels seen during search. It's unclear whether K-Search can effectively optimize kernels outside the attention/MoE domain without significant reconfiguration or additional prior knowledge.

### 6.10 Summary of Limitations

| Limitation | Severity | Addressed in Paper |
|------------|----------|---------------------|
| Small batch performance degradation | High | Yes (Section 4.4) |
| In-context learning only (no fine-tuning) | Medium | Yes (Section 3.3) |
| Initial program requirement | Medium | Partially (Section 4.3) |
| Computational overhead unquantified | Medium | No |
| Hardware specificity (H100/B200 only) | High | No |
| Objective function masks edge cases | High | Yes (Section 4.4) |
| Scalability beyond tested budgets | Medium | No |
| Model dependence and switching | Medium | Partially (Section 4.7) |
| Limited kernel type coverage | High | No |

**Bottom Line:** K-Search represents a significant advance in automated kernel optimization, but it is not a universal solution. The method excels when:
- The kernel has hierarchical optimization structure
- Average-case performance is the priority
- Sufficient LLM capacity is available
- Target hardware matches the search environment

Practitioners should be cautious when:
- Edge-case performance (e.g., small batches) is critical
- Cross-architecture portability is required
- Search time/cost is a primary constraint
- The kernel type differs substantially from attention/MoE patterns

These limitations define a clear agenda for future work: multi-objective optimization, cross-architecture search, fine-tuned world models, and broader kernel coverage.

## 7. Implications and Future Directions

K-Search represents more than an incremental improvement in kernel auto-tuning; it signals a conceptual shift in how we leverage Large Language Models for systems optimization. By demonstrating that LLMs can function as **intrinsic world models** rather than mere code generators, this work opens new avenues for research in automated system design, compiler construction, and infrastructure management. This section outlines how this work changes the landscape, identifies concrete follow-up research opportunities, describes practical deployment scenarios, and provides guidance for practitioners considering adoption.

### 7.1 Shifting the Paradigm: From Heuristic Search to Model-Based Planning

The most significant implication of K-Search is the validation of **model-based planning** in domains previously dominated by heuristic search. Traditional compiler autotuners like TVM/Ansor (Section 2) rely on learned cost models or exhaustive search over scheduling primitives. Evolutionary LLM methods like OpenEvolve (Section 2) rely on quality-diversity heuristics (e.g., MAP-Elites) to mutate program text directly.

K-Search demonstrates that introducing an explicit **search state** $S_t$ with prioritized actions $V(a \mid S_t)$ (Section 3.2) allows the system to navigate **non-monotonic optimization paths**. This changes the landscape in two key ways:

1.  **Reasoning Over Sampling:** The field moves away from treating LLMs as stochastic samplers ($x_{t+1} \sim \pi_{\text{LLM}}(\text{context})$) toward treating them as reasoning engines that maintain beliefs about the search space. The Conclusion states this explicitly: LLMs possess "latent planning capabilities" that are unlocked when decoupled from immediate implementation.
2.  **Structured Knowledge Retention:** Unlike flat archives that store programs without relationships, K-Search's tree structure (Figure 1) encodes *why* certain transformations work together. This suggests future automated systems should maintain explicit structural memories of optimization trajectories rather than just logging final scores.

This paradigm shift suggests that future "AI for Systems" research should focus less on prompt engineering for code generation and more on **architecting search states** that allow models to reason over time.

### 7.2 Follow-Up Research Opportunities

The limitations identified in Section 6 and the design choices in Section 3 point to several high-value research directions:

#### 1. Fine-Tuned World Models vs. In-Context Learning
Currently, "the world model evolution is merely performed by in-context learning of past observations" (Section 3.3). This limits the depth of knowledge retention to the context window.
*   **Research Question:** Can fine-tuning the LLM on successful search trajectories from multiple kernels improve sample efficiency on new kernels?
*   **Hypothesis:** A fine-tuned world model could learn universal optimization priors (e.g., "split-K composes well with fusion") that persist across tasks, reducing the need for large context windows and enabling cross-task transfer (currently absent per Section 6.2).

#### 2. Multi-Objective and Worst-Case Optimization
K-Search optimizes for average speedup $J(x)$ over a workload distribution (Section 3.1). This led to performance degradation on small batch sizes for GQA decode (Section 4.4).
*   **Research Question:** How can the search state $S_t$ be extended to track Pareto frontiers rather than single scalar scores?
*   **Potential Approach:** Modify the priority score $V(a \mid S_t)$ to be a vector representing performance across different workload clusters (e.g., small vs. large batches). The `Prune` operation (Section 3.3) would then remove actions that are dominated across all objectives, not just the average.

#### 3. Cross-Architecture Generalization
Experiments are confined to NVIDIA H100 and B200 GPUs (Section 4.1). The world model learns architecture-specific priors (e.g., Hopper TMA instructions).
*   **Research Question:** Can a world model trained on one architecture (e.g., Hopper) transfer its planning logic to another (e.g., Blackwell or AMD MI300)?
*   **Potential Approach:** Decouple the "intent" layer ($\delta$) from architecture-specific primitives. The world model could plan high-level strategies (e.g., "reduce memory traffic") while a lower-level policy handles architecture-specific instantiation.

#### 4. Reducing Search Overhead
The paper does not quantify the wall-clock time or API cost of the search process (Section 6.4). With $K=7$ stagnation tolerance (Section 4.2), each action selection may trigger multiple LLM calls.
*   **Research Question:** Can smaller, specialized models serve as the world model while larger models handle code instantiation?
*   **Potential Approach:** Distill the planning capability into a smaller model (e.g., 7B parameters) to reduce latency during Action Selection, reserving large models for the Program Instantiation phase.

### 7.3 Practical Applications and Downstream Use Cases

K-Search is not merely a research prototype; it addresses concrete pain points in ML infrastructure development.

#### 1. Rapid Adaptation to New Hardware
The Introduction notes that "new architectures... introduce new instructions and architectural characteristics that fundamentally alter performance trade-offs." Manually retuning libraries like FlashInfer for every new GPU generation is costly.
*   **Use Case:** When NVIDIA releases a new architecture (e.g., Rubin), K-Search can be deployed to automatically retune critical kernels (Attention, MoE) using the previous generation's kernels as initial programs $x_{\text{parent}}$. The co-evolving world model can quickly identify which old optimizations remain valid and which need restructuring.

#### 2. Custom Operator Optimization for Emerging Models
New model architectures (e.g., Mamba, RWKV mentioned in Section 2) require custom operators that lack mature library support.
*   **Use Case:** Infrastructure teams can use K-Search to generate optimized kernels for novel attention mechanisms or routing algorithms without waiting for vendor-optimized libraries. The GPUMODE TriMul result (Table 3) demonstrates this capability, achieving state-of-the-art performance on a specialized AlphaFold3 operator.

#### 3. Competition and Leaderboard Optimization
The GPUMODE TriMul evaluation (Section 4.7) shows K-Search surpassing human-designed solutions with only 300 iterations.
*   **Use Case:** In kernel optimization competitions or internal performance sprints, K-Search can serve as a force multiplier, exploring optimization spaces that human engineers might overlook due to time constraints. The 14.3× gain on MoE kernels (Section 4.4) suggests it is particularly valuable for complex, irregular operators where human intuition is less reliable.

### 7.4 Reproduction and Integration Guidance

For practitioners considering K-Search, the following guidance synthesizes the experimental setup (Section 4) and limitations (Section 6) to help determine fit.

#### When to Prefer K-Search
*   **Complex Kernels:** Use K-Search for kernels requiring coordinated, multi-step transformations (e.g., MoE, MLA). The 14.3× gain on MoE vs. 18% on MLA Decode (Section 4.4) indicates benefits scale with complexity.
*   **Limited Evaluation Budget:** If compilation/profiling is expensive (limiting you to $B \approx 100-300$ evaluations), K-Search's planning efficiency outperforms evolutionary baselines that waste budget on invalid candidates.
*   **Available Initial Program:** K-Search performs best when provided a working initial program (Section 4.3). If you lack a correct baseline, you may need a bootstrap phase to generate one.

#### When to Consider Alternatives
*   **Simple Kernels:** For straightforward operators (e.g., element-wise additions), traditional compiler autotuners (TVM) or simple evolutionary search may be sufficient and incur less overhead.
*   **Edge-Case Critical:** If your workload includes significant small-batch traffic, be aware that K-Search may optimize for average performance at the expense of edge cases (Section 4.4). You may need to modify the objective function $J(x)$ to weight specific workload clusters.
*   **Strict Cost Constraints:** If LLM API costs are a primary concern, note that the local refinement loop (Algorithm 1) may consume multiple tokens per action.

#### Integration Steps
1.  **Access the Code:** The authors provide open-source code at `https://github.com/caoshiyi/K-Search` (Abstract).
2.  **Define Task Interface:** Implement the unified Task interface (Section 4.1) requiring:
    *   A task specification (PyTorch reference, optimization objective).
    *   An evaluator (compilation, correctness check, benchmarking).
3.  **Configure Hyperparameters:**
    *   Set budget $B$ (e.g., 120 for FlashInfer tasks, Section 4.1).
    *   Set stagnation limit $K$ (e.g., 7 for CUDA, 5 for Triton, Section 4.2/4.7).
    *   Select LLM backend (e.g., `gemini-3-pro-preview` for FlashInfer, Section 4.2).
4.  **Provide Initial Program:** Supply a correct (even if slow) CUDA/Triton implementation to initialize the search tree root (Section 4.3).
5.  **Monitor Search Trace:** Use the tree visualization (like Figure 2) to understand which optimization intents are being selected and pruned. This provides interpretability often missing in black-box autotuners.

#### Critical Success Factors
*   **Evaluator Correctness:** The evaluator $\mathcal{E}(x)$ must strictly enforce correctness ($s \in \{0, 1\}$). Incorrect kernels receiving non-zero scores will corrupt the world model's priority estimates $V$.
*   **Workload Representation:** Ensure the benchmark workloads used in $J(x)$ accurately reflect production traffic distributions. As seen in the GQA small-batch case (Section 4.4), the optimizer will faithfully optimize for whatever distribution you provide.

By treating the LLM as a planning engine rather than a code generator, K-Search offers a robust framework for navigating the increasing complexity of GPU kernel optimization. While limitations around hardware specificity and search overhead remain, the demonstrated ability to discover state-of-the-art kernels with limited budgets establishes a new baseline for future automated systems research.
