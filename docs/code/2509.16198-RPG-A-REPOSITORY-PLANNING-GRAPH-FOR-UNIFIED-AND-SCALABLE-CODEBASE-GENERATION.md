## 1. Executive Summary

This paper introduces the Repository Planning Graph (RPG) and the ZeroRepo framework to solve the fundamental challenge of generating complete, coherent software repositories from scratch, overcoming the ambiguity and inconsistency inherent in natural language planning. Evaluated on the new RepoCraft benchmark comprising 1,052 tasks across six real-world projects, ZeroRepo achieves 81.5% functional coverage and 69.7% test accuracy, outperforming the strongest baseline (Claude Code) by 27.3 and 35.8 percentage points while generating repositories 3.9× larger. These results establish structured graph-based planning as a critical enabler for scalable, long-horizon code generation that maintains structural fidelity and functional correctness.

## 2. Context and Motivation

### The Core Challenge: From Functions to Repositories

Large language models have demonstrated remarkable capability at generating individual functions and single files of code from natural language descriptions. However, there exists a fundamental gap between this function-level competence and the ability to generate **complete software repositories from scratch**. This gap is not merely a matter of scale—it represents a qualitative shift in the planning and reasoning requirements.

Generating a repository requires navigating what the paper terms a "progressive planning" process across two complementary phases:

1. **Proposal-level planning**: Determining *what* to build by defining the functional scope and key capabilities of the system
2. **Implementation-level planning**: Determining *how* to build it by specifying file structure, interfaces, dependencies, and data flows

The core difficulty lies in bridging high-level user intent (e.g., "build a machine learning library") with the repository's intricate network of files, classes, and dependencies. This requires maintaining coherence across hundreds or thousands of code entities while ensuring that inter-module contracts, data flows, and functional boundaries remain consistent throughout the generation process.

### Why This Problem Matters

The ability to generate complete repositories from high-level specifications has both practical and theoretical significance:

**Real-World Impact:**
- **Automation at scale**: Current code generation tools assist developers at the function or file level, but repository-level generation could enable rapid prototyping of entire software systems from specifications
- **Reduced engineering overhead**: Building coherent software systems from scratch requires significant architectural planning; automated repository generation could accelerate this process
- **Consistency and maintainability**: Human-designed repositories can suffer from inconsistent patterns; structured generation could enforce architectural coherence from the outset

**Theoretical Significance:**
- **Long-horizon reasoning**: Repository generation tests an AI system's ability to plan and execute over extended sequences of dependent decisions—a key capability for general autonomous agents
- **Structured representation learning**: The problem requires encoding and manipulating complex hierarchical relationships (modules → files → classes → functions) with explicit dependency constraints
- **Planning under ambiguity**: Translating vague natural language specifications into precise, executable code structures requires disambiguating intent and making architectural decisions

### Prior Approaches and Their Limitations

The paper identifies three dominant paradigms in prior work, all of which share a critical weakness:

| Paradigm | Examples | Approach |
|----------|----------|----------|
| **Distributed Planning Frameworks** | MetaGPT, ChatDev | Assign specialized roles (manager, architect, engineer) to negotiate between requirements and implementations |
| **Workflow-Based Systems** | Paper2Code, AutoP2C | Follow fixed pipelines that build architectural skeletons before filling in details |
| **Iterative Terminal Agents** | OpenHands, Claude Code, Gemini CLI | Externalize intermediate plans (often in markdown) and refine them step-by-step |

**The Shared Limitation: Natural Language as Planning Medium**

Despite their architectural differences, all three paradigms rely on **natural language as the intermediate representation for planning**. While natural language is flexible and human-readable, the paper argues it is fundamentally ill-suited for large-scale repository generation due to three inherent weaknesses:

1. **Ambiguity**: Natural language blurs distinctions between intent and constraints, making it difficult to enforce precise architectural decisions (Wang et al., 2024)

2. **Lack of Explicit Hierarchy**: Dependency tracking becomes particularly difficult without structured representations that encode hierarchical relationships (Besta et al., 2024)

3. **Plan Degradation**: Static natural language plans may gradually degrade over long horizons without adaptive adjustment, leading to drift across iterations (Sun et al., 2023)

**Concrete Failure Modes:**

When extended to automatic repository generation, these limitations manifest as:

- **Unstable proposal-level planning**: Functionalities become incomplete, overlapping, or unevenly scoped (Zhu et al., 2025)
- **Fragmented implementation-level planning**: Plans drift across iterations, introducing inconsistencies in dependencies, data flows, and modular boundaries (Almorsi et al., 2024; Ashrafi et al., 2025)

The paper's empirical results support this critique: on the RepoCraft benchmark, natural language-based baselines like Claude Code achieve only 54.2% functional coverage and 33.9% test pass rate, while generating repositories that are 3.9× smaller than ZeroRepo's output (Table 2).

### Positioning Relative to Existing Work

This paper positions itself as a **structural alternative** to natural language planning rather than an incremental improvement to existing frameworks. The key distinction is the introduction of the **Repository Planning Graph (RPG)**—a persistent, evolvable graph representation that encodes:

- **Nodes**: Hierarchical capabilities (modules → files → classes → functions)
- **Edges**: Semantic relations and data flows between components

By replacing free-form natural language with an explicit blueprint, RPG provides what the paper calls a "compact, interpretable basis for consistent long-horizon planning."

**Benchmark Contribution:**

The paper also addresses a gap in evaluation infrastructure. Prior benchmarks mainly target:
- Incremental development (editing, refactoring, or bug fixing in existing codebases)
- Repository generation with detailed skeletons and specifications that limit autonomous planning

The **RepoCraft benchmark** (6 projects, 1,052 tasks) fills this gap by requiring agents to build complete repositories from high-level natural language descriptions and evaluating them against real-world projects on scale, functionality, and correctness (Table 1).

**Key Differentiators:**

| Aspect | Prior Work | This Paper (ZeroRepo) |
|--------|------------|----------------------|
| Planning representation | Natural language (markdown, text) | Structured graph (RPG) |
| Planning persistence | Ephemeral, iteration-specific | Persistent, evolvable |
| Dependency encoding | Implicit in text | Explicit edges in graph |
| Evaluation | Incremental tasks or constrained generation | End-to-end repository generation |

The paper's central claim is that structured graph-based planning is not merely an implementation detail but a **fundamental enabler** for scalable repository generation—allowing near-linear scaling of functionality and code size while maintaining structural coherence.

## 3. Technical Approach

This is a **systems paper** that introduces a new architectural framework for repository-level code generation, with the core idea being that **structured graph representations** (RPG) can replace ambiguous natural language planning to enable consistent, scalable software synthesis.

### 3.1 Reader orientation (approachable technical breakdown)

**What the system is:** ZeroRepo is an automated software engineering framework that takes a high-level natural language specification (e.g., "build a machine learning library") and generates a complete, working code repository with hundreds of files, proper dependencies, and passing tests.

**What problem it solves and the solution shape:** The system solves the problem of **long-horizon planning inconsistency** in code generation by replacing ephemeral natural language plans with a **persistent graph structure** (the Repository Planning Graph) that explicitly encodes what to build, how files relate, and how data flows between components—this graph serves as a stable blueprint that guides code generation from start to finish.

### 3.2 Big-picture architecture (diagram in words)

The ZeroRepo system consists of **three sequential stages** that progressively transform a user query into executable code:

1. **Proposal-Level Construction Module**: Takes the user's natural language query and constructs a **functionality graph** by searching and refining a subtree from the EpiCoder Feature Tree (a 1.5M-node ontology of software capabilities). Output: A graph of what features the repository should have, organized into modules.

2. **Implementation-Level Construction Module**: Takes the functionality graph and enriches it with **file structure, data flows, and interface specifications**. This stage assigns folders to modules, files to components, and defines how data moves between modules. Output: The complete Repository Planning Graph (RPG) with implementation details.

3. **Graph-Guided Code Generation Module**: Traverses the RPG in **topological order** (dependencies before dependents), generating code for each leaf node with test-driven development. Includes localization tools for debugging and editing. Output: A complete, tested code repository.

**Information Flow:** User Query → Feature Tree Search → Functionality Graph → File/Interface Encoding → Complete RPG → Topological Code Generation → Tested Repository

### 3.4 Detailed, sentence-based technical breakdown

#### 3.4.1 The Repository Planning Graph (RPG) Structure

The Repository Planning Graph is a **directed, hierarchical graph** where nodes represent software components at multiple levels of abstraction and edges represent dependencies and data flows between them.

**Node Semantics (Dual Representation):**

Each node in the RPG carries two complementary meanings that bridge functionality and structure:

- **Functional Level**: Nodes represent progressively refined capabilities, where high-level modules (e.g., "ML Algorithms") decompose into mid-level components (e.g., "Linear Regression") and ultimately into leaf nodes corresponding to concrete algorithms (e.g., "Lasso", "Elastic Net").

- **Structural Level**: This hierarchy mirrors actual repository organization, where root nodes align with file regions or directories (e.g., `src/algos/`), intermediate nodes with files (e.g., `linear.py`), and leaf nodes with specific functions or classes (e.g., `class LassoRegression`).

This dual semantics is the key innovation: **the same graph simultaneously represents what the system does and how the code is organized**, ensuring that functional decomposition aligns with code structure.

**Edge Types (Explicit Dependencies):**

The RPG encodes two types of edges that capture execution dependencies:

1. **Inter-module edges** (shown as black arrows in Figure 2): These encode data flows between different modules. For example, outputs from a "Data Loading" module feed into "ML Algorithms" and then into "Evaluation". These edges ensure that the global execution pipeline is coherent.

2. **Intra-module edges** (shown as gray dashed arrows in Figure 2): These capture file-level orderings within a module. For instance, `load_data.py` must precede `preprocess.py`, with outputs passed to preprocessing. These edges impose a local topological order.

**Mathematical Representation:**

The RPG can be formalized as a directed graph $G = (V, E)$ where:

- $V$ is the set of nodes, partitioned into levels $V = V_{\text{root}} \cup V_{\text{intermediate}} \cup V_{\text{leaf}}$
- $E = E_{\text{inter}} \cup E_{\text{intra}}$ where $E_{\text{inter}}$ represents inter-module data flows and $E_{\text{intra}}$ represents intra-module orderings

The graph must satisfy a **topological constraint**: for any edge $(u, v) \in E$, node $u$ must be generated before node $v$ in the code generation phase. This ensures dependencies precede dependents.

**Why Graph Over Natural Language:**

The paper makes a critical design choice to use a graph rather than natural language for planning. The reasoning is:

- **Ambiguity elimination**: Natural language blurs distinctions between intent and constraints, whereas graph nodes and edges have precise, machine-interpretable semantics.
- **Explicit hierarchy**: Dependency tracking is difficult in text without structured representations, whereas the graph explicitly encodes parent-child and dependency relationships.
- **Persistence**: Natural language plans degrade over long horizons (30+ iterations), whereas the graph provides a stable substrate that can be incrementally refined without losing coherence.

The empirical evidence supports this: natural language baselines plateau at 3–4K lines of code, while RPG-guided generation sustains near-linear growth to 30K+ lines (Figure 6).

#### 3.4.2 Proposal-Level Construction (Stage 1)

The first stage transforms a high-level user specification into a **functionality graph** by searching and refining a subtree from a global knowledge base.

**The EpiCoder Feature Tree as Knowledge Base:**

ZeroRepo uses the **EpiCoder Feature Tree** (Wang et al., 2025) as its knowledge base—a pre-constructed ontology containing **1.5 million software capabilities** organized into 7 hierarchical levels:

| Level | # Elements | Examples |
|-------|-----------|----------|
| 1 | 17 | functionality, data structures, data processing |
| 2 | 1,527 | text processing, process monitoring, create flowchart |
| 3 | 21,739 | heap allocation, dayjs, affine transformation |
| 4 | 113,348 | update record by ID, automated versioning, M5 Model Tree |
| 5 | 613,311 | add vertices, angular velocity, find minimum by group |
| 6 | 33,801 | min with inclusion, multiple with keyword |
| 7 | 781 | validate against thesaurus, swipe event detection |

This tree serves as a **structured prior** that mitigates randomness and bias in capability enumeration. Without this knowledge base, LLMs alone provide unstable and biased capability enumeration with incomplete coverage (as shown in the ablation study in Section 8).

**Node Embedding for Retrieval:**

For efficient retrieval, each node in the Feature Tree is embedded in a vector space using **`infly/inf-retriever-v1`** (Yang et al., 2025), with its hierarchical path stored as metadata in a vector index. This preserves both semantic similarity and structural context, enabling precise, scalable grounding during subtree selection.

**Explore–Exploit Subtree Selection (Algorithm 2):**

The system cannot enumerate all 1.5M nodes, so it incrementally expands a **repo-aligned subtree** via an explore–exploit strategy over **30 iterations**:

1. **Exploitation (Precision)**: The system retrieves the top-$k$ feature paths most aligned with the user goal and augments them with keywords suggested by LLM queries. This ensures the selected features are relevant to the specification.

2. **Exploration (Diversity)**: The system deliberately expands into unvisited regions of the ontology to capture less obvious but relevant functionalities. This prevents the plan from becoming too narrow.

3. **LLM Filtering**: Candidates from both strategies are filtered by the LLM and integrated into the evolving subtree. The LLM performs batch-level self-checks to ensure internal consistency.

4. **Termination**: The process continues until the iteration budget (30 iterations) is reached.

**Diversity-Aware Rejection Sampling (Algorithm 1):**

To encourage broader feature space exploration, the system uses a rejection sampling mechanism:

- At each step, a candidate tree is accepted only if its **overlap with previously sampled nodes** is below a specified threshold $\rho$.
- Otherwise, the tree with the minimal overlap is returned as a fallback.
- This is formalized in Algorithm 1, where `REJECTSAMPLE` retries up to $T_{\text{max}}$ times to find a candidate with overlap $\leq \rho$.

**Refactoring by Goal Alignment:**

The repo-aligned subtree, though capturing relevant functionalities, still inherits the generic organization of the global ontology. To align it with the user's repository goal, the system refactors feature placements into a **modular functionality graph**:

- The LLM partitions functionalities into cohesive modules following software engineering principles of **cohesion and coupling**.
- For instance, in a machine learning library, metrics such as `silhouette_score` are reorganized under an "Evaluation" module rather than within "Clustering Algorithms".
- The resulting graph establishes clear functional boundaries, encoding proposal-level planning directly into the representation.

**Configuration:**
- **30 iterations** for feature selection in Proposal-Level Construction
- **Top-$k$ retrieval** for exploitation (exact $k$ not specified in main text)
- **Batch size $B$** for LLM self-checks (exact value in Appendix B.1)

#### 3.4.3 Implementation-Level Construction (Stage 2)

After proposal-level construction establishes the multi-level functional plan, the graph is further enriched with implementation details, culminating in the complete Repository Planning Graph.

**File Structure Encoding (§3.3.1):**

While proposal-level planning defines modular functionalities, it remains abstract and detached from implementation. This stage bridges the gap by extending the graph with folder and file layouts.

**Folder-Level Encoding:**

- Root nodes are enriched with **folder-level specifications**, assigning each subgraph a directory namespace (e.g., `algos/`, `eval/`).
- This encoding couples semantic modularity with structural separation, ensuring descendants inherit a consistent namespace.
- The repository skeleton aligns with high-level capability decomposition.

**File-Level Encoding:**

- Once folder regions are encoded at root nodes, the graph is enriched by assigning **files to intermediate nodes**, specifying how module functionalities map to executable files.
- For example, preprocessing utilities consolidate into `preprocess.py`, while models like linear regression and variants group into `linear_models.py`.
- Embedding file structure preserves semantic cohesion, reduces cross-file coupling, and yields a **file-augmented graph** anchoring design.

**Data Flow and Functions Encoding (§3.3.2):**

After obtaining the file-augmented graph, this stage finalizes the full RPG by assigning executable roles to leaf nodes.

**Data-Flow Encoding:**

- The graph is augmented with **data-flow edges** capturing inter- and intra-module relations.
- **Globally** (Figure 2): Typed input–output flows connect subgraph roots. For example, a data-loading module may provide an array of training data to algorithms.
- **Locally**: Files within a module are ordered to ensure coherent, dependency-aware implementation.
- These flows impose a **hierarchical order** constraining and organizing interface design.

**Abstracting Global Interfaces:**

- To improve scalability and maintainability, recurring input–output patterns across modules are abstracted into **common data structures or base classes**.
- For example, algorithms can be unified under a `BaseEstimator` class to standardize interaction with preprocessing and evaluation modules.
- These abstractions serve as anchors that enforce consistency and reduce redundancy.

**Adaptive Interface Design:**

- Within each file-level subgraph, leaf features cluster into executable interfaces by semantic relatedness.
- **Independent features** become standalone functions, while **interdependent ones** form shared classes with methods.
- For example, in Figure 2, `load_json` and `load_csv` are grouped into a `DataLoader` class, while `elastic_net` is implemented as an `ElasticNet` class.
- This adaptive mapping balances granularity and cohesion, yielding an RPG that preserves modularity and semantic consistency at repository scale.

**Prompt Templates (Appendix C.1):**

The implementation-level construction is guided by several prompt templates that structure the LLM's output:

- **Data Flow Template**: Requires the LLM to define inter-subtree data flow as a structured, directed graph with fields for `from`, `to`, `data_id`, `data_type`, and `transformation`. The output must form a **Directed Acyclic Graph (DAG)** with full connectivity (every subtree appears in at least one edge).

- **Skeleton Mapping Template**: Guides the transformation from graph subtrees into modular code skeletons, requiring folder names to follow Python conventions (snake_case) and balance clarity, scalability, and maintainability.

- **Base Class Template**: Instructs the LLM to design reusable abstractions for nodes with high in-degree or out-degree, typically producing 1–3 global base classes.

- **Interface Template**: For each feature, the LLM defines exactly one interface (function or class) with imports, signature, and detailed docstring, but **no implementation** (uses `pass`).

#### 3.4.4 Graph-Guided Code Generation (Stage 3)

Given a user query and the completed RPG, ZeroRepo generates repositories by traversing the graph in topological order, ensuring dependencies precede dependents.

**Topological Traversal:**

- The system traverses the RPG in **topological order**, which is determined by the data-flow edges encoded during implementation-level construction.
- This ensures that when generating code for a node, all its dependencies have already been generated and validated.
- For example, data loading modules are generated before ML algorithm modules, which are generated before evaluation modules.

**Test-Driven Development at Leaf Nodes:**

At each leaf node, **test-driven development (TDD)** is applied in the following sequence:

1. **Test Derivation**: Tests are derived from the specification (the node's docstring and interface definition).
2. **Implementation**: The corresponding functions or classes are implemented.
3. **Validation**: Tests are executed; failing cases trigger revisions until passing or the iteration limit is reached.
4. **Commitment**: Only functions that pass all tests are committed, enabling incremental expansion while preserving stability.

**Configuration:**
- Each function allows up to **8 debugging iterations**.
- **20 localization attempts** are permitted for finding the correct code location.
- For test failures, **5-round majority voting** is used for attribution (distinguishing implementation errors from environment/test issues).
- Up to **20 remediation attempts** are allowed for test or environment errors.

**Patch-Oriented Testing (Algorithm 3):**

The testing framework combines three types of tests into a unified process:

1. **Unit Tests**: Validate each new patch at a fine-grained level. For each patch $p$, the system creates or updates a unit test node $n_{\text{new}}$ and executes it.

2. **Regression Tests**: Re-execute existing nodes to ensure backward compatibility. If an existing unit node $n_{\text{old}}$ has the same signature or logic as the patch, it is reused for regression testing.

3. **Integration Tests**: Verify that multiple patched components interact correctly when combined. Patch groups $G$ are clustered by integration-node, and integration test nodes are created or reused.

The algorithm is formalized in Algorithm 3, which returns $T_{\text{unit}} \cup T_{\text{inte}}$ (the union of unit and integration test results) along with trajectory information $T_{\text{traj}}$.

**End-to-End Test Pipeline (Algorithm 4):**

The testing pipeline proceeds in a sequence of stages:

1. **Branch Planning**: A candidate test branch is created for the given code unit(s).
2. **Test Generation**: Test code is generated and wrapped into a `TestNode` or `IntegrationTestNode`.
3. **Execution**: The test is executed inside a controlled **Docker environment**.
4. **Judgment**: An LLM judges the outcome; if failures are detected, the framework automatically generates fix queries.
5. **Repair**: The test is iteratively repaired until a validated version is obtained.

This is formalized in Algorithm 4, which takes as input the repo skeleton $R$, tested unit(s) $U$, source code $C$, optional prior test node $n_{\text{old}}$, and maximum retries $T_{\text{max}}$.

#### 3.4.5 Graph-Guided Localization and Editing

To handle implementation and debugging requests, ZeroRepo adopts a **two-stage workflow**: first localizing the target in the RPG, then editing the associated code.

**Localization Tools (Appendix D.1):**

Localization leverages the graph's global structure and five tools:

1. **`view_file_interface_feature_map(file_path)`**: Inspects a single Python file to list the interface structures (functions, classes, methods) it contains, along with the feature mappings they support.

2. **`get_interface_content(target_specs)`**: Retrieves the full implementation code of a specific function, class, or method, given its fully qualified name (file path + entity name).

3. **`expand_leaf_node_info(feature_path)`**: Given a feature path from the implemented feature tree, this tool expands and lists all associated interfaces (functions or classes) in a structural summary.

4. **`search_interface_by_functionality(keywords)`**: Performs a fuzzy semantic search for interfaces based on given keywords and returns the top-5 most relevant interface implementations.

5. **`Terminate(result)`**: Terminates the localization exploration and returns the final ranked list of located interfaces in JSON-style format.

**Editing Tools (Appendix D.2):**

Once localized, the agent revises or generates the corresponding code using four editing tools:

1. **`edit_whole_class_in_file(file_path, class_name)`**: Replaces an entire class (including all methods and docstring). Output must provide the full class definition.

2. **`edit_method_of_class_in_file(file_path, class_name, method_name)`**: Replaces a single method inside a class. Output must return the full class block containing only the target method (not the method alone).

3. **`edit_function_in_file(file_path, function_name)`**: Replaces a top-level function. Output must provide the full function, including signature, logic, and docstring.

4. **`edit_imports_and_assignments_in_file(file_path)`**: Adds or corrects import statements or top-level assignments. Must follow strict import order: (1) standard library, (2) third-party packages, (3) local modules.

**Localization Efficiency (Table 4):**

The paper evaluates the impact of RPG guidance by comparing localization steps with and without graph support:

| Category | ZeroRepo | Without Graph |
|----------|----------|---------------|
| Integration Testing | 6.2 ± 2.1 | 13.3 ± 11.1 |
| Incremental Development | 6.8 ± 1.8 | 10.8 ± 2.6 |
| Debugging | 5.8 ± 2.8 | 8.5 ± 2.9 |

Graph-guided search reduces effort by **30–50%** across all categories. This shows that RPG equips agents with a principled navigation mechanism, enabling faster dependency tracing, more accurate bug localization, and smoother module integration.

**Localization Behavior Pattern (Figure 15):**

Figure 15 shows that with graph guidance, localization behavior follows a structured **CCG pattern** (Coarse Search → Content Inspection → Global Graph Exploration):

1. The agent begins by traversing the RPG at a coarse level to identify high-level candidates.
2. Then inspects content-rich nodes for detailed signals.
3. Finally explores semantically related structures across the graph.
4. Termination calls rise as the search converges.

This progression indicates that the RPG reshapes the agent's behavior into a systematic, relation-aware search process, replacing ad hoc or repetitive probing.

#### 3.4.6 Key Design Choices and Rationale

**Why EpiCoder Feature Tree?**

The ablation study in Section 8 isolates the impact of the EpiCoder Feature Tree (EFT):

- **ZeroRepo (with EFT)**: 95.7% coverage, 7.9% novelty, 31,596 LOC at iteration 30
- **ZeroRepo (without EFT)**: 87.2% coverage, 6.4% novelty, 25,202 LOC at iteration 30
- **Claude-Code (unstructured)**: 59.6% coverage, 0.0% novelty, 3,559 LOC at iteration 30

The EFT acts as an **accelerator** rather than a prerequisite: ZeroRepo without EFT reaches 97.9% coverage by iteration 40, matching the EFT-enhanced version at iteration 30. Both configurations exhibit robust linear growth in LOC over iterations:

- **EFT-enhanced**: $y \approx 983x + 2992$, $R^2 = 0.97$
- **EFT-free**: $y \approx 800x + 989$, $R^2 = 0.98$

This indicates the **RPG structure ensures stable planning**, with EFT optimizing the initial context (intercept) and generation velocity (slope).

**Why Topological Order?**

The topological traversal is critical for maintaining dependency coherence:

- If code were generated in arbitrary order, later modules might reference classes or functions that do not yet exist, causing import errors and broken dependencies.
- The RPG's data-flow edges explicitly encode the dependency graph, making topological sorting straightforward.
- This mirrors how human developers build systems: foundational components first, then higher-level abstractions.

**Why Test-Driven Development?**

TDD at leaf nodes serves multiple purposes:

- **Early error detection**: Failing tests trigger immediate revision, preventing error accumulation.
- **Incremental validation**: Each component is verified before being used as a dependency for downstream components.
- **Regression prevention**: When components are modified, regression tests ensure backward compatibility.

The paper reports that automated evaluation achieves strong Pearson correlations with human scores (0.893 for coverage, 0.958 for novelty), indicating the test-based evaluation is reliable (Table 5).

**Why 30 Iterations?**

The 30-iteration budget for feature selection balances coverage and efficiency:

- Figure 5 shows ZeroRepo exhibits near-linear growth, surpassing 1,100 leaf features with o3-mini at iteration 30.
- Natural language baselines show limited scalability: Claude Code grows steadily but with diminishing rates, Gemini CLI converges by round 30, and Codex ceases features after 4–5 iterations.
- Extending beyond 30 iterations yields diminishing returns for EFT-enhanced ZeroRepo (Figure 7 shows coverage plateaus near 96%).

## 4. Key Insights and Innovations

This section identifies the fundamental innovations that distinguish ZeroRepo from prior work. Rather than incremental improvements to existing frameworks, the paper introduces architectural changes that enable capabilities previously unattainable with natural language planning.

---

### 4.1 Innovation 1: RPG as a Persistent Planning Substrate (Fundamental)

**What Prior Work Does:**
All three paradigms discussed in Section 2—distributed planning frameworks, workflow-based systems, and iterative terminal agents—rely on **ephemeral natural language plans**. These plans exist only within individual iterations or conversation turns. When the agent moves to the next step, the plan must be reconstructed from context or re-summarized, leading to information loss and drift.

**What ZeroRepo Does Differently:**
The Repository Planning Graph serves as a **persistent, evolvable substrate** that survives across all 30+ iterations of the generation process. Once a node or edge is added to the RPG, it remains as an anchor that constrains and guides subsequent decisions. This is not merely a different data structure—it represents a shift from *stateless* to *stateful* planning.

**Why This Matters:**

The persistence property explains the dramatic scaling difference observed in Figure 6. Natural language baselines plateau at 3–4K lines of code because their plans degrade: each iteration introduces small inconsistencies that accumulate until the system can no longer maintain coherence. The RPG prevents this accumulation by providing a **single source of truth** that all iterations reference.

**Evidence of Impact:**
- ZeroRepo sustains near-linear growth to 30K+ LOC over 30 iterations (Figure 6)
- Claude Code plateaus at ~10K LOC despite using the same iteration budget (Table 2)
- The ablation study (Section 8) shows that removing the EpiCoder Feature Tree slows convergence but does not eliminate scaling—confirming that the **graph structure itself**, not just the knowledge base, enables sustained growth

**Classification:** This is a **fundamental innovation** because it changes the theoretical limits of what automated code generation can achieve. Prior systems were bounded by context window degradation and plan drift; RPG removes this bound by externalizing the plan into a persistent structure.

### 4.2 Innovation 2: Dual-Semantics Node Design (Fundamental)

**What Prior Work Does:**
Existing frameworks separate proposal-level planning ("what to build") from implementation-level planning ("how to build it"). Paper2Code, for example, follows a fixed three-stage pipeline where architectural decisions are made first, then implementation details are filled in separately. This separation creates a **representation gap**: the functional specification does not directly map to code structure, requiring translation that introduces errors.

**What ZeroRepo Does Differently:**
As detailed in Section 3.4.1, each RPG node carries **dual semantics**: it simultaneously represents a functional capability (e.g., "linear regression") AND a structural element (e.g., `src/algos/linear.py`). This is not a mapping between two representations—it is a **unified representation** where functionality and structure are two views of the same graph.

**Why This Matters:**

The dual-semantics design eliminates the translation layer that causes inconsistencies in prior work. When the LLM decides to add a "clustering metrics" feature, it simultaneously decides where that feature lives in the file hierarchy and how it connects to data flows. There is no opportunity for the functional plan and structural plan to diverge because they are the same object.

**Evidence of Impact:**
- Figure 2 shows how the ML Algorithms module decomposes into leaf nodes that directly correspond to classes like `LassoRegression` and `ElasticNet`
- Table 9 demonstrates that subgraphs map isomorphically to code regions: the "ML Algorithms" subgraph produces 58 files, 171 classes, and 67 functions with coherent feature density (5.57 features/file)
- The 81.5% functional coverage (Table 2) indicates that proposed features are actually implemented, not lost in translation

**Classification:** This is a **fundamental innovation** because it solves the representation alignment problem that has plagued multi-stage code generation systems. Prior work treats functionality and structure as separate concerns; ZeroRepo proves they can be unified without loss of expressiveness.

---

### 4.3 Innovation 3: Near-Linear Scaling as an Emergent Property (Empirical Discovery)

**What Prior Work Shows:**
Previous studies of LLM-based code generation report performance on fixed tasks (single functions, file edits, or small repositories). None have systematically measured how capability scales with iteration count because their planning representations do not support sustained growth.

**What ZeroRepo Reveals:**
Figures 5 and 6 show that ZeroRepo exhibits **near-linear scaling** in both feature count and lines of code across 30 iterations. This is not a design goal stated upfront—it is an emergent property discovered through evaluation. The linear fit for EFT-enhanced generation is $y \approx 983x + 2992$ with $R^2 = 0.97$ (Section 8), indicating remarkably stable growth.

**Why This Matters:**

This discovery has theoretical implications for long-horizon planning. It suggests that the bottleneck in repository generation is not model capability but **planning representation stability**. When the plan is encoded in a graph with explicit dependencies, the system can continue adding functionality without the coherence degradation that plagues natural language planning.

**Evidence of Impact:**
- ZeroRepo generates repositories 3.9× larger than Claude Code and 68× larger than other baselines (Abstract, Table 2)
- Feature count grows from ~200 at iteration 5 to 1,100+ at iteration 30 (Figure 5)
- Natural language baselines show diminishing returns: Codex CLI ceases adding features after 4–5 iterations

**Classification:** This is an **empirical discovery with theoretical implications**. The paper does not claim to have designed for linear scaling; rather, it discovered that structured planning enables scaling that was previously thought to require human oversight.

### 4.4 Innovation 4: Graph-Guided Localization for Maintenance (Practical Innovation)

**What Prior Work Does:**
Existing code generation systems focus on initial generation. Debugging and localization are typically handled through ad hoc methods: searching file contents, reading error messages, or asking the LLM to "find the bug." These approaches scale poorly as repositories grow because they lack global structural awareness.

**What ZeroRepo Does Differently:**
As shown in Table 4 and Figure 15, ZeroRepo leverages the RPG for **localization during debugging and iteration**. The graph provides a global index that allows the agent to navigate from functionality to code location systematically. The localization workflow follows a CCG pattern (Coarse Search → Content Inspection → Global Graph Exploration) that reduces effort by 30–50% compared to graph-free approaches.

**Why This Matters:**

This innovation extends the value of RPG beyond initial generation into the **maintenance lifecycle**. Real software systems spend more time in maintenance than in initial development. By encoding the repository's functional structure in a graph, ZeroRepo enables efficient navigation and modification that would be difficult with text-based plans.

**Evidence of Impact:**
- Integration testing localization drops from 13.3 steps (without graph) to 6.2 steps (with graph) (Table 4)
- Debugging localization drops from 8.5 steps to 5.8 steps
- Figure 15 shows systematic search patterns rather than random probing across all six repositories

**Classification:** This is a **practical innovation** that demonstrates RPG's utility beyond the core generation task. It suggests that structured representations may be valuable for any long-lived codebase that requires iterative modification.

### 4.5 Innovation 5: RepoCraft Benchmark Design (Infrastructure Contribution)

**What Prior Work Lacks:**
As noted in Section 2, prior benchmarks target either incremental development (editing existing codebases) or constrained generation (with detailed skeletons provided). None evaluate end-to-end repository generation from high-level specifications with comprehensive functional and correctness metrics.

**What ZeroRepo Contributes:**
RepoCraft fills this gap with six real-world projects, 1,052 tasks, and multi-dimensional evaluation (coverage, novelty, accuracy, scale). The benchmark's design—deriving tasks from reference repository test suites while anonymizing project names to prevent memorization—establishes a rigorous evaluation protocol for repository-level generation.

**Why This Matters:**

Without appropriate benchmarks, progress in repository generation cannot be measured reliably. RepoCraft provides the infrastructure needed to compare future systems on the same tasks, enabling cumulative research progress rather than isolated demonstrations.

**Evidence of Impact:**
- Table 1 shows the benchmark spans diverse domains (ML, data analysis, web frameworks, symbolic computation)
- Table 5 demonstrates automated evaluation achieves strong Pearson correlation with human judgments (0.893 for coverage, 0.958 for novelty)
- The benchmark reveals gaps invisible to prior evaluations: Claude Code achieves 54.2% coverage but only 33.9% pass rate, indicating many proposed features are not correctly implemented

**Classification:** This is an **infrastructure contribution** that enables future research. While not a technical innovation in the algorithmic sense, it is essential for establishing repository generation as a rigorous research area.

## 5. Experimental Analysis

This section examines how the authors validate their claims about RPG and ZeroRepo. The experimental design is comprehensive, targeting three core assertions: (1) structured graph planning outperforms natural language planning, (2) the system scales near-linearly with iteration count, and (3) the evaluation methodology reliably captures repository-level generation quality. I walk through the benchmark construction, baseline selection, quantitative results, and ablation studies to assess whether the evidence supports these claims.

### 5.1 Evaluation Methodology

#### 5.1.1 The RepoCraft Benchmark

The paper introduces **RepoCraft**, a new benchmark designed specifically for end-to-end repository generation evaluation. This addresses a critical gap: prior benchmarks either focus on incremental tasks (editing or debugging existing code) or provide detailed skeletons that limit autonomous planning.

**Benchmark Composition (Table 1):**

RepoCraft comprises **six real-world Python projects** spanning diverse domains:

| Original Project | Anonymized Name | Domain | # Files | LOC | Code Tokens | Tasks |
|-----------------|-----------------|--------|---------|-----|-------------|-------|
| scikit-learn | MLKit-Py | Machine Learning | 185 | 65,972 | 592,187 | 236 |
| pandas | TableKit | Data Analysis | 217 | 106,447 | 943,873 | 175 |
| sympy | SymbolicMath | Symbolic Computation | 699 | 218,924 | 2,339,881 | 192 |
| statsmodels | StatModeler | Statistical Modeling | 271 | 83,325 | 893,824 | 234 |
| requests | HttpEasy | HTTP Client | 17 | 2,793 | 22,297 | 50 |
| django | PyWebEngine | Web Framework | 681 | 109,457 | 917,622 | 165 |
| **Total** | — | — | **2,070** | **586,918** | **5,709,884** | **1,052** |

**Key Design Choices:**

1. **Anonymization**: Project names and descriptions are paraphrased before providing them to agents. This mitigates pretraining leakage—models cannot simply retrieve memorized implementations from training data.

2. **Task Derivation (Figure 3)**: Tasks are extracted from reference repository test suites through a four-step pipeline:
   - **Test file filtering**: Remove trivial or non-algorithmic tests
   - **Hierarchical parsing**: Organize tests following each project's modular structure
   - **Stratified sampling**: Ensure representative coverage across functional categories
   - **Task generation**: Create natural language descriptions with ground-truth tests

3. **Evaluation Dimensions**: Each task includes (i) a natural language description of the target algorithm, (ii) a ground-truth test, and (iii) auxiliary materials for interface matching.

**Why This Matters:**

The benchmark's scale is notable: **1,052 tasks** across **six domains** provides statistical power for detecting meaningful differences between methods. The anonymization protocol ensures that performance reflects genuine planning capability rather than memorization—a critical consideration given that models like Claude and GPT were likely trained on these open-source repositories.

#### 5.1.2 Metrics

RepoCraft evaluates generated repositories along **four complementary dimensions** (Section 5.1.2, Appendix E.3.1):

**Functionality Metrics:**

1. **Coverage**: The proportion of functional categories defined in official documentation that are represented in the generated repository.
   $$
   \text{Coverage} = \frac{1}{|C|} \sum_{j=1}^{K} \mathbb{1}[\exists g_i \in G, f(g_i) = c_j]
   $$
   where $C$ is the reference category set, $G$ is generated functionalities, and $f$ maps generated features to categories.

2. **Novelty**: The proportion of generated functionalities **not** in the reference taxonomy, capturing the system's ability to propose coherent new capabilities.
   $$
   \text{Novelty} = \frac{1}{|G|} \sum_{i=1}^{N} \mathbb{1}[f(g_i) = c_{\text{OOD}}]
   $$
   where $c_{\text{OOD}}$ is the out-of-distribution centroid.

**Accuracy Metrics:**

3. **Pass Rate**: The fraction of task-level tests that pass execution.

4. **Voting Rate**: The fraction of tasks confirmed by **5-round majority-vote checks**, which separates genuine implementation errors from environment or test issues.

**Code-Level Statistics:**

5. **File Count**: Number of valid source files (excluding tests and examples).

6. **Normalized LOC**: Effective lines of code after removing comments, docstrings, and blank lines.

7. **Token Count**: Number of tokens in normalized code, measured with a standard tokenizer.

**Why Multiple Metrics:**

Using only coverage or only accuracy would give an incomplete picture. A system could achieve high coverage by generating many shallow features, or high accuracy by implementing only a few simple functions. The combination reveals whether systems generate **both** comprehensive **and** correct repositories.

#### 5.1.3 Baselines and Experimental Setup

**Baseline Categories (Section 5.2):**

The paper compares three paradigms plus terminal agents:

| Category | Systems | Backbone Models |
|----------|---------|-----------------|
| Multi-agent Frameworks | MetaGPT, ChatDev | o3-mini, Qwen3-Coder-480B-A35B-Instruct |
| Workflow-Based | Paper2Code | o3-mini, Qwen3-Coder |
| Terminal Agents | OpenHands, Codex CLI, Gemini CLI, Claude Code CLI | Official strongest models |

**Configuration Details (Section 5.3):**

- **Iteration Budget**: All methods run for **30 iterations** to ensure fair comparison
- **Feature Selection**: 30 iterations for proposal-level graph construction
- **Node Embedding**: `infly/inf-retriever-v1` for Feature Tree retrieval
- **Debugging**: Up to **8 iterations** per function, **20 localization attempts**
- **Test Validation**: **5-round majority voting** for error attribution, up to **20 remediation attempts**
- **Terminal Agents**: Enabled with web search for real-world knowledge retrieval

**Evaluation Model:**

The automated evaluation pipeline uses **o3-mini** as the base model for localization, validation, and judgment. The paper validates this choice against human annotators (discussed in Section 5.4).

### 5.6 Critical Assessment: Do the Experiments Support the Claims?

#### 5.6.1 Claim 1: Structured Graph Planning Outperforms Natural Language

**Evidence Strength: Strong**

The coverage gap (81.5% vs. 54.2%), accuracy gap (69.7% vs. 33.9%), and scale gap (3.9× LOC) are substantial and consistent across all six repositories. The ablation study (Section 8) isolates the RPG structure as the primary driver, not the EpiCoder Feature Tree. The manual validation (Table 15) confirms the gaps reflect real capability differences, not evaluation artifacts.

**Potential Concerns:**

- All baselines use the same 30-iteration budget, but natural language methods may benefit from different iteration strategies. The paper does not explore iteration tuning for baselines.
- Terminal agents (Claude Code, Gemini CLI) use proprietary models that may have different optimization objectives than the research-focused baselines.

#### 5.6.2 Claim 2: Near-Linear Scaling is Enabled by RPG

**Evidence Strength: Strong**

Figures 5 and 6 show near-linear growth with $R^2 > 0.97$ for both EFT-enhanced and EFT-free settings. Natural language baselines plateau early (Codex at iteration 5, Claude Code at ~10K LOC). The ablation confirms the graph structure—not just the knowledge base—enables sustained growth.

**Potential Concerns:**

- Scaling is demonstrated only up to 30 iterations. Whether linear growth continues to 100+ iterations remains unknown.
- The linear fit is reported for MLKit-Py only; per-repository scaling curves (Appendix F.2) show some variance.

#### 5.6.3 Claim 3: Evaluation Methodology is Reliable

**Evidence Strength: Moderate to Strong**

Pearson correlations above 0.89 with human judgments, manual validation of 200 tasks, and test code audits all support the evaluation pipeline's reliability. The consistency across different judge models (DeepSeek, GPT-5, o3-mini) further strengthens confidence.

**Potential Concerns:**

- Human evaluation was performed on a subset of tasks; full human annotation of all 1,052 tasks would be prohibitively expensive but would provide stronger validation.
- The evaluation focuses on functional correctness but does not assess code quality dimensions like readability, maintainability, or adherence to best practices.

#### 5.6.4 Unanswered Questions and Limitations

**What the Experiments Do Not Address:**

1. **Long-Term Maintainability**: The evaluation measures initial generation quality but does not assess how easily generated repositories can be extended or modified by human developers over time.

2. **Domain Generalization**: All six projects are Python repositories. Whether RPG generalizes to other languages (JavaScript, Rust, Go) remains untested.

3. **Computational Cost**: The paper does not report token consumption, API costs, or runtime for ZeroRepo vs. baselines. Generating 3.9× more code likely incurs proportionally higher costs.

4. **Failure Modes**: The paper emphasizes successes but provides limited analysis of failure cases. What types of specifications cause ZeroRepo to fail? Are there domains where RPG provides no benefit?

5. **Human-in-the-Loop**: The evaluation is fully automated. How does ZeroRepo perform when human developers intervene during generation? Does the graph representation facilitate human-AI collaboration?

| **Scaling** | Near-linear growth with $R^2 > 0.97$ over 30 iterations | Figure 5, Figure 6, Section 8 |
| **Ablation** | RPG structure (not EFT) drives 7× volume increase | Table 6, Figures 7–8 |
| **Localization** | Graph guidance reduces effort by 30–50% | Table 4, Figure 15 |
| **Evaluation Reliability** | 0.89–0.96 Pearson correlation with human judgments | Table 5, Tables 13–16 |

**Overall Assessment:**

The experiments provide **convincing evidence** for the paper's core claims. The performance gaps are large, consistent across domains, and validated through multiple ablation studies and robustness checks. The primary limitation is the focus on Python repositories and initial generation quality—future work should address long-term maintainability, cross-language generalization, and computational efficiency.

The RepoCraft benchmark itself is a significant contribution: it establishes a rigorous evaluation protocol that future work can build upon. The combination of coverage, novelty, accuracy, and scale metrics provides a holistic view of repository generation capability that prior benchmarks lacked.

## 6. Limitations and Trade-offs

While ZeroRepo demonstrates substantial improvements over prior work, the paper reveals several important limitations and trade-offs that constrain its applicability and highlight open research questions. I organize these into four categories: computational and resource constraints, domain and language limitations, evaluation scope gaps, and model dependencies.

---

### 6.1 Computational and Resource Constraints

**Token Consumption and Cost:**

The paper reports that ZeroRepo generates repositories **3.9× larger** than the strongest baseline (Claude Code) in terms of lines of code and **68× larger** than other baselines (Table 2). While this demonstrates superior scaling capability, it implies proportionally higher computational costs:

- **ZeroRepo (Qwen3-Coder)**: 445,511 code tokens generated
- **Claude Code**: 105,236 code tokens generated
- **Other baselines**: ~5,000–15,000 code tokens generated

The paper **does not report** API costs, total token consumption (including prompts and intermediate reasoning), or runtime for any method. This is a significant omission because:

1. Generating 3.9× more code likely requires 3.9× more generation tokens
2. The 30-iteration proposal-level construction plus implementation-level encoding adds substantial overhead
3. Each function allows up to **8 debugging iterations** with **20 localization attempts** (Section 5.3), multiplying token consumption for complex repositories

**Trade-off:** Users must decide whether the improved coverage (81.5% vs. 54.2%) and accuracy (69.7% vs. 33.9%) justify the potentially **4–10× higher computational cost**. For rapid prototyping where cost matters more than completeness, natural language baselines may remain preferable.

**Iteration Budget Sensitivity:**

The system is evaluated with a fixed **30-iteration budget** for both proposal construction and baseline comparison. However:

- ZeroRepo without EpiCoder Feature Tree requires **40 iterations** to match the coverage of the EFT-enhanced version at iteration 30 (Section 8)
- Natural language baselines plateau early (Codex CLI ceases features after 4–5 iterations, Figure 5)

This reveals a **hidden dependency**: ZeroRepo's advantages assume sufficient iteration budget. If constrained to 10–15 iterations (e.g., for cost or latency reasons), the coverage gap would narrow significantly (Table 3 shows 70.2% coverage at iteration 5 vs. 95.7% at iteration 30).

**Trade-off:** The RPG structure enables sustained growth, but realizing this benefit requires committing to long-horizon generation. Short-horizon tasks may not fully leverage the structured planning advantage.

---

### 6.2 Domain and Language Limitations

**Python-Only Evaluation:**

All six repositories in RepoCraft are **Python projects** (scikit-learn, pandas, sympy, statsmodels, requests, django). The paper does not evaluate:

- **Statically-typed languages** (Java, Rust, Go, TypeScript) where type constraints add planning complexity
- **Multi-language repositories** (e.g., Python backend with JavaScript frontend)
- **Domain-specific languages** or configuration-heavy systems (e.g., Kubernetes manifests, Terraform)

The RPG's design assumptions may not generalize:

1. **File-to-function mapping**: Python's flexible module structure differs from Java's strict class-per-file convention or Rust's module system
2. **Import resolution**: Python's dynamic imports contrast with compile-time dependency resolution in compiled languages
3. **Build systems**: The paper does not address language-specific build configurations (Maven, Cargo, webpack) that affect repository structure

**Open Question:** Does the dual-semantics node design (functionality + structure) transfer to languages where structural constraints are more rigid? This remains untested.

**Domain Coverage:**

While the six projects span diverse domains (ML, data analysis, web frameworks, symbolic computation, HTTP clients), they share characteristics that may bias results:

- **Well-documented**: All have comprehensive test suites and official documentation, enabling reliable task extraction
- **Mature**: These are established projects with stable APIs, not rapidly evolving codebases
- **Library-focused**: All are libraries/frameworks rather than end-user applications with UI components or deployment configurations

**Limitation:** The benchmark does not include:
- **Application repositories** with frontend/backend separation
- **Microservice architectures** with inter-service contracts
- **Legacy codebases** with technical debt or inconsistent patterns
- **Real-time or embedded systems** with timing constraints

The paper's claims about "repository-level generation" should be interpreted as **library-level generation** for well-structured Python projects. Generalization to other repository types remains an open question.

---

### 6.3 Evaluation Scope Gaps

**Test Generation as Bottleneck:**

The paper explicitly identifies test generation quality as a limiting factor. Table 11 and Figure 14 reveal:

| Model | Success Rate | Test Coverage |
|-------|-------------|---------------|
| o3-mini | 75–90% | 59–65% |
| Qwen3-Coder | 50–55% | 45–48% |

**Key Observation:** Test coverage **declines as code length increases** (Figure 14). Shorter implementations reach high class-level coverage, but function-level and overall coverage drop significantly with greater complexity.

The paper states: *"These results suggest that while current models are increasingly effective at generating functional code, their ability to produce comprehensive and high-quality test cases remains limited, highlighting test generation as a key bottleneck for practical deployment"* (Section D.5).

**Implication:** ZeroRepo's 69.7% pass rate may be **test-limited** rather than code-limited. Some generated code may be correct but fail tests due to inadequate test adaptation (though Table 17 shows 90–91% test adaptation correctness).

**Code Quality Dimensions Not Assessed:**

The evaluation focuses on functional correctness but does not measure:

1. **Readability**: Variable naming, code structure, comment quality
2. **Maintainability**: Ease of modification by human developers
3. **Best practices**: Adherence to language idioms, design patterns, security considerations
4. **Performance**: Runtime efficiency, memory usage, scalability

**Limitation:** A repository could achieve 81.5% coverage and 69.7% pass rate while producing code that is difficult for humans to understand or extend. The paper does not address whether RPG-guided generation produces more or less maintainable code than natural language planning.

**Long-Term Maintainability:**

The evaluation measures **initial generation quality** but does not assess:

- How easily generated repositories can be **extended** with new features
- How well the code supports **debugging** by human developers
- Whether the graph representation facilitates **human-AI collaboration** during maintenance

**Open Question:** Does the persistent RPG structure enable better long-term maintainability, or does it encode design decisions that constrain future evolution? The paper's localization analysis (Table 4) suggests RPG aids debugging, but this is measured during automated generation, not human-led maintenance.

---

### 6.4 Model and Knowledge Base Dependencies

**Model Backbone Trade-offs:**

ZeroRepo is evaluated with two backbones, revealing an **accuracy-scale trade-off** (Section 5.5.2):

| Model | Coverage (%) | Pass Rate (%) | LOC | Tokens |
|-------|--------------|---------------|-----|--------|
| o3-mini | 81.5 | 69.7 | 23,977 | 260,761 |
| Qwen3-Coder | 75.1 | 57.3 | 36,941 | 445,511 |

**Interpretation:** The RPG structure enables scaling for both models, but **model capability affects the accuracy-scale trade-off**. o3-mini produces more correct code per line, while Qwen3-Coder generates larger but less accurate repositories.

**Implication:** ZeroRepo **amplifies model capabilities but does not eliminate model differences**. Users must choose between:
- Higher accuracy with smaller repositories (o3-mini)
- Larger repositories with lower accuracy (Qwen3-Coder)

This dependency means ZeroRepo's absolute performance is bounded by the underlying model's code generation capability.

**EpiCoder Feature Tree Dependency:**

While the ablation study (Section 8) shows the RPG structure is the fundamental innovation, the EpiCoder Feature Tree (EFT) provides meaningful acceleration:

| Setting | Coverage at Iteration 30 | LOC at Iteration 30 |
|---------|-------------------------|---------------------|
| With EFT | 95.7% | 31,596 |
| Without EFT | 87.2% | 25,202 |

**Trade-off:** The EFT adds **8.5 percentage points coverage** and **6,394 LOC** at iteration 30. However:
- The EFT contains **1.5 million nodes** (Table 7), requiring significant storage and retrieval infrastructure
- Node embedding uses `infly/inf-retriever-v1`, adding dependency on external models
- The feature distribution is **highly skewed** (Figure 10, Appendix B.2), with data processing dominating and many categories having few nodes

**Limitation:** The EFT's long-tail distribution may bias generation toward common patterns (data processing, utilities) while underrepresenting specialized functionalities. The paper notes this requires "large-scale filtering and reorganization" to mitigate (Appendix B.2), but does not quantify residual bias.

**Localization Efficiency Variance:**

Table 10 (Appendix D.5) shows localization efficiency varies by model and task:

| Model | Integration Testing | Incremental Development | Debugging |
|-------|--------------------|------------------------|-----------|
| o3-mini | 13.33 ± 2.92 | 12.30 ± 5.19 | 11.59 ± 5.74 |
| Qwen3-Coder | 6.16 ± 2.37 | 6.81 ± 1.87 | 6.75 ± 2.21 |

**Observation:** o3-mini achieves higher localization efficiency but with **larger variance** (standard deviation 2.92–5.74 vs. 1.77–2.42 for Qwen3-Coder).

**Implication:** Model capacity and stability jointly influence localization performance. The RPG structure provides guidance, but model differences affect how effectively agents use that guidance.

### 6.5 Open Questions and Future Directions

**Unanswered Questions:**

The paper explicitly identifies several open questions (Section 5.6.4):

1. **Long-Horizon Scaling Beyond 30 Iterations**: Whether linear growth continues to 100+ iterations remains unknown. The paper demonstrates scaling to 30 iterations but does not test further.

2. **Cross-Language Generalization**: Whether RPG generalizes to languages beyond Python is untested.

3. **Human-in-the-Loop Performance**: How ZeroRepo performs when human developers intervene during generation is not evaluated. Does the graph representation facilitate human-AI collaboration?

4. **Failure Mode Analysis**: The paper emphasizes successes but provides limited analysis of failure cases. What types of specifications cause ZeroRepo to fail? Are there domains where RPG provides no benefit?

5. **Computational Efficiency**: Token consumption, API costs, and runtime are not reported, making it difficult to assess practical deployment feasibility.

**Research Directions:**

Based on the paper's findings, promising directions include:

1. **Test Generation Improvement**: Since test coverage declines with code length (Figure 14), dedicated research on LLM-based test generation could unlock higher pass rates.

2. **Multi-Language RPG**: Extending the graph structure to handle language-specific constraints (types, build systems, deployment configurations).

3. **Human-AI Collaboration Tools**: Developing interfaces that allow developers to inspect, modify, and extend the RPG during generation.

4. **Cost Optimization**: Exploring whether selective graph expansion (focusing RPG detail on critical modules) can reduce token consumption while maintaining quality.

5. **Long-Term Maintenance Studies**: Evaluating how RPG-guided repositories perform when extended by human developers over weeks or months.

## 7. Implications and Future Directions

This section examines how ZeroRepo and the Repository Planning Graph reshape the research landscape for automated code generation, identifies concrete follow-up research directions enabled by this work, outlines practical deployment scenarios, and provides guidance on when to adopt this approach versus alternatives.

---

### 7.1 Paradigm Shift: From Ephemeral Plans to Persistent Structures

**The Fundamental Change:**

ZeroRepo establishes that **planning representation stability**, not model capability, is the primary bottleneck for long-horizon code generation. This reframes a central assumption in the field: prior work treated LLM context degradation and plan drift as inherent limitations of language models. ZeroRepo demonstrates these are limitations of **natural language as a planning medium**.

**Why This Matters for the Field:**

The near-linear scaling observed in Figures 5 and 6 ($R^2 > 0.97$ over 30 iterations) reveals that repository generation is not fundamentally bounded by model reasoning capacity. Instead, the bottleneck is **representation coherence**—the ability to maintain consistent architectural decisions across hundreds of dependent generation steps.

**Three Paradigm Shifts:**

| Prior Assumption | ZeroRepo's Demonstration | Implication |
|-----------------|-------------------------|-------------|
| LLMs cannot sustain coherent planning beyond ~10 iterations | RPG enables 30+ iterations with stable growth | Planning horizon is a **representation choice**, not a model limitation |
| Natural language is sufficient for intermediate planning | Graph structures outperform text by 27.3 points coverage (Table 2) | Structured representations should replace free-form planning for complex tasks |
| Repository generation requires detailed human specifications | ZeroRepo achieves 81.5% coverage from high-level queries | Autonomous architectural decision-making is feasible with proper grounding |

**Theoretical Implications:**

This work suggests that **externalized state**—maintaining planning artifacts outside the model's context window—is critical for long-horizon tasks. The RPG functions as what cognitive science terms a "cognitive scaffold": an external structure that offloads working memory demands and enables reasoning beyond immediate context limits.

**Research Community Impact:**

Future work on autonomous agents should consider:
1. **What persistent structures** (graphs, databases, knowledge bases) can replace ephemeral conversation history?
2. **What operations** (traversal, update, query) should these structures support for agent interaction?
3. **What consistency constraints** (topological ordering, type checking, dependency validation) should be enforced during construction?

ZeroRepo provides a concrete instantiation of this paradigm for code generation; the broader question is whether similar structured representations enable long-horizon reasoning in other domains (scientific discovery, business process automation, legal document generation).

### 7.2 Concrete Follow-Up Research Directions

The paper's findings and limitations point to specific, actionable research questions. I organize these by technical area with concrete hypotheses that could be tested.

#### 7.2.1 Test Generation and Validation

**The Bottleneck:**

Section 5.6.3 and Figure 14 identify test generation as the limiting factor: test coverage declines as code length increases, and pass rates (69.7%) lag behind coverage (81.5%). This suggests some generated code is correct but fails due to inadequate test adaptation.

**Research Questions:**

1. **Can dedicated test-generation models improve pass rates?**
   - *Hypothesis*: Training a model specifically on test-code pairs (rather than general code) could improve test adaptation correctness beyond the current 90–91% (Table 17).
   - *Method*: Fine-tune on repositories with high-quality test suites; evaluate on RepoCraft tasks.

2. **Can property-based testing complement example-based tests?**
   - *Hypothesis*: Generating invariant-based tests (e.g., "output shape matches input shape") rather than specific input-output pairs could improve coverage for edge cases.
   - *Method*: Extend the RPG to encode type signatures and invariants; generate property tests from these specifications.

3. **Can test execution feedback improve code generation iteratively?**
   - *Hypothesis*: Using test failure messages as direct training signals (not just for debugging) could improve initial code quality.
   - *Method*: Implement a reinforcement learning loop where test pass/fail rewards guide code generation policy updates.

#### 7.2.2 Cross-Language Generalization

**The Gap:**

All six RepoCraft repositories are Python projects (Table 1). The RPG's design assumptions—flexible module structure, dynamic imports, minimal build configuration—may not transfer to languages with stricter constraints.

**Research Questions:**

1. **How should RPG encode type systems?**
   - *Hypothesis*: For statically-typed languages (Java, Rust, TypeScript), the RPG must encode type signatures at node level, not just interface names.
   - *Method*: Extend leaf nodes to include type annotations; validate that generated code compiles without type errors.

2. **How should RPG encode build and deployment configurations?**
   - *Hypothesis*: Multi-language repositories require RPG nodes for build files (`pom.xml`, `Cargo.toml`, `package.json`) and deployment configs (Docker, Kubernetes).
   - *Method*: Add configuration subgraphs to RPG; evaluate on repositories with complex build pipelines.

3. **Does RPG benefit vary by language paradigm?**
   - *Hypothesis*: Object-oriented languages (Java) may benefit more from RPG's class hierarchy encoding than functional languages (Haskell).
   - *Method*: Construct cross-language RepoCraft benchmarks; compare RPG vs. natural language planning gains by language.

#### 7.2.3 Human-AI Collaboration

**The Open Question:**

ZeroRepo is evaluated as a fully autonomous system. Real-world deployment will involve human developers inspecting, modifying, and extending generated code. The paper does not address how RPG facilitates human-AI collaboration.

**Research Questions:**

1. **Can developers navigate RPG more efficiently than raw code?**
   - *Hypothesis*: A visual RPG interface allows developers to understand repository structure faster than reading file hierarchies.
   - *Method*: User study comparing time-to-comprehension for RPG visualization vs. IDE file tree.

2. **Can humans modify RPG to guide regeneration?**
   - *Hypothesis*: Allowing developers to edit RPG nodes (add/remove features, change data flows) and regenerate code produces more targeted modifications than editing code directly.
   - *Method*: Build an RPG editor; measure modification accuracy and effort vs. direct code editing.

3. **Does RPG preserve design intent across iterations?**
   - *Hypothesis*: When humans specify architectural constraints in RPG (e.g., "use microservices pattern"), these constraints persist across regeneration cycles better than text-based specifications.
   - *Method*: Iterative modification study tracking constraint preservation over 10+ cycles.

#### 7.2.4 Computational Efficiency

**The Cost Question:**

ZeroRepo generates 3.9× more code than Claude Code (Table 2), implying proportionally higher token consumption. The paper does not report costs, but practical deployment requires efficiency optimization.

**Research Questions:**

1. **Can selective graph expansion reduce costs?**
   - *Hypothesis*: Expanding RPG detail only for critical modules (not all 1,258 features at iteration 30) maintains quality with fewer tokens.
   - *Method*: Implement importance-weighted graph expansion; measure coverage/cost trade-off.

2. **Can smaller models achieve comparable results with RPG?**
   - *Hypothesis*: RPG structure compensates for model capability, allowing 7B-parameter models to match 100B+ models with natural language planning.
   - *Method*: Evaluate ZeroRepo with models from 7B to 400B parameters; plot accuracy vs. model size curve.

3. **Can caching and reuse reduce redundant generation?**
   - *Hypothesis*: Common patterns (e.g., CRUD operations, standard algorithms) can be cached and reused across repositories, reducing generation tokens.
   - *Method*: Build a pattern library from generated repositories; measure token savings on new tasks.

#### 7.2.5 Long-Term Maintainability

**The Unanswered Question:**

The evaluation measures initial generation quality but does not assess how easily generated repositories can be extended months later by human developers.

**Research Questions:**

1. **Do RPG-guided repositories have lower technical debt?**
   - *Hypothesis*: Explicit dependency encoding in RPG produces more modular code with fewer circular dependencies and tighter cohesion.
   - *Method*: Static analysis of generated repositories comparing coupling metrics (afferent/efferent coupling, instability) vs. baselines.

2. **Can RPG support incremental feature addition?**
   - *Hypothesis*: Adding features to an existing RPG-guided repository requires fewer modification steps than adding to natural-language-planned repositories.
   - *Method*: Measure steps-to-integration for 10 new features added to ZeroRepo vs. Claude Code repositories.

3. **Does RPG documentation improve onboarding?**
   - *Hypothesis*: The RPG itself serves as living documentation, reducing time for new developers to understand repository structure.
   - *Method*: Onboarding study measuring time-to-first-contribution for developers using RPG visualization vs. traditional documentation.

### 7.3 Practical Applications and Deployment Scenarios

ZeroRepo's capabilities suggest several concrete use cases where structured graph planning provides distinct advantages over natural language approaches.

#### 7.3.1 Rapid Prototyping of Libraries and Frameworks

**Use Case:**

A team needs to build a new machine learning library with specific algorithms (e.g., custom clustering methods, novel evaluation metrics) but lacks bandwidth for full implementation from scratch.

- **Coherence requirement**: Algorithms must share common interfaces (`BaseEstimator`, `fit`/`predict` methods)—RPG's dual-semantics nodes ensure interface consistency (Section 3.4.1).

**Expected Outcome:**

Based on MLKit-Py results (Table 18), expect ~30K LOC with 70%+ test pass rate, covering 85%+ of specified functionality. Human developers can then refine the remaining 30% manually.

#### 7.3.2 Legacy System Modernization

**Use Case:**

An organization has a legacy codebase with poor documentation and inconsistent patterns. They want to regenerate core functionality with modern architecture while preserving behavior.

**Why ZeroRepo Fits:**

- **Localization capability**: RPG-guided search reduces localization effort by 30–50% (Table 4), enabling efficient mapping from legacy functions to new structures.
- **Test preservation**: Existing tests can be adapted to the new RPG structure, ensuring behavioral equivalence.
- **Architecture enforcement**: RPG encodes desired architectural patterns (microservices, clean architecture) that the legacy system lacks.

**Expected Outcome:**

Based on localization results (Table 10), expect 6–8 steps per function for mapping legacy to new structure, compared to 10–13 steps without graph guidance.

#### 7.3.3 Educational Code Generation

**Use Case:**

An educational platform wants to generate example repositories demonstrating specific concepts (e.g., "build a REST API with authentication" or "implement a neural network from scratch").

**Why ZeroRepo Fits:**

- **Coverage control**: Instructors can specify exact functional categories to cover; ZeroRepo achieves 81.5% coverage on specified requirements (Table 2).

- **Test inclusion**: Generated tests serve as grading rubrics and learning aids.

**Expected Outcome:**

#### 7.3.4 Research Code Reproduction

**Use Case:**

A researcher wants to reproduce results from a paper that describes algorithms but provides no code implementation.

**Why ZeroRepo Fits:**

- **Paper2Code lineage**: ZeroRepo builds on Paper2Code's approach but achieves 3× better coverage (81.5% vs. 30.2% for Paper2Code in Table 2).
- **Algorithm specification**: The RPG can encode algorithm descriptions from papers as functionality nodes.
- **Test validation**: Generated tests verify that implementations match described behavior.

**Expected Outcome:**

Based on SymbolicMath results (Table 22), expect 75%+ coverage of described algorithms with 65%+ pass rate—sufficient for initial reproduction, with manual refinement for publication-quality code.

### 7.5 Summary: The Path Forward

ZeroRepo establishes that **structured graph planning** is a foundational capability for large-scale code generation, not an incremental optimization. The key takeaways for researchers and practitioners are:

**For Researchers:**
1. **Investigate persistent structures** for other long-horizon tasks (scientific workflows, business process automation)
2. **Address test generation** as the current bottleneck limiting pass rates
3. **Extend RPG to other languages** and repository types (applications, microservices)
4. **Study human-AI collaboration** patterns with graph-based planning interfaces

**For Practitioners:**
1. **Use ZeroRepo for large-scale library generation** where natural language planning fails
2. **Budget for 30+ iterations** to realize RPG's scaling benefits
3. **Plan for manual review** of ~30% of generated code (based on 69.7% pass rate)
4. **Consider accuracy-scale trade-off** when selecting model backbones (o3-mini vs. Qwen3-Coder)

**For the Field:**

This work shifts the conversation from "Can LLMs generate repositories?" to "What planning representations enable LLMs to generate repositories reliably?" The answer—for code generation—is structured graphs with explicit dependencies. Whether this generalizes to other domains is one of the most important open questions emerging from this research.

The RepoCraft benchmark (1,052 tasks across 6 domains) provides the infrastructure for cumulative progress. Future work should build on this foundation rather than creating new evaluation protocols, enabling direct comparison and clear measurement of advances beyond ZeroRepo's 81.5% coverage and 69.7% pass rate ceiling.
