## 1. Executive Summary

This paper introduces **sleep-time compute**, a paradigm that shifts inference workload from test-time to offline periods by prompting models to pre-process available context ($c$) into a reasoned representation ($c'$) before user queries ($q$) arrive. By applying this method to stateful reasoning benchmarks like **Stateful GSM-Symbolic** and **Stateful AIME**, the authors demonstrate a Pareto improvement that reduces the test-time compute required to achieve baseline accuracy by approximately **5×**, while scaling sleep-time compute further boosts accuracy by up to **13%** and **18%** respectively. Additionally, in multi-query settings using the **Multi-Query GSM-Symbolic** dataset, amortizing these offline computations across related questions decreases the average cost per query by **2.5×**, effectively decoupling reasoning depth from real-time latency constraints.

## 2. Context and Motivation

### The Latency-Cost Bottleneck in Test-Time Scaling
The primary problem this paper addresses is the prohibitive **latency and cost** associated with scaling compute at test-time. Recent advancements in Large Language Models (LLMs) have demonstrated that forcing models to "think" longer—generating extended chains of thought or sampling multiple solutions—significantly improves performance on difficult reasoning tasks (Section 1). This approach, known as **test-time scaling**, has become a key ingredient for solving complex problems in mathematics and coding.

However, this performance gain comes with a severe trade-off:
*   **High Latency:** Users may wait several minutes for a single response as the model generates thousands of tokens of reasoning before producing an answer.
*   **High Cost:** Extended reasoning can cost "up to tens of dollars per query" due to the massive number of tokens generated (Section 1).

The root cause of this inefficiency lies in the standard assumption that inference problems are **stateless**. In the conventional paradigm, the model receives the context ($c$, e.g., a document or codebase) and the user's query ($q$) simultaneously at "test-time." Consequently, every time a user asks a new question about the same context, the model must re-perform all the heavy reasoning from scratch, even if the underlying context hasn't changed. This leads to redundant computation and wasted resources.

### The Reality of Stateful Applications
The paper argues that the stateless assumption is a mismatch for many real-world LLM applications, which are inherently **stateful**. In practice, users often interact with persisted contexts over multiple turns:
*   **Document Question-Answering:** Users ask multiple questions about the same long document.
*   **Coding Agents:** Developers debug and iterate on a single large codebase over many sessions.
*   **Conversational Assistants:** Models must maintain and reason over long dialogue histories.

In these scenarios, the context $c$ is available *before* the specific query $q$ arrives. The current infrastructure fails to leverage this temporal gap, leaving the model "idling" during periods when it could be pre-processing the context. The gap this paper fills is the lack of a mechanism to utilize this idle time to perform useful inference, thereby reducing the burden on the critical path (test-time).

### Limitations of Prior Approaches
Before introducing sleep-time compute, the field primarily relied on two strategies to improve reasoning, both of which have distinct limitations regarding latency and redundancy:

1.  **Sequential Test-Time Scaling:**
    *   *Mechanism:* The model generates a long chain of thought or performs multiple reasoning steps sequentially after receiving the query.
    *   *Limitation:* While effective for accuracy, this directly increases latency linearly with the amount of compute. The user must wait for the entire reasoning trace to complete (Section 2).

2.  **Parallel Test-Time Scaling (e.g., Pass@k):**
    *   *Mechanism:* The model generates $k$ independent solutions simultaneously and selects the best one (often requiring an oracle verifier).
    *   *Limitation:* While this avoids increasing latency, it drastically increases the total compute cost (multiplying it by $k$). Furthermore, it assumes access to a ground-truth verifier at test-time, which is often unrealistic in open-ended tasks. It also fails to reuse information across related queries about the same context (Section 5.1).

3.  **Speculative Decoding:**
    *   *Mechanism:* A smaller model drafts tokens that a larger model verifies, aiming to speed up token generation.
    *   *Limitation:* As noted in Section 2, speculative decoding focuses on accelerating the *decoding* process itself. It does not address the semantic redundancy of re-reasoning about a static context for different queries. Sleep-time compute differs fundamentally because the generated tokens (the pre-processed context $c'$) are used as *input* regardless of the specific user query, acting as a cached reasoning state rather than a speculative draft of the answer.

### Positioning Sleep-Time Compute
This paper positions **sleep-time compute** not as a replacement for test-time scaling, but as an orthogonal dimension that enhances it. The core innovation is the decomposition of the input prompt into **Context ($c$)** and **Query ($q$)**, and the introduction of an offline phase where the model transforms $c$ into a richer representation $c'$ *before* $q$ is known.

*   **From Stateless to Stateful:** Unlike prior work that treats every query as a fresh start, this approach explicitly leverages the persistence of context.
*   **Amortization:** By performing heavy lifting offline, the cost of reasoning is amortized across multiple future queries. As illustrated in **Figure 1**, the model can pre-compute facts (e.g., "there are 200 tennis balls") during "sleep-time," allowing it to answer specific questions (e.g., "how many are marked?") instantly at test-time with minimal additional compute.
*   **Pareto Improvement:** The method aims to shift the efficiency frontier, offering either the same accuracy with significantly lower test-time latency (Section 5.1) or higher accuracy for the same test-time budget by scaling the offline compute (Section 5.2).

In essence, the paper reframes the inference problem from "how do we think faster when asked?" to "how much can we think *before* we are asked?"

## 3. Technical Approach

This section details the mechanism of **sleep-time compute**, a method that decouples context reasoning from query answering by introducing an offline pre-processing phase. Unlike standard inference which treats every interaction as a stateless event, this approach explicitly models the temporal gap between context availability and query arrival, utilizing that gap to generate a compressed, reasoned representation of the context.

### 3.1 Reader orientation (approachable technical breakdown)
The system is an inference pipeline that splits the traditional prompt into a static "context" and a dynamic "query," inserting an offline "sleep" phase where the model pre-computes useful facts about the context before the user ever asks a question. It solves the problem of redundant, high-latency reasoning in stateful applications by shifting the heavy computational load to idle periods, allowing the model to answer complex questions instantly using pre-derived insights rather than re-deriving them from scratch.

### 3.2 Big-picture architecture (diagram in words)
The architecture consists of three distinct logical components operating in a sequential flow:
1.  **Context Ingestion Module:** Receives the raw context $c$ (e.g., a codebase, document, or problem statement) during an idle period and holds it in memory without a user query.
2.  **Sleep-Time Processor ($S$):** An LLM instance prompted to analyze $c$, predict potential relevant questions, and perform deductive reasoning to generate a new, enriched context representation $c'$. This component outputs $c'$ via a specific function call interface (`rethink_memory`).
3.  **Test-Time Responder ($T_b$):** The inference engine that receives the user's actual query $q$ along with the pre-processed context $c'$. It uses a significantly reduced compute budget $b$ to generate the final answer $a$, leveraging the insights already embedded in $c'$.

### 3.3 Roadmap for the deep dive
*   **Formal Problem Definition:** We first define the mathematical distinction between standard test-time compute and the proposed sleep-time formulation to establish the theoretical baseline.
*   **The Sleep-Time Mechanism:** We explain exactly how the model transforms raw context into reasoned context ($c \to c'$), including the specific prompting strategies and function-calling interfaces used.
*   **Test-Time Integration:** We describe how the pre-processed context is injected into the test-time prompt and how the model is instructed to utilize it without redundant recomputation.
*   **Scaling Strategies:** We detail the two methods used to increase sleep-time compute (parallel generations and increased reasoning effort) and how they impact the final representation.
*   **Amortization Logic:** We explain the cost model for multi-query settings, showing how the fixed cost of sleep-time compute is distributed across multiple questions to reduce average latency and expense.

### 3.4 Detailed, sentence-based technical breakdown

#### Formalizing the Shift from Stateless to Stateful Inference
The core technical contribution is a reformulation of the inference function to separate context processing from query answering. In the standard **test-time compute** paradigm, the model $T$ takes a context $c$ and a query $q$ simultaneously and produces an answer $a$ using a compute budget $B$. This is denoted as:
$$T_B(q, c) \to a$$
Here, the model must perform all necessary reasoning about $c$ *after* seeing $q$, meaning that for $N$ different queries on the same context, the reasoning about $c$ is repeated $N$ times.

In contrast, **sleep-time compute** introduces an offline function $S$ that operates solely on the context $c$ to produce a transformed context $c'$. This process is defined as:
$$S(c) \to c'$$
The function $S$ represents any standard test-time scaling technique (such as extended chain-of-thought or parallel sampling) applied specifically to pre-process the context. Once $c'$ is generated, the test-time function $T$ operates with a much smaller budget $b$ (where $b \ll B$) on the pair $(q, c')$:
$$T_b(q, c') \to a$$
The fundamental hypothesis is that $c'$ contains sufficient intermediate reasoning steps and derived facts that $T_b$ can answer $q$ correctly without needing the large budget $B$ required to derive those facts from raw $c$.

#### The Sleep-Time Processing Mechanism
The implementation of $S(c)$ relies on a specialized prompting strategy that forces the model to act as an offline analyst rather than a direct responder. The system prompts the model with the raw context $c$ and instructs it to anticipate potential questions and perform relevant deductions.
*   **Prediction and Deduction:** The prompt explicitly tells the model: "Make as many inferences as possible about the part of the problem you are given so as to help yourself answer the fully problem more quickly once it is given to you later" (Figure 18). The model is encouraged to "predict possible ways the rest of the problem might go and compute results that could be helpful."
*   **Function-Calling Interface:** To structurally separate this offline reasoning from standard generation, the system implements a function-calling interface with two specific tools: `rethink_memory` and `finish_rethinking`.
    *   The `rethink_memory` function accepts a string argument `new_memory` and updates an internal memory block labeled `rethink_memory_block`. The model calls this function iteratively to store intermediate deductions, hypotheses, and derived facts.
    *   The model is permitted to call `rethink_memory` up to **10 times** per context, allowing for multiple steps of refinement before finalizing the pre-processed state.
    *   The `finish_rethinking` function signals the end of the sleep-time phase, locking in the final $c'$.
*   **Representation Format:** Unlike traditional representation learning which modifies internal model weights or activation vectors, sleep-time compute generates $c'$ in the space of **natural language**. The output $c'$ is a text string containing logical conclusions, calculated values, and structural summaries that are concatenated with the user query at test-time.

#### Test-Time Integration and Constraint Enforcement
At test-time, the system constructs the input prompt by combining the user query $q$ with the pre-processed context $c'$. Crucially, the system prompt is modified to enforce the usage of this pre-computed information and prevent redundant work.
*   **Explicit Instruction to Reuse:** The test-time prompt (Figures 12–16) explicitly instructs the model: "You check the 'rethink memory block' for potential questions and answers and intermediate reasoning traces that can help answer the question. You use the information in the 'rethink memory block' to answer the questions rather than thinking on the spot."
*   **Prohibition of Recomputation:** To ensure efficiency, the prompt includes a negative constraint: "Do not recompute anything that already exists in the 'rethink memory block'." This forces the model to treat the sleep-time output as ground truth or established premises, skipping the derivation steps that would normally consume tokens.
*   **Verbosity Control:** To precisely measure the reduction in test-time compute, the authors employ a set of **verbosity-controlled prompts** (Levels 0–4).
    *   **Level 0:** Requires a single sentence answer with no explanation.
    *   **Level 4:** Requires step-by-step reasoning, double-checking numbers from the memory block, and explaining each step aloud.
    *   By varying these levels, the researchers can sweep across different test-time token budgets to construct the Pareto frontier curves shown in Figures 3 and 4.

#### Scaling Sleep-Time Compute
The paper explores two distinct dimensions for scaling the amount of compute applied during the sleep-time phase ($S$), analogous to sequential and parallel scaling at test-time.
*   **Parallel Scaling (Non-Reasoning Models):** For models like GPT-4o and GPT-4o-mini, the system generates $k$ independent versions of the pre-processed context ($c'_1, c'_2, \dots, c'_k$) in parallel. At test-time, these are concatenated into a single input: $c'_{final} = [c'_1; c'_2; \dots; c'_k]$.
    *   **Hyperparameters:** Experiments tested $k \in \{1, 2, 5, 10\}$. Results indicated diminishing returns, with **5 parallel generations** generally outperforming 10, suggesting a limit to how much diverse pre-computed information can be effectively utilized in a single context window (Section 5.2).
*   **Reasoning Effort Scaling (Reasoning Models):** For models with native reasoning capabilities (e.g., o1, o3-mini), the system varies the "reasoning effort" during the sleep-time prompt.
    *   **Configuration:** This is achieved by modifying the system instructions to encourage "low," "medium," or "high" reasoning effort, effectively controlling the length and depth of the chain-of-thought generated before the `finish_rethinking` call.
    *   **Outcome:** Higher reasoning effort during sleep-time shifts the accuracy Pareto curve outward, improving performance by up to **18%** on Stateful AIME compared to low-effort sleep-time (Figure 8).

#### Amortization Across Multi-Query Settings
A critical component of the technical approach is the cost model for scenarios where a single context $c$ supports multiple queries $q_1, q_2, \dots, q_N$. The system leverages the fact that $c'$ is computed once and reused $N$ times.
*   **Cost Function:** The total inference cost is modeled as a linear combination of sleep-time and test-time tokens, weighted by their relative latency/cost impact. The paper assumes test-time tokens are **10× more expensive** than sleep-time tokens ($t=10$) due to the strict latency requirements of interactive applications (Section 5.3).
    $$ \text{Total Cost} = \text{Cost}(S(c)) + \sum_{i=1}^{N} t \cdot \text{Cost}(T_b(q_i, c')) $$
*   **Amortization Effect:** As $N$ increases, the fixed cost of generating $c'$ is spread over more queries. The paper demonstrates that with **10 queries per context**, the average cost per query decreases by **2.5×** compared to a baseline where every query incurs the full test-time reasoning cost (Figure 9).
*   **Dataset Construction:** To validate this, the authors created the **Multi-Query GSM-Symbolic** dataset. Using `o3-mini`, they synthetically generated additional question-answer pairs for existing GSM-Symbolic contexts. For example, a single context about a juggler's balls was expanded to include 10 distinct questions ranging from simple counts to conditional hypotheticals (Figure 20, Table 1).

#### Design Choices and Rationale
The choice to implement sleep-time compute via **natural language representation** ($c'$) rather than hidden state caching or fine-tuning is deliberate and driven by flexibility.
*   **Interpretability:** By keeping the pre-computed knowledge in natural language, the system allows for debugging and verification of what the model "learned" during sleep-time. Users can inspect the `rethink_memory_block` to see exactly which facts were derived.
*   **Model Agnosticism:** This approach works with any LLM capable of following instructions and using function calls, without requiring access to internal model weights or activation states.
*   **Handling Uncertainty:** The prompt design encourages the model to generate "hypotheses" and "potential conclusions" if uncertain (Figure 17). This allows the test-time model to evaluate multiple pre-computed possibilities against the actual query $q$, rather than being locked into a single potentially incorrect derivation path.
*   **Decoupling Latency:** By moving the compute to "sleep-time," the system effectively hides the latency of complex reasoning. The user experiences only the latency of $T_b$, which is minimal, while the heavy lifting of $S$ occurs in the background when the system is idle.

This technical framework transforms the inference pipeline from a reactive, stateless process into a proactive, stateful one, enabling significant efficiency gains without sacrificing the depth of reasoning required for complex tasks.

## 4. Key Insights and Innovations

The paper's contributions extend beyond a simple engineering optimization; they represent a fundamental shift in how we conceptualize the timing and statefulness of LLM inference. While prior work focused on making test-time reasoning faster or more parallel, this work introduces a new temporal dimension to the compute budget. The following insights distinguish between incremental efficiency gains and the novel theoretical frameworks proposed.

### 4.1 The Predictability-Utility Correlation: A New Heuristic for Compute Allocation
Perhaps the most profound theoretical insight of this work is the empirical discovery that the efficacy of sleep-time compute is not uniform but is strictly governed by the **predictability of the query given the context**.

*   **The Insight:** The authors quantify "query predictability" by measuring the log-probability of a question $q$ given the context $c$ under a base model (Llama2-70B). They find a strong positive correlation: as the predictability of the query increases, the performance gap between sleep-time compute and standard test-time compute widens significantly (Figure 10).
*   **Why It Differs from Prior Work:** Traditional test-time scaling (e.g., Chain-of-Thought, Best-of-N) operates under the assumption that *any* difficult problem benefits from more compute, regardless of the input structure. It treats the query and context as a unified, opaque block. In contrast, sleep-time compute reveals that the *relationship* between $c$ and $q$ dictates the optimal strategy. If $q$ is highly unpredictable (low log-probability), pre-computing facts about $c$ yields diminishing returns because the pre-computed inferences are unlikely to align with the specific needs of $q$.
*   **Significance:** This finding transforms sleep-time compute from a blanket policy into a dynamic resource allocation problem. It suggests a future where systems can estimate the entropy of incoming queries in real-time:
    *   **High Predictability:** Aggressively invest in sleep-time compute to amortize costs.
    *   **Low Predictability:** Revert to standard test-time scaling to avoid wasting cycles on irrelevant pre-computations.
    *   This moves the field toward **adaptive inference architectures** that modulate their behavior based on the statistical properties of the input stream, rather than applying a static "think longer" heuristic.

### 4.2 Natural Language as a Persistent, Editable Cache (Representation Learning in Token Space)
The paper innovates by implementing "representation learning" not in the model's weight space or activation vectors, but explicitly in the **space of natural language tokens**.

*   **The Innovation:** Standard caching mechanisms (like KV-cache) store low-level attention states that are brittle and specific to exact token sequences. Fine-tuning modifies weights, which is computationally expensive and lacks granularity. Sleep-time compute instead prompts the model to generate a "re-represented context" ($c'$) consisting of derived facts, hypotheses, and logical summaries in plain English (Section 3.4).
*   **Why It Differs:** This approach treats the context window itself as a writable memory buffer. By using the `rethink_memory` function (Figure 17), the model performs a form of **semantic compression**: it distills hundreds of tokens of raw context into a denser, higher-value set of propositions.
    *   Unlike speculative decoding, which guesses the *next tokens of the answer*, sleep-time compute guesses the *intermediate states of understanding*.
    *   Because $c'$ is natural language, it is **interpretable** (developers can read what the model "learned") and **composable** (multiple $c'$ versions can be concatenated or edited).
*   **Significance:** This validates the hypothesis that LLMs can perform useful statistical modeling and data compression purely through prompting, without parameter updates. It opens a new avenue for "prompt-based programming" where the model's "state" is an evolving document of reasoned facts, enabling complex agentic behaviors that persist across sessions without requiring external vector databases or fine-tuning loops.

### 4.3 Decoupling Reasoning Depth from User-Latency via Temporal Shifting
While parallel test-time scaling (Pass@k) reduces latency by distributing work across GPUs, it does not reduce the *total* compute required per query, nor does it address the fundamental constraint that the user must wait for the reasoning to complete if it happens sequentially. Sleep-time compute introduces a **temporal decoupling** of reasoning depth and response latency.

*   **The Innovation:** The method shifts the "critical path" of inference. In standard settings, Reasoning Time $\subseteq$ Response Time. With sleep-time compute, the heavy reasoning component $S(c)$ is moved to the "idle" period between interactions, making Response Time $\approx$ Retrieval Time + Lightweight Verification ($T_b$).
*   **Why It Differs:**
    *   **Vs. Sequential Scaling:** Sequential scaling (e.g., o1-pro) forces the user to wait minutes for deep reasoning. Sleep-time allows for "deep reasoning" (via extensive $S(c)$ generation) while maintaining sub-second response times at test-time.
    *   **Vs. Parallel Scaling:** Parallel scaling multiplies the cost ($k \times$ compute) to reduce variance. Sleep-time compute *amortizes* the cost. As shown in the Multi-Query GSM-Symbolic experiments, the cost of deep reasoning is paid once and shared across $N$ queries, leading to a **2.5× reduction in average cost per query** when $N=10$ (Section 5.3).
*   **Significance:** This enables a new class of applications where **high-accuracy reasoning is compatible with real-time interaction**. For example, a coding agent can spend minutes analyzing a entire repository overnight (sleep-time) to answer a developer's specific debugging question instantly (test-time). It effectively breaks the trade-off triangle of Accuracy, Latency, and Cost by introducing "Time-Shifting" as a fourth variable.

### 4.4 Amortization as a First-Class Optimization Objective
The paper formally introduces **amortization across related queries** as a primary metric for inference efficiency, moving beyond the standard single-query benchmark paradigm.

*   **The Innovation:** Most LLM benchmarks evaluate performance on isolated $(c, q)$ pairs. This paper constructs the **Multi-Query GSM-Symbolic** dataset specifically to measure how well a system leverages shared context across a sequence of queries. The core metric shifts from "tokens per query" to "tokens per session" or "average cost per query over a context lifecycle."
*   **Why It Differs:** Prior work optimizes for the single-shot case. Even multi-turn dialogue research often treats each turn as a distinct inference event or relies on simple context concatenation. Sleep-time compute explicitly optimizes for the **marginal cost of the $N$-th query**. The system is designed such that the marginal cost of answering $q_{10}$ is near-zero regarding context processing, as the heavy lifting was done for $q_1$.
*   **Significance:** This aligns LLM evaluation more closely with real-world economic models. In enterprise settings (e.g., legal document review, codebase maintenance), users rarely ask just one question. By demonstrating that sleep-time compute creates a **Pareto improvement** in the cost-accuracy curve specifically under amortization (Figure 9), the paper provides a blueprint for building economically viable agentic systems that would otherwise be prohibitively expensive under standard test-time scaling regimes.

### Summary of Novelty vs. Incrementalism
| Feature | Incremental Improvement | Fundamental Innovation (This Paper) |
| :--- | :--- | :--- |
| **Compute Timing** | Faster token generation (Speculative Decoding). | **Temporal Shifting:** Moving compute to idle time ($t &lt; 0$) relative to the query. |
| **State Management** | KV-Cache or Vector DB retrieval. | **Semantic Re-representation:** Generating new natural language facts ($c'$) as a persistent state. |
| **Optimization Goal** | Minimize latency for a single query. | **Amortization:** Minimizing average cost across a *sequence* of related queries. |
| **Decision Logic** | "Think longer for hard problems." | **Predictability-Gated:** "Think offline only if the query is predictable from context." |

These insights collectively argue that the future of efficient LLM inference lies not just in faster hardware or better sampling strategies, but in smarter **temporal scheduling** of compute resources based on the statistical structure of the user's interaction history.

## 5. Experimental Analysis

This section dissects the empirical validation of sleep-time compute. The authors do not merely claim efficiency; they construct a rigorous experimental framework to quantify the trade-offs between offline pre-computation and online latency. The analysis moves from establishing a baseline Pareto improvement to exploring scaling laws, amortization benefits, and the specific conditions under which the method succeeds or fails.

### 5.1 Evaluation Methodology: Constructing Stateful Benchmarks

To evaluate a method that relies on the separation of context and query, standard benchmarks (which typically provide both simultaneously) are insufficient. The authors therefore engineered new datasets and metrics specifically designed to isolate the benefits of stateful inference.

#### Dataset Construction: From Stateless to Stateful
The core experimental innovation is the transformation of existing reasoning datasets into **stateful** variants.
*   **Stateful GSM-Symbolic:** Derived from the GSM-Symbolic dataset (Mirzadeh et al., 2024), which itself extends GSM8K with additional clauses to increase difficulty. The authors split each problem instance into a **Context ($c$)** (the problem statement up to the final question) and a **Query ($q$)** (the specific question asked).
    *   **Scale:** The dataset includes the **P1** split (5,000 examples with one added clause) and **P2** split (2,500 examples with two added clauses).
    *   **Example:** As shown in **Figure 2**, a problem about a juggler's balls is split such that the context contains the counts and ratios ("800 balls... 1/4 are tennis..."), while the query isolates the specific target ("How many marked indigo tennis balls are there?").
*   **Stateful AIME:** Constructed from the American Invitational Mathematics Examination (AIME) 2024 and 2025 problems.
    *   **Splitting Logic:** Problems are segmented by punctuation. All statements except the final one form the context $c$; the final statement forms the query $q$.
    *   **Sanitization:** To prevent data leakage, LaTeX representations of figures that might contain explicit side lengths or answers were manually stripped if the problem remained solvable without them (Appendix J).
    *   **Scale:** A total of **60 questions** combined from the 2024 and 2025 exams.
*   **Multi-Query GSM-Symbolic:** Created specifically to test amortization.
    *   **Generation:** Using `o3-mini`, the authors synthetically generated multiple distinct questions for each existing context in GSM-Symbolic.
    *   **Constraint:** Generated questions were required to have the same reasoning depth as the original.
    *   **Statistics:** As detailed in **Table C** (Appendix C), the P1 split contains **1,095 contexts** with a total of **12,043 questions** (approx. 11 questions per context), while P2 has **500 contexts** with **5,497 questions**. **Figure 20** illustrates a single context about toys generating diverse queries ranging from simple counts ("How many action figures?") to hypothetical scenarios ("If Sofia received 10 extra...").

#### Models and Baselines
The evaluation covers a spectrum of model capabilities, distinguishing between non-reasoning models and native reasoning models.
*   **Non-Reasoning Models:** `GPT-4o-mini` and `GPT-4o` were tested on Stateful GSM-Symbolic.
*   **Reasoning Models:** `o1`, `o3-mini`, `Claude 3.7 Sonnet (Extended Thinking)`, and `DeepSeek-R1` were tested on Stateful AIME.
*   **Baselines:**
    1.  **Standard Test-Time Compute:** The model receives $c$ and $q$ simultaneously. Compute is scaled via **sequential** methods (verbosity prompts) or **parallel** methods (Pass@k).
    2.  **Context-Only Baseline:** To ensure the queries are not trivially guessable from the context alone, a baseline was run where the model sees only $c$ and must guess the most likely $q$ and answer it. As shown in **Figures 21 and 22** (Appendix I), this baseline performs poorly, confirming that $q$ is not deterministic from $c$ and that sleep-time compute must rely on robust pre-processing rather than simple memorization.

#### Metrics and Cost Modeling
*   **Primary Metric:** The **Pareto frontier** of **Accuracy vs. Average Test-Time Tokens**. The goal is to shift this curve upward (higher accuracy for same tokens) or leftward (same accuracy for fewer tokens).
*   **Amortization Cost Model:** For multi-query scenarios, the authors define a linear cost function where test-time tokens are weighted higher due to latency constraints.
    $$ \text{Total Cost} = \text{Tokens}_{sleep} + t \times \sum \text{Tokens}_{test} $$
    They set **$t = 10$**, reflecting that latency-optimized inference is roughly **10× more expensive** per token than batched offline inference (Section 5.3).

### 5.2 Main Results: The Pareto Shift

The central claim of the paper is that sleep-time compute offers a Pareto improvement over standard test-time scaling. The results strongly support this, demonstrating significant reductions in required test-time compute.

#### Reducing Test-Time Compute by ~5×
On both Stateful GSM-Symbolic and Stateful AIME, sleep-time compute allows models to achieve baseline accuracy levels with drastically reduced online computation.
*   **Stateful GSM-Symbolic:** As illustrated in **Figure 3**, for both `GPT-4o-mini` and `GPT-4o`, the sleep-time compute curve (green/orange lines) sits well above the standard test-time curve (blue/purple lines).
    > "At lower test-time budgets, the performance of sleep-time compute is significantly better than the baseline, achieving performance comparable to that of the baseline with **5× less test-time tokens**." (Section 5.1)
    For example, on GSM-Symbolic P1, `GPT-4o-mini` with sleep-time compute achieves ~0.75 accuracy with roughly **100 test-time tokens**, whereas the standard baseline requires nearly **500 tokens** to reach the same accuracy.
*   **Stateful AIME:** **Figure 4** shows similar trends for reasoning models.
    *   `o3-mini`, `Claude 3.7 Sonnet`, and `DeepSeek-R1` all exhibit a clear leftward shift in the Pareto curve.
    *   **Exception:** The `o1` model shows **limited gains**. The authors hypothesize this may be because `o1`'s internal reasoning mechanisms are already highly optimized for stateless inputs, or its proprietary "chain of thought" is less amenable to external pre-computation hints compared to other models.

#### Comparison Against Parallel Scaling (Pass@k)
A critical comparison is made against **Pass@k**, the standard method for parallel test-time scaling.
*   **Methodology:** The authors apply parallel sampling to the lowest sequential compute setting. This is a strong baseline because parallel scaling typically maintains low latency.
*   **Results:** As shown in **Figures 5 and 6**, sleep-time compute **consistently Pareto dominates** Pass@k across all tasks and models.
    *   On Stateful GSM-Symbolic (**Figure 5**), `GPT-4o` with sleep-time compute achieves higher accuracy at every token budget compared to Pass@k.
    *   On Stateful AIME (**Figure 6**), the gap is even more pronounced for `o3-mini` and `DeepSeek-R1`.
    *   **Reasoning:** Pass@k relies on redundancy (sampling multiple full solutions), whereas sleep-time compute relies on **information density**. The pre-processed context $c'$ provides unique, high-value deductions that a single pass of standard inference cannot match, whereas Pass@k merely increases the probability of hitting a correct path by brute force.

### 5.3 Scaling Sleep-Time Compute

Does adding *more* offline compute yield further benefits? The authors investigate two scaling dimensions: parallel generations and reasoning effort.

#### Parallel Scaling (Non-Reasoning Models)
For `GPT-4o` and `GPT-4o-mini`, the authors generated $k$ independent pre-processed contexts ($c'_1 \dots c'_k$) and concatenated them at test-time.
*   **Findings:** **Figure 7** demonstrates that increasing $k$ shifts the Pareto curve outward.
    *   **Magnitude:** Scaling sleep-time compute improves accuracy by up to **13%** on Stateful GSM-Symbolic at similar test-time budgets.
    *   **Diminishing Returns:** Interestingly, **5 parallel generations** generally outperformed **10**. This suggests a "context saturation" point where adding more pre-computed text introduces noise or exceeds the model's effective attention window, diluting the benefit of the additional information.

#### Reasoning Effort Scaling (Reasoning Models)
For models like `o1` and `o3-mini`, compute was scaled by varying the "reasoning effort" prompt during the sleep-time phase (Low, Medium, High).
*   **Findings:** **Figure 8** confirms that deeper offline reasoning leads to better test-time performance.
    *   **Magnitude:** On Stateful AIME, scaling sleep-time compute improves accuracy by up to **18%**.
    *   **Trend:** The gap between "low reasoning effort" and "high reasoning effort" sleep-time curves is substantial, indicating that the *quality* of the pre-computed deductions matters immensely. A shallow pre-analysis fails to uncover the deep structural insights needed for AIME-level problems.

### 5.4 Amortization: The Multi-Query Advantage

The most compelling economic argument for sleep-time compute lies in the multi-query setting, where the cost of generating $c'$ is shared across many questions.

*   **Experimental Setup:** Using the **Multi-Query GSM-Symbolic** dataset, the authors measured the total inference cost per query as the number of questions per context ($N$) increased from 1 to 10.
*   **Results:** **Figure 9** plots Accuracy vs. Total Inference Cost (weighted by $t=10$).
    *   **Low $N$ (1-2 queries):** Sleep-time compute is **less favorable**. The overhead of generating $c'$ is not yet amortized, making the total cost higher than simply answering each question with standard test-time compute.
    *   **High $N$ (10 queries):** The trend reverses dramatically. With 10 queries per context, sleep-time compute decreases the average cost per query by **2.5×** compared to the single-query baseline.
    *   **Interpretation:** At $N=10$, the fixed cost of the offline phase becomes negligible per query, while the test-time cost remains minimal because the model simply retrieves facts from $c'$. This validates the hypothesis that sleep-time compute is ideally suited for **session-based interactions** (e.g., debugging a codebase, analyzing a document) rather than isolated, one-off queries.

### 5.5 Critical Insight: The Role of Query Predictability

Not all contexts benefit equally from sleep-time compute. The authors conduct a nuanced analysis to determine *when* the method works best.

*   **Metric:** They define **Query Predictability** as the log-probability of the question $q$ given the context $c$, calculated using a base model (`Llama2-70B`).
    $$ \text{Predictability}(q|c) = \log P(q|c) $$
*   **Methodology:** Examples in Stateful GSM-Symbolic were binned into five quantiles based on this score.
*   **Results:** **Figure 10** reveals a striking correlation.
    *   **Low Predictability (Bin 1):** The accuracy difference between sleep-time and standard test-time compute is near zero. If the question is unpredictable (e.g., a complex hypothetical unrelated to the obvious facts in $c$), the pre-computed inferences are likely irrelevant.
    *   **High Predictability (Bin 5):** The accuracy gap widens significantly (up to **0.40 difference** on P2). When the query follows naturally from the context (e.g., "How many total items?"), the sleep-time phase successfully anticipates the necessary deductions.
*   **Implication:** This finding serves as a crucial boundary condition. Sleep-time compute is not a universal silver bullet; it is a **predictability-gated optimization**. Systems implementing this should ideally estimate query entropy before deciding to invest in offline compute.

### 5.6 Case Study: Agentic Software Engineering (SWE-Features)

To move beyond math problems, the authors introduced **SWE-Features**, a benchmark for software engineering tasks requiring multi-file edits.
*   **Task:** Agents must implement new features based on Pull Request (PR) descriptions, using related PRs as context $c$.
*   **Metric:** F1 score between the set of files modified by the agent and the ground-truth modified files.
*   **Results:** **Figure 11** shows results for `Claude 3.7 Sonnet`.
    *   **Low Budget:** Sleep-time compute improves F1 score, achieving a **1.5× decrease** in test-time tokens for equivalent performance. The agent uses the offline phase to explore the repository structure, allowing it to identify relevant files faster at test-time.
    *   **High Budget:** Standard test-time compute eventually surpasses sleep-time compute.
    *   **Failure Mode Analysis:** The authors note that at high budgets, standard test-time agents exhibit **higher precision**. The sleep-time agent, having explored broadly offline, tends to edit *more* files (higher recall) but occasionally includes unnecessary changes (lower precision). This suggests that while sleep-time compute improves coverage, it may introduce "over-eager" modifications if the offline exploration is too broad relative to the specific query.

### 5.7 Assessment of Experimental Validity

The experiments convincingly support the paper's core claims, with specific strengths and minor limitations:

*   **Strengths:**
    *   **Robust Baselines:** The inclusion of both sequential and parallel (Pass@k) baselines ensures the improvement is not just an artifact of poor baseline tuning.
    *   **Cost Modeling:** The explicit modeling of test-time tokens as 10× more expensive ($t=10$) grounds the results in real-world economic constraints, making the 2.5× cost reduction claim highly relevant for deployment.
    *   **Predictability Analysis:** The ablation on query predictability (Section 5.4) adds significant scientific depth, moving the work from a simple "it works" demonstration to a nuanced understanding of *why* it works.

*   **Limitations & Conditions:**
    *   **Model Dependence:** The limited gains on `o1` suggest the method's efficacy may vary depending on the underlying model architecture and its native reasoning capabilities.
    *   **Context Saturation:** The diminishing returns observed with 10 parallel generations indicate there is an upper bound on how much pre-computed text a model can effectively utilize in a single context window.
    *   **Precision Trade-off:** In the SWE-Features case study, the slight drop in precision at high budgets highlights a potential risk: pre-computed context might bias the model toward certain actions (editing more files) that are not strictly necessary for the specific query.

In summary, the experimental analysis provides strong evidence that sleep-time compute fundamentally alters the efficiency landscape of LLM inference. By shifting the workload to idle periods, it achieves a **5× reduction in test-time compute** for equivalent accuracy and enables **2.5× cost savings** in multi-query sessions, provided the user queries are reasonably predictable from the available context.

## 6. Limitations and Trade-offs

While **sleep-time compute** demonstrates a compelling shift in the efficiency frontier of LLM inference, its efficacy is not universal. The approach relies on specific structural assumptions about the relationship between context and query, and it introduces new trade-offs regarding precision, context window capacity, and temporal dynamics. A critical examination of the paper reveals several boundaries where the method struggles or remains unproven.

### 6.1 The Predictability Dependency: A Hard Boundary Condition
The most significant limitation of sleep-time compute is its strict dependence on the **predictability of the user query** given the context. As established in Section 5.4 and visualized in **Figure 10**, the method's utility collapses when the query entropy is high.

*   **The Mechanism of Failure:** Sleep-time compute operates by anticipating useful inferences. If the user's actual query $q$ diverges significantly from the distribution of queries the model anticipated during the sleep phase, the pre-computed context $c'$ becomes irrelevant or even distracting.
*   **Empirical Evidence:** The authors bin questions by predictability (log-probability of $q$ given $c$). In the lowest predictability bin (Bin 1), the accuracy gap between sleep-time compute and standard test-time compute is **near zero**. This implies that for highly creative, adversarial, or non-obvious queries, the offline compute is effectively wasted.
*   **Operational Risk:** In real-world applications, users often ask the *least* predictable questions (e.g., edge cases in debugging, novel interpretations of a document). A system optimized for sleep-time compute might perform exceptionally well on routine queries but fail to provide any latency or cost benefit precisely when the user asks something unexpected. The paper acknowledges this, noting that "in settings where the queries are challenging to predict... it may be preferable to apply standard test-time scaling instead" (Section 7). However, it does not provide a dynamic switching mechanism to handle this hybrid workload efficiently.

### 6.2 The Precision-Recall Trade-off in Agentic Tasks
In the **SWE-Features** case study (Section 6), the paper uncovers a subtle but critical trade-off: sleep-time compute can degrade **precision** while improving recall.

*   **The Observation:** At high test-time compute budgets, the standard baseline (test-time only) achieved higher precision than the sleep-time variant. The sleep-time agent, having broadly explored the repository and pre-computed potential modifications offline, tended to edit **more files** than strictly necessary.
*   **Why This Happens:** The sleep-time prompt encourages the model to "make as many inferences as possible" and "predict possible ways the rest of the problem might go" (Figure 18). This broadens the model's attention scope. When the actual query arrives, the model may be biased by its extensive offline exploration, leading it to propose changes to files that are tangentially related but not required for the specific fix.
*   **Implication:** For tasks requiring surgical precision (e.g., patching a security vulnerability without touching unrelated modules), sleep-time compute might introduce "over-eager" behavior. The pre-computed context acts as a prior that pushes the model toward broader modifications, potentially increasing the burden on the user to review and reject unnecessary changes.

### 6.3 Context Window Saturation and Diminishing Returns
Scaling sleep-time compute by generating multiple parallel pre-processed contexts ($c'_1, \dots, c'_k$) encounters a hard limit imposed by the model's context window and attention mechanism.

*   **Diminishing Returns:** In Section 5.2, the authors report that while scaling from 1 to 5 parallel generations improved accuracy, scaling to **10 parallel generations generally resulted in lower performance** than 5.
*   **Hypothesized Cause:** The paper suggests this is due to "distracting information" or the model's inability to effectively attend to all relevant parts of a massively expanded context. As the pre-processed context $c'$ grows, it may dilute the signal-to-noise ratio, making it harder for the test-time model to locate the specific fact needed for $q$.
*   **Scalability Constraint:** This imposes a ceiling on how much "offline thinking" can be injected into a single interaction. Unlike test-time scaling, where one can theoretically sample indefinitely (until budget runs out), sleep-time compute is constrained by the **input capacity** of the model. Once the context window is saturated with pre-computed hypotheses, adding more compute yields negative returns.

### 6.4 Temporal Assumptions and Dynamic Contexts
The entire framework rests on the assumption that the context $c$ is **static** during the sleep-time interval. This assumption breaks down in highly dynamic environments.

*   **The Staleness Problem:** The paper defines sleep-time as occurring "between interactions... while it would otherwise be idle" (Section 1). However, in many stateful applications (e.g., collaborative coding, live document editing, real-time monitoring), the context $c$ may change *during* the sleep period.
    *   If the codebase is modified by another developer while the agent is "sleeping," the pre-computed $c'$ becomes stale or contradictory.
    *   The current architecture lacks a mechanism for **incremental updates** to $c'$. If $c$ changes, the system must discard the expensive $c'$ and re-run the entire sleep-time process, negating the amortization benefits.
*   **Variable Idle Times:** The length of the "sleep" period is unpredictable. It could be seconds (between user keystrokes) or days (between project sessions).
    *   **Short Intervals:** If the idle time is too short, the model cannot generate a sufficiently deep $c'$ to be useful.
    *   **Long Intervals:** If the idle time is very long, the probability of context drift (staleness) increases.
    The paper acknowledges this in Section 7, stating that "real-world LLM use cases can be more complex... and the length of the sleep-time may also vary significantly," but offers no solution for adaptive compute allocation based on available idle time.

### 6.5 Economic Viability in Single-Query Regimes
The economic model of sleep-time compute is heavily skewed toward **multi-query sessions**.

*   **The Break-Even Point:** As shown in **Figure 9**, for contexts with only **1 or 2 queries**, sleep-time compute is **less favorable** than standard test-time compute in terms of total cost. The fixed overhead of generating $c'$ dominates the total cost equation when $N$ (number of queries) is small.
*   **Workload Mismatch:** Many real-world LLM interactions are single-turn or very short sessions (e.g., a user asks one question about a document and leaves). In these "one-and-done" scenarios, sleep-time compute actively **increases** the total compute cost and latency (if the sleep-time compute is not perfectly hidden) without providing accuracy benefits.
*   **Deployment Challenge:** To realize the claimed **2.5× cost reduction**, a system must reliably predict or enforce multi-turn interactions. Applying this technique to a workload with high churn (many unique contexts, few queries per context) would be economically inefficient.

### 6.6 Model-Specific Efficacy
The benefits of sleep-time compute are not uniform across all model architectures.

*   **The `o1` Anomaly:** In the Stateful AIME experiments (**Figure 4**), the **o1** model demonstrated **limited gains** from sleep-time compute compared to `o3-mini`, `Claude 3.7`, and `DeepSeek-R1`.
*   **Potential Causes:**
    *   **Internal Optimization:** `o1` may already employ internal mechanisms that mimic sleep-time compute (e.g., extensive internal pre-processing of the prompt before generating the visible chain of thought), rendering external pre-computation redundant.
    *   **Instruction Following:** The specific prompting strategy used to generate $c'$ might not align well with `o1`'s training distribution.
*   **Uncertainty:** This variability introduces uncertainty for deployers. It is unclear *a priori* which models will benefit from this paradigm. If a proprietary model's internal reasoning dynamics change (as they often do with updates), the efficacy of sleep-time compute could vanish overnight, requiring costly re-evaluation.

### 6.7 Summary of Open Questions
The paper leaves several critical questions unanswered, pointing to necessary future work:
1.  **Dynamic Switching:** How can a system automatically detect low-predictability queries in real-time and switch to standard test-time scaling without incurring a "double cost" (paying for failed sleep-time compute + full test-time compute)?
2.  **Incremental Updates:** Can $c'$ be updated incrementally when $c$ changes slightly, rather than recomputed from scratch?
3.  **Optimal Sleep Duration:** What is the optimal amount of compute to spend during sleep-time given a probabilistic distribution of idle times and query arrival rates?
4.  **Precision Control:** How can the "over-eagerness" observed in the SWE-Features case study be mitigated to ensure high precision in code modification tasks?

In conclusion, while sleep-time compute offers a powerful lever for optimizing stateful, multi-turn interactions with predictable queries, it is not a panacea. Its success is contingent on stable contexts, multi-query amortization, and the inherent predictability of the user's intent. Deploying this technique requires careful workload analysis to ensure the specific constraints of the application align with the method's strengths.

## 7. Implications and Future Directions

The introduction of **sleep-time compute** fundamentally alters the architectural assumptions of Large Language Model (LLM) deployment. By demonstrating that inference efficiency is not solely a function of model size or sampling strategy, but also of **temporal scheduling**, this work shifts the field's focus from "how fast can we think?" to "when should we think?" The implications extend beyond mathematical benchmarks, suggesting a new paradigm for building stateful, agentic systems where latency and cost are decoupled through proactive computation.

### 7.1 Reshaping the Inference Landscape: From Reactive to Proactive
Historically, LLM inference has been treated as a strictly **reactive** process: a query arrives, and the model consumes compute to generate a response. This paper challenges that orthodoxy by proving that a significant portion of the reasoning workload can be shifted to **idle periods** without sacrificing accuracy.

*   **The Fourth Dimension of Scaling:** Prior scaling laws focused on parameters (pre-training), data (pre-training), and tokens (test-time). Sleep-time compute introduces **time-shifting** as a fourth dimension. It suggests that the "compute budget" for a problem is not fixed at the moment of the query but is a flexible resource that can be accumulated over time.
*   **Redefining "State":** Current stateful applications often rely on simple context concatenation or vector retrieval (RAG). This work elevates the concept of state from passive storage to **active semantic representation**. The `rethink_memory_block` (Section 3.4) acts as a dynamic, natural-language cache that evolves as the model "sleeps," enabling a form of **continuous learning** within a single session without parameter updates.
*   **Economic Realignment:** The finding that test-time tokens are effectively **10× more expensive** than sleep-time tokens (due to latency constraints) creates a new economic incentive structure. Cloud providers and application developers may begin to offer "batched offline reasoning" as a distinct service tier, encouraging users to tolerate slight delays in background processing in exchange for instant, low-cost responses later.

### 7.2 Enabling Follow-Up Research Directions
The empirical results and limitations identified in this paper open several critical avenues for future research:

*   **Adaptive Compute Orchestration:**
    The strong correlation between **query predictability** and sleep-time efficacy (Section 5.4) suggests the need for a **hybrid inference controller**. Future work should explore systems that dynamically estimate the entropy of an incoming query $H(q|c)$ in real-time.
    *   *Research Question:* Can a lightweight router predict whether a query falls into a "high-predictability" bin and automatically toggle between sleep-time and standard test-time modes?
    *   *Goal:* To avoid the "double cost" penalty of running sleep-time compute for unpredictable queries that yield no benefit.

*   **Incremental Context Updating:**
    A major limitation identified is the **staleness** of $c'$ if the underlying context $c$ changes during the sleep period (Section 6.4).
    *   *Research Question:* Can sleep-time compute be made **incremental**? Instead of regenerating $c'$ from scratch, can the model apply a "diff" operation to update only the relevant portions of the pre-computed memory when $c$ changes slightly?
    *   *Potential Approach:* Adapting techniques from continuous pre-training or delta-compression to the natural language representation space.

*   **Optimizing the Sleep-Test Trade-off:**
    The paper observes diminishing returns when scaling parallel sleep-time generations beyond $k=5$ (Section 5.2), likely due to context window saturation.
    *   *Research Question:* What is the optimal information density for $c'$? Can we develop compression algorithms that distill multiple parallel reasoning traces ($c'_1 \dots c'_k$) into a single, non-redundant super-context $c'_{optimal}$ that fits within standard context limits?

*   **Synthetic Data Generation at Scale:**
    Section 7 hints at using sleep-time compute to generate high-quality synthetic data for pre-training.
    *   *Research Question:* Can we use massive clusters to run sleep-time compute on vast corpora of raw text (e.g., Wikipedia, GitHub) to generate "reasoned versions" of these datasets? These enriched datasets could then be used to pre-train models that inherently possess deeper reasoning capabilities, effectively distilling test-time compute into model weights.

### 7.3 Practical Applications and Downstream Use Cases
The sleep-time compute paradigm is uniquely suited for applications characterized by **long-lived contexts** and **multi-turn interactions**.

*   **Enterprise Coding Agents (SWE-Bench++):**
    As demonstrated in the **SWE-Features** case study (Section 6), coding agents can utilize overnight compute to analyze entire repositories, map dependency graphs, and hypothesize potential bug fixes.
    *   *Workflow:* An agent "sleeps" while developers are offline, producing a detailed architectural summary ($c'$). When a developer arrives and asks, "Why is the build failing?", the agent answers instantly using the pre-computed map, rather than spending minutes re-parsing thousands of files.

*   **Legal and Financial Document Review:**
    In domains where users ask sequences of related questions about massive documents (contracts, audit reports), sleep-time compute can pre-extract entities, clauses, and logical inconsistencies.
    *   *Benefit:* Lawyers can interrogate a 500-page contract with sub-second latency, as the heavy lifting of parsing and cross-referencing was completed during the document upload phase.

*   **Personalized Conversational Companions:**
    Long-term memory in chatbots often suffers from retrieval latency or loss of nuance. Sleep-time compute allows a companion to "reflect" on past conversations during idle hours, synthesizing new insights about the user's preferences or emotional state into its memory block.
    *   *Outcome:* The model appears more empathetic and context-aware, not because it is larger, but because it has "thought about" the user before the conversation resumed.

*   **Real-Time Strategy and Simulation:**
    In gaming or simulation environments where the world state ($c$) is complex but changes discretely, AI NPCs can use sleep-time compute to plan strategies or update their world models between player turns, enabling sophisticated behavior without delaying the game loop.

### 7.4 Reproducibility and Integration Guidance
For practitioners looking to adopt or reproduce these findings, the following guidelines clarify when and how to integrate sleep-time compute:

*   **When to Prefer Sleep-Time Compute:**
    *   **High Context Reuse:** Adopt this method only if the ratio of queries to context ($N$) is expected to be **$\geq 3$**. As shown in **Figure 9**, the cost benefits only materialize after the fixed cost of generating $c'$ is amortized over multiple questions.
    *   **Predictable Query Patterns:** Ideal for domains where queries follow standard templates (e.g., "What is the value of X?", "Summarize section Y", "Debug function Z"). Avoid for open-ended, creative, or adversarial prompting where the query distribution is high-entropy.
    *   **Latency-Sensitive Interfaces:** Use when the user experience demands sub-second response times, but the backend has access to idle compute cycles (e.g., overnight batch processing, background threads).

*   **When to Stick with Standard Test-Time Scaling:**
    *   **Single-Turn Interactions:** For one-off queries (e.g., search engine snippets), the overhead of sleep-time compute increases total cost without benefit.
    *   **Dynamic/Streaming Contexts:** If the context $c$ changes rapidly (e.g., live stock tickers, real-time collaborative editing), the risk of $c'$ staleness outweighs the benefits.
    *   **Precision-Critical Tasks:** Be cautious in tasks requiring surgical precision (e.g., security patching). The "over-eagerness" observed in the SWE-Features study suggests sleep-time agents may propose broader changes than necessary.

*   **Implementation Checklist:**
    1.  **Decouple Input:** Modify your ingestion pipeline to separate static context ($c$) from dynamic queries ($q$).
    2.  **Define the Sleep Prompt:** Implement a prompt similar to **Figure 17** or **Figure 18**, explicitly instructing the model to "predict useful inferences" and "store them in memory."
    3.  **Function Calling Interface:** Utilize function calling (e.g., `rethink_memory`) to structurally isolate the pre-computed state, ensuring the test-time prompt can explicitly reference it (Section 3.4).
    4.  **Enforce Reuse:** Crucially, the test-time system prompt **must** include negative constraints (e.g., "Do not recompute anything that already exists in the memory block") to prevent the model from ignoring the sleep-time work.
    5.  **Monitor Predictability:** Log the log-probability of incoming queries relative to the context. If the average predictability drops, dynamically reduce the sleep-time budget.

By treating compute as a storable resource rather than a fleeting expense, sleep-time compute offers a pragmatic path toward building faster, cheaper, and more intelligent AI systems. It invites the community to rethink the very definition of "inference time," transforming the idle moments between interactions into opportunities for deeper understanding.
