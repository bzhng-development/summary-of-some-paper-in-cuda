## 1. Executive Summary

This paper introduces **Splitwise**, a system architecture that decouples the compute-intensive **prompt computation** phase from the memory-intensive **token generation** phase of generative LLM inference (e.g., **Llama2-70B**, **BLOOM-176B**) onto separate, hardware-optimized machines. By addressing the severe underutilization of compute resources during token generation and the mismatched scaling of GPU memory versus FLOPs, Splitwise enables clusters to achieve up to **1.4× higher throughput at 20% lower cost**, or alternatively, **2.35× more throughput** within the same power and cost budgets compared to traditional homogeneous deployments.

## 2. Context and Motivation

### The Fundamental Mismatch in LLM Inference Hardware
To understand the necessity of Splitwise, one must first recognize a critical divergence in how modern GPU hardware has evolved versus how Large Language Model (LLM) inference actually behaves.

Generative LLM inference is not a monolithic operation; it consists of two distinct phases with opposing resource requirements:
1.  **Prompt Computation Phase:** The model processes the entire input prompt (e.g., a user's question or code context) in parallel to generate the first output token. This phase is **compute-intensive**, requiring massive Floating Point Operations per Second (FLOPs) to perform matrix multiplications across all input tokens simultaneously.
2.  **Token Generation Phase:** Subsequent tokens are generated sequentially (auto-regressively). Each new token depends on the previous one and the cached context (Key-Value or KV cache) of the entire sequence. Because this process cannot be parallelized across tokens, it becomes **memory-bandwidth and memory-capacity bound**. The GPU spends most of its time waiting for data to move from High Bandwidth Memory (HBM) to the compute cores, leaving the actual compute units idle.

The problem arises because the semiconductor industry has optimized GPUs primarily for the compute-heavy needs of *training* and the prompt phase, leading to a severe imbalance for the token generation phase. As detailed in **Table I** of the paper, when comparing the NVIDIA A100 to the newer H100:
*   Compute power (TFLOPs) increased by **3.43×**.
*   Power consumption increased by **1.75×**.
*   Cost per machine increased by **2.16×**.
*   **However**, HBM capacity remained identical (**1.00×**), and bandwidth only increased by **1.64×**.

This creates a "memory wall" for token generation. Deploying expensive, power-hungry H100 GPUs for the token generation phase results in massive inefficiency: the extra compute capability sits unused, while the system is bottlenecked by memory bandwidth that has not scaled proportionally. The paper argues that continuing to run both phases on the same high-end hardware leads to underutilized resources, inflated costs, and unnecessary power consumption.

### Limitations of Prior Approaches: Batching and Scheduling
Before Splitwise, the research community focused on optimizing inference *within* a single machine or homogeneous cluster. The primary technique employed is **batching**, which attempts to hide latency and improve throughput by processing multiple requests simultaneously. The paper analyzes three specific batching strategies in **Section II-D** and **Figure 2**, highlighting their inherent flaws:

*   **Request-Level Batching:** Requests are grouped, but the system waits for all requests in the batch to finish their entire sequence (prompt + all tokens) before starting the next batch. This causes high latency for new arrivals because they must wait for long token-generation sequences to complete.
*   **Continuous Batching (e.g., Orca [81]):** The scheduler makes decisions at every iteration. However, to simplify memory management, a batch typically contains *either* only prompt phases *or* only token phases. Since prompt phases are critical for Time-To-First-Token (TTFT) latency, they often preempt token phases. While this improves TTFT, it drastically increases the **Time-Between-Tokens (TBT)** for ongoing conversations, leading to poor user experience in streaming applications.
*   **Mixed Batching:** Prompts and token generations run in the same batch. While this mitigates some latency issues, it does not solve the underlying hardware mismatch. A machine running a mixed batch still suffers from the fact that the token-generation portion of the workload cannot utilize the GPU's full compute power, yet it occupies the same expensive memory bandwidth as the compute-heavy prompt portion.

Furthermore, prior work on **heterogeneous scheduling** (Section VIII) typically assigns entire workloads to different processors (e.g., CPU vs. GPU) based on average characteristics. These approaches fail to recognize that a *single* LLM request changes its character dynamically from compute-bound to memory-bound. No existing system splits a single request's lifecycle across different machines to match these shifting phases.

### The Real-World Impact: Cost, Power, and Availability
The motivation for solving this problem extends beyond theoretical efficiency; it addresses three pressing constraints in the deployment of generative AI:

1.  **The GPU Capacity Crunch:** The demand for LLM inference far exceeds supply, leading to a global shortage of flagship GPUs like the H100. By allowing the token generation phase to run on older, less compute-capable hardware (like the A100), systems can free up scarce high-end GPUs for tasks that actually require them (prompt computation and training).
2.  **The Power Wall:** Data centers are hitting physical limits on power delivery. As shown in **Table I**, newer GPUs consume significantly more power (700W for H100 vs. 400W for A100). Since the token generation phase does not need peak compute, running it on power-capped or older GPUs reduces the overall energy footprint of the cluster without sacrificing performance.
3.  **Economic Efficiency:** Cloud Service Providers (CSPs) and users face exorbitant costs. The paper demonstrates that running token generation on mismatched hardware inflates the cost per token. By decoupling the phases, providers can construct clusters using a mix of hardware generations, optimizing the **Performance-per-Dollar** and **Performance-per-Watt** metrics.

### Positioning of Splitwise
Splitwise positions itself as a paradigm shift from **request-level scheduling** to **phase-level scheduling**.

Unlike prior works that treat an inference request as an atomic unit bound to a single machine, Splitwise treats the request as a workflow with two distinct stages. It proposes a architecture where:
*   **Prompt Machines:** Equipped with high-compute GPUs (e.g., H100) to minimize TTFT.
*   **Token Machines:** Equipped with memory-optimized or older GPUs (e.g., A100 or power-capped H100) to maximize throughput and efficiency.
*   **State Transfer:** The system efficiently transfers the generated KV-cache state between these machines over high-speed interconnects (InfiniBand), masking the communication latency through overlapping techniques.

By doing so, Splitwise does not merely tweak scheduling algorithms; it fundamentally re-architects the deployment topology to align hardware capabilities with the specific computational characteristics of each inference phase. This allows for the design of **heterogeneous clusters** that were previously impossible, achieving higher throughput at lower costs and power budgets than any homogeneous design could support.

## 3. Technical Approach

This section details the architecture and mechanisms of **Splitwise**, a system that fundamentally re-architects LLM inference by decoupling the compute-bound prompt phase from the memory-bound token generation phase onto separate, specialized hardware pools. Unlike traditional systems that treat an inference request as an atomic unit bound to a single GPU, Splitwise treats the request as a distributed workflow, transferring the internal model state (the KV-cache) between machines to match the specific resource demands of each phase.

### 3.1 Reader orientation (approachable technical breakdown)
Splitwise is a cluster-level scheduling and serving system that splits a single LLM inference request across two different physical machines: one optimized for heavy calculation (prompt processing) and another optimized for memory bandwidth (token generation). It solves the problem of wasted compute power during token generation by dynamically moving the request's internal state over a high-speed network, allowing each phase to run on the most cost-effective and efficient hardware available.

### 3.2 Big-picture architecture (diagram in words)
The Splitwise architecture consists of three primary logical components operating in a hierarchical structure:
*   **Cluster-Level Scheduler (CLS):** A central controller that manages three distinct pools of machines (Prompt Pool, Token Pool, and a dynamic Mixed Pool) and routes incoming requests to specific machine pairs based on queue lengths and hardware availability.
*   **Machine-Level Scheduler (MLS):** A local agent running on each GPU node that manages memory, handles local batching of requests, and executes the specific logic for either prompt computation, token generation, or mixed workloads.
*   **Optimized Communication Layer:** A specialized data plane built on top of InfiniBand and GPU-direct libraries (MSCCL++) that handles the low-latency transfer of the Key-Value (KV) cache state from the Prompt Machine to the Token Machine, overlapping communication with computation to hide latency.

In operation, a request enters the CLS, which assigns it a Prompt Machine and a Token Machine; the Prompt Machine computes the initial context, streams the KV-cache to the Token Machine via the Communication Layer, and the Token Machine completes the sequence generation, effectively creating a pipelined assembly line for LLM tokens.

### 3.3 Roadmap for the deep dive
To fully understand how Splitwise achieves its efficiency gains without sacrificing latency, we will explore the system in the following logical order:
*   **Hierarchical Scheduling Logic:** We first explain how the Cluster-Level Scheduler manages heterogeneous machine pools and routes requests, as this defines the system's global resource allocation strategy.
*   **Local Batching and Execution:** We then detail the Machine-Level Scheduler's distinct algorithms for prompt vs. token machines, showing how local policies differ to maximize hardware utilization.
*   **State Transfer Mechanism:** We provide a deep dive into the core innovation—the KV-cache transfer protocol—explaining how the system overlaps communication with computation to minimize the penalty of splitting the request.
*   **Cluster Provisioning Strategies:** We analyze how Splitwise enables four specific cluster configurations (e.g., heterogeneous hardware, power-capped nodes) to optimize for cost, power, or throughput.
*   **Fault Tolerance and Scalability:** Finally, we address practical deployment concerns, including how the system handles machine failures and scales to large cluster sizes.

### 3.4 Detailed, sentence-based technical breakdown

#### Hierarchical Scheduling and Resource Management
The core intelligence of Splitwise lies in its two-level scheduling hierarchy, which separates global resource allocation from local execution details.

**Cluster-Level Scheduler (CLS)**
The Cluster-Level Scheduler acts as the central brain of the deployment, responsible for maintaining the health and composition of three distinct machine pools: the **Prompt Pool**, the **Token Pool**, and the **Mixed Pool**.
*   The CLS initially assigns machines to the Prompt or Token pools based on the expected workload distribution (e.g., the ratio of input prompt tokens to output tokens derived from production traces).
*   To handle transient load spikes and prevent resource fragmentation, the CLS dynamically manages a **Mixed Pool**, into which machines from the Prompt or Token pools can be temporarily moved.
*   When a machine enters the Mixed Pool, it retains its original identity (e.g., a "Prompt Machine" running token tasks) and automatically returns to its original pool once its queue of opposite-type tasks is drained, ensuring no permanent reconfiguration overhead.
*   For request routing, the CLS employs a **Join-the-Shortest-Queue (JSQ)** policy, but unlike standard implementations that look at the number of requests, it defines queue length by the **number of pending tokens**.
*   The CLS simultaneously assigns *both* a Prompt Machine and a Token Machine to a new request upon arrival, a critical design choice that allows the system to pre-establish the communication path for the KV-cache transfer before the prompt computation even begins.
*   If the pending queue for a specific pool exceeds a defined threshold, the CLS attempts to pull machines from the Mixed Pool; if the Mixed Pool is empty, it forcibly borrows a machine from the opposite pool (e.g., borrowing a Token Machine to run prompts), temporarily converting it to mixed-mode operation.

**Machine-Level Scheduler (MLS)**
Each physical machine runs a local Machine-Level Scheduler that is aware of its assigned role (Prompt, Token, or Mixed) and adjusts its batching strategy accordingly.
*   On **Prompt Machines**, the MLS uses a strict **First-Come-First-Serve (FCFS)** policy but enforces a hard limit on the total number of prompt tokens in a batch to prevent performance degradation.
*   Based on the characterization in **Figure 6a**, the MLS restricts the batch size to a maximum of **2048 prompt tokens**; beyond this threshold, the throughput per token drops significantly due to memory capacity constraints, so the scheduler queues additional requests rather than expanding the batch.
*   On **Token Machines**, the MLS also uses FCFS but aims to maximize the batch size up to the limit of available GPU memory, as **Figure 6b** demonstrates that token generation throughput scales linearly with batch size until memory exhaustion occurs.
*   The MLS on Token Machines continuously monitors memory usage and begins queueing new token requests once the machine approaches its memory capacity limit, ensuring that active batches never spill or cause swapping.
*   On **Mixed Machines** (those temporarily borrowed from the other pool), the MLS prioritizes **Time-To-First-Token (TTFT)** Service Level Objectives (SLOs) by immediately scheduling any waiting prompt phases.
*   To prevent the starvation of token generation tasks on Mixed Machines, the MLS implements an **aging mechanism** that increases the priority of token requests over time and limits the number of times a single token request can be preempted by a new prompt.

#### Optimized KV-Cache Transfer Mechanism
The most critical technical challenge in Splitwise is the latency introduced by transferring the **Key-Value (KV) cache**—the internal state of the model containing attention keys and values for all processed tokens—from the Prompt Machine to the Token Machine.

**The Latency Problem**
In a naive implementation, the system would wait for the Prompt Machine to finish computing the entire prompt, generate the full KV-cache, and *then* initiate a network transfer before the Token Machine could begin generating the first output token.
*   This serialized approach, illustrated in **Figure 11a**, directly adds the network transfer time to the **Time-Between-Tokens (TBT)** for the second token, potentially violating strict latency SLOs for interactive applications.
*   The size of the KV-cache is proportional to the number of prompt tokens and the model size; for large prompts, this transfer could take tens of milliseconds even on high-bandwidth InfiniBand links.

**Layer-Wise Overlapping Strategy**
Splitwise eliminates this penalty by overlapping the KV-cache transfer with the ongoing prompt computation using a **layer-wise asynchronous transfer** protocol.
*   Modern Transformer models consist of multiple layers (e.g., 80 layers for Llama2-70B); as the Prompt Machine finishes the forward pass for Layer $L$, the KV-cache for that specific layer is immediately available.
*   Instead of waiting for all layers to complete, the Prompt Machine triggers an asynchronous transfer of Layer $L$'s KV-cache to the Token Machine while the GPU immediately proceeds to compute Layer $L+1$.
*   This pipelining, shown in **Figure 11b**, ensures that by the time the Prompt Machine finishes the final layer, the majority (or all) of the KV-cache has already been transmitted, reducing the visible transfer latency to near zero.
*   The Token Machine can begin its own computation for the first output token as soon as the first few layers of the KV-cache arrive, further hiding the tail end of the transfer latency.

**Adaptive Transfer Protocol**
Recognizing that layer-wise transfer introduces synchronization overhead that may outweigh benefits for small prompts, Splitwise implements an adaptive selection mechanism.
*   The system calculates the total expected KV-cache size at the start of the request based on the known prompt length.
*   For **small prompts** (specifically those under **512 tokens** on H100 setups, as noted in **Section VI-A**), the system defaults to a **serialized transfer** because the total transfer time is negligible and the complexity of layer-wise synchronization is unnecessary.
*   For **large prompts**, the system switches to the **layer-wise transfer** mode to maximize overlap.
*   The implementation leverages **MSCCL++**, a GPU-driven communication library, to execute **zero-copy one-sided put primitives**.
*   In this model, the Prompt Machine directly writes KV-cache blocks into the Token Machine's memory over InfiniBand without requiring the Token Machine's CPU to issue receive instructions, minimizing software overhead.
*   Synchronization is handled via **semaphores** signaled over the same InfiniBand connection; the Prompt Machine signals a semaphore only after all layers for a specific request have been pushed, allowing the Token Machine to safely begin processing.

#### Cluster Provisioning and Heterogeneous Designs
Splitwise enables the construction of cluster architectures that were previously impractical, allowing operators to mix hardware generations and power configurations to optimize for specific goals (Cost, Power, or Throughput).

**Hardware Configurations**
The paper evaluates four distinct cluster designs, denoted by a two-letter code representing the hardware used for the Prompt Pool and Token Pool respectively:
1.  **Splitwise-AA:** Uses **DGX-A100** machines for both pools. This design targets users who want to maximize throughput using older, more readily available hardware, avoiding the scarcity and high cost of H100s.
2.  **Splitwise-HH:** Uses **DGX-H100** machines for both pools. This represents a high-performance homogeneous baseline where Splitwise is used primarily to improve batching efficiency rather than hardware heterogeneity.
3.  **Splitwise-HA:** Uses **DGX-H100** for the Prompt Pool and **DGX-A100** for the Token Pool. This is the primary heterogeneous design, leveraging the H100's superior FLOPs for the compute-bound prompt phase and the A100's sufficient memory bandwidth for the token phase, achieving optimal **Performance-per-Dollar**.
4.  **Splitwise-HHcap:** Uses **DGX-H100** for both pools but applies aggressive **power capping** to the Token Machines. Based on **Figure 9**, the token generation phase is insensitive to power limits; thus, Token Machines are capped at **50% of their rated power** (reducing consumption from 700W to ~350W per GPU) with negligible impact on latency, while Prompt Machines run at full power.

**Provisioning Methodology**
Determining the optimal number of machines for each pool is a non-trivial optimization problem solved via an event-driven simulator.
*   The provisioning framework takes as input: the target cluster design, a performance model (derived from profiling), workload traces (prompt/output distributions), SLO constraints, and an optimization goal (e.g., minimize cost).
*   It searches a two-dimensional space of **Number of Prompt Machines** vs. **Number of Token Machines** to find the configuration that meets all latency SLOs (P50, P90, P99 for TTFT, TBT, and E2E) while optimizing the objective function.
*   For example, in an **iso-power** scenario (fixed total power budget), the simulator might determine that a **Splitwise-HH** cluster for a coding workload requires **35 Prompt Machines and 5 Token Machines**, reflecting the compute-heavy nature of coding tasks, whereas a conversation workload might require **25 Prompt and 15 Token Machines** due to longer generation sequences.

#### Mathematical Modeling and Performance Constraints
To ensure the system meets Service Level Objectives, Splitwise relies on precise performance modeling of latency components.

**Latency Composition**
The total End-to-End (E2E) latency for a request in Splitwise is modeled as the sum of queuing delays, computation times, and transfer overheads:
$$ T_{E2E} = T_{queue\_prompt} + T_{prompt\_comp} + T_{transfer\_overlap} + T_{queue\_token} + T_{token\_gen} $$
Where:
*   $T_{prompt\_comp}$ is the time to process all input tokens on the Prompt Machine, which scales linearly with the number of prompt tokens $N_{prompt}$.
*   $T_{token\_gen}$ is the time to generate $N_{output}$ tokens sequentially on the Token Machine, which is dominated by memory bandwidth limits.
*   $T_{transfer\_overlap}$ represents the residual transfer latency after overlapping; in the ideal layer-wise case, this approaches the latency of transferring a single layer rather than the full cache.

**Throughput Constraints**
The system throughput is constrained by the bottleneck phase. For a cluster with $M_p$ prompt machines and $M_t$ token machines:
$$ \text{Throughput} = \min\left( \frac{M_p \cdot \text{Rate}_p}{\text{Avg}(N_{prompt})}, \frac{M_t \cdot \text{Rate}_t}{\text{Avg}(N_{output})} \right) $$
Where $\text{Rate}_p$ and $\text{Rate}_t$ are the token processing rates of the respective hardware configurations. Splitwise allows operators to balance $M_p$ and $M_t$ such that neither pool becomes a severe bottleneck, unlike homogeneous clusters where every machine must be sized for the worst-case combination of both phases.

#### Fault Tolerance and Practical Considerations
The paper addresses the inherent risks of distributed state management in Splitwise.
*   **Failure Handling:** In the event of a machine failure (either Prompt or Token), the current implementation defaults to **restarting the request from scratch**, similar to standard LLM serving systems like vLLM.
*   **Checkpointing Alternative:** The authors propose an optional optimization where the KV-cache generated after the prompt phase is checkpointed to an in-memory database. If a Token Machine fails, the system can retrieve the cached state and resume token generation on a different machine without recomputing the prompt, though this specific mechanism is noted as out of scope for the current prototype.
*   **Scalability:** While the central CLS could theoretically become a bottleneck for massive clusters, the authors note that LLM requests are significantly longer than typical microservice requests, reducing the frequency of scheduling decisions. They suggest that future work could incorporate partitioned or replicated scheduling strategies from prior distributed systems research to scale the CLS horizontally.
*   **Accuracy Guarantees:** Splitwise guarantees bit-wise identical output to single-machine inference because the KV-cache transfer is lossless and the mathematical operations performed on the Token Machine are identical to those on a unified machine; the split is purely a system-level optimization with no algorithmic approximation.

## 4. Key Insights and Innovations

The contributions of Splitwise extend beyond simple engineering optimizations; they represent a fundamental shift in how the community conceptualizes LLM serving architectures. While prior work focused on refining scheduling *within* a single hardware domain, Splitwise introduces a new design space where hardware heterogeneity and phase-specific resource management are first-class citizens. The following insights distinguish Splitwise's novel contributions from incremental improvements.

### 4.1 Phase-Decoupled Architecture: From Atomic Requests to Distributed Workflows
**The Innovation:**
The most profound conceptual leap in Splitwise is the rejection of the "atomic request" model. Prior systems, including state-of-the-art schedulers like Orca [81] and vLLM [51], treat an inference request as an indivisible unit that must reside on a single GPU from the first prompt token to the final output token. Even "mixed batching" approaches (Section II-D) only interleave different phases on the *same* machine; they do not separate the phases onto *different* hardware.

Splitwise is the first system to explicitly decouple the lifecycle of a single request across physically distinct machines with divergent hardware characteristics.
*   **Why it is Fundamental:** This breaks the constraint that every GPU in a cluster must be provisioned for the *union* of prompt and token requirements. By treating the request as a distributed workflow (Prompt $\rightarrow$ State Transfer $\rightarrow$ Token), Splitwise enables the construction of **heterogeneous clusters** where the prompt pool can consist of bleeding-edge, compute-heavy GPUs (e.g., H100), while the token pool utilizes older, memory-bandwidth-optimized, or power-capped hardware (e.g., A100 or capped H100).
*   **Significance:** This architectural shift unlocks efficiency gains that are mathematically impossible in homogeneous designs. As shown in **Table IV**, running token generation on an A100 incurs only a marginal latency penalty compared to an H100, yet the cost and power savings are substantial. Prior work could not exploit this because it lacked the mechanism to migrate the request state mid-execution without prohibitive latency.

### 4.2 Latency-Hiding State Migration via Layer-Wise Pipelining
**The Innovation:**
A naive implementation of phase splitting would introduce a fatal flaw: the network latency required to transfer the Key-Value (KV) cache would dominate the Time-Between-Tokens (TBT), ruining the user experience for streaming applications. Splitwise solves this not by making the network faster, but by fundamentally changing *when* the transfer occurs relative to computation.

The paper introduces **layer-wise asynchronous KV-cache transfer** (Section IV-C, Figure 11b). Instead of waiting for the entire prompt forward pass to complete before initiating a bulk transfer, Splitwise triggers the network transmission of Layer $L$'s KV-cache immediately after that specific layer's computation finishes, overlapping it with the computation of Layer $L+1$.
*   **Distinction from Prior Work:** Traditional distributed inference systems (e.g., pipeline parallelism) overlap communication between layers to scale a *single* forward pass across devices. Splitwise repurposes this technique for a completely different goal: masking the *inter-machine state handoff* between two distinct phases of inference.
*   **Significance:** This technique reduces the visible transfer overhead to near zero. As demonstrated in **Figure 14**, while serialized transfers scale linearly with prompt size (becoming a bottleneck for large contexts), the optimized layer-wise approach maintains a constant, negligible overhead (~5ms on H100). This makes the "split" transparent to the end-user, enabling the architectural benefits of heterogeneity without the latency penalty that previously made such designs impractical.

### 4.3 The "Power-Insensitive" Token Phase and Dynamic Power Capping
**The Innovation:**
Splitwise provides the first rigorous characterization demonstrating that the token generation phase is effectively **power-insensitive** within a wide operating range. While the prompt phase is compute-bound and highly sensitive to frequency and power limits (Figure 9a), the token phase is strictly memory-bandwidth bound. Consequently, reducing the GPU power cap by over 50% (from 700W to 350W) has almost no impact on token generation latency (Figure 9b).

Based on this insight, Splitwise introduces the **Splitwise-HHcap** design (Section IV-D), where token machines are aggressively power-capped while prompt machines run at full TDP.
*   **Why it is Novel:** Prior power management strategies in data centers typically apply uniform caps or react to thermal throttling. They do not exploit the *phase-specific* insensitivity of LLM workloads to dynamically reconfigure power budgets at the machine-pool level.
*   **Significance:** This allows Cloud Service Providers (CSPs) to effectively double the density of token-generation nodes within a fixed power envelope. In **Figure 19a**, the Splitwise-HHcap design achieves the same throughput as a baseline H100 cluster while consuming **25% less power**. This is a critical capability for data centers hitting the "power wall," allowing them to scale capacity without requiring new power infrastructure.

### 4.4 Morphable Resource Pools via the "Mixed Pool" Abstraction
**The Innovation:**
A static split between prompt and token machines risks resource fragmentation if the workload distribution shifts (e.g., a sudden spike in long-context prompts vs. short chat responses). Splitwise addresses this with a dynamic **Mixed Pool** abstraction (Section IV-A).

Unlike static partitioning, the Mixed Pool allows machines to temporarily fluidly switch roles. A machine originally assigned to the Token Pool can be borrowed to run prompt tasks during a surge, operating in "mixed mode" until the backlog clears, at which point it seamlessly returns to its native role.
*   **Distinction from Prior Work:** Traditional heterogeneous schedlers often require static partitioning or complex migration of entire VMs/containers, which incurs high overhead. Splitwise's approach leverages the fact that all machines are pre-loaded with the model weights; only the *scheduling policy* (Section IV-B) and *queue identity* change.
*   **Significance:** This provides robustness against workload volatility. As shown in **Figure 20**, when running a conversation trace on a cluster optimized for coding, Splitwise designs utilizing the Mixed Pool maintain performance, whereas rigid heterogeneous designs might suffer. This ensures that the efficiency gains of specialization do not come at the cost of flexibility.

### Summary of Impact
These innovations collectively move the field from **homogeneous over-provisioning** to **heterogeneous right-sizing**.
*   **Throughput:** By matching hardware to phase characteristics, Splitwise achieves **2.35× higher throughput** under iso-power/cost constraints (Abstract).
*   **Cost:** It enables the use of older, cheaper GPUs for the majority of the inference lifecycle (token generation), reducing cluster costs by **20%** for equivalent throughput.
*   **Sustainability:** It introduces a viable path to reduce the carbon footprint of LLM serving by nearly halving the power draw of token nodes without sacrificing user-perceived latency.

Splitwise does not merely optimize an existing pipeline; it redefines the topology of LLM serving, proving that the most efficient path forward is not a single, faster GPU, but a coordinated ecosystem of specialized machines.

## 5. Experimental Analysis

This section dissects the experimental validation of Splitwise. The authors do not rely solely on theoretical modeling; they combine **real-hardware implementation** on Azure cloud infrastructure with a **large-scale event-driven simulator** to prove that phase splitting works in practice and scales to production levels. The evaluation is rigorous, testing not just ideal conditions but also workload shifts, hardware heterogeneity, and strict latency Service Level Objectives (SLOs).

### 5.1 Evaluation Methodology: Traces, Models, and Baselines

To ensure the results reflect real-world usage rather than synthetic benchmarks, the evaluation is grounded in production data and state-of-the-art models.

**Datasets and Workloads**
The study utilizes **production traces** from two distinct Azure LLM inference services captured on November 11, 2023 (**Section III**):
*   **Coding Trace:** Characterized by large input prompts (median **1,500 tokens**) as users provide code context, but very short outputs (median **13 tokens**) representing code completions. This workload is heavily skewed toward the **prompt computation phase**.
*   **Conversation Trace:** Features moderate input prompts (median **1,020 tokens**) but significantly longer outputs (median **129 tokens**), with a bimodal distribution indicating some very long responses. This workload is dominated by the **token generation phase**.

These traces drive the simulator, ensuring that the mix of short/long prompts and outputs matches actual user behavior.

**Models and Hardware**
The experiments focus on two large-scale decoder-only models (**Table III**):
*   **BLOOM-176B:** 70 layers, 14,336 hidden size.
*   **Llama2-70B:** 80 layers, 8,192 hidden size.

The hardware testbed consists of **NVIDIA DGX-A100** and **DGX-H100** virtual machines on Microsoft Azure, connected via **InfiniBand** (200 Gbps for A100 pairs, 400 Gbps for H100 pairs). The authors implemented the core KV-cache transfer mechanism by modifying **vLLM** [51] and using **MSCCL++** [11] for GPU-direct communication.

**Baselines and SLOs**
Splitwise is compared against two homogeneous baselines that use standard **mixed continuous batching** (the state-of-the-art approach):
*   **Baseline-A100:** A cluster composed entirely of DGX-A100 machines.
*   **Baseline-H100:** A cluster composed entirely of DGX-H100 machines.

Performance is measured against strict latency SLOs defined as slowdowns relative to a non-contended A100 run (**Table VI**):
*   **TTFT (Time to First Token):** $2\times$ (P50), $3\times$ (P90), $6\times$ (P99).
*   **TBT (Time Between Tokens):** $1.25\times$ (P50/P90), $5\times$ (P99).
*   **E2E (End-to-End):** $1.25\times$ (P50/P90), $5\times$ (P99).

A request is only considered "successful" for throughput calculations if it meets **all nine** of these SLO constraints.

### 5.2 The Overhead of Splitting: KV-Cache Transfer Latency

The most critical risk in Splitwise is the network latency introduced by moving the KV-cache. If this overhead is too high, the benefits of heterogeneous hardware are nullified.

**Quantifying the Overhead**
**Figure 14** presents the latency of KV-cache transfers as prompt size increases.
*   **Serialized Transfer (Naive):** Latency grows linearly with prompt size. For large prompts, this adds significant delay.
*   **Layer-Wise Transfer (Optimized):** By overlapping transfer with computation, the visible latency becomes nearly constant.
    *   On **H100** (400 Gbps), the non-overlapped residue is only **~5 ms**.
    *   On **A100** (200 Gbps), the residue is **~8 ms**.

**Impact on User Experience**
**Figure 15** translates this raw transfer time into user-perceived latency (TTFT and E2E) for the coding trace.
*   **Serialized Approach:** Adds up to **3%** to E2E latency and a massive **64%** overhead to the second token's arrival time. This would be perceptible to users as a "stutter" in streaming text.
*   **Splitwise (Optimized):** Reduces the E2E impact to just **0.8%**. Crucially, the latency added to the second token is only **16.5%**, which the authors argue is "hardly perceivable" in interactive applications.

> **Key Takeaway:** The layer-wise pipelining strategy successfully hides the communication cost, making the physical split of the request transparent to the end-user.

### 5.3 Iso-Power Throughput Optimization: Doing More with the Same Energy

The first major experiment asks: *If we fix the total power budget of a data center, how much more throughput can we get with Splitwise?*

**Experimental Setup**
The authors fix the power budget to that of **40 DGX-H100 machines** (Baseline-H100).
*   **Baseline-H100:** 40 machines (Mixed batching).
*   **Baseline-A100:** Since A100s consume less power (400W vs 700W), the same budget allows for **70 machines**.
*   **Splitwise Designs:** Various combinations (e.g., Splitwise-HA uses H100s for prompts, A100s for tokens) sized to fit the same power envelope.

**Results: The Conversation Workload**
**Figure 16b** and **Figure 18a** show the results for the conversation trace (token-heavy).
*   **Baseline-H100 Failure:** As load increases beyond ~90 requests per second (RPS), the P90 TBT violates the SLO (dashed red line) because the H100s are bottlenecked by memory bandwidth during token generation, despite having excess compute.
*   **Splitwise-HHcap Victory:** This design (H100 prompts, power-capped H100 tokens) sustains the highest load. It achieves **2.35× higher throughput** than Baseline-H100 within the same power budget.
    *   *Why?* By power-capping the token machines to 50% (insignificant latency impact per **Figure 9**), the system fits more effective token-processing capacity into the power envelope.
*   **Splitwise-HA Performance:** Using A100s for tokens yields **1.18× more throughput** than Baseline-A100 at **10% lower cost**, proving that older hardware is highly effective for the token phase.

**Results: The Coding Workload**
**Figure 16a** shows the coding trace (prompt-heavy).
*   Here, **Splitwise-HA** shines. Because coding requires heavy prompt computation, the H100 prompt pool handles the load efficiently, while the A100 token pool (which is rarely busy due to short outputs) handles the tail end cheaply.
*   Baseline-H100 again suffers from high TBT tails due to mixed batching interference, whereas Splitwise isolates the phases, smoothing the latency distribution.

### 5.4 Iso-Cost and Iso-Throughput Optimizations

The paper further explores two other economic dimensions: minimizing cost for fixed throughput, and minimizing power for fixed throughput.

**Iso-Cost Throughput-Optimized Clusters**
*Goal: Maximize throughput for a fixed dollar budget.*
**Figure 18b** compares clusters with identical hourly costs.
*   **Splitwise-AA** (All A100s, split phases) delivers **1.4× higher throughput** than Baseline-H100.
    *   *Trade-off:* This comes at the cost of **2× the rack space** and **25% higher power consumption**.
    *   *Analysis:* This is a vital finding for customers who have access to older A100 inventory (which is more abundant than H100s) and care more about throughput than power density. It proves Splitwise can extend the life of legacy hardware.
*   **Splitwise-HA** offers a middle ground, providing better throughput than Baseline-A100 with lower power than Baseline-H100.

**Iso-Throughput Power-Optimized Clusters**
*Goal: Minimize power consumption for a fixed throughput target.*
**Figure 19a** shows the power savings required to hit a specific RPS.
*   **Splitwise-HHcap** is the clear winner, achieving the target throughput with **25% less power** than Baseline-H100.
*   **Splitwise-AA** achieves the same throughput as Baseline-H100 at **25% lower cost** (**Figure 19b**), reinforcing the economic argument for using heterogeneous or older hardware when phases are decoupled.

### 5.5 Robustness and Ablation Studies

A common critique of specialized systems is that they are brittle—optimized for one specific workload but failing when conditions change. The authors address this directly.

**Workload Mismatch Test**
The authors take a cluster provisioned for the **Coding** workload (many prompt machines, few token machines) and run the **Conversation** workload on it (**Figure 20a**).
*   **Result:** Splitwise designs with a **Mixed Pool** (Splitwise-AA, Splitwise-HH) adapt seamlessly. The CLS dynamically moves machines into the Mixed Pool to handle the unexpected surge in token generation, suffering **no throughput or latency impact**.
*   **Heterogeneous Penalty:** Designs with rigid hardware splits (Splitwise-HA, Splitwise-HHcap) see a modest **7% throughput reduction** because they cannot instantly change the physical hardware type (A100 vs H100) assigned to a pool. However, they still significantly outperform the Baselines.

**Model Mismatch Test**
The authors run **Llama2-70B** (smaller model) on a cluster provisioned for **BLOOM-176B** (larger model) (**Figure 20b**).
*   **Result:** All Splitwise designs handle the smaller model effortlessly, supporting much higher throughput than the baselines. The latency remains low even as load increases, demonstrating that the phase-splitting logic is model-agnostic.

**Batch Job Stress Test**
For batch jobs where latency SLOs are irrelevant, the system can be stressed to maximum capacity.
*   **Finding:** Under extreme load, Splitwise naturally devolves into a "mixed batching" mode where all machines join the Mixed Pool.
*   **Efficiency:** Even in this degraded mode, **Baseline-A100** and **Splitwise-AA** achieve **0.89 RPS/$**, outperforming the H100-based equivalents (0.75 RPS/$). This confirms that the benefits of Splitwise persist even when the sophisticated scheduling is bypassed by sheer volume.

### 5.6 Critical Assessment: Do the Experiments Support the Claims?

The experimental evidence strongly supports the paper's central thesis: **decoupling prompt and token phases enables superior resource utilization.**

**Strengths of the Evaluation:**
1.  **Realistic Traces:** Using production traces with distinct characteristics (Coding vs. Conversation) prevents the results from being artifacts of synthetic, uniform workloads.
2.  **Hardware Implementation:** The prototype on real Azure VMs validates that the theoretical "layer-wise transfer" actually works on InfiniBand, reducing overhead to negligible levels (~5ms).
3.  **Multi-Dimensional Optimization:** The paper doesn't just claim "it's faster." It provides a Pareto frontier of options:
    *   Need **Power Efficiency**? Use Splitwise-HHcap (25% savings).
    *   Need **Cost Efficiency** with old hardware? Use Splitwise-AA (1.4× throughput).
    *   Need **Maximum Throughput**? Use Splitwise-HA (2.35× gain iso-power).
4.  **Robustness:** The "mismatch" experiments (**Figure 20**) effectively counter the argument that heterogeneous clusters are too rigid for dynamic cloud environments.

**Limitations and Conditions:**
*   **Interconnect Dependency:** The success of Splitwise relies heavily on high-bandwidth interconnects (InfiniBand). The paper notes in **Section VII** that while lower bandwidth (e.g., Ethernet/RoCE) might still work due to the optimized transfer, the latency benefits would diminish. The results are specific to datacenters with fast back-plane networks.
*   **Complexity of Management:** While the simulator handles the provisioning, the real-world operational complexity of managing three pools (Prompt, Token, Mixed) and dynamically moving nodes is non-trivial. The paper acknowledges this but treats the scheduling logic as a solved algorithmic problem.
*   **Fault Tolerance Scope:** The evaluation assumes a "restart on failure" model. While the authors propose checkpointing the KV-cache to an in-memory database for faster recovery, this feature was not implemented or evaluated in the current study (**Section IV-E**). In a large cluster, frequent restarts of long conversation threads could impact user experience, a factor not fully quantified in the throughput numbers.

**Conclusion of Analysis**
The experiments convincingly demonstrate that the "memory wall" in token generation is a solvable architectural problem, not a fundamental hardware limitation. By splitting the workload, Splitwise unlocks efficiency gains (up to **2.35×**) that are mathematically inaccessible to homogeneous clusters, regardless of how advanced their batching algorithms become. The trade-offs (slightly higher complexity, dependency on fast networking) are vastly outweighed by the gains in cost, power, and throughput.

## 6. Limitations and Trade-offs

While Splitwise demonstrates significant gains in throughput, cost, and power efficiency, its architecture introduces specific dependencies, operational complexities, and boundary conditions that constrain its applicability. Understanding these limitations is crucial for determining where Splitwise is the optimal solution versus where traditional homogeneous deployments remain preferable.

### 6.1 Critical Dependency on High-Bandwidth Interconnects
The fundamental viability of Splitwise rests on the assumption that the latency cost of transferring the KV-cache between machines is negligible compared to the computation time. The paper's optimized **layer-wise transfer** (Section IV-C) successfully masks most of this latency, but only under specific network conditions.

*   **InfiniBand Requirement:** The experimental validation relies entirely on **InfiniBand** interconnects with bandwidths of **200 Gbps** (A100 pairs) to **400 Gbps** (H100 pairs) (Section V-A). At these speeds, the residual non-overlapped transfer latency is merely **5–8 ms** (Figure 14).
*   **The Ethernet/RoCE Gap:** The authors explicitly acknowledge in **Section VII** that Splitwise-HA (heterogeneous H100/A100 pools) may not be readily available with native InfiniBand connections in all cloud environments, as mixed-generation clusters often rely on Ethernet or RoCE (RDMA over Converged Ethernet).
    *   *Trade-off:* If the interconnect bandwidth drops by an order of magnitude (e.g., to standard 25–100 Gbps Ethernet), the transfer time could exceed the masking capability of the layer-wise pipeline. The paper hypothesizes that even with **10× lower bandwidth**, the system might still be beneficial due to the optimized transfer, but this is **not experimentally validated**. In low-bandwidth scenarios, the "stutter" in Time-Between-Tokens (TBT) could violate strict interactive SLOs, rendering the split counter-productive.
*   **Network Contention:** The evaluation assumes a dedicated high-bandwidth data plane. In multi-tenant cloud environments where network bandwidth is shared or oversubscribed, variable network latency could introduce jitter into the KV-cache transfer, directly impacting the determinism of the TBT metric.

### 6.2 Operational Complexity and Management Overhead
Splitwise shifts the complexity from simple homogeneous scaling to dynamic, heterogeneous resource orchestration. This introduces several operational trade-offs:

*   **Scheduler Bottleneck:** The **Cluster-Level Scheduler (CLS)** maintains global state for three distinct pools (Prompt, Token, Mixed) and makes routing decisions based on queue lengths.
    *   *Scalability Concern:* While the authors argue that long LLM request durations reduce scheduling frequency compared to microservices, they admit in **Section IV-E** that the CLS "may become a scalability bottleneck for large clusters."
    *   *Mitigation Gap:* The paper suggests that insights from prior work on partitioned or replicated scheduling (e.g., Apollo [27], Sparrow [61]) could help, but **Splitwise does not implement a distributed CLS**. The current prototype relies on a centralized controller, which poses a single point of failure and a potential throughput ceiling for massive deployments (e.g., tens of thousands of GPUs).
*   **Pool Fragmentation and Thrashing:** The **Mixed Pool** mechanism (Section IV-A) is designed to handle workload shifts by temporarily borrowing machines. However, if the workload oscillates rapidly between prompt-heavy and token-heavy bursts, machines might frequently switch roles.
    *   *Risk:* While the paper states switching pools incurs "no noticeable latency," frequent context switching at the scheduler level could lead to **cache thrashing** or suboptimal batching decisions if the system cannot stabilize long enough to fill large token batches. The evaluation in **Figure 20** shows robustness to *static* mismatches, but the dynamic performance under *rapidly fluctuating* loads is not explicitly quantified.

### 6.3 Fault Tolerance and Recovery Latency
The paper adopts a simplified approach to failure handling that may be insufficient for long-running conversational sessions.

*   **Restart-on-Failure Default:** In the current implementation, if either the Prompt or Token machine fails, the **entire request is restarted from scratch** (Section IV-E).
    *   *Impact:* For short coding completions (median 13 tokens), this is acceptable. However, for long conversation traces (median 129 tokens, with a long tail), restarting a request near completion wastes significant compute resources and incurs a high latency penalty for the user, potentially violating E2E SLOs.
*   **Unimplemented Checkpointing:** The authors propose an optimization where the KV-cache is checkpointed to an in-memory database after the prompt phase to allow resumption on a new Token Machine.
    *   *Critical Gap:* They explicitly state that **"designing safe and efficient failure recovery is out of scope for our paper"** (Section IV-E). Consequently, the reported throughput and cost metrics **do not account for the overhead of checkpointing** (network write latency, storage costs) or the probability of failure in large clusters. In a large-scale deployment, the cost of frequent restarts could erode the theoretical 20% cost savings.

### 6.4 Assumptions on Workload Characteristics
Splitwise's efficiency gains are derived from specific characteristics of current generative LLM workloads. Changes in model architecture or usage patterns could diminish its advantages.

*   **Auto-Regressive Dependency:** The approach assumes the strict separation between a parallel prompt phase and a sequential token phase inherent to standard decoder-only transformers.
    *   *Future Models:* While the authors claim extensibility to Mixture-of-Experts (MoE) models (Section VII), architectures that alter the attention mechanism (e.g., retrieval-augmented generation with massive external context lookups during generation, or non-autoregressive decoding) might blur the line between the two phases, reducing the benefit of static phase splitting.
*   **Context Caching Assumption:** The characterization assumes that **KV-caches are not reused** between requests due to security/privacy constraints (Section III).
    *   *Scenario Shift:* The paper notes in **Section VII** that future chat APIs might cache conversation context on the GPU to avoid recomputation for multi-turn dialogues. If the prompt phase is skipped for subsequent turns in a conversation, the "Prompt Pool" would be underutilized, and the system would effectively become a token-generation cluster. While Splitwise can handle this via the Mixed Pool, the specific benefit of *splitting* diminishes if the prompt phase becomes a rare event rather than a per-request constant.

### 6.5 Hardware Availability and Heterogeneity Constraints
The most powerful configuration, **Splitwise-HA** (H100 for prompts, A100 for tokens), relies on the ability to deploy heterogeneous hardware within the same high-speed network fabric.

*   **Deployment Reality:** In many public cloud offerings, instances are sold as homogeneous clusters. Creating a custom cluster with mixed GPU generations (H100 and A100) connected via low-latency InfiniBand may require specialized enterprise agreements or private cloud deployments, limiting the immediate accessibility of the optimal Splitwise configuration for smaller users.
*   **Memory Capacity Limits:** The Token Machines are memory-bound. While Splitwise allows using older GPUs (A100) which have sufficient bandwidth, it does not solve the fundamental constraint of **HBM capacity**. If model sizes grow beyond 80GB (the limit of both A100 and H100 in this study), neither pool can host the model without extensive model parallelism, which would increase the communication overhead of the split significantly. The paper does not evaluate Splitwise with models that require tensor parallelism across nodes *within* a single phase.

### Summary of Trade-offs

| Feature | Benefit | Trade-off / Limitation |
| :--- | :--- | :--- |
| **Phase Splitting** | Enables heterogeneous hardware (H100+A100) for max efficiency. | Requires high-bandwidth InfiniBand; performance degrades on slower networks. |
| **Layer-wise Transfer** | Hides KV-cache transfer latency (~5ms). | Adds implementation complexity; synchronization overhead for small prompts. |
| **Mixed Pool** | Prevents resource fragmentation during load spikes. | Centralized scheduler (CLS) may become a bottleneck; potential thrashing under volatile loads. |
| **Power Capping** | Reduces token phase power by 50% with no latency loss. | Requires fine-grained power management capabilities; only applicable to memory-bound phases. |
| **Fault Handling** | Simple implementation (restart on failure). | Inefficient for long conversations; checkpointing overhead is unquantified and unimplemented. |

In conclusion, Splitwise is not a universal panacea but a highly specialized architecture optimized for **large-scale, InfiniBand-connected data centers** running **standard auto-regressive LLMs** with **strict power or cost constraints**. Its effectiveness diminishes in environments with limited network bandwidth, highly volatile workloads that exceed scheduler capacity, or use cases requiring robust, low-latency fault recovery for long-running sessions.

## 7. Implications and Future Directions

The introduction of **Splitwise** marks a paradigm shift in Large Language Model (LLM) serving, moving the field from a reliance on monolithic, homogeneous hardware scaling toward **phase-aware, heterogeneous architectural design**. By demonstrating that the compute-intensive prompt phase and the memory-intensive token generation phase can be decoupled without perceptible latency penalties, Splitwise fundamentally alters how researchers and practitioners approach the cost, power, and scalability of generative AI infrastructure.

### 7.1 Reshaping the Landscape: From "One-Size-Fits-All" to Specialized Pipelines
Prior to Splitwise, the dominant strategy for LLM deployment was **homogeneous over-provisioning**. Systems were forced to provision every GPU for the *union* of prompt and token requirements, leading to the inefficient use of expensive, power-hungry hardware (like the H100) for memory-bound tasks where its extra compute capability sat idle.

Splitwise changes this landscape by validating three critical shifts:
*   **Hardware Decoupling is Viable:** The paper proves that a single request's lifecycle can span physically distinct machines with different hardware generations (e.g., H100 for prompts, A100 for tokens). This breaks the assumption that inference must be atomic on a single device, opening the door for **lifecycle-aware cluster design**.
*   **Power as a First-Class Constraint:** By showing that token generation is insensitive to power capping (Section IV-D, Figure 9), Splitwise introduces **dynamic power provisioning** as a viable optimization lever. Data centers can now treat power budgets not as a static limit per machine, but as a fluid resource that can be reallocated from token pools to prompt pools to maximize overall throughput.
*   **Extending Hardware Lifecycles:** The success of **Splitwise-AA** (using only older A100s) and **Splitwise-HA** (mixing generations) suggests that "legacy" hardware is not obsolete for LLM serving. Instead, older GPUs can be repurposed specifically for the token generation phase, potentially extending their useful life by years and alleviating the global shortage of flagship accelerators.

### 7.2 Enabling Follow-Up Research Directions
Splitwise opens several fertile avenues for future research, moving beyond the initial proof-of-concept to address the complexities of large-scale, production-grade deployment.

**A. Advanced Fault Tolerance and State Checkpointing**
The current implementation relies on restarting requests upon failure, which is inefficient for long conversations. Future work must address the **checkpointing overhead** explicitly mentioned as out-of-scope in Section IV-E.
*   *Research Question:* How can we design low-latency, distributed KV-cache checkpointing mechanisms (e.g., using CXL memory pools or NVMe-over-Fabrics) that allow token generation to resume on a different machine with sub-millisecond overhead?
*   *Potential Impact:* Solving this would make Splitwise robust enough for mission-critical, long-context applications where restarting a 10-minute conversation is unacceptable.

**B. Scalable Distributed Scheduling**
The paper identifies the centralized **Cluster-Level Scheduler (CLS)** as a potential bottleneck for massive clusters (Section IV-E).
*   *Research Question:* Can we adapt hierarchical or partitioned scheduling algorithms (e.g., from Sparrow [61] or Apollo [27]) to create a **distributed CLS** that maintains global optimality for phase splitting while scaling to tens of thousands of GPUs?
*   *Potential Impact:* This would enable Splitwise architectures to scale to the size of modern AI superclusters without introducing a single point of failure or contention.

**C. Interconnect Optimization and Compression**
While Splitwise works well with InfiniBand, its performance on cheaper, ubiquitous Ethernet networks remains an open question (Section VII).
*   *Research Question:* Can **KV-cache compression** techniques (e.g., quantization, sparsity exploitation [55]) be integrated into the layer-wise transfer pipeline to reduce bandwidth requirements by 4–8×?
*   *Potential Impact:* This would democratize Splitwise, allowing it to run efficiently on standard RoCE (RDMA over Converged Ethernet) networks, making heterogeneous clusters accessible to a broader range of cloud providers and on-premise deployments.

**D. Dynamic Model Adaptation**
Current Splitwise designs assume a static model per cluster. Future systems could dynamically swap models or adjust parallelism strategies based on real-time demand.
*   *Research Question:* How can we integrate **Mixture-of-Experts (MoE)** routing with Splitwise? Since MoE models activate only a subset of parameters per token, does the "memory-bound" nature of the token phase change, and how should the split strategy adapt?

### 7.3 Practical Applications and Downstream Use Cases
The principles of Splitwise have immediate applicability across various domains of AI deployment:

*   **Cost-Optimized Cloud Serving:** Cloud Service Providers (CSPs) can deploy **Splitwise-HA** clusters to offer lower-cost inference tiers. By routing token generation to older, depreciated hardware, providers can significantly reduce the cost-per-token for users, making high-volume applications (e.g., coding assistants, bulk summarization) more economically viable.
*   **Green AI and Sustainability:** For organizations with strict carbon footprint goals, **Splitwise-HHcap** offers a direct path to reduce energy consumption. By power-capping token machines by 50% with no performance loss, data centers can serve the same workload with **25% less power** (Figure 19a), directly contributing to sustainability targets without sacrificing user experience.
*   **Edge-Cloud Hybrid Architectures:** The decoupling concept could extend to edge computing. A powerful edge device (or local server) could handle the prompt phase for privacy-sensitive inputs, while offloading the lengthy token generation to a lower-power, cloud-based token pool, optimizing both latency and bandwidth usage.
*   **Legacy Infrastructure Modernization:** Enterprises sitting on large inventories of V100 or A100 GPUs can immediately upgrade their LLM serving capabilities by adopting the **Splitwise-AA** architecture. Instead of waiting for H100 availability, they can re-architect their serving stack to achieve **1.4× higher throughput** using existing assets.

### 7.4 Reproducibility and Integration Guidance
For practitioners considering the adoption of Splitwise or building upon its codebase, the following guidance outlines when and how to integrate these techniques.

**When to Prefer Splitwise:**
*   **High-Throughput Scenarios:** If your application is throughput-bound (e.g., batch processing, high-volume APIs) rather than strictly latency-bound for single requests, Splitwise offers immediate gains.
*   **Heterogeneous Hardware Availability:** If you have access to a mix of GPU generations (e.g., a fleet of A100s and a few H100s), Splitwise is the optimal strategy to utilize them jointly.
*   **Power-Constrained Environments:** If your data center is hitting power limits, the **HHcap** variant is the superior choice to maximize capacity within the existing envelope.
*   **Long-Context Workloads:** Applications with large prompt sizes (e.g., document analysis, code completion) benefit most from the layer-wise transfer optimization, as the relative overhead of state transfer diminishes compared to the compute time saved by using specialized hardware.

**When to Stick with Homogeneous Baselines:**
*   **Low-Bandwidth Networks:** If your cluster relies on standard Ethernet (<100 Gbps) without RDMA support, the KV-cache transfer latency may outweigh the benefits of phase splitting. In such cases, traditional mixed batching on homogeneous hardware remains safer.
*   **Small-Scale Deployments:** For clusters with fewer than 8–16 GPUs, the complexity of managing three pools (Prompt, Token, Mixed) and the overhead of the centralized scheduler may not yield significant efficiency gains over simple continuous batching.
*   **Strict Real-Time Constraints with Short Prompts:** For applications with very short prompts (<512 tokens) and ultra-strict TBT SLOs, the synchronization overhead of layer-wise transfer (even if small) might be undesirable compared to the simplicity of a single-machine execution, unless the hardware mismatch is extreme.

**Integration Steps:**
1.  **Profile Your Workload:** Use the provided **production traces** (available in the Azure Public Dataset [4]) or collect your own to determine the ratio of prompt-to-token tokens. This dictates the optimal split ratio for your machine pools.
2.  **Implement Layer-Wise Transfer:** If building a custom solver, integrate **MSCCL++** [11] or similar GPU-direct libraries to implement the asynchronous, layer-wise KV-cache transfer. Do not rely on serialized CPU-mediated copies.
3.  **Adopt the Simulator:** Before deploying hardware, use the open-source **SplitwiseSim** [20] to model your specific hardware mix and workload traces. This allows you to find the optimal number of prompt vs. token machines to meet your specific SLOs (TTFT, TBT, E2E).
4.  **Start with Power Capping:** The lowest-hanging fruit is implementing **Splitwise-HHcap**. If you have a homogeneous H100 cluster, simply configure your token-serving nodes to run at 50% power cap. This requires no code changes to the model serving logic, only infrastructure configuration, and yields immediate power savings.

By shifting the focus from "faster GPUs" to "smarter architectures," Splitwise provides a blueprint for the next generation of efficient, scalable, and sustainable LLM infrastructure. It demonstrates that the future of AI serving lies not just in silicon innovation, but in the intelligent orchestration of diverse computational resources.
