## 1. Executive Summary
This paper addresses the lack of unified comparison between integer (INT) and floating-point (FP) quantization by systematically evaluating trade-offs across fine-grained block-wise formats on models like Llama-3.1-8B and Qwen3 (Section 5.2). The authors reveal a critical performance crossover where fine-grained MXINT8 consistently outperforms MXFP8 in accuracy and hardware efficiency (reducing energy by 37% per Table 5), while NVINT4 surpasses NVFP4 when combined with Hadamard rotation. These findings challenge the current FP-centric hardware trajectory, advocating for fine-grained INT formats to achieve a more optimal balance of accuracy and efficiency in future AI accelerators.

## 2. Context and Motivation

To understand the significance of this study, we must first examine the evolving landscape of Large Language Model (LLM) deployment and the specific numerical challenges that arise when compressing these models. This section outlines the problem gap, the real-world stakes of hardware design, the limitations of prior quantization research, and how this paper positions itself to resolve these ambiguities.

### The Core Problem: Outliers and Quantization Granularity

The proliferation of LLMs has led to a surge in computational and memory demands, making **quantization**—the process of mapping high-precision tensors to lower bit-widths—indispensable for efficient deployment (Section 1). However, quantizing LLMs is not straightforward. A central challenge lies in the **activation distributions** of Transformer-based models, which often contain significant **outliers** (Section 1).

*   **Outliers:** These are values characterized by large magnitude but infrequent occurrence (Section 1).
*   **The Conflict:** Low-precision formats have a limited **dynamic range** (the span between the smallest and largest representable numbers). If a format cannot accommodate outliers without clipping them to the maximum value, significant information is lost, degrading model accuracy.

To handle this wide dynamic range, the AI hardware industry has increasingly pivoted towards low-precision **floating-point (FP)** formats, such as FP8 and FP4 (Section 1). Prominent examples include NVIDIA's Blackwell architecture, which favors the superior dynamic range of FP to handle outliers more gracefully than traditional **integer (INT)** formats (Section 1).

However, this industry-wide momentum towards FP formats is based on an incomplete picture. The comparative advantages of FP and INT have not been systematically evaluated across different **quantization granularities** in a unified framework (Section 1).

*   **Quantization Granularity:** This specifies how scale factors apply across a tensor (Section 2.3).
    *   **Coarse-grained (e.g., Per-tensor, Per-channel):** A single scale factor covers a large portion of the tensor. Outliers dominate the scale, forcing most non-outlier values into a narrow quantization range, increasing error.
    *   **Fine-grained (Block-wise):** The tensor is partitioned into small blocks (e.g., 32 elements), and each block has its own scale factor (Section 2.3). This isolates outliers within specific blocks, preventing them from distorting the scale for the rest of the tensor.

The critical gap this paper addresses is: **How does the performance trade-off between INT and FP evolve as granularity becomes finer?** (Section 1). Since fine-grained quantization is now a standard technique for mitigating outliers, understanding its interaction with the underlying number format is essential for effective algorithm-hardware co-design (Section 1).

### Why This Problem Matters: Hardware Co-Design and Efficiency

The stakes of this comparison extend beyond academic metrics; they directly influence the architecture of future AI accelerators.

1.  **Hardware Trajectory:** Current hardware design is increasingly FP-centric (Section 1). Building dedicated silicon for FP operations (multipliers, adders, exponent logic) is costly and fixed once manufactured.
2.  **Efficiency Trade-offs:** If INT formats are actually superior at fine-grained levels but hardware is optimized solely for FP, the industry risks suboptimal efficiency in terms of power, area, and performance (Abstract).
3.  **Algorithm-Hardware Co-Design:** Without a unified comparison, algorithm developers and hardware engineers lack clear guidance on which formats to prioritize for future generations of chips (Abstract).

The paper argues that a "one-size-fits-all FP approach is suboptimal" and that fine-grained INT formats may offer a better balance of accuracy, power, and efficiency (Abstract). Validating this claim requires rigorous evidence, as it challenges the prevailing industry trajectory.

### Limitations of Prior Approaches

Prior research and industry standards have laid the groundwork but fall short in providing a direct, granular comparison:

*   **Focus on Single Formats:** Most existing studies focus on a single format (either INT or FP) or compare them only at coarse granularities (e.g., per-channel) (Section 1). This fails to capture the nuances of block-wise quantization where local dynamic range is reduced.
*   **Existing Block Formats:**
    *   **Microscaling (MX):** Proposed by OCP, this format uses a shared `UE8M0` scale for each block of 32 elements (Section 2.4).
    *   **NVIDIA (NV) Formats:** NVIDIA proposes `NVFP4`, which reduces block size to 16 and uses an `E4M3` scale (Section 2.4).
    *   **Gap:** While hardware supports low-bit fine-grained *floating-point* formats (e.g., `MXFP8`, `MXFP4`), corresponding *integer* variants (e.g., `MXINT8`, `NVINT4`) have not been systematically evaluated alongside them (Section 2.4).
*   **Training Limitations:** Existing works on low-bit training mainly focus on FP8 training, demonstrating it can be nearly lossless (Section 5.3). There is a lack of comprehensive comparison for INT training at similar bit-widths, particularly regarding gradient bias and stability (Section 3.2).

### Positioning of This Work

This paper positions itself as the first comprehensive, systematic comparison of fine-grained INT and FP quantization within a unified framework (Section 1). It distinguishes itself from prior work through three key strategic choices:

1.  **Direct Format Pairing:** The authors introduce and evaluate integer variants (`MXINT8`, `MXINT6`, `MXINT4`, `NVINT4`) alongside their standard FP counterparts (`MXFP8`, `MXFP6`, `MXFP4`, `NVFP4`) to ensure fair comparison (Section 1, Table 1).
2.  **Theoretical Framework:** Instead of relying solely on empirical results, the paper develops a theoretical framework modeling the **Quantization Signal-to-Noise Ratio (QSNR)** for both INT and FP formats (Section 1, Section 4). This allows for a direct theoretical comparison of performance trade-offs and clarifies "crossover points" where one format surpasses the other based on data statistics (specifically the **crest factor** `$\kappa$`).
3.  **Holistic Evaluation:** The study covers multiple levels of analysis:
    *   **Tensor-wise:** Analyzing intermediate tensors (weights, activations, gradients) across the compute flow (Section 5.1, Figure 1).
    *   **Inference:** Direct-cast inference across 12 LLMs ranging from 0.6B to 235B parameters (Section 5.2).
    *   **Training:** Low-bit training comparisons including a proposed **symmetric clipping method** to resolve gradient bias in INT training (Section 5.3, Section 3.2).
    *   **Hardware Cost:** A comparative hardware cost analysis evaluating energy and area efficiency (Section 6).

By bridging the gap between theoretical analysis, empirical benchmarking, and hardware cost modeling, this work aims to provide the clear guidance necessary for future AI accelerator design (Section 1).

### 3.4 Detailed, sentence-based technical breakdown

This is a comparative systems analysis paper that evaluates two competing quantization paradigms—integer versus floating-point formats—by building matched format pairs, deriving theoretical performance bounds, and validating across inference and training workloads. The core idea is that fine-grained block-wise quantization reduces the local dynamic range within each block, which changes the relative advantage between INT and FP formats compared to coarse-grained scenarios.

**Quantization Format Specifications and Matched Pairs.** The paper defines four integer variants to match existing floating-point formats, ensuring fair comparison across all experiments. For the Microscaling (MX) family with block size 32, the authors introduce `MXINT8`, `MXINT6`, and `MXINT4` to correspond with `MXFP8` (E4M3), `MXFP6` (E2M3), and `MXFP4` (E2M1) respectively as shown in Table 1. For the NVIDIA (NV) family with block size 16, `NVINT4` is introduced to match `NVFP4`. All MX formats use a `UE8M0` scale factor, which is an 8-bit unsigned floating-point format with eight exponent bits and zero mantissa bits that represents scale as powers of two. The NV formats use an `E4M3` scale factor with a second-level per-tensor scale to prevent overflow. The representable ranges differ significantly: `MXINT8` covers `$\pm 127$` with dynamic range 127, while `MXFP8` covers `$\pm 448$` with dynamic range `$1.75 \times 2^{17}$`. For 4-bit formats, `MXINT4` covers `$\pm 7$` while `MXFP4` covers `$\pm 6$`, but FP formats gain advantage through subnormal numbers that provide finer granularity near zero.

**Compute Flow for Low-Bit Forward and Backward Propagation.** Figure 1 illustrates the six quantization operations that occur in one linear layer during training, and understanding this flow is essential because different tensors exhibit different crest factors that determine INT versus FP performance. The forward pass quantizes activations `$X$` (operation ①) and weights `$W$` (operation ②) before the GEMM (general matrix multiply) produces output `$Y$`. The backward pass quantizes output gradients `$dY$` (operation ③) and transposed weights `$W^T$` (operation ④) to compute input gradients `$dX$`, then quantizes transposed activations `$X^T$` (operation ⑤) and transposed output gradients `$dY^T$` (operation ⑥) to compute weight gradients `$dW$`. Block-wise quantization requires tensors to be quantized along the GEMM reduction dimension to gain hardware benefits, which means operations ① and ⑤, ② and ④, and ③ and ⑥ are quantized along different axes. The paper separately analyzes quantization error for all six operations in Section 5.1 because activation tensors (①, ⑤) typically have higher crest factors than weight tensors (②, ④), making them more challenging for INT formats.

**Scale Factor Computation and Precision Considerations.** The scale factor `$s$` in quantization normalizes high-precision values to the target integer or floating-point range, and how this scale is computed and represented significantly impacts quantization error. For AbsMax quantization, the scale is computed as `$s = \frac{\text{AbsMax}(X)}{Q_{\text{max}}}$` where `AbsMax(X)` is the maximum absolute value within the group sharing a single scale factor and `$Q_{\text{max}}$` is the maximum representable value of the quantized type from Table 1. For MX formats, the high-precision scale factor is converted to `UE8M0` format using `$s' = 2^{\text{clip}(\lceil \log_2(s) \rceil, -127, 127)}$` where the ceiling operation rounds up to avoid clipping error. The paper chooses to round up rather than round down (as in prior OCP work) because rounding down introduces extra clipping error when the scale is slightly underestimated. For NV formats using `E4M3` scale, the scale factor closely matches BFloat16 precision, so the overhead factor `$\rho \approx 1$` compared to `$\rho \in [1, 2)$` for `UE8M0` scales.

**Theoretical QSNR Framework for INT Formats.** The paper develops Theorem 1 to model Quantization Signal-to-Noise Ratio for integer quantization, where QSNR measures the ratio of original signal power to quantization noise power expressed in decibels. The QSNR metric is defined as `$\text{QSNR} = -10 \log_{10} \left( \frac{\|X - X_q\|^2}{\|X\|^2} \right)$` where higher QSNR indicates better preservation of the original vector's magnitude and direction. Under the assumption of block vectors `$X \in \mathbb{R}^k$` with independently and identically distributed entries `$X_i \sim \mathcal{N}(0, \sigma^2)$`, the crest factor is defined as `$\kappa := \frac{\max(|X|)}{\sigma}$` where `$\sigma$` is the block root-mean-square. For `$b$`-bit INT quantization with `UE8M0` scale, the QSNR in decibels is approximately `$\text{QSNR}_{\text{INT}} \approx 4.78 + 6.02 b - 20 \log_{10}(\rho) - 20 \log_{10}(\kappa)$`. This equation reveals four key insights: each additional bit provides approximately 6.02 dB improvement, `UE8M0` scaling incurs up to 6.02 dB loss due to `$\rho \in [1, 2)$`, larger crest factor `$\kappa$` reduces QSNR, and smaller blocks typically reduce `$\kappa$` and improve QSNR. For `E4M3` scale (used in `NVINT4`), the formula becomes `$\text{QSNR}_{\text{INT}} \approx 4.78 + 6.02 b - 20 \log_{10}(\kappa) + 10 \log_{10} \left( \frac{g}{g-1} \right)$` where `$g$` is block size, and the last term accounts for one near error-free element per block since `$\rho \approx 1$`.

**Theoretical QSNR Framework for FP Formats.** Theorem 2 models QSNR for floating-point quantization, which differs from INT because FP error depends on mantissa resolution and subnormal number handling rather than uniform step size. For FP quantization with `UE8M0` scale, the QSNR is approximately `$\text{QSNR}_{\text{FP}} \approx -10 \log_{10} \left( \alpha_M w_{\text{norm}} + \beta (\rho \kappa)^2 p_{\text{sub}} \right)$` where `$\alpha_M = \frac{1}{24 \cdot 2^{2M}}$` is the mantissa resolution term with `$M$` mantissa bits, `$w_{\text{norm}}$` is the fraction of signal energy in normal FP numbers, `$\beta = \frac{2^{2(1-B-M)}}{12 Q_{\text{max}}^2}$` with `$B$` exponent bias, and `$p_{\text{sub}}$` is the probability a value encodes as subnormal. This equation reveals that mantissa bit width sets the upper bound on FP QSNR, and with ample dynamic range where `$w_{\text{norm}} \approx 1$` and `$p_{\text{sub}} \approx 0$`, QSNR approximates `$13.80 + 6.02 M$` dB independent of block granularity. For `E4M3` scale (used in `NVFP4`), the formula adjusts to `$\text{QSNR}_{\text{FP}} \approx -10 \log_{10} \left( \alpha_M (w_{\text{norm}} - \frac{\kappa^2}{g}) + \beta \kappa^2 p_{\text{sub}} \right)$` where the `$\frac{\kappa^2}{g}$` term excludes the block maximum from error budget since it maps with near-zero error. Figure 3 plots these theoretical QSNR curves across crest factors, revealing crossover points where INT surpasses FP: `MXINT8` outperforms `MXFP8` when `$\kappa < 7.55$`, `MXINT6` outperforms `MXFP6` when `$\kappa < 1.96$`, `MXINT4` outperforms `MXFP4` when `$\kappa < 2.04$`, and `NVINT4` outperforms `NVFP4` when `$\kappa < 2.39$`.

**Crest Factor Statistics and Their Implications.** Table 2 reports measured crest factor statistics across 10,752 tensors from Llama-3.1-8B, and these statistics determine which format wins in practice. For channel-wise quantization (block size -1), the 75th percentile crest factor is 11.97, which exceeds all crossover points, explaining why FP outperforms INT at coarse granularity. For MX-format with block size 32, the 75th percentile crest factor is 2.96, which is well below the `MXINT8` versus `MXFP8` crossover of 7.55 but above the `MXINT6`/`MXINT4` crossovers of 1.96 and 2.04. After applying random Hadamard rotation (a 32×32 orthogonal transformation that spreads outlier energy across dimensions), the 75th percentile decreases from 2.96 to 2.39 for MX formats and from 2.39 to 2.11 for NV formats with block size 16. This rotation is critical because it enables `NVINT4` to surpass `NVFP4` post-rotation, as the reduced crest factor falls below the 2.39 crossover point. Figure 4 confirms these theoretical predictions with empirical QSNR measurements: `MXINT8` achieves 40.35 dB average QSNR versus 31.50 dB for `MXFP8` (100% win rate), while `MXINT6` and `MXINT4` consistently lag behind their FP counterparts with win rates below 1%.

**Symmetric Clipping Method for INT Training.** The paper introduces a symmetric clipping method to resolve gradient bias in fine-grained low-bit INT training, which is essential because standard two's complement integers have asymmetric ranges. For a `$b$`-bit signed integer, the standard range is `$Q_{\text{min}} = -2^{b-1}$` and `$Q_{\text{max}} = 2^{b-1} - 1$`, meaning there is one extra negative value (e.g., -128 for INT8). Figure 2 demonstrates that this asymmetric range degrades INT8 training loss because finer-grained quantization creates more blocks where values fall into the unique negative endpoint `$Q_{\text{min}}$`, introducing persistent negative bias in gradients. The paper adopts a symmetric integer range where `$Q_{\text{min}} = -(2^{b-1} - 1)$` and `$Q_{\text{max}} = 2^{b-1} - 1$` (e.g., [-127, 127] for INT8 instead of [-128, 127]). Table 10 in the appendix confirms this choice: with BFloat16 scale factors, asymmetric [-128, 127] range achieves 3.1354 loss at block size 32 while symmetric [-127, 127] achieves 3.1251 loss. The asymmetry problem is exacerbated by BFloat16 arithmetic precision: even when the scale is set to `AbsMax(X)/127`, the minimum value in a group can still map to -128 due to rounding, and 16.82% of values incorrectly map to -128 in BFloat16 precision according to Algorithm 1 in the appendix.

**Direct-Cast Inference Evaluation Protocol.** For inference comparison, the paper performs direct-cast quantization from pretrained BFloat16 models without fine-tuning, quantizing all forward GEMM operations. The evaluation covers 12 LLMs spanning 0.6B to 235B parameters: Qwen3-0.6B/1.7B/4B/8B/14B/32B/30B-A3B/235B-A22B and Llama-3.1-8B/70B and Llama-3.2-1B/3B. The metric is KL divergence on WikiText2 between each quantized model and its BFloat16 counterpart, computed over the softmax distribution restricted to the top-25 logits to reduce noise. Table 3 summarizes results: without rotation, `MXINT8` wins on all 12 models against `MXFP8`, while `MXINT6`, `MXINT4`, and `NVINT4` lose to their FP counterparts. With random Hadamard rotation (block size 32 for MX, 16 for NV), both `MXINT8` and `NVINT4` win on all 12 models, while `MXINT6` wins only 1 of 12 and `MXINT4` loses all 12. Table 12 and Table 13 in the appendix provide per-model KL divergence numbers multiplied by `$10^6$`: for example, `MXINT8` achieves 191 versus `MXFP8`'s 579 on Qwen3-0.6B, demonstrating nearly 3× lower divergence.

**Low-Bit Training Evaluation Protocol.** For training comparison, the paper focuses on 8-bit settings (`MXINT8` versus `MXFP8`) because FP8 training is already demonstrated as nearly lossless in prior work. The experiments train 1B and 3B Llama-3-style models on the OLMo2-Mix-1124 pretraining dataset with 100B and 200B training tokens respectively. Model architectures use Group Query Attention and SwiGLU feed-forward modules with hyperparameters in Table 9: 1B model has 16 layers, 2048 hidden size, 8192 FFN hidden size, 32 attention heads, 8 KV heads, batch size 512, max learning rate 6e-4 with cosine schedule and 500 warmup steps. Performance is measured using training loss (smoothed with exponential moving average coefficient 0.9) and task accuracy on six common-sense reasoning tasks (WinoGrande, HellaSwag, Arc_Challenge, Arc_Easy, PIQA, Openbookqa) via 5-shot evaluation with lm_eval. Figure 5 shows loss curves where `MXINT8` and `MXFP8` nearly overlap with BFloat16 baseline, but `MXINT8` consistently achieves approximately 0.001 lower loss than `MXFP8`. Table 4 confirms this: for 1B model at 100B tokens, `MXINT8` achieves 2.6758 loss versus 2.6767 for `MXFP8` and 57.02% average accuracy versus 56.86% for `MXFP8`.

**Hardware Cost Modeling Methodology.** Section 6 and Appendix C describe a gate-level model estimating chip area and energy for a Matrix-Multiply Unit supporting MX formats at matched throughput. The model includes three components: the MAC (multiply-and-accumulate) unit with `$k$` lanes, a shared FP32 accumulator, and a dequantizer, while excluding the quantizer to isolate multiplication and accumulation costs. For MX formats, `$k=32$` lanes; for NV formats, `$k=16$` lanes. The mantissa aligner width `$n$` is set to `$n = \min(2^{x+1} + 2^y, \text{psum\_bit\_width})$` where `$x$` and `$y$` are exponent and mantissa widths (with `$x=0$` for INT) and psum_bit_width=24. Table 6 reports gate complexity: INT multipliers require `$k(y+1)^2$` cells versus FP multipliers requiring `$k(x+y+1)^2$` cells, and FP adders require exponent comparator/subtractor, barrel aligner, and normalizer logic that INT adders avoid. The evaluation considers reuse schemes: for INT, two `int8×(u)int4` lanes reconfigure as either int8 or int4 (lowest area cost); for FP, one `e4m3` lane serves as `e2m1` path directly. Table 5 shows results normalized to `MXFP8` baseline: `MXINT8` reduces energy by 37% (0.63×) and area by 21% (0.79×), while `NVINT4` reduces energy by 66% (0.34×) and area by 62% (0.38×). For mixed 8-bit + 4-bit configurations matching NVIDIA Blackwell's 1:2 throughput ratio, `MXINT8+NVINT4` reduces area by 34% (0.66×) relative to `MXFP8+NVFP4` because circuit reuse is simpler in the INT pipeline.

**Design Choices and Their Rationale.** The paper makes several critical design choices that enable fair comparison and valid conclusions. First, matched format pairs (`MXINT8`/`MXFP8`, `NVINT4`/`NVFP4`) with identical block sizes ensure differences arise from number format rather than granularity. Second, rounding up the `UE8M0` scale (Equation 9) rather than rounding down (as in OCP) avoids systematic underestimation that would unfairly penalize MX formats. Third, symmetric integer clipping resolves gradient bias that would otherwise invalidate INT training comparisons, as Figure 2 demonstrates asymmetric clipping increases loss by 0.0103 at block size 32. Fourth, random Hadamard rotation is applied consistently to both INT and FP formats, ensuring rotation benefits are not attributed to format choice. Fifth, the hardware model excludes the quantizer because quantization can be fused into the accumulation pipeline (two 8-bit integer additions for `UE8M0` scale or two floating-point multiplications for `E4M3` scale), and including it would conflate quantization overhead with MAC costs. Sixth, FP32 accumulation is used for all formats to prevent error growth and preserve scalability, which is standard in TPU-like designs. These choices collectively ensure the INT versus FP comparison isolates format differences rather than confounding implementation details.

## 4. Key Insights and Innovations

This paper moves beyond empirical benchmarking to establish fundamental design principles for low-precision AI hardware. While prior work often treated quantization formats as fixed constraints, this study reveals that the optimal choice depends dynamically on data statistics and granularity. Below, we dissect the four most significant contributions, distinguishing between incremental improvements and fundamental shifts in understanding.

### 1. The Crest Factor Crossover Theory (Fundamental Innovation)
Prior research generally assumed floating-point (FP) formats are inherently superior for handling outliers due to their exponential dynamic range. This paper challenges that assumption with a theoretical framework showing that **integer (INT) formats surpass FP formats when the data crest factor `$\kappa$` drops below a specific threshold**.

*   **What Changed:** Instead of treating FP as the default for outliers, the authors derive closed-form Quantization Signal-to-Noise Ratio (QSNR) equations (Theorem 1 and Theorem 2 in Section 4.1) that explicitly model the trade-off between uniform INT precision and non-uniform FP precision.
*   **The Insight:** Fine-grained block-wise quantization isolates outliers, drastically reducing the local crest factor `$\kappa$` within each block. As shown in **Figure 3**, there is a critical crossover point: for 8-bit formats, INT outperforms FP when `$\kappa < 7.55$`. Since fine-grained blocks (size 32) typically exhibit `$\kappa \approx 2.96$` (Table 2), the data naturally falls into the INT-superior region.
*   **Significance:** This is a fundamental shift in hardware design logic. It proves that **granularity mitigates the need for FP dynamic range**. If the block size is small enough, the uniform precision of INT becomes more valuable than the dynamic range of FP. This provides a mathematical justification for choosing INT formats in fine-grained accelerators, rather than relying on heuristic assumptions.

### 2. Symmetric Clipping for Lossless INT Training (Algorithmic Innovation)
While FP8 training has been demonstrated as nearly lossless in prior work, low-bit INT training historically suffered from stability issues. This paper identifies and resolves a specific numerical bias that previously hindered INT training competitiveness.

*   **What Changed:** Standard two's complement integers have an asymmetric range (e.g., `[-128, 127]` for 8-bit). The authors发现 (discover) that in fine-grained quantization, this asymmetry causes a persistent negative bias in gradients because the unique minimum value (`-128`) is triggered more frequently across many small blocks.
*   **The Insight:** By enforcing a **symmetric clipping range** (e.g., `[-127, 127]`), the gradient bias is eliminated. **Figure 2** illustrates that asymmetric clipping degrades training loss, while symmetric clipping restores it. **Table 10** (Appendix D.2) confirms this: asymmetric INT8 training yields a loss of `3.1354`, while symmetric INT8 achieves `3.1251` at block size 32.
*   **Significance:** This enables **MXINT8 training to be nearly lossless**, matching or slightly exceeding MXFP8 performance (Table 4). Without this fix, INT training would remain inferior, validating the industry's FP-centric trajectory. This innovation unlocks INT formats for training workloads, not just inference, expanding their utility in future accelerators.

### 3. Rotation-Dependent Viability for 4-bit Formats (Strategic Insight)
The study reveals a nuanced distinction between 8-bit and 4-bit formats regarding their dependency on algorithmic preprocessing. This insight is critical for system architects deciding whether to support INT4 hardware natively.

*   **What Changed:** Prior comparisons often evaluated formats in isolation. This paper evaluates them with and without **Hadamard rotation**, a technique that spreads outlier energy across dimensions to reduce `$\kappa$`.
*   **The Insight:** There is a divergence in robustness. **MXINT8** outperforms MXFP8 on all 12 models *without* rotation (Table 3). In contrast, **NVINT4** loses to NVFP4 on all 12 models without rotation but wins on all 12 *with* rotation.
*   **Significance:** This indicates that **8-bit INT is robust enough to stand alone**, whereas **4-bit INT requires algorithmic support (rotation) to be viable**. For hardware designers, this implies that an INT4 accelerator must either assume rotation is performed in software or integrate rotation logic on-chip. It warns against adopting INT4 hardware without considering the full algorithmic stack, as raw INT4 quantization is insufficient against FP4 in the presence of outliers.

### 4. Quantified Hardware Efficiency Gap (Practical Innovation)
Beyond accuracy, the paper provides a rigorous gate-level cost analysis that quantifies the efficiency penalty of supporting FP formats. This moves the argument from "INT is accurate enough" to "INT is significantly cheaper."

*   **What Changed:** Previous hardware claims often relied on high-level estimates. This study uses a detailed gate-complexity model (Appendix C) accounting for multipliers, adders, aligners, and normalizers across 32 lanes (MX) or 16 lanes (NV).
*   **The Insight:** At matched throughput, INT formats eliminate the need for complex FP-specific logic (exponent comparison, mantissa alignment, normalization). **Table 5** shows that **MXINT8 reduces energy by 37%** (`0.63x`) and **NVINT4 reduces energy by 66%** (`0.34x`) compared to their FP counterparts. In mixed-precision configurations (8-bit + 4-bit), the INT combination reduces area by 34% (`0.66x`).
*   **Significance:** This provides the economic justification for shifting hardware trajectories. Even if accuracy were identical (which the paper shows it is not—INT often wins), the **37% energy reduction for 8-bit** represents a massive operational cost saving for data centers. This insight challenges the industry to reconsider the area/energy budget allocated to FP units in future architectures like successors to NVIDIA Blackwell.

## 5. Experimental Analysis

This section critically examines the empirical evidence provided in the paper. We move beyond the theoretical framework established in Section 4 to analyze how the proposed formats perform in real-world inference and training scenarios. The goal is to determine whether the experimental data convincingly supports the claim that fine-grained integer formats can supersede floating-point formats in specific regimes. We break this down by evaluation methodology, quantitative results across inference and training, hardware cost validation, and robustness checks.

### 5.1 Evaluation Methodology

To ensure a fair comparison between Integer (INT) and Floating-Point (FP) formats, the authors employ a multi-tiered evaluation strategy. This strategy is designed to isolate the impact of the number format from other variables like model architecture or quantization granularity.

*   **Datasets:**
    *   **Inference:** The primary metric is computed on **WikiText2** [25], a standard language modeling benchmark. This dataset is used to calculate both KL divergence and perplexity.
    *   **Training:** Models are trained on **OLMo2-Mix-1124** [33], a large-scale pretraining dataset. The scale is significant: **100B tokens** for 1B parameter models and **200B tokens** for 3B parameter models (Section 5.3).
*   **Models:**
    *   The inference study covers **12 LLMs** ranging from **0.6B to 235B parameters** (Section 5.2). This includes the Qwen3 series (e.g., 0.6B, 8B, 235B-A22B) and Llama-3 series (e.g., 8B, 70B). This breadth ensures results are not artifacts of a specific model size or architecture.
    *   The training study uses **Llama-3-style models** (1B and 3B) to maintain consistency with industry standards (Section 5.3).
*   **Metrics:**
    *   **KL Divergence:** Computed on the softmax distribution restricted to the **top-25 logits** of the BFloat16 baseline. This focuses the metric on the most probable tokens, reducing noise from low-probability tails (Section 5.2).
    *   **Task Accuracy:** Evaluated using **lm_eval** [17] via **5-shot evaluation** across six common-sense reasoning tasks (e.g., WinoGrande, HellaSwag, ARC) (Section 5.3).
    *   **Hardware Cost:** Estimated using a gate-level model (Appendix C) that calculates energy and area for a Matrix-Multiply Unit (MMU) at **matched throughput** (Section 6).
*   **Baselines:**
    *   **Numerical:** All quantized models are compared against their **BFloat16 (BF16)** counterparts.
    *   **Formats:** INT formats (e.g., `MXINT8`) are directly paired with their FP equivalents (e.g., `MXFP8`) as defined in Table 1.
    *   **Preprocessing:** Experiments are run both with and without **random Hadamard rotation** to test robustness against outliers (Section 5.2).

### 5.2 Inference Performance Results

The inference experiments provide the most direct evidence for the "crossover" theory introduced in Section 4. The results are nuanced: INT dominates at 8-bit, but FP retains an advantage at 4-bit unless specific mitigation techniques are used.

#### 8-Bit Formats: Clear INT Superiority
The data strongly supports the claim that `MXINT8` is superior to `MXFP8` in fine-grained settings.
*   **Win Rate:** As shown in **Table 3**, `MXINT8` outperforms `MXFP8` on **all 12 models** (100% win rate), both with and without Hadamard rotation.
*   **Magnitude of Improvement:** The advantage is not marginal. In **Table 12** (Appendix D.3), for the **Qwen3-0.6B** model, `MXINT8` achieves a KL divergence of **191** (×10⁶), whereas `MXFP8` scores **579**. This indicates that `MXINT8` preserves the original model distribution roughly **3× better** than `MXFP8` in this configuration.
*   **Perplexity:** **Table 14** confirms this trend in perplexity. For **Qwen3-8B**, `MXINT8` achieves **6.5174**, compared to **6.5444** for `MXFP8`. While the absolute difference seems small, in language modeling, consistent improvements across 12 models suggest a systematic advantage rather than noise.

#### 4-Bit Formats: Conditional INT Viability
The 4-bit results validate the claim that INT requires algorithmic support (rotation) to compete with FP at very low bit-widths.
*   **Without Rotation:** `NVINT4` loses to `NVFP4` on **all 12 models** (Table 3). For **Llama-3.1-8B** (Table 13), `NVINT4` has a KL divergence of **4224**, while `NVFP4` is lower at **3718**. This confirms that raw 4-bit integer quantization struggles more with outliers than FP.
*   **With Rotation:** The trend flips completely. With random Hadamard rotation, `NVINT4` wins on **all 12 models**. For **Llama-3.1-8B**, `NVINT4` KL divergence drops to **3609**, surpassing `NVFP4`'s **4752**.
*   **Interpretation:** This conditional success is critical. It implies that adopting `NVINT4` hardware is viable *only if* the software stack includes rotation techniques like Hadamard transforms. Without this co-design, FP remains the safer choice for 4-bit.

#### 6-Bit Formats: The Weak Link
Interestingly, `MXINT6` does not share the success of `MXINT8`.
*   **Performance:** `MXINT6` loses to `MXFP6` on **11 out of 12 models** without rotation, and **11 out of 12** even with rotation (Table 3).
*   **Reasoning:** Referring back to the theoretical crossover in **Figure 3**, `MXINT6` only outperforms `MXFP6` when the crest factor `$\kappa < 1.96$`. However, **Table 2** shows the typical crest factor for block size 32 is **2.96** (Q3). Even with rotation, it only drops to **2.39**, which remains above the 1.96 threshold. This alignment between theory and experiment strengthens the validity of the QSNR model.

### 5.3 Training Performance Results

The training experiments address whether INT formats can support *learning*, not just inference. This is a higher bar, as gradient errors can accumulate over billions of steps.

*   **Loss Curves:** **Figure 5** displays training loss over 100B tokens for 1B models. The curves for `MXINT8`, `MXFP8`, and `BF16` are nearly indistinguishable to the naked eye. However, the enlarged view reveals `MXINT8` consistently maintains a loss approximately **0.001 lower** than `MXFP8`.
*   **Final Metrics:** **Table 4** provides the concrete numbers.
    *   **1B Model (100B tokens):** `MXINT8` achieves a loss of **2.6758**, compared to **2.6767** for `MXFP8` and **2.6727** for `BF16`. The gap between `MXINT8` and `BF16` is only **0.0031**, validating the "nearly lossless" claim.
    *   **Accuracy:** On common-sense reasoning tasks, `MXINT8` averages **57.02%** accuracy, slightly edging out `MXFP8` at **56.86%**.
    *   **3B Model (200B tokens):** The trend holds. `MXINT8` loss is **2.4812** vs `MXFP8` at **2.4821**.
*   **Significance:** These results demonstrate that the **symmetric clipping method** (discussed in Section 3.2) successfully mitigates the gradient bias that historically plagued INT training. Without this fix, INT training would likely diverge or converge to a poorer minimum.

### 5.4 Hardware Efficiency Results

The paper complements accuracy metrics with a hardware cost analysis to argue for the *practicality* of INT formats. The results quantify the efficiency penalty of supporting FP logic.

*   **Energy Efficiency:** **Table 5** normalizes costs to `MXFP8` (1.0x).
    *   `MXINT8` consumes **0.63x** the energy of `MXFP8` (a **37% reduction**).
    *   `NVINT4` consumes **0.34x** the energy of `NVFP4` (a **66% reduction**).
*   **Area Efficiency:**
    *   `MXINT8` requires **0.79x** the area of `MXFP8`.
    *   `NVINT4` requires **0.38x** the area of `NVFP4`.
*   **Mixed-Precision Scenario:** Real-world chips often support multiple precisions. The paper models a mixed 8-bit + 4-bit configuration (matching NVIDIA Blackwell's 1:2 throughput ratio).
    *   The `MXINT8+NVINT4` configuration reduces area to **0.66x** of the `MXFP8+NVFP4` baseline.
    *   **Reasoning:** As noted in Section 6, circuit reuse is simpler in the INT pipeline. FP addition requires complex alignment and normalization logic that INT addition avoids.
*   **Implication:** Even if accuracy were identical (which the paper shows it is not—INT often wins), the **37% energy saving** for 8-bit operations provides a compelling economic argument for data center deployments.

### 5.5 Robustness Checks and Ablation Studies

To ensure the results are not artifacts of specific implementation choices, the authors conduct several ablation studies. These are critical for validating the *necessity* of their proposed methods.

#### Necessity of Symmetric Clipping
The most critical ablation is the clipping range for INT training.
*   **Experiment:** **Table 10** (Appendix D.2) compares asymmetric clipping `[-128, 127]` vs. symmetric clipping `[-127, 127]` for INT8 training.
*   **Result:** With BFloat16 scale factors at block size 32, asymmetric clipping yields a loss of **3.1354**, while symmetric clipping achieves **3.1251**.
*   **Granularity Impact:** The asymmetry penalty worsens with finer granularity. For per-channel quantization, asymmetric loss is **3.2544** vs symmetric **3.2560** (negligible difference), but at block size 32, the gap widens. This confirms the hypothesis that fine-grained quantization triggers the unique negative endpoint (`-128`) more frequently, causing bias.
*   **Numerical Stability:** **Table 11** (Appendix D.2) reveals a subtle pitfall in BFloat16 arithmetic. When scaling values to map to 127, **16.82%** of values incorrectly map to **-128** due to precision limits. This numerical instability justifies the forced symmetric clipping step.

#### Crest Factor Validation
The paper validates its theoretical QSNR model against empirical data.
*   **Measurement:** **Figure 4** plots measured QSNR against crest factor for 10,752 tensors.
*   **Alignment:** The empirical crossover points match the theoretical predictions in **Figure 3**. For example, `MXINT8` empirically outperforms `MXFP8` across the observed crest factor range (mostly `$\kappa < 3.5$`), consistent with the theoretical crossover at `$\kappa = 7.55$`.
*   **Rotation Effect:** **Table 2** shows Hadamard rotation reduces the 75th percentile crest factor for MX formats from **2.96** to **2.39**. This reduction is what enables `NVINT4` to cross the performance threshold against `NVFP4` (crossover `$\kappa = 2.39$`).

### 5.6 Critical Assessment of Claims

Do the experiments convincingly support the paper's conclusions?

*   **Claim 1: "MXINT8 is superior to MXFP8."**
    *   **Verdict:** **Strongly Supported.** The 100% win rate across 12 models (Table 3) and consistent training loss improvements (Table 4) leave little room for doubt. The hardware savings (Table 5) further reinforce this.
*   **Claim 2: "NVINT4 can surpass NVFP4 with rotation."**
    *   **Verdict:** **Supported, but Conditional.** The win rate jumps from 0% to 100% with rotation (Table 3). However, this dependency is a limitation. It means `NVINT4` hardware cannot be treated as a drop-in replacement for `NVFP4` without software modifications.
*   **Claim 3: "Fine-grained INT offers a better balance of accuracy and efficiency."**
    *   **Verdict:** **Supported for 8-bit, Nuanced for 4-bit.** For 8-bit, INT wins on both accuracy and efficiency. For 4-bit, INT wins on efficiency but requires algorithmic help to win on accuracy. The 6-bit results (`MXINT6`) serve as a honest failure case, showing that INT is not universally superior at all bit-widths.
*   **Limitations:**
    *   **Training Scale:** The training experiments are limited to 1B and 3B models. While indicative, performance on 70B+ models during training is not explicitly shown (only inference). Scaling laws might shift the trade-offs at larger scales.
    *   **Hardware Model:** The hardware cost is based on a gate-level model (Appendix C), not silicon measurements. While rigorous, real-world chip constraints (memory bandwidth, control logic) might alter the exact energy ratios.

In summary, the experimental analysis is robust. The authors do not hide the conditions under which INT fails (e.g., 6-bit, 4-bit without rotation). This transparency strengthens the core argument: fine-grained INT is not a magic bullet, but where it works (specifically 8-bit), it offers a compelling advantage over the industry's current FP-centric trajectory.

## 6. Limitations and Trade-offs

While this paper provides compelling evidence that fine-grained integer formats can outperform floating-point counterparts in specific regimes, a critical educator must highlight where the approach relies on assumptions, where it falls short, and what trade-offs remain unresolved. Understanding these limitations is essential for researchers and engineers who might consider adopting these findings for production systems or future hardware designs.

### 6.1 Training Scale and Generalizability

The most significant constraint in the empirical validation is the scale of the training experiments. While the inference evaluation covers models up to **235B parameters** (Section 5.2), the low-bit training experiments are limited to **1B and 3B parameter models** (Section 5.3, Table 4).

*   **Why This Matters:** Quantization behavior does not always scale linearly. As model size increases, the distribution of activations and gradients can shift, potentially altering the **crest factor `$\kappa$`** statistics that drive the INT vs. FP crossover theory (Section 4.1).
*   **The Risk:** The paper demonstrates that `MXINT8` training is nearly lossless for 1B/3B models (loss difference `$\approx 0.001$` vs. BF16 in Figure 5). However, it remains an open question whether this stability holds for 70B+ models trained from scratch. Prior work on scaling laws suggests that quantization noise can accumulate differently at massive scales [8, 19]. Without training data for larger models, we cannot definitively claim `MXINT8` is a universal replacement for FP8 training in frontier model development.
*   **Evidence:** Table 4 shows training results only for 1B and 3B models. The 235B results in Table 3 are strictly **direct-cast inference** (post-training quantization), which is generally less sensitive to quantization noise than full training loops.

### 6.2 Hardware Cost Modeling Assumptions

The hardware efficiency claims (e.g., **37% energy reduction** for `MXINT8` in Table 5) are derived from a **gate-level model** (Appendix C), not measured silicon data. While this model is rigorous, it relies on specific abstractions that may not hold in all chip designs.

*   **Excluded Quantizer Overhead:** The model explicitly **excludes the quantizer block** from cost accounting (Appendix C). The authors argue that quantization can be fused into the accumulation pipeline (e.g., via integer additions for `UE8M0` scales). However, in architectures where quantization is a distinct stage or requires complex rounding logic, this exclusion could underestimate the true cost of INT formats.
*   **Accumulation Precision:** The model assumes **FP32 accumulation** for all formats (Appendix C). This is a standard design choice to prevent error growth, but it favors INT formats less than a lower-precision accumulator might. If a design opted for INT32 accumulation to match the INT MACs, the area/energy benefits might shift. Conversely, if FP accumulation logic is shared across formats, the marginal cost of supporting FP might be lower than modeled.
*   **Toggle Rate Simplification:** The energy model assumes a uniform toggle rate `$\tau_g = \tau$` for all gate cells (Appendix C). In reality, FP exponent logic might switch less frequently than INT carry chains depending on data distribution. This simplification allows for relative comparison but may not capture absolute power dynamics accurately.

### 6.3 Algorithmic Dependencies for 4-Bit Formats

A critical trade-off emerges at 4-bit precision: `NVINT4` is **not** a drop-in replacement for `NVFP4`. Its superiority is conditional on algorithmic preprocessing.

*   **Rotation Dependency:** As shown in **Table 3**, `NVINT4` loses to `NVFP4` on **all 12 models** without Hadamard rotation. It only wins (100% win rate) **with random Hadamard rotation**.
*   **Hidden Cost:** The paper's hardware model (Section 6) does not explicitly quantify the energy or latency cost of performing Hadamard rotation on-chip. While rotation is linear complexity `$O(N \log N)$`, it requires additional memory bandwidth and compute units. If the rotation overhead exceeds the MAC efficiency gains (e.g., the **66% energy reduction** claimed in Table 5), the net system benefit diminishes.
*   **Design Implication:** Hardware architects cannot simply swap `NVFP4` units for `NVINT4` units. They must co-design the accelerator with rotation support, adding complexity to the software stack and potentially the chip interface.

### 6.4 Bit-Width Specificity: The 6-Bit Failure Case

The paper honestly reports that INT formats are **not universally superior** across all bit-widths. Specifically, `MXINT6` fails to outperform `MXFP6`.

*   **The Data:** In **Table 3**, `MXINT6` wins only **1 of 12 models** (with rotation) and **0 of 12** (without rotation).
*   **The Theoretical Reason:** This failure is predicted by the QSNR theory in **Figure 3**. The crossover point for 6-bit formats is `$\kappa < 1.96$`. However, **Table 2** shows the typical crest factor for block size 32 is **2.96** (75th percentile). Even with rotation, it only drops to **2.39**, which remains above the 1.96 threshold.
*   **The Trade-off:** This reveals a "dead zone" in the design space. 8-bit INT is robust enough to handle the residual outliers in fine-grained blocks, but 6-bit INT lacks the precision to compensate for the remaining dynamic range, while 6-bit FP retains enough mantissa bits to handle the variation. This suggests that hardware support for 6-bit INT might be a poor investment compared to sticking with FP6 or jumping directly to INT4 with rotation.

### 6.5 Symmetric Clipping and Dynamic Range

To enable lossless INT training, the authors introduce **symmetric clipping** (Section 3.2), restricting the INT8 range from `[-128, 127]` to `[-127, 127]`.

*   **The Benefit:** This resolves the gradient bias caused by the asymmetric two's complement range, reducing training loss from **3.1354** to **3.1251** at block size 32 (Table 10, Appendix D.2).
*   **The Cost:** This choice sacrifices one representable value (the `-128` endpoint). In scenarios where the data distribution is heavily skewed or where every bit of dynamic range is critical (e.g., extremely coarse-grained quantization), this reduction could theoretically limit the maximum representable magnitude.
*   **Numerical Stability Pitfall:** The paper highlights that **BFloat16 arithmetic precision** can cause values to incorrectly map to `-128` even when the scale is set for `127` (Table 11, Appendix D.2). **16.82%** of values mapped incorrectly in their stability test. This implies that implementing symmetric clipping requires careful handling of the scale factor computation to avoid overflow during the quantization step itself, adding a subtle implementation constraint for hardware designers.

### 6.6 Scale Factor Overhead (`$\rho$`)

The theoretical framework distinguishes between `UE8M0` scales (used in MX formats) and `E4M3` scales (used in NV formats). This introduces a trade-off in scale factor precision.

*   **UE8M0 Overhead:** The `UE8M0` scale rounds the scale factor to a power of two, introducing an overhead factor `$\rho \in [1, 2)$`. Theorem 1 (Section 4.1) shows this incurs up to **6.02 dB loss** in QSNR (`$-20 \log_{10}(\rho)$`).
*   **E4M3 Precision:** NV formats use `E4M3` scales, where `$\rho \approx 1$`, avoiding this penalty. However, `E4M3` scales are more complex to store and compute than power-of-two shifts.
*   **The Trade-off:** `MXINT8` succeeds despite the `$\rho$` penalty because the 8-bit mantissa precision is high enough to absorb the loss. However, for lower bit-widths, this scale overhead becomes more critical. This suggests that future INT formats might need higher-precision scales (like `E4M3`) to compete at 4-bit, complicating the hardware simplicity argument for INT.

### 6.7 Summary of Open Questions

Based on these limitations, several critical questions remain for future work:

1.  **Scaling Laws:** Does `MXINT8` training remain lossless at **70B+ parameters**, or does quantization noise accumulate differently at scale?
2.  **Rotation Cost:** What is the exact energy cost of Hadamard rotation on silicon, and does it negate the **34% area reduction** claimed for mixed INT formats in Table 5?
3.  **Real Silicon Validation:** Do the gate-level model predictions (e.g., **0.63x energy** for `MXINT8`) hold when measured on actual taped-out accelerators with memory bandwidth constraints?
4.  **Domain Specificity:** The evaluation focuses on language models (Llama, Qwen). Does this INT advantage hold for **multi-modal models** (vision-language) or **code generation**, where activation distributions might differ significantly?

By acknowledging these trade-offs, we see that while fine-grained INT formats offer a promising path forward, they are not a universal solution. They require careful co-design of algorithms (rotation), data types (symmetric clipping), and hardware (scale factors) to realize their theoretical potential.

## 7. Implications and Future Directions

This study does more than compare number formats; it fundamentally challenges the assumption that floating-point (FP) is the necessary default for low-precision AI hardware. By demonstrating that integer (INT) formats can surpass FP in both accuracy and efficiency under fine-grained conditions, the paper forces a reevaluation of the algorithm-hardware co-design landscape. This section outlines how these findings shift industry priorities, identifies critical gaps for future research, and provides concrete guidance for engineers looking to integrate these methods.

### 7.1 Shifting the Hardware Design Paradigm

The most profound implication of this work is the validation of **fine-grained INT formats** as a viable, and often superior, alternative to the industry's current FP-centric trajectory (Abstract). Historically, hardware architects have prioritized FP units (e.g., NVIDIA Blackwell's support for `MXFP8`/`NVFP4`) to handle activation outliers (Section 1). This paper argues that this approach is suboptimal when quantization granularity is fine enough to isolate those outliers.

*   **Architectural Reallocation:** The hardware cost analysis in **Table 5** shows that `MXINT8` reduces energy by **37%** (`0.63x`) and area by **21%** (`0.79x`) compared to `MXFP8`. For chip designers, this suggests that silicon area currently dedicated to complex FP logic (exponent comparators, mantissa aligners) could be reallocated to increase compute density or memory bandwidth if INT formats are adopted.
*   **Granularity-Format Coupling:** The study establishes that format performance is inextricably linked to granularity. The "crossover point" theory (Section 4.2) implies that hardware cannot be designed for a format in isolation. An accelerator optimized for `MXINT8` must support **block size 32** quantization to ensure the crest factor `$\kappa$` remains below the critical threshold of **7.55** (Figure 3). Coarse-grained INT support (e.g., per-channel) would fail to capture these benefits, as `$\kappa$` typically exceeds **11.97** at that granularity (Table 2).
*   **Mixed-Precision Strategies:** The efficiency gains extend to mixed-precision workloads. The `MXINT8+NVINT4` configuration reduces area by **34%** (`0.66x`) relative to `MXFP8+NVFP4` (Table 5). This encourages hardware vendors to design flexible MAC units that can efficiently switch between 8-bit and 4-bit INT operations, rather than hardwiring separate FP pipelines for each.

### 7.2 Critical Avenues for Follow-up Research

While this paper provides a robust framework, several open questions remain that define the next wave of research in low-precision computing.

*   **Scaling Laws for INT Training:** The training experiments are limited to **1B and 3B parameter models** (Section 5.3). A critical follow-up direction is validating whether `MXINT8` remains "nearly lossless" at frontier scales (e.g., **70B+ parameters**). Quantization noise can accumulate non-linearly over billions of training steps [8, 19], and the stability observed in smaller models may not hold without additional regularization or modified optimizers.
*   **Quantifying Rotation Overhead:** The viability of `NVINT4` depends entirely on **Hadamard rotation** (Section 5.2). However, the paper's hardware model (Appendix C) does not account for the energy or latency cost of performing this rotation on-chip. Future work must measure the **system-level energy balance**: does the **66% energy reduction** in the MAC unit (Table 5) outweigh the cost of the rotation unit? If rotation requires high-bandwidth memory access, the net system benefit could diminish.
*   **Domain Generalization:** The evaluation focuses exclusively on Large Language Models (Llama-3, Qwen3) (Section 5.2). Activation distributions in **Vision Transformers (ViT)** or **multimodal models** may exhibit different crest factor statistics. Research is needed to determine if the `$\kappa < 7.55$` crossover point holds for image or audio data, where outlier structures might differ from text tokens.
*   **Dynamic Scale Formats:** The paper identifies the `UE8M0` scale overhead (`$\rho \in [1, 2)$`) as a source of QSNR loss (Theorem 1, Section 4.1). Future formats could explore hybrid scales that retain the hardware simplicity of power-of-two shifts for most blocks but switch to higher-precision scales (like `E4M3`) only for blocks with extreme dynamic range, optimizing the trade-off between scale precision and hardware cost.

### 7.3 Practical Deployment Guidelines

For engineers and practitioners integrating low-bit quantization into production systems, this paper offers a clear decision matrix based on bit-width, granularity, and workload type.

*   **When to Prefer `MXINT8`:**
    *   **Use Case:** Fine-grained inference or training where hardware supports block-wise quantization (block size 32).
    *   **Reasoning:** `MXINT8` outperforms `MXFP8` on **100% of tested models** (Table 3) and reduces hardware energy by **37%** (Table 5). It is robust enough to work **without Hadamard rotation**, simplifying the software stack.
    *   **Constraint:** Ensure the hardware accumulator uses **FP32 precision** to prevent error growth, as modeled in Appendix C.

*   **When to Prefer `NVINT4`:**
    *   **Use Case:** Extreme memory-constrained inference (4-bit) where hardware supports block size 16.
    *   **Condition:** **Mandatory** use of outlier mitigation techniques like **Hadamard rotation**. Without rotation, `NVINT4` loses to `NVFP4` on all 12 tested models (Table 3).
    *   **Reasoning:** With rotation, `NVINT4` wins on **100% of models** and offers **66% energy savings** over `NVFP4` (Table 5).

*   **When to Avoid INT Formats:**
    *   **6-Bit Regime:** Avoid `MXINT6`. The paper identifies this as a "dead zone" where INT lacks the precision to compensate for residual outliers, yet FP retains enough mantissa bits to handle them. `MXINT6` loses to `MXFP6` on **11 of 12 models** even with rotation (Table 3).
    *   **Coarse Granularity:** Avoid INT for per-channel or per-tensor quantization. The crest factor `$\kappa$` at coarse granularity (e.g., **11.97** for per-channel) exceeds the crossover point for all INT formats, making FP the safer choice for dynamic range (Section 5.1).

### 7.4 Reproduction and Integration Guidance

The authors provide open-source code to facilitate adoption and verification (Abstract). Practitioners aiming to reproduce results or integrate these formats should attend to the following implementation details:

*   **Symmetric Clipping for Training:** If implementing `MXINT8` training, you **must** use symmetric clipping ranges. Standard two's complement ranges (e.g., `[-128, 127]`) introduce gradient bias that degrades loss (Figure 2). Implement the range as `$Q_{\text{min}} = -(2^{b-1} - 1)$` and `$Q_{\text{max}} = 2^{b-1} - 1$` (e.g., `[-127, 127]` for 8-bit) (Section 3.2).
*   **Scale Factor Rounding:** When computing `UE8M0` scales, round **up** rather than down. The paper specifies `$s' = 2^{\text{clip}(\lceil \log_2(s) \rceil, -127, 127)}$` (Equation 9). Rounding down underestimates the scale, causing systematic clipping of maximum values and reducing QSNR (Section 3.2).
*   **Rotation Integration:** For 4-bit INT, integrate Hadamard rotation as a preprocessing step. The paper uses random Hadamard matrices of size `$h \times h$` where `$h$` matches the block size (32 for MX, 16 for NV) (Section 5.2). Ensure this rotation is applied consistently to both weights and activations during inference.
*   **Code Availability:** The full implementation, including the QSNR modeling and training recipes, is available at `https://github.com/ChenMnZ/INT_vs_FP` (Abstract). Researchers should use this baseline to test alternative scale factors or rotation methods before committing to hardware design changes.

By adhering to these guidelines, the community can move beyond the default assumption of FP dominance and leverage fine-grained INT formats to build more efficient, accurate, and sustainable AI systems.
