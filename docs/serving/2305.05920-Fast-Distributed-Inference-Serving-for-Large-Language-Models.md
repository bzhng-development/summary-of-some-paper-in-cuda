## 1. Executive Summary

Existing LLM serving systems like vLLM suffer from severe head-of-line blocking due to run-to-completion scheduling, where queuing delay accounts for up to 90% of total latency in real-world workloads such as ShareGPT and Alpaca. This paper presents FastServe, a distributed inference serving system that exploits the autoregressive pattern of LLMs to enable iteration-level preemption using a novel skip-join Multi-Level Feedback Queue (MLFQ) scheduler and proactive key-value cache memory management. Experiments on OPT-175B across 16 NVIDIA A100 GPUs demonstrate that FastServe improves throughput by up to 31.4× and 17.9× compared to vLLM under the same average and tail latency requirements, respectively, addressing the critical latency demands of interactive AI applications.

## 2. Context and Motivation

To understand the necessity of FastServe, we must first examine the unique operational characteristics of Large Language Models (LLMs) and why traditional serving infrastructure fails to meet their latency requirements. This section dissects the specific bottleneck of head-of-line blocking, explains the underlying mechanics of LLM inference that create this bottleneck, and evaluates why prior state-of-the-art systems remain insufficient.

### The Latency Crisis in Interactive AI

The primary driver for this research is the shift from batch-oriented AI tasks to interactive applications. Systems like ChatGPT, New Bing, and Claude-3 require immediate responses to maintain user engagement. In this context, **latency** is the critical metric. Users expect inputs to be responded to instantly.

However, the paper identifies a severe inefficiency in existing infrastructure: **head-of-line (HOL) blocking**. This occurs when a long-running job prevents shorter, subsequent jobs from being processed. In standard queueing theory, if jobs are processed First-Come-First-Served (FCFS), a single large job at the front of the queue delays all others behind it.

The magnitude of this problem is quantified in **Figure 1** of the paper. The authors analyze real-world workloads from datasets like `ShareGPT` and `Alpaca`. They observe that:
*   In idealized scenarios where all jobs have identical input/output lengths, queuing delay is negligible even at high load (`load ≈ 1`).
*   In real-world skewed workloads, **up to 90% of total latency is queuing delay**, not execution time.

> "Figure 1 shows that up to 90% of total latency is the queuing delay for real-world datasets. In this case, optimizing execution time is not enough... Instead, we need to optimize the queuing delay."

This finding fundamentally shifts the optimization target. Prior systems focused on speeding up the GPU computation (execution time). FastServe argues that because execution time is a minor component of total latency in skewed workloads, the system must prioritize reducing the time jobs spend waiting in the queue.

### Unique Characteristics of LLM Inference

To understand why HOL blocking is particularly acute for LLMs, we must distinguish LLM inference from traditional Deep Neural Network (DNN) inference (e.g., ResNet for image classification).

1.  **Autoregressive Generation:**
    Traditional DNN inference is typically "one-shot": an input enters, and a result exits in a single deterministic pass. LLMs operate **autoregressively**. As described in **Section 2.1**, an LLM generates text token-by-token.
    *   **Iteration:** Each step where the model generates one output token is called an iteration.
    *   **Variable Length:** The total execution time depends on the input length (known) and the output length (unknown). The model stops generating when it produces an end-of-sequence (`<EOS>`) token or reaches a maximum length.
    *   **Unpredictability:** Because the output length depends on the semantic content of the generation, it cannot be known prior to execution. This variability is the root cause of HOL blocking; a job requesting a long paragraph blocks a job requesting a single word.

2.  **Initialization vs. Decoding Phases:**
    LLM inference is not uniform across iterations. **Section 2.2** and **Figure 3** explain the **Key-Value (KV) Cache** optimization.
    *   **Initialization Phase (First Iteration):** The model processes the entire input prompt. It must compute and cache the key and value tensors for every token in the input sequence. This is computationally expensive.
    *   **Decoding Phase (Subsequent Iterations):** The model only computes the key and value for the *newly* generated token, retrieving previous tokens from the KV cache.
    *   **Performance Implication:** As shown in **Figure 5**, the execution time of the first iteration is significantly higher than subsequent iterations, and it scales with input length. For example, an input length of 1024 takes much longer for the first iteration than an input length of 128.

This asymmetry creates a specific scheduling challenge: the cost of starting a job is high, but the cost of continuing it is low.

### Limitations of Prior Approaches

Existing solutions fall into three categories, each with specific shortcomings regarding LLM serving.

#### 1. Traditional DNN Serving Systems
Systems like `TensorFlow Serving` or `Triton Inference Server` are model-agnostic. They batch jobs to maximize GPU utilization. However, they typically employ **run-to-completion** processing for batches.
*   **Shortcoming:** They do not account for the autoregressive nature of LLMs. They cannot interleave iterations from different jobs dynamically. A long job in a batch forces short jobs to wait until the entire batch completes.

#### 2. LLM-Specific Systems (Orca, vLLM)
Recent systems have adapted to LLM architectures.
*   **Orca [10]:** Introduces **iteration-level scheduling**. It allows the system to add new jobs or remove completed ones at the end of each iteration.
*   **vLLM [11]:** Introduces **`PagedAttention`**, which manages the KV cache in blocks to reduce memory fragmentation, allowing larger batch sizes.
*   **Shortcoming:** Despite these improvements, both systems fundamentally rely on **FCFS scheduling** for job ordering. As stated in **Section 1**:
    > "Once a job is scheduled, it runs until it finishes. Due to the limited GPU memory... a long job may block the incoming ones, known as head-of-line blocking."
    They lack **preemption**. If a high-priority (short) job arrives while a low-priority (long) job is running, the system cannot pause the long job to serve the short one immediately.

#### 3. Classic Scheduling Algorithms
In operating systems and networking, algorithms exist to minimize latency, but they fail in this specific context.
*   **Shortest Remaining Processing Time (SRPT):** This is theoretically optimal for minimizing average latency. It always processes the job with the least work remaining.
    *   **Failure Mode:** SRPT requires knowing the total job size (output length). As established, LLM output length is unknown until generation finishes. Therefore, SRPT cannot be directly applied (**Section 2.3, Challenge 1**).
*   **Multi-Level Feedback Queue (MLFQ):** A classic approach for information-agnostic settings. Jobs start at high priority and are demoted if they use too much CPU time.
    *   **Failure Mode:** Naive MLFQ assumes uniform task costs. In LLMs, the first iteration (initialization) is much heavier than decoding iterations (**Figure 5**). A job with a long input might exhaust its time quantum in the very first iteration and get demoted immediately, even if it only needs one more token to finish. This violates the spirit of priority scheduling and wastes resources on context switching.

### The Memory Overhead Challenge

Enabling preemption introduces a secondary, critical constraint: **GPU Memory**.

To preempt a job (pause it and resume later), the system must save its intermediate state. For LLMs, this state is the **KV Cache**.
*   **Scale:** The memory required is substantial. **Section 2.3** notes that for `OPT-175B` with an input length of 512, a single job requires at least **2.3 GB** of GPU memory for the KV cache.
*   **Capacity:** An NVIDIA A100 GPU has 80 GB of memory. A significant portion is already occupied by the model weights (e.g., 350 GB for OPT-175B, split across 16 GPUs).
*   **The Trade-off:** In FCFS systems, the cache only stores state for currently running jobs. In a preemptive system, the cache must store state for *all* started but unfinished jobs (including those waiting in queues).
*   **Risk:** Without careful management, the cache will overflow, causing Out-Of-Memory (OOM) errors. Naive solutions like killing low-priority jobs waste computation (re-computation), while deferring new jobs reintroduces HOL blocking (**Section 4.2**).

### Positioning FastServe

FastServe positions itself at the intersection of these challenges. It does not merely optimize kernel execution (like vLLM) nor does it rely on unknown future information (like SRPT). Instead, it exploits the **semi information-agnostic** nature of LLMs:
1.  **Unknown:** Output length (total iterations).
2.  **Known:** Input length (cost of the first iteration).

By leveraging the known input length, FastServe can make informed scheduling decisions without violating the constraints of the autoregressive generation process. It introduces a system that supports **iteration-level preemption** (addressing HOL blocking) while managing the massive memory footprint of preempted jobs through **proactive swapping** (addressing GPU memory constraints). This dual focus on scheduling logic and memory infrastructure distinguishes it from prior work that addressed only one aspect or relied on FCFS ordering.

## 3. Technical Approach

This is a systems paper that presents FastServe, a distributed inference serving system for Large Language Models that combines a novel skip-join Multi-Level Feedback Queue scheduler with proactive key-value cache memory management to eliminate head-of-line blocking while respecting GPU memory constraints.

### 3.1 Reader orientation (approachable technical breakdown)

FastServe is a distributed system that sits between users submitting LLM inference requests and the GPU cluster that executes them, making real-time decisions about which requests to process and when to pause or resume them. The system solves the head-of-line blocking problem by treating each token generation as an interruptible unit of work, then manages the resulting memory pressure by intelligently moving intermediate computation state between GPU and host memory before it becomes critical.

### 3.2 Big-picture architecture (diagram in words)

The FastServe architecture consists of four major components that work together in a coordinated pipeline. First, the **Job Pool** receives incoming inference requests from clients through a RESTful API frontend, where each request contains a prompt and sampling parameters like maximum output length and temperature. Second, the **Skip-Join MLFQ Scheduler** (Section 4.1) examines each arriving job, profiles its expected initialization time based on input length, and places it into an appropriate priority queue rather than always starting at the highest priority. Third, the **Distributed Execution Engine** (Section 4.3) pulls jobs from the scheduler according to priority, forms batches up to a maximum batch size, and executes one iteration (one token generation) per job across multiple GPUs using tensor and pipeline parallelism. Fourth, the **Key-Value Cache Manager** (Section 4.2) proactively swaps intermediate state between GPU memory and host memory based on estimated next scheduled time, ensuring that active jobs reside in GPU memory while inactive jobs are offloaded without blocking the critical path of inference. Information flows unidirectionally from the Job Pool through the Scheduler to the Execution Engine, with the Cache Manager operating in parallel to handle memory state asynchronously.

### 3.3 Roadmap for the deep dive

- We first explain the skip-join MLFQ scheduler because it is the core innovation that enables preemption and directly addresses the head-of-line blocking problem identified in Section 2.
- We then detail the proactive key-value cache management mechanism because preemption creates memory pressure that must be resolved before the scheduler can function effectively at scale.
- We next describe the distributed execution support because large models like OPT-175B cannot fit on a single GPU and require parallelism strategies that interact with both scheduling and memory management.
- We conclude with implementation details and configuration parameters because understanding the concrete system requirements helps contextualize the design choices and performance claims.

### 3.4 Detailed, sentence-based technical breakdown

#### Core Scheduling Innovation: Skip-Join Multi-Level Feedback Queue

The skip-join MLFQ scheduler represents a fundamental departure from both traditional FCFS scheduling and classic MLFQ algorithms by exploiting what the paper terms the "semi information-agnostic" setting of LLM inference. In this setting, while the total output length (number of tokens to generate) remains unknown before execution begins, the input length (prompt size) is known at arrival time and directly determines the execution time of the first iteration. **Figure 5** demonstrates this relationship empirically: for the OPT-2.7B model on an NVIDIA A100, the first iteration time increases monotonically with input length, ranging from approximately 0.02 seconds for input length 128 to over 0.08 seconds for input length 1024, while subsequent decoding iterations remain roughly constant at around 0.02 seconds regardless of input length.

The scheduler maintains $n$ priority queues denoted as $Q_1, Q_2, ..., Q_n$, where $Q_1$ represents the highest priority and $Q_n$ represents the lowest priority. Each queue $Q_i$ has an associated time quantum $q_i$, which represents the maximum cumulative execution time a job can consume in that queue before being demoted to a lower priority queue. The quantum values follow a geometric progression where $q_1 < q_2 < ... < q_n$, and FastServe specifically sets the quantum of each lower priority queue to two times that of the higher priority queue (i.e., $q_{i+1} = 2 \times q_i$), which aligns with prior work on MLFQ scheduling in datacenter systems. The quantum of the highest priority queue $q_1$ is set to the minimum iteration time observed during profiling, ensuring that even the fastest decoding iteration can complete within one quantum.

When a new job arrives, the scheduler does not place it in $Q_1$ as classic MLFQ would. Instead, the scheduler invokes a lightweight profiling function `P.getNextIterTime(job)` that predicts the initialization phase execution time $t_{init}$ based on the job's input length, model architecture, and hardware configuration. The scheduler then finds the highest priority queue $Q_i$ such that $q_i \geq t_{init}$, and places the job directly into that queue. This "skip-join" operation prevents jobs with long input prompts from immediately exhausting their quantum in the highest priority queue and being incorrectly demoted, which would happen under naive MLFQ. **Algorithm 1** (lines 6–9) formalizes this skip-join logic: for each newly arrived job, compute `init_time`, find the minimum priority level $p_{job}$ such that $q_{p_{job}} \geq init\_time$, and push the job to $Q_{p_{job}}$.

After a job is placed in a queue and begins execution, the scheduler monitors its cumulative execution time. When a job generates an output token (completing one iteration), the scheduler checks whether the job has depleted its allocated quantum for the current priority queue. If the job has consumed its quantum but has not finished generating all output tokens (i.e., no `<EOS>` token produced and maximum output length not reached), the scheduler demotes the job to a lower priority queue. **Algorithm 1** (lines 16–17) specifies that the job is popped from $Q_{p_{job}}$ and pushed to $Q_{p_{job} + \eta}$, where $\eta$ represents the demotion step size. FastServe sets $\eta$ such that the job moves to a queue with quantum two times larger, allowing longer-running jobs to progress without excessive preemption overhead.

The scheduler must also prevent starvation, where jobs with long input and output lengths never receive GPU time because shorter jobs continuously arrive and occupy higher priority queues. To address this, the scheduler periodically examines all jobs and tracks their "starve time," which measures how long a job has waited without being scheduled. **Algorithm 1** (lines 19–21) implements starvation prevention: if a job's `starveTime` exceeds a threshold $\alpha$, the job is popped from its current queue and promoted directly to $Q_1$, the highest priority queue, and its `starveTime` is reset to zero. FastServe tunes $\alpha$ based on the user-specified Service Level Objective (SLO), with a default value of 300 milliseconds. This mechanism ensures that even long jobs eventually receive service, protecting tail latency while still prioritizing short jobs for average latency optimization.

**Figure 7** provides a concrete execution timeline comparison across four scheduling algorithms: FCFS, Original MLFQ, Skip-Join MLFQ (FastServe), and SRPT (the theoretical optimum). In this example, three jobs $J_1, J_2, J_3$ arrive simultaneously, where $T_1(J_i)$ denotes the initialization phase time and $T_2(J_i)$ denotes the decoding phase time for job $J_i$. The four priority queues have quantum values of 1, 2, 4, and 8 time units. FCFS processes jobs strictly in arrival order, causing $J_1$ to block $J_2$ and $J_3$, resulting in an average latency of 4.23. Original MLFQ places all jobs in $Q_1$ initially, but $J_1$ exhausts its quantum quickly due to long initialization time and gets demoted, yet still causes blocking with average latency of 5.0. Skip-Join MLFQ correctly places $J_1$ in a lower priority queue based on its initialization time, allowing $J_2$ and $J_3$ to complete first, achieving an average latency of 3.3, which approaches the SRPT optimum of 3.0. This example demonstrates that skip-join MLFQ effectively approximates SRPT performance without requiring knowledge of total job size.

#### Proactive Key-Value Cache Management

The skip-join MLFQ scheduler introduces a critical secondary challenge: GPU memory consumption increases dramatically under preemptive scheduling compared to FCFS. In FCFS systems like vLLM, the key-value cache only needs to store intermediate state for jobs currently in the processing batch, limited by the maximum batch size. In contrast, the skip-join MLFQ scheduler must maintain key-value cache for all started but unfinished jobs, including those preempted and waiting in lower priority queues. **Figure 8** quantifies this overhead: for OPT-2.7B under a synthetic Gamma Process workload with rate=64 and coefficient of variation=4, with maximum output length set to 20, the peak KV cache memory overhead for skip-join MLFQ can be 7× larger than FCFS, reaching over 20 GB compared to approximately 3 GB for FCFS.

The memory requirement for a single job's key-value cache can be calculated precisely. For an LLM with $l$ transformer layers, hidden dimension $h$, input sequence length $s$, and output sequence length $t$, if model weights and computations use FP16 precision (2 bytes per value), the total bytes required for the key-value cache is:

$$\text{KV\_cache\_bytes} = 4 \times l \times h \times (s + t)$$

The factor of 4 arises because each token requires both key and value tensors (2 tensors), and each tensor element is stored in FP16 (2 bytes), giving $2 \times 2 = 4$ bytes per element per layer. For OPT-175B specifically, **Table 1** shows $l = 96$ layers and $h = 12288$ hidden dimension. With input length $s = 512$ and minimum output length $t = 1$, a single job requires:

$$4 \times 96 \times 12288 \times (512 + 1) = 2,416,969,728 \text{ bytes} \approx 2.3 \text{ GB}$$

This 2.3 GB per job creates severe capacity constraints. An NVIDIA A100 GPU provides 80 GB of memory, but a large portion must store model weights. For OPT-175B, the model weights alone require 350 GB in FP16, which must be distributed across 16 GPUs (approximately 22 GB per GPU for weights), leaving limited space for KV cache. Without careful management, the cache will overflow, causing Out-Of-Memory errors that halt inference.

The paper evaluates two strawman solutions before presenting the proactive swapping approach. The first strawman defers newly arrived jobs when GPU memory is insufficient, keeping current in-memory jobs until they finish. This approach degenerates MLFQ back to FCFS under memory-constrained settings, reintroducing head-of-line blocking. The second strawman kills low-priority jobs and releases their KV cache to make room for high-priority jobs. This causes two problems: killed jobs lose their generation state and must rebuild KV tensors from scratch (wasting computation), and the starvation prevention mechanism may promote killed jobs back to high priority, potentially creating deadlock cycles where jobs repeatedly kill each other.

FastServe's solution extends the key-value cache storage from GPU memory to host memory through proactive swapping. The key observation is that KV tensors only need to reside in GPU memory when their corresponding jobs are actively scheduled for execution. For inactive jobs (preempted and waiting in queues), the KV tensors can be stored in host memory and swapped back to GPU memory when the job is about to be scheduled. However, swapping introduces latency overhead that must be managed carefully. When deploying OPT-175B on 16 NVIDIA A100 GPUs, the KV tensors for a single job occupy 2.3 GB. Token generation time in the decoding phase is approximately 60 milliseconds, while swapping 2.3 GB between host and GPU memory over PCIe 4.0×16 at full bandwidth takes approximately 36 milliseconds. A reactive swapping mechanism that performs swapping and inference sequentially would add 60% overhead to each token generation, which is unacceptable.

FastServe employs proactive swapping that overlaps memory transfer with GPU computation. **Figure 9** illustrates the difference between reactive and proactive approaches. In reactive swapping (Figure 9a), when job $J_1$ finishes or is preempted, the system swaps out $J_1$'s KV cache and then swaps in $J_2$'s KV cache before $J_2$ can execute, placing swapping on the critical path. In proactive swapping (Figure 9b), the system swaps in $J_2$'s KV cache while $J_1$ is still executing, so that when $J_1$ completes, $J_2$ is ready to execute immediately without waiting for memory transfer. This overlapping hides the swapping latency behind computation, maintaining high GPU utilization.

The job swapping order is determined by calculating the Estimated Next Scheduled Time (ENST) for each job. The ENST represents when a job will next be scheduled for execution, and jobs with larger ENST should be swapped out first (they will not be needed soon), while jobs with smaller ENST should be swapped in first (they will be needed soon). For job $i$, FastServe considers two factors: the time until starvation prevention promotes the job ($T_{promote}(i)$), and the sum of execution time for all higher-priority jobs that must complete before job $i$ can execute ($T_{execute}(i)$). The execution time for a higher-priority job $j$ is calculated as the sum of quantum values for all queues between job $j$'s current priority and job $i$'s priority:

$$T_{execute}(i, j) = \sum_{i.priority < k \leq j.priority} q_k$$

where $i.priority$ is the priority level of job $i$, $j.priority$ is the priority level of job $j$, and $q_k$ is the quantum of priority queue $k$. The total execution time considering all higher-priority jobs is then:

$$T_{execute}(i) = \frac{1}{B} \sum_{i.priority < j.priority} T_{execute}(i, j)$$

where $B$ is the maximum batch size (representing parallel execution of $B$ jobs). The final ENST for job $i$ is the minimum of promotion time and execution time:

$$ENST(i) = \min(T_{promote}(i), T_{execute}(i))$$

This ENST definition ensures that jobs likely to be promoted soon (due to starvation prevention) or jobs with few higher-priority competitors remain in GPU memory, while jobs with distant scheduling times are offloaded to host memory.

To handle bursts of new high-priority jobs, FastServe reserves idle key-value cache slots specifically for new arrivals. The number of reserved slots is determined based on historical job arrival patterns: higher frequency of job bursts necessitates more reserved slots. This reservation prevents the cache management system from being forced into reactive eviction when new jobs arrive, which would adversely affect new job performance. **Figure 10** shows how intermediate result transmission between pipeline stages can occur in parallel with key-value cache swapping instructions, further reducing overhead in distributed settings.

#### Distributed Execution Support

Large language models like OPT-175B cannot fit in a single GPU's memory and require distributed inference across multiple GPUs. **Table 1** shows that OPT-175B requires 350 GB for model weights alone in FP16, far exceeding the 80 GB capacity of even the largest NVIDIA A100 GPUs. FastServe supports two parallelism strategies that can be combined: tensor parallelism and pipeline parallelism.

Tensor parallelism splits each operator (e.g., matrix multiplication in attention layers) across multiple devices, with each device executing a portion of the computation in parallel. Additional communication overhead is required to split the input and gather the output from different GPUs. Tensor parallelism increases both computational and memory resources available to a single job, reducing the time per iteration. Pipeline parallelism splits the entire computation graph into multiple stages, with each stage assigned to different GPUs. During inference, each stage processes a different batch simultaneously in a pipeline fashion, transmitting intermediate results to the next stage. Pipeline parallelism requires less communication overhead than tensor parallelism and allows the model to exceed individual GPU memory constraints.

Job scheduling in distributed settings requires modification from traditional MLFQ semantics. In standard MLFQ, if no new job arrives, the scheduler keeps scheduling the same job until it finishes or is demoted. With pipeline parallelism, the scheduler operates at the granularity of individual pipeline stages. Once a job completes its first stage and transmits intermediate results to the next stage, the first stage GPU becomes available for a new job. FastServe cannot follow traditional MLFQ that keeps scheduling the same job until demotion, because the job is still in progress through later pipeline stages. To preserve MLFQ semantics, FastServe keeps the running job in the priority queue but schedules the highest priority job in the pending state for the now-available stage. This allows early jobs in a queue to expedite their quantum completion while maintaining pipeline utilization.

The key-value cache management is also distributed across multiple GPUs. Since each key-value tensor is used by the same stage of the LLM during inference, FastServe partitions key-value tensors according to tensor parallelism requirements and assigns each tensor to the corresponding GPU. This ensures that all computation on a GPU only requires local key-value tensors on the same GPU, minimizing cross-GPU memory access. The proactive swapping mechanism operates in distributed fashion: because different pipeline stages process different jobs simultaneously, each stage may offload or upload different key-value tensors independently. To reduce redundant control overhead, before processing intermediate results from the previous stage, the current stage performs the same offloading or uploading action as the previous stage. **Figure 10** illustrates this: when intermediate results are sent to the next stage, the next stage receives swapping instructions and can swap the key-value cache simultaneously if needed. The key-value cache swapping mechanism only needs to decide offloading or uploading for the first stage. When using tensor parallelism to split the first stage into multiple chunks, a centralized key-value cache swapping manager instructs all chunks in the first stage to offload or upload the key-value tensors owned by the same job.

#### Implementation Details and Configuration Parameters

FastServe is implemented as a distributed system with three main components. The frontend and scheduler are implemented with 2.9K lines of Python code, providing a RESTful API that supports OpenAI API-compatible interfaces where clients can specify sampling parameters like maximum output length and temperature. The distributed execution engine is implemented with 8.1K lines of C++/CUDA code for performance-critical operations. The system uses Ray actors to implement GPU workers that execute LLM inference and manage the key-value cache in a distributed manner. FastServe implements popular open-source LLMs such as OPT in C++ to achieve better performance and scalability than Python implementations like HuggingFace, and includes custom CUDA kernels supporting Orca's iteration-level scheduling and vLLM's PagedAttention.

**Table 2** compares FastServe's capabilities against baseline systems. FasterTransformer supports iteration-level preemption (IP) and pipeline parallelism (PP) but lacks iteration-level scheduling (IS) and PagedAttention (PA). vLLM supports IS and PA but lacks IP and PP. FastServe-FCFS is a variant using FastServe's execution engine without the scheduling innovations, supporting IS, PA, and PP but not IP. Full FastServe supports all four capabilities: IP, IS, PA, and PP.

The evaluation testbed uses two AWS EC2 p4d.24xlarge instances for end-to-end experiments, each configured with eight NVIDIA A100 40GB GPUs connected over NVLink, 1152 GB host memory, and PCIe 4.0×16. Design choice experiments use one NVIDIA A100 40GB GPU in a local testbed. All experiments use FP16 precision. The latency SLO is set to 10× the latency of a single iteration in the decoding phase, specifically 0.3 seconds based on profiling. Workloads are generated from ShareGPT and Alpaca datasets with arrival times following a Poisson process parameterized by arrival rate.

The starvation prevention threshold $\alpha$ is set to 300 milliseconds by default, tuned based on user-specified SLO. The quantum progression follows $q_{i+1} = 2 \times q_i$, with $q_1$ set to the minimum iteration time from profiling. The demotion step $\eta$ moves jobs to queues with two times larger quantum. For handling job bursts, the number of idle cache slots is determined from historical arrival patterns, though the paper does not specify the exact formula for calculating this number.

## 4. Key Insights and Innovations

FastServe introduces three fundamental innovations that distinguish it from prior serving systems like vLLM or Orca. These contributions are not merely incremental optimizations of GPU kernels; they represent structural changes to how LLM inference is scheduled and managed in memory. Below, we dissect each insight, explaining why prior approaches failed in this specific domain and how FastServe's design choices unlock significant performance gains.

### 4.1 Exploiting the "Semi Information-Agnostic" Nature of LLMs

The most theoretical contribution of FastServe is the reclassification of the LLM scheduling problem. In classical scheduling theory, algorithms typically fall into two categories:
1.  **Information-Agnostic:** The scheduler knows nothing about job size (e.g., Round Robin, classic MLFQ).
2.  **Information-Perfect:** The scheduler knows the exact remaining processing time (e.g., Shortest Remaining Processing Time, SRPT).

FastServe identifies that LLM inference fits neither category perfectly. It is **semi information-agnostic**. As established in **Section 2**, the total output length (job size) is unknown until generation completes. However, the input length is known at arrival, and crucially, the input length determines the execution time of the *first iteration* (initialization phase).

**Why this distinction matters:**
*   **Failure of Classic MLFQ:** A naive Multi-Level Feedback Queue (MLFQ) assumes all jobs start equal. It places every job in the highest priority queue. Because the initialization phase is computationally heavy (processing the entire prompt), a job with a long prompt exhausts its time quantum immediately during the first iteration. The scheduler incorrectly demotes this job to a lower priority, even if it only needs one more token to finish. This misclassification wastes GPU cycles on context switching and delays short jobs.
*   **Failure of SRPT:** SRPT is optimal for latency but requires knowing the total output length, which is impossible for autoregressive models.

**FastServe's Innovation:**
FastServe bridges this gap by using the *known* input length to bypass the "learning phase" of MLFQ. Instead of starting every job at the highest priority and waiting for demotion, the scheduler profiles the initialization time and **skip-joins** the job directly to the appropriate priority queue.

*   **Significance:** This allows FastServe to approximate the performance of SRPT (the theoretical optimum) without requiring oracle knowledge of the output length. **Figure 7** demonstrates this empirically: Skip-Join MLFQ achieves an average latency of 3.3 time units, closely trailing SRPT's 3.0, while naive MLFQ suffers 5.0 due to misclassification.
*   **Fundamental vs. Incremental:** This is a **fundamental innovation** in scheduling theory applied to ML. It modifies the classic MLFQ algorithm to accommodate a specific workload characteristic (bimodal iteration cost) that standard operating system schedulers do not encounter.

### 4.2 Preemption Without Penalty via Proactive Swapping

Enabling preemption (pausing a job to run another) is standard in CPUs but notoriously difficult in GPU-accelerated LLM serving due to memory constraints. As noted in **Section 2.3**, the Key-Value (KV) cache for a single OPT-175B job can consume **2.3 GB** of GPU memory. In a preemptive system, hundreds of jobs might be paused simultaneously, quickly exhausting the 80 GB capacity of an NVIDIA A100.

Prior systems handle memory pressure in ways that degrade performance:
*   **vLLM (FCFS):** Defers new jobs until memory is free. This reintroduces head-of-line blocking, negating the benefit of scheduling.
*   **Recompute Strawman:** Kills low-priority jobs to free memory. This wastes computation because the KV cache must be rebuilt from scratch when the job resumes.

**FastServe's Innovation:**
FastServe treats host memory (CPU RAM) as an extension of GPU memory and swaps KV caches proactively. The key insight is that swapping latency (data transfer) can be hidden entirely if it occurs *while* the GPU is computing other jobs.

*   **Mechanism:** Instead of reacting to memory pressure (swapping when OOM is imminent), FastServe predicts when a job will be needed using the Estimated Next Scheduled Time (ENST). It swaps out jobs with large ENST and swaps in jobs with small ENST *before* they are scheduled.
*   **Significance:** This makes preemption "free" in terms of critical path latency. **Figure 15(b)** shows that swapping time accounts for **less than 5%** of the end-to-end latency. Without this proactive overlap, swapping would add ~36 ms per token (on PCIe 4.0) to a ~60 ms generation task, a 60% overhead.
*   **Fundamental vs. Incremental:** This is a **fundamental systems innovation**. While swapping is a known OS technique, applying it to LLM KV caches with predictive timing (ENST) to enable *iteration-level preemption* at scale is novel. It removes the memory capacity barrier that previously made preemptive scheduling impractical for large models.

### 4.3 Shifting the Optimization Target from Execution to Queuing

The third insight is conceptual but drives the entire system design. Prior state-of-the-art systems like vLLM focus heavily on optimizing **execution time**. They introduce techniques like `PagedAttention` to reduce memory fragmentation, allowing larger batch sizes and faster kernel execution.

FastServe argues that for interactive workloads, execution time is no longer the primary bottleneck.

**Evidence from Real Workloads:**
*   **Figure 1** shows that in real-world datasets (ShareGPT, Alpaca), **queuing delay accounts for up to 90%** of total latency.
*   Optimizing the remaining 10% (execution time) yields diminishing returns. Even if execution time were reduced to zero, latency would only improve by 10% if queuing delay remains high.

**FastServe's Innovation:**
FastServe explicitly targets **queuing delay** via preemptive scheduling. By allowing short jobs to bypass long jobs (mitigating head-of-line blocking), it reduces the time jobs spend waiting in the queue.

*   **Significance:** This shift in focus explains the massive throughput gains reported in the **Abstract**. FastServe improves throughput by **31.4×** compared to vLLM under the same average latency requirements. This gain does not come from faster matrix multiplication; it comes from keeping the GPU busy with *ready* jobs rather than blocked jobs.
*   **Fundamental vs. Incremental:** This is a **fundamental shift in optimization strategy**. It suggests that for interactive LLM serving, scheduling policy (software logic) yields higher returns than kernel optimization (hardware utilization) once a baseline efficiency is reached. **Table 2** highlights this: FastServe adds Iteration-level Preemption (IP) to the feature set, which neither vLLM nor FasterTransformer possesses, directly addressing the queuing bottleneck.

### 4.4 Summary of Contributions

| Innovation | Prior State (e.g., vLLM) | FastServe Approach | Impact |
| :--- | :--- | :--- | :--- |
| **Scheduling** | FCFS or Naive MLFQ | **Skip-Join MLFQ** | Approximates SRPT latency without knowing output length (**Figure 7**). |
| **Memory** | Static Allocation or Deferral | **Proactive Swapping** | Enables preemption for large models with <5% latency overhead (**Figure 15**). |
| **Focus** | Execution Time Optimization | **Queuing Delay Optimization** | 31.4× throughput gain by eliminating head-of-line blocking (**Abstract**). |

These innovations collectively transform LLM serving from a batch-processing problem into a low-latency interactive service, capable of handling skewed workloads where job sizes vary wildly.

## 5. Experimental Analysis

This section examines how the authors validate FastServe's claims through systematic experimentation. We will walk through the evaluation methodology, dissect the quantitative results, and critically assess whether the evidence supports the paper's central thesis: that preemptive scheduling with proactive memory management can eliminate head-of-line blocking while respecting GPU memory constraints.

### 5.1 Evaluation Methodology

The authors design their experiments to answer three distinct questions: (1) Does FastServe outperform state-of-the-art systems in end-to-end latency and throughput? (2) Does it maintain tail latency guarantees despite using preemptive scheduling? (3) Which specific design components contribute to the performance gains?

#### Experimental Setup and Infrastructure

The end-to-end performance evaluation uses a production-grade cluster configuration. **Section 6.1** specifies:
*   **Hardware:** Two AWS EC2 `p4d.24xlarge` instances, each equipped with eight NVIDIA A100 40GB GPUs connected via NVLink.
*   **Host Memory:** 1152 GB per instance, connected via PCIe 4.0×16.
*   **Total Scale:** 16 GPUs for OPT-175B experiments, 4 GPUs for OPT-66B, and 1 GPU for OPT-13B.

This configuration is critical because it mirrors real-world deployment scenarios for large models. The OPT-175B model alone requires 350 GB for weights in FP16 (**Table 1**), necessitating distribution across all 16 GPUs. The design choice experiments (ablation studies) use a single NVIDIA A100 40GB GPU in a local testbed to isolate component effectiveness without distributed system complexity.

#### Models and Workloads

The evaluation covers three model scales from the OPT family (**Table 1**):

| Model | Memory (Weights) | Layers | Hidden Size | GPUs Required |
| :--- | :--- | :--- | :--- | :--- |
| OPT-13B | 26 GB | 40 | 5120 | 1 |
| OPT-66B | 132 GB | 64 | 9216 | 4 |
| OPT-175B | 350 GB | 96 | 12288 | 16 |

All experiments use **FP16 precision**. The workloads are derived from two real-world datasets:
*   **ShareGPT [13]:** User-shared conversations with ChatGPT, representing highly skewed, interactive workloads with variable input/output lengths.
*   **Alpaca [14]:** Instruction-following data generated by GPT-3.5, with generally shorter job sizes than ShareGPT.

Since these datasets lack arrival timestamps, the authors generate request arrival times following a **Poisson process** parameterized by arrival rate (jobs per second). This is standard practice in queuing theory experiments and allows controlled stress-testing at different load levels.

#### Baselines and Comparison Metrics

**Table 2** outlines the four systems compared:

| System | Iteration Preemption | Iteration Scheduling | PagedAttention | Pipeline Parallelism |
| :--- | :---: | :---: | :---: | :---: |
| FasterTransformer v5.3 | ✗ | ✗ | ✗ | ✓ |
| vLLM v0.1.7 | ✗ | ✓ | ✓ | ✗ |
| FastServe-FCFS | ✗ | ✓ | ✓ | ✓ |
| **FastServe (Full)** | **✓** | **✓** | **✓** | **✓** |

This comparison is carefully constructed:
*   **FasterTransformer:** Represents production-grade inference engines without LLM-specific scheduling.
*   **vLLM:** Represents the state-of-the-art in memory-efficient LLM serving (via PagedAttention) but uses FCFS scheduling.
*   **FastServe-FCFS:** A critical ablation baseline that uses FastServe's execution engine *without* the skip-join MLFQ scheduler. This isolates the scheduling innovation from implementation efficiency gains.

**Metrics:**
*   **Average Per-Token Latency:** Mean end-to-end latency divided by output length. This normalizes for variable job sizes.
*   **P95 Tail Latency:** The 95th percentile of per-token latency, critical for SLO compliance.
*   **Throughput under SLO:** Maximum job arrival rate achievable while meeting a latency Service Level Objective.
*   **SLO Definition:** Set to **10× the latency of a single decoding iteration**, specifically **0.3 seconds** based on profiling (**Section 6.1**).

> **Teaching Note:** The choice of *per-token* latency rather than *per-job* latency is deliberate. Since LLM jobs have variable output lengths, per-token latency provides a fair comparison across workloads. A job generating 100 tokens should not be penalized for taking longer than a job generating 5 tokens.

### 5.2 End-to-End Performance Results

The primary claim is that FastServe significantly outperforms existing systems under realistic workloads. The results are presented across multiple dimensions.

#### Average Latency and Throughput

**Figure 11** presents average latency versus job arrival rate for all three models under ShareGPT (first row) and Alpaca (second row) workloads. The key observation is the **maximum sustainable arrival rate** before latency exceeds the SLO (marked by the horizontal red line at 0.3 s/token).

**ShareGPT Workload (Highly Skewed):**
*   **OPT-13B (1 GPU):** FastServe sustains approximately **5.5 jobs/s** before violating SLO, while vLLM saturates at approximately **2.5 jobs/s** and FasterTransformer at approximately **0.5 jobs/s**.
*   **OPT-66B (4 GPUs):** FastServe achieves approximately **3.5 jobs/s**, vLLM approximately **1.5 jobs/s**, FasterTransformer approximately **0.3 jobs/s**.
*   **OPT-175B (16 GPUs):** FastServe achieves approximately **2.0 jobs/s**, vLLM approximately **0.6 jobs/s**, FasterTransformer approximately **0.1 jobs/s**.

The paper quantifies these gains in **Section 6.2**:
> "FastServe outperforms FasterTransformer by 31.5–74.9× in terms of throughput under the SLO... FastServe can significantly reduce the queuing delay and outperform vLLM by 2.3–18.3×."

**Alpaca Workload (Shorter Jobs):**
Since Alpaca jobs are shorter on average, all systems handle higher arrival rates. However, the relative gains persist:
*   FastServe outperforms vLLM by **3–31.4×**.
*   FastServe outperforms FasterTransformer by **9.5–15.8×**.

**Critical Observation: FastServe-FCFS Performance**
FastServe-FCFS (same execution engine, no preemptive scheduling) also outperforms vLLM, but by a smaller margin. The paper states:
> "FastServe-FCFS also outperforms vLLM, because it uses more efficient C++ implementation and fuses more operations into fewer GPU kernels. But it still suffers from the head-of-line blocking problem, which makes it slower than FastServe by 2–4×."

This is a crucial ablation result. It demonstrates that:
1.  **Implementation matters:** FastServe's C++ engine provides baseline improvements over vLLM's Python-heavy implementation.
2.  **Scheduling matters more:** The skip-join MLFQ scheduler adds an additional 2–4× improvement on top of implementation gains.

#### Tail Latency (P95)

A potential concern with preemptive scheduling is that long jobs might starve, increasing tail latency. **Figure 12** addresses this by showing P95 latency versus arrival rate.

**Key Results:**
*   For **OPT-175B**, FastServe improves throughput by up to **1.5×** compared to FastServe-FCFS under the same tail latency SLO.
*   FastServe outperforms vLLM by up to **17.9×** and FasterTransformer by up to **59.8×** for tail latency.

> **Teaching Note:** This result is counterintuitive but important. Prioritizing short jobs *also* helps long jobs. Why? Because long jobs spend less time waiting in the queue behind other long jobs. The starvation prevention mechanism (**Section 4.1**, parameter $\alpha = 300$ ms) ensures that no job waits indefinitely. The net effect is that *all* jobs experience reduced queuing delay.

#### Goodput Under Different SLOs

**Figure 13** examines P95 goodput (throughput where 95% of jobs meet SLO) under varying SLO strictness (5×, 10×, 20× of baseline latency).

**Results:**
*   FastServe outperforms vLLM by **4.1× to 4.7×** across all SLO settings.
*   FastServe outperforms FastServe-FCFS by **1.46× to 1.64×**.

This demonstrates that FastServe's benefits are **robust across different latency requirements**. Whether the application demands strict (5×) or relaxed (20×) SLOs, FastServe consistently achieves higher goodput.

### 5.3 Ablation Studies and Design Choice Validation

The paper includes targeted experiments to validate each core innovation independently.

#### Effectiveness of Skip-Join MLFQ (**Figure 14**)

This experiment compares four scheduling policies on OPT-13B with ShareGPT workload, varying the ratio between input and output lengths (to simulate industry trends toward longer context windows).

| Scheduler | Performance vs. FCFS | Performance vs. Naive MLFQ | Performance vs. Fixed Priority |
| :--- | :--- | :--- | :--- |
| Skip-Join MLFQ | Up to **8.9×** | Up to **1.87×** | Up to **13.9×** |

**Interpretation:**
*   **FCFS:** Consistently high latency regardless of input/output ratio due to head-of-line blocking.
*   **Naive MLFQ:** Performs well when input/output ratio is small (initialization phase is short), but degrades as ratio increases because jobs are incorrectly demoted after the heavy initialization phase.
*   **Fixed Priority:** Excels when initialization dominates (high ratio) but underperforms when decoding dominates (low ratio) because it ignores decoding phase characteristics.
*   **Skip-Join MLFQ:** Consistently optimal across all ratios because it leverages the semi information-agnostic setting—using input length to place jobs correctly without needing output length.

> **Teaching Note:** This ablation validates the core theoretical contribution. The skip-join mechanism is not just an engineering tweak; it fundamentally addresses the bimodal cost structure of LLM inference (heavy initialization, light decoding).

#### Effectiveness of Proactive Key-Value Cache Management (**Figure 15**)

This experiment compares three memory management strategies on OPT-13B with ShareGPT workload:

1.  **Recompute:** Kill low-priority jobs and recompute KV cache when resumed.
2.  **Reactive Swapping:** Swap KV cache to host memory only when GPU memory is full.
3.  **Proactive Swapping (FastServe):** Predictively swap based on ENST, overlapping transfer with computation.

**Quantitative Results:**
*   Proactive swapping outperforms Recompute by **2.7×**.
*   Proactive swapping outperforms Reactive by **1.7×**.

**Latency Breakdown (Figure 15b):**
The authors decompose end-to-end latency into three components:
*   **Queuing Delay:** Time waiting in scheduler queues.
*   **Execution Time:** GPU kernel execution for token generation.
*   **Swapping Time:** Time blocked by KV cache transfer.

> "As shown in Figure 15(b), the swapping time is less than 5% of the end-to-end latency, which is negligible compared to the execution time and queuing delay."

This is a critical validation. If swapping were on the critical path, it would add approximately 36 ms per token (for 2.3 GB over PCIe 4.0×16) to a 60 ms generation task—a 60% overhead. By overlapping swapping with computation, FastServe reduces this to less than 5%, making preemption effectively "free" in terms of latency.

### 5.4 Critical Assessment of Experimental Claims

#### Do the Experiments Support the Claims?

**Yes, with strong evidence.** The experimental design addresses the key claims systematically:

| Claim | Supporting Evidence | Strength |
| :--- | :--- | :--- |
| HOL blocking causes 90% queuing delay | **Figure 1** (ShareGPT/Alpaca breakdown) | Strong - Direct measurement |
| Skip-join MLFQ approximates SRPT | **Figure 7** (3.3 vs. 3.0 avg latency), **Figure 14** (8.9× over FCFS) | Strong - Multiple validations |
| Proactive swapping hides overhead | **Figure 15b** (<5% swapping time) | Strong - Direct breakdown |
| 31.4× throughput improvement | **Abstract**, **Section 6.2** (Alpaca OPT-175B) | Conditional - Depends on workload |

#### Conditions and Trade-offs

The results are **workload-dependent**. The 31.4× improvement is achieved under specific conditions:
*   **Skewed Workloads:** ShareGPT and Alpaca have long-tailed output length distributions. If all jobs had identical lengths (as in the "Fix-Len" columns of **Figure 1**), the improvement would be minimal because HOL blocking would not occur.
*   **Memory-Constrained Settings:** The benefits of proactive swapping are most pronounced when GPU memory is insufficient to hold all active KV caches. In memory-abundant scenarios, the advantage diminishes.
*   **Interactive Applications:** The SLO of 0.3 seconds is tuned for interactive use cases. Batch-oriented workloads with relaxed latency requirements may not benefit as much from preemptive scheduling.

#### Potential Limitations

1.  **Model Diversity:** All experiments use the OPT family. While OPT is representative, the paper does not evaluate on other architectures (e.g., LLaMA, PaLM) which may have different KV cache structures or attention mechanisms.

2.  **Burst Handling:** **Section 4.2** mentions reserving idle cache slots for job bursts, but **Section 6** does not include experiments specifically testing burst arrival patterns. The number of reserved slots is determined from "historical job arrival patterns," but the exact formula is not specified.

3.  **Multi-Tenancy:** The experiments assume a single tenant submitting jobs. Real-world deployments often involve multiple tenants with different priority levels and SLOs. The starvation prevention mechanism ($\alpha = 300$ ms) is tuned for a single SLO; multi-tenant scenarios may require more complex priority structures.

4.  **Swapping Bandwidth Sensitivity:** The 5% swapping overhead assumes PCIe 4.0×16 at full bandwidth. Systems with slower interconnects (e.g., PCIe 3.0, or shared bandwidth in cloud environments) may experience higher swapping overhead, though the proactive overlapping mechanism should still provide benefits.

### 5.5 Summary of Experimental Findings

The experimental evaluation is comprehensive and well-structured. The authors use appropriate baselines (including a critical FastServe-FCFS ablation), realistic workloads (ShareGPT, Alpaca), and production-grade hardware (16× A100 for OPT-175B). The key findings are:

1.  **Throughput Gains Are Real:** FastServe achieves **2.3–31.4×** throughput improvement over vLLM under the same latency SLOs, with the highest gains on skewed workloads.
2.  **Tail Latency Is Protected:** Despite prioritizing short jobs, FastServe improves tail latency by up to **17.9×** over vLLM, validated by the starvation prevention mechanism.
3.  **Both Innovations Matter:** The skip-join MLFQ scheduler provides up to **8.9×** improvement over FCFS, and proactive swapping provides **1.7×** improvement over reactive swapping.
4.  **Overhead Is Minimal:** Swapping accounts for **<5%** of end-to-end latency, confirming that proactive overlapping successfully hides memory transfer costs.

The experiments convincingly demonstrate that FastServe's dual focus on scheduling policy and memory infrastructure addresses the fundamental bottlenecks of LLM inference serving. The results are not merely incremental kernel optimizations; they represent a systemic rethinking of how LLM jobs are scheduled and managed in memory-constrained environments.

## 6. Limitations and Trade-offs

While FastServe demonstrates significant performance improvements over state-of-the-art systems like vLLM, its design introduces specific trade-offs and relies on assumptions that may not hold in all deployment scenarios. Understanding these limitations is crucial for practitioners deciding whether to adopt this architecture. We categorize these constraints into foundational assumptions, system complexity trade-offs, and unaddressed operational scenarios.

### 6.1 Foundational Assumptions and Stability Risks

FastServe's core scheduling logic depends on the stability and accuracy of specific system properties. If these assumptions are violated, the performance guarantees may degrade.

*   **Profiling Accuracy and Hardware Stability:**
    The skip-join MLFQ scheduler relies on a profiling function `P.getNextIterTime(job)` to estimate the initialization phase time based on input length (**Section 4.1**). The system assumes this profiling data remains accurate across time.
    *   **Risk:** In multi-tenant cloud environments, GPU performance can fluctuate due to noise, thermal throttling, or interference from co-located processes. If the actual initialization time exceeds the profiled time, a job might exhaust its quantum earlier than expected, triggering unnecessary demotions.
    *   **Impact:** Frequent misclassification could cause the scheduler to behave more like a naive MLFQ, reducing the approximation quality to SRPT. The paper does not quantify the sensitivity of the scheduler to profiling errors (e.g., what happens if iteration time varies by $\pm 10\%$?).

*   **Static Input Length at Arrival:**
    The skip-join mechanism requires the input length to be known at job arrival to determine the initial queue (**Section 4.1**).
    *   **Limitation:** This assumes the prompt is fully available before inference begins. It does not explicitly support **streaming inputs** where the user continues typing while the model generates responses, or scenarios where the prompt is dynamically modified during generation.
    *   **Consequence:** Supporting dynamic prompt growth would require re-profiling and potentially re-queueing jobs mid-execution, adding significant complexity to the state machine.

*   **Memory Bandwidth Dependence:**
    The proactive swapping mechanism assumes high-bandwidth interconnects to hide data transfer latency behind computation. **Section 4.2** specifies that swapping 2.3 GB over PCIe 4.0×16 takes approximately 36 ms, compared to a 60 ms token generation time.
    *   **Trade-off:** The system trades GPU memory capacity for host memory bandwidth. If the host-to-GPU bandwidth is lower (e.g., PCIe 3.0, or shared bandwidth in virtualized environments), the swap time could exceed the generation time.
    *   **Threshold:** If swap time $> $ generation time, the proactive overlapping strategy fails to hide the latency, and swapping becomes a bottleneck on the critical path. The paper does not provide a breakdown of performance degradation under lower bandwidth conditions.

### 6.2 System Complexity and Overhead

FastServe achieves its performance gains by introducing significant system complexity compared to simpler FCFS-based systems. This complexity manifests in code maintenance, CPU overhead, and energy consumption.

*   **Implementation Complexity:**
    FastServe requires custom management of distributed state that simpler systems avoid.
    *   **Evidence:** The implementation consists of 2.9K lines of Python for the scheduler and 8.1K lines of C++/CUDA for the execution engine (**Section 5**). It also requires custom CUDA kernels to support iteration-level preemption and integration with PagedAttention.
    *   **Trade-off:** While vLLM focuses on memory fragmentation reduction via `PagedAttention`, FastServe adds a complex scheduling layer on top. This increases the engineering burden for maintenance, debugging, and integration with new model architectures. For teams prioritizing simplicity over maximum throughput, this complexity may be a deterrent.

*   **CPU Scheduling Overhead:**
    The scheduler must calculate the Estimated Next Scheduled Time (ENST) for every job in the queue to determine swapping order (**Section 4.2**).
    *   **Mechanism:** The ENST calculation involves summing quantum values for all higher-priority jobs:
        $$T_{execute}(i) = \frac{1}{B} \sum_{i.priority < j.priority} T_{execute}(i, j)$$
    *   **Scalability Concern:** As the number of pending jobs grows into the thousands, this calculation could become computationally expensive on the CPU. The paper does not report the CPU utilization of the scheduler itself. At extreme scales, the scheduler could become a bottleneck, limiting the maximum job arrival rate regardless of GPU capacity.

*   **Energy Efficiency Trade-off:**
    FastServe prioritizes latency and throughput over energy efficiency.
    *   **Observation:** Proactive swapping involves moving large tensors (e.g., 2.3 GB per job for OPT-175B) between host and GPU memory continuously. Data movement is energy-intensive.
    *   **Open Question:** While latency improves by reducing queuing delay, the total energy per token might increase due to the overhead of PCIe transfers and additional kernel launches for context switching. The paper does not measure power consumption, leaving the energy-latency trade-off unquantified.

### 6.3 Unaddressed Operational Scenarios

Certain real-world deployment scenarios are either simplified in the evaluation or not addressed in the design.

*   **Multi-Tenancy and Tiered SLOs:**
    The evaluation assumes a single Service Level Objective (SLO) for all jobs (0.3 seconds per token, **Section 6.1**). The starvation prevention threshold $\alpha$ is tuned based on this user-specified SLO (**Section 4.1**).
    *   **Limitation:** Real-world serving platforms often host multiple tenants with different priority levels and SLOs (e.g., premium users vs. free users).
    *   **Gap:** FastServe's current MLFQ design prioritizes based on job length (via input profiling) and starvation time, not tenant identity. Supporting tiered SLOs would require modifying the priority queue structure to incorporate tenant weights, which is not discussed.

*   **Burst Handling Specifics:**
    **Section 4.2** mentions that FastServe reserves idle key-value cache slots for bursts of new jobs to avoid reactive eviction.
    *   **Missing Detail:** The paper states the number of reserved slots is "based on historical job arrival patterns," but does not provide the exact formula or algorithm for determining this number.
    *   **Risk:** If the burst prediction is inaccurate (e.g., a sudden viral event spikes traffic beyond historical norms), the reserved slots may be insufficient, forcing the system into reactive swapping anyway. **Section 6** does not include experiments specifically testing extreme burst scenarios, leaving the robustness of this mechanism unverified.

*   **Model Architecture Diversity:**
    All experiments use the OPT family of models (**Table 1**).
    *   **Assumption:** The design assumes KV cache memory scales linearly with sequence length and layer count ($4 \times l \times h \times (s+t)$ bytes, **Section 4.2**).
    *   **Limitation:** Newer architectures like LLaMA-2 or models using **Grouped-Query Attention (GQA)** have different KV cache memory profiles (often smaller due to shared key/value heads). FastServe's memory management and profiling might need recalibration for these models. For instance, if the KV cache is smaller, the benefit of swapping diminishes relative to the overhead. The paper does not evaluate performance on non-OPT architectures.

*   **Small Model Regime:**
    FastServe is optimized for large models (e.g., OPT-175B) that require distributed inference and face severe memory constraints.
    *   **Trade-off:** For smaller models (e.g., OPT-13B on a single GPU) where memory is abundant, the overhead of proactive swapping and complex scheduling might outweigh the benefits.
    *   **Evidence:** In **Figure 11(a)** (OPT-13B, 1 GPU), the throughput gap between FastServe and vLLM is smaller compared to OPT-175B (**Figure 11(c)**). In memory-abundant scenarios, simpler FCFS scheduling with large batches might be more efficient due to lower CPU overhead.

### 6.4 Parameter Sensitivity and Tuning

FastServe introduces several hyperparameters that require tuning for optimal performance. The paper provides defaults but does not fully explore the sensitivity landscape.

*   **Starvation Threshold ($\alpha$):**
    Set to 300 ms by default (**Section 4.1**).
    *   **Trade-off:** A lower $\alpha$ reduces tail latency for long jobs but increases context switching overhead (more promotions). A higher $\alpha$ improves throughput for short jobs but risks violating tail latency SLOs for long jobs.
    *   **Limitation:** The paper does not provide a sensitivity analysis showing how performance varies if $\alpha$ is set to 100 ms or 1000 ms. Operators may need to perform expensive grid searches to find the optimal value for their specific workload.

*   **Quantum Progression:**
    The queue quantums follow a geometric progression $q_{i+1} = 2 \times q_i$ (**Section 4.1**).
    *   **Assumption:** This progression assumes a specific distribution of job lengths. If the workload distribution changes (e.g., suddenly many medium-length jobs appear), this fixed progression might not align with the optimal demotion points, leading to suboptimal queue placement.

*   **Reserved Cache Slots:**
    As noted in **Section 4.2**, the number of idle slots for burst handling is based on historical patterns.
    *   **Risk:** This introduces a dependency on workload stationarity. If the workload distribution shifts conceptually (e.g., from chat to code generation), the historical data may become stale, leading to poor memory reservation decisions.

### 6.5 Summary of Trade-offs

| Feature | Benefit | Cost / Limitation |
| :--- | :--- | :--- |
| **Skip-Join MLFQ** | Approximates SRPT latency without output length knowledge. | Requires accurate input profiling; sensitive to hardware noise. |
| **Proactive Swapping** | Enables preemption without OOM; hides swap latency. | Depends on high PCIe bandwidth; increases energy consumption. |
| **Distributed State** | Supports massive models (OPT-175B). | High implementation complexity (8.1K C++ lines); CPU scheduling overhead. |
| **Starvation Prevention** | Protects tail latency for long jobs. | Requires tuning $\alpha$; adds context switch overhead. |

In summary, FastServe is not a universal replacement for all serving systems. It is specifically optimized for **interactive, latency-sensitive applications** running **large models** on **memory-constrained clusters** with **skewed workloads**. For batch-oriented tasks, memory-abundant deployments, or scenarios requiring strict multi-tenant isolation, the complexity and overhead of FastServe may not be justified. Practitioners must weigh the 31.4× throughput gains against the increased engineering burden and dependency on stable hardware profiling.

## 7. Implications and Future Directions

FastServe does more than improve throughput numbers—it fundamentally reframes how the systems community should approach LLM serving. This section examines how the work shifts the field's optimization priorities, identifies concrete research opportunities it unlocks, and provides practical guidance for practitioners deciding whether to adopt these techniques.

### 7.1 Paradigm Shift: From Execution to Scheduling Optimization

The most significant implication of FastServe is its demonstration that **scheduling policy matters more than kernel optimization** for interactive LLM workloads. Prior to this work, the serving community focused heavily on execution efficiency: `PagedAttention` [11] reduces memory fragmentation, custom CUDA kernels minimize kernel launch overhead, and quantization compresses model weights. FastServe shows that these optimizations address only the minority component of latency.

**Evidence from Figure 1:**
> "Figure 1 shows that up to 90% of total latency is the queuing delay for real-world datasets. In this case, optimizing execution time is not enough... Instead, we need to optimize the queuing delay."

This finding has three cascading implications for the field:

1.  **Re-evaluation of Prior Work:** Systems like vLLM achieve impressive throughput through memory efficiency, but FastServe demonstrates that adding preemptive scheduling on top yields an additional **2–4× improvement** (comparing FastServe to FastServe-FCFS in **Section 6.2**). This suggests that future serving systems should treat scheduling as a first-class optimization target, not an afterthought.

2.  **Workload-Dependent Optimization:** The optimal serving strategy depends on workload characteristics. For skewed workloads like ShareGPT and Alpaca, preemptive scheduling is essential. For uniform workloads (the "Fix-Len" columns in **Figure 1**), FCFS performs adequately. This implies that serving systems should include **workload characterization** as a prerequisite for selecting scheduling policies.

3.  **Scheduling Theory for ML:** FastServe bridges classical scheduling theory (MLFQ, SRPT) with ML-specific constraints (autoregressive generation, KV cache memory). This opens a new research direction: **ML-aware scheduling algorithms** that exploit model architecture properties (e.g., attention patterns, layer sparsity) rather than treating inference as a black box.

### 7.2 Research Opportunities Enabled

FastServe's design choices reveal several open questions that warrant follow-up investigation. We categorize these into scheduling extensions, memory management innovations, and system architecture evolution.

#### Scheduling Extensions

*   **Multi-Tenant Priority Scheduling:** FastServe's current MLFQ design prioritizes jobs based on input length and starvation time (**Section 4.1**). Real-world deployments often require **tiered SLOs** (e.g., premium vs. free users). Future work could extend skip-join MLFQ to incorporate tenant weights, creating a hybrid priority system that balances fairness with latency optimization. The starvation prevention threshold $\alpha = 300$ ms (**Section 4.1**) could become tenant-configurable.

*   **Adaptive Quantum Tuning:** FastServe uses a fixed geometric progression for queue quantums ($q_{i+1} = 2 \times q_i$, **Section 4.1**). However, workload distributions change over time (e.g., daytime chat vs. nighttime batch processing). Research on **online quantum adaptation** could adjust queue parameters dynamically based on observed job length distributions, potentially improving the SRPT approximation quality.

*   **Streaming Input Support:** FastServe assumes the full input prompt is available at arrival time (**Section 4.1**). Emerging applications like real-time transcription or interactive coding assistants involve **streaming inputs** where the prompt grows during generation. Extending skip-join MLFQ to handle dynamic input lengths—potentially with mid-execution re-queueing—would broaden applicability.

#### Memory Management Innovations

*   **NVMe-Based Swapping:** FastServe swaps KV caches between GPU and host memory over PCIe 4.0×16 (**Section 4.2**). With the rise of GPU-direct storage and NVMe-based memory tiers, future systems could extend the cache hierarchy to include **persistent storage**, enabling even larger batch sizes at the cost of higher swap latency. The proactive ENST-based swapping algorithm (**Section 4.2**) would need recalibration for slower storage tiers.

*   **Compression-Aware Swapping:** The KV cache formula ($4 \times l \times h \times (s + t)$ bytes, **Section 4.2**) assumes FP16 precision. Recent work on KV cache quantization (e.g., 4-bit or 8-bit KV caches) could reduce swap bandwidth requirements by 2–4×. Integrating compression with proactive swapping could enable preemption for even larger models or memory-constrained edge deployments.

*   **Energy-Aware Memory Management:** FastServe does not measure power consumption (**Section 6.4**). Data movement between host and GPU memory is energy-intensive. Future work could develop **energy-aware swapping policies** that trade off latency for power efficiency, particularly relevant for datacenters with strict power budgets.

#### System Architecture Evolution

*   **Disaggregated Prefill-Decode:** Concurrent work like Splitwise [53] and DistServe [41] disaggregates the initialization (prefill) and decoding phases onto separate hardware. FastServe could integrate with this architecture: the skip-join MLFQ scheduler could route jobs to prefill or decode clusters based on phase, while the proactive swapping mechanism manages KV cache transfer between clusters. This hybrid approach could combine the benefits of both designs.

*   **Heterogeneous GPU Clusters:** FastServe's evaluation uses homogeneous A100 clusters (**Section 6.1**). Real-world deployments often mix GPU generations (e.g., A100 + V100) or include specialized accelerators (TPUs, NPUs). Extending the ENST calculation (**Section 4.2**) to account for heterogeneous execution speeds would enable FastServe to operate in mixed-hardware environments.

*   **Model Architecture Diversity:** All experiments use the OPT family (**Table 1**). Newer architectures like LLaMA-2 with **Grouped-Query Attention (GQA)** [29] have smaller KV caches due to shared key/value heads. Research on how FastServe's memory management adapts to different attention mechanisms would validate generalizability. For GQA models, the swap overhead might be lower, potentially changing the optimal swapping threshold.

### 7.3 Practical Applications and Use Cases

FastServe's design targets specific deployment scenarios. Understanding when to adopt these techniques prevents unnecessary complexity in scenarios where simpler systems suffice.

#### Recommended Deployment Scenarios

| Scenario | Recommendation | Rationale |
| :--- | :--- | :--- |
| **Interactive Chatbots** (ChatGPT-like) | **Adopt FastServe** | Highly skewed workloads with variable output lengths; latency SLOs are critical (**Figure 11**). |
| **Large Model Serving** (100B+ parameters) | **Adopt FastServe** | Memory constraints make preemption essential; OPT-175B requires 350 GB for weights alone (**Table 1**). |
| **Memory-Constrained Clusters** | **Adopt FastServe** | Proactive swapping enables more concurrent jobs than GPU memory alone permits (**Figure 8**). |
| **Batch-Oriented Workloads** | **Consider vLLM** | If latency SLOs are relaxed (e.g., >5 seconds), FCFS batching may be more efficient due to lower CPU overhead. |
| **Small Models** (<30B parameters) | **Evaluate Carefully** | For OPT-13B on 1 GPU, FastServe's advantage over vLLM is smaller (**Figure 11a**); complexity may not be justified. |
| **Uniform Workloads** | **Consider vLLM** | If input/output lengths are predictable and uniform, HOL blocking is minimal (**Figure 1**, Fix-Len columns). |

#### Integration Considerations

Practitioners considering FastServe should evaluate the following factors:

1.  **Profiling Infrastructure:** FastServe requires accurate iteration time profiling (`P.getNextIterTime(job)`, **Section 4.1**). Organizations must maintain profiling data for each model-hardware combination. If hardware performance fluctuates (e.g., multi-tenant cloud), profiling accuracy may degrade.

2.  **Host Memory Capacity:** Proactive swapping extends GPU memory to host memory (**Section 4.2**). For OPT-175B with hundreds of concurrent jobs, host memory requirements can exceed 100 GB. Ensure sufficient RAM is available; otherwise, swapping to disk would introduce unacceptable latency.

3.  **PCIe Bandwidth:** The <5% swapping overhead (**Figure 15b**) assumes PCIe 4.0×16 at full bandwidth. On systems with PCIe 3.0 or shared bandwidth (common in cloud VMs), swap times may increase, reducing the effectiveness of proactive overlapping.

4.  **SLO Tuning:** The starvation prevention threshold $\alpha = 300$ ms (**Section 4.1**) is tuned for interactive workloads. For applications with different latency requirements, operators should tune $\alpha$ based on their SLO. The paper does not provide automated tuning guidance, so empirical testing is necessary.

### 7.4 Reproduction and Deployment Guidance

For teams interested in reproducing FastServe's results or integrating its techniques into existing systems, we provide the following practical guidance.

#### Minimum Viable Implementation

Teams need not implement the full system to benefit from FastServe's insights. A **minimal viable implementation** could include:

1.  **Skip-Join Logic Only:** Implement the skip-join MLFQ scheduler (**Algorithm 1**, lines 6–9) on top of an existing serving system like vLLM. This requires:
    *   Profiling initialization time vs. input length (one-time cost per model).
    *   Modifying the job queue to support multiple priority levels.
    *   Adding starvation prevention with a configurable $\alpha$ threshold.

2.  **Reactive Swapping First:** Before implementing proactive ENST-based swapping (**Section 4.2**), start with reactive swapping (swap when GPU memory is full). This provides most memory capacity benefits with simpler implementation. Upgrade to proactive swapping once the baseline is stable.

3.  **Single-GPU Prototype:** The design choice experiments in **Section 6.3** use a single A100 GPU. Teams can validate the scheduling logic on a single GPU before scaling to distributed inference, reducing debugging complexity.

#### Key Metrics to Monitor

When deploying FastServe or similar systems, monitor these metrics to ensure correct operation:

*   **Queue Distribution:** Track the number of jobs in each priority queue ($Q_1$ through $Q_n$). An imbalance (e.g., all jobs in $Q_n$) indicates profiling or quantum configuration issues.
*   **Swap Ratio:** Measure the percentage of jobs that require swapping. If >50% of jobs swap frequently, consider increasing host memory or reducing maximum batch size.
*   **Starvation Events:** Count how often jobs are promoted due to starvation prevention (**Algorithm 1**, lines 19–21). High promotion rates suggest $\alpha$ is too large or workload is too skewed.
*   **Swapping Overhead:** Track swapping time as a percentage of end-to-end latency. If >10% (vs. <5% in **Figure 15b**), investigate PCIe bandwidth or proactive prediction accuracy.

#### When to Prefer Alternatives

FastServe is not universally optimal. Consider alternatives in these scenarios:

*   **vLLM:** Prefer when workloads are uniform, models are small (<30B parameters), or latency SLOs are relaxed (>1 second per token). vLLM's simpler FCFS scheduling has lower CPU overhead.
*   **FasterTransformer:** Prefer when pipeline parallelism is not needed and production-grade kernel optimization is the priority. FasterTransformer's C++ implementation is mature and well-supported by NVIDIA.
*   **DistServe/Splitwise:** Prefer when prefill and decode phases can be physically separated onto different hardware. These systems optimize for phase-specific resource allocation rather than preemptive scheduling.

### 7.5 Long-Term Vision: Intelligent LLM Serving Infrastructure

FastServe represents a step toward **intelligent serving infrastructure** that adapts to workload characteristics, model architectures, and hardware constraints dynamically. Future systems could incorporate:

*   **Learned Scheduling Policies:** Instead of fixed MLFQ rules, use reinforcement learning to optimize queue placement and demotion thresholds based on observed workload patterns.
*   **Cross-Model Optimization:** Serve multiple models (e.g., OPT-13B + OPT-175B) on the same cluster with unified scheduling, balancing resource allocation across model sizes.
*   **Edge-Cloud Hybrid Serving:** Extend proactive swapping to include edge devices, where KV caches could be partially stored on edge hardware and swapped to cloud GPUs for large-scale generation.

The core insight—that **queuing delay dominates execution time for interactive LLM workloads** (**Figure 1**)—should guide all future serving system design. FastServe demonstrates that addressing this bottleneck requires co-design of scheduling algorithms and memory management, not just kernel optimization. As LLMs continue to grow in size and deployment scale, this holistic approach will become increasingly critical for delivering responsive, cost-effective AI services.
