## 1. Executive Summary

This paper presents KernelEvolve, an agentic kernel coding framework that automates high-performance kernel generation across heterogeneous AI accelerators including NVIDIA, AMD, and Meta's MTIA v3, addressing the combinatorial explosion of model architecture and hardware diversity. By leveraging graph-based search and retrieval-augmented prompt synthesis, KernelEvolve reduces development time from weeks to hours while achieving 100% correctness across 480 operator-platform configurations and delivering up to 17× speedups over PyTorch baselines on production recommendation models like WuKong and InterFormer (Figure 4). This system enables efficient monolithic deployment on proprietary hardware, eliminating the 10–20ms latency overhead associated with disaggregated serving architectures caused by missing kernel support (Table 2).

## 2. Context and Motivation

To understand why KernelEvolve matters, we must first grasp the scale of the problem it solves. This is not an academic optimization exercise—it addresses a fundamental bottleneck in Meta's advertising infrastructure serving **hundreds of trillions of ad-ranking inferences daily** across global data centers. The challenge stems from what the paper terms "The Curse of Dimensionality" (Section 1.1): a combinatorial explosion across three independent dimensions that makes manual kernel development economically infeasible.

### 2.1 The Three-Dimensional Diversity Challenge

The paper identifies three orthogonal sources of complexity that together create an unsustainable optimization space:

**Hardware Diversity Across Vendors and Generations.** Production deployment spans heterogeneous accelerators including NVIDIA GPUs, AMD GPUs, and Meta's custom MTIA chips (Figure 1). Each architecture exhibits fundamentally different characteristics:

- **Memory hierarchy heterogeneity:** NVIDIA employs multi-level cache hierarchies with tens of megabytes of L2 cache; AMD features large Infinity Cache structures as shared L3-equivalent storage; MTIA implements custom on-chip SRAM subsystems optimized for recommendation inference patterns. These differences necessitate architecture-specific memory access patterns and tiling strategies.

- **Programming model fragmentation:** Each platform exposes hardware through incompatible abstractions—CUDA's thread-block model, Triton's tile-based DSL, AMD's ROCm/HIP extensions, CuTe's layout algebra for NVIDIA Hopper, and MTIA's C++ kernel DSL. As Figure 2 illustrates, Triton compiles through different MLIR dialects for each target (TritonGPU-MLIR, TritonAMDGPU-MLIR, TritonMTIA-MLIR), but the underlying execution models differ fundamentally—from Triton's automatic memory coalescing to CUDA's explicit shared memory management.

- **Generational architectural discontinuities:** Even within a single vendor, generational transitions introduce fundamentally different execution models. NVIDIA's Ampere-to-Hopper evolution added the Tensor Memory Accelerator (TMA) for asynchronous bulk tensor transfers, introduced a 128-thread warp-group execution model for WGMMA tensor operations, and exposed multiple asynchronous execution pipelines. These features require kernel developers to adopt new pipeline structures that differ significantly from traditional warp-centric Ampere kernels.

The economic consequence is stark: **each platform-specific implementation demands 2-8 weeks of expert optimization effort**, with 12-18 month hardware generation update cycles that invalidate existing optimizations.

**Model Architecture Diversity Across Ranking Stages.** Modern ads ranking employs multi-stage architectures where computational characteristics vary dramatically (Table 1):

| Ranking Stage | Candidates Processed | Model Count | Complexity |
|--------------|---------------------|-------------|------------|
| Early stage | 10K ads | ≥150 models | 0.01-0.1 GFLOPS/request |
| Late stage | 100 ads | ≥200 models | 0.2-2 GFLOPS/request |
| Transformer-based | 100 ads | — | ~80 GFLOPS/request† |

†Transformer-based sequence models require 10-100× computational complexity increases over traditional dense ranking models.

Each stage demands distinct kernel optimization strategies: retrieval favors batched operations maximizing memory bandwidth utilization; early-stage ranking requires balanced compute-memory workloads; final-stage ranking demands maximum single-request performance through aggressive operator fusion. This architectural diversity—spanning from traditional MLPs to attention-based sequence models—prevents one-size-fits-all kernel optimization approaches.

**Kernel Primitive Diversity Beyond GEMM.** While dense matrix multiplication (GEMM) operations benefit from mature, highly-optimized libraries (cuBLAS, FBGEMM, DeepGEMM), production ads ranking workloads execute **over 200 distinct data preprocessing operators** as integral components of model inference pipelines. These transformations span:

- Feature derivation (bucketization, set operations, n-gram hashing)
- Dense normalization (variance-stabilizing transformations, one-hot encoding)
- Sparse normalization (cryptographic hashing with modulo reduction, top-k truncation)

Critically, the paper emphasizes that **preprocessing kernel absence creates a binary deployment constraint rather than incremental performance loss**. A single missing preprocessing operator—regardless of arithmetic intensity—blocks entire model deployments on new accelerators.

### 2.2 Why This Problem Matters: The Disaggregation Penalty

The real-world impact becomes clear when examining serving architecture constraints. When critical operators lack native implementations on AI accelerators, production systems must adopt **disaggregated serving topologies** where missing operators execute on separate CPU tiers. Table 2 quantifies these costs for a production MTIA deployment:

| Serving Paradigm | P50 | P75 | P90 | P99 |
|-----------------|-----|-----|-----|-----|
| Paradigm 1: Client → MTIA Tier | 39ms | 44ms | 46ms | 61ms |
| Paradigm 2: Client → CPU → MTIA | 58ms | 65ms | 73ms | 97ms |
| Extra Network Latency (δ) | — | — | — | **10-20ms** |

The 10-20ms network overhead represents **pure architectural tax consuming a substantial portion of the sub-100ms latency budget** required for ads serving. This fragmentation compounds through cross-node serialization costs, cascading failure modes across service tiers, and increased Total Cost of Ownership from redundant CPU infrastructure.

> "While suboptimal GEMM implementations degrade performance incrementally, missing preprocessing kernels impose architectural penalties preventing monolithic deployment entirely."

This constraint paradoxically elevates preprocessing operators to equal or greater priority than compute-intensive kernels. Comprehensive kernel coverage enabling monolithic accelerator deployment emerges as a first-order requirement for cost-effective serving.

### 2.3 Prior Approaches and Their Limitations

The paper surveys existing AI-powered kernel authoring systems in Section 1 ("Emerging AI-Powered Kernel Authoring") and Section 7 ("Related Work"). These fall into two categories:

**Traditional Kernel Optimization Frameworks.** Systems like Halide, TVM, and Triton provide abstractions for kernel development but still require substantial domain expertise. Halide decouples algorithm specification from scheduling; TVM extends this with learned cost models; Triton provides a Python-embedded DSL abstracting GPU programming at the block level. Recent work introduces higher-level abstractions (NVIDIA's CuTe DSL, Meta's TLX, OpenAI's Gluon, TileLang, Helion). However, **these abstractions reduce development effort but still require substantial domain expertise for novel kernel transformations and struggle to generalize across heterogeneous hardware without manual adaptation**.

**LLM-Based Kernel Generation Systems.** Recent academic and industry efforts include:

- **KernelBench** (Ouyang et al., 2025): Benchmarks LLM capabilities across three difficulty levels spanning canonical GPU operators
- **AutoTriton** (Li et al., 2025): Applies reinforcement learning to Triton programming
- **KernelLLM** (Fisches et al., 2025): Explores code generation with world models
- **TritonRL** (Woo et al., 2025): Trains LLMs for Triton synthesis
- **GEAK-agent** (Wang et al., 2025b): Targets Triton kernel generation on AMD MI300X
- **Kevin** (Baronio et al., 2025): Employs multi-turn RL for CUDA kernel generation

While these systems demonstrate that AI agents can generate competitive implementations on isolated benchmarks, the paper identifies **six fundamental limitations preventing production deployment**:

1. **Narrow optimization scope:** Systems target isolated subproblems without addressing end-to-end kernel lifecycle management from synthesis to deployment.

2. **Synthetic evaluation:** Benchmarks use canonical operators with static tensor shapes rather than production workloads exhibiting dynamic batching, variable sequence lengths, and domain-specific transformations.

3. **Single-platform focus:** Most target homogeneous NVIDIA environments without cross-platform synthesis for heterogeneous accelerator fleets.

4. **Limited agent capabilities:** Existing systems lack fully autonomous workflows encompassing automated synthesis, multi-level correctness verification, hierarchical profiling feedback, and persistent knowledge bases.

5. **Absence of inference-time scaling:** No system employs large-scale search strategies (greedy, Monte Carlo Tree Search, evolutionary algorithms) that iterate hundreds to thousands of optimization steps per kernel.

6. **No checkpointing support:** Systems restart from scratch on failure rather than resuming from intermediate states, making multi-hour optimization runs brittle.

Figure 3 illustrates the programming model evolution at Meta: Triton has grown to over 8,000 kernels with 60% expansion rate, surpassing CUDA's stagnant legacy codebase, while emerging DSLs (CuTe, TLX, Helion) remain under 600. This shift toward higher-level DSLs creates programming model fragmentation across 5+ languages, motivating KernelEvolve's automated synthesis approach.

## 3. Technical Approach

### 3.1 Reader orientation (approachable technical breakdown)

KernelEvolve is an autonomous AI agent system that generates, tests, and optimizes low-level compute kernels for AI accelerators by iteratively searching through possible implementations using feedback from actual hardware execution. The system solves the combinatorial explosion of kernel optimization across heterogeneous hardware by treating kernel generation as a graph-based search problem where each iteration learns from compilation results, correctness validation, and performance profiling to guide subsequent optimization attempts.

### 3.2 Big-picture architecture (diagram in words)

The system comprises five major components connected in a closed-loop optimization pipeline. First, the **LLM Synthesizer** receives a kernel specification and generates initial Triton kernel candidates using dynamic prompts augmented with retrieved context. Second, the **Tree Search and State Machine** maintains a search graph where each node represents a kernel implementation, tracking parent-child relationships, fitness scores, and correctness flags to guide exploration strategy. Third, the **Agentic Retrieval System** (consisting of the Deep Search Sub-Agent and Context Memory Sub-Agent) dynamically loads relevant hardware constraints, optimization patterns, and historical execution data from a persistent knowledge base into the LLM's context window. Fourth, the **Evaluation and Tooling Framework** executes generated kernels on target hardware interpreters (NVIDIA, AMD, MTIA) using multiple profiling tools (TritonBench, Torch Profiler, NCU, Proton, MTIA Insight) to produce correctness validation and performance metrics. Fifth, the **Persistent Storage Layer** (metadata store and object store) maintains complete exploration history enabling checkpointing, cross-session knowledge reuse, and distributed concurrent exploration across multiple agents. Information flows cyclically: the LLM generates a kernel candidate, the evaluation framework executes it on hardware interpreters, profiling results update the search graph state, the context memory sub-agent analyzes artifacts to diagnose bottlenecks, the deep search sub-agent retrieves targeted knowledge base content, and augmented prompts guide the next iteration.

### 3.4 Detailed, sentence-based technical breakdown

This is a systems paper presenting an agentic framework for automated kernel generation, and the core idea is to formulate kernel optimization as a graph-based search algorithm augmented with retrieval-augmented prompt synthesis that dynamically adapts to runtime execution context.

**Graph-Based Search Framework Formalization.**

KernelEvolve models the optimization process as a graph-based search algorithm specified by the tuple $(F, \pi_{sel}, O, \tau)$, where each component serves a distinct function in the exploration process. The **fitness function** $F: S \rightarrow \mathbb{R}_{\geq 0}$ estimates the quality of a kernel implementation node $v \in V_t$ through the speedup achieved by the generated Triton kernel relative to the PyTorch compiled reference code. Specifically, for a generated Triton kernel with execution time $t_{triton}$ and PyTorch compiled baseline with execution time $t_{pytorch}$, the fitness is computed as:

$$F(v) = \frac{t_{pytorch}}{t_{triton}}$$

Kernels that fail correctness validation (numerical accuracy checks against reference code) or encounter compilation/runtime errors are assigned $F(v) = 0$, ensuring correctness serves as a hard constraint before performance optimization. The **selection policy** $\pi_{sel}: 2^{V_t} \rightarrow 2^{V_t}$ chooses a subset of nodes $U_t \subseteq V_t$ for expansion, guided by a heuristic function $h: V_t \rightarrow \mathbb{R}$ that assigns scalar estimates to each node. Different instantiations support various search strategies: greedy search selects the single highest-scoring node, Monte Carlo Tree Search (MCTS) balances exploration-exploitation via upper confidence bounds for trees (UCT), and evolutionary algorithms maintain populations of diverse candidates for crossover and mutation operations. The **universal operator** $O: S \times C \rightarrow S$ is a single transformation function that generates new kernel candidates from existing implementations, where $C$ represents the contextual information (profiling results, error messages, hardware constraints, historical optimizations) that guides the transformation. Unlike traditional approaches that employ multiple specialized operators (Draft, Debug, Improve) with fixed prompting strategies, KernelEvolve's universal operator dynamically adapts its behavior based on runtime context through retrieval-augmented prompt synthesis. The **termination rule** $\tau$ halts search when computational budgets (wall-clock time or maximum number of artifacts) are exhausted, progress stalls, or fitness thresholds are achieved.

The search operates in two distinct phases as illustrated in Figure 10. In the **draft phase** (steps 0–10), KernelEvolve generates candidate kernels through independent sampling without feedback, establishing initial exploration diversity. In the **tree expansion phase** (steps 10–50), each node incorporates execution feedback—profiling data, compilation status, and correctness results—from its ancestors, enabling iterative refinement. Figure 10 shows fitness score trajectories for six representative ATen operators over 50 search steps, where four of six operators achieve fitness scores exceeding 1.0×, confirming that KernelEvolve-generated kernels can outperform compiler-generated baselines. For example, `torch.cos` improves from 2.8× to 3.05× during tree expansion, indicating that feedback-guided search discovers superior implementations, while `torch.ops.aten.add.Tensor` shows early-stage improvement from 0.64× to 0.70×, demonstrating that iterative refinement benefits even initially suboptimal kernels.

**Universal Operator and Dynamic Prompting Design.**

The paper identifies a fundamental limitation in prior multi-operator frameworks: each operator (e.g., Debug, Improve) is associated with a static prompt template that cannot adapt to runtime execution context. For instance, Debug operators employ fixed error-focused prompts regardless of whether the underlying issue stems from algorithmic errors, memory access patterns, or hardware-specific constraints, while Improve operators use performance-focused prompts that remain unchanged despite varying bottleneck characteristics (compute-bound vs. memory-bound vs. synchronization overhead). This static prompting strategy imposes cognitive constraints on the model's reasoning process, potentially misleading the model by framing optimization problems through predefined lenses that may not align with the actual runtime context.

To address this limitation, KernelEvolve employs a single universal operator that dynamically adapts its behavior based on runtime context rather than predefined operator categories, as illustrated in Figure 6. Instead of fixed prompt templates associated with specific operators, the system introduces a retrieval-augmented dynamic prompting mechanism that synthesizes contextually optimal prompts at each iteration. Figure 6 shows the universal operator's agentic kernel generation workflow for a Swish activation kernel, where the agent iteratively reads, analyzes, modifies, and validates code through tool invocations (`read_file`, `write_to_file`, `replace_in_file`, `lint`), guided by LLM reasoning at each step. The workflow completes after 20 steps, producing a lint-free optimized Triton kernel with autotune configurations. This design choice allows the LLM to reason holistically about kernel optimization without artificial constraints imposed by operator boundaries. By unifying operator functionality under a single context-aware interface, KernelEvolve enables more flexible exploration strategies where each generation step can simultaneously address multiple optimization objectives—correcting numerical errors, improving memory access patterns, exploiting hardware-specific features, and refining algorithmic approaches.

**Persistent Knowledge Base and Hierarchical Retrieval.**

KernelEvolve employs a retrieval-augmented approach where contextual information is dynamically loaded into the LLM's context at runtime rather than maintaining complete historical knowledge in working memory. This design philosophy aligns with recent agentic coding systems such as Anthropic's Claude Code, which demonstrates that complex analysis over large datasets can be performed through targeted queries and incremental loading rather than exhaustive context consumption. The persistent knowledge base implements a hierarchical file system encoding domain expertise across hardware platforms, optimization strategies, debugging patterns, and language constraints, as shown in Figure 5.

The knowledge base partitions content across three primary categories addressing distinct optimization concerns. The **constraints/** directory enforces correctness requirements through anti-cheating rules (prohibiting cross-platform abstractions, external library dependencies), forbidden patterns (direct CUDA API usage, incomplete test coverage), and output format specifications. These constraints ensure generated kernels represent authentic Triton implementations rather than wrappers around pre-compiled libraries or high-level PyTorch operations that bypass kernel-level optimization. The **guidance/** directory provides platform-agnostic optimization knowledge organized by concern: debugging methodologies (error interpretation, numerical stability), performance tuning (autotuning, block sizing, fusion strategies), and Triton language idioms (data types, indexing patterns, memory primitives). This cross-platform content transfers across hardware architectures, establishing foundational implementation patterns before platform-specific specialization. The **hardware/** directory encodes platform-specific knowledge for NVIDIA GPUs, AMD GPUs, and MTIA accelerators. Each platform subtree maintains architectural documentation (compute hierarchies, memory subsystems, execution models), platform-specific debugging guidance (common pitfalls, precision issues), and advanced optimization techniques. NVIDIA content includes Hopper-generation features (Tensor Memory Accelerator, warp specialization via Triton Low-level Extensions (TLX)), persistent kernel patterns, and generation-specific optimizations. Hardware modules comprise 15-40 documents per platform—totaling ≥100 documents—reflecting the depth of architectural specialization required for production-grade kernel optimization.

A comprehensive index (`index.md`, Figure 5) maintains the complete directory tree with inline module documentation, serving dual purposes: human-readable reference and machine-parseable navigation structure for automated retrieval. Upon receiving runtime feedback (profiling metrics, error diagnostics), the deep search sub-agent executes two-stage retrieval: first, it queries the index to identify relevant modules based on hardware platform, bottleneck type, and optimization phase; second, it fetches targeted content from identified modules. For example, memory bandwidth bottlenecks on NVIDIA H100 trigger index queries returning `hardware/nvidia/optimization/{tma, shared_memory, on_device_tma}.md`. The hierarchical structure enables efficient pruning: top-level platform selection (`hardware/{nvidia|amd|mtia}/`) eliminates irrelevant architectures; folder hierarchies encode concern taxonomies (`arch/`, `debug/`, `optimization/`); file naming conventions signal specificity (`memory_hierarchy.md` versus `on_device_tma.md`).

Content organization supports iterative refinement throughout optimization trajectories. Initial generation retrieves broad guidance—Triton basics, general optimization principles, correctness requirements. Subsequent iterations navigate progressively specialized content guided by profiling feedback. A representative trajectory for compute-intensive GEMM on H100 follows this progression: (1) `hardware/nvidia/arch/tensor_cores.md` establishing Tensor Core capabilities; (2) `hardware/nvidia/tlx/{overview, warp_specialization, async_tensor_core_operations}.md` introducing fine-grained control through producer-consumer warp patterns and asynchronous matrix operations; (3) `code_samples/{hopper-gemm-pipelined, hopper-gemm-ws}.py` providing complete reference implementations. This progression from high-level tensor core usage to expert-level TLX pipelined and warp specialization exemplifies support for both novice implementations and production-grade optimizations approaching theoretical hardware limits.

**Context Memory Sub-Agent and Two-Tier Storage Architecture.**

The context memory sub-agent bridges the persistent knowledge base and runtime optimization state through a two-tier storage architecture separating metadata from content, as shown in Figure 5. Each explored search graph node persists to a metadata store (relational database) containing: unique identifiers (`id`), parent references (`pid`) encoding tree structure, fitness scores (`score`), correctness flags (`is_buggy`), and path references (`path_ref`) linking to an object store containing kernel implementations (`kernel_n.py`), profiling results, and LLM-generated analysis reports (`overview.md`). This separation enables efficient metadata queries without loading large source files or profiling traces.

The relational storage architecture provides four critical capabilities beyond simple persistence. First, it enables **distributed concurrent exploration**: multiple agents can simultaneously expand different nodes in the search graph, with the database providing transaction isolation and consistency guarantees. When KernelEvolve scales to dozens or hundreds of concurrent agents exploring thousands of optimization steps, maintaining an in-memory graph representation becomes infeasible. The metadata store allows agents to operate independently, querying only relevant subgraphs on demand. Second, it supports **complex contextual queries** through SQL operations that logically reconstruct graph views without materializing the entire structure in memory. The relational schema enables efficient graph traversal via recursive Common Table Expressions (CTEs). These queries enable sophisticated context construction: analyzing sibling node outcomes, retrieving strategies from high-performing ancestors, identifying global best solutions for comparison. Third, it enables **cross-session knowledge reuse**: persistence enables querying historically explored similar operators (matched by operator type, input shapes, hardware platform). When high-quality solutions exist, KernelEvolve initializes search with these implementations rather than generating from scratch. Consider a new GEMM variant for transformer attention on AMD MI350: metadata queries identify 15 historical GEMM kernels, three achieving > 1.5× speedup through TLX warp specialization. KernelEvolve retrieves the highest-performing implementation (score 1.5) with its optimization report documenting successful strategies (async tensor core operations, double-buffered shared memory), then focuses exploration on problem-specific adaptations (different dimensions, fusion opportunities) rather than rediscovering fundamental patterns. This approach provides reduced time-to-solution starting from proven implementations, inference cost reduction eliminating redundant token consumption, and environmental impact reduction through decreased computational overhead. Fourth, it enables **fault tolerance and checkpointing**: persistent storage enables fault-tolerant search with automatic checkpointing. When search processes crash or are interrupted, KernelEvolve reconstructs the complete search state from metadata: loading explored nodes, their parent-child relationships, fitness scores, and associated artifacts. Tree search resumes from the last successful iteration rather than restarting from scratch. This proves critical for long-running optimization campaigns where generating production-grade kernels may require hundreds of iterations spanning hours: hardware failures, deployment updates, or resource preemption no longer discard accumulated exploration progress. The metadata store serves as continuous checkpoint, with each node insertion atomically persisting exploration state.

At each search node, KernelEvolve generates execution artifacts: kernel source, execution logs, correctness validation results, performance measurements (execution time, speedup), and profiling metrics (instruction latency, memory throughput, occupancy, synchronization overhead). The context memory sub-agent invokes the LLM to analyze these artifacts, producing structured reports diagnosing bottlenecks and recommending optimization strategies. For example, profiling revealing 30% occupancy on H100 with high shared memory pressure triggers analysis identifying register spilling and bank conflicts as root causes, recommending register usage reduction through value recomputation and warp-level memory access optimizations. The context memory sub-agent composes prompts for the universal operator combining: (1) current kernel implementation and execution history, (2) LLM-generated analysis reports, (3) retrieved knowledge base content, (4) hardware-specific constraints. This implements self-managed context windows maintaining only task-relevant information within token budgets (64K-1M tokens depending on LLM backend). When multiple optimization opportunities exist, the sub-agent prioritizes based on profiling evidence, loading content for dominant bottlenecks while deferring secondary optimizations to subsequent iterations. The context memory maintains summaries of previous optimization attempts, enabling learning from successes and failures. If increasing block size fails to improve speedup or introduces correctness violations, subsequent iterations avoid that strategy and explore alternatives. This mirrors human debugging workflows where engineers track attempted fixes, analyze failures, and systematically explore solution spaces while avoiding dead ends.

**Evaluation Framework and Hardware Interpreters.**

KernelEvolve generates kernel implementations conforming to a standardized dual-implementation interface enabling automated correctness validation and performance benchmarking. Each generated artifact comprises three components: a PyTorch baseline, an optimized Triton kernel, and input data generation, structured to support systematic evaluation and downstream compilation integration. Generated kernels provide two `nn.Module` implementations with identical signatures. The PyTorch baseline prioritizes correctness over performance, providing ground truth for numerical validation. The Triton implementation consists of a `@triton.jit` kernel, a wrapper managing grid configuration and kernel launch, and an `nn.Module` encapsulation. Input generation produces test cases spanning diverse sizes, exposing performance characteristics under varying computational and memory pressure. The `nn.Module` structure enables integration with PyTorch's `torch.compile` infrastructure, supporting hybrid optimization strategies combining compiler-driven graph transformations with hand-optimized kernels. Standardized interfaces enable automated evaluation: correctness validation executes both implementations on identical inputs, comparing outputs via `torch.allclose()` with precision-dependent tolerances; performance profiling measures execution time and computes speedup ratios.

Generated Triton kernels require execution on target hardware platforms for correctness validation and performance profiling. To enable automated evaluation across heterogeneous accelerators (NVIDIA GPUs, AMD GPUs, MTIA), KernelEvolve establishes dedicated interpreter environments providing standardized execution contexts with complete software stacks, compilation toolchains, and runtime dependencies for each hardware target. Meta's Bento platform—the standard Jupyter notebook environment—bundles external libraries (PyTorch, Triton, CUDA/ROCm) with internal frameworks including hardware-specific software stacks (MTIA runtime, vendor-specific compilers) and build systems. KernelEvolve configures three hardware-specific interpreters: `meta_kernel_gpu_interpreter` (NVIDIA), `meta_kernel_amd_interpreter` (AMD), and `meta_kernel_mtia_interpreter` (MTIA). Each interpreter encapsulates platform-specific compilation toolchains (TritonGPU/AMDGPU/MTIA-MLIR backends), runtime libraries (`mtia_triton_launcher` for MTIA, cuDNN for NVIDIA), and profiling tools (NCU, Proton).

Interpreters integrate with Meta's Conveyor continuous deployment system, which monitors dependency updates and automatically publishes new versions on regular schedules, as shown in Figure 8. When underlying components update (Triton compiler, runtime libraries, build systems), Conveyor rebuilds and deploys interpreter packages with current dependencies. This eliminates manual environment maintenance: kernel artifacts submitted to interpreters execute directly against up-to-date software stacks. Interpreter isolation ensures reproducible profiling across tree search iterations. This infrastructure abstracts hardware-specific complexities from kernel generation, enabling unified evaluation interfaces while maintaining platform-specific optimization capabilities.

KernelEvolve employs multiple profiling tools targeting different analysis objectives. TritonBench validates correctness through numerical comparison against PyTorch baselines and measures speedup ratios across production input shapes. PyTorch Profiler captures system-level execution timelines including kernel launch overhead and host-device synchronization. NCU provides kernel-level hardware metrics including occupancy, memory throughput, and instruction mix. Proton delivers intra-kernel instruction-level latency and pipeline behavior. MTIA Insight provides comprehensive MTIA-specific instrumentation: PE utilization, fixed-function accelerator metrics (DPE/SFU/MLU utilization and stall cycles), per-PE CPU runtime, cache analysis (CPU I/D-cache hit rates, branch prediction, LLC behavior), memory bandwidth (LLC/DRAM), and load-store throughput with per-PE read/write counters. KernelEvolve employs a deterministic code generator transforming LLM-generated kernel implementations into platform-specific evaluation scripts invoking these profiling tool APIs, as illustrated in Figure 9. The evaluation code generator accepts standardized kernel artifacts as input and produces executable Python scripts for each profiling tool. For TritonBench, the generator creates benchmark harnesses wrapping both `PytorchModel` and `TritonModel` within the `BenchmarkOperator` framework, configuring correctness validation (`baseline=True`) and speedup measurement. For Torch Profiler, generated scripts insert profiler contexts (`torch.profiler.profile()`) around kernel invocations capturing execution traces. For NCU and Proton (via Triton MPP), the generator synthesizes instrumentation invoking respective APIs with hardware-specific configurations. Each generated script imports models from standardized artifacts, instantiates profiler contexts, executes across test cases from `get_inputs()`, and formats results as structured data consumable by the context memory sub-agent.

Generated evaluation scripts leverage pre-deployed interpreter environments eliminating compilation overhead. Since hardware interpreters bundle complete toolchains (Triton compilers, profiling frameworks, runtime libraries) via Conveyor's continuous deployment, evaluation code executes immediately without dependency resolution or environment setup. This architectural separation—LLM-generated kernel logic versus deterministically-generated evaluation harness—provides critical benefits: compilation occurs once during interpreter deployment rather than per-kernel evaluation (reducing latency from ≥10 minutes to seconds); evaluation code remains consistent across kernel variants, ensuring reproducible profiling; profiling tool APIs update independently through interpreter redeployment without modifying kernel generation prompts.

**Unified Profiling: Triton MPP Integration.**

KernelEvolve integrates Triton MPP (Multi-Pass Profiler) from Meta addressing fundamental GPU profiling challenges. Modern profiling workflows exhibit fragmentation: practitioners orchestrate IR-level tracers, assembly profilers (NCU), and binary instrumentation (NVBit), each with incompatible interfaces and vendor-specific assumptions. Profilers target human interpretation through textual reports and dashboards rather than structured data, forcing brittle text parsing unsuitable for automation. MPP provides a compiler-centric abstraction unifying heterogeneous profiling tools. The framework composes analysis through job graphs: compiler transforms insert MLIR-level instrumentation, profiling passes collect metrics, trace synthesis produces structured output. This proves critical for modern Triton kernels employing TMA operations, warp-specialized pipelines, and overlapped data movement.

Traditional GPU profilers expose asynchronous behavior coarsely, failing to reveal instruction-level overlap between memory and computation. Direct wait insertion perturbs execution: initial synchronization disrupts overlap, cascading timing changes through subsequent operations. MPP addresses this through minimally-invasive profiling: capturing unmodified base traces, applying targeted probe passes isolating single instructions in specific iterations, guarding instrumentation to prevent interference, and fusing results for attribution. This profiles warp-group operations, async copies, and TMA transfers at TTGIR level with negligible perturbation. MPP integration provides KernelEvolve with structured, instruction-level performance data without vendor-specific parser implementations, transforming profiling from manual orchestration to programmatic composition.

**Agentic Debugging in JIT Flow.**

Beyond profiling metrics, MTIA-Triton provides compiler introspection capabilities critical for kernel debugging and optimization. MTIA-Triton supports optional C++ code emission exposing the compiler's intermediate representation before final RISC-V binary generation:

```python
compiled_kernel = kernel[grid](x, *x.shape, BLOCK_SIZE=1024, cb_multiplier=8, emit_cxx=True)
cpp_source = compiled_kernel.asm["cpp"]
```

The emitted C++ code reveals low-level MTIA operations including RISC-V vector intrinsics, circular buffer pointer management, SFU initialization, and core affinity queries. When generated kernels crash, fail correctness validation, or exhibit unexpected performance, the context memory sub-agent retrieves emitted C++ code alongside error diagnostics. The universal operator analyzes this representation—identifying incorrect buffer management, missing synchronization, or suboptimal SFU usage—and generates corrected C++ implementations. Modified C++ kernels can be tested immediately without full recompilation through MTIA's replay mechanism:

```python
from triton_mtia.python.mtia.eager.debug.replay_cpp import replay_cpp
replay_cpp(modified_cpp_source, compiled_kernel, args=(1, 1, 1, x, *x.shape))
```

This rapid iteration cycle—emit C++, modify, replay—eliminates full Triton recompilation overhead (constexpr resolution, MLIR lowering, backend code generation), enabling agents to validate hypothesized fixes within seconds rather than minutes. Compiler introspection transforms opaque execution failures into actionable optimization opportunities grounded in hardware-specific implementation details.

**Kernel Evaluation on FaaS.**

Tree search execution decomposes each node expansion into two phases: kernel generation and kernel evaluation. Generation comprises prompt synthesis, knowledge base retrieval, and LLM invocation—CPU-bound operations requiring no accelerator access. Evaluation executes generated kernels on target hardware through TritonBench correctness validation, Torch Profiler timeline capture, and Triton MPP instruction-level analysis. This workload asymmetry—generation on CPU hosts versus evaluation on AI accelerators—motivates architectural disaggregation enabling independent, non-blocking evaluation execution on remote hardware.

KernelEvolve migrates kernel evaluation to Meta's FaaS (Function-as-a-Service) platform, which abstracts infrastructure complexity including service routing, Twine autoscaling, and lifecycle management. FaaS runtime auto-generates Thrift server interfaces for evaluation handlers, packaging them as fbpkg distributions with hardware interpreter dependencies. The system extended FaaS's Tasklet resource model from CPU/RAM to include GPU resources, enabling hardware-specific evaluation functions targeting NVIDIA, AMD, and MTIA platforms. When tree search generates a kernel candidate, it asynchronously dispatches evaluation requests to FaaS endpoints corresponding to target hardware. Remote workers load pre-deployed interpreter environments, execute evaluation harnesses, and return structured results (correctness status, speedup ratios, profiling metrics) consumed by the context memory sub-agent.

Disaggregation addresses resource contention: a single host may run hundreds of generation agents but possess limited accelerators (8 GPUs or 24 MTIA devices per host). Without separation, agents serialize through available hardware—each occupying a device for 8-12 minutes (mostly idle during generation) while other agents queue. FaaS-based evaluation provides: (1) resource decoupling—generation agents execute CPU-bound work locally while dispatching evaluation to remote accelerator pools, preventing device occupation during idle generation phases; (2) elastic capacity—evaluation distributes across FaaS worker pools with hundreds of GPUs/MTIA devices rather than serializing through local hardware. This architecture maximizes both CPU (generation) and accelerator (evaluation) utilization, eliminating the mismatch between abundant generation parallelism and scarce local hardware resources.

**MTIA Knowledge Injection for Proprietary Accelerators.**

MTIA presents unique challenges for LLM-based kernel generation: unlike widely-documented GPU architectures (NVIDIA CUDA, AMD ROCm), MTIA's proprietary architecture and programming model remain largely absent from public training corpora. Pretrained LLMs lack knowledge of MTIA-specific hardware features, extended Triton language constructs, and optimization patterns. KernelEvolve addresses this knowledge gap through systematic injection of MTIA domain expertise into the persistent knowledge base, enabling the deep search sub-agent to retrieve MTIA-specific content that educates the LLM during kernel generation.

While Triton originated as a GPU-focused language, Triton-MTIA extends the base language to expose hardware-specific features fundamentally different from GPU programming models. The knowledge base documents three categories of extensions addressing distinct requirements. **Hardware Feature Exposure:** MTIA v2i architectures expose unique capabilities including Specialized Function Units (SFU) with lookup table (LUT) operations, inter-Processing Element (PE) communication primitives, and dual-core synchronization mechanisms. The knowledge base provides detailed documentation of libdevice APIs and hardware-specific compiler directives. Libdevice APIs map Triton operations to hardware primitives: `tl.extra.libdevice.gelu(x)` compiles to SFU LUT queries rather than mathematical approximations, providing higher performance at potential accuracy cost. Documented operations include `exp`, `gelu`, `log`, `sigmoid`, `tanh`—each mapping to dedicated SFU instructions unavailable on GPU targets. MTIA exposes hardware-specific performance tuning through compilation options during kernel invocation. Two critical options documented in the knowledge base enable pipeline parallelism optimization: (1) `cb_multiplier` (integer) increases Circular Buffer allocation by specified factors, allowing multiple operations to execute concurrently by expanding on-chip memory capacity; (2) `use_dual_core` (boolean) instructs the compiler to distribute operations between core A and core B—executing DMAs on core A while core B performs vector instructions—improving throughput through heterogeneous execution. These options can be explored statically via `@triton.autotune` decorators that evaluate configuration combinations (e.g., `BLOCK_SIZE ∈ {32, 1024}`, `cb_multiplier ∈ {1, 8}`), with autotuning keyed to input dimensions (`key=["N"]`) to rerun when problem characteristics change.

**Compute Helper Functions:** MTIA provides three categories of optimized compute helpers documented in the knowledge base: (a) `unary_elemwise_compute(op, x)` supporting 30+ operations including mathematical functions (`exp`, `log`, `sqrt`), activations (`relu`, `gelu`, `sigmoid`), and logical operations; (b) `binary_elemwise_compute(op, x, y)` for tensor-tensor operations including arithmetic, comparisons, and ML-specific functions (`gelu_backward_tanh`, `log_sigmoid_backward`); (c) `binary_elemwise_const_compute(op, x, const)` for tensor-scalar operations. These helpers compile to optimized vector instructions, providing higher performance than manual Triton implementations expressing equivalent semantics.

**Custom Type System:** MTIA kernels operate on device-specific data structures including `TensorView` (tensor metadata with shape, stride, addressing information), `CoreID` (PE identification and chip topology), and `ExecutionGrid` (kernel launch configuration). The knowledge base documents type definitions via `@core.struct_type` decorators and associated utility functions, enabling kernels to directly manipulate MTIA runtime structures rather than relying on compiler-managed abstractions.

**Advanced Synchronization and Communication Primitives:** MTIA's multi-PE architecture requires explicit control over inter-PE data movement and synchronization—capabilities absent from standard Triton. The knowledge base documents four critical extensions. **Cross-PE Broadcasting** (`direction` attribute in `tl.load`): enables streaming memory between neighboring PEs, allowing multiple PEs reading identical memory to receive data from immediate neighbors rather than independent memory accesses. The `direction` parameter ("down" or "right") specifies propagation direction through the PE grid. Complementary `tl.consume()` operator reads and discards memory, maintaining functional correctness by ensuring all participating PEs execute identical load sequences. **Cross-PE Reduction** (`direction` attribute in `tl.store`): extends `tl.store` to send computed results directly to neighboring PEs, enabling collaborative computation patterns where multiple program instances cooperate to produce results. This mechanism supports efficient row-wise and column-wise reductions across PE arrays. **Runtime Barriers** (`tl.pe_runtime_barrier()`): introduces runtime synchronization across all PEs, enabling cross-PE reduction mechanisms and eliminating kernel splitting for explicit synchronization. This primitive maps directly to `libjit_fba_runtime_barrier()`, requiring careful placement to execute exactly once per PE in the physical grid. **Explicit Tensor Copies** (`tl.copy()`): creates deep tensor copies facilitating producer-consumer synchronization between MTIA cores. The compiler automatically detects data race conditions and fails compilation if copy operations prove insufficient for correctness.

The MTIA hardware subtree (`hardware/mtia/`) in the persistent knowledge base contains multiple documents spanning architectural overviews, language extensions, optimization patterns, and complete code examples. When the deep search sub-agent receives queries targeting MTIA hardware, it retrieves relevant documentation—libdevice API references for activation functions, cross-PE communication patterns for multi-PE kernels, custom type definitions for runtime structure manipulation. This retrieved content enters the LLM's context window during prompt synthesis, effectively teaching the model MTIA-specific programming idioms absent from pretraining data. The context memory sub-agent further refines retrieval based on runtime feedback: compilation errors citing undefined MTIA primitives trigger retrieval of language extension documentation; profiling results indicating suboptimal SFU utilization trigger retrieval of libdevice mapping tables. This knowledge injection strategy proves critical for MTIA kernel generation quality. Without MTIA-specific documentation in context, LLMs generate standard Triton code targeting GPU semantics, producing compilation failures or functionally incorrect kernels when executed on MTIA hardware. With systematic knowledge base retrieval, KernelEvolve successfully generates production-grade MTIA kernels leveraging hardware-specific features—SFU operations, inter-PE communication, dual-core synchronization—approaching hand-optimized performance while maintaining the productivity benefits of automated generation. This approach generalizes to emerging accelerator architectures: as new hardware platforms enter production, corresponding documentation injected into the knowledge base enables immediate LLM-based kernel generation without model retraining.

### 4.1 Universal Operator: From Fixed Templates to Context-Aware Synthesis

**What Prior Systems Do.** Existing LLM-based kernel generation systems employ multiple specialized operators with static prompt templates—separate operators for drafting, debugging, and improving, each with predetermined prompting strategies (Section 2.3). **Why This Is Fundamentally Different.** This is not merely an architectural simplification—it addresses a core limitation in how LLMs reason about optimization problems. Static operator templates impose artificial boundaries on the solution space by framing problems through predefined lenses that may not align with actual runtime characteristics. A "Debug" operator with fixed error-focused prompts cannot adapt when the underlying issue stems from hardware-specific constraints rather than algorithmic errors. The universal operator enables holistic reasoning where each generation step can address multiple optimization objectives simultaneously.

**Incremental vs. Fundamental.** The graph-based search framework itself builds on established patterns (MCTS, evolutionary algorithms). The universal operator design is the fundamental innovation—it changes what information the LLM can reason about at each step, enabling adaptation to runtime execution characteristics rather than predefined operator semantics.

### 5.1 Evaluation Methodology

**Datasets and Workloads.** The evaluation spans three distinct workload categories, each serving a different validation purpose:

- **Production Recommendation Workloads:** Real operators extracted from deployed Meta models including WuKong, InterFormer, and Convolutional Transformer architectures. **Metrics.** The paper employs multiple evaluation metrics depending on the validation objective:

**Baselines.** Two baseline categories are used depending on the operator type:

**Experimental Setup.** Generated kernels execute on dedicated interpreter environments (`meta_kernel_gpu_interpreter`, `meta_kernel_amd_interpreter`, `meta_kernel_mtia_interpreter`) deployed via Meta's Conveyor continuous deployment system (Figure 8). Evaluation harnesses are deterministically generated to invoke multiple profiling tools (TritonBench, Torch Profiler, NCU, Proton, MTIA Insight) with consistent configurations across kernel variants. Tree search operates in two phases: draft phase (steps 0–10) with independent sampling, and tree expansion phase (steps 10–50) incorporating execution feedback from ancestors.

This architecture ensures automated synthesis delivers performance improvements where possible without risking regressions.

#### 5.4.3 WuKong Optimized FM (Section 5.3.1)

**Setup.** Optimized FM is a core computational primitive in Meta's Wukong recommendation model. Computing pairwise dot product $XX^\top$ has $O(N^2D)$ complexity; Wukong exploits low-rank property by introducing learnable projection matrix $Y \in \mathbb{R}^{N \times K}$ where $K \ll N$, reducing output dimensionality. The computation reorders to:

$$\text{out} = X \cdot (X^\top Y)$$

where $X \in \mathbb{R}^{B \times N \times D}$ and $Y \in \mathbb{R}^{B \times N \times K}$. Computing $X^\top Y$ first reduces complexity from $O(N^2D)$ to $O(NKD)$.

**Optimization Strategy.** KernelEvolve generates a fused Triton kernel exploiting two key optimizations:

1. **Operator Fusion:** PyTorch baseline executes two independent matrix multiplications with intermediate HBM round-trip. KernelEvolve fuses both operations into single kernel: inputs loaded once, intermediate result $X^\top Y$ remains in SRAM, only final output writes to HBM. This reduces memory traffic by approximately 2×.

2. **Shape-Specific Tiling:** Custom tiling configurations tailored to production shapes ensure complete computation chain fits within SRAM capacity.

**Results (Figure 14).** Speedup varies by feature count $N$ and batch size $B$:

- **Small N (24 features):** 3.6-3.9× speedup for batch sizes 128-2048, maintaining stable performance as batch size scales.
- **Medium N (40, 48 features):** 2.1-3.0× at small batch sizes, gradually decreasing to 2.2-2.3× as $B$ increases—reflecting trade-off between fusion benefits and tiling overhead.
- **Output dimension K variation:** Small N values (24-32) maintain 3.0-3.5× speedup across entire K range (256-2304). Medium N (40-64) achieve 2.0-2.5×. Larger N (96-256) show diminishing returns, approaching 1× as N increases.

**Degradation Explanation.** For larger feature counts, more tiles are required to fit in SRAM. As the number of tiles grows, overhead of tile management and accumulation eventually surpasses benefits of on-chip computation, making direct HBM execution competitive.

**Deployment Strategy.** Generated kernel demonstrates consistent 2-4× speedups on production shapes where $N \leq 64$, covering majority of deployed Wukong variants. For configurations with larger feature counts where tiling overhead dominates, system falls back to PyTorch's unfused baseline. This shape-specific dispatch ensures performance gains on target workloads without risking regressions on out-of-distribution inputs.

**Critical Assessment.** This case study demonstrates KernelEvolve's ability to discover fusion strategies and tiling configurations tailored to production distributions. The shape-dependent performance is not a limitation but an expected characteristic of specialized optimization—the system explicitly handles this through fallback mechanisms. The results validate that automated synthesis can achieve competitive performance with expert manual implementations while reducing development time from weeks to hours.

#### 5.4.4 InterFormer PFFN (Section 5.3.2)

**Setup.** Personalized FeedForward Network (PFFN) comprises five operations executed sequentially: feed-forward neural network, GELU activation, RMSNorm, another feed-forward layer, and final RMSNorm. PyTorch baseline with `torch.compile` generates two separate kernels with three total memory round-trips.

**Optimization Strategy.** KernelEvolve generates two kernel variants targeting different operator chains, selecting highest-performing variant for production. Key optimizations:

1. **Shape-Specific Tiling:** Customized tiling configurations maximize SRAM utilization for target dimensions ($D \in [96, 256]$, $K \in [96, 256]$).
2. **Cross-Operation Tile Reuse:** Unified single-pass kernel loads tiles once, performs complete operator chain while data resides in SRAM, writes final results to HBM—requiring only one load and one write per tile.

**Results (Figure 15).** Speedup varies by batch size $B$ and input dimension $D$:

- **Batch size variation:** Peak speedups of 2.0-2.6× occur at small batch sizes ($B \leq 256$), where fused kernel's reduced memory traffic dominates. As batch size increases beyond 512, speedup stabilizes at 1.2-1.4× across all configurations—larger batches amortize kernel launch overhead for both optimized and baseline implementations.
- **Input dimension variation:** Speedup peaks at 1.6-1.9× for small $D$ ($D \leq 100$), drops to local minimum of 1.1-1.2× around $D = 200$, then recovers to 1.2-1.4× for larger dimensions ($D > 200$).

**Non-Monotonic Behavior Explanation.** This pattern arises from interplay between tile size and SRAM capacity. At small $D$, tiles fit comfortably in SRAM enabling effective fusion. At intermediate $D \approx 200$, tile dimensions approach SRAM limits causing partial spilling that negates fusion benefits. At large $D > 200$, kernel adapts tiling strategy to maintain SRAM residency, recovering performance.

**Critical Assessment.** All configurations maintain speedup $\geq 1.0$ across tested parameter space, with majority achieving 1.2-2.0× improvements. The absence of performance regressions validates KernelEvolve's shape-aware optimization approach. The non-monotonic behavior is insightful—it reveals that fusion benefits are not universal but depend on hardware resource constraints. The system's ability to adapt tiling strategies through search rather than fixed heuristics is a key advantage.

#### 5.4.5 Data Preprocessing on MTIA (Sections 5.4.1-5.4.3)

**Setup.** MTIA presents unique kernel development challenges: not all PyTorch ATen operators are natively supported across hardware variants. Table 5 summarizes unsupported operators for MapIdTransform and MergeBucketizedDenseTransform (MBDT) across MTIA generations:

| Hardware | Operator | Missing ATen Ops |
|----------|----------|------------------|
| MTIA v2i | MapId | `clamp.out`, `gather.out`, `sort.values_stable`, `all.all_out`, `_unique2` |
| MTIA v2i | MBDT | `all.all_out`, `unique_consecutive` |
| MTIA v3 | MapId | `clamp.out`, `sort.values_stable`, `_unique2` |
| MTIA v3 | MBDT | `unique_consecutive` |

**MapIdTransform Results (Table 7).** KernelEvolve synthesizes fused Triton kernel consolidating four PyTorch operators into single accelerator invocation:

| Hardware | Batch Size | PyTorch (ms) | Triton (ms) | Speedup |
|----------|------------|--------------|-------------|---------|
| MTIA v2i | 10000 | 1.623-1.688 | 0.466-0.523 | 3.28-3.48× |
| MTIA v2i | 50000 | 8.090 | 1.989 | 4.07× |
| MTIA v3 | 10000 | 0.063 | 0.048 | 1.31× |
| MTIA v3 | 8000 | 0.053 | 0.039 | 1.36× |

**Key Observations:**

1. **MTIA v2i shows larger speedups:** 3.28-4.07× because PyTorch executes CPU fallbacks for unsupported operators with expensive host-device synchronization. KernelEvolve provides both functional enablement and performance optimization.

2. **MTIA v3 shows modest speedups:** 1.05-1.36× because native operator coverage is improved, resulting in stronger PyTorch baselines with less room for optimization.

3. **Performance scales with batch size:** On v2i, speedup increases from 0.78× at batch 2000 to 4.07× at batch 50000, reflecting trade-off between fixed kernel launch overhead and batch-dependent computation benefits.

4. **Regression at extreme scales:** On v3, batch size 50000 shows 0.80× (regression), demonstrating that optimization is shape-dependent.

**MBDT Results (Figure 17).** Configuration format is Batch × Features × Borders:

| Hardware | Configuration | Speedup |
|----------|---------------|---------|
| MTIA v2i | 64 × 2 × 2 | 3.19× |
| MTIA v2i | 2048 × 2 × 4 | 9.25× |
| MTIA v3 | 64 × 2 × 2 | 2.31× |
| MTIA v3 | 2048 × 2 × 4 | 3.07× |

**Optimization Mechanisms:**

- **Kernel Fusion:** Entire bucketization pipeline (border lookup, binary search, offset computation) executes in single kernel launch.
- **Vectorized Counting:** SIMD-vectorized counting applied to blocks of 64-256 elements simultaneously. For typical small border arrays (3-10 elements), this $O(n)$ approach outperforms $O(\log n)$ binary search due to reduced control flow overhead.
- **Register-Resident Computation:** Intermediate results remain in registers throughout computation with no intermediate tensor allocations.

**Critical Assessment.** These results demonstrate KernelEvolve's dual value proposition on emerging hardware platforms: **enablement and optimization**. On MTIA v2i with limited native operator coverage, KernelEvolve-generated kernels provide the only viable on-device execution path—without them, PyTorch falls back to CPU incurring order-of-magnitude latency penalties. On MTIA v3 with more complete coverage, KernelEvolve still delivers 2-3× speedups through kernel fusion and hardware-specific tuning. This highlights a key insight: as new accelerators emerge, the gap between hardware availability and software ecosystem maturity creates critical need for automated kernel generation.

#### 5.4.6 Batch Event Truncate (Section 5.5)

**Setup.** Event-based features encode user behavior sequences in ads ranking models as nested jagged tensors. Each batch contains three tensors: `outer_lengths` (events per user), `inner_lengths` (attributes per event), and `values` (flattened attribute data). The operation requires coordinated index arithmetic across three nested levels and multiple features—a pattern poorly suited to standard tensor primitives.

**Results (Table 8).** KernelEvolve generates batched Triton kernel processing multiple features in parallel:

| Description | Feature Counts | Events | Max Length | PyTorch (ms) | Triton (ms) | Speedup |
|-------------|---------------|--------|------------|--------------|-------------|---------|
| Single feature | 1 | 200 | 100 | 0.148 | 0.313 | 1.0× |
| Single feature | 1 | 200 | 200 | 0.148 | 0.109 | 1.4× |
| Prod multi-feature | 5 | 200 | 100 | 0.788 | 0.571 | 1.4× |
| Prod multi-feature | 9 | 200 | 200 | 1.443 | 0.148 | 9.8× |
| Large feature count | 32 | 200 | 100 | 5.078 | 2.548 | 2.0× |
| Large feature count | 32 | 200 | 200 | 5.085 | 0.350 | 14.5× |

**Performance Drivers:**

1. **No truncation needed (Max N ≥ actual event count):** PyTorch loops through each batch element comparing lengths individually, while batched kernel performs single vectorized comparison—yielding 9.8× and 14.5× speedups at higher feature counts.

2. **Truncation required:** Batched kernel uses constant kernel launches for parallel processing versus PyTorch's sequential iteration, achieving 1.4-2.0× speedups.

3. **Production end-to-end:** 2× speedup over PyTorch implementation. Speedup scales with batch size, indicating batched kernel enables further model scaling by supporting additional event-based features without proportional latency increase.

**Critical Assessment.** This demonstrates KernelEvolve can generate efficient batched implementations for operators where only sequential baselines exist. The dramatic speedups at higher feature counts (9.8×, 14.5×) reflect that the optimization addresses a fundamental algorithmic limitation in the baseline—sequential processing versus parallel batched execution. This is not incremental tuning but discovering a superior algorithmic approach that manual engineering would require significant effort to implement.

### 5.5 Assessment of Experimental Evidence

**Strengths.** The experimental evaluation convincingly supports the paper's claims through several mechanisms:

1. **Multi-tier validation:** Correctness is validated on 480 operator-platform configurations (100% pass rate), performance is validated on canonical benchmarks (KernelBench 100% pass), and production relevance is validated through detailed case studies on deployed models.

2. **Cross-platform evidence:** Results span three hardware vendors (NVIDIA, AMD, Meta) demonstrating that the approach generalizes beyond homogeneous environments.

3. **Mechanism transparency:** Profiling traces (Figure 11), search trajectories (Figure 10, 12), and kernel breakdowns (Table 4) reveal not just that improvements occur but why they occur.

4. **Honest reporting of limitations:** The paper explicitly acknowledges shape-dependent performance, out-of-distribution regressions, and operators with limited optimization headroom. This transparency strengthens credibility.

5. **Production deployment evidence:** Figure 8 shows continuous deployment pipeline with automated recovery, demonstrating operational robustness beyond benchmark results.

**Limitations and Conditions.** Several important caveats qualify the results:

1. **Shape specialization:** Generated kernels are deliberately specialized for production shapes. On out-of-distribution shapes, performance can degrade below baselines (0.49-0.63× for conv1d on large shapes). This is appropriate for production but limits generalization.

2. **Operator-dependent headroom:** Simple ATen operators show limited optimization potential (torch.amax, torch.div near 1.0×). Substantial speedups emerge primarily in operators with fusion opportunities or missing hardware support.

3. **Hardware maturity matters:** Speedups are largest on MTIA (up to 17×) where vendor library coverage is nascent, modest on NVIDIA (1.2-2.3×) where cuDNN is mature. This is expected but important context.

4. **No ablation studies:** The paper does not isolate the contribution of individual components (e.g., how much does retrieval-augmented prompting contribute versus tree search?). This makes it difficult to attribute performance gains to specific design choices.

5. **Search cost not quantified:** While development time is reduced from weeks to hours, the computational cost of tree search (LLM invocations, hardware evaluation time) is not reported. This is relevant for sustainability and cost-benefit analysis.

**Overall Assessment.** The experiments convincingly demonstrate that KernelEvolve achieves its stated objectives: 100% correctness across heterogeneous platforms, substantial speedups on production workloads, and reduced development time. The evidence is strongest for the claim that automated synthesis can match or exceed expert manual implementations for domain-specific operators. The claim that KernelEvolve enables monolithic deployment (eliminating 10-20ms disaggregation overhead) is supported by the MTIA preprocessing kernel results showing that missing operators force CPU fallbacks.

The most compelling evidence is the detailed case studies revealing optimization mechanisms—kernel fusion reducing memory round-trips, shape-specific tiling maximizing SRAM utilization, hardware-specific features (SFU, inter-PE communication) leveraged through knowledge injection. These are not black-box improvements but explainable optimizations grounded in hardware architecture.

However, readers should understand that KernelEvolve is not a universal optimizer—performance gains are conditional on operator complexity, fusion opportunities, hardware maturity, and input shape distributions. The system's value is highest for (1) preprocessing operators with irregular access patterns, (2) proprietary accelerators with limited vendor library support, and (3) production workloads with stable input distributions enabling shape specialization. For simple primitives on mature platforms with arbitrary shapes, gains are modest or non-existent.

This nuanced understanding is critical for practitioners evaluating whether KernelEvolve applies to their use cases. The paper succeeds in demonstrating production-grade effectiveness while honestly reporting conditions and trade-offs.

## 6. Limitations and Trade-offs

KernelEvolve demonstrates compelling production-grade results, but understanding its limitations is critical for practitioners evaluating whether this approach applies to their use cases. This section examines the assumptions underlying the system, scenarios where performance gains diminish or regress, computational and scalability constraints, and open questions that remain.

### 6.1 Shape Specialization: Production Optimization vs. General-Purpose Kernels

**The Trade-off.** KernelEvolve deliberately optimizes for production input distributions rather than arbitrary shapes. This is an explicit design choice with measurable consequences.

**Evidence from the Paper.** Table 3 presents conv1d performance across different tensor shapes. On production shapes ($2048 \times 96 \times 96 \times 200$), the generated kernel achieves 2.30× speedup over PyTorch conv1d and 1.62× over the optimized conv2d baseline in FP16. However, **on out-of-distribution shapes** (e.g., $64 \times 768 \times 768 \times 1024$), performance degrades to 0.49-0.63× of baseline—meaning the generated kernel is nearly 2× slower than PyTorch.

The paper states explicitly:
> "The generated kernel is deliberately specialized: on out-of-distribution shapes, it underperforms baselines, confirming that optimization targets production distributions rather than arbitrary inputs."

**Why This Matters.** This specialization is appropriate for Meta's production environment where input distributions are stable and well-characterized. However, it limits applicability to:
- Research settings where tensor shapes vary widely
- Multi-tenant serving platforms with unpredictable workloads
- Early-stage model development where architectures are still evolving

**Mitigation Strategy.** The paper describes shape-aware dispatch: "for each platform, the system generates and validates platform-specific kernels during an offline optimization phase, then deploys the highest-performing variant while maintaining fallback paths to vendor libraries (conv1d/conv2d) when generated kernels underperform." This adds operational complexity—maintaining both optimized and fallback paths—and requires upfront profiling to determine shape boundaries.

### 6.2 Operator-Dependent Optimization Headroom

**The Limitation.** Not all operators benefit equally from automated optimization. Simple primitives with mature compiler support show minimal improvement.

**Evidence from the Paper.** Figure 10 presents fitness score trajectories for six ATen operators over 50 search steps:

| Operator | Final Speedup | Optimization Headroom |
|----------|---------------|----------------------|
| `torch.cos` | 3.05× | High |
| `torch.ops.aten.add.Tensor` | 0.70× | Low (underperforms baseline) |
| `torch.div` | ~1.0× | Minimal |
| `torch.amax` | ~1.0× | Minimal |

The paper acknowledges:
> "Four of six operators achieve fitness scores exceeding 1.0×, confirming that KernelEvolve-generated kernels can outperform compiler-generated baselines. However, the paper explicitly acknowledges that these basic ATen operators serve primarily to validate end-to-end correctness rather than demonstrate optimization potential."

**Why This Matters.** Substantial speedups emerge primarily in operators with:
- **Fusion opportunities** (multiple operations that can be combined)
- **Missing hardware support** (operators requiring CPU fallbacks)
- **Irregular memory access patterns** (preprocessing operators with jagged tensors)

Simple element-wise operations or reductions with mature vendor library support (cuDNN, rocBLAS) offer limited headroom. Practitioners should not expect 3× speedups on every operator—the optimization potential depends heavily on operator complexity and existing compiler maturity.

### 6.3 Hardware Maturity Dependency

**The Trade-off.** Speedups correlate inversely with vendor library maturity. KernelEvolve achieves largest gains where existing support is weakest.

**Evidence from the Paper.** Figure 13 compares conv1d performance across five hardware platforms:

| Platform | vs. conv1d | vs. conv2d (optimized) |
|----------|------------|------------------------|
| MTIA v3 | 6.54× | 4.71× |
| AMD MI350 | 2.54× | 1.06× |
| NVIDIA H100 | 2.30× | 1.62× |
| NVIDIA A100 | 1.77× | 1.35× |
| AMD MI300 | 1.75× | 1.25× |

The paper notes:
> "NVIDIA GPUs show modest improvements (1.62× on H100, 1.35× on A100), reflecting cuDNN's maturity on these architectures. AMD platforms demonstrate smaller gains (1.25× on MI300, 1.06× on MI350). MTIA v3 achieves the largest speedup at 4.71× over conv2d, demonstrating that KernelEvolve's automated synthesis can effectively target custom accelerator architectures where vendor library coverage is less mature."

**Why This Matters.** This is both a strength and a limitation:
- **Strength:** KernelEvolve provides the most value where it's needed most—on emerging or proprietary hardware lacking mature vendor libraries.
- **Limitation:** On mature platforms (NVIDIA with cuDNN), the marginal benefit is smaller, potentially not justifying the computational cost of automated search.

For organizations primarily using NVIDIA GPUs with standard workloads, the business case for deploying KernelEvolve may be weaker than for organizations deploying custom accelerators like MTIA.

### 6.4 Regression Cases and Fallback Requirements

**The Limitation.** Generated kernels can underperform baselines on specific configurations, requiring runtime dispatch logic.

**Evidence from the Paper.** Table 7 shows MapIdTransform performance on MTIA v3:

| Batch Size | PyTorch (ms) | Triton (ms) | Speedup |
|------------|--------------|-------------|---------|
| 10000 | 0.063 | 0.048 | 1.31× |
| 50000 | 0.140 | 0.174 | **0.80×** (regression) |

The paper states:
> "For edge cases where regressions occur on v2i and v3, runtime dispatch based on input dimensions ensures fallback to PyTorch, preventing performance degradation in deployment."

**Why This Matters.** This reveals that KernelEvolve does not guarantee improvement across all input configurations. Production deployment requires:
1. **Comprehensive profiling** across the expected input distribution
2. **Runtime dispatch logic** to select between generated and baseline kernels
3. **Continuous validation** to detect regressions as hardware or software stacks update

This adds operational overhead that the paper acknowledges but does not quantify in terms of engineering effort or latency impact from dispatch decisions.

### 6.5 Computational Cost of Search Not Quantified

**The Gap.** While the paper claims development time reduction from "weeks to hours," it does not report the computational cost of the tree search process itself.

**Evidence from the Paper.** The paper mentions:
- Search operates over 50-300 steps per kernel (Figure 10, Figure 12)
- Each step involves LLM invocation, kernel compilation, hardware execution, and profiling
- FaaS-based evaluation distributes across "hundreds of GPUs/MTIA devices"
- Context windows range from 64K-1M tokens depending on LLM backend

However, the paper does not report:
- Total LLM token consumption per kernel
- Aggregate GPU/MTIA hours consumed during search
- Carbon footprint or monetary cost of the optimization process
- Comparison of search cost vs. manual engineering cost

**Why This Matters.** Section 6 ("Future Directions") acknowledges this gap:
> "As LLM-based generation scales, resource efficiency becomes critical. Future work will quantify token consumption per kernel, optimize prompt design for minimal inference cost, and track carbon footprint across the search process."

For organizations considering deployment, the cost-benefit analysis remains incomplete. A kernel achieving 2× speedup may not justify the search cost if it's only used infrequently. The economic case is strongest for high-volume kernels (ads ranking serving hundreds of trillions of inferences daily) but may not hold for lower-volume workloads.

### 6.6 No Ablation Studies on Component Contributions

**The Gap.** The paper does not isolate the contribution of individual system components to overall performance.

**Evidence from the Paper.** KernelEvolve integrates multiple innovations:
- Graph-based search with selection policy
- Universal operator with dynamic prompting
- Retrieval-augmented knowledge base
- Persistent metadata storage with checkpointing
- Multi-tool profiling integration
- FaaS-based evaluation disaggregation

However, the paper does not answer:
- How much does retrieval-augmented prompting contribute vs. tree search alone?
- What is the marginal benefit of the universal operator over fixed operator templates?
- How much does MTIA knowledge injection improve success rate vs. generic Triton knowledge?
- What is the impact of checkpointing on search efficiency?

**Why This Matters.** Without ablation studies, it's difficult to determine which components are essential vs. nice-to-have. Organizations with resource constraints may want to deploy a simplified version but lack guidance on which components can be omitted without significant performance degradation.

### 6.7 Proprietary Infrastructure Dependencies

**The Limitation.** Many critical components rely on Meta-internal infrastructure not available externally.

**Evidence from the Paper.** The system depends on:
- **Bento platform:** Meta's standard Jupyter notebook environment bundling hardware-specific software stacks
- **Conveyor:** Meta's continuous deployment system for interpreter environments (Figure 8)
- **FaaS platform:** Meta's Function-as-a-Service infrastructure with GPU resource extensions
- **Internal LLMs:** Meta's CWM and Llama models on Twine cluster management
- **MTIA Insight:** Proprietary MTIA-specific profiling tool
- **fbsource/Glean:** Meta's monolithic code repository and search infrastructure

**Why This Matters.** External organizations cannot directly replicate KernelEvolve's deployment architecture. While the core algorithmic contributions (graph-based search, retrieval-augmented prompting, universal operator) are portable, the production-grade reliability depends on infrastructure investments that may not be feasible for smaller organizations. This limits the paper's direct applicability as a deployment blueprint—readers must adapt the concepts to their own infrastructure constraints.

### 6.8 Knowledge Base Maintenance Burden

**The Trade-off.** The persistent knowledge base enables hardware-specific optimization without model retraining, but requires ongoing documentation effort.

**Evidence from the Paper.** Section 3.2.3 describes the MTIA knowledge injection mechanism:
> "The MTIA hardware subtree (hardware/mtia/) in the persistent knowledge base contains multiple documents spanning architectural overviews, language extensions, optimization patterns, and complete code examples."

The paper notes hardware modules comprise "15-40 documents per platform—totaling ≥100 documents."

**Why This Matters.** While this approach avoids model retraining for new hardware, it shifts the burden to documentation maintenance:
- New hardware generations require new documentation subtrees
- Compiler updates may invalidate optimization guidance
- Bug fixes in hardware runtimes may require knowledge base updates
- Documentation quality directly impacts kernel generation quality

The paper does not quantify the engineering effort required to maintain this knowledge base or describe processes for keeping documentation synchronized with hardware/software updates. Section 6 mentions future work on "hardware abstraction primitives that capture memory hierarchies, compute capabilities, and ISA constraints in a format amenable to LLM reasoning," suggesting the current document-based approach may not scale indefinitely.

### 6.9 Token Budget Constraints on Context

**The Limitation.** Context window size limits how much information can be loaded for prompt synthesis.

**Evidence from the Paper.** Section 3.2.2 states:
> "This implements self-managed context windows maintaining only task-relevant information within token budgets (64K-1M tokens depending on LLM backend)."

**Why This Matters.** For complex kernels requiring extensive hardware documentation, historical optimization reports, and profiling data, the system must prioritize which content to load. The context memory sub-agent "prioritizes based on profiling evidence, loading content for dominant bottlenecks while deferring secondary optimizations to subsequent iterations." This sequential approach may miss interactions between optimization opportunities that would be apparent with full context. As kernels grow more complex (e.g., multi-operator fusion across entire model blocks), token budget constraints may become a limiting factor.

### 6.10 Limited Scope: Operators and Small Modules, Not Full Models

**The Gap.** Current optimizations target individual operators and small modules, not end-to-end model graphs.

**Evidence from the Paper.** Section 6 ("Future Directions") explicitly acknowledges:
> "Current optimizations target individual operators and small modules, but the greatest performance gains lie at model level. Future work will extend KernelEvolve to reason about cross-layer fusion, global memory allocation, and end-to-end computation graphs."

**Why This Matters.** Operator-level optimization misses opportunities for:
- **Cross-operator fusion:** Combining operations across layer boundaries
- **Global memory planning:** Coordinating memory allocation across the entire model
- **End-to-end graph optimization:** Reordering operations to minimize data movement

While KernelEvolve demonstrates 1.25-17× speedups on individual operators, the ultimate performance ceiling for model-level optimization remains unexplored. This is an acknowledged limitation rather than a failure—the paper positions KernelEvolve as a "first step" toward broader automated optimization—but readers should understand the current scope boundaries.

### 6.11 Summary: When KernelEvolve Applies (and When It Doesn't)

Based on the limitations above, KernelEvolve is best suited for:

| Scenario | Fit | Rationale |
|----------|-----|-----------|
| Proprietary accelerators with limited vendor support | **High** | Largest speedups (4-17×) where vendor libraries are nascent |
| Production workloads with stable input distributions | **High** | Shape specialization is appropriate when distributions are known |
| Preprocessing operators with irregular access patterns | **High** | Fusion opportunities and missing hardware support create headroom |
| High-volume kernels justifying search cost | **High** | Amortizes computational cost of tree search over many inferences |
| Mature platforms (NVIDIA + cuDNN) with simple operators | **Low** | Limited optimization headroom (1.0-1.6×) |
| Research settings with variable tensor shapes | **Low** | Shape specialization causes regressions on out-of-distribution inputs |
| Low-volume kernels | **Low** | Search cost may not be justified by infrequent use |
| Organizations without infrastructure for continuous deployment | **Low** | Requires significant infrastructure investment for production reliability |

**The Core Insight.** KernelEvolve is not a universal optimizer—it is a production-grade system optimized for Meta's specific constraints: heterogeneous hardware fleets, stable production distributions, and massive inference volume justifying search cost. Understanding these boundaries is essential for practitioners evaluating whether the approach applies to their context.

The paper's honesty about these limitations strengthens its credibility. By explicitly acknowledging shape-dependent regressions, operator-dependent headroom, and infrastructure dependencies, the authors provide readers with the information needed to make informed deployment decisions rather than overpromising universal applicability.

## 7. Implications and Future Directions

KernelEvolve represents more than an incremental improvement in kernel optimization—it signals a fundamental shift in how the field approaches the hardware-software gap in AI infrastructure. This section examines how the work changes the landscape, what research directions it enables, practical deployment considerations, and guidance for practitioners evaluating whether to adopt similar approaches.

### 7.1 Landscape Shift: From Manual Kernel Engineering to AI-Assisted Compilation

**Reframing the Optimization Problem.** Prior to KernelEvolve, the field treated kernel optimization as a manual, expertise-dependent process requiring weeks of specialist effort per platform. The paper demonstrates that **LLM agents can serve as the universal compilation layer for heterogeneous AI systems** (Section 8), automatically adapting to new hardware through knowledge injection rather than manual porting. This reframes kernel development from a craft requiring deep hardware expertise to an automated service that scales with compute investment.

**The Coverage-Performance Paradigm Shift.** Perhaps the most significant conceptual contribution is recognizing that **kernel coverage—not just kernel performance—determines deployment architecture**. Section 1.1 establishes that "a single missing preprocessing operator—regardless of arithmetic intensity—blocks entire model deployments on new accelerators." This insight elevates preprocessing operators to equal or greater priority than compute-intensive kernels, fundamentally changing optimization priorities. Prior work focused on squeezing incremental gains from GEMM operations; KernelEvolve demonstrates that comprehensive operator coverage enables monolithic deployment, eliminating the 10-20ms architectural overhead from disaggregated serving (Table 2).

**Knowledge Injection Without Retraining.** The approach to proprietary hardware support represents a methodological advance. Rather than fine-tuning models on hardware-specific data (which requires exposing proprietary details and incurs retraining costs), KernelEvolve treats hardware expertise as **retrievable context** through the persistent knowledge base (Section 3.2.3). This decouples hardware knowledge from model parameters, enabling immediate support for new accelerators by populating documentation. Section 3.2.3 notes this "generalizes to emerging accelerator architectures: as new hardware platforms enter production, corresponding documentation injected into the knowledge base enables immediate LLM-based kernel generation without model retraining."

**Production-Grade Automation as a Research Target.** KernelEvolve demonstrates that research systems must address failure modes, debugging strategies, performance validation methodologies, and organizational integration patterns to achieve production deployment. Section 4.3 emphasizes that "the gap between research prototype and production deployment is not merely scale—it requires addressing failure modes... that research systems ignore." This raises the bar for what constitutes meaningful progress in automated code generation: benchmark performance alone is insufficient without operational robustness.

### 7.2 Follow-Up Research Directions

The paper explicitly outlines several research directions in Section 6 ("Future Directions"), each representing open problems that build on KernelEvolve's foundation:

**From Operators to Model-Level Optimization.** Current optimizations target individual operators and small modules, but Section 6 acknowledges "the greatest performance gains lie at model level." Follow-up research should extend KernelEvolve to reason about:
- **Cross-layer fusion:** Combining operations across layer boundaries rather than within single operators
- **Global memory allocation:** Coordinating memory placement across the entire model graph
- **End-to-end computation graphs:** Reordering operations to minimize data movement across the full inference pipeline

This extension would require representing entire model architectures as search spaces rather than individual kernel specifications—a significantly larger optimization problem that may demand new search algorithms beyond the current graph-based approach.

**Deeper Code Generation Beyond Triton.** Triton provides a productive abstraction, but Section 6 notes "certain optimizations require lower-level control." Extending KernelEvolve to modify **MLIR dialects, direct PTX/SASS, or hardware diagnostic routines** would unlock performance-critical scenarios where Triton's abstractions become limiting. This vertical integration—from high-level DSLs to bare-metal code—positions LLM agents as general-purpose compilers rather than domain-specific tools. Research questions include: How should agents navigate multiple abstraction levels simultaneously? What verification mechanisms ensure correctness when generating lower-level code?

**Massively Parallel Search at Scale.** The current tree search explores candidates sequentially through the state machine (Section 3.1), but Section 6 proposes "relaxing consistency guarantees enables thousands of candidates to be evaluated simultaneously across distributed infrastructure." This "infinite-width" search paradigm combined with inference-time scaling laws suggests "kernel quality may improve predictably with compute investment." Research opportunities include:
- Characterizing the relationship between search compute and kernel quality (scaling laws for kernel optimization)
- Designing distributed search algorithms that maintain exploration diversity without central coordination
- Developing early termination criteria that identify promising candidates before full evaluation

**Hardware-Specific Adaptation via Reinforcement Learning.** Section 6 proposes "reinforcement learning from execution feedback—rewarding compilation success, correctness, and latency—offers a path to adapt generic models to specialized hardware without exposing proprietary details." This could enable "rapid onboarding of new accelerators: deploy hardware, collect execution traces, fine-tune the agent, and generate optimized kernels within days rather than months." Key research questions:
- What reward signals best capture optimization objectives across different hardware architectures?
- How can RL fine-tuning preserve generalization while adapting to hardware-specific constraints?
- What is the minimum amount of execution feedback needed for effective adaptation?

**Sustainable AI Infrastructure.** Section 6 explicitly acknowledges that "as LLM-based generation scales, resource efficiency becomes critical." Future work should "quantify token consumption per kernel, optimize prompt design for minimal inference cost, and track carbon footprint across the search process." This research direction addresses an important gap: the paper reports development time reduction from "weeks to hours" but does not quantify the computational cost of tree search (Section 6.5). Understanding the energy and cost implications is essential for organizations evaluating deployment.

**Automated Knowledge Base Curation.** The current knowledge base requires manual documentation (15-40 documents per platform, totaling ≥100 documents per Section 3.2.1). Section 6 envisions "developing hardware abstraction primitives that capture memory hierarchies, compute capabilities, and ISA constraints in a format amenable to LLM reasoning." Research should explore:
- Automatic extraction of hardware constraints from compiler documentation and specification sheets
- Self-updating knowledge bases that learn from compilation failures and profiling results
- Quality assurance mechanisms ensuring documentation accuracy as hardware evolves

**Organizational Integration Patterns.** Section 4.3 shares "organizational integration patterns for adopting automated kernel generation workflows." Practitioners should consider:

1. **Start with coverage gaps.** Prioritize operators missing native hardware support (like MTIA v2i preprocessing operators in Table 5) where KernelEvolve provides both enablement and optimization.

2. **Establish fallback paths.** Section 5.2 notes the system "deploys the highest-performing variant while maintaining fallback paths to vendor libraries when generated kernels underperform." Production deployment requires runtime dispatch logic to handle shape-dependent regressions (Section 6.4).

3. **Invest in profiling infrastructure.** The evaluation framework integrates multiple profiling tools (TritonBench, Torch Profiler, NCU, Proton, MTIA Insight per Section 3.4.3). Organizations lacking this instrumentation will need to build it before deploying automated optimization.

4. **Plan for knowledge base maintenance.** Section 3.2.1 describes the hierarchical knowledge base structure requiring ongoing documentation effort. Organizations should allocate engineering resources for maintaining hardware-specific documentation as compilers and runtimes update.

**Downstream System Impacts.** Successful kernel optimization cascades through the infrastructure stack:

- **Reduced infrastructure costs.** Section 1 notes "marginal kernel-level performance improvements translate to multi-million dollar reductions in infrastructure operating costs." A 2× speedup on high-volume kernels directly reduces accelerator count requirements.

- **Enabling new model architectures.** Section 1.1 states "incomplete kernel coverage blocks model launches, directly constraining algorithmic innovation." Comprehensive kernel support allows algorithm teams to experiment with novel operators without deployment blockers.

- **Simplified serving architecture.** Section 1.1 quantifies that missing preprocessing operators force "disaggregated serving architectures partitioning models across compute tiers" with 10-20ms network overhead (Table 2). Monolithic deployment simplifies operations and reduces latency.

### 7.4 Reproducibility and Integration Guidance

**What Can Be Replicated Externally.** While KernelEvolve depends on Meta-internal infrastructure (Bento, Conveyor, FaaS, fbsource per Section 6.7), the core algorithmic contributions are portable:

| Component | External Viability | Notes |
|-----------|-------------------|-------|
| Graph-based search framework | **High** | MCTS, evolutionary algorithms are well-documented (Section 3.1) |
| Universal operator with dynamic prompting | **High** | Retrieval-augmented prompting is implementable with public LLMs |
| Persistent knowledge base | **Medium** | Requires documentation effort but no proprietary infrastructure |
| Hardware interpreters | **Medium** | Can use public cloud GPUs; MTIA-specific tooling unavailable |
| FaaS-based evaluation | **Low** | Requires GPU-enabled serverless infrastructure (Section 3.4.6) |
| MTIA-specific knowledge | **Low** | Proprietary hardware documentation not publicly available |

**When to Prefer KernelEvolve Over Alternatives.** Based on the limitations identified in Section 6, practitioners should consider KernelEvolve-style approaches when:

| Condition | Recommendation | Rationale |
|-----------|---------------|-----------|
| Deploying on proprietary/custom accelerators | **Strongly prefer** | Largest speedups (4-17×) where vendor libraries are nascent (Figure 13) |
| High-volume kernels (>1M inferences/day) | **Prefer** | Amortizes search cost over many inferences |
| Stable input distributions | **Prefer** | Shape specialization is appropriate when distributions are known (Section 6.1) |
| Preprocessing operators with irregular access | **Prefer** | Fusion opportunities create optimization headroom (Section 5.4) |
| NVIDIA GPUs with standard workloads | **Consider alternatives** | Limited headroom (1.2-1.6×) may not justify search cost (Figure 13) |
| Research with variable tensor shapes | **Avoid** | Shape specialization causes regressions on out-of-distribution inputs (Table 3) |
| Low-volume kernels | **Avoid** | Search cost may not be justified by infrequent use (Section 6.5) |

**Minimum Viable Implementation.** Organizations lacking Meta-scale infrastructure can implement a simplified KernelEvolve variant:

1. **Start with single-platform Triton generation** using public LLMs and cloud GPU access
2. **Implement basic tree search** with greedy selection before adding MCTS complexity
3. **Use file-based storage** instead of relational metadata store for small-scale exploration
4. **Manual profiling** with existing tools (NCU, PyTorch Profiler) before building automated pipelines
5. **Focus on 5-10 critical operators** rather than comprehensive coverage initially

This incremental approach allows organizations to validate the approach before investing in production-grade infrastructure.

**Key Success Factors.** Based on the deployment experience shared in Section 4.3, successful adoption requires:

- **Executive sponsorship** for the infrastructure investment (interpreters, profiling, FaaS)
- **Cross-team collaboration** between ML engineers, compiler engineers, and infrastructure teams
- **Patience with iteration**—Figure 12 shows 300 search steps for conv1d optimization
- **Willingness to maintain fallback paths** for shape-dependent regressions
- **Commitment to knowledge base curation** as hardware and compilers evolve

### 7.5 The Broader Vision: AI Agents as Systems Software

KernelEvolve points toward a future where **LLM agents serve as general-purpose compilers** rather than domain-specific tools. Section 8 articulates this vision: "We envision a future where LLM agents serve as the universal compilation layer for heterogeneous AI systems, automatically adapting to new hardware through knowledge injection rather than manual porting."

This vision has implications beyond kernel optimization:

- **Database query optimization:** Agents could generate hardware-specific query execution plans
- **Network protocol stacks:** Agents could optimize packet processing for specific NIC architectures
- **Operating system schedulers:** Agents could generate scheduling policies tuned to workload characteristics
- **Security hardening:** Agents could generate platform-specific mitigations for known vulnerabilities

The design principles established by KernelEvolve—retrieval-augmented context management, graph-based search with execution feedback, persistent knowledge bases, multi-granularity validation—provide a blueprint for AI-assisted systems software across domains.

**The Path Forward.** Section 8 concludes that "the path forward requires not just better kernels, but fundamentally new approaches to bridging the hardware-software gap at scale." As AI infrastructure evolves toward increasingly diverse accelerator fleets—integrating custom silicon, next-generation GPUs, and specialized processors—the gap between hardware innovation and software ecosystem maturity widens. KernelEvolve demonstrates that automated synthesis can close this gap, transforming kernel availability from a deployment blocker to an enabler.

The insights from this deployment—multi-granularity profiling integration, automated validation frameworks, and architecture-aware optimization—establish design principles that will prove essential as the hardware landscape continues to fragment and diversify. For researchers and practitioners, the question is not whether AI agents will participate in systems software development, but how to design these systems for correctness, efficiency, and maintainability at production scale.
