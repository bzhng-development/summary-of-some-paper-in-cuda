## 1. Executive Summary
This paper introduces Reinforcement Pre-Training (RPT), a new scaling paradigm that reframes standard next-token prediction as a reasoning task trained via reinforcement learning with verifiable rewards, specifically utilizing the OmniMATH dataset for experiments. By transforming vast unannotated text into a massive dataset for general-purpose RL without external annotations, RPT addresses the scalability limitations of current RL methods like RLHF and RLVR while significantly improving next-token prediction accuracy. Experiments demonstrate that RPT-14B not only surpasses standard baselines (e.g., 45.11% vs 41.90% accuracy on easy splits) but also matches the performance of significantly larger models like R1-Distill-Qwen-32B, establishing a stronger foundation for subsequent reinforcement fine-tuning.

## 2. Context and Motivation

To understand why Reinforcement Pre-Training (RPT) represents a meaningful shift, we must first examine the fundamental tension in modern language model development: the gap between **scalable pre-training objectives** and **powerful but limited reinforcement learning techniques**. This section unpacks the specific problem RPT addresses, why it matters, what prior approaches attempted, and where they fell short.

### The Core Problem: RL Cannot Scale to Pre-Training

The paper identifies a critical scalability bottleneck in how reinforcement learning (RL) is currently applied to large language models. While RL has proven effective for **post-training** tasks like alignment and reasoning enhancement, it cannot be applied to the **pre-training** stage where models learn from vast text corpora. The gap exists for two interconnected reasons:

1. **Data availability mismatch**: Standard pre-training uses billions of tokens from unannotated web text. RL traditionally requires labeled data with verifiable outcomes (e.g., question-answer pairs with correct solutions).

2. **Reward signal limitations**: Pre-training lacks the clear correctness signals that RL needs. In next-token prediction, there is no external "verifier" to say whether a token prediction is right or wrong beyond matching the corpus itself.

As stated in the Introduction:

> However, current applications of RL in LLM training face scalability and generality challenges. Reinforcement learning from human feedback [OWJ+22], while effective for alignment, relies on costly human preference data, and its learned reward models can be susceptible to reward hacking, limiting scalability.

This limitation matters because pre-training is where models acquire their foundational knowledge. If RL cannot operate at this stage, its benefits remain confined to fine-tuning on narrow tasks.

### Why This Problem Matters

The importance of bridging this gap operates on both theoretical and practical levels:

**Theoretical significance**: The standard next-token prediction objective treats language modeling as a **pattern matching problem**. The model learns statistical correlations between tokens without necessarily understanding *why* a token should follow. The paper argues this limits deeper comprehension:

> By explicitly encouraging next-token reasoning patterns, RPT promotes deeper understanding and generalization instead of merely memorizing next tokens. The model learns to explore and validate hypotheses about why a certain token should follow, fostering more robust representations.

This reframing suggests that prediction accuracy alone may not capture whether a model truly understands the content it processes.

**Practical impact**: Three concrete benefits emerge if RL can scale to pre-training:

- **Compute allocation**: RPT allows models to spend more computational effort on each prediction during training, similar to test-time scaling but applied during pre-training. This is described as "a form of inference-time scaling applied at training time for each token."

- **Reward hacking mitigation**: Rule-based rewards (correct/incorrect token matching) are inherently more stable than learned reward models. The paper notes this "inherently minimizes the risk of reward hacking often associated with complex, learned reward models."

- **Foundation quality**: A model pre-trained with reasoning patterns transfers better to downstream RL fine-tuning. Section 4.3 shows RPT-14B achieves 56.3% before RL fine-tuning versus 51.2% for the baseline (Table 2).

### Prior Approaches and Their Limitations

Two main RL paradigms existed before RPT, each with distinct constraints:

#### Reinforcement Learning from Human Feedback (RLHF)

RLHF [OWJ+22] fine-tunes pre-trained models using human preference data. The workflow involves:

1. Collecting human rankings of model outputs
2. Training a reward model to predict these preferences
3. Using RL (typically PPO) to maximize the reward model's score

**Where it falls short**:

- **Cost**: Human annotation does not scale to web-text corpus sizes. Pre-training requires billions of tokens; RLHF typically uses thousands to millions of preference labels.
- **Reward hacking**: Learned reward models can be exploited. The model may discover shortcuts that maximize reward without improving actual capability.
- **Scope**: RLHF targets alignment (making outputs helpful, harmless, honest), not foundational language understanding.

#### Reinforcement Learning with Verifiable Rewards (RLVR)

RLVR [LMP+25] uses objective, rule-based rewards instead of learned reward models. For a question-answer pair $(q, a)$, a deterministic verifier $V$ calculates:

$$r = V(o, a)$$

where $o$ is the model's output. The objective maximizes expected reward:

$$J_{\text{RLVR}}(\theta) = \mathbb{E}_{(q,a) \sim \mathcal{D}, o \sim \pi_\theta(\cdot|q)} [r(o, a)]$$

**Where it falls short**:

- **Data scarcity**: RLVR requires annotated question-answer pairs with verifiable answers. As the Introduction states: "RLVR is typically constrained by the scarcity of annotated data with verifiable answers, restricting its application to domain-specific fine-tuning rather than general-purpose pre-training."
- **Domain specificity**: Math problems, code execution, and factual QA have clear correctness criteria. General text (news articles, narratives, explanations) does not.

#### Quiet-Star: The Closest Prior Work

The Related Work section identifies [ZHS+24] as most relevant:

> [ZHS+24] is the most relevant work, which encourages language models to generate helpful rationales for next-token prediction. The helpfulness-based reward tends to be hacked by repeating the target token in the generated rationale, where the shortcut potentially harms the model.

Quiet-Star attempted to add reasoning before token prediction but used a **helpfulness-based reward** that could be gamed. The model could simply repeat the target token in its rationale without genuine reasoning. RPT addresses this by using **correctness-based rewards** (does the prediction match the ground truth?) rather than subjective helpfulness scores.

### How RPT Positions Itself

RPT occupies a unique position by combining elements from both pre-training and RL paradigms:

| Aspect | Standard Pre-Training | RLHF / RLVR | RPT (This Work) |
|--------|----------------------|-------------|-----------------|
| Data source | Unannotated web text | Human preferences or QA pairs | Unannotated web text |
| Reward signal | Implicit (likelihood) | Learned or rule-based | Rule-based (token correctness) |
| Scalability | High | Limited | High |
| Reasoning encouragement | None | Task-specific | Built into objective |

The key innovation is **reframing next-token prediction as next-token reasoning**. Instead of directly predicting $x_t$ given context $x_{&lt;t}$, the model generates a chain-of-thought $c_t$ before producing prediction $y_t$. The reward is binary:

$$r^i_t = \begin{cases} 1 & \text{if } y^i_t = x_{\geq t}[1:l] \text{ and } l \in \mathcal{L}_{\text{gt}} \\ 0 & \text{otherwise} \end{cases}$$

This transforms every token position in the pre-training corpus into a verifiable RL problem without requiring external annotations.

Figure 1 illustrates this conceptual shift: RPT scales RL to the web-text corpus by treating next-token prediction as a reasoning task with intrinsic verifiable rewards. The "cherry-on-top cake" metaphor (from LeCun's slides [LeC16]) suggests RPT adds a reasoning layer atop standard pre-training.

### The Scaling Question

A final motivational element concerns whether RL-based pre-training can exhibit favorable **scaling properties**. Traditional pre-training follows power-law relationships between compute and performance [KMH+20, HBM+22b]. If RPT is to be viable long-term, it must demonstrate similar scaling behavior.

Section 4.2 investigates this explicitly, fitting a power-law form:

$$P(C) = \frac{A}{C^\alpha} + P^*$$

where $P(C)$ denotes next-token prediction accuracy and $C$ is training compute. Figure 5 shows high coefficients of determination ($R^2 = 0.995$ for easy, $0.997$ for medium, $0.989$ for hard splits), indicating RPT performance improves reliably with increased compute. This scaling evidence is crucial for positioning RPT as a sustainable paradigm rather than a one-off improvement.

### Summary of the Gap

The motivation crystallizes around a single observation: **the data for pre-training exists, but the RL framework to leverage it does not**. RPT closes this gap by:

1. Using the pre-training corpus itself as the reward source (the ground-truth next token)
2. Requiring reasoning before prediction (chain-of-thought generation)
3. Applying on-policy RL to optimize reasoning trajectories

This positions RPT not as a replacement for standard pre-training, but as an enhancement that preserves scalability while adding reasoning capabilities at the foundational level.

## 3. Technical Approach

This is a methods paper proposing a new training paradigm that converts standard language model pre-training into a reinforcement learning problem by requiring the model to reason before predicting each token. The core idea is simple but transformative: instead of directly predicting the next token, the model generates a chain-of-thought reasoning sequence first, then receives a binary reward based on whether its final prediction matches the ground-truth token from the pre-training corpus itself.

### 3.1 Reader orientation (approachable technical breakdown)

The system being built is a language model training framework that forces the model to "think out loud" before predicting each token, then uses reinforcement learning to reward correct predictions. The problem it solves is that standard pre-training teaches models to match patterns without understanding why tokens follow each other, and the solution shape is to insert a reasoning step between context and prediction, verified by a simple correct/incorrect reward signal derived from the training data itself.

### 3.2 Big-picture architecture (diagram in words)

The RPT pipeline consists of four major components connected in sequence. First, the **Data Preparation Module** takes raw text from the pre-training corpus and filters it based on token difficulty, keeping only positions where prediction requires genuine reasoning effort. Second, the **Rollout Generator** takes each context prefix and samples G different reasoning trajectories from the current model, where each trajectory includes both a chain-of-thought sequence and a final token prediction. Third, the **Reward Verifier** compares each prediction against the ground-truth continuation from the corpus using byte-level prefix matching, assigning 1 for correct and 0 for incorrect. Fourth, the **RL Update Module** uses the GRPO algorithm to adjust model parameters, increasing the probability of trajectories that received positive rewards. Information flows from corpus to filtered tokens to multiple sampled responses to reward signals to parameter updates, completing one training iteration.

- First, I explain the next-token reasoning task formulation because understanding what the model is asked to do is prerequisite to understanding how it is trained.
- Second, I detail the reinforcement learning objective and reward function because these define what "success" means and how the model receives feedback.

### 3.4 Detailed, sentence-based technical breakdown

**Paper Type and Core Mechanism**

This is a methods paper introducing a new training paradigm that reframes next-token prediction as a verifiable reasoning task trainable via reinforcement learning. The fundamental mechanism transforms every token position in a pre-training corpus into an individual reinforcement learning problem where the model must reason before predicting, and correctness is verified against the actual next token in the corpus.

**The Next-Token Reasoning Task**

The task reformulation is the conceptual foundation of RPT. Given an input sequence $x_0 \cdots x_T$ from the training corpus, for each position $t \in \{1, \ldots, T\}$, the prefix $x_{&lt;t}$ is treated as the context and the ground-truth next token is $x_t$. In standard next-token prediction, the model directly outputs a prediction for $x_t$. In next-token reasoning, the model $\pi_\theta$ must first generate a chain-of-thought reasoning sequence, denoted by $c_t$, before generating a prediction $y_t$ for the next token. The overall model response is $o_t = (c_t, y_t)$, where $o_t \sim \pi_\theta(\cdot | x_{&lt;t})$. Figure 2 illustrates this comparison visually: standard prediction estimates the next token directly, while next-token reasoning performs reasoning over multiple tokens before making the prediction. The chain-of-thought process can involve various reasoning patterns such as brainstorming, self-critique, and self-correction, as shown in the example where the model considers multiple possibilities ("Alternatively, it could be...", "Wait, perhaps...") before committing to a final answer in a `\boxed{}` format. This task reconstruction shifts pre-training beyond learning superficial token-level correlations to understanding the hidden knowledge behind them, making RL scaling possible on web-text corpora.

**Reinforcement Learning Objective**

RPT trains LLMs to perform next-token reasoning via on-policy reinforcement learning, as illustrated in Figure 3. For the context $x_{&lt;t}$, the framework prompts the language model $\pi_\theta$ to generate $G$ responses (thinking trajectories), denoted $\{o^i_t\}^G_{i=1}$. Each response $o^i_t = (c^i_t, y^i_t)$ consists of a chain-of-thought reasoning sequence $c^i_t$ and a final prediction sequence $y^i_t$. The model is trained to maximize the expected reward across all sampled trajectories. The formal objective function is:

$$J_{\text{RPT}}(\theta) = \mathbb{E}_{(x_{&lt;t}, x_{\geq t}) \sim \mathcal{D}, \{o^i_t\}^G_{i=1} \sim \pi_\theta(\cdot|x_{&lt;t})} \left[r^i_t\right]$$

where $\mathcal{D}$ is the set of all context prefixes $\{x_{&lt;t}\}^T_{t=1}$ from the corpus, and $r^i_t$ is the reward for the $i$-th output given context $x_{&lt;t}$. This objective differs from standard next-token prediction, which maximizes log-likelihood:

$$J_{\text{NTP}}(\theta) = \sum_{t=1}^T \log P(x_t | x_0, x_1, \ldots, x_{t-1}; \theta)$$

The key distinction is that NTP optimizes probability distributions over tokens, while RPT optimizes expected rewards over complete reasoning trajectories.

**Reward Function Design**

The reward mechanism is critical because it determines what behaviors the model learns to reinforce. The paper introduces a prefix matching reward, which supports verifying predictions that span multiple tokens or involve out-of-vocabulary tokens. Let $x_{\geq t}$ and $y^i_t$ denote the byte sequences of the ground-truth completion sequence and the prediction $y^i_t$, respectively. Denote the byte length of $y^i_t$ by $l$. The system defines the cumulative byte lengths of the tokens in the ground-truth completion sequence as valid boundaries, and denotes this set by $\mathcal{L}_{\text{gt}}$. Formally, the reward $r^i_t$ for the $i$-th output for $x_{&lt;t}$ is defined as:

$$r^i_t = \begin{cases} 1 & \text{if } y^i_t = x_{\geq t}[1:l] \text{ and } l \in \mathcal{L}_{\text{gt}} \\ 0 & \text{otherwise} \end{cases}$$

The reward is 1 if the byte sequence of the prediction is an exact prefix of the ground-truth completion sequence and its length $l$ matches any valid token boundary; otherwise, the reward is 0. This design choice matters for several reasons. First, byte-level matching handles cases where the predicted token may not align perfectly with the tokenizer's vocabulary boundaries. Second, the binary reward structure (1 or 0) minimizes reward hacking compared to learned reward models that can be exploited. Third, using the ground-truth token from the corpus itself as the verification signal means no external annotations are required—every token in the pre-training data becomes a verifiable RL problem. Figure 3 illustrates this process: given a context with a missing continuation, the LLM performs on-policy rollouts to generate $G$ different thinking trajectories, each including an intermediate reasoning step and a final prediction, and a positive reward is assigned if the prediction matches the ground-truth token.

**Training Configuration and Hyperparameters**

The experimental setup uses specific configurations that determine training feasibility and reproducibility. The dataset is OmniMATH, which contains 4,428 competition-level mathematical problems and solutions from official websites such as AoPS Wiki and AoPS forum. The base model is Deepseek-R1-Distill-Qwen-14B, which serves as a good starting point for reinforcement learning due to its basic reasoning capabilities. The training framework is implemented with the verl library and uses vllm for inference. The RL algorithm is GRPO (Group Relative Policy Optimization), which is an on-policy method suitable for this setting. Key hyperparameters from Section 3.3 and Appendix B include: training sequence length of 8k tokens, learning rate of $1 \times 10^{-6}$, zero KL penalty (meaning no regularization against deviation from the initial model), batch size of 256 questions, $G=8$ responses sampled per question, rollout temperature of 0.8, and total training steps of 1,000. Starting from step 500, the system utilizes dynamic sampling to boost training efficiency. For extracting predictions, the framework directly extracts the full sequence inside the last `\boxed{}` following the special token `</think>` as the model prediction for the next token. Adam optimizer parameters are $\beta = (0.9, 0.999)$, weight decay is 0.01, actor gradient clip is 0.2, maximum prompt length is 4096 tokens, and maximum response length is 8192 tokens. The entropy loss coefficient is set to 0, following exact on-policy reinforcement learning settings.

**Data Filtering Strategy**

Not all token positions benefit equally from reasoning-based training, so the paper employs token-level data filtering before reinforcement pre-training. Since many tokens are easily predictable even without reasoning, training on them would waste compute without improving reasoning capabilities. The filtering process uses Deepseek-R1-Distill-Qwen-1.5B as a small proxy model. For each token position, the system calculates the proxy model entropy on the top-16 next tokens. By applying an entropy threshold, the framework filters out low-entropy positions, prioritizing training on challenging tokens that require greater computational effort to predict. This design choice is important because it focuses training compute on positions where reasoning actually matters—high-entropy positions indicate uncertainty in the proxy model, suggesting the token is not trivially predictable from context. During evaluation in Section 4.1, the paper categorizes token positions in the validation set according to their difficulty using entropy thresholds: positions are designated as easy, medium, or hard splits if their entropy exceeds thresholds of 0.5, 1.0, and 1.5, respectively. This same filtering logic applies during training to ensure the model spends effort on tokens that genuinely require reasoning.

**Evaluation Methodology**

The paper evaluates pretrained models along two dimensions: language modeling capability and transfer to downstream RL fine-tuning. For language modeling evaluation, the system reports next-token prediction accuracy on a held-out validation set of 200 samples from OmniMATH. The evaluation compares RPT-14B against two baselines evaluated in different ways: standard next-token prediction (selecting the token with the highest probability) and next-token reasoning (generating a chain-of-thought before the final prediction) using R1-Distill-Qwen-14B, plus Qwen2.5-14B as the base model for R1-Distill-Qwen-14B. Table 1 presents results across three test splits of varying difficulty, showing RPT-14B achieves 45.11% accuracy on easy splits versus 41.90% for the standard baseline. For reinforcement fine-tuning evaluation, the paper conducts continual RL fine-tuning with RPT models in a pretrain-then-finetune manner on Skywork-OR1 data, using 256 examples for training and 200 for testing, with both training batch size and PPO mini-batch size set to 64, trained for 15 epochs. During evaluation, the maximum number of tokens for validation is set to 32,000 with a temperature of 0.6. Zero-shot performance is evaluated on MMLU-Pro and SuperGPQA benchmarks under reasoning settings with maximum 12,288 tokens and temperature 0.8, using multiple-choice question format.

**Design Choices and Alternatives**

The paper explicitly investigates alternative reward designs to assess robustness of the core approach. Appendix A describes three variations beyond the prefix matching reward. The first is first-token matching, where the reward reflects only whether the first token of the model prediction matches the ground-truth next token, ignoring all tokens after the first. The second is a dense reward scheme, where correctly predicted next tokens receive a full reward of 1, but for incorrect predictions, the reward is a positive, smaller value equal to the language model probability of generating that particular incorrect token, $P(y^i_t[0] | x_{&lt;t}; \theta)$, providing denser feedback than binary rewards. The third is conditional dense reward, where the dense reward structure is used only for training instances where at least one of the $G$ sampled rollouts correctly predicted the next token; if all rollouts are incorrect, a different reward scheme applies. Experiments indicate these alternative reward designs generally achieved performance comparable to the prefix matching reward, suggesting the RPT framework is relatively robust to these particular modifications. The paper also explores seven prompt template variants in Appendix D, with Table 8 showing clear prompts significantly improve initial performance (v6 achieves 19.0% Pass@8 versus 8.5% for v0), though the main experiments used v0 and prompt engineering is left for future work. The choice to use GRPO over other RL algorithms like PPO is motivated by GRPO's efficiency in on-policy settings with multiple sampled trajectories per input. The decision to initialize from a reasoning-capable model (R1-Distill-Qwen-14B) rather than a standard base model is acknowledged as a limitation in Section 6, with future work planned to investigate RPT training from standard base language models. The selection of mathematical documents for the pre-training corpus is also noted as a limitation, with future work exploring broader general-domain text.

**Scaling Analysis Framework**

The paper investigates whether RPT exhibits favorable scaling properties comparable to standard pre-training. Traditional next-token pre-training loss empirically follows a power-law decay with respect to model size, number of training tokens, and training compute. For RPT, the paper analyzes scaling behavior specifically with respect to training compute $C$, modeling the relationship using the power-law form:

$$P(C) = \frac{A}{C^\alpha} + P^*$$

where $P(C)$ denotes the next-token prediction accuracy on the validation set, and $P^*$, $\alpha$, and $A$ are parameters to be estimated. The evaluation measures next-token prediction accuracy at various training steps (100, 200, 400, 800, 1000, and 1200) and converts them into corresponding training compute. Figure 5 shows the results with fitted curves exhibiting high coefficients of determination: $R^2 = 0.995$ for easy, $0.997$ for medium, and $0.989$ for hard splits, indicating the scaling curves accurately capture performance trends and that RPT performance improves reliably as training compute scales up. This scaling evidence is crucial for positioning RPT as a sustainable paradigm rather than a one-off improvement, demonstrating that increased compute consistently yields better next-token prediction accuracy across all difficulty levels.

## 4. Key Insights and Innovations

This section distills the conceptual breakthroughs that distinguish RPT from prior work. Rather than cataloging technical specifications (covered in Section 3), I focus on what makes each contribution fundamentally novel versus incrementally improved, and why these distinctions matter for the field's trajectory.

### 4.1 Intrinsic Verifiability: Every Token Becomes a RL Problem

**The Innovation**: RPT discovers that the pre-training corpus *already contains* verifiable reward signals—the ground-truth next token itself. This eliminates the need for external annotations, human preferences, or domain-specific question-answer pairs.

**What Makes This Different**: Prior RL approaches for LLMs all require *external* verification:

| Approach | Reward Source | Annotation Required |
|----------|---------------|---------------------|
| RLHF [OWJ+22] | Human preference rankings | Yes (expensive) |
| RLVR [LMP+25] | Question-answer correctness | Yes (domain-specific) |
| Quiet-Star [ZHS+24] | Helpfulness-based scoring | No, but reward hackable |
| **RPT (This Work)** | **Ground-truth token match** | **No (intrinsic)** |

The Quiet-Star comparison is particularly instructive. As noted in Section 2, Quiet-Star attempted reasoning-before-prediction but used a helpfulness-based reward that could be gamed by repeating the target token in the rationale. RPT's correctness-based reward (Section 3.2) cannot be hacked this way—the prediction must actually match the corpus token, regardless of what the reasoning says.

**Why This Matters**: This transforms the scalability equation. Pre-training corpora contain trillions of tokens; RLVR datasets contain thousands to millions of annotated examples. By making every token position a verifiable RL problem, RPT unlocks RL's benefits (explicit reasoning, exploration, reward optimization) at pre-training scale. This is not an incremental improvement to RLVR—it is a fundamental redefinition of what counts as a "verifiable" reward signal.

**Evidence of Significance**: Table 1 shows RPT-14B achieves 45.11% accuracy on easy splits versus 41.90% for standard next-token prediction—a 3.21 percentage point gain without any external annotations. More importantly, Figure 4 shows RPT-14B matches R1-Distill-Qwen-32B performance, suggesting the reasoning objective extracts more capability per parameter than simply scaling model size.

### 4.2 Training-Time Inference Scaling: Compute Allocation During Pre-Training

**The Innovation**: RPT applies what is traditionally *test-time* compute scaling (spending more inference compute per query) to the *training* phase. Each token prediction during pre-training receives G=8 sampled reasoning trajectories, with the model learning which reasoning patterns lead to correct predictions.

**What Makes This Different**: The scaling literature distinguishes two paradigms [ZLS+25a, JKL+24]:

- **Training-time scaling**: Increase model parameters or training tokens
- **Test-time scaling**: Increase inference compute (e.g., chain-of-thought, multiple samples, search)

RPT blurs this boundary. As stated in the Introduction:

> The internal reasoning process during pre-training effectively allows the model to allocate more "thought" or computational effort to each prediction step, akin to a form of inference-time scaling applied at training time for each token.

This is conceptually distinct from standard pre-training, where each token receives exactly one forward pass with no explicit reasoning. It is also distinct from test-time scaling, where reasoning happens only during inference on downstream tasks.

**Why This Matters**: This creates a new axis for capability improvement. Instead of asking "should we make the model bigger or train longer?", RPT enables "should we spend more compute per token during training to build stronger reasoning foundations?" The hypothesis is that models trained with explicit reasoning patterns generalize better because they learn *why* tokens follow, not just *that* they follow.

**Evidence of Significance**: Section 4.5's reasoning pattern analysis (Figure 6) shows RPT-14B uses hypothesis patterns 161.8% more and deduction patterns 26.2% more than the baseline model doing problem-solving. Table 4's case study reveals the model engages in deliberative processes—considering alternatives ("Alternatively, it could be..."), reflecting on structural cues ("markdown with headers"), and weighing multiple plausible continuations. This is qualitatively different from the pattern-matching behavior of standard next-token prediction.

**Caveat**: This innovation comes with a compute cost. Generating G=8 trajectories per token position is substantially more expensive than single-pass prediction. The scaling curves in Figure 5 show this cost is justified (performance improves reliably with compute), but the trade-off remains an open question for production deployments.

### 4.3 Objective Alignment: RL Pre-Training Enables RL Fine-Tuning

**The Innovation**: RPT minimizes the objective gap between pre-training and post-training. Both stages use reinforcement learning with verifiable rewards, creating a continuous optimization landscape rather than a discontinuous shift from likelihood maximization to reward maximization.

**What Makes This Different**: Standard training pipelines create an objective mismatch:

```
Standard Pipeline:
Pre-training: Maximize log-likelihood (next-token prediction)
    ↓ (objective shift)
RL Fine-tuning: Maximize expected reward (RLVR/RLHF)

RPT Pipeline:
Pre-training: Maximize expected reward (next-token reasoning)
    ↓ (objective continuity)
RL Fine-tuning: Maximize expected reward (RLVR)
```

**Why This Matters**: Objective mismatches create optimization friction. A model trained to maximize likelihood may have learned representations that are suboptimal for reward maximization. RPT's pre-trained model already operates in the reward-maximization regime, so fine-tuning requires less adaptation.

**Evidence of Significance**: Table 2 provides direct evidence. When RPT-14B undergoes RLVR fine-tuning, it starts at 56.3% before RL and reaches 58.3% after. In contrast, R1-Distill-Qwen-14B with continual next-token prediction training on the same corpus drops to 10.7% before RL and only reaches 13.0% after. The paper states:

> The reasoning ability of the model significantly declines when continually trained on the same data using a next-token prediction objective. Subsequent RLVR yields only slow performance improvements.

This is not merely a performance gain—it demonstrates that the *training objective itself* shapes what capabilities are preserved or degraded during continual training. RPT's RL-based objective preserves reasoning patterns that standard next-token prediction erodes.

**Incremental vs. Fundamental**: This is a fundamental innovation in training pipeline design, not an incremental hyperparameter tuning. It suggests that pre-training objectives should be chosen with post-training requirements in mind, rather than treating pre-training as a generic foundation stage.

### 4.4 Sustainable Scaling: Power-Law Evidence for RL-Based Pre-Training

**The Innovation**: RPT demonstrates favorable scaling properties—performance improves predictably with increased training compute, following a power-law relationship comparable to standard pre-training.

**What Makes This Different**: Many novel training methods show one-off improvements but fail to scale. If RPT were merely a clever trick that works at 14B parameters but plateaus, it would have limited long-term value. The scaling analysis in Section 4.2 tests whether RPT is a sustainable paradigm.

The paper fits the power-law form (Equation 5, Section 4.2):

$$P(C) = \frac{A}{C^\alpha} + P^*$$

where $P(C)$ is next-token prediction accuracy and $C$ is training compute.

**Evidence of Significance**: Figure 5 shows coefficients of determination $R^2 = 0.995$ (easy), $0.997$ (medium), and $0.989$ (hard). These exceptionally high values indicate the scaling curves accurately capture performance trends—RPT is not a one-off improvement but a predictable, scalable paradigm.

**Why This Matters**: Scaling laws guide investment decisions. If a method does not scale, there is a ceiling on its value. The power-law evidence suggests RPT can benefit from continued compute increases, similar to how standard pre-training has scaled over the past decade. This positions RPT as a viable long-term direction, not a temporary workaround.

**Caveat**: The current experiments use 1,000 training steps on 4,428 mathematical problems (OmniMATH). As acknowledged in Section 6, future work must validate scaling on broader corpora and larger compute budgets. The power-law fit is promising but based on a relatively narrow range of compute values.

### 4.5 Distinguishing Incremental from Fundamental Contributions

To clarify the novelty landscape, I categorize RPT's contributions:

| Contribution | Type | Rationale |
|--------------|------|-----------|
| Correctness-based reward (vs. helpfulness) | **Fundamental** | Eliminates reward hacking vulnerability present in Quiet-Star |
| Intrinsic verifiability from corpus | **Fundamental** | Removes annotation bottleneck that limits RLVR/RLHF scalability |
| Training-time inference scaling | **Fundamental** | Creates new axis for capability improvement beyond model size/data |
| Objective alignment for fine-tuning | **Fundamental** | Redesigns training pipeline to minimize optimization friction |
| GRPO algorithm selection | Incremental | Standard on-policy RL choice; other algorithms could work |
| OmniMATH dataset usage | Incremental | Mathematical domain is convenient but not essential to method |
| Prompt template variants | Incremental | Engineering detail; Appendix D shows v6 outperforms v0 but main experiments used v0 |
| Entropy-based data filtering | **Fundamental** | Focuses compute on tokens requiring reasoning; without this, training efficiency drops significantly |

The fundamental innovations are those that would remain valuable even if specific implementation details changed. The incremental contributions are engineering choices that could be improved without altering the core paradigm.

**Summary**: RPT's primary contribution is conceptual—reframing what counts as a verifiable reward signal and demonstrating that RL can operate at pre-training scale without external annotations. The performance gains (Table 1, Table 3), scaling evidence (Figure 5), and fine-tuning benefits (Table 2) validate this conceptual shift, but the lasting impact lies in opening a new design space for pre-training objectives rather than any single benchmark result.

## 5. Experimental Analysis

This section examines how the authors validate RPT's claims through empirical evaluation. The experiments address four core questions: Does RPT improve next-token prediction accuracy? Does it exhibit favorable scaling properties? Does it provide a better foundation for RL fine-tuning? Does it enhance zero-shot performance on downstream tasks? I walk through each evaluation dimension, present the specific numbers, and assess whether the evidence supports the paper's conclusions.

### 5.1 Evaluation Methodology

**Datasets and Splits**

The experimental design uses multiple datasets to evaluate different aspects of RPT:

- **OmniMATH** [GSY+24]: The primary pre-training corpus containing 4,428 competition-level mathematical problems and solutions from AoPS Wiki and AoPS forum. A held-out validation set of 200 samples is used for language modeling evaluation.

- **Skywork-OR1** [HLL+25]: Used for reinforcement fine-tuning experiments. The authors sample 256 examples for training and 200 for testing, following the data filtering pipeline from Skywork-OR1 to identify challenging instances.

- **MMLU-Pro** [HBB+20]: A comprehensive multi-task understanding benchmark evaluating LLM capabilities across various domains. Used for zero-shot evaluation.

- **SuperGPQA** [DYM+25]: A large-scale benchmark of graduate-level reasoning questions spanning 285 disciplines. Used for zero-shot evaluation.

**Difficulty Stratification**

A critical methodological choice is the entropy-based difficulty stratification for next-token prediction evaluation. Following the data filtering strategy from Section 3.3, the authors calculate entropy at each token position using R1-Distill-Qwen-14B and categorize positions into three splits:

| Split | Entropy Threshold | Interpretation |
|-------|-------------------|----------------|
| Easy | > 0.5 | Low uncertainty, predictable tokens |
| Medium | > 1.0 | Moderate uncertainty |
| Hard | > 1.5 | High uncertainty, requires reasoning |

This stratification matters because it tests whether RPT's benefits are concentrated on easy tokens (which would suggest superficial improvements) or extend to challenging positions where reasoning genuinely matters.

**Baselines and Comparison Models**

The evaluation includes multiple baselines to isolate RPT's contribution:

- **Qwen2.5-14B**: The base model for R1-Distill-Qwen-14B, evaluated with standard next-token prediction.

- **R1-Distill-Qwen-14B**: Evaluated in two modes:
  1. Standard next-token prediction (selecting token with highest probability)
  2. Next-token reasoning (generating chain-of-thought before prediction)

- **R1-Distill-Qwen-32B**: A significantly larger model (more than 2× parameters) used to test whether RPT can match larger model performance through better training objectives.

- **Continual NTP training**: A control condition where R1-Distill-Qwen-14B is continually trained on the same corpus as RPT-14B but using standard next-token prediction objective. This isolates whether improvements come from the RL objective versus simply more training on the data.

**Metrics**

- **Next-token prediction accuracy**: Percentage of correctly predicted tokens on the validation set. This is the primary metric for language modeling capability.

- **RL fine-tuning performance**: Accuracy on Skywork-OR1 test set before and after RLVR fine-tuning.

- **Zero-shot accuracy**: Multiple-choice accuracy on MMLU-Pro and SuperGPQA without task-specific fine-tuning.

- **Reasoning pattern statistics**: Percentage of responses containing reasoning-indicative keywords across six pattern categories (transition, reflection, breakdown, hypothesis, divergent, deduction).

**Training Configuration**

From Section 3.3 and Appendix B, the main experiment uses:
- Base model: Deepseek-R1-Distill-Qwen-14B
- Training steps: 1,000 (with dynamic sampling from step 500)
- Batch size: 256 questions
- Rollout count (G): 8 responses per question
- Learning rate: $1 \times 10^{-6}$
- Temperature: 0.8 for rollout, 0.6 for evaluation
- Training length: 8k tokens
- KL penalty: 0 (no regularization against initial model)

### 5.2 Next-Token Prediction Accuracy (Table 1)

The primary claim is that RPT improves language modeling accuracy. Table 1 presents next-token prediction accuracy across three difficulty splits:

| Model | Easy | Medium | Hard |
|-------|------|--------|------|
| **Standard next-token prediction** | | | |
| Qwen2.5-14B | 41.90% | 30.03% | 20.65% |
| R1-Distill-Qwen-14B | 41.60% | 29.46% | 20.43% |
| **Next-token reasoning** | | | |
| R1-Distill-Qwen-14B | 3.31% | 1.66% | 1.41% |
| **RPT-14B** | **45.11%** | **33.56%** | **23.75%** |

**Key Observations:**

1. **RPT vs. Standard NTP**: RPT-14B outperforms both standard baselines across all difficulty levels. On easy splits, the gain is 3.21 percentage points over Qwen2.5-14B (45.11% vs 41.90%). On hard splits, the gain is 3.10 percentage points (23.75% vs 20.65%). The improvement is consistent across difficulty levels, suggesting RPT does not merely exploit easy tokens.

2. **RPT vs. Reasoning Baseline**: The R1-Distill-Qwen-14B evaluated in reasoning mode achieves only 3.31% on easy splits—dramatically lower than RPT-14B's 45.11%. This is a critical comparison because it shows that simply adding chain-of-thought to a standard pre-trained model does not work. The reasoning capability must be *trained* via RL, not just prompted at inference time.

3. **Absolute Performance Context**: Even RPT-14B's 45.11% on easy splits may seem low compared to typical language modeling perplexity metrics. However, this is token-level accuracy on mathematical text with high entropy positions included. The hard split at 23.75% reflects genuinely challenging prediction positions where multiple continuations are plausible.

**Does This Support the Claim?**

Yes, but with caveats. The consistent improvement across all difficulty levels supports the claim that RPT improves next-token prediction. However, the evaluation is on OmniMATH (mathematical text), not general web text. The Introduction claims RPT is "general-purpose," but the experiments are domain-specific. This limitation is acknowledged in Section 6.

### 5.3 Model Size Comparison (Figure 4)

Figure 4 compares average next-token prediction accuracy across data of different difficulty levels for different model sizes:

- **R1-Qwen-14B** (R1-Distill-Qwen-14B): ~30% average accuracy
- **R1-Qwen-32B** (R1-Distill-Qwen-32B): ~34% average accuracy
- **RPT-14B**: ~34% average accuracy (matching 32B)

**Key Observation:**

> Remarkably, it matches the performance of a significantly larger model, i.e., R1-Distill-Qwen-32B (Figure 4).

This is a strong claim: RPT-14B achieves comparable performance to a model with more than 2× the parameters. The implication is that better training objectives can extract more capability per parameter than simply scaling model size.

**Does This Support the Claim?**

The figure supports the claim, but the exact numbers are not provided in tabular form—only visual representation in Figure 4. The text states RPT-14B "matches" the 32B model, which suggests the difference is within measurement noise. This is significant because it positions RPT as an alternative scaling axis: instead of increasing parameters, improve the training objective.

### 5.4 Scaling Properties (Figure 5, Section 4.2)

Section 4.2 investigates whether RPT exhibits favorable scaling properties. The authors fit a power-law relationship between training compute $C$ and next-token prediction accuracy $P(C)$:

$$P(C) = \frac{A}{C^\alpha} + P^*$$

Evaluation is conducted at training steps: 100, 200, 400, 800, 1000, and 1200, converted to corresponding training compute.

**Scaling Curve Fit Quality:**

| Difficulty | $R^2$ Value |
|------------|-------------|
| Easy | 0.995 |
| Medium | 0.997 |
| Hard | 0.989 |

**Key Observations:**

1. **High $R^2$ Values**: All three difficulty levels show $R^2 > 0.989$, indicating the power-law model accurately captures performance trends. This is comparable to scaling law fits in standard pre-training literature [KMH+20, HBM+22b].

2. **Consistent Improvement**: Figure 5 shows accuracy increases monotonically with compute across all difficulty levels. There is no plateau visible within the tested range (up to 1,200 steps).

3. **Difficulty Gap Persists**: The three curves (easy, medium, hard) remain separated across the compute range, indicating that harder tokens remain harder even with more training. However, all three improve at similar rates.

**Does This Support the Claim?**

Yes, this is strong evidence for RPT as a sustainable scaling paradigm. The high $R^2$ values indicate predictable improvement with compute, which is essential for long-term investment. However, the compute range tested is relatively narrow (100 to 1,200 steps on 4,428 problems). The paper acknowledges in Section 6 that future work must validate scaling on "broader corpora and larger compute budgets."

### 5.5 Reinforcement Fine-Tuning Transfer (Table 2)

Table 2 evaluates whether RPT provides a better foundation for subsequent RL fine-tuning:

| Model | Before RL | After RL |
|-------|-----------|----------|
| R1-Distill-Qwen-14B | 51.2% | 52.7% |
| + Continual NTP training | 10.7% | 13.0% |
| **RPT-14B** | **56.3%** | **58.3%** |

**Key Observations:**

1. **RPT vs. Baseline**: RPT-14B starts 5.1 percentage points higher than R1-Distill-Qwen-14B before any fine-tuning (56.3% vs 51.2%). After RLVR, the gap is 5.6 percentage points (58.3% vs 52.7%).

2. **Continual NTP Catastrophe**: The "Continual NTP training" condition is striking—performance drops from 51.2% to 10.7% when the baseline model is continually trained on the same corpus using standard next-token prediction. This suggests the NTP objective actively degrades reasoning capabilities that were present in the R1-Distill-Qwen-14B starting point.

3. **Fine-Tuning Gains**: RPT-14B gains 2.0 percentage points from RLVR (56.3% → 58.3%), while the baseline gains 1.5 percentage points (51.2% → 52.7%). The absolute gain is similar, but RPT starts from a much higher baseline.

**Does This Support the Claim?**

Yes, this is compelling evidence for objective alignment. The paper states:

> The reasoning ability of the model significantly declines when continually trained on the same data using a next-token prediction objective.

This supports the claim that RPT's RL-based objective preserves reasoning patterns that standard NTP erodes. The catastrophic drop in the "Continual NTP" condition (10.7%) is particularly striking—it suggests objective mismatch can actively harm capabilities, not just fail to improve them.

**Caveat**: The Skywork-OR1 dataset is math-focused, similar to OmniMATH. Transfer to non-mathematical domains is not tested here.

### 5.6 Zero-Shot Performance on End Tasks (Table 3)

Table 3 evaluates zero-shot performance on general-domain benchmarks:

| Model | Mode | SuperGPQA | MMLU-Pro |
|-------|------|-----------|----------|
| R1-Distill-Qwen-14B | Standard NTP | 32.0% | 48.4% |
| R1-Distill-Qwen-32B | Standard NTP | 37.2% | 56.5% |
| R1-Distill-Qwen-14B | Reasoning | 36.1% | 68.9% |
| **RPT-14B** | **Reasoning** | **39.0%** | **71.1%** |

**Key Observations:**

1. **RPT vs. 14B Baseline**: RPT-14B outperforms R1-Distill-Qwen-14B in reasoning mode by 2.9 points on SuperGPQA (39.0% vs 36.1%) and 2.2 points on MMLU-Pro (71.1% vs 68.9%).

2. **RPT vs. 32B Baseline**: RPT-14B surpasses R1-Distill-Qwen-32B (standard NTP mode) by 7.0 points on SuperGPQA (39.0% vs 37.2%) and approximately 14.6 points on MMLU-Pro (71.1% vs 56.5%). The paper claims "~22 points" on MMLU-Pro, but the table shows 71.1 - 56.5 = 14.6. This discrepancy should be noted.

3. **Reasoning Mode Advantage**: The R1-Distill-Qwen-14B in reasoning mode (36.1% SuperGPQA, 68.9% MMLU-Pro) significantly outperforms the same model in standard NTP mode (32.0%, 48.4%). This confirms that reasoning at inference time helps, but RPT-trained models reason better.

**Does This Support the Claim?**

Yes, with one discrepancy. The results support the claim that RPT enhances zero-shot performance. However, the paper states "approximately 22 points on MMLU-Pro" when comparing RPT-14B to R1-Distill-Qwen-32B, but Table 3 shows 71.1% - 56.5% = 14.6 percentage points. This may be a calculation error in the paper, or the 22 points refers to a different comparison (perhaps RPT-14B reasoning vs. R1-Distill-Qwen-32B reasoning, but that model is not evaluated in reasoning mode in the table).

Appendix C (Tables 6 and 7) provides per-category breakdowns. For example, on MMLU-Pro, RPT-14B shows particular strength in Math (91.5% vs 86.0% for R1-Distill-Qwen-14B reasoning) and Physics (82.5% vs 80.0%).

### 5.7 Reasoning Pattern Analysis (Figure 6, Table 4)

Section 4.5 analyzes qualitative differences in reasoning patterns between RPT-14B (next-token reasoning) and R1-Distill-Qwen-14B (problem solving).

**Pattern Statistics (Figure 6):**

| Pattern | RPT-14B vs. Baseline |
|---------|---------------------|
| Hypothesis | +161.8% |
| Deduction | +26.2% |
| Breakdown | Baseline uses more heavily |

**Key Observations:**

1. **Hypothesis Pattern**: RPT-14B uses hypothesis patterns (proposing and verifying assumptions) 161.8% more than the baseline. This aligns with the goal of encouraging exploratory reasoning before committing to a prediction.

2. **Breakdown Pattern**: The baseline problem-solving relies more heavily on breakdown (decomposing the problem). This suggests next-token reasoning elicits qualitatively different reasoning than structured problem-solving.

3. **Case Study (Table 4)**: The example shows the model analyzing semantic context ("calculating vector magnitude"), identifying pivotal phrases ("go over some..."), brainstorming alternatives ("Alternatively, it could be..."), and reflecting on structural cues ("markdown with headers"). This is deliberative, not pattern-matching.

**Does This Support the Claim?**

Yes, this provides mechanistic evidence for *how* RPT works. The pattern analysis shows RPT-trained models engage in genuinely different reasoning processes, not just superficially longer outputs. The 161.8% increase in hypothesis patterns is particularly strong evidence that the RL objective shapes reasoning behavior.

**Caveat**: The analysis is based on 200 sampled responses from each model. While sufficient for statistical comparison, larger samples would strengthen confidence.

### 5.8 Ablation Studies and Robustness Checks

**Alternative Reward Designs (Appendix A)**

The paper investigates three alternative reward functions:

1. **First-token matching**: Reward based only on whether the first predicted token matches ground truth.

2. **Dense reward**: Correct predictions get 1, incorrect predictions get $P(y^i_t[0] | x_{&lt;t}; \theta)$ (the model's probability for that token).

3. **Conditional dense reward**: Dense reward applied only when at least one of $G$ rollouts is correct.

**Result**: "Alternative reward designs generally achieved performance comparable to the prefix matching reward."

**Assessment**: This is a robustness check showing the core framework is not overly sensitive to specific reward design choices. However, exact numbers are not provided—only the qualitative statement that performance is "comparable."

**Prompt Template Variants (Appendix D, Table 8)**

Seven prompt templates are tested:

| Template | Random@1 | Pass@8 |
|----------|----------|--------|
| v0 (used in main experiments) | 3.0% | 8.5% |
| v6 (best) | 6.0% | 19.0% |

**Key Observation**: Clear prompts significantly improve initial performance. v6 achieves 19.0% Pass@8 versus 8.5% for v0—more than 2× improvement.

**Assessment**: This is both a robustness check and a limitation. The main experiments use v0, but v6 performs substantially better. The paper states: "We leave prompt engineering based on other template variants for future work, which tends to improve the final performance." This suggests reported results are conservative—better prompt engineering could yield higher absolute performance.

**Data Filtering Impact**

The entropy-based data filtering is not ablated (no "no filtering" baseline is reported). However, the design rationale is clear: training on low-entropy tokens would waste compute on trivially predictable positions. The evaluation stratification (easy/medium/hard) indirectly validates this by showing RPT improves across all levels, including hard tokens where filtering prioritizes training.

### 5.9 Limitations and Conditional Results

**Domain Limitation**

All experiments use mathematical text (OmniMATH, Skywork-OR1). Section 6 acknowledges:

> The current pre-training corpus predominantly consists of mathematical documents; future work will explore its efficacy on broader, general-domain text.

This is a significant limitation. Mathematical text has clear correctness criteria and logical structure. General web text (news, narratives, conversations) may not benefit equally from reasoning-before-prediction.

**Base Model Limitation**

RPT is initialized from a reasoning-capable model (R1-Distill-Qwen-14B), not a standard base language model:

> Investigating RPT training from a standard base language model would provide further insights into its foundational impact.

This matters because the starting point already has reasoning capabilities. It is unclear how much RPT adds versus how much it refines existing reasoning.

**Compute Cost**

Generating $G=8$ trajectories per token position is substantially more expensive than single-pass next-token prediction. The scaling curves (Figure 5) show this cost is justified by performance gains, but the trade-off is not quantified in terms of FLOPs per accuracy point compared to standard pre-training.

**Result Discrepancy**

As noted in Section 5.6, the paper claims "~22 points on MMLU-Pro" when comparing RPT-14B to R1-Distill-Qwen-32B, but Table 3 shows 14.6 percentage points. This discrepancy is not explained and should be treated as a potential error.

| Claim | Evidence Strength | Notes |
|-------|-------------------|-------|
| RPT improves next-token prediction | **Strong** | Table 1 shows consistent gains across difficulty levels |
| RPT matches larger models | **Moderate-Strong** | Figure 4 supports this, but exact numbers not tabulated |
| RPT scales with compute | **Strong** | Figure 5 shows $R^2 > 0.989$ for all difficulty levels |
| RPT improves RL fine-tuning | **Strong** | Table 2 shows 5.1 point advantage before RL, objective alignment demonstrated |

**Summary Assessment:**

The experiments convincingly support the core technical claims: RPT improves next-token prediction, exhibits favorable scaling, and provides better RL fine-tuning foundations. The reasoning pattern analysis provides mechanistic evidence for *how* RPT works. However, the "general-purpose" claim is not fully validated—experiments are confined to mathematical domains. The prompt template ablation (Appendix D) suggests reported results are conservative, and the reward design ablation (Appendix A) shows robustness to specific choices.

The most compelling evidence is Table 2's "Continual NTP" condition, which demonstrates that objective mismatch can actively degrade capabilities. This supports the fundamental claim that pre-training objectives should align with post-training requirements.

## 6. Limitations and Trade-offs

No training paradigm is without constraints. This section examines the assumptions RPT relies on, the scenarios it does not address, the computational and data requirements it imposes, and the open questions that remain. Understanding these limitations is essential for evaluating whether RPT is suitable for a given use case and for identifying where future research should focus.

### 6.1 Core Assumptions

RPT's effectiveness depends on several foundational assumptions that are not universally guaranteed:

**Assumption 1: Ground-truth tokens represent correct continuations.** The reward mechanism treats the next token in the pre-training corpus as the verification signal. This assumes the corpus is high-quality and that the ground-truth token is unambiguously correct. However, web text contains errors, ambiguities, and multiple valid continuations. As the reward function in Section 3.2 shows:

$$r^i_t = \begin{cases} 1 & \text{if } y^i_t = x_{\geq t}[1:l] \text{ and } l \in \mathcal{L}_{\text{gt}} \\ 0 & \text{otherwise} \end{cases}$$

This binary reward penalizes any prediction that differs from the corpus token, even if the alternative is semantically valid. In mathematical text (used in experiments), correctness is objective. In narrative or conversational text, multiple continuations may be equally reasonable. This assumption limits RPT's applicability to domains where ground-truth tokens are reliably correct.

**Assumption 2: Reasoning before prediction improves accuracy.** The entire paradigm rests on the hypothesis that generating chain-of-thought before token prediction leads to better predictions than direct prediction. Section 4.1 validates this for mathematical text (Table 1 shows RPT-14B at 45.11% vs. standard NTP at 41.90% on easy splits). However, this assumption may not hold for all token types. Highly predictable tokens (e.g., punctuation after common phrases) may not benefit from reasoning, which is why the entropy-based filtering in Section 3.3 is necessary. The paper does not ablate this assumption—there is no "no reasoning" control within the RPT framework itself.

**Assumption 3: Reasoning-capable initialization is sufficient.** Section 3.3 states: "R1-Distill-Qwen-14B serves as a good starting point for reinforcement learning due to its basic reasoning capabilities." This assumes the base model already possesses enough reasoning ability to generate meaningful chain-of-thought sequences. Section 6 explicitly acknowledges this as a limitation:

> Furthermore, RPT training is initialized from a reasoning model; investigating RPT training from a standard base language model would provide further insights into its foundational impact.

If RPT requires a reasoning-capable starting point, its utility as a pre-training method is constrained—you cannot apply it to models that lack basic reasoning. This creates a chicken-and-egg problem for deploying RPT on new model architectures.

**Assumption 4: On-policy RL is stable without KL regularization.** Appendix B shows the KL penalty is set to 0, following "exact on-policy reinforcement learning [HDW+25]" settings. This assumes the model will not diverge significantly from its initial behavior during training. While this worked for 1,000 training steps, longer training runs or larger learning rates could destabilize the model. Standard RLHF typically uses KL penalties to prevent catastrophic forgetting—RPT's zero KL penalty is a design choice that trades stability for optimization freedom.

### 6.2 Unaddressed Scenarios and Edge Cases

Several important scenarios are not evaluated in the paper:

**General-domain text.** All experiments use mathematical documents (OmniMATH for pre-training, Skywork-OR1 for fine-tuning). Section 6 acknowledges:

> The current pre-training corpus predominantly consists of mathematical documents; future work will explore its efficacy on broader, general-domain text.

This is a significant gap. Mathematical text has clear logical structure and objective correctness criteria. News articles, fiction, code comments, and conversational data have different properties:
- Multiple valid continuations are common
- Correctness is often subjective or context-dependent
- Reasoning patterns may differ (e.g., narrative flow vs. logical deduction)

The zero-shot evaluation on MMLU-Pro and SuperGPQA (Table 3) shows RPT-14B performs well on general-domain benchmarks, but these are evaluation tasks, not pre-training domains. The model was still trained on mathematical text. Whether RPT trained on web-scale general text would show similar benefits remains untested.

**Non-reasoning token types.** The entropy-based filtering in Section 3.3 prioritizes "challenging tokens that require greater computational effort to predict." However, the paper does not analyze which token types benefit most from reasoning. For example:
- Function words (the, a, is) vs. content words (mathematical terms, named entities)
- Tokens within sentences vs. sentence boundaries
- Tokens in quoted text vs. narrative text

Without this analysis, it is unclear whether RPT's compute expenditure is optimally allocated. The filtering threshold choices (0.5, 1.0, 1.5 for easy/medium/hard in Section 4.1) are heuristic, not derived from first principles.

**Long-context reasoning.** The training uses 8k token sequences (Section 3.3), but the reasoning is localized to next-token prediction. The paper does not evaluate whether RPT improves long-range dependency modeling—whether reasoning about token $t$ helps the model understand relationships between tokens at positions $t-100$ and $t+100$. This matters for tasks requiring document-level understanding.

**Multi-token predictions.** The reward function supports predictions spanning multiple tokens (Section 3.2: "prefix matching reward, which supports verifying predictions that span multiple tokens"). However, all reported results focus on single-token accuracy. The paper does not evaluate whether RPT improves multi-token completion quality, which is more relevant for practical generation tasks.

### 6.3 Computational and Data Constraints

RPT introduces significant resource requirements compared to standard pre-training:

**Compute overhead per token.** For each token position, RPT samples $G=8$ reasoning trajectories (Section 3.3). Each trajectory includes:
- A chain-of-thought sequence (variable length, up to 8192 tokens per Appendix B)
- A final prediction

This is substantially more expensive than standard next-token prediction, which requires one forward pass per token. The paper does not quantify this overhead in FLOPs or training time. Figure 5 shows scaling with "RL Compute (FLOPs)" on the x-axis, but does not compare this to equivalent compute spent on standard pre-training. The critical question—does RPT deliver better accuracy per FLOP than standard pre-training?—is not answered.

**Memory requirements.** On-policy RL with $G=8$ rollouts requires storing all trajectories for gradient computation. Appendix B shows a batch size of 256 questions with 8 rollouts each, meaning 2,046 responses must be stored per batch. For 8k token sequences, this imposes significant memory pressure. The paper mentions using the verl library [SZY+24] and vllm for inference but does not discuss memory optimization strategies or hardware requirements beyond acknowledging MI300 GPUs in the Acknowledgements.

**Data efficiency.** The main experiment uses 4,428 mathematical problems (OmniMATH) for 1,000 training steps (Section 3.3). This is orders of magnitude smaller than standard pre-training corpora (typically trillions of tokens). Section 6 states future work will "scale up the training corpus, including data size, and domain coverage." Until this is demonstrated, RPT's data efficiency at web-scale is unknown. The scaling curves in Figure 5 cover only 100–1,200 steps—a narrow range compared to standard pre-training scaling studies that span multiple orders of magnitude.

**Inference latency.** While RPT is a training-time method, the resulting model is expected to use reasoning at inference time (Table 3 evaluates "Reasoning mode"). This means RPT-trained models will have higher inference latency than standard models, as they must generate chain-of-thought before producing outputs. The paper does not quantify this latency cost or discuss whether the accuracy gains justify the inference overhead for production deployments.

### 6.4 Weaknesses in Experimental Validation

Several aspects of the experimental design limit confidence in the results:

**Missing ablations.** Key design choices are not ablated:
- **Data filtering**: No "no filtering" baseline is reported. The entropy-based filtering is motivated but not validated—does it actually improve efficiency compared to training on all tokens?
- **Number of rollouts ($G$)**: The paper uses $G=8$ throughout but does not test whether fewer rollouts (e.g., $G=4$) would achieve similar performance with less compute, or whether more rollouts ($G=16$) would yield additional gains.
- **KL penalty**: Zero KL penalty is used, but no comparison to non-zero KL values is provided. This is a standard regularization technique in RLHF—its absence should be justified empirically.

**Prompt template sensitivity.** Appendix D (Table 8) shows significant variation across prompt templates:

| Template | Random@1 | Pass@8 |
|----------|----------|--------|
| v0 (used in main experiments) | 3.0% | 8.5% |
| v6 (best) | 6.0% | 19.0% |

The main experiments use v0, but v6 achieves more than 2× better Pass@8. The paper states: "We leave prompt engineering based on other template variants for future work, which tends to improve the final performance." This means reported results are conservative—but it also means RPT's performance is sensitive to prompt engineering, which introduces variability and reduces reproducibility. Different prompt templates could lead to different conclusions about RPT's effectiveness.

**Result discrepancy.** Section 4.4 claims RPT-14B surpasses R1-Distill-Qwen-32B by "approximately 22 points on MMLU-Pro." However, Table 3 shows:
- RPT-14B (Reasoning): 71.1%
- R1-Distill-Qwen-32B (Standard NTP): 56.5%
- Difference: 14.6 percentage points

This 7.4-point discrepancy is not explained. It may be a calculation error, or it may refer to a different comparison not shown in the table. Either way, it undermines confidence in the reported numbers.

**Reward design robustness.** Appendix A states alternative reward designs "generally achieved performance comparable to the prefix matching reward" but provides no exact numbers. Without quantitative comparison, it is unclear whether "comparable" means within 1% or within 10%. This limits understanding of how sensitive RPT is to reward design choices.

### 6.5 Open Questions

Several important questions remain unanswered:

**Question 1: Does RPT scale to larger models?** All experiments use 14B parameter models (with one 32B comparison in Figure 4). Section 6 states: "Our experiments are primarily conducted using a 14B parameter model." Whether RPT's benefits persist at 70B, 100B, or larger scales is unknown. Scaling laws may change at different model sizes—the $R^2 > 0.989$ fits in Figure 5 cover only a narrow compute range.

**Question 2: Can RPT train from scratch?** As noted in Section 6, RPT is initialized from a reasoning-capable model. Whether RPT can train a standard base language model from scratch (without prior reasoning capabilities) is untested. This matters for understanding whether RPT creates reasoning capabilities or merely refines existing ones.

**Question 3: What is the optimal compute allocation?** RPT spends more compute per token during training. The optimal balance between:
- Number of rollouts ($G$)
- Reasoning sequence length
- Training steps
- Model size

is not explored. Different allocations may be optimal for different domains or tasks.

**Question 4: Does RPT reduce or increase reward hacking risk?** The Introduction claims rule-based rewards "inherently minimizes the risk of reward hacking." However, the model could still learn to game the system—for example, by generating reasoning that appears deliberative but is actually shallow, or by exploiting tokenization quirks in the byte-level matching. No adversarial analysis is provided.

**Question 5: How does RPT interact with other training techniques?** The paper does not explore combinations with:
- Curriculum learning (progressively harder tokens)
- Multi-task training (combining RPT with other objectives)
- Hybrid reasoning [JWH+25] (mentioned in Section 6 as future work)

Understanding these interactions is essential for integrating RPT into production training pipelines.

### 6.6 Summary of Trade-offs

| Aspect | Benefit | Cost/Limitation |
|--------|---------|-----------------|
| **Reward signal** | Intrinsic, no annotations needed | Assumes corpus tokens are correct; no handling of ambiguity |
| **Reasoning encouragement** | Deeper understanding, better generalization | Higher compute per token; inference latency |
| **Objective alignment** | Better RL fine-tuning transfer | Requires reasoning-capable initialization |
| **Scaling properties** | Predictable improvement with compute | Tested on narrow compute range; domain-limited |
| **Data efficiency** | Uses unannotated text | Experiments on small mathematical corpus only |
| **Reward design** | Robust to variations (Appendix A) | Exact robustness not quantified |

**Bottom line**: RPT is a promising paradigm with strong experimental support for its core claims within the tested domain. However, its general-purpose applicability, compute efficiency, and scalability to larger models and broader corpora remain open questions. The method is best suited for domains with objective correctness criteria (mathematics, code, factual QA) where the compute overhead is justified by accuracy gains. For general web-text pre-training, further validation is needed before RPT can be recommended as a replacement for standard next-token prediction.

## 7. Implications and Future Directions

This section examines how Reinforcement Pre-Training (RPT) reshapes the research landscape, what new avenues it opens for investigation, where it can be practically deployed, and how practitioners should decide whether to adopt this paradigm. The analysis draws from the paper's stated future work (Section 6), experimental findings (Section 4), and identified limitations (Section 6 of this analysis).

### 7.1 Landscape Shift: Redefining the Pre-Training Objective

RPT represents a conceptual pivot in how the field thinks about language model pre-training. The implications extend beyond incremental performance gains to fundamental questions about what pre-training should optimize.

**From Pattern Matching to Reasoning Optimization**

Standard pre-training treats language modeling as statistical pattern learning. The objective maximizes log-likelihood:

$$J_{\text{NTP}}(\theta) = \sum_{t=1}^T \log P(x_t | x_0, x_1, \ldots, x_{t-1}; \theta)$$

RPT reframes this as a reasoning optimization problem where the model must generate chain-of-thought $c_t$ before predicting $y_t$, with rewards based on correctness:

$$J_{\text{RPT}}(\theta) = \mathbb{E}_{(x_{&lt;t}, x_{\geq t}) \sim \mathcal{D}, \{o^i_t\}^G_{i=1} \sim \pi_\theta(\cdot|x_{&lt;t})} \left[r^i_t\right]$$

This shift matters because it changes what capabilities are prioritized during the most compute-intensive phase of model development. As Table 2 demonstrates, models trained with standard next-token prediction on the same corpus as RPT experienced catastrophic capability degradation (dropping from 51.2% to 10.7% before RL fine-tuning). This suggests the pre-training objective itself shapes which reasoning patterns are preserved or erased.

**Blurring Training-Time and Test-Time Scaling**

The scaling literature has traditionally distinguished two paradigms [ZLS+25a, JKL+24]:

| Scaling Type | When Compute Is Spent | Typical Approach |
|--------------|----------------------|------------------|
| Training-time | During pre-training | Larger models, more tokens |
| Test-time | During inference | Chain-of-thought, multiple samples |

RPT collapses this distinction by applying test-time reasoning patterns during training. As stated in the Introduction:

> The internal reasoning process during pre-training effectively allows the model to allocate more "thought" or computational effort to each prediction step, akin to a form of inference-time scaling applied at training time for each token.

This creates a third axis for capability improvement: **training-time inference scaling**. Instead of asking only "should we make the model bigger?" practitioners can now ask "should we spend more compute per token during training to build stronger reasoning foundations?"

**Democratizing RL at Pre-Training Scale**

Prior RL approaches for LLMs faced a fundamental scalability barrier:

- **RLHF** [OWJ+22]: Requires human preference annotations (expensive, does not scale)
- **RLVR** [LMP+25]: Requires question-answer pairs with verifiable answers (domain-specific, limited data)

RPT eliminates the annotation bottleneck by using the pre-training corpus itself as the reward source. Every token position becomes a verifiable RL problem because the ground-truth next token provides the correctness signal. This transforms trillions of tokens from unannotated web text into a massive dataset for general-purpose RL without external annotations.

The scaling evidence in Figure 5 supports this scalability claim, showing coefficients of determination $R^2 > 0.989$ across all difficulty levels, indicating predictable improvement with increased compute comparable to standard pre-training scaling laws [KMH+20, HBM+22b].

### 7.2 Research Directions Enabled

RPT opens several concrete research avenues that were previously inaccessible due to the RL scalability bottleneck.

**Direction 1: Scaling RPT to Web-Scale Corpora**

The current experiments use 4,428 mathematical problems (OmniMATH) for 1,000 training steps (Section 3.3). Section 6 explicitly identifies scaling as a priority:

> We would like to scale up the training corpus, including data size, and domain coverage. Large-scale general Internet data can be utilized during reinforcement pre-training.

Key research questions include:
- Does RPT maintain favorable scaling properties ($R^2 > 0.98$) when trained on trillions of tokens instead of thousands of problems?
- How does performance vary across domains (code, news, conversations, scientific text)?
- What data filtering strategies optimize the reasoning/compute trade-off at web scale?

The entropy-based filtering in Section 3.3 (using a 1.5B proxy model to identify high-entropy tokens) provides a starting point, but web-scale deployment would require more efficient filtering mechanisms.

**Direction 2: Training from Standard Base Models**

Current RPT experiments initialize from Deepseek-R1-Distill-Qwen-14B, which already possesses "basic reasoning capabilities" (Section 3.3). Section 6 acknowledges this limitation:

> Investigating RPT training from a standard base language model would provide further insights into its foundational impact.

This matters for understanding whether RPT:
- **Creates** reasoning capabilities de novo, or
- **Refines** existing reasoning capabilities

Research should compare RPT initialized from:
1. Standard base models (e.g., Qwen2.5-14B without reasoning distillation)
2. Reasoning-capable models (current approach)
3. Random initialization (theoretical, likely impractical)

If RPT works from standard base models, it becomes a true pre-training method rather than a post-training enhancement.

**Direction 3: Hybrid Reasoning Integration**

Section 6 mentions integrating hybrid thinking [JWH+25] with RPT:

> We are interested in integrating hybrid thinking with RPT to enable fine-grained adaptive thinking by adaptively triggering next-token reasoning.

The current approach applies reasoning to every token position (after entropy filtering). Hybrid reasoning would enable the model to decide **when** to reason versus when to predict directly. This could dramatically improve compute efficiency by reserving reasoning trajectories for genuinely uncertain predictions.

Research questions include:
- Can the model learn to predict which tokens require reasoning?
- Should reasoning be triggered by entropy thresholds, semantic cues, or learned policies?
- How does hybrid RPT compare to always-reason or never-reason baselines?

**Direction 4: Reward Design Optimization**

Appendix A explores three alternative reward designs (first-token matching, dense reward, conditional dense reward) and finds they achieve "performance comparable to the prefix matching reward." However, exact numbers are not provided, limiting understanding of sensitivity.

Future work should systematically evaluate:
- **Reward granularity**: Binary (0/1) vs. dense (probability-based) vs. hybrid
- **Matching scope**: First token only vs. prefix matching vs. multi-token completion
- **Negative rewards**: Should incorrect predictions receive penalties, or only zero reward?
- **Reward shaping**: Can intermediate reasoning steps receive partial credit?

The robustness finding in Appendix A suggests the core framework is not overly sensitive to reward design, but optimization could yield additional gains.

**Direction 5: Reasoning Pattern Engineering**

Section 4.5 analyzes reasoning patterns and finds RPT-14B uses hypothesis patterns 161.8% more and deduction patterns 26.2% more than the baseline (Figure 6). This suggests the RL objective shapes **how** the model reasons, not just **whether** it reasons.

Future research could:
- Identify which reasoning patterns correlate most strongly with accuracy gains
- Design reward bonuses for specific reasoning behaviors (e.g., self-correction, alternative generation)
- Analyze whether different domains benefit from different reasoning patterns (mathematical deduction vs. narrative coherence)

Table 4's case study shows the model engages in deliberative processes (considering alternatives, reflecting on structural cues), but systematic pattern engineering could amplify beneficial behaviors.

**Direction 6: Establishing RPT Scaling Laws**

Section 4.2 fits a power-law relationship between training compute $C$ and accuracy $P(C)$:

$$P(C) = \frac{A}{C^\alpha} + P^*$$

With $R^2 > 0.989$ across difficulty levels, this provides initial evidence for RPT scaling. However, Section 6 notes the tested range is narrow (100 to 1,200 steps on 4,428 problems).

Future work should:
- Extend the compute range by orders of magnitude
- Fit scaling laws with respect to model size (not just compute)
- Fit scaling laws with respect to data size (not just steps)
- Compare RPT scaling exponents to standard pre-training exponents

Establishing reliable scaling laws would guide investment decisions and help predict RPT performance at larger scales.

### 7.3 Practical Applications and Use Cases

RPT's characteristics make it suitable for specific application domains while less appropriate for others. The following guidance is based on experimental evidence and identified limitations.

**High-Fit Applications**

| Domain | Rationale | Evidence |
|--------|-----------|----------|
| **Mathematical reasoning** | Objective correctness, logical structure | OmniMATH experiments (Table 1, Table 3) |
| **Code generation** | Verifiable execution, clear syntax rules | Similar to math: verifiable rewards available |
| **Scientific QA** | Factual correctness can be verified | SuperGPQA performance (39.0% vs. 37.2% for 32B baseline) |
| **Formal logic / theorem proving** | Proof verification is deterministic | Aligns with RLVR-style verifiable rewards |
| **RL fine-tuning foundations** | Objective alignment between pre/post-training | Table 2 shows 5.1 point advantage before RL |

These domains share a key property: **ground-truth tokens are reliably correct**. Mathematical text, code, and factual QA have objective correctness criteria that align with RPT's binary reward structure.

**Moderate-Fit Applications**

| Domain | Rationale | Caveats |
|--------|-----------|---------|
| **General knowledge QA** | Factual answers exist but ambiguity possible | MMLU-Pro shows gains (71.1% vs. 68.9%) but corpus was math-focused |
| **Technical documentation** | Structured text with clear continuations | May benefit from reasoning about API usage, examples |
| **Educational content** | Pedagogical structure benefits from reasoning | OmniMATH is educational; transfer plausible |

These applications may benefit from RPT but require validation on domain-specific corpora. The current experiments are math-focused, so general-domain performance is inferred from zero-shot benchmarks rather than direct pre-training evidence.

**Low-Fit Applications**

| Domain | Rationale | Concerns |
|--------|-----------|----------|
| **Creative writing** | Multiple valid continuations common | Binary reward penalizes valid alternatives |
| **Conversational AI** | Context-dependent appropriateness | Ground-truth tokens may not represent optimal responses |
| **Subjective tasks** | No objective correctness criterion | RPT's reward mechanism requires verifiable signals |
| **Low-latency inference** | RPT models reason at inference time | Chain-of-thought generation increases latency |

These domains face fundamental mismatches with RPT's design. Creative and conversational tasks often have multiple valid continuations, making binary correctness rewards inappropriate. Additionally, RPT-trained models are expected to use reasoning at inference time (Table 3 evaluates "Reasoning mode"), which increases latency compared to standard next-token prediction.

**Decision Framework for Practitioners**

When considering RPT adoption, practitioners should evaluate:

1. **Does my domain have objective correctness criteria?**
   - Yes → RPT is suitable
   - No → Consider standard pre-training or alternative RL approaches

2. **Is inference latency a critical constraint?**
   - No → RPT's reasoning mode is acceptable
   - Yes → Standard pre-training may be preferable

3. **Do I plan RL fine-tuning on verifiable tasks?**
   - Yes → RPT provides better foundation (Table 2 evidence)
   - No → Benefit is limited to zero-shot performance gains

4. **Do I have compute budget for G=8 rollouts per token?**
   - Yes → RPT is feasible
   - No → Consider standard pre-training or reduce G (untested)

5. **Is my base model reasoning-capable?**
   - Yes → RPT can refine existing capabilities
   - No → Wait for research on training from standard base models

### 7.4 Reproduction and Integration Guidance

For researchers and practitioners interested in implementing RPT, the following guidance synthesizes implementation details from Section 3, Appendix B, and identified limitations.

**Minimum Viable Configuration**

Based on the main experiments (Section 3.3, Appendix B):

| Component | Recommended Setting | Rationale |
|-----------|--------------------|-----------|
| Base model | Reasoning-capable (e.g., R1-Distill-Qwen-14B) | Current experiments require reasoning initialization |
| Rollout count ($G$) | 8 | Used in all main experiments |
| Learning rate | $1 \times 10^{-6}$ | Appendix B |
| Batch size | 256 questions | Appendix B |
| Training length | 8k tokens | Section 3.3 |
| Temperature | 0.8 (rollout), 0.6 (evaluation) | Section 3.3, Section 4.3 |
| KL penalty | 0 | Appendix B (exact on-policy RL) |
| Reward type | Prefix matching | Section 3.2 (most robust per Appendix A) |

**Critical Implementation Details**

1. **Prediction extraction**: "From each response, we directly extract the full sequence inside the last `\boxed{}` following the special token `</think>` as the model prediction" (Section 3.3). This parsing logic is essential for reward computation.

2. **Byte-level matching**: The reward function uses byte sequences, not token IDs: "Let $x_{\geq t}$ and $y^i_t$ denote the byte sequences of the ground-truth completion sequence and the prediction $y^i_t$, respectively" (Section 3.2). This handles out-of-vocabulary tokens and tokenization boundary issues.

3. **Entropy filtering**: Use a small proxy model (e.g., 1.5B) to calculate entropy on top-16 next tokens, then filter low-entropy positions (Section 3.3). This focuses compute on tokens requiring reasoning.

4. **Dynamic sampling**: "Starting from 500 steps, we utilize dynamic sampling to boost training efficiency" (Section 3.3). This likely reduces rollouts for easy examples as training progresses.

**Known Sensitivities**

| Factor | Sensitivity | Guidance |
|--------|-------------|----------|
| Prompt template | High (v6 achieves 19.0% Pass@8 vs. v0's 8.5%) | Experiment with templates; Appendix D Table 8 provides variants |
| Reward design | Low (Appendix A reports comparable performance) | Start with prefix matching; alternatives are viable |
| Base model reasoning capability | High (untested from standard base models) | Use reasoning-capable initialization for now |
| Data domain | Unknown (only math tested) | Validate on target domain before large-scale deployment |

**Integration with Existing Pipelines**

RPT can be integrated into training pipelines in several ways:

1. **Pre-training replacement**: Use RPT instead of standard next-token prediction for the entire pre-training phase. This maximizes benefit but requires the most compute.

2. **Mid-training enhancement**: Apply RPT for a portion of pre-training (e.g., after initial standard pre-training). This balances compute cost with reasoning benefits.

3. **Domain-specific pre-training**: Use RPT for domains with objective correctness (math, code) while using standard pre-training for general text. This hybrid approach leverages RPT where it fits best.

4. **Fine-tuning foundation**: Use RPT specifically to create better starting points for RL fine-tuning. Table 2 shows RPT provides 5.1 point advantage before any fine-tuning.

**Reproduction Checklist**

Researchers attempting to reproduce or extend RPT should verify:

- [ ] Base model has basic reasoning capabilities (can generate chain-of-thought)
- [ ] Reward function implements byte-level prefix matching (Section 3.2, Equation 3)
- [ ] Prediction extraction parses `\boxed{}` content after `</think>` token
- [ ] Entropy filtering uses proxy model to identify high-entropy positions
- [ ] GRPO algorithm is used for on-policy RL updates (Section 3.3)
- [ ] Prompt template is clearly specified (Appendix D shows significant variation)
- [ ] Evaluation distinguishes between standard NTP mode and reasoning mode

**Open Implementation Questions**

The paper leaves several implementation details unspecified:

- **Dynamic sampling algorithm**: Section 3.3 mentions it but does not describe the mechanism
- **Memory optimization**: Storing $G=8$ rollouts for 8k sequences requires significant memory; strategies are not detailed
- **Gradient accumulation**: Appendix B shows batch size 256 but does not specify gradient accumulation steps
- **Hardware requirements**: Acknowledgements mention MI300 GPUs but do not specify cluster size or training duration

Researchers should contact the authors or monitor future releases for these details.

### 7.5 Summary: Positioning RPT in the Research Landscape

RPT occupies a unique position in the pre-training paradigm space:

| Paradigm | Data Requirement | Reward Source | Reasoning Encouraged | Scalability |
|----------|-----------------|---------------|---------------------|-------------|
| Standard NTP | Unannotated text | Implicit (likelihood) | No | High |
| RLHF | Human preferences | Learned reward model | Task-specific | Limited |
| RLVR | QA pairs | Rule-based verifier | Task-specific | Limited |
| Quiet-Star | Unannotated text | Helpfulness-based | Yes | Moderate (hackable) |
| **RPT** | **Unannotated text** | **Ground-truth token** | **Yes** | **High** |

The key differentiator is that RPT achieves RL's benefits (explicit reasoning, reward optimization) at standard pre-training's scale (unannotated web text) without Quiet-Star's vulnerability to reward hacking.

**When to Adopt RPT**

Adopt RPT when:
- Your domain has objective correctness criteria (math, code, factual QA)
- You plan RL fine-tuning on verifiable tasks
- You have compute budget for multi-trajectory training
- Inference latency is not a primary constraint

Stick with standard pre-training when:
- Your domain has subjective or multiple-valid-continuation scenarios
- Inference latency is critical
- Compute budget is constrained
- You lack a reasoning-capable base model for initialization

**The Path Forward**

RPT is an early but promising demonstration that RL can scale to pre-training without external annotations. The experimental evidence (Table 1, Table 2, Table 3, Figure 5) supports the core claims within the tested domain. However, Section 6's acknowledged limitations—mathematical corpus only, reasoning-capable initialization, narrow compute range—mean the paradigm requires further validation before becoming a standard recommendation.

The most impactful next steps are:
1. Scale RPT to web-text corpora (trillions of tokens, not thousands of problems)
2. Demonstrate RPT training from standard base models (not just reasoning-capable initialization)
3. Establish comprehensive scaling laws (model size, data size, compute)
4. Validate on non-mathematical domains (code, science, general text)

If these directions confirm the initial findings, RPT could become a foundational component of future language model training pipelines, particularly for applications requiring strong reasoning capabilities and RL fine-tuning compatibility.
