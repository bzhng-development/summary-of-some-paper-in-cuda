## 1. Executive Summary

This paper presents Relax, a compiler abstraction that solves the challenge of optimizing end-to-end dynamic machine learning workloads by unifying computational graphs, tensor programs, and external libraries into a single cross-level representation with first-class symbolic shape annotations. This design enables global dynamic shape tracking and composable optimizations, such as static memory planning and CUDA Graph offloading for dynamic models, delivering up to a 27% reduction in decode token latency for Llama3-8B on NVIDIA RTX 4090 compared to state-of-the-art systems. Crucially, Relax facilitates ahead-of-time compilation for emerging large language models across diverse backends previously unsupported by established frameworks, including mobile phones (iPhone 14 Pro, Samsung S23), embedded devices, and web browsers via WebGPU.

## 2. Context and Motivation

### The Dynamic Shape Challenge in Modern Machine Learning

Modern machine learning workloads, particularly large language models (LLMs), fundamentally depend on **dynamic tensor shapes** — dimensions that cannot be determined at compile time and instead depend on runtime values. Consider a language model processing user input: the sequence length varies with each message, the batch size changes based on concurrent requests, and the KV-cache context grows as conversation history accumulates. These are not edge cases; they are the core operating conditions of production ML systems.

The paper identifies this as a critical tension:

> Dynamic shapes contain variables that may depend on program values, which makes it more difficult to perform crucial optimizations like static memory planning.

To understand why this matters, consider what compilers typically want to do. When tensor shapes are known statically (e.g., every tensor is exactly $1 \times 512 \times 768$), the compiler can:
- Pre-allocate all memory upfront in a fixed layout
- Fuse operators knowing their exact output dimensions
- Generate specialized kernel code tuned for those dimensions
- Capture execution graphs for replay (e.g., CUDA Graphs)

When shapes are dynamic, each of these optimizations becomes uncertain. If you do not know that tensor $A$ has shape $(n, 512)$ and tensor $B$ has shape $(n, 256)$ where $n$ is the same variable, you cannot prove they can share memory buffers. You cannot fuse operators across function boundaries because you lose track of shape relations. You cannot use CUDA Graphs because the driver requires statically allocated memory.

### Why Existing Compiler Abstractions Fall Short

The paper identifies a structural problem in how traditional ML compilers organize their intermediate representations (IRs). Most compilers use **multiple levels of abstraction** with **single-shot lowering** between them:

| Abstraction Level | Purpose | Typical Representation |
|-------------------|---------|------------------------|
| Computational Graph | Global computation rewrites, operator fusion | Dataflow graph of high-level operators (matmul, reshape) |
| Tensor Program | Fine-grained kernel optimizations | Loop-level code with indexed buffer accesses |
| Operator Libraries | Vendor-optimized routines | External function calls (cuDNN, CUTLASS) |

The critical limitation is that **dynamic shape handling occurs independently at each level**, without tracking relationships across level boundaries. The paper explains:

> Relay [35] and IREE [27] handles dynamic shapes in computational graphs through "unknown" annotations and do not track relations between dynamic shapes.

This "unknown" annotation approach erases valuable information. If a dimension is marked as `unknown` rather than as a symbolic variable $n$, the compiler cannot deduce that two tensors both depend on $n$ and therefore have related sizes. Consider this concrete example from the paper (Figure 3):

```python
# With "unknown" annotations (prior approach)
def any_shape_fn(x: Tensor((?, 2, 2), "f32")):
    lv0: Tensor((?, 4), "f32") = reshape(x, (n, 4))  # n obtained at runtime
    lv1: Tensor((?,), "f32") = flatten(lv0)           # Shape relation lost
```

versus Relax's approach:

```python
# With first-class symbolic shapes (Relax)
def symbolic_shape_fn(x: Tensor(("n", 2, 2), "f32")):
    n = sym_var()
    lv0: Tensor((n, 4), "f32") = reshape(x, shape(n, 4))
    lv1: Tensor((n * 4,), "f32") = flatten(lv0)  # Compiler knows output is 4n elements
```

In the second case, the compiler can prove that `lv1` contains exactly $4n$ elements — the same as the input — enabling buffer reuse optimizations that are impossible with `?` annotations.

The PyTorch compiler takes a different approach using **just-in-time (JIT) graph tracing**, which eliminates the need for shape tracking across function boundaries by tracing at runtime. However, the paper notes this "limits its applications on emerging platforms with constrained environments, such as mobile and WebGPU." JIT tracing requires runtime infrastructure that may not be available on embedded devices or browsers, and it prevents ahead-of-time (AOT) compilation — a requirement for many deployment scenarios.

### The Deployment Gap for Emerging Platforms

Beyond the technical compiler challenges, there is a practical deployment problem. The success of LLMs has "led to growing demand for their deployment across a diverse set of backend environments, including servers, personal computers, vehicles, and mobile devices." However, existing frameworks create a fragmentation problem:

- **Server-optimized systems** (vLLM, TensorRT-LLM) assume powerful GPUs with abundant memory and do not target mobile or web backends
- **Mobile-optimized systems** (llama.cpp) achieve good performance on Apple GPUs but "perform less effectively on NVIDIA GPUs" and lack GPU support on Android devices
- **General ML frameworks** (PyTorch, TensorFlow) require significant engineering effort to support each new hardware backend and often lack dynamic shape optimizations

The paper's evaluation section reveals concrete gaps: PyTorch compile mode and vLLM lack Apple GPU support entirely. llama.cpp "only utilizes CPU due to lack of kernels for Android GPUs." This means model developers must maintain multiple code paths or accept suboptimal performance on certain platforms.

The memory constraint issue is particularly acute on emerging platforms. The paper notes that "without memory planning that pre-allocates all needed memory and keeps it within the budget, these models are not even runnable on some of the environments due to memory constraints." Dynamic allocation at runtime can fragment memory or exceed limits on devices with 8–16 GB of unified memory (mobile phones, embedded devices).

### How Relax Positions Itself

Relax positions itself as a **holistic ahead-of-time compiler abstraction** that addresses both the technical and deployment gaps simultaneously. Its key differentiators relative to prior work are:

1. **Cross-level abstraction**: Unlike traditional compilers that treat tensor programs and libraries as opaque foreign functions, Relax "encapsulates computational graphs, loop-level tensor programs and external libraries calls in a single representation." This enables optimizations that span abstraction boundaries — for example, fusing a graph-level operator with a tensor program that was previously opaque.

2. **First-class symbolic shapes**: Rather than using `unknown` annotations or runtime tracing, Relax introduces symbolic variables that track shape relations "globally across tensor operators, subgraph function calls and foreign function calls." This preserves information that enables static analyses even for dynamic models.

3. **AOT compilation for emerging backends**: Relax is designed for ahead-of-time compilation, producing deployable modules that do not require JIT infrastructure. This enables "deployment of emerging models to a broader set of emerging environments, including mobile phones, embedded devices, and web browsers."

4. **Composable optimizations**: The cross-level abstraction enables optimization patterns that are impossible with single-shot lowering, including:
   - Dynamic shape–aware operator fusion across graph and tensor program levels
   - Cross-level tensor program workspace lifting (moving allocations from tensor programs to graph level for global memory planning)
   - CUDA Graph offloading for dynamic models (via static memory planning with symbolic upper bounds)

The paper explicitly positions Relax as complementary to existing work rather than a replacement: "Relax complements the PyTorch compiler by abstracting and globally tracks cross-function symbolic shape for full programs, enabling AOT compilation and holistic deployment onto emerging platforms." It can serve as a backend for frameworks like PyTorch or JAX while extending their deployment reach.

This positioning is significant because it acknowledges that no single compiler solves all problems. Instead, Relax addresses a specific gap: **full-program symbolic shape tracking with cross-level optimizations for AOT deployment on diverse backends**. Prior systems excel at individual pieces (JIT compilation, tensor program optimization, library dispatch) but lack the unified abstraction needed to compose these techniques for dynamic models on emerging platforms.

## 3. Technical Approach

This is a systems paper that presents a compiler intermediate representation (IR) design; the core idea is to unify multiple abstraction levels (computational graphs, tensor programs, and external libraries) into a single representation while tracking symbolic shape relationships globally across all function boundaries.

### 3.1 Reader orientation (approachable technical breakdown)

Relax is a compiler intermediate representation that lets machine learning models with variable-sized inputs be optimized as if they had fixed sizes, by tracking mathematical relationships between dimensions using symbolic variables instead of marking them as "unknown." It solves the problem that traditional compilers lose shape information when crossing between high-level graph operations and low-level kernel code, which prevents crucial optimizations like memory reuse and kernel fusion for dynamic models.

### 3.2 Big-picture architecture (diagram in words)

The Relax system consists of four major components that process a model from high-level description to deployable code. First, the **Relax IR** serves as the unified representation that encapsulates computational graph operators, tensor program functions, and external library calls in a single data structure with symbolic shape annotations attached to every tensor value. Second, the **Shape Deduction Engine** traverses the IR and propagates symbolic shape relationships forward through operations, proving equalities between expressions like $n \times 4$ and $4 \times n$ to enable optimizations. Third, the **Cross-Level Optimization Passes** apply transformations that span abstraction boundaries, including operator fusion, memory planning, and workspace lifting, all while preserving symbolic shape information. Fourth, the **Code Generation Backend** lowers the optimized IR into runtime executable code, converting symbolic variables into runtime integer tensors that are evaluated before kernel launches. Information flows unidirectionally through these components: a model enters as Relax IR with symbolic annotations, gets transformed by optimization passes that query the shape deduction engine, and exits as a compiled module containing both virtual machine instructions for graph-level control flow and GPU kernels for tensor computations.

### 3.3 Roadmap for the deep dive

- We start with the **language constructs** (annotations, dataflow blocks, function calls) because these form the vocabulary everything else builds upon — you cannot understand shape tracking without knowing what gets annotated.
- We then explain **first-class symbolic shapes** and how they differ from prior "unknown" annotation approaches, since this is the key innovation that enables downstream optimizations.
- We describe the **cross-level abstraction mechanisms** (`call_tir` and `call_dps_library`) that bridge graph-level and tensor-program-level code, because these enable optimizations to span traditional compiler boundaries.
- We walk through the **shape deduction algorithm** and its three design principles, since this explains how symbolic relationships are actually computed and maintained during transformations.
- We detail the **optimization passes** (fusion, memory planning, workspace lifting, CUDA Graph offloading) in the order they appear in the compilation pipeline, showing how each depends on symbolic shape information.
- We conclude with the **end-to-end lowering pipeline** that produces deployable modules, explaining how symbolic variables become runtime values and how the final executable is structured.

### 3.4 Detailed, sentence-based technical breakdown

#### Core Language Constructs: Annotations, Dataflow, and Function Calls

Relax is an imperative compiler abstraction with first-class functions that operates on tensors at the graph level while allowing calls into lower-level tensor programs and external libraries. Every value in Relax carries an **annotation** that conveys structural information similar to a static type system, and these annotations form the foundation for symbolic shape tracking. Table 1 in the paper enumerates the annotation types: `Object` represents any runtime value, `Shape` represents symbolic shape values (e.g., `Shape([n, 4])` for a two-dimensional shape with symbolic first dimension), `Tensor` represents tensors with optional shape and dtype information (e.g., `Tensor((n, 4), "f32")`), `Tuple` represents heterogeneous collections, and `Callable` represents function signatures with parameter and return type annotations. The annotation syntax is designed to be embeddable in Python AST, and symbolic expressions are quoted into strings (e.g., `"n*4"`) in function signatures when symbolic variables are not yet declared, though these strings are read as normal symbolic expressions by the compiler.

Relax programs are organized into **dataflow blocks** that demarcate side-effect-free regions without control flow, meaning they contain straight-line sequences of pure operations. This design simplifies program transformations because compiler passes can safely remove unused operators during dead code elimination without worrying about visible behavior changes from removing operations with side effects. Within dataflow blocks, Relax supports three types of **function calls**: calls within the same abstraction level (one graph-level function invoking another), calls across levels to foreign tensor program functions via `call_tir`, and calls to external library functions via `call_dps_library`. The paper uses TensorIR as the loop-level tensor program abstraction for `call_tir`, though the design principle applies to other loop-level abstractions.

#### First-Class Symbolic Shape Abstraction

The key innovation in Relax is treating symbolic shapes as first-class annotations rather than marking dynamic dimensions as unknown. Prior systems like ONNX, Relay, and some MLIR dialects introduce an `any` or `unknown` value to represent dynamic dimensions, but this approach erases relationships between shape dimensions. For example, if one tensor has dimension $n$ and another has dimension $4n$, marking both as `unknown` prevents the compiler from recognizing that the second tensor is exactly four times larger than the first. Relax instead describes each shape dimension using a **symbolic expression** comprised of integer variables and constants, enabling the compiler to track relationships like $4n$ versus $n$ throughout the program.

For fully static models, Relax shape annotations subsume existing static shape-based annotations, meaning they work identically when all dimensions are known constants. For models with mixed dynamic and static dimensions, Relax not only expresses shape dimensions symbolically but also tracks symbolic relations over dimensions, which enables dynamic shape-aware optimizations. The paper provides a concrete example in Figure 3: after a `flatten` operator, the compiler knows the total number of elements is $4n$ and matches the input, suggesting potential buffer reuse that would be impossible with `unknown` annotations.

Symbolic shapes can also be used as first-class values in computation, not just as annotations. For instance, the `reshape` operator in Figure 3 takes the symbolic shape $(n, 4)$ as input and outputs a tensor value with the same shape, and symbolic shape expressions can construct arguments to tensor functions. However, the paper acknowledges that tracking shape relations at compile time is not always possible, particularly for data-dependent operators like `unique` whose output tensor shape depends on runtime values of its input. For such cases, Relax provides coarse-grained annotations (e.g., `Shape(ndim=2)` meaning the shape has two dimensions but both are unknown) and introduces a special construct called `match_cast` that asserts a symbolic shape annotation for a value, allowing introduction of new symbolic variables. The compiler inserts runtime checks for each `match_cast`, throwing an error if a constraint is violated, and `match_cast` can be inserted by both front-ends and compiler passes to suggest more specific symbolic shapes within programs.

#### Cross-Level Abstraction Mechanisms

The cross-level abstraction design reconciles different characteristics of computational graphs versus tensor programs and libraries. Computational graph abstractions favor pure operators that return a new tensor for each operation, enabling organization through directed acyclic graphs and effective graph rewriting without side effect concerns. In contrast, most tensor programs and libraries adopt **destination-passing style (DPS)** interfaces, which take computation result tensors as inputs and directly mutate them rather than allocating and returning new tensors. Relax requires passing input and output memory explicitly to low-level tensor programs, conforming to DPS, which abstracts away memory management concerns from low-level code and simplifies code generation.

The paper introduces two foreign function call primitives to bridge abstraction levels, shown in Figure 4. The first primitive, `call_tir`, allows direct invocation of a tensor program from the graph level. Figure 5 explains the semantics: `call_tir` allocates an output tensor based on the provided annotation shape and dtype, then calls the low-level function in destination-passing style with the output buffer passed explicitly, and finally returns the output. Notably, `call_tir` takes the shape annotation of the output as well as potentially other symbolic expressions as arguments, in order to pass shape information from the graph level to loop-level tensor programs. This shape information is crucial for optimizations on loop-level programs like operator fusion, and by flowing symbolic shape information from graph level to tensor programs, Relax allows tensor programs to generate code that specializes to most static dimensions and only uses dynamic dimensions when necessary.

The second primitive, `call_dps_library`, allows direct calls into foreign operator libraries from the graph level. The convention mirrors `call_tir` except that the callee is the name of a library function rather than a tensor program function, and these functions are supplied by a registry and linked to the final runnable module. This design introduces great flexibility in prototyping since external routines can be easily called from a Relax program. The paper emphasizes that graphs, tensor programs, and libraries are closely combined in Relax, all of which are critical in ML compilers, while other lower layers in compilers are leveraged for GPU source code generation.

The benefits of cross-level abstractions are summarized in Figure 6 with three common optimization patterns. **Partial lowering** allows a pass to make dispatch decisions or loop-level optimizations for part of the computations rather than making all lowering decisions in a single shot, which is helpful when replacing the lowering decision of a fused operator to different libraries while passing the program to later passes to handle other operators. **Analysis feedback** enables analyzing loop patterns of tensor programs and automatically annotating their operator properties, greatly reducing the engineering cost of manual annotation on high-level operators. **Cross-level transforms** allow optimization opportunities that are only disclosed after low-level optimizations, such as when tensor program analysis decides a tensor program needs a temporary workspace and the system jointly transforms the tensor program and graph level to insert workspace allocation at graph level, allowing the workspace to participate in global memory planning.

#### Shape Annotation Deduction Algorithm

The symbolic shapes in annotations serve as an important source of information for optimization passes, and Relax automatically tracks and deduces symbolic shape annotations of intermediate values not only during model construction but also between compiler passes. Each tensor operator has a registered shape deduction rule that takes the inputs' shape annotations and values (such as the case of `reshape`) and returns the output annotations. Relax adopts a **forward deduction method** that deduces the annotation of an expression based on its inputs, which offers benefits of simplicity and locality by effectively avoiding synchronization complexities across global contexts during processing. For foreign function call primitives `call_tir` and `call_dps_library`, the output annotations are part of their arguments and are directly used for deduction, and forward deduction leverages the explicit information of `match_cast` so that shape annotations of any new variables introduced during compiler passes can be efficiently deduced locally.

The paper recognizes the importance of propagating interprocedural shape relations globally across subgraph function calls to accommodate intermediate results of optimization passes like fusion, where subgraph functions themselves contain dynamic inputs and outputs. Figure 7 exemplifies symbolic shape deduction for subgraph function calls: the system takes symbolic relations at function `subfn` and propagates the shapes correctly in the callers. The paper summarizes three key design principles for shape annotation and deduction.

The first principle is **isolated symbolic relations at function boundaries**: function signatures in Relax give parameter and return value annotations, allowing for annotation inference of function calls with only the function signature. This allows functions to be used as first-class values with the `Callable` annotation, and the function signature also serves as a source to generate runtime checks to ensure that passed arguments and return values match the annotations in signature. These shape checks are lightweight and do not impact overall performance.

The second principle is **forward symbolic deduction**: Relax performs forward shape deduction based on symbolic shape relations, and as a safety net, coarse-grained annotations are returned when more specific information cannot be inferred (such as for data-dependent operators). It is permitted to pass values with coarse-grained annotations (e.g., `Shape(ndim=2)`) to functions that contain more specific annotations like `Shape((n,m))`, since lightweight runtime checks at the function boundary ensure the shape matches. The paper chooses forward deduction by default to maintain efficiency of symbolic shape deductions across passes — a full-graph forward deduction takes time linear to the number of operations — while still supporting introduction of more powerful but less efficient deduction methods via compiler passes as needed.

The third principle is **support for symbolic expressions in parameter annotations**: besides symbolic variables, Relax supports general arithmetic expressions in function parameter annotations, which is an important capability to simplify operator fusion and other transformations across subgraph functions in dynamic shape settings. Figure 8 provides an example: when fusing `add` and `relu` operators where both intermediate values contain expression $2 \times n$, a naïve approach would create a new function with parameters having shape $(2n,)$ but $n$ is nowhere supplied. To address this, the operator fusion pass passes an extra parameter $s$ which is bound to the runtime value of $n$, and passing extra symbolic arguments is a common pattern when designing passes that lift out function regions and recombine.

#### Cross-Level Dynamic Shape-Aware Operator Fusion

Operator fusion helps bring multiple operators together and reduce the overall memory loading cost of operators, and Figure 9 shows the general flow of operator fusion in Relax. A Relax program can contain tensor programs for both standard operators (e.g., `matmul`) and customized operators that may not have corresponding graph-level operators, such as quantization decode written in loops. The fusion process begins with an **analysis feedback pass** that annotates the pattern kind of each tensor program by pattern matching on tensor programs, and Algorithm 1 in the paper shows simplified pseudocode for this pass. The candidate pattern kinds include `ElementWise`, `Broadcast`, `Injective`, `Reduction`, `OutputEwiseFusible`, or `Opaque` as the fallback, and these pattern kinds describe mathematical properties of tensor programs.

The pattern kind information is then used by the **FuseOps pass** (Algorithm 2) to group tensor program calls into subgraph functions via pattern-match-based graph partitioning. An example pattern is the fusion of `ElementWise` tensor programs into the back of `OutputEwiseFusible` ones, such as `matmul` + ReLU. Finally, Relax applies **FuseTensorIR**, a cross-level transformation that jointly updates tensor programs and the graph-level calling site by merging tensor programs called in each subgraph function into a single one. Unlike applying naïve static-shape fusion, all steps handle symbolic shapes by tracking symbolic variables and generating extra symbolic variable parameters to support symbolic expressions in parameter annotations, as discussed in the shape deduction section.

The analysis feedback significantly reduces manual effort because the system can automate tensor program pattern analysis with a lightweight pass, whereas traditional single-shot abstractions require heavy and inflexible manual operator annotations. The three fusion sub-steps bring opportunities for further composition and customization: for example, a pass can fuse new sets of patterns not covered by FuseOps (e.g., fusing all sub-operators in scaled dot-product attention), and use FuseOps for the remainder, then FuseTensorIR can transform the fused subgraph function from both customized and standard fusion. This approach allows quick composition of different fusion strategies, improving overall productivity of continuous compiler development.

#### Dynamic Shape-Aware Memory Planning

Memory is an essential resource in modern ML applications, and most ML compilers can plan memory reuse by comparing sizes of static-shape tensors and allocating a fixed set of memory blocks ahead of time to reduce runtime memory allocation. Normally, compilers cannot take the same approach for compile-time unknown shapes and must rely on runtime memory allocators, but with symbolic shape abstractions, Relax can analyze and compare dynamic tensor shapes and plan for their reuse accordingly. Figure 10 and Algorithm 3 show how memory planning works with dynamic shapes: the system first lowers foreign function calls `call_tir` and `call_dps_library` to explicit memory allocation and DPS calls, exposing these allocations for planning.

To support dynamic shapes, the `RequestReuseWithSymShape` function in Algorithm 3 leverages symbolic expression analysis to prove whether two symbolic expressions are equal, and then appropriately picks a reusable storage from the pool. The system can further take the upper bound of symbolic values when they are known (e.g., annotated by users, such as inherent context lengths in LLMs) and statically allocate adequate memory, which allows creating a static memory allocation plan ahead of time even in the presence of dynamic shapes. This upper bound approach is related to `Func::bound()` in Halide which marks bounds in tensor programs, and Relax brings it to both graph and tensor program levels. Such predictable memory usage estimation is crucial for deploying dynamic ML models on memory-limited backends, and static planning can improve model performance by enabling CUDA Graph offloading, which relies on static memory allocation.

Figure 10 provides a concrete example: before memory planning, four intermediate tensors (`lv0` through `lv3`) are individually allocated, but after memory planning, the intermediate tensors reuse two allocated storage chunks (`s0` and `s1`). The tensors `lv0` and `lv2` instantiate from `s0`, while `lv1` and `lv3` instantiate from `s1`, demonstrating how symbolic shape equality proofs enable buffer reuse even when dimensions contain symbolic variables like $n$.

#### Cross-Level Tensor Program Workspace Lifting

In addition to memory allocation at the graph level, sometimes a tensor program may also require global memory allocation for intermediate workspace. For example, the Stream-K schedule of `matmul` decomposes a matrix multiplication into two phases of reduction: the first phase writes partial accumulation results into an intermediate global memory buffer, which is then consumed and accumulated by the second phase. As shown in Figure 11, Relax can detect such global memory allocation in tensor programs from analysis feedback, and jointly rewrite the tensor program and the graph-level caller site to lift the allocation to graph level.

Importantly, the lifted allocation can be planned by the memory planning mechanism described above, which further increases overall memory reuse. This optimization is only possible with cross-level abstractions when shape relations are preserved throughout all cross-level transformations in Relax, and optimization opportunities for planning such memory reuse may not arise in traditional single-shot lowering flow. Figure 11 shows the transformation: before workspace lifting, the tensor program `mm_split_k` internally allocates an 8 MB workspace buffer, but after lifting, the workspace allocation moves to the graph-level `main` function and is passed explicitly via `call_tir` as a parameter to the updated tensor program.

#### CUDA Graph Offloading

CUDA Graph is an optimization that reduces GPU kernel launch overhead at the GPU driver level and thereby improves overall system performance. It works by capturing multiple GPU kernel launches and then replaying them as a group rather than launching individual kernels separately. To replay captured kernel launches, the GPU driver requires all global memory accessed by the kernels to be constant-sized and statically allocated ahead of capturing, which poses a significant challenge for applying CUDA Graph to dynamic-shape ML models.

With static memory planning, Relax can pre-allocate all memory statically even for dynamic-shaped tensors. The system builds a pass to analyze the computational graph and lift subgraphs that meet CUDA Graph conditions into subgraph functions after memory planning, and the pass inserts runtime builtin functions that handle CUDA Graph capture or replay for these offloaded subgraph functions. At runtime, only the first run of a subgraph function triggers CUDA Graph capture, and subsequent runs automatically replay the captured CUDA Graph. Through these mechanisms, Relax extends CUDA Graph, typically available only for static models, to a broad range of dynamic workloads, and the paper notes this principle can generally apply to any GPU backend that supports static execution graphs in the future.

#### Tensor Operator Optimizations via Partial Lowering

Modern ML frameworks optimize tensor operators in two approaches: offloading computation to platform-specific operator libraries, or leveraging compiler tensor program optimizations and code generation. Most existing ML compilers make these decisions at the boundary between graph level and lower levels, making it hard to compose different lowering approaches. For instance, introducing a new operator library requires carefully examining existing lowering strategies and updating accordingly, and the complexity of the lowering layer grows as more approaches are incorporated, such as different ways of auto-scheduling.

Relax applies tensor operator optimizations via **partial lowering**, shown in Figure 12. The system registers a set of "(subgraph pattern, library function)" pairs in Relax, and builds pattern-match-and-rewrite passes that detect specific patterns (e.g., `matmul` with epilogue) in the graph level and partially lower detected regions to foreign library calls. Relax also allows users to register patterns for customizability. Additionally, based on TensorIR scheduling transformations, Relax builds a set of analysis-based dynamic shape-aware schedule rules to optimize tensor programs by minimizing memory loading. The system can also include passes to apply Ansor-style auto-tuning for rare tensor programs (e.g., complicated convolutions) that analysis-based schedule rules fail to handle. Importantly, all these transformations can be composed together and work collectively, and Relax enables fast development requiring only a single relatively simple partial lowering pass to customize operator optimizations.

#### End-to-End Optimization and Lowering Pipeline

Relax uses a fixed-order pipeline (without fixed point) on the cross-level abstraction to optimize, lower, and finally build an end-to-end model into a runnable module. Figure 13 shows an example pipeline: the system prioritizes partial library lowering to leverage external library functions on the target platform, then goes through the whole program to generate tensor programs for all high-level operator calls and lower operator calls to `call_tir` of corresponding tensor programs. Next, the system applies operator fusion, tensor program workspace lifting, memory planning, and CUDA Graph offloading. Notably, some tensor program optimizations (e.g., workspace lifting) are applied before graph optimizations, which necessitates Relax's cross-level abstraction design.

At the end of the pipeline is building the model into a runnable module. At the graph level, a fundamental task is to associate symbolic variables with concrete shape values and compute symbolic expressions at runtime. Relax creates an integer host tensor to store runtime values of all symbolic expressions in the program, and at the start of transformation, the system populates values of symbolic variables in program input tensors. Then, Relax generates tensor programs that load from the tensor, evaluate symbolic expressions, and store results to corresponding locations. Finally, the system inserts function calls to construct shape tuples when tensor shapes are needed as first-class values.

After this transformation, Relax erases all annotations, leaving a program comprised mainly of low-level function calls. The calls are then translated to a sequence of virtual machine instructions, each of which is a call into a generated or builtin function. For optimized low-level tensor programs, Relax directly generates corresponding GPU code. The system packages the graph-level virtual machine instructions and the GPU code together into a single holistic end-to-end module, which can then run on the target platform of compilation.

#### Design Rationale and Trade-offs

The paper makes several deliberate design choices that distinguish Relax from prior work. The decision to use **forward deduction** rather than constraint solving trades completeness for efficiency: a full-graph forward deduction takes time linear to the number of operations, whereas constraint solving (as in Axon) can be more powerful but incurs additional compile time costs. The paper notes Relax could still apply a similar constraint-solving approach despite these costs. The choice to support **coarse-grained annotations** as a fallback (e.g., `Shape(ndim=2)` when specific dimensions cannot be inferred) ensures the symbolic deduction succeeds for common cases while supporting general cases with runtime checks at function boundaries.

The **cross-level abstraction** design specifically addresses the single-shot lowering limitation of prior compilers. Traditional ML compilers use multi-level abstractions with optimizations in each level and single-shot lowering between levels, which makes it harder to analyze or transform tensor programs first and then use results to inform high-level graph optimizations. Relax's unified representation enables incremental optimization or partial lowering of portions of computation using different approaches, with analyses from all abstraction levels taken into account. The **destination-passing style** requirement for low-level calls abstracts away memory management from low-level code, simplifying code generation while enabling cross-level memory planning.

The paper acknowledges limitations: symbolic shape tracking cannot handle all data-dependent operators (requiring `match_cast` with runtime checks), and the forward deduction approach may miss some shape relationships that constraint solving could prove. However, these trade-offs enable Ahead-of-Time compilation for emerging platforms where JIT infrastructure is unavailable, which is a core deployment goal of the system.

## 4. Key Insights and Innovations

This section distills the core conceptual breakthroughs in Relax. While the previous section detailed *how* the system works, here we examine *why* these design choices represent fundamental shifts from prior ML compiler architectures. We distinguish between incremental engineering improvements and the structural innovations that enable Relax's unique capabilities.

### 4.1 Dissolving the Graph-Kernel Abstraction Boundary

**The Insight:** Traditional ML compilers treat the boundary between high-level computational graphs and low-level tensor programs as a one-way wall. Once a graph operator is lowered into a tensor program (kernel), it becomes an opaque blob. Relax identifies this as an artificial constraint that hides optimization opportunities.

**Why This Is Fundamental:**
In prior systems like TVM Relay or standard MLIR flows, lowering is "single-shot" (Section 2). Once a `matmul` operator is lowered to a TensorIR function, the graph optimizer no longer sees its internal structure. This creates two specific blind spots:
1.  **Lost Optimization Context:** The graph optimizer cannot see that a tensor program requires a temporary workspace, preventing global memory planning.
2.  **Manual Annotation Burden:** Developers must manually annotate properties (e.g., "this op is element-wise") for every new operator so the graph optimizer knows how to fuse it.

Relax innovations address this by maintaining a **cross-level abstraction** where graph operators and tensor programs coexist in the same IR. This enables two novel optimization patterns described in Figure 6:
*   **Analysis Feedback:** Instead of manual annotation, the compiler inspects the loop structure of a tensor program to automatically deduce its properties (e.g., `ElementWise` vs. `Reduction`). This reduces the engineering cost of adding custom operators, as the system infers fusibility rather than relying on hardcoded rules.
*   **Workspace Lifting:** Because the graph level can "see" allocations inside tensor programs, it can lift them to the graph level (Section 4.4). This allows a workspace allocated inside a kernel to participate in global memory reuse planning, which is impossible when the kernel is opaque.

**Comparison to Prior Work:**
*   **Halide [33]:** Supports external function calls but primarily within tensor programs, not bridging back to a graph level for global planning.
*   **PyTorch Inductor [2]:** Uses JIT tracing which avoids cross-function shape tracking entirely. Relax enables this analysis **ahead-of-time (AOT)**, which is critical for environments without JIT infrastructure (e.g., web browsers).

### 4.2 Symbolic Relations Over "Unknown" Annotations

**The Insight:** Dynamic shapes are often treated as "unknown" values in existing compilers (e.g., ONNX, Relay). Relax posits that while specific shape *values* may be unknown at compile time, the *relations* between shapes are often known and trackable.

**Why This Is Fundamental:**
Marking a dimension as `unknown` (or `?`) erases mathematical relationships. If tensor $A$ has shape $(n, 512)$ and tensor $B$ has shape $(n, 256)$, an `unknown` annotation sees two unrelated dynamic tensors. Relax's **first-class symbolic shapes** (Section 3.2) preserve the variable $n$, allowing the compiler to prove that $A$ and $B$ share the same first dimension.

This distinction enables optimizations that were previously mutually exclusive with dynamic shapes:
*   **Buffer Reuse Proof:** The compiler can prove that two tensors have equal size ($n \times 512 = n \times 512$) even if $n$ is unknown, allowing them to share memory buffers.
*   **Expression Propagation:** Shape expressions like $2n$ or $n + 1$ are tracked through operations (Figure 8). This allows operator fusion passes to generate correct function signatures for fused kernels without losing track of the symbolic variable $n$.

**Comparison to Prior Work:**
*   **Relay / MLIR [27, 35]:** Use "unknown" annotations that break shape relations across operators.
*   **Axon [11]:** Uses constraint solving for shapes, which is powerful but incurs higher compile-time costs. Relax chooses **forward deduction** (Section 3.4) as a trade-off, sacrificing some completeness for linear-time efficiency ($O(\text{operations})$), making it viable for large models.
*   **PyTorch JIT [2]:** Tracks symbols but limits scope to traced subgraphs. Relax tracks symbols **globally across function boundaries**, enabling whole-program AOT optimization.

### 4.3 Enabling Static Optimizations for Dynamic Workloads

**The Insight:** A major contribution of Relax is demonstrating that "static" optimization techniques can be applied to "dynamic" models if shape relations are preserved and upper bounds are known. This challenges the conventional wisdom that dynamic shapes necessitate dynamic memory allocation and prevent graph capture.

**Why This Is Significant:**
This innovation directly translates to performance and deployment capabilities that were previously unavailable for dynamic LLMs:

1.  **Static Memory Planning for Dynamic Shapes:**
    By proving shape relations and allowing users to specify upper bounds (e.g., max context length), Relax can pre-allocate memory pools statically.
    *   **Result:** Table 2 shows this reduces activation memory by **22% during prefill** and **40% during decode** for Llama3-8B.
    *   **Impact:** Without this, dynamic allocation fragments memory on constrained devices. The paper notes that without this planning, some models are "not even runnable on some of the environments due to memory constraints" (Section 5.3).

2.  **CUDA Graph Offloading for Dynamic Models:**
    CUDA Graphs typically require static memory addresses and sizes, making them incompatible with dynamic shapes. Relax's static memory planning allows it to capture CUDA Graphs even for dynamic workloads.
    *   **Result:** Figure 17 shows CUDA Graph offloading contributes **1–2% performance gain** by reducing kernel launch overhead.
    *   **Impact:** While 1–2% seems modest, it enables the use of driver-level optimizations on dynamic models that were previously restricted to static graphs.

3.  **Universal Deployment:**
    The combination of AOT compilation, memory planning, and cross-level optimization allows Relax to target backends that lack robust JIT or memory management infrastructure.
    *   **Result:** Table 3 demonstrates LLM inference on **WebGPU**, **iPhone 14 Pro**, and **Orange Pi 5**.
    *   **Impact:** Competitors like vLLM lack Apple GPU support; llama.cpp lacks Android GPU support. Relax fills this gap by generating optimized code automatically rather than relying on hand-written kernels for each backend.

### 4.4 Critical Trade-offs and Limitations

To fully understand the innovation, one must recognize the deliberate trade-offs Relax makes:

*   **Forward Deduction vs. Constraint Solving:** Relax uses forward deduction for efficiency. This means it cannot solve complex circular shape constraints that a constraint solver (like Axon) might handle. The paper acknowledges this limitation but prioritizes compilation speed for large models.
*   **Runtime Checks for Coarse Annotations:** When shape relations cannot be proven (e.g., data-dependent operators like `unique`), Relax falls back to coarse annotations (e.g., `Shape(ndim=2)`) and inserts runtime checks via `match_cast` (Section 3.2). This ensures safety but introduces minor runtime overhead if checks fail.
*   **Fixed-Order Pipeline:** The optimization pipeline (Figure 13) is fixed-order without fixed-point iteration. This simplifies implementation but may miss optimization opportunities that require re-evaluating earlier passes after later transformations.

These trade-offs highlight that Relax is engineered for **practical deployment efficiency** rather than theoretical completeness. The innovation lies not in solving every possible shape constraint, but in solving the *common* constraints efficiently enough to enable AOT deployment on emerging hardware.

## 5. Experimental Analysis

This section examines how the Relax authors validate their claims through systematic experimentation. The evaluation is designed to answer four specific questions about performance competitiveness, optimization effectiveness, platform coverage, and model generality. Understanding the experimental methodology is crucial because the claims about "competitive performance" and "broad deployment" rest entirely on these results.

### 5.1 Evaluation Methodology

**Hardware Platforms.** The evaluation spans three categories of hardware to test Relax's deployment claims:

| Platform Category | Specific Devices | Purpose |
|-------------------|------------------|---------|
| Desktop GPUs | NVIDIA RTX 4090, AMD Radeon 7900 XTX, Apple M2 Ultra | Test performance against established baselines on powerful hardware |
| Mobile/Embedded | iPhone 14 Pro (A16), Samsung S23 (Snapdragon 8 Gen 2), Orange Pi 5 (ARM Mali), Steam Deck (AMD APU), Jetson Orin | Test deployment on memory-constrained emerging platforms |
| Web | WebGPU on Apple M3 Max | Test browser-based deployment without native installation |

**Models Evaluated.** The paper tests across multiple model families to ensure results are not model-specific:

- **LLMs for main evaluation:** Llama3-8B, Gemma1.1-7B, Qwen2-7B (all with float16 weights and activations)
- **Quantized LLMs for mobile:** Llama2-7B (3-bit on iPhone, 4-bit on Samsung S23), Phi3-mini-4k (4-bit), RedPajama-3B (4-bit)
- **Other modalities:** Whisper-large-v3 (speech recognition), LLaVA (vision-language)

**Baselines.** The authors compare against a comprehensive set of existing systems:

- **HuggingFace Transformers (v4.41.2)** with PyTorch (v2.3.1) in both eager and `torch.compile` modes
- **vLLM (v0.5.0.post1)** — a production LLM serving system optimized for throughput
- **llama.cpp (172c825)** — a hand-optimized CPU/metal inference system
- **Domain-specific baselines:** WhisperX, Faster Whisper, whisper.cpp for speech; various combinations for LLaVA

**Metrics.** The evaluation uses three primary metrics:

1. **Decode token latency (ms/tok):** Time to generate one token during the autoregressive decode phase, measured across different batch sizes (1, 16, 32, 64)
2. **Throughput (tokens/sec):** For mobile/embedded evaluation where single-sequence generation is the primary use case
3. **Memory usage (MiB):** Total allocated activation memory during inference, measured for both prefill and decode phases

**Measurement Protocol.** For LLM evaluation, the authors "measure the decode time of generating 32 tokens and compute the per-token latency per sequence." Importantly, "Relax compiles models only once for arbitrary batch sizes and sequence lengths" — this is a critical detail because some baselines may require recompilation for different shapes, which would unfairly advantage them if compilation time were excluded.

### 5.2 LLM Inference Performance on Desktop GPUs

The first evaluation question asks whether Relax provides competitive performance compared to existing frameworks on established GPU platforms. Figures 14, 15, and 16 present results across three different GPU architectures.

**NVIDIA RTX 4090 Results (Figure 14).** On the most powerful test platform, Relax demonstrates competitive performance across all three LLM models:

- **Llama3-8B:** Relax reduces decode token latency by up to **27%** compared to baselines at certain batch sizes
- **Gemma1.1-7B:** Similar competitive performance across batch sizes 1–64
- **Qwen2-7B:** The paper notes "we omitted the results of HF with torch.compile for Qwen2-7B due to the lack of support" — this is an important transparency detail showing that not all baselines work on all models

The key observation from Figure 14 is that Relax does not simply match baselines at one batch size — it remains competitive across the full range from batch size 1 to 64. This matters because batch size 1 represents latency-critical interactive use, while larger batch sizes represent throughput-optimized serving scenarios.

**AMD Radeon 7900 XTX Results (Figure 15).** This platform tests Relax's ability to target non-NVIDIA GPUs without hand-written kernels:

- Relax "brings optimized performance with up to **1.50x** under case of batch size 1" for Llama3-8B
- The performance advantage is most pronounced at batch size 1, which the paper attributes to "compiler-optimized matrix-vector multiplication tensor programs at batch size 1, while being able to apply partial library lowering to leverage operator libraries for other batch sizes"

This result is significant because it demonstrates Relax's cross-level abstraction enables automatic adaptation to different batch sizes without manual kernel selection — a capability that typically requires engineering effort in other frameworks.

**Apple M2 Ultra Results (Figure 16).** This platform reveals a critical deployment gap that Relax addresses:

- **PyTorch compile mode and vLLM lack Apple GPU support entirely** — they simply do not run on this platform
- **llama.cpp has strong performance on Apple GPUs** but "performs less effectively on NVIDIA GPUs"
- **Relax achieves competitive performance comparing to the hand-optimized llama.cpp baseline** on Apple hardware

The paper explicitly states: "In contrast, Relax is able to support all the platforms with competitive performance." This is not a marginal improvement — it represents the difference between having any GPU-accelerated solution versus none at all on certain platforms.

### 5.3 Ablation Study: Effects of Composable Optimizations

Section 5.2 isolates the contribution of individual optimization techniques to verify that the proposed abstractions actually enable the claimed benefits. Figure 17 presents an ablation study on Llama3-8B inference across different batch sizes on NVIDIA RTX 4090.

**Partial Library Lowering.** This optimization contributes the most to performance:

> "Partial library lowering contributes the most, up to 27% of the performance for large batch sizes, where it lowers heavy-load matrix multiplications (which count for about 1/3 of all operators) to cuBLAS library kernels."

This result validates the cross-level abstraction design: by allowing graph-level patterns to be selectively lowered to library calls while keeping other operators in tensor program form, Relax achieves the best of both worlds — library performance for heavy operators and compiler flexibility for the rest.

**Operator Fusion.** The fusion optimization provides moderate but consistent benefits:

> "Operator fusion helps by fusing about 1/5 of all operators (such as RMSNorm and element-wise addition) to reduce launched kernels and GPU global memory accesses."

Fusing 20% of operators may seem modest, but the paper notes these are specifically memory-bound operators where fusion reduces global memory traffic. The benefit is most visible at smaller batch sizes where kernel launch overhead dominates.

**CUDA Graph Offloading.** This optimization provides the smallest but still measurable benefit:

> "CUDA Graph offloading overall brings about 1–2% of performance gain by reducing kernel launch overheads at GPU driver level."

The paper is transparent that this is a modest gain. However, the significance is not the magnitude but the **feasibility**: CUDA Graphs were previously unavailable for dynamic-shape models. Relax's static memory planning with symbolic upper bounds makes this optimization possible at all.

**Memory Usage Reduction (Table 2).** The ablation study also measures memory consumption with and without static memory planning:

| Phase | Relax w/o Planning | Relax w/ Planning | Reduction |
|-------|-------------------|-------------------|-----------|
| Prefill (successive inputs) | 192.7 MiB | 149.7 MiB | **22%** |
| Decode (successive batches) | 150.0 MiB | 88.2 MiB | **40%** |

The paper explains the mechanism: "With static memory planning, we always reuse memory across all input lengths and batch sizes, even as they vary over time. In contrast, without memory planning, the system repeatedly allocates dynamic-sized memory whenever the input shape changes."

This 40% reduction during decode is particularly important for memory-constrained devices. The paper makes a strong claim: "Without memory planning that pre-allocates all needed memory and keeps it within the budget, these models are not even runnable on some of the environments due to memory constraints." This is not a performance optimization — it is an **enabling technology** for deployment.

Section 5.3 addresses whether Relax can actually deploy models to platforms that existing frameworks do not support. Table 3 presents throughput results across six emerging platforms that are "less supported by existing solutions."

**Mobile Phone Performance.** The results show Relax achieves usable inference speeds on consumer mobile devices:

This is an important methodological detail — the largest model (Llama3-8B) does not fit in mobile VRAM even with 4-bit quantization, so a smaller model is used instead.

**Embedded Device Performance.** The Orange Pi 5 represents a low-cost embedded platform:

- The paper states Relax provides "a throughput of over 5 tokens/s for mobile devices, and 2.3 tokens/s for Orange Pi 5"

Section 5.4 tests whether Relax's benefits extend beyond LLMs to other model architectures. This is important because the abstractions could theoretically be LLM-specific.

**Whisper Speech Recognition (Figure 19).** The evaluation uses Whisper-large-v3 to transcribe a 30-second speech file:

- **NVIDIA RTX 4090:** Relax brings a **14% speedup** compared to baselines
- **Apple M2 Ultra:** "Competitive performance on Apple GPU"
- **Baseline limitations:** "WhisperX and Faster Whisper have no Apple GPU support"

The 14% speedup on NVIDIA hardware shows that Relax's optimizations benefit encoder-decoder Transformer architectures, not just decoder-only LLMs. The Apple GPU result again demonstrates deployment coverage that baselines lack.

**LLaVA Vision-Language Model (Figure 20).** LLaVA combines a CLIP visual encoder with an LLM (Vicuna), testing Relax's ability to handle multi-modal architectures:

- The evaluation measures "time of generating 32 tokens for an image"
- Baselines include HuggingFace Transformers, vLLM, and llama.cpp
- Result: "Relax efficiently supports the vision encoder together with the prefill and decode phases of LLM"

The paper does not report specific speedup numbers for LLaVA, which is a minor gap. However, the claim that Relax "achieves competitive optimized performance for LLaVA generation on both platforms" (NVIDIA and Apple) suggests the cross-level abstractions handle heterogeneous model components (vision encoder + language model) without special-case engineering.

### 5.6 Critical Assessment of Experimental Claims

**Do the experiments support the main claims?**

| Claim | Evidence | Assessment |
|-------|----------|------------|
| "Competitive performance to state-of-the-art systems" | Figures 14–16 show Relax matches or exceeds baselines on NVIDIA, AMD, Apple GPUs | **Strongly supported** — 27% latency reduction on RTX 4090 is substantial |
| "Enables deployment to emerging environments" | Table 3 shows functioning LLM inference on iPhone, Samsung, Orange Pi, WebGPU | **Strongly supported** — multiple platforms where baselines have no GPU support |
| "Static memory planning reduces memory usage" | Table 2 shows 22–40% memory reduction | **Strongly supported** — clear quantitative evidence |
| "Cross-level optimizations enable new capabilities" | Figure 17 ablation shows each optimization contributes measurably | **Moderately supported** — individual contributions are clear, but interaction effects are not fully explored |

**Limitations and Gaps.**

The evaluation has several notable limitations that readers should recognize:

1. **No compilation time measurements.** The paper focuses on inference latency and memory but does not report how long Relax takes to compile models. For AOT compilation, this matters less than JIT, but it is still relevant for development workflows.

2. **Limited model size range.** All LLM evaluations use 7–8B parameter models. The paper does not test larger models (70B+) where memory planning benefits might be more pronounced or where compilation complexity might increase.

3. **No failure cases reported.** The paper does not discuss models or operators where symbolic shape deduction fails and requires `match_cast` with runtime checks. Understanding the frequency of such cases would help assess practical usability.

4. **Ablation study is single-model.** Figure 17 only shows ablation results for Llama3-8B on RTX 4090. It is unclear whether the relative contributions of partial lowering, fusion, and CUDA Graph offloading hold across different models and platforms.

5. **WebGPU results lack baseline comparison.** Table 3 shows Relax works on WebGPU, but there is no baseline because (as the paper states) no other framework supports this. While this demonstrates capability, it makes it harder to assess whether Relax's WebGPU performance is optimal.

**Trade-offs and Conditions.**

The results are not universally positive — there are conditions where Relax's advantages are more or less pronounced:

- **Batch size 1 advantage is largest on AMD GPU** (1.50x) but more modest on NVIDIA. This suggests Relax's compiler-generated kernels provide the most benefit where vendor libraries are less optimized.

- **Memory planning benefits are largest during decode** (40%) versus prefill (22%). This makes sense because decode has more repeated allocations across autoregressive steps.

- **Mobile performance requires quantization.** The footnote in Table 3 reveals that full-precision models do not fit on mobile devices — Relax enables deployment but does not eliminate the need for quantization on memory-constrained platforms.

- **CUDA Graph benefit is modest (1–2%).** This optimization is enabled by Relax's abstractions but provides limited performance gain. The value is in enabling the optimization at all for dynamic models, not in the magnitude of improvement.

### 5.7 Summary of Experimental Findings

The experimental evaluation provides strong evidence for Relax's core claims:

1. **Performance competitiveness:** Relax achieves up to **27% lower decode latency** on NVIDIA GPUs and up to **1.50x speedup** on AMD GPUs at batch size 1, while matching hand-optimized systems like llama.cpp on Apple hardware.

2. **Memory efficiency:** Static memory planning reduces activation memory by **22% during prefill** and **40% during decode**, enabling models to run on memory-constrained devices where they would otherwise fail.

3. **Platform coverage:** Relax deploys to **six emerging platforms** (iPhone, Samsung, Orange Pi, Steam Deck, Jetson Orin, WebGPU) where established frameworks either lack support or provide CPU-only fallbacks.

4. **Optimization composition:** The ablation study confirms that partial library lowering, operator fusion, and CUDA Graph offloading each contribute measurably, validating the composable abstraction design.

The experiments do have limitations — no compilation time measurements, limited model size range, and no reported failure cases. However, within the scope of what is measured, the results convincingly demonstrate that Relax's cross-level abstractions and symbolic shape tracking enable capabilities that prior systems cannot provide, particularly for AOT deployment on emerging platforms.

## 6. Limitations and Trade-offs

While Relax demonstrates significant advances in dynamic shape handling and cross-level optimization, its design involves deliberate trade-offs that constrain its applicability in certain scenarios. Understanding these limitations is crucial for practitioners deciding whether to adopt Relax for their specific workloads. The following analysis breaks down the structural assumptions, edge cases, and open questions inherent in the approach.

### 6.1 Symbolic Deduction: Efficiency Over Completeness

The most fundamental trade-off in Relax is the choice of **forward symbolic deduction** over constraint solving. As described in Section 3.4 and Section 4.4, Relax propagates shape relationships linearly through the graph ($O(\text{operations})$). This ensures compilation remains fast even for large models, but it sacrifices **completeness**.

*   **Missed Relations:** Forward deduction cannot resolve circular dependencies or complex constraints that require solving a system of equations. For example, if shape $A$ depends on $B$ and $B$ depends on $A$ through a chain of operations, a constraint solver (like Axon [11]) might resolve this, but Relax's forward pass may fail to deduce the precise relationship.
*   **Fallback to Coarse Annotations:** When deduction fails, Relax falls back to coarse-grained annotations like `Shape(ndim=2)` (Table 1, Section 3.1). While this ensures compilation succeeds, it erases the specific symbolic relations needed for optimizations like buffer reuse. The paper notes this is a "safety net," but in practice, frequent fallbacks degrade the performance benefits of symbolic planning.
*   **Runtime Checks:** To handle cases where static deduction is impossible (e.g., data-dependent operators), Relax inserts runtime checks via `match_cast` (Section 3.2). This introduces a **reliability vs. performance trade-off**:
    *   If the check passes, execution proceeds with optimized paths.
    *   If the check fails, the program throws an error at runtime.
    *   This shifts some burden to the developer to ensure inputs adhere to assumed constraints, whereas a fully dynamic system might simply reallocate memory.

### 6.2 Handling Data-Dependent Dynamism

Relax excels at **symbolic dynamism** (shapes depend on variables like batch size $n$), but it struggles with **data-dependent dynamism** (shapes depend on tensor values).

*   **The `unique` Operator Case:** Section 3.2 explicitly cites the `unique` operator as a case where output shape depends on runtime values (the number of unique elements in a tensor). Relax cannot symbolically deduce this count.
*   **Impact on Optimization:** When such operators appear, the compiler must insert `match_cast` to introduce new symbolic variables (e.g., $m$) for the output shape. However, the relationship between $n$ (input size) and $m$ (output size) is lost. This breaks downstream optimizations:
    *   **Memory Planning:** The compiler cannot prove that the output buffer fits within a pre-allocated pool sized for $n$.
    *   **Fusion:** Operators following `unique` may not fuse correctly if their shape relations to previous layers are obscured.
*   **Workload Suitability:** This limitation suggests Relax is best suited for workloads where dynamism is primarily structural (sequence length, batch size) rather than content-dependent. Models heavy in dynamic control flow or data-dependent indexing may see reduced benefits.

### 6.3 Pipeline Rigidity and Optimization Order

Relax employs a **fixed-order pipeline** without fixed-point iteration (Section 4.7, Section 4.4). This design choice simplifies implementation and ensures predictable compilation behavior, but it limits optimization potential.

*   **Missed Interaction Effects:** In compiler theory, fixed-point iteration allows passes to re-run until no further changes occur. For example, memory planning might enable a fusion opportunity that was previously blocked by allocation constraints. In Relax's fixed-order flow (Figure 13), once a pass completes, it is not revisited.
*   **Ordering Sensitivity:** The pipeline prioritizes partial library lowering before operator fusion (Section 4.7). If a user registers a custom pattern that conflicts with a built-in fusion rule, the fixed order dictates the outcome. This reduces flexibility compared to systems that allow users to define pass sequences dynamically.
*   **Justification:** The authors justify this by noting that "some tensor program optimizations (e.g., workspace lifting) are applied before graph optimizations, which necessitates Relax's cross-level abstraction design" (Section 4.7). However, this rigidity means that novel optimization strategies requiring different ordering must modify the core pipeline rather than simply adding a pass.

### 6.4 Engineering Effort for Partial Lowering

While Relax reduces the burden of writing full kernels, it does not eliminate engineering effort entirely. The **partial lowering** approach (Section 4.6) requires users to register "(subgraph pattern, library function)" pairs.

*   **Pattern Registration:** Users must define patterns (e.g., "matmul with epilogue") to trigger library dispatch. This is less effort than writing a CUDA kernel but more effort than fully automatic systems that rely purely on auto-tuning (like Ansor [8, 49]).
*   **Maintenance Overhead:** As new library versions emerge (e.g., cuBLAS updates), patterns may need adjustment to match new API signatures or performance characteristics. The paper states Relax "enables fast development and only requires a single relatively simple partial lowering pass," but this assumes the pattern matching infrastructure is robust enough to handle variant operator signatures without frequent manual intervention.
*   **Coverage Gaps:** If a specific operator combination lacks a registered pattern, it falls back to compiler-generated code. While functional, this code may not match the performance of vendor-optimized libraries, leading to inconsistent performance across different model architectures.

### 6.5 Unverified Scalability and Compilation Cost

The experimental evaluation provides strong evidence for 7–8B parameter models but leaves open questions about scalability to larger systems.

*   **Compilation Time:** Section 5.6 notes that "no compilation time measurements" are reported. For Ahead-of-Time (AOT) compilation, this is critical. Symbolic shape deduction and cross-level optimization add complexity to the compiler passes. If compilation time scales super-linearly with model size, Relax might become impractical for 70B+ parameter models where iteration speed matters during development.
*   **Model Size Limits:** All LLM evaluations use models up to 8B parameters (Section 5.1). The paper does not test larger models where memory planning benefits might be more pronounced but where symbolic expression complexity could also increase.
*   **IR Bloat:** Maintaining symbolic expressions and annotations across cross-level boundaries (Section 3.3) increases the size of the Intermediate Representation (IR). For very large graphs, this could increase memory consumption during compilation, potentially limiting the maximum model size that can be compiled on a given host machine.

### 6.6 Memory Planning Assumptions

The static memory planning optimization (Section 4.3) relies on specific assumptions that may not hold in all deployment scenarios.

*   **Upper Bound Requirement:** To statically allocate memory for dynamic shapes, Relax requires users to provide **upper bounds** (e.g., maximum context length, maximum batch size). Section 4.3 states: "We can further take the upper bound of the symbolic values when they are known (e.g., annotated by users...)."
    *   **Risk:** If users overestimate bounds to be safe, memory is wasted (fragmentation or excessive reservation). If they underestimate, the model fails at runtime.
    *   **Unknown Workloads:** In serving scenarios where request lengths are highly unpredictable (e.g., open-ended chat), setting a tight upper bound is difficult. This limits the effectiveness of static planning compared to dynamic allocators that adapt precisely to each request.
*   **Fragmentation:** While static planning reduces total allocation (Table 2), it does not necessarily eliminate internal fragmentation within the pre-allocated pools. If the symbolic shapes vary widely (e.g., sequence lengths ranging from 10 to 2048), fixed pools may leave gaps that cannot be reused by incompatible shapes.

### 6.7 Performance Variability Across Backends

While Relax supports diverse backends, performance parity is not guaranteed across all of them.

*   **Library Dependency:** Partial lowering relies on external libraries (e.g., cuBLAS for NVIDIA, Metal for Apple). Section 5.1 notes that "PyTorch compile mode and vLLM lack Apple GPU support," which Relax addresses. However, Relax's performance on Apple GPUs is "competitive comparing to the hand-optimized llama.cpp baseline" (Figure 16), not necessarily superior. This suggests that while Relax enables deployment, it may not always beat highly specialized, hand-tuned stacks on specific hardware.
*   **WebGPU Overhead:** Table 3 shows WebGPU throughput is functional (e.g., 37.8 tok/s for Llama on M3 Max), but the paper acknowledges "no baseline comparison" exists because no other framework supports this. Without a baseline, it is unclear how much overhead the Relax abstraction layer introduces compared to raw WebGPU compute limits.
*   **CUDA Graph Benefits:** The benefit of CUDA Graph offloading is modest (1–2% in Figure 17). While enabling it for dynamic models is a technical achievement, practitioners should not expect this optimization to drastically change latency profiles on its own. It is a complementary gain rather than a primary performance driver.

### 6.8 Summary of Trade-offs

The table below summarizes the key design trade-offs in Relax:

| Design Choice | Benefit | Cost / Limitation |
| :--- | :--- | :--- |
| **Forward Deduction** | Fast compilation ($O(\text{ops})$) | Cannot solve complex/circular constraints |
| **Static Memory Planning** | Enables CUDA Graphs, reduces fragmentation | Requires known upper bounds; risks over-allocation |
| **Fixed-Order Pipeline** | Predictable, simpler implementation | Misses optimizations requiring fixed-point iteration |
| **Partial Lowering** | Flexible library integration | Requires manual pattern registration |
| **`match_cast`** | Handles data-dependent shapes | Introduces runtime checks and potential failure modes |

In conclusion, Relax is not a universal solver for all ML compilation problems. It is specifically optimized for **AOT deployment of dynamic models on emerging platforms** where memory constraints and lack of JIT infrastructure are the primary bottlenecks. For workloads with highly unpredictable data-dependent shapes, or where compilation time is critical for large-scale models, the trade-offs described above may require careful consideration.

## 7. Implications and Future Directions

This section examines how Relax's contributions reshape the machine learning compiler landscape, what research trajectories it opens, and when practitioners should consider adopting this approach. We ground these implications in the paper's concrete results and stated design goals.

### 7.1 Shifting the Compiler Abstraction Paradigm

Relax challenges a foundational assumption in ML compiler design: that **abstraction levels must be separated with single-shot lowering** between them. The paper demonstrates that maintaining a unified cross-level representation enables optimizations that were previously impossible for dynamic models. This has three landscape-level implications:

**1. Dynamic Shapes No Longer Preclude Static Optimizations.** The conventional wisdom held that dynamic tensor shapes necessitate dynamic memory allocation and prevent graph capture techniques like CUDA Graphs. Relax's evaluation contradicts this: Table 2 shows **40% memory reduction during decode** through static planning, and Figure 17 demonstrates CUDA Graph offloading provides **1–2% performance gain** even for dynamic workloads. This shifts the design space — compilers can now target "static optimization for dynamic models" rather than accepting dynamic allocation as inevitable.

**2. Cross-Function Symbolic Tracking Becomes Viable for AOT Compilation.** Prior systems like PyTorch Inductor track symbolic shapes but limit scope to traced subgraphs (Section 6 of the paper). Relax proves that **global symbolic shape tracking across function boundaries** is feasible for ahead-of-time compilation. This enables whole-program optimizations that JIT systems cannot perform, particularly important for emerging platforms without runtime infrastructure.

**3. Automatic Code Generation Competes with Hand-Written Kernels on Diverse Backends.** Figure 18 shows Relax achieves **up to 55% better throughput than llama.cpp on Samsung S23** because it automatically generates GPU kernels where llama.cpp falls back to CPU. This reduces the engineering burden of maintaining hand-optimized kernels for each backend — a significant cost for framework developers supporting emerging hardware.

The paper explicitly positions these insights as transferable: "Relax's insights for supporting dynamic shapes and cross-level optimizations can be used to improve these ML compiler frameworks" (Section 6). This suggests Relax's abstractions may influence future compiler designs beyond the TVM ecosystem.

### 7.2 Follow-Up Research Directions

The paper's design choices and acknowledged limitations point to several concrete research trajectories:

**Constraint Solving for Shape Deduction.** Section 4.4 notes that Relax uses forward deduction for efficiency but acknowledges "Relax could still apply a similar constraint-solving approach, despite its additional compile time costs." This opens a research question: **what is the practical trade-off space between forward deduction and constraint solving for production ML workloads?** Axon [11] demonstrates constraint solving is feasible but incurs higher compile costs. A hybrid approach — forward deduction for common cases with constraint solving fallback for complex relations — could capture completeness without sacrificing typical compilation speed.

**Fixed-Point Optimization Iteration.** Section 6.7 identifies the fixed-order pipeline as a limitation: "once a pass completes, it is not revisited." This prevents optimizations that emerge from pass interactions (e.g., memory planning enabling new fusion opportunities). Future work could explore **selective fixed-point iteration** on critical optimization subsets while maintaining overall compilation efficiency. The cross-level abstraction makes this feasible because shape relations are preserved throughout transformations.

**Data-Dependent Shape Handling.** Section 3.2 explicitly cites operators like `unique` as cases where output shape depends on runtime values, requiring `match_cast` with runtime checks. This remains an open challenge: **can symbolic abstractions extend to data-dependent dynamism without excessive runtime overhead?** One direction is probabilistic shape bounds (e.g., "output size is 95% likely to be ≤ 0.5n"), enabling optimistic memory planning with fallback paths.

**Cross-Abstraction Integration.** Section 6 states: "We use TensorIR [16] as the tensor program abstraction in our cross-level design implementation, but we could combine our approaches with other abstractions for tensor supports programs." This suggests research on **integrating Relax with alternative tensor program IRs** like Triton [42] or Graphene [22]. Each offers different optimization capabilities — Triton excels at GPU kernel customization, while TensorIR provides automatic scheduling. A unified cross-level framework could dispatch to the most appropriate tensor program abstraction per operator.

**Compilation Time Characterization.** Section 5.6 notes "no compilation time measurements" are reported. For AOT compilation, this matters less than JIT but remains relevant for development workflows. Future work should **quantify compilation time scaling** with model size, particularly for 70B+ parameter models where symbolic expression complexity may increase super-linearly.

### 7.3 Practical Applications and Downstream Use Cases

The evaluation results identify specific deployment scenarios where Relax provides unique value:

**Mobile and Embedded LLM Deployment.** Table 3 demonstrates functional LLM inference on iPhone 14 Pro (5.1 tok/s), Samsung S23 (7.9 tok/s), and Orange Pi 5 (2.3 tok/s). The paper states: "Without memory planning that pre-allocates all needed memory and keeps it within the budget, these models are not even runnable on some of the environments due to memory constraints" (Section 5.3). This makes Relax particularly valuable for:
- **On-device AI applications** requiring offline inference (privacy-sensitive use cases)
- **Edge devices** with limited or intermittent connectivity
- **IoT deployments** where cloud inference latency is unacceptable

**Web-Based ML via WebGPU.** The WebGPU results (37.8 tok/s for Llama on M3 Max) represent a novel capability: "To our best knowledge, Relax is the first solution to enable GPU-accelerated LLM inference on these platforms except the NVIDIA Jetson Orin" (Section 5.3). This enables:
- **Browser-native AI applications** without native installation
- **Cross-platform ML demos** accessible via URL
- **Enterprise deployments** where software installation is restricted

**Multi-Backend Model Distribution.** Figure 14–16 show Relax achieves competitive performance across NVIDIA, AMD, and Apple GPUs with a single compilation flow. This addresses the fragmentation problem identified in Section 2: model developers currently must maintain separate code paths for different backends (vLLM for NVIDIA, llama.cpp for Apple, CPU fallback for Android). Relax enables **single-source deployment** across heterogeneous hardware, reducing engineering overhead for model distributors.

**Custom Operator Support.** Section 4.2 demonstrates fusion of customized quantization decode operators that "cannot be easily represented on graph level without introducing specialized operators." This is valuable for:
- **Proprietary model architectures** with non-standard operators
- **Research prototypes** requiring rapid operator iteration
- **Domain-specific models** (e.g., scientific computing) with custom tensor operations

### 7.4 Reproducibility and Integration Guidance

Based on the paper's evaluation and design documentation, here is practical guidance for when to adopt Relax versus alternative approaches:

**When to Prefer Relax:**

| Scenario | Rationale | Paper Evidence |
|----------|-----------|----------------|
| **AOT deployment required** | Relax produces deployable modules without JIT infrastructure | Section 2: "JIT tracing... limits its applications on emerging platforms" |
| **Memory-constrained targets** | Static memory planning reduces activation memory by 22–40% | Table 2: Memory usage comparison |
| **Multiple backend targets** | Single compilation flow across NVIDIA, AMD, Apple, WebGPU | Figures 14–16, Table 3 |
| **Custom operators needed** | Cross-level abstraction enables fusion of user-defined tensor programs | Figure 9: Customized quantization decode fusion |
| **Dynamic batch/sequence lengths** | Symbolic shape tracking preserves relations across variable dimensions | Figure 3: Symbolic vs. unknown annotations |

**When Alternatives May Be Preferable:**

| Scenario | Alternative | Rationale |
|----------|-------------|-----------|
| **Server-only NVIDIA deployment** | vLLM, TensorRT-LLM | These systems are highly optimized for this specific case; Relax's cross-platform benefits are less relevant |
| **Maximum Apple GPU performance** | llama.cpp | Figure 16 shows llama.cpp is competitive on Apple hardware; if only targeting Apple, the hand-optimized stack may suffice |
| **Complex data-dependent shapes** | PyTorch JIT | If workload has many operators like `unique` where shape depends on values, PyTorch's runtime tracing may handle this more gracefully than Relax's `match_cast` approach |
| **Rapid prototyping with large models** | PyTorch Eager | Relax compilation time is not reported (Section 5.6); for development iteration, eager execution may be faster despite inferior inference performance |
| **70B+ parameter models** | Unverified | All LLM evaluations use 7–8B models (Section 5.1); scalability to larger models is not demonstrated |

**Integration Pathway.** The paper states Relax "is incorporated into a major open-source project" (Abstract) and "can be used as a backend for frameworks like PyTorch, JAX" (Section 6). This suggests a practical integration path:

1. **Start with PyTorch front-end:** Use Relax's PyTorch-like `nn.Module` interface (Section 5.1) to minimize code changes
2. **Enable partial library lowering first:** Figure 17 shows this provides the largest performance gain (up to 27%)
3. **Add memory planning for constrained targets:** Table 2 shows 40% memory reduction during decode — critical for mobile deployment
4. **Iterate on custom operators:** Use cross-level fusion (Section 4.2) for model-specific optimizations once baseline deployment works

**Reproducibility Considerations.** The paper provides specific version numbers for baselines (HuggingFace Transformers v4.41.2, PyTorch v2.3.1, vLLM v0.5.0.post1, llama.cpp 172c825) in Section 5.1, enabling exact reproduction. However, several factors may affect reproducibility:
- **Hardware variability:** GPU performance depends on specific GPU models; RTX 4090 results may not transfer to other NVIDIA GPUs
- **Quantization details:** Table 3 footnote specifies 3-bit vs. 4-bit quantization for different mobile devices — exact quantization parameters affect both performance and memory
- **Upper bound annotations:** Memory planning requires user-provided upper bounds (Section 4.3); different bounds will produce different memory usage results

### 7.5 Long-Term Vision

The paper's conclusion articulates a broader aspiration: "We hope this work will encourage additional studies of dynamic shape–aware program abstractions and highlight new possibilities for ML compilers." Relax positions itself not as a final solution but as a **proof of concept for cross-level, symbolically-aware compilation** that other systems can adopt.

If the field embraces these abstractions, we may see:
- **Unified deployment stacks** where models compile once and run across servers, phones, and browsers without modification
- **Memory-safe dynamic models** where static planning prevents out-of-memory errors on constrained devices
- **Composable optimization ecosystems** where library dispatch, compiler-generated kernels, and custom operators coexist in a single optimization pipeline

The evaluation demonstrates this vision is technically feasible — the question is whether the ecosystem will adopt the abstractions needed to realize it at scale.
