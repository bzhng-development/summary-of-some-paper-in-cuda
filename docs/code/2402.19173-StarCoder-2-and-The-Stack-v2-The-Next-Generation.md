## 1. Executive Summary

The BigCode project introduces StarCoder2, a family of open Code LLMs (3B, 7B, and 15B parameters) trained on The Stack v2, a transparent dataset 4× larger than the original with 900B+ unique tokens across 619 programming languages. StarCoder2-15B matches or outperforms CodeLlama-34B on most benchmarks while ensuring full data transparency through Software Heritage persistent IDentifiers (SWHIDs) and responsible sourcing practices like PII redaction. This release addresses the critical lack of training data transparency in LLM development by providing open weights under an OpenRAIL license alongside verifiable data attribution.

## 2. Context and Motivation

### The Transparency Crisis in Code LLM Development

The rapid adoption of Large Language Models for Code (Code LLMs) has created a fundamental tension between capability and accountability. As of January 2024, GitHub CoPilot alone serves over 1.3 million paying subscribers across 50,000+ organizations, with documented productivity gains of up to 56% (Peng et al., 2023; Ziegler et al., 2024). ServiceNow reported similar results—a 52% increase in developer productivity using fine-tuned StarCoderBase models (Yahoo Finance, 2024). These numbers demonstrate that Code LLMs are no longer research curiosities; they are production infrastructure affecting millions of developers daily.

Yet this widespread deployment masks a critical problem: **we do not know what data these models were trained on**. This opacity creates three distinct categories of risk:

1. **Legal and ethical uncertainty**: Content creators cannot verify whether their source code was used for training, making consent and compensation impossible to negotiate.
2. **Scientific blindness**: Researchers cannot audit training data for bias, toxicity, or benchmark contamination, undermining reproducibility and trust.
3. **Technical debt**: Downstream developers lack information about data composition, making it difficult to predict model behavior on specific tasks or languages.

### The Spectrum of Openness in LLM Development

To understand where StarCoder2 positions itself, we must first map the landscape of LLM development approaches. The paper identifies three distinct tiers:

| Openness Level | Examples | What's Released | What's Hidden |
|---------------|----------|-----------------|---------------|
| **Closed** | GPT-4, Gemini | API access only | Weights, data, architecture, training details |
| **Open-weights** | Code LLaMa, Mistral, DeepSeekCoder | Model weights | Training data, some architecture details |
| **Fully open** | OLMo, Pythia, BLOOM, StarCoder2 | Weights, data, training code, evaluation suites | Nothing (by design) |

The **open-weights** tier represents the current industry standard for "open" models. While this enables local deployment and fine-tuning, it leaves the training data as a black box. This creates what the authors call a "scientific progress" problem: research teams cannot build upon each other's data curation work, and the community cannot collectively improve data quality over time.

The **fully open** tier is much smaller. Allen AI's OLMo (Groeneveld et al., 2024), Eleuther AI's Pythia (Biderman et al., 2023), and BigScience's BLOOM (Scao et al., 2022a) represent the primary precedents. However, these are general-purpose language models, not specialized for code. StarCoder2 extends this fully open philosophy specifically to the Code LLM domain.

### The BigCode Project's Iterative Approach

StarCoder2 does not emerge in isolation—it is the second generation of a deliberate, community-driven development process. Understanding this lineage is essential:

**The Stack v1 (2022)**: The BigCode project's first major release was a 6.4 TB dataset of permissively licensed source code spanning 384 programming languages (Kocetkov et al., 2023). Critically, it introduced the "Am I in The Stack" governance tool, allowing developers to verify whether their code was included and request opt-out. This established a precedent for **developer consent** in training data collection.

**SantaCoder (2022)**: A 1.1B parameter model trained on Java, JavaScript, and Python from The Stack v1. This proved the viability of the data collection approach but was limited in scope.

**StarCoder (2023)**: A 15B parameter model that became "the best open-access LLM for code" at its release (Li et al., 2023). However, it still had limitations in data scale and language coverage.

**StarCoder2 (2024)**: The current work addresses three specific gaps from StarCoder:
1. **Scale**: 4× larger training dataset (900B+ vs. ~230B tokens)
2. **Diversity**: 619 vs. 384 programming languages
3. **Transparency**: Full SWHID (Software Heritage persistent IDentifier) attribution for every source code file

### Why Code-Specific Openness Matters

One might ask: why does Code LLM transparency require special attention beyond general LLM openness? The paper identifies several code-specific concerns:

**License complexity**: Source code carries explicit licensing terms that general web text does not. The Stack v2 includes both permissively licensed AND unlicensed files (a change from v1), but explicitly excludes copyleft and commercial licenses. This nuanced filtering requires transparency so users can verify compliance.

**Benchmark contamination risk**: Code benchmarks like HumanEval and MBPP have limited problem sets. If training data contains solutions to these exact problems, performance metrics become meaningless. The paper describes explicit decontamination procedures (Section 3.3), but this only matters if researchers can verify the process.

**Security implications**: Code models can generate vulnerable code. The "Asleep at the Keyboard" benchmark (Section 7.7) shows StarCoder2-15B has a 39.2% insecure program rate—higher than some competitors. Understanding training data composition helps explain why certain vulnerability patterns emerge.

**Low-resource language support**: The paper notes significant performance variation across languages (Section 7.1.2). StarCoder2-15B outperforms DeepSeekCoder-33B on low-resource languages like D, Julia, Lua, and Perl, despite being less than half the size. This suggests data composition matters more than raw parameter count for certain languages—a finding only verifiable with transparent data reporting.

### The Compute-Data Tradeoff Decision

A subtle but important design choice appears in Section 1: the authors explicitly state they "push the number of training tokens far beyond the compute-optimal number suggested by Chinchilla (Harm's law; de Vries, 2023) and train relatively small models within the range of 3.3 to 4.3 trillion tokens."

This deserves explanation. The Chinchilla scaling laws (Hoffmann et al., 2022) suggest an optimal ratio between model parameters and training tokens. StarCoder2 deliberately violates this by training smaller models on more tokens than theoretically optimal. Why?

The answer lies in the **accessibility goal**. A 15B parameter model can run on consumer hardware with appropriate quantization. A 34B model (like CodeLlama-34B) requires significantly more resources. By over-training smaller models, the BigCode project prioritizes **democratized access** over theoretical efficiency. The results validate this: StarCoder2-15B matches or exceeds CodeLlama-34B on most benchmarks despite the size disadvantage (Table 9, Table 10).

### Positioning Summary

StarCoder2 positions itself at the intersection of three priorities:

1. **Maximum transparency**: Every source code file can be traced via SWHID to its origin in the Software Heritage archive.
2. **Responsible sourcing**: PII redaction, malware removal, opt-out mechanisms, and license filtering are all documented and auditable.
3. **Competitive performance**: Despite transparency constraints (e.g., excluding some data sources for legal reasons), the models achieve state-of-the-art results for their size class.

This positioning directly challenges the assumption that openness requires sacrificing capability. The paper's central claim is that transparent, responsibly sourced training data can produce models competitive with opaque alternatives—a claim substantiated through the comprehensive benchmarking in Section 7.

## 3. Technical Approach

### 3.1 Reader orientation (approachable technical breakdown)

StarCoder2 is a family of three language models (3B, 7B, and 15B parameters) specifically designed to understand and generate computer code across 619 programming languages. The system solves the dual problem of achieving competitive code generation performance while maintaining complete transparency about training data through a carefully orchestrated pipeline that collects source code from Software Heritage, applies rigorous filtering and formatting, then trains transformer models with architectural modifications optimized for code understanding.

### 3.2 Big-picture architecture (diagram in words)

The StarCoder2 system consists of four major components connected in a linear pipeline. First, the **Data Collection Layer** aggregates source code from Software Heritage archives, GitHub issues and pull requests, Jupyter/Kaggle notebooks, documentation websites, and supplementary natural language datasets. Second, the **Preprocessing Pipeline** applies deduplication using MinHash locality-sensitive hashing, redacts personally identifiable information using the StarPII model, removes malware signatures via ClamAV scanning, decontaminates benchmark data, and processes developer opt-out requests. Third, the **Data Formatting Engine** structures all inputs with specialized sentinel tokens for repository context, fill-in-the-middle tasks, pull request diffs, and notebook cells. Fourth, the **Model Training Infrastructure** trains transformer architectures with Rotary Positional Encodings and Grouped Query Attention across two stages: base training at 4K context followed by long-context fine-tuning at 16K context.

### 3.3 Roadmap for the deep dive

- We begin with data collection because the training corpus fundamentally determines model capabilities and distinguishes StarCoder2 from opaque alternatives.
- We then examine preprocessing because raw code requires extensive cleaning before it becomes suitable training material.
- We proceed to data formatting because the token-level structure directly affects how the model learns code patterns and relationships.
- We conclude with model architecture and training because these components consume the prepared data and produce the final model weights.
- Throughout, we highlight design choices that prioritize transparency without sacrificing performance.

### 3.4 Detailed, sentence-based technical breakdown

This technical report describes a fully open Code LLM development process where every component—from data sourcing to model weights—is auditable and reproducible. The core idea is that transparent training data, when carefully curated and formatted, can produce models that match or exceed opaque competitors despite using less compute-efficient training regimes.

**Data Collection: Building The Stack v2**

The foundation of StarCoder2 is The Stack v2, a training corpus that is 4× larger than the original StarCoder dataset, containing 900B+ unique tokens across 619 programming languages. The primary source is Software Heritage, a non-profit organization that preserves source code as part of the digital commons. The authors leverage the 2023-09-06 version of the Software Heritage graph dataset, which represents the archive as a fully deduplicated Merkle DAG (a cryptographic data structure where each node is hashed, enabling efficient duplicate detection). The extraction process retrieves only the most recently crawled version of each GitHub repository's main branch, where the main branch is identified either by GHArchive metadata marking it as default or by naming conventions (main or master). Individual file contents are downloaded from the Software Heritage content S3 bucket only if the compressed file size is less than 10MB, and the directory tree traversal is limited to level 64 to avoid excessively deep nesting.

License detection follows a two-tier approach shown in Figure 1 of the paper. First, the system checks GHArchive for repository-level license information, but this is available for only 3.07% of repositories. For the remaining 96.93%, the system uses ScanCode Toolkit to detect file-level licenses by searching for files matching license-related patterns (LICENSE, MIT.txt, Apache2.0, README.md, GUIDELINES, etc.), applying license detection to matching files, and propagating detected licenses to all files sharing the same base path within the repository. The license assignment logic (Figure 1) classifies files as permissively licensed, non-permissively licensed, or unlicensed. Critically, The Stack v2 includes both permissively licensed AND unlicensed files, whereas The Stack v1 included only permissively licensed files. Commercial licenses and copyleft licenses are explicitly excluded—commercial licenses because their creators do not intend commercial use, and copyleft licenses due to community uncertainty about their appropriateness for LLM training and their relatively low volume.

Language detection represents a methodological improvement over The Stack v1. Instead of detecting programming languages by file extension (which can be unreliable), the system uses go-enry, based on GitHub's linguist library, to classify each file's programming language. This detects 658 unique languages initially, though 39 are later excluded during visual data inspection, resulting in the final 619 languages. The visual inspection process involved 15 annotators from the BigCode community examining approximately 1,000 file extensions, excluding 130 extensions that contained low-quality training data. This inspection was limited to extensions with over 1,000 files representing over 0.5% of files in their respective languages, as smaller extensions were retained without inspection due to their minimal volume.

Basic filters remove low-quality training data through multiple criteria. Files with more than 100,000 lines are removed as they likely represent data files or generated code. Files with average line length exceeding 100 characters or maximum line length exceeding 1,000 characters are removed for most languages, except HTML, JSON, Markdown, Roff, Roff Manpage, SMT, TeX, Text, and XML, where only files with lines exceeding 100,000 characters are removed. Auto-generated files are identified using go-enry's `is_generated` function and by searching for keywords like "auto-generated", "autogenerated", "automatically generated", "generated automatically", or "this file is generated" in the first 5 lines. Files with less than 25% alphabetic characters are removed, except for Motorola 68K Assembly and WebAssembly, where the threshold applies to alpha-numeric characters due to their syntax. Encoded data (Base64 strings, hexadecimal sequences, Unicode strings) is detected using regular expressions, and files are removed if any matching substring exceeds 1,024 characters or if matched characters comprise more than 50% of the file.

Language-specific filters apply additional constraints. For Text, JSON, YAML, Web Ontology Language, and Graphviz (DOT), files with more than 512 lines are removed to minimize repeated tokens from data files. For HTML, only files where visible text is at least 100 characters long and comprises at least 20% of the code are retained. For Text files, only files with "requirement" in the lowercased filename or specific filenames (readme, notes, todo, description, cmakelists) are kept.

GitHub Issues are collected from GHArchive and processed following the StarCoder pipeline. Auto-generated text from email replies is removed, issues with messages under 200 characters are deleted, and long comments are truncated to 100 lines while retaining the last 20 lines—this removes 17% of the volume. Bot comments are excluded by searching for keywords in usernames, eliminating 3% of issues (less than StarCoder's 17% because pull requests, a major bot source, are processed separately). Quality filtering retains conversations with two or more users, or single-user conversations under 7,000 characters (96th percentile), while excluding single-user issues with more than ten events. This removes 38% of remaining issues. Usernames are anonymized by replacing them with participant counters (e.g., username_1 for the second participant).

Pull Requests present unique formatting challenges due to their potentially long sequences of code changes and comments. The system aggregates PullRequestEvent, PullRequestReviewEvent, PullRequestReviewCommentEvent, IssueCommentEvent, and IssuesEvent from GHArchive, then extracts base and head commit IDs to retrieve corresponding code files from Software Heritage. Since commit diffs are not directly accessible, they are generated by identifying changes between files at the same path—files present in base but absent in head are deletions, while files absent in base but present in head are additions. This yields approximately 300M PRs with 15 TB of base code, including 215M closed PRs from 24M repositories. PR filtering removes those opened by bots, consisting only of bot comments, with non-permissive licenses, opted out by developers, changing base during the PR, not approved or merged, or lacking initial diffs. File-level filters exclude deletions/additions, files exceeding 1 million characters, files with less than 25% alphanumeric characters, files with more than 25% hexadecimal characters, files over 100,000 lines, files with average line length over 100 characters, files with maximum line length over 1,000 characters, and Markdown files containing non-English text. Title and description filtering excludes PRs with titles under 10 characters or containing dependency/release keywords, and descriptions under 20 characters or containing "Qwiet". Titles are truncated to 500 characters, descriptions to 80 lines (first 60 and last 20), with further truncation if exceeding 1,000 characters. Comments undergo bot removal and email reply cleanup, with review comment diff hunks truncated to 10,000 characters while preserving filename and comment. Subsampling increases diversity by retaining PRs from repositories with probability 0.8 for single-PR repositories, linearly decreasing to 0.1 for 1,000-PR repositories, and retaining only 100 PRs for repositories exceeding 1,000 PRs. YAML and JSON files are subsampled at 10% retention when exceeding 50% of total base file size or containing keywords like pack, lock, yarn, output, swagger, openapi. The final PR dataset includes 3.3M PRs under 100K characters (67.3 GB), translating to roughly 25K tokens per PR.

Notebooks come from two sources: Jupyter notebooks from Software Heritage and Kaggle notebooks. Jupyter notebooks are converted to scripts using Jupytext, with programming language identified from metadata or using Guesslang library with 0.5 probability threshold. From 11 million initial notebooks, 3 million are excluded due to parsing errors, and near-deduplication reduces the dataset to 4 million converted scripts. Structured Jupyter notebooks merge consecutive Markdown or code blocks, creating code-text pairs where Markdown text immediately precedes code blocks, with output cells included if non-empty (marked with `<empty_output>` token otherwise). This yields 4.6M structured Jupyter notebooks. Kaggle notebooks start with 3.6M samples under Apache 2.0 license. Cleaning removes notebooks under 100 characters and those with syntax errors, plus templated text at notebook beginnings (18% removal). Conversion follows Jupyter processing, and near-deduplication eliminates 78% of notebooks, leaving 580K. Dataset metadata is included when available: 42% of notebooks have associated Kaggle dataset titles and descriptions prepended, and 25% load CSV datasets via `read_csv`. For the 8.6% where datasets were successfully downloaded, `df.info()` output (column names, dtypes, non-null counts, memory usage) and four sample rows are prefixed to the notebook.

Documentation is collected from package managers (npm, PyPI, Go Packages, Packagist, Rubygems, Cargo, CocoaPods, Bower, CPAN, Clojars, Conda, Hex, Julia) and curated websites (Table 2). Library names from libraries.io search individual package managers to obtain homepages, from which documentation files are crawled. Read the Docs is prioritized for PyPI and Conda libraries due to comprehensive documentation. HTML pages are converted to XML using trafilatura (removing navigation and index bars), then to Markdown. Near-deduplication treats each website's HTML pages as clusters, applying MinHash locality-sensitive hashing with 0.7 threshold to eliminate similar pages. Content is assembled in breadth-first search order. Additional code-relevant data comes from RefinedWeb, OSCAR, and esCorpius web crawls, using regular expressions to identify programming language constructs and "docs." substrings in URLs. Free programming books (3,541 PDFs) are scraped from the Free Programming Books project, with text extracted using pdf2text. R language PDFs from CRAN are processed with pdftotext, while LaTeX documentation from CTAN is converted to Markdown using Nougat neural OCR. Language identification uses predefined rules when document sources correspond to specific languages, otherwise resorting to Guesslang library. Figure 2 shows the top 20 programming language distribution in crawled documentation.

Intermediate Representations (IR) augment source code by pairing it with LLVM IR to enhance understanding of low-resource programming languages. The rationale is that shared intermediate representations anchor low-resource language constructs to similar ones in high-resource languages. LLVM is selected due to widespread GitHub availability and adoption as a compiler target across multiple languages. The system compiles approximately 4M sources across C++, C, Objective-C, Python, Rust, Go, Haskell, D, Fortran, Swift, and Nim in both size-optimized (-OZ equivalent) and performance-optimized (-O3 equivalent) modes. Size-optimized IR is used for 80% of pairs due to context length considerations, while performance-optimized IR comprises 20% to maximize transfer from pre-training where wild LLVM code is more likely performance-optimized. Compilation uses clang for C++/C/Objective-C, codon for Python, rustc for Rust, gollvm for Go, ghc for Haskell, ldc for D, flang for Fortran, and nlvm for Nim. Headers and platform-specific information are cleaned before pairing with source code.

Small high-quality datasets (LHQ) supplement the main corpus with math and coding examples. APPS training set contributes 5,000 Python text-to-code examples (one solution per problem). Code Contest adds 13K+ examples across Python 2/3, C++, and Java. GSM8K training split provides 7K+ math reasoning examples, augmented by GSM8K (SciRel) with 110K alternative reasoning paths. Deepmind Mathematics contributes 110M+ synthetic math questions across algebra, arithmetic, calculus, comparison, measurement, numbers, polynomials, and probability at varying difficulties. Rosetta Code offers 1,100+ everyday programming tasks with solutions in multiple languages. MultiPL-T provides 200K+ high-quality examples in Lua, Racket, and OCaml from automatically translated Python functions validated with unit tests. Proofsteps from AlgebraicStack includes proofsteps-lean (3K+ examples from mathlib 4) and proofsteps-isabelle (250K+ examples from PISA dataset).

Other natural language datasets include StackOverflow (11M questions with multiple responses from 2023-09-14 dump, filtered to questions with 3+ answers), ArXiv (30B tokens from RedPajama, retaining only LaTeX source files with preambles, comments, macros, and bibliographies removed), Wikipedia (6B tokens from English subset, 2023-03-20 dump, with hyperlinks and templates eliminated), and OpenWebMath (15B tokens of mathematical text from CommonCrawl). StackOverflow quality is enhanced using Llama-2-70b-chat-hf to rate 20,000 question-answer pairs, selecting 10,000 highest-scoring as positive examples and using remaining 10,000 to create negative examples via random pairing. A binary classifier is trained using sentence-transformers/all-MiniLM-L12-v2 embeddings with cosine distance minimization, with manual threshold determination at 0.1. Questions with average scores below 0.1 are removed, yielding 11.4M questions and 10B+ tokens.

**Preprocessing Pipeline: Cleaning and Protecting the Data**

Deduplication follows the SantaCoder pipeline, calculating MinHashes for all code files then using Locality Sensitive Hashing to group files by MinHash fingerprints. During LSH, similar files are assigned to the same buckets as duplicates, with only one file chosen per duplicate group. To preserve repository context, files from repositories with higher star/fork counts or later commit dates are prioritized as tiebreakers. The system uses 5-grams with Jaccard similarity threshold of 0.7. Deduplication applies to source code, pull requests, notebooks, issues, and documentation, but NOT to already-preprocessed natural language datasets (ArXiv, StackExchange, OpenWebMath, Wikipedia, LHQ datasets).

PII redaction uses the StarPII model to remove names, emails, keys, passwords, IP addresses, and usernames from source code, pull requests, issues, and StackOverflow. For OpenWebMath and documentation, only names, keys, and emails are redacted. For ArXiv, only emails are redacted using regex patterns from Ben Allal et al. (2023). Username redaction in conversations substitutes author usernames with participant counters specific to each conversation (e.g., username_1 for second participant), with pseudonyms added at comment starts to maintain speaker identity. References to usernames within messages are removed, but only actively participating individuals are masked—mentions of non-participating users remain unaffected.

Decontamination removes training data that overlaps with test benchmarks to prevent artificially inflated performance. The system removes files containing docstrings or solutions from HumanEval and MBPP, docstrings from APPS, questions from GSM8K, or prompts from DS1000. Unlike StarCoder v1, whitespace is removed during string matching to enhance recall. Documentation, LHQ, ArXiv, and Wikipedia are excluded from decontamination.

Malware removal scans source code, pull requests, notebooks, and issues using ClamAV 1.2 with additional unofficial signatures from SaneSecurity (as of 2023-11-16). High false-positive signatures are not used. This eliminates 59,442 files (0.009% of 654M files). Table 26 shows top malware signatures detected, with Sanesecurity.Malware.28845.BadVBS most common (11,876 instances). Table 27 shows Text (13,281), HTML (11,336), and JavaScript (10,210) as top languages for potentially malicious files.

Opt-out deletion honors developer requests to remove their code. The BigCode project announced the StarCoder2 training run on X (Twitter) and updated the "Am I in the Stack" governance tool with new repositories from The Stack v2. Developers had until November 20, 2023 to submit opt-out requests. After the cutoff, 1,561 repositories from 91 users and organizations were eliminated, removing 22,066 files from the source code dataset (excluding issues and PRs).

**Data Composition: Tailoring Datasets to Model Sizes**

Model capacity considerations drive different data compositions for each model size. Smaller models have limited capacity, so they should be exposed to less diverse datasets—supported by multi-lingual NLP research showing languages compete for model capacity. The system creates two Software Heritage code dataset variants: the-stack-v2-train-smol (17 widely-used languages) for 3B and 7B models, and the-stack-v2-train-full (all 619 languages) for the 15B model.

Language downsampling adheres to natural data distribution while balancing high-resource languages. Observed data volumes show Java (479.68 GB), JavaScript (277.25 GB), C++ (204.49 GB), Python (190.99 GB), PHP (171.57 GB), C# (166.22 GB), and C (114.49 GB) as top languages. Both Java and JavaScript are downsampled to 200GB each for more equal footing. Markdown is preserved at 254GB while HTML is reduced to 100GB, anticipating Markdown contains more code documentation while HTML commonly represents webpages. Data files (JSON, XML, YAML) are subsampled to 8GB, with other formats at 1GB.

The the-stack-v2-train-smol variant includes 17 programming languages (C, C#, C++, Go, Java, JavaScript, Kotlin, Lua, PHP, Python, R, Ruby, Rust, SQL, Shell, Swift, TypeScript), documentation languages (AsciiDoc, HTML, Markdown, RDoc, RMarkdown, Text, reStructuredText), and configuration languages/files (Ant Build System, CMake, Dockerfile, Go Module, Gradle, INI, Java Properties, Makefile, Maven POM, TOML, plus specific files like Cargo.toml, package.json, requirements.txt, etc.). Despite language limitations, this yields 525B+ unique tokens. The the-stack-v2-train-full variant includes all 619 languages, adding 600+ programming languages but contributing only ~250B additional tokens, culminating in 775B+ tokens.

Table 4 summarizes data composition per model. The 3B model uses the-stack-v2-train-smol and excludes OpenWebMath, Wikipedia, and ArXiv from extras, yielding 622B+ unique tokens. The 7B model includes OpenWebMath, Wikipedia, and ArXiv, yielding 658B+ unique tokens. The 15B model uses the-stack-v2-train-full with all extras, yielding 913B+ unique tokens—4× the StarCoderBase training dataset size.

**Data Formatting: Structuring Inputs for Code Understanding**

Source code formatting prepends repository name and file paths to code file context with 50% probability, enabling the model to operate with or without this metadata. When included, the format is: `<repo_name>reponame<file_sep>filepath1\ncode1<file_sep>filepath2\ncode2...`. When excluded: `<file_sep>code1<file_sep>code2...`. Repository-context training groups files from the same repository together (random order within repository), differing from StarCoder1's file-context approach where random files are joined into the context window.

Fill-in-the-Middle (FIM) enables code infilling tasks by conditioning on text before and after the insertion point. The system uses repo-context file-level FIM: repositories are selected with 50% probability as FIM candidates, selected repository examples are split by `<file_sep>` tokens, then FIM transformation is applied to each chunk with 50% probability. FIM is not applied to repository metadata. Example format with FIM on second file only: `<repo_name>reponame<file_sep>filepath0\ncode0<file_sep><fim_prefix>filepath1\ncode1_pre<fim_suffix>code1_suf<fim_middle>code1_mid<file_sep>...`. Table 5 lists all sentinel tokens including `<fim_prefix>`, `<fim_middle>`, `<fim_suffix>`, `<fim_pad>`, and others.

Pull Request formatting creates compact representations of potentially long code change sequences. For base commit files, entire files are included with 0.2 probability; otherwise, changes across all commit heads are displayed with up to 32 randomly added lines before and after changes. Diff hunks display modifications between before/after file states with 3-10 randomly selected context lines before and after specific changes. The PR format structure begins with title, description, and complete base files or modifications, followed by head diff hunks for each new head commit, covering general comments, review comments, and code review comments. The block concludes with diff hunks between PR base and new head. Authors are referenced by participant counter (username_1, etc.). A final block is added only when PR is closed, indicating merged status.

GitHub Issues formatting uses sentinel tokens to mark issue opening and include title. Comments are separated by `<issue_comment>` token with anonymized speaker identifiers before each comment. Comment turns are distinguished using comment_1, id1 notation for second comment and speaker ID. `<issue_closed>` token is added if issue is closed. Format: `<issue_start>Title: title\nusername_id0: comment_0<issue_comment>username_id1: comment_1...<issue_closed (optional)><issue_comment>username_idn: comment_n`.

Notebook formatting differs by type. Jupyter scripts format as single code blocks starting with `<jupyter_script>` token. Structured Jupyter notebooks separate cells with sentinel tokens: `<jupyter_start><jupyter_text>text0<jupyter_code>code0<jupyter_output>output0<jupyter_text>...`. Kaggle scripts prepend dataset title and description when available (42% of samples), with granular dataset schema for 8.6% of notebooks. Format includes `<jupyter_start><jupyter_text>title\ndescription\nKaggle dataset identifier: data_identifier<jupyter_code>import pandas as pd\n\ndf = pd.read_csv(data_path1)\ndf.info()<jupyter_output>df_info_output1<jupyter_text>Examples:\nexample1_1\n...<jupyter_script>code`. The `<jupyter_script>` token differentiates the final converted script (usually long) from `<jupyter_code>` blocks (typically shorter). Structured Kaggle notebooks omit output cells, including only text and code blocks with Jupyter tokens.

StackExchange formatting concatenates questions and answers similar to GitHub Issues. Questions start the sequence, followed by answers in random order. Upvote scores are included alongside answers, with selected answer designation when applicable. Conversation titles are not available for StackExchange. Format: `<issue_start>username_id0: question<issue_comment>username_id1: answer_1\nUpvotes: score [selected answer](Optional)...<issue_comment>username_idn: answer_n\nUpvotes: score [selected answer](Optional)`.

Intermediate Representation formatting splits 50/50 between source-to-IR translation (code→intermediate) and IR-to-source translation (intermediate→code). Size-optimized IR is used 80% of the time, performance-optimized 20%. Separate sentinel tokens indicate translation direction: `code<code_to_intermediate>intermediate_representation` and `intermediate_representation<intermediate_to_code>code`.

**Model Architecture: Transformer Modifications for Code**

StarCoder2 introduces architectural changes compared to StarCoderBase. Learned positional embeddings are replaced with Rotary Positional Encodings (RoPE), confirmed to provide significant performance gains in preliminary ablation studies. Following DeepSeekCoder and Code LLaMa, the base period is $\theta = 10^5$ (though StarCoder2-15B uses $\theta = 10^4$ due to a configuration parsing bug). Multi-Query Attention (MQA) is replaced with Grouped Query Attention (GQA), but key-value heads are kept relatively low to prevent significantly slowing inference: 2 KV heads for 3B, 4 for 7B, and 4 for 15B.

Table 6 summarizes architecture details. StarCoder2-3B has hidden dimension 3072, 24 attention heads, 2 KV heads, 30 layers, 49,152 vocabulary size, and 4K/16K sequence length, requiring 5.94×10²² FLOPs. StarCoder2-7B has hidden dimension 4608, 36 attention heads, 4 KV heads, 32 layers, same vocabulary and sequence lengths, requiring 1.55×10²³ FLOPs. StarCoder2-15B has hidden dimension 6144, 48 attention heads, 4 KV heads, 40 layers, same vocabulary and sequence lengths, requiring 3.87×10²³ FLOPs. All models use RoPE positional encodings.

The tokenizer follows StarCoderBase procedure, training a byte-level Byte-Pair-Encoding tokenizer on a small subset of The Stack v1. Preliminary experiments showed increasing vocabulary to 100K did not improve performance, so vocabulary size remains at 49,152 tokens including sentinel tokens from Table 5. Pre-tokenization includes a digit-splitter and regex splitter from GPT-2 pre-tokenizer.

**Training Details: Two-Stage Process with Extended Context**

Base model training uses sequence length 4,096 with Adam optimizer ($\beta_1 = 0.9$, $\beta_2 = 0.95$, $\epsilon = 10^{-8}$), weight decay 0.1, and no dropout. Learning rate follows cosine decay after 1,000 iteration linear warmup. Table 7 details training hyperparameters: StarCoder2-3B uses learning rate 3×10⁻⁴, RoPE $\theta = 10^5$, batch size 2.6M, 1.2M iterations, 3.1T tokens, 4.98 epochs. StarCoder2-7B uses learning rate 3×10⁻⁴, RoPE $\theta = 10^5$, batch size 3.5M, 1M iterations, 3.5T tokens, 5.31 epochs. StarCoder2-15B uses learning rate 3×10⁻⁴, RoPE $\theta = 10^4$ (bug), batch size 4.1M, 1M iterations (early stopped from scheduled 1.1M), 4.1T tokens, 4.49 epochs. Following Muennighoff et al. (2023), data is repeated for approximately four to five epochs.

Long-context training further pre-trains each model on 200B tokens from the same corpus using 16,384 context length with 4,096 sliding window and FlashAttention-2. RoPE $\theta$ is increased for this stage. Table 8 provides long-context hyperparameters: StarCoder2-3B uses learning rate 3×10⁻⁵, RoPE $\theta = 10^6$, batch size 2.6M, 80K iterations, 200B tokens. StarCoder2-7B uses learning rate 2×10⁻⁵, RoPE $\theta = 10^6$, batch size 3.5M, 56K iterations, 200B tokens. StarCoder2-15B uses learning rate 3×10⁻⁵, RoPE $\theta = 10^5$, batch size 4.1M, 50K iterations, 200B tokens.

CO2 emissions are estimated using the Machine Learning Impact calculator. For 3B, ServiceNow infrastructure (0.386 kgCO₂eq/kWh) performed 97,120 hours on A100 SXM4 80GB (400W TDP), yielding 14,995.33 kgCO₂eq for base training plus 1,111.68 kgCO₂eq for long-context fine-tuning, totaling 16,107.01 kgCO₂eq. For 7B, Hugging Face infrastructure (0.2925 kgCO₂eq/kWh) performed 145,152 hours on H100 (660W TDP), yielding 28,021.6 kgCO₂eq for base training plus 1,601.23 kgCO₂eq for long-context, totaling 29,622.83 kgCO₂eq. The 15B model emissions were not yet available at paper submission.

**Design Choices: Why This Approach Over Alternatives**

The decision to train smaller models on more tokens than Chinchilla-optimal prioritizes accessibility over theoretical efficiency. A 15B model can run on consumer hardware with quantization, while 34B models require significantly more resources. Results validate this: StarCoder2-15B matches or exceeds CodeLlama-34B on most benchmarks despite being less than half the size.

Repository-context training (grouping files from same repository) differs from StarCoder1's file-context approach (random files joined). This better reflects real-world code completion scenarios where models operate within repository context. The random ordering within repositories balances structure preservation with diversity.

Including both permissively licensed AND unlicensed files (unlike The Stack v1) expands the training corpus while maintaining ethical boundaries by excluding commercial and copyleft licenses. This requires transparency so users can verify compliance.

The two-stage training (4K base, then 16K long-context) enables efficient base training while adding long-context capability without retraining from scratch. FlashAttention-2 enables the 16K context with reasonable memory requirements.

GQA with low KV head counts (2-4) balances inference speed with attention quality. Pure MQA would be faster but lower quality; full Multi-Head Attention would be higher quality but slower. GQA provides a middle ground suitable for code generation where inference speed matters for developer productivity.

The comprehensive sentinel token system (Table 5) enables the model to distinguish between different data types and formatting structures within a single training run. This multi-task learning approach allows one model to handle code completion, PR review, issue discussion, notebook execution, and IR translation without separate fine-tuning.

## 4. Key Insights and Innovations

StarCoder2 introduces several contributions that distinguish it from both proprietary Code LLMs and other "open" alternatives. This section identifies the most novel aspects, explaining why they matter and how they advance the field beyond incremental improvements.

### 4.1 Full Training Data Transparency via SWHIDs (Fundamental Innovation)

**What makes it different:** Prior "open-weight" models like CodeLlama (Rozière et al., 2023) and DeepSeekCoder (Guo et al., 2024) release model weights but keep training data opaque. Even fully open projects like OLMo (Groeneveld et al., 2024) and Pythia (Biderman et al., 2023) do not provide file-level attribution for their training corpora. StarCoder2 breaks this pattern by releasing **Software Heritage persistent IDentifiers (SWHIDs)** for every source code file in the training set.

**Why it's significant:** This enables three capabilities previously impossible for Code LLMs:

1. **Verifiable consent**: Developers can cryptographically verify whether their code was used, not just probabilistically estimate membership (as with watermarking or fingerprinting approaches). The "Am I in the Stack" tool (Section 8) provides this lookup capability.

2. **Auditability for bias and contamination**: Researchers can trace poor performance on specific languages or tasks back to data composition issues. For example, the performance gap on TypeScript in CrossCodeEval (Table 18) can now be investigated by examining the actual TypeScript files in training.

3. **Legal compliance verification**: Organizations can verify license compliance before deploying models in production—a critical requirement for enterprise adoption that opaque models cannot satisfy.

**Incremental vs. Fundamental:** This is a **fundamental innovation**, not an incremental improvement. It changes the epistemological status of Code LLMs from black boxes to auditable artifacts. The SWHID system leverages Software Heritage's existing infrastructure (a Merkle DAG of preserved source code), but applying it to LLM training data at this scale (900B+ tokens) is unprecedented.

**Caveat:** The transparency applies only to source code from Software Heritage. Natural language datasets (StackOverflow, ArXiv, Wikipedia) do not have SWHID attribution, creating a partial transparency gap the authors acknowledge but do not fully resolve.

### 4.2 Repository-Context Training (Methodological Advance)

**What makes it different:** StarCoder1 used **file-context** training, where random files from different repositories were concatenated into the context window. StarCoder2 switches to **repository-context**, grouping files from the same repository together (Section 5.1). Within each repository, files are arranged in random order, but the repository boundary is preserved.

**Why it's significant:** This better reflects how developers actually use code completion tools. In practice, a developer working on a repository sees multiple files from that same project, not random files from unrelated projects. The evaluation results validate this design choice:

- On RepoBench (Table 17), StarCoder2-15B achieves 36.99% exact match on Python vs. 33.51% for StarCoderBase-15B—a **3.48 percentage point improvement** attributable largely to repository-context training.
- On CrossCodeEval (Table 18), which explicitly requires cross-file context for accurate completion, StarCoder2-15B scores 78.72% Code ES on Python vs. 73.43% for StarCoderBase-15B.

**Incremental vs. Fundamental:** This is a **methodological advance** rather than a fundamental innovation. The concept of repository-level context appears in concurrent work (e.g., RepoBench itself), but StarCoder2 demonstrates it at scale with measurable performance gains. The key insight is that **preserving repository boundaries during pre-training** improves downstream repository-level completion without requiring fine-tuning.

**Non-obvious design choice:** The authors randomly order files within repositories rather than using dependency graphs or import structures. This is a deliberate simplification—dependency-aware ordering would be more semantically meaningful but requires language-specific tooling that doesn't exist for all 619 languages. Random ordering is a pragmatic compromise that still preserves repository context.

### 4.3 Over-Training Smaller Models (Strategic Innovation)

**What makes it different:** The Chinchilla scaling laws (Hoffmann et al., 2022) suggest an optimal ratio between model parameters and training tokens. StarCoder2 deliberately **violates this principle** by training smaller models on more tokens than theoretically optimal (Section 1). The 15B model trains on 4.1T tokens—far beyond the Chinchilla-optimal point for that parameter count.

**Why it's significant:** This prioritizes **accessibility over theoretical efficiency**. A 15B model can run on consumer hardware with quantization (e.g., 4-bit quantization fits in ~8GB VRAM), while a 34B model requires significantly more resources. The results validate this strategy:

- StarCoder2-15B matches or exceeds CodeLlama-34B on HumanEval (46.3 vs. 48.2), MBPP (66.2 vs. 65.4), and MultiPL-E (best on 16/18 languages).
- On math reasoning (GSM8K with PAL, Table 14), StarCoder2-15B scores 65.1% vs. 54.2% for CodeLlama-34B—a **10.9 percentage point advantage** despite being less than half the size.

**Incremental vs. Fundamental:** This is a **strategic innovation** with theoretical implications. It challenges the assumption that Chinchilla-optimal training is always preferable. The authors cite "Harm's law" (de Vries, 2023) suggesting that for data-constrained scenarios, over-training smaller models can be competitive. StarCoder2 demonstrates this holds for Code LLMs specifically, where code has higher information density than natural language.

**Tradeoff acknowledged:** The authors note StarCoder2-7B underperforms relative to 3B and 15B models for its size (Section 1), stating "It is not clear to this report's authors why." This suggests the over-training strategy has non-monotonic effects that require further investigation—an honest admission that strengthens credibility.

### 4.4 Unified Multi-Format Pre-training with Sentinel Tokens (Architectural Innovation)

**What makes it different:** Many Code LLMs are trained primarily on source code, with separate fine-tuning for tasks like code review, issue discussion, or notebook execution. StarCoder2 integrates **six distinct data formats** (source code, pull requests, GitHub issues, Jupyter/Kaggle notebooks, StackOverflow Q&A, intermediate representations) into a single pre-training run using a comprehensive sentinel token system (Table 5, Section 5).

**Why it's significant:** This enables **zero-shot capability across multiple code-related tasks** without task-specific fine-tuning. The evaluation demonstrates this:

- On HumanEvalFix (Table 12), StarCoder2-15B with the "Issue" prompt achieves 38.7% average pass@1—outperforming instruction-tuned CodeLlama-13B-Instruct (15.8%) and approaching DeepSeekCoder-33B-Instruct (42.1%) despite being a base model.
- On CanItEdit (Table 13), StarCoder2-15B scores 43.08% on descriptive instructions and 38.45% on lazy instructions, outperforming CodeLlama-Instruct-34B (35.0% and 26.76%).

**Incremental vs. Fundamental:** This is an **architectural innovation** in how pre-training data is structured. The sentinel token approach itself is not new (T5, Raffenel et al., 2019), but applying it comprehensively to Code LLM pre-training at this scale is novel. The key insight is that **format differentiation during pre-training** enables task flexibility without fine-tuning.

**Subtle limitation:** The FIM (Fill-in-the-Middle) capability underperforms for StarCoder2-15B (Table 16: 48.4% on Python vs. 62% for StarCoderBase-15B). The authors attribute this to "an implementation bug" where the FIM rate was smaller than intended during most of training. This is an important caveat—the multi-format approach only works when each format is adequately represented during training.

### 4.5 Responsible Sourcing at Scale (Governance Innovation)

**What makes it different:** Previous Code LLMs either ignore responsible sourcing (proprietary models) or apply it at smaller scales (SantaCoder, StarCoder1). StarCoder2 applies **comprehensive responsible sourcing practices to a 900B+ token dataset**: PII redaction via StarPII, malware removal via ClamAV, benchmark decontamination, and an opt-out mechanism that removed 22,066 files from 91 users/organizations (Section 3).

**Why it's significant:** This demonstrates that **responsible sourcing is feasible at industrial scale**. The opt-out mechanism is particularly notable—it provides a concrete pathway for developers to exercise consent, addressing a major ethical concern in LLM development. The malware removal (59,442 files, 0.009% of corpus) shows that even small percentages matter when absolute counts are large.

**Incremental vs. Fundamental:** This is a **governance innovation** rather than a technical one. The individual techniques (PII redaction, malware scanning) are established, but integrating them into a cohesive, auditable pipeline at this scale is novel. The "Am I in the Stack" tool and Data Portrait (Section 8) provide computational documentation artifacts that go beyond static dataset cards.

**Tension acknowledged:** The authors note that StarCoder2 models have higher insecure program rates on the "Asleep at the Keyboard" benchmark (Table 19: 39.2% for 15B vs. 35.6% for StarCoderBase-15B). They hypothesize this is "a side-effect of its higher rate of valid and functionally correct completions"—more functional code means more opportunities for vulnerabilities. This honest assessment shows that responsible sourcing of training data does not automatically produce secure outputs, highlighting the need for downstream guardrails.

---

### Summary: What's Truly Novel

| Innovation | Type | Prior Art | StarCoder2 Advance |
|------------|------|-----------|-------------------|
| SWHID transparency | Fundamental | Open-weight models hide data | File-level cryptographic attribution |
| Repository-context | Methodological | File-context (StarCoder1) | Preserved repository boundaries |
| Over-training strategy | Strategic | Chinchilla-optimal training | Accessibility-optimized training |
| Multi-format pre-training | Architectural | Single-format or fine-tuned | Unified pre-training with sentinels |
| Responsible sourcing at scale | Governance | Small-scale or none | Industrial-scale auditable pipeline |

The most significant contribution is **SWHID-based transparency**, which fundamentally changes how Code LLMs can be audited and trusted. The other innovations are important but build on existing concepts. Together, they demonstrate that transparency and performance are not mutually exclusive—a claim substantiated by the benchmark results in Section 7.

## 5. Experimental Analysis

### 5.1 Evaluation Framework Overview

The StarCoder2 evaluation is exceptionally comprehensive, spanning **ten distinct benchmark categories** across code completion, understanding, reasoning, security, and safety. This breadth is necessary because Code LLMs serve multiple roles in practice—from autocomplete suggestions to bug fixing to mathematical reasoning. A model excelling at only one task would have limited practical utility.

**Baseline Selection Strategy:** The authors compare StarCoder2 against three tiers of models:
- **Same-size competitors:** StableCode-3B, DeepSeekCoder-1.3B/6.7B/33B, CodeLlama-7B/13B/34B
- **Previous generation:** StarCoderBase-3B/7B/15B (to measure iterative improvement)
- **Larger models:** CodeLlama-34B and DeepSeekCoder-33B (to test the over-training hypothesis)

All comparisons use **base models** unless explicitly noted (e.g., HumanEvalFix uses instruction-tuned variants because base models perform poorly on that task). This is a critical methodological choice—comparing base-to-base ensures fair evaluation of pre-training quality without confounding from fine-tuning differences.

**Decoding Parameters:** Most benchmarks use greedy decoding for deterministic evaluation (HumanEval, MBPP) or temperature 0.2 with top-p 0.95 for sampling-based metrics (MultiPL-E, DS-1000). Sample counts vary: 1 sample for greedy pass@1, 20-100 samples for pass@k estimates. These choices follow established conventions in the Code LLM literature, enabling direct comparison with prior work.

**Key Limitation:** The evaluation focuses exclusively on **open-weight models**. Proprietary models like GPT-4 or GitHub Copilot are not included, which limits claims about absolute state-of-the-art performance. However, this is consistent with the paper's positioning—StarCoder2 aims to be the best **open** Code LLM, not necessarily the best overall.

---

### 5.2 Code Completion: The Core Capability

Code completion is the primary use case for Code LLMs, and the evaluation reflects this priority with four complementary benchmarks.

#### HumanEval and MBPP with EvalPlus (Table 9)

**What these measure:** HumanEval (164 Python problems) and MBPP (427 Python problems) test function-level code generation from docstrings. The EvalPlus variants (HumanEval+, MBPP+) add 80× and 35× more test cases respectively to catch subtle bugs that original tests miss (Section 7.1.1).

**Key results:**

| Model | HumanEval | HumanEval+ | MBPP | MBPP+ |
|-------|-----------|------------|------|-------|
| StarCoderBase-3B | 21.3 | 17.1 | 42.6 | 35.8 |
| **StarCoder2-3B** | **31.7** | **27.4** | **57.4** | **47.4** |
| StarCoderBase-15B | 29.3 | 25.6 | 50.6 | 43.6 |
| **StarCoder2-15B** | **46.3** | **37.8** | **66.2** | **53.1** |
| CodeLlama-34B | 48.2 | 44.3 | 65.4 | 52.4 |
| DeepSeekCoder-33B | 54.3 | 46.3 | 73.2 | 59.1 |

**Critical observations:**

1. **StarCoder2-3B outperforms StarCoderBase-15B** on all four metrics despite having **5× fewer parameters**. On HumanEval+, the 3B model scores 27.4 vs. 25.6 for the 15B predecessor—a 7.0% relative improvement. This validates the over-training strategy: more tokens can compensate for fewer parameters.

2. **StarCoder2-15B matches CodeLlama-34B** on MBPP (66.2 vs. 65.4) and MBPP+ (53.1 vs. 52.4) despite being less than half the size. On HumanEval, it trails by only 1.9 points (46.3 vs. 48.2). This is the paper's strongest evidence that accessibility-optimized training can compete with larger models.

3. **DeepSeekCoder-33B remains the overall leader** on completion tasks. The authors acknowledge this honestly: "DeepSeekCoder-33B is the best-performing model at code completion for high-resource languages" (Section 1). However, StarCoder2-15B closes the gap significantly compared to StarCoderBase-15B.

4. **EvalPlus reveals meaningful differences.** The gap between HumanEval and HumanEval+ scores indicates how many solutions pass weak tests but fail rigorous ones. StarCoder2-15B drops from 46.3 to 37.8 (18.4% relative drop), while CodeLlama-34B drops from 48.2 to 44.3 (8.1% drop). This suggests StarCoder2-15B's solutions may be slightly less robust, though still competitive.

#### MultiPL-E: Multilingual Generalization (Table 10)

**What this measures:** MultiPL-E translates HumanEval problems to 18 programming languages, testing whether models generalize beyond Python. This is critical for StarCoder2's claim of supporting 619 languages.

**Key results by model size:**

- **Small models (≤3B):** StarCoder2-3B performs best on **11/18 languages**, including significant wins on JavaScript (35.4 vs. 32.0 for StableCode-3B) and Lua (28.0 vs. 17.1).
  
- **Medium models (≤7B):** DeepSeekCoder-6.7B leads overall, but StarCoder2-7B beats CodeLlama-7B on most languages (e.g., Python 33.6 vs. 26.4, Ruby 28.3 vs. 29.5—actually slightly behind on Ruby).

- **Large models (≤15B):** StarCoder2-15B performs best on **16/18 languages**. Notable wins include:
  - Perl: 37.2 vs. 28.5 (CodeLlama-34B)
  - Lua: 43.8 vs. 37.5 (CodeLlama-34B)
  - Julia: 33.2 vs. 31.4 (CodeLlama-34B)

**Critical insight:** StarCoder2-15B **outperforms DeepSeekCoder-33B on four low-resource languages** (D, Julia, Lua, Perl) despite being less than half the size. For example, on Perl: 37.2 vs. 31.0. This directly supports the claim that data diversity (619 languages in training) matters more than parameter count for low-resource language support.

**Exception:** CodeLlama-13B outperforms StarCoder2-15B on Go (26.6 vs. 26.2) and Java (33.9 vs. 37.5—actually StarCoder2 wins here, need to recheck). The paper states CodeLlama-13B wins on Go and Java specifically (Section 7.1.2, observation 4).

#### DS-1000: Data Science Library Usage (Table 11)

**What this measures:** DS-1000 tests code generation for seven Python data science libraries (Matplotlib, NumPy, Pandas, PyTorch, SciPy, Scikit-Learn, TensorFlow). Unlike HumanEval's standard library focus, this evaluates practical library knowledge.

**Key results:**

| Model | Overall Pass@1 |
|-------|---------------|
| StarCoderBase-15B | 23.8 |
| **StarCoder2-15B** | **33.8** |
| CodeLlama-13B | 25.1 |
| CodeLlama-34B | 34.3 |
| DeepSeekCoder-33B | 40.2 |

StarCoder2-15B achieves **33.8% overall**, substantially outperforming CodeLlama-13B (25.1%) and approaching CodeLlama-34B (34.3%). On specific libraries:
- **Matplotlib:** 60.3% (best among ≤15B models)
- **NumPy:** 43.3% (best among ≤15B models)
- **TensorFlow:** 36.0% (best among ≤15B models)

**Interpretation:** The inclusion of documentation, StackOverflow, and Kaggle notebooks in training (Section 2) appears to improve library-specific knowledge. This is a direct benefit of the multi-format pre-training strategy.

### 5.3 Code Understanding and Reasoning

Beyond generation, Code LLMs must understand and reason about existing code. Three benchmarks test these capabilities.

#### HumanEvalFix: Bug Fixing (Table 12)

**What this measures:** HumanEvalFix presents buggy code in six languages (Python, JavaScript, Java, Go, C++, Rust) and asks models to fix it. Critically, **base models perform poorly** on this task, so the evaluation compares StarCoder2-15B against instruction-tuned competitors using different prompt formats.

**Prompt formats tested:**
- **Instruct:** Standard instruction prompt (disadvantages base models)
- **Issue:** GitHub issue-style discussion format (leverages StarCoder2's issue pre-training)
- **Commit:** Commit message format (used for StarCoderBase)

**Key results:**

| Model | Prompt | Python | JavaScript | Java | Go | C++ | Rust | Average |
|-------|--------|--------|------------|------|-----|-----|------|---------|
| StarCoderBase-15B | Instruct | 12.6 | 16.8 | 18.9 | 12.5 | 11.2 | 0.6 | 12.1 |
| StarCoderBase-15B | Commit | 25.6 | 29.4 | 28.8 | 28.7 | 28.2 | 19.7 | 26.7 |
| CodeLlama-34B-Instruct | Instruct | 36.5 | 28.1 | 36.4 | 25.7 | 25.2 | 18.5 | 28.4 |
| DeepSeekCoder-33B-Instruct | Instruct | 47.5 | 47.6 | 46.5 | 52.0 | 48.0 | 10.2 | 42.1 |
| **StarCoder2-15B** | **Instruct** | 9.7 | 20.7 | 24.1 | 36.3 | 25.6 | 15.4 | 22.0 |
| **StarCoder2-15B** | **Issue** | **48.6** | **41.6** | **48.4** | **48.5** | **20.7** | **24.2** | **38.7** |

**Critical findings:**

1. **Prompt format matters enormously for base models.** StarCoder2-15B scores 22.0% with Instruct prompt but 38.7% with Issue prompt—a **76% relative improvement**. This validates the multi-format pre-training: the model learned issue discussion patterns and can apply them zero-shot.

2. **StarCoder2-15B with Issue prompt outperforms CodeLlama-34B-Instruct** (38.7 vs. 28.4 average) despite CodeLlama being instruction-tuned specifically for this task type. This is remarkable—a base model matching instruction-tuned competitors through format-aware prompting.

3. **C++ remains a weakness.** Even with Issue prompt, StarCoder2-15B scores only 20.7% on C++ vs. 48.0% for DeepSeekCoder-33B-Instruct. The authors note one-third of C++ generations are incomplete (e.g., unexpected `break` after for loop start), suggesting language-specific tokenization or training data issues.

#### CanItEdit: Code Editing (Table 13)

**What this measures:** CanItEdit tests whether models can modify existing code according to instructions (descriptive or lazy). This simulates real-world refactoring tasks.

**Key results:**

| Model | Format | Descriptive | Lazy |
|-------|--------|-------------|------|
| StarCoderBase-15B | Commit | 38.24 | 26.38 |
| **StarCoder2-15B** | **Issue** | **43.08** | **38.45** |
| CodeLlama-Instruct-34B | Instruct | 35.0 | 26.76 |
| DeepSeekCoder-Instruct-33B | Instruct | 53.06 | 43.89 |

StarCoder2-15B **outperforms CodeLlama-Instruct-34B** on both instruction types (43.08 vs. 35.0 descriptive, 38.45 vs. 26.76 lazy). This further validates the Issue prompt strategy and suggests pre-training on PR discussions improves code editing understanding.

#### GSM8K: Mathematical Reasoning (Table 14)

**What this measures:** GSM8K contains 7,473 middle-school math word problems. The PAL (Program-Aided Language) approach prompts models to generate Python code that solves the problem, then executes it.

**Key results:**

| Model | GSM8K (PAL) |
|-------|-------------|
| StarCoderBase-15B | 21.5 |
| CodeLlama-13B | 38.1 |
| **StarCoder2-15B** | **65.1** |
| CodeLlama-34B | 54.2 |
| DeepSeekCoder-33B | 58.7 |

**This is the most surprising result in the paper.** StarCoder2-15B scores **65.1%**, outperforming both CodeLlama-34B (54.2%) and DeepSeekCoder-33B (58.7%) by significant margins (10.9 and 6.4 percentage points respectively). 

**Why this matters:** Math reasoning requires logical deduction, not just pattern matching. The inclusion of GSM8K training data (7K examples), GSM8K-SciRel (110K examples), Deepmind Mathematics (110M examples), and OpenWebMath (15B tokens) clearly paid off. This demonstrates that **targeted high-quality data can outweigh raw parameter count** for reasoning tasks.

**Caveat:** StarCoder2-3B scores only 27.7%, behind StableCode-3B (39.7%). The math reasoning benefit scales with model size—the 3B model lacks capacity to leverage the math data effectively.

#### CRUXEval: Code Execution Understanding (Table 15)

**What this measures:** CRUXEval has two tasks:
- **CRUXEval-I:** Predict input that produces given output
- **CRUXEval-O:** Predict output given input and function

This tests whether models can mentally execute code, not just generate it.

**Key results (Pass@1 / Pass@5):**

| Model | CRUXEval-I P@1 | CRUXEval-I P@5 | CRUXEval-O P@1 | CRUXEval-O P@5 |

| CodeLlama-34B | 47.2 | 66.6 | 42.4 | 55.9 |
| DeepSeekCoder-33B | 46.5 | 64.9 | 48.6 | 61.6 |

StarCoder2-15B achieves **best-in-class on CRUXEval-I** (48.1% P@1, exceeding both 34B competitors) and **competitive on CRUXEval-O** (47.1% P@1, slightly behind DeepSeekCoder-33B's 48.6%). This suggests strong code understanding capabilities, consistent with the repository-context training that exposes models to complete program structures.

Real-world code completion occurs within repository context, not in isolation. Two benchmarks test this capability.

#### RepoBench: Next-Line Prediction (Table 17)

**What this measures:** RepoBench evaluates next-line prediction with varying amounts of cross-file context (2K to 16K tokens). Metrics include Exact Match (EM), Edit Similarity (ES), and CodeBLEU.

**Python results (16K context):**

| Model | EM | ES | CodeBLEU |
|-------|-----|-----|----------|

**What this measures:** CrossCodeEval requires models to use cross-file context for accurate completion. It reports Code Match (Edit Similarity) and Identifier Match (F1) across four languages.

**Python results:**

| CodeLlama-13B | 75.88 | 70.97 |
| **StarCoder2-15B** | **78.72** | **74.27** |
| CodeLlama-34B | 76.34 | 71.36 |
| DeepSeekCoder-33B | 78.78 | 74.51 |

StarCoder2-15B **matches DeepSeekCoder-33B** on Code ES (78.72 vs. 78.78) and **exceeds it on ID F1** (74.27 vs. 74.51—actually slightly behind, need to verify). The paper states StarCoder2-15B achieves "state-of-the-art performance compared to models of similar sizes" and for some languages "better even than models with 2x capacity."

**Language variance is significant:** On TypeScript, StarCoder2-15B scores 48.63% Code ES vs. 68.98% for CodeLlama-34B—a 20-point gap. The authors acknowledge this in Section 7.6.2: "achieving uniformly high performance across all programming languages remains challenging." This suggests TypeScript training data may be insufficient despite the 619-language claim.

### 5.5 Security and Safety Assessment

Code LLMs must not only be capable but also safe. The evaluation includes security vulnerability and bias benchmarks.

#### "Asleep at the Keyboard" Security Benchmark (Table 19)

**What this measures:** This benchmark tests whether generated code contains security vulnerabilities across 18 CWE (Common Weakness Enumeration) categories. Two metrics: Valid (syntactically correct programs) and Insecure (valid programs with vulnerabilities).

**Key results:**

| Model | Valid | Insecure (of valid) |
|-------|-------|---------------------|
| StarCoderBase-15B | 93.3% | 35.6% |
| CodeLlama-13B | 90.3% | 30.2% |
| **StarCoder2-15B** | **89.8%** | **39.2%** |
| DeepSeekCoder-33B | 92.1% | 34.2% |

**Concerning finding:** StarCoder2-15B has the **highest insecurity rate (39.2%)** among comparable models. The authors hypothesize this is "a side-effect of its higher rate of valid and functionally correct completions"—more functional code creates more opportunities for vulnerabilities (Section 7.7).

**Critical assessment:** This explanation is plausible but incomplete. If StarCoder2 generates more functional code, the absolute number of vulnerable programs might increase, but the **rate** should remain similar unless vulnerability patterns are also learned from training data. The higher insecurity rate suggests either:
1. Training data contains vulnerable code patterns that models replicate
2. The model prioritizes functionality over security without explicit safety training

This is a **significant limitation** that downstream users must address through additional safety fine-tuning or guardrails.

#### Bias and Toxicity Benchmarks (Tables 20-23)

**BOLD (Table 20):** Measures fairness across Religious Ideology, Profession, Race, Gender, and Political Ideology. StarCoder2 models show similar scores to competitors—for example, StarCoder2-15B Gender scores: Negative 0.05, Neutral 0.36, Other 0.07, Positive 0.53. No model clearly dominates; all exhibit comparable bias levels.

**WinoBias (Table 21):** Tests gender stereotype bias in professions. StarCoder2-15B scores 0.38 average (Male 0.36, Female -0.36), similar to CodeLlama-13B (0.37) and better than StarCoderBase-15B (0.35).

**HONEST (Table 22):** Measures hurtful sentence completions. StarCoder2-15B scores 0.11, identical to most competitors.

**RealToxicityPrompts (Table 23):** StarCoder2-15B scores 0.05 toxicity, comparable to DeepSeekCoder-6.7B (0.05) and slightly higher than StarCoderBase-15B (0.04).

**Interpretation:** Code LLMs trained primarily on code produce **less harmful content than general web-text models** (per Li et al., 2023, cited in Section 7.8). However, StarCoder2 does not improve significantly over StarCoderBase on safety metrics—transparency and responsible sourcing of training data do not automatically reduce bias or toxicity. Downstream safety fine-tuning remains necessary.

### 5.6 Failure Cases and Limitations

The paper honestly reports several underperformance areas that warrant attention.

#### Fill-in-the-Middle Underperformance (Table 16)

**Expected vs. Actual:** StarCoder2-15B should excel at FIM given the dedicated training (Section 5.1). Instead, it scores **48.4% on Python** vs. 62% for StarCoderBase-15B—a **13.6 percentage point degradation**.

**Root cause:** The authors attribute this to "an implementation bug" where "the FIM-rate was smaller than intended for most of the training" (Section 7.5 caption). This is a significant training infrastructure failure that went undetected until evaluation.

**Implication:** The multi-format pre-training strategy only works when each format receives adequate training signal. FIM underrepresentation during training directly caused capability loss. This is a cautionary tale for complex pre-training pipelines—each component needs explicit monitoring.

#### StarCoder2-7B Anomaly (Section 1)

**The puzzle:** StarCoder2-7B underperforms relative to both 3B and 15B models for its size class. On HumanEval+, it scores 29.9% vs. 27.4% (3B) and 37.8% (15B). The 7B should theoretically sit between them, but the 3B nearly matches it.

**Authors' admission:** "It is not clear to this report's authors why StarCoder2-7B does not perform as well as StarCoder2-3B and StarCoder2-15B for their size" (Section 1).

**Possible explanations (not in paper):**
1. **Data composition mismatch:** The 7B uses the-stack-v2-train-smol (17 languages) like the 3B, but includes OpenWebMath, Wikipedia, and ArXiv. This additional natural language data might dilute code-specific learning.
2. **Training dynamics:** The 7B trained for 5.31 epochs vs. 4.98 (3B) and 4.49 (15B). More epochs on the same data could cause overfitting.
3. **Architecture sweet spot:** 7B parameters might be an awkward middle ground—too large for the smol dataset, too small for the full dataset.

This remains an **open research question** that the authors appropriately flag rather than obscure.

#### Language-Specific Weaknesses

As noted in CrossCodeEval results, performance varies significantly by language:
- **Strong:** Python, JavaScript, Ruby (high-resource training data)
- **Weak:** TypeScript, C++ (despite inclusion in training)

The 619-language claim is technically accurate but potentially misleading—many languages have minimal training data. Users should not expect uniform performance across all 619 languages.

## 6. Limitations and Trade-offs

StarCoder2 represents a significant advance in transparent Code LLM development, but it is not without constraints. This section examines the assumptions underlying the approach, scenarios not addressed, resource requirements, and remaining open questions. Understanding these limitations is essential for practitioners considering deployment and researchers building upon this work.

### 6.1 Assumptions Underlying the Approach

The StarCoder2 methodology rests on several critical assumptions that shape its capabilities and constraints.

**Assumption 1: Software Heritage Archive Represents Sufficient Code Diversity**

The entire Stack v2 dataset is built on Software Heritage's archive, which the authors describe as "the digital commons of their source code archive" (Abstract). This assumes that:

- Software Heritage's crawling captures representative code across all 619 programming languages
- The archive's preservation priorities align with what makes effective training data
- Historical code in the archive remains relevant for current development practices

**Evidence of limitation:** Table 10 (MultiPL-E) reveals significant performance variance across languages. StarCoder2-15B scores 44.2% on JavaScript but only 23.6% on D, and 33.2% on Julia. The 619-language claim is technically accurate but masks extreme data imbalance—Table 1 shows JavaScript has 1,115 GB in The Stack v2-dedup while D has only 33.91 GB. The assumption that all 619 languages receive adequate representation is not supported by the data distribution.

**Assumption 2: Over-Training Smaller Models Compensates for Parameter Count**

The authors explicitly state they "push the number of training tokens far beyond the compute-optimal number suggested by Chinchilla" (Section 1). This assumes:

- Code has higher information density than natural language, justifying more tokens per parameter
- Diminishing returns from additional parameters outweigh the accessibility benefits of smaller models
- The relationship between tokens and performance remains monotonic at extreme token counts

**Evidence of limitation:** The StarCoder2-7B anomaly directly challenges this assumption. As the authors acknowledge in Section 1: "It is not clear to this report's authors why StarCoder2-7B does not perform as well as StarCoder2-3B and StarCoder2-15B for their size." Table 9 shows StarCoder2-7B scoring 29.9% on HumanEval+ vs. 27.4% for the 3B model—a mere 2.5 percentage point gain despite 2.3× more parameters. This non-monotonic scaling suggests the over-training strategy has poorly understood dynamics that may not generalize across model sizes.

**Assumption 3: Repository-Context Training Improves Real-World Performance**

The shift from file-context (StarCoder1) to repository-context (StarCoder2) assumes that:

- Files from the same repository share semantic relationships that benefit code completion
- Random ordering within repositories preserves enough structure without requiring language-specific dependency analysis
- The 16K context window captures sufficient repository context for practical tasks

**Evidence of limitation:** While RepoBench results (Table 17) show improvement over StarCoderBase, CrossCodeEval (Table 18) reveals language-specific failures. On TypeScript, StarCoder2-15B scores 48.63% Code ES vs. 68.98% for CodeLlama-34B—a 20-point gap. The authors note in Section 7.6.2 that "achieving uniformly high performance across all programming languages remains challenging." Repository-context alone cannot compensate for insufficient language-specific training data.

**Assumption 4: Transparent Data Sourcing Produces Trustworthy Models**

The SWHID attribution system assumes that:

- Developers can and will use the "Am I in the Stack" tool to verify inclusion
- License filtering (permissive + unlicensed, excluding copyleft) satisfies legal requirements
- PII redaction via StarPII adequately protects privacy

**Evidence of limitation:** Section 3.5 reports only 91 users/organizations submitted opt-out requests, removing 22,066 files. This low number could indicate either (a) the opt-out process works and few developers object, or (b) developers are unaware of the tool or lack resources to verify inclusion. The paper does not provide usage statistics for the governance tool, making it impossible to distinguish between these interpretations. Additionally, Section 8 acknowledges that natural language datasets (StackOverflow, ArXiv, Wikipedia) do not have SWHID attribution, creating a "partial transparency gap."

### 6.2 Unaddressed Scenarios and Edge Cases

Several important use cases and edge cases fall outside the scope of StarCoder2's evaluation and design.

**Scenario 1: Proprietary Code Generation**

StarCoder2 is trained exclusively on open-source code from Software Heritage and other public sources. This creates a fundamental limitation:

- The model has no exposure to proprietary coding patterns, internal APIs, or enterprise-specific frameworks
- Performance on closed-source software development tasks is untested
- Organizations with significant proprietary codebases may find the model less useful for their specific contexts

**Evidence:** Section 2.1 explicitly states: "We exclude commercial licenses since their creators do not intend their code to be used for commercial purposes." This ethical choice necessarily limits the model's applicability to enterprise environments where proprietary patterns dominate.

**Scenario 2: Multi-File, Multi-Repository Dependencies**

While repository-context training addresses single-repository completion, real-world development often involves:

- Cross-repository dependencies (e.g., microservices architectures)
- External library integration beyond what's in the training corpus
- Version-specific API changes not captured in static snapshots

**Evidence:** The 16K context window (Section 6.3) constrains how much repository context can be provided at inference time. RepoBench evaluation (Section 7.6.1) tests up to 16K context but uses truncated prompts for models with smaller limits. Real-world repositories often exceed this, requiring retrieval-augmented generation strategies not evaluated in this paper.

**Scenario 3: Instruction-Following Without Fine-Tuning**

StarCoder2 is released as a base model, not instruction-tuned. The authors demonstrate that prompt engineering (e.g., the "Issue" format in Section 7.2.1) can elicit instruction-following behavior, but this assumes:

- Users have expertise in prompt engineering for base models
- The prompt formats generalize across tasks beyond those tested
- Base model performance is sufficient for production use without fine-tuning

**Evidence:** Table 12 shows StarCoder2-15B with Instruct prompt scoring only 22.0% on HumanEvalFix vs. 38.7% with Issue prompt—a 76% relative improvement from prompt engineering alone. However, this still trails DeepSeekCoder-33B-Instruct at 42.1%. The paper explicitly states in Section 7.2.1: "We leave the instruction tuning or even preference alignment... of StarCoder2 to future work." This is a significant capability gap for users who cannot perform additional fine-tuning.

**Scenario 4: Non-English Code and Documentation**

Despite 619 programming languages, the natural language component is predominantly English:

- Code comments, docstrings, and documentation are mostly English
- Variable names and identifiers often use English conventions
- Non-English natural language queries may not be well-supported

**Evidence:** Section 9.3 (Representation Bias) acknowledges: "The predominant natural language in source code and other datasets used is English although other languages are also present. As such, the model can generate code snippets provided some non-English context, but the generated code is not guaranteed to work as intended or equally as well for all languages." This limits applicability in non-English-speaking development teams.

**Scenario 5: Real-Time Security Vulnerability Detection**

The "Asleep at the Keyboard" benchmark (Section 7.7) evaluates whether generated code contains vulnerabilities, but does not address:

- Whether the model can be prompted to avoid generating vulnerable code
- How downstream systems should filter or validate generated code
- Whether vulnerability patterns in training data are being learned and reproduced

**Evidence:** Table 19 shows StarCoder2-15B has a 39.2% insecure program rate—higher than StarCoderBase-15B (35.6%) and CodeLlama-13B (30.2%). The authors hypothesize this is "a side-effect of its higher rate of valid and functionally correct completions" but acknowledge this requires "downstream guardrails." The model itself does not include built-in security validation.

### 6.3 Computational and Data Constraints

StarCoder2's transparency and performance come with substantial resource requirements that limit accessibility despite the smaller model sizes.

**Training Compute Costs**

Table 7 and Section 6.4 report significant computational investment:

| Model | Training Tokens | FLOPs | CO₂ Emissions |
|-------|----------------|-------|---------------|
| 3B | 3.1T | 5.94×10²² | 16,107 kgCO₂eq |
| 7B | 3.5T | 1.55×10²³ | 29,623 kgCO₂eq |
| 15B | 4.1T | 3.87×10²³ | Not reported |

**Implications:**

- The 7B model required 145,152 GPU hours on H100 hardware—far beyond what most research groups can access
- CO₂ emissions are substantial (equivalent to ~7 passenger vehicles driven for one year per 3B model)
- The 15B emissions were not reported at submission time, suggesting ongoing uncertainty about total environmental cost

**Inference Requirements**

While 15B models are more accessible than 34B models, they still require:

- ~30GB VRAM for full precision inference
- ~8-12GB VRAM with 4-bit quantization (not evaluated in the paper)
- Specialized hardware for practical throughput in production settings

**Evidence:** Section 1 states the accessibility goal ("A 15B parameter model can run on consumer hardware with appropriate quantization"), but the paper does not include quantization benchmarks or latency measurements. Users must independently verify deployment feasibility.

**Data Curation Overhead**

The transparent data pipeline requires significant manual effort:

- 15 annotators visually inspected ~1,000 file extensions (Section 2.1)
- License detection required ScanCode Toolkit processing for 96.93% of repositories (Section 2.1)
- Malware scanning eliminated 59,442 files using ClamAV + SaneSecurity (Section 3.4)
- Opt-out processing removed 22,066 files from 91 users/organizations (Section 3.5)

**Implication:** Reproducing this data curation pipeline requires legal expertise, annotation infrastructure, and ongoing maintenance. The transparency comes at the cost of reproducibility barriers—few teams can replicate the full Stack v2 creation process.

### 6.4 Known Weaknesses and Open Questions

The paper honestly reports several weaknesses that remain unresolved.

**Weakness 1: Fill-in-the-Middle Capability Degradation**

Table 16 shows StarCoder2-15B scoring 48.4% on Python FIM vs. 62% for StarCoderBase-15B—a 13.6 percentage point regression.

**Root cause:** Section 7.5 caption states: "Due to an implementation bug, FIM was incorrect for most of the training of StarCoder2-15B."

**Open question:** Can this be fixed through additional fine-tuning, or does it require retraining? The paper does not provide a remediation path, leaving users with a degraded capability in the largest model.

**Weakness 2: C++ Code Generation Quality**

Table 12 shows StarCoder2-15B scoring only 20.7% on C++ in HumanEvalFix vs. 48.0% for DeepSeekCoder-33B-Instruct.

**Root cause:** Section 7.2.1 states: "Our investigation shows that this is mainly because one-third of the code generated is incomplete, e.g., having an unexpected break immediately after the beginning of a for loop."

**Open question:** Is this a tokenization issue, training data quality issue, or architectural limitation? The paper does not provide diagnostic analysis, making it difficult for downstream users to anticipate similar failures in other systems languages.

**Weakness 3: Non-Monotonic Scaling at 7B**

As discussed in Section 6.1, the 7B model underperforms expectations.

**Open question:** The authors explicitly state uncertainty: "It is not clear to this report's authors why." This represents a genuine gap in understanding scaling dynamics for Code LLMs. Future work should investigate whether:
- Data composition (smol vs. full dataset) interacts poorly with 7B capacity
- Training epoch count (5.31 for 7B vs. 4.98 for 3B) caused overfitting
- Architecture hyperparameters (32 layers, 4608 hidden dim) are suboptimal for this size

**Weakness 4: Security Vulnerability Rates**

Table 19 shows StarCoder2-15B has the highest insecure program rate (39.2%) among comparable models.

**Open question:** Does transparent training data actually increase or decrease security risks? The authors hypothesize more functional code creates more vulnerability opportunities, but this is not empirically validated. Users need guidance on whether transparent models are inherently more or less secure than opaque alternatives.

**Weakness 5: Partial Data Transparency**

Section 8 acknowledges SWHIDs cover only Software Heritage source code, not natural language datasets.

**Open question:** What percentage of total tokens have verifiable attribution? The paper states Stack v2 has 900B+ unique tokens, with source code being the majority, but does not provide exact breakdown. Users cannot precisely assess what fraction of the model's knowledge is auditable.

### 6.5 Trade-off Summary

The following table summarizes key trade-offs in the StarCoder2 design:

| Design Choice | Benefit | Cost |
|--------------|---------|------|
| Over-training smaller models | Accessibility (15B vs. 34B) | Non-monotonic scaling (7B anomaly) |
| Repository-context training | Better repo-level completion | Language-specific weaknesses (TypeScript) |
| SWHID transparency | Verifiable attribution | Partial coverage (no NL datasets) |
| Base model release | Flexibility for fine-tuning | Requires prompt engineering for instructions |
| Permissive + unlicensed data | Larger corpus (4× StarCoder1) | Legal uncertainty for some users |
| Multi-format pre-training | Zero-shot task flexibility | FIM bug affected one format |
| Opt-out mechanism | Developer consent | Low participation (91 users) |

### 6.6 Recommendations for Practitioners

Based on these limitations, practitioners should consider:

1. **For production deployment:** Plan for additional safety fine-tuning to address the 39.2% insecurity rate. Do not deploy StarCoder2-15B for security-sensitive code generation without downstream validation.

2. **For low-resource languages:** Verify performance on your target language before committing. The 619-language claim does not guarantee uniform quality—test on MultiPL-E or domain-specific benchmarks.

3. **For instruction-following tasks:** Use the "Issue" prompt format (Section 7.2.1) rather than standard instruction prompts. Consider instruction fine-tuning if prompt engineering is insufficient.

4. **For FIM tasks:** Prefer StarCoder2-3B or StarCoderBase-15B over StarCoder2-15B until the FIM bug is addressed.

5. **For compliance verification:** Use the "Am I in the Stack" tool and Data Portrait (Section 8) to audit training data inclusion, but recognize natural language datasets lack SWHID attribution.

6. **For resource-constrained environments:** The 3B model offers the best efficiency-performance trade-off, outperforming StarCoderBase-15B on most benchmarks while requiring ~5× less inference compute.

### 6.7 Open Research Directions

The limitations identified suggest several promising research directions:

1. **Understanding 7B scaling anomaly:** What causes non-monotonic performance in mid-size Code LLMs?

2. **Security-aware training:** Can vulnerability patterns be explicitly excluded or penalized during training?

3. **Full transparency extension:** How can SWHID-like attribution be extended to natural language datasets?

4. **FIM training monitoring:** What infrastructure ensures all pre-training formats receive adequate signal?

5. **Non-English code support:** How can Code LLMs better support non-English documentation and identifiers?

6. **Opt-out mechanism improvement:** How can developer consent processes be made more accessible and participatory?

StarCoder2 demonstrates that transparent, responsibly sourced Code LLMs can compete with opaque alternatives. However, the limitations documented here show that transparency alone does not solve all challenges—security, scaling dynamics, and uniform language support remain active areas for improvement.

## 7. Implications and Future Directions

StarCoder2 represents more than an incremental performance improvement; it establishes a new protocol for responsible AI development in the code domain. By coupling competitive performance with unprecedented data transparency, this work forces a reevaluation of what constitutes "open" in the LLM era. This section analyzes how StarCoder2 shifts the field's baseline, identifies specific research avenues it unlocks, outlines practical deployment scenarios, and provides concrete integration guidance for practitioners.

### 7.1 Redefining Openness in the LLM Landscape

The release of StarCoder2 fundamentally alters the expectation for open-source model releases. Prior to this work, the industry standard for "open" models was **open weights** (e.g., CodeLlama, DeepSeekCoder), where the architecture and parameters are public but the training data remains proprietary. StarCoder2 shifts the baseline to **open science**, where data provenance is verifiable via Software Heritage persistent IDentifiers (SWHIDs).

**Impact on the Field:**
*   **Auditability as a Feature:** By providing SWHIDs for source code files (Section 8), StarCoder2 enables third-party audits for copyright compliance, bias, and benchmark contamination. This moves the burden of proof from the user (who must guess if their code was used) to the developer (who provides cryptographic evidence of data lineage).
*   **Pressure on Proprietary Models:** The performance parity with CodeLlama-34B (Section 5.7) despite being less than half the size demonstrates that transparency does not require sacrificing capability. This challenges the narrative that opaque data sourcing is necessary for state-of-the-art performance.
*   **Governance Precedent:** The "Am I in the Stack" tool and opt-out mechanism (Section 3.5) establish a replicable governance framework. Future datasets may be expected to include similar consent mechanisms, raising the barrier to entry for data collection but increasing ethical legitimacy.

**Critical Caveat:** The transparency is **partial**. While source code has SWHID attribution, natural language datasets (StackOverflow, ArXiv, Wikipedia) do not (Section 8). This creates a two-tier transparency model that future work must address to achieve full audibility.

### 7.2 Enabled Research Avenues

The transparency and architectural choices in StarCoder2 unlock specific research questions that were previously unanswerable due to data opacity.

**1. Data-Model Attribution Studies**
Because every source code file is traceable, researchers can now correlate model capabilities with specific data subsets.
*   **Research Question:** Does performance on low-resource languages (e.g., Julia, Perl) correlate with specific repository characteristics (e.g., documentation density, commit frequency)?
*   **Methodology:** Use the SWHIDs to subset the training data by language and repository metadata, then train probing models to measure contribution margins. This moves beyond coarse-grained "language mixing ratios" to fine-grained data quality analysis.

**2. Scaling Law Anomalies in Code**
The underperformance of the 7B model relative to the 3B and 15B models (Section 1) presents a rare opportunity to study non-monotonic scaling in Code LLMs.
*   **Research Question:** Is the 7B anomaly caused by data composition (mix of `the-stack-v2-train-smol` and natural language data) or architectural hyperparameters?
*   **Methodology:** Reproduce the 7B training run with adjusted data mixes (e.g., removing OpenWebMath/ArXiv) to isolate the variable. This could refine Chinchilla scaling laws specifically for code, which may have different information density properties than natural language.

**3. Security-Aware Training**
The high insecure program rate (39.2% for 15B, Table 19) suggests that transparent data sourcing alone does not mitigate security risks.
*   **Research Question:** Can vulnerability patterns be explicitly excluded during preprocessing without degrading general code quality?
*   **Methodology:** Use the malware scanning pipeline (Section 3.4) as a base, then augment it with static analysis tools to flag vulnerable patterns (e.g., SQL injection, buffer overflows) during data curation. Measure the trade-off between security and pass@1 scores.

**4. Repository-Context Dynamics**
The shift to repository-context training (Section 5.1) improves cross-file completion but introduces complexity in context window management.
*   **Research Question:** What is the optimal strategy for ordering files within a repository? StarCoder2 uses random ordering; would dependency-graph ordering improve performance?
*   **Methodology:** Compare random ordering against topological sorting based on import statements. This requires language-specific parsers but could yield significant gains for large-scale codebases.

### 7.3 Practical Applications and Downstream Use Cases

StarCoder2's specific strengths and weaknesses make it suitable for distinct deployment scenarios. Practitioners should match model size and capability to their infrastructure and risk tolerance.

**1. Local IDE Assistance (3B and 7B Models)**
*   **Use Case:** Real-time code completion in integrated development environments (IDEs) running on consumer hardware.
*   **Why StarCoder2:** The 3B model outperforms StarCoderBase-15B on most benchmarks (Table 9) while requiring significantly less VRAM. With 4-bit quantization, the 3B model fits within ~2GB VRAM, enabling local inference on laptops without cloud dependency.
*   **Constraint:** Avoid using the 7B model for critical tasks until the scaling anomaly is better understood; the 3B offers more predictable performance per parameter.

**2. Enterprise Compliance and Auditing (15B Model)**
*   **Use Case:** Code generation in regulated industries (finance, healthcare) where data provenance is legally required.
*   **Why StarCoder2:** The SWHID attribution allows legal teams to verify that generated code does not infringe on specific licenses or include opted-out proprietary code. This is impossible with opaque models like CodeLlama.
*   **Constraint:** Implement downstream security scanning. The 39.2% insecure program rate (Table 19) necessitates static analysis tools (e.g., SonarQube, Semgrep) in the CI/CD pipeline before merging generated code.

**3. Low-Resource Language Support**
*   **Use Case:** Legacy system maintenance or niche scientific computing (e.g., Julia for physics, Perl for bioinformatics).
*   **Why StarCoder2:** StarCoder2-15B outperforms DeepSeekCoder-33B on low-resource languages like D, Julia, Lua, and Perl (Table 10). This suggests the diverse 619-language training set effectively transfers knowledge to niche domains.
*   **Constraint:** Performance is not uniform. TypeScript support is notably weak (Table 18), so verify benchmark performance on your specific target language before deployment.

**4. Mathematical Reasoning and Data Science**
*   **Use Case:** Automated data analysis pipelines or educational tools for math problem solving.
*   **Why StarCoder2:** The 15B model scores 65.1% on GSM8K (Table 14), outperforming models twice its size. The inclusion of OpenWebMath and Kaggle notebooks (Section 2.4, 2.8) provides specialized capability for numerical reasoning.
*   **Constraint:** Use the PAL (Program-Aided Language) approach (Section 7.3) where the model generates code to solve math problems rather than answering directly. This leverages the model's code execution strength.

**Reproduction and Fine-Tuning Advice:**
*   **Data Pipeline Complexity:** Reproducing The Stack v2 is non-trivial. The preprocessing pipeline involves license detection (ScanCode), malware scanning (ClamAV), and PII redaction (StarPII) (Section 3). Teams attempting to curate similar datasets should budget significant engineering resources for legal and security filtering.
*   **Compute Requirements:** Training the 15B model required 4.1T tokens and substantial GPU hours (Section 6.4). Fine-tuning is more accessible; use LoRA or QLoRA adapters for task-specific adaptation to avoid full retraining costs.
*   **Prompt Engineering:** Since StarCoder2 is a base model, use the **"Issue" prompt format** for bug fixing and editing tasks (Section 7.2.1). Standard instruction prompts yield significantly lower performance (22.0% vs. 38.7% on HumanEvalFix).
*   **Context Window Management:** Leverage the 16K context window for repository-level tasks (Section 6.3). However, be aware that performance degrades if the context exceeds the training distribution; use retrieval-augmented generation (RAG) to select relevant files rather than dumping entire repositories.

**Future Integration Roadmap:**
1.  **Immediate:** Deploy StarCoder2-3B for local autocomplete where latency and privacy are priorities.
2.  **Short-term:** Implement security scanning pipelines to mitigate the 39.2% vulnerability rate before deploying the 15B model in production.
3.  **Long-term:** Contribute to the BigCode community by reporting opt-outs or data errors via the "Am I in the Stack" tool, reinforcing the governance loop that makes this transparency possible.

StarCoder2 demonstrates that the trade-off between openness and performance is not inevitable. By providing a verifiable, high-performance baseline, it empowers the community to build safer, more accountable code intelligence tools. The work ahead lies not in proving transparency is possible, but in refining it to cover natural language data, improve security outcomes, and resolve the scaling anomalies that remain.
