## 1. Executive Summary

DistServe addresses the critical inefficiency in Large Language Model (LLM) serving caused by "prefill-decoding interference," where colocating the compute-bound prefill phase and memory-bound decoding phase on the same GPU forces systems to over-provision resources to meet latency Service Level Objectives (SLOs). By disaggregating these phases onto separate GPUs and co-optimizing their resource allocation and parallelism strategies, DistServe eliminates contention and enables tailored execution plans for Time to First Token (TTFT) and Time Per Output Token (TPOT). Evaluations on OPT models (13B to 175B) across real-world workloads like ShareGPT, HumanEval, and LongBench demonstrate that DistServe serves up to **7.4× more requests** or supports **12.6× tighter SLOs** compared to state-of-the-art systems like vLLM and DeepSpeed-MII, while maintaining >90% SLO attainment.

## 2. Context and Motivation

To understand why DistServe is necessary, we must first dissect the unique computational anatomy of Large Language Model (LLM) inference and why traditional serving architectures struggle to meet modern latency demands efficiently.

### The Dual-Phase Nature of LLM Inference
Unlike standard database queries or image classification tasks, generating a response from an LLM occurs in two distinct computational phases with fundamentally different characteristics:

1.  **The Prefill Phase:** When a user submits a prompt (e.g., "Write a summary of..."), the model processes all input tokens simultaneously to generate the *first* output token.
    *   **Computational Characteristic:** This phase is **compute-bound**. The workload scales superlinearly with the number of input tokens because the model must compute attention scores between every pair of tokens in the prompt.
    *   **Latency Metric:** Performance is measured by **Time to First Token (TTFT)**. Applications like real-time chatbots require this to be extremely low (e.g., < 250ms) to feel responsive.
2.  **The Decoding Phase:** Once the first token is generated, the model enters an autoregressive loop, generating one token at a time based on the previous output.
    *   **Computational Characteristic:** This phase is **memory-bandwidth-bound**. Although it processes only one new token per step, it must load the entire model weights and the accumulated history (KV cache) from GPU memory for every single step. The arithmetic intensity is low, meaning the GPU spends most of its time waiting for data transfer rather than computing.
    *   **Latency Metric:** Performance is measured by **Time Per Output Token (TPOT)**. For reading fluency, this typically needs to be faster than human reading speed (e.g., < 100ms per token).

### The Core Problem: Prefill-Decoding Interference
The central gap this paper addresses is the inefficiency caused by **colocating** these two phases on the same GPU resources. Existing state-of-the-art systems (such as vLLM and DeepSpeed-MII) employ a strategy called **continuous batching**. In this approach, the server mixes new prefill requests with ongoing decoding requests in the same batch to maximize overall GPU utilization.

While continuous batching maximizes *throughput* (total tokens generated per second), it creates severe **prefill-decoding interference** when strict latency Service Level Objectives (SLOs) are required:

*   **Decoding Starvation:** A single long prefill request in a batch can dominate the GPU's compute units. Since decoding steps cannot proceed until the prefill step finishes, the TPOT for all ongoing decoding requests spikes dramatically.
*   **Prefill Delay:** Conversely, if the system prioritizes decoding to maintain smooth output, new prefill requests sit in the queue, increasing TTFT.
*   **The Trade-off Trap:** As illustrated in **Figure 1** of the paper, when serving a 13B model on an NVIDIA A100 GPU, existing systems hit a wall. To meet a 90% SLO attainment (where 90% of requests meet both TTFT and TPOT targets), the system can only handle **1.6 requests per second (rps)**. Beyond this rate, either TTFT or TPOT violations skyrocket.

This interference forces service providers into a costly dilemma: they must **over-provision** hardware. They might deploy enough GPUs to handle the peak load of the *slower* phase, leaving the other phase underutilized, simply to avoid latency violations. The paper notes that if these phases were run independently on separate GPUs, the prefill phase could handle **5.6 rps** and the decoding phase **10 rps** on the same hardware. The gap between the collocated performance (1.6 rps) and the theoretical independent potential highlights the magnitude of the interference problem.

### Limitations of Prior Approaches
Before DistServe, the community attempted to mitigate this interference through software scheduling and batching optimizations, but these solutions offer only partial relief:

*   **Chunked Prefill (e.g., SARATHI, DeepSpeed-MII):** This technique splits a long prefill request into smaller chunks and interleaves them with decoding steps.
    *   *Why it falls short:* While this reduces the blocking time for decoding, it does not eliminate interference. Furthermore, it introduces significant overhead. As explained in **Section 2.3**, splitting a prefill job into $N$ chunks increases the memory access cost for the KV cache from $O(N)$ to $O(N^2)$ because previous chunks must be reloaded repeatedly. This makes chunked prefill inefficient for long contexts.
*   **Priority Scheduling:** Some systems attempt to prioritize one phase over the other.
    *   *Why it falls short:* This is a zero-sum game. Prioritizing prefill hurts TPOT; prioritizing decoding hurts TTFT. It cannot satisfy stringent SLOs for *both* simultaneously under high load.
*   **Colocation Constraints:** All these methods suffer from **resource coupling**. Because prefill and decoding share the same GPU memory and compute units, the system cannot apply the optimal parallelism strategy for each.
    *   *Prefill* benefits from **intra-operator parallelism** (splitting a single matrix multiplication across GPUs) to reduce execution time.
    *   *Decoding* often benefits from **inter-operator parallelism** (pipelining layers) or simple replication to scale throughput linearly.
    *   Colocation forces a single, compromised configuration that is suboptimal for both.

### DistServe's Positioning: Disaggregation for Goodput
DistServe positions itself not as an incremental scheduler improvement, but as an architectural shift from **throughput-optimization** to **goodput-optimization**.

*   **Goodput vs. Throughput:** Traditional systems optimize for *throughput* (max tokens/sec), ignoring whether individual requests meet their latency deadlines. DistServe optimizes for *goodput*: the maximum request rate that can be served while adhering to specific TTFT and TPOT SLOs for a target percentage of requests (e.g., 90%).
*   **The Disaggregation Strategy:** DistServe proposes physically separating the prefill and decoding phases onto different sets of GPUs.
    *   **Eliminating Interference:** By assigning dedicated resources, a long prefill job never blocks a decoding step.
    *   **Tailored Parallelism:** The system can independently choose the optimal parallelism strategy for each phase. For example, it might assign 3 GPUs with high intra-op parallelism for prefill to minimize TTFT, while assigning 4 GPUs with a different configuration for decoding to maximize TPOT efficiency.
*   **Handling Communication Overhead:** A natural concern with disaggregation is the cost of transferring the intermediate state (KV cache) from the prefill GPU to the decoding GPU. DistServe argues that in modern clusters with high-bandwidth interconnects (like NVLINK or InfiniBand), this overhead is negligible compared to the gains from eliminating interference. The paper specifically designs placement algorithms (**Section 4**) to ensure prefill and decoding instances are located to minimize this transfer cost, even in clusters with limited cross-node bandwidth.

In summary, DistServe challenges the prevailing assumption that colocation is necessary for efficiency. It demonstrates that for latency-sensitive LLM applications, the cost of interference far outweighs the benefits of shared resources, and that disaggregation is the key to unlocking cost-effective, high-performance serving.

## 3. Technical Approach

### 3.1 Reader orientation (approachable technical breakdown)
DistServe is a distributed serving system that physically separates the "prefill" (prompt processing) and "decoding" (token generation) phases of Large Language Model inference onto different sets of GPUs to eliminate performance interference. It solves the problem of unpredictable latency in collocated systems by treating these two phases as independent services with their own resource allocations and parallelism strategies, connected only by a high-speed data transfer of intermediate states.

### 3.2 Big-picture architecture (diagram in words)
The system operates as a coordinated pipeline consisting of four primary logical components:
1.  **Centralized Controller:** The entry point that receives incoming user requests, monitors the queue lengths of all instances, and dispatches new prompts to the least-loaded **Prefill Instance**.
2.  **Prefill Instances:** Dedicated GPU clusters that hold a copy of the model weights; they process the full input prompt in parallel to generate the first token and the corresponding Key-Value (KV) cache, then output this state rather than continuing generation.
3.  **Decoding Instances:** Separate GPU clusters (potentially with different parallelism configurations) that receive the KV cache and first token from the prefill stage; they perform the autoregressive loop to generate subsequent tokens one by one until the response is complete.
4.  **Placement & Simulation Engine:** An offline optimization module that analyzes the target model, cluster bandwidth, and workload statistics to automatically calculate the optimal number of GPUs and parallelism strategy (intra-op vs. inter-op) for both prefill and decoding instances before deployment.

### 3.3 Roadmap for the deep dive
*   **Phase Characterization:** We first analyze the distinct computational bottlenecks of prefill (compute-bound) and decoding (memory-bound) to mathematically justify why they require different parallelism strategies.
*   **Queueing Theory Models:** We derive the latency equations for both phases under disaggregation, showing how execution time and queuing delay interact differently for each phase.
*   **Placement Algorithms:** We detail the two specific algorithms DistServe uses to assign GPUs and parallelism strategies, distinguishing between clusters with high cross-node bandwidth and those restricted to intra-node connections.
*   **Runtime Mechanics:** We explain the online scheduling policies, including how the system handles "pull-based" KV cache transmission to prevent memory overload during traffic bursts.
*   **Replanning Strategy:** We describe the mechanism for adapting to changing workload patterns over time without service interruption.

### 3.4 Detailed, sentence-based technical breakdown

**Core Idea and Paper Type**
This is a systems research paper proposing an architectural disaggregation of LLM inference, where the core idea is to maximize "goodput" (the rate of requests meeting latency SLOs) by decoupling the resource allocation and parallelism plans of the prefill and decoding phases.

**System Data Pipeline and Flow**
The operational flow begins when a client sends a request to the **Controller**, which immediately routes the prompt to a **Prefill Instance** selected via a "shortest queue" policy. Inside the Prefill Instance, the GPU workers execute the model layers on the entire input sequence in parallel using a specific parallelism strategy (e.g., tensor parallelism) optimized for compute throughput. Once the prefill computation finishes, the instance generates the first output token and the associated **KV Cache** (the intermediate attention states required for future steps), but instead of proceeding to generate more tokens, it halts and transmits this KV cache to a **Decoding Instance**. The transmission method depends on the cluster topology: if the instances are on the same node, it uses high-bandwidth NVLINK; if across nodes, it uses InfiniBand or TCP, managed by a "pull" mechanism where the Decoding Instance fetches the data only when ready. Upon receiving the KV cache and the first token, the Decoding Instance enters an autoregressive loop, generating one token at a time by loading the model weights and the received KV cache from memory, continuing this process until an end-of-sequence token is produced or the maximum length is reached, at which point the final text is streamed back to the client.

**Mathematical Analysis of Prefill Latency**
To determine the optimal configuration for the prefill phase, DistServe models the average Time to First Token (TTFT) using queuing theory, specifically treating the disaggregated prefill instance as an **M/D/1 queue** (Markovian arrivals, Deterministic service time, 1 server).
The fundamental equation for average TTFT without parallelism is:
$$ \text{Avg\_TTFT} = D + \frac{R D^2}{2(1 - R D)} $$
Here, $D$ represents the constant execution time for a single request (since prefill length is assumed uniform for the model), and $R$ is the request arrival rate. The first term ($D$) is the pure computation time, while the second term represents the **queuing delay** caused by requests waiting for the GPU to become free.
When applying **2-way inter-operator parallelism** (pipelining the model across two GPUs), the execution time remains roughly $D$, but the slowest stage takes $D_m \approx D/2$. The resulting latency equation becomes:
$$ \text{Avg\_TTFT}_{\text{inter}} = D + \frac{R D^2}{4(2 - R D)} $$
This shows that inter-op parallelism significantly reduces the queuing delay term (the denominator increases), making it superior at **high arrival rates** where queuing dominates.
Conversely, for **2-way intra-operator parallelism** (splitting matrix multiplications across two GPUs), the execution time reduces to $D/K$ where $K$ is a speedup coefficient ($1 < K < 2$) accounting for communication overhead. The latency equation is:
$$ \text{Avg\_TTFT}_{\text{intra}} = \frac{D}{K} + \frac{R D^2}{2K(K - R D)} $$
Intra-op parallelism reduces the first term (execution time), making it superior at **low arrival rates** or when the SLO is extremely stringent, as it minimizes the base processing time regardless of queue length.
DistServe uses these equations to automatically select the parallelism strategy: if the target TTFT SLO is tight or the rate is low, it chooses intra-op; if the rate is high and queuing is the bottleneck, it chooses inter-op.

**Mathematical Analysis of Decoding Latency**
The decoding phase is analyzed differently because it is **memory-bandwidth-bound** rather than compute-bound.
In the decoding phase, the system must load the entire model weights and the growing KV cache for every single token generated.
DistServe observes that as the batch size increases, the decoding phase eventually becomes compute-bound, behaving similarly to the prefill phase.
However, for typical batch sizes constrained by memory, the latency is dominated by memory access.
The key insight for decoding is that **inter-operator parallelism** allows the system to linearly scale throughput by adding more GPUs (forming a pipeline), whereas **intra-operator parallelism** offers diminishing returns due to high communication costs relative to the small amount of computation per token.
Therefore, the default strategy for decoding is to use inter-op parallelism or simple model replication to maximize the number of concurrent requests (batch size) without violating the Time Per Output Token (TPOT) SLO.
The system calculates the maximum batch size $B_{max}$ supported by GPU memory $C$ given the model size $G$ and KV cache size per token, ensuring that:
$$ \text{Memory Usage} = \text{Weights} + B_{max} \times \text{Context Length} \times \text{KV Size per Token} \leq C $$

**Placement Algorithm for High Node-Affinity Clusters (Algorithm 1)**
For clusters equipped with high-speed cross-node networks (e.g., InfiniBand), DistServe employs a two-level placement strategy that assumes communication overhead between any two GPUs is negligible.
The algorithm first iterates through all feasible parallelism configurations for the prefill instance independently, simulating the goodput for each using the workload distribution and the M/D/1 latency model derived above.
It selects the configuration $(inter\_op_p, intra\_op_p)$ that maximizes the per-GPU goodput for the prefill phase.
It repeats this exhaustive search for the decoding instance to find its optimal configuration $(inter\_op_d, intra\_op_d)$.
Once the optimal single-instance configurations are found, the algorithm calculates the number of replicas needed for each phase to meet the total target traffic rate $R_{target}$:
$$ N_{prefill} = \lceil \frac{R_{target}}{\text{Goodput}_{prefill}} \rceil, \quad N_{decode} = \lceil \frac{R_{target}}{\text{Goodput}_{decode}} \rceil $$
This approach allows the system to allocate, for example, 3 GPUs for prefill and 5 GPUs for decoding per replica, perfectly tailoring resources to the specific bottlenecks of each phase.
The complexity of this search is $O(N M^2)$, where $N$ is the node limit and $M$ is the GPUs per node, which remains computationally tractable (under 1.3 minutes in experiments).

**Placement Algorithm for Low Node-Affinity Clusters (Algorithm 2)**
In many practical clusters, cross-node bandwidth is limited, making the transfer of large KV caches between nodes a potential bottleneck.
For these environments, DistServe enforces a constraint that the prefill and decoding segments communicating with each other must reside on the **same physical node** to utilize high-bandwidth NVLINK (e.g., 600 GB/s on A100 nodes).
The algorithm modifies the search space to enumerate **instance segments**: it splits the model layers into stages using inter-op parallelism and ensures that the prefill segment for stage $i$ and the decoding segment for stage $i$ are colocated on the same node.
Inside a single node with $M$ GPUs, the algorithm searches for a pair of configurations $(Config_p, Config_d)$ such that the sum of their GPU requirements does not exceed $M$:
$$ \text{GPUs}(Config_p) + \text{GPUs}(Config_d) \leq M $$
It simulates the combined goodput of this colocated pair, accounting for the negligible intra-node transmission time, and selects the pair that maximizes efficiency.
This effectively creates a "super-instance" per node that handles both phases locally, scaling out by adding more nodes rather than mixing phases across the network.

**Online Scheduling and Burst Handling**
At runtime, DistServe implements a **First-Come-First-Served (FCFS)** dispatch policy with specific optimizations to handle real-world workload irregularities.
To mitigate **pipeline bubbles** caused by variable prompt lengths (which disrupt inter-op parallelism), the scheduler groups requests such that the total number of tokens in a batch is close to a critical threshold $L_m$ that saturates the GPU.
If a request is longer than $L_m$, it is scheduled alone; if shorter, multiple requests are batched together to reach $L_m$, ensuring consistent execution times across pipeline stages.
To handle **burstiness** (sudden spikes in traffic) without overwhelming the decoding instances' memory, DistServe uses a **"pull" mechanism** for KV cache transmission.
Instead of the prefill instance pushing data immediately (which could fill up the decoding instance's memory if it is lagging), the decoding instance requests the KV cache only when it is ready to process the next step.
This design uses the prefill instance's GPU memory as a natural queue buffer, allowing the prefill phase to continue processing new prompts at full speed even if the decoding phase is temporarily saturated.

**Replanning and Adaptation**
Recognizing that workload patterns (average input/output length, arrival rate) change over time (e.g., hourly), DistServe includes a **periodic replanning** mechanism.
A background profiler continuously monitors key workload statistics; if a significant shift is detected (e.g., users start sending much longer prompts), the system triggers the placement algorithm again using the recent historical data.
Because the simulation-based search is fast (seconds to minutes) and reloading model weights takes only minutes, the system can dynamically re-optimize the number of prefill vs. decoding instances and their parallelism strategies to maintain optimal goodput as the application usage evolves.

**Design Choices and Trade-offs**
The decision to use **disaggregation** over **chunked prefill** (used in systems like DeepSpeed-MII) is driven by the $O(N^2)$ memory access penalty of chunking; DistServe avoids this entirely by processing the prompt in one go on dedicated hardware.
The choice to rely on **simulation** rather than real-time profiling for placement decisions is due to the prohibitive time cost of testing every parallelism configuration on physical hardware; the paper validates that the simulator's error margin is less than 2% compared to real runs.
Finally, the system explicitly chooses **not** to implement complex preemption or fault tolerance in this version, acknowledging that while valuable, these features are orthogonal to the core disaggregation architecture and can be added as future extensions without altering the fundamental goodput gains.

## 4. Key Insights and Innovations

DistServe's contributions extend beyond simple architectural separation; they represent a fundamental shift in how we model, optimize, and deploy LLM serving systems. The following insights distinguish DistServe from prior incremental optimizations like chunked batching or priority scheduling.

### 1. The Paradigm Shift from Throughput to Goodput Optimization
Most existing LLM serving systems (e.g., vLLM, Orca) are designed to maximize **throughput**—the total number of tokens generated per second across all users. This metric assumes that higher aggregate output automatically translates to better service. However, DistServe identifies a critical flaw in this assumption for latency-sensitive applications: maximizing throughput often comes at the expense of individual request latency, leading to Service Level Objective (SLO) violations.

*   **The Innovation:** DistServe introduces **goodput** as the primary optimization target. Goodput is defined strictly as the maximum request rate the system can sustain while ensuring a specific percentage of requests (e.g., 90%) meet *both* their TTFT and TPOT constraints.
*   **Why It Matters:** This shift changes the optimization landscape entirely. A configuration that yields the highest token throughput might fail the goodput test because it causes tail-latency spikes in either the prefill or decoding phase. By optimizing for goodput, DistServe aligns the system's objective directly with the user experience and cost-efficiency (requests served per GPU within SLO), rather than raw hardware utilization.
*   **Distinction from Prior Work:** Prior works like SARATHI or DeepSpeed-MII attempt to *balance* TTFT and TPOT within a throughput-maximizing framework using heuristics (e.g., chunking). DistServe proves that under strict SLOs, the throughput-maximizing configuration is often suboptimal or even invalid, necessitating a dedicated goodput-oriented search space.

### 2. Decoupling Parallelism Strategies via Phase-Specific Queuing Models
A subtle but profound innovation in DistServe is the realization that the optimal **model parallelism strategy** differs fundamentally between the prefill and decoding phases, and that colocation forces a detrimental compromise.

*   **The Innovation:** DistServe derives distinct analytical queuing models (M/D/1) for each phase to mathematically prove that:
    *   **Prefill** benefits from **intra-operator parallelism** (tensor parallelism) when arrival rates are low or SLOs are tight, as this reduces the base execution time ($D$). However, at high loads, **inter-operator parallelism** (pipeline parallelism) becomes superior because it reduces the queuing delay term ($\frac{RD^2}{2(1-RD)}$) more effectively.
    *   **Decoding**, being memory-bound, scales throughput linearly with **inter-operator parallelism** or replication, while intra-operator parallelism offers diminishing returns due to communication overhead outweighing compute gains.
*   **Why It Matters:** In collocated systems, the parallelism strategy must be uniform for the entire model instance. This forces the system to choose a "middle-ground" configuration that is inefficient for both phases. DistServe's disaggregation allows the system to assign, for example, a 3-way intra-op strategy to prefill instances and a 4-way inter-op strategy to decoding instances simultaneously.
*   **Distinction from Prior Work:** Previous research on model parallelism (e.g., Alpa, Megatron-LM) focused primarily on training or static inference latency. DistServe is the first to dynamically select parallelism strategies based on **arrival rate ($R$)** and **SLO strictness**, leveraging the mathematical trade-off between execution time reduction and queuing delay reduction specific to each phase's computational profile.

### 3. Bandwidth-Aware Placement Algorithms for Heterogeneous Clusters
While disaggregation introduces a communication overhead (transferring KV caches), DistServe innovates by treating network topology not as a fixed constraint but as a variable in the optimization problem. It provides two distinct placement algorithms that adapt to the cluster's bandwidth capabilities, ensuring the theoretical benefits of disaggregation are realized in practice.

*   **The Innovation:**
    *   **High Node-Affinity (Algorithm 1):** For clusters with high-speed interconnects (InfiniBand), the algorithm treats the cluster as a flat resource pool, freely placing prefill and decoding instances on any node to maximize per-phase goodput.
    *   **Low Node-Affinity (Algorithm 2):** For clusters with limited cross-node bandwidth, the algorithm introduces a novel **"instance segment" colocation constraint**. It splits the model into pipeline stages and ensures that the prefill segment for stage $i$ and the decoding segment for stage $i$ reside on the same physical node. This forces KV cache transfers to occur over high-bandwidth NVLINK (600 GB/s) rather than slower network links.
*   **Why It Matters:** This approach eliminates the "communication penalty" often cited as a drawback of disaggregation. As shown in **Figure 10**, even for massive models (OPT-175B), the transmission latency accounts for less than 0.1% of total request time because the placement algorithm successfully confines traffic to intra-node links.
*   **Distinction from Prior Work:** Existing disaggregation proposals often assume ideal network conditions or suffer significant performance degradation on commodity clusters. DistServe's explicit handling of low-bandwidth environments via stage-aware colocation makes the architecture viable for real-world, heterogeneous deployments without requiring expensive network upgrades.

### 4. Simulation-Driven Auto-Tuning vs. Heuristic Scheduling
DistServe replaces runtime heuristic scheduling with an offline, simulation-driven auto-tuning engine. Instead of reacting to load changes with simple rules (e.g., "prioritize prefill if queue > X"), DistServe proactively computes the optimal global configuration.

*   **The Innovation:** The system employs a lightweight simulator that models FLOPs, memory access patterns, and queuing dynamics to predict SLO attainment for thousands of potential configurations in minutes. It enumerates the search space of parallelism degrees, GPU counts, and placement strategies to find the global maximum for goodput.
*   **Why It Matters:** The search space for optimal serving configurations is too complex for human intuition or simple heuristics. As noted in **Section 6.2**, the optimal placement for OPT-175B involved a non-trivial mix of inter-op and intra-op degrees (e.g., prefill: 3-inter/3-intra, decode: 3-inter/4-intra) that would be nearly impossible to derive manually. The simulator achieves this with &lt;2% error compared to real runs (**Table 2**), enabling reliable "what-if" analysis before deployment.
*   **Distinction from Prior Work:** While systems like Pollux or Sia use feedback loops for resource adjustment, they often rely on online exploration which can be unstable or slow to converge. DistServe's approach leverages the high predictability of DNN workloads to perform an exhaustive, offline search, guaranteeing a near-optimal starting point that heuristic schedulers cannot match.

### 5. The "Pull-Based" KV Cache Flow Control
A critical runtime innovation that enables stable disaggregation is the shift from a "push" to a "pull" data transmission model for intermediate states.

*   **The Innovation:** Instead of the prefill instance pushing the KV cache to the decoding instance immediately upon completion (which risks overwhelming the decoder's memory during traffic bursts), the decoding instance **pulls** the data only when it is ready to process the next step. The prefill instance retains the KV cache in its own GPU memory, effectively using it as a distributed queue buffer.
*   **Why It Matters:** This design decouples the processing rates of the two phases. If the decoding phase becomes a bottleneck (e.g., due to a sudden spike in output length), the prefill phase can continue operating at full capacity without stalling or causing memory overflow errors on the decoder side. It naturally absorbs burstiness without complex coordination logic.
*   **Distinction from Prior Work:** Traditional pipelined systems often rely on fixed buffer sizes or backpressure mechanisms that can stall the entire pipeline. DistServe's pull mechanism leverages the abundant memory of the prefill GPUs (which are often underutilized in terms of memory capacity compared to compute) to create a robust, self-regulating flow control system.

## 5. Experimental Analysis

To validate the hypothesis that disaggregating prefill and decoding phases improves goodput, the authors conduct a comprehensive evaluation across multiple model scales, real-world workloads, and cluster configurations. This section dissects the experimental methodology, quantifies the performance gains, and critically assesses the robustness of the results.

### 5.1 Evaluation Methodology

**Cluster Testbed and Hardware**
The experiments are deployed on a physical cluster consisting of **4 nodes**, each equipped with **8 NVIDIA SXM A100-80GB GPUs**.
*   **Intra-node Connectivity:** GPUs within a node are connected via **NVLINK**, providing high bandwidth (600 GB/s peak).
*   **Inter-node Connectivity:** Cross-node bandwidth is limited to **25 Gbps**, simulating a realistic constraint where cross-node communication is significantly slower than intra-node transfers.
*   **Implication:** Due to this limited cross-node bandwidth, most experiments utilize the **Low Node-Affinity Placement Algorithm (Algorithm 2)**, which forces prefill and decoding segments of the same pipeline stage to reside on the same node to leverage NVLINK.

**Models and Workloads**
The evaluation uses the **OPT** model family (13B, 66B, and 175B parameters) in FP16 precision. The authors selected OPT specifically because it uses standard Multi-Head Attention (MHA), which generates larger KV caches compared to newer models using Grouped-Query Attention (GQA). This choice intentionally stresses the communication overhead of disaggregation; the authors note that DistServe would perform even better on GQA models due to reduced transfer sizes.

Three distinct real-world applications are simulated using specific datasets (Table 1):
1.  **Chatbot:** Uses the **ShareGPT** dataset.
    *   *Characteristics:* Moderate input length (avg ~755 tokens) and output length (avg ~200 tokens).
    *   *SLOs:* Stringent TTFT (0.25s for 13B, up to 4.0s for 175B) and TPOT (0.1s–0.2s).
2.  **Code Completion:** Uses the **HumanEval** dataset.
    *   *Characteristics:* Shorter inputs (avg ~171 tokens) and outputs (avg ~98 tokens).
    *   *SLOs:* Extremely stringent TTFT (0.125s) to mimic real-time coding assistance.
3.  **Summarization:** Uses the **LongBench** dataset.
    *   *Characteristics:* Very long inputs (avg ~1738 tokens) and shorter outputs (avg ~90 tokens).
    *   *SLOs:* Loose TTFT (15s) but stringent TPOT (0.15s) to ensure fast summary generation.

**Baselines**
DistServe is compared against two state-of-the-art collocated systems:
*   **vLLM:** Represents the standard approach using continuous batching and PagedAttention. It colocates prefill and decoding, optimizing for throughput. The authors configure it with optimal intra-operator parallelism degrees (1, 4, and 8 for 13B, 66B, and 175B models respectively).
*   **DeepSpeed-MII:** Represents advanced scheduling using **chunked prefill** (splitting prompts into chunks to interleave with decoding). This baseline tests whether software-level mitigation of interference can match hardware disaggregation. Note: DeepSpeed-MII could not serve OPT-175B due to kernel constraints regarding vocabulary size and parallelism degrees.

**Metrics**
The primary metric is **SLO Attainment**: the percentage of requests that meet *both* TTFT and TPOT constraints.
*   **Goodput Definition:** The maximum request rate (requests per second, rps) achievable while maintaining **90% SLO attainment**.
*   **Secondary Metric:** **SLO Scale**, which measures how much the latency requirements can be tightened (scaled down) while still maintaining 90% attainment at a fixed rate.

### 5.2 Quantitative Results: Goodput and Latency SLOs

The results overwhelmingly support the claim that disaggregation eliminates the interference bottleneck, allowing DistServe to serve significantly higher loads or meet tighter latency constraints.

**Chatbot Workload (ShareGPT)**
Figure 8 presents the performance across three model sizes.
*   **Throughput Gain (Rate):** At the 90% SLO attainment threshold, DistServe sustains **2.0× to 4.6× higher request rates** compared to vLLM.
    *   For **OPT-175B**, DistServe achieves a per-GPU rate where vLLM fails to meet SLOs entirely at lower loads. The gap widens as model size increases because the compute/memory disparity between prefill and decoding becomes more pronounced.
*   **Comparison with DeepSpeed-MII:** DistServe outperforms DeepSpeed-MII by **1.6× to 7.4×** in sustainable request rate.
    *   *Analysis:* While chunked prefill in DeepSpeed-MII mitigates some interference, it incurs the $O(N^2)$ memory access penalty described in Section 2.3. As the model grows, this overhead prevents DeepSpeed-MII from meeting the stringent TTFT SLOs that DistServe handles easily via dedicated compute.
*   **Latency Robustness (SLO Scale):** When fixing the request rate and tightening the latency requirements, DistServe supports **1.8× to 3.2× tighter SLOs** than vLLM. This indicates that DistServe not only handles higher volume but also provides more consistent low-latency performance for individual users.

**Code Completion (HumanEval)**
Figure 9(a) highlights performance on the code completion task, which demands ultra-low TTFT.
*   **Result:** DistServe sustains **5.7× higher request rates** than vLLM and **1.6× higher** than DeepSpeed-MII.
*   **Reasoning:** In this scenario, the system is constrained primarily by TTFT. Collocated systems (vLLM) suffer because decoding jobs in the batch delay the start of prefill jobs. DistServe's dedicated prefill instances, optimized with high intra-operator parallelism (as determined by the placement algorithm), minimize the base execution time $D$, directly satisfying the tight 0.125s requirement.

**Summarization (LongBench)**
Figure 9(b) evaluates the long-context scenario.
*   **Result:** DistServe achieves **4.3× higher request rates** than vLLM and a staggering **12.6× tighter SLO**.
*   **Reasoning:** Here, the bottleneck shifts to TPOT. Long prefill jobs in collocated systems block decoding steps for extended periods, causing massive TPOT spikes. By isolating decoding on dedicated GPUs, DistServe ensures that the generation of summary tokens proceeds at a constant, high speed, unaffected by the arrival of new, long prompts.

**Specific Configuration Insight**
The paper highlights a non-trivial configuration chosen by DistServe for OPT-175B on ShareGPT (Section 6.2):
*   **Prefill Instance:** 3-way inter-op $\times$ 3-way intra-op.
*   **Decoding Instance:** 3-way inter-op $\times$ 4-way intra-op.
This heterogeneous assignment confirms the paper's thesis: a single uniform parallelism strategy (required by vLLM) cannot simultaneously optimize both phases. The automatic search found a "sweet spot" that manual tuning would likely miss.

### 5.3 Latency Breakdown and Communication Overhead

A critical counter-argument to disaggregation is the cost of transferring KV caches between GPUs. Figure 10 provides a detailed breakdown to address this.

*   **Transmission Overhead:** For the largest model (OPT-175B), KV cache transmission accounts for **less than 0.1%** of the total end-to-end latency.
*   **Absolute Delay:** The Cumulative Distribution Function (CDF) in Figure 10(b) shows that **>95% of requests** experience a transmission delay of **less than 30ms**.
*   **Why it works:** Despite the testbed having slow cross-node links (25 Gbps), the **Low Node-Affinity Placement Algorithm (Algorithm 2)** successfully colocated communicating pipeline stages on the same node. This forced all KV cache transfers to occur over NVLINK (600 GB/s), rendering the communication cost negligible.
*   **Conclusion:** The experiment proves that with intelligent placement, the theoretical communication penalty of disaggregation is practically invisible, even for massive models with large context windows.

### 5.4 Ablation Studies and Simulator Accuracy

To isolate the sources of improvement, the authors conducted ablation studies (Figure 11) and validated their simulation tools (Table 2).

**Simulator Accuracy (Table 2)**
DistServe relies on a simulator to search the vast configuration space. The authors compared simulator predictions against real system runs for vLLM and DistServe-Low across varying rates.
*   **Result:** The error in predicted SLO attainment is **consistently less than 2%**.
*   **Significance:** This high accuracy validates the underlying latency models (Section 3) and justifies using simulation for offline optimization rather than expensive online probing.

**Ablation: Disaggregation vs. Parallelism Tuning**
The authors created a variant called **"vLLM++"**, which applies DistServe's exhaustive search to find the *best possible* parallelism configuration for vLLM, but *without* disaggregation.
*   **Result:** "vLLM++" performed identically to standard vLLM.
*   **Insight:** This is a crucial finding. It demonstrates that simply tuning parallelism strategies within a collocated architecture yields **zero benefit** because the phases remain coupled. The interference persists regardless of how the model is split. The performance gains come strictly from **disaggregation**, not just better tuning.

**Ablation: High vs. Low Node-Affinity**
The study compares **DistServe-High** (simulated ideal infinite bandwidth) vs. **DistServe-Low** (realistic constrained bandwidth).
*   **Result:** DistServe-High shows further improvements over DistServe-Low, but the gap is not prohibitive.
*   **Trade-off:** While ideal networks allow more flexible placement (e.g., placing all prefill on Node A and all decoding on Node B), the node-affinity constraint in DistServe-Low effectively mitigates most bandwidth issues. This confirms the system's viability on commodity clusters without requiring expensive InfiniBand upgrades.

### 5.5 Critical Assessment and Limitations

**Strengths of the Evaluation**
*   **Realistic Workloads:** Using ShareGPT, HumanEval, and LongBench ensures the results reflect actual usage patterns (variable lengths, burstiness) rather than synthetic uniform distributions.
*   **Hardware Constraints:** Testing on a cluster with limited cross-node bandwidth (25 Gbps) is a strong design choice. It proves the system works in "messy" real-world environments, not just idealized lab setups.
*   **Comprehensive Baselines:** Including DeepSpeed-MII addresses the most sophisticated existing alternative (chunked prefill), showing that software tricks cannot fully replace architectural separation.

**Limitations and Conditional Results**
*   **Model Family:** The evaluation is limited to the OPT family. While the authors argue this is a "worst-case" for communication due to large KV caches, results on newer architectures (e.g., Llama-3 with GQA) might show different scaling behaviors, though likely favorable to DistServe.
*   **Small-Scale Scenarios:** The Discussion section (Section 7) acknowledges that DistServe struggles in resource-constrained environments (e.g., single GPU or very small clusters). In such cases, the overhead of managing separate instances and the inability to share memory pools may outweigh the benefits. The evaluation does not cover sub-8-GPU setups extensively, as the placement algorithms require sufficient degrees of freedom to split phases.
*   **Fault Tolerance:** The experiments do not evaluate fault tolerance. As noted in Section 4.3, a failure in a decoding instance could cascade to dependent prefill instances. While the authors propose this as future work, the current evaluation assumes a failure-free environment, which is a simplification for production systems.
*   **Dynamic Replanning Latency:** While the paper mentions replanning takes minutes, the evaluation does not quantify the performance dip *during* the transition period when weights are being reloaded or instances are being migrated.

**Conclusion on Experimental Validity**
The experiments convincingly support the paper's core claims. The data clearly demonstrates that colocation creates an unavoidable ceiling on goodput due to interference, and that disaggregation shatters this ceiling. The specific numbers (up to **7.4× rate increase**, **12.6× tighter SLO**) are robust across different model sizes and application types. The latency breakdown definitively refutes the concern that communication overhead negates the benefits of separation, provided the placement algorithm is respected. The ablation studies further cement the argument that the gain is architectural (disaggregation) rather than algorithmic (better scheduling of collocated tasks).

## 6. Limitations and Trade-offs

While DistServe demonstrates significant gains in goodput for latency-sensitive applications, its architectural disaggregation introduces specific constraints, assumptions, and scenarios where the approach may be suboptimal or inapplicable. A critical understanding of these limitations is essential for determining when to deploy DistServe versus alternative systems.

### 6.1 Dependency on Cluster Topology and Bandwidth
The efficacy of DistServe is heavily contingent on the underlying network infrastructure, creating a trade-off between deployment flexibility and performance optimization.

*   **The "Low Node-Affinity" Constraint:** In clusters with limited cross-node bandwidth (e.g., the 25 Gbps testbed used in **Section 6.1**), DistServe cannot freely place prefill and decoding instances anywhere in the cluster. Instead, it must resort to **Algorithm 2**, which forces prefill and decoding segments of the same pipeline stage to reside on the **same physical node** to utilize high-bandwidth NVLINK.
    *   *Trade-off:* This constraint reduces the search space for optimal resource allocation. As shown in the ablation study (**Figure 11**), **DistServe-High** (simulated infinite bandwidth) outperforms **DistServe-Low** (constrained placement). While DistServe-Low still vastly outperforms collocated baselines, the inability to pool all prefill resources on one set of nodes and all decoding resources on another limits the granularity of scaling.
*   **Communication Overhead Sensitivity:** Although **Figure 10** shows transmission latency is negligible (&lt;0.1%) under optimal placement, this relies on the assumption that KV cache transfers occur over high-speed links. If a deployment forces cross-node transfers for large models (e.g., OPT-175B with massive KV caches) on slow networks, the "pull" mechanism may introduce visible latency spikes, eroding the TPOT gains. The system assumes the cluster topology allows for at least some high-bandwidth intra-node communication; it is not designed for purely distributed, low-bandwidth environments (e.g., geo-distributed serving).

### 6.2 Inefficiency in Resource-Constrained and Small-Scale Settings
DistServe is explicitly designed for large-scale serving clusters, and its benefits diminish or vanish in resource-constrained environments.

*   **Minimum Scale Requirement:** The disaggregation strategy requires sufficient GPUs to instantiate separate prefill and decoding pipelines with meaningful parallelism. In **Section 7 (Discussion)**, the authors acknowledge that for small-scale enterprises or individual researchers with only a few GPUs (or a single GPU), the design space is "significantly limited."
    *   *Why it fails:* On a single GPU, one cannot physically separate the phases. Even with 2–4 GPUs, the overhead of maintaining two distinct model copies (weights + KV cache buffers) and the complexity of the orchestration layer may outweigh the latency benefits. In these cases, simpler non-disaggregated systems like vLLM or DeepSpeed-MII remain more operationally efficient.
*   **Memory Duplication Overhead:** Disaggregation inherently requires storing model weights on both prefill and decoding instances. While collocated systems share a single copy of weights across phases, DistServe duplicates them.
    *   *Constraint:* For very large models (e.g., >175B) on memory-constrained GPUs, this duplication reduces the memory available for KV caches, potentially limiting the maximum batch size for decoding. While the paper mitigates this via inter-op parallelism (splitting layers), the fundamental memory cost of holding two active instances per "logical" model replica is higher than colocation.

### 6.3 Suboptimality for Throughput-Oriented (Offline) Workloads
DistServe optimizes for **goodput** (requests meeting SLOs), not raw **throughput** (total tokens/second). This creates a trade-off where DistServe may underperform in offline or latency-insensitive scenarios.

*   **The Utilization Gap:** In offline applications (e.g., batch data processing, non-real-time summarization), the strict TTFT and TPOT SLOs are irrelevant. Here, the goal is to saturate the GPU compute units completely.
    *   *Evidence:* **Section 7** notes that techniques like **chunked-prefill with piggyback** (used in SARATHI or DeepSpeed-MII) are superior in these contexts. By interleaving prefill chunks with decoding steps, collocated systems can fill every GPU cycle, achieving near-100% utilization. DistServe's dedicated phases may suffer from "straggler" effects where one phase (e.g., decoding) finishes early while the other (prefill) is still busy, leading to lower aggregate hardware utilization compared to a perfectly tuned continuous batching system.
*   **Cost Implication:** For users whose primary metric is "cost per token" rather than "cost per request within SLO," DistServe's architectural overhead might result in higher costs if the latency guarantees are not strictly required.

### 6.4 Operational Complexity and Dynamic Adaptation
The shift from a monolithic serving instance to a distributed, disaggregated pipeline introduces operational challenges that the current implementation does not fully resolve.

*   **Replanning Latency:** DistServe relies on periodic replanning to adapt to workload shifts (e.g., changing from short chat prompts to long document summaries). While **Section 4.3** states the algorithm runs in seconds and weight reloading takes minutes, this process is not instantaneous.
    *   *Weakness:* During the transition window (minutes), the system operates with a suboptimal configuration. If workload patterns change rapidly (e.g., flash crowds with unpredictable distributions), the system may lag in adaptation, temporarily violating SLOs. The current "periodic" approach assumes workload stability on the scale of hours, which may not hold for all viral or event-driven traffic.
*   **Fault Tolerance and Error Propagation:** The paper explicitly identifies fault tolerance as an open challenge in **Section 4.3**.
    *   *Risk:* In collocated systems, a failure in one instance typically affects only the requests currently on that GPU. In DistServe, the tight coupling via KV cache handoffs creates a risk of **fault propagation**. If a decoding instance fails, all prefill instances mapped to it must halt or buffer indefinitely, potentially stalling a large portion of the cluster. The current implementation lacks built-in mechanisms for seamless failover or state migration between decoding instances, leaving this as a critical gap for production readiness.
*   **Lack of Preemption:** DistServe uses a simple First-Come-First-Served (FCFS) policy. It does not implement **preemption** (pausing long jobs to let short jobs finish), which is a feature in systems like FastServe.
    *   *Consequence:* As noted in **Section 4.3**, this can lead to the "convoy effect," where a single extremely long prefill request delays all subsequent requests in its queue, increasing tail latency. While the batching strategy mitigates this by grouping requests to saturation ($L_m$), the inability to preempt a running job remains a limitation for handling highly variable request lengths dynamically.

### 6.5 Model and Context Length Assumptions
The evaluation and design rely on specific characteristics of the models and workloads tested, which may not generalize perfectly to all future LLM architectures.

*   **KV Cache Size Assumption:** The experiments use the OPT family with standard Multi-Head Attention (MHA), which generates large KV caches. The authors argue in **Section 6.1** that DistServe would perform *better* on models with Grouped-Query Attention (GQA) or Multi-Query Attention (MQA) (e.g., Llama-2/3, Mistral) because the communication overhead would be lower.
    *   *Caveat:* While likely true for bandwidth, the *compute* characteristics of these newer models might shift the balance. If the prefill phase becomes less compute-bound relative to memory due to architectural changes, the optimal parallelism strategies derived in **Section 3** might need re-calibration.
*   **Extreme Long-Context Scaling:** **Section 7** discusses the trend toward million-token context windows. While disaggregation helps isolate the heavy prefill compute, the linear growth of KV cache size with context length poses a scaling challenge for the "pull" mechanism.
    *   *Open Question:* As context lengths reach millions of tokens, the KV cache for a single request could exceed the memory of a single GPU or even a node. The current placement algorithms assume the KV cache can be transferred and stored efficiently; extreme contexts might require new sharding strategies for the KV cache itself, which DistServe does not currently address.

### Summary of Trade-offs

| Feature | DistServe Approach | Trade-off / Limitation |
| :--- | :--- | :--- |
| **Optimization Target** | Goodput (SLO-aware) | May yield lower raw throughput/utilization than collocated systems for offline batches. |
| **Resource Allocation** | Disaggregated (Separate GPUs) | Requires more total GPUs to start; inefficient for single-GPU or small-cluster setups. |
| **Placement** | Topology-Aware (Algo 1 & 2) | Constrained by network bandwidth; optimal placement requires high intra-node bandwidth (NVLINK). |
| **Scheduling** | FCFS + Periodic Replanning | No preemption; adaptation to workload shifts takes minutes (not real-time). |
| **Reliability** | Basic Orchestration | No built-in fault tolerance; risk of fault propagation between phases. |
| **Memory** | Dedicated per Phase | Duplicates model weights; reduces total available memory for KV caching compared to shared pools. |

In conclusion, DistServe is a specialized solution optimized for **large-scale, latency-critical online serving** where SLO adherence is paramount. It is not a universal replacement for all serving systems; for small-scale deployments, offline batch processing, or environments with severe network constraints, traditional collocated architectures with advanced batching (like chunked prefill) may remain the pragmatic choice.

## 7. Implications and Future Directions

DistServe fundamentally alters the architectural assumptions of Large Language Model (LLM) serving, shifting the field's focus from monolithic throughput maximization to modular goodput optimization. By demonstrating that the computational interference between prefill and decoding phases is an architectural bottleneck rather than a scheduling nuisance, this work opens new avenues for system design, hardware utilization, and service level agreement (SLA) management.

### 7.1 Shifting the Landscape: From Monoliths to Modular Pipelines
The most profound implication of DistServe is the validation of **phase-specific resource provisioning**. Prior to this work, the prevailing dogma in LLM serving (exemplified by vLLM, Orca, and TGI) was that colocation and continuous batching were necessary to maintain high GPU utilization. DistServe proves that for latency-sensitive workloads, the cost of interference outweighs the benefits of shared resources.

*   **Decoupling Hardware Requirements:** This work suggests that future serving clusters may not need homogeneous GPU pools. Instead, infrastructure could be specialized:
    *   **Prefill Nodes:** Optimized for high compute density (e.g., GPUs with higher FLOPS, potentially lower memory bandwidth) and configured with aggressive intra-operator parallelism to minimize Time to First Token (TTFT).
    *   **Decoding Nodes:** Optimized for high memory bandwidth and large capacity (to hold massive KV caches) and configured with inter-operator parallelism or replication to maximize Time Per Output Token (TPOT) throughput.
*   **Redefining Efficiency Metrics:** The industry standard metric of "tokens per second" is revealed to be insufficient for user-facing applications. DistServe establishes **goodput** (requests meeting SLOs per GPU) as the true north star for cost-efficiency. This shifts the economic model of LLM APIs from selling raw compute cycles to selling guaranteed latency tiers, allowing providers to offer premium low-latency services without blindly over-provisioning hardware.

### 7.2 Enabling Follow-Up Research
DistServe provides a foundational architecture that invites several critical lines of future inquiry:

*   **Fault Tolerance and State Migration:** As noted in **Section 4.3**, DistServe currently lacks robust fault tolerance. The tight coupling via KV cache handoffs creates a risk where a single decoding instance failure could stall multiple prefill instances. Future research must develop **stateless handoff protocols** or **KV cache replication strategies** that allow a failed decoding instance to be replaced mid-generation without restarting the user's request. This might involve streaming KV caches to a shared memory pool (e.g., CXL-linked memory) rather than holding them solely on the prefill GPU.
*   **Dynamic Preemption and Scheduling:** The current First-Come-First-Served (FCFS) policy is susceptible to the "convoy effect" from long prompts. Future work could integrate **iteration-level preemption** (as seen in FastServe) into the disaggregated architecture. Since prefill and decoding are separate, preempting a long prefill job to clear a queue for high-priority short prompts becomes easier to implement without corrupting the state of ongoing decoding streams.
*   **Heterogeneous Hardware Orchestration:** DistServe's placement algorithms (**Section 4**) currently assume homogeneous GPU clusters (all A100s). A natural extension is to explore **heterogeneous disaggregation**, where prefill runs on newer, faster training-grade GPUs (e.g., H100s) while decoding runs on older, cost-effective inference GPUs (e.g., A10s or L40s). The challenge lies in adapting the simulator to model the performance delta and ensuring the KV cache format is compatible across different hardware architectures.
*   **Extreme Long-Context Scaling:** With the rise of models supporting 1M+ token contexts (e.g., Claude-3, Gemini-1.5), the KV cache size becomes massive. DistServe's "pull" mechanism assumes the cache fits in the prefill GPU's memory. Future research must address **distributed KV cache sharding**, where the intermediate state itself is split across multiple nodes, requiring a more complex communication protocol than the current point-to-point transfer.

### 7.3 Practical Applications and Downstream Use Cases
The architectural shifts proposed by DistServe directly enable new classes of applications and operational strategies:

*   **Tiered Service Levels:** Cloud providers can now technically enforce distinct SLAs.
    *   **"Turbo" Tier:** Assign requests to a dedicated pool of prefill instances with high intra-op parallelism and low-queue decoding instances, guaranteeing sub-100ms TTFT for real-time conversational agents.
    *   **"Standard" Tier:** Route requests to a shared pool with higher batching, accepting higher latency for lower cost.
    *   *Impact:* This allows providers to monetize latency guarantees, a capability previously difficult to enforce without massive over-provisioning.
*   **Real-Time Interactive Agents:** Applications requiring strict interactivity, such as **voice assistants** or **pair programming tools** (HumanEval workload in **Section 6**), benefit disproportionately. The elimination of decoding starvation ensures that once the agent starts speaking or coding, the stream never hiccups, regardless of how many new users are joining the chat simultaneously.
*   **Hybrid Batch/Online Clusters:** An organization could run a single physical cluster that dynamically shifts resources. During the day, the cluster operates in DistServe mode for interactive chatbots. At night, the decoding instances could be repurposed for offline batch summarization tasks (where throughput matters more than TPOT), while prefill instances handle bulk data ingestion. The periodic replanning mechanism (**Section 4.3**) facilitates this diurnal rhythm.

### 7.4 Reproducibility and Integration Guidance
For practitioners and researchers looking to adopt or build upon DistServe, the following guidance outlines when and how to deploy this architecture:

*   **When to Prefer DistServe:**
    *   **Latency is Critical:** If your application has strict TTFT (&lt;500ms) or TPOT (&lt;100ms) SLOs that must be met for >90% of requests.
    *   **Scale is Sufficient:** You have access to a cluster with at least **8–16 GPUs**. The overhead of managing separate instances and the memory duplication of model weights makes DistServe inefficient for single-GPU or very small (2–4 GPU) setups.
    *   **Workload is Variable:** Your traffic patterns exhibit burstiness or varying input lengths, where collocated systems struggle to balance the queue.
*   **When to Stick with Colocation (vLLM/DeepSpeed-MII):**
    *   **Offline/Batch Processing:** If you are generating datasets or summarizing documents where total completion time matters more than individual request latency, continuous batching (colocation) will likely yield higher raw throughput.
    *   **Resource Constrained:** If you are deploying on edge devices or small clusters where memory is the primary bottleneck, the duplication of weights in DistServe may prevent you from loading the model at all.
    *   **Network Limitations:** If your cluster lacks high-bandwidth intra-node links (NVLINK) and has very slow inter-node connectivity (&lt;10 Gbps), the communication overhead may negate the benefits of disaggregation unless you strictly adhere to the node-affinity constraints.

*   **Integration Steps:**
    1.  **Profile Workload:** Before deployment, collect traces of input/output lengths and arrival rates. DistServe's simulator (**Section 4.1**) requires these distributions to find the optimal placement.
    2.  **Select Placement Algorithm:**
        *   Use **Algorithm 1** if your cluster has high-speed InfiniBand/RoCE across all nodes.
        *   Use **Algorithm 2** (Low Node-Affinity) if cross-node bandwidth is limited; ensure your physical topology groups GPUs with NVLINK.
    3.  **Configure Replanning:** Set the workload profiler to trigger re-optimization every 1–2 hours or upon detecting a statistical shift in request length distributions.
    4.  **Monitor KV Cache Memory:** Since prefill instances act as buffers in the "pull" model, monitor their memory usage closely during traffic bursts to ensure they do not OOM (Out Of Memory) while holding pending KV caches.

In summary, DistServe does not merely offer a better scheduler; it proposes a new **operating system for LLMs** where compute phases are treated as independent microservices. This paradigm shift enables a level of latency guarantee and resource efficiency that monolithic architectures cannot achieve, paving the way for the next generation of responsive, cost-effective AI services.
