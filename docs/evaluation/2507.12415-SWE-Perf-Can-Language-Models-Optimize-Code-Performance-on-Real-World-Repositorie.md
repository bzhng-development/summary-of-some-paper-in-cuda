## 1. Executive Summary

This paper introduces **SWE-Perf**, the first benchmark designed to systematically evaluate Large Language Models (LLMs) on repository-level code performance optimization, comprising **140** curated instances derived from real-world pull requests across **9** popular GitHub repositories like `scikit-learn` and `xarray`. By comparing leading models (e.g., **Claude-3.7-sonnet**, **OpenAI-o3**) and autonomous agents (**OpenHands**, **Agentless**) against expert human patches, the study reveals a critical capability gap: while experts achieve an average performance gain of **10.85%**, the best-performing agent (OpenHands) only reaches **2.26%**, and direct model approaches struggle to exceed **1.76%**. This work matters because it shifts the evaluation focus from mere code correctness to actual runtime efficiency in complex, multi-file contexts, demonstrating that current LLMs largely fail to replicate the high-level abstraction and strategic refactoring skills required for production-grade optimization.

## 2. Context and Motivation

### The Unaddressed Gap: Correctness vs. Efficiency
While Large Language Models (LLMs) have achieved remarkable proficiency in generating syntactically correct code and fixing functional bugs, a critical dimension of software engineering remains largely unexplored: **repository-level performance optimization**.

Existing benchmarks and tools operate under a binary success metric: does the code pass the unit tests? This focus on **correctness** ignores a fundamental requirement of production systems: **efficiency**. As noted in the Introduction, while correctness is foundational, performance optimization often yields "more profound system-wide benefits." The specific problem this paper addresses is the lack of a standardized, rigorous framework to evaluate whether LLMs can do more than just make code *work*—can they make code *fast*?

The gap is twofold:
1.  **Scope Limitation:** Prior efficiency benchmarks (e.g., `Mercury`, `EffiBench`, `EvalPerf`) focus on **function-level** optimizations. They present isolated algorithmic problems (like sorting or mathematical computations) where the context is self-contained. However, real-world optimization rarely happens in a vacuum. It often requires understanding data flows across multiple files, refactoring module interactions, or altering architectural patterns—complexities that isolated function benchmarks completely miss.
2.  **Lack of Ground Truth:** In bug-fixing benchmarks like `SWE-Bench`, the "correct" solution is defined by passing tests. In performance optimization, passing tests is necessary but insufficient. Without a human-authored "gold standard" patch that demonstrates a specific runtime improvement, it is impossible to determine if an LLM's optimization is truly effective or if further improvements are possible. The field lacks a dataset where the *optimal* efficiency target is known and measurable.

### Why This Problem Matters
The inability of current models to optimize performance at the repository level poses significant risks and limitations for deploying AI in serious software engineering workflows:

*   **Real-World Impact:** In production environments, inefficient code translates directly to higher computational costs, increased energy consumption, and degraded user experience. An LLM that generates functionally correct but computationally expensive code might pass all unit tests while causing system slowdowns or budget overruns in deployment.
*   **The "Ceiling" of Automation:** If AI agents cannot optimize code, their utility is capped at junior-developer levels. Senior engineering value often comes from refactoring legacy systems for scalability—a task requiring deep contextual understanding of how changes in one module affect the performance of another. Without this capability, LLMs cannot fully automate the software development lifecycle.
*   **Theoretical Significance:** Performance optimization is an **open-ended problem**. Unlike bug fixing, where the goal is to reach a state of "no errors," optimization has no clear stopping point. Evaluating this requires new metrics that measure *degree of improvement* rather than binary success, pushing the boundaries of how we assess AI reasoning and planning capabilities.

### Limitations of Prior Approaches
To understand the novelty of `SWE-Perf`, one must examine where existing methodologies fall short:

#### 1. Function-Level Efficiency Benchmarks
Works such as `Mercury` (Du et al., 2024), `EffiBench` (Huang et al., 2024), and `KernelBench` (Ouyang et al., 2025) have made strides in evaluating code efficiency. However, they suffer from **oversimplification**.
*   **The Flaw:** These benchmarks treat code as isolated snippets. They ignore **cross-cutting concerns** like dataflow refactoring or parallelism that span multiple files.
*   **The Consequence:** A model might excel at optimizing a single sorting function in isolation but fail to recognize that changing a data structure in a shared module could speed up ten different downstream processes. Real-world optimization potential is often unlocked by these repository-scale interactions, which function-level benchmarks cannot capture.

#### 2. Repository-Level Correctness Benchmarks
Benchmarks like `SWE-Bench` (Jimenez et al., 2024) and `SWE-Dev` (Du et al., 2025) successfully evaluate LLMs on repository-level tasks, but exclusively for **bug fixing** or **feature implementation**.
*   **The Flaw:** These tasks have well-defined objectives (e.g., "fix this crash"). Performance optimization, by contrast, is ambiguous. It requires the agent to *identify* the bottleneck, *plan* a refactor, and *verify* the speedup without breaking functionality.
*   **The Consequence:** Existing frameworks do not provide the infrastructure to measure runtime differences or the "expert" patches needed to define what a successful optimization looks like. They assume that if the tests pass, the job is done.

#### 3. Agent Frameworks Not Designed for Optimization
Current autonomous agents like `Agentless` (Xia et al., 2024) and `OpenHands` (Wang et al., 2024) are designed primarily for issue resolution.
*   **The Flaw:** Their workflows (fault localization -> repair -> test) are optimized for restoring correctness, not maximizing efficiency. They lack specific mechanisms to profile code, identify hotspots, or reason about algorithmic complexity changes across a large codebase.
*   **The Consequence:** When applied to performance tasks, these agents often make superficial changes or fail to navigate the repository effectively to find the root cause of slowness.

### Positioning of SWE-Perf
`SWE-Perf` positions itself as the bridge between **repository-level context** and **performance-specific evaluation**. It differentiates itself through three key design choices:

1.  **Real-World Derivation:** Unlike synthetic algorithmic problems, `SWE-Perf` is constructed from **140 actual pull requests** from popular repositories (e.g., `xarray`, `scikit-learn`) where human experts successfully improved performance. This ensures the tasks reflect genuine engineering challenges rather than contrived puzzles.
2.  **Executable Ground Truth:** The benchmark provides not just the code and tests, but also the **expert-authored patch** and a containerized environment to measure runtime. This allows for a direct comparison: *How close does the LLM get to the human expert's speedup?*
3.  **Dual Evaluation Settings:** The paper introduces two distinct modes to probe model capabilities:
    *   **Oracle (File-Level):** Gives the model the exact files and functions to optimize, testing pure coding ability.
    *   **Realistic (Repo-Level):** Forces the model to act as an autonomous agent, navigating the entire repository to find and fix bottlenecks, testing planning and retrieval capabilities.

By shifting the metric from "Did the tests pass?" to "How much faster is the code compared to the expert?", `SWE-Perf` establishes a new standard for evaluating the next frontier of AI coding capabilities: **performance-aware software engineering**.

## 3. Technical Approach

This section details the construction of the `SWE-Perf` benchmark, moving from the high-level concept of extracting performance data from GitHub to the rigorous statistical methods used to verify that a code change actually makes software faster. Unlike standard benchmarks that simply collect code and tests, `SWE-Perf` implements a multi-stage filtering pipeline designed to isolate *stable, measurable, and attributable* performance gains from the noise of real-world software development.

### 3.1 Reader orientation (approachable technical breakdown)
The `SWE-Perf` system is an automated data engineering pipeline that crawls popular GitHub repositories, executes thousands of unit tests in isolated containers, and applies statistical hypothesis testing to identify pull requests that genuinely improve code execution speed. It solves the problem of "optimization ambiguity" by transforming vague code changes into quantifiable performance metrics, creating a dataset where every example has a verified "before" and "after" runtime and a human-expert solution to serve as a gold standard.

### 3.2 Big-picture architecture (diagram in words)
The architecture functions as a five-stage funnel, progressively narrowing down millions of potential code changes into a curated set of 140 high-quality optimization tasks:
1.  **Collection Module:** Crawls GitHub for Pull Requests (PRs) from popular repositories, filtering for those that resolve issues but ignoring whether they add new tests.
2.  **Measurement Engine:** Builds Docker containers for both the original and modified code versions of each PR, executing all unit tests to record raw execution times (runtimes).
3.  **Identification Filter:** Applies initial heuristics to discard PRs where tests fail or where the speedup is negligible (less than 30%), keeping only candidates with significant potential.
4.  **Statistical Verifier:** Re-runs promising tests 20 times with warm-ups and outlier removal, using the Mann-Whitney U test to calculate a statistically significant minimum performance gain ($\delta$).
5.  **Target Extractor:** Analyzes the code and test execution traces to define specific "target functions" for the model to optimize, creating two distinct evaluation modes: "Oracle" (file-level) and "Realistic" (repo-level).

### 3.3 Roadmap for the deep dive
To fully understand how `SWE-Perf` ensures data quality and evaluation rigor, we will proceed in the following logical order:
*   **Task Formulation:** We first define the exact input and output of the optimization task, explaining why "target functions" are provided to constrain the problem space.
*   **Data Collection Pipeline:** We walk through the five phases of data extraction, detailing the specific constraints (e.g., CPU limits, retry counts) used to ensure fair comparison.
*   **Statistical Verification:** We deep-dive into the mathematical method (Algorithm 1) used to prove that a speedup is real and not just random noise, defining the conservative gain metric $\delta$.
*   **Target Extraction Strategy:** We explain the distinction between the "Oracle" and "Realistic" settings and how the system uses dynamic tracing to prevent data leakage.
*   **Evaluation Metrics:** We define the three-tiered scoring system (Apply, Correctness, Performance) used to grade LLMs, emphasizing how the final score is calculated.

### 3.4 Detailed, sentence-based technical breakdown

#### Task Formulation and Scope
The core task in `SWE-Perf` is defined as generating a code patch that, when applied to an original codebase, reduces the execution time of specific target functions while maintaining functional correctness. The input to the system consists of the full repository codebase and a set of **target functions**, while the output is a modified version of the code (a patch). The inclusion of target functions is a deliberate design choice to restrict the evaluation scope; running the full test suite for large repositories like `xarray` (which contains over 220,000 test cases) would take more than one hour per sample on a single-core CPU, making large-scale evaluation impractical. Furthermore, without targeted guidance, the search space for optimization across a massive codebase is too vast for current models to navigate effectively. While the current benchmark provides these targets, the authors note that future work should aim to remove this constraint as models improve.

#### Phase 1: Collecting Pull Requests
The data collection begins by aggregating raw material from the software engineering ecosystem, specifically targeting high-quality, real-world optimization efforts. The system collects Pull Requests (PRs) from 12 popular GitHub repositories previously used in the `SWE-Bench` study, ensuring a baseline of community vetting and popularity. Unlike prior works that filter for PRs contributing new tests, `SWE-Perf` retains any PR that resolves an issue, explicitly allowing PRs that do not modify unit tests because the focus is on execution time rather than test coverage changes. This initial crawl yields a massive pool of **102,241** pull requests, which are then subjected to attribute filtering to prepare for performance measurement.

#### Phase 2: Measuring Codebase Performance
To accurately compare the speed of code before and after a change, the system must eliminate environmental variability, which is achieved through strict containerization and resource isolation. For every collected PR, the pipeline builds two distinct Docker images: one for the original codebase and one for the modified codebase. To ensure comparability, each Docker container is constrained to exactly **1 CPU core** and **16 GB of memory** during this measurement phase (though later phases use 5 cores). The system then executes all unit tests within these containers using `pytest`, recording the runtime for every individual test case. This step is computationally expensive, resulting in **25,264** distinct codebases being built and tested, with some repositories like `xarray` requiring over an hour per codebase to complete the full test suite. To account for transient system noise, each codebase is evaluated in **three repeated experimental runs**, producing three runtime measurements per unit test.

#### Phase 3: Identifying Performance-Optimizing Pull Requests
Once raw runtime data is collected, the system applies a two-step filtering process to identify PRs that demonstrate genuine optimization potential. First, a **Correctness Filter** ensures that a unit test passes in both the original and modified versions; if a test fails in either version, it is discarded because a faster but broken function is not a valid optimization. Second, a **Performance Ratio Filter** calculates the relative improvement for each test using the mean of the three replicates. The optimization ratio is defined as:

$$ \text{Ratio} = \frac{R_{\text{original}} - R_{\text{modified}}}{R_{\text{original}}} $$

where $R$ represents the runtime of a specific unit test. To be considered a candidate, this ratio must be below a threshold of **0.3** (indicating at least a 30% reduction in runtime, or conversely, the modified time is significantly lower). Additionally, the system dynamically executes the tests to ensure they actually exercise the patched code segments while avoiding any tests that were themselves modified in the PR, guaranteeing that the speedup comes from the logic change, not the test definition. This phase narrows the dataset from tens of thousands of codebases to **1,696** valid instances.

#### Phase 4: Verifying Stable Performance Improvements
The most critical technical contribution of `SWE-Perf` is its rigorous statistical verification process, which distinguishes true performance gains from random fluctuations inherent in computer systems. The system subjects the candidate tests from Phase 3 to a high-precision measurement protocol consisting of four steps. First, a **warm-up** phase executes the test three times to mitigate initialization costs and JIT (Just-In-Time) compilation delays. Second, the test is executed **20 times** to generate a robust distribution of runtime values. Third, outliers are removed using the **Interquartile Range (IQR)** method with a multiplier $k=1$; any data point $r_i$ is discarded if it falls outside the range:

$$ r_i &lt; Q_1 - 1 \times \text{IQR} \quad \text{or} \quad r_i > Q_3 + 1 \times \text{IQR} $$

where $Q_1$ and $Q_3$ are the first and third quartiles, and $\text{IQR} = Q_3 - Q_1$. Finally, the system computes a **Statistically Significant Minimum Performance Gain ($\delta$)** using **Algorithm 1**. This algorithm iteratively tests hypothetical gain values $x$ (starting from 0.0 and incrementing by 0.01) by pessimistically adjusting the modified runtimes ($B_{\text{adj}} = B \times (1-x)$) and performing a **Mann-Whitney U test** against the original runtimes ($A$). The value $\delta$ is the largest $x$ for which the adjusted modified distribution remains statistically significantly faster than the original distribution at a significance level of $\alpha = 0.1$. Only tests with a $\delta$ exceeding **0.05** (5% guaranteed gain) are retained, resulting in the final dataset of **140** instances.

#### Phase 5: Extracting Optimization Targets
To facilitate evaluation, the system extracts specific "target functions" for the models to optimize, creating two distinct experimental settings that test different capabilities. The **Oracle (File-Level)** setting provides the model with the exact files and functions modified in the human expert's patch, extracted via Abstract Syntax Tree (AST) analysis and unified diff matching; this isolates the model's ability to write efficient code without needing to search the repository. In contrast, the **Realistic (Repo-Level)** setting simulates an autonomous agent scenario by providing only the functions measured during testing as targets. These targets are identified using `yappi` to record dynamic execution traces during unit test runs, combined with AST parsing to pinpoint the specific functions directly invoked. Crucially, the system avoids using the unit test code itself as the target to prevent **functional pruning**, a failure mode where a model might delete necessary functionality just to make the specific test run faster. This phase yields an average of **7.6** target functions for the Oracle setting and **30.1** for the Realistic setting per instance.

#### Evaluation Methodology and Metrics
The final component of the technical approach is the three-level evaluation framework used to score LLMs, designed to separate the ability to generate code from the ability to optimize it. The first metric, **Apply**, measures the percentage of generated patches that can be successfully applied to the codebase without syntax errors or conflicts, calculated as $N_{\text{apply}} / N_{\text{total}}$. The second metric, **Correctness**, assesses functional integrity by checking if all performance-related unit tests pass after the patch is applied; a sample is only considered correct if every single test returns a "pass" result. The third and most novel metric, **Performance**, quantifies the actual speedup achieved by the model. It reuses the statistical methodology from Phase 4: for each correct sample, the system runs the tests 20 times, filters outliers, and calculates the minimum significant gain $p_{i,j}$ for each test $j$ using Algorithm 1. The final Performance score is the average of these gains across all tests and samples:

$$ \text{Performance} = \frac{1}{N_{\text{total}}} \sum_{i=1}^{N_{\text{total}}} P_i, \quad \text{where } P_i = \frac{1}{n_i} \sum_{j=1}^{N_i} p_{i,j} $$

This rigorous approach ensures that the reported results reflect genuine, statistically verifiable improvements rather than noisy or lucky outcomes.

## 4. Key Insights and Innovations

The `SWE-Perf` benchmark does more than simply aggregate data; it introduces fundamental shifts in how we conceptualize, measure, and evaluate code optimization. While prior work treated efficiency as a secondary metric or isolated it to algorithmic puzzles, `SWE-Perf` establishes performance as a primary, statistically rigorous objective within complex, real-world software ecosystems. The following innovations distinguish this work from existing literature.

### 1. The Shift from Binary Correctness to Continuous Performance Gain
**Innovation Type:** Fundamental Paradigm Shift

Prior benchmarks in software engineering, including the influential `SWE-Bench`, operate on a **binary success metric**: a solution is either "correct" (all tests pass) or "incorrect" (tests fail). This binary view is insufficient for performance optimization, where a solution can be functionally correct but vary wildly in efficiency. A model could theoretically pass all tests by implementing a brute-force $O(n^2)$ algorithm when an $O(n \log n)$ solution exists, receiving a perfect score despite being suboptimal.

`SWE-Perf` innovates by replacing this binary switch with a **continuous performance metric**.
*   **The Mechanism:** Instead of asking "Did it work?", the benchmark asks "How much faster is it?" using the **Statistically Significant Minimum Performance Gain ($\delta$)** derived in Phase 4 (Section 3.2). This metric quantifies the *magnitude* of improvement, ensuring that a 1% speedup is distinguished from a 50% speedup.
*   **Significance:** This allows researchers to measure *degrees of competence*. As shown in **Table 2**, while the best models achieve high "Correctness" scores (e.g., 78.57% for Claude-4-opus in the Oracle setting), their "Performance" scores remain abysmal (1.28% vs. the expert's 10.85%). Without this continuous metric, these models would appear nearly successful; with it, the critical capability gap is exposed. This shift forces the field to move beyond "does it compile?" to "is it production-ready?"

### 2. Statistical Rigor in Noisy Environments (Algorithm 1)
**Innovation Type:** Methodological Advance

Measuring code performance is notoriously difficult due to system noise, caching effects, and JIT compilation variances. Naive benchmarks often rely on single-run averages or simple percentage differences, which are prone to false positives (claiming a speedup where none exists) or false negatives (missing subtle optimizations).

`SWE-Perf` introduces a novel statistical verification pipeline centered on **Algorithm 1** (Section 3.2), which computes a **conservative lower bound** on performance gain.
*   **The Mechanism:** Rather than simply comparing means, the algorithm iteratively tests hypothetical gain values ($x$) by pessimistically shrinking the modified runtime distribution ($B_{adj} = B \times (1-x)$). It then applies the **Mann-Whitney U test** to determine if this "weakened" modified version is still statistically faster than the original. The output $\delta$ is the maximum gain that can be claimed with statistical confidence ($p &lt; 0.1$).
*   **Significance:** This approach prioritizes **precision over recall**. By filtering out unstable improvements and requiring a minimum $\delta > 0.05$, the benchmark ensures that every data point represents a genuine, reproducible engineering win. This rigor is absent in prior function-level benchmarks like `EffiBench` or `Mercury`, which often lack such robust noise-filtering mechanisms for repository-scale execution. It sets a new standard for empirical validity in AI code evaluation.

### 3. Dynamic Trace-Based Target Extraction to Prevent "Gaming"
**Innovation Type:** Novel Defense Against Adversarial Optimization

A unique challenge in performance benchmarking is preventing models from "gaming" the metrics. If a model knows exactly which unit test will be run, it might engage in **functional pruning**—stripping away essential logic that the test doesn't check, thereby making the code faster but broken for any other use case.

`SWE-Perf` solves this through a sophisticated **Dynamic Trace-Based Target Extraction** strategy (Phase 5, Section 3.2).
*   **The Mechanism:** Instead of giving the model the unit test code as a target, the system uses `yappi` to record the dynamic execution trace of the test run. It identifies the specific functions *invoked* during the test but explicitly excludes the test code itself from the optimization scope. In the **Realistic** setting, the target is defined as the set of functions executed, not the files modified.
*   **Significance:** This decouples the *verification signal* (the test) from the *optimization target* (the function). It forces the model to understand the semantic role of the code rather than just minimizing operations to satisfy a specific test assertion. This design choice addresses a subtle but critical failure mode in automated evaluation that prior static-analysis-based benchmarks overlook.

### 4. Revealing the "Low-Level vs. High-Level" Optimization Divide
**Innovation Type: Empirical Discovery via Novel Analysis**

By constructing a dataset with expert-authored patches, `SWE-Perf` enables a comparative analysis that was previously impossible: directly contrasting *how* humans optimize versus *how* LLMs optimize. The keyword analysis in **Section 5.3.4** (Figures 10 and 11) reveals a profound qualitative divergence.

*   **The Insight:**
    *   **Experts** focus on **high-level abstractions and data integrity**. Their patches frequently involve terms like "dtype," "label," "value," and "type," indicating strategic refactoring of data structures and type safety to improve efficiency.
    *   **LLMs** (specifically agents like OpenHands) focus on **low-level infrastructure and syntax**. Their patches are dominated by terms like "miniconda3," "envs," "importlib," and "bootstrap," suggesting they attempt to tweak environment configurations or make superficial syntactic changes rather than restructuring algorithmic logic.
*   **Significance:** This finding explains *why* the performance gap exists. It is not merely a lack of coding skill; it is a lack of **architectural reasoning**. Current models act like junior developers tweaking configuration files, whereas experts act like senior engineers refactoring core data flows. This insight directs future research away from simply scaling model size and toward improving long-context planning and abstract reasoning capabilities.

### 5. The "Realistic" Repository-Level Stress Test
**Innovation Type:** Scalability Benchmarking

While some prior works evaluate code generation in isolation, `SWE-Perf` introduces a **Realistic (Repo-Level)** setting that serves as a stress test for autonomous agents.
*   **The Mechanism:** Unlike the **Oracle** setting (which provides the exact files to edit), the Realistic setting requires the agent to navigate a repository with an average of **447 files** and **170k lines of code** to locate the bottleneck.
*   **Significance:** The results highlight a sharp decline in capability as context expands. As shown in **Figure 8**, model performance degrades rapidly as the number of target functions increases, whereas experts maintain higher efficiency. This demonstrates that current agents struggle with **cross-module dependency reasoning**. The innovation here is the provision of a testbed where "finding the problem" is as hard as "solving it," exposing limitations in retrieval and planning that function-level benchmarks completely mask.

## 5. Experimental Analysis

This section dissects the empirical evaluation of `SWE-Perf`, moving beyond the headline numbers to understand *how* and *why* current models succeed or fail at repository-level optimization. The experiments are designed not just to rank models, but to isolate specific bottlenecks: Is the failure due to an inability to generate valid patches? A lack of correctness? Or a fundamental inability to reason about algorithmic efficiency? By decoupling these factors and analyzing performance across different repository characteristics, the paper reveals a nuanced landscape where models show promise in narrow contexts but collapse under the complexity of real-world software engineering.

### 5.1 Evaluation Methodology: Datasets, Baselines, and Setup

To ensure a fair and rigorous comparison, the authors constructed a controlled experimental environment that mirrors the data collection pipeline's strictness.

**Datasets and Task Settings**
The evaluation utilizes the full **140-instance** `SWE-Perf` dataset derived from **9** popular repositories (including `xarray`, `scikit-learn`, and `sympy`). As established in the technical approach, the evaluation is split into two distinct settings to probe different capabilities:
*   **Oracle (File-Level):** The model is given the specific files and target functions identified from the expert patch. This tests the model's raw coding ability to optimize known bottlenecks without the noise of repository navigation.
*   **Realistic (Repo-Level):** The model acts as an autonomous agent, receiving only the repository context and high-level target functions derived from dynamic tracing. It must locate the relevant code, understand dependencies, and apply fixes. This tests planning, retrieval, and full-stack reasoning.

**Baselines and Models**
The study evaluates a diverse set of approaches representing the current state-of-the-art:
1.  **Direct Model Approaches (Oracle):** Ten leading LLMs are tested in a single-pass generation setting using Chain-of-Thought prompting. These include **Claude-3.7-sonnet**, **Claude-4-sonnet**, **Claude-4-opus**, **GPT-4o**, **OpenAI-o1**, **OpenAI-o3**, **DeepSeek-V3**, **DeepSeek-R1**, **Gemini-2.5-Pro**, and **Qwen3-235B-A22B**.
2.  **Pipeline-Based Agents:** **Agentless** (Xia et al., 2024), which uses a fixed multi-stage workflow (localization -> repair -> selection), is evaluated using **Claude-3.7-sonnet** as its backend.
3.  **Agent-Based Systems:** **OpenHands** (Wang et al., 2024), a flexible autonomous agent platform allowing iterative reasoning, is also evaluated with **Claude-3.7-sonnet**.

**Experimental Setup**
All evaluations occur within the provided Dockerized environments to ensure hardware consistency. Crucially, the authors re-evaluate the *original* codebase runtime during the testing phase for every sample, rather than relying on historical data from the collection phase. This eliminates environmental drift as a confounding variable. The metrics strictly follow the three-tiered framework: **Apply** (can the patch be applied?), **Correctness** (do all tests pass?), and **Performance** (what is the statistically significant speedup?).

### 5.2 Main Quantitative Results: The Capability Gap

The core finding of the experiments is a stark divergence between functional correctness and performance optimization. While models are increasingly capable of writing code that *works*, they struggle profoundly to write code that runs *fast*.

**The Performance Chasm**
**Table 2** presents the primary results, highlighting a massive gap between human experts and AI agents.
*   **Expert Baseline:** Human experts achieve an average performance gain of **10.85%** while maintaining **100%** Apply and Correctness rates. This sets the ceiling for what is achievable on these tasks.
*   **Model Performance:** No model comes close to expert-level optimization.
    *   In the **Oracle** setting, the best-performing model is **Claude-4-sonnet**, achieving a mere **1.76%** performance gain. Even **Gemini-2.5-Pro**, which shows strong correctness (**83.57%**), only reaches **1.48%** speedup.
    *   In the **Realistic** setting, **OpenHands** emerges as the top performer with **2.26%** gain, significantly outperforming the pipeline-based **Agentless** (**0.41%**).
    *   Reasoning-heavy models like **OpenAI-o1** and **OpenAI-o3** do not offer a magic bullet; o3 achieves **1.37%** in the Oracle setting, while o1 lags at **0.41%**.

> **Key Observation:** The best agent (OpenHands at 2.26%) achieves less than **21%** of the expert's performance gain (10.85%). This indicates that current LLMs are not just "slightly worse" than experts; they are operating on a fundamentally different plane of optimization capability.

**Correctness vs. Efficiency Decoupling**
A critical insight from **Table 2** is that high correctness does not correlate with high performance.
*   **Gemini-2.5-Pro** achieves the highest correctness in the Oracle setting (**83.57%**) but only modest performance (**1.48%**).
*   **Claude-4-opus** has high correctness (**78.57%**) but low performance (**1.28%**).
*   Conversely, some models with lower correctness manage slightly better relative gains, but the absolute numbers remain negligible.

This decoupling suggests that models are successfully preserving functional behavior (passing tests) but failing to identify or implement the structural changes necessary for speed. They are effectively "treading water"—keeping the code alive but not making it swim faster.

**Repository-Specific Variance**
**Figure 4** breaks down performance by repository, revealing that the difficulty of optimization is highly context-dependent.
*   On **scikit-learn (`sklearn`)**, **OpenHands** actually *surpasses* the expert benchmark, achieving a **0.4%** higher gain than the human patch. This suggests that for certain well-structured, algorithmic libraries, models can find competitive optimizations.
*   However, on **`xarray`**, the gap is catastrophic. The expert achieves a massive **31.8%** gain, while OpenHands manages only **5.1%** and the Oracle model (Claude-3.7) achieves just **2.6%**.
*   **`sympy`** shows a similar trend: Expert **14.5%** vs. OpenHands **2.7%**.

This variance implies that models struggle most with repositories involving complex data flow, heavy I/O, or intricate module interactions (like `xarray`'s rolling windows), whereas they perform relatively better on repositories with more isolated, algorithmic bottlenecks.

### 5.3 Deep Dive: Analyzing Model Capabilities and Failure Modes

To understand the root causes of these failures, the authors conduct a series of targeted analyses that peel back the layers of the aggregate metrics.

#### 5.3.1 Decoupling Performance from Correctness
Does the low performance score stem from models generating broken code that gets filtered out? **Figure 5** and **Figure 6** address this by recalculating performance *only* on the subset of samples where the model's patch was correct (i.e., all tests passed).
*   **Result:** Even when restricting the analysis to correct patches, the performance gap persists. **OpenHands** improves to roughly **3%** higher than its raw score when considering only correct samples, but it still trails the "Correct-Only Expert" baseline significantly.
*   **Implication:** The bottleneck is not primarily *correctness*. Even when models manage to write bug-free code, that code is rarely optimized. The failure is in the *optimization reasoning* itself, not in the basic ability to write valid Python.

#### 5.3.2 The Complexity Ceiling: Target Function Count
How does task complexity affect performance? **Figure 8** plots performance gain against the number of target functions in the Realistic setting.
*   **Trend:** As the number of target functions increases, model performance degrades rapidly. For instances with few targets (1-2 functions), **OpenHands** approaches expert-level performance. However, as the count rises to 8, 16, or 32 functions, the model's gain flattens near zero, while the expert's performance, though also declining, remains substantially higher.
*   **Interpretation:** This indicates a **context window or reasoning bottleneck**. Models can optimize a single isolated function reasonably well. However, when optimization requires coordinating changes across multiple functions or understanding a chain of dependencies, their capability collapses. This confirms that repository-level optimization is not just "more" of the same problem; it introduces a qualitative jump in difficulty related to long-context planning.

#### 5.3.3 The Runtime Paradox
Does the potential for speedup correlate with the original runtime? **Figure 9** examines performance relative to the original execution time of the tests.
*   **Expert Behavior:** Experts show a positive correlation: as the original runtime increases (indicating heavier computation), the percentage gain *increases*, reaching over **20%** for long-running tests. This suggests experts know how to attack expensive operations with algorithmic improvements.
*   **Model Behavior:** Models exhibit a **plateau**. Their performance gain remains flat (around 1-2%) regardless of whether the test takes 0.01 seconds or 1.0 second.
*   **Conclusion:** Models fail to leverage "low-hanging fruit" in computationally intensive tasks. They do not seem to recognize that longer runtimes often imply loops or heavy data processing that are prime candidates for algorithmic refactoring (e.g., vectorization, caching). Instead, they apply superficial fixes that yield constant, negligible returns.

#### 5.3.4 Qualitative Analysis: Low-Level Tweaks vs. High-Level Refactoring
Perhaps the most illuminating finding comes from the keyword analysis of the generated patches (**Figures 10 and 11**). By visualizing the most frequent terms in the code added by models versus experts, a clear strategic divergence emerges.

*   **Expert Strategy (High-Level Abstraction):**
    *   **Keywords:** `dtype`, `label`, `value`, `type`, `literal`.
    *   **Action:** Experts focus on **data integrity and type safety**. They refactor data structures, enforce stricter typing to enable compiler optimizations, and alter the semantic flow of data. For example, changing a generic object array to a specific numeric `dtype` can yield massive speedups in libraries like `numpy` or `xarray`.
    *   **Reasoning:** This reflects a deep understanding of *how* the code interacts with the underlying hardware and libraries.

*   **Model Strategy (Low-Level Infrastructure):**
    *   **Keywords (OpenHands):** `children`, `identifier`, `time`, `attributes`.
    *   **Keywords (Oracle/General):** `miniconda3`, `envs`, `frozen`, `importlib`, `bootstrap`.
    *   **Action:** Models obsess over **environmental configuration and syntactic structure**. The presence of terms like `miniconda3` and `envs` suggests models are often trying to tweak the execution environment or dependency loading rather than the algorithmic logic. They focus on "plumbing" (imports, basic attribute access) rather than "architecture."
    *   **Reasoning:** This indicates that models are acting like junior developers who know how to make code *run* (fixing imports, adjusting paths) but lack the senior-level intuition to restructure data flows for efficiency. They are optimizing the *container*, not the *content*.

### 5.4 Critical Assessment: Do the Experiments Support the Claims?

The experimental design robustly supports the paper's central claim: **There is a substantial capability gap between existing LLMs and expert-level performance optimization.**

**Strengths of the Evidence:**
1.  **Statistical Rigor:** By using the conservative $\delta$ metric and 20-repetition testing, the paper eliminates the possibility that the results are due to noise. The gains (or lack thereof) are real.
2.  **Multi-Dimensional Analysis:** The paper doesn't just report a single accuracy number. By breaking down results by repository (Fig 4), function count (Fig 8), and runtime (Fig 9), it provides a comprehensive map of *where* models fail.
3.  **Qualitative Validation:** The word cloud analysis provides a mechanistic explanation for the quantitative gaps, moving the discussion from "models are bad" to "models are optimizing the wrong things."

**Limitations and Nuances:**
*   **The "Sklearn" Outlier:** The fact that OpenHands beat the expert on `sklearn` suggests the gap is not insurmountable. It implies that for specific domains (perhaps those with cleaner abstractions or more algorithmic focus), models are already competitive. The claim of a "gap" is an average phenomenon, not a universal law.
*   **Oracle vs. Realistic:** The relatively small difference between the best Oracle result (1.76%) and the best Realistic result (2.26%) is counter-intuitive; one might expect the Oracle setting to be much easier. This suggests that even when *given* the location of the bottleneck, models simply do not know *how* to fix it. The primary bottleneck is **optimization knowledge**, not **retrieval**.
*   **Baseline Selection:** The study relies on automated agents (OpenHands, Agentless) for the Realistic setting. It does not explore whether a human-in-the-loop workflow (where a human guides the model to the right file) would close the gap. The results strictly reflect *autonomous* capability.

**Conclusion on Robustness:**
The experiments are highly convincing. The use of a "gold standard" expert patch allows for a direct, apples-to-apples comparison that prior benchmarks lacked. The consistent underperformance across diverse models (from Claude to DeepSeek to OpenAI) and settings (Oracle and Realistic) confirms that this is a systemic limitation of current LLM architectures, not an artifact of a specific model or prompt. The data clearly shows that while we have solved "code generation," we have barely scratched the surface of "code optimization."

## 6. Limitations and Trade-offs

While `SWE-Perf` establishes a rigorous new standard for evaluating code performance, its design involves necessary trade-offs that constrain the scope of its findings. Understanding these limitations is critical for interpreting the results correctly and for guiding future research directions. The authors explicitly acknowledge several constraints in **Appendix A.1**, while others emerge from the methodological choices detailed in **Section 3**.

### 6.1 The "Target Function" Constraint: Artificial Guidance vs. Realism
The most significant trade-off in `SWE-Perf` is the inclusion of **target functions** as part of the input prompt.
*   **The Assumption:** The benchmark assumes that providing models with specific functions to optimize is a necessary scaffold to make the evaluation tractable. As noted in **Section 3.1**, running the full test suite for repositories like `xarray` (averaging **220,000** test cases) would take over **one hour** per sample on a single-core CPU, rendering large-scale evaluation impossible. Furthermore, without guidance, the search space across **170,000+ lines of code** (average per repo, **Table 1**) is too vast for current models to navigate effectively.
*   **The Limitation:** This design choice artificially lowers the barrier to entry. In a true real-world scenario, a developer (or agent) must first *identify* the bottleneck before optimizing it. By providing the target, `SWE-Perf` isolates the *optimization capability* but masks the *diagnostic capability*.
*   **The Trade-off:** The authors mitigate this with the **Realistic** setting, which uses dynamic tracing (`yappi`) to define targets based on executed functions rather than the expert's specific edits. However, even this setting provides more guidance than a completely open-ended task where the agent must decide *which* tests to run or *where* the slowness originates. The paper explicitly encourages future work to omit these targets entirely, acknowledging that the current setup is a stepping stone rather than the final destination for autonomous optimization.

### 6.2 The "Expert Ceiling" Problem: Is Human Performance Optimal?
The benchmark relies on human-authored pull requests as the "gold standard" for performance gains.
*   **The Assumption:** The expert patch represents a high-quality, near-optimal solution against which model performance can be measured.
*   **The Limitation:** As stated in **Appendix A.1**, "human-written patches... may not represent the optimal achievable performance." Human engineers are fallible; a PR merged into a popular repository might improve performance by 10%, but a different approach (perhaps one the model could theoretically discover) might yield 50%.
*   **The Consequence:** This creates a potential **underestimation of the true upper bound**. If a model discovers a novel optimization strategy that surpasses the human expert's speedup, the current evaluation metrics might struggle to contextualize this success relative to the "expert baseline." Conversely, if the expert patch is suboptimal, the "capability gap" reported in **Table 2** (e.g., 10.85% vs. 2.26%) might be slightly distorted, though the fundamental conclusion—that models are far behind—likely remains valid. The benchmark measures the gap to *current human practice*, not to *theoretical optimality*.

### 6.3 Dataset Scope and Generalizability
The construction of `SWE-Perf` involved a rigorous filtering pipeline that reduced **102,241** initial pull requests down to just **140** final instances (**Section 3.3**).
*   **The Constraint:** This aggressive filtering ensures high data quality (stable, statistically significant gains) but severely limits the dataset's size and diversity.
*   **Repository Bias:** The 140 instances are drawn from only **9** popular Python repositories (e.g., `scikit-learn`, `sympy`, `xarray`). While these are influential projects, they represent a specific slice of the software ecosystem: mostly scientific computing and data processing libraries written in Python.
*   **Missing Domains:** The benchmark does not address:
    *   **Systems Programming:** Repositories in C++, Rust, or Go, where memory management and pointer arithmetic offer different optimization levers.
    *   **Concurrency & Parallelism:** Many optimizations in production systems involve multi-threading or distributed computing, which are difficult to measure reliably in the single-core Docker environments used in Phase 2 (**Section 3.2**).
    *   **I/O-Bound Tasks:** The focus on runtime metrics in a controlled container may favor CPU-bound optimizations over those involving disk I/O or network latency, which are notoriously noisy to benchmark.
*   **Generalizability Risk:** A model that performs well on `SWE-Perf` might be overfitted to the patterns found in these specific 9 repositories. It is unclear if these results translate to web backends, embedded systems, or legacy enterprise codebases.

### 6.4 Computational and Scalability Constraints
The methodology imposes heavy computational costs, which limits the frequency and scale of future evaluations.
*   **Resource Intensity:** The data collection pipeline required building **34,397** distinct Docker images and executing millions of unit tests. Phase 2 alone involved running tests on **19,499** codebases, with some taking over an hour each (**Section 3.2**).
*   **Evaluation Bottleneck:** Evaluating a single model on the full 140-instance benchmark is expensive. For the **Realistic** setting, where agents like `OpenHands` may take multiple steps (up to 50 iterations, **Appendix C.1**), the cost multiplies. This high barrier to entry may discourage widespread adoption or rapid iteration by the research community compared to lighter-weight benchmarks like `HumanEval` or `Mercury`.
*   **Hardware Sensitivity:** Despite using Docker containers with fixed resources (1 CPU core, 16 GB RAM in Phase 2; 5 cores in Phase 4), micro-architectural differences in the host machines (the study used machines with **256 logical cores**, **Appendix B**) can still introduce variance. The rigorous statistical filtering (Algorithm 1) mitigates this, but it cannot eliminate it entirely.

### 6.5 The "Correctness" Definition Gap
The benchmark defines correctness strictly as "passing the existing unit tests."
*   **The Edge Case:** This binary definition misses subtle regressions that do not trigger test failures. For example, an optimization might increase memory usage significantly, alter floating-point precision slightly (within test tolerance), or change error message formats, none of which would cause a standard `pytest` assertion to fail.
*   **Functional Pruning Risk:** Although the authors take steps to prevent functional pruning by excluding test code from the target functions (**Section 3.2, Phase 5**), a sufficiently adversarial model might still find ways to "cheat" the specific performance metrics without preserving the broader semantic contract of the function, provided the narrow set of provided tests still passes. The reliance on a limited set of **8.1 related tests** per instance (average, **Table 1**) means the coverage of the function's behavior is incomplete.

### 6.6 Open Questions and Future Directions
The limitations of `SWE-Perf` highlight several open questions for the field:
*   **Can models optimize without hints?** The field needs a follow-up benchmark that removes the "target function" crutch, forcing agents to profile and diagnose bottlenecks autonomously.
*   **How do models handle multi-language repos?** Real-world systems often mix Python with C extensions (common in `scikit-learn` and `xarray`). `SWE-Perf` focuses on the Python layer; optimizing the C backend requires different skills entirely.
*   **What about energy efficiency?** The benchmark measures *time*, but not *energy*. An optimization that speeds up code by 10% but increases CPU power draw by 50% might be undesirable in green computing contexts. `SWE-Perf` does not capture this trade-off.

In summary, `SWE-Perf` is a highly rigorous but narrow slice of the performance optimization problem. It successfully proves that current LLMs cannot match human experts in *guided* optimization tasks within *Python scientific libraries*. Extending these findings to broader contexts requires acknowledging that the benchmark's strict controls (target functions, specific repos, time-only metrics) are both its greatest strength (validity) and its primary limitation (generalizability).

## 7. Implications and Future Directions

The introduction of `SWE-Perf` marks a pivotal transition in the evaluation of Large Language Models (LLMs) for software engineering: the field is moving from assessing **functional correctness** ("Does it work?") to evaluating **operational efficiency** ("How well does it work?"). By exposing a stark capability gap where even state-of-the-art agents achieve less than 21% of the performance gains realized by human experts, this work fundamentally reshapes the research agenda. It signals that current scaling laws and training objectives, which heavily favor syntactic validity and test passing, are insufficient for producing production-grade optimization skills. The implications extend beyond benchmarking, offering a roadmap for next-generation model architectures, training curricula, and deployment strategies.

### 7.1 Reshaping the Research Landscape: From Syntax to Strategy

The primary implication of `SWE-Perf` is the revelation that **optimization is a distinct cognitive capability** from code generation, requiring a shift in how we design and train AI systems.

*   **The End of "Correctness-Only" Evaluation:** Prior to this work, a model that passed all unit tests was considered successful. `SWE-Perf` demonstrates that this metric is dangerously incomplete. A model can be functionally perfect yet computationally wasteful, leading to inflated cloud costs and poor user experience in production. Future benchmarks must adopt **continuous performance metrics** (like the statistically significant gain $\delta$ used here) rather than binary pass/fail states.
*   **The "Junior Developer" Ceiling:** The keyword analysis (Section 5.3.4) reveals that current models behave like junior developers, focusing on low-level syntax, imports, and environment configuration (`miniconda3`, `importlib`). In contrast, experts operate at a higher level of abstraction, refactoring data structures and enforcing type safety (`dtype`, `label`). This suggests that simply increasing model size or context window will not solve the problem; architectures must be enhanced with **strategic reasoning modules** capable of understanding algorithmic complexity and data flow across modules.
*   **The Retrieval vs. Reasoning Bottleneck:** The marginal difference between the **Oracle** (file-level) and **Realistic** (repo-level) settings implies that the primary bottleneck is not finding the code to optimize, but knowing *how* to optimize it once found. Even when given the exact location of the bottleneck, models fail to generate efficient patches. This redirects research focus from improved retrieval-augmented generation (RAG) toward **deep code understanding and refactoring capabilities**.

### 7.2 Catalysts for Follow-Up Research

`SWE-Perf` provides the infrastructure and data necessary to pursue several critical lines of inquiry that were previously impossible due to the lack of ground-truth performance data.

#### A. Training for Efficiency: The "SWE-Perf" Curriculum
The dataset of 140 expert-authored patches paired with original codebases offers a unique opportunity for **Supervised Fine-Tuning (SFT)** and **Reinforcement Learning from Human Feedback (RLHF)** specifically targeted at performance.
*   **Proposed Direction:** Researchers can train models on the `(Original Code, Expert Patch)` pairs to learn the specific patterns of high-impact optimizations (e.g., vectorization, caching, type specialization).
*   **Reward Modeling:** The executable environment allows for **process-supervised reward models**. Instead of rewarding a model only for passing tests, a reward signal can be generated based on the runtime reduction $\delta$. This enables RL algorithms (like PPO) to directly optimize for speed, potentially closing the gap observed in Table 2.

#### B. Autonomous Profiling and Diagnosis
Since `SWE-Perf` currently provides "target functions" to make the task tractable, a natural follow-up is to remove this scaffold.
*   **Proposed Direction:** Develop agents capable of **autonomous profiling**. Such an agent would need to integrate tools like `cProfile` or `py-spy` within the Docker environment to identify hotspots dynamically, formulate a hypothesis about the bottleneck, and then attempt an optimization.
*   **Challenge:** This requires solving the "credit assignment" problem in long-horizon tasks: determining which specific code change led to a measured speedup amidst system noise.

#### C. Cross-Module and Architectural Refactoring
The failure of models on repositories with complex data flows (e.g., `xarray`) highlights a need for **architectural reasoning**.
*   **Proposed Direction:** Future work should explore methods that allow models to reason about **global state and side effects**. This might involve graph-based representations of the codebase where nodes are functions and edges represent data dependencies, allowing the model to simulate the impact of a change on downstream consumers before applying it.

#### D. Beyond Python: Multi-Language and Systems Optimization
`SWE-Perf` is currently Python-centric. The methodologies established here (statistical verification, containerized execution) can be extended to systems programming.
*   **Proposed Direction:** Constructing benchmarks for C++, Rust, or Go where optimization involves memory management, pointer arithmetic, or concurrency control. This would test whether the "low-level vs. high-level" divide holds across different programming paradigms.

### 7.3 Practical Applications and Downstream Use Cases

While current models fall short of expert performance, the framework established by `SWE-Perf` enables immediate practical applications in software development workflows.

*   **Automated Performance Regression Detection:** CI/CD pipelines can integrate `SWE-Perf`-style statistical testing to detect not just functional breakages, but **performance regressions**. By running new commits through the 20-repetition, outlier-filtered protocol, teams can catch subtle slow-downs (e.g., a 5% increase in latency) that traditional testing misses.
*   **AI-Assisted Refactoring Tools:** IDE plugins can use the insights from `SWE-Perf` to offer **targeted optimization suggestions**. Rather than generating full patches autonomously (which currently risks correctness), these tools can highlight specific functions with high runtime potential and suggest expert-like patterns (e.g., "Consider converting this list comprehension to a numpy vectorized operation") based on the keyword patterns identified in the study.
*   **Cost Optimization for Cloud Deployments:** For organizations running large-scale Python services (e.g., data processing pipelines in `pandas` or `xarray`), even a 2% improvement in code efficiency (the current model baseline) can translate to significant cost savings at scale. Deploying `OpenHands`-like agents to scan non-critical paths for easy wins could yield a positive ROI despite the capability gap.

### 7.4 Reproducibility and Integration Guidance

For researchers and practitioners looking to adopt or extend `SWE-Perf`, the following guidance outlines when and how to utilize this benchmark effectively.

#### When to Prefer SWE-Perf
*   **Evaluate Optimization Capabilities:** Use `SWE-Perf` when your goal is to measure a model's ability to improve runtime efficiency, not just fix bugs. It is superior to `EffiBench` or `Mercury` when the task involves **repository-scale context** and **real-world dependencies**.
*   **Test Autonomous Agents:** The **Realistic** setting is the gold standard for evaluating end-to-end agents (like `OpenHands` or `SWE-Agent`) on open-ended engineering tasks where navigation and planning are required.
*   **Benchmark Statistical Rigor:** If your research involves measuring small performance deltas, adopt the **Algorithm 1** verification pipeline (warm-up, 20 repetitions, IQR filtering, Mann-Whitney U test) to ensure your results are statistically significant and not artifacts of system noise.

#### Integration Strategy
1.  **Start with the Oracle Setting:** If you are developing a new model architecture or prompting strategy, begin with the **Oracle (File-Level)** setting. This isolates the model's coding ability from its retrieval capabilities, providing a clearer signal of optimization competence.
2.  **Leverage the Docker Environment:** The provided Docker images are critical for reproducibility. Do not attempt to replicate the environment manually; subtle differences in library versions or OS configurations can invalidate runtime comparisons. Use the provided containers to ensure your baselines match the paper's reported numbers.
3.  **Adopt the Three-Tiered Metric:** When reporting results, always include **Apply**, **Correctness**, and **Performance** scores. Reporting only correctness masks the failure to optimize, while reporting only performance without correctness risks validating broken code.
4.  **Extend the Dataset:** Given the limited scope (9 repos, 140 instances), consider contributing new instances from other domains (e.g., web frameworks, NLP libraries) using the open-source data collection pipeline described in Section 3.2. This will help determine if the observed capability gaps are universal or domain-specific.

#### Caveats for Deployment
*   **Do Not Deploy Blindly:** Current models (even the best performers in Table 2) have low success rates and may introduce subtle bugs or non-optimal changes. Do not use these agents to automatically merge code into production without human review.
*   **Beware of Functional Pruning:** When designing your own evaluation tasks based on `SWE-Perf`, ensure you exclude the test code from the optimization target (as done in Phase 5) to prevent models from "cheating" by removing functionality to gain speed.

In conclusion, `SWE-Perf` does more than measure a gap; it defines the terrain for the next decade of AI software engineering research. By shifting the focus to **statistically verified performance gains** in **real-world repository contexts**, it challenges the community to build models that are not just coders, but true **software engineers** capable of delivering efficient, scalable, and production-ready systems.
