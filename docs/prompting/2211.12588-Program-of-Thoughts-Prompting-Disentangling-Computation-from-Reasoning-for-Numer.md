## 1. Executive Summary
This paper introduces Program of Thoughts (PoT) prompting to address the limitation of Chain-of-Thought (CoT) methods where large language models (LLMs) prone to arithmetic errors perform both reasoning and computation. PoT disentangles these tasks by prompting models like Codex to generate Python programs that delegate calculation to an external interpreter, evaluated across five math word problem datasets (e.g., GSM8K, AQuA) and three financial datasets (e.g., FinQA). This approach yields an average performance gain of approximately 12% over CoT across all datasets, achieving state-of-the-art results such as 80.0% accuracy on GSM8K with self-consistency decoding.

## 2. Context and Motivation

To understand the necessity of Program of Thoughts (PoT), we must first examine the fundamental mismatch between how Large Language Models (LLMs) operate and the requirements of numerical reasoning. This section details the specific limitations of prior methods, the theoretical gap this paper addresses, and how PoT positions itself within the landscape of reasoning strategies.

### The Challenge of Numerical Reasoning in AI
Numerical reasoning is a long-standing benchmark for artificial intelligence, encompassing tasks ranging from elementary math word problems (MWP) to complex financial question answering. The paper evaluates performance across a heterogeneous set of benchmarks summarized in **Table 1**, including:
*   **Math Word Problems:** Datasets like `GSM8K`, `AQuA`, and `SVAMP`, where models must derive numerical answers from text descriptions.
*   **Financial QA:** Datasets like `FinQA` and `TATQA`, which often involve hybrid inputs (tables + text) and require precise arithmetic on large numbers (e.g., millions of dollars).

The core difficulty lies not just in understanding the language of the problem, but in executing the mathematical operations required to solve it. While humans can separate the *logic* of a problem (e.g., "calculate compound interest") from the *computation* (e.g., multiplying specific floats), standard language models attempt to handle both simultaneously within the same generative process.

### Limitations of Language Models as Calculators
Prior to PoT, the state-of-the-art approach was **Chain-of-Thought (CoT)** prompting (Wei et al., 2022). In CoT, the model generates a natural language rationale step-by-step before producing the final answer. While CoT significantly improved reasoning capabilities, the authors argue in the **Introduction (Paragraph 2)** that using LLMs for actual computation is fundamentally flawed for three specific reasons:

1.  **Arithmetic Precision:** LLMs are probabilistic next-token predictors, not deterministic calculators. They are prone to calculation errors, especially with large numbers or high-precision floats. For example, in financial datasets like `FinQA`, CoT struggles with values in the millions due to tokenization and probability distribution limits.
2.  **Complex Expressions:** LLMs cannot natively solve complex mathematical structures like polynomial equations or differential equations using only natural language reasoning.
3.  **Iteration Efficiency:** Expressing iterative processes (loops) in natural language is verbose and error-prone. The paper notes in **Footnote 1** of the Introduction that if an LLM has a 90% accuracy per addition step, performing 50 sequential additions results in a likelihood of correct output of less than 1% ($0.9^{50} \approx 0.005$).

**Figure 1** visually contrasts this limitation. In the provided examples, CoT fails on a cubic equation and an iterative problem because it attempts to compute the values within the text generation. In contrast, PoT delegates these steps to an external interpreter.

### Evolution of Prompting Strategies
To contextualize PoT, we must look at the progression of methods used to solve these tasks:

*   **Fine-tuning:** Early approaches involved training models from scratch or fine-tuning them on datasets with expert-annotated steps (**Introduction, Paragraph 3**). This method is data-intensive, requiring significant labeled examples for every new task.
*   **In-Context Learning:** Later work (Brown et al., 2020) showed that LLMs could solve tasks by observing a few input-output exemplars in the prompt without parameter updates.
*   **Chain-of-Thought (CoT):** This extended in-context learning by including intermediate "thought processes" in the exemplars. However, CoT still relies on the model to generate the computed values as text tokens.
*   **CoT + Calculator:** Some prior work attempted to mitigate CoT errors by using an external calculator to verify equations generated in the text chain. However, the authors note in **Section 3.2** that this approach yields only mild improvements. The limitation is "rigid post-processing," which often fails to correctly extract or calibrate the calculation results from the natural language text, leading to low recall.

### The Specific Gap: Coupling Reasoning and Computation
The critical gap identified by this paper is the **coupling of reasoning and computation in a single medium (natural language)**. When a model writes "The total is 500 + 500 = 1000," it is performing two distinct cognitive tasks:
1.  **Reasoning:** Deciding that addition is the correct operation.
2.  **Computation:** Executing the addition $500 + 500$.

In CoT, both tasks are subject to the same probabilistic errors of the language model. If the model hallucinates the reasoning, the answer is wrong. If the model reasons correctly but miscalculates the token sequence for the sum, the answer is also wrong. PoT posits that by disentangling these tasks, we can leverage the strengths of different systems: the LLM for semantic reasoning and a program interpreter for deterministic computation.

### Positioning PoT Relative to Existing Work
PoT distinguishes itself from related approaches through specific design choices detailed in **Section 2.2**:

*   **Vs. Direct Equation Generation:** One might ask why the model cannot simply generate a single mathematical equation (e.g., `solve(20000*(1+x)^3 ...)`). The authors argue that directly generating such equations is challenging for LLMs. PoT differs by breaking the equation into a **multi-step thought process** packaged as code.
*   **Vs. CoT + Calculator:** Unlike appending a calculator to CoT, PoT generates executable code (Python) where the computation logic is explicit. This avoids the ambiguity of parsing natural language math expressions.
*   **Semantic Binding:** A key theoretical contribution is the concept of **semantic binding**. In PoT, variables are named meaningfully (e.g., `interest_rate` rather than `x`). The paper argues in **Section 2.2** that binding semantic meanings to variables helps ground the model in language, eliciting better reasoning capabilities than abstract equation generation.

By positioning itself at the intersection of symbolic execution and neural reasoning, PoT aims to solve the precision issues of CoT while maintaining the flexibility of natural language understanding. This motivation is validated across the diverse input formats shown in **Table 1**, proving that the disentanglement of computation is beneficial regardless of whether the input is text, tables, or conversation.

## 3. Technical Approach

This is a prompting methodology paper that proposes a system for disentangling reasoning from computation in large language models. The core idea is to have language models generate executable Python code that expresses the reasoning logic, while delegating all numerical computation to an external program interpreter that guarantees deterministic, precise results.

### 3.1 Reader orientation (approachable technical breakdown)

The system being built is a two-stage pipeline where a large language model writes Python programs to solve math problems, and then a separate Python interpreter executes those programs to produce the final numerical answer. This approach solves the problem of language models making arithmetic errors by ensuring that all computation happens in a deterministic environment rather than within the probabilistic text generation of the model itself.

### 3.2 Big-picture architecture (diagram in words)

The PoT system consists of four major components connected in sequence. First, the **Input Preprocessor** takes heterogeneous inputs (text questions, tables, conversation history) and linearizes them into a unified text format suitable for prompting. Second, the **Prompt Constructor** assembles the task instruction, optional few-shot exemplars, and the current question into a complete prompt string. Third, the **Code-Generating Language Model** (primarily Codex) receives this prompt and generates Python code that expresses the reasoning process with semantically meaningful variable names. Fourth, the **Program Interpreter** (Python 3.8 with SymPy library) executes the generated code and returns the computed answer, which may optionally feed into a second-stage CoT prompt for problems requiring additional textual reasoning. Information flows unidirectionally from input through code generation to execution, with the interpreter's output serving as the final answer or as intermediate context for further reasoning.

### 3.3 Roadmap for the deep dive

- We begin with the prompt construction process because understanding what the model receives is prerequisite to understanding what it generates.
- We then examine the code generation mechanism, including the critical design choice of semantic variable binding that distinguishes PoT from simple equation generation.
- We proceed to the execution environment and how the system handles the interface between generated code and the interpreter.
- We cover the few-shot and zero-shot prompting configurations, including the specific hyperparameters for token suppression in zero-shot mode.
- We explain the self-consistency decoding strategy and how it integrates with PoT to reduce generation randomness.
- We conclude with the hybrid PoT+CoT approach for multi-stage reasoning problems that require both symbolic computation and textual understanding.

### 3.4 Detailed, sentence-based technical breakdown

**Paper Type and Core Mechanism**

This is a prompting methodology paper that introduces a neuro-symbolic approach where neural language models handle semantic reasoning while symbolic program interpreters handle deterministic computation. The fundamental mechanism replaces natural language reasoning chains with executable Python programs, allowing the language model to focus exclusively on expressing the logical structure of the solution without being responsible for calculating intermediate values.

**Input Preprocessing and Linearization**

The system must handle heterogeneous input formats across the eight evaluated datasets, including pure text questions, table-text hybrids, and conversational histories. For table inputs, the system adopts a linearization strategy where columns are separated by the pipe character `|` and rows are separated by newline characters `\n`, as described in **Section 3.1**. When a table cell is empty, it is filled with a dash character `-` to maintain structural consistency. For hybrid inputs containing both text and tables, the system separates tables and text with newline characters `\n`. For conversational question answering datasets like ConvFinQA, the system concatenates all dialog history turns into the prompt, with each turn separated by newline characters. The final prompt is constructed by concatenating the task instruction, any linearized table content, and the question itself in that specific order. This linearization ensures that the language model receives structured data in a format it can process as continuous text, which is necessary because the underlying models (Codex, GPT-3) accept text-only inputs rather than structured data objects.

**Prompt Construction for Few-Shot Setting**

Under the few-shot setting, the prompt contains between 4 and 8 exemplar pairs of (question, program) demonstrations prefixed before the current question, as specified in **Section 3.1 Implementation Details**. The exact number of shots varies based on dataset difficulty: simpler datasets like FinQA use fewer shots (closer to 4), while more challenging datasets like AQuA and TATQA use 8 shots to cover more diverse problem types. The exemplars are sampled from the training set of each dataset, and the authors report writing prompts for 10-20 candidate examples before tuning the exemplar selection on a small validation set to choose the best 4-8 shots for full evaluation. Each exemplar demonstrates the expected output format: Python code with semantically meaningful variable names, computation logic, and a final `ans` variable that stores the result. **Figure 3 (Left)** shows the few-shot PoT prompting structure with multiple exemplars followed by the target question. The exemplars teach the language model both the reasoning pattern and the code syntax expected, establishing a clear input-output mapping that the model imitates during inference.

**Prompt Construction for Zero-Shot Setting**

Under the zero-shot setting, the prompt contains only an instruction without any exemplar demonstrations, as shown in **Figure 3 (Right)**. Unlike zero-shot Chain-of-Thought which requires an extra step to extract the answer from the reasoning chain, zero-shot PoT can return the answer straightforwardly because the program execution directly produces the numerical result. However, a critical caveat in zero-shot mode is that the language model can fall back to generating a reasoning chain in comments (using the `#` token) rather than in executable program statements. To suppress this behavior, the system applies a logit bias of `-2` to the `#` token, which decreases its probability and encourages the model to generate actual code instead of comments. The authors report in **Section 3.1** that this simple strategy greatly improves performance in zero-shot settings. The zero-shot prompt includes an instruction that encourages the model to generate reasonable programs without demonstration, and the detailed prompt template is shown in **Figure 3**.

**Code Generation with Semantic Binding**

The language model generates Python code that expresses the reasoning process through a sequence of variable assignments and operations, rather than generating a single monolithic equation. This design choice is critical: the paper argues in **Section 2.2** that directly generating equations like `solve(20000*(1+x)^3 - 2000 - x*20000*3 - 1000, x)` is challenging for language models. PoT differs from equation generation in two key aspects. First, PoT breaks down the equation into a multi-step thought process where each step corresponds to a logical deduction in the reasoning chain. Second, PoT binds semantic meanings to variables to help ground the model in language, using names like `interest_rate` or `total_cost` rather than abstract variables like `x` or `a`. The authors found in **Table 6** that removing semantic binding (using `a, b, c` as variable names) hurts performance, with GSM8K accuracy dropping from 71.6% to 60.2%. Similarly, prompting the model to directly generate target equations without multi-step breakdown reduces GSM8K accuracy to 45.8%. This ablation demonstrates that both semantic binding and multi-step reasoning are essential components of the PoT approach, not merely stylistic choices.

**Variable Naming and Grounding**

Each variable in the generated program carries semantic meaning that corresponds to entities or quantities mentioned in the problem statement. For example, in the GSM8K exemplars shown in the Appendix (**Section 7.2**), variables are named `total_eggs`, `eaten_eggs`, `baked_eggs`, `sold_eggs`, and `dollars_per_egg` rather than generic names like `x1`, `x2`, `x3`. This semantic binding serves a grounding function: it forces the model to maintain correspondence between the natural language problem description and the symbolic representation in code. When the model assigns `total_eggs = 16`, it must first identify that "16 eggs per day" corresponds to the total production, then bind that value to a semantically appropriate variable name. This process requires the model to perform entity recognition and value extraction before computation, which improves the accuracy of value grounding. The error analysis in **Section 3.3** shows that 47% of failures on TAT-QA are value grounding errors (assigning incorrect values to variables), indicating that this remains a challenge even with semantic binding, but the binding mechanism at least makes such errors more detectable and interpretable.

**Program Execution Environment**

The generated Python code is executed in a Python 3.8 environment with the SymPy library available, as specified in **Section 3.1 Implementation Details**. SymPy is a Python library for symbolic mathematics that allows the program to solve equations algebraically rather than numerically, which is essential for problems requiring polynomial equation solving or symbolic manipulation. The execution environment is restricted: the system blocks the language model from importing additional modules beyond the pre-defined set (primarily SymPy and standard math libraries) to prevent dangerous code execution like `import os; os.rmdir()`. This restriction is noted as a limitation in **Section 6**, as it may hurt generalization to unknown symbolic tasks that require other libraries. The interpreter executes the code deterministically, meaning that given the same program, it will always produce the same output, eliminating the probabilistic errors that occur when language models attempt to compute values as text tokens. The execution returns either a numerical value stored in the `ans` variable, or in some cases a dictionary of solutions when solving systems of equations.

**Answer Extraction and Post-Processing**

After program execution, the system extracts the answer from the `ans` variable that the generated code is required to populate. For most datasets (GSM8K, SVAMP, MultiArith), the system uses exact match scores as the evaluation metric, rounding the predicted number to a specific precision before comparing with the reference number. For the FinQA dataset, the evaluation is relaxed because language models cannot perform computation precisely with high-precision floats and large numbers in CoT baselines, so the system adopts `math.isclose` with a relative tolerance of 0.001 to compare answers. For the AQuA dataset, which has multiple-choice options, the system uses PoT to compute the intermediate numerical answer and then prompts the language model again to output the closest option to measure accuracy. For TabMWP, ConvFinQA, and TATQA, the system uses the official evaluation scripts provided on Github for each dataset. This variation in evaluation metrics reflects the heterogeneous nature of the datasets and ensures fair comparison across different answer formats (numbers, options, text).

**Self-Consistency Decoding Integration**

The system integrates self-consistency decoding to understand the upper bound of the method and reduce randomness in the generation procedure. Self-consistency works by sampling multiple completions from the language model and taking a majority vote over the results. Specifically, the system sets a temperature of 0.4 and samples K=40 different completions throughout the experiments, as stated in **Section 3.2**. Each of the 40 samples produces a different program (due to the stochastic sampling at temperature 0.4), and each program is executed independently. The final prediction is determined by taking the majority vote over the 40 executed answers. According to **Table 2**, PoT combined with self-consistency (PoT-SC) outperforms CoT combined with self-consistency (CoT-SC) by an average of 10% across all datasets. On GSM8K specifically, PoT-SC achieves 80.0% accuracy compared to 78.0% for CoT-SC. The self-consistency mechanism is particularly effective for math word problem datasets, while on financial datasets the improvement from self-consistency is less impactful for both PoT and CoT, suggesting that financial reasoning errors are more systematic rather than random.

**PoT as Intermediate Step for Multi-Stage Reasoning**

For certain problems requiring additional textual reasoning beyond computation, the system employs a hybrid approach where PoT serves as an intermediate step that feeds into Chain-of-Thought reasoning. This workflow is depicted in **Figure 8** in the Appendix (**Section 7.1**). The process works as follows: first, the language model generates a PoT program for the question. Second, the program is executed to produce an intermediate result (e.g., a float number like `ans=2.05`). Third, if the output is a dictionary (from solving equations), the system extracts the first key-value pair. Fourth, the system constructs an additional prompt that includes the original question plus the intermediate result (e.g., "according to the program: ans = 2.05"). Fifth, the language model is prompted again to derive the final answer through CoT reasoning using this intermediate result as context. This approach is necessary for datasets like AQuA where the computed value needs additional interpretation (e.g., converting 2.05 hours to minutes to match the multiple-choice time format). The pseudo-code in **Section 7.1** shows this logic: if the answer is a dictionary, extract and format it; otherwise, return the answer directly. The authors note in **Section 2.3** that this prompting strategy is only needed for AQuA, as other datasets can be solved by PoT-only prompting.

**Model Backend Configurations**

The primary model used for experiments is OpenAI Codex (code-davinci-002) with 175 billion parameters, as stated in **Section 3.1**. The system also tested alternative backends for ablation studies, including GPT-3 (text-davinci-002), ChatGPT (gpt-turbo-3.5), CodeGen (codegen-16B-multi and codegen-16B-mono), CodeT5+, and Xgen. **Table 4** shows the performance comparison across these backends on GSM8K and SVAMP. Notably, gpt-3.5-turbo achieves the highest score (76.3% on GSM8K), outperforming Codex (71.6%) by a remarkable margin. In contrast, text-davinci-002 is weaker than code-davinci-002 (60.4% vs 71.6% on GSM8K), which the authors attribute to text-based instruction tuning undermining the model's capabilities to generate code. Open-source models like CodeGen (16B parameters) perform significantly worse (8.2% on GSM8K for codegen-16B-multi), which the authors conjecture is due to insufficient pre-training and smaller model size. This ablation demonstrates that PoT performance is highly dependent on the underlying model's code generation capabilities, with larger code-specialized models performing best.

**Exemplar Sensitivity and Selection**

The system's performance is sensitive to the choice of few-shot exemplars, and the authors conduct a sensitivity analysis to quantify this effect. Specifically, they wrote 20 total exemplars and for k-shot learning randomly sampled k = (2, 4, 6, 8) exemplars three times (labeled v1, v2, v3) from the pool of 20. **Figure 5** shows the results for GSM8K and FinQA across different shot counts and exemplar versions. The analysis reveals two key findings. First, increasing the number of shots helps more for GSM8K than FinQA, which the authors attribute to the greater diversity of questions in GSM8K; adding more exemplars allows the language model to better generalize to diverse question types. Second, when given fewer exemplars, PoT's performance variance is larger: at K=2, the performance variance can be as large as 7% for both datasets. With more exemplars (K=8), the performance becomes more stable. This finding has practical implications for deployment: for datasets with high question diversity, using more shots (6-8) is recommended to reduce variance and improve generalization, while for simpler datasets fewer shots may suffice.

**Comparison with Alternative Computation Strategies**

The paper compares PoT against CoT+calc, which leverages an external calculator to correct calculation results in the generated chain-of-thought text. According to **Table 2**, CoT+calc shows only mild improvement over CoT on math word problem datasets (e.g., 65.4% vs 63.1% on GSM8K for Codex), which is much behind PoT's 71.6%. The authors explain in **Section 3.2** that the main reason for CoT+calc's poor performance is its "rigid post-processing step," which can lead to low recall in terms of calibrating the calculation results. The fundamental issue is that CoT+calc still requires the language model to generate mathematical expressions in natural language text, which must then be parsed and extracted for calculator evaluation. This parsing step is error-prone: if the model generates an expression in an unexpected format, the calculator cannot process it. In contrast, PoT generates executable code directly, eliminating the parsing ambiguity. The code is guaranteed to be syntactically valid Python (or the interpreter raises an error), and the computation logic is explicit rather than embedded in natural language. This architectural difference explains why PoT significantly outperforms CoT+calc despite both approaches using external computation.

**Mathematical Formulation of the Disentanglement**

The core mathematical insight of PoT can be expressed as separating the reasoning function from the computation function. In Chain-of-Thought, the language model $M$ attempts to learn a combined function:

$$M: \text{Question} \rightarrow \text{Reasoning Steps} + \text{Computation} \rightarrow \text{Answer}$$

In Program of Thoughts, the system decomposes this into two separate functions:

$$M: \text{Question} \rightarrow \text{Program } P$$

$$I: P \rightarrow \text{Answer}$$

where $M$ is the language model that generates the program $P$, and $I$ is the program interpreter that executes $P$ to produce the answer. The key advantage is that $I$ is deterministic and exact, while $M$ only needs to learn the mapping from questions to correct program structures. This decomposition reduces the learning burden on $M$ because it no longer needs to learn arithmetic facts or computation rules; it only needs to learn which operations to apply and in what order. The semantic binding mechanism further constrains the learning problem by requiring that variable names in $P$ correspond to entities in the Question, which provides additional supervision signal during the in-context learning process.

**Error Types and Failure Modes**

The system experiences two primary types of errors as identified in the error analysis in **Section 3.3**. The first type is value grounding error, where the model fails to assign correct values to the variables relevant to the question (e.g., extracting the wrong number from the problem text). The second type is logic generation error, where the model fails to generate the correct computation process to answer the question based on the defined variables. Manual examination of 198 failure cases on TAT-QA with the PoT greedy method reveals that 47% have value grounding errors, 33% have logic errors, 15% have both types of errors, and 5% are believed to actually be correct (false negatives in evaluation). The majority of errors being value grounding errors indicates that the computation disentanglement successfully addresses arithmetic errors, but the language understanding component (extracting correct values from text) remains a challenge. This is consistent with the observation that PoT shows larger improvements on datasets requiring complex computation (FinQA with large numbers, AQuA with iterative problems) but similar performance to CoT on simpler arithmetic tasks where computation errors are less likely.

**Design Choice Rationale Summary**

The PoT approach makes several deliberate design choices that distinguish it from alternatives. Using Python rather than a domain-specific language provides access to a rich ecosystem of libraries (SymPy for symbolic math, standard math functions) and ensures that the generated code is human-readable and debuggable. Using semantic variable names rather than abstract symbols grounds the reasoning in the problem context and improves model performance as shown in **Table 6**. Using few-shot exemplars rather than fine-tuning maintains the flexibility of in-context learning while teaching the model the expected output format. Using an external interpreter rather than internal computation eliminates arithmetic errors entirely for the execution phase. Using self-consistency sampling rather than greedy decoding reduces the variance from stochastic generation and improves robustness. Each of these choices addresses a specific limitation of prior approaches, and the ablation studies in **Section 3.3** provide empirical evidence for their importance.

## 4. Key Insights and Innovations

The contributions of Program of Thoughts (PoT) extend beyond simple performance metrics. While the empirical gains are substantial, the paper offers deeper conceptual shifts in how we understand the capabilities and limitations of Large Language Models (LLMs) in reasoning tasks. Below, we dissect the fundamental innovations that distinguish PoT from prior prompting strategies like Chain-of-Thought (CoT).

### 4.1 Fundamental Innovation: Architectural Disentanglement of Reasoning and Computation
The most significant contribution of this work is the architectural decision to **disentangle reasoning from computation**. Prior to PoT, the prevailing assumption in methods like CoT was that a single model could jointly optimize for semantic understanding and arithmetic precision within the same generative stream. PoT challenges this by positing that these are distinct cognitive tasks requiring different tools.

*   **The Shift:** In CoT, the model operates as a unified function $f_{\text{CoT}}(x)$ that attempts to output both logic and numbers. In PoT, the system is split into a reasoning engine (the LLM) and a computation engine (the interpreter). The LLM is no longer responsible for the *value* of the answer, only the *procedure* to derive it.
*   **Why It Matters:** This change fundamentally alters the error profile of the system. As noted in the **Introduction**, LLMs are probabilistic next-token predictors. When asked to compute $12345 \times 67890$, the model must predict tokens that happen to form the correct product, which is statistically unlikely without explicit training. By delegating this to a Python interpreter, the computation error rate drops to effectively zero (barring syntax errors).
*   **Evidence:** This is validated in **Table 2**, where PoT outperforms CoT by roughly 20% on financial datasets like `FinQA` and `ConvFinQA`. These datasets involve large numbers (e.g., millions of dollars) where CoT typically fails due to tokenization limits and arithmetic hallucination. The gain here is not incremental; it represents a capability unlock for tasks previously inaccessible to pure language modeling.

### 4.2 Theoretical Insight: Semantic Binding as a Reasoning Anchor
A subtle but critical innovation in PoT is the emphasis on **semantic binding** within the generated code. One might assume that for a calculator, variable names are irrelevant (i.e., `x = 5` is computationally identical to `price = 5`). However, the paper demonstrates that for the *LLM generating the code*, variable names are crucial reasoning aids.

*   **The Mechanism:** PoT prompts the model to use semantically meaningful variable names (e.g., `interest_rate`, `total_cost`) rather than abstract symbols (e.g., `x`, `a`). This forces the model to maintain a grounding between the natural language problem statement and the symbolic code representation.
*   **Why It Matters:** This finding suggests that LLMs rely on linguistic continuity to maintain reasoning coherence. When a model writes `total_cost = price + tax`, it is explicitly reasoning about the relationship between entities. If it writes `a = b + c`, that semantic link is severed, increasing the likelihood of logic errors.
*   **Evidence:** The ablation study in **Table 6** provides direct proof. When semantic binding is removed (using `a, b, c`), performance on `GSM8K` drops from 71.6% to 60.2%. This 11.4% decrease confirms that semantic binding is not merely a stylistic choice for human readability, but a functional component that stabilizes the model's reasoning process. This distinguishes PoT from standard program synthesis, where variable naming is often optimized for brevity rather than semantic grounding.

### 4.3 Engineering Innovation: Zero-Shot Code Enforcement via Logit Bias
Enabling PoT in a zero-shot setting (without exemplars) presents a unique challenge: LLMs trained on code often default to writing explanations in comments rather than executable logic when not given examples. The paper introduces a specific engineering solution to enforce code generation behavior.

*   **The Technique:** In the zero-shot setting, the authors apply a logit bias of `-2` to the `#` token (the comment character in Python). This artificially suppresses the probability of the model generating comments, forcing it to express its reasoning in executable statements.
*   **Why It Matters:** This is a novel method for controlling the *mode* of generation without fine-tuning. It acknowledges that the model possesses the capability to write code but lacks the incentive to do so in a zero-shot context. By penalizing the "lazy" path (writing comments), the system unlocks the model's coding capabilities without needing few-shot demonstrations.
*   **Evidence:** As described in **Section 3.1**, this strategy "greatly improves performance" in zero-shot settings. **Table 3** shows that Zero-Shot PoT achieves 57.0% on `GSM8K`, significantly outperforming Zero-Shot CoT (40.5%). Without the logit bias suppression, the model would likely revert to natural language rationales, negating the benefits of PoT. This insight is transferable to other domains where specific token types need to be encouraged or discouraged to steer model behavior.

### 4.4 Empirical Insight: Performance Gains are Task-Dependent
PoT does not uniformly improve all reasoning tasks; rather, its utility scales with the complexity of the computation required. This provides a nuanced map of where LLMs struggle versus where they succeed.

*   **The Observation:** The performance gap between PoT and CoT widens as the computational demand increases. For simple arithmetic or tasks requiring mostly semantic understanding, CoT performs competitively. For tasks requiring iteration, complex equations, or high-precision math, PoT dominates.
*   **Why It Matters:** This validates the core hypothesis that LLMs are capable reasoners but unreliable calculators. It suggests that future systems should be adaptive: using CoT for semantic tasks (like commonsense reasoning) and PoT for numerical tasks.
*   **Evidence:** **Figure 6** breaks down performance on the `AQuA` dataset by question type. PoT shows massive gains in categories like `linear/polynomial equation`, `iterative`, and `symbolic` reasoning. However, on `arithmetic` and `probability` questions, the performance between PoT and CoT is similar. This indicates that PoT specifically targets the "computation bottleneck" rather than providing a general boost to all reasoning capabilities. Furthermore, **Section 3.3** notes that on `SVAMP` (a simpler dataset), the improvement is only 4%, compared to 20% on `FinQA`. This differential gain is a key insight for practitioners: PoT is most valuable when the cost of arithmetic error is high.

### 4.5 Distinction from Contemporary Work (PaL and CoT+Calc)
It is important to distinguish PoT from concurrent works that also utilize external tools, specifically **PaL** (Program-aided Language models) and **CoT+Calc**.

*   **Vs. CoT+Calc:** As discussed in **Section 3.2**, CoT+Calc attempts to extract equations from natural language text to verify them with a calculator. PoT outperforms this because it eliminates the extraction step. By generating code directly, PoT avoids the "rigid post-processing" errors where a calculator fails to parse a naturally language expression.
*   **Vs. PaL:** While PaL also uses code, **Table 5** shows PoT generally outperforming PaL (e.g., 6% higher on `SVAMP`). The paper attributes this to the "thoughtful" nature of PoT programs—specifically the multi-step breakdown and semantic binding discussed in **Section 2.2**. PoT is not just about executing code; it is about generating *interpretable* code that mirrors the reasoning chain, which appears to better align with the LLM's training distribution than direct equation generation.

In summary, PoT's innovations are not merely about adding a Python interpreter to a prompt. They represent a rethinking of the LLM's role: from a universal solver to a specialized reasoning engine that orchestrates deterministic tools. The insights regarding semantic binding, token suppression, and task-dependent utility provide a blueprint for future neuro-symbolic systems.

## 5. Experimental Analysis

This section provides a comprehensive examination of the experimental methodology and results that validate the Program of Thoughts (PoT) approach. We will walk through the evaluation design, analyze the quantitative outcomes with specific numbers, and critically assess whether the experiments convincingly support the paper's central claims about disentangling computation from reasoning.

### 5.1 Evaluation Methodology

**Dataset Selection and Diversity**

The authors evaluate PoT across eight distinct datasets spanning two major domains, as summarized in **Table 1**. This diverse selection is intentional: it tests whether PoT generalizes beyond a single task type.

*   **Math Word Problems (5 datasets):**
    *   `GSM8K`: 1,318 test examples, grade-school math problems requiring multi-step reasoning.
    *   `AQuA`: 253 test examples, multiple-choice algebra questions with answer options.
    *   `SVAMP`: 1,000 test examples, structurally varied arithmetic problems designed to test robustness.
    *   `MultiArith`: 600 test examples, multi-step arithmetic reasoning.
    *   `TabMWP`: 7,861 test examples, math problems involving table inputs.

*   **Financial Question Answering (3 datasets):**
    *   `FinQA`: 1,147 test examples, numerical reasoning over financial reports (table + text hybrid).
    *   `ConvFinQA`: 421 test examples, conversational financial QA with multi-turn dialogue.
    *   `TATQA`: 1,668 dev examples, hybrid table-text financial reasoning.

The input formats vary significantly: pure text (GSM8K, SVAMP), tables + text (TabMWP, FinQA, TATQA), and conversation history (ConvFinQA). This heterogeneity tests whether PoT's core mechanism—delegating computation to an interpreter—remains effective regardless of how the problem is presented.

**Evaluation Metrics**

The paper adopts dataset-specific metrics to ensure fair comparison, as detailed in **Section 3.1**:

*   **Exact Match:** Used for `GSM8K`, `SVAMP`, and `MultiArith`. The predicted number is rounded to a specific precision before comparing with the reference.
*   **Multiple-Choice Accuracy:** For `AQuA`, PoT computes the intermediate numerical answer, then the LLM is prompted again to select the closest option from the provided choices.
*   **Official Scripts:** `TabMWP`, `ConvFinQA`, and `TATQA` use the official evaluation scripts provided on Github for each dataset.
*   **Relaxed Matching for FinQA:** Because CoT baselines struggle with high-precision floats and large numbers, the authors adopt `math.isclose` with a relative tolerance of 0.001 for FinQA. This relaxation acknowledges that CoT's computation errors are systematic, not just minor rounding differences.

**Baseline Methods**

The experiments compare PoT against several established approaches, as described in **Section 3.1**:

*   **Direct Answer:** The model outputs only the final answer without intermediate reasoning.
*   **Chain-of-Thought (CoT):** The model generates natural language rationales step-by-step before the answer (Wei et al., 2022).
*   **CoT + Calculator (CoT+calc):** An external calculator verifies equations extracted from the CoT text. This tests whether simply adding computation tools to CoT achieves similar gains.
*   **Self-Consistency Decoding (SC):** Samples 40 completions at temperature 0.4 and takes a majority vote over the executed answers (Wang et al., 2022b). This reduces randomness in generation.
*   **Published State-of-the-Art:** Results from prior work including PaLM (540B parameters), LaMDA (137B parameters), and fine-tuned models.

**Model Backends**

The primary experiments use OpenAI Codex (`code-davinci-002`, 175B parameters). However, **Section 3.3** includes ablation studies with alternative backends:
*   GPT-3 (`text-davinci-002`, 175B)
*   ChatGPT (`gpt-turbo-3.5`)
*   CodeGen (`codegen-16B-multi` and `codegen-16B-mono`, 16B each)
*   CodeT5+ (16B)
*   Xgen (7B)

This range tests whether PoT's benefits depend on model scale or code-specific training.

**Prompt Configuration**

For few-shot settings, the authors use 4-8 exemplars based on dataset difficulty (**Section 3.1**):
*   Simpler datasets (e.g., `FinQA`): ~4 shots
*   Complex datasets (e.g., `AQuA`, `TATQA`): 8 shots

Exemplars are sampled from training sets, with 10-20 candidate examples written and tuned on a validation set to select the best 4-8. For zero-shot settings, only an instruction is provided without exemplars, with a logit bias of `-2` applied to the `#` token to suppress comment generation.

### 5.2 Main Quantitative Results

**Few-Shot Performance (Greedy Decoding)**

**Table 2** presents the core few-shot results comparing PoT against CoT and other baselines. The numbers reveal a consistent pattern: PoT outperforms CoT across all datasets, with larger gains on computationally intensive tasks.

| Dataset | Codex CoT | PoT-Codex | Gain |
|---------|-----------|-----------|------|
| GSM8K | 63.1% | 71.6% | +8.5% |
| AQuA | 45.3% | 54.1% | +8.8% |
| SVAMP | 76.4% | 85.2% | +8.8% |
| TabMWP | 65.2% | 73.2% | +8.0% |
| FinQA | 40.4% | 64.5% | +24.1% |
| ConvFinQA | 45.6% | 64.6% | +19.0% |
| TATQA | 61.4% | 69.0% | +7.6% |
| **Average** | **56.7%** | **68.9%** | **+12.2%** |

Several observations stand out:

1.  **Financial Datasets Show Largest Gains:** `FinQA` and `ConvFinQA` exhibit improvements of 24.1% and 19.0% respectively. The authors explain in **Section 3.2** that this is due to CoT's miscalculations on large numbers (e.g., values in the millions). PoT's external interpreter handles these precisely, while CoT's token-based computation fails.

2.  **Math Datasets Show Consistent Improvement:** All five MWP datasets show gains between 7.6% and 8.8%. This consistency suggests PoT's benefits are not dataset-specific but generalize across numerical reasoning tasks.

3.  **CoT+Calculator Underperforms:** The `Codex CoT+calc` baseline achieves only 65.4% on GSM8K, compared to PoT's 71.6%. The authors attribute this to "rigid post-processing" that fails to extract equations from natural language text reliably (**Section 3.2**).

4.  **PoT Approaches Published SoTA:** The average PoT score of 68.9% is close to the published SoTA average of 70.7% (which includes fine-tuned models and GPT-4 results excluded). On GSM8K specifically, PoT's 71.6% is competitive with fine-tuned approaches.

**Few-Shot + Self-Consistency Performance**

**Table 2** also reports results with self-consistency decoding (K=40 samples, temperature 0.4). This tests the upper bound of each method when generation randomness is reduced.

| Dataset | Codex CoT-SC | PoT-SC-Codex | Gain |
|---------|--------------|--------------|------|
| GSM8K | 78.0% | 80.0% | +2.0% |
| AQuA | 52.0% | 58.6% | +6.6% |
| SVAMP | 86.8% | 89.1% | +2.3% |
| TabMWP | 75.4% | 81.8% | +6.4% |
| FinQA | 44.4% | 68.1% | +23.7% |
| ConvFinQA | 47.9% | 67.3% | +19.4% |
| TATQA | 63.2% | 70.2% | +7.0% |
| **Average** | **63.9%** | **73.6%** | **+9.7%** |

Key insights:

*   **Self-Consistency Helps Both Methods:** CoT-SC improves from 56.7% to 63.9% (average), while PoT-SC improves from 68.9% to 73.6%. This confirms that sampling multiple completions reduces variance for both approaches.

*   **PoT-SC Maintains Lead:** Despite both methods benefiting from SC, PoT-SC still outperforms CoT-SC by 9.7% on average. This indicates PoT's advantage is not merely due to reduced randomness but stems from the fundamental architecture.

*   **Financial Datasets Still Dominate:** The gap on `FinQA` (23.7%) and `ConvFinQA` (19.4%) remains large even with self-consistency, reinforcing that computation precision is the bottleneck for these tasks.

*   **GSM8K Gap Narrows:** On GSM8K, the gap shrinks from 8.5% (greedy) to 2.0% (SC). This suggests that for simpler arithmetic, CoT with sufficient sampling can approach PoT's performance, but PoT still maintains an edge.

**Zero-Shot Performance**

**Table 3** evaluates zero-shot settings where no exemplars are provided. This tests PoT's ability to generalize without task-specific demonstrations.

| Dataset | Zero-shot CoT (GPT-3) | Zero-shot PoT (Codex) | Gain |
|---------|----------------------|----------------------|------|
| GSM8K | 40.5% | 57.0% | +16.5% |
| AQuA | 31.9% | 43.9% | +12.0% |
| SVAMP | 63.7% | 70.8% | +7.1% |
| TabMWP | 53.5% | 66.5% | +13.0% |
| MultiArith | 79.3% | 92.2% | +12.9% |
| **Average** | **53.7%** | **66.1%** | **+12.4%** |

Notable findings:

*   **Larger Zero-Shot Gap:** The average zero-shot gain (12.4%) exceeds the few-shot greedy gain (12.2%), suggesting PoT generalizes better without exemplars. On `GSM8K`, zero-shot PoT (57.0%) even exceeds few-shot CoT (63.1%) by a smaller margin, but the relative improvement over zero-shot CoT (40.5%) is substantial.

*   **MultiArith Excellence:** Zero-shot PoT achieves 92.2% on `MultiArith`, demonstrating that for structured arithmetic tasks, PoT requires no task-specific tuning.

*   **Codex vs. GPT-3:** The zero-shot CoT baseline uses GPT-3, while PoT uses Codex. However, **Table 4** shows that even when comparing Codex CoT (63.1% few-shot) to PoT-Codex (71.6%), the gap persists, indicating the improvement is not solely due to model differences.

Critical observations:

*   **Code-Specialized Models Excel:** `gpt-3.5-turbo` achieves the highest scores (76.3% GSM8K), outperforming Codex by 4.7%. This suggests newer models with better code capabilities further amplify PoT's benefits.

*   **Text-Tuned Models Underperform:** `text-davinci-002` scores 60.4% on GSM8K, 11.2% below Codex. The authors attribute this to text-based instruction tuning undermining code generation capabilities (**Section 3.3**).

*   **Open-Source Models Struggle:** All 16B and 7B open-source models score below 13% on GSM8K. The authors conjecture this is due to insufficient pre-training and smaller model size. This is a significant limitation: PoT's benefits may not transfer to smaller, open-source models without substantial architectural improvements.

**Exemplar Sensitivity Analysis**

**Figure 5** examines how performance varies with the number of shots (k=2, 4, 6, 8) and exemplar selection (three random samples v1, v2, v3 from a pool of 20).

*   **GSM8K Benefits More from Additional Shots:** Performance increases steadily from 2 shots (~62%) to 8 shots (~73%). The diversity of GSM8K questions means more exemplars help the model generalize.

*   **FinQA Stabilizes Earlier:** FinQA shows less improvement beyond 4 shots, suggesting its problem types are more uniform.

*   **Variance Decreases with More Shots:** At k=2, performance variance across v1/v2/v3 can reach 7%. At k=8, variance is minimal. This has practical implications: for high-diversity datasets, using 6-8 shots is recommended to ensure stability.

|---------|-----|-----|

| ASDIV | 79.6% | 85.2% |
| ADDSUB | 92.5% | 92.2% |
| MultiArith | 99.2% | 99.5% |

PoT outperforms PaL on most datasets, particularly `SVAMP` (+5.8%) and `ASDIV` (+5.6%). The authors attribute this to PoT's "thoughtful" multi-step breakdown and semantic binding, which better align with the LLM's reasoning capabilities than PaL's more direct equation generation approach (**Section 3.3**).

### 5.3 Ablation Studies: Validating Core Design Choices

The paper conducts targeted ablations to verify that PoT's specific design choices—not just the use of code—drive the improvements.

**Semantic Binding Ablation (Table 6)**

This study tests whether semantically meaningful variable names (e.g., `interest_rate`) are necessary, or if abstract names (e.g., `a`, `b`, `c`) suffice:

| Dataset | PoT (Full) | PoT - Binding | PoT - MultiStep |
|---------|------------|---------------|-----------------|
| GSM8K | 71.6% | 60.2% | 45.8% |
| SVAMP | 85.2% | 83.8% | 81.9% |
| FinQA | 64.5% | 61.6% | 58.9% |

*   **Removing Semantic Binding:** GSM8K drops 11.4% (71.6% → 60.2%). This confirms that variable naming is not cosmetic; it grounds the model's reasoning in the problem context.

*   **Removing Multi-Step Breakdown:** GSM8K drops 25.8% (71.6% → 45.8%). Direct equation generation is significantly harder for LLMs than step-by-step code. This validates the core PoT hypothesis: breaking reasoning into executable steps is essential.

**Breakdown by Question Type (Figure 6)**

The authors manually classify `AQuA` questions into categories to identify where PoT helps most:

| Category | CoT | PoT | Gain |
|----------|-----|-----|------|
| Linear/Polynomial Equation | 20% | 56% | +36% |
| Iterative | 40% | 62% | +22% |
| Symbolic | 50% | 66% | +16% |
| Combinatorics | 50% | 86% | +36% |
| Arithmetic | 86% | 88% | +2% |
| Probability | 38% | 40% | +2% |
| Geometry | 10% | 20% | +10% |

*   **Largest Gains:** Categories requiring complex computation (`linear/polynomial`, `iterative`, `combinatorics`) show gains of 16-36%. This directly supports the claim that PoT targets computation bottlenecks.

*   **Minimal Gains:** Simple `arithmetic` and `probability` questions show only 2% improvement. CoT already handles these reasonably well, so PoT's computation delegation provides less marginal benefit.

*   **Implication:** PoT is not a universal improvement; it is most valuable when computation complexity is high. Practitioners should consider task characteristics when choosing between CoT and PoT.

### 5.4 Error Analysis and Failure Modes

**Section 3.3** provides a detailed error analysis on `TAT-QA`, examining 198 failure cases from the PoT greedy method:

| Error Type | Percentage |
|------------|------------|
| Value Grounding Error | 47% |
| Logic Generation Error | 33% |
| Both Errors | 15% |
| Actually Correct (False Negative) | 5% |

*   **Value Grounding Dominates:** Nearly half of failures (47%) occur when the model extracts incorrect values from the problem text (e.g., assigning the wrong number to a variable). This is significant: PoT successfully eliminates *computation* errors, but *language understanding* errors remain.

*   **Logic Errors Persist:** 33% of failures involve incorrect computation logic (e.g., using addition instead of multiplication). This indicates that while PoT improves reasoning structure, it does not eliminate reasoning errors entirely.

*   **Comparison to CoT:** The authors note that value grounding errors are "also common for other methods such as CoT." This suggests that PoT shifts the error profile from computation mistakes to language understanding mistakes—a favorable trade-off since computation errors are harder to debug and more systematic.

**Figure 7** illustrates two error cases:
1.  **Value Grounding:** The model fetches variable values incorrectly but computes correctly.
2.  **Logic Generation:** The model grounds variables correctly but generates flawed computation logic.

This analysis is crucial for understanding PoT's limitations: it solves the computation problem but not the full reasoning problem.

### 5.5 Critical Assessment: Do Experiments Support the Claims?

**Claim 1: PoT Outperforms CoT on Numerical Reasoning**

*   **Supported:** The average 12.2% gain across eight datasets (**Table 2**) is substantial and consistent. Financial datasets show even larger gains (19-24%), validating that computation precision matters most for large numbers.

*   **Caveat:** The gap narrows with self-consistency decoding (9.7% average gain), suggesting that CoT can partially close the gap with sufficient sampling. However, PoT-SC still maintains a lead, indicating the architectural advantage persists.

**Claim 2: PoT Generalizes Across Input Formats**

*   **Supported:** PoT performs well on text-only (GSM8K), table+text (FinQA, TATQA), and conversational (ConvFinQA) inputs. The linearization strategy in **Section 3.1** successfully handles heterogeneous formats.

*   **Caveat:** Performance varies by dataset complexity. `SVAMP` shows only 4% improvement in few-shot greedy setting, while `FinQA` shows 24%. The benefit is task-dependent, not uniform.

**Claim 3: Semantic Binding and Multi-Step Breakdown Are Essential**

*   **Strongly Supported:** **Table 6** shows dramatic drops when either component is removed (11.4% and 25.8% on GSM8K respectively). This is among the strongest evidence in the paper, as it isolates specific design choices rather than comparing entire systems.

**Claim 4: PoT Works in Zero-Shot Settings**

*   **Supported:** Zero-shot PoT achieves 66.1% average (**Table 3**), outperforming zero-shot CoT by 12.4%. The logit bias trick for suppressing comments is a practical contribution that enables zero-shot code generation.

*   **Caveat:** Zero-shot PoT still lags behind few-shot PoT (66.1% vs 68.9%), indicating that exemplars provide meaningful guidance. Zero-shot is viable but not optimal.

**Claim 5: PoT Is Model-Dependent**

*   **Supported:** **Table 4** shows open-source 16B models scoring below 13% on GSM8K, while Codex scores 71.6%. This is a significant limitation: PoT's benefits may not transfer to smaller or less code-capable models.

*   **Implication:** Practitioners using open-source models may not realize PoT's full benefits without substantial model improvements. This is an important constraint for reproducibility and deployment.

**Claim 6: PoT Shifts Error Profile from Computation to Grounding**

*   **Supported:** The error analysis in **Section 3.3** shows 47% value grounding errors vs. 33% logic errors. Since PoT eliminates computation errors by design, the remaining failures are primarily language understanding issues.

*   **Implication:** Future improvements should focus on better value extraction and entity grounding, not computation accuracy.

### 5.6 Limitations and Trade-offs

**Computational Cost**

PoT requires executing generated code for each sample. With self-consistency decoding (K=40), this means 40 program executions per question. While the interpreter is fast, this adds latency compared to direct text generation. The paper does not report exact inference times, which is a gap in the experimental reporting.

**Security Concerns**

The authors acknowledge in **Section 6 (Limitations)** that executing LLM-generated code poses security risks (e.g., `import os; os.rmdir()`). They mitigate this by blocking additional module imports, but note this "might hurt the PoT's generalization" to tasks requiring other libraries. This is a practical constraint for deployment in production environments.

**Open-Source Model Gap**

The poor performance of open-source models (**Table 4**) limits PoT's accessibility. Researchers without access to Codex or GPT-3.5 may not achieve comparable results. This raises questions about the reproducibility of PoT's benefits in resource-constrained settings.

**Task Specificity**

PoT excels on numerical reasoning but may not generalize to semantic tasks. The authors note in **Section 5 (Discussion)** that for commonsense reasoning (e.g., StrategyQA), CoT may be more appropriate. PoT is specialized, not universal.

**AQuA Performance Ceiling**

Despite improvements, PoT achieves only 58.6% on `AQuA` with self-consistency (**Table 2**). The authors attribute this to the dataset's diversity, which few-shot exemplars cannot fully cover. This suggests that for highly varied algebraic problems, additional strategies (e.g., dynamic exemplar selection) may be needed.

### 5.7 Summary of Experimental Evidence

The experiments provide strong, multi-faceted support for PoT's core claims:

| Claim | Evidence Strength | Key Supporting Data |
|-------|-------------------|---------------------|
| PoT > CoT on numerical tasks | Strong | Table 2: +12.2% average gain |
| Largest gains on complex computation | Strong | Table 2: FinQA +24.1%, ConvFinQA +19.0% |
| Semantic binding is essential | Very Strong | Table 6: -11.4% without binding |
| Multi-step breakdown is essential | Very Strong | Table 6: -25.8% without multi-step |
| Zero-shot PoT is viable | Strong | Table 3: 66.1% average, +12.4% over CoT |
| Model capability matters | Strong | Table 4: Open-source models &lt;13% |
| Error profile shifts to grounding | Strong | Section 3.3: 47% grounding errors |

The experimental design is thorough: multiple datasets, multiple decoding strategies, ablation studies, and error analysis. The main gaps are inference time reporting and open-source model performance, which limit practical deployment guidance. Nevertheless, the quantitative evidence convincingly demonstrates that disentangling computation from reasoning via program generation yields substantial, consistent improvements on numerical reasoning tasks.

## 6. Limitations and Trade-offs

While Program of Thoughts (PoT) demonstrates substantial improvements over Chain-of-Thought (CoT) on numerical reasoning tasks, the approach is not without significant constraints. This section critically examines the assumptions underlying PoT, the scenarios where it struggles, the computational and security trade-offs involved, and the open questions that remain for future work.

### 6.1 Core Assumptions and Their Fragility

PoT relies on several foundational assumptions that may not hold universally across all reasoning tasks or model architectures.

**Assumption 1: The LLM Can Generate Syntactically Valid Code**

PoT assumes the language model will produce executable Python code. However, **Table 4** reveals this assumption is highly model-dependent. Open-source models like `codegen-16B-multi` achieve only 8.2% on GSM8K, compared to 71.6% for Codex. This 63.4% gap indicates that PoT's benefits are contingent on access to large, code-specialized models. The paper acknowledges in **Section 3.3** that "open source model like codegen is significantly behind across different benchmarks," conjecturing this is due to "non-sufficient pre-training and model size."

*   **Trade-off:** PoT gains computation accuracy but loses accessibility. Researchers without access to Codex or GPT-3.5 may not realize comparable benefits.
*   **Implication:** The approach is not easily reproducible in resource-constrained settings, limiting its democratization.

**Assumption 2: Computation Is the Primary Bottleneck**

PoT is designed to address arithmetic errors, assuming computation is the main failure mode. However, the error analysis in **Section 3.3** reveals a different reality:

> "Among the 198 failure cases of numerical reasoning questions with the PoT (greedy) method, 47% have value grounding errors and 33% have logic errors."

This means PoT successfully eliminates computation errors, but **47% of remaining failures are value grounding errors** (extracting wrong numbers from text). PoT shifts the error profile rather than eliminating errors entirely.

*   **Trade-off:** PoT trades computation errors for language understanding errors. While computation errors are deterministic and easier to debug, grounding errors remain as challenging as in CoT.
*   **Implication:** Future improvements should focus on entity extraction and value grounding, not computation accuracy.

**Assumption 3: Python Is Sufficient for All Numerical Reasoning**

The system restricts the model to Python 3.8 with SymPy and standard math libraries. **Section 6 (Limitations)** explicitly notes:

> "We have blocked the LLM from importing any additional modules and restrict it to using the pre-defined modules. Such brutal-force blocking works reasonable for math QA, however, for other unknown symbolic tasks, it might hurt the PoT's generalization."

*   **Trade-off:** Security is prioritized over flexibility. The system prevents dangerous code execution but may fail on tasks requiring specialized libraries (e.g., numerical integration, statistical modeling).
*   **Implication:** PoT's applicability is bounded by the pre-approved module set, limiting extension to domains like scientific computing or advanced finance.

### 6.2 Scenarios and Edge Cases Not Addressed

The paper identifies several problem settings where PoT provides limited or no benefit.

**Semantic Reasoning Tasks**

PoT is specialized for numerical reasoning. In **Section 5 (Discussion)**, the authors state:

> "For semantic reasoning tasks like commonsense reasoning (StrategyQA), we conjecture that PoT is not the best option. In contrast, CoT can solve more broader reasoning tasks."

*   **Limitation:** PoT does not generalize to non-numerical reasoning domains. Tasks requiring world knowledge, causal inference, or qualitative judgment remain better suited for CoT.
*   **Evidence:** The paper evaluates only math and financial datasets. No experiments on commonsense or logical reasoning benchmarks are provided.

**Highly Diverse Algebraic Problems (AQuA)**

Despite PoT's improvements, performance on `AQuA` remains modest. **Table 2** shows PoT-SC achieves 58.6% on AQuA, compared to 80.0% on GSM8K. The authors explain in **Section 6 (Limitations)**:

> "Another limitation is that PoT still struggles with AQuA dataset with complex algebraic questions with only 58% accuracy. It's mainly due to the diversity questions in AQuA, which the demonstration cannot possibly cover."

*   **Limitation:** Few-shot exemplars cannot cover the full diversity of algebraic problem types in AQuA. The model fails to generalize to unseen equation structures.
*   **Implication:** For highly varied symbolic tasks, static few-shot prompting is insufficient. Dynamic exemplar selection or fine-tuning may be required.

**Iterative Problems with Large Step Counts**

While PoT handles iteration better than CoT (as shown in **Figure 1**), the paper does not evaluate extreme cases. **Footnote 1** in the Introduction notes that 50 sequential additions reduce CoT accuracy to less than 1%, but PoT's performance on iterations beyond 50 steps is not tested.

*   **Limitation:** The upper bound of PoT's iteration handling is unknown. Very long loops may exceed interpreter timeouts or memory limits.
*   **Open Question:** How does PoT scale to problems requiring thousands of iteration steps?

**Multi-Modal Inputs Beyond Tables**

The evaluated datasets include text, tables, and conversation history (**Table 1**). However, PoT is not tested on images, diagrams, or geometric figures, which are common in math problems.

*   **Limitation:** PoT's linearization strategy works for tabular data but may not extend to visual inputs without additional preprocessing.
*   **Implication:** Integration with vision-language models would be required for diagram-based reasoning.

### 6.3 Computational and Scalability Constraints

PoT introduces computational overhead that the paper does not fully quantify.

**Inference Latency**

With self-consistency decoding, the system samples K=40 completions and executes each program independently (**Section 3.2**). This means:

*   40 LLM forward passes per question
*   40 Python interpreter executions per question

The paper does not report exact inference times, which is a notable gap. However, we can estimate the overhead:

| Component | Estimated Cost |
|-----------|---------------|
| LLM Generation (40 samples) | ~40× single-pass latency |
| Program Execution (40 runs) | Additional interpreter overhead |
| Total | Significantly higher than greedy decoding |

*   **Trade-off:** Accuracy is improved at the cost of latency. For real-time applications, this overhead may be prohibitive.
*   **Open Question:** What is the exact inference time per question, and how does it compare to CoT-SC?

**Exemplar Selection Sensitivity**

**Figure 5** shows that performance variance can reach 7% when using only 2 shots. At K=8, performance stabilizes, but this requires:

*   Writing 10-20 candidate exemplars per dataset (**Section 3.1**)
*   Tuning exemplar selection on a validation set
*   Maintaining dataset-specific exemplar pools

*   **Trade-off:** Better performance requires more manual effort in exemplar curation. This reduces the "plug-and-play" appeal of prompting methods.
*   **Implication:** For new datasets, significant prompt engineering is required before PoT reaches optimal performance.

**Memory and Token Limits**

Few-shot PoT with 8 exemplars consumes substantial prompt tokens. For long questions (e.g., financial reports in FinQA), this may approach model context limits.

*   **Limitation:** The paper does not discuss token budget management. As models handle longer contexts, exemplar count may need adjustment.
*   **Open Question:** What is the optimal balance between exemplar count and question length within fixed context windows?

### 6.4 Security and Deployment Concerns

Executing LLM-generated code introduces risks that the paper acknowledges but does not fully resolve.

**Code Injection Risks**

**Section 6 (Limitations)** states:

> "PoT would require execution of 'generated code' from LLMs, which could contain certain dangerous or risky code snippets like 'import os; os.rmdir()', etc."

The mitigation is "brutal-force blocking" of additional module imports. However:

*   **Limitation:** Blocking imports may not prevent all malicious patterns (e.g., infinite loops, memory exhaustion).
*   **Trade-off:** Stricter sandboxing improves security but may break legitimate code that requires restricted operations.
*   **Implication:** Production deployment requires robust sandboxing infrastructure, not just import blocking.

**Determinism vs. Flexibility**

By restricting the interpreter to pre-defined modules, PoT ensures safety but limits flexibility. Tasks requiring external APIs, database queries, or file I/O cannot be handled.

*   **Limitation:** PoT is confined to in-memory computation. Real-world financial or scientific tasks often require external data sources.
*   **Open Question:** How can PoT be extended to safely interact with external tools without compromising security?

### 6.5 Open Questions and Future Directions

The paper identifies several areas where PoT's capabilities remain unexplored or incomplete.

**Adaptive Method Selection**

Since PoT excels on computation-heavy tasks but CoT may be better for semantic reasoning, an open question is:

*   **When should a system use PoT vs. CoT?** The paper does not propose a mechanism for automatic method selection based on problem characteristics.
*   **Future Direction:** A classifier could predict whether a question requires heavy computation (favoring PoT) or semantic understanding (favoring CoT).

**Integration with Tool-Use Frameworks**

**Section 4.4** mentions concurrent work on tool-use (Schick et al., 2023; Paranjape et al., 2023) that generalizes Python execution to broader API calls.

*   **Open Question:** How does PoT integrate with multi-tool systems that include search engines, calculators, and databases?
*   **Future Direction:** PoT could become one tool among many in a broader reasoning framework.

**Fine-Tuning vs. Prompting**

All experiments use in-context learning without fine-tuning. However, **Section 3.3** shows open-source models struggle with PoT prompting.

*   **Open Question:** Would fine-tuning smaller models on PoT-style examples close the gap with Codex?
*   **Future Direction:** Fine-tuning code-specialized models on PoT demonstrations could improve accessibility.

**Error Correction and Self-Refinement**

**Section 4.4** mentions follow-up work on self-critic (Gou et al., 2023) and self-eval (Xie et al., 2023) that enhance PoT with self-evaluation.

*   **Open Question:** Can PoT programs be automatically debugged when execution fails?
*   **Future Direction:** Integrating error feedback loops could improve robustness when generated code raises exceptions.

However, practitioners must weigh these benefits against the costs of model access, computational overhead, security infrastructure, and task specificity. Future work should address the open questions around adaptive method selection, fine-tuning for accessibility, and integration with broader tool-use frameworks.

## 7. Implications and Future Directions

The Program of Thoughts (PoT) framework represents more than an incremental improvement in numerical reasoning accuracy. It establishes a new architectural paradigm for how we conceptualize the role of Large Language Models (LLMs) in reasoning systems. This section examines how PoT reshapes the research landscape, what avenues it opens for future investigation, and how practitioners should decide when to deploy this approach.

### 7.1 Shifting the Paradigm: LLMs as Reasoning Orchestrators

PoT fundamentally repositions the LLM from a **universal solver** to a **reasoning orchestrator**. This conceptual shift has ripple effects across multiple research domains.

**The Decoupling Principle**

Prior to PoT, the dominant assumption was that reasoning and computation should be handled within a single generative process. Chain-of-Thought (CoT) exemplified this: the model generates both the logical steps and the computed values as text tokens. PoT challenges this by demonstrating that **disentangling these tasks yields superior performance**.

The evidence is clear in **Table 2**: PoT outperforms CoT by an average of 12.2% across eight datasets. More importantly, the gain is not uniform—it scales with computational complexity. On `FinQA`, where large numbers (millions of dollars) are common, the improvement reaches 24.1%. This differential gain validates the core hypothesis: LLMs are capable reasoners but unreliable calculators.

**Implication for Model Design**

This finding suggests that future model architectures should not attempt to optimize for all reasoning capabilities uniformly. Instead, models could be specialized:

*   **Reasoning-specialized models:** Optimized for generating correct program structures and semantic bindings.
*   **Computation-specialized tools:** Deterministic interpreters (Python, SymPy) that guarantee arithmetic precision.

The ablation in **Table 4** supports this: `gpt-3.5-turbo` (76.3% on GSM8K) outperforms `text-davinci-002` (60.4%) because it has better code generation capabilities, not necessarily better arithmetic understanding. This suggests that **code-specific training** is more valuable than general reasoning training for numerical tasks.

**Broader Applicability Beyond Math**

The paper explicitly states in **Section 6 (Conclusions)**:

> "We believe our work can inspire more work to combine symbolic execution with LLMs to achieve better performance on other symbolic reasoning tasks."

The disentanglement principle extends beyond arithmetic. Any task where:
1.  Reasoning can be expressed as executable logic
2.  Execution can be delegated to a deterministic tool

...is a candidate for PoT-style architectures. This includes:
*   **Logical reasoning:** Generating SAT solver inputs instead of natural language deductions.
*   **Database queries:** Generating SQL instead of text descriptions of data retrieval.
*   **API orchestration:** Generating function calls instead of describing actions.

The concurrent work mentioned in **Section 4.4** (Schick et al., 2023; Paranjape et al., 2023) already generalizes PoT's Python execution to broader API calls, including search engines and string extraction tools. PoT provides the theoretical foundation for this generalization.

### 7.2 Follow-Up Research Directions Enabled by PoT

The paper identifies several concrete avenues for future investigation, both explicitly stated and implicitly suggested by the results.

**Direction 1: Improving Value Grounding**

The error analysis in **Section 3.3** reveals that 47% of PoT failures are **value grounding errors** (assigning incorrect values to variables), while only 33% are logic generation errors. Since PoT eliminates computation errors by design, **value grounding becomes the new bottleneck**.

*   **Research Question:** How can we improve the model's ability to extract correct values from problem text?
*   **Potential Approaches:**
    *   Attention visualization to identify which text spans the model grounds to each variable.
    *   Intermediate supervision signals that verify value extraction before code generation.
    *   Multi-stage prompting where value extraction is separated from program generation.

This direction is explicitly mentioned in **Section 6 (Limitations)**:

> "Future research should discuss how to further prompt LLMs to generate code for highly diversified Math questions."

**Direction 2: Adaptive Method Selection**

**Section 5 (Discussion)** acknowledges that PoT is not universally superior:

> "For semantic reasoning tasks like commonsense reasoning (StrategyQA), we conjecture that PoT is not the best option. In contrast, CoT can solve more broader reasoning tasks."

This raises a critical question: **When should a system use PoT vs. CoT?**

*   **Research Question:** Can we build a classifier that predicts whether a question requires heavy computation (favoring PoT) or semantic understanding (favoring CoT)?
*   **Potential Approaches:**
    *   Train a lightweight model to detect numerical density, iteration requirements, or equation complexity.
    *   Use meta-prompting to let the LLM decide which method to apply.
    *   Hybrid systems that attempt both methods and select the most confident answer.

The breakdown analysis in **Figure 6** provides guidance: PoT shows massive gains on `linear/polynomial equation` (+36%), `iterative` (+22%), and `combinatorics` (+36%), but minimal gains on `arithmetic` (+2%) and `probability` (+2%). A classifier could use these patterns to route questions appropriately.

**Direction 3: Fine-Tuning for Accessibility**

**Table 4** reveals a stark gap: open-source 16B models score below 13% on GSM8K, while Codex (175B) scores 71.6%. This limits PoT's reproducibility and democratization.

*   **Research Question:** Would fine-tuning smaller models on PoT-style demonstrations close the gap with Codex?
*   **Potential Approaches:**
    *   Fine-tune code-specialized models (CodeGen, CodeT5+) on PoT exemplars.
    *   Use distillation to transfer PoT capabilities from large models to smaller ones.
    *   Develop training curricula that progressively increase program complexity.

This is particularly important given that the paper's code and data will be released (per the **Abstract**), enabling the community to pursue this direction.

**Direction 4: Self-Correction and Debugging**

**Section 4.4** mentions follow-up work on self-critic (Gou et al., 2023) and self-eval (Xie et al., 2023) that enhance PoT with self-evaluation.

*   **Research Question:** Can PoT programs be automatically debugged when execution fails?
*   **Potential Approaches:**
    *   When the interpreter raises an exception, feed the error message back to the LLM for correction.
    *   Use unit tests on intermediate variables to catch grounding errors before final execution.
    *   Implement iterative refinement loops where the model revises code based on execution feedback.

This direction addresses the 33% logic generation errors identified in **Section 3.3**, where the model generates syntactically valid but semantically incorrect code.

**Direction 5: Extending the Tool Set**

The current implementation restricts the model to Python 3.8 with SymPy and standard math libraries (**Section 3.1**). **Section 6 (Limitations)** acknowledges this constraint:

> "For other unknown symbolic tasks, it might hurt the PoT's generalization."

*   **Research Question:** How can PoT safely interact with external tools without compromising security?
*   **Potential Approaches:**
    *   Develop sandboxed execution environments that allow controlled API access.
    *   Create a registry of pre-approved tools (calculators, databases, search engines) that the model can invoke.
    *   Implement permission systems where certain operations require explicit user approval.

This extends PoT from a math-solving tool to a general-purpose reasoning framework.

### 7.3 Practical Applications and Downstream Use Cases

PoT's strengths align with specific real-world applications where numerical precision is critical.

**Financial Analysis and Reporting**

The largest performance gains occur on financial datasets: `FinQA` (+24.1%) and `ConvFinQA` (+19.0%) in **Table 2**. These datasets involve:
*   Large numbers (millions of dollars)
*   Complex calculations (compound interest, ratios, growth rates)
*   Hybrid inputs (tables + text from financial reports)

*   **Use Case:** Automated financial report analysis where accuracy is non-negotiable.
*   **Why PoT:** Computation errors in financial contexts can have legal and regulatory consequences. PoT's deterministic execution eliminates arithmetic hallucination.
*   **Integration:** Combine PoT with document parsing to extract tables from PDF reports, then generate Python programs for calculation.

**Educational Technology**

Math word problem datasets (`GSM8K`, `SVAMP`, `MultiArith`) represent the type of problems students encounter in K-12 education.

*   **Use Case:** AI tutoring systems that solve problems step-by-step and explain the reasoning.
*   **Why PoT:** The generated code is human-readable and can be displayed to students as a transparent solution method. Variable names like `total_eggs` and `sold_eggs` (from **Section 7.2 Appendix**) provide semantic clarity.
*   **Integration:** Use PoT to generate solutions, then add natural language explanations derived from the code structure.

**Scientific Computing Assistance**

While not evaluated in the paper, the PoT architecture extends to scientific domains where symbolic math is common.

*   **Use Case:** Assisting researchers with formula derivation, unit conversion, or equation solving.
*   **Why PoT:** SymPy integration (**Section 3.1**) enables symbolic manipulation, not just numeric computation. The `solve_it` function shown in the **Appendix (Section 7.2)** demonstrates polynomial equation solving.
*   **Integration:** Extend the allowed module set to include scientific libraries (NumPy, SciPy) with appropriate sandboxing.

**Data Analysis and Business Intelligence**

The linearization strategy for tables (**Section 3.1**) enables PoT to handle structured data inputs.

*   **Use Case:** Natural language interfaces to spreadsheets or databases.
*   **Why PoT:** Users can ask questions in natural language, and PoT generates executable code to compute answers from tabular data.
*   **Integration:** Combine with SQL generation for database queries, or pandas code for spreadsheet manipulation.

### 7.4 Reproducibility and Integration Guidance

Practitioners considering PoT should evaluate whether their use case aligns with the method's strengths and constraints.

**When to Prefer PoT Over CoT**

Based on the experimental evidence, PoT is recommended when:

| Condition | Recommendation | Evidence |
|-----------|---------------|----------|
| Large numbers involved (e.g., millions) | **Use PoT** | FinQA +24.1% gain (**Table 2**) |
| Iterative computation required | **Use PoT** | Figure 1 shows CoT fails on 50-step iteration |
| Polynomial/symbolic equations | **Use PoT** | Figure 6: +36% on linear/polynomial |
| Simple arithmetic only | **CoT acceptable** | Figure 6: only +2% on arithmetic |
| Semantic/commonsense reasoning | **Use CoT** | Section 5: PoT not suitable for StrategyQA |
| Open-source models only | **CoT may be better** | Table 4: open-source PoT &lt;13% on GSM8K |

**Implementation Checklist**

For practitioners deploying PoT, the paper provides specific guidance:

1.  **Model Selection:** Use code-specialized models (Codex, `gpt-3.5-turbo`). Avoid text-only models like `text-davinci-002` which score 11.2% lower on GSM8K (**Table 4**).

2.  **Exemplar Count:** Use 6-8 shots for high-diversity datasets (GSM8K, AQuA), 4 shots for simpler datasets (FinQA). **Figure 5** shows variance drops from 7% at K=2 to minimal at K=8.

3.  **Zero-Shot Configuration:** Apply logit bias of `-2` to the `#` token to suppress comment generation (**Section 3.1**). Without this, the model may revert to natural language rationales.

4.  **Security Sandbox:** Block additional module imports beyond SymPy and standard math libraries (**Section 6**). Implement timeout limits to prevent infinite loops.

5.  **Self-Consistency:** For maximum accuracy, use K=40 samples at temperature 0.4 (**Section 3.2**). This adds 40× inference cost but improves GSM8K from 71.6% to 80.0%.

6.  **Value Grounding Verification:** Consider adding an intermediate step to verify extracted values before code execution, since 47% of errors are grounding-related (**Section 3.3**).

**Cost-Benefit Trade-offs**

| Factor | PoT | CoT |
|--------|-----|-----|
| Accuracy (numerical tasks) | Higher (+12% average) | Lower |
| Inference latency | Higher (code execution + SC) | Lower |
| Model requirements | Code-specialized (Codex, GPT-3.5) | General LLMs acceptable |
| Security considerations | Code execution risks | Minimal |
| Interpretability | High (readable code) | Medium (natural language) |
| Task scope | Numerical/symbolic only | Broad (including semantic) |

**Reproducibility Notes**

The paper states in the **Abstract**: "All of our data and code will be released." This enables direct reproduction of results. Key reproducibility factors:

*   **Prompt Templates:** The **Appendix (Section 7.2)** provides full exemplars for GSM8K and AQuA.
*   **Evaluation Scripts:** The paper uses official evaluation scripts for `TabMWP`, `ConvFinQA`, and `TATQA` (**Section 3.1**).
*   **Model Access:** Codex and GPT-3.5 require API access. Open-source alternatives perform significantly worse (**Table 4**).
*   **Random Seeds:** Self-consistency decoding (K=40) introduces variance. Multiple runs should be averaged for stable reporting.

### 7.5 Summary: PoT's Place in the Reasoning Landscape

Program of Thoughts establishes a new design pattern for neuro-symbolic reasoning systems. Its core contribution—the architectural disentanglement of reasoning and computation—extends beyond numerical tasks to any domain where executable logic can be separated from semantic understanding.

**Key Takeaways for Researchers:**
*   PoT validates that LLMs are better at expressing reasoning procedures than executing them.
*   Semantic binding (meaningful variable names) is a functional requirement, not a stylistic choice.
*   The error profile shifts from computation errors to value grounding errors, redirecting future improvement efforts.

**Key Takeaways for Practitioners:**
*   PoT is most valuable when computation complexity is high (large numbers, iteration, symbolic math).
*   Code-specialized models are essential; open-source alternatives currently underperform significantly.
*   Security sandboxing is mandatory for production deployment.

**Open Questions for the Community:**
*   Can fine-tuning close the gap between open-source models and Codex for PoT?
*   How can PoT be extended to safely interact with external APIs and databases?
*   What mechanisms enable automatic selection between PoT and CoT based on problem characteristics?

The release of code and data positions PoT as a foundation for subsequent research. As noted in **Section 4.4**, follow-up work on self-critic, self-eval, and plan-and-solve already builds on this foundation. The trajectory suggests that PoT-style architectures will become a standard component in the toolkit for building reliable, interpretable reasoning systems.
