## 1. Executive Summary

This paper introduces ACE (Agentic Context Engineering), a framework that solves the "brevity bias" and "context collapse" limitations of existing context adaptation methods by treating contexts as evolving playbooks that accumulate strategies through modular generation, reflection, and curation. ACE consistently outperforms strong baselines with gains of +10.6% on the AppWorld agent benchmark and +8.6% on financial reasoning tasks (FiNER, Formula), enabling a smaller open-source model (DeepSeek-V3.1) to match the top-ranked production agent (IBM-CUGA/GPT-4.1) on the AppWorld leaderboard. Beyond accuracy, the framework reduces adaptation latency by 86.9% on average and operates effectively without labeled supervision by leveraging natural execution feedback, demonstrating a scalable path for self-improving LLM systems.

## 2. Context and Motivation

To understand the contribution of this paper, we must first establish the shifting paradigm in how Large Language Models (LLMs) are adapted for specific tasks. Traditionally, improving an LLM involved **weight updates** (fine-tuning), where the model's internal parameters are modified via gradient descent. This paper focuses on **context adaptation** (or context engineering), a paradigm where performance is improved by modifying the *inputs* to the model—such as instructions, examples, or memory—rather than its weights.

### 2.1 The Paradigm of Context Adaptation

Context adaptation has emerged as a central strategy for building scalable AI systems, particularly for agents and domain-specific reasoning. The paper outlines three primary advantages of this approach over weight updates (Section 1):

*   **Interpretability:** Contexts are human-readable. Developers can inspect system prompts or memory to understand why a model behaves a certain way, whereas weight updates are opaque.
*   **Rapid Integration:** New knowledge can be integrated at runtime without retraining. This is critical for applications where facts change frequently (e.g., financial regulations or API documentation).
*   **Shareability:** Contexts can be shared across different models or modules in a compound AI system, decoupling knowledge from the underlying model architecture.

This paradigm is enabled by recent advances in **long-context LLMs**, which can process hundreds of thousands of tokens, and efficient inference techniques like **KV cache reuse**, which make handling large inputs computationally feasible (Section 1). However, despite these infrastructure improvements, the *methods* used to construct and update these contexts remain flawed.

### 2.2 Critical Failure Modes: Brevity Bias and Context Collapse

The paper identifies two specific structural failures in existing context adaptation methods that prevent them from scaling effectively. Understanding these failure modes is essential to grasping why ACE is designed the way it is.

#### Brevity Bias
**Brevity bias** refers to the tendency of prompt optimization methods to converge toward short, generic instructions rather than comprehensive, domain-specific strategies.
*   **Mechanism:** Optimizers often prioritize concise prompts that perform well on validation metrics. For example, GEPA (a baseline discussed in Section 2.1) highlights brevity as a strength.
*   **Consequence:** This abstraction omits critical details such as domain-specific heuristics, tool-use guidelines, or common failure modes. In complex domains like financial analysis or agent planning, success depends on accumulating detailed tactics, not compressing them (Section 2.2).
*   **Example:** The paper cites prior work where iterative optimization produced near-identical instructions like "Create unit tests to ensure methods behave as expected," sacrificing diversity and omitting specific constraints needed for the task (Section 2.2).

#### Context Collapse
**Context collapse** is a phenomenon where iterative rewriting of the context by an LLM causes accumulated knowledge to be abruptly erased or compressed into less informative summaries.
*   **Mechanism:** Many methods (e.g., Dynamic Cheatsheet) task an LLM with fully rewriting the accumulated context at each adaptation step. As the context grows large, the model tends to compress it to fit its own processing preferences or attention limits.
*   **Evidence:** The paper provides a concrete case study in Figure 2. At step 60 of an adaptation process on the AppWorld benchmark, the context contained 18,282 tokens and achieved 66.7% accuracy. At the very next step, the context collapsed to just 122 tokens, and accuracy dropped to 57.1%—worse than the baseline without adaptation (Section 2.2).
*   **Significance:** This demonstrates that monolithic rewriting is unstable. It creates a risk where valuable historical knowledge is lost not because it was irrelevant, but because the rewriting process itself was destructive.

### 2.3 Limitations of Prior Approaches

The paper positions ACE against several representative methods in the context adaptation landscape. Each falls short in addressing the failure modes described above.

*   **In-Context Learning (ICL):** Provides fixed demonstrations in the input prompt. While effective for format inference, it lacks the ability to evolve or accumulate strategies over time. It is static and cannot adapt to new failure modes encountered during execution (Section 4.2).
*   **Prompt Optimizers (e.g., GEPA, MIPROv2):** These methods use natural language feedback to refine prompts.
    *   **GEPA:** Uses reflective prompt evolution and genetic search. While sample-efficient, it suffers from **brevity bias**, optimizing for concise instructions that may lack the detailed heuristics needed for complex agents (Section 2.2, 4.2).
    *   **MIPROv2:** Uses Bayesian optimization to tune instructions and demonstrations. Like GEPA, it tends toward static or compressed prompt structures rather than expanding knowledge bases (Section 4.2).
*   **Adaptive Memory Systems (e.g., Dynamic Cheatsheet):** Constructs an external memory of strategies from past successes and failures.
    *   **Strength:** Does not require ground-truth labels; can curate memory from generations.
    *   **Weakness:** Susceptible to **context collapse**. Because it relies on monolithic rewriting to update the memory, it risks erasing detailed knowledge as the context grows (Section 2.2, Figure 2).

### 2.4 Positioning ACE: From Summaries to Playbooks

ACE positions itself as a solution to the structural instability of prior methods. It argues that contexts should not function as concise summaries (which humans prefer) but as **comprehensive, evolving playbooks** (which LLMs benefit from).

*   **Philosophical Shift:** Unlike humans, who benefit from concise generalization, LLMs are more effective when provided with long, detailed contexts and can distill relevance autonomously at inference time (Section 1). Therefore, the goal is to preserve detailed domain-specific heuristics rather than compress them.
*   **Architectural Shift:** ACE builds on the agentic architecture of Dynamic Cheatsheet but introduces a **modular workflow** (Generation, Reflection, Curation) to prevent collapse.
    *   Instead of monolithic rewriting, ACE uses **incremental delta updates**. Only relevant parts of the context are modified, preserving the rest (Section 3.1).
    *   Instead of a single model doing all work, ACE separates **evaluation** (Reflector) from **integration** (Curator), improving the quality of insights added to the context (Section 3).
*   **Operational Shift:** ACE is designed for both **offline adaptation** (optimizing system prompts before deployment) and **online adaptation** (updating memory during test-time execution). This flexibility allows it to handle both static domain knowledge and dynamic environment interactions (Section 1, 3).

By addressing brevity bias and context collapse directly through structural design choices, ACE aims to enable self-improving systems that scale with long-context models without incurring the latency or instability of prior adaptive methods.

## 3. Technical Approach

This is a systems architecture paper that proposes a modular framework for evolving language model contexts through structured generation, reflection, and curation. The core idea is to replace unstable monolithic context rewriting with incremental, itemized updates that preserve accumulated knowledge while preventing redundancy.

### 3.1 Reader orientation (approachable technical breakdown)

ACE is a three-agent system where one model generates task solutions, a second model critiques those solutions to extract lessons, and a third model decides which lessons to permanently add to a shared knowledge playbook. It solves the problem of context adaptation becoming unstable or losing information over time by treating contexts as versioned collections of bullet points rather than documents that get completely rewritten at each step.

### 3.2 Big-picture architecture (diagram in words)

The ACE framework consists of three specialized components connected in a sequential pipeline (Figure 4, Section 3):

*   **Generator:** Receives a task query and the current context playbook, then produces a reasoning trajectory (e.g., code execution steps for an agent task). Its output includes both the solution attempt and metadata about which context bullets were helpful or misleading during generation.
*   **Reflector:** Receives the Generator's trajectory along with execution feedback (e.g., test results, ground truth comparisons). It analyzes errors and successes to produce structured natural language insights explaining what went wrong and what strategy should be remembered.
*   **Curator:** Receives the Reflector's insights and the current playbook. It decides which new bullets to add, which existing bullets to update (e.g., incrementing helpful/harmful counters), and performs de-duplication. The merging logic is deterministic and non-LLM-based, ensuring stable integration.

Information flows unidirectionally: Query → Generator → Trajectory → Reflector → Insights → Curator → Updated Playbook → (loops back to Generator for next query).

*   First, I explain the **bullet structure** that forms the atomic unit of ACE contexts, since all downstream mechanisms operate on this representation.
*   Second, I detail the **Generator's role** in producing trajectories and providing feedback on context usefulness, which seeds the adaptation loop.
*   Third, I describe the **Reflector's analysis process**, including how it extracts actionable insights from execution traces and why separating this from curation matters.
*   Fourth, I cover the **Curator's integration logic**, explaining how delta updates are merged without monolithic rewriting.
*   Fifth, I explain the **grow-and-refine mechanism** that balances context expansion with redundancy control.

### 3.4 Detailed, sentence-based technical breakdown

**Paper Type and Core Mechanism.** This is a systems architecture paper that introduces a modular workflow for context adaptation, where the fundamental innovation is replacing end-to-end context rewriting with structured, incremental delta updates guided by separate agent roles. The system treats context not as a single prompt string but as a versioned collection of itemized knowledge bullets, each with metadata tracking its utility across tasks.

**Atomic Context Unit: The Bullet Structure.** Each piece of knowledge in ACE's context playbook is stored as a structured bullet with two components (Section 3.1). The first component is **metadata**, which includes a unique identifier (e.g., `ctx-00263`) and counters tracking how many times the bullet was marked as helpful or harmful during task execution. The second component is **content**, which captures a small, reusable unit such as a domain-specific strategy, a conceptual definition, or a documented failure mode. This itemized design enables three critical properties: localization (only relevant bullets are updated during adaptation), fine-grained retrieval (the Generator can focus on the most pertinent knowledge for a given query), and incremental adaptation (efficient merging, pruning, and de-duplication during inference without full context regeneration).

**Generator: Trajectory Production and Context Feedback.** The workflow begins when the Generator receives a new task query along with the current context playbook (Figure 4, Section 3). The Generator produces a reasoning trajectory—for example, a sequence of code execution steps for an agent task or a chain of numerical computations for financial analysis. Critically, during generation, the model also provides feedback on which bullets from the playbook were useful or misleading for solving the current task. This feedback is not optional; it serves as the signal that guides the Reflector in proposing corrective updates. The Generator uses the same underlying LLM as the other components (DeepSeek-V3.1 in non-thinking mode) to prevent knowledge transfer from a stronger Reflector or Curator to a weaker Generator, isolating the benefit of context construction itself (Section 4.2). The batch size is set to 1, meaning a delta context is constructed from each individual sample rather than aggregating across multiple samples before updating (Section 4.2).

**Reflector: Insight Extraction Through Structured Critique.** The Reflector receives the Generator's trajectory along with execution feedback, which may include test results, ground truth comparisons, or environment signals such as code execution success or failure (Section 3, Figure 10). The Reflector's job is to diagnose what went wrong (or right) and extract concrete, actionable insights that can prevent similar errors in the future. This component separates evaluation and insight extraction from curation, which is a key design choice that improves context quality and downstream performance (Section 3, ablation in §4.5). The Reflector can refine its insights across multiple iterations, with a maximum of 5 refinement rounds allowed per sample (Section 4.2). The output is structured natural language feedback containing fields such as `reasoning`, `error_identification`, `root_cause_analysis`, `correct_approach`, and `key_insight` (Figure 10). Importantly, the Reflector also tags each bullet from the playbook that was used during generation as `helpful`, `harmful`, or `neutral`, providing granular feedback that the Curator uses to update bullet metadata (Figure 13). This separation of concerns matters because a single model tasked with both critiquing and integrating knowledge may optimize for different objectives—for example, prioritizing concise summaries over detailed heuristics—which leads back to the brevity bias problem ACE aims to solve.

**Curator: Deterministic Delta Integration.** The Curator receives the Reflector's insights and the current playbook, then decides which new bullets to add and which existing bullets to update (Figure 11, Section 3). Unlike prior methods that task an LLM with fully rewriting the accumulated context, the Curator produces compact **delta contexts**: small sets of candidate bullets distilled by the Reflector and integrated by lightweight, non-LLM logic. This is the mechanism that prevents context collapse. Because updates are itemized and localized, multiple deltas can be merged in parallel, enabling batched adaptation at scale (Section 3). The Curator's output is a pure JSON object specifying operations to perform on the playbook, such as `ADD` operations that create new bullet points with fresh IDs in specific sections (e.g., `strategies_and_hard_rules`, `apis_to_use_for_specific_information`, `verification_checklist`) (Figure 11). The merging logic is deterministic, meaning the same delta applied to the same playbook will always produce the same updated playbook, eliminating the variance introduced by monolithic context rewriting. The Curator is instructed to avoid redundancy—if similar advice already exists in the playbook, it should only add new content that complements existing bullets rather than regenerating the entire playbook (Figure 11).

**Grow-and-Refine Mechanism: Balancing Expansion and Compression.** Beyond incremental growth, ACE ensures that contexts remain compact and relevant through a **grow-and-refine** process (Section 3.2). In this mechanism, bullets with new identifiers are appended to the playbook, while existing bullets are updated in place (e.g., incrementing helpful/harmful counters based on Reflector feedback). A de-duplication step then prunes redundancy by comparing bullets via semantic embeddings, removing bullets that are semantically similar to existing ones. This refinement can be performed in two modes depending on application requirements: **proactively** (after each delta update) for maximum accuracy, or **lazily** (only when the context window is exceeded) for minimum latency (Section 3.2). This design choice directly addresses the context collapse problem: instead of compressing the entire context at once (which risks losing information), ACE grows steadily and only prunes when necessary, using semantic similarity rather than LLM summarization to identify redundancy.

**Offline vs. Online Adaptation Modes.** ACE supports two distinct adaptation settings, each with different configuration parameters (Section 1, 4.1). In **offline adaptation**, methods are optimized on the training split and evaluated on the test split with pass@1 accuracy. The maximum number of epochs in offline adaptation is set to 5, meaning the same training queries can be revisited up to 5 times to progressively strengthen the context (Section 4.2). In **online adaptation**, methods are evaluated sequentially on the test split: for each sample, the model first predicts with the current context, then updates its context based on that sample's execution feedback. The same shuffled test split is used across all methods to ensure fair comparison (Section 4.1). ACE can also combine these modes through **offline warmup**, where the context is initialized through offline adaptation before online adaptation begins, which Table 3 shows improves online performance from 56.1% to 59.5% average accuracy on AppWorld (Section 4.5).

**Configuration and Hyperparameters.** Several implementation choices are critical for reproducibility and fair comparison (Section 4.2). ACE uses the same LLM for the Generator, Reflector, and Curator (non-thinking mode of DeepSeek-V3.1), preventing knowledge transfer from a stronger Reflector or Curator to a weaker Generator. The batch size is 1, meaning a delta context is constructed from each sample individually. The maximum number of Reflector refinement rounds is 5, and the maximum number of epochs in offline adaptation is also 5. For the AppWorld benchmark, ACE builds on top of the official ReAct implementation released by the benchmark authors, ensuring that gains come from context engineering rather than changes to the underlying agent framework (Section 4.2). These choices isolate the benefit of context construction itself, making the reported improvements attributable to ACE's architecture rather than model capacity or agent design.

**Why Incremental Updates Over Monolithic Rewriting.** The decision to use incremental delta updates rather than full context regeneration is the core technical contribution that prevents context collapse (Section 3.1). When an LLM is tasked with fully rewriting an accumulated context at each adaptation step, it tends to compress the context into much shorter, less informative summaries as the context grows large—as evidenced by the case study where context dropped from 18,282 tokens to 122 tokens in a single step (Figure 2, Section 2.2). By contrast, ACE's incremental approach ensures that past knowledge is preserved and new insights are steadily appended, with de-duplication handled via semantic embeddings rather than LLM summarization. This design choice directly trades off some flexibility (the Curator cannot reorganize the entire playbook structure) for stability (knowledge is not abruptly erased). The ablation study in Table 3 confirms this matters: removing the Reflector or multi-epoch refinement substantially reduces performance, showing that the modular workflow is not just an implementation detail but a key driver of ACE's success (Section 4.5).

**Feedback Signals and Supervision Requirements.** ACE can adapt effectively without labeled supervision by leveraging natural execution feedback, such as code execution success or failure, test results, or environment signals (Section 1, 4.3). In the AppWorld benchmark, ReAct + ACE achieves an average improvement of 14.8% over the ReAct baseline even without access to ground-truth labels during adaptation (Table 1). This robustness arises because ACE leverages signals naturally available during execution to guide the Reflector and Curator in forming structured lessons of successes and failures. However, the paper also notes a limitation: when ground-truth supervision or reliable execution signals are absent, both ACE and other adaptive methods such as Dynamic Cheatsheet may degrade in performance (Table 2, Section 4.4). In such cases, the constructed context can be polluted by spurious or misleading signals, highlighting that context adaptation depends critically on feedback quality. This is not a flaw unique to ACE but a fundamental constraint of inference-time adaptation without reliable feedback signals.

**Cost and Latency Optimizations.** ACE achieves significant efficiency gains through its support for incremental delta updates and non-LLM-based context merging (Section 4.6). On offline adaptation of AppWorld, ACE achieves 82.3% reduction in adaptation latency and 75.1% reduction in the number of rollouts compared to GEPA (Table 4a). On online adaptation of FiNER, ACE achieves 91.5% reduction in adaptation latency and 83.6% reduction in token dollar cost compared to Dynamic Cheatsheet (Table 4b). These gains arise because ACE avoids the computational cost and latency of full rewrites: instead of regenerating the entire context at each step, it produces compact delta contexts that are merged deterministically. Additionally, the paper notes that longer contexts do not necessarily translate to higher serving cost, as modern serving infrastructures are increasingly optimized for long-context workloads through techniques such as KV cache reuse, compression, and offload (Section 5). This means the amortized cost of handling ACE's comprehensive playbooks will continue to decrease as infrastructure improves.

## 4. Key Insights and Innovations

This section distills the conceptual breakthroughs that distinguish ACE from prior context adaptation methods. While Section 3 detailed the mechanical implementation (the "how"), this section explains the underlying design philosophy (the "why") and why these choices represent fundamental shifts rather than incremental tweaks. We focus on three core innovations: the stabilization of context growth, the decoupling of critique from integration, and the reliance on execution feedback for autonomy.

### 4.1 Structural Stability: Context as Versioned State vs. Rewritten Document

The most fundamental innovation in ACE is the treatment of context as a **versioned collection of items** rather than a **monolithic document** subject to rewriting. Prior adaptive memory systems, such as Dynamic Cheatsheet (DC), operate on the assumption that an LLM can successfully summarize and integrate new knowledge into an existing text block at every step. ACE challenges this assumption, identifying it as the root cause of **context collapse** (Section 2.2).

*   **The Innovation:** ACE replaces generative rewriting with **incremental delta updates**. Instead of asking an LLM to "rewrite the memory to include this new lesson," ACE asks the Curator to "append this specific bullet point." The merging logic is deterministic and non-LLM-based (Section 3.4).
*   **Why It Matters:** This design choice fundamentally changes the stability properties of the system.
    *   **Preventing Catastrophic Forgetting:** In the case study shown in Figure 2, a monolithic rewriting approach caused context length to plummet from 18,282 tokens to 122 tokens in a single step, destroying accumulated knowledge. ACE's itemized approach ensures that once a bullet is added, it persists unless explicitly pruned by semantic de-duplication.
    *   **Scaling Behavior:** Because updates are localized, the computational cost of adaptation does not grow linearly with context size. Table 4 demonstrates that this leads to an **82.3% reduction in latency** for offline adaptation compared to GEPA, and a **91.5% reduction** for online adaptation compared to DC.
    *   **Theoretical Implication:** This suggests that for long-context adaptation, **deterministic insertion** is superior to **generative integration**. LLMs are excellent at generating content (Reflector) but unreliable at editing large stateful documents without loss (Curator). By offloading the "edit" operation to deterministic logic, ACE guarantees information preservation.

### 4.2 Architectural Decoupling: Separating Critique from Integration

ACE introduces a strict separation of concerns between the **Reflector** (who identifies errors) and the **Curator** (who updates the playbook). In prior work like GEPA or standard prompt optimization, the same model process often diagnoses a failure and immediately rewrites the prompt. ACE argues that these are conflicting objectives.

*   **The Innovation:** The Reflector is optimized for **diagnostic depth**, while the Curator is optimized for **structural integrity**. The Reflector produces detailed JSON insights including `root_cause_analysis` and `key_insight` (Figure 10), while the Curator focuses solely on whether this insight fits into the existing playbook structure without redundancy (Figure 11).
*   **Why It Matters:** This decoupling directly mitigates **brevity bias** (Section 2.2).
    *   **Preserving Nuance:** If a single model is tasked with both finding errors and updating the prompt, it tends to optimize for the shortest prompt that fixes the immediate error, often discarding broader heuristics. By isolating the Reflector, ACE allows for detailed, verbose diagnostic traces that are then carefully filtered by the Curator.
    *   **Empirical Validation:** The ablation study in Table 3 provides strong evidence for this design. Removing the Reflector (or disabling multi-epoch refinement) causes performance to drop from **59.4% to 55.1%** average accuracy on AppWorld. This **4.3% gap** confirms that the quality of insights extracted by a dedicated Reflector is a primary driver of performance, not just the presence of memory itself.
    *   **Modularity:** This separation allows future iterations to upgrade the Reflector (e.g., using a larger model for better diagnosis) without changing the Curator's merging logic, offering a clearer path for system improvement than monolithic optimizers.

### 4.3 Operational Viability: Label-Free Adaptation via Execution Feedback

A critical barrier to deploying adaptive agents in real-world settings is the scarcity of ground-truth labels. Most prompt optimizers (like GEPA or MIPROv2) rely on validation metrics or correct answers to guide optimization. ACE demonstrates that **execution feedback** alone is sufficient for significant improvement.

*   **The Innovation:** ACE leverages natural signals available during agent execution—such as code success/failure, API return values, or test report assertions—as the reward signal for the Reflector. The Reflector compares the generated trajectory against these environment signals rather than a gold-standard answer (Section 4.3).
*   **Why It Matters:** This capability shifts context adaptation from a **training-time optimization** problem to a **runtime learning** problem.
    *   **Real-World Applicability:** In domains like personal assistants or enterprise automation, ground truth is often unavailable (e.g., there is no single "correct" email to draft). ACE's ability to learn from execution success allows it to adapt in these ambiguous environments.
    *   **Performance Without Supervision:** Table 1 shows that ReAct + ACE without ground-truth (GT) labels achieves **57.2%** average accuracy, compared to the baseline ReAct score of **42.4%**. This **14.8% gain** is achieved purely through self-reflection on execution traces.
    *   **Limitation Boundary:** It is important to note the boundary of this innovation. Table 2 shows that on financial benchmarks (FiNER), performance degrades without GT labels (dropping from 81.9% to 77.1% for ACE). This indicates that execution feedback is most effective when the environment provides **unambiguous success signals** (e.g., code passes tests) rather than semantic correctness (e.g., financial entity labeling). This distinction is crucial for practitioners deciding where to deploy ACE.

### 4.4 Summary of Contributions

Table 5 synthesizes how these innovations map to the limitations identified in Section 2.

| Limitation | Prior Approach | ACE Innovation | Resulting Benefit |
| :--- | :--- | :--- | :--- |
| **Context Collapse** | Monolithic Rewriting (LLM edits full text) | **Incremental Delta Updates** (Deterministic merging) | Stable knowledge accumulation; **91.5% lower latency** (Table 4b). |
| **Brevity Bias** | Single-model optimization (Diagnose + Edit) | **Modular Reflection** (Separate Reflector/Curator) | Higher quality insights; **+4.3% accuracy** vs. ablation (Table 3). |

These innovations collectively demonstrate that scalable self-improvement in LLMs requires **structural constraints** on how context is updated, not just better prompting. By enforcing stability through deterministic merges and quality through modular reflection, ACE enables contexts to grow comprehensive without becoming unstable—a prerequisite for the next generation of autonomous agents.

## 5. Experimental Analysis

This section dissects the empirical evidence supporting ACE's claims. We move beyond the architectural description to examine how the framework performs under rigorous benchmarking conditions. The experiments are designed to answer three critical questions: Does ACE outperform existing adaptation methods across diverse tasks? Can it operate effectively without ground-truth labels? And does it achieve these gains efficiently?

### 5.1 Evaluation Methodology

The evaluation strategy is split across two distinct domains that stress-test context adaptation in different ways: autonomous agent interaction and static domain-specific reasoning.

**Datasets and Tasks.**
*   **AppWorld [43]:** A benchmark for autonomous agents involving API understanding, code generation, and environment interaction (e.g., email, file system, Spotify). It features two difficulty splits: `test-normal` and `test-challenge`. The environment provides executable feedback, making it suitable for testing online adaptation without ground-truth labels (Section 4.1).
*   **Financial Analysis:** Two benchmarks testing specialized knowledge.
    *   **FiNER [33]:** Financial Numeric Entity Recognition. Requires labeling tokens in XBRL documents with one of 139 fine-grained entity types. This tests precise information extraction (Section 4.1).
    *   **Formula [44]:** Numerical reasoning over XBRL filings. Requires extracting values and performing computations. This tests multi-step logical reasoning (Section 4.1).

**Metrics.**
*   **Agents:** We report **Task Goal Completion (TGC)** and **Scenario Goal Completion (SGC)**. TGC measures if the specific task objective was met, while SGC measures if the broader scenario state was correctly achieved (Table 1).
*   **Finance:** We report **Accuracy**, defined as the proportion of predicted answers exactly matching the ground truth (Section 4.1).

**Baselines.**
ACE is compared against a hierarchy of methods representing the state of the art in context adaptation (Section 4.2):
*   **Base LLM:** DeepSeek-V3.1 with default prompts (no adaptation).
*   **In-Context Learning (ICL):** Static few-shot/many-shot demonstrations.
*   **Prompt Optimizers:** **MIPROv2** (Bayesian optimization) and **GEPA** (Genetic-Pareto reflective evolution). These represent methods that optimize static prompts.
*   **Adaptive Memory:** **Dynamic Cheatsheet (DC)** in cumulative mode (DC-CU). This is the closest prior work, utilizing an external memory updated at test-time.

**Experimental Setup.**
All methods use **DeepSeek-V3.1** (non-thinking mode) to ensure fair comparison. ACE uses a batch size of $1$, constructing a delta context from each sample individually. For offline adaptation, models are optimized on the training split and evaluated on the test split (pass@1). For online adaptation, models are evaluated sequentially on the test split, updating context after each sample (Section 4.1, 4.2).

### 5.2 Performance on Agent Benchmarks

The AppWorld results (Table 1) provide the strongest evidence for ACE's effectiveness in dynamic, interactive environments.

**Offline Adaptation Gains.**
When ground-truth (GT) labels are available during optimization, ACE significantly outperforms static prompt optimizers.
*   **ReAct + ACE** achieves an average accuracy of **59.4%**, compared to **46.4%** for **ReAct + GEPA** and **46.0%** for **ReAct + ICL**.
*   This represents a gain of **$+13.0\%$** over GEPA, validating the claim that evolving playbooks capture complex agent strategies better than optimized static prompts (Table 1).

**Online Adaptation and Label-Free Learning.**
A key claim of ACE is its ability to self-improve using only execution feedback (e.g., code success/failure) without needing ground-truth labels.
*   **ReAct + ACE (Online, No GT)** achieves **59.5%** average accuracy. This is effectively identical to the offline setting with GT labels (**59.4%**), demonstrating that execution signals are sufficient proxies for supervision in this domain (Table 1).
*   In contrast, **ReAct + DC (Online, No GT)** achieves only **51.9%**. ACE outperforms DC by **$+7.6\%$** on average in the online setting, highlighting the stability benefits of incremental delta updates over DC's monolithic rewriting (Section 4.3).

**Leaderboard Comparison.**
The paper contextualizes these results against the public AppWorld leaderboard (Figure 5, Section 4.3).
*   **ReAct + ACE (59.4%)** matches the top-ranked production agent **IBM-CUGA (60.3%)**, which is powered by **GPT-4.1**.
*   Crucially, ACE achieves this using a smaller open-source model (**DeepSeek-V3.1**).
*   On the harder **test-challenge** split, ACE surpasses IBM-CUGA. The text notes ACE exceeds IBM-CUGA by **$8.4\%$ in TGC** and **$0.7\%$ in SGC** on this split, suggesting ACE's accumulated strategies generalize better to difficult, unseen scenarios (Section 4.3).

### 5.3 Performance on Domain-Specific Reasoning

The financial benchmarks (Table 2) reveal a more nuanced picture, highlighting the dependency of context adaptation on feedback quality.

**Offline Performance.**
With ground-truth labels, ACE demonstrates strong domain mastery.
*   **ACE (Offline, w/ GT)** achieves an average of **81.9%**, surpassing **GEPA (72.5%)** and **ICL (69.6%)** by wide margins.
*   On the **Formula** benchmark specifically, ACE reaches **85.5%** accuracy, a **$+18.0\%$** improvement over the Base LLM (67.5%). This indicates that structured playbooks effectively capture numerical reasoning tactics (Table 2).

**The Feedback Dependency Constraint.**
The online results without ground-truth labels expose a critical boundary condition for ACE.
*   **Formula Benchmark:** ACE maintains strong performance without GT. **ACE (Online, No GT)** achieves **78.5%** on Formula, still a **$+11.0\%$** gain over Base (67.5%). This is because formula correctness provides a clear binary signal (correct/incorrect) for the Reflector (Section 4.4).
*   **FiNER Benchmark:** Performance degrades without reliable signals. **ACE (Online, No GT)** drops to **67.3%** on FiNER, which is **$-3.4\%$** below the Base LLM (70.7%).
*   **Dynamic Cheatsheet** suffers even more, dropping to **65.4%** average (vs. 69.1% Base).
*   **Interpretation:** In entity recognition tasks (FiNER), there is no natural execution signal (like code passing tests) to confirm correctness without ground truth. The Reflector receives noisy feedback, leading to "polluted" context updates (Section 4.4). This confirms that while ACE reduces the *need* for labels, it does not eliminate the need for *reliable feedback signals*.

### 5.4 Ablation Studies and Design Validation

To verify that ACE's performance stems from its specific architectural choices rather than just scale or compute, the authors conduct ablation studies on the AppWorld benchmark (Table 3, Section 4.5).

**Impact of the Reflector.**
The Reflector is responsible for diagnosing errors and extracting insights.
*   Removing the Reflector (and multi-epoch refinement) drops average accuracy from **59.4%** (Full ACE) to **55.1%**.
*   This **$-4.3\%$** decrease confirms that the modular separation of critique (Reflector) from integration (Curator) is essential for high-quality context updates. Without it, the system reverts toward simpler, less effective adaptation (Section 4.5).

**Impact of Multi-Epoch Adaptation.**
ACE revisits training queries multiple times to strengthen context.
*   Disabling multi-epoch adaptation (single pass) reduces accuracy to **56.8%**.
*   This **$-2.6\%$** drop indicates that iterative refinement over the same data allows the system to consolidate lessons more effectively than a single pass (Table 3).

**Impact of Offline Warmup.**
ACE can initialize context offline before switching to online adaptation.
*   **Online ACE without warmup** achieves **56.1%**.
*   **Online ACE with offline warmup** achieves **59.5%**.
*   This **$+3.4\%$** gain suggests that seeding the playbook with foundational strategies before encountering test-time data provides a stable base for online learning (Table 3).

### 5.5 Efficiency and Cost Analysis

A common criticism of long-context methods is computational overhead. Table 4 (Section 4.6) demonstrates that ACE's incremental design actually reduces costs compared to baselines.

**Offline Adaptation Efficiency (AppWorld).**
Compared to **GEPA**, which requires extensive rollouts for genetic search:
*   **Latency:** ACE takes **9,517 seconds**, an **$82.3\%$ reduction** from GEPA's **53,898 seconds**.
*   **Rollouts:** ACE requires **357 rollouts**, a **$75.1\%$ reduction** from GEPA's **1,434 rollouts**.
*   **Reason:** GEPA must regenerate and evaluate entire prompts repeatedly. ACE only generates small delta updates and merges them deterministically (Section 4.6).

**Online Adaptation Efficiency (FiNER).**
Compared to **Dynamic Cheatsheet (DC)**, which rewrites memory at every step:
*   **Latency:** ACE takes **5,503 seconds**, a **$91.5\%$ reduction** from DC's **65,104 seconds**.
*   **Token Cost:** ACE costs **$2.9**, an **$83.6\%$ reduction** from DC's **$17.7**.
*   **Reason:** DC's monolithic rewriting forces the model to process the entire growing context repeatedly. ACE's delta updates avoid re-processing stable knowledge, leveraging KV cache reuse more effectively (Section 4.6, Section 5).

### 5.6 Critical Assessment of Results

The experimental data robustly supports the paper's core claims, but with important boundary conditions that practitioners must heed.

**Strengths.**
*   **Scalability:** The efficiency metrics (Table 4) convincingly show that comprehensive contexts do not require prohibitive compute. The $>90\%$ latency reduction over DC proves that incremental updates are computationally superior to rewriting.
*   **Agent Performance:** The AppWorld results (Table 1) are compelling. Matching a GPT-4.1 system with a smaller open-source model via context engineering alone is a significant result, validating the "playbook" philosophy.
*   **Modularity:** The ablation study (Table 3) confirms that the Reflector and warmup strategies are not just engineering details but primary drivers of performance.

**Limitations and Trade-offs.**
*   **Feedback Quality Dependency:** The FiNER online results (Table 2) serve as a critical caution. ACE is not a magic bullet for label-free learning; it requires *verifiable* execution signals. In tasks where correctness is ambiguous (e.g., entity labeling without a checker), ACE can degrade below the baseline. Users must ensure their environment provides clear success/failure signals before deploying online adaptation.
*   **Context Length vs. Cost:** While the paper argues that longer contexts do not linearly increase serving cost due to KV cache optimization (Section 5), this relies on infrastructure support (e.g., cache reuse [17, 51]). In environments without optimized serving stacks, the token volume of a comprehensive playbook could still incur higher inference costs than a concise prompt.
*   **Task Suitability:** Appendix B notes that not all tasks benefit from rich contexts. Simple tasks (e.g., HotPotQA) may suffer from overhead if the playbook becomes too large relative to the task complexity. ACE is best suited for complex, multi-step domains where detailed heuristics are actually reusable (Section 5, Appendix B).

**Conclusion on Evidence.**
The experiments successfully demonstrate that **structural stability** (incremental updates) and **modular reflection** (separate Reflector) solve the brevity bias and context collapse problems identified in Section 2. The results are consistent across offline and online settings, provided that reliable feedback signals exist. The efficiency gains further validate that this approach is viable for deployment, not just research.

## 6. Limitations and Trade-offs

While ACE demonstrates substantial improvements over prior context adaptation methods, the paper explicitly acknowledges several important limitations and boundary conditions. Understanding these constraints is critical for practitioners deciding whether ACE is appropriate for their use case. This section examines the assumptions ACE relies on, the scenarios where it may fail, and the open questions that remain.

### 6.1 Dependency on Reliable Feedback Signals

The most significant limitation identified in the paper is ACE's dependence on **verifiable execution feedback** for effective adaptation. While ACE can operate without ground-truth labels, it cannot operate without *any* signal of success or failure.

**Evidence from Financial Benchmarks.** Table 2 (Section 4.4) reveals a stark contrast between two financial tasks:

| Benchmark | Signal Type | ACE (Online, No GT) | Base LLM | Degradation |
| :--- | :--- | :--- | :--- | :--- |
| **Formula** | Computation correctness (binary) | 78.5% | 67.5% | **+11.0%** |
| **FiNER** | Entity labeling (no verifier) | 67.3% | 70.7% | **−3.4%** |

On the Formula benchmark, correctness is unambiguous: a computed value is either right or wrong. The Reflector can use this binary signal to extract meaningful lessons. On FiNER, however, there is no natural execution signal to verify entity labels without ground truth. The paper states:

> "In such cases, the constructed context can be polluted by spurious or misleading signals, highlighting a potential limitation of inference-time adaptation without reliable feedback." (Section 4.4)

**Implication for Practitioners.** ACE is best suited for domains where the environment provides clear success/failure signals:
*   **Well-suited:** Code execution (tests pass/fail), API interactions (returns expected data), mathematical computation (verifiable answers), game states (win/loss).
*   **Poorly-suited:** Open-ended generation (creative writing, email drafting), semantic tasks without verifiers (entity recognition, sentiment analysis), tasks where multiple answers could be correct.

This is not a flaw unique to ACE—Dynamic Cheatsheet shows similar degradation (Table 2)—but it is a fundamental constraint of *any* inference-time adaptation method that relies on execution feedback rather than labeled supervision.

### 6.2 Reflector Quality as a Bottleneck

ACE's performance hinges on the Reflector's ability to extract meaningful insights from execution traces. Appendix B explicitly states:

> "A potential limitation of ACE is its reliance on a reasonably strong Reflector: if the Reflector fails to extract meaningful insights from generated traces or outcomes, the constructed context may become noisy or even harmful." (Appendix B)

**Why This Matters.** The Reflector performs several cognitively demanding tasks:
*   Diagnosing root causes from execution errors (not just surface-level symptoms).
*   Distinguishing between task-specific failures and generalizable lessons.
*   Tagging existing playbook bullets as `helpful`, `harmful`, or `neutral` (Figure 10, 13).

If the Reflector model is too weak, it may:
*   Misattribute errors to the wrong causes.
*   Extract overly specific lessons that don't generalize.
*   Fail to identify when existing playbook entries are actually harmful.

**Ablation Evidence.** Table 3 (Section 4.5) shows that removing the Reflector (along with multi-epoch refinement) drops AppWorld accuracy from **59.4% to 55.1%**—a **4.3% decrease**. This confirms that the Reflector is not just an auxiliary component but a primary driver of performance.

**Open Question.** The paper uses the same model (DeepSeek-V3.1) for Generator, Reflector, and Curator to ensure fair comparison (Section 4.2). It remains unclear whether:
*   A stronger Reflector (e.g., GPT-4.1) would yield further gains.
*   A weaker Reflector would cause catastrophic degradation.
*   There is a minimum model capability threshold below which ACE becomes ineffective.

This dependency mirrors Dynamic Cheatsheet's limitation (Appendix B), suggesting it is a broader challenge for adaptive memory systems rather than an ACE-specific weakness.

### 6.3 Task Suitability: Not All Problems Need Playbooks

The paper explicitly acknowledges that comprehensive, evolving contexts are not universally beneficial. Appendix B states:

> "Not all applications require rich or detailed contexts. Tasks like HotPotQA often benefit more from concise, high-level instructions... Similarly, games with fixed strategies such as Game of 24 may only need a single reusable rule, rendering additional context redundant." (Appendix B)

**When ACE Adds Overhead.**
*   **Simple QA Tasks:** For questions with straightforward answers (e.g., HotPotQA [50]), the overhead of maintaining a detailed playbook may exceed the benefit. Concise prompts often suffice.
*   **Fixed-Strategy Domains:** In games or tasks with a small, static solution space (e.g., Game of 24 [41]), a single rule may be all that's needed. Accumulating extensive strategies provides diminishing returns.
*   **One-Off Tasks:** If a task is never repeated, there is no opportunity to reuse accumulated knowledge, making context adaptation unnecessary.

**When ACE Excels.** The paper positions ACE for domains where detailed heuristics are *reusable* across episodes:
*   **Multi-Turn Agents:** AppWorld tasks involve repeated API interactions where strategies (e.g., pagination handling, identity resolution) apply across many tasks (Section 4.3).
*   **Domain-Specific Reasoning:** Financial analysis requires mastering XBRL rules and numerical computation tactics that recur across documents (Section 4.4).
*   **Complex Tool Use:** Environments with many APIs or tools benefit from accumulated documentation of correct usage patterns (Figure 3).

**Practical Guidance.** Practitioners should assess whether their domain has:
1.  **Recurring patterns** that can be codified as reusable strategies.
2.  **Sufficient complexity** that detailed heuristics provide value over concise instructions.
3.  **Multiple episodes** where accumulated knowledge can be reused.

If the answer to these is "no," ACE's overhead may not be justified.

### 6.4 Infrastructure Assumptions for Long-Context Serving

ACE produces longer contexts than methods like GEPA (Section 5 discusses this explicitly). The paper argues this does not linearly increase serving cost:

> "Longer Context ≠ Higher Serving Cost. Although ACE produces longer contexts than methods such as GEPA, this does not translate to linearly higher inference cost or GPU memory usage. Modern serving infrastructures are increasingly optimized for long-context workloads through techniques such as the reuse, compression, and offload of KV cache." (Section 5)

**The Assumption.** This claim relies on deployment infrastructure that supports:
*   **KV Cache Reuse** [17, 51]: Caching frequently reused context segments to avoid repetitive prefill operations.
*   **KV Cache Compression** [30, 32]: Reducing memory footprint of cached contexts.
*   **KV Cache Offload** [25]: Moving cached contexts to cheaper storage when not actively used.

**The Risk.** In environments without these optimizations:
*   Longer contexts increase **prefill latency** proportionally to token count.
*   Memory requirements grow, potentially limiting batch sizes or requiring more expensive hardware.
*   Token costs for inference may be higher if the serving stack charges per input token without cache optimization.

**Evidence Gap.** The paper reports adaptation latency (Table 4) but does not report **inference latency** during task execution with the full playbook. While the authors argue amortized costs will decrease with infrastructure improvements (Section 5), practitioners deploying ACE today on non-optimized stacks may experience higher per-query latency than the paper's efficiency metrics suggest.

### 6.5 Scalability at Extreme Scale

While ACE demonstrates efficiency gains over baselines (Table 4), the paper does not address scalability at extreme scales:

**Unanswered Questions.**
*   **Playbook Size Limits:** How large can the playbook grow before retrieval becomes inefficient? The paper mentions de-duplication via semantic embeddings (Section 3.2), but does not specify at what point this becomes a bottleneck.
*   **Retrieval Overhead:** As the playbook grows, does the Generator spend more time searching for relevant bullets? The paper mentions "fine-grained retrieval" (Section 3.1) but does not quantify retrieval latency at scale.
*   **Parallel Adaptation:** The paper notes that "multiple deltas can be merged in parallel" (Section 3), but does not report experiments with batched adaptation across many concurrent agents or tasks.

**Potential Constraints.**
*   **Semantic De-duplication Cost:** Comparing new bullets against all existing bullets via embeddings has $O(n)$ complexity. At very large playbook sizes, this could become expensive unless indexed appropriately.
*   **Context Window Limits:** Even with long-context models, there is a hard limit on total tokens. The paper mentions lazy refinement "when the context window is exceeded" (Section 3.2), but does not detail what happens when pruning is insufficient.

These are not immediate blockers—the paper demonstrates ACE works effectively at the scales tested—but they represent open engineering challenges for production deployment at massive scale.

### 6.6 Context Pollution and Error Propagation

When feedback signals are unreliable, ACE risks accumulating **harmful knowledge** that degrades performance over time. Section 4.4 notes:

> "In such cases, the constructed context can be polluted by spurious or misleading signals... This suggests that while ACE is robust under rich feedback... its effectiveness depends on the availability of signals that allow the Reflector and Curator to make sound judgments." (Section 4.4)

**Mechanism of Pollution.**
1.  The Generator produces an incorrect trajectory.
2.  The Reflector receives noisy feedback and extracts an incorrect lesson.
3.  The Curator adds this lesson to the playbook as a new bullet.
4.  Future Generators retrieve this harmful bullet and repeat the error.

**Mitigation in ACE.** The paper includes several safeguards:
*   **Helpful/Harmful Counters:** Bullets track how often they were marked helpful or harmful (Section 3.1), allowing the system to identify and potentially prune harmful entries.
*   **De-duplication:** Semantic similarity checks prevent redundant or conflicting bullets from accumulating (Section 3.2).
*   **Curator Filtering:** The Curator is instructed to avoid redundancy and only add complementary content (Figure 11).

**Remaining Risk.** These mechanisms reduce but do not eliminate pollution risk. If the Reflector systematically misdiagnoses errors (e.g., due to model weakness or ambiguous feedback), harmful bullets may accumulate faster than they are pruned. The paper does not report long-term stability experiments (e.g., adaptation over thousands of tasks) that would reveal whether pollution compounds over time.

### 6.7 Summary of Trade-offs

Table 6 synthesizes the key trade-offs practitioners must consider when adopting ACE.

| Trade-off | Benefit | Cost/Risk | Mitigation |
| :--- | :--- | :--- | :--- |
| **Feedback Quality** | Label-free adaptation possible | Degrades without verifiable signals | Ensure environment provides clear success/failure signals |
| **Reflector Strength** | High-quality insights extracted | Weak Reflectors produce noisy context | Use capable models; monitor Reflector output quality |
| **Context Comprehensiveness** | Captures detailed heuristics | Overhead for simple tasks | Assess task complexity before deployment |
| **Long-Context Serving** | Comprehensive playbooks scalable | Requires optimized infrastructure | Deploy with KV cache optimization support |
| **Incremental Updates** | Prevents context collapse | Less flexible than full rewriting | Accept structural constraints for stability |

**Final Assessment.** ACE represents a significant advance in context adaptation, but it is not a universal solution. Its strengths—stability, efficiency, and label-free operation—are most valuable in **complex, multi-episode domains with verifiable feedback**. Practitioners working in simple QA tasks, open-ended generation, or environments without clear success signals should carefully evaluate whether ACE's overhead is justified. The paper's transparency about these limitations (particularly in Appendix B and Section 4.4) is a strength, enabling informed deployment decisions rather than overpromising capabilities.

## 7. Implications and Future Directions

This section examines how ACE reshapes the landscape of context adaptation, identifies concrete research directions it enables, and provides practical guidance for practitioners considering adoption. The implications extend beyond immediate performance gains to fundamental questions about how LLM systems should accumulate and preserve knowledge over time.

### 7.1 Paradigm Shift: From Prompt Optimization to Context Engineering

ACE represents a conceptual shift in how the field approaches LLM adaptation. Prior work treated context as a **static optimization target**—find the best prompt, then deploy it. ACE reframes context as a **dynamic, versioned state** that evolves through structured interactions with the environment.

**Key Conceptual Changes:**

*   **Stability Over Compression:** The paper demonstrates that preventing information loss (context collapse) is more important than minimizing context length. Figure 2 showed accuracy dropping from 66.7% to 57.1% when context collapsed from 18,282 to 122 tokens. This challenges the prevailing assumption that shorter prompts are inherently better.
*   **Deterministic Merging Over Generative Editing:** Section 3.1 establishes that offloading context integration to deterministic logic (rather than LLM rewriting) guarantees knowledge preservation. This suggests a broader principle: **LLMs should generate content, but system logic should manage state.**
*   **Execution Feedback as Supervision:** Section 4.3 shows ACE achieving +14.8% accuracy on AppWorld without ground-truth labels. This shifts context adaptation from a training-time optimization problem to a runtime learning capability, enabling deployment in domains where labels are unavailable.

**Field-Level Impact:**

> "Context adaptation is emerging as a central paradigm for building capable, scalable, and self-improving AI systems." (Section 1)

ACE provides the first framework that makes this paradigm practically viable at scale. The 86.9% average reduction in adaptation latency (Section 4.6) removes a key barrier that previously made context engineering prohibitively expensive compared to static prompting.

### 7.2 Research Directions Enabled by ACE

The architectural choices in ACE open several concrete avenues for follow-up research. Each addresses limitations acknowledged in the paper or extends capabilities demonstrated in the results.

**Direction 1: Heterogeneous Agent Roles.** The paper uses the same model (DeepSeek-V3.1) for Generator, Reflector, and Curator to ensure fair comparison (Section 4.2). However, Section 4.5's ablation study shows the Reflector contributes +4.3% accuracy. This suggests:

*   **Research Question:** Does upgrading only the Reflector to a stronger model (e.g., GPT-4.1) yield disproportionate gains compared to upgrading all components?
*   **Hypothesis:** A stronger Reflector could extract higher-quality insights from the same execution traces, improving context quality without increasing generation costs.
*   **Measurement:** Compare ACE with homogeneous models vs. ACE with Reflector-only upgrades, measuring both accuracy and cost per adaptation step.

**Direction 2: Long-Term Stability and Pollution Dynamics.** Section 6.6 notes the risk of context pollution when feedback is unreliable, but the paper does not report experiments beyond the evaluated benchmark scales.

*   **Research Question:** How does ACE's performance evolve over thousands of adaptation steps? Do harmful bullets accumulate faster than they are pruned?
*   **Hypothesis:** The helpful/harmful counters (Section 3.1) provide a natural decay mechanism, but the threshold for pruning harmful bullets requires tuning.
*   **Measurement:** Track playbook composition over extended adaptation runs, measuring the ratio of helpful-to-harmful bullets and correlating with task performance over time.

**Direction 3: Retrieval Optimization at Scale.** Section 3.1 mentions "fine-grained retrieval" allowing the Generator to focus on pertinent knowledge, but Section 6.5 acknowledges unanswered questions about retrieval overhead as playbooks grow.

*   **Research Question:** At what playbook size does retrieval latency become a bottleneck, and what indexing strategies mitigate this?
*   **Hypothesis:** Semantic embedding-based retrieval (mentioned in Section 3.2 for de-duplication) could be extended to bullet selection, but requires efficient approximate nearest neighbor indexing.
*   **Measurement:** Profile retrieval latency as a function of playbook size, comparing linear scan vs. indexed retrieval approaches.

**Direction 4: Selective Unlearning and Compliance.** Section 5 explicitly connects ACE to responsible AI:

> "Because contexts are human-interpretable, ACE enables selective unlearning—whether due to privacy or legal constraints, or when outdated or incorrect information is identified by domain experts." (Section 5)

*   **Research Question:** Can ACE's itemized bullet structure support efficient compliance-driven removal of specific knowledge (e.g., GDPR "right to erasure")?
*   **Hypothesis:** Removing individual bullets is more tractable than unlearning from model weights, but may require tracking bullet provenance (which training samples contributed to each bullet).
*   **Measurement:** Implement bullet-level provenance tracking and measure the completeness of knowledge removal after deletion requests.

**Direction 5: Cross-Domain Context Transfer.** ACE builds playbooks within a single domain (AppWorld or finance). The modular bullet structure suggests potential for transfer:

*   **Research Question:** Can playbooks trained on one domain (e.g., API interaction patterns) accelerate adaptation in related domains?
*   **Hypothesis:** Generalizable strategies (e.g., pagination handling, error recovery) could be transferred, while domain-specific bullets (e.g., XBRL rules) remain isolated.
*   **Measurement:** Initialize ACE with a pre-trained playbook from a related domain and measure adaptation speed compared to cold-start.

### 7.3 Practical Applications and Use Cases

ACE's design makes it particularly suitable for specific application categories. Practitioners should evaluate their use case against the criteria in Table 7.

**High-Value Applications:**

| Application Type | Why ACE Fits | Evidence from Paper |
| :--- | :--- | :--- |
| **Autonomous Agents** | Multi-turn interactions generate rich execution feedback; strategies generalize across tasks | AppWorld: +17.1% accuracy, matches GPT-4.1 agent with smaller model (Table 1, Section 4.3) |
| **Financial/Legal Analysis** | Domain rules are stable and reusable; numerical reasoning benefits from accumulated tactics | Formula: +18.0% accuracy with structured playbooks (Table 2, Section 4.4) |
| **API Integration Systems** | API behaviors are deterministic; error messages provide clear feedback signals | AppWorld API tasks show strong online adaptation without labels (Table 1) |
| **Code Generation Assistants** | Test execution provides binary success/failure; debugging patterns are reusable | Reflector extracts root causes from execution errors (Figure 10) |

**Low-Value Applications:**

| Application Type | Why ACE Is Overhead | Evidence from Paper |
| :--- | :--- | :--- |
| **Simple QA (e.g., HotPotQA)** | Tasks benefit from concise instructions, not detailed playbooks | Appendix B explicitly notes HotPotQA as unsuitable |
| **Open-Ended Generation** | No verifiable success signal for Reflector to learn from | FiNER degrades -3.4% without ground truth (Table 2) |
| **One-Off Tasks** | No opportunity to reuse accumulated knowledge across episodes | ACE requires multiple episodes for adaptation to pay off (Section 3) |

**Decision Framework for Practitioners:**

Before adopting ACE, practitioners should verify three conditions:

1.  **Verifiable Feedback:** Does the environment provide clear success/failure signals (e.g., test results, API responses, computation correctness)? If not, online adaptation may degrade performance (Section 4.4, Table 2).
2.  **Recurring Patterns:** Do tasks share reusable strategies, heuristics, or failure modes? If each task is unique, accumulated knowledge provides no benefit (Appendix B).
3.  **Infrastructure Support:** Does the deployment stack support KV cache optimization? Without cache reuse, longer contexts increase inference latency proportionally (Section 5).

### 7.4 Integration and Reproducibility Guidance

For teams considering ACE adoption, the paper provides sufficient detail for implementation, but several practical considerations warrant attention.

**Implementation Prerequisites:**

*   **Model Selection:** The paper uses DeepSeek-V3.1 (non-thinking mode) for all components (Section 4.2). Practitioners should ensure their chosen model has sufficient capability for the Reflector role—Appendix B warns that weak Reflectors produce noisy context.
*   **Batch Size:** ACE uses batch size of 1, constructing delta contexts from individual samples (Section 4.2). This simplifies implementation but may limit throughput for high-volume applications.
*   **Refinement Rounds:** Maximum of 5 Reflector refinement rounds and 5 offline adaptation epochs (Section 4.2). These are tunable hyperparameters that trade adaptation quality for latency.

**Integration Checklist:**

1.  **Define Bullet Structure:** Implement the metadata + content bullet format (Section 3.1), including unique identifiers and helpful/harmful counters.
2.  **Implement Deterministic Merger:** Build non-LLM logic for merging delta updates (Section 3.1). This is critical for preventing context collapse.
3.  **Configure Feedback Pipeline:** Ensure execution traces, test results, or environment signals are captured and passed to the Reflector (Figure 10, 13).
4.  **Set Up De-duplication:** Implement semantic embedding comparison for pruning redundant bullets (Section 3.2). Decide between proactive (after each delta) or lazy (when context window exceeded) refinement.
5.  **Monitor Context Growth:** Track playbook size and composition over time to detect pollution or excessive growth (Section 6.5, 6.6).

**When to Prefer ACE Over Alternatives:**

| Scenario | Recommended Approach | Rationale |
| :--- | :--- | :--- |
| **Agent with execution feedback, no labels** | ACE (Online) | Table 1 shows +14.8% over baseline without GT labels |
| **Domain reasoning with GT labels available** | ACE (Offline) | Table 2 shows +12.8% average over baselines with GT |
| **Simple QA with limited context budget** | ICL or Static Prompt | Appendix B notes ACE overhead not justified for simple tasks |
| **High-throughput, latency-sensitive deployment** | GEPA or MIPROv2 | ACE's longer contexts may increase inference latency without KV cache optimization (Section 5) |
| **Tasks with ambiguous success signals** | Static Prompt + Human Review | Table 2 shows ACE degrades on FiNER without reliable feedback |

**Reproducibility Resources:**

The paper provides extensive implementation details:

*   **Code Repository:** https://github.com/ace-agent/ace (Abstract)
*   **Project Page:** https://ace-agent.github.io (Abstract)
*   **Prompts:** Full Generator, Reflector, and Curator prompts for AppWorld and FiNER included in Appendix D (Figures 6-14)
*   **Baseline Configurations:** DSPy implementations specified for GEPA and MIPROv2 with `auto="heavy"` setting (Section 4.2)

These resources significantly lower the barrier to replication compared to papers that omit prompt details or rely on proprietary implementations.

### 7.5 Broader Impact on AI System Design

ACE's success suggests a shift in how compound AI systems should be architected. The separation of generation, reflection, and curation mirrors software engineering principles of separation of concerns, applied to LLM workflows.

**Design Principle 1: Stateful Context Management.** Traditional LLM applications treat context as ephemeral—constructed fresh for each query. ACE demonstrates that **persistent, versioned context** enables cumulative improvement. This suggests future systems should include dedicated context storage and versioning infrastructure, similar to how databases manage application state.

**Design Principle 2: Feedback-Driven Adaptation.** ACE's reliance on execution feedback (Section 4.3) points toward a broader pattern: LLM systems should be designed with **instrumentation for self-evaluation** from the start. This includes capturing execution traces, test results, and user feedback as first-class inputs to the adaptation loop.

**Design Principle 3: Interpretability as a Feature.** Section 5 highlights that human-readable contexts enable selective unlearning and expert review. This positions interpretability not as a constraint but as an **operational capability**—allowing domain experts to audit, modify, and curate system knowledge without model retraining.

**Long-Term Vision:**

> "ACE could play a central role in advancing continuous and responsible learning." (Section 5)

If ACE's approach generalizes, future LLM systems may ship with empty playbooks that accumulate domain expertise through deployment, similar to how humans learn through experience. This shifts the deployment model from "train once, deploy forever" to "deploy and continuously improve," with the critical difference that improvements are interpretable and auditable rather than embedded in opaque weights.

### 7.6 Summary: The Path Forward

ACE establishes that comprehensive, evolving contexts are both technically feasible and practically valuable for complex LLM applications. The framework's key contributions—incremental delta updates, modular reflection, and execution-feedback adaptation—provide a foundation for self-improving systems that scale with long-context models.

**Immediate Next Steps for Researchers:**

*   Validate ACE's stability over extended adaptation horizons (thousands of episodes)
*   Explore heterogeneous model assignments across Generator, Reflector, and Curator roles
*   Develop efficient retrieval indexing for large-scale playbook management

**Immediate Next Steps for Practitioners:**

*   Assess whether your domain provides verifiable execution feedback
*   Evaluate infrastructure support for long-context serving (KV cache optimization)
*   Start with offline warmup before deploying online adaptation (Table 3 shows +3.4% gain)

**Open Questions:**

*   What is the maximum effective playbook size before retrieval becomes a bottleneck?
*   Can ACE's approach transfer across domains, or must playbooks be built from scratch for each application?
*   How does ACE interact with other adaptation methods (e.g., fine-tuning, retrieval-augmented generation)?

The paper's transparency about limitations (Appendix B, Section 4.4) provides a realistic foundation for building on this work. ACE is not a universal solution, but for the right applications—complex, multi-episode domains with verifiable feedback—it represents a significant step toward scalable, self-improving LLM systems.
