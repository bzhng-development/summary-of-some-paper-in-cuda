## 1. Executive Summary
Kimi K2.5 is an open-source multimodal agentic model that advances general agentic intelligence through joint text-vision optimization and a novel parallel agent orchestration framework called Agent Swarm. It solves the latency and scalability bottlenecks of sequential agent execution by dynamically decomposing tasks into heterogeneous sub-problems, reducing inference latency by up to 4.5× while improving item-level F1 from 72.8% to 79.0% on WideSearch. The model achieves state-of-the-art results across domains including coding (76.8% on SWE-Bench Verified) and reasoning (96.1% on AIME 2025), demonstrating that visual reinforcement learning can enhance textual performance (e.g., +2.1% on GPQA-Diamond) rather than degrade it.

## 2. Context and Motivation

To understand why Kimi K2.5 matters, we need to examine two converging challenges in modern AI systems: the **multimodal alignment problem** and the **agentic scalability bottleneck**. These are not isolated issues—they interact in ways that limit what current models can achieve in real-world applications.

### The Multimodal Alignment Problem

**What is the core tension?** When building models that process both text and images, practitioners face a fundamental design question: how should vision and language capabilities be integrated during training? The conventional wisdom, established by prior vision-language models like Qwen3-VL and Seed1.5-VL, treats multimodal capability as a **post-hoc add-on** to linguistic competence. The standard approach introduces vision tokens predominantly in the later stages of training at high ratios (50% or higher), under the assumption that this accelerates multimodal capability acquisition once the model has mastered language.

**Why does this approach fall short?** Kimi K2.5's experiments reveal a counterintuitive finding documented in **Table 1** and **Figure 9**: late-stage vision injection actually degrades final performance. When vision data is introduced late, the model experiences what the authors call a "dip-and-recover" pattern—text capability initially degrades before gradually recovering. This happens because the sudden introduction of vision tokens disrupts the established linguistic representation space, forcing the model to temporarily sacrifice text-specific competence for cross-modal alignment. The paper attributes this to **modality domain shift**: the model must reorganize its internal representations to accommodate a new input type, creating instability.

**The theoretical significance:** This finding challenges the assumption that vision and language are separable competencies that can be stacked. Instead, Kimi K2.5 demonstrates that **early fusion with lower vision ratios** (10%:90% vision-to-text) yields better results when total token budgets are fixed. This suggests that unified multimodal representations must be cultivated from the outset, not grafted on later.

### The Agentic Scalability Bottleneck

**What is the operational constraint?** Existing agentic models—including advanced systems like Kimi K2-Thinking, Claude Opus 4.5, and GPT-5.2—rely on **sequential execution** of reasoning and tool-calling steps. While effective for simple, short-horizon tasks, this architecture creates a fundamental scalability problem: inference time scales linearly with task complexity. As the paper states in **Section 1**:

> "Even systems capable of hundreds of reasoning steps, such as Kimi K2-Thinking [1], suffer from linear scaling of inference time, leading to unacceptable latency and limiting task complexity."

**Why does this matter in practice?** Consider a real-world software engineering task that requires researching multiple documentation sources, designing an architecture, implementing code across several files, and testing the result. A sequential agent must complete each step before beginning the next. If each step takes 30 seconds and the task requires 50 steps, the total latency exceeds 25 minutes—and this grows worse as tasks become more complex. The paper identifies this as the primary barrier preventing agentic systems from handling "broad information gathering and intricate, multi-branch reasoning."

**Prior approaches and their limitations:** Several strategies have been attempted to address sequential bottlenecks:

| Approach | Mechanism | Limitation |
|----------|-----------|------------|
| Context truncation (Discard-all, Summary) | Remove or compress old context when limits are reached | Reactive rather than proactive; sacrifices structural information |
| Pre-specified parallelization | Hard-code which tasks can run in parallel | Inflexible; cannot adapt to task structure |
| End-to-end multi-agent co-optimization | Train all agents jointly | Suffers from credit assignment ambiguity and training instability |

The paper explicitly notes in **Section 3** that end-to-end co-optimization faces two fundamental challenges: **credit assignment ambiguity** (a correct final answer does not guarantee flawless subagent execution) and **training instability** (outcome-based rewards are sparse and noisy in multi-agent settings).

### The Cold-Start Problem for Multimodal Agents

**A hidden challenge:** Pretrained vision-language models do not naturally perform vision-based tool-calling. This creates a **cold-start problem** for multimodal reinforcement learning. Conventional approaches address this through manually annotated or prompt-engineered chain-of-thought data, but such methods are "limited in diversity, often restricting visual reasoning to simple diagrams and primitive tool manipulations (crop, rotate, flip)" (**Section 2.2**).

**Why is this a bottleneck?** High-quality text SFT (Supervised Fine-Tuning) data is relatively abundant and diverse, but high-quality vision-specific trajectories are scarce. This asymmetry creates a training imbalance: models can be extensively fine-tuned on text reasoning but struggle to generalize those capabilities to visual domains.

### How Kimi K2.5 Positions Itself

Kimi K2.5 positions itself as a **unified architecture for general-purpose agentic intelligence** that addresses all three challenges simultaneously:

1. **Native multimodal pre-training:** Rather than treating vision as an add-on, Kimi K2.5 mixes text and vision tokens with a constant ratio throughout the entire training process, starting early with moderate vision ratios. This builds on the Kimi-VL architecture but extends it with the MoonViT-3D encoder that shares parameters between image and video processing.

2. **Zero-vision SFT:** The paper introduces a novel technique where text-only SFT data activates visual, agentic capabilities during post-training. This works because joint pretraining already establishes strong vision-text alignment, enabling capabilities to generalize naturally across modalities. This directly addresses the cold-start problem without requiring scarce vision-specific trajectories.

3. **Agent Swarm with PARL:** Instead of sequential execution or brittle end-to-end co-optimization, Kimi K2.5 introduces a decoupled architecture with a trainable orchestrator and frozen subagents. The Parallel-Agent Reinforcement Learning (PARL) paradigm updates only the orchestrator while treating subagent outputs as environmental observations. This circumvents credit assignment ambiguity while enabling dynamic task decomposition.

4. **Proactive context management:** Unlike reactive truncation strategies, Agent Swarm enables **context sharding**—long-horizon tasks are decomposed into parallel, semantically isolated subtasks, each executed by a specialized subagent with bounded local context. Only task-relevant outputs are routed back to the orchestrator, preserving reasoning integrity while scaling effective context length.

### Real-World Impact

The significance of these contributions extends beyond benchmark scores. The paper demonstrates Agent Swarm on a concrete workload: analyzing 24 hours of continuous gameplay across 32 videos (40GB total) from *Black Myth: Wukong* (**Figure 11**). A sequential agent would require prohibitive latency to process this content. Agent Swarm completes the task by spawning 32 subagents in parallel, each analyzing one video segment, then aggregating results into a comprehensive HTML showcase.

Similarly, the cross-modal transfer finding—that visual RL improves textual performance on MMLU-Pro (+1.7%) and GPQA-Diamond (+2.1%) as shown in **Table 2**—suggests that multimodal training need not compromise language capabilities. This has implications for how the field approaches model specialization versus generalization.

By open-sourcing the post-trained checkpoints, Kimi K2.5 enables the research community to build on this foundation rather than treating these capabilities as proprietary. This positioning is significant given that competing state-of-the-art models (GPT-5.2, Claude Opus 4.5, Gemini 3 Pro) remain closed-source.

## 3. Technical Approach

This technical report presents a training methodology and system architecture for building general-purpose agentic intelligence, with the core idea that joint text-vision optimization combined with parallel agent orchestration can simultaneously improve multimodal capabilities and reduce inference latency.

### 3.1 Reader orientation (approachable technical breakdown)

Kimi K2.5 is a multimodal language model system that can see images and videos, reason about them, and coordinate multiple AI agents working in parallel to solve complex tasks. The system solves two interconnected problems: how to train a model that genuinely understands both text and vision without one modality compromising the other, and how to execute complex multi-step tasks without waiting for each step to complete sequentially.

### 3.2 Big-picture architecture (diagram in words)

The Kimi K2.5 system consists of four major components arranged in a training-to-inference pipeline. First, the **MoonViT-3D vision encoder** processes images and videos at native resolution, converting visual input into token embeddings that share a unified representation space with text. Second, the **Kimi K2 MoE language backbone** (1.04 trillion total parameters, 32 billion activated per forward pass) receives these visual tokens interleaved with text tokens and performs reasoning and generation. Third, the **Unified Agentic RL Environment** provides the training infrastructure for reinforcement learning, supporting both single-agent and multi-agent (Agent Swarm) paradigms with tool execution sandboxes. Fourth, the **Agent Swarm Orchestrator** operates at inference time, dynamically creating frozen sub-agents, decomposing tasks into parallel sub-problems, and aggregating results while managing context across all agents. Information flows from visual/text input through the vision encoder, into the language model for reasoning, through tool interfaces for action execution, and back through the orchestrator for result synthesis.

### 3.3 Roadmap for the deep dive

- We begin with the foundation model architecture because understanding the parameter scale and mixture-of-experts design establishes what capabilities the system has before any specialized training.
- We then examine the native multimodal pre-training strategy because this determines how vision and language representations become aligned—the prerequisite for all downstream multimodal behavior.
- We proceed to post-training methods (zero-vision SFT and joint RL) because these activate the agentic capabilities that pre-training alone does not produce.
- We analyze the Agent Swarm and PARL framework next because this represents the inference-time architecture that distinguishes K2.5 from sequential agent systems.
- We conclude with training infrastructure (Decoupled Encoder Process) because this explains how the system achieves efficient training at scale despite multimodal input variability.

### 3.4 Detailed, sentence-based technical breakdown

**Foundation Model Architecture**

Kimi K2.5 builds upon Kimi K2, a trillion-parameter mixture-of-experts transformer that serves as the language backbone for all multimodal and agentic capabilities. The model contains 1.04 trillion total parameters but activates only 32 billion parameters per token through a sparse routing mechanism, achieving computational efficiency while maintaining representational capacity. The architecture employs 384 experts with 8 experts activated per token, yielding a sparsity ratio of 48 (meaning only 1/48th of expert parameters are used per forward pass). Training stability is maintained through the MuonClip optimizer with QK-Clip, which prevents gradient explosion in deep transformer layers by clipping query-key attention scores. The multimodal extension adds two components to this foundation: MoonViT-3D, a three-dimensional native-resolution vision encoder, and an MLP projector that maps visual embeddings into the language model's token embedding space. MoonViT-3D is initialized from SigLIP-SO-400M and incorporates the NaViT patch packing strategy, which allows images of varying resolutions to be processed without complex sub-image splitting operations. For video understanding, the system groups up to four consecutive frames into a spatiotemporal volume, applies the shared MoonViT encoder across all frames, and performs temporal averaging at the patch level before the MLP projector. This design achieves 4× temporal compression, enabling the model to process videos up to four times longer within the same context window while maintaining complete weight sharing between image and video encoders. The context window extends to 256k tokens during evaluation, with long-context activation achieved through YaRN interpolation during mid-training.

**Native Multimodal Pre-Training Strategy**

The pre-training pipeline processes approximately 15 trillion mixed visual and text tokens across three distinct stages, each with specific objectives and data compositions. The first stage performs standalone ViT training on 1 trillion tokens, updating only the MoonViT-3D encoder to align it with the Moonlight-16B-A3B language model via caption generation loss, consuming minimal training FLOPs while establishing high-resolution image and video understanding. The second stage conducts joint pre-training over 15 trillion vision-text tokens at 4,096 sequence length, continuing from a near-end Kimi K2 checkpoint while introducing vision data with controlled ratios and data source epoch limits. The third stage performs long-context mid-training with 500 billion to 200 billion tokens, sequentially extending context length from 32,768 to 262,144 tokens through YaRN interpolation to refine long-context text understanding and long video comprehension. A critical design decision involves the vision-to-text token ratio and injection timing during joint pre-training. Conventional approaches introduce vision tokens predominantly in later training stages at high ratios (50% or higher), treating multimodal capability as a post-hoc addition to linguistic competence. Kimi K2.5's ablation studies, documented in **Table 1**, test three configurations: early fusion with 10%:90% vision-to-text ratio, mid fusion with 20%:80% ratio at 50% training completion, and late fusion with 50%:50% ratio at 80% training completion, all under fixed total vision-text token budgets. The results show that early fusion with lower vision ratios yields superior performance across all measured capabilities: vision knowledge (25.8 vs. 25.0 vs. 24.2), vision reasoning (43.8 vs. 40.7 vs. 39.0), OCR (65.7 vs. 64.1 vs. 61.5), text knowledge (45.5 vs. 43.9 vs. 43.1), text reasoning (58.5 vs. 58.6 vs. 57.8), and code (24.8 vs. 24.0 vs. 24.0). The paper attributes this to modality domain shift: when vision data is introduced late, the model experiences a "dip-and-recover" pattern where text capability initially degrades before gradually recovering, as shown in **Figure 9**. Early fusion avoids this shock by co-optimizing vision and language from the outset, allowing unified multimodal representations to evolve naturally without late-stage domain migration. The data recipe extends Kimi K2's pre-training distribution by introducing unique tokens, adjusting data proportions with increased weight on coding-related content, and controlling maximum epochs per data source to prevent overfitting.

**Zero-Vision Supervised Fine-Tuning**

Post-training begins with a novel technique called zero-vision SFT, which uses only text SFT data to activate visual and agentic capabilities during the supervised fine-tuning phase. This approach addresses the cold-start problem for multimodal reinforcement learning: pretrained vision-language models do not naturally perform vision-based tool-calling, and conventional approaches rely on manually annotated or prompt-engineered chain-of-thought data that is limited in diversity. Zero-vision SFT works because joint pre-training already establishes strong vision-text alignment, enabling capabilities learned on text to generalize naturally across modalities. All image manipulations are proxied through programmatic operations in IPython, effectively serving as a generalization of traditional vision tool-use. This activation enables diverse reasoning behaviors including pixel-level operations such as object size estimation via binarization and counting, and generalizes to visually grounded tasks such as object localization, counting, and OCR. The paper explicitly notes that adding human-designed visual trajectories at this stage hurts generalization, and preliminary experiments show that text-vision SFT yields much worse performance on visual agentic tasks, possibly due to the lack of high-quality vision data. The SFT data is synthesized from K2, K2 Thinking, and proprietary in-house expert models, employing specialized pipelines tailored to specific domains with human annotation, advanced prompt engineering, and multi-stage verification. This produces a large-scale instruction-tuning dataset featuring diverse prompts and intricate reasoning trajectories, training the model to prioritize interactive reasoning and precise tool-calling for complex real-world applications.

**Joint Multimodal Reinforcement Learning**

Following zero-vision SFT, the model undergoes joint multimodal reinforcement learning to refine its ability to incorporate visual inputs into reasoning and to activate agentic capabilities. The RL framework employs outcome-based rewards on tasks that explicitly require visual comprehension for correct solutions, categorized into three domains: visual grounding and counting (accurate localization and enumeration of objects within images), chart and document understanding (interpretation of structured visual information and text extraction), and vision-critical STEM problems (mathematical and scientific questions filtered to require visual inputs). The policy optimization objective departs from standard PPO clipping by introducing a token-level clipping mechanism designed to mitigate off-policy divergence amplified by discrepancies between training and inference frameworks. The loss function is defined as:

$$L_{RL}(\theta) = \mathbb{E}_{x \sim \mathcal{D}} \left[ \frac{1}{N} \sum_{j=1}^{K} \sum_{i=1}^{|y_j|} \text{Clip}\left( \frac{\pi_\theta(y_i^j|x,y_{0:i}^j)}{\pi_{\text{old}}(y_i^j|x,y_{0:i}^j)}, \alpha, \beta \right) (r(x,y_j) - \bar{r}(x)) - \tau \left( \log \frac{\pi_\theta(y_i^j|x,y_{0:i}^j)}{\pi_{\text{old}}(y_i^j|x,y_{0:i}^j)} \right)^2 \right]$$

where $x$ is the problem sampled from dataset $\mathcal{D}$, $y_j$ represents the $j$-th response among $K$ generated responses, $y_{0:i}^j$ is the prefix up to the $i$-th token of the $j$-th response, $N = \sum_{i=1}^{K} |y_i|$ is the total number of generated tokens in a batch, $\bar{r}(x) = \frac{1}{K} \sum_{j=1}^{K} r(x,y_j)$ is the mean reward of all generated responses, and $\alpha, \beta, \tau > 0$ are hyperparameters controlling the clipping bounds and KL penalty. The clipping mechanism functions as a gradient masking scheme: policy gradients are computed normally for tokens with log-ratios within the interval $[\alpha, \beta]$, while gradients for tokens falling outside this range are zeroed out. This approach aligns with recent strategies proposed to stabilize large-scale RL training and is empirically essential for maintaining training stability in complex domains requiring long-horizon, multi-step tool-use reasoning. A critical finding from this phase is that visual RL enhances textual performance rather than degrading it, as documented in **Table 2**: MMLU-Pro improves from 84.7% to 86.4% (+1.7%), GPQA-Diamond improves from 84.3% to 86.4% (+2.1%), and LongBench v2 improves from 56.7% to 58.9% (+2.2%). Analysis suggests that visual RL enhances calibration in areas requiring structured information extraction, reducing uncertainty on queries that resemble visually grounded reasoning such as counting and OCR. The joint multimodal RL paradigm organizes RL domains not by input modality but by abilities—knowledge, reasoning, coding, agentic, etc.—with domain experts jointly learning from both pure-text and multimodal queries while the Generative Reward Model optimizes across heterogeneous traces without modality barriers.

**Token-Efficient Reinforcement Learning via Toggle**

Token efficiency is central to LLMs with test-time scaling, but rigid budget constraints create a length-overfitting phenomenon where models fail to generalize to higher compute scales. To address this, the paper proposes Toggle, a training heuristic that alternates between inference-time scaling and budget-constrained optimization. For learning iteration $t$, the reward function is defined by:

$$\tilde{r}(x,y) = \begin{cases} r(x,y) \cdot \mathbb{I}\left\{ \frac{1}{K} \sum_{i=1}^{K} r(x,y_i) &lt; \lambda \text{ or } |y_i| \leq \text{budget}(x) \right\} & \text{if } \lfloor t/m \rfloor \pmod{2} = 0 \text{ (Phase0)} \\ r(x,y) & \text{if } \lfloor t/m \rfloor \pmod{2} = 1 \text{ (Phase1)} \end{cases}$$

where $\lambda$ and $m$ are hyperparameters of the algorithm, $K$ is the number of rollouts per problem, and $\text{budget}(x)$ is the problem-dependent budget estimated from the $\rho$-th percentile of token lengths among correct responses:

$$\text{budget}(x) = \text{Percentile}(\{|y_j| \mid r(x,y_i) = 1, i = 1, \ldots, K\}, \rho)$$

The algorithm alternates between two optimization phases every $m$ iterations. Phase0 (budget limited phase) trains the model to solve problems within a task-dependent token budget, but this constraint is conditionally applied only when the model's mean accuracy for a given problem exceeds threshold $\lambda$ to prevent premature sacrifice of quality for efficiency. Phase1 (standard scaling phase) allows the model to generate responses up to the maximum token limit, encouraging the model to leverage computation for better inference-time scaling. This budget is estimated once at the beginning of training and remains fixed thereafter. Toggle functions as a stochastic alternating optimization for a bi-objective problem, specifically designed to reconcile reasoning capabilities with computational efficiency. Evaluation on K2 Thinking shows consistent output length reduction across nearly all benchmarks, with Toggle decreasing output tokens by 25–30% on average with negligible impact on performance. The mechanism also reduces redundant patterns in chain-of-thought reasoning, such as repeated verifications and mechanical calculations, and shows strong domain generalization: when trained exclusively on mathematics and programming tasks, the model still achieves consistent token reductions on GPQA and MMLU-Pro with only marginal degradation in performance.

**Agent Swarm and Parallel-Agent Reinforcement Learning**

Agent Swarm is a dynamic framework for parallel agent orchestration that departs from traditional sequential agent execution by enabling complex tasks to be decomposed into heterogeneous sub-problems executed concurrently by domain-specialized agents. The PARL framework adopts a decoupled architecture comprising a trainable orchestrator and frozen subagents instantiated from fixed intermediate policy checkpoints, deliberately avoiding end-to-end co-optimization to circumvent credit assignment ambiguity and training instability. In this multi-agent setting, outcome-based rewards are inherently sparse and noisy: a correct final answer does not guarantee flawless subagent execution, just as a failure does not imply universal subagent error. By freezing the subagents and treating their outputs as environmental observations rather than differentiable decision points, the system disentangles high-level coordination logic from low-level execution proficiency, leading to more robust convergence. The PARL reward function is defined as:

$$r_{\text{PARL}}(x,y) = \lambda_1 \cdot r_{\text{parallel}} + \lambda_2 \cdot r_{\text{finish}} + r_{\text{perf}}(x,y)$$

where $r_{\text{perf}}$ evaluates the overall success and quality of the solution $y$ for a given task $x$, $r_{\text{parallel}}$ is an instantiation reward introduced to mitigate serial collapse (a local optimum where the orchestrator defaults to single-agent execution), and $r_{\text{finish}}$ focuses on successful completion of assigned subtasks to prevent spurious parallelism (a reward-hacking behavior where the orchestrator spawns many subagents without meaningful task decomposition). The hyperparameters $\lambda_1$ and $\lambda_2$ are annealed to zero over the course of training to ensure the final policy optimizes for the primary objective. To measure computational time cost in a parallel-agent setting, the system defines critical steps by analogy to the critical path in a computation graph. An episode is modeled as a sequence of execution stages indexed by $t = 1, \ldots, T$, where in each stage the main agent executes an action corresponding to either direct tool invocation or instantiation of a group of subagents running in parallel. Let $S_{\text{main}}^{(t)}$ denote the number of steps taken by the main agent in stage $t$ (typically $S_{\text{main}}^{(t)} = 1$), and $S_{\text{sub},i}^{(t)}$ denote the number of steps taken by the $i$-th subagent in that parallel group. The duration of stage $t$ is governed by the longest-running subagent within that cohort, and the total critical steps for an episode are defined as:

$$\text{CriticalSteps} = \sum_{t=1}^{T} \left( S_{\text{main}}^{(t)} + \max_{i} S_{\text{sub},i}^{(t)} \right)$$

By constraining training and evaluation using critical steps rather than total steps, the framework explicitly incentivizes effective parallelization: excessive subtask creation that does not reduce the maximum execution time of parallel groups yields little benefit, while well-balanced task decomposition that shortens the longest parallel branch directly reduces critical steps. The orchestrator is equipped with two specialized tools for sub-agent creation and scheduling: `create_subagent` instantiates a specialized sub-agent with a custom system prompt and identifier for reuse across tasks, and `assign_task` dispatches assignments to created sub-agents. During training, sub-agents are frozen and their execution trajectories are excluded from the optimization objective; only the orchestrator is updated via reinforcement learning. To improve efficiency, the system first trains the orchestrator using small-size subagents before transitioning to larger models, and the RL framework supports dynamically adjusting the inference instance ratios between subagents and the orchestrator to maximize resource usage across the cluster.

**Training Infrastructure: Decoupled Encoder Process**

Kimi K2.5 inherits the training infrastructure from Kimi K2 with minimal modifications, but introduces Decoupled Encoder Process (DEP) to address load imbalance caused by multimodal input variability during pipeline parallel training. In typical multimodal training utilizing Pipeline Parallelism, the vision encoder and text embedding are co-located in the first stage of the pipeline (Stage-0), but due to inherent variations in multimodal input size (e.g., image counts and resolutions), Stage-0 suffers from drastic fluctuations in both computational load and memory usage. DEP leverages the unique topological position of the visual encoder within the computation graph—specifically, its role as the start of the forward pass and the end of the backward pass—and is composed of three stages in each training step. First, Balanced Vision Forward executes the forward pass for all visual data in the global batch: because the vision encoder is small, it is replicated on all GPUs regardless of other parallelism strategies, and during this phase the forward computational workload is evenly distributed across all GPUs based on load metrics such as image or patch counts, eliminating load-imbalance caused by Pipeline Parallelism and visual token counts. To minimize peak memory usage, all intermediate activations are discarded, retaining only the final output activations, and results are gathered back to PP Stage-0. Second, Backbone Training performs the forward and backward passes for the main transformer backbone: by discarding intermediate activations in the preceding phase, the system can fully leverage any efficient parallel strategies validated in pure text training, and after this phase gradients are accumulated at the visual encoder output. Third, Vision Recomputation and Backward re-computes the vision encoder forward pass, followed by a backward pass to compute gradients for parameters in the vision encoder. DEP not only achieves load-balance but also decouples the optimization strategy of the vision encoder and the main backbone, enabling K2.5 to seamlessly inherit the parallel strategy of K2 and achieve a multimodal training efficiency of 90% relative to text-only training. The system is trained on NVIDIA H800 GPU clusters with 8×400 Gbps RoCE interconnects across nodes, employing a flexible parallelism strategy combining 16-way Pipeline Parallelism with virtual stages, 16-way Expert Parallelism, and ZeRO-1 Data Parallelism, enabling training on any number of nodes that is a multiple of 32. Expert Parallelism all-to-all communication is overlapped with computation under interleaved 1F1B scheduling, and to fit activations within GPU memory constraints, the system applies selective recomputation for LayerNorm, SwiGLU, and MLA up-projections, compresses insensitive activations to FP8-E4M3, and offloads remaining activations to CPU with overlapped streaming.

**Reward Function Design**

The reinforcement learning framework employs multiple reward function types tailored to different task categories and evaluation requirements. For tasks with verifiable solutions such as reasoning and agentic tasks, the system applies a rule-based outcome reward, and to optimize resource consumption, it incorporates a budget-control reward aimed at enhancing token efficiency. For general-purpose tasks, Generative Reward Models provide granular evaluations aligned with Kimi's internal value criteria, functioning not as binary adjudicators but as fine-grained evaluators capturing nuanced preference gradients including helpfulness, response readiness, contextual relevance, appropriate level of detail, aesthetic quality of generated artifacts, and strict instruction following. For visual tasks, the system designs task-specific reward functions to provide fine-grained supervision: for visual grounding and point localization tasks, an F1-based reward with soft matching is employed where grounding tasks derive soft matches from Intersection over Union and point tasks derive soft matches from Gaussian-weighted distances under optimal matching; for polygon segmentation tasks, the predicted polygon is rasterized into a binary mask and segmentation IoU is computed against the ground-truth mask to assign the reward; for OCR tasks, normalized edit distance quantifies character-level alignment between predictions and ground-truth; for counting tasks, rewards are assigned based on the absolute difference between predictions and ground-truth. To mitigate reward hacking and overfitting to a single preference signal, the system employs multiple alternative GRM rubrics tailored to different task contexts, and for complex visual puzzle problems, it synthesizes problems and utilizes an LLM verifier (Kimi K2) to provide feedback.

## 4. Key Insights and Innovations

Kimi K2.5 introduces several contributions that challenge established assumptions in multimodal learning and agentic systems. This section distinguishes between incremental engineering improvements and fundamental innovations that reshape how we think about model training and agent orchestration.

### 4.1 Zero-Vision SFT: Activating Vision Without Vision Data

**What makes this novel:** The conventional approach to multimodal post-training assumes that visual capabilities require visual training data. Kimi K2.5 demonstrates the counterintuitive finding that **text-only supervised fine-tuning can activate visual reasoning capabilities** when preceded by joint text-vision pre-training. As stated in **Section 2.2**:

> "We find that adding human-designed visual trajectories at this stage hurts generalization. In contrast, text-only SFT performs better—likely because joint pretraining already establishes strong vision-text alignment, enabling capabilities to generalize naturally across modalities."

**Why this matters theoretically:** This finding challenges the assumption that modality-specific fine-tuning data is necessary for modality-specific capabilities. The mechanism relies on **representation transfer**: joint pre-training creates a unified embedding space where reasoning patterns learned on text can generalize to vision without explicit visual supervision. This is analogous to how humans can apply abstract reasoning skills across different sensory domains.

**Prior work limitation:** Previous vision-language models (Qwen3-VL, Seed1.5-VL) rely on manually annotated or prompt-engineered chain-of-thought data for visual tool-use, which the paper notes is "limited in diversity, often restricting visual reasoning to simple diagrams and primitive tool manipulations (crop, rotate, flip)" (**Section 2.2**). This creates a data bottleneck: high-quality text SFT data is abundant, but high-quality vision-specific trajectories are scarce.

**Significance:** Zero-vision SFT transforms a data scarcity problem into a training strategy problem. The performance curves in **Figure 2** show that starting from minimal zero-vision SFT, scaling vision RL FLOPs produces continuous improvement across MMMU Pro, MathVision, CharXiv, and OCRBench. This suggests the bottleneck was never the absence of visual SFT data, but rather the failure to leverage cross-modal alignment established during pre-training.

**Fundamental vs. incremental:** This is a **fundamental innovation** in post-training methodology. It changes the requirement structure for multimodal capability acquisition—from "need vision data for vision tasks" to "need aligned representations, then reasoning generalizes."

### 4.2 Bidirectional Cross-Modal Transfer: Visual RL Improves Text

**What makes this novel:** The standard assumption in multimodal training is that there exists a **trade-off** between modalities: improving vision capabilities comes at the cost of text performance, or vice versa. Kimi K2.5's results in **Table 2** directly contradict this:

| Benchmark | Before Vision-RL | After Vision-RL | Improvement |
|-----------|-----------------|-----------------|-------------|
| MMLU-Pro | 84.7 | 86.4 | +1.7 |
| GPQA-Diamond | 84.3 | 86.4 | +2.1 |
| LongBench v2 | 56.7 | 58.9 | +2.2 |

**Why this matters theoretically:** The paper's analysis suggests that "visual RL enhances calibration in areas requiring structured information extraction, reducing uncertainty on queries that resemble visually grounded reasoning (e.g., counting, OCR)" (**Section 2.3**). This implies that **visual reasoning trains structural reasoning patterns** that transfer back to text-only domains. The mechanism is not modality-specific capability, but rather **reasoning discipline**: visual tasks force precise, verifiable outputs, and this discipline generalizes to textual reasoning.

**Prior work limitation:** Most multimodal training pipelines treat text and vision as separate optimization objectives, often with modality-specific expert divisions. The paper notes that Kimi K2.5 departs from this by organizing "RL domains not by input modality but by abilities—knowledge, reasoning, coding, agentic, etc." (**Section 2.3**). This ability-centric organization enables cross-modal transfer rather than modality siloing.

**Significance:** This finding has implications for how the field approaches model specialization. If visual training enhances textual reasoning, then multimodal training should be viewed not as a compromise but as a **mutual enhancement strategy**. This justifies the computational cost of joint training beyond the obvious benefit of handling visual inputs.

**Fundamental vs. incremental:** This is a **fundamental insight** about the nature of reasoning capabilities. It suggests that reasoning is not modality-bound but rather a transferable skill that can be strengthened through cross-modal training—a claim that warrants further investigation in future research.

### 4.3 Decoupled Orchestrator Architecture: Solving Credit Assignment in Multi-Agent RL

**What makes this novel:** Multi-agent reinforcement learning faces a well-known challenge: **credit assignment ambiguity**. When multiple agents contribute to a final outcome, determining which agent deserves credit (or blame) for success (or failure) is inherently difficult. Kimi K2.5's Agent Swarm addresses this through a **decoupled architecture** where only the orchestrator is trainable while subagents remain frozen. As explained in **Section 3**:

> "By freezing the subagents and treating their outputs as environmental observations rather than differentiable decision points, we disentangle high-level coordination logic from low-level execution proficiency, leading to more robust convergence."

**Why this matters theoretically:** This design transforms a multi-agent RL problem into a **single-agent RL problem with a richer action space**. The orchestrator's actions include subagent creation and task delegation, but the subagents' internal decisions are treated as environment dynamics rather than policy decisions. This eliminates the gradient propagation challenges that plague end-to-end multi-agent training.

**Prior work limitation:** The paper explicitly identifies two challenges of end-to-end co-optimization in **Section 1**: "credit assignment ambiguity and training instability." Prior approaches either accept these challenges (training all agents jointly) or avoid them through pre-specified parallelization heuristics (hard-coding which tasks can run in parallel). Agent Swarm learns parallelization policies without suffering from these limitations.

**Significance:** The training curves in **Figure 4** show smooth convergence with increasing parallelism over training, demonstrating that the decoupled architecture enables stable learning of orchestration policies. This is not merely an engineering choice but a **structural solution** to a fundamental RL problem.

**Fundamental vs. incremental:** This is a **fundamental innovation** in multi-agent system design. It changes the optimization problem from "train N agents jointly" to "train one coordinator with N fixed tools," which is tractable where the former is not.

### 4.4 Critical Steps Metric: Incentivizing Effective Parallelization

**What makes this novel:** Simply measuring total computational steps does not capture the benefit of parallelization. Kimi K2.5 introduces the **CriticalSteps** metric, defined in **Section 3** as:

$$\text{CriticalSteps} = \sum_{t=1}^{T} \left( S_{\text{main}}^{(t)} + \max_{i} S_{\text{sub},i}^{(t)} \right)$$

This measures wall-clock time by taking the maximum of parallel subagent steps at each stage, analogous to the critical path in a computation graph.

**Why this matters theoretically:** This metric directly incentivizes **balanced task decomposition**. Creating many subagents that do not reduce the maximum execution time of parallel groups yields little benefit under this metric. The orchestrator learns to allocate work in a way that minimizes end-to-end latency rather than merely maximizing concurrency.

**Prior work limitation:** Existing agentic benchmarks typically measure total token usage or total steps, which do not distinguish between sequential and parallel execution. This creates a misalignment: a system could appear efficient by total step count while suffering from sequential bottlenecks.

**Significance:** The results in **Figure 8** demonstrate the practical impact: Agent Swarm achieves 3×–4.5× faster execution time compared to single-agent baselines as target Item-F1 increases from 30% to 70% on WideSearch. More importantly, the single agent's execution time grows from approximately 1.8× to over 7.0× the baseline as difficulty increases, whereas Agent Swarm maintains near-constant low latency (0.6×–1.6×). This shows that the CriticalSteps metric successfully incentivizes parallelization that **scales with task complexity**.

**Fundamental vs. incremental:** This is an **incremental innovation with fundamental implications**. The metric itself is a straightforward adaptation of critical path analysis, but its application to agent RL training objectives changes what the system optimizes for—from "minimize total work" to "minimize wall-clock time."

### 4.5 Early Fusion with Lower Vision Ratios: Challenging Multimodal Training Conventions

**What makes this novel:** The conventional wisdom in multimodal pre-training, established by models like Qwen3-VL and Seed1.5-VL, introduces vision tokens predominantly in later training stages at high ratios (50% or higher). Kimi K2.5's ablation studies in **Table 1** show that **early fusion with lower vision ratios (10%:90%) yields better results** across all measured capabilities when total token budgets are fixed.

**Why this matters theoretically:** The paper attributes this to **modality domain shift**: when vision data is introduced late, the model experiences a "dip-and-recover" pattern where text capability initially degrades before gradually recovering (**Appendix B.1**, **Figure 9**). Early fusion avoids this shock by co-optimizing vision and language from the outset, allowing unified multimodal representations to evolve naturally.

**Prior work limitation:** The assumption that vision should be treated as a post-hoc add-on to linguistic competence reflects a view of modalities as separable competencies. Kimi K2.5's results suggest that **unified representations must be cultivated from the start**, not grafted on later.

**Significance:** This finding has practical implications for training efficiency. If early fusion with lower ratios produces better results, then practitioners can avoid the computational waste of late-stage vision injection that temporarily degrades text capabilities. The performance differences in **Table 1** are consistent across all domains: vision knowledge (25.8 vs. 24.2), vision reasoning (43.8 vs. 39.0), OCR (65.7 vs. 61.5), text knowledge (45.5 vs. 43.1), text reasoning (58.5 vs. 57.8), and code (24.8 vs. 24.0).

**Fundamental vs. incremental:** This is a **fundamental innovation** in training strategy. It changes the temporal structure of multimodal pre-training from "language first, vision later" to "language and vision together from the beginning."

### 5.3 Agent Swarm: Latency and Performance Gains

The Agent Swarm evaluation provides the most direct evidence for the parallel orchestration claims. **Figure 8** shows execution time savings as task difficulty increases:

The key insight: **single-agent execution time grows superlinearly with task difficulty** (from 1.8× to 7.0× as target F1 increases from 30% to 70%), while **Agent Swarm maintains near-constant low latency** (0.6× to 1.6×). This demonstrates that parallelization benefits **scale with task complexity**—exactly what the CriticalSteps metric was designed to incentivize.

### 5.5 Critical Assessment: Do the Experiments Support the Claims?

**Strong Support**

1. **Agent Swarm reduces latency while improving performance:** The evidence is compelling. **Figure 8** shows 3×–4.5× speedup with increasing task complexity, and **Table 6** demonstrates consistent accuracy gains (6.3%–17.8%) across benchmarks. The experimental design directly measures both latency and accuracy, addressing the core claim.

2. **Early fusion with lower vision ratios outperforms late fusion:** **Table 1** and **Figure 9** provide clear ablation evidence under fixed token budgets. The "dip-and-recover" pattern documented in Appendix B.1 offers mechanistic explanation (modality domain shift), strengthening the causal claim.

3. **Visual RL improves textual performance:** **Table 2** shows consistent improvements across three text-only benchmarks after visual RL. This is a clean before/after comparison that directly tests the cross-modal transfer hypothesis.

**Moderate Support**

1. **Zero-vision SFT activates visual capabilities:** While **Figure 2** shows improving vision RL curves from zero-vision SFT starting points, the paper does not provide a direct comparison showing zero-vision SFT versus text-vision SFT on the same vision benchmarks with matched training budgets. The claim that text-vision SFT yields "much worse performance" is stated but not quantified with specific numbers.

2. **Token efficiency via Toggle:** **Figure 5** shows token reduction, but the comparison is against Kimi K2 Thinking rather than a non-Toggle variant of K2.5. A cleaner ablation would compare K2.5 with and without Toggle under identical conditions.

**Limitations and Missing Information**

1. **Proprietary model access constraints:** The paper acknowledges in **Appendix E.2** that GPT-5.2 API access was inconsistent, leading to skipped benchmarks (e.g., WideSearch for GPT-5.2) and potential underestimation of GPT-5.2 vision scores due to 10% failure rates. This limits the conclusiveness of some SOTA claims.

2. **In-house benchmark opacity:** The In-house Swarm Bench results (58.3% for Agent Swarm vs. 41.6% single-agent) show large gains, but the benchmark is not publicly available. Without external validation, these results cannot be independently verified.

3. **Compute budget disclosure:** The paper does not specify the total compute budget (FLOPs, GPU-hours) for training Kimi K2.5, making it difficult to assess efficiency relative to competing models. **Figure 2** mentions "RL flops" on the x-axis but does not provide absolute values.

4. **Failure cases:** The paper does not systematically document failure modes or error analysis. For instance, on ZeroBench, K2.5 achieves only 11% even with tools—understanding why the model fails on 89% of cases would provide insight into current limitations.

5. **Agent Swarm overhead:** While **Figure 8** shows wall-clock time savings, the paper does not report the computational cost (total GPU-hours, token consumption) of running Agent Swarm versus single-agent. Parallel execution reduces latency but may increase total compute cost—a trade-off that deserves explicit discussion.

## 6. Limitations and Trade-offs

No technical system is without constraints. Kimi K2.5's innovations come with specific assumptions, operational trade-offs, and unresolved questions that are important to understand before deploying or building upon this work. This section examines what the approach requires to function, where it may struggle, and what remains unknown.

### 6.1 Foundational Assumptions

**Joint Pre-Training as a Prerequisite**

The zero-vision SFT technique—one of the paper's most novel contributions—depends critically on strong vision-text alignment established during pre-training. The paper states in **Section 2.2**:

> "This phenomenon is likely due to the joint pretraining of text and vision data as described in Section 2.1."

**What this means in practice:** Zero-vision SFT is not a standalone technique that can be applied to any pretrained language model. It requires a foundation model that has already undergone extensive joint multimodal pre-training (15 trillion mixed vision-text tokens in K2.5's case). Practitioners cannot simply take a text-only LLM, apply zero-vision SFT, and expect visual capabilities to emerge. The representation transfer mechanism only works if the embedding space already unifies both modalities.

**Implication:** This limits the generalizability of the approach. Organizations without access to large-scale multimodal pre-training infrastructure cannot replicate the zero-vision SFT benefits, even if they adopt the post-training methodology.

**Frozen Subagents Require Stable Checkpoints**

The Agent Swarm architecture relies on subagents instantiated from "fixed intermediate policy checkpoints" (**Section 3**). This design choice solves credit assignment ambiguity but introduces a dependency:

- Subagents must be competent enough to execute delegated tasks reliably
- The orchestrator cannot adapt subagent behavior during training—only its own delegation policy is updated

**What could go wrong:** If the frozen subagent checkpoints are weak on certain task types, the orchestrator cannot teach them to improve. It can only learn to avoid delegating those tasks. This creates a **capability ceiling** determined by the weakest subagent in the swarm.

**The paper's mitigation:** Section 3 notes that training first uses "small-size subagents before transitioning to larger models," suggesting an iterative checkpoint selection process. However, the paper does not specify how subagent checkpoints are selected or validated, leaving this as an implementation detail that could significantly affect reproducibility.

**Reward Functions Must Be Verifiable**

The PARL reward function includes a task-level outcome reward $r_{\text{perf}}(x,y)$ that "evaluates the overall success and quality of the solution" (**Section 3**). For this to work:

- Tasks must have clear success criteria
- Outcome verification must be automated or semi-automated

**Limitation:** This constrains Agent Swarm to domains where outcomes are verifiable—mathematical problems, code execution, information retrieval with ground truth. Open-ended creative tasks (writing, design, strategic planning) lack clear verification signals, making them unsuitable for PARL training without significant reward model engineering.

### 6.2 Scenarios and Edge Cases Not Addressed

**Low-Resource Deployment**

The paper reports Kimi K2.5 as a 1.04 trillion parameter MoE model with 32 billion activated parameters per forward pass (**Section 4.1**). While sparse activation improves efficiency, the model still requires:

- Significant GPU memory for inference (even with MoE sparsity)
- High-bandwidth interconnects for expert parallelism
- Specialized infrastructure for the 256k token context window

**What's missing:** The paper does not discuss quantization, distillation, or other compression techniques that might enable deployment on consumer hardware. There is no mention of latency or throughput characteristics at different batch sizes, nor any guidance on minimum hardware requirements.

**Implication:** Kimi K2.5 is designed for cloud-scale deployment. Researchers or organizations without access to large GPU clusters cannot practically run the full model, limiting the real-world applicability of the open-sourced checkpoints.

**Single-Image and Short-Context Tasks**

The architecture emphasizes long-context understanding (256k tokens) and video processing (up to 2,000 frames via MoonViT-3D's 4× temporal compression). However, this focus may introduce inefficiencies for simpler tasks:

- **Table 5** shows K2.5 uses 14k–36k output tokens on reasoning benchmarks, compared to 7k–16k for Gemini-3.0 Pro and DeepSeek-V3.2 Thinking
- The MoonViT-3D encoder processes all images at native resolution, which may be overkill for low-resolution inputs

**Trade-off:** The system is optimized for complex, multi-step agentic workloads. For simple query-response tasks, the overhead of the full architecture (vision encoder, MoE routing, orchestrator logic) may not be justified.

**Real-Time and Latency-Sensitive Applications**

Agent Swarm reduces wall-clock time by 3×–4.5× on complex tasks (**Figure 8**), but this assumes:

- Sufficient parallel compute capacity is available to run all subagents concurrently
- Network overhead for inter-agent communication is negligible
- Subagent instantiation time is small relative to task execution time

**What's not measured:** The paper reports execution time savings but does not disclose:

- Total GPU-hours consumed (parallel execution may increase total compute cost)
- Cold-start latency for subagent instantiation
- Communication overhead between orchestrator and subagents

**Implication:** In resource-constrained environments where total compute cost matters more than wall-clock time, Agent Swarm may not provide net benefits.

**Non-English and Low-Resource Languages**

The benchmark suite in **Section 5.1.1** is predominantly English-language:

- Reasoning benchmarks (AIME, HMMT, GPQA) are English
- Coding benchmarks use English documentation and comments
- Agentic benchmarks (BrowseComp, WideSearch) query English-language web sources

**Exception:** SWE-Bench Multilingual (73.0% score) and OCRBench (92.3%) suggest some multilingual capability, but the paper does not provide a systematic evaluation across languages.

**Gap:** There is no discussion of how joint pre-training handles non-Latin scripts, right-to-left languages, or languages with limited web presence. The vision-text alignment learned primarily from English-centric data may not generalize equally across all languages.

### 6.3 Computational and Data Constraints

**Training Compute Not Disclosed**

The paper specifies data volumes (15 trillion vision-text tokens, 1T ViT training tokens, 500B–200B mid-training tokens in **Table 3**) but does not report:

- Total training FLOPs
- GPU-hours consumed
- Training duration
- Energy consumption

**Why this matters:** Without compute budgets, it is impossible to assess efficiency relative to competing models. A model that achieves SOTA results but requires 10× more compute than alternatives represents a different trade-off than one achieving similar results with comparable compute.

**Comparison point:** **Figure 2** shows "RL flops" on the x-axis for vision RL training curves, but these are normalized values without absolute scales. Readers cannot determine whether K2.5's RL phase is computationally intensive relative to its pre-training or relative to other models' RL phases.

**Vision Data Quality and Diversity**

The paper acknowledges in **Section 2.2** that text-vision SFT yields "much worse performance" than zero-vision SFT, "possibly because of the lack of high-quality vision data." This admission reveals a constraint:

- High-quality text SFT data is abundant and diverse
- High-quality vision-specific trajectories are scarce

**Implication:** The zero-vision SFT technique is partly a workaround for data scarcity, not purely a methodological preference. If high-quality vision trajectories were available at scale, the optimal training strategy might differ.

**Appendix B.3** describes the vision data corpus (caption, interleaving, OCR, knowledge, perception, video, agent data) but does not quantify the proportion of each category or provide quality metrics beyond "rigorous filtering, deduplication, and quality control."

**Agent Swarm Compute Overhead**

While **Figure 8** demonstrates wall-clock time savings, the paper does not report whether Agent Swarm increases total token consumption or GPU utilization:

| Metric | Reported | Not Reported |
|--------|----------|--------------|
| Wall-clock time | ✓ (3×–4.5× speedup) | — |
| Total tokens generated | — | ✓ |
| GPU-hours per task | — | ✓ |
| Subagent instantiation cost | — | ✓ |

**Trade-off:** Parallel execution reduces latency but may increase total compute cost if subagents generate redundant reasoning or if the orchestrator requires additional tokens to coordinate them. For cost-sensitive deployments, this trade-off is critical but unquantified.

### 6.4 Weaknesses and Open Questions

**ZeroBench Performance Remains Low**

On the challenging ZeroBench visual perception benchmark, Kimi K2.5 achieves only 9% without tools and 11% with tool augmentation (**Table 4**). While this exceeds competing models (Claude Opus 4.5: 3%, GPT-5.2: 9%), the absolute performance indicates:

- Fundamental limitations in pixel-level visual reasoning persist
- Tool augmentation provides marginal benefit (only +2 percentage points)

**Open question:** The paper does not analyze why the model fails on 89% of ZeroBench tasks. Is this due to visual perception limits, reasoning limits, or tool execution limits? Understanding failure modes would guide future improvements.

**Thinking Mode Incompatibilities**

**Appendix E.5** notes that Terminal Bench 2.0 was evaluated "under non-thinking mode because our current context management implementation for thinking mode is technically incompatible with Terminus-2's conversation state handling."

**What this reveals:** The thinking mode (extended chain-of-thought reasoning) and certain agent frameworks have integration issues. This suggests:

- The system architecture is not fully unified across all modes
- Some benchmark results may not reflect the model's full capabilities

**Implication:** Users deploying K2.5 in agentic settings may encounter similar compatibility issues that require workarounds or mode switching.

**Context Management Trade-Offs**

**Appendix E.6** states: "Unless otherwise specified below, no context management is applied to agentic evaluations; tasks exceeding the model's supported context window are directly counted as failures rather than truncated."

**Limitation:** This evaluation protocol is strict—it does not test how the model behaves when context management is required mid-task. The Hide-Tool-Result strategy for HLE and Discard-all strategy for BrowseComp are exceptions, not the default.

**Open question:** How does performance degrade as task length approaches and exceeds the 256k context window? The paper demonstrates capability within the window but does not characterize the failure boundary.

**Claude and GPT Evaluation Discrepancies**

**Appendix E.2** and **E.7** note several evaluation adjustments for fair comparison:

- GPT-5.2-xhigh had ~10% failure rate on vision evaluations (treated as incorrect)
- Some GPT-5.2 benchmarks were skipped due to API access limitations
- Claude Opus 4.5 was evaluated "solely with computer-use tools (excluding browser tools)" for OSWorld-Verified, departing from its System Card configuration

**Implication:** Some SOTA claims rest on comparisons where baseline models were not evaluated under their optimal configurations. While this ensures apples-to-apples comparison for the specific task, it may not reflect real-world performance differences.

**Reward Hacking Mitigation Not Fully Detailed**

The paper mentions employing "multiple alternative GRM rubrics tailored to different task contexts" to mitigate reward hacking (**Section 4.4.2**), but does not specify:

- How many alternative rubrics exist
- How rubrics are selected during training
- Whether reward hacking was observed during development

**Open question:** Without transparency on reward hacking incidents and mitigations, it is unclear how robust the RL training is to optimization pressure. This is a common opacity in RLHF/RLAIF work but worth noting.

### 6.5 Summary: Key Trade-Offs

| Trade-Off | Benefit | Cost |
|-----------|---------|------|
| Zero-vision SFT | Activates vision without scarce vision data | Requires extensive joint pre-training first |
| Frozen subagents in Agent Swarm | Solves credit assignment, stable training | Subagents cannot adapt; capability ceiling |
| Early fusion with low vision ratio | Better final performance, stable training | Vision capabilities develop more slowly initially |
| CriticalSteps optimization | Incentivizes effective parallelization | May not minimize total compute cost |
| 256k context window | Handles long documents and videos | High memory requirements, inefficient for short tasks |
| Outcome-based RL rewards | Clear optimization signal | Limited to verifiable tasks |

### 6.6 What Remains Unknown

The paper leaves several important questions unanswered:

1. **Compute efficiency:** What is the total training cost, and how does K2.5's performance-per-FLOP compare to alternatives?

2. **Failure analysis:** Why does the model fail on 89% of ZeroBench tasks? What are the dominant error patterns?

3. **Agent Swarm economics:** Does parallel execution increase or decrease total compute cost per task?

4. **Multilingual generalization:** How does performance vary across non-English languages and scripts?

5. **Long-term stability:** Do frozen subagents become outdated as tasks evolve, requiring periodic checkpoint updates?

6. **Reward model robustness:** How susceptible is the GRM-based reward system to adversarial exploitation or distribution shift?

These open questions represent opportunities for future research—and cautionary notes for practitioners considering deployment. The Kimi K2.5 technical report demonstrates impressive capabilities, but like all current AI systems, it operates within specific constraints that define its appropriate use cases.

## 7. Implications and Future Directions

Kimi K2.5 represents more than an incremental improvement in model capabilities—it introduces architectural and training paradigm shifts that reshape how the field approaches multimodal agentic intelligence. This section examines how the work changes the research landscape, what follow-up investigations it enables, where practical deployment makes sense, and guidance for practitioners considering adoption.

### 7.1 How This Work Changes the Field Landscape

**Challenging Multimodal Training Conventions**

The finding that **early fusion with lower vision ratios (10%:90%) outperforms late fusion (50%:50%)** under fixed token budgets (**Table 1**) directly contradicts established practice in vision-language model training. Prior work—including Qwen3-VL [8] and Seed1.5-VL [21]—treats multimodal capability as a post-hoc add-on, introducing vision tokens predominantly in later training stages. Kimi K2.5 demonstrates this approach is suboptimal:

> "Early fusion maintains a healthier and more stable text performance curve throughout training. By co-optimizing vision and language from the outset, the model naturally evolves unified multimodal representations without the shock of late-stage domain migration." (**Appendix B.1**)

**Implication:** Future multimodal models should reconsider their training schedules. The "language first, vision later" paradigm may be actively harmful to final capability, not merely inefficient.

**Redefining Cross-Modal Transfer**

The demonstration that **visual RL improves textual performance** (+1.7% on MMLU-Pro, +2.1% on GPQA-Diamond in **Table 2**) challenges the assumption of modality trade-offs. This suggests reasoning capabilities are not modality-bound but rather transferable skills strengthened through cross-modal training:

> "Visual RL enhances calibration in areas requiring structured information extraction, reducing uncertainty on queries that resemble visually grounded reasoning (e.g., counting, OCR)." (**Section 2.3**)

**Implication:** Multimodal training should be viewed as a **mutual enhancement strategy** rather than a compromise. This justifies the computational cost of joint training beyond the obvious benefit of handling visual inputs.

**Structural Solution to Multi-Agent RL**

The **decoupled orchestrator architecture** transforms multi-agent reinforcement learning from an intractable joint optimization problem into a tractable single-agent problem with a richer action space. By freezing subagents and treating their outputs as environmental observations, Agent Swarm circumvents credit assignment ambiguity without sacrificing adaptability:

> "By freezing the subagents and treating their outputs as environmental observations rather than differentiable decision points, we disentangle high-level coordination logic from low-level execution proficiency, leading to more robust convergence." (**Section 3**)

**Implication:** This architectural pattern may generalize beyond agent orchestration to any hierarchical RL setting where credit assignment is ambiguous—robotics, distributed control systems, or multi-stage decision pipelines.

**Open-Source Agentic Intelligence**

By releasing post-trained checkpoints, Kimi K2.5 makes state-of-the-art agentic capabilities accessible to the research community. Competing models at this capability level (GPT-5.2, Claude Opus 4.5, Gemini 3 Pro) remain proprietary. This has downstream effects:

- Researchers can build on the foundation without API access constraints
- Independent verification of claims becomes possible
- The open-source ecosystem gains a reference implementation for agentic architectures

**Implication:** This accelerates community progress toward General Agentic Intelligence by reducing the infrastructure barrier to entry.

### 7.2 Follow-Up Research Directions

**Zero-Vision SFT Generalization**

The zero-vision SFT technique activates visual capabilities using only text SFT data, relying on representation transfer from joint pre-training. This suggests several research questions:

| Research Question | Why It Matters | Feasibility |
|------------------|----------------|-------------|
| Can zero-vision SFT generalize to other modality pairs (e.g., audio-text, sensor-text)? | Extends the technique beyond vision | Requires joint pre-training foundation |
| What is the minimum joint pre-training required for zero-vision activation to work? | Reduces pre-training cost for downstream applications | Can be tested via ablation |
| Does zero-vision SFT work for domains without abundant text SFT data? | Determines boundary conditions | May require synthetic data generation |

**Agent Swarm Extensions**

The current Agent Swarm architecture uses frozen subagents instantiated from fixed checkpoints. Future work could explore:

- **Dynamic subagent training:** Allow subagents to adapt during orchestration while maintaining credit assignment clarity through hierarchical reward structures
- **Heterogeneous architectures:** Deploy subagents with different specializations (e.g., coding-focused, reasoning-focused, retrieval-focused) rather than homogeneous checkpoints
- **Cross-swarm communication:** Enable subagents to coordinate directly rather than routing all communication through the orchestrator

**CriticalSteps Metric Adoption**

The CriticalSteps metric incentivizes effective parallelization by measuring wall-clock time rather than total computation:

$$\text{CriticalSteps} = \sum_{t=1}^{T} \left( S_{\text{main}}^{(t)} + \max_{i} S_{\text{sub},i}^{(t)} \right)$$

This metric could be adopted by:

- **Agentic benchmark suites:** Add CriticalSteps as a standard evaluation metric alongside accuracy
- **RL training frameworks:** Incorporate CriticalSteps into reward functions for any parallel execution system
- **Cloud inference platforms:** Optimize scheduling algorithms using CriticalSteps-aware cost models

**Token Efficiency Mechanisms**

The Toggle heuristic reduces output tokens by 25–30% with negligible performance impact (**Figure 5**). Follow-up research could:

- Apply Toggle to non-reasoning domains (creative writing, dialogue, code generation)
- Investigate whether alternating optimization prevents length-overfitting in other contexts
- Explore adaptive budget estimation beyond the $\rho$-th percentile approach

**Failure Mode Analysis**

The paper does not systematically document why Kimi K2.5 fails on 89% of ZeroBench tasks despite outperforming competitors. Understanding these failure modes would:

- Identify bottlenecks in pixel-level visual reasoning
- Guide targeted improvements in perception modules
- Reveal whether failures stem from visual perception, reasoning, or tool execution limits

### 7.3 Practical Applications and Downstream Use Cases

**Long-Form Video Analysis**

The **Figure 11** demonstration—analyzing 24 hours of *Black Myth: Wukong* gameplay across 32 videos (40GB total)—illustrates a concrete use case where Agent Swarm provides unique value:

- **Media production:** Automated highlight reel generation, content tagging, chronological timeline creation
- **Security monitoring:** Parallel analysis of multiple camera feeds with event detection and aggregation
- **Educational content:** Lecture video indexing, key moment extraction, searchable transcript generation

**When to use:** Tasks requiring analysis of hours-long video content where sequential processing would exceed practical latency bounds.

**Software Engineering Workflows**

Kimi K2.5's strong performance on SWE-Bench Verified (76.8%), SWE-Bench Multilingual (73.0%), and LiveCodeBench v6 (85.0%) in **Table 4** indicates readiness for real-world coding assistance:

- **Code maintenance:** Automated bug fixing, refactoring, and test generation across repository-scale codebases
- **Security auditing:** Vulnerability detection (41.3 on CyberGym) in open-source projects
- **Documentation:** Generating documentation from code, creating usage examples, answering technical questions

**When to use:** Complex software engineering tasks requiring cross-file reasoning, architectural understanding, or security analysis. Single-agent mode suffices for simple queries; Agent Swarm excels for multi-file, multi-step workflows.

**Complex Research and Information Synthesis**

Agent Swarm's performance on BrowseComp (78.4% with Agent Swarm vs. 60.6% single-agent in **Table 6**) and WideSearch (79.0% vs. 72.7%) demonstrates capability for deep research tasks:

- **Literature reviews:** Parallel exploration of multiple academic databases with synthesis of findings
- **Market research:** Simultaneous analysis of competitors, trends, and customer feedback across sources
- **Due diligence:** Multi-source verification of claims, financial data aggregation, risk assessment

**When to use:** Information-gathering tasks requiring broad exploration across independent sources. The 3×–4.5× latency reduction (**Figure 8**) makes previously impractical workloads feasible.

**Computer Use Automation**

Performance on OSWorld-Verified (63.3%) and WebArena (58.9%) in **Table 4** indicates capability for GUI-based task automation:

- **Workflow automation:** Repetitive desktop tasks (data entry, form filling, report generation)
- **Web operations:** Automated browsing, form submission, data extraction from web interfaces
- **IT operations:** System administration tasks requiring GUI interaction

**When to use:** Tasks requiring interaction with graphical interfaces where API access is unavailable. Note that Claude Opus 4.5 still leads slightly (66.3% on OSWorld-Verified), so task-specific evaluation is recommended.

### 7.4 Reproduction and Integration Guidance

**When to Prefer Kimi K2.5 Over Alternatives**

| Use Case | Recommended Approach | Rationale |
|----------|---------------------|-----------|
| Complex agentic tasks with parallelizable subtasks | Kimi K2.5 with Agent Swarm | 3×–4.5× latency reduction, +6.3% to +17.8% accuracy gains |
| Long video understanding (>1 hour) | Kimi K2.5 with MoonViT-3D | 4× temporal compression, SOTA on LVBench (75.9%) and LongVideoBench (79.8%) |
| Software engineering across repositories | Kimi K2.5 single-agent or Agent Swarm | 76.8% on SWE-Bench Verified, strong cross-file reasoning |
| Simple query-response tasks | Consider lighter alternatives | K2.5's 1.04T parameter MoE may be overkill; token usage 14k–36k vs. 7k–16k for competitors (**Table 5**) |
| Non-English languages | Evaluate carefully | Benchmark suite is predominantly English; multilingual capability not systematically tested |
| Cost-sensitive deployments | Evaluate total compute cost | Agent Swarm reduces latency but may increase total GPU-hours; not quantified in paper |

**Infrastructure Requirements**

Based on **Section 4.1** and **Appendix C**:

- **Model size:** 1.04 trillion total parameters, 32 billion activated per forward pass (MoE sparsity of 48)
- **Context window:** 256k tokens during evaluation
- **Training infrastructure:** NVIDIA H800 GPU clusters with 8×400 Gbps RoCE interconnects, 16-way Pipeline Parallelism, 16-way Expert Parallelism, ZeRO-1 Data Parallelism
- **Inference considerations:** High-bandwidth interconnects required for expert parallelism; specialized infrastructure for 256k context window

**Practical constraint:** The model is designed for cloud-scale deployment. Researchers or organizations without access to large GPU clusters cannot practically run the full model, limiting real-world applicability despite open-sourced checkpoints.

**Integration Considerations**

**Agent Swarm Configuration:** From **Appendix E.8**, the orchestrator requires two specialized tools:

```json
{
  "name": "create_subagent",
  "parameters": {
    "name": "Unique name for this agent configuration",
    "system_prompt": "System prompt defining the agent's role, capabilities, and boundaries"
  }
}
{
  "name": "assign_task",
  "parameters": {
    "agent": "Specify which created agent to use",
    "prompt": "The task for the agent to perform"
  }
}
```

**Step limits:** BrowseComp constrains orchestrator to 15 steps with 100 steps per subagent; WideSearch allocates 100 steps to both orchestrator and subagents.

**Thinking mode compatibility:** **Appendix E.5** notes that thinking mode is "technically incompatible with Terminus-2's conversation state handling" for Terminal Bench 2.0. Users deploying K2.5 in agentic settings may encounter similar compatibility issues requiring workarounds or mode switching.

**Context management:** **Appendix E.6** states that "tasks exceeding the model's supported context window are directly counted as failures rather than truncated" unless specific context management strategies (Hide-Tool-Result, Discard-all) are applied. Deployments should implement context management proactively rather than relying on default behavior.

**Reproduction Checklist**

For researchers attempting to reproduce or build upon Kimi K2.5:

- [ ] Verify access to sufficient GPU infrastructure for 1.04T parameter MoE inference
- [ ] Confirm compatibility between thinking mode and target agent frameworks
- [ ] Implement context management strategies for tasks approaching 256k token limits
- [ ] Evaluate Agent Swarm overhead (total GPU-hours) for cost-sensitive applications
- [ ] Test performance on domain-specific tasks before production deployment
- [ ] Consider whether zero-vision SFT benefits apply (requires joint pre-training foundation)

### 7.5 Summary: The Path Forward

Kimi K2.5 demonstrates that scalable general agentic intelligence requires both **representational unity** (joint text-vision optimization) and **architectural parallelism** (Agent Swarm orchestration). The work's contributions extend beyond benchmark scores:

1. **Training paradigm shifts:** Early fusion, zero-vision SFT, and cross-modal transfer challenge established conventions
2. **Architectural innovations:** Decoupled orchestrator and CriticalSteps metric solve fundamental multi-agent RL problems
3. **Open-source acceleration:** Released checkpoints enable community building on state-of-the-art agentic capabilities

The open questions—compute efficiency, failure mode analysis, multilingual generalization, reward model robustness—represent opportunities for the research community to extend this foundation. By making these capabilities accessible, Kimi K2.5 positions itself not as an endpoint but as a platform for continued progress toward General Agentic Intelligence.
