## 1. Executive Summary

This paper presents Ray, a distributed framework designed to support emerging AI applications like reinforcement learning (RL) that require the tight coupling of simulation, training, and serving within a single system. Ray unifies task-parallel and actor-based computations on a dynamic execution engine backed by a sharded Global Control Store (GCS), demonstrating linear scalability beyond 1.8 million tasks per second with millisecond-level latencies (Section 5.1). The framework's impact is evidenced by RL benchmarks (Figure 14) where a Ray-based Evolution Strategies implementation solves the Humanoid-v1 task in 3.7 minutes, outperforming the previous best published result of 10 minutes while eliminating the need for specialized one-off system engineering.

## 2. Context and Motivation

To understand why Ray was built, we must first understand how AI workloads were evolving at the time of this paper's writing. The industry was transitioning from static data analysis to dynamic, interactive intelligence. This section breaks down the specific systemic gaps that emerged during this shift, why existing tools failed to address them, and the precise requirements a new system needed to meet.

### 2.1 The Shift from Static Data to Interactive Learning

For two decades prior to this work, distributed computing frameworks were optimized for "Big Data" analytics. Systems like MapReduce, Apache Spark, and Dryad were designed to process large, static datasets in batch modes (Section 1). Later, deep learning frameworks like TensorFlow and MXNet emerged to handle supervised learning, where the goal is to map data points to labels using deep neural networks. These systems focused on reducing training time in a **batch setting**, often leveraging specialized hardware like GPUs.

However, the paper argues that the promise of AI extends beyond classical supervised learning. Emerging applications, particularly in **Reinforcement Learning (RL)**, require systems that can:
*   Operate continuously in dynamic environments.
*   React to changes in real-time.
*   Take sequences of actions to accomplish long-term goals (Section 1).

In RL, an agent learns a **policy**—a mapping from the state of the environment to a choice of action. Unlike supervised learning, where data is provided upfront, RL agents must **explore** the space of possible actions and learn from delayed feedback (Section 1). This fundamental difference changes the systems requirements from "process this large file" to "interact, learn, and act continuously."

### 2.2 The Reinforcement Learning Loop

The core challenge Ray addresses is the tight coupling of three distinct workloads within a single RL application. As illustrated in **Figure 1**, an RL system consists of a continuous loop involving:

1.  **Simulation:** The agent interacts with an environment (often a simulator) to generate **trajectories**. A trajectory is a sequence of state-reward tuples $(s_0, (s_1, r_1), \dots, (s_n, r_n))$ produced by the current policy. Simulations can vary wildly in duration, from milliseconds (e.g., a chess move) to minutes (e.g., a self-driving car scenario) (Section 2).
2.  **Training:** The system uses the generated trajectories to improve the policy, typically via stochastic gradient descent (SGD). This step updates the model weights to maximize future rewards (Section 2).
3.  **Serving:** The trained policy is used to render actions based on the current state of the environment. This requires minimizing latency to maximize decisions per second (Section 2).

**Figure 2** provides pseudocode for this process. The function `rollout(policy, environment)` generates trajectories (Simulation + Serving), while `train_policy(environment)` uses those trajectories to update the policy (Training). Crucially, these steps are not independent pipelines; they are tightly interwoven. The results of a simulation determine whether more simulations are needed, and the updated policy immediately affects the next simulation step.

### 2.3 Specific System Requirements

This tight coupling imposes demanding requirements that existing frameworks were not designed to handle. The paper identifies three critical dimensions where new systems must excel (Section 2):

*   **Fine-Grained, Heterogeneous Computations:**
    *   **Time:** Task durations range from milliseconds (taking an action) to hours (training a complex policy).
    *   **Hardware:** Training often requires GPUs or TPUs, while simulations typically run on CPUs. A system must manage these heterogeneous resources efficiently.
*   **Flexible Computation Model:**
    *   RL requires both **stateless computations** (e.g., fine-grained simulation tasks that can be load-balanced across any node) and **stateful computations** (e.g., parameter servers or GPU-backed training loops that maintain internal state).
    *   Existing systems typically support one or the other, but not both unified in a single engine.
*   **Dynamic Execution:**
    *   The order in which computations finish is not known in advance (e.g., simulations finish at different times).
    *   The results of a computation determine future computations (e.g., a simulation result triggers a policy update).
    *   This contrasts with static graph frameworks where the computation structure is fixed before execution begins.

To achieve high utilization in large clusters, the paper calculates a specific throughput requirement. Assuming 5ms single-core tasks on a cluster of 200 nodes with 32 cores each, the system must handle:

$$
\left(\frac{1\text{s}}{5\text{ms}}\right) \times 32 \times 200 = 1.28\text{M tasks/sec}
$$

This footnote calculation in Section 2 establishes the baseline: a viable framework must handle **millions of tasks per second** with **millisecond-level latencies**.

### 2.4 Limitations of Prior Approaches

The paper conducts a thorough critique of existing frameworks, categorizing them by their primary design focus and highlighting why they fall short for RL workloads (Section 1):

*   **Bulk-Synchronous Parallel (BSP) Systems:** Frameworks like MapReduce, Spark, and Dryad rely on barrier synchronization. They do not support fine-grained simulation or low-latency policy serving. Their coarse-grained nature introduces latency unacceptable for interactive loops.
*   **Task-Parallel Systems:** Systems like CIEL and Dask provide little support for distributed training and serving. They lack the stateful abstractions needed for parameter servers or persistent actors.
*   **Streaming Systems:** Frameworks like Naiad and Storm focus on data streams but do not naturally support the stateful training loops required for RL.
*   **Distributed Deep Learning:** TensorFlow and MXNet excel at batch training but do not naturally support simulation and serving within the same graph. They are optimized for static DAGs (Directed Acyclic Graphs) of linear algebra operations.
*   **Model Serving Systems:** Tools like TensorFlow Serving and Clipper support neither training nor simulation. They are designed for external clients, not embedded simulation loops.

### 2.5 The Cost of Specialization

One might propose stitching together specialized systems—for example, using Horovod for training, Clipper for serving, and CIEL for simulation. However, the paper argues this approach is **untenable in practice** (Section 1).

*   **Tight Coupling:** The components are so interdependent within RL applications that data movement and latency between separate systems become prohibitive.
*   **Engineering Burden:** Researchers and practitioners are forced to build **one-off systems** for specialized RL applications (e.g., AlphaGo, robotic manipulation). This pushes standard systems challenges—scheduling, fault tolerance, data movement—onto every application developer, slowing down research and innovation.

### 2.6 Ray's Positioning

Ray positions itself not as a replacement for all existing tools, but as a **general-purpose cluster-computing framework** specifically designed to unify these coupled workloads (Section 1).

*   **What Ray Is:** A system that enables simulation, training, and serving for RL applications within a single dynamic execution engine. It unifies **task-parallel** (stateless) and **actor-based** (stateful) abstractions to handle heterogeneity.
*   **What Ray Is Not:**
    *   It is not a substitute for serving systems like Clipper when broader model management, testing, and composition are required.
    *   It is not a substitute for generic data-parallel frameworks like Spark, as it lacks rich functionality like query optimization and straggler mitigation (Section 1).

By addressing the specific gap between static data processing and dynamic interactive learning, Ray aims to remove the systems engineering burden from AI researchers, allowing them to focus on algorithm design rather than infrastructure.

## 3. Technical Approach

This is a systems paper that presents Ray, a distributed execution framework designed to unify task-parallel and actor-based computations under a single dynamic engine. The core idea is to decouple control state from computation by storing all lineage and metadata in a sharded Global Control Store (GCS), enabling stateless schedulers and workers that can scale horizontally while maintaining fault tolerance through lineage-based reconstruction.

### 3.1 Reader orientation (approachable technical breakdown)

Ray is a distributed computing system that lets programmers write Python code that automatically runs across many machines, handling both stateless functions (tasks) and stateful objects (actors) through a unified interface. It solves the problem of running reinforcement learning workloads that require millions of fine-grained operations per second with millisecond latencies by distributing the scheduler and metadata store rather than centralizing them in a single bottleneck.

### 3.2 Big-picture architecture (diagram in words)

Imagine Ray as a three-layer system where the top layer is your Python code (the Driver), the middle layer consists of worker processes that execute your functions (Workers and Actors), and the bottom layer is the infrastructure that coordinates everything (the System Layer). The System Layer contains three critical components: first, the **Global Control Store (GCS)** acts as a distributed database storing all task metadata, object locations, and lineage information; second, the **Distributed Scheduler** has two levels—local schedulers on each node that handle most scheduling decisions, and global schedulers that balance load across nodes when local capacity is exceeded; third, the **Object Store** on each node holds input and output data in shared memory using Apache Arrow format, enabling zero-copy data sharing between tasks on the same machine. When you call a remote function, the request flows from your Driver to the local scheduler, which may forward it to a global scheduler if the node is overloaded, the global scheduler consults the GCS to find where input data lives, assigns the task to an appropriate node, that node's local scheduler dispatches it to a worker, the worker reads inputs from the local object store (replicating from remote nodes if needed), executes the function, writes outputs to the local object store, and registers the output location in the GCS for future tasks to discover.

### 3.3 Roadmap for the deep dive

- First, we explain the programming model (tasks and actors) because understanding what programmers write is prerequisite to understanding how the system executes it.
- Second, we detail the computation graph construction (data edges, control edges, stateful edges) because this determines how the system tracks dependencies for scheduling and fault tolerance.
- Third, we examine the Global Control Store architecture because it is the central design innovation that enables all other components to be stateless and scalable.
- Fourth, we analyze the bottom-up distributed scheduler because this is how Ray achieves million-task-per-second throughput without centralized bottlenecks.
- Fifth, we describe the object store and data movement because this determines actual task execution latency and locality optimization.
- Finally, we cover fault tolerance mechanisms because lineage-based reconstruction depends on all the previously described components working together.

### 3.4 Detailed, sentence-based technical breakdown

#### 3.4.1 Programming Model: Tasks and Actors

Ray provides two computational abstractions that map to different workload characteristics, and understanding when to use each is critical for performance. **Tasks** represent stateless remote function executions where a function `f` is invoked via `f.remote(args)` and immediately returns a **future**—a placeholder for a result that will be computed asynchronously (Table 1). The key property of tasks is that they operate on immutable objects and are expected to be side-effect free, meaning their outputs are determined solely by their inputs, which implies **idempotence** (the function can be re-executed without changing the result) and simplifies fault tolerance through function re-execution on failure (Section 3.1). **Actors** represent stateful computations where a class is instantiated as `Class.remote(args)` and returns a handle that can be passed to other tasks or actors, and methods on that actor are invoked via `actor.method.remote(args)` (Table 1). Unlike tasks, actor methods execute serially on a stateful worker, meaning each method execution can depend on the internal state modified by previous method calls on the same actor (Section 3.1).

The tradeoffs between tasks and actors are explicit and documented in **Table 2**, which shows that tasks enable fine-grained load balancing because each task can be scheduled on any node storing its inputs, while actors provide coarse-grained load balancing because once an actor is placed on a node, all its methods execute there. Tasks support object locality because the scheduler can place a task on the node where its input data resides, whereas actors have poor locality support because their state is fixed to one node. However, actors have low overhead for small state updates because updates modify internal state directly without serialization, while tasks have high overhead for small updates because each update requires serializing and deserializing data across the network. For fault tolerance, tasks have efficient failure handling through re-execution since they are stateless, while actors require checkpointing overhead to recover state after failures (Section 3.1, Table 2).

To handle the heterogeneity requirements from Section 2, Ray augments the basic API with three critical features. First, `ray.wait(futures, k, timeout)` returns the first `k` completed futures instead of waiting for all results like `ray.get()`, which is essential when simulations have variable durations ranging from milliseconds to minutes (Section 3.1). Second, developers can specify resource requirements like `@ray.remote(num_gpus=1)` so the scheduler can match tasks to nodes with appropriate hardware, enabling heterogeneous clusters with CPU-only and GPU-equipped machines (Section 3.1, Figure 3). Third, Ray enables **nested remote functions**, meaning a remote function can invoke other remote functions, which is critical for scalability because it allows multiple processes to spawn tasks in a distributed fashion rather than funneling all task creation through a single driver (Section 3.1).

#### 3.4.2 Computation Graph Construction

Ray employs a **dynamic task graph computation model**, meaning the graph of dependent tasks evolves during execution rather than being fixed before the program starts (Section 3.2). This contrasts with static graph frameworks like TensorFlow where the computation structure must be defined before execution begins. The computation graph contains three types of nodes and three types of edges, as illustrated in **Figure 4**. The node types are: **data objects** (the actual values produced by computations), **tasks** (remote function invocations), and **actor method invocations** (which are similar to tasks but execute on stateful workers) (Section 3.2).

The edge types capture different dependency relationships. **Data edges** connect tasks to the objects they produce or consume: if data object `D` is an output of task `T`, there is a data edge from `T` to `D`, and if `D` is an input to `T`, there is a data edge from `D` to `T` (Section 3.2). **Control edges** capture computation dependencies from nested remote functions: if task `T1` invokes task `T2`, there is a control edge from `T1` to `T2`, which allows the system to track which tasks spawned other tasks (Section 3.2). **Stateful edges** are unique to actors and connect successive method invocations on the same actor: if method `Mj` is called right after method `Mi` on the same actor, there is a stateful edge from `Mi` to `Mj` (Section 3.2). This chain of stateful edges captures the implicit data dependency between successive method invocations that share the actor's internal mutable state, as shown in **Figure 4** where methods `A11`, `A12` on actor `A1` are connected by stateful edges.

The inclusion of stateful edges in the lineage graph is a critical design choice that enables fault tolerance for stateful computations. By explicitly tracking the order of method invocations on an actor, Ray can reconstruct lost actor state by replaying the method calls from a checkpoint, rather than requiring full state replication (Section 3.2, Section 4.2.3). This design embeds actors into an otherwise stateless task graph while preserving the ability to track data lineage for reconstruction, which is essential for the fault tolerance mechanism described in Section 4.2.3.

#### 3.4.3 System Architecture: Application and System Layers

Ray's architecture is explicitly divided into two layers as shown in **Figure 5**: an **Application Layer** that implements the programming model from Section 3, and a **System Layer** that provides scalability and fault tolerance (Section 4). The Application Layer consists of three process types. The **Driver** is the process executing the user program, which submits tasks and actors to the system (Section 4.1). **Workers** are stateless processes that execute tasks (remote functions) invoked by a driver or another worker, and they are started automatically by the system and assigned tasks by the local scheduler (Section 4.1). When a remote function is declared, it is automatically published to all workers, and each worker executes tasks serially with no local state maintained across tasks (Section 4.1). **Actors** are stateful processes that execute only the methods they expose when invoked, and unlike workers, actors are explicitly instantiated by a worker or driver (Section 4.1). Like workers, actors execute methods serially, but each method depends on the state resulting from the previous method execution on that actor (Section 4.1).

The System Layer consists of three major components, all designed to be horizontally scalable and fault-tolerant (Section 4.2). The **Global Control Store (GCS)** maintains the entire control state of the system and is implemented as a key-value store with pub-sub functionality (Section 4.2.1). The **Distributed Scheduler** has two levels: local schedulers on each node and global schedulers that coordinate across nodes (Section 4.2.2). The **Distributed Object Store** stores the inputs and outputs of every task on each node using shared memory (Section 4.2.3). The implementation comprises approximately 40,000 lines of code, with 72% in C++ for the system layer and 28% in Python for the application layer (Section 4.2.4).

#### 3.4.4 Global Control Store: The Central Design Innovation

The Global Control Store is the most distinctive architectural feature of Ray, and understanding its design explains why Ray can scale to millions of tasks per second while maintaining fault tolerance. The GCS is fundamentally a **sharded key-value store with pub-sub functionality**, where sharding enables horizontal scale and per-shard **chain replication** provides fault tolerance (Section 4.2.1). The primary reason for the GCS is to maintain fault tolerance and low latency for a system that can dynamically spawn millions of tasks per second, which requires decoupling durable lineage storage from other system components so each can scale independently (Section 4.2.1).

The GCS stores three types of metadata that are critical for system operation. The **Object Table** tracks the location and size of every data object in the system, which allows schedulers to make locality-aware placement decisions without querying the object store directly (Section 4.2.1, Figure 7). The **Task Table** maintains lineage information for every task, including its arguments, function identity, and output object IDs, which enables reconstruction after failures (Section 4.2.1). The **Function Table** stores the code for remote functions so it can be distributed to all workers automatically (Section 4.2.1, Figure 7). Additionally, the GCS maintains **Event Logs** for debugging and profiling tools (Figure 5).

The design choice to store object metadata in the GCS rather than in the scheduler is critical for performance. Many existing dataflow systems like Spark and CIEL couple object location tracking with the scheduler, which works when the scheduler is not a bottleneck but becomes prohibitively expensive at Ray's target scale (Section 4.2.1). For example, distributed training primitives like **allreduce** are both communication-intensive and latency-sensitive, and involving the scheduler in each object transfer would add unacceptable overhead (Section 4.2.1). By storing object metadata in the GCS, Ray fully decouples **task dispatch** (retrieving remote inputs from other nodes) from **task scheduling** (choosing where to execute), which keeps the scheduler off the critical path for data movement (Section 4.2.1).

The GCS implementation uses one Redis key-value store per shard, with entirely single-key operations to simplify replication semantics (Section 4.2.4). GCS tables are sharded by object and task IDs to scale, and every shard is chain-replicated for fault tolerance using the chain replication protocol (Section 4.2.4, reference [61]). During failure recovery, the maximum client-observed latency is under 30 milliseconds, which includes both failure detection and recovery delays (Section 5.1, Figure 10a). To bound memory usage for long-running applications, Ray implements **GCS flushing** that periodically writes lineage to disk, which caps the memory footprint at a user-configurable level and provides natural snapshotting for lineage (Section 5.1, Figure 10b). Without GCS flushing, the memory footprint grows linearly with the number of tasks tracked and eventually causes the workload to fail (Figure 10b).

The GCS significantly simplifies Ray's overall design by enabling every component in the system to be stateless (Section 4.2.1). On failure, components simply restart and read the lineage from the GCS, rather than requiring complex state recovery protocols. This also makes it easy to scale the distributed object store and scheduler independently, as all components share the needed state via the GCS (Section 4.2.1). An added benefit is the easy development of debugging, profiling, and visualization tools, since the entire system state is queryable through the GCS (Section 4.2.1, Section 7).

#### 3.4.5 Bottom-Up Distributed Scheduler

Ray's scheduler architecture is designed to handle millions of tasks per second with millisecond-level latencies, which no existing cluster scheduler could achieve at the time of this paper (Section 4.2.2). The scheduler is a **two-level hierarchical** design consisting of global schedulers and per-node local schedulers, as illustrated in **Figure 6** (Section 4.2.2). The key innovation is the **bottom-up** scheduling strategy: tasks created at a node are submitted first to that node's local scheduler, and the local scheduler schedules tasks locally unless the node is overloaded or cannot satisfy the task's resource requirements (Section 4.2.2).

The local scheduler uses a threshold-based mechanism to decide when to forward tasks to the global scheduler. If the local task queue exceeds a predefined threshold, or if the task requires resources the node lacks (e.g., a GPU on a CPU-only node), the local scheduler forwards the task to the global scheduler (Section 4.2.2). This design keeps most scheduling decisions at the leaves of the hierarchy, reducing load on the global scheduler and minimizing network traffic for task submission (Section 4.2.2, Figure 6 shows the thickness of arrows proportional to request rate, with most traffic staying at the local level).

The global scheduler makes placement decisions by considering each node's load and the task's constraints. Specifically, the global scheduler identifies the set of nodes that have enough resources of the type requested by the task, and among these nodes selects the one which provides the lowest estimated waiting time (Section 4.2.2). The estimated waiting time at a given node is computed as:

$$
\text{waiting time} = (\text{queue size} \times \text{average task execution time}) + \left(\frac{\text{total size of remote inputs}}{\text{average bandwidth}}\right)
$$

The first term estimates how long the task will wait in the queue before execution begins, and the second term estimates how long it will take to transfer the task's inputs from remote nodes (Section 4.2.2). The global scheduler obtains queue size and resource availability at each node via heartbeats, and obtains the location and size of task inputs from the GCS (Section 4.2.2). The scheduler computes average task execution time and average transfer bandwidth using simple exponential averaging, which adapts to changing workload characteristics without requiring manual tuning (Section 4.2.2).

If the global scheduler becomes a bottleneck, Ray can instantiate more replicas all sharing the same information via the GCS, which makes the scheduler architecture highly scalable (Section 4.2.2). This is possible because the GCS stores all the state the scheduler needs, so multiple scheduler instances can read the same metadata without coordination overhead (Section 4.2.2). The scalability of this design is demonstrated in **Figure 8b**, which shows near-linear scalability reaching 1 million tasks per second at 60 nodes and continuing to scale beyond 1.8 million tasks per second at 100 nodes (Section 5.1).

The scheduler's performance is critical for implementing distributed primitives like allreduce. **Figure 12b** shows an ablation study where artificial scheduler latency is injected, and performance drops nearly 2× with just a few milliseconds of extra latency (Section 5.1). Systems with centralized schedulers like Spark and CIEL typically have scheduler overheads in the tens of milliseconds, making such workloads impractical on Ray's target scale (Section 5.1). This quantifies why the bottom-up distributed design is necessary rather than optional.

#### 3.4.6 In-Memory Distributed Object Store

The object store is designed to minimize task latency by keeping all data in memory and enabling zero-copy data sharing between tasks on the same node (Section 4.2.3). On each node, the object store is implemented via **shared memory**, which allows tasks running on the same node to access the same data without copying (Section 4.2.3). The data format used is **Apache Arrow**, which provides a language-independent columnar memory layout that avoids serialization overhead when passing data between processes (Section 4.2.3).

When a task's inputs are not local, the inputs are replicated to the local object store before execution begins (Section 4.2.3). This replication eliminates potential bottlenecks from hot data objects (objects accessed by many tasks) and minimizes task execution time because a task only reads and writes data from local memory (Section 4.2.3). For low latency, objects are kept entirely in memory and evicted to disk using an **LRU (Least Recently Used)** policy only when memory pressure requires it (Section 4.2.3).

The object store is limited to **immutable data**, similar to existing cluster computing frameworks like Spark and Dryad (Section 4.2.3). This design choice obviates the need for complex consistency protocols because objects are never updated after creation, and it simplifies fault tolerance since there is no need to track concurrent modifications (Section 4.2.3). In the case of node failure, Ray recovers any needed objects through lineage re-execution rather than replication, which trades storage overhead for computation overhead (Section 4.2.3). The lineage stored in the GCS tracks both stateless tasks and stateful actors during initial execution, and Ray uses this lineage to reconstruct objects in the store after failures (Section 4.2.3).

For simplicity, the object store does not support **distributed objects**, meaning each object must fit on a single node (Section 4.2.3). Distributed objects like large matrices or trees can be implemented at the application level as collections of futures, which gives developers flexibility but requires them to manage object partitioning explicitly (Section 4.2.3). To transfer large objects between different object stores, Ray stripes the object across multiple TCP connections, which maximizes network utilization on high-bandwidth links (Section 4.2.4).

The performance of the object store is measured in **Figure 9**, which shows write throughput and IOPS from a single client on a 16-core instance (m4.4xlarge) (Section 5.1). For large objects, throughput exceeds 15 GB/s, limited primarily by `memcpy` operations for object creation (Section 5.1, Figure 9). For small objects, the system achieves 18,000 IOPS, with the main overheads in serialization and inter-process communication between the client and object store (Section 5.1, Figure 9). The implementation uses 8 threads to copy objects larger than 0.5 MB and 1 thread for small objects, which optimizes for both throughput and latency depending on object size (Figure 9 caption).

#### 3.4.7 End-to-End Execution Flow

**Figure 7** provides a step-by-step illustration of how Ray executes a simple task that adds two objects `a` and `b` to produce result `c`, and understanding this flow integrates all the previously described components (Section 4.3). The execution begins when the remote function `add()` is automatically registered with the GCS upon initialization and distributed to every worker in the system (step 0 in Figure 7a) (Section 4.3).

When a driver invokes `add.remote(a, b)` where `a` is stored on node N1 and `b` is stored on node N2, the following sequence occurs (Figure 7a):

1. The driver submits `add(a, b)` to the local scheduler on N1 (step 1).
2. The local scheduler forwards it to a global scheduler (step 2).
3. The global scheduler looks up the locations of `add(a, b)`'s arguments in the GCS (step 3).
4. The global scheduler decides to schedule the task on node N2, which stores argument `b` (step 4).
5. The local scheduler at node N2 checks whether the local object store contains `add(a, b)`'s arguments (step 5).
6. Since the local store doesn't have object `a`, it looks up `a`'s location in the GCS (step 6).
7. Learning that `a` is stored at N1, N2's object store replicates it locally (step 7).
8. As all arguments of `add()` are now stored locally, the local scheduler invokes `add()` at a local worker (step 8).
9. The worker accesses the arguments via shared memory and executes the function (step 9).

When the driver calls `ray.get(idc)` to retrieve the result (Figure 7b):

1. The driver checks the local object store for value `c` using the future `idc` returned by `add()` (step 1).
2. Since the local object store doesn't store `c`, it looks up its location in the GCS, but there is no entry yet because `c` has not been created (step 2).
3. N1's object store registers a callback with the Object Table to be triggered when `c`'s entry has been created (step 2).
4. Meanwhile, at N2, `add()` completes execution and stores result `c` in the local object store (step 3).
5. N2's object store adds `c`'s entry to the GCS (step 4).
6. The GCS triggers a callback to N1's object store with `c`'s entry (step 5).
7. N1 replicates `c` from N2 (step 6).
8. N1 returns `c` to `ray.get()`, which completes the task (step 7).

While this example involves a large number of RPCs, the paper notes that in many cases this number is much smaller because most tasks are scheduled locally and GCS replies are cached by the global and local schedulers (Section 4.3). The locality-aware scheduling shown in **Figure 8a** demonstrates that with locality-aware policy, task latency remains independent of the size of task inputs, whereas without locality awareness, latency grows by 1-2 orders of magnitude at 10-100 MB input data sizes (Section 5.1).

#### 3.4.8 Fault Tolerance Mechanisms

Ray provides two types of fault tolerance depending on whether the computation is stateless or stateful, and both mechanisms depend on the lineage information stored in the GCS (Section 4.2.3, Section 5.1). For **stateless tasks**, Ray uses **lineage-based reconstruction**: when a node fails, the system re-executes the tasks that produced the lost objects by reading the task lineage from the GCS (Section 4.2.3). **Figure 11a** demonstrates this mechanism: as nodes are removed at 25s, 50s, and 100s, the local schedulers reconstruct previous results in the task chain to continue execution, and overall per-node throughput remains stable throughout (Section 5.1).

For **stateful actors**, Ray combines lineage-based reconstruction with **user-defined checkpoint functions** to bound reconstruction time (Section 5.1). By encoding actor method calls as stateful edges directly in the dependency graph (Section 3.2), Ray can reuse the same object reconstruction mechanism as for stateless tasks, but actor state must also be recovered (Section 5.1). **Figure 11b** shows actor reconstruction: at t=200s, when 2 of 10 nodes are killed, 400 of the 2000 actors in the cluster are recovered on the remaining nodes between t=200s and t=270s (Section 5.1, Figure 11b caption). With checkpointing, only 500 methods need to be re-executed versus 10,000 re-executions without checkpointing, demonstrating the importance of bounding reconstruction scope (Section 5.1).

The GCS itself is fault-tolerant through chain replication, where each shard is replicated across multiple nodes and reconfiguration after a node failure completes within 30 milliseconds maximum client-observed latency (Section 5.1, Figure 10a). This includes both failure detection and recovery delays, which is critical for maintaining low-latency task scheduling during failures (Section 5.1).

The paper explicitly addresses whether fault tolerance is necessary for AI applications given their statistical nature (Section 7). The authors argue "yes" for three reasons: first, the ability to ignore failures makes applications easier to write and reason about; second, deterministic replay dramatically simplifies debugging by allowing errors to be reproduced; third, fault tolerance enables running on cheap resources like AWS spot instances, which reduces costs significantly (Section 7). The overhead of fault tolerance is found to be minimal for Ray's target workloads (Section 7).

#### 3.4.9 Design Choices and Tradeoffs

Several critical design choices distinguish Ray from alternative approaches, and understanding these tradeoffs explains why Ray is architected as it is. The decision to **decouple control state from computation** via the GCS contrasts with systems like Spark and CIEL that store metadata in a centralized scheduler (Section 4.2.1, Section 6). This decoupling enables independent scaling of the scheduler and object store, but requires an additional network hop to query the GCS for object locations (Section 4.2.1). The paper argues this tradeoff is favorable because involving the scheduler in each object transfer is prohibitively expensive for communication-intensive primitives like allreduce (Section 4.2.1).

The choice of a **bottom-up distributed scheduler** contrasts with centralized schedulers (Spark, CIEL, Dryad) and fully decentralized schedulers (Sparrow, work-stealing) (Section 4.2.2, Section 6). Centralized schedulers can provide better locality but have latencies in the tens of milliseconds, while fully decentralized schedulers either don't consider data locality or assume tasks belong to independent jobs (Section 4.2.2). Ray's hierarchical approach captures the locality benefits of centralized scheduling while achieving the scale of decentralized approaches (Section 4.2.2).

The decision to use **immutable objects** in the object store contrasts with systems that support mutable distributed state (Section 4.2.3). This obviates complex consistency protocols and simplifies fault tolerance, but requires applications to manage state explicitly through actors rather than shared mutable objects (Section 4.2.3). The paper argues this is appropriate for RL workloads where state is naturally encapsulated in actors (e.g., parameter servers, simulators) (Section 3.1).

The choice to **not support distributed objects** (objects spanning multiple nodes) contrasts with systems that provide this abstraction natively (Section 4.2.3). This simplifies the object store implementation and avoids the complexity of distributed consistency, but requires developers to partition large data structures manually as collections of futures (Section 4.2.3). The paper notes this is acceptable because AI workloads typically process data in batches that fit on single nodes (Section 4.2.3).

Finally, the decision to implement Ray as a **general-purpose framework** rather than an RL-specific system contrasts with one-off solutions built for specific algorithms (Section 1, Section 5.3). This requires more upfront engineering but enables reuse across multiple RL algorithms and reduces the burden on application developers (Section 5.3). The evaluation in Section 5.3 shows this approach matches or exceeds the performance of specialized systems while providing much greater flexibility (Section 5.3).

## 4. Key Insights and Innovations

While the technical mechanics of Ray are detailed in Section 3, the true novelty lies in the specific design principles that distinguish it from prior distributed systems. This section isolates the fundamental innovations that enable Ray to meet the demanding requirements of emerging AI workloads (Section 2). We distinguish between incremental improvements (e.g., faster serialization) and fundamental architectural shifts (e.g., decoupling control state).

### 4.1 Principle of Stateless Components via Sharded Control Store

The most foundational innovation in Ray is the architectural decision to store **all** control state in a dedicated Global Control Store (GCS), rendering every other system component stateless.

*   **Prior Approach:** Traditional dataflow systems like Spark [64] or CIEL [40] typically embed control state (lineage, object locations, task status) within a centralized master or scheduler process. While this simplifies consistency, it creates a single point of contention. As the number of tasks grows, the master becomes a bottleneck for both scheduling decisions and metadata updates.
*   **Ray's Innovation:** Ray externalizes this state into the GCS, a sharded key-value store (Section 4.2.1). This allows the schedulers, workers, and object stores to be **stateless**. If a scheduler crashes, it can simply restart and reload its state from the GCS without losing cluster-wide information.
*   **Significance:** This decoupling enables **independent horizontal scaling**. The storage layer (GCS) can scale via sharding, while the computation layer (schedulers/workers) can scale via replication.
    *   **Fault Tolerance Simplification:** Because components are stateless, fault tolerance reduces to restarting the component and replaying lineage from the GCS. This avoids complex state recovery protocols required in stateful masters.
    *   **Debuggability:** Having a single source of truth for system state enables powerful debugging tools that can query the entire cluster state without interrupting execution (Section 7).
    *   **Performance:** By removing object metadata lookups from the scheduler's critical path, Ray avoids the latency penalties seen in centralized systems. As shown in **Figure 12b**, adding just a few milliseconds of scheduler latency can degrade allreduce performance by nearly $2\times$. The GCS ensures the scheduler remains off the critical path for data movement.

This is a fundamental shift from "smart master, dumb workers" to "dumb components, smart store," enabling the system to handle millions of tasks per second without a centralized bottleneck.

### 4.2 Bottom-Up Hierarchical Scheduling for Fine-Grained Latency

Ray introduces a **bottom-up distributed scheduling** strategy to resolve the trade-off between scheduling scalability and data locality.

*   **Prior Approach:** Existing schedulers generally fall into two categories:
    1.  **Centralized (e.g., Spark, Dryad):** Excellent data locality but limited scalability. Scheduler overheads are typically in the tens of milliseconds [62, 38], which is too high for fine-grained RL tasks.
    2.  **Fully Decentralized (e.g., Sparrow [45], Work Stealing [12]):* High scalability but often ignore data locality or assume independent jobs. This leads to excessive data movement for dependent tasks.
*   **Ray's Innovation:** Ray employs a two-level hierarchy where tasks are submitted to a **local scheduler** first. The local scheduler attempts to place the task on the local node. It only forwards the task to a **global scheduler** if the node is overloaded or lacks resources (Section 4.2.2).
*   **Significance:** This design captures the benefits of both approaches:
    *   **Locality:** By scheduling locally first, Ray minimizes data transfer for tasks whose inputs are already on the node. **Figure 8a** demonstrates that locality-aware placement keeps task latency independent of input size, whereas unaware policies suffer latency increases of 1–2 orders of magnitude for 10–100 MB inputs.
    *   **Scalability:** By filtering tasks at the local level, the global scheduler is shielded from the full load. This enables near-linear scalability. **Figure 8b** shows Ray reaching 1 million tasks per second at 60 nodes and scaling beyond 1.8 million tasks per second at 100 nodes.
    *   **Fine-Grained Support:** The low overhead of the local scheduler allows Ray to handle tasks with durations as short as a few milliseconds, which is critical for the simulation workloads described in Section 2.

This innovation is not merely an optimization; it is a requirement for supporting the fine-grained, dynamic task graphs inherent in reinforcement learning, where centralized schedulers would collapse under the request rate.

### 4.3 Unified Execution Graph for Heterogeneous Workloads

Ray's unification of task-parallel and actor-based abstractions within a single dynamic execution engine addresses the fragmentation of prior AI infrastructure.

*   **Prior Approach:** Practitioners typically stitched together specialized systems: Horovod [53] for training, Clipper [19] for serving, and custom scripts for simulation. This introduces high latency due to data movement between systems and imposes a massive engineering burden to manage compatibility and fault tolerance across boundaries (Section 1).
*   **Ray's Innovation:** Ray expresses both stateless simulations (tasks) and stateful training/serving (actors) in a single computation graph connected by data, control, and **stateful edges** (Section 3.2). This allows the system to optimize across the entire RL loop rather than within isolated components.
*   **Significance:** This unification enables application-level optimizations that are difficult or impossible in stitched systems:
    *   **Hierarchical Aggregation:** In the Evolution Strategies (ES) benchmark (Section 5.3.1), Ray users implemented an aggregation tree of actors to combine results from 10,000 tasks. The reference implementation failed beyond 1,024 cores due to driver bottlenecks, whereas Ray scaled to 8,192 cores.
    *   **Performance:** **Figure 14a** shows the Ray ES implementation solving the Humanoid-v1 task in 3.7 minutes, compared to 10 minutes for the specialized reference system. This $2.7\times$ speedup is achieved not by faster primitives, but by the flexibility to restructure the computation graph dynamically.
    *   **Resource Efficiency:** For Proximal Policy Optimization (PPO), Ray's heterogeneity awareness allowed asymmetric resource allocation (CPU for simulation, GPU for training). **Figure 14b** shows Ray outperforming an MPI implementation while using fewer GPUs, reducing costs by a factor of 4.5 solely through better resource mapping.

This insight transforms the system from a passive executor into an active enabler of algorithmic innovation. By removing the boundaries between simulation, training, and serving, Ray allows researchers to optimize the entire learning loop as a single distributed program.

### 4.4 Summary of Contributions vs. Prior Art

The table below summarizes how Ray's innovations contrast with the state of the art at the time of publication.

| Feature | Prior Systems (Spark, TensorFlow, MPI) | Ray | Innovation Significance |
| :--- | :--- | :--- | :--- |
| **Control State** | Centralized in Master/Scheduler | Sharded in GCS | Enables stateless components & independent scaling (Section 4.2.1). |
| **Scheduling** | Centralized or Fully Decentralized | Bottom-Up Hierarchical | Balances locality and scale for fine-grained tasks (Figure 8b). |
| **Computation Model** | Task-Parallel OR Actor-Based | Unified Tasks + Actors | Supports coupled RL workloads without system stitching (Section 5.3). |
| **Fault Tolerance** | Checkpointing or Replication | Lineage-Based Reconstruction | Reduces storage overhead; simplifies debugging via deterministic replay (Section 7). |
| **Task Granularity** | Coarse (Seconds) | Fine (Milliseconds) | Supports interactive simulation loops required for RL (Section 2). |

These innovations collectively address the specific gaps identified in Section 2, providing a system that is both flexible enough for dynamic AI workloads and performant enough for large-scale cluster deployment.

## 5. Experimental Analysis

The paper evaluates Ray across three dimensions: microbenchmarks testing core system capabilities, building block experiments comparing Ray to specialized systems for individual RL workloads, and end-to-end RL application benchmarks. This section walks through the experimental methodology, presents specific quantitative results, and critically assesses whether the evidence supports the paper's claims about scalability, latency, and flexibility.

### 5.1 Evaluation Methodology

**Experimental Setup.** All experiments were conducted on Amazon Web Services, ensuring reproducible hardware configurations (Section 5). The primary instance types used include:
- `m4.16xlarge` CPU instances for general computation benchmarks
- `p3.16xlarge` GPU instances for deep learning workloads (V100 GPUs)
- `p2.16xlarge` GPU instances for PPO experiments (K80 GPUs)
- `m4.4xlarge` and `m4.xlarge` for specific microbenchmarks

Network connectivity between nodes uses 25Gbps Ethernet, which is critical for communication-intensive primitives like allreduce (Section 5.1).

**Metrics.** The evaluation focuses on four key metrics aligned with the requirements from Section 2:
1. **Throughput:** Tasks per second, images per second, or timesteps per second
2. **Latency:** Mean task latency in milliseconds or microseconds
3. **Scalability:** How performance changes with increasing cluster size (nodes or cores)
4. **Fault Tolerance:** Recovery time and throughput stability during node failures

**Baselines.** Ray is compared against a range of specialized systems rather than a single competitor:
- **OpenMPI (v1.10 and v3.0):** For allreduce and PPO benchmarks
- **Horovod + TensorFlow:** For distributed training comparisons
- **Distributed TensorFlow:** For training throughput
- **Clipper:** For model serving throughput
- **Reference Implementations:** Special-purpose systems built for Evolution Strategies [49] and PPO [5]

This multi-baseline approach is methodologically sound because no single system addresses all RL workloads. However, it also means Ray is evaluated against best-in-class systems for each individual workload, raising the bar for demonstrating competitiveness.

### 5.2 Microbenchmark Results

#### 5.2.1 Locality-Aware Task Placement

**Figure 8a** tests whether Ray's scheduler effectively places tasks near their input data. The experiment schedules 1000 tasks with random object dependencies onto one of two nodes, varying object sizes from 100KB to 100MB.

The results show a stark contrast:
- **With locality-aware scheduling:** Task latency remains independent of input size across the entire range
- **Without locality awareness:** Latency increases by **1–2 orders of magnitude** at 10–100MB input sizes

This demonstrates that the scheduler's ability to query object locations from the GCS (Section 4.2.1) translates into tangible performance benefits. For RL workloads where simulations may produce large observations (e.g., images, video), avoiding unnecessary data transfer is critical.

#### 5.2.2 End-to-End Scalability

**Figure 8b** measures task throughput as cluster size increases from 10 to 100 nodes using an embarrassingly parallel workload of empty tasks.

Key results:
- **60 nodes:** Ray exceeds **1 million tasks per second**
- **100 nodes:** Ray scales beyond **1.8 million tasks per second**
- **100 million tasks:** Completed in **54 seconds** with minimum variability

The scaling is described as "near-perfect linearity," which is significant because centralized schedulers typically degrade under high request rates. The rightmost datapoint (x ∈ {70, 80, 90} omitted due to cost) suggests the authors tested intermediate points but chose not to display them, which is a minor transparency limitation.

The paper notes that increasing task duration reduces throughput proportionally, but overall scalability remains linear. This is an important caveat: realistic workloads with object dependencies and inherent parallelism limits may exhibit more constrained scalability than this microbenchmark suggests.

#### 5.2.3 Object Store Performance

**Figure 9** evaluates the in-memory object store on a 16-core `m4.4xlarge` instance, measuring both IOPS (for small objects) and write throughput (for large objects).

Specific results:
- **Large objects:** Write throughput exceeds **15 GB/s** (red line in figure)
- **Small objects:** **18,000 IOPS** (cyan line in figure)
- **Threading strategy:** 8 threads for objects >0.5MB, 1 thread for smaller objects

The performance bottleneck shifts based on object size:
- For large objects, `memcpy` dominates object creation time
- For small objects, serialization and inter-process communication (IPC) between client and object store are the main overheads

This characterization helps developers understand when to expect bottlenecks. For example, workloads with many small objects should minimize serialization overhead, while workloads with large objects should optimize memory copy operations.

#### 5.2.4 GCS Fault Tolerance and Flushing

**Figure 10a** simulates GCS read/write operations during node failures. The chain starts with 2 replicas, and at t ≈ 4.2s, a chain member is killed and immediately replaced.

Key finding:
- **Maximum client-observed latency:** Under **30 milliseconds** during reconfiguration (includes both failure detection and recovery delays)

This is critical for maintaining low-latency task scheduling during failures. The 30ms bound is achieved through chain replication [61], which provides strong consistency without the coordination overhead of consensus protocols like Paxos.

**Figure 10b** tests GCS memory management by submitting 50 million empty tasks sequentially:
- **Without GCS flushing:** Memory footprint grows linearly until reaching system capacity, causing the workload to fail (indicated by red cross in figure)
- **With GCS flushing:** Memory footprint remains constant at a user-configurable level

This ablation demonstrates that GCS flushing is not optional for long-running applications. The flushing mechanism also provides natural snapshotting for lineage, enabling checkpoint-and-restart functionality.

#### 5.2.5 Task and Actor Reconstruction

**Figure 11a** demonstrates lineage-based fault tolerance for stateless tasks. The workload consists of linear chains of 100ms tasks on `m4.xlarge` instances. Nodes are removed at 25s, 50s, and 100s.

Result:
- **Per-node throughput remains stable** throughout node removals
- Lost task dependencies are transparently reconstructed from lineage stored in GCS

**Figure 11b** tests actor fault tolerance with checkpointing. At t = 200s, 2 of 10 nodes are killed, causing 400 of 2000 actors to recover on remaining nodes between t = 200s and t = 270s.

Critical comparison:
- **With checkpointing:** Only **500 methods** re-executed
- **Without checkpointing:** **10,000 methods** re-executed

This **20× reduction** in re-execution overhead demonstrates why checkpointing is essential for stateful computations. The paper notes future work could further reduce reconstruction time by allowing users to annotate methods that do not mutate state, which would enable more granular recovery.

#### 5.2.6 Allreduce Performance and Scheduler Ablation

**Figure 12a** compares Ray's allreduce implementation against OpenMPI (v1.10) on 16 `m4.16xlarge` nodes:

| Object Size | Ray Time | OpenMPI Time | Speedup |
|-------------|----------|--------------|---------|
| 100MB | ~200ms | ~300ms | **1.5×** |
| 1GB | ~1200ms | ~2400ms | **2×** |

Ray outperforms OpenMPI despite being a general-purpose framework. The authors attribute this to Ray's use of multiple threads for network transfers, fully utilizing the 25Gbps AWS connection, whereas OpenMPI sequentially sends and receives on a single thread [22].

For smaller objects, OpenMPI outperforms Ray by switching to a lower-overhead algorithm—an optimization the authors plan to implement in the future. This is an honest acknowledgment of a current limitation.

**Figure 12b** is a critical ablation study injecting artificial scheduler latency:
- **Added latency:** Just a few milliseconds
- **Performance impact:** Drops nearly **2×**

This quantifies why centralized schedulers with tens of milliseconds overhead (like Spark and CIEL [62, 38]) are impractical for communication-intensive primitives. The scheduler must operate at sub-millisecond latency to avoid becoming the bottleneck.

### 5.3 Building Block Comparisons

#### 5.3.1 Distributed Training

**Figure 13** compares Ray's synchronous parameter-server SGD implementation against Horovod and distributed TensorFlow using a ResNet-101 model on `p3.16xlarge` instances with 4 GPUs per node.

Results:
- **Ray vs. Horovod:** Ray **matches** Horovod performance
- **Ray vs. Distributed TensorFlow:** Ray is within **10%** of distributed TensorFlow (in distributed replicated mode)

The comparison is methodologically careful: only TensorFlow-based systems are compared to isolate Ray's overhead rather than differences between deep learning frameworks. The key optimization enabling this performance is pipelining gradient computation, transfer, and summation within a single iteration, using a custom TensorFlow operator to write tensors directly to Ray's object store.

#### 5.3.2 Model Serving

**Table 3** compares Ray against Clipper [19], a dedicated serving system, for embedded serving workloads where client and server are co-located on the same `p3.8xlarge` instance.

| System | Small Input (4KB) | Larger Input (100KB) |
|--------|-------------------|----------------------|
| Clipper | 4400 ± 15 states/sec | 290 ± 1.3 states/sec |
| Ray | 6200 ± 21 states/sec | 6900 ± 150 states/sec |

Ray achieves:
- **1.4× higher throughput** for small inputs (fully connected network, 5ms eval time)
- **23.8× higher throughput** for larger inputs (residual network, 10ms eval time)

The dramatic difference for larger inputs is attributed to Ray's low-overhead serialization and shared memory abstractions, which are optimized for co-located processes. Clipper is designed for external web-serving workloads over REST, which introduces additional overhead not present in Ray's embedded serving model.

This result is conditional: Ray excels when client and server are within the same task graph, but Clipper may be preferable for broader model management, testing, and external client serving (Section 1).

#### 5.3.3 Simulation

**Table 4** compares Ray against an MPI implementation using bulk-synchronous parallelism (BSP) with global barriers between rounds. The benchmark uses the Pendulum-v0 simulator from OpenAI Gym [13].

| System | 1 CPU | 16 CPUs | 256 CPUs |
|--------|-------|---------|----------|
| MPI (BSP) | 22.6K | 208K | 2.16M timesteps/sec |
| Ray (Async) | 22.3K | 290K | 4.03M timesteps/sec |

At 256 CPUs, Ray achieves **1.86× higher throughput** than MPI. The advantage comes from Ray's ability to dynamically spawn and collect results from fine-grained simulation tasks without waiting at barriers.

The footnote acknowledges that MPI experts could use asynchronous primitives to avoid barriers, but this increases program complexity. The authors chose the BSP implementation to simulate typical usage patterns, which is reasonable but represents a somewhat favorable comparison for Ray.

### 5.4 End-to-End RL Applications

#### 5.4.1 Evolution Strategies

**Figure 14a** compares Ray's Evolution Strategies (ES) implementation against a reference implementation [49] specifically built for this algorithm. The benchmark measures time to reach a score of 6000 on the Humanoid-v1 task [13].

Key results:
- **Ray scalability:** Scales to **8192 cores**
- **Reference system:** Fails to complete at **2048 cores** (driver bottleneck)
- **Speedup from doubling cores:** Average **1.6×** completion time reduction
- **Median time to solve:** **3.7 minutes** (Ray) vs. **10 minutes** (best published result)
- **Improvement factor:** **2.7× faster** than prior state-of-the-art

The engineering effort comparison is striking:
- **Ray implementation:** Required modifying only **7 lines of code** from a serial implementation
- **Reference implementation:** Several hundred lines dedicated to communication protocols

Hierarchical aggregation (using an aggregation tree of actors) was easy to implement in Ray due to support for nested tasks and actors, whereas the reference system would require substantial re-engineering to support this optimization. This demonstrates Ray's claim that flexibility enables algorithmic innovations that are impractical in specialized systems.

One caveat: ES shows greater runtime variance than PPO on this benchmark (noted in Figure 14a caption), which is an algorithmic characteristic rather than a system limitation.

#### 5.4.2 Proximal Policy Optimization

**Figure 14b** compares Ray's PPO implementation against an optimized MPI reference [5] using asymmetric resource allocation. The algorithm collects 320,000 simulation steps per iteration, with policy updates performing 20 steps of SGD (batch size 32,768).

Resource efficiency:
- **MPI implementation:** Required **1 GPU for every 8 CPUs** (symmetric architecture)
- **Ray implementation:** Required at most **8 GPUs total** (never more than 1 GPU per 8 CPUs)

Ray's heterogeneity-aware scheduling allows CPU-only tasks (simulation) to run on cheaper high-CPU instances while GPU resources are reserved for training. This asymmetric allocation is difficult in MPI, which typically requires all processes to run identical code with identical resources.

Cost analysis:
- **Resource heterogeneity alone:** **4.5× cost reduction** [4]
- **With spot instances (4× cheaper) and fault tolerance:** **18× total cost reduction**

The 18× figure combines Ray's ability to use spot instances (enabled by transparent fault tolerance) with better resource mapping. This is a compelling economic argument for Ray's architecture, though it depends on AWS pricing at the time of writing [4].

### 5.5 Critical Assessment of Experimental Claims

#### 5.5.1 Strengths of the Evaluation

The experimental design has several notable strengths:

1. **Multi-dimensional validation:** The paper tests scalability (Figure 8b), latency (Figure 8a, 12b), fault tolerance (Figures 10, 11), and application performance (Figure 14), covering all requirements from Section 2.

2. **Ablation studies:** Figure 12b (scheduler latency) and Figure 10b (GCS flushing) isolate specific design choices and quantify their impact, which strengthens causal claims about architecture decisions.

3. **Realistic workloads:** The RL benchmarks (ES, PPO) are representative algorithms from the literature, not synthetic workloads designed to favor Ray.

4. **Honest limitations:** The paper acknowledges where Ray underperforms (small object allreduce vs. OpenMPI) and where comparisons are favorable (MPI BSP vs. Ray async).

#### 5.5.2 Limitations and Open Questions

Several limitations warrant consideration:

1. **Cluster size ceiling:** The scalability benchmark stops at 100 nodes (1.8M tasks/sec), with intermediate points (70, 80, 90 nodes) omitted due to cost. While the trend is linear, extrapolation beyond 100 nodes is not empirically validated.

2. **Workload specificity:** All end-to-end benchmarks are RL-focused. The paper claims Ray is general-purpose, but the evaluation does not test non-RL workloads (e.g., data preprocessing pipelines, hyperparameter search without RL).

3. **Comparison fairness:** Some baselines may not represent best practices. For example, the MPI simulation benchmark uses BSP with barriers, which experts could avoid with asynchronous primitives (acknowledged in Table 4 footnote).

4. **Long-running stability:** The GCS flushing experiment (Figure 10b) tests 50 million tasks, but the paper does not report multi-day or multi-week deployment experience, which would be relevant for production use.

5. **Debugging overhead:** While Section 7 mentions the GCS enables debugging tools, the evaluation does not quantify the development time savings from these tools compared to debugging one-off systems.

#### 5.5.3 Do the Experiments Support the Claims?

The experimental evidence **largely supports** the paper's central claims:

| Claim | Evidence | Assessment |
|-------|----------|------------|
| Scalability to 1.8M tasks/sec | Figure 8b (100 nodes) | **Supported** with empirical data |
| Millisecond-level latencies | Figure 8a, 10a, 12b | **Supported** (30ms during failures, sub-ms normal) |
| Better than specialized systems | Figure 14a (2.7× ES), Figure 14b (PPO) | **Supported** for tested algorithms |
| Unified model enables optimizations | 7 lines vs. hundreds for ES | **Supported** with engineering effort comparison |
| Fault tolerance with minimal overhead | Figure 11a, 11b | **Supported** (stable throughput, 20× checkpoint benefit) |

The most compelling evidence is the end-to-end RL benchmarks, where Ray not only matches but exceeds specialized systems while requiring less engineering effort. This validates the core thesis that a flexible, general-purpose framework can outperform one-off solutions when the workload requires tight coupling of simulation, training, and serving.

The scalability claims are well-supported up to 100 nodes, though extrapolation to larger clusters remains an open question. The fault tolerance mechanisms are demonstrated but not stress-tested under cascading failures or network partitions, which would be important for production deployments.

Overall, the evaluation is rigorous for a systems paper of this scope, though some claims (particularly around general-purpose applicability beyond RL) would benefit from additional workload diversity in future evaluations.

## 6. Limitations and Trade-offs

No system design is without compromises. Ray's architecture makes deliberate choices that optimize for specific workload characteristics while accepting limitations in other areas. This section critically examines the assumptions underlying Ray's design, the scenarios it does not address well, and the open questions that remain.

### 6.1 Architectural Assumptions

Ray's design rests on several foundational assumptions that shape its performance characteristics and applicability.

**Assumption 1: Control State Can Be Externalized Without Prohibitive Latency**

The GCS architecture assumes that storing all control state in a sharded key-value store will not introduce unacceptable latency for scheduling decisions. This is a calculated tradeoff:

> "Involving the scheduler in each object transfer is prohibitively expensive for primitives important to distributed training like allreduce, which is both communication-intensive and latency-sensitive. Therefore, we store the object metadata in the GCS rather than in the scheduler, fully decoupling task dispatch from task scheduling." (Section 4.2.1)

The evidence supports this assumption for Ray's target workloads—**Figure 12b** shows that scheduler latency directly impacts allreduce performance, with just a few milliseconds of added latency causing nearly **2× performance degradation**. However, this design introduces an additional network hop for object location lookups that may not be optimal for all workload patterns. Workloads with highly localized data access patterns might benefit from tighter scheduler-object store coupling.

**Assumption 2: Immutable Objects Are Sufficient for AI Workloads**

Ray's object store is limited to **immutable data**, which "obviates the need for complex consistency protocols" (Section 4.2.3). This assumption holds well for RL workloads where:
- Simulation outputs are read-only trajectories
- Model weights are updated through actor state, not shared objects
- Training gradients are aggregated through parameter server actors

However, this assumption may not hold for workloads requiring fine-grained mutable shared state. The paper acknowledges this limitation implicitly by noting that "distributed objects like large matrices or trees can be implemented at the application level as collections of futures" (Section 4.2.3). This pushes complexity onto the application developer, who must manually partition and manage large data structures.

**Assumption 3: Lineage-Based Fault Tolerance Is Preferable to Replication**

Ray reconstructs lost objects through lineage re-execution rather than maintaining replicated copies. This trades **computation overhead for storage overhead**:

> "In the case of node failure, Ray recovers any needed objects through lineage re-execution. The lineage stored in the GCS tracks both stateless tasks and stateful actors during initial execution; we use this lineage to reconstruct objects in the store." (Section 4.2.3)

This assumption is valid when:
- Task computation is cheaper than storing redundant copies
- Failures are infrequent enough that re-execution cost is amortized
- Deterministic replay is possible (which Ray ensures through idempotent tasks)

However, for workloads with expensive computations (e.g., hours-long simulations) or non-deterministic operations, lineage-based recovery may be prohibitively expensive. The paper addresses this partially through actor checkpointing (**Figure 11b** shows checkpointing reduces re-execution from 10,000 to 500 methods), but checkpointing overhead is application-dependent and requires user-defined checkpoint functions.

### 6.2 Workload Constraints and Unsupported Scenarios

Ray is explicitly positioned as a general-purpose framework, but several workload types fall outside its design scope.

**Constraint 1: Not a Replacement for Specialized Serving Systems**

The paper is explicit that Ray does not aim to substitute for dedicated model serving infrastructure:

> "Ray does not aim to substitute for serving systems like Clipper [19] and TensorFlow Serving [6], as these systems address a broader set of challenges in deploying models, including model management, testing, and model composition." (Section 1)

This limitation is evident in **Table 3**, where Ray outperforms Clipper for **embedded serving** (co-located client and server) but the comparison explicitly notes this is "often the case for RL applications but not for the general web serving workloads addressed by systems like Clipper" (Section 5.2.2). For production deployments requiring:
- A/B testing across model versions
- Canary deployments with traffic splitting
- Multi-tenant isolation
- External API access with authentication

Ray would need to be complemented with or replaced by specialized serving infrastructure.

**Constraint 2: Not a Replacement for Data-Parallel Frameworks**

Similarly, Ray is not positioned as a Spark replacement:

> "Ray is not a substitute for generic data-parallel frameworks, such as Spark [64], as it currently lacks the rich functionality and APIs (e.g., straggler mitigation, query optimization) that these frameworks provide." (Section 1)

This limitation stems from Ray's focus on fine-grained, dynamic task graphs rather than bulk data processing. Workloads requiring:
- SQL-like query interfaces
- Automatic straggler detection and mitigation
- Complex join optimizations
- Mature ecosystem of data connectors

would find Ray's lower-level API insufficient. The paper acknowledges the API is "relatively low-level" and notes they are "considering enhancing the API to include higher level primitives and libraries" (Section 7).

**Constraint 3: Single-Node Object Size Limit**

The object store does not support distributed objects spanning multiple nodes:

> "For simplicity, our object store does not support distributed objects, i.e., each object fits on a single node." (Section 4.2.3)

This constraint imposes a hard ceiling on individual object size based on per-node memory capacity. For workloads with:
- Models larger than single-node memory
- Datasets that cannot be partitioned into node-sized chunks
- Large intermediate results from distributed computations

developers must manually implement object partitioning as "collections of futures" (Section 4.2.3). This increases application complexity and may introduce inefficiencies if partitioning strategies are suboptimal.

**Constraint 4: Limited Evaluation Beyond RL Workloads**

While Ray is positioned as a general-purpose framework, the evaluation focuses almost exclusively on RL applications:

> "All end-to-end benchmarks are RL-focused. The paper claims Ray is general-purpose, but the evaluation does not test non-RL workloads (e.g., data preprocessing pipelines, hyperparameter search without RL)." (Section 5.5.2)

The microbenchmarks (Section 5.1) test system capabilities in isolation, and the building block experiments (Section 5.2) isolate training, serving, and simulation. However, the end-to-end applications (Section 5.3) are limited to Evolution Strategies and PPO. This leaves open questions about Ray's performance on:
- Hyperparameter tuning without RL
- Distributed data preprocessing pipelines
- Mixed workloads combining batch and interactive processing
- Multi-tenant cluster scenarios with competing applications

### 6.3 Scalability Boundaries

The experimental evaluation demonstrates impressive scalability, but several boundaries and open questions remain.

**Boundary 1: Evaluation Ceiling at 100 Nodes**

The scalability benchmark in **Figure 8b** demonstrates linear scaling up to 100 nodes (1.8 million tasks/sec), but the paper explicitly notes:

> "The rightmost datapoint shows that Ray can process 100 million tasks in less than a minute (54s), with minimum variability. As expected, increasing task duration reduces throughput proportionally to mean task duration, but the overall scalability remains linear. **x ∈ {70, 80, 90} omitted due to cost.**" (Section 5.1, Figure 8b caption)

The omission of intermediate datapoints due to cost is understandable but limits confidence in extrapolation beyond 100 nodes. Key questions remain:
- Does linearity hold at 500+ nodes?
- Does GCS sharding scale indefinitely, or does coordination overhead emerge?
- How does network topology affect performance at larger scales?

**Boundary 2: GCS Memory Growth Requires Active Management**

**Figure 10b** demonstrates that without GCS flushing, memory consumption grows linearly with task count until the system fails:

> "Without GCS flushing, the memory footprint reaches a maximum capacity and the workload fails to complete within a predetermined duration (indicated by the red cross)." (Section 5.1, Figure 10b caption)

With flushing enabled, memory is capped at a "user-configurable level," but this introduces tradeoffs:
- More aggressive flushing reduces memory but increases disk I/O
- Flushing frequency affects recovery time after failures (less lineage in memory = more disk reads)
- The paper notes garbage collection policies are "actively developing" (Section 7), indicating this is not a solved problem

For long-running applications (days to weeks), GCS memory management becomes a critical operational concern that requires tuning.

**Boundary 3: Scheduler Performance Degrades with Added Latency**

**Figure 12b** quantifies how sensitive Ray is to scheduler latency:

> "Ray's scheduler performance is critical to implementing primitives such as allreduce. In Figure 12b, we inject artificial task execution delays and show that performance drops nearly 2× with just a few ms of extra latency." (Section 5.1)

This sensitivity creates operational constraints:
- Network latency between schedulers and GCS must be minimized
- GCS must respond within sub-millisecond timeframes under load
- Scheduler replication (for high availability) must not introduce coordination overhead

The paper notes that global scheduler replicas "share the same information via GCS" (Section 4.2.2), but does not quantify the coordination overhead of multiple scheduler instances reading from the same sharded store.

### 6.4 Open Questions and Future Work

The paper explicitly identifies several areas for future improvement, which represent acknowledged limitations.

**Open Question 1: Garbage Collection for Lineage Storage**

> "Storing lineage for each task requires the implementation of garbage collection policies to bound storage costs in the GCS, a feature we are actively developing." (Section 7)

This is a critical open problem. Without garbage collection:
- Long-running applications accumulate unbounded lineage
- GCS flushing helps but does not address the fundamental growth problem
- Determining which lineage can be safely deleted requires tracking object dependencies across the entire computation graph

The complexity of this problem is non-trivial, as lineage deletion must not compromise fault tolerance guarantees.

**Open Question 2: Scheduling Without Full Graph Knowledge**

> "Given the workload generality, specialized optimizations are hard. For example, we must make scheduling decisions without full knowledge of the computation graph. Scheduling optimizations in Ray might require more complex runtime profiling." (Section 7)

This limitation contrasts with static graph frameworks (TensorFlow, MXNet) that can optimize placement with complete graph knowledge. Ray's dynamic graph model enables flexibility but sacrifices:
- Global optimization opportunities (e.g., co-locating dependent tasks proactively)
- Resource reservation across task chains
- Predictive scaling based on known graph structure

The paper suggests "runtime profiling" as a potential solution but does not provide implementation details or evaluation.

**Open Question 3: Actor Reconstruction Optimization**

> "In the future, we hope to further reduce actor reconstruction time, e.g., by allowing users to annotate methods that do not mutate state." (Section 5.1)

Currently, actor recovery requires re-executing all methods since the last checkpoint. The proposed optimization—annotating read-only methods—would enable:
- Skipping re-execution of methods that don't modify state
- More granular recovery checkpoints
- Reduced recovery time for large actor state

However, this requires additional API complexity and developer annotations, which may reduce usability.

**Open Question 4: Small Object Allreduce Performance**

> "For smaller objects, OpenMPI outperforms Ray by switching to a lower overhead algorithm, an optimization we plan to implement in the future." (Section 5.1)

This acknowledgment reveals that Ray's allreduce implementation is not universally optimal. The performance gap for small objects suggests:
- Ray's multi-threaded transfer strategy favors large objects
- Algorithm selection based on object size is not yet implemented
- There is room for optimization in communication primitives

### 6.5 Summary of Trade-offs

The table below summarizes the key trade-offs inherent in Ray's design:

| Design Choice | Benefit | Cost/Limitation |
|--------------|---------|-----------------|
| GCS for control state | Stateless components, independent scaling | Additional network hop for metadata lookups |
| Immutable objects | Simplified consistency, easier fault tolerance | No distributed objects, manual partitioning required |
| Lineage-based recovery | Lower storage overhead, deterministic replay | Computation overhead for expensive tasks |
| Bottom-up scheduling | Locality + scalability balance | May miss global optimization opportunities |
| Low-level API | Flexibility for diverse workloads | Higher developer burden vs. specialized frameworks |
| Dynamic task graphs | Supports RL's dynamic execution | Cannot optimize with full graph knowledge |

These trade-offs reflect Ray's prioritization of **flexibility and fine-grained performance** over **abstraction and automatic optimization**. For RL workloads requiring tight coupling of simulation, training, and serving, these trade-offs are favorable. For other workload types, the limitations may be more significant.

## 7. Implications and Future Directions

This paper does more than present a new system—it identifies a fundamental shift in how distributed AI applications should be architected. By demonstrating that a general-purpose framework can outperform specialized one-off systems while requiring less engineering effort, Ray challenges the prevailing assumption that performance requires specialization. This section examines how Ray reshapes the distributed AI landscape, what research directions it opens, and when practitioners should adopt this approach.

### 7.1 Shifting the Distributed AI Systems Paradigm

Ray's most significant contribution is establishing that **flexibility and performance are not mutually exclusive** in distributed AI systems. Prior to this work, the field operated under an implicit tradeoff:

| Prior Belief | Ray's Demonstration |
|-------------|---------------------|
| Specialized systems outperform general frameworks | Ray matches or exceeds specialized systems (Figure 14a: 2.7× faster ES) |
| Fine-grained tasks require centralized coordination | Bottom-up scheduling achieves 1.8M tasks/sec without central bottleneck (Figure 8b) |
| Fault tolerance imposes prohibitive overhead | Lineage-based recovery maintains stable throughput during failures (Figure 11a) |
| Tight coupling requires custom engineering | 7 lines of code to parallelize ES vs. hundreds in reference implementation (Section 5.3.1) |

This paradigm shift has three concrete implications for the field:

**1. Unified Infrastructure Reduces Research Burden**

The paper quantifies the engineering burden that Ray eliminates. For Evolution Strategies, the reference implementation required "several hundred lines of code dedicated to a protocol for communicating tasks and data between workers" (Section 5.3.1). In contrast, the Ray implementation required modifying only **7 lines of code** from a serial implementation. This 50–100× reduction in infrastructure code means researchers can focus on algorithm design rather than distributed systems engineering.

The implication extends beyond ES. Section 7 notes that Ray has been used to implement "many state-of-the-art RL algorithms," including A3C, PPO, DQN, ES, DDPG, and Ape-X. The unified abstraction means each new algorithm does not require rebuilding the distributed infrastructure from scratch.

**2. Dynamic Execution Enables Algorithmic Innovation**

Ray's dynamic task graph model enables optimizations that are impractical in static systems. The hierarchical aggregation tree used in the ES implementation (Section 5.3.1) would require "substantial engineering to port to custom-built systems" but is "transparently supported by Ray's dynamic task graph execution engine" (Section 5.3).

This suggests a broader principle: **system flexibility enables algorithmic flexibility**. When the computation graph can evolve during execution, researchers can implement adaptive strategies (e.g., dynamic batch sizes, early stopping, hierarchical aggregation) without restructuring the entire system.

**3. Cost Efficiency Through Heterogeneity Awareness**

The PPO benchmark demonstrates that heterogeneity-aware scheduling has direct economic impact. By allowing CPU-only simulation tasks to run on cheaper high-CPU instances while reserving GPUs for training, Ray reduced costs by **4.5×** through resource mapping alone (Section 5.3.2). Combined with fault tolerance enabling spot instance usage (4× cheaper), the total cost reduction reached **18×**.

This quantifies a previously anecdotal claim: systems that understand workload heterogeneity can dramatically reduce cloud computing costs. The implication is that future AI infrastructure should expose resource requirements at task granularity rather than assuming homogeneous clusters.

### 7.2 Follow-Up Research Directions

The paper explicitly identifies several open problems that represent natural extensions of this work. These directions fall into three categories: system improvements, workload extensions, and theoretical foundations.

#### 7.2.1 System Improvements

**Garbage Collection for Lineage Storage**

The paper states: "Storing lineage for each task requires the implementation of garbage collection policies to bound storage costs in the GCS, a feature we are actively developing" (Section 7). This is a critical open problem because:

- Without garbage collection, long-running applications accumulate unbounded lineage
- GCS flushing (Figure 10b) caps memory but does not address fundamental growth
- Determining which lineage can be safely deleted requires tracking object dependencies across the entire computation graph

Follow-up research could explore:
- Reference counting schemes for lineage objects
- Time-based retention policies with configurable windows
- Dependency-aware garbage collection that identifies orphaned lineage

**Scheduling with Partial Graph Knowledge**

The authors note: "We must make scheduling decisions without full knowledge of the computation graph. Scheduling optimizations in Ray might require more complex runtime profiling" (Section 7). This contrasts with static graph frameworks (TensorFlow, MXNet) that optimize placement with complete graph knowledge.

Potential research directions include:
- Runtime profiling to learn task duration distributions
- Predictive scheduling based on historical execution patterns
- Hybrid approaches that combine static hints with dynamic adaptation

**Actor Reconstruction Optimization**

The paper suggests: "In the future, we hope to further reduce actor reconstruction time, e.g., by allowing users to annotate methods that do not mutate state" (Section 5.1). Currently, actor recovery requires re-executing all methods since the last checkpoint.

Follow-up work could explore:
- API extensions for read-only method annotations
- Incremental checkpointing based on state mutation tracking
- Selective replay that skips non-mutating operations

**Small Object Communication Optimization**

The authors acknowledge: "For smaller objects, OpenMPI outperforms Ray by switching to a lower overhead algorithm, an optimization we plan to implement in the future" (Section 5.1). This reveals that Ray's multi-threaded transfer strategy favors large objects.

Research opportunities include:
- Adaptive algorithm selection based on object size
- Batched communication primitives for small messages
- Integration with specialized collective communication libraries (e.g., NCCL)

#### 7.2.2 Workload Extensions

**Beyond Reinforcement Learning**

While Ray is positioned as general-purpose, the evaluation focuses almost exclusively on RL workloads (Section 5.5.2). The paper explicitly notes this limitation: "All end-to-end benchmarks are RL-focused."

Follow-up research should evaluate Ray on:
- Hyperparameter tuning without RL (e.g., population-based training)
- Distributed data preprocessing pipelines
- Mixed workloads combining batch and interactive processing
- Multi-tenant cluster scenarios with competing applications

**Integration with Deep Learning Frameworks**

The paper demonstrates integration with TensorFlow (Figure 13) but notes this requires "a custom TensorFlow operator to write tensors directly to Ray's object store" (Section 5.2.1). This suggests room for deeper integration.

Potential directions include:
- Native operators in PyTorch, MXNet, and JAX
- Zero-copy tensor sharing between frameworks and Ray
- Automatic gradient aggregation through Ray's actor model

#### 7.2.3 Theoretical Foundations

**Scheduling Guarantees for Dynamic Graphs**

Ray's bottom-up scheduler achieves empirical scalability (Figure 8b), but the paper does not provide theoretical bounds on scheduling quality. Follow-up work could explore:

- Competitive ratios compared to optimal offline scheduling
- Bounds on locality violations under adversarial workloads
- Convergence guarantees for distributed RL algorithms on Ray

**Fault Tolerance Complexity Analysis**

The lineage-based recovery mechanism trades computation for storage, but the paper does not characterize the break-even point. Research could address:

- When is lineage-based recovery cheaper than replication?
- How does checkpoint frequency affect expected recovery time?
- What are the storage complexity bounds for different workload patterns?

### 7.3 Practical Applications and Downstream Use Cases

Based on the paper's evaluation and discussion, several concrete application patterns emerge where Ray provides clear advantages.

#### 7.3.1 When to Use Ray

**Scenario 1: Tightly Coupled Simulation-Training Loops**

Ray excels when simulation, training, and serving must operate within a single application with low-latency communication between components. Examples include:

- Reinforcement learning with environment simulators (Section 5.3)
- Robotics control with real-time policy evaluation (Section 1 mentions UAVs [42] and robotic manipulation [25, 60])
- Game AI with in-game simulation (Section 1 mentions AlphaGo [54])

The key indicator is **bidirectional data flow**: simulation results feed training, and updated policies immediately affect simulation.

**Scenario 2: Heterogeneous Resource Requirements**

Ray's resource-aware scheduling provides value when workloads have asymmetric hardware needs. The PPO benchmark demonstrates this (Section 5.3.2):

- CPU-intensive simulation tasks → CPU-only instances
- GPU-intensive training tasks → GPU-equipped instances
- Mixed workloads → Asymmetric cluster configurations

Use Ray when you cannot assume homogeneous nodes or when cost optimization requires mixing instance types.

**Scenario 3: Fine-Grained Task Parallelism**

Ray's million-task-per-second throughput (Figure 8b) benefits workloads with many short-duration tasks. Indicators include:

- Task durations in the millisecond range (Section 2 footnote calculates 5ms tasks)
- Millions of tasks per training iteration (Section 5.3.1 mentions 10,000 tasks for ES)
- Dynamic task creation based on intermediate results (Section 3.2 dynamic graph model)

**Scenario 4: Rapid Algorithm Prototyping**

The 7-line parallelization of ES (Section 5.3.1) demonstrates Ray's value for research iteration. Use Ray when:

- You need to experiment with multiple algorithm variants
- Infrastructure code should not dominate development time
- You want to reuse distributed primitives across algorithms

#### 7.3.2 When to Use Alternatives

The paper is explicit about Ray's limitations (Section 1, Section 6). Alternatives are preferable in these scenarios:

**Scenario 1: Production Model Serving**

> "Ray does not aim to substitute for serving systems like Clipper [19] and TensorFlow Serving [6], as these systems address a broader set of challenges in deploying models, including model management, testing, and model composition." (Section 1)

Use dedicated serving systems when you need:
- A/B testing across model versions
- Canary deployments with traffic splitting
- Multi-tenant isolation
- External API access with authentication

Ray excels at **embedded serving** (co-located with simulation), not production serving infrastructure.

**Scenario 2: Bulk Data Processing**

> "Ray is not a substitute for generic data-parallel frameworks, such as Spark [64], as it currently lacks the rich functionality and APIs (e.g., straggler mitigation, query optimization) that these frameworks provide." (Section 1)

Use Spark or similar frameworks when you need:
- SQL-like query interfaces
- Automatic straggler detection and mitigation
- Complex join optimizations
- Mature ecosystem of data connectors

**Scenario 3: Large Distributed Objects**

> "For simplicity, our object store does not support distributed objects, i.e., each object fits on a single node." (Section 4.2.3)

Use alternative systems when you need:
- Single objects larger than node memory
- Native distributed matrix or tensor abstractions
- Automatic object partitioning across nodes

**Scenario 4: Static Computation Graphs**

If your workload has a fixed computation structure known before execution, static graph frameworks (TensorFlow, MXNet) may provide better optimization opportunities through:
- Global placement optimization
- Operation fusion
- Memory planning with complete graph knowledge

$$

Practitioners should evaluate:

For teams considering adoption:

1. **Start with simulation workloads**: The stateless task model is simplest to adopt (Section 3.1)

- Exact GCS shard count and replication factor

Practitioners should expect to tune these parameters for their workloads.

### 7.5 Long-Term Vision

The paper concludes with a forward-looking statement: "Ray provides a powerful combination of flexibility, performance, and ease of use for the development of future AI applications" (Section 8). This vision extends beyond the evaluated RL workloads.

**Emerging AI Application Patterns**

Section 1 identifies characteristics of next-generation AI applications:
- Continuous interaction with the environment
- Learning from interactions in real-time
- Dynamic adaptation to changing conditions

Ray's architecture—dynamic task graphs, heterogeneous resource support, low-latency scheduling—aligns with these requirements. The implication is that Ray (or systems following similar design principles) will become foundational infrastructure for:

- Autonomous systems (UAVs, robotics)
- Interactive AI (dialogue systems, game AI)
- Real-time decision making (trading, ad placement)
- Continual learning systems

**Design Principles for Future Systems**

Ray's innovations suggest design principles for future distributed AI systems:

1. **Decouple control state from computation** (GCS architecture, Section 4.2.1)
2. **Support both stateless and stateful abstractions** (tasks + actors, Section 3.1)
3. **Enable dynamic graph evolution** (not static DAGs, Section 3.2)
4. **Optimize for fine-grained latency** (millisecond tasks, Section 2)
5. **Provide transparent fault tolerance** (lineage-based recovery, Section 4.2.3)

These principles extend beyond Ray and represent a blueprint for next-generation AI infrastructure.

**Open Ecosystem Development**

Section 7 notes that "several hundreds of people have used it and several companies are running it in production." This early adoption suggests Ray's potential to become a community standard, similar to how Spark became standard for batch processing.

The open-source nature (GitHub repository, Section 4.2.4) enables:
- Community-contributed integrations (deep learning frameworks, simulators)
- Third-party libraries building on Ray primitives
- Production hardening through real-world deployment feedback

The long-term impact depends on whether this ecosystem develops sufficient breadth to support diverse AI workloads beyond the RL focus of this paper.
