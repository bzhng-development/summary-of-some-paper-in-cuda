## 1. Executive Summary

FlashFFTConv addresses the hardware inefficiency of long-filter convolutions, which lag behind Transformers despite strong reasoning abilities, by decomposing the FFT into matrix multiplications compatible with GPU Tensor Cores to reduce I/O costs and enable kernel fusion. This optimization speeds up exact FFT convolutions by up to 7.93× over PyTorch and enables end-to-end model speedups of 4.4×, allowing Hyena-GPT-s to achieve 2.3 points better perplexity on the PILE dataset and M2-BERT-base to match models with twice the parameters on GLUE. Furthermore, it unlocks unprecedented sequence lengths, achieving 96.1% accuracy on the Path-512 vision task and enabling the first DNA model to process the longest human genes (2.3M base pairs) at single-nucleotide resolution.

## 2. Context and Motivation

### The Long Sequence Modeling Challenge

A fundamental challenge in modern machine learning is efficiently reasoning over long sequences. This problem appears across multiple domains:

- **Language modeling**: Processing documents with tens of thousands of tokens
- **DNA modeling**: Analyzing genomic sequences at single-nucleotide resolution (up to millions of base pairs)
- **Audio processing**: Working with raw waveforms sampled at high frequencies (e.g., 64 KHz)
- **Computer vision**: Treating high-resolution images as flattened sequences

Convolutional sequence models have emerged as a compelling alternative to Transformers for these tasks. Unlike classical convolutions in vision (which use short filters like $3 \times 3$ or $7 \times 7$), sequence modeling convolutions often use **filters as long as the input sequence itself**. This design choice enables better scaling with sequence length and greater training stability compared to attention mechanisms.

However, despite demonstrating state-of-the-art reasoning abilities, convolutional models face a critical adoption barrier: they **lag behind optimized Transformers in wall-clock time**. This performance gap limits their practical deployment, even when they offer theoretical advantages.

### The FFT Convolution Algorithm: Asymptotically Efficient, Practically Slow

To understand the bottleneck, we must first understand how long convolutions are computed. A direct convolution between input $u$ and kernel $k$ requires:

$$(u * k)[i] = \sum_{j} u[j] k[i - j]$$

This naive approach incurs $O(N^2)$ FLOPs for sequence length $N$—prohibitively expensive for long sequences. The standard solution uses the **Fast Fourier Transform (FFT)** to convert convolution into frequency-domain multiplication:

$$u * k = \mathcal{F}^{-1}(\mathcal{F}u \odot \mathcal{F}k)$$

where $\mathcal{F}$ denotes the FFT and $\odot$ is elementwise multiplication. This reduces computational complexity to $O(N \log N)$, which is asymptotically efficient.

> **Key misconception**: Asymptotic efficiency does not guarantee fast wall-clock time. The FFT convolution algorithm has poor hardware utilization on modern accelerators despite its favorable complexity.

### Why FFT Struggles on Modern GPUs

The paper identifies two fundamental bottlenecks that emerge as sequence length increases:

#### Bottleneck 1: Poor Tensor Core Utilization

Modern GPUs (since V100) include specialized **Tensor Cores**—matrix-matrix multiply units that dramatically outperform general-purpose compute:

| Compute Unit | H100 Performance |
|--------------|------------------|
| Tensor Cores | 1.0 PetaFLOP/s |
| General Arithmetic | 67 TeraFLOP/s |

Traditional FFT implementations (like cuFFT) do not effectively map to Tensor Cores. They rely on general arithmetic operations, leaving the most powerful compute units underutilized. This represents a **15× performance gap** that the paper aims to close.

#### Bottleneck 2: Memory Hierarchy I/O Costs

GPUs have a multi-level memory hierarchy (Figure 1, Left):

| Memory Level | Size | Speed |
|--------------|------|-------|
| Registers | 64 KB | Fastest |
| SRAM (Shared Memory) | ~20 MB per SM | Medium |
| HBM (Global Memory) | 40-80 GB | Slowest |

For short sequences, intermediate FFT results fit in SRAM, enabling **kernel fusion**—loading data once, performing multiple operations, and writing results back. However, as sequences grow:

1. Intermediate outputs exceed SRAM capacity
2. Kernel fusion fails
3. Expensive I/O between HBM and registers dominates runtime

The paper notes that sequences become "too large to fit in SRAM, and kernel fusion fails, resulting in expensive I/O costs." This I/O overhead is exacerbated by:
- Padding operations required for causal convolutions
- Conversions between real-valued inputs and complex-valued FFT intermediates

### Prior Approaches and Their Limitations

#### Standard PyTorch FFT Convolution

The baseline implementation uses PyTorch's FFT operations without hardware-specific optimizations. For short sequences, this works adequately because:
- Filter FFTs can be pre-computed and reused across batches: $k_f = \mathcal{F}k$
- Operations are parallel across batches and filters
- Intermediate outputs cache in SRAM via kernel fusion

However, this approach **does not scale** to longer sequences due to the bottlenecks described above.

#### cuFFTdx (NVIDIA's Optimized Library)

NVIDIA provides cuFFTdx, a library for kernel-fused FFT operations. The paper evaluates this as an ablation baseline ("Fusion-Only/cuFFTdx" in Table 3). While cuFFTdx improves over standard PyTorch, it:
- Still relies on general arithmetic rather than Tensor Cores
- Cannot support sequences longer than 32K due to SRAM limitations
- Achieves only modest speedups compared to FlashFFTConv (e.g., 2.85× vs. 4.78× at 32K sequence length)

#### Prior Tensor Core FFT Work

The paper acknowledges existing research on using Tensor Cores for FFT convolution (citing works [43, 44, 69]). However, these approaches:
- Do not scale to the sequence lengths FlashFFTConv achieves (up to 4 million)
- Lack the systematic cost modeling for choosing decomposition order
- Do not integrate architectural extensions like partial and frequency-sparse convolutions

### Positioning Relative to Existing Work

FlashFFTConv positions itself as a **systems-level optimization** that enables algorithmic innovation—drawing a direct parallel to FlashAttention's impact on Transformer models:

> "Just as systems advances such as FlashAttention yielded improvements in modeling quality and the development of new attention algorithms, we hope that understanding how to optimize the FFT convolution can also inspire algorithmic innovation."

This positioning is significant for two reasons:

1. **Hardware-aware algorithm design**: Rather than treating the FFT as a black box, FlashFFTConv restructures the algorithm to match GPU architecture (Tensor Cores, memory hierarchy). This is analogous to how FlashAttention restructured attention to minimize HBM accesses.

2. **Enabling new capabilities**: The optimization is not merely about speed—it unlocks previously infeasible sequence lengths. For example:
   - Path-512 (256K sequence length): No model previously achieved better than 50% accuracy; FlashFFTConv achieves 96.1%
   - DNA modeling: First model to process the longest human genes (2.3M base pairs) at single-nucleotide resolution

### The Monarch Decomposition Foundation

FlashFFTConv builds on the **Monarch FFT decomposition** from prior work [23]. An order-$p$ Monarch decomposition rewrites the $N \times N$ discrete Fourier matrix $\mathbf{F}_N$ as a series of $p$ matrix-matrix multiply operations:

$$\mathbf{F}_N = \mathbf{P}(\mathbf{I}_{N_2} \otimes \mathbf{F}_{N_1})\mathbf{D}\mathbf{P}^{-1}(\mathbf{I}_{N_1} \otimes \mathbf{F}_{N_2})\mathbf{P}$$

where $\otimes$ denotes the Kronecker product, $\mathbf{P}$ is a permutation matrix, and $\mathbf{D}$ contains "Twiddle factors" (correction values).

**Key insight**: This decomposition transforms the FFT from a specialized signal processing operation into a sequence of matrix multiplications—operations that Tensor Cores excel at. However, the original Monarch formulation broadcasts across batch and hidden dimensions, which limits sequence length due to SRAM constraints. FlashFFTConv's innovation is **adapting this decomposition to broadcast across the sequence dimension instead**, enabling kernel fusion at much longer sequence lengths.

### Why This Matters Beyond Convolution Speed

The motivation extends beyond making convolutions faster:

1. **Architectural diversity**: If convolutions remain slower than attention, researchers will default to Transformers even when convolutions offer theoretical advantages (stability, length scaling). Efficient convolutions enable fair architectural comparison.

2. **New application domains**: DNA modeling at single-nucleotide resolution requires processing millions of base pairs. Without FlashFFTConv, such models are memory-prohibited or prohibitively slow.

3. **Compute budget efficiency**: For the same compute budget, FlashFFTConv allows models to see more training data, directly improving downstream quality (2.3 points better perplexity, 3.3 points higher GLUE score).

4. **Sparsity analogues**: The matrix view of FFT convolution enables "partial convolutions" and "frequency-sparse convolutions"—convolutional analogues to sparse attention in Transformers. These can be implemented by simply skipping blocks in the matrix decomposition.

In summary, FlashFFTConv addresses a critical gap between the theoretical promise of long convolutions and their practical deployment. By restructuring the FFT to match modern GPU architecture, it closes the wall-clock time gap with Transformers while unlocking new sequence length regimes previously inaccessible to convolutional models.

## 3. Technical Approach

### 3.1 Reader orientation (approachable technical breakdown)

FlashFFTConv is a hardware-aware system that restructures how Fast Fourier Transform (FFT) convolutions are computed on modern GPUs by decomposing the FFT into matrix multiplications that run on Tensor Cores. It solves the problem of poor hardware utilization in long-sequence convolutions by transforming the FFT from a memory-bound operation into a compute-bound operation that can be kernel-fused, reducing expensive data movement between GPU memory levels while enabling sequence lengths up to 4 million tokens.

### 3.2 Big-picture architecture (diagram in words)

The FlashFFTConv system consists of four major components that process data in sequence. First, the **Monarch FFT Decomposition Engine** takes the input sequence and convolution kernel, then rewrites the FFT operation as a series of matrix-matrix multiplications parameterized by an order-$p$ value that controls the number of decomposition steps. Second, the **Kernel Fusion Scheduler** determines which operations can be fused together in SRAM based on sequence length and available memory, loading data from HBM once and performing multiple computation steps before writing back. Third, the **Tensor Core Mapper** converts the decomposed matrix operations into Tensor Core-compatible formats (16×16×16 blocks on A100/H100), ensuring the specialized matrix multiply units are fully utilized. Fourth, the **Architectural Extension Module** optionally applies partial convolutions or frequency-sparse convolutions by skipping specific blocks in the matrix decomposition, reducing both memory footprint and compute cost. Data flows from HBM through SRAM into registers where Tensor Core operations execute, then results flow back through the inverse decomposition path to produce the final convolution output.

### 3.3 Roadmap for the deep dive

- We begin with the core Monarch FFT decomposition adaptation because understanding how the FFT becomes matrix multiplication is prerequisite to all subsequent optimizations.
- We then explain the kernel fusion strategy since it depends on the decomposition structure and determines which sequence lengths are feasible.
- We cover the cost model next because it explains how to choose the decomposition order-$p$ for different sequence lengths, which is critical for performance.
- We describe domain-specific optimizations (real-valued FFT, padding handling) since these build on the fused kernel structure to extract additional speedup.
- We conclude with the architectural extensions (partial and frequency-sparse convolutions) because they leverage the matrix view established in earlier sections to enable new capabilities.

### 3.4 Detailed, sentence-based technical breakdown

This is a systems optimization paper that restructures the FFT convolution algorithm to match modern GPU architecture, with the core idea being that the FFT can be decomposed into matrix multiplications that run on Tensor Cores rather than general arithmetic units.

#### The Monarch FFT Decomposition: From Signal Processing to Matrix Multiplication

The foundational insight is that an $N \times N$ discrete Fourier transform matrix $\mathbf{F}_N$ can be factored into a product of smaller matrices using the Monarch decomposition from prior work [23]. For an order-2 decomposition where $N = N_1 \times N_2$, the Fourier matrix is rewritten as:

$$\mathbf{F}_N = \mathbf{P}(\mathbf{I}_{N_2} \otimes \mathbf{F}_{N_1})\mathbf{D}\mathbf{P}^{-1}(\mathbf{I}_{N_1} \otimes \mathbf{F}_{N_2})\mathbf{P}$$

where $\otimes$ denotes the Kronecker product (a way of combining two matrices into a larger block matrix), $\mathbf{P}$ is a permutation matrix that reshapes and transposes the input, $\mathbf{D}$ is a diagonal matrix containing "Twiddle factors" (complex correction values that account for phase shifts), and $\mathbf{F}_{N_1}$ and $\mathbf{F}_{N_2}$ are smaller Fourier matrices. Figure 2 illustrates this decomposition visually for $N=16$ with $N_1 = N_2 = 4$, showing five steps: reshape input, FFT on columns, apply Twiddle factors, FFT on rows, and transpose output.

The critical observation is that each $\mathbf{F}_{N_i}$ operation is now a matrix-matrix multiplication rather than a specialized FFT operation. This matters because modern GPUs have Tensor Cores that compute matrix multiplication at 1.0 PetaFLOP/s on H100, compared to only 67 TeraFLOP/s for general arithmetic—a 15× performance difference that the paper exploits. Higher-order decompositions (order-3, order-4) recursively apply the order-2 decomposition to the smaller Fourier matrices, which reduces FLOP cost through smaller matrices but increases I/O cost due to more intermediate results that must be stored and retrieved.

#### Adapting Monarch for Kernel Fusion: Broadcasting Across Sequence Dimension

The original Monarch FFT decomposition broadcasts matrix operations across the batch dimension $B$ and hidden dimension $H$, as shown in Figure 3 (top left). This traditional approach requires loading at least 16 sequences simultaneously into SRAM to fill out the Tensor Core matrix multiply unit, which limits feasible sequence length to around 2K on A100 and H100 GPUs. FlashFFTConv makes a crucial design change: it broadcasts the matrix operation across the entire sequence dimension instead, running the algorithm in parallel across batch and hidden dimensions as shown in Figure 3 (top right).

This broadcasting change has three concrete effects on the system. First, it reduces SRAM requirements for kernel fusion because only a single sequence needs to be loaded into SRAM at a time, allowing fusion for sequences up to 32K on A100 and H100. Second, the permutation operations $\mathbf{P}$ in the Monarch decomposition become simple matrix transpose operations that can be executed quickly using established on-chip routines, as illustrated in Figure 3 (bottom). Third, the computation is tiled across $B$ and $H$ dimensions to reduce the cost of loading the convolution kernel $k_f$, FFT matrices $\mathbf{F}$, and Twiddle factors from HBM.

Algorithm 1 provides the core algorithm for order-2 decomposition. The algorithm loads FFT matrices $\mathbf{F}$, $\mathbf{F}^{-1}$, Twiddle factors $t$, $t_{inv}$ from HBM once per Streaming Multiprocessor (SM). For each hidden dimension tile $h$ and batch tile $b$, it loads the input $X$ and kernel $K_f$ reshaped to $N_1 \times N_1$, computes the FFT as $((\mathbf{F}^\top X) * t)\mathbf{F}$, performs elementwise multiplication with the kernel $X * K_f^\top$, then computes the inverse FFT as $((X\mathbf{F}^{-1})^\top * t_{inv})\mathbf{F}^{-1}$, and finally writes the transposed output $Y^\top$ back to HBM. The asterisk $*$ denotes elementwise multiplication, and $\top$ denotes matrix transpose.

#### Kernel Fusion and Recomputation Strategy

The Monarch decomposition enables kernel fusion for long sequences because inner layers of the decomposition do not require the entire sequence to be present in SRAM simultaneously. For long sequences, FlashFFTConv fuses the innermost matrix operations and elementwise multiplications together, taking only one I/O operation for the outermost matrix operations. This is fundamentally different from standard PyTorch FFT convolution, which writes intermediate results to HBM after each operation (FFT, pointwise multiply, inverse FFT), incurring three separate HBM accesses.

For the backward pass, FlashFFTConv uses recomputation rather than storing intermediate activations. Instead of storing intermediate results like $\mathbf{F}u$ on HBM for gradient computation, the system recomputes them during the backward pass. This trades additional compute for reduced memory footprint and I/O cost—a common optimization pattern also used in checkpointing [65, 111]. The memory savings from this approach are substantial: Table 16 shows FlashFFTConv uses 8.21× less memory than PyTorch at sequence length 256, and still maintains 2.63× savings at sequence length 4M.

#### Cost Model for Choosing Decomposition Order-$p$

Section 3.2 presents a formal cost model that determines the optimal order-$p$ decomposition based on sequence length, analogous to a roofline analysis [54]. The model accounts for both compute cost and I/O cost, recognizing that different sequence lengths favor different decomposition strategies. Let $B$ be batch size, $H$ be hidden dimension, $N$ be sequence length (assumed to be a power of 2), and $N = \prod_{i=1}^p N_i$ be the product of $p$ factors. Let $\mu$ be the Tensor Core matrix size (16 for A100 and H100). Let $\tau_G$ and $\tau_M$ be empirically-achievable FLOPs for general-purpose and matrix-matrix multiply arithmetic respectively. Let $\sigma_H$ and $\sigma_S$ be empirically-achievable bandwidth for HBM and SRAM respectively. Table 19 in Appendix C provides measured values for A100-40GB: $\sigma_H = 1.35$ TB/s, $\sigma_S = 9.5$ TB/s, $\tau_M = 234$ TFLOPs, $\tau_G = 17.6$ TFLOPs.

The cost equation (Equation 2) is:

$$C = BH \sum_{i=1}^p \left( \frac{16NN_i}{\gamma(N_i)} + \frac{4N}{\omega(i)} \right)$$

where $\gamma(N_i)$ returns $\tau_G$ if $N_i &lt; \mu$ (matrix too small for Tensor Cores) and $\tau_M$ if $N_i \geq \mu$ (matrix large enough for Tensor Cores), and $\omega(i)$ returns the bandwidth of the memory level where intermediate results of decomposition step $i$ are stored. The first term represents compute cost (FLOPs divided by achievable FLOP rate), and the second term represents I/O cost (bytes divided by achievable bandwidth).

Figure 4 graphs this cost model for $p \in \{2, 3, 4\}$ across sequence lengths from 256 to 4M on A100. The graph reveals two critical tradeoff points. First, there are early bumps for higher-order decompositions at short sequence lengths where matrices become smaller than the Tensor Core unit size ($\mu = 16$), making general arithmetic slower than matrix multiplication. Second, there is a bump for $p=3$ between 32K and 64K where SRAM capacity is exceeded, requiring an extra decomposition ($p=4$) to mediate the I/O cost. The cost per token $C/N$ shows that order-2 decomposition is optimal for sequences up to approximately 32K, while order-3 and order-4 become favorable for longer sequences despite increased I/O, because the smaller matrices maintain Tensor Core utilization.

#### Domain-Specific Optimizations for Sequence Learning Workloads

FlashFFTConv applies three domain-specific optimizations that exploit properties of convolutional sequence models in machine learning. First, since convolutions in sequence learning are real-to-real (real-valued inputs and outputs with real kernel weights), the system uses a classic algorithm called one-stage decimation in time to compute the FFT of a sequence of length $N$ using a complex FFT of length $N/2$, as detailed in Appendix A.1. This algorithm exploits symmetry properties of the Fourier transform: for real input $x(n)$, the output satisfies $X(k) = X^*(N-k)$ where $*$ denotes complex conjugation. The implementation creates a complex sequence $z(n) = x(2n) + i x(2n+1)$ of length $N/2$, computes its DFT $Z(k)$, then recovers the even and odd parts using:

$$X_e[k] = \frac{Z[k] + Z^*[N/2 - k]}{2}$$

$$X_o[k] = \frac{-iZ[k] - Z^*[N/2 - k]}{2i}$$

The full DFT is then $X[k] = X_e[k \mod N/2] + X_o[k \mod N/2] W_N^k$ where $W_N = e^{-2\pi i/N}$. This cuts the FFT cost in half compared to using a complex FFT of length $N$.

Second, inputs and outputs are often zero-padded to compute causal convolutions (where output at position $i$ depends only on inputs at positions $j \leq i$). FlashFFTConv special-cases this padding to eliminate half of the outermost matrix multiply operations in the FFT and inverse FFT. When the input is zero-padded, certain portions of the matrix multiplication produce known zero outputs that can be skipped entirely. Table 13 and Table 14 in Appendix B show results for causal convolutions where input size is half the FFT size, demonstrating this optimization works across all sequence lengths.

Third, the system fuses additional operations around the convolution, such as elementwise gating. A common pattern in convolutional and state-space models is the gated convolution $y = v \odot ((u \odot w) * k)$ where $v$ and $w$ are linear projections of input $u$ [43, 44, 78, 94]. A PyTorch implementation incurs additional I/O overhead from loading and storing intermediate gating results, whereas FlashFFTConv fuses the gating operations into the convolution kernel. Table 4 shows this fusion achieves up to 7.93× speedup over PyTorch for gated convolutions at sequence length 1K, compared to 6.54× for standard convolutions in Table 3.

#### Low-Level CUDA Implementation Details

Appendix A.2 provides implementation specifics for achieving high performance on CUDA-enabled GPUs. Tensor Cores can multiply two matrices of dimensions $m \times k$ and $k \times n$ for bfloat16 or float16 elements, where $m \times k \times n$ must be one of: $16 \times 16 \times 16$, $32 \times 8 \times 16$, or $8 \times 32 \times 16$. FlashFFTConv breaks down matrix-matrix multiplications into blocked operations where $m \times k \times n = 16 \times 16 \times 16$, which informs the choice of radix for decomposition. Tensor Core operands are held in register fragments using the Warp Level Matrix Multiply Accumulate (WMMA) API, with operands in `wmma::matrix_a` and `wmma::matrix_b` fragments and results in `wmma::accumulator` fragments.

A key optimization is register reuse: the output from the accumulator fragment is moved directly into the `matrix_a` fragment for subsequent matrix multiplications, avoiding an extra trip to SRAM. This is only possible when the output does not need transposition before the next operation; when transposition is required, data must travel to SRAM and back. Algorithm 2 annotates the I/O movements from SRAM to register fragments for the two-way decomposition, showing exactly where data moves between memory levels. The algorithm is parallelized across batch $B$ and hidden $H$ dimensions, allowing tiling at the threadblock level with warp-tiling for inner loops.

#### Generalization to Higher-Order Decompositions

Appendix A.3 provides algorithms for 3-way and 4-way Monarch decompositions. For 3-way decomposition where $N = N_1^3$, Algorithm 3 adds one extra matrix multiply operation on either side of the FFT and inverse FFT compared to Algorithm 1, proceeding through an inner loop over $N_1$ iterations. The algorithm loads the kernel $K_f$ reshaped to $N_1 \times N_1^2$, transposes the last two dimensions, then reshapes to $N_1^2 \times N_1$. For each batch tile, it loads input $X$ reshaped to $N_1 \times N_1 \times N_1$, performs FFT over columns with Twiddle correction $t_2$, then loops over rows applying the 2-way FFT decomposition with kernel multiplication, applies inverse Twiddle $t_{2,inv}$, and performs inverse FFT over columns.

For 4-way decomposition where $N = N_1^4$, Algorithm 4 treats the 3-way decomposition as a subroutine and assumes intermediate outputs must be written to HBM due to SRAM capacity limits. The algorithm reshapes input $u$ to $B \times H \times N_1 \times (N/N_1)$ and kernel $k_f$ to $H \times N_1 \times (N/N_1)$, transposes the kernel, computes FFT over columns of $u$, reshapes to move $N_1$ into the $H$ dimension, calls the 3-way FlashFFTConv kernel over rows, then computes inverse FFT over columns. This hierarchical approach allows FlashFFTConv to scale to sequence lengths of 4 million while maintaining reasonable I/O costs.

#### Partial Convolutions: Learning with Shorter Kernels

Section 3.3 introduces partial convolutions as the first architectural extension, where later portions of the convolution kernel are zeroed out, analogous to local attention in Transformers. This has two concrete benefits. First, it reduces memory footprint because fewer kernel elements need to be held in GPU memory simultaneously. Second, it enables natural extension of pretrained models to longer sequences via a sliding window approach, since the kernel length is independent of the input sequence length.

Table 7 evaluates partial convolutions on Hyena-s-8K across different kernel lengths from 256 to 8K. The perplexity remains stable (13.8 to 14.2) even when reducing kernel length from 8K to 2K, while memory footprint drops from 32.5G to 11.8G—a 2.75× reduction. This demonstrates that a large proportion of convolution filters can be pruned without impacting downstream quality. Table 8 shows partial convolutions extending HyenaDNA-1M to 4M sequence length while maintaining perplexity (2.91 to 2.90), enabling the first model to embed the longest human genes (2.3M base pairs) at single-nucleotide resolution.

#### Frequency-Sparse Convolutions: Sparsity in Frequency Space

The second architectural extension is frequency-sparse convolutions, where portions of the convolution kernel $k_f$ are zeroed out in frequency space rather than time space. This can be viewed as a variant of partial convolutions operating in the frequency domain. The specific sparsity pattern yields computational benefits: zeroing out portions of the kernel obviates the need to compute corresponding portions of the matrix-matrix multiplies in the Monarch decomposition.

Appendix A.4 describes how sparsity along each dimension of a reshaped kernel enables skipping specific operations in the 4-way decomposition. For a kernel reshaped to $N_1 \times N_1 \times N_1 \times N_1$ (or $32 \times 32 \times 32 \times 64$ for 2M sequence length), sparsity in the first dimension allows skipping computation in matrix $B$ (the second FFT operation), sparsity in the second dimension allows skipping computation in matrix $A$ (the first FFT operation), sparsity in the third dimension reduces the inner loop iterations $\beta$, and sparsity in the fourth dimension reduces the outer loop iterations $\alpha$. Table 10 provides the specific sparsity patterns used in experiments, parameterized by variables $a, b, c, d$ describing how much of each dimension is set to zero, with sparsity fraction $S = 1 - \frac{(32-a)(32-b)(32-c)(64-d)}{32 \times 32 \times 32 \times 64}$.

Table 9 shows experimental results applying frequency-sparsity to pretrained HyenaDNA-1M. At 75% sparsity, perplexity actually improves slightly from 2.91 to 2.90, potentially due to removing high-frequency noise. At 79% sparsity, perplexity returns to 2.91 with 1.4× convolution speedup. Even at 91% sparsity, perplexity only degrades to 2.98 while achieving 1.8× speedup. This demonstrates that frequency-sparse convolutions can yield significant speedup while maintaining or even improving model quality.

#### Design Choices and Alternative Approaches

The paper makes several critical design choices that distinguish FlashFFTConv from alternative approaches. First, broadcasting across sequence dimension rather than batch/hidden dimension was chosen because it reduces SRAM requirements from 16 sequences to 1 sequence, enabling kernel fusion at 32K instead of 2K sequence length. The alternative (traditional broadcasting) would limit applicability to short sequences only.

Second, the order-$p$ decomposition is chosen adaptively based on sequence length rather than fixed. Figure 4 shows that order-2 is optimal for sequences up to 32K, but order-3 and order-4 become necessary for longer sequences to maintain Tensor Core utilization. A fixed order would either waste compute (too high for short sequences) or fail to scale (too low for long sequences).

Third, recomputation is used in the backward pass rather than storing all intermediates. This trades compute for memory, which is favorable on GPUs where compute capacity exceeds memory bandwidth. The alternative (storing all intermediates) would limit maximum sequence length due to HBM capacity constraints.

Fourth, real-valued FFT optimization is applied because sequence learning convolutions use real-valued inputs and kernels. The alternative (complex FFT) would waste 50% of compute on redundant complex arithmetic. This optimization is domain-specific and would not apply to signal processing applications with complex inputs.

Fifth, frequency-sparse convolutions zero out kernel portions in frequency space rather than time space. This choice enables skipping matrix multiplication blocks in the Monarch decomposition directly, whereas time-domain sparsity would require additional transformations. The tradeoff is that frequency-domain sparsity patterns are less interpretable than time-domain local attention patterns.

#### Integration with Existing Model Architectures

FlashFFTConv integrates with existing convolutional sequence models without requiring architectural changes. The system works with M2-BERT-base (BERT-style masked language modeling with 110M parameters, sequence length 128), Hyena-s (GPT-style next token prediction with 155M parameters, sequence length 4K), long convolution models for Path-X (102M parameters, sequence length 16K), SaShiMi audio generation (5.4M parameters, sequence length 64K), and HyenaDNA (sequence length 1M to 4M). Table 5 shows end-to-end speedups ranging from 1.3× (SaShiMi, where convolutions are interleaved with other operations) to 4.4× (HyenaDNA, where PyTorch is bottlenecked by small batch size).

For models using multiplicative gating (M2-BERT, Hyena), FlashFFTConv fuses the gating operations into the convolution kernel, as described in the domain-specific optimizations. For models requiring causal convolutions (language modeling, DNA), the system applies implicit padding without running extra padding operations. For models with very long sequences (HyenaDNA-4M), the system uses order-4 decomposition with HBM intermediate storage. This flexibility allows FlashFFTConv to accelerate diverse model architectures across four orders of magnitude in sequence length (256 to 4M) without requiring model-specific customization.

## 4. Key Insights and Innovations

FlashFFTConv introduces several contributions that distinguish it from prior FFT optimization work. While some components build on existing algorithms (like Monarch decomposition), the *integration* and *adaptation* for modern GPU architectures represent fundamental shifts in how we implement long convolutions. Below, we analyze the four most significant innovations, distinguishing between incremental engineering improvements and fundamental algorithmic changes.

### 4.1 Reorienting Parallelism for Memory Residency

The most critical systems innovation in FlashFFTConv is the decision to **broadcast parallelism across the sequence dimension** rather than the batch or hidden dimensions. This contradicts conventional wisdom in GPU kernel design, where maximizing batch parallelism is typically prioritized to hide latency.

*   **Prior Assumption:** Traditional Monarch decomposition [23] and standard FFT libraries parallelize matrix operations across the batch size $B$ and hidden dimension $H$. This assumes that keeping many sequences in flight is the best way to utilize compute units.
*   **The Innovation:** FlashFFTConv sacrifices batch-level parallelism granularity to ensure that a *single* sequence's intermediate data fits entirely within SRAM during the inner decomposition steps. As detailed in Section 3.1, this reduces the SRAM requirement from needing ~16 sequences simultaneously to just one.
*   **Why It Matters:** This design choice directly enables **kernel fusion** at much longer sequence lengths.
    *   Prior approaches (including cuFFTdx) fail to fuse operations beyond ~2K sequence length because intermediate activations overflow SRAM when broadcasting over batch.
    *   FlashFFTConv extends the fusion limit to **32K sequence length** on A100/H100 GPUs.
*   **Significance:** This is a **fundamental innovation** in memory hierarchy management. It demonstrates that for long-sequence models, optimizing for *data residency* (keeping data in SRAM) yields higher throughput than optimizing for *compute occupancy* (filling all SMs with batch items). This insight mirrors the philosophy of FlashAttention [24] but applies it to the distinct structural constraints of the FFT.

### 4.2 Hardware-Aware Adaptive Decomposition

While the Monarch decomposition provides a mechanism to rewrite FFT as matrix multiplications, FlashFFTConv introduces a **cost model-driven strategy** to select the decomposition order dynamically. This moves FFT optimization from a static kernel selection problem to a dynamic algorithm selection problem.

*   **Prior Approach:** Most FFT implementations use a fixed algorithm (e.g., Cooley-Tukey radix-2) regardless of sequence length. Even prior Tensor Core FFT work [69] typically fixes the decomposition strategy.
*   **The Innovation:** The paper derives a formal cost model (Equation 2 in Section 3.2) that balances compute cost (FLOPs) against I/O cost (memory bandwidth). This model accounts for the specific threshold where matrices become too small for Tensor Cores ($\mu = 16$) and where SRAM capacity is exceeded.
*   **Why It Matters:** The optimal decomposition order $p$ changes as sequence length $N$ grows.
    *   For short sequences ($N &lt; 32\text{K}$), order-$p=2$ minimizes I/O while keeping matrices large enough for Tensor Cores.
    *   For long sequences ($N > 32\text{K}$), higher orders ($p=3, 4$) become necessary to keep intermediate matrices small enough to fit in SRAM, despite the increased I/O overhead.
*   **Significance:** This is a **fundamental innovation** in algorithm selection. Figure 4 illustrates how the cost per token shifts at specific breakpoints (e.g., the bump at 32K for $p=3$). By automating this selection, FlashFFTConv achieves smooth scaling across **four orders of magnitude** (256 to 4M) without manual kernel tuning. This adaptive approach is rare in deep learning primitives, which typically rely on hardcoded kernels for specific sizes.

### 4.3 Frequency-Domain Sparsity Primitives

FlashFFTConv leverages its matrix-based view of the FFT to introduce **frequency-sparse convolutions**, a new architectural primitive that has no direct analogue in standard convolution libraries.

*   **Prior Approach:** Sparsity in convolutions is typically applied in the *time domain* (e.g., pruning kernel weights or using dilated convolutions). Sparse attention in Transformers operates on the attention matrix directly.
*   **The Innovation:** By viewing the FFT as a sequence of matrix multiplications, FlashFFTConv allows users to zero out portions of the kernel $k_f$ in *frequency space*. As described in Section 3.3 and Appendix A.4, this maps directly to skipping specific blocks in the Monarch matrix decomposition.
*   **Why It Matters:** This enables two distinct capabilities:
    1.  **Partial Convolutions:** Learning with kernels shorter than the input sequence. This allows extending pretrained models (like HyenaDNA-1M) to longer sequences (4M) via sliding window without retraining, as shown in Table 8.
    2.  **Frequency Sparsity:** Zeroing out high-frequency components. Table 9 shows that **75% sparsity** in frequency space maintains perplexity (2.90 vs. 2.91) while yielding **1.4× speedup** in convolution time.
*   **Significance:** This is a **fundamental algorithmic innovation**. It provides a mechanism to apply sparse attention concepts to convolutions without breaking the $O(N \log N)$ efficiency of the FFT. The ability to skip matrix blocks based on frequency content opens new avenues for model compression and architecture design that were previously inaccessible with black-box FFT libraries.

### 4.4 Efficiency as a Quality Multiplier

Beyond raw speedup, FlashFFTConv validates the hypothesis that **systems efficiency directly translates to model quality** under fixed compute budgets. This insight bridges the gap between systems research and machine learning outcomes.

*   **Prior Assumption:** Systems optimizations are often evaluated solely on wall-clock time or FLOP utilization. The assumption is that faster training saves money but does not necessarily improve the final model.
*   **The Innovation:** The evaluation in Section 4.1 explicitly compares models trained with the *same compute budget* (same total FLOPs), not the same number of epochs. Because FlashFFTConv achieves higher throughput, models see more data within the budget.
*   **Why It Matters:** The results show significant quality gains purely from efficiency:
    *   **Hyena-GPT-s:** Achieves **2.3 points better perplexity** on the PILE dataset (Table 1).
    *   **M2-BERT-base:** Achieves **3.3 points higher GLUE score**, matching models with **twice the parameter count** (Table 1).
*   **Significance:** This is a **critical empirical insight** for the ML community. It demonstrates that optimizing primitives like convolution is not just an engineering task but a pathway to better models. The 4.4× end-to-end speedup (Abstract) allows researchers to iterate faster or train larger contexts within the same budget, effectively shifting the Pareto frontier of model quality vs. compute cost.

### 4.5 Summary of Contributions

| Contribution | Type | Prior State | FlashFFTConv Innovation | Impact |
| :--- | :--- | :--- | :--- | :--- |
| **Sequence Broadcasting** | Fundamental | Batch/Hidden parallelism | Sequence parallelism for SRAM residency | Enables fusion up to 32K sequences (Section 3.1) |
| **Adaptive Decomposition** | Fundamental | Fixed FFT algorithms | Cost-model driven order-$p$ selection | Smooth scaling from 256 to 4M sequence length (Figure 4) |
| **Frequency Sparsity** | Fundamental | Time-domain pruning | Matrix block skipping in frequency space | 1.4× speedup, enables 4M DNA modeling (Table 9) |
| **Efficiency-Quality Link** | Empirical | Speed = Cost Savings | Speed = More Data = Better Quality | 2.3 PPL gain, matches 2× parameter models (Table 1) |

In summary, FlashFFTConv is not merely a faster FFT implementation; it is a rethinking of how convolutional primitives map to modern hardware. By prioritizing memory residency over batch parallelism, automating algorithm selection via cost modeling, and exposing frequency-domain sparsity, it enables capabilities (like 4M sequence modeling) that were previously infeasible. These innovations collectively demonstrate that hardware-aware algorithm design remains a potent lever for advancing machine learning capabilities.

## 5. Experimental Analysis

The authors evaluate FlashFFTConv across three dimensions: **quality improvements** under fixed compute budgets, **efficiency gains** in raw convolution speed and end-to-end model throughput, and **new capabilities** enabled by longer sequence lengths. This section walks through the experimental methodology, presents the key quantitative results, and critically assesses whether the evidence supports the paper's claims.

### 5.1 Evaluation Methodology

#### Datasets and Tasks

The evaluation spans four distinct domains, each representing a different sequence modeling challenge:

| Domain | Dataset/Task | Sequence Length | Model |
|--------|--------------|-----------------|-------|
| Language (GPT-style) | PILE | 4K tokens | Hyena-s-155M |
| Language (BERT-style) | C4 + GLUE | 128 tokens | M2-BERT-base-110M |
| Vision | Path-X, Path-512 (LRA benchmark) | 16K, 256K | Long convolution model |
| DNA | Human genes (Ensembl biotypes) | 1M to 4M base pairs | HyenaDNA |
| Audio | Raw waveform (64 KHz sampling) | 64K samples | SaShiMi-5.4M |

The **Path-512 task** is particularly challenging: it requires classifying whether two dots in a 512×512 image are connected by a path, with the image flattened into a 256K-token sequence. Prior to this work, no model achieved better than 50% accuracy (random chance) on this task.

#### Metrics

The paper uses domain-appropriate metrics:

- **Perplexity (PPL)** for language modeling (lower is better)
- **GLUE Score** for natural language understanding (higher is better, max 100)
- **Classification Accuracy** for Path-X/Path-512 (higher is better, max 100%)
- **Throughput** in sequences/second, tokens/second, images/second, or audio clips/second
- **Wall-clock time** in milliseconds for convolution benchmarks
- **Memory usage** in GB
- **FLOP utilization** as a percentage of theoretical peak

#### Baselines

The authors compare against three primary baselines:

1. **PyTorch FFT Convolution**: The standard implementation using PyTorch's `fft` operations without hardware-specific optimizations. This represents the common practice in the research community.

2. **cuFFTdx (Fusion-Only)**: NVIDIA's optimized library for kernel-fused FFT operations. This appears as an ablation baseline in Table 3 ("Fusion-Only/cuFFTdx") to isolate the benefit of Tensor Core utilization from kernel fusion alone.

3. **FlashAttention-v2**: For the Transformer comparison in Table 6, the authors use the official FlashAttention-v2 implementation [22] with a 2.7B-parameter GPT model matched to a 2.7B-parameter Hyena model.

#### Experimental Setup

All convolution benchmarks use **batch size 64** and **hidden dimension 768**, scaled across sequence lengths from 256 to 4 million. The hardware platforms are:

- **A100-40GB**: Used for most benchmarks and the cost model (Table 19 provides measured constants: $\sigma_H = 1.35$ TB/s HBM bandwidth, $\sigma_S = 9.5$ TB/s SRAM bandwidth, $\tau_M = 234$ TFLOPs Tensor Core, $\tau_G = 17.6$ TFLOPs general arithmetic)
- **H100-SXM**: Used for convolution speedup tables (Tables 3, 4, 11-15)

For the **fixed compute budget experiments** (Table 1), the key methodological choice is training for the same total FLOPs rather than the same number of epochs. Because FlashFFTConv achieves higher throughput, models trained with it see more data within the budget:

- **M2-BERT-base**: FlashFFTConv trains for 70,000 steps vs. 16,000 steps for PyTorch (both with 64K token batch size)
- **Hyena-s**: FlashFFTConv trains for 15B tokens vs. 5B tokens for PyTorch

This design directly tests whether efficiency translates to quality, rather than just cost savings.

### 5.2 Quality Improvements Under Fixed Compute Budget

The paper's first major claim is that FlashFFTConv improves model quality when trained under the same compute budget. Table 1 presents the core results:

| Model (Metric) | PyTorch | FlashFFTConv | Improvement |
|----------------|---------|--------------|-------------|
| M2-BERT-base-110M (GLUE Score ↑) | 77.6 | 80.9 | **+3.3 points** |
| Hyena-s-155M (PPL ↓) | 13.4 | 11.1 | **-2.3 points** |

> **Critical interpretation**: A 3.3-point GLUE improvement is substantial. The authors note in Appendix B (Table 18) that doubling the parameter count of M2-BERT-base (from 110M to 260M) yields a GLUE score of 81.0—nearly identical to the 80.9 achieved by FlashFFTConv with the original parameter count. This supports the claim that FlashFFTConv achieves quality "equivalent to doubling the parameters."

Similarly, for Hyena-s, the perplexity improvement from 13.4 to 11.1 represents a **17% reduction** in perplexity. The Appendix B reference shows that a larger Hyena-m-355M model (more than double the parameters) achieves 11.1 PPL—again matching the FlashFFTConv result with the smaller model.

**Why this matters**: These results validate the hypothesis that systems efficiency directly impacts model quality. The mechanism is straightforward: higher throughput means more training data within the same compute budget, and more data leads to better generalization. This is not merely an engineering optimization—it shifts the Pareto frontier of model quality vs. compute cost.

**Potential limitation**: The fixed compute budget comparison conflates two factors: (1) the efficiency of the convolution implementation, and (2) the benefit of seeing more data. A purer comparison would train both models for the same number of tokens and compare wall-clock time. However, the chosen methodology is more practically relevant: most researchers have fixed compute budgets, not fixed token budgets.

### 5.3 Efficiency: Convolution Speedup and Memory Savings

#### Raw Convolution Benchmarks (Tables 3, 4, 11-15)

Table 3 presents the forward pass time for standard convolutions across sequence lengths from 256 to 4M on H100-SXM:

| Sequence Length | PyTorch (ms) | FlashFFTConv (ms) | Speedup | Memory Savings |
|-----------------|--------------|-------------------|---------|----------------|
| 256 | 0.43 | 0.09 | **4.78×** | 8.21× |
| 1K | 1.57 | 0.24 | **6.54×** | 7.73× |
| 4K | 6.65 | 1.37 | 4.85× | 7.61× |
| 16K | 28.6 | 9.27 | 3.09× | 7.21× |
| 32K | 62.1 | 21.8 | 2.85× | 6.57× |
| 1M | 2,346.3 | 1,492.8 | 1.57× | 2.64× |
| 4M | 10,127.6 | 7,587.0 | 1.33× | 2.63× |

Several patterns emerge:

1. **Speedup is highest at short sequences** (up to 6.54× at 1K) because PyTorch is dominated by I/O overhead that FlashFFTConv eliminates through kernel fusion.

2. **Speedup decreases at longer sequences** because both implementations become I/O-bound between SRAM and HBM. However, FlashFFTConv still maintains 1.33× speedup at 4M.

3. **Memory savings are substantial** across all lengths, from 8.21× at 256 to 2.63× at 4M. This comes from kernel fusion (reducing intermediate storage) and recomputation in the backward pass.

Table 4 shows **gated convolutions**, which are common in modern architectures (Hyena, M2-BERT). The speedup is even higher:

| Sequence Length | PyTorch (ms) | FlashFFTConv (ms) | Speedup |
|-----------------|--------------|-------------------|---------|
| 256 | 0.62 | 0.11 | 5.64× |
| 1K | 2.30 | 0.29 | **7.93×** |
| 4K | 9.49 | 1.43 | 6.64× |
| 32K | 84.8 | 26.3 | 3.22× |
| 4M | 13,031.2 | 10,049.4 | 1.30× |

The **7.93× speedup at 1K** is the maximum reported in the paper. This exceeds the standard convolution speedup because FlashFFTConv fuses the gating operations ($v \odot ((u \odot w) * k)$) into the kernel, while PyTorch incurs separate I/O for each gating operation.

#### Ablation: Fusion-Only Without Tensor Cores

Table 3 includes an ablation ("Fusion-Only/cuFFTdx") that uses kernel fusion but does not use the Monarch decomposition for Tensor Cores:

| Sequence Length | Fusion-Only (ms) | FlashFFTConv (ms) |
|-----------------|------------------|-------------------|
| 256 | 0.21 | 0.09 |
| 1K | 0.67 | 0.24 |
| 32K | 45.5 | 21.8 |
| 1M+ | – (OOM) | 1,492.8 |

Two critical observations:

1. **Fusion-Only is 2-3× slower** than FlashFFTConv at short sequences, demonstrating that Tensor Core utilization provides significant additional speedup beyond kernel fusion alone.

2. **Fusion-Only fails beyond 32K** due to SRAM limitations, while FlashFFTConv scales to 4M. This validates the design choice to broadcast across the sequence dimension (Section 3.1), which reduces SRAM requirements.

#### End-to-End Model Throughput (Table 5)

Raw convolution speedup does not always translate to proportional end-to-end speedup, since models include other operations (MLPs, normalizations, etc.). Table 5 shows end-to-end throughput across five models:

| Model (Size, Sequence Length) | PyTorch | FlashFFTConv | Speedup |
|-------------------------------|---------|--------------|---------|
| M2-BERT-base (110M, 128) | 4,480 seqs/s | 8,580 seqs/s | **1.9×** |
| Hyena-s-4K (155M, 4K) | 84.1 seqs/s | 147 seqs/s | 1.7× |
| Long convs, Path-X (102M, 16K) | 126 images/s | 308 images/s | **2.4×** |
| SaShiMi (5.4M, 64K) | 38.7 clips/s | 50.3 clips/s | 1.3× |
| HyenaDNA (1M) | 0.69 seqs/s | 3.03 seqs/s | **4.4×** |

The variation in speedup reveals important insights:

- **HyenaDNA achieves 4.4×** because the PyTorch implementation is bottlenecked by small batch size (batch 1 on 80GB GPU vs. batch 4 for FlashFFTConv). This demonstrates how memory savings enable larger batches.

- **SaShiMi achieves only 1.3×** because the model interleaves convolutions with SSM-based filter generation, pooling layers, and MLPs. The convolution is a smaller fraction of total runtime.

- **Path-X achieves 2.4×** despite 16K sequence length, showing that FlashFFTConv's benefits persist at moderate lengths.

### 5.4 New Capabilities: Longer Sequence Models

The most striking claim is that FlashFFTConv enables sequence lengths previously infeasible for convolutional models.

#### Path-512: First Model Above Random Chance (Table 2)

| Task (Sequence Length) | PyTorch | FlashFFTConv |
|------------------------|---------|--------------|
| Path-X (16K) | 96.9% | 96.9% |
| Path-512 (256K) | ✗ (OOM) | **96.1%** |

The Path-512 result is particularly significant. The paper states in the Abstract that "no model had previously achieved better than 50%" on this task. FlashFFTConv achieves 96.1%, which is:

1. **Far above random chance** (50% for binary classification)
2. **Comparable to Path-X performance** (96.9%), despite 16× longer sequences
3. **Only possible with FlashFFTConv**, as PyTorch runs out of memory

This result directly supports the claim that FlashFFTConv "yields the first model that can solve the challenging Path-512 task."

#### DNA Modeling: Longest Human Genes at Single-Nucleotide Resolution

Table 8 shows partial convolutions extending HyenaDNA to 4M sequence length:

| Base Filter Length | 1M | 2M | 4M |
|--------------------|----|----|----|
| HyenaDNA-450K | 2.91 | 2.91 | 2.91 |
| HyenaDNA-1M | 2.91 | 2.91 | 2.90 |

The perplexity remains stable (2.90-2.91) even at 4M sequence length. The paper notes in the Abstract that this enables "the first DNA model that can process the longest human genes (2.3M base pairs)" at single-nucleotide resolution. Figure 5 in Appendix B shows a t-SNE visualization of gene embeddings, with the dystrophin gene (2.3M base pairs) annotated.

**Why this matters**: Prior DNA models either tokenized sequences (losing nucleotide resolution) or downsampled (losing information). FlashFFTConv enables modeling at the natural resolution of DNA, which the paper argues yields downstream improvements in quality.

### 5.5 Comparison to FlashAttention-v2 (Table 6)

The paper compares a 2.7B-parameter Hyena model using FlashFFTConv against a 2.7B-parameter GPT model using FlashAttention-v2:

| Model | 2K (tokens/s) | 8K (tokens/s) | 16K (tokens/s) |
|-------|---------------|---------------|----------------|
| GPT-2.7B, FA-v2 | 33.8 | 27.8 | 21.6 |
| Hyena-2.7B, FlashFFTConv | **35.2** | **35.2** | **32.3** |
| FA-v2 FLOP Utilization | 65.7% | 72.1% | 78.5% |
| FlashFFTConv FLOP Utilization | 62.3% | 61.9% | 56.5% |
| FlashFFTConv Speedup | 1.1× | 1.3× | **1.5×** |

Three observations:

1. **Hyena is faster at all sequence lengths**, despite lower FLOP utilization. This is because convolutions have **fewer total FLOPs** than attention ($O(N \log N)$ vs. $O(N^2)$).

2. **The speedup gap widens at longer sequences** (1.1× at 2K to 1.5× at 16K). This is because attention's quadratic complexity becomes increasingly expensive.

3. **FLOP utilization is lower** for FlashFFTConv (56.5-62.3%) compared to FlashAttention-v2 (65.7-78.5%). The authors acknowledge this in Section 4.2: "FlashFFTConv achieves lower end-to-end FLOP utilization than FlashAttention-v2 but achieves higher throughput, since convolutions incur fewer overall FLOPs."

**Critical assessment**: This comparison is somewhat asymmetric. FlashAttention-v2 is highly optimized for attention, while FlashFFTConv is optimized for convolution. A fairer comparison would be convolution vs. attention with both using their best implementations. However, the result is practically relevant: it shows that convolutional models can compete with Transformers in wall-clock time, removing a key adoption barrier.

### 5.6 Architectural Extensions: Partial and Frequency-Sparse Convolutions

#### Partial Convolutions (Tables 7, 8)

Table 7 evaluates partial convolutions on Hyena-s-8K with varying kernel lengths:

| Kernel Length | 8K | 4K | 2K | 1K | 512 | 256 |
|---------------|----|----|----|----|----|----|
| PPL (↓) | 13.8 | 13.8 | 13.8 | 13.9 | 14.0 | 14.2 |
| Memory (↓) | 32.5G | 15.3G | 11.8G | 8.4G | 6.1G | 5.8G |

The perplexity remains nearly stable (13.8 to 14.2) even when reducing the kernel from 8K to 256, while memory drops from 32.5G to 5.8G—a **5.6× reduction**. This supports the claim that "a large proportion of the convolution filters can be pruned without impacting downstream quality."

Table 8 (discussed in Section 5.4) shows that partial convolutions enable extending pretrained models to longer sequences via sliding window, maintaining perplexity at 4M sequence length.

#### Frequency-Sparse Convolutions (Table 9)

Table 9 applies frequency-sparsity to a pretrained HyenaDNA-1M model:

| Sparsity Fraction | 0% | 50% | 75% | 79% | 84% | 91% |
|-------------------|----|----|----|----|----|----|
| PPL (↓) | 2.91 | 2.91 | **2.90** | 2.91 | 2.93 | 2.98 |
| Convolution Speedup (↑) | 1.0× | 1.2× | 1.3× | **1.4×** | 1.5× | 1.8× |

Two notable findings:

1. **Quality improves slightly at 75% sparsity** (2.91 to 2.90). The authors hypothesize this is due to "removing high-frequency noise." This is counterintuitive—sparsity typically degrades quality—but suggests that frequency-domain sparsity may have a regularizing effect.

2. **Speedup scales with sparsity**, reaching 1.8× at 91% sparsity with only modest quality degradation (2.98 vs. 2.91). This demonstrates that frequency-sparse convolutions can trade quality for speed in a controlled manner.

**Mechanism**: As described in Appendix A.4, sparsity in different dimensions of the reshaped kernel allows skipping specific matrix multiplication blocks in the Monarch decomposition. For example, sparsity in the first dimension skips computation in matrix $B$ (the second FFT operation), while sparsity in the fourth dimension reduces outer loop iterations.

### 5.7 Critical Assessment of Experimental Claims

#### Claims Well-Supported by Evidence

| Claim | Supporting Evidence | Assessment |
|-------|---------------------|------------|
| 7.93× speedup over PyTorch | Table 4 (gated convolution at 1K) | **Strong**: Specific numbers provided, ablation confirms Tensor Core benefit |
| 4.4× end-to-end speedup | Table 5 (HyenaDNA) | **Strong**: Consistent across multiple models, mechanism explained (batch size increase) |
| Path-512 solved at 96.1% | Table 2 | **Strong**: PyTorch baseline fails (OOM), first result above 50% |
| Quality gain equivalent to 2× parameters | Table 1 + Appendix B Table 18 | **Strong**: Direct comparison to larger models provided |

#### Claims with Conditions or Limitations

| Claim | Condition/Limitation | Assessment |
|-------|---------------------|------------|
| Speedup across 4 orders of magnitude | Speedup decreases from 6.54× to 1.33× | **Qualified**: Speedup is real but diminishes at longest sequences |
| Faster than FlashAttention-v2 | Only at 2K+ sequence length | **Qualified**: At 2K, speedup is 1.1×; advantage grows with length |
| Frequency-sparsity maintains quality | Up to ~79% sparsity | **Qualified**: Quality degrades beyond 79% (2.98 PPL at 91%) |
| Memory savings up to 8.21× | Only at short sequences (256) | **Qualified**: Savings drop to 2.63× at 4M |

#### Missing or Unclear Results

1. **Training stability**: The paper does not report on training stability (e.g., gradient norms, loss curves) for FlashFFTConv vs. PyTorch. Given that recomputation is used in the backward pass, numerical precision could be a concern.

2. **Multi-GPU scaling**: All benchmarks appear to be single-GPU. The paper does not evaluate how FlashFFTConv scales across multiple GPUs, which is critical for large model training.

3. **Non-power-of-2 sequence lengths**: The cost model assumes $N$ is a power of 2. The paper does not discuss performance for arbitrary sequence lengths, which are common in practice (e.g., variable-length documents).

4. **Precision effects**: All benchmarks use half precision (fp16/bf16). The paper does not evaluate fp32 performance, which may be needed for some applications.

### 5.8 Summary of Experimental Findings

The experiments provide strong evidence for FlashFFTConv's core claims:

1. **Efficiency gains are real and substantial**: Up to 7.93× convolution speedup and 4.4× end-to-end model speedup, with consistent improvements across four orders of magnitude in sequence length.

2. **Efficiency translates to quality**: Under fixed compute budgets, FlashFFTConv-trained models match the quality of models with twice the parameters, validating the hypothesis that throughput improvements enable better generalization.

3. **New capabilities are unlocked**: Path-512 (256K sequences) and DNA modeling at 2.3M base pairs were previously infeasible; FlashFFTConv makes them tractable.

4. **Architectural extensions are effective**: Partial and frequency-sparse convolutions provide additional memory and speed benefits with minimal quality impact.

The main limitations are that speedup diminishes at the longest sequence lengths (though remains positive), and some comparisons (e.g., vs. FlashAttention-v2) are sequence-length-dependent. However, these are expected trade-offs given the underlying algorithmic complexity differences, not failures of the approach.

Overall, the experimental evaluation is thorough and convincing. The authors test across multiple domains, provide ablation studies to isolate contributions, and report both positive results and limitations (e.g., decreasing speedup at longer sequences). The evidence strongly supports adopting FlashFFTConv for long-sequence convolutional models.

## 6. Limitations and Trade-offs

While FlashFFTConv demonstrates substantial improvements over prior FFT convolution implementations, the approach relies on specific assumptions and exhibits trade-offs that constrain its applicability. This section critically examines the limitations, edge cases, and open questions that remain after the paper's evaluation.

### 6.1 Diminishing Speedup at Extreme Sequence Lengths

The most prominent trade-off is that **speedup decreases as sequence length increases**. Table 3 shows convolution speedup dropping from **6.54× at 1K sequence length to 1.33× at 4M**. Similarly, memory savings decline from **8.21× at 256 to 2.63× at 4M** (Table 16).

**Why this happens**: At short sequences, PyTorch is dominated by I/O overhead that FlashFFTConv eliminates through kernel fusion. However, at very long sequences (1M+), both implementations become bottlenecked by I/O between SRAM and HBM. The paper acknowledges this in Section 4.2: "Speedup is more modest for longer sequences, which incur additional I/O costs (between registers and SRAM for the $p=3$ and between SRAM and HBM for $p=4$)."

**Practical implication**: FlashFFTConv provides the greatest benefit for moderate sequence lengths (256 to 32K). For extreme lengths (1M+), the absolute time savings remain substantial (e.g., 2,540 seconds saved at 4M), but the relative advantage narrows. Researchers expecting uniform 5-8× speedup across all lengths may be disappointed.

### 6.2 Lower FLOP Utilization Than FlashAttention-v2

Table 6 reveals that FlashFFTConv achieves **56.5-62.3% end-to-end FLOP utilization**, compared to **65.7-78.5% for FlashAttention-v2**. The authors acknowledge this gap explicitly: "FlashFFTConv achieves lower end-to-end FLOP utilization than FlashAttention-v2 but achieves higher throughput, since convolutions incur fewer overall FLOPs."

**Why this matters**: Lower FLOP utilization indicates that FlashFFTConv leaves some compute capacity unused. While the $O(N \log N)$ complexity of convolutions means fewer total FLOPs than attention's $O(N^2)$, the utilization gap suggests there is room for further optimization. This could become a bottleneck if future GPU architectures increase compute capacity faster than memory bandwidth.

**Open question**: The paper does not analyze whether the FLOP utilization gap stems from the Monarch decomposition itself, the kernel fusion strategy, or other implementation details. Understanding this could guide future improvements.

### 6.3 Hardware Generation Constraints

Appendix A.5 explicitly states: "**Older generations of GPU such as V100 are not supported, since the sizes of the tensor cores are different.**" FlashFFTConv was developed on A100 GPUs and tested on A100 and H100 GPUs.

**Why this matters**: This limits adoption for researchers and organizations with older hardware. The V100, while older, remains in widespread use in academic and industrial settings. The paper notes they "look forward to integrating more general libraries such as Cutlass [88] to support a wider range of GPUs," but this support does not exist at the time of publication.

**Additional constraint**: The implementation relies on specific Tensor Core matrix sizes ($16 \times 16 \times 16$ on A100/H100). Future GPU architectures with different Tensor Core configurations may require re-tuning the decomposition radix and tiling parameters.

### 6.4 Power-of-2 Sequence Length Assumption

The cost model in Section 3.2 explicitly assumes "**$N$ is a power of 2**." The Monarch decomposition factors $N = \prod_{i=1}^p N_i$, and the algorithms in Appendix A assume $N = N_1^p$ for simplicity.

**Why this matters**: Real-world sequences are rarely perfect powers of 2. Documents, DNA sequences, and audio clips have arbitrary lengths. The paper does not discuss:
- How padding to the next power of 2 affects efficiency
- Whether the cost model holds for non-power-of-2 lengths
- Performance degradation from padding overhead

**Practical implication**: Users may need to pad sequences to the nearest power of 2, incurring additional compute and memory costs that are not reflected in the benchmark tables. This could reduce the effective speedup in production settings with variable-length data.

### 6.5 Half-Precision Only Evaluation

All benchmarks in Tables 3, 4, 11-15 use **half precision (fp16/bf16)**. The paper does not evaluate fp32 performance.

**Why this matters**: Some applications require full precision for numerical stability:
- Scientific computing applications may need fp32 for accuracy
- Training very deep models can accumulate numerical errors in half precision
- Some optimization algorithms (e.g., certain second-order methods) require fp32

**Open question**: The real-valued FFT optimization (Appendix A.1) exploits symmetry properties that may behave differently in fp32. The paper does not report whether the 2× cost reduction from real-valued FFT holds in full precision, or whether Tensor Core support for fp32 matrix multiplication is sufficient to maintain speedup.

### 6.6 Single-GPU Evaluation Only

All experiments appear to be **single-GPU benchmarks**. The paper does not evaluate:
- Multi-GPU data parallelism scaling
- Model parallelism for very large models
- Communication overhead in distributed training

**Why this matters**: Modern large-scale training typically uses 8-512 GPUs. The efficiency characteristics of FlashFFTConv may change in distributed settings:
- Kernel fusion benefits may be reduced if communication dominates
- The memory savings (2.63-8.21×) could enable larger per-GPU batches, but the paper does not test this
- Gradient synchronization overhead is not measured

**Practical implication**: Researchers planning to use FlashFFTConv for large-scale distributed training lack guidance on expected multi-GPU efficiency. The 4.4× end-to-end speedup for HyenaDNA (Table 5) is measured on a single H100-SXM.

### 6.7 Recomputation Trades Compute for Memory

The backward pass uses **recomputation** to reduce memory footprint (Section 3.1): "Instead of storing intermediate results on HBM for the backward pass (e.g., the intermediate result of $\mathbf{F}u$), we simply recompute them in the backward pass."

**Trade-off**: This reduces memory usage but **increases compute cost** during backpropagation. The paper does not quantify:
- The exact compute overhead from recomputation
- Whether this affects training stability (e.g., gradient accumulation errors)
- How recomputation interacts with gradient checkpointing at the model level

**Open question**: For models that already use gradient checkpointing, does FlashFFTConv's recomputation compound the compute overhead, or can the two be coordinated? The paper does not address this interaction.

### 6.8 Frequency-Sparsity Quality Degradation at High Sparsity

Table 9 shows that frequency-sparse convolutions maintain quality up to ~79% sparsity, but **degrade beyond that**:

| Sparsity Fraction | 0% | 75% | 79% | 84% | 91% |
|-------------------|----|----|----|----|----|
| PPL (↓) | 2.91 | 2.90 | 2.91 | 2.93 | **2.98** |
| Speedup (↑) | 1.0× | 1.3× | 1.4× | 1.5× | **1.8×** |

**Why this matters**: While 75-79% sparsity provides speedup without quality loss, users seeking maximum speedup (1.8× at 91% sparsity) must accept a **2.4% perplexity degradation**. This trade-off may be acceptable for inference but could impact training convergence.

**Additional concern**: The paper evaluates sparsity on a **pretrained model** (HyenaDNA-1M). It does not test whether training from scratch with sparse kernels yields the same quality, or whether sparsity patterns need to be tuned per task.

### 6.9 End-to-End Speedup Varies by Model Architecture

Table 5 shows end-to-end speedup ranging from **1.3× (SaShiMi) to 4.4× (HyenaDNA)**:

| Model | Speedup | Reason for Variation |
|-------|---------|---------------------|
| SaShiMi | 1.3× | Convolutions interleaved with SSM, pooling, MLPs |
| M2-BERT-base | 1.9× | Short sequence length (128), convolution is smaller fraction |
| HyenaDNA | 4.4× | PyTorch bottlenecked by batch size 1 vs. FlashFFTConv batch 4 |

**Why this matters**: The benefit of FlashFFTConv depends heavily on **what fraction of the model's runtime is spent in convolution**. Models with significant non-convolution components (MLPs, normalizations, other layers) will see proportionally smaller end-to-end speedup.

**Practical implication**: Researchers should profile their specific architecture before expecting the 4.4× speedup reported for HyenaDNA. The 1.3× for SaShiMi may be more representative for models with mixed layer types.

### 6.10 No Training Stability Analysis

The paper reports final quality metrics (perplexity, GLUE score, accuracy) but **does not analyze training dynamics**:
- No loss curves comparing FlashFFTConv vs. PyTorch
- No gradient norm statistics
- No discussion of numerical precision issues from recomputation
- No ablation on learning rate sensitivity

**Why this matters**: Systems optimizations can subtly affect training stability. For example:
- Recomputation may introduce small numerical differences that accumulate over training
- Half-precision FFT operations may have different rounding behavior than standard FFT
- The Monarch decomposition's matrix operations may have different conditioning properties

**Open question**: Does FlashFFTConv require different hyperparameters (learning rate, warmup, weight decay) than standard PyTorch FFT convolution? The paper uses the same hyperparameters (Appendix C.2) but does not test whether tuning could yield additional improvements.

### 6.11 Order-$p$ Selection Requires Cost Model Tuning

The cost model (Section 3.2, Equation 2) requires **empirically-measured hardware constants** ($\tau_G$, $\tau_M$, $\sigma_H$, $\sigma_S$) that vary by GPU generation. Table 19 provides values for A100-40GB, but these must be re-measured for each new GPU.

**Why this matters**: Users deploying FlashFFTConv on new hardware must:
1. Profile their GPU to measure achievable FLOP rates and bandwidth
2. Potentially re-tune the order-$p$ selection thresholds
3. Verify that the cost model predictions match actual performance

**Practical implication**: This adds implementation complexity compared to a fixed-algorithm FFT library. While the paper provides a "simple heuristic for selecting $p$" (Section 3), the heuristic depends on hardware-specific constants that are not universal.

### 6.12 Causal Convolution Padding Overhead

The paper mentions that inputs are "often padded with zeros to compute a causal convolution" (Section 2.1) and that they "special-case this padding" (Section 3.1). However, the **overhead from padding is not quantified separately**.

**Why this matters**: Causal convolutions require the FFT size to be at least $2N$ (to avoid circular convolution artifacts). This effectively doubles the sequence length for the FFT computation, increasing both compute and memory costs.

**Evidence**: Tables 13 and 14 in Appendix B show results for "input is half the length of the convolution size" (causal setting). Comparing Table 3 (standard) to Table 13 (causal) at 4K sequence length:
- Standard: 1.37 ms
- Causal: 1.37 ms (same in this case, but FFT size is doubled)

The paper claims to "eliminate half of the outermost matrix multiply operations" for padded inputs, but does not show an ablation isolating this optimization's contribution.

### 6.13 Summary of Limitations

| Limitation | Severity | Mitigation |
|------------|----------|------------|
| Diminishing speedup at 4M+ sequences | Moderate | Absolute time savings still substantial |
| Lower FLOP utilization than FlashAttention-v2 | Low | Convolution has fewer total FLOPs |
| A100/H100 only (no V100 support) | Moderate | Await Cutlass integration |
| Power-of-2 sequence length assumption | Moderate | Padding overhead not quantified |
| Half-precision only | Moderate | fp32 support needed for some applications |
| Single-GPU evaluation | High | Multi-GPU scaling untested |
| Recomputation compute overhead | Low | Trade-off is explicit and favorable |
| Frequency-sparsity degrades at 91% | Low | 75-79% sparsity maintains quality |
| End-to-end speedup varies by model | Moderate | Profile specific architecture |
| No training stability analysis | Moderate | Final quality metrics are strong |
| Order-$p$ selection requires tuning | Low | Cost model provided, constants measurable |
| Causal padding overhead not isolated | Low | Optimization claimed but not ablated |

### 6.14 Open Questions for Future Work

Based on the limitations above, several open questions remain:

1. **Multi-GPU scaling**: How does FlashFFTConv perform in data-parallel or model-parallel distributed training? Does the memory savings enable larger effective batch sizes across GPUs?

2. **Full-precision support**: Can FlashFFTConv maintain speedup in fp32, and are there numerical stability benefits for certain applications?

3. **Non-power-of-2 sequences**: What is the performance impact of padding arbitrary-length sequences to powers of 2? Can the Monarch decomposition be adapted for arbitrary factorizations?

4. **Training dynamics**: Does FlashFFTConv require different hyperparameters than standard FFT convolution? Are there training stability differences from recomputation?

5. **Broader hardware support**: Can FlashFFTConv be ported to TPUs, non-NVIDIA GPUs, or specialized AI accelerators using libraries like Cutlass?

6. **Sparsity pattern optimization**: The frequency-sparsity patterns in Table 10 are hand-designed. Can optimal sparsity patterns be learned automatically per task?

These limitations do not undermine FlashFFTConv's core contributions but highlight areas where future work could extend its applicability and performance.

## 7. Implications and Future Directions

FlashFFTConv does more than optimize a single linear algebra primitive; it fundamentally alters the trade-off space between convolutional and attention-based architectures for long-sequence modeling. By closing the wall-clock time gap with Transformers, it validates convolutions as a viable, scalable alternative for tasks requiring context lengths beyond the typical 2K–8K range. This section explores how this work shifts the research landscape, identifies concrete avenues for follow-up research, outlines high-impact application domains, and provides guidance for practitioners considering integration.

### 7.1 Shifting the Architectural Landscape

The primary implication of FlashFFTConv is the restoration of **architectural diversity** in long-context modeling. Prior to this work, Transformers dominated long-sequence tasks not necessarily because attention was theoretically superior, but because optimized implementations (like FlashAttention-v2) made them faster in practice. FlashFFTConv removes this systems-level barrier.

*   **Convolution vs. Attention Competitiveness**: Table 6 demonstrates that a 2.7B-parameter Hyena model using FlashFFTConv achieves **1.5× higher throughput** than a matched GPT model using FlashAttention-v2 at 16K sequence length. This implies that for sequence lengths $N \geq 2\text{K}$, convolutions are no longer the "slow alternative." Researchers can now choose architectures based on inductive biases (e.g., convolutions for locality, attention for global routing) rather than hardware constraints.
*   **Efficiency as a Quality Lever**: Section 4.1 establishes that efficiency gains directly translate to model quality under fixed compute budgets. The observation that FlashFFTConv allows M2-BERT-base to match models with **twice the parameter count** (Table 1) suggests that future model scaling laws may need to account for implementation efficiency as a primary variable. A model architecture is only as good as its fastest implementation.
*   **Unlocking "Impossible" Sequence Lengths**: The ability to process sequences up to **4 million tokens** (Abstract) shifts the boundary of what is considered "long context." Tasks previously requiring downsampling or tokenization (like genomic modeling) can now be approached at native resolution. This enables new scientific questions—such as modeling entire human genes at single-nucleotide resolution—that were computationally prohibited before.

### 7.2 Follow-Up Research Directions

The paper explicitly identifies several areas where future work can extend FlashFFTConv's capabilities. These directions range from hardware portability to algorithmic refinements.

#### Hardware Portability and Generalization
Currently, FlashFFTConv relies on specific Tensor Core configurations available on **A100 and H100 GPUs** (Appendix A.5). Older generations like V100 are unsupported due to differing Tensor Core sizes.
*   **Cutlass Integration**: The authors note plans to integrate with **Cutlass** [88], a CUDA template library for linear algebra. This would generalize the implementation to support a wider range of GPU architectures and potentially other accelerators (e.g., TPUs) that support matrix-matrix multiplications.
*   **Non-Power-of-2 Sequences**: The cost model assumes sequence length $N$ is a power of 2 (Section 3.2). Real-world data often has arbitrary lengths. Future work could adapt the Monarch decomposition to handle arbitrary factorizations $N = \prod N_i$ without excessive padding overhead, or develop efficient padding strategies that minimize the compute waste identified in Section 6.12.

#### Algorithmic Extensions
The matrix view of the FFT opens new avenues for algorithmic innovation beyond simple speedup.
*   **Learned Sparsity Patterns**: Appendix A.4 describes **frequency-sparse convolutions** using hand-designed sparsity patterns (e.g., zeroing out high-frequency blocks). A natural follow-up is to make these sparsity patterns **learnable** during training. Instead of fixed masks, the model could learn which frequency bands are relevant for a specific task, potentially improving the 1.8× speedup observed at 91% sparsity (Table 9) while maintaining quality.
*   **Distributed Training**: All experiments in this paper are **single-GPU** (Section 6.6). Large-scale model training typically requires data or model parallelism across 8–512 GPUs. Future work must evaluate how FlashFFTConv's kernel fusion and memory savings interact with gradient synchronization overhead. The **2.63×–8.21× memory savings** (Table 16) could enable larger per-GPU batch sizes in distributed settings, but communication patterns may need adjustment to match the fused kernel structure.
*   **Full-Precision Support**: Benchmarks are limited to **half precision (fp16/bf16)** (Section 6.5). Scientific applications (e.g., climate modeling, physics simulations) often require **fp32** for numerical stability. Extending FlashFFTConv to fp32 while maintaining Tensor Core utilization is an open engineering challenge, as fp32 Tensor Core throughput is typically lower than fp16.

#### Training Dynamics and Stability
While final quality metrics are strong, the paper does not analyze **training dynamics** (Section 6.10).
*   **Recomputation Effects**: The backward pass uses recomputation to save memory (Section 3.1). Future work should investigate whether this introduces gradient noise or stability issues over long training runs compared to storing intermediates.
*   **Hyperparameter Sensitivity**: It remains unclear if FlashFFTConv requires different learning rates or warmup schedules than standard FFT convolutions. Systematic hyperparameter sweeps could reveal if the optimization allows for more aggressive training configurations.

### 7.3 Practical Applications and Use Cases

FlashFFTConv is particularly well-suited for domains where sequence length is the primary bottleneck. The following use cases represent high-impact opportunities for adoption.

#### Genomics and DNA Modeling
This is the most immediate application. Prior DNA models either tokenized sequences (losing resolution) or downsampled (losing information).
*   **Use Case**: Modeling entire human genes at single-nucleotide resolution.
*   **Evidence**: Table 8 shows stable perplexity (2.90) at **4M sequence length**, enabling the embedding of the **2.3M base pair dystrophin gene** (Abstract, Figure 5).
*   **Impact**: Researchers can now train models on full gene contexts without truncation, potentially improving variant effect prediction and regulatory element detection.

#### High-Resolution Vision
Treating images as flattened sequences allows convolutional models to capture long-range spatial dependencies.
*   **Use Case**: Classifying high-resolution images where local features are insufficient.
*   **Evidence**: Table 2 shows **96.1% accuracy on Path-512** (256K sequence length), a task where prior models achieved ≤50%.
*   **Impact**: This approach can be extended to medical imaging (e.g., whole-slide histology) or satellite imagery, where context spans millions of pixels.

#### Raw Audio Processing
Audio waveforms sampled at high frequencies (e.g., 64 KHz) generate massive sequences quickly.
*   **Use Case**: Speech synthesis and audio generation directly from raw waveforms.
*   **Evidence**: Table 5 shows a **1.3× end-to-end speedup** for SaShiMi, an audio generation model processing 1-second clips at 64K samples.
*   **Impact**: Faster training enables iteration on higher-fidelity audio models and longer context audio understanding (e.g., minute-long recordings).

#### Long-Context Language Modeling
While Transformers dominate language modeling, convolutions offer stability and linear-logarithmic scaling.
*   **Use Case**: Document-level understanding or retrieval-augmented generation with large context windows.
*   **Evidence**: Table 6 shows Hyena outperforming GPT with FlashAttention-v2 at **16K sequence length** (32.3K vs. 21.6K tokens/s).
*   **Impact**: For tasks where attention's quadratic cost is prohibitive, FlashFFTConv provides a linear-logarithmic alternative that remains fast at extreme lengths.

### 7.4 Reproduction and Integration Guidance

Practitioners considering FlashFFTConv should evaluate their hardware, sequence lengths, and precision requirements against the system's constraints.

#### When to Use FlashFFTConv
*   **Hardware**: You must have **NVIDIA A100 or H100 GPUs**. The implementation relies on specific Tensor Core matrix sizes ($16 \times 16 \times 16$) unavailable on older cards like V100 (Appendix A.5).
*   **Sequence Length**: The benefits are most pronounced for **$N \geq 256$**. Table 3 shows speedups peak at **6.54× for 1K sequences** and remain positive (1.33×) up to **4M**. For very short sequences ($N &lt; 256$), standard PyTorch may suffice.
*   **Precision**: Your model must tolerate **half precision (fp16/bf16)**. If your task requires fp32 for stability, FlashFFTConv is not yet suitable (Section 6.5).
*   **Sequence Shape**: Ideally, your sequence length should be a **power of 2**. While padding is possible, the cost model and decomposition assume $N = 2^k$ (Section 3.2).

#### When to Avoid
*   **Older Hardware**: Do not attempt to run on V100 or consumer-grade GPUs without Tensor Core support matching A100/H100 specifications.
*   **Mixed Layers**: If your model interleaves convolutions with many non-fusible operations (like complex pooling or SSM layers), end-to-end speedup may be limited. Table 5 shows SaShiMi achieves only **1.3× speedup** because convolutions are not the sole bottleneck.
*   **Distributed Training**: If you require multi-GPU training, be aware that scaling characteristics are **untested** (Section 6.6). You may need to implement custom data parallelism wrappers to preserve the memory benefits.

#### Integration Steps
1.  **Verify Hardware**: Confirm access to A100/H100 GPUs.
2.  **Check Sequence Lengths**: Align your data pipeline to power-of-2 sequence lengths where possible to maximize Tensor Core utilization.
3.  **Replace Convolution Kernels**: Swap standard FFT convolution calls with FlashFFTConv kernels. For models using gating (e.g., Hyena, M2-BERT), ensure the gating operations are fused as described in Section 3.1 to achieve the **7.93× speedup** (Table 4).
4.  **Monitor Memory**: Expect **2.63×–8.21× memory reduction** (Table 16). You can use this headroom to increase batch size or sequence length rather than simply reducing cost.
5.  **Validate Quality**: Run a small-scale training run to verify that half-precision FFT operations do not introduce numerical instability in your specific task, especially if using recomputation in the backward pass.

In summary, FlashFFTConv provides a robust pathway for deploying convolutional sequence models at scales previously reserved for Transformers. By adhering to the hardware and precision constraints, practitioners can unlock significant efficiency gains and access new problem domains involving million-token contexts.
