## 1. Executive Summary

This paper introduces `Nemotron-Nano-9B-v2`, a hybrid Mamba-Transformer reasoning model derived from a 12-billion-parameter base pre-trained on 20 trillion tokens using an FP8 recipe. Compared to similarly-sized models like `Qwen3-8B`, it achieves on-par or better accuracy on reasoning benchmarks while delivering up to 6.3× higher inference throughput for generation-heavy workloads such as 8k input and 16k output tokens (Figure 1). By employing the Minitron compression strategy to prune and distill the model, the authors enable 128k token context inference on a single NVIDIA A10G GPU with 22GiB of memory, addressing the critical cost and latency barriers of long-context reasoning.

## 2. Context and Motivation

To understand why `Nemotron-Nano-9B-v2` matters, we need to first examine the tension at the heart of modern language model deployment: the trade-off between **reasoning capability** and **inference efficiency**. This paper addresses a specific gap that has become increasingly critical as reasoning-focused models have emerged.

### The Core Problem: Long Thinking Traces Are Expensive

Recent advances in language models have shown that **reasoning performance scales with the length of the model's "thinking trace"** — the intermediate tokens the model generates before producing a final answer. For complex math, code, or scientific reasoning tasks, models may need to generate 8,000 to 16,000+ output tokens just to work through a problem step-by-step. This is fundamentally different from typical chat or completion workloads where outputs are often under 1,000 tokens.

The problem is that **inference cost scales linearly with output length**. If a model generates 16k tokens instead of 1k, it consumes 16× more compute time and memory bandwidth. For organizations deploying reasoning models at scale, this creates a prohibitive cost barrier. The paper explicitly targets this pain point, noting that `Nemotron-Nano-9B-v2` achieves "up to 6.3× higher throughput for such workloads" compared to `Qwen3-8B` (Figure 1).

### The Memory Wall: 128k Context on Consumer Hardware

A second, compounding challenge is **context length**. Modern reasoning tasks often require models to process long documents, codebases, or multi-turn conversations — easily exceeding 100,000 tokens. However, running a 12-billion-parameter model at 128k context length creates a memory problem:

> "Storing just the weights of a 12B parameter model in bfloat16 precision requires 22.9 GiB, which is more than the 22 GiB memory capacity of an A10G GPU" (Section 4).

This is not a minor inconvenience — it means the model literally cannot fit on the hardware. The KV cache (which stores attention states for each token in the context) adds additional memory pressure that scales linearly with sequence length. The paper's goal of enabling "inference over context lengths of 128k tokens on a single NVIDIA A10G GPU (22 GiB of memory, bfloat16 precision)" (Abstract) is therefore a hard constraint that requires architectural innovation, not just engineering optimization.

### Prior Approaches and Their Limitations

To appreciate what this paper does differently, we need to understand the landscape of approaches that came before:

**Pure Transformer Architectures.** Standard Transformer models use self-attention at every layer. While highly accurate, this creates a quadratic memory bottleneck for long sequences. The KV cache grows linearly with context length, and at 128k tokens, this cache can consume more memory than the model weights themselves. This makes pure Transformer designs impractical for long-context reasoning on memory-constrained hardware.

**Pure State Space Models (Mamba).** The Mamba architecture (Dao & Gu, 2024) offers linear-time sequence modeling with constant memory per token, theoretically solving the long-context problem. However, pure Mamba models have shown **accuracy gaps** on certain reasoning benchmarks compared to Transformers. The community has observed that attention mechanisms still provide unique benefits for specific tasks, particularly those requiring precise token-to-token relationships.

**Nemotron-H (Prior NVIDIA Work).** The immediate predecessor to this work, `Nemotron-H` (NVIDIA, 2025), introduced a **hybrid Mamba-Transformer architecture** where roughly 8% of layers use self-attention while the rest use Mamba-2 layers. This preserved accuracy while improving throughput. However, `Nemotron-H` was not specifically optimized for **reasoning workloads** with long thinking traces, nor was it compressed to run on single-GPU deployments with 128k context.

**Minitron Compression.** The Minitron framework (Muralidharan et al., 2024; Sreenivas et al., 2024) demonstrated effective pruning and distillation for **base pre-trained models**. However, as the paper notes: "While Minitron was originally designed for compressing pretrained base models targeting user-defined parameter budgets, in this work, we extend it to compress reasoning models while also incorporating the memory constraints and throughput-based objectives" (Section 4). Compressing a **reasoning-aligned model** is fundamentally different from compressing a base model — the alignment procedures (SFT, RLHF, DPO) create behavioral patterns that can be disrupted by naive pruning.

**Competing Reasoning Models (e.g., Qwen3-8B).** Models like `Qwen3-8B` (Yang et al., 2025) achieve strong reasoning accuracy but use standard Transformer architectures. As Figure 1 shows, they achieve comparable accuracy to `Nemotron-Nano-9B-v2` on benchmarks like AIME24, GPQA-D, and LiveCodeBench, but at **1× relative throughput** compared to Nano 2's **3.3×–6.3×**. This represents a real deployment cost difference: serving the same workload on `Qwen3-8B` would require 3–6× more GPUs.

### How This Paper Positions Itself

`Nemotron-Nano-9B-v2` positions itself at the intersection of three competing objectives that prior work has typically optimized separately:

| Objective | Prior Work Trade-off | This Paper's Approach |
|-----------|---------------------|----------------------|
| **Accuracy** | High accuracy required large models or pure Transformers | Hybrid Mamba-Transformer preserves accuracy at 9B parameters |
| **Throughput** | Efficiency required sacrificing model quality | 6.3× throughput gain **without** accuracy loss |
| **Memory** | Long context required multi-GPU setups | 128k context on **single** A10G (22 GiB) |

The key insight is that **none of these objectives need to be sacrificed**. By combining:
1. A hybrid Mamba-Transformer architecture (from Nemotron-H),
2. Specialized reasoning-focused pre-training and alignment data,
3. Extended Minitron compression adapted for reasoning models,

the authors demonstrate that efficient reasoning models are achievable without compromising capability. This is not an incremental improvement — it represents a shift in what is considered possible for sub-10B parameter models.

### Why This Matters Beyond Benchmarks

The real-world significance extends beyond benchmark scores:

- **Cost Reduction.** A 6× throughput improvement directly translates to 6× lower inference costs for reasoning workloads. For organizations running thousands of reasoning queries daily, this is the difference between a viable product and an unsustainable expense.

- **Accessibility.** Enabling 128k context inference on a single A10G GPU (a commonly available cloud instance) democratizes access to long-context reasoning. Organizations no longer need multi-GPU clusters to deploy these capabilities.

- **Thinking Budget Control.** The paper introduces a novel "budget control" feature where users can specify how many thinking tokens the model may generate (Section 3.4). This allows dynamic trade-offs between response quality and latency at inference time — a capability not present in prior reasoning models.

- **Open Science.** The authors release not just model checkpoints, but "the majority of our pre- and post-training datasets" (Abstract). This includes the `Nemotron-CC-Math-v1` (133B tokens), `Nemotron-Pretraining-Code-v1`, and `Nemotron-Post-Training-Dataset-v2`. This transparency enables the community to reproduce, audit, and build upon the work — addressing a common criticism of proprietary model development.

In summary, this paper addresses a concrete deployment gap: **how to deliver state-of-the-art reasoning capability at a cost and latency that makes real-world deployment feasible**. It does so not by accepting accuracy-efficiency trade-offs, but by re-architecting the model, training process, and compression strategy in tandem. The following sections will unpack exactly how each of these components works.

## 3. Technical Approach

This is a **model development and compression paper** that demonstrates how to build an efficient reasoning language model by combining hybrid architecture design, specialized training recipes, and constraint-aware pruning. The core idea is that **accuracy, throughput, and memory efficiency can be optimized simultaneously** rather than traded off against each other, provided the architecture, training data, and compression strategy are co-designed from the start.

### 3.1 Reader orientation (approachable technical breakdown)

The system being built is a 9-billion-parameter language model that can reason through complex problems while fitting entirely on a single consumer-grade GPU, even when processing documents up to 128,000 tokens long. The solution takes the shape of a three-stage pipeline: first, pre-train a larger 12-billion-parameter hybrid model on carefully curated data; second, align it for reasoning and instruction-following through multiple fine-tuning stages; third, compress it via pruning and distillation while preserving the reasoning behaviors learned during alignment.

### 3.2 Big-picture architecture (diagram in words)

Think of the system as four major components connected in sequence. **Component 1 (Base Model Architecture)** defines the neural network structure itself — a hybrid of Mamba-2 layers and self-attention layers arranged in a specific pattern (Figure 2). **Component 2 (Pre-training Pipeline)** feeds 20 trillion tokens through this architecture using FP8 precision training with a three-phase data curriculum (Figure 3). **Component 3 (Alignment Pipeline)** takes the pre-trained base model and applies supervised fine-tuning, reinforcement learning, and preference optimization to teach reasoning behaviors and instruction-following (Figure 4). **Component 4 (Compression Pipeline)** then prunes layers and neurons from the aligned 12B model and distills knowledge back into it, producing the final 9B model that fits within the 19.66 GiB memory budget for 128k context inference.

### 3.3 Roadmap for the deep dive

- We start with the **model architecture** because every subsequent decision (training, alignment, compression) depends on understanding what layers exist and how they compute.
- We then explain **pre-training** because the base model's capabilities determine the ceiling for what alignment and compression can preserve.
- We cover **alignment** next because reasoning behaviors must be learned before compression — pruning an unaligned model would lose these behaviors permanently.
- We detail **compression** afterward because it modifies the already-aligned model and must be done carefully to avoid degrading reasoning quality.
- We conclude with **budget control** because this is an inference-time feature that depends on all prior stages being correctly implemented.

### 3.4 Detailed, sentence-based technical breakdown

#### 3.4.1 Hybrid Mamba-Transformer Architecture

The foundation of `Nemotron-Nano-12B-v2-Base` is a **hybrid architecture** that replaces most self-attention layers with Mamba-2 layers while keeping a small fraction of attention for tasks that require precise token-to-token relationships. The model contains exactly 62 layers total, of which only 6 are self-attention layers, 28 are FFN (feed-forward network) layers, and 28 are Mamba-2 layers (Section 2.1, Figure 2). This means approximately 8% of layers use self-attention, which matches the ratio found effective in the prior `Nemotron-H` work. The attention layers are **evenly dispersed throughout the model** rather than clustered at the beginning or end, which ensures that long-range dependency modeling is available at all depths of the network.

The model uses a hidden dimension of $d_{\text{model}} = 5120$ and an FFN hidden dimension of $d_{\text{ffn}} = 20480$ (Table 1). For attention, the model employs **Grouped-Query Attention (GQA)** with 40 query heads and 8 key-value heads, which reduces KV cache memory compared to multi-head attention while maintaining accuracy. The Mamba-2 layers use 8 groups, a state dimension of 128, a head dimension of 64, an expansion factor of 2, and a convolution window size of 4. Several architectural choices reduce memory overhead: the model uses **no position embeddings** (relying on Mamba's inherent sequence modeling), **RMSNorm** instead of LayerNorm, **separate embedding and output layer weights**, **no dropout**, and **no bias weights** for linear layers. Each of these decisions removes parameters or computation that would otherwise consume memory without proportional accuracy gains for reasoning tasks.

#### 3.4.2 Pre-training Data Curation and Mixture

The pre-training dataset consists of **20 trillion tokens** drawn from thirteen data categories, organized into a three-phase curriculum that shifts from diversity to quality over the course of training (Section 2.3, Figure 3). **Phase 1** uses a diverse mixture including 18.3% crawl-medium, 14.8% crawl-medium-high, 11.1% crawl-high, 16.2% syn-crawl-high, 20.0% code, 3.2% math, 4.4% academic, 5.0% multilingual, and 3.1% stem-sft data. **Phase 2** begins at the 60% training point and increases high-quality data: syn-crawl-high rises to 21.0%, math increases to 9.5%, and stem-sft jumps to 14.5%. **Phase 3** begins at the 90% training point and further concentrates on high-quality sources: stem-sft reaches 32.0%, math is 11.0%, and code-sft is 10.9%. This curriculum design reflects the insight that **early training benefits from diverse exposure** while **later training benefits from high-quality signals** that refine the model's representations.

The math data deserves special attention because prior extraction pipelines frequently **discard or distort equations**, severely limiting utility for reasoning pre-training (Section 2.2.1). The authors built a new pipeline that aggregates math-related URLs from prior datasets, re-fetches raw HTML from 98 Common Crawl snapshots (2014–2024), renders each page using the `lynx` text-based browser to preserve layout and math structure, and then applies the Phi-4 (14B-parameter) model to remove boilerplate and standardize notation into LaTeX. A FineMath classifier retains high-quality documents, followed by MinHash-based Locality Sensitive Hashing for fuzzy deduplication via the NeMo-Curator framework. This process produced a 133-billion-token corpus (`Nemotron-CC-Math-3+`) and a higher-quality 52-billion-token subset (`Nemotron-CC-Math-4+`). When used for pretraining, this dataset yields **substantial improvements across math, code, and general-domain evaluations**, surpassing all existing open math datasets according to the authors.

For multilingual data, the authors conducted an ablation study comparing four sources: Common Crawl extractions, FineWeb-2, DiverseQA-wiki (generated from Wikipedia), and DiverseQA-crawl (translated from English) (Section 2.3.1, Table 2). The results showed that **synthesized multilingual QA pairs performed much better than curated web crawl data**, with DiverseQA-crawl achieving the highest average score over 8 languages (47.0% average vs. 37.0% for Common Crawl). Consequently, the final data mixture assigns **much higher weight to DiverseQA-crawl** than other multilingual categories. This finding is important because it suggests that **quality-controlled synthetic data can outperform raw web scrapes** for multilingual capability, which has implications for how future models should approach non-English training.

The authors also introduced **Fundamental Reasoning (FR) focused SFT-style data** during pre-training, which targets analytical reasoning, logical reasoning, and reading comprehension tasks (Section 2.2.2, Section 2.3.2). They collected existing datasets including LSAT, LogiQA, and AQuA-RAT, then prompted DeepSeek-V3 and Qwen3-30B-A3B to synthesize similar questions with chain-of-thought solutions. An ablation study showed this data **improved MMLU-Pro from 44.24 to 56.36** and increased average MATH score by around 2 points, with no decrease in commonsense reasoning or code benchmarks (Table 3). This demonstrates that **reasoning-specific data during pre-training** provides measurable benefits that persist even after alignment and compression.

#### 3.4.3 FP8 Training Recipe and Hyperparameters

The entire pre-training run uses **FP8 precision** following DeepSeek's training recipe, which significantly reduces memory usage and increases throughput compared to BF16 training (Section 2.4). Specifically, the model uses **E4M3 format for all tensors**, with 128×128 quantization blocks for weights and 1×128 tiles for activations. Unlike the original `Nemotron-H`, the authors **natively kept model weights in E4M3** so that distributed optimizer parameter all-gather operations could be performed in FP8, though master weights are still kept in FP32. One exception is that the **first and last four linear layers remain in BF16**, as done with `Nemotron-H`, because these layers are particularly sensitive to quantization error. All optimizer state remains in FP32, and the authors observed **no training instabilities** from this choice of numerics.

The training hyperparameters are precisely specified (Section 2.5). The model trains on a **token horizon of 20 trillion tokens** with a sequence length of 8192 and a global batch size of 768, which equals 6,029,312 tokens per batch. No batch size ramp-up is used. The learning rate schedule follows a **Warmup-Stable-Decay (WSD)** pattern with a stable learning rate of $4.5 \times 10^{-4}$ and a minimum value of $4.5 \times 10^{-6}$. The learning rate decays over the final 3.6 trillion tokens. Weight decay is set to 0.1, and Adam $\beta_1$ and $\beta_2$ are set to 0.9 and 0.95 respectively. These hyperparameters are important because they define the **optimization trajectory** that determines how the model learns from the 20 trillion tokens — too aggressive a learning rate would cause instability, while too conservative would waste compute.

#### 3.4.4 Long-Context Extension Phase

After the three pre-training phases, the model undergoes a **long-context extension phase (Phase LC)** to enable 128k token inference without degrading other benchmarks (Section 2.6). This phase uses continuous pre-training with a context length of **524,288 tokens (512k)** at a constant learning rate of $4.5 \times 10^{-6}$. Although the target context length is 128k, preliminary studies on `Nemotron-H 8B` found that **training at 512k sequence length works better** than training at 256k or 128k (Table 4). The authors' intuition is that longer training sequences effectively lower the chance of long coherent documents being cut and separated by the Concat & Chunk algorithm for pretraining data loading. At 512k sequence length with synthetic data, the model achieves 81.04 on RULER-128k compared to 73.68 at 128k training length.

Phase LC consists of 18.9 billion tokens using a data blend based on Phase 3, but with all Phase 3 data weights proportionally downscaled to 80% of their original values, allocating the remaining 20% to newly added **long-context document-QA data** (Section 2.6). This long-context synthetic data is generated from academic documents longer than 32k tokens: each document is split into 1,024-token chunks, 10% of chunks are randomly selected and fed into Qwen-2.5-72B-Instruct to generate QA pairs, and these pairs are concatenated and appended to the end of the original document. This provides material for the model to learn **long-context dependencies** — the ability to retrieve and reason over information that appears far from the query position. The training uses 8-way tensor model parallelism and 16-way context parallelism to ensure 512k sequence lengths fit in GPU memory, with a global batch size of 12 to maintain approximately 6M tokens per global batch (matching pre-training).

#### 3.4.5 Alignment Pipeline: From Base to Reasoning Model

The alignment process converts the 12B base checkpoint into an aligned model through multiple stages, as illustrated in Figure 4 (Section 3). The process begins with **Stage 1 SFT**, which trains on approximately 80 billion tokens of prompt-response pairs across domains: 1.5M math samples, 1.1M coding, 2.0M science, 400K tool-calling, 1.5M conversational, 2K safety, and 5.0M multilingual (Table 7, Section 3.1). Importantly, Stage 1 augments the dataset with roughly **10% of prompts paired with outputs stripped of reasoning traces**, exposing the model to "empty" traces so it can produce direct answers in a reasoning-off mode. To preserve long-context ability from pretraining, samples are **concatenated into sequences of approximately 128k tokens**, reducing padding overhead and encouraging long-range learning.

**Stage 2 SFT** specifically targets tool-calling because Stage 1's sample concatenation at 128k disrupted learning of tool-calling patterns (Section 3.2). Stage 2 trains **without concatenation** using the full tool-calling dataset and a representative subsample of other domains. The tool-calling dataset includes single-turn, multi-turn, and multi-step conversations sampled from xlam-function-calling-60k, glaive-function-calling-v2, and NVIDIA-When2Call, with responses generated using Qwen3-235B-A22B. Inspired by ToolACE and APIGen-MT, the authors extend this to multi-turn settings by simulating conversations where Qwen3-235B-A22B plays the roles of User-Agent, Assistant-Agent, and API-Server-Agent. A **lightweight rule-based tool-call verification layer** ensures outputs are consistent and verifiable, retaining only successful trajectories.

**Stage 3 SFT** reinforces long-context capability by incorporating long-context data following the `Nemotron-H` preparation recipe, along with augmented examples where **reasoning traces were abruptly truncated to 1–2k tokens** while preserving the final answer (Section 3.2). This truncation strategy is critical: it improves robustness under varying inference-time thinking budgets by teaching the model to produce coherent answers even when reasoning is cut short. About 5% of the post-training data contained deliberately truncated reasoning traces, enabling **fine-grained thinking budget control at inference time** (Abstract, Section 3.4).

After SFT, the model undergoes multiple optimization stages. **IFeval RL** samples 16,000 prompts from LMSYS Chat and augments them with IFEval-style instructions; a rule-based verifier scores outputs based on instruction satisfaction, creating a reward signal that prioritizes following directions with precision. **DPO (Direct Preference Optimization)** is applied in a separate branch to improve tool-calling, evaluated using the BFCL v3 benchmark which emphasizes multi-step and multi-turn tool calls (Section 3.2). The model undergoes reinforcement learning in the WorkBench environment through iterative DPO stages, generating on-policy data with positive examples (successful tool calls) and negative examples (failed generations) for every WorkBench prompt. **GRPO (Group Relative Policy Optimization)** trains candidate checkpoints from the SFT stage using English-only contexts from HelpSteer3, generating responses both with and without thinking traces and using a Qwen-based reward model to judge rollouts.

A key observation during training was a **trade-off between reasoning capabilities and chat capabilities** (Section 3.2). To address this, the authors opted for **checkpoint interpolation**, blending an RL checkpoint with strong reasoning capabilities with an RL checkpoint with strong chat capabilities. Checkpoint interpolation is performed by linearly interpolating model weights:

$$
w_{\text{merged}} = (1 - \alpha) \cdot w_{\text{model1}} + \alpha \cdot w_{\text{model2}}
$$

where $w_{\text{model1}}$ is the reasoning-focused checkpoint, $w_{\text{model2}}$ is the chat-focused checkpoint, and $\alpha$ controls the blend ratio. The authors experimented with a parameter sweep over $\alpha$ values from 0.1 to 0.9 in increments of 0.1, finding that **values around 0.5 offered a good trade-off**. This model merging technique allows the final model to excel at both reasoning and conversation without requiring additional training compute.

#### 3.4.6 Budget Control Mechanism

`Nemotron Nano V2` implements a novel **thinking budget control** feature that allows users to specify how many thinking tokens the model may generate before producing the final answer (Section 3.4). The final answer is the portion of text typically shown to end users, while thinking tokens are the intermediate reasoning steps. This feature is implemented by **counting tokens after the model begins generating the `` token**. Once the budget is reached, the inference setup attempts to insert a closing `</think>` tag. Rather than inserting it immediately, the system **lets the model finish its current sentence** and places the tag at the next newline. In extreme cases where no newline appears, the system enforces closure within 500 tokens past the budget: if no newline occurs by the (budget + 500)th token, the `</think>` tag is forcibly inserted.

The truncation training in Stage 3 SFT is essential for this feature to work correctly. Without truncated training examples, the model exhibits a **compensation effect** where it uses more tokens in the final answer to make up for restrictions in the thinking traces (Figure 5a, center). With truncated training, this effect is absent (Figure 5b, center). Another failure mode is that the model can remain in "thinking mode" even after the closing `</think>` tag is inserted, evident when the model generates the closing tag again after forced insertion. The authors evaluate this using **"Well-Formedness"**, where a well-formed response should contain only a single closing tag. Figure 5a (right) shows that for short budgets without truncation training, the percentage of well-formed responses drops sharply. With truncation training, the model **consistently produces well-formed responses** (Figure 5b, right). This demonstrates that **training-time preparation is required for inference-time features** — you cannot add budget control to an already-trained model without degrading output quality.

#### 3.4.7 Compression via Pruning and Distillation

The compression pipeline reduces the aligned 12B model to 9B parameters to enable 128k context inference on a single NVIDIA A10G GPU with 22 GiB memory (Section 4). Storing just the weights of a 12B parameter model in bfloat16 precision requires 22.9 GiB, which exceeds the A10G's capacity — this **hard memory constraint** drives the need for compression. The strategy builds on Minitron, a lightweight model pruning framework, but extends it to compress **reasoning models** (which have alignment behaviors) while incorporating memory constraints and throughput-based objectives.

**Importance estimation** determines which components to prune by computing sensitivity scores for each model component using only forward passes (Section 4.1). For **layer importance**, the model computes scores iteratively: for each candidate layer, it temporarily removes the layer and computes the mean squared error (MSE) between the original model's logits and those produced by the pruned model. At each pruning step, the layer with the lowest MSE is removed, as it has the least influence on the final output. For **FFN and embedding channel importance**, the model examines the outputs of each neuron in the first linear operator of each FFN layer using a calibration dataset of 1024 samples. Formally, each neuron's importance score is computed by aggregating its outputs given an input batch $X$:

$$
F^{(i)}_{\text{neuron}} = \sum_{B,S} \delta\left(X \left(W^i_1\right)^T\right)
$$

where $W^i_1$ refers to the $i$th row of the weight matrix $W_1$, and $\sum_{B,S}$ refers to aggregation along the batch and sequence dimensions using mean and L2-norm functions. For **Mamba importance**, the model adopts a nested activation-based scoring strategy over the calibration dataset, computing activation scores from the $W_x$ projection and ranking heads within each Mamba group to preserve group-aware computation semantics. The authors experimented with pruning Mamba heads but found this axis caused **severe accuracy degradation**, so they focus on head pruning only when it yields accuracy-throughput trade-off benefits.

The **Neural Architecture Search (NAS)** process explores multiple axes within the 19.66 GiB memory budget through combinatorial pruning (Section 4.2). The search space includes depth reduction (removing 6–10 layers from the original 62-layer architecture) combined with width pruning of embedding channels (4480–5120), FFN dimension (13440–20480), and Mamba heads (112–128). To find the optimal architecture efficiently, the authors break the problem into two parts: first find the optimal depth, then find the optimal width-pruned architecture given that depth. For depth, they compare 52, 54, and 56 layer candidates while keeping attention layers fixed at 4 (to achieve 7–8% attention-to-total-layers ratio). After 6B tokens of distillation, reducing depth beyond 56 layers results in **significant accuracy degradation** (Table 9: 52 layers = 44.92%, 54 layers = 47.35%, 56 layers = 51.48%), so depth is fixed at 56 for further width pruning.

For width pruning, the authors enumerate all candidate pruned architectures meeting the memory budget and sort them by estimated memory consumption at 128k context length and batch size 1. The top 3 candidates undergo short Knowledge Distillation for 19B tokens, and throughput is benchmarked to pick the final architecture (Table 10). **Candidate 2** achieves the best accuracy (63.02%) while maintaining reasonable throughput (156.42 tokens/s), with 56 layers, 4480 hidden dimension, 15680 FFN dimension, and 128 Mamba heads, totaling 8.89B parameters. The authors ablated Mamba head pruning but found limited benefit at the compression ratios explored (less than 15% after depth pruning), so **pruning only FFN and embedding dimensions proves sufficient**.

**Retraining with distillation** recovers accuracy lost due to pruning (Section 4.3). The reasoning model is distilled in stages: depth pruning to 56 layers with KD using ~60B tokens at 8,192 sequence length; width pruning and KD with ~50B tokens at 8,192, ~25B tokens at 49,152, and ~1B tokens at 262,144 sequence length; followed by DPO, GRPO, KD with ~0.4B tokens at 262,144 to recover post-RL drops, RLHF for alignment, and model merging via 0.5 linear interpolation. The dataset mix matters: **70% post-training stage 2 data and 30% pretraining data yields the highest accuracy** after ~6B tokens of KD (Table 11: 50/50 = 57.5%, 70/30 = 58.5%, 90/10 = 57.2%). Figure 6 shows how task accuracy evolves through the distillation pipeline: DPO and GRPO enhance function-calling (BFCL v3) and instruction-following (IFEval), though IFEval temporarily degrades multi-task understanding (MMLU-Pro), which is recovered in the post-GRPO KD step. RLHF enhances alignment (Arena-Hard) but causes additional benchmark drops, recovered through model merging.

The final compressed model, `Nemotron-Nano-9B-v2`, retains 56 layers with embedding channels pruned from 5120 to 4480 and FFN intermediate size pruned from 20480 to 15680 (Section 4.4). As shown in Figure 1 and Tables 5–6, this model achieves **3×–6× higher throughput than Qwen3-8B** for generation-heavy scenarios while surpassing it in accuracy and remaining comparable to the 12B teacher on most benchmarks. This demonstrates that **careful compression preserves capability** — the 9B model is not a degraded version of the 12B, but an optimized variant that trades unused capacity for efficiency.

## 4. Key Insights and Innovations

This paper delivers several contributions that extend beyond incremental engineering improvements. Below, I distinguish between **fundamental innovations** (new capabilities or paradigms) and **significant extensions** (substantial improvements to existing methods). Understanding this distinction helps readers appreciate what can be directly adopted in their own work versus what represents NVIDIA-specific optimization.

---

### 4.1 Thinking Budget Control via Truncation Training (Fundamental Innovation)

**What it is.** The paper introduces the first reasoning model that allows users to **specify a token budget for thinking at inference time**, dynamically trading response quality against latency. This is implemented through the `` / `</think>` tag mechanism described in Section 3.4, but the innovation lies in *how* the model is trained to respect these budgets.

**What makes it different.** Prior reasoning models (including DeepSeek-R1 and o1-style models) generate thinking traces of **unpredictable length** — the model decides when to stop reasoning based on internal convergence. Users cannot constrain this without degrading output quality. The key insight in this paper is that **budget control must be baked into training**, not added at inference:

> "About 5% of the data contained deliberately truncated reasoning traces, enabling fine-grained thinking budget control at inference time" (Abstract).

Without this truncation training, the model exhibits a **compensation effect**: when thinking is cut short, it "makes up for it" by generating longer final answers, defeating the purpose of budget control (Figure 5a, center). With truncation training, this effect disappears (Figure 5b, center). The model also learns to produce **well-formed responses** with exactly one closing `</think>` tag, whereas untrained models often generate the tag multiple times after forced insertion (Figure 5a vs. 5b, right).

**Why it matters.** This enables **production-grade deployment** of reasoning models where latency guarantees are required. A customer-facing application can specify "spend at most 2k thinking tokens" to ensure sub-second responses, while an internal analysis tool might allow 16k tokens for maximum accuracy. Prior to this work, such dynamic trade-offs were not possible without fine-tuning separate models for each budget tier.

**Adoption consideration.** This innovation is **directly transferable** to other reasoning models. Any team training a chain-of-thought model can incorporate truncated reasoning traces into their SFT data to enable budget control. The paper's finding that ~5% truncated data is sufficient provides a concrete starting point.

### 4.2 Reasoning-Aware Model Compression (Fundamental Innovation)

**What it is.** The paper extends the Minitron pruning framework — originally designed for **base pre-trained models** — to compress **reasoning-aligned models** while preserving alignment behaviors learned through SFT, DPO, GRPO, and RLHF.

**What makes it different.** Compressing a base model and compressing an aligned model are **fundamentally different problems**:

| Aspect | Base Model Compression | Reasoning Model Compression |
|--------|----------------------|----------------------------|
| **Objective** | Preserve next-token prediction accuracy | Preserve reasoning behaviors AND accuracy |
| **Risk** | Accuracy degradation | Behavioral regression (e.g., lost tool-calling ability) |
| **Recovery** | Standard fine-tuning | Multi-stage distillation with alignment data |

The paper demonstrates that naive pruning of an aligned model **disrupts learned behaviors**. For example, Stage 1 SFT's sample concatenation at 128k improved long-context ability but **degraded tool-calling accuracy**, requiring a dedicated Stage 2 SFT without concatenation (Section 3.2). Similarly, RLHF improves chat capability (Arena-Hard) but causes benchmark drops that must be recovered through post-RL knowledge distillation (Figure 6).

The compression pipeline interleaves pruning with **behavioral recovery stages**: DPO for tool-calling, GRPO for instruction-following, and model merging to balance reasoning vs. chat capabilities. This is not just "prune then fine-tune" — it is a **co-designed compression-alignment pipeline**.

**Why it matters.** This unlocks **efficient deployment of aligned models** without re-training from scratch. The 12B→9B compression achieves 3×–6× throughput gains (Figure 1) while remaining "comparable to the 12B teacher on most benchmarks" (Section 4.4). For organizations that have invested in expensive alignment procedures, this means they can compress models post-alignment rather than discarding alignment work.

**Adoption consideration.** The importance estimation methods (MSE-based layer pruning, activation-based neuron scoring) are **directly reusable**. However, the multi-stage distillation recipe (Section 4.3) is tightly coupled to NVIDIA's alignment pipeline. Teams adopting this should expect to design their own recovery stages matching their alignment procedures.

---

### 4.3 Data Quality Over Quantity for Reasoning (Significant Extension)

**What it is.** The paper demonstrates that **carefully curated synthetic data** outperforms larger volumes of raw web data for reasoning capabilities, with specific innovations in math data extraction and Fundamental Reasoning SFT data.

**What makes it different.** The math data pipeline addresses a previously overlooked problem: **prior extraction tools discard or distort mathematical expressions**. As Section 2.2.1 notes:

> "These tools frequently discard or distort equations and flatten code formatting, severely limiting the utility of the extracted content for pretraining."

The solution — rendering HTML via `lynx`, standardizing to LaTeX with Phi-4, and filtering with FineMath classifiers — produced a 133B-token corpus that "surpasses all existing open math datasets" on benchmarks. This is not just more data; it is **higher-fidelity data** that preserves the structure reasoning models need.

The Fundamental Reasoning SFT data ablation (Table 3) shows that adding just 5% of this data (replacing Common Crawl) improved MMLU-Pro from **44.24 to 56.36** — a 12-point gain from a small data substitution. This challenges the prevailing "more tokens = better" assumption for reasoning tasks.

**Why it matters.** This provides a **data-centric pathway** to reasoning improvements that does not require architectural changes or massive compute increases. For teams with limited training budgets, investing in data quality (particularly for math and reasoning domains) may yield better returns than scaling model size.

**Adoption consideration.** The math extraction pipeline is **fully described** and can be replicated. The Fundamental Reasoning SFT approach (synthesizing LSAT/LogiQA-style questions with chain-of-thought solutions) is also directly applicable. However, the specific finding that 70% post-training / 30% pretraining data mix works best for distillation (Table 11) may be dataset-dependent.

---

### 4.4 Constraint-Aware Neural Architecture Search (Significant Extension)

**What it is.** The paper performs architecture search under a **hard memory constraint** (19.66 GiB for 128k context on A10G) rather than optimizing for accuracy alone. This produces a model that is **guaranteed to fit** the target hardware.

**What makes it different.** Standard NAS optimizes for accuracy or latency in isolation. This work optimizes for **feasibility first**: the search space is restricted to architectures that satisfy the memory budget, then accuracy and throughput are optimized within that feasible set. The memory model accounts for both **parameter memory** (constant) and **KV cache memory** (scales with sequence length), which is critical for long-context workloads.

The finding that **56 layers is the minimum viable depth** (Table 9: 52 layers drops to 44.92% accuracy vs. 51.48% at 56 layers) provides a concrete design rule. Similarly, the decision to keep **4 attention layers** (maintaining ~7% attention-to-total ratio) balances KV cache size against long-context performance.

**Why it matters.** This demonstrates that **deployment constraints can drive architecture design** without sacrificing capability. The resulting 9B model achieves better accuracy than Qwen3-8B while running on the same hardware class — proving that constraint-aware design is not a compromise but an optimization opportunity.

**Adoption consideration.** The memory estimation approach (parameter + KV cache budgeting) is **directly reusable** for any long-context deployment. The specific layer counts and dimension choices are NVIDIA-specific, but the methodology of "find feasible set, then optimize within it" generalizes.

### 4.5 What Is Incremental vs. Fundamental

| Contribution | Type | Reasoning |
|--------------|------|-----------|
| Thinking Budget Control | **Fundamental** | New inference-time capability not present in prior reasoning models |
| Reasoning-Aware Compression | **Fundamental** | First demonstration of compressing aligned reasoning models with behavioral recovery |
| Data Quality for Reasoning | **Significant Extension** | Builds on prior data curation work but with novel math-specific pipeline |
| Constraint-Aware NAS | **Significant Extension** | Extends Minitron with memory constraints, but uses established pruning methods |
| Hybrid Mamba-Transformer | **Incremental** | Directly inherits from Nemotron-H; contribution is reasoning-specific optimization |

**Key takeaway for practitioners.** The **budget control mechanism** and **reasoning-aware compression pipeline** are the most novel contributions with direct applicability to other reasoning model projects. The data curation insights provide actionable guidance for improving reasoning without architectural changes. The hybrid architecture itself is less novel but demonstrates that existing efficient architectures can be adapted for reasoning workloads with careful tuning.

## 5. Experimental Analysis

This section examines how the authors validate their claims about `Nemotron-Nano-9B-v2`. Understanding the evaluation methodology is critical because **reasoning model benchmarks have become increasingly contested** — different papers use different evaluation setups that can dramatically affect reported scores. We will walk through what was measured, how it was measured, and whether the evidence supports the paper's central claims about accuracy, throughput, and memory efficiency.

### 5.1 Evaluation Methodology

#### 5.1.1 Benchmark Suite and Task Categories

The authors evaluate across **seven distinct capability categories**, each targeting a different aspect of model behavior. This comprehensive suite is important because a model could excel at math while failing at tool use, or achieve high throughput while sacrificing instruction-following quality.

**Mathematical Reasoning.** The paper uses four math benchmarks with increasing difficulty:
- `GSM8K CoT` (grade school math with chain-of-thought)
- `MATH` (competition-level problems)
- `MATH Level 5` (the hardest subset of MATH)
- `AIME-2024` and `AIME-2025` (American Invitational Mathematics Examination, representing elite competition math)

For AIME benchmarks, the authors report **Pass@1 averaged over 16 runs**, acknowledging the high variance in competition math where a single model can produce different answers across generations. For MATH-500, they average over 4 runs. This multi-run approach is more rigorous than single-pass evaluation, which can be noisy for difficult reasoning tasks.

**Code Generation.** Three benchmarks assess coding capability:
- `HumanEval+ avg@32` (164 Python programming problems, evaluated with 32 generations per prompt)
- `MBPP+ avg@32` (Mostly Basic Python Programming, also 32 generations)
- `LiveCodeBench (07/24–12/24)` (contamination-free coding benchmark with recent problems)

The use of `avg@32` rather than `pass@1` is notable — it measures the model's **sampling capability** rather than greedy decoding, which is more relevant for reasoning models that benefit from multiple attempts.

**General Knowledge and Reasoning.** The paper evaluates:
- `MMLU` (57 subjects across STEM, humanities, and social sciences)
- `MMLU-Pro 5-shot` (a more challenging version with 10 answer choices instead of 4)
- `AGIEval English CoT` (college admission and graduate school entrance exams)
- `ARC Challenge`, `HellaSwag`, `OpenBookQA`, `PIQA`, `WinoGrande` (commonsense reasoning)

**Long Context.** The `RULER-128K` benchmark measures the model's ability to retrieve and reason over information at 128,000 token context lengths. This is reported as an **average score over all 13 tasks** included in RULER, not a single metric. This is important because some models can retrieve information but fail at reasoning over it — the multi-task average captures both capabilities.

**Instruction Following and Tool Use.** Two specialized benchmarks:
- `IFEval (Inst. Strict)` (Instruction Following Evaluation with strict formatting requirements)
- `BFCL v3` (Berkeley Function Calling Leaderboard, version 3, emphasizing multi-step and multi-turn tool calls)

**Chat Capability.** `ArenaHard` measures overall helpfulness and conversational ability, which is distinct from reasoning capability. This separation is important because a model can be excellent at math but poor at conversation.

**Multilingual Capability.** The paper uses `Global-MMLU-Lite` across 8 languages (German, Spanish, French, Italian, Japanese, Korean, Portuguese, Chinese) and `MGSM` (Multilingual GSM8K) for math reasoning in non-English languages.

#### 5.1.2 Baseline Models

The primary baseline is `Qwen3-8B`, which the paper compares against throughout. This is an appropriate choice because:
1. It is a **similarly-sized model** (8B vs. 9B parameters)
2. It is a **strong reasoning model** with published benchmark scores
3. It uses a **standard Transformer architecture**, making the throughput comparison meaningful

Secondary baselines include `Qwen3-14B` (to show the 9B model can approach 14B performance) and `Gemma3-12B-Base` (for base model comparisons). The paper is careful to compare **base models to base models** and **aligned models to aligned models**, avoiding the common mistake of mixing checkpoint types.

#### 5.1.3 Experimental Setup and Hardware

All throughput measurements are conducted on a **single NVIDIA A10G GPU with 22 GiB of memory** in `bfloat16` precision (Figure 1 caption). This is a critical detail because throughput is highly hardware-dependent. The authors specify:
- Input sequence length (ISL): 8,000 tokens
- Output sequence length (OSL): 16,000 tokens
- Batch size: maximum that fits in memory (varies by model)
- Throughput metric: output tokens per second per GPU

For accuracy evaluations, the authors use `NeMo-Skills` and `lm-evaluation-harness` with specific modifications noted in Section 2.7. For example, for `ARC-Challenge`, they present all options at the same time (similar to MMLU) rather than one-at-a-time, which can affect scores.

**Transparency note:** The authors state "We run evaluations of all models ourselves unless otherwise stated" (Section 2.7), which reduces the risk of comparing against inconsistently-evaluated baselines. However, they do not release the exact evaluation code, which limits reproducibility.

### 5.2 Main Quantitative Results

#### 5.2.1 Accuracy vs. Throughput Trade-off (Figure 1)

Figure 1 is the paper's **central claim visualization**, showing both accuracy and throughput for `Nemotron-Nano-9B-v2` vs. `Qwen3-8B`. The data reveals:

| Benchmark | Nemotron-Nano-9B-v2 | Qwen3-8B | Difference |
|-----------|---------------------|----------|------------|
| IFBench (Instruction Following) | 78.9% | 74.1% | +4.8 pts |
| AIME24 (Math) | 81.9% | 75.8% | +6.1 pts |
| AIME25 (Math) | 72.0% | 69.3% | +2.7 pts |
| GPQA-D (Science) | 64.0% | 59.6% | +4.4 pts |
| LiveCodeBench (Coding) | 71.1% | 59.5% | +11.6 pts |
| BFCLv3 (Tool Use) | 66.9% | 66.3% | +0.6 pts |
| RULER 128k (Long Context) | 78.9% | 74.1% | +4.8 pts |

**Throughput comparison:**
- At 1k input / 8k output: **3.3×** higher throughput
- At 8k input / 16k output: **6.3×** higher throughput

This is the paper's most significant result: **higher accuracy AND higher throughput** simultaneously. Prior work typically showed trade-offs (e.g., smaller models are faster but less accurate). The hybrid Mamba-Transformer architecture enables this by reducing KV cache memory (fewer attention layers) while preserving accuracy (attention layers at critical positions).

**Critical assessment:** The throughput numbers are measured on a single A10G GPU, which is appropriate for the paper's deployment target. However, the comparison assumes `Qwen3-8B` is also run in `bfloat16` without quantization. If `Qwen3-8B` were quantized to INT8 or FP8, its throughput would increase, narrowing the gap. The paper does not report quantized baselines, which is a limitation for real-world deployment comparisons.

Several observations stand out:

1. **The 9B pruned base model outperforms Qwen3-8B-Base on most benchmarks** despite having only 1B more parameters. This suggests the hybrid architecture and training data are more important than raw parameter count.

2. **MATH and MATH Level 5 show the largest gaps** — Nemotron-Nano-9B-Base scores 80.50% on MATH vs. 55.40% for Qwen3-8B-Base, a 25-point difference. This indicates the math-specific data curation (Section 2.2.1) had substantial impact.

3. **AIME 2024 pass@32 drops from 56.67% (12B) to 30.00% (9B)** — a significant degradation from pruning. However, the 9B model still outperforms Qwen3-8B (20.00%). This shows compression does cost some capability, but the remaining capability is still state-of-the-art for the size class.

4. **RULER-128K at 82.22% for the 9B base model** confirms the long-context extension phase (Section 2.6) was effective. The 12B base achieves 84.74%, so pruning costs about 2.5 points on long-context tasks.

Table 6 shows multilingual benchmarks. The 9B base model averages **69.94% on Global-MMLU-Lite** vs. 72.81% for Qwen3-8B-Base — slightly behind on multilingual general knowledge. However, on **Multilingual Math (MGSM)**, the 9B base averages **84.67%** vs. 80.93% for Qwen3-8B-Base, showing stronger math reasoning even in non-English languages.

Key insights:

1. **The 12B model outperforms Qwen3-14B on AIME-2024 (85.42 vs. 81.53) and AIME-2025 (76.25 vs. 66.60)** despite having 2B fewer parameters. This is strong evidence that the architecture and training data matter more than scale alone for reasoning tasks.

2. **ArenaHard is the only benchmark where the 12B model trails both Qwen3 variants** (74.00 vs. 78.40 for 8B and 87.70 for 14B). This aligns with the paper's observation that there is a **trade-off between reasoning and chat capabilities** (Section 3.2), which they address through model merging.

3. **RULER @ 128k at 83.36%** is substantially higher than Qwen3-8B (74.13) and Qwen3-14B (73.55), confirming the long-context extension phase was effective.

4. **Humanity's Last Exam scores are low across all models** (6.30% for Nemotron, 4.40% for Qwen3-8B), indicating this benchmark is extremely difficult for current models. The 2-point advantage is meaningful but should not be overinterpreted.

#### 5.2.4 Compression Results (Tables 9, 10, 11)

The compression experiments show how accuracy degrades with pruning and how distillation recovers it.

**Depth Ablation (Table 9):**

| Layers | Accuracy (Avg) |
|--------|----------------|
| 52 | 44.92% |
| 54 | 47.35% |
| 56 | 51.48% |

This shows a **non-linear relationship** between depth and accuracy. Removing 6 layers (62→56) costs some accuracy, but removing 10 layers (62→52) causes catastrophic degradation. The authors correctly identify 56 layers as the minimum viable depth.

**Architecture Candidates (Table 10):**

| Candidate | Layers | Hidden | FFN | Mamba Heads | Params (B) | Accuracy | Throughput |
|-----------|--------|--------|-----|-------------|------------|----------|------------|
| 1 | 56 | 4480 | 17920 | 112 | 8.92 | 59.07% | 161.02 |
| 2 | 56 | 4480 | 15680 | 128 | 8.89 | 63.02% | 156.42 |
| 3 | 56 | 4800 | 14400 | 120 | 8.97 | 62.94% | 155.86 |

Candidate 2 was selected as the final architecture. Notably, **Candidate 1 has higher throughput but lower accuracy**, while Candidate 2 achieves the best accuracy with only a 3% throughput penalty. This demonstrates the authors prioritized accuracy within the memory budget rather than maximizing throughput alone.

**Data Mix for Distillation (Table 11):**

| Reasoning-SFT % | Pretraining % | Accuracy (Avg) |
|-----------------|---------------|----------------|
| 50 | 50 | 57.5% |
| 70 | 30 | 58.5% |
| 90 | 10 | 57.2% |

The **70/30 split yields the highest accuracy**, suggesting that post-training data (which includes reasoning traces and alignment signals) is more important than raw pretraining data for recovering accuracy after pruning. However, 100% post-training data was not tested, so the optimal ratio may be higher.

### 5.3 Ablation Studies and Robustness Checks

The paper includes several ablation studies that strengthen confidence in the results:

#### 5.3.1 Multilingual Data Ablation (Table 2)

The authors compare four multilingual data sources on Global-MMLU:

| Data Source | Average (8 languages) |
|-------------|----------------------|
| Common Crawl | 37.0% |
| FineWeb-2 | 35.1% |
| DiverseQA-wiki | 42.1% |
| DiverseQA-crawl | 47.0% |

This ablation directly informed the final data mixture, with `DiverseQA-crawl` receiving the highest weight. The **10-point gap between curated crawl data and synthetic QA data** is surprising and challenges the assumption that raw web data is superior. This finding has implications beyond this paper — it suggests synthetic data generation may be more effective than previously believed for multilingual capability.

#### 5.3.2 Fundamental Reasoning SFT Data Ablation (Table 3)

Adding 5% Fundamental Reasoning SFT data to `Nemotron-H 8B`:

| Model | Avg Math | MMLU-Pro |
|-------|----------|----------|
| Nemotron-H 8B | 37.92 | 44.24 |
| + FR-SFT data | 39.70 | 56.36 |

The **12-point improvement on MMLU-Pro** from just 5% data substitution is remarkable. This ablation validates the paper's claim that reasoning-specific data during pre-training provides measurable benefits. However, the ablation was done on `Nemotron-H 8B`, not the final Nano 2 model, so the exact magnitude may differ.

#### 5.3.3 Long-Context Training Length (Table 4)

The authors tested different training sequence lengths for the long-context extension phase:

| Train Length | Synthetic Data | RULER-128k |
|--------------|----------------|------------|
| 128k | yes | 73.68 |
| 256k | no | 70.19 |

Two findings stand out:
1. **Training at 512k is better than training at 128k** for 128k inference, supporting the authors' intuition that longer training sequences prevent document fragmentation.
2. **Synthetic data provides a 9-point boost** (70.19→79.04 at 256k), showing that long-context synthetic QA generation is effective.

#### 5.3.4 Budget Control Failure Modes (Figure 5)

Figure 5 shows before/after comparisons of budget control behavior:

- **Without truncation training (Figure 5a):** At short budgets, the model compensates by generating longer final answers, and well-formedness drops sharply (many responses have multiple `</think>` tags).
- **With truncation training (Figure 5b):** The compensation effect disappears, and well-formedness remains high across all budgets.

This ablation is critical because it shows the budget control feature **cannot be added post-hoc** — it requires training-time preparation. The paper does not report exact well-formedness percentages, which is a minor limitation, but the visual trend is clear.

#### 5.3.5 Distillation Pipeline Stages (Figure 6)

Figure 6 tracks accuracy across distillation stages:
- **DPO and GRPO** improve BFCL v3 (tool use) and IFEval (instruction following) but temporarily degrade MMLU-Pro.
- **Post-GRPO KD** recovers MMLU-Pro.
- **RLHF** improves ArenaHard but causes additional benchmark drops.
- **Model merging** recovers the drops while preserving ArenaHard gains.

This shows the **non-monotonic nature of alignment** — improvements in one capability can degrade others, requiring careful checkpoint selection and merging. The paper does not report exact values at each stage, which limits quantitative assessment, but the trend supports the multi-stage approach.

### 5.4 Critical Assessment of Experimental Claims

#### 5.4.1 Claims That Are Well-Supported

**Claim 1: "Up to 6.3× higher throughput than Qwen3-8B"**

**Verdict: Strongly supported.** Figure 1 clearly shows 6.3× throughput at 8k input / 16k output. The hardware setup is specified (single A10G, bfloat16), and the comparison is apples-to-apples. The caveat is that quantized baselines are not reported, which could narrow the gap in practice.

**Claim 2: "On-par or better accuracy on reasoning benchmarks"**

**Verdict: Supported with conditions.** Tables 5, 6, and 8 show Nemotron-Nano-9B-v2 outperforms Qwen3-8B on most reasoning benchmarks (AIME, MATH, GPQA, LiveCodeBench). However, on `ArenaHard` (chat capability), the 12B model scores 74.00 vs. 78.40 for Qwen3-8B (Table 8). The paper acknowledges this trade-off and addresses it through model merging, but the final 9B aligned model's ArenaHard score is not reported. This is a gap — readers cannot verify the merged model's chat capability.

**Claim 3: "128k context inference on a single A10G GPU"**

**Verdict: Supported.** The memory budget calculation is explicit (19.66 GiB available after framework overhead), and the RULER-128k scores (83.36% for 12B, 78.9% for 9B in Figure 1) confirm long-context capability is preserved. The compression to 8.89B parameters (Table 10) makes this feasible.

**Claim 4: "Budget control enables dynamic trade-offs between quality and latency"**

**Verdict: Partially supported.** Figure 5 shows the mechanism works (well-formed responses at various budgets), but the paper does not report **accuracy vs. budget curves**. Readers cannot see how much accuracy is lost at 2k vs. 8k vs. 16k thinking tokens. This is a significant gap for practitioners who need to set budgets.

#### 5.4.2 Limitations and Missing Information

**Missing: Final 9B Aligned Model Benchmarks**

The paper reports extensive benchmarks for the 12B aligned model (Table 8) and the 9B base model (Tables 5–6), but **does not report the full benchmark suite for the final 9B aligned model** (`Nemotron-Nano-9B-v2`). Figure 1 shows selected benchmarks, but readers cannot verify performance on all tasks after compression and alignment. This is a notable omission because compression could affect different capabilities differently.

**Missing: Quantized Baseline Comparisons**

All throughput comparisons assume `bfloat16` precision. In production, models are often quantized to INT8 or FP8, which would increase baseline throughput. The paper does not report how the throughput gap changes under quantization, which limits real-world applicability of the claims.

**Missing: Multi-GPU Scaling**

The paper focuses on single-GPU deployment, which is appropriate for the target use case. However, readers may wonder how the model scales across multiple GPUs for higher throughput. The paper does not address this, which is acceptable given the stated goals but worth noting.

**Missing: Training Compute Costs**

The paper specifies 20 trillion tokens for pretraining but does not report the **total compute hours or GPU costs**. For practitioners considering replication, this information would be valuable. The FP8 training recipe reduces costs compared to BF16, but the absolute cost is not disclosed.

#### 5.4.3 Potential Confounds

**Evaluation Contamination Risk**

The paper states they used `LLM Decontaminator` (Section 2.2.1) and contamination-free benchmarks like `LiveCodeBench`. However, the synthetic data generation used models like `Qwen3-30B-A3B` and `DeepSeek-V3`, which may have been trained on overlapping data. The paper does not report decontamination results for the synthetic data, which could inflate scores if test set leakage occurred.

**Checkpoint Selection Bias**

The model merging process involved a parameter sweep over α values from 0.1 to 0.9 (Section 3.2). The paper reports the best trade-off (α ≈ 0.5) but does not show the full sweep results. This introduces potential selection bias — the reported scores may represent the best checkpoint rather than a typical outcome.

**Baseline Version Sensitivity**

The paper compares against `Qwen3-8B` but does not specify which version (there may be multiple releases). Small differences in baseline versions can affect comparisons. The authors should have included model hashes or release dates for full reproducibility.

### 5.5 Summary of Experimental Evidence

The experimental analysis provides **strong but incomplete evidence** for the paper's claims:

| Claim | Evidence Strength | Key Supporting Data | Gaps |
|-------|-------------------|---------------------|------|
| 6.3× throughput gain | **Strong** | Figure 1, Section 4.2 | No quantized baselines |
| Better reasoning accuracy | **Strong** | Tables 5, 6, 8; Figure 1 | Missing full 9B aligned benchmarks |
| 128k context on A10G | **Strong** | RULER-128k scores, memory budget | None significant |
| Budget control works | **Moderate** | Figure 5 | No accuracy vs. budget curves |
| Compression preserves capability | **Strong** | Tables 9, 10, 11 | ArenaHard score for 9B not reported |

**Overall assessment:** The experiments are well-designed and the results are convincing for the core claims about throughput and reasoning accuracy. The main limitations are missing data for the final 9B aligned model on all benchmarks and lack of quantized baseline comparisons. For practitioners, the evidence is sufficient to conclude that `Nemotron-Nano-9B-v2` represents a meaningful advance in efficient reasoning models, but exact deployment performance should be validated on target hardware with representative workloads.

## 6. Limitations and Trade-offs

No technical approach is without constraints. Understanding where `Nemotron-Nano-9B-v2` may underperform, what assumptions it relies on, and which scenarios remain unaddressed is critical for practitioners considering adoption. This section provides a critical but fair assessment grounded in evidence from the paper.

### 6.1 Architectural Assumptions and Their Consequences

#### 6.1.1 The 8% Attention Layer Ratio May Not Generalize

The hybrid architecture relies on a specific design choice: **approximately 8% of layers use self-attention** (6 out of 62 layers, Section 2.1). This ratio was inherited from `Nemotron-H` and maintained during compression (4 attention layers out of 56, Section 4.2.2). While this works well for the benchmarks tested, several questions remain:

- **Task-specific attention needs.** Certain tasks may require more frequent attention mechanisms. For example, tasks requiring precise token-to-token alignment (such as certain translation or structured output tasks) might degrade more than math or code reasoning when attention is sparse. The paper does not evaluate such tasks.

- **Compression sensitivity.** During compression, the authors experimented with pruning Mamba heads but found this caused **"severe accuracy degradation"** (Section 4.1). This limits the compression axes available — essentially, the model can only be compressed via depth reduction, FFN pruning, and embedding channel pruning. If further compression is needed beyond the 12B→9B reduction, the architecture may not support it without significant capability loss.

- **Scaling uncertainty.** It is unclear whether the 8% attention ratio scales to larger or smaller models. The paper does not ablate this ratio, so practitioners cannot know if 5% or 15% attention layers would work better for their specific use cases.

#### 6.1.2 No Position Embeddings: A Double-Edged Sword

The model uses **no position embeddings**, relying on Mamba's inherent sequence modeling (Section 2.1). This reduces parameters and simplifies the architecture, but introduces assumptions:

- **Absolute position information is implicit.** While Mamba's state space model captures sequential order, certain tasks benefit from explicit position information (e.g., tasks where token position carries semantic meaning independent of content).

- **Long-context extrapolation risk.** The long-context extension phase trained at 512k tokens for 128k inference (Section 2.6), which helps, but models without position embeddings can sometimes struggle with **length extrapolation** beyond training distribution. The paper shows strong RULER-128k scores (83.36% for 12B, 78.9% for 9B in Figure 1), but does not test beyond 128k.

### 6.2 Capability Trade-offs: What Was Sacrificed

#### 6.2.1 Chat Capability vs. Reasoning Capability

The paper explicitly acknowledges a **trade-off between reasoning and chat capabilities** (Section 3.2). This is evident in Table 8:

| Model | AIME-2024 (Reasoning) | ArenaHard (Chat) |
|-------|----------------------|------------------|
| Nemotron-Nano-12B | 85.42% | 74.00% |
| Qwen3-8B | 75.83% | 78.40% |
| Qwen3-14B | 81.53% | 87.70% |

The 12B Nemotron model **outperforms Qwen3-14B on AIME-2024** (85.42% vs. 81.53%) but **trails significantly on ArenaHard** (74.00% vs. 87.70%). The authors address this through model merging (α ≈ 0.5 interpolation), but:

- **The final 9B aligned model's ArenaHard score is not reported.** Figure 1 shows selected benchmarks for `Nemotron-Nano-9B-v2`, but ArenaHard is absent. Practitioners cannot verify whether the merged 9B model preserves chat capability after compression.

- **Model merging is heuristic.** The α parameter sweep (0.1 to 0.9 in increments of 0.1) found 0.5 worked well, but this is **dataset and task-dependent**. Teams adopting this approach may need to re-sweep for their own use cases.

#### 6.2.2 Compression Costs: Capability Loss Is Real

While the paper emphasizes that compression preserves capability, the data shows **non-trivial degradation**:

| Benchmark | 12B Base | 9B Base | Loss |
|-----------|----------|---------|------|
| AIME 2024 pass@32 | 56.67% | 30.00% | -26.67 pts |
| MATH Level 5 | 67.61% | 63.64% | -3.97 pts |
| RULER-128K | 84.74% | 82.22% | -2.52 pts |

(Table 5)

The **AIME 2024 pass@32 drop from 56.67% to 30.00%** is substantial — nearly halving the elite math reasoning capability. While the 9B model still outperforms Qwen3-8B (20.00%), this shows that **compression does cost capability**, particularly on the hardest reasoning tasks. Practitioners should expect similar degradation if applying comparable compression ratios.

#### 6.2.3 Multilingual General Knowledge Gap

On multilingual benchmarks, the 9B base model **trails Qwen3-8B-Base**:

| Benchmark | N-Nano-V2 9B Base | Qwen3 8B Base |
|-----------|-------------------|---------------|
| Global-MMLU-Lite (Average) | 69.94% | 72.81% |

(Table 6)

However, on **Multilingual Math (MGSM)**, the 9B base model leads (84.67% vs. 80.93%). This suggests the model is **specialized for reasoning over general multilingual knowledge**. For applications requiring broad multilingual capability (e.g., customer support in multiple languages), this gap may be significant.

### 6.3 Computational and Data Constraints

#### 6.3.1 Training Compute Costs Are Undisclosed

The paper specifies **20 trillion tokens** for pretraining (Section 2.5) and approximately **90 billion tokens** for post-training (Introduction), but does not report:

- **Total GPU hours** required
- **Number of GPUs** used in parallel
- **Wall-clock training time**
- **Estimated dollar cost**

For practitioners considering replication, this is a significant gap. The FP8 training recipe reduces memory and compute compared to BF16, but without absolute cost figures, teams cannot budget for similar efforts. The scale (20T tokens) suggests this is a **multi-million dollar training run**, limiting accessibility to well-resourced organizations.

#### 6.3.2 Heavy Reliance on Synthetic Data

The pretraining and post-training pipelines rely extensively on **synthetically generated data**:

- **STEM data:** Generated using Qwen2.5-VL-72B-Instruct, Qwen3-30B-A3B, Qwen3-235B-A22B, DeepSeek-R1, and DeepSeek-V3 (Section 2.2.2)
- **Math data:** Nemotron-MIND regenerated using Phi-4 (Section 2.2.2)
- **Multilingual QA:** Translated using Qwen3-30B-A3B (Section 2.2.2)
- **Post-training responses:** Generated using DeepSeek-R1-0528 and Qwen3-235B-A22B (Section 3.1)

This introduces several concerns:

- **Potential contamination.** The synthetic data generators may have been trained on overlapping data with the evaluation benchmarks. While the paper mentions using `LLM Decontaminator` (Section 2.2.1), **decontamination results for synthetic data are not reported**.

- **Quality ceiling.** Synthetic data quality is bounded by the generator models. If Qwen3-235B-A22B makes systematic errors, those errors propagate into the training data. The paper does not report human evaluation of synthetic data quality.

- **Reproducibility dependency.** Teams wishing to replicate this approach need access to similarly capable generator models (72B+ parameters). This creates a **barrier to entry** for organizations without access to large proprietary models.

#### 6.3.3 Hardware Specificity: A10G Optimization

The compression pipeline is explicitly optimized for the **NVIDIA A10G GPU with 22 GiB memory** (Section 4.2):

> "Our goal was to be able to perform inference at a sequence length of 128k and a batch size of at least 1 within a memory budget of 19.66 GiB. We obtained the budget as follows: from the 22.06 GiB available memory on an NVIDIA A10G GPU, we subtract a 5% buffer for frameworks such as vLLM and TensorRT-LLM and another 1.3 GiB to allow sufficient space for a vision encoder." (Section 4.2)

This specificity creates limitations:

- **Other GPUs may not benefit equally.** The architecture search found optimal parameters for the A10G's memory profile. Running on A100 (40/80 GiB), H100 (80 GiB), or consumer GPUs (RTX 4090 with 24 GiB) may yield different optimal architectures.

- **Framework overhead assumptions.** The 5% buffer for vLLM/TensorRT-LLM and 1.3 GiB for vision encoder are **specific assumptions**. Teams using different inference frameworks or without vision encoder needs may have different memory budgets, potentially allowing less aggressive compression.

- **No multi-GPU scaling discussion.** The paper focuses entirely on single-GPU deployment. For organizations needing higher throughput via multi-GPU parallelism, the paper provides no guidance on how the model scales.

### 6.4 Evaluation Gaps and Missing Information

#### 6.4.1 Incomplete Benchmark Coverage for Final 9B Aligned Model

As noted in Section 5.4.2, the paper reports:

- **Full benchmarks for 12B aligned model** (Table 8)
- **Full benchmarks for 9B base model** (Tables 5–6)
- **Selected benchmarks for 9B aligned model** (Figure 1 only)

This means practitioners **cannot verify the final product's performance** on all tasks. Key missing data includes:

- ArenaHard score for `Nemotron-Nano-9B-v2`
- Full multilingual benchmark suite for the aligned 9B model
- Complete tool-use benchmark breakdown (BFCL v3 sub-tasks)

This gap is significant because compression and alignment interactions could affect different capabilities differently.

#### 6.4.2 Budget Control: No Accuracy vs. Budget Curves

The budget control feature (Section 3.4) is a key innovation, but the paper does not report:

- **How accuracy degrades as budget decreases.** Is there a linear relationship, or are there threshold effects?
- **Optimal budget recommendations per task.** What budget should be used for math vs. code vs. general reasoning?
- **Latency measurements at different budgets.** The feature enables latency control, but actual latency numbers at 2k, 8k, 16k budgets are not provided.

Figure 5 shows well-formedness improves with truncation training, but **well-formedness is not accuracy**. A response can be well-formed (single `</think>` tag) but incorrect. Practitioners need accuracy-budget curves to make informed deployment decisions.

#### 6.4.3 Quantized Baseline Comparisons Missing

All throughput comparisons assume **bfloat16 precision** (Figure 1 caption). In production, models are often quantized to INT8 or FP8, which would:

- **Increase baseline throughput** for both Nemotron and Qwen3
- **Potentially narrow the throughput gap** if Qwen3 quantizes more efficiently

The paper does not report:
- Throughput comparisons under INT8/FP8 quantization
- Accuracy degradation from quantization for either model
- Whether Nemotron's hybrid architecture quantizes as well as pure Transformer architectures

This limits real-world applicability of the throughput claims.

#### 6.4.4 Evaluation Code Not Released

The authors state:

> "We run evaluations of all models ourselves unless otherwise stated" (Section 2.7)

While this reduces baseline comparison inconsistencies, the paper also notes:

> "However, they do not release the exact evaluation code, which limits reproducibility." (Section 5.1.3)

Without evaluation code, independent verification of benchmark scores is difficult. This is particularly relevant for contested benchmarks like MATH and AIME, where evaluation details (e.g., answer parsing, tolerance for equivalent expressions) can affect scores by several points.

### 6.5 Unaddressed Scenarios and Edge Cases

#### 6.5.1 Extremely Long Context Beyond 128k

The model is optimized for **128k context inference**, trained at 512k sequence length (Section 2.6). However:

- **Performance beyond 128k is not tested.** Applications requiring 256k, 512k, or 1M token contexts cannot rely on this model without empirical validation.
- **The 512k training for 128k inference ratio may not scale.** Training at 4× the target context length worked here, but whether this generalizes to 1M+ token contexts is unknown.

#### 6.5.2 Real-Time Streaming Inference

The paper measures throughput at **8k input / 16k output** with maximum batch size (Section 4.2). This represents batched offline processing, not:

- **Streaming inference** where tokens are generated and consumed in real-time
- **Interactive applications** with user feedback loops
- **Variable-length workloads** where input/output lengths vary significantly

The hybrid Mamba-Transformer architecture may have different latency characteristics for streaming vs. batched inference, but this is not explored.

#### 6.5.3 Domain-Specific Fine-Tuning

The model is aligned for general reasoning, tool use, and conversation. However:

- **Domain adaptation costs are unknown.** Fine-tuning for specific domains (e.g., legal, medical, finance) may interact differently with the hybrid architecture compared to pure Transformers.
- **LoRA/adapter compatibility** is not discussed. Many organizations use parameter-efficient fine-tuning rather than full fine-tuning; the paper does not address whether the hybrid architecture supports this efficiently.

#### 6.5.4 Safety and Robustness

The safety dataset is notably small:

| Domain | Number of Samples |
|--------|-------------------|
| Safety | 2K |
| Math | 1.5M |
| Coding | 1.1M |
| Science | 2.0M |

(Table 7)

With only **2K safety samples** compared to millions of reasoning samples, several concerns arise:

- **Jailbreak resistance** may be weaker than models with larger safety datasets.
- **Adversarial robustness** is not evaluated. The paper does not test the model against adversarial prompts or jailbreak attempts.
- **Compression effects on safety** are unknown. Pruning could potentially remove safety-related neurons, but this is not analyzed.

### 6.6 Open Questions for Future Work

Based on the limitations above, several open questions remain:

1. **What is the minimum attention layer ratio for different task families?** The 8% ratio works for reasoning, but is it optimal for all tasks?

2. **How does the model scale beyond 9B parameters?** The paper compresses 12B→9B, but what about 12B→6B or 12B→3B? Is there a compression floor?

3. **Can budget control be extended to multi-turn conversations?** The current implementation counts tokens per response; multi-turn budget allocation is not addressed.

4. **How does the hybrid architecture interact with speculative decoding?** Mamba layers have different computation patterns than attention; speculative decoding efficiency may differ.

5. **What are the energy efficiency implications?** Throughput is measured, but power consumption (tokens per watt) is not reported. For cost-sensitive deployments, this may matter more than raw throughput.

| Scenario | Recommendation | Reasoning |
|----------|---------------|-----------|
| **Long-context reasoning on single GPU** | ✅ Strong fit | Core design target; 128k on A10G demonstrated |
| **High-throughput math/code reasoning** | ✅ Strong fit | 6.3× throughput gain with accuracy improvement |
| **General multilingual chat** | ⚠️ Evaluate carefully | Global-MMLU-Lite trails Qwen3-8B; ArenaHard not reported for 9B |

**Key takeaway:** `Nemotron-Nano-9B-v2` excels at its stated design goal — **efficient long-context reasoning on memory-constrained hardware**. However, practitioners should validate performance on their specific tasks, particularly for chat capability, multilingual general knowledge, and safety-critical applications. The missing 9B aligned model benchmarks and lack of quantized comparisons mean real-world performance should be verified on target hardware before deployment commitments.

## 7. Implications and Future Directions

This paper does more than introduce a new model checkpoint — it establishes a **blueprint for efficient reasoning model development** that challenges several prevailing assumptions in the field. In this section, we examine how `Nemotron-Nano-9B-v2` shifts the landscape, what research directions it opens, and when practitioners should adopt this approach versus alternatives.

### 7.1 How This Work Changes the Landscape

#### 7.1.1 Breaking the Accuracy-Efficiency Trade-off Myth

Prior to this work, the community largely accepted that **reasoning capability requires large models or pure Transformer architectures**. The prevailing wisdom held that:
- Sub-10B models could not compete with 14B+ models on elite math benchmarks like AIME
- Efficient architectures (Mamba, hybrid designs) would sacrifice accuracy for throughput
- Long-context inference (128k+) required multi-GPU setups

`Nemotron-Nano-9B-v2` directly contradicts all three assumptions. As Figure 1 shows, the 9B model achieves **higher accuracy than Qwen3-8B** on AIME24 (81.9% vs. 75.8%), GPQA-D (64.0% vs. 59.6%), and LiveCodeBench (71.1% vs. 59.5%) while delivering **6.3× higher throughput**. Table 8 further shows the 12B base outperforms `Qwen3-14B` on AIME-2024 (85.42% vs. 81.53%) despite having 2B fewer parameters.

**The landscape shift:** Efficiency and accuracy are no longer orthogonal objectives. With co-designed architecture, training data, and compression, models can improve on both dimensions simultaneously. This reframes how teams should approach model development — rather than accepting trade-offs, they should invest in **holistic optimization** across all pipeline stages.

#### 7.1.2 Thinking Budget Control as a First-Class Feature

Before this work, reasoning models operated with **unpredictable inference costs**. A model might generate 2k thinking tokens for one query and 20k for another, making latency guarantees impossible. Production systems had to either:
- Over-provision resources for worst-case scenarios
- Accept variable latency that degrades user experience
- Use separate models for different latency tiers

The budget control mechanism (Section 3.4) introduces **deterministic inference costing** for reasoning workloads. By training with ~5% truncated reasoning traces, the model learns to produce coherent answers at any budget level without compensation effects (Figure 5b).

**The landscape shift:** Reasoning models can now be deployed with **SLA-bound latency guarantees**, similar to how traditional APIs specify response time commitments. This enables reasoning capabilities in latency-sensitive applications (real-time coding assistants, interactive tutoring) that were previously impractical.

#### 7.1.3 Compression of Aligned Models (Not Just Base Models)

The Minitron framework (Muralidharan et al., 2024) originally targeted **base pre-trained models**. This paper demonstrates that **reasoning-aligned models** — which have undergone SFT, DPO, GRPO, and RLHF — can be compressed while preserving alignment behaviors. This is non-trivial because:

> "Compressing a reasoning-aligned model is fundamentally different from compressing a base model — the alignment procedures (SFT, RLHF, DPO) create behavioral patterns that can be disrupted by naive pruning" (Section 4).

The multi-stage distillation pipeline (Section 4.3) interleaves pruning with behavioral recovery: DPO for tool-calling, GRPO for instruction-following, and model merging to balance reasoning vs. chat capabilities.

**The landscape shift:** Organizations that have invested in expensive alignment procedures no longer need to discard that work when compressing models. They can **compress post-alignment**, preserving behavioral capabilities while reducing deployment costs. This changes the economics of model development — alignment investments have longer useful lifespans.

#### 7.1.4 Data Quality Over Quantity for Reasoning

The paper challenges the "more tokens = better" assumption that dominates pre-training discussions. The Fundamental Reasoning SFT data ablation (Table 3) shows that replacing just 5% of Common Crawl data with reasoning-focused synthetic data improved MMLU-Pro from **44.24 to 56.36** — a 12-point gain from a small data substitution. Similarly, the math extraction pipeline produced a 133B-token corpus that "surpasses all existing open math datasets" (Section 2.2.1) through **higher-fidelity extraction**, not larger scale.

**The landscape shift:** For reasoning capabilities, **data curation quality may matter more than raw token count**. This opens a pathway for teams with limited training budgets to compete through superior data engineering rather than compute scaling.

### 7.2 Follow-up Research Directions

The paper's contributions enable several concrete research directions that the community should pursue:

#### 7.2.1 Attention Ratio Ablation Studies

The model uses **~8% attention layers** (6 of 62 in the 12B base, 4 of 56 in the 9B compressed model). This ratio was inherited from `Nemotron-H` without systematic ablation. Key open questions:

- **Task-specific optimal ratios.** Does code generation benefit from more attention than math reasoning? Does translation require different attention density than summarization?
- **Dynamic attention allocation.** Could models learn to route tokens through attention vs. Mamba layers based on task demands, rather than using a fixed pattern?
- **Scaling laws for attention ratio.** Does the 8% ratio hold at 3B, 30B, or 100B parameter scales, or does it follow a scaling law?

**Suggested methodology:** Train a family of models with 4%, 8%, 12%, and 16% attention layers on identical data, then evaluate across task families. This would establish whether the 8% ratio is a universal optimum or task-dependent.

#### 7.2.2 Compression Beyond 9B Parameters

The paper compresses 12B→9B (~25% reduction). Table 9 shows accuracy degrades significantly at 52 layers (44.92% vs. 51.48% at 56 layers), suggesting a **compression floor** near 9B for this architecture. Open questions:

- **What is the minimum viable parameter count?** Can the hybrid architecture support 6B or 3B models while retaining reasoning capability?
- **Alternative compression axes.** The paper found Mamba head pruning caused "severe accuracy degradation" (Section 4.1). Are there other compression methods (quantization, low-rank adaptation) that work better for hybrid models?
- **Progressive compression.** Could iterative compression (12B→9B→6B) with distillation at each step preserve more capability than single-step compression?

**Suggested methodology:** Apply the same Minitron pipeline to reach 6B parameters, then measure the accuracy-throughput frontier. Compare against native 6B training to determine if compression or training-from-scratch is preferable.

#### 7.2.3 Budget Control Extensions

The current budget control implementation counts tokens per response (Section 3.4). Several extensions would increase practical utility:

- **Multi-turn budget allocation.** For conversations spanning 10+ turns, how should budgets be distributed? Should early turns receive more budget for context establishment, or should all turns receive equal budgets?
- **Adaptive budgeting.** Could models learn to request more budget when encountering difficult subproblems, similar to how humans spend more time on hard questions?
- **Budget-aware training.** Beyond truncation, could models be trained with explicit budget signals (e.g., "you have 5k tokens remaining") to improve budget utilization efficiency?

**Suggested methodology:** Extend the truncation training to multi-turn conversations, then evaluate conversation quality at various per-turn and total budget configurations.

#### 7.2.4 Quantization Studies for Hybrid Architectures

All throughput comparisons in the paper assume **bfloat16 precision** (Figure 1 caption). However, production deployments often use INT8 or FP8 quantization. Open questions:

- **Does the hybrid architecture quantize as well as pure Transformers?** Mamba's state space computations may have different numerical sensitivity than attention.
- **Quantization-aware training for hybrid models.** Would training with quantization simulation improve post-training quantization accuracy?
- **Mixed-precision strategies.** Could attention layers remain in BF16 while Mamba layers use INT8, optimizing the accuracy-throughput trade-off?

**Suggested methodology:** Apply standard INT8 quantization (e.g., AWQ, GPTQ) to `Nemotron-Nano-9B-v2` and measure accuracy degradation vs. `Qwen3-8B` under the same quantization. This would determine whether the hybrid architecture has quantization advantages or disadvantages.

#### 7.2.5 Energy Efficiency Measurements

The paper reports **throughput (tokens/second)** but not **energy efficiency (tokens/watt)**. For cost-sensitive and environmentally-conscious deployments, energy metrics may matter more than raw throughput. Open questions:

- **Does the hybrid architecture reduce energy consumption?** Mamba layers have different compute patterns than attention — do they consume less power per token?
- **Energy-accuracy trade-offs.** At what point does accuracy degradation from aggressive compression outweigh energy savings?
- **Hardware-specific optimization.** Do different GPU architectures (A10G vs. H100 vs. consumer GPUs) favor different points on the energy-accuracy frontier?

**Suggested methodology:** Measure power draw during inference using GPU telemetry (e.g., `nvidia-smi`), then compute tokens/watt for various model configurations. This would enable energy-aware model selection for sustainability-focused deployments.

#### 7.2.6 Domain Adaptation for Hybrid Models

The model is aligned for general reasoning, tool use, and conversation. However, many applications require **domain-specific fine-tuning** (legal, medical, finance). Open questions:

- **Do hybrid architectures adapt differently than pure Transformers?** The Mamba layers may capture domain patterns differently than attention.
- **Parameter-efficient fine-tuning compatibility.** Can LoRA adapters be applied to Mamba layers as effectively as to attention layers?
- **Catastrophic forgetting risk.** Does the sparse attention pattern make hybrid models more or less susceptible to forgetting general capabilities during domain adaptation?

**Suggested methodology:** Fine-tune `Nemotron-Nano-9B-v2` on domain-specific datasets (e.g., legal contracts, medical notes) using LoRA, then evaluate both domain performance and general capability retention compared to pure Transformer baselines.

### 7.5 Summary: The Path Forward

`Nemotron-Nano-9B-v2` demonstrates that **efficient reasoning models are achievable without sacrificing capability** — but only through co-design of architecture, training data, alignment, and compression. The key takeaways for the field are:

1. **Hybrid architectures are viable for reasoning.** The Mamba-Transformer combination preserves accuracy while enabling 6.3× throughput gains.

2. **Budget control requires training-time preparation.** Inference-time features must be baked into training (via truncation) to work correctly.

3. **Compression of aligned models is feasible.** With multi-stage distillation, behavioral capabilities can be preserved through pruning.

4. **Data quality matters more than quantity for reasoning.** Carefully curated synthetic data outperforms larger volumes of raw web data.

5. **Deployment constraints can drive better design.** Optimizing for specific hardware (A10G, 128k context) produced a model that outperforms less-constrained alternatives.
