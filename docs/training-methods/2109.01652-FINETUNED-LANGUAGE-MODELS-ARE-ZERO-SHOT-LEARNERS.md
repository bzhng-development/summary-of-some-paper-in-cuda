## 1. Executive Summary
This paper introduces instruction tuning, a method where a 137B parameter pretrained language model (LaMDA-PT) is finetuned on over 60 NLP datasets described via natural language instructions to create FLAN (Finetuned Language Net). FLAN substantially improves zero-shot performance on unseen task types, surpassing zero-shot 175B GPT-3 on 20 of 25 evaluated datasets and even outperforming few-shot GPT-3 on specific tasks like ANLI, RTE, and BoolQ. This work demonstrates that supervised finetuning on diverse instructions enables large language models to generalize to unseen tasks without exemplars, effectively bridging the gap between the pretrain–finetune and prompting paradigms.

## 2. Context and Motivation

### The Zero-Shot Learning Gap in Large Language Models

This paper addresses a critical limitation in large-scale language models: **the performance gap between few-shot and zero-shot learning**. While models like GPT-3 (175B parameters) demonstrated remarkable few-shot learning capabilities—where the model receives a handful of input-output examples before making a prediction—their zero-shot performance (no examples, only an instruction or prompt) remained substantially weaker on many tasks.

The Introduction explicitly states: *"GPT-3's zero-shot performance is much worse than few-shot performance on tasks such as reading comprehension, question answering, and natural language inference."* This gap matters because zero-shot interaction is more practical for real-world deployment. Requiring users to provide few-shot exemplars for every new task creates friction—it demands expertise in prompt engineering, increases token costs, and limits accessibility for non-technical users.

### Why This Problem Matters

The significance of this problem operates on two levels:

**Practical Impact:** If large language models could reliably follow natural language instructions without examples, they would become dramatically more accessible. A user could simply ask "Translate this sentence to Spanish" or "Is this movie review positive?" without needing to construct carefully engineered prompts with training examples. This lowers the barrier to entry and reduces computational costs at inference time.

**Theoretical Significance:** The zero-shot gap reveals something fundamental about how language models learn. If a model truly understands language and tasks, it should generalize from instructions alone. The fact that few-shot examples are often necessary suggests the model is pattern-matching to pretraining data formats rather than genuinely understanding task semantics. Closing this gap tests whether models can learn *how to follow instructions* as a transferable skill.

### Prior Approaches and Their Limitations

Figure 2 illustrates three distinct paradigms that existed before this work:

| Paradigm | Example Models | Key Limitation |
|----------|---------------|----------------|
| **Pretrain–Finetune** | BERT, T5 | Requires many task-specific examples; produces one specialized model per task |
| **Prompting** | GPT-3 | Zero-shot performance is weak; relies on few-shot exemplars or careful prompt engineering |
| **Instruction Tuning** | FLAN (this work) | Combines benefits of both—single model, instruction-based, generalizes to unseen tasks |

**Pretrain–Finetune Paradigm:** Traditional supervised finetuning (e.g., BERT, T5) achieves strong performance but requires substantial labeled data for each downstream task. More critically, it produces *specialist models*—one checkpoint per task. This approach doesn't scale when you want a single model to handle hundreds of different task types, and it provides no mechanism for zero-shot generalization to unseen tasks.

**Prompting Paradigm:** GPT-3 introduced in-context learning, where a single pretrained model handles multiple tasks via text prompts. However, as the Introduction notes, *"without few-shot exemplars, it is harder for models to perform well on prompts that are not similar to the format of the pretraining data."* The model wasn't trained to interpret instructions—it was trained to predict the next token. When you ask it a question in an unfamiliar format, it struggles because that format rarely appears naturally in web text.

**Early Instruction-Based Work:** Prior research explored instruction-like formulations but with critical limitations:
- **QA-based task formulation** (McCann et al., 2018; Kumar et al., 2016) cast NLP tasks as question-answering but focused on *multi-task learning* (improving performance on seen tasks) rather than *zero-shot generalization* to unseen tasks.
- **Mishra et al. (2021)** finetuned a 140M parameter BART model on instructions with few-shot exemplars, evaluating few-shot abilities on unseen tasks. However, the model scale was three orders of magnitude smaller than FLAN.
- **Sanh et al. (2021)** finetuned T5-11B on prompts and reported zero-shot improvements, but at 11B parameters, this is still an order of magnitude smaller than FLAN's 137B.

### How This Paper Positions Itself

FLAN occupies a unique position in the research landscape by combining three key elements that prior work did not simultaneously address:

1. **Scale:** At 137B parameters, FLAN operates at the scale where emergent capabilities appear. The ablation studies (Section 4.2) reveal that instruction tuning actually *hurts* performance on models below ~8B parameters, suggesting the benefits only emerge at sufficient scale. This distinguishes FLAN from smaller instruction-tuned models.

2. **Zero-Shot Focus:** Unlike multi-task learning work that optimizes for seen tasks, FLAN explicitly evaluates on *held-out task clusters*. For example, when evaluating natural language inference, no NLI datasets appeared during instruction tuning. This tests genuine generalization rather than task interference or multi-task benefits.

3. **Natural Language Instructions:** Rather than using dataset names or tags as prompts (as T5 does), FLAN uses diverse natural language instructions like "Does the premise entail the hypothesis?" This makes the model more accessible to non-technical users and tests whether models can learn instruction-following as a general skill.

The Discussion (Section 6) frames this contribution as bridging a conceptual divide: *"Instruction tuning is a simple method that... combines appealing aspects of both the pretrain–finetune and prompting paradigms by using supervision via finetuning to improve language model's responses to inference-time text interactions."* The work demonstrates that labeled data—typically used for specialist models—can instead train *generalist models* that generalize across task boundaries.

This positioning is crucial because it challenges the assumed tradeoff between specialist and generalist models. The paper shows that task-specific training data, when structured as instructions across many tasks, can actually *enhance* cross-task generalization rather than narrowing the model's capabilities.

## 3. Technical Approach

This is an empirical systems paper that demonstrates how supervised finetuning on diverse natural language instructions enables a large language model to generalize to unseen tasks. The core idea is straightforward: instead of training a model to predict the next token in arbitrary text, we train it to respond to explicit task descriptions, teaching it the *skill* of instruction-following rather than just task-specific patterns.

### 3.1 Reader orientation (approachable technical breakdown)

The system being built is FLAN (Finetuned Language Net), a 137-billion parameter language model that has been specially trained to understand and execute natural language instructions like "Translate this to Spanish" or "Does this premise entail this hypothesis?" without needing example demonstrations. The problem it solves is the zero-shot learning gap—large models perform poorly when given only instructions without examples—and the solution shape is to expose the model to many different tasks during finetuning, all formatted as natural language instructions, so it learns the general pattern of "instruction → appropriate response" rather than memorizing specific task formats.

### 3.2 Big-picture architecture (diagram in words)

The FLAN system consists of four major components connected in a sequential pipeline. First, the **Base Pretrained Model** (LaMDA-PT, 137B parameters) serves as the foundation—a decoder-only transformer that has already learned general language patterns from 2.49 trillion tokens of web text, dialog data, and Wikipedia. Second, the **Instruction Dataset Constructor** transforms 62 existing NLP datasets into a unified instructional format by wrapping each input-output pair with one of ten manually written natural language templates per dataset, organized into 12 task clusters (Figure 3). Third, the **Instruction Tuning Pipeline** mixes all datasets together, randomly samples instruction templates during training, and finetunes the base model for 30,000 gradient steps using specific hyperparameters (batch size 8,192 tokens, learning rate 3e-5). Fourth, the **Zero-Shot Evaluation Framework** tests the finetuned model on entirely held-out task clusters—for example, evaluating natural language inference performance when no NLI datasets appeared during instruction tuning—ensuring genuine generalization rather than task memorization.

### 3.3 Roadmap for the deep dive

- We start with the **base model architecture and pretraining** because understanding what FLAN builds upon clarifies what instruction tuning actually changes.
- We then explain the **instruction dataset construction** because the quality and diversity of training instructions directly determine what the model learns to generalize.
- We detail the **instruction tuning procedure** including data mixing, template selection, and optimization hyperparameters because these engineering choices determine training stability and effectiveness.
- We cover the **classification output handling** because decoder-only models naturally generate free text, requiring special treatment for tasks with fixed answer sets.
- We describe the **evaluation methodology** because the held-out task cluster design is critical for proving zero-shot generalization rather than memorization.
- We conclude with **scaling and ablation findings** because these reveal when and why instruction tuning works, which is essential for understanding the method's boundaries.

### 3.4 Detailed, sentence-based technical breakdown

**Base Model Architecture and Pretraining Foundation**

The foundation of FLAN is LaMDA-PT, a dense left-to-right decoder-only transformer language model with 137 billion parameters, which means every parameter is active during every forward pass rather than using sparse mixture-of-experts routing. This model was pretrained on a collection of 2.49 trillion Byte Pair Encoding (BPE) tokens using a vocabulary of 32,000 tokens created with the SentencePiece library, where approximately 10% of the pretraining data was non-English text. The pretraining corpus included web documents (including computer code), dialog data, and Wikipedia, but critically, LaMDA-PT only underwent language model pretraining and was *not* finetuned for dialog applications (unlike the later LaMDA model). This distinction matters because it ensures that any instruction-following capability emerges from the instruction tuning process itself, not from prior dialog optimization. The decoder-only architecture means the model processes text autoregressively—generating one token at a time conditioned on all previous tokens—which naturally suits free-text generation but requires modification for classification tasks with fixed output spaces.

**Instruction Dataset Construction and Task Clustering**

The instruction tuning dataset aggregates 62 publicly available text datasets from TensorFlow Datasets, which are then organized into 12 task clusters based on their underlying task type (Figure 3). Each task cluster groups datasets that share the same fundamental objective: for example, the Natural Language Inference cluster contains 7 datasets (ANLI R1–R3, CB, MNLI, QNLI, RTE, SNLI, WNLI) that all ask the model to determine whether a hypothesis follows from a premise. The 12 clusters include Natural Language Inference (7 datasets), Commonsense Reasoning (4 datasets), Sentiment Analysis (4 datasets), Struct-to-Text Generation (4 datasets), Closed-Book QA (3 datasets), Coreference Resolution (3 datasets), Translation (8 datasets), Summarization (11 datasets), Reading Comprehension (5 datasets), Paraphrase Detection (4 datasets), Reading Comprehension with Commonsense (2 datasets), and Miscellaneous (7 datasets). For each of the 62 datasets, the authors manually composed 10 unique instruction templates that use natural language to describe the task, such as "Does the premise entail the hypothesis?" for NLI or "Translate this sentence to Spanish:" for translation. To increase template diversity and prevent overfitting to any single phrasing, up to 3 of the 10 templates per dataset "turn the task around"—for sentiment classification, this might mean generating a movie review given a sentiment label rather than classifying a review's sentiment. During training, one template is randomly selected for each example, which forces the model to learn the underlying task semantics rather than memorizing template-specific patterns.

**Instruction Tuning Procedure and Data Mixing**

The instruction tuning pipeline mixes all 62 datasets together and randomly samples from each dataset during training, but with careful balancing to prevent larger datasets from dominating the training signal. To balance different dataset sizes, the number of training examples per dataset is capped at 30,000 examples, ensuring no single dataset contributes disproportionately to the gradient updates. The pipeline follows an examples-proportional mixing scheme with a mixing rate maximum of 3,000, which means that a dataset does not receive additional sampling weight for examples beyond 3,000—this prevents very large datasets from overwhelming the mixture even after the 30,000 example cap. The model is finetuned for exactly 30,000 gradient steps with a batch size of 8,192 tokens, using the Adafactor optimizer with a learning rate of $3 \times 10^{-5}$. The input sequence length during finetuning is 1,024 tokens and the target sequence length is 256 tokens, which constrains both how much context the model can receive and how long its responses can be. The training uses packing, a technique that combines multiple training examples into a single sequence by separating inputs from targets using a special end-of-sequence (EOS) token, which improves computational efficiency by reducing padding. This entire instruction tuning process takes approximately 60 hours on a TPUv3 pod with 128 cores, and all reported evaluations use the final checkpoint trained for the full 30,000 steps.

**Classification Output Handling with Options Suffix**

A critical technical detail addresses how FLAN handles classification tasks versus generation tasks, since decoder-only language models naturally produce free text rather than selecting from fixed categories. For generation tasks like translation or summarization, no modification is needed—the model simply generates text until it reaches the target length or produces an EOS token. For classification tasks, prior work like GPT-3 used rank classification, where only specific output tokens (e.g., "yes" and "no") are considered and the higher probability token is taken as the prediction. However, this approach is imperfect because the probability mass for answers may have an undesired distribution among different ways of expressing each answer—for instance, many alternative phrasings of "yes" may each receive low probability, collectively lowering the total mass assigned to affirmative answers. FLAN addresses this by appending an options suffix to classification tasks, where the token "OPTIONS" is added to the end of the instruction along with a list of valid output classes for that task. For example, a natural language inference task would end with "OPTIONS: -yes -no -it is not possible to tell", which makes the model explicitly aware of which choices are valid when responding. This design choice appears in the NLI and commonsense examples in Figure 1, where the model can see the available answer choices before generating its response, reducing ambiguity about the expected output format.

**Zero-Shot Evaluation Methodology with Held-Out Task Clusters**

The evaluation framework is designed to test genuine zero-shot generalization rather than memorization of specific datasets, which requires a careful definition of what counts as an "unseen task." Whereas some prior work defines unseen tasks by simply disallowing the same dataset to appear in training, FLAN uses a more conservative definition that leverages the task clusters from Figure 3. A dataset $D$ is considered unseen at evaluation time only if *no datasets* from any task clusters that $D$ belongs to were seen during instruction tuning. For instance, if $D$ is an entailment task from the Natural Language Inference cluster, then no entailment datasets appeared in instruction tuning—the model was trained on all other clusters but held out the entire NLI cluster. To evaluate zero-shot FLAN on $c$ task clusters, the authors instruction tune $c$ separate models, where each model holds out a different task cluster for evaluation. This means there isn't a single FLAN checkpoint evaluated on all tasks; rather, each evaluation task cluster uses a different checkpoint that was never exposed to that cluster type during instruction tuning. When evaluating on the Reading Comprehension with Commonsense cluster, both Reading Comprehension and Commonsense Reasoning clusters were dropped from instruction tuning, and conversely, the Reading Comprehension with Commonsense cluster was not used for instruction tuning when evaluating on Reading Comprehension or Commonsense Reasoning separately. The paraphrase cluster is also dropped from instruction tuning when evaluating on NLI tasks and vice-versa, since paraphrase detection and NLI are closely related tasks. For each dataset, performance is reported as the mean across all 10 instruction templates, which proxies the expected performance given a typical natural language instruction, though the authors also report results using the template with the best development set performance for comparison with GPT-3 baselines.

**Scaling Laws and Model Size Dependencies**

A crucial finding from the ablation studies reveals that instruction tuning benefits are strongly dependent on model scale, which has important implications for when this technique should be applied. The authors evaluate instruction tuning on models of size 422 million, 2 billion, 8 billion, 68 billion, and 137 billion parameters using the same cluster split as the main experiments. Figure 7 shows that for the two models on the order of 100 billion parameters (68B and 137B), instruction tuning substantially improves performance on held-out tasks, consistent with the main results. However, for the 8 billion parameter model and smaller, instruction tuning actually *hurts* performance on held-out tasks compared to the untuned baseline. The authors propose an explanation: for small-scale models, learning the approximately 40 tasks used during instruction tuning fills the entire model capacity, causing these models to overfit to the training tasks and perform worse on new tasks. Under this explanation, for larger scale models, instruction tuning fills up some model capacity but also teaches these models *how to follow instructions* as a transferable skill, allowing them to generalize to new tasks with the remaining capacity. This scaling dependency is a critical design consideration—it means instruction tuning is not universally beneficial and requires sufficient model scale (appearing to emerge around the 100B parameter range) to provide zero-shot generalization benefits rather than causing catastrophic forgetting of pretrained capabilities.

**Number of Instruction Tuning Clusters and Task Diversity**

The ablation study in Section 4.1 examines how performance on unseen tasks is affected by the number of task clusters and individual tasks used during instruction tuning, holding out Natural Language Inference, Closed-Book QA, and Commonsense Reasoning as evaluation clusters while using the seven remaining clusters for instruction tuning. Figure 6 shows results for one through seven instruction tuning clusters, where clusters are added in decreasing order of number of tasks per cluster. The average performance across the three held-out clusters improves as additional clusters and tasks are added to instruction tuning, with the exception of the sentiment analysis cluster which provides minimal added value. Starting from 49.9% average accuracy with one cluster, performance increases to 55.0% with two clusters, 59.3% with three, 59.2% with four, 60.8% with five, 61.9% with six, and 63.5% with seven clusters. Notably, performance does not appear to saturate at seven clusters, implying that performance may further improve with even more clusters added to instruction tuning. When examining datasets per task cluster (Appendix B.1), using more datasets per cluster improved performance by almost 10% on average across the three held-out clusters, while using more templates per dataset had a comparatively negligible effect on performance when there was sufficient dataset diversity per cluster. This suggests that task *diversity* (many different types of tasks) matters more than template *variety* (many ways of asking the same task) for zero-shot generalization.

**Few-Shot Exemplars Combined with Instruction Tuning**

Although the primary focus is zero-shot learning, the paper also explores how instruction tuning can be combined with few-shot exemplars at inference time, creating a complementary method for further performance improvement. The few-shot format builds on the zero-shot format: for some input $x$ and output $y$, let $\text{instruct}(x)$ denote the zero-shot instructions, then given $k$ few-shot exemplars $(x_i, y_i)_{i=1}^k$ and a new input $x$, the instruction format is $\text{instruct}(x_1) \oplus y_1 \oplus \text{instruct}(x_2) \oplus y_2 \oplus \ldots \oplus \text{instruct}(x_k) \oplus y_k \oplus \text{instruct}(x)$, where $\oplus$ denotes string concatenation with a delimiter token inserted between components. At both training and inference time, exemplars are randomly drawn from the training set, and the number of exemplars is capped at 16 such that the total sequence length remains less than 960 tokens. Figure 9 shows that few-shot exemplars improve performance on all task clusters compared with zero-shot FLAN, with exemplars being especially effective for tasks with large or complex output spaces such as struct-to-text generation, translation, and closed-book QA—potentially because exemplars help the model better understand the expected output format. Additionally, for all task clusters, the standard deviation among templates is lower for few-shot FLAN, indicating reduced sensitivity to prompt engineering and more stable performance across different instruction phrasings.

**Instruction Tuning Facilitates Prompt Tuning**

As further analysis, the paper demonstrates that instruction tuning makes models more amenable to prompt tuning, where continuous variables (soft prompts) are optimized via backpropagation rather than using discrete text instructions. The authors train continuous prompts for each SuperGLUE task following the cluster splits from Section 2.2, such that when prompt-tuning on task $T$, no tasks in the same cluster as $T$ were seen during instruction tuning. The prompt tuning setup follows the procedure of Lester et al. (2021) except with a prompt length of 10, weight decay of $1 \times 10^{-4}$, and no dropout on attention scores—changes that preliminary experiments showed improved LaMDA-PT performance. Figure 10 shows results for both fully-supervised training and a low-resource setting with only 32 training examples. In all scenarios, prompt tuning works better with FLAN than with the untuned LaMDA-PT baseline, and in many cases—especially for the low-resource setting—prompt tuning on FLAN achieves more than 10% improvement over prompt tuning on LaMDA-PT. Table 4 in Appendix B.3 shows per-dataset results where, for example, on BoolQ with 32 training examples, FLAN achieves 77.5% accuracy versus 55.5% for LaMDA-PT, and on RTE with 32 examples, FLAN achieves 83.0% versus 52.4% for LaMDA-PT. This result exemplifies how instruction tuning produces a checkpoint that is more desirable for performing NLP tasks generally, not just for zero-shot instruction following—the model has learned a more task-amenable representation space that responds better to both discrete instructions and continuous prompt optimization.

**Design Choices and Alternative Approaches Considered**

Several design choices in the FLAN approach reflect deliberate trade-offs against alternative methods. The decision to use natural language instructions rather than dataset names or tags (as T5 does) is validated by the ablation study in Figure 8 and Table 3, which compares three finetuning setups: (1) no template, where only inputs and outputs are given; (2) dataset name, where each input is prepended with the task and dataset name like "[Translation: WMT'14 to French]"; and (3) natural instructions, which is the full FLAN approach. Both ablation configurations performed substantially worse than FLAN—the no template setup achieved 37.3% average zero-shot performance, the dataset name setup achieved 46.6% with instruction evaluation, while FLAN with natural instructions achieved 55.2%. This indicates that training with genuine natural language instructions is crucial for zero-shot performance on unseen tasks, not just multi-task finetuning generally. The choice to hold out entire task clusters rather than individual datasets for evaluation is more conservative than prior work but necessary to prove genuine task-type generalization rather than dataset-specific memorization. The 30,000 step training duration and 8,192 token batch size were selected to balance training stability with computational cost—the 60-hour training time on 128 TPUv3 cores represents less than 2% of the pretraining compute cost, making instruction tuning relatively efficient compared to full pretraining. The input length of 1,024 tokens is a limitation that prevents evaluation on most summarization tasks where mean input length exceeds this constraint, representing a known boundary of the current approach.

## 4. Key Insights and Innovations

This section identifies the conceptual breakthroughs that distinguish FLAN from prior work. While the Technical Approach (Section 3) explained *how* the system was built, these insights reveal *why* it works and what it teaches us about language model capabilities. I distinguish between fundamental innovations—ideas that change how we think about model training—and incremental improvements that refine existing methods.

---

### 4.1 Instruction Tuning as a Paradigm Bridge (Fundamental Innovation)

**What Makes This Novel:** Prior work treated labeled data and unlabeled pretraining as serving fundamentally different purposes. Labeled data created *specialist models* (one checkpoint per task, as with BERT finetuning), while unlabeled pretraining created *generalist models* that relied on prompting for task adaptation (as with GPT-3). FLAN demonstrates that labeled data, when structured as diverse natural language instructions across many tasks, can train a *single generalist model* that generalizes to unseen tasks.

This is not merely multi-task learning. As the Related Work (Section 5) notes, prior multi-task work like McCann et al. (2018) focused on improving performance on *seen tasks* or adapting to new *domains* of the same task type. FLAN's innovation is testing generalization to entirely held-out *task types*—for example, evaluating natural language inference when no NLI datasets appeared during instruction tuning. The held-out task cluster methodology (Section 2.2) ensures this is genuine cross-task transfer, not just reduced overfitting.

**Why This Matters:**

- **Reframes the Role of Labeled Data:** Labeled data is typically viewed as narrowing a model's capabilities to specific tasks. FLAN shows labeled data can *expand* capabilities by teaching instruction-following as a transferable skill. The Discussion (Section 6) states: *"task-specific training is complementary to general language modeling and motivates further research on generalist models."*

- **Bridges Two Paradigms:** Figure 2 illustrates how instruction tuning combines the reliability of supervised finetuning (the model learns explicit task mappings) with the flexibility of prompting (a single model handles many tasks via text instructions). This resolves a key tension in the field: do we build many specialized models or one general model? FLAN suggests we can have both.

- **Practical Accessibility:** Because the model learns to respond to natural language instructions rather than dataset-specific formats, non-technical users can interact with it without prompt engineering expertise. The FAQ (Appendix E) contrasts FLAN prompts with GPT-3 prompts: FLAN uses instructions like "Does the premise entail the hypothesis?" while GPT-3 requires prompts that mimic pretraining data formats.

**Incremental vs. Fundamental:** This is a **fundamental innovation**. While Sanh et al. (2021) and Mishra et al. (2021) explored similar ideas at smaller scales, FLAN demonstrates the approach at the scale where emergent capabilities appear (137B parameters), and rigorously evaluates zero-shot generalization to unseen task clusters rather than seen tasks or domains.

---

### 4.2 Scale-Dependent Emergence of Instruction Following (Fundamental Discovery)

**What Makes This Novel:** The ablation study in Figure 7 reveals a critical threshold effect: instruction tuning *improves* zero-shot performance on held-out tasks for models at 68B and 137B parameters, but *degrades* performance for models at 8B parameters and below. This is counterintuitive—one might expect any supervised signal to help, regardless of scale.

The authors propose an explanation grounded in model capacity: for small models, learning the ~40 instruction tuning tasks consumes all available capacity, causing overfitting to training tasks and worse generalization. For large models, instruction tuning fills some capacity but also teaches instruction-following as a skill, leaving remaining capacity for generalization to new tasks.

**Why This Matters:**

- **Challenges Universal Applicability Assumptions:** Many techniques in deep learning are assumed to scale gracefully—what works at small scale should work at large scale, or vice versa. This finding shows instruction tuning has a *minimum viable scale* around 100B parameters. Practitioners cannot apply this method to smaller models and expect benefits.

- **Evidence for Emergent Capabilities:** The scaling law provides empirical support for the theory that certain capabilities only emerge at sufficient model scale. Instruction-following is not a skill that gradually improves with size—it appears to "turn on" once the model is large enough to both memorize training tasks and retain generalization capacity.

- **Guides Resource Allocation:** For organizations considering instruction tuning, this finding prevents wasted compute on models too small to benefit. The 60-hour training time on 128 TPUv3 cores (Section 2.4) is only justified for models in the 100B+ parameter range.

**Incremental vs. Fundamental:** This is a **fundamental discovery** about language model behavior. Prior instruction-based work (Sanh et al., 2021 at 11B; Mishra et al., 2021 at 140M) did not systematically evaluate across scales, so this threshold effect was unknown. The finding has implications beyond instruction tuning—it suggests other capabilities may have similar scale thresholds.

---

### 4.3 Natural Language Instructions as the Critical Signal (Fundamental Innovation)

**What Makes This Novel:** The ablation study in Figure 8 and Table 3 (Appendix B.2) isolates what component of instruction tuning drives zero-shot generalization. Three finetuning setups were compared:

| Finetuning Format | Evaluation Format | Zero-Shot Performance (4-cluster avg.) |
|-------------------|-------------------|----------------------------------------|
| No template (input/output only) | Natural instructions | 37.3% |
| Dataset name (e.g., "[Translation: WMT'14]") | Natural instructions | 46.6% |
| Natural language instructions (FLAN) | Natural instructions | 55.2% |

The 17.9 percentage point gap between "no template" and FLAN demonstrates that multi-task finetuning alone is insufficient—the model must learn to interpret *natural language instructions* as commands. The dataset name ablation (46.6%) shows that even task identification helps, but natural language instructions provide substantially more signal.

**Why This Matters:**

- **Instructions as Learned Skill:** This proves the model is not just learning task patterns—it is learning to *parse and execute instructions* as a transferable capability. When evaluated on unseen tasks, the model can interpret novel instructions because it learned instruction semantics, not just task-specific mappings.

- **Validates Design Choice:** The decision to manually compose 10 templates per dataset (Section 2.1) is justified by this result. Simpler alternatives like dataset tags would not achieve the same zero-shot performance.

- **Distinguishes from T5-Style Prompts:** T5 uses dataset names as prefixes (e.g., "translate English to German:"), which works for seen tasks but fails in zero-shot settings. FLAN's natural language instructions enable genuine zero-shot generalization because they teach the model to understand instruction *meaning*, not just recognize task labels.

**Incremental vs. Fundamental:** This is a **fundamental innovation** in how we conceptualize model training. Prior work assumed multi-task learning benefits came from shared representations across tasks. FLAN shows the *instruction format itself* is the key—teaching models to respond to natural language commands is a distinct capability from learning task patterns.

---

### 4.4 Complementarity with Other Adaptation Methods (Important Finding)

**What Makes This Novel:** Instruction tuning is not positioned as replacing other adaptation methods but as *complementing* them. Section 4.4 shows few-shot exemplars combined with instruction tuning improve performance on all task clusters, with particular gains on tasks with complex output spaces (struct-to-text, translation, closed-book QA). Section 4.5 and Figure 10 demonstrate that instruction-tuned models respond better to prompt tuning (continuous soft prompts optimized via backpropagation), achieving over 10% improvement in low-resource settings (32 training examples).

**Why This Matters:**

- **Orthogonal Improvements:** Instruction tuning, few-shot prompting, and prompt tuning operate through different mechanisms—instruction tuning changes model weights to learn instruction-following, few-shot provides in-context examples, and prompt tuning optimizes continuous inputs. Their complementarity means practitioners can stack these methods for cumulative gains.

- **Reduced Prompt Sensitivity:** Figure 9 shows that few-shot FLAN has lower standard deviation among templates compared to zero-shot FLAN, indicating reduced sensitivity to prompt engineering. This makes the model more robust in production settings where prompt quality may vary.

- **Enables Efficient Adaptation:** The prompt tuning result (Section 4.5) shows instruction tuning creates a checkpoint more amenable to efficient adaptation. When only 32 labeled examples are available, prompt tuning on FLAN outperforms prompt tuning on the base model by substantial margins (e.g., 83.0% vs. 52.4% on RTE, Table 4).

**Incremental vs. Fundamental:** This is an **important finding** rather than a fundamental innovation. The complementarity itself is not surprising—different adaptation methods often stack. However, quantifying the gains and demonstrating reduced prompt sensitivity provides practical guidance for deployment.

---

### 4.5 Summary: What Is Truly New

| Contribution | Type | Prior Art | FLAN's Advance |
|--------------|------|-----------|----------------|
| Instruction tuning for zero-shot generalization | Fundamental | Multi-task learning (seen tasks), Prompting (no finetuning) | Labeled data trains generalist model for unseen tasks |
| Scale-dependent emergence | Fundamental | Scaling laws for pretraining | First demonstration of instruction tuning scale threshold |
| Natural language instructions as critical signal | Fundamental | Dataset tags (T5), No templates | Instructions teach transferable skill, not just task patterns |
| Complementarity with few-shot/prompt tuning | Important | Individual methods studied separately | Quantified stacked benefits and reduced prompt sensitivity |

The first three contributions represent fundamental shifts in how we understand language model training and capabilities. The fourth provides practical guidance for combining methods. Together, these insights establish instruction tuning not as an incremental improvement but as a new paradigm for building generalist language models.

## 5. Experimental Analysis

This section examines how the authors validated their claims about instruction tuning. We will walk through the evaluation methodology, analyze the quantitative results with specific numbers, assess the strength of the evidence, and examine ablation studies and failure cases. Understanding *how* the experiments were designed is as important as the results themselves—the held-out task cluster methodology is what makes the zero-shot claims credible.

### 5.1 Evaluation Methodology: Rigorous Zero-Shot Testing

**Task Clusters and Held-Out Evaluation Design**

The evaluation framework is built around the 12 task clusters shown in Figure 3 (Section 2.1). These clusters group datasets by underlying task type: Natural Language Inference (7 datasets), Reading Comprehension (5 datasets), Closed-Book QA (3 datasets), Translation (8 datasets), Commonsense Reasoning (4 datasets), Coreference Resolution (3 datasets), Struct-to-Text (4 datasets), Sentiment Analysis (4 datasets), Paraphrase Detection (4 datasets), Summarization (11 datasets), Reading Comprehension with Commonsense (2 datasets), and Miscellaneous (7 datasets).

The critical methodological choice is the **held-out task cluster evaluation** described in Section 2.2. A dataset $D$ is considered unseen at evaluation time only if *no datasets* from any task clusters that $D$ belongs to appeared during instruction tuning. For example, when evaluating natural language inference performance, the model was instruction-tuned on all clusters *except* the NLI cluster. This is more conservative than prior work that merely holds out individual datasets—FLAN holds out entire *task types*.

To evaluate on $c$ task clusters, the authors train $c$ separate FLAN checkpoints, where each checkpoint holds out a different task cluster. This means there is no single "FLAN model" evaluated on all tasks; each evaluation task cluster uses a different checkpoint that was never exposed to that cluster type during instruction tuning. When evaluating Reading Comprehension with Commonsense, both Reading Comprehension and Commonsense Reasoning clusters were dropped from instruction tuning. The paraphrase cluster is also dropped when evaluating NLI tasks and vice-versa, since these task types are closely related.

**Metrics and Template Averaging**

For each dataset, performance is reported as the **mean across all 10 instruction templates** (Section 3). This proxies the expected performance given a typical natural language instruction and reduces sensitivity to any single template phrasing. The authors also report results using the template with the best development set performance for direct comparison with GPT-3 baselines.

Metrics vary by task type:
- **Accuracy** for classification tasks (NLI, sentiment, multiple-choice QA)
- **F1 score** for DROP, MultiRC, and SQuAD v1/v2 (Table 2, Appendix A)
- **BLEU** for translation tasks (Table 1, Appendix A)
- **ROUGE** for struct-to-text tasks (Table 1, Appendix A)

**Baseline Models**

The paper compares FLAN against multiple baselines to isolate the effect of instruction tuning:

| Baseline | Parameters | Purpose |
|----------|------------|---------|
| LaMDA-PT (zero-shot) | 137B | Direct ablation of instruction tuning effect |
| LaMDA-PT (few-shot) | 137B | Tests if instruction tuning beats few-shot on same architecture |
| GPT-3 (zero-shot) | 175B | Industry standard for zero-shot comparison |
| GPT-3 (few-shot) | 175B | Tests if FLAN beats few-shot GPT-3 |
| GLaM (zero-shot/one-shot) | 64B/64E | Sparse mixture-of-experts baseline |
| Supervised models | Varies | T5, BERT, or task-specific models (Table 1, Table 2) |

For LaMDA-PT baselines, the authors use the same prompts as GPT-3 (from Brown et al., 2020) without additional prompt engineering, since LaMDA-PT is not suitable for natural instructions without instruction tuning (FAQ, Appendix E). This provides the most direct ablation of how much instruction tuning helps.

### 5.3 Ablation Studies: What Drives the Improvement?

**Number of Instruction Tuning Clusters (Section 4.1, Figure 6)**

Figure 6 examines how performance on held-out tasks changes as more task clusters are added to instruction tuning. The evaluation clusters are NLI, Closed-Book QA, and Commonsense Reasoning; the seven remaining clusters are used for instruction tuning.

| # Clusters | Avg Performance on Held-Out Clusters |
|------------|--------------------------------------|
| 1 | 49.9% |
| 2 | 55.0% |
| 3 | 59.3% |
| 4 | 59.2% |
| 5 | 60.8% |
| 6 | 61.9% |
| 7 | 63.5% |

Performance improves as additional clusters are added, with the exception of the sentiment analysis cluster which provides minimal added value. Notably, performance does not appear to saturate at seven clusters, implying further improvement may be possible with more task diversity. Appendix B.1 (Figure 11) shows that using more datasets per cluster improved performance by almost 10% on average, while using more templates per dataset had a negligible effect when there was sufficient dataset diversity.

**Scaling Laws (Section 4.2, Figure 7)**

Figure 7 reveals the scale-dependent emergence of instruction tuning benefits. The authors evaluate instruction tuning on models of size 422M, 2B, 8B, 68B, and 137B parameters:

| Model Size | Instruction Tuning Effect on Held-Out Tasks |
|------------|---------------------------------------------|
| 422M | Hurts performance |
| 2B | Hurts performance |
| 8B | Hurts performance |
| 68B | Substantially improves |
| 137B | Substantially improves |

For models at 8B parameters and below, instruction tuning *degrades* performance on held-out tasks compared to the untuned baseline. The authors propose an explanation: for small-scale models, learning the ~40 instruction tuning tasks fills the entire model capacity, causing overfitting to training tasks. For larger models, instruction tuning fills some capacity but also teaches instruction-following as a transferable skill, leaving remaining capacity for generalization.

This is a critical finding—instruction tuning is not universally beneficial and requires sufficient model scale (appearing to emerge around 100B parameters) to provide zero-shot generalization benefits.

**Role of Instructions (Section 4.3, Figure 8, Table 3)**

Figure 8 and Table 3 (Appendix B.2) isolate what component of instruction tuning drives zero-shot generalization. Three finetuning setups were compared:

| Finetuning Format | Evaluation Format | Zero-Shot Performance (4-cluster avg.) |
|-------------------|-------------------|----------------------------------------|
| No template (input/output only) | Natural instructions | 37.3% |
| Dataset name (e.g., "[Translation: WMT'14]") | Natural instructions | 46.6% |
| Dataset name | Dataset name | 47.0% |
| Natural language instructions (FLAN) | Natural instructions | 55.2% |

The 17.9 percentage point gap between "no template" and FLAN demonstrates that multi-task finetuning alone is insufficient—the model must learn to interpret *natural language instructions* as commands. Both ablation configurations performed substantially worse than FLAN, indicating that training with instructions is crucial for zero-shot performance on unseen tasks.

Section 4.4 explores combining instruction tuning with few-shot exemplars at inference time. Figure 9 shows few-shot exemplars improve performance on all task clusters compared with zero-shot FLAN:

Exemplars are especially effective for tasks with large/complex output spaces (struct-to-text, translation, closed-book QA), potentially because exemplars help the model better understand the output format. ### 5.4 Robustness Checks: Data Contamination Analysis

**Contamination Methodology (Appendix C)**

A reasonable concern is that examples from evaluation datasets may have appeared verbatim in the 2.49 trillion token pretraining corpus, inflating purported zero-shot performance. The authors follow the GPT-3 contamination analysis procedure (Brown et al., 2020): for each benchmark, they produce a "clean" version that removes all potentially leaked examples, defined as examples for which any n-gram (n varies per dataset but is roughly 13) overlapped with anything in the pretraining corpus.

**Contamination Results (Figure 12, Table 5)**

Figure 12 and Table 5 (Appendix C) summarize the contamination analysis. Key findings:

1. **Many datasets had substantial overlap:** DROP and SQuADv2 had almost total overlap with pretraining data (99.4% and 99.1% dirty, respectively). However, upon manual inspection, most overlapping n-grams were only in the *contexts* of examples (99.6% for DROP, 97.2% for SQuADv2), not in questions and answers. Overlaps never occurred in both question and answer for DROP, and only occurred for both in 5 of 11,153 SQuADv2 evaluation examples.

2. **No correlation between contamination and performance:** Across all datasets, there is no correlation that evaluating on clean data performs worse than evaluating on the total dataset. Figure 12 shows percent change in performance when evaluating on clean data versus all data—some datasets perform better on clean data, some worse, with no systematic pattern.

3. **High variance with small clean subsets:** When the percent of clean data is very small (e.g., ANLI R1 at 1.4% clean, ANLI R2 at 2.1% clean), there is higher variance in percent change due to fewer examples for computing clean performance.

4. **Two datasets marked for contamination:** Only ReCoRD and PIQA had clean subset performance lower than overall evaluation set performance by more than 1%. These results are marked with an asterisk (*) in Table 2. The authors note Brown et al. (2020) also reported substantial contamination rates for these datasets (61% dirty for ReCoRD, 29% for PIQA).

The authors conclude: *"we do not find any evidence that data overlap substantially impacted the results"* (Section 6, Discussion). However, they acknowledge the conservative nature of n-gram matching likely introduces false positives, and accurately detecting test contamination remains an open research area.

### 5.5 Failure Cases and Limitations

**Documented Failure Cases (Appendix F, Figure 22)**

Appendix F provides qualitative examples of FLAN failures:

1. **Simple rule-based tasks:** FLAN fails at returning the second word in a sentence. When asked "What is the second word in this sentence? Conan went to the store to buy some pencils," FLAN outputs "pencils," "Conan," "store," or "buy" instead of "went."

2. **Instruction misinterpretation:** When asked to answer a question in Danish ("What is Barack Obama's profession? Answer in Danish"), FLAN translates the *question* to Danish rather than answering it ("Hvad er Barack Obamas beskæftigelse?" instead of "Præsident").

3. **Context length limitation:** The 1,024 token input length prevents evaluation on most summarization tasks where mean input length exceeds this constraint (Section 3, Appendix A).

**Task Types Where Instruction Tuning Does Not Help**

As noted in Section 5.2, instruction tuning does not improve performance for tasks directly formulated as language modeling where instructions would be largely redundant:

> *"instruction tuning is very effective on tasks naturally verbalized as instructions (e.g., NLI, QA, translation, struct-to-text) and is less effective on tasks directly formulated as language modeling, where instructions would be largely redundant (e.g., commonsense reasoning and coreference resolution tasks that are formatted as finishing an incomplete sentence or paragraph)."* (Section 3)

For seven commonsense reasoning and coreference resolution tasks, FLAN only outperforms LaMDA-PT on three of seven tasks (Table 2).

**Scale Dependency**

The scaling law ablation (Figure 7) reveals instruction tuning is not beneficial for models below ~8B parameters—it actually hurts performance. This is a critical limitation for practitioners with smaller models.

**Other Limitations (Section 6, Discussion)**

The authors acknowledge several limitations:
- Subjectivity in assigning tasks to clusters (though they use accepted categorizations)
- Only relatively short instructions of typically a single sentence are explored
- Individual examples might have appeared in pretraining data (though contamination analysis suggests minimal impact)
- The 137B scale makes FLAN costly to serve
- Pretraining data was ~90% English, limiting cross-lingual capabilities

### 5.6 Assessment: Do the Experiments Support the Claims?

**Strengths of the Experimental Design**

1. **Conservative Evaluation:** The held-out task cluster methodology is more rigorous than holding out individual datasets. By ensuring no datasets from a task type appear during instruction tuning when evaluating that task type, the authors demonstrate genuine cross-task generalization rather than dataset memorization.

2. **Multiple Baselines:** Comparing against LaMDA-PT (same architecture, no instruction tuning), GPT-3 (industry standard), and GLaM (sparse MoE) isolates the effect of instruction tuning from model architecture differences.

3. **Template Averaging:** Reporting mean performance across 10 templates reduces sensitivity to prompt engineering and provides a more robust estimate of typical performance.

4. **Contamination Analysis:** The thorough data contamination check (Appendix C) addresses a critical validity threat for zero-shot claims.

5. **Ablation Studies:** The scaling law, number of clusters, and role of instructions ablations reveal *when* and *why* instruction tuning works, not just *that* it works.

**Weaknesses and Open Questions**

1. **No Single Checkpoint:** Because each evaluation task cluster uses a different checkpoint, there is no single FLAN model that demonstrates all capabilities simultaneously. This is methodologically sound for proving generalization but makes deployment more complex.

2. **Summarization Not Evaluated:** Despite including 11 summarization datasets in instruction tuning, summarization is not evaluated due to the 1,024 token input length limitation (Appendix A). This leaves a gap in understanding instruction tuning's effectiveness on long-context generation tasks.

3. **Limited Cross-Lingual Evaluation:** With ~90% English pretraining data and an English sentencepiece tokenizer, translation *from* English is weaker than translation *to* English. The paper does not explore non-English instruction tuning.

4. **Scale Threshold Uncertainty:** The scaling law shows instruction tuning hurts models ≤8B and helps models ≥68B, but the exact threshold is unclear. More granular scale evaluation (e.g., 16B, 32B) would clarify the emergence point.

5. **Template Quality Variation:** While template averaging reduces sensitivity, the best-dev-template performance often substantially exceeds average-template performance (e.g., CB: 64.1% avg vs. 83.9% best dev, Table 2). This suggests instruction quality matters significantly.

**Overall Assessment**

The experiments convincingly support the paper's central claim: instruction tuning substantially improves zero-shot performance on unseen task types for sufficiently large models. The held-out task cluster methodology, multiple baselines, contamination analysis, and comprehensive ablation studies provide strong evidence. The negative results (commonsense/coreference tasks, small model scales) are appropriately reported and actually strengthen the claims by defining clear boundary conditions.

The key contribution is not just that FLAN outperforms GPT-3 on 20 of 25 datasets, but that the ablation studies reveal *why*: task diversity matters more than template variety, natural language instructions are crucial (not just dataset tags), and the benefits emerge only at sufficient scale. These insights provide actionable guidance for practitioners beyond the specific FLAN checkpoint.

## 6. Limitations and Trade-offs

No method is universally applicable, and instruction tuning is no exception. This section examines the boundary conditions, assumptions, and trade-offs that define when instruction tuning is appropriate—and when it is not. Understanding these limitations is critical for practitioners considering whether to adopt this approach, as the paper itself identifies several scenarios where instruction tuning provides no benefit or may even degrade performance.

### 6.1 Scale Dependency: A Minimum Viable Model Size

**The Limitation:** Instruction tuning is not universally beneficial across model sizes. Figure 7 (Section 4.2) reveals a stark threshold effect: instruction tuning *improves* zero-shot performance on held-out tasks for models at 68B and 137B parameters, but *degrades* performance for models at 8B parameters and below.

| Model Size | Instruction Tuning Effect |
|------------|---------------------------|
| 422M | Hurts performance |
| 2B | Hurts performance |
| 8B | Hurts performance |
| 68B | Substantially improves |
| 137B | Substantially improves |

**Why This Matters:** This is not a minor caveat—it fundamentally restricts who can use this technique. Organizations without access to 100B+ parameter models cannot expect instruction tuning benefits. The authors propose an explanation: for small-scale models, learning the ~40 instruction tuning tasks consumes all available capacity, causing overfitting to training tasks. For larger models, instruction tuning fills some capacity but also teaches instruction-following as a transferable skill, leaving remaining capacity for generalization.

**Trade-off:** The computational cost of training and serving 100B+ parameter models is substantial. The paper notes FLAN's 137B scale "makes it costly to serve" (Section 6, Discussion). For practitioners with smaller models, instruction tuning may actively harm performance, requiring alternative adaptation strategies.

**Open Question:** The exact emergence threshold is unclear. The study evaluates 422M, 2B, 8B, 68B, and 137B—there is a 56B parameter gap between the largest model where instruction tuning hurts (8B) and the smallest where it helps (68B). More granular evaluation at 16B, 32B, and 48B would clarify where the transition occurs.

### 6.2 Task Type Limitations: When Instructions Are Redundant

**The Limitation:** Instruction tuning does not improve performance for tasks that are already naturally formulated as language modeling objectives. Section 3 states:

> *"instruction tuning is very effective on tasks naturally verbalized as instructions (e.g., NLI, QA, translation, struct-to-text) and is less effective on tasks directly formulated as language modeling, where instructions would be largely redundant (e.g., commonsense reasoning and coreference resolution tasks that are formatted as finishing an incomplete sentence or paragraph)."*

**Evidence:** For seven commonsense reasoning and coreference resolution tasks (Table 2, Appendix A), FLAN only outperforms LaMDA-PT on three of seven tasks:

| Dataset | LaMDA-PT Zero-Shot | FLAN Zero-Shot (avg) | Improvement? |
|---------|-------------------|----------------------|--------------|
| COPA | 90.0% | 90.6% | Minimal (+0.6%) |
| HellaSwag | 57.0% | 56.4% | No (−0.6%) |
| PIQA | 80.3% | 80.9% | Minimal (+0.6%) |
| StoryCloze | 79.5% | 92.2% | Yes (+12.7%) |
| Winogrande | 68.3% | 67.3% | No (−1.0%) |
| WSC273 | 81.0% | 80.8% | No (−0.2%) |
| ReCoRD | 87.8% | 67.8%* | No (−20.0%)* |

**Why This Happens:** When a task is already formatted as sentence completion (e.g., "Conan went to the store to buy some pencils. What is the second word?"), the pretrained language model already knows how to continue the text. Adding an instruction like "What is the second word in this sentence?" does not provide additional signal—the model was already trained to predict the next token. Instruction tuning adds value primarily when the task format differs from standard language modeling, such as answering questions or following commands.

**Trade-off:** Practitioners must assess whether their target tasks are naturally verbalizable as instructions. For sentence completion tasks, instruction tuning may provide no benefit despite the 60-hour training cost on 128 TPUv3 cores (Section 2.4).

### 6.3 Context Length Constraints: Summarization Not Evaluated

**The Limitation:** FLAN uses an input sequence length of 1,024 tokens during finetuning (Section 2.4). This prevents evaluation on most summarization tasks, despite 11 summarization datasets being included in the instruction tuning mixture.

**Evidence:** Appendix A states:

> *"Though we use a summarization task cluster during instruction tuning, we leave evaluation of summarization for future work, as the mean input of most summarization datasets exceeds FLAN's input length of 1024 tokens."*

**Why This Matters:** Summarization is a high-value application for instruction-tuned models. The inability to evaluate on this task cluster leaves a significant gap in understanding instruction tuning's effectiveness on long-context generation tasks. Many real-world documents (news articles, reports, emails) exceed 1,024 tokens, limiting FLAN's practical applicability.

**Trade-off:** Longer context lengths increase computational cost quadratically for transformer attention. The 1,024 token limit represents a practical engineering constraint, but it excludes important use cases. Future work would need to address this limitation through architectural changes (e.g., sparse attention, retrieval augmentation) or extended context training.

### 6.4 Cross-Lingual Limitations: English-Centric Training

**The Limitation:** FLAN's pretraining data was approximately 90% English (Section 2.4), and it uses an English SentencePiece tokenizer. This limits cross-lingual capabilities, particularly for translation *from* English into other languages.

**Evidence:** Table 1 (Appendix A) shows translation results where FLAN performs better translating *into* English than *from* English:

| Direction | FLAN Zero-Shot (best dev) | GPT-3 Few-Shot |
|-----------|---------------------------|----------------|
| FR→EN | 35.9 BLEU | 37.9 BLEU |
| DE→EN | 38.9 BLEU | 40.7 BLEU |
| RO→EN | 37.3 BLEU | 38.1 BLEU |
| EN→FR | 33.9 BLEU | 33.8 BLEU |
| EN→DE | 27.0 BLEU | 26.1 BLEU |
| EN→RO | 18.9 BLEU | 20.5 BLEU |

Section 3 notes: *"Translating from English into other languages, however, was relatively weaker, as might be expected given that FLAN uses an English sentencepiece tokenizer and that the majority of pretraining data is English."*

**Why This Matters:** For global applications requiring multilingual support, FLAN's English-centric training is a significant constraint. The model may struggle with non-English instructions or tasks requiring deep understanding of non-English languages.

**Trade-off:** Multilingual pretraining increases data collection and compute costs. The authors note cross-lingual experiments as future work (Section 6, Discussion), but practitioners needing multilingual capabilities today would need to explore alternative approaches.

### 6.5 Multiple Checkpoints: No Single Universal Model

**The Limitation:** There is no single FLAN checkpoint that demonstrates all capabilities simultaneously. Because each evaluation task cluster uses a different checkpoint (trained with that cluster held out), the reported results come from multiple models.

**Evidence:** Section 2.2 states:

> *"Hence, to evaluate zero-shot FLAN on c task clusters, we instruction tune c models, where each model holds out a different task cluster for evaluation."*

**Why This Matters:** This is methodologically sound for proving generalization—it ensures the model was never exposed to the task type during training. However, it complicates deployment. A practitioner cannot download one FLAN checkpoint and expect it to handle all task types with guaranteed zero-shot performance. Each task cluster would ideally use its corresponding held-out checkpoint.

**Trade-off:** The held-out cluster design strengthens the scientific claim (genuine cross-task generalization) but weakens practical utility (no single deployable model). In practice, one might train a single FLAN model on all tasks, but the paper does not evaluate whether this sacrifices zero-shot performance on any individual task type.

### 6.6 Instruction Quality Sensitivity: Template Variation Matters

**The Limitation:** While template averaging reduces sensitivity to any single instruction phrasing, the best-development-template performance often substantially exceeds average-template performance, indicating instruction quality matters significantly.

**Evidence:** Table 2 (Appendix A) shows notable gaps between average and best-dev template performance:

| Dataset | Average Template | Best Dev Template | Gap |
|---------|-----------------|-------------------|-----|
| CB (NLI) | 64.1% | 83.9% | +19.8% |
| RTE (NLI) | 78.3% | 84.1% | +5.8% |
| BoolQ | 80.2% | 82.9% | +2.7% |
| ARC-challenge | 61.7% | 63.1% | +1.4% |

**Why This Matters:** The 19.8 percentage point gap on CB suggests that instruction phrasing can dramatically affect performance. While the paper's template averaging approach provides a robust estimate of typical performance, practitioners deploying FLAN would need to invest in prompt engineering to achieve best-case results.

**Trade-off:** The paper's goal is to reduce prompt engineering burden, yet the template variation results suggest some engineering is still valuable. The few-shot FLAN results (Section 4.4) show reduced template standard deviation, indicating exemplars can mitigate this sensitivity—but at the cost of additional tokens and complexity.

### 6.7 Subjectivity in Task Clustering

**The Limitation:** The held-out task cluster methodology depends on how tasks are grouped into clusters. The authors acknowledge:

> *"As for limitations of our study, there is a degree of subjectivity in assigning tasks to clusters (though we try to use accepted categorizations in the literature)."* (Section 6, Discussion)

**Why This Matters:** If task clusters are not truly distinct—if, for example, paraphrase detection and NLI share substantial underlying capabilities—then holding out one cluster may not genuinely test zero-shot generalization. The paper does drop the paraphrase cluster when evaluating NLI and vice-versa (Section 2.2, footnote 1), acknowledging this overlap.

**Trade-off:** More granular clustering (more, smaller clusters) would provide stricter evaluation but requires training more checkpoints. Coarser clustering (fewer, larger clusters) is more practical but may allow task type leakage. The 12-cluster design represents a middle ground but is not definitively optimal.

### 6.8 Short Instructions Only

**The Limitation:** The paper explores only relatively short instructions, typically a single sentence. Section 6 (Discussion) notes:

> *"we only explore the use of relatively short instructions of typically a single sentence (c.f. detailed instructions given to crowd-workers)."*

**Why This Matters:** Real-world instructions may be complex, multi-step, or require nuanced constraints (e.g., "Summarize this article in 3 bullet points, using simple language suitable for a 10-year-old, and highlight any controversial claims"). FLAN's training on short instructions may not generalize to such complex directives.

**Open Question:** Would instruction tuning on longer, more detailed instructions improve performance on complex tasks? This remains unexplored.

### 6.9 Data Contamination: Residual Uncertainty

**The Limitation:** Despite thorough contamination analysis (Appendix C), residual uncertainty remains. The authors note:

> *"the conservative nature of our n-gram matching procedure likely introduces additional false positives; there are no guarantees that the clean subset is drawn from the same distribution as the overall subset; and, accurately detecting test contamination is a relatively new research area without established best practices."* (Appendix C)

**Evidence:** Some datasets had extremely high contamination rates:

| Dataset | % Clean Data | Performance Change (Clean vs. All) |
|---------|--------------|-----------------------------------|
| DROP | 0.6% | +47.4% (better on clean) |
| SQuADv2 | 0.9% | −6.2% (worse on clean) |
| ANLI R1 | 1.4% | +18.8% (better on clean) |
| ANLI R2 | 2.1% | −11.2% (worse on clean) |

**Why This Matters:** When 99%+ of evaluation examples overlap with pretraining data, the "zero-shot" claim becomes questionable. The authors argue that for DROP and SQuADv2, overlaps occurred in contexts but not questions/answers, so the model gains only background information. However, this interpretation requires trust in manual inspection.

**Trade-off:** Perfect contamination detection is impossible without knowing the exact pretraining corpus. The paper's analysis follows GPT-3's methodology, providing reasonable confidence but not certainty.

### 6.10 Summary: When to Use Instruction Tuning

| Condition | Recommendation |
|-----------|----------------|
| Model size < 8B parameters | **Do not use** — instruction tuning hurts performance |
| Model size ≥ 68B parameters | **Consider using** — benefits emerge at this scale |
| Tasks are sentence completion | **Limited benefit** — instructions are redundant |
| Tasks are instruction-based (QA, NLI, translation) | **Strong benefit** — natural fit for instruction tuning |
| Need single deployable checkpoint | **Caution** — paper uses multiple held-out checkpoints |
| Require long-context (>1024 tokens) | **Not supported** — summarization not evaluated |
| Need multilingual support | **Limited** — English-centric training |
| Have resources for prompt engineering | **Combine with few-shot** — reduces template sensitivity |

**Final Assessment:** Instruction tuning is a powerful technique with clear boundary conditions. It is not a universal solution but rather a tool that works exceptionally well within specific constraints: large models, instruction-verbalizable tasks, and English-language applications. Practitioners should evaluate their use case against these limitations before adopting the approach. The paper's transparent reporting of negative results (small models, commonsense tasks) strengthens its credibility and provides actionable guidance for the research community.

## 7. Implications and Future Directions

This section examines how FLAN reshapes the research landscape, what questions it opens for future investigation, and how practitioners should think about deploying instruction-tuned models. The implications extend beyond the specific 137B checkpoint—the methodological insights about scale, task diversity, and instruction format have broader relevance for how we build and deploy language models.

### 7.1 Landscape Impact: Reframing the Role of Labeled Data

**From Specialist Training to Generalist Capability**

FLAN fundamentally challenges a prevailing assumption in NLP: that labeled data inherently narrows a model's capabilities to specific tasks. The Discussion (Section 6) states: *"Although one might expect labeled data to have the most natural role in improving specialist models, instruction tuning demonstrates how labeled data can be used to help large language models perform many, unseen tasks."*

This reframing has three concrete implications for the field:

1. **Labeled Data as Capability Expansion:** Prior to FLAN, the dominant paradigm treated labeled data as creating specialist models (one checkpoint per task, as with BERT finetuning). FLAN shows labeled data can *expand* capabilities by teaching instruction-following as a transferable skill. This shifts how organizations should think about investing in labeled datasets—they become infrastructure for generalist models, not just task-specific solutions.

2. **Bridging the Pretrain–Finetune Divide:** Figure 2 illustrates how instruction tuning occupies a middle ground between traditional finetuning (many specialized models) and prompting (one general model with weak zero-shot). The paper demonstrates these paradigms are not mutually exclusive—supervised signals can enhance a generalist model's flexibility rather than constraining it.

3. **Scale-Dependent Method Selection:** The scaling law in Figure 7 establishes that instruction tuning is not universally applicable. This introduces a new decision boundary for practitioners: model size now determines *which adaptation methods are viable*. Below ~8B parameters, instruction tuning actively harms performance; above ~68B, it substantially helps. This creates a tiered landscape where different model sizes require different adaptation strategies.

**Positioning Relative to Concurrent Work**

The Related Work (Section 5) notes several contemporaneous efforts that validate instruction tuning as an emerging paradigm:

| Work | Model Size | Key Difference from FLAN |
|------|------------|--------------------------|
| Sanh et al. (2021) | T5-11B | Smaller scale; similar instruction-tuning approach |
| Mishra et al. (2021) | BART-140M | Focus on few-shot with instructions, not zero-shot |
| InstructGPT (Ouyang et al., 2022) | GPT-3 scale | Uses reinforcement learning from human feedback in addition to finetuning |
| FLAN (this work) | 137B | First demonstration at scale where emergence occurs; rigorous held-out task cluster evaluation |

FLAN's contribution is establishing instruction tuning at the scale where emergent capabilities appear, with methodology (held-out task clusters) that proves genuine cross-task generalization rather than task interference benefits.

### 7.2 Research Directions Enabled by This Work

The paper explicitly identifies several open questions and implicitly suggests others through its findings and limitations.

**Explicitly Stated Future Work (Section 6, Discussion)**

The authors directly propose four research directions:

1. **Gathering More Task Clusters:** *"Future work on instruction tuning could include gathering/generating even more task clusters for finetuning."* Figure 6 shows performance does not saturate at seven instruction tuning clusters (63.5% at seven clusters vs. 49.9% at one cluster), implying diminishing returns have not been reached. This suggests research on:
   - Automated template generation to reduce manual composition cost (10 templates × 62 datasets = 620 templates manually written)
   - Identifying which task clusters provide the most transferable instruction-following skills
   - Understanding whether certain task combinations interfere with each other

2. **Cross-Lingual Experiments:** The paper notes FLAN's pretraining was ~90% English with an English SentencePiece tokenizer (Section 2.4), and translation *from* English was weaker than translation *to* English (Table 1, Appendix A). Cross-lingual instruction tuning would investigate:
   - Whether instruction-following transfers across languages
   - Whether non-English instructions require separate tuning
   - How multilingual pretraining interacts with instruction tuning

3. **Using FLAN for Data Generation:** *"Using FLAN to generate data for training downstream classifiers"* is proposed as future work. This creates a bootstrap opportunity: an instruction-tuned model could generate labeled training data for tasks where labeled data is scarce. The qualitative examples in Appendix F (Figures 17–21) show FLAN can generate utterances, search queries, and stories—capabilities directly applicable to data augmentation.

4. **Bias and Fairness Improvements:** *"Using finetuning to improve model behavior with respect to bias and fairness"* is noted, referencing Solaiman & Dennison (2021). Instruction tuning could potentially steer models away from harmful outputs by including fairness-oriented tasks in the instruction mixture, though this requires careful evaluation to avoid unintended consequences.

**Implicitly Suggested Research Directions**

The paper's findings and limitations point to additional research questions:

1. **Understanding the Scale Threshold:** Figure 7 evaluates 422M, 2B, 8B, 68B, and 137B parameters—a 56B gap between the largest model where instruction tuning hurts (8B) and the smallest where it helps (68B). The FAQ (Appendix E) acknowledges: *"Ideally, we would have performed the Figure 7 ablation with cross-validation instead of a single split, which likely would have smoothed out that artifact."* Research at 16B, 32B, and 48B parameters would clarify:
   - The exact emergence point for instruction-following capabilities
   - Whether the threshold is sharp or gradual
   - Whether different task types have different scale requirements

2. **Context Length Extensions:** Appendix A states summarization was not evaluated because *"the mean input of most summarization datasets exceeds FLAN's input length of 1024 tokens."* This limitation excludes a high-value application area. Research directions include:
   - Instruction tuning with extended context (2048, 4096, or more tokens)
   - Whether instruction-following degrades with longer contexts
   - Architectural changes (sparse attention, retrieval augmentation) that enable longer contexts without prohibitive compute costs

3. **Instruction Complexity:** Section 6 notes the paper explores *"only relatively short instructions of typically a single sentence."* Real-world instructions are often multi-step with constraints (e.g., "Summarize in 3 bullet points using simple language"). Research could investigate:
   - Whether instruction tuning on complex, multi-step instructions improves performance on such tasks
   - Whether there is a maximum instruction complexity the model can handle
   - How instruction complexity interacts with model scale

4. **Task Cluster Boundaries:** The Discussion acknowledges *"a degree of subjectivity in assigning tasks to clusters."* The paper drops the paraphrase cluster when evaluating NLI and vice-versa (Section 2.2, footnote 1), acknowledging overlap. Research could:
   - Develop objective metrics for task cluster similarity
   - Test whether finer-grained clustering changes the generalization conclusions
   - Investigate whether certain cluster combinations provide synergistic benefits

5. **Instruction Quality Optimization:** Table 2 (Appendix A) shows substantial gaps between average-template and best-dev-template performance (e.g., CB: 64.1% vs. 83.9%, a 19.8 percentage point difference). This suggests instruction quality matters significantly. Research could:
   - Develop methods for automatically identifying high-quality instruction templates
   - Study what makes an instruction effective (specificity, clarity, task framing)
   - Create instruction optimization procedures that don't require held-out development sets

### 7.3 Practical Applications and Downstream Use Cases

**Direct Deployment Scenarios**

Based on FLAN's demonstrated capabilities, several practical applications become feasible:

1. **Zero-Shot Task Automation:** Organizations can deploy instruction-tuned models for tasks without collecting task-specific training data. The paper shows FLAN outperforms zero-shot GPT-3 on 20 of 25 datasets (Abstract), including high-value tasks like:
   - **Natural Language Inference:** Determining whether documents contradict each other (RTE: 78.3% vs. GPT-3's 70.8%)
   - **Reading Comprehension:** Extracting answers from provided text (BoolQ: 80.2% vs. GPT-3's 80.0%)
   - **Translation:** Converting text between languages (EN→FR: 32.9 BLEU vs. GPT-3's 32.6 BLEU)

2. **Prompt Engineering Reduction:** Figure 9 shows few-shot FLAN has lower standard deviation among templates compared to zero-shot FLAN, indicating reduced sensitivity to prompt engineering. For production systems where prompt quality may vary, this stability reduces operational risk.

3. **Low-Resource Adaptation:** Section 4.5 and Figure 10 demonstrate that instruction-tuned models respond better to prompt tuning. With only 32 training examples, prompt tuning on FLAN achieves more than 10% improvement over prompt tuning on the base model (e.g., RTE: 83.0% vs. 52.4%). This enables efficient adaptation when labeled data is scarce but not absent.

4. **Data Generation for Training:** Appendix F (Figure 17) shows FLAN can generate utterances consistent with a given intent ("Generate utterances with the intent 'get COVID vaccine'"). This capability supports:
   - Training data augmentation for downstream classifiers
   - Synthetic data generation for privacy-sensitive applications
   - Rapid prototyping of task-specific datasets

**Integration Patterns**

The paper suggests several ways to integrate instruction tuning into existing workflows:

| Use Case | Recommended Approach | Rationale |
|----------|---------------------|-----------|
| Zero-shot with no labeled data | Use FLAN directly with natural instructions | Section 3 shows FLAN outperforms zero-shot GPT-3 on most tasks |
| Few-shot with some examples | Combine FLAN with few-shot exemplars (Section 4.4) | Figure 9 shows complementary gains, especially for complex output spaces |
| Low-resource supervised (32–100 examples) | Prompt tuning on FLAN checkpoint (Section 4.5) | Figure 10 shows 10%+ improvement over base model prompt tuning |
| High-resource supervised (1000+ examples) | Traditional finetuning may suffice | Instruction tuning benefits diminish when ample task-specific data exists |
| Model size < 8B parameters | Do not use instruction tuning | Figure 7 shows instruction tuning hurts performance at this scale |

**Cost-Benefit Considerations**

The Environmental Considerations section notes instruction tuning gradient steps are *"less than 2% of the number of pretraining steps,"* with estimated additional energy cost comparatively smaller than pretraining (451 MWh, 26 tCO2e for pretraining). However, the Discussion acknowledges *"the scale of FLAN 137B makes it costly to serve."*

Practitioners should weigh:
- **Training Cost:** ~60 hours on 128 TPUv3 cores (Section 2.4) for instruction tuning
- **Serving Cost:** 137B parameter models require substantial inference infrastructure
- **Alternative Costs:** Few-shot GPT-3 requires more tokens per query (exemplars increase token count); traditional finetuning requires labeled data collection

For organizations already operating 100B+ parameter models, instruction tuning represents a relatively low-cost enhancement. For organizations without such infrastructure, the serving costs may outweigh benefits unless zero-shot capabilities provide unique value.

### 7.4 Reproducibility and Integration Guidance

**When to Prefer Instruction Tuning Over Alternatives**

Based on the paper's findings, instruction tuning is preferable when:

1. **Model Scale ≥ 68B Parameters:** Figure 7 establishes this as the approximate threshold where benefits emerge. Below this scale, instruction tuning actively degrades zero-shot performance.

2. **Tasks Are Naturally Verbalizable as Instructions:** Section 3 states instruction tuning is *"very effective on tasks naturally verbalized as instructions (e.g., NLI, QA, translation, struct-to-text) and is less effective on tasks directly formulated as language modeling."* If your task is sentence completion (e.g., HellaSwag, Winogrande), instruction tuning provides minimal benefit (Table 2 shows FLAN underperforms LaMDA-PT on several commonsense tasks).

3. **Zero-Shot Capability Is Required:** If you cannot provide few-shot exemplars at inference time (due to token limits, latency constraints, or user experience requirements), instruction tuning provides substantial zero-shot improvements over untuned models.

4. **Task Diversity Is High:** If you need a single model to handle many different task types, instruction tuning enables cross-task generalization. Figure 6 shows performance improves with more task clusters, suggesting benefits compound with task diversity.

**When to Use Alternative Approaches**

Instruction tuning is not optimal when:

1. **Model Scale < 8B Parameters:** Figure 7 shows instruction tuning hurts performance. Use traditional finetuning or prompting instead.

2. **Single Task with Ample Labeled Data:** If you have thousands of labeled examples for one task, traditional supervised finetuning likely provides better performance with less complexity.

3. **Tasks Are Sentence Completion Format:** For tasks like HellaSwag or Winogrande where the pretrained model already knows how to continue text, instructions are redundant (Section 3).

4. **Long-Context Requirements (>1024 tokens):** Appendix A notes summarization was not evaluated due to input length constraints. If your application requires longer contexts, instruction tuning as described in this paper is not suitable.

5. **Multilingual Requirements:** With ~90% English pretraining data and an English tokenizer, cross-lingual capabilities are limited (Section 3, Translation results).

**Reproducibility Considerations**

The paper provides several resources for reproducibility:

- **Code Availability:** *"Source code for loading the instruction tuning dataset used for FLAN is publicly available at https://github.com/google-research/flan"* (Section 1)
- **Dataset Details:** Appendix G provides complete dataset listings, example inputs/outputs, and template information
- **Training Hyperparameters:** Section 2.4 specifies batch size (8,192 tokens), learning rate ($3 \times 10^{-5}$), steps (30,000), sequence lengths (1024 input, 256 target)
- **Evaluation Methodology:** Section 2.2 describes the held-out task cluster approach in detail

However, full reproducibility requires:
- Access to a 137B parameter pretrained model (LaMDA-PT is not publicly available)
- Substantial compute resources (~60 hours on 128 TPUv3 cores)
- The 62 datasets used for instruction tuning (all publicly available via TensorFlow Datasets per Section 2.1)

For researchers without access to 100B+ parameter models, the paper's methodological insights (scale dependency, task diversity importance, instruction format criticality) remain applicable even if the exact FLAN checkpoint cannot be reproduced.

**Integration Checklist for Practitioners**

Before adopting instruction tuning:

- [ ] Model size ≥ 68B parameters (Figure 7 threshold)
- [ ] Tasks are instruction-verbalizable, not sentence completion (Section 3)
- [ ] Zero-shot or low-shot capability is required (Section 4.4, 4.5)
- [ ] Context length ≤ 1024 tokens (Section 2.4 limitation)
- [ ] English-language focus acceptable (Section 2.4, ~90% English pretraining)
- [ ] Compute budget supports 137B parameter serving (Section 6, Discussion)
- [ ] Task diversity justifies generalist approach (Figure 6 shows benefits scale with task clusters)

If any of these conditions are not met, alternative approaches (traditional finetuning, few-shot prompting, or smaller-scale adaptation methods) may be more appropriate.

### 7.5 Summary: The Instruction Tuning Paradigm

FLAN establishes instruction tuning as a viable third paradigm alongside pretrain–finetune and prompting. The key insights that will shape future work are:

1. **Labeled data can train generalists, not just specialists**—when structured as diverse natural language instructions
2. **Capabilities emerge at scale**—instruction tuning requires ~100B+ parameters to provide benefits
3. **Instruction format matters**—natural language instructions outperform dataset tags or no templates by 17.9 percentage points (Figure 8)
4. **Methods are complementary**—instruction tuning stacks with few-shot exemplars and prompt tuning for cumulative gains

The paper's transparent reporting of negative results (small models, commonsense tasks, summarization limitations) provides as much value as the positive findings, establishing clear boundary conditions for when instruction tuning is appropriate. Future work will likely extend these findings to larger scales, more task clusters, multilingual settings, and more complex instruction formats—but the foundational insight that instruction-following is a learnable, transferable skill at scale will remain central to how we build and deploy language models.
