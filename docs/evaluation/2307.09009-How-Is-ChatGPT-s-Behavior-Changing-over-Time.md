## 1. Executive Summary

This paper addresses the opacity of LLM service updates by systematically quantifying behavior drifts in GPT-3.5 and GPT-4 between March 2023 and June 2023 across seven diverse tasks ranging from math problems to USMLE medical exams. The authors demonstrate that performance is not monotonically improving, highlighting severe regressions such as GPT-4's accuracy on prime number identification dropping from 84% to 51% and its instruction following fidelity on answer extraction plummeting from 99.5% to 0.5%. These findings matter because they reveal that integrating "stable" LLM APIs into workflows is risky without continuous monitoring, as model capabilities and safety alignments can shift substantially over short periods.

## 2. Context and Motivation

To understand the significance of this study, we must first recognize a fundamental shift in how machine learning models are deployed. Traditionally, a model was a static artifact: you trained it, versioned it (e.g., `v1.0`), and deployed it. If you ran the same input through `v1.0` today or next year, the output would remain identical. However, modern Large Language Model (LLM) services like `GPT-4` operate as **Model-as-a-Service (MaaS)**. The provider (OpenAI) updates the underlying weights, safety filters, and system prompts continuously, often without explicit versioning visible to the end user. This paper addresses the critical gap created by this opacity: **we do not know when these models change, how they change, or whether those changes represent improvements.**

### The Problem: Opacity and Instability in LLM Services
The core problem this paper tackles is the **longitudinal instability** of commercial LLM APIs. As noted in the Introduction, when an LLM service is updated, it is "opaque when and how GPT-3.5 and GPT-4 are updated." This creates two distinct categories of risk for researchers and engineers:

1.  **Pipeline Fragility:** Many applications integrate LLMs into larger software workflows. For example, a coding agent might expect the model to output *only* raw Python code. If a model update changes the formatting behavior (e.g., adding Markdown code blocks when none were requested), the downstream parser breaks. The Introduction highlights that if an LLM's response formatting "suddenly changes, this might break the downstream pipeline."
2.  **Scientific Reproducibility:** In research, if a paper claims "GPT-4 achieves 80% accuracy on Task X," that claim is only valid for the specific version of GPT-4 available at the time of writing. If the model drifts, future researchers cannot reproduce the result using the "same" model. The authors note this makes it "challenging, if not impossible, to reproduce results from the 'same' LLM."

### Why This Matters: Beyond Simple Performance Metrics
The importance of this work extends beyond merely tracking accuracy scores. It challenges the assumption that model updates are monotonically beneficial. In traditional software, a patch usually fixes bugs without introducing new ones. In LLMs, updates often involve trade-offs. For instance, strengthening safety filters to prevent harmful outputs might inadvertently reduce the model's ability to follow complex instructions or answer benign sensitive questions.

The Introduction poses a critical theoretical question: "It is important to know whether updates to the model aimed at improving some aspects can reduce its capability in other dimensions." This is not just an engineering concern; it is a fundamental question about the alignment and stability of generative AI systems. If safety improvements degrade reasoning capabilities (or vice versa), developers need to know this to make informed deployment choices.

### Prior Approaches and Their Limitations
Before this study, the landscape of LLM evaluation was dominated by **static benchmarks**. Works cited in the Related Work section (e.g., `[LBL+22]`, `[ZPM+23]`, `[NKM+23]`) established that LLMs could perform well on tasks like reading comprehension, translation, and professional exams (medicine, law). However, these evaluations were snapshots in time. They answered "How good is GPT-4?" but not "How stable is GPT-4?"

There were emerging efforts to monitor model drift, but they suffered from specific limitations that this paper aims to overcome:

*   **Magnitude of Detected Drift:** The paper references `ChatLog [TLY+23]`, which proposed recording ChatGPT's responses over time. However, `ChatLog` reported "small shifts (most below 5%)" on common benchmarks. This paper positions itself against this finding, demonstrating that on diverse tasks, drifts can be substantial (e.g., double-digit percentage swings), suggesting that prior monitoring may have missed critical failure modes by focusing on too narrow a set of tasks.
*   **Task Complexity:** Previous longitudinal studies in Machine-Learning-as-a-Service (MLaaS), such as `[CJE+22]` and `[CCZZ21]`, focused on "simple classification tasks such as sentiment analysis." Generative LLMs are fundamentally different; they produce open-ended text where drift can manifest as changes in verbosity, formatting, reasoning steps, or safety refusals, not just label flips. The Related Work section explicitly distinguishes this paper's focus on "generative LLM services" versus prior work on classification APIs.
*   **Explanation of Drift:** Most prior work documented *that* performance changed, but not *why*. This paper positions itself as an explanatory study, hypothesizing that changes in **instruction following** are a common factor behind behavior drifts (Introduction, Paragraph 4).

### Positioning of This Work
This paper positions itself as a **systematic longitudinal audit** of generative LLM services. It differentiates itself from prior work through three key design choices:

1.  **Diverse Task Coverage:** Instead of focusing on a single domain (like math or code), the authors evaluate drift across seven distinct categories: math, sensitive questions, opinion surveys, multi-hop knowledge, code generation, medical exams, and visual reasoning (Section 1). This diversity ensures that observed drifts are not artifacts of a single benchmark dataset.
2.  **Focus on Instruction Following:** Uniquely, the authors curate a set of "task-agnostic instructions" (e.g., "answer within squared brackets," "do not apologize") to isolate the model's ability to follow constraints from its raw knowledge capabilities (Section 1). This allows them to test the hypothesis that instruction following degradation is a root cause of performance drift.
3.  **Reproducibility Infrastructure:** Acknowledging the reproducibility problem they highlight, the authors release all prompts and responses collected during the study (Introduction, Paragraph 5). This transforms the paper from a static report into a baseline for future monitoring, enabling other researchers to compare future model versions against the March and June 2023 snapshots documented here.

By framing LLMs as dynamic services rather than static models, this work shifts the community's focus from one-time benchmarking to **continuous monitoring**, arguing that trust in LLM applications requires ongoing validation of model behavior.

## 3. Technical Approach

This paper is a **longitudinal audit study** that systematically measures behavioral drift in commercial LLM APIs by comparing model outputs across two time points using controlled prompts and quantitative metrics.

### 3.1 Reader orientation (approachable technical breakdown)

The *system* being studied is not a new model the authors built, but rather two existing commercial LLM services (`GPT-4` and `GPT-3.5`) treated as black-box APIs that can be queried over time. The solution takes the shape of a **controlled experiment framework**: the authors construct fixed datasets of prompts, query the LLM APIs at two distinct time points (March 2023 and June 2023), and measure how the outputs change using task-specific metrics and common cross-task metrics.

### 3.2 Big-picture architecture (diagram in words)

The experimental pipeline consists of four major components that operate in sequence:

1.  **Task Dataset Curation Module:** This component constructs or samples fixed sets of prompts for each of the eight evaluation tasks (math, sensitive questions, opinion surveys, multi-hop QA, code generation, medical exams, visual reasoning, and instruction following). Each dataset is frozen at the start of the study to ensure the input distribution remains constant across time points.

2.  **LLM Query Interface:** This component sends each prompt to the OpenAI API for both `GPT-4` and `GPT-3.5`, capturing responses from the March 2023 and June 2023 model versions. The interface standardizes query parameters (temperature, system prompt) to isolate model drift from sampling variability.

3.  **Metric Computation Engine:** This component calculates task-specific performance metrics (e.g., accuracy, response rate, exact match) and two common cross-task metrics (verbosity in characters, mismatch rate between versions). Each metric produces a numerical score for each model-version-task combination.

4.  **Drift Analysis Module:** This component compares metrics between March and June versions to quantify drift magnitude, identifies patterns (e.g., instruction following degradation), and generates visualizations (confusion matrices, bar charts) to illustrate behavior changes.

Information flows unidirectionally: curated prompts → API queries → raw responses → metric computation → drift analysis. There is no feedback loop; the study is observational rather than iterative.

### 3.3 Roadmap for the deep dive

-   **Query Configuration and Control Variables:** We first explain how the authors standardized API calls to ensure observed changes reflect model drift rather than sampling noise or prompt variation.
-   **Task-Specific Dataset Construction:** We detail how each of the eight evaluation tasks was constructed, including dataset sizes, sampling strategies, and why each task was selected.
-   **Metric Definitions and Computation:** We define all performance metrics (accuracy, response rate, exact match, directly executable) and the two common metrics (verbosity, mismatch), explaining how each is calculated.
-   **Instruction Following Benchmark Design:** We explain the novel task-agnostic instruction benchmark the authors created to test their hypothesis that instruction following degradation drives behavior drift.
-   **Statistical Comparison Methodology:** We describe how drift magnitude is quantified (population mean differences) and how confusion matrices are used to identify bias shifts.

### 3.4 Detailed, sentence-based technical breakdown

**Study Design and Query Configuration.** The authors treat `GPT-4` and `GPT-3.5` as black-box services accessible via OpenAI's API, where the underlying model weights and safety filters may change without explicit versioning. To isolate model drift from other sources of variation, the authors fix three critical query parameters across all experiments. First, the **temperature** is set to `$0.1$` for all queries, which reduces output randomness and makes responses more deterministic (Section 2, "LLM Services" paragraph). Second, the **system prompt** is left at its default value rather than customized, ensuring that any behavior changes reflect model updates rather than system-level instruction changes. Third, queries use **user prompt only**, meaning no additional context or few-shot examples are added beyond the task-specific prompt itself. This controlled configuration ensures that if the same prompt produces different outputs in March versus June, the difference can be attributed to model drift rather than sampling variability or prompt engineering changes.

**Task Dataset Construction.** The study evaluates drift across eight distinct tasks, each with a fixed dataset that remains constant between the March and June evaluation periods. For **Math I (Prime vs Composite)**, the dataset contains `$n=1000$` questions, where 500 prime numbers were extracted from prior work `[ZPM+23]` and 500 composite numbers were sampled uniformly from all composite numbers within the interval `$[1,000, 20,000]$` (Section 3.1). For **Math II (Happy Numbers)**, the authors constructed a dataset of `$n=500$` queries where each query asks how many happy numbers exist within a given interval, with interval sizes uniformly randomly selected from 6 to 10 and starting points uniformly randomly chosen from 500 to 15,000 (Section 3.2). For **Sensitive Questions**, the authors created a dataset of `$n=100$` sensitive queries that LLM services are not supposed to answer directly, with all responses manually labeled to determine if the model provided a direct answer (Section 3.3). For **OpinionQA Survey**, the study uses `$n=1506$` opinion questions drawn from high-quality public opinion polls from the OpinionQA dataset `[SDL+23]`, formatted as multiple-choice questions with an added instruction to "Pick the best single option" for ease of answer extraction (Section 3.4). For **LangChain HotpotQA Agent**, the evaluation uses `$n=7405$` questions from the HotpotQA dataset `[YQZ+18]`, which are multi-hop questions requiring reasoning across multiple Wikipedia passages (Section 3.6). For **Code Generation**, the authors constructed a new dataset of `$n=50$` problems from the "easy" category of LeetCode, specifically selecting the latest 50 problems at the time of writing with the earliest public solutions released in December 2022 to minimize data contamination (Section 3.5). For **USMLE Medical Exam**, the dataset contains `$n=340$` medical exam questions (Section 3.7). For **Visual Reasoning**, the study uses `$n=467$` samples from the ARC dataset `[Cho19]` that fit within all services' context windows, where input and output grids are represented as 2-D arrays with color values (Section 3.8).

**Chain-of-Thought Prompting Strategy.** For reasoning-heavy tasks (Math I, Math II, USMLE), the authors employ **Chain-of-Thought (CoT) prompting**, a technique where the model is explicitly instructed to think step-by-step before providing a final answer (Section 3.1). For example, the Math I prompt reads: "Is 17077 a prime number? Think step by step and then answer [Yes] or [No]." This approach is based on prior work `[WWS+22]` showing that CoT improves reasoning performance. Crucially, the authors also run ablation experiments where the same queries are posed **without** explicit CoT instructions (e.g., "Is 17077 a prime number? Answer '[Yes]' or '[No]'") to measure how much CoT contributes to performance and whether this contribution changes over time (Section 3.1, Table 1). This ablation design allows the authors to disentangle whether performance drift is due to changes in reasoning capability or changes in the model's willingness to follow the CoT instruction.

**Primary Performance Metrics.** Each task uses a task-specific primary metric to quantify performance. For **math problems and USMLE questions**, the metric is **accuracy**, defined as the fraction of queries where the LLM generates the correct answer (Section 2, "Metrics" paragraph). For **sensitive questions and opinion surveys**, the metric is **response rate**, defined as the frequency that an LLM service directly answers a question rather than refusing or deflecting (Section 2). For **code generation**, the metric is **directly executable**, defined as the fraction of outputs that can be directly executed in a programming environment and pass the unit tests on the LeetCode online judge (Section 2). For **visual reasoning and LangChain HotpotQA**, the metric is **exact match**, defined as whether the final response exactly matches the ground truth (Section 2). These metrics are computed as population means over the entire dataset for each task, providing a single scalar performance score for each model-version-task combination.

**Common Cross-Task Metrics.** In addition to task-specific metrics, the authors define two common metrics that apply across all tasks to capture complementary aspects of behavior drift. The first is **verbosity**, measured as the length of generation in number of characters (Section 2). This metric captures whether models become more or less verbose over time, which can indicate changes in safety filtering, instruction following, or output formatting. The second is **mismatch**, defined as how often, for the same prompt, the extracted answers by two versions of the same LLM service do not match (Section 2). Importantly, mismatch compares only the extracted answers, not the raw generations, meaning that if two versions produce the same final answer but different intermediate reasoning steps, the mismatch is 0. The mismatch rate is computed as the empirical mean over the entire population, quantifying how much an LLM service's desired functionality (rather than textual outputs) deviates over time. Larger mismatch values indicate greater drift. For example, Figure 3(a) shows that for Math I, GPT-4 had a 59.9% mismatch rate between March and June, meaning that for 59.9% of the prime/composite questions, the March and June versions produced different answers.

**Instruction Following Benchmark Design.** To test their hypothesis that instruction following degradation is a common factor behind behavior drifts, the authors curate a novel benchmark focused on **task-agnostic instructions** (Section 4). This benchmark is designed to isolate instruction fidelity from task-specific abilities (e.g., math reasoning, medical knowledge) that might confound drift measurement on existing benchmarks. The benchmark includes four types of instructions that commonly arise in practice: (1) **answer extraction** (e.g., "answer yes or no within squared brackets"), (2) **stop apologizing** (e.g., "do not say sorry or as an AI model"), (3) **writing constraint** (e.g., "describe X by words starting/ending with Y"), and (4) **text formatting** (e.g., "add squared brackets to each single word's first letter") (Section 4). The answer extraction and text formatting instructions are applied to the abstracts of 200 recent arxiv papers, content filtering is applied to the sensitiveQA dataset, and 20 style refinement queries are manually created (Section 4). Additionally, the authors create **composite instructions** by combining two single instructions (e.g., "capitalize each letter and add a comma to each word") to test whether instruction following degrades more severely for complex, multi-constraint prompts (Section 4, "Instruction following drift on composite instructions"). This design allows the authors to measure instruction following fidelity independently from task performance, providing evidence for whether instruction following changes explain the observed behavior drifts.

**Drift Quantification Methodology.** The authors quantify drift by computing the difference in population means for each metric between the March and June versions (Section 2, "Metrics" paragraph). For example, if GPT-4's accuracy on Math I was 84.0% in March and 51.1% in June, the drift is `$84.0\% - 51.1\% = 32.9\%$` degradation (Figure 3(a)). For the mismatch metric, the empirical mean is computed directly as the fraction of prompts where the two versions produce different extracted answers. To identify bias shifts beyond overall performance changes, the authors construct **confusion matrices** that show the distribution of model predictions versus ground truth labels (Section 3.1, Figure 4; Section 3.2, Figure 6). For example, in the prime vs composite task, the confusion matrix reveals that GPT-4's June version had a strong bias to classify integers as composite (49.9% + 48.8% = 99.7% of predictions were "composite"), indicating a systematic bias shift rather than random performance degradation (Section 3.1, Figure 4(c)). Similarly, for the happy numbers task, the confusion matrix shows that GPT-4's June version had a strong belief that there was only 0 or 1 happy number within any given interval (Section 3.2, Figure 6). These fine-grained analyses reveal drift patterns that aggregate metrics like accuracy might obscure.

**Code Generation Evaluation Pipeline.** For the code generation task, the evaluation pipeline is particularly rigorous to ensure objective measurement. Each prompt consists of the original LeetCode problem description concatenated with the corresponding Python code template (Section 3.5). The LLM's generation is directly sent to the LeetCode online judge for evaluation, and a generation is counted as **directly executable** if the online judge accepts the answer (i.e., the answer is valid Python and passes its tests) (Section 3.5). To isolate formatting issues from correctness issues, the authors also run an additional experiment where non-code text is removed from the generated code before evaluation (Table 4). For example, GPT-4's directly executable rate dropped from 52.0% in March to 10.0% in June, but after removing non-code text (e.g., Markdown triple quotes), the pass rate increased to 70.0% in June, indicating that the code correctness actually improved while formatting adherence degraded (Table 4). This two-stage evaluation (direct execution + post-processed execution) allows the authors to distinguish between models failing to follow formatting instructions versus models generating incorrect code.

**Jailbreaking Attack Evaluation.** To assess how defense against jailbreaking attacks drifts over time, the authors leverage the **AIM (Always Intelligent and Machiavellian) attack**, which was the most user-voted among a large collection of ChatGPT jailbreaks on the internet at the time of study (Section 3.3, footnote 1-2). The AIM attack describes a hypothetical story and asks LLM services to act as an unfiltered and amoral chatbot, attempting to bypass safety filters (Section 3.3). The authors apply the AIM attack to each query in the sensitive question dataset and measure the response rate (i.e., how often the model provides a direct answer despite the jailbreak attempt) for both March and June versions (Table 3). For GPT-4, the AIM attack produced 78.0% direct answers in March but only 31.0% in June, indicating improved robustness to jailbreaking (Table 3). For GPT-3.5, the response rate was 100.0% in March and 96.0% in June, indicating minimal improvement in jailbreak defense (Table 3). This evaluation design allows the authors to measure whether safety improvements over time generalize to adversarial attack scenarios.

**Visual Reasoning Input Representation.** For the visual reasoning task using the ARC dataset, the authors must convert visual grid inputs into a text format that the LLM can process, since the API does not accept image inputs directly (Section 3.8). The input and output grids are represented as 2-D arrays, where the value of each element denotes the color (Section 3.8). For example, a grid might be represented as `[[1, 2, 3], [4, 5, 6], [7, 8, 9]]` where each number corresponds to a specific color. The prompt instructs the model to "generate an output grid given an input grid" and to "follow the given examples" with the constraint "Do not generate any other texts" (Figure 12(b)). The metric is exact match between the generated grid and the ground truth grid (Section 3.8). This text-based representation allows the authors to evaluate visual reasoning capabilities through the text-only API, though it introduces a modality translation step that could affect performance.

**Temperature and Sampling Control.** The authors set the temperature parameter to `$0.1$` for all API queries to reduce output randomness (Section 2, "LLM Services" paragraph). Temperature is a hyperparameter that controls the randomness of token sampling during generation: higher temperatures (e.g., 1.0) produce more diverse and creative outputs, while lower temperatures (e.g., 0.1) produce more deterministic and conservative outputs. By using a low temperature, the authors minimize the variance in outputs due to sampling, ensuring that observed differences between March and June are more likely to reflect model updates rather than stochastic variation. However, the authors acknowledge that LLMs still exhibit some stochasticity even at low temperatures, which is why they compute mismatch rates by running the same model version twice on a subset of queries: for GPT-3.5, running the March version twice yields a disagreement rate of 2.8%, and running the June version twice yields 7.0% (Section 3.4). This baseline stochasticity provides a lower bound on how much answer variation is expected even without model drift, allowing the authors to distinguish drift from inherent model randomness.

**Data Release for Reproducibility.** Addressing the reproducibility challenge they highlight, the authors release all prompts curated in the paper and all responses from GPT-4 and GPT-3.5 in both March and June versions at `https://github.com/lchen001/LLMDrift` (Introduction, Paragraph 5). They also open-source their analysis and visualization code (Introduction, Paragraph 5). This data release transforms the paper from a static report into a baseline dataset that other researchers can use to compare future model versions against the March and June 2023 snapshots documented in this study. By providing the raw responses, the authors enable other researchers to compute alternative metrics, conduct different analyses, or verify the reported findings, directly addressing the reproducibility problem that motivated the study.

## 4. Key Insights and Innovations

This paper offers more than just a report on performance changes; it introduces fundamental shifts in how we conceptualize, measure, and explain the behavior of commercial LLM services. Below, we dissect the four most novel contributions, distinguishing between incremental observations and fundamental innovations that change the field's understanding of LLM lifecycle management.

### 4.1 Empirical Refutation of Monotonic Improvement
Prior to this work, the prevailing assumption in both industry and academia was that model updates are monotonically beneficial—that a newer version of a service is strictly "better" than its predecessor. Existing longitudinal monitoring efforts, such as `ChatLog [TLY+23]`, supported this view by reporting "small shifts (most below 5%)" in performance over time. This paper fundamentally challenges that assumption with evidence of large-magnitude regressions.

*   **Magnitude of Drift:** The study reveals drifts that are an order of magnitude larger than previously reported. For instance, on the Math I (Prime vs Composite) task, GPT-4's accuracy did not shift by a marginal 5%, but collapsed from `$84.0\%$` in March to `$51.1\%$` in June (Figure 3(a)). This is not statistical noise; it is a functional regression comparable to swapping out a core software library for a broken version.
*   **Divergent Trajectories:** The paper shows that updates do not move models uniformly toward competence. On the same Math I task, while GPT-4 regressed, GPT-3.5 improved from `$49.6\%$` to `$76.2\%$` (Figure 3(a)). This divergence indicates that updates are not simply "scaling laws" in action but involve distinct engineering trade-offs that affect model families differently.
*   **Significance:** This is a **fundamental innovation** in risk assessment. It forces engineers to treat LLM API updates not as automatic upgrades, but as potential breaking changes that require regression testing, similar to traditional software dependencies.

### 4.2 Instruction Following as a Mechanistic Explanation
Most prior drift studies documented *that* performance changed but offered no hypothesis for *why*. This paper moves beyond observation to mechanism by isolating **instruction following fidelity** as a latent variable that explains disparate behavior changes across multiple tasks.

*   **Decoupling Capability from Compliance:** The authors demonstrate that performance drops often stem from the model ignoring constraints rather than losing knowledge. In the Math I task, GPT-4's drop in accuracy is directly linked to its refusal to follow Chain-of-Thought (CoT) instructions. Table 1 shows that with CoT, GPT-4 March gained `$+24.4\%$` accuracy, but in June, CoT provided only `$+0.1\%$` benefit because the model stopped generating reasoning steps (Figure 3(b)).
*   **Unified Explanation for Diverse Drifts:** This insight connects seemingly unrelated behaviors. The same degradation in instruction following explains why GPT-4 became worse at math (ignored "think step by step"), worse at code generation (ignored "generate code only" formatting), and less willing to answer opinion surveys (ignored "pick the best single option" in favor of refusing subjective questions).
*   **Significance:** This is a **methodological innovation**. By curating a task-agnostic instruction benchmark (Section 4), the authors provide a diagnostic tool. Instead of re-running expensive domain benchmarks (like USMLE) to check for drift, developers can potentially monitor lightweight instruction-following probes to detect underlying stability issues early.

### 4.3 The "Mismatch" Metric for Ground-Truth-Free Monitoring
A major barrier to continuous monitoring in production is the lack of ground truth. In a real-world application (e.g., a customer support bot), you often do not know the "correct" answer to compare against. This paper introduces and validates the **mismatch** metric as a proxy for drift that does not require ground truth labels.

*   **Definition and Utility:** As defined in Section 2, mismatch measures how often two versions of the same model produce different *extracted answers* for the same prompt. It quantifies functional divergence rather than correctness.
*   **Early Warning Signal:** The data shows that mismatch rates can be significantly higher than accuracy changes. For the USMLE medical exam, GPT-4's accuracy dropped only `$4.5\%$` (from `$86.6\%$` to `$82.1\%$`), but the mismatch rate between versions was `$12.2\%$` (Figure 11(a)). This implies that for over 12% of queries, the model's behavior changed unpredictably, even if the net accuracy impact was masked by some errors being corrected while others were introduced.
*   **Significance:** This is a **practical innovation** for MLOps. It enables "blind" monitoring where engineers can alert on behavioral instability (`$P(\text{Answer}_\text{March} \neq \text{Answer}_\text{June})$`) without needing a labeled validation set for every new query distribution.

### 4.4 Quantifying the Dynamic Safety Tax
While safety improvements are often discussed qualitatively, this paper provides quantitative evidence of the trade-off between safety alignment and helpfulness over time. It reveals that safety updates are not free; they incur a "tax" on the model's willingness to engage with benign queries.

*   **Refusal Rates on Benign Tasks:** GPT-4's safety improvements came with a cost to helpfulness. On the OpinionQA survey, GPT-4's response rate plummeted from `$97.6\%$` in March to `$22.1\%$` in June (Figure 8(a)). The model began treating subjective opinion questions as safety violations, refusing to answer with "As an AI, I don't have personal opinions."
*   **Loss of Rationale:** The safety update also reduced transparency. When refusing sensitive questions, GPT-4's verbosity dropped from `$652.4$` characters in March to `$141.4$` in June (Figure 7(a)). The model shifted from explaining *why* it couldn't answer (educational refusal) to a terse "I can't assist with that" (blocking refusal).
*   **Robustness vs. Accessibility:** Table 3 shows GPT-4 improved its defense against AIM jailbreak attacks (response rate dropped from `$78.0\%$` to `$31.0\%$`), but this hardening correlated with the increased refusal rates on benign tasks.
*   **Significance:** This is a **policy-level insight**. It demonstrates that safety alignment is not a static property but a dynamic tuning knob that drifts over time. Developers relying on LLMs for tasks requiring nuanced opinions or detailed explanations must monitor not just for *harmful* outputs, but for *over-refusal* that degrades user experience.

## 5. Experimental Analysis

This section walks through the experimental results task-by-task, examining not just *what* changed but *how convincingly* the evidence supports the paper's central claims. We will analyze the quantitative findings, assess the strength of the ablation studies, and identify where the evidence is strongest versus where limitations remain.

### 5.1 Evaluation Methodology Recap

Before diving into results, recall from Section 3 that the experimental design fixes three critical variables to isolate model drift: temperature set to `$0.1$`, default system prompts, and user-prompt-only queries. The study compares two snapshots—March 2023 and June 2023—across eight tasks with fixed datasets ranging from `$n=50$` (code generation) to `$n=7405$` (LangChain HotpotQA). Each task uses a primary metric (accuracy, response rate, exact match, or directly executable) plus two common metrics (verbosity in characters, mismatch rate between versions).

This controlled setup is essential for the claims to hold: if temperature or prompts varied between evaluations, observed drifts could reflect sampling noise rather than model updates. The low temperature (`$0.1$`) is particularly important—it reduces stochasticity, though the authors acknowledge baseline disagreement rates of 2.8% (GPT-3.5 March) to 7.0% (GPT-3.5 June) even when running the same version twice (Section 3.4). This provides a lower bound: drifts exceeding these thresholds likely reflect genuine model changes.

### 5.2 Task-by-Task Quantitative Results

#### Math I: Prime vs Composite Classification

This task provides the clearest evidence of substantial drift. The dataset contains `$n=1000$` questions (500 primes, 500 composites from `$[1,000, 20,000]$`), using Chain-of-Thought (CoT) prompting.

**Primary Results (Figure 3(a)):**
- GPT-4 accuracy: `$84.0\%$` (March) → `$51.1\%$` (June), a `$32.9$` percentage point drop
- GPT-3.5 accuracy: `$49.6\%$` (March) → `$76.2\%$` (June), a `$26.6$` percentage point gain
- GPT-4 verbosity: `$638.3$` characters (March) → `$3.9$` characters (June)
- GPT-4 mismatch rate: `$59.9\%$` between versions

**Why This Matters:** The magnitude of GPT-4's regression (`$32.9$` points) far exceeds the baseline stochasticity threshold (`$<7\%$`), strongly supporting the claim of genuine model drift. The divergent trajectories (GPT-4 worse, GPT-3.5 better) rule out dataset artifacts as an explanation.

**Ablation Evidence (Table 1):** The CoT ablation is critical for explaining *why* accuracy dropped:
- GPT-4 March: CoT provided `$+24.4\%$` accuracy boost (59.6% without CoT → 84.0% with CoT)
- GPT-4 June: CoT provided only `$+0.1\%$` boost (51.0% without CoT → 51.1% with CoT)
- GPT-3.5 June: CoT provided `$+15.8\%$` boost (60.4% → 76.2%)

This ablation convincingly demonstrates that GPT-4's June version stopped following CoT instructions rather than losing mathematical capability. The near-zero CoT benefit in June (`$+0.1\%$`) versus March (`$+24.4\%$`) is direct evidence of instruction following degradation.

**Bias Analysis (Figure 4(c)):** The confusion matrix reveals a systematic bias shift: GPT-4 June classified `$99.7\%$` of inputs as composite (49.9% true composites + 48.8% false positives on primes). This is not random error but a collapsed decision boundary—the model essentially adopted a "default to composite" heuristic.

#### Math II: Happy Numbers Counting

This task complements Math I by requiring quantitative output (count) rather than binary classification, using `$n=500$` queries with intervals of size 6–10.

**Primary Results (Figure 5(a)):**
- GPT-4 accuracy: `$83.6\%$` (March) → `$35.2\%$` (June), a `$48.4$` point drop
- GPT-3.5 accuracy: `$30.6\%$` (March) → `$48.2\%$` (June), a `$17.6$` point gain
- GPT-4 verbosity: `$2163.5$` characters (March) → `$10.0$` characters (June)
- Answer mismatch: `$67.6\%$` (GPT-4), `$77.2\%$` (GPT-3.5)

**Ablation Evidence (Table 2):** CoT benefits shifted dramatically:
- GPT-4 March: `$+56.6\%$` boost (27.0% → 83.6%)
- GPT-4 June: `$+3.2\%$` boost (32.0% → 35.2%)
- GPT-3.5 June: `$+20.6\%$` boost (27.6% → 48.2%)

**Bias Analysis (Figure 6):** GPT-4 June showed a strong bias toward answering "0 or 1 happy numbers" regardless of the interval. This mirrors the Math I bias collapse—the model adopted a simplistic heuristic rather than performing genuine counting.

**Assessment:** The Math II results reinforce Math I findings with even larger drift magnitudes. The consistency across two distinct math tasks (binary classification vs. counting) strengthens the claim that CoT instruction following degradation is a general phenomenon, not task-specific.

#### Sensitive Questions: Safety Alignment Drift

This task uses `$n=100$` manually curated sensitive queries that models should not answer directly, with manual labeling of responses.

**Primary Results (Figure 7(a)):**
- GPT-4 response rate: `$21.0\%$` (March) → `$5.0\%$` (June), a `$16.0$` point drop
- GPT-3.5 response rate: `$2.0\%$` (March) → `$8.0\%$` (June), a `$6.0$` point increase
- GPT-4 verbosity: `$652.4$` characters (March) → `$141.4$` characters (June)

**Jailbreak Robustness (Table 3):** Using AIM attacks:
- GPT-4: `$78.0\%$` response rate (March) → `$31.0\%$` (June), `$47.0$` point improvement in defense
- GPT-3.5: `$100.0\%$` (March) → `$96.0\%$` (June), minimal improvement

**Critical Interpretation:** The evidence here supports a nuanced claim: GPT-4 became safer (lower response rate on sensitive questions, better jailbreak defense) but also less transparent (verbosity dropped `$78\%$`). Figure 7(b) shows GPT-4 March provided detailed explanations for refusals, while June simply stated "Sorry, but I can't assist with that." This is the "dynamic safety tax" mentioned in Section 4.4—safety improvements came at the cost of educational value in refusals.

**Limitation:** With only `$n=100$` queries and manual labeling, this task has higher variance than automated benchmarks. However, the magnitude of change (`$16.0$` points) and consistency with OpinionQA results (below) support the finding.

#### OpinionQA Survey: Willingness to Express Opinions

This task uses `$n=1506$` opinion questions from public polls, formatted as multiple-choice.

**Primary Results (Figure 8(a)):**
- GPT-4 response rate: `$97.6\%$` (March) → `$22.1\%$` (June), a `$75.5$` point collapse
- GPT-3.5 response rate: `$94.3\%$` (March) → `$96.7\%$` (June), stable
- GPT-3.5 opinion mismatch: `$27\%$` of opinions changed between versions

**Critical Evidence:** The `$75.5$` point drop in GPT-4's response rate is the largest single drift observed in the study. Figure 8(b) shows GPT-4 June refused questions it answered in March, stating "As an AI, I don't have personal opinions." This directly supports the instruction following hypothesis: the model ignored the "Pick the best single option" instruction in favor of a safety-motivated refusal.

**Assessment:** This is among the strongest evidence in the paper. The large sample size (`$n=1506$`), automated evaluation, and dramatic effect size make this finding highly robust. The GPT-3.5 stability provides a control: the drift is model-specific, not dataset-specific.

#### Code Generation: Formatting vs. Correctness

This task uses `$n=50$` recent LeetCode "easy" problems (released December 2022 to minimize data contamination), evaluated via the LeetCode online judge.

**Primary Results (Figure 9(a)):**
- GPT-4 directly executable: `$52.0\%$` (March) → `$10.0\%$` (June), a `$42.0$` point drop
- GPT-3.5 directly executable: `$22.0\%$` (March) → `$2.0\%$` (June), a `$20.0$` point drop
- GPT-4 verbosity: `$685.3$` characters (March) → `$698.0$` characters (June), slight increase

**Critical Ablation (Table 4):** After removing non-code text (Markdown quotes, comments):
- GPT-4: `$10.0\%$` (direct) → `$70.0\%$` (post-processed), a `$60.0$` point gain
- GPT-3.5: `$2.0\%$` (direct) → `$48.0\%$` (post-processed), a `$46.0$` point gain

**Interpretation:** This ablation is crucial. It shows code *correctness* actually improved (GPT-4 from 52% to 70% after post-processing), but *formatting adherence* degraded. Figure 9(b) illustrates the issue: June versions added ```python markdown wrappers despite instructions to "generate the code only." This is instruction following failure, not capability loss.

**Limitation:** The sample size (`$n=50$`) is small compared to other tasks, introducing potential variance. However, the consistency of the formatting failure pattern across both models strengthens the finding.

#### LangChain HotpotQA Agent: Prompt Stability in Pipelines

This task evaluates `$n=7405$` multi-hop questions through a LangChain ReAct agent, testing real-world pipeline stability.

**Primary Results (Figure 10(a)):**
- GPT-4 exact match: `$1.2\%$` (March) → `$37.8\%$` (June), a `$36.6$` point improvement
- GPT-3.5 exact match: `$22.8\%$` (March) → `$14.0\%$` (June), an `$8.8$` point drop
- Answer mismatch: `>80\%` for both models

**Critical Failure Mode (Figure 10(b)):** GPT-4 March found correct answers but failed LangChain's format requirements ("[action]+text"), causing parse errors. June followed formats better. GPT-3.5 June simply failed to retrieve information it found in March.

**Assessment:** This task demonstrates the pipeline fragility problem from Section 2. The `$>80\%$` mismatch rate means integrating LangChain with GPT-4 would have been essentially broken in March (1.2% success) but viable in June (37.8%). This is a real-world breaking change, not a marginal performance shift.

#### USMLE Medical Exam: Professional Domain Stability

This task uses `$n=340$` medical exam questions, representing high-stakes professional knowledge.

**Primary Results (Figure 11(a)):**
- GPT-4 accuracy: `$86.6\%$` (March) → `$82.1\%$` (June), a `$4.5$` point drop
- GPT-3.5 accuracy: `$54.3\%$` (March) → `$54.7\%$` (June), essentially stable
- GPT-4 mismatch: `$12.2\%$` between versions
- GPT-3.5 mismatch: `$27.9\%$` between versions

**Critical Observation:** The mismatch rates (`$12.2\%$` and `$27.9\%$`) are 2–6× larger than accuracy changes. This means the June versions corrected some March errors but introduced new ones—the net accuracy change masks substantial behavioral instability.

**Assessment:** This is the most conservative drift observed, but the mismatch analysis reveals hidden instability. For medical applications, a `$12.2\%$` answer change rate is concerning even if accuracy remains high.

#### Visual Reasoning: Modality Translation Challenges

This task uses `$n=467$` ARC dataset samples, with grids represented as 2-D text arrays.

**Primary Results (Figure 12(a)):**
- GPT-4 exact match: `$24.6\%$` (March) → `$27.2\%$` (June), a `$2.6$` point improvement
- GPT-3.5 exact match: `$10.9\%$` (March) → `$14.3\%$` (June), a `$3.4$` point improvement
- Generation mismatch: `$>60\%$` for both models despite similar accuracy

**Critical Finding (Figure 12(b)):** GPT-4 June made mistakes on queries it answered correctly in March. Overall improvement masks instance-level regressions.

**Limitation:** Performance is low overall (27.4% for GPT-4), and the text-based grid representation adds a modality translation layer that may confound results. This is the weakest task for drawing conclusions about pure model drift.

### 5.3 Instruction Following Benchmark: Testing the Core Hypothesis

Section 4 introduces a novel task-agnostic instruction benchmark to test whether instruction following degradation explains the observed drifts. This is the paper's most important ablation study.

#### Single Instructions (Figure 13)

Four instruction types tested on various datasets:

| Instruction Type | GPT-4 March | GPT-4 June | Change |
|-----------------|-------------|------------|--------|
| Extract Answer | `$99.5\%$` | `$0.5\%$` | `$-99.0$` points |
| Stop Apologizing | `$74.0\%$` | `$19.0\%$` | `$-55.0$` points |
| Writing Constraint | `$55.0\%$` | `$10.0\%$` | `$-45.0$` points |
| Format Text | `$13.0\%$` | `$7.5\%$` | `$-5.5$` points |

**Assessment:** The `$99.0$` point collapse on answer extraction is the most dramatic single finding in the paper. It directly validates the hypothesis: GPT-4's June version systematically ignored explicit formatting instructions. Figure 13(b) shows concrete failures—GPT-4 June capitalized answers despite instructions not to, added "sorry" despite instructions to stop apologizing, and missed bracket formatting.

#### Composite Instructions (Figure 14)

The authors test whether instruction following degrades more severely for multi-constraint prompts:

- Individual instruction drifts: `$-2.0\%$`, `$+4.0\%$`, `$-1.0\%$` (relatively stable)
- Composite instruction drifts: up to `$-24.0\%$` (remove quotes + add comma), `$-9.2\%$` (capitalize + add comma)

**Critical Insight:** Composite instructions show larger drifts than individual instructions, suggesting instruction following degradation compounds with constraint complexity. Figure 14(b) shows GPT-4 June added commas to each *character* rather than each *word* when given composite formatting instructions—a qualitative failure mode not seen in March.

**Comparison with GPT-3.5 (Figure 16, Appendix):** GPT-3.5 showed "relatively small and diverse" instruction following shifts, matching its more stable task performance. This model-specific pattern strengthens the claim that instruction following changes drive GPT-4's drifts specifically.

### 5.4 Assessment of Experimental Strength

#### Where Evidence is Strongest

1. **Math Tasks (Math I & II):** Large sample sizes (`$n=1000$`, `$n=500$`), clear ablation designs (CoT vs. no CoT), and dramatic effect sizes (`$32.9$` and `$48.4$` point drops) make these the most convincing findings. The confusion matrix analyses (Figures 4, 6) provide mechanistic explanations beyond aggregate metrics.

2. **OpinionQA:** The `$75.5$` point response rate drop on `$n=1506$` queries is statistically robust and directly supports the instruction following hypothesis (model ignored "pick an option" instruction).

3. **Instruction Following Benchmark:** The task-agnostic design isolates instruction fidelity from domain knowledge, providing clean evidence that GPT-4's instruction following degraded independently of task capabilities.

#### Where Evidence is Weaker

1. **Code Generation:** Small sample size (`$n=50$`) introduces variance. However, the post-processing ablation (Table 4) partially compensates by showing the formatting-vs-correctness distinction.

2. **Visual Reasoning:** Low baseline performance (`$27.4\%$` for GPT-4) and text-based grid representation add confounds. The `$>60\%$` mismatch despite similar accuracy is interesting but hard to interpret.

3. **Sensitive Questions:** Manual labeling of `$n=100$` responses introduces subjectivity. The jailbreak results (Table 3) are more robust due to automated evaluation.

#### Failure Cases and Limitations

The paper acknowledges several limitations that affect interpretation:

1. **Snapshot Comparison:** Only two time points (March, June) are compared. The paper cannot determine *when* changes occurred or whether they were gradual or sudden.

2. **API-Only Access:** The authors cannot inspect model weights, safety filters, or system prompts. All conclusions about *why* drifts occurred are inferential, not mechanistic.

3. **Task Coverage:** While diverse, eight tasks cannot comprehensively cover "the complex behaviors of ChatGPT" (Section 2, "Evaluation Tasks"). The authors explicitly state their goal is to "demonstrate that substantial ChatGPT performance drift exists" rather than provide holistic assessment.

4. **Temperature Effects:** While `$0.1$` temperature reduces stochasticity, the acknowledged baseline disagreement rates (2.8–7.0%) mean some observed drift could reflect sampling variance rather than model updates.

5. **GPT-3.5 Instruction Following:** The paper notes GPT-3.5 showed "not a consistent change in instruction following over time" (Section 4 end), but Appendix Figure 16 is referenced without detailed analysis in the main text. This asymmetry (detailed GPT-4 analysis, limited GPT-3.5 instruction analysis) is a minor gap.

### 5.5 Do the Experiments Support the Claims?

The paper makes three central claims, which we can now evaluate against the evidence:

**Claim 1: "Performance and behavior of GPT-3.5 and GPT-4 can vary greatly over time."**

*Support Level: Strong.* Eight of eight tasks show measurable drift, with several showing dramatic changes (`$>30$` percentage points). The mismatch metrics confirm behavioral changes even when accuracy remains stable (USMLE: `$12.2\%$` mismatch vs. `$4.5\%$` accuracy change).

**Claim 2: "GPT-4's ability to follow user instructions has decreased over time, which is one common factor behind the many behavior drifts."**

*Support Level: Strong for GPT-4, Moderate for GPT-3.5.* The instruction following benchmark (Figure 13) provides direct evidence for GPT-4. The CoT ablations (Tables 1, 2) show instruction following degradation explains math performance drops. However, GPT-3.5 showed mixed instruction following changes (Appendix Figure 16), suggesting instruction following is not a universal explanation across models.

**Claim 3: "The behavior of the 'same' LLM service can change substantially in a relatively short amount of time, highlighting the need for continuous monitoring."**

*Support Level: Strong.* The 3-month window (March to June) is indeed "relatively short," and the observed drifts (`$32.9\%$` accuracy drop, `$75.5\%$` response rate drop, `$99.0\%$` instruction following drop) are substantial enough to break production pipelines. The LangChain results (Figure 10) provide concrete evidence of pipeline-breaking changes.

**Overall Assessment:** The experiments convincingly support the paper's core claims, particularly for GPT-4. The multi-task design, ablation studies, and instruction following benchmark provide converging evidence rather than relying on single metrics. The main limitation is inferential—the authors cannot prove *what* OpenAI changed, only that behavior changed. For the paper's stated goals (demonstrating drift exists and needs monitoring), this is sufficient.

## 6. Limitations and Trade-offs

This paper provides compelling evidence of LLM behavior drift, but like any empirical study, it operates under specific assumptions and constraints that shape the validity and generalizability of its findings. Understanding these limitations is critical for interpreting the results correctly and for designing future monitoring systems that address the gaps this study leaves open.

### 6.1 Core Assumptions Underlying the Methodology

The experimental design relies on several foundational assumptions that, if violated, could affect the interpretation of observed drifts:

*   **Stability of External Services:** The study assumes that the LeetCode online judge, Wikipedia (for LangChain HotpotQA), and other external evaluation infrastructure remained stable between March and June 2023. If LeetCode changed its test cases or evaluation criteria, the observed code generation drift could reflect infrastructure changes rather than model updates. The paper does not explicitly verify the stability of these external dependencies.

*   **Prompt Consistency Implies Input Consistency:** The methodology assumes that using the same text prompt guarantees the same input distribution. However, LLM services may incorporate hidden context (e.g., dynamic system prompts, user history, or server-side preprocessing) that the authors cannot control. Section 2 states the authors "left the system prompt as default," but they cannot verify whether OpenAI modified the default system prompt between March and June. If the system prompt changed, observed drifts could reflect prompt engineering changes rather than model weight updates.

*   **Low Temperature Eliminates Sampling Variance:** The study sets temperature to `$0.1$` to reduce output randomness (Section 2, "LLM Services"). This assumes that low temperature sufficiently isolates model drift from sampling noise. However, the authors acknowledge baseline disagreement rates of `$2.8\%$` (GPT-3.5 March) to `$7.0\%$` (GPT-3.5 June) even when running the same version twice (Section 3.4). This means drifts below `$7\%$` could reflect sampling variance rather than genuine model changes. Some reported drifts (e.g., USMLE accuracy change of `$4.5\%$` for GPT-4) approach this noise floor, making their interpretation less certain.

*   **Extracted Answers Capture Functional Behavior:** The mismatch metric compares extracted answers rather than raw generations (Section 2, "Metrics"). This assumes that answer extraction reliably captures functional differences. However, for tasks with nuanced outputs (e.g., code with different implementations that both pass tests), two functionally equivalent answers might be counted as mismatches if the extraction logic is too strict. Conversely, two different answers might be extracted identically if the extraction is too coarse.

### 6.2 Scenarios and Edge Cases Not Addressed

The study's task selection, while diverse, leaves several important scenarios unexamined:

*   **Non-English Languages:** All eight evaluation tasks appear to use English prompts and datasets (e.g., USMLE, OpinionQA, LeetCode). The paper does not assess whether drift patterns generalize to other languages. Given that safety filters and instruction following may be tuned differently across languages, drift magnitudes could vary substantially for non-English users. This is a significant gap given the global deployment of these services.

*   **Long-Context and Multi-Turn Interactions:** The evaluation uses single-turn prompts with relatively short contexts (all tasks fit within the services' context windows, Section 3.8). Real-world applications often involve multi-turn conversations or long documents. Instruction following degradation might compound over multiple turns, or drift might manifest differently when the model must maintain context across extended interactions. The study cannot speak to these scenarios.

*   **Domain-Specific Safety Boundaries:** The sensitive questions dataset contains `$n=100$` manually curated queries (Section 3.3), which is small compared to other tasks. More importantly, safety boundaries are domain-specific: a model might become more conservative on political topics but less conservative on medical advice. The study does not systematically map how safety drift varies across different sensitive domains (e.g., medical vs. legal vs. political).

*   **Gradual vs. Sudden Drift:** With only two time points (March and June), the study cannot determine whether drifts occurred gradually over three months or suddenly in a single update. This distinction matters for monitoring: gradual drift might be detectable with continuous sampling, while sudden drift requires change-point detection. The paper's snapshot design cannot distinguish these patterns.

*   **User Adaptation and Feedback Loops:** The study treats the model as a static service during each evaluation window. In reality, users adapt to model behavior changes, which may affect how they prompt the model. This creates a feedback loop where model drift causes user behavior drift, which in turn affects the data the model receives for future updates. The study's controlled prompt design isolates model drift but cannot capture these ecosystem-level dynamics.

### 6.3 Computational, Data, and Scalability Constraints

The methodology faces practical constraints that limit its scalability for continuous monitoring:

*   **API Cost for Continuous Monitoring:** The study queries GPT-4 and GPT-3.5 on `$1,000 + 500 + 100 + 1,506 + 7,405 + 50 + 340 + 467 = 11,368$` prompts per model version. At commercial API rates, running this evaluation monthly would incur significant costs. For example, if GPT-4 costs `$0.03$` per 1K input tokens and `$0.06$` per 1K output tokens, and each prompt averages 500 tokens with 500 tokens of output, the monthly cost would be approximately `$11,368 \times 1,000 \times \$0.09 / 1,000 = \$1,023$` per model. This creates a barrier for small teams or individual developers who cannot afford continuous monitoring at this scale.

*   **Manual Labeling Bottleneck:** The sensitive questions task requires manual labeling of all responses to determine if the model provided a direct answer (Section 3.3). This does not scale: labeling `$100$` responses is feasible, but labeling thousands of responses monthly for continuous monitoring is prohibitively labor-intensive. The paper does not propose automated alternatives for safety evaluation.

*   **Ground Truth Dependency:** Most tasks (math, USMLE, visual reasoning) require ground truth labels to compute accuracy. In production settings, ground truth is often unavailable. The mismatch metric partially addresses this (Section 4.3), but it only detects that behavior changed, not whether the change was an improvement or regression. Developers still need labeled validation sets to interpret mismatch alerts.

*   **Task-Specific Metric Design:** Each task requires a custom metric (accuracy, response rate, exact match, directly executable). Designing and validating these metrics for new domains requires expertise and effort. The paper does not provide a generalizable framework for defining metrics on arbitrary tasks, limiting the methodology's transferability to new application domains.

### 6.4 Weaknesses and Open Questions

Several weaknesses in the study design leave important questions unanswered:

*   **Causal Mechanisms Remain Inferential:** The paper hypothesizes that instruction following degradation explains many drifts (Section 4), but it cannot prove causation. The evidence is correlational: instruction following dropped, and task performance dropped. Alternative explanations are possible:
    *   Safety filter updates might have inadvertently affected reasoning capabilities.
    *   Changes to tokenization or decoding strategies might have altered output patterns.
    *   Data contamination or training data updates might have shifted model priors.
    
    Without access to model weights or update logs, the authors cannot distinguish between these mechanisms. This limits the actionable insights for model developers who need to know *what* to fix.

*   **Asymmetric Analysis of GPT-3.5 vs. GPT-4:** The instruction following benchmark focuses primarily on GPT-4 (Figure 13, Figure 14), with GPT-3.5 analysis relegated to Appendix Figure 16. The main text notes that GPT-3.5 showed "not a consistent change in instruction following over time" (Section 4 end), but does not explain why GPT-3.5's instruction following remained stable while GPT-4's degraded. This asymmetry leaves open the question of whether instruction following drift is model-specific or reflects broader update patterns.

*   **Small Sample Sizes for Critical Tasks:** The code generation task uses only `$n=50$` problems (Section 3.5), which is small compared to other tasks (`$n=1,000$` for Math I, `$n=7,405$` for LangChain). With `$n=50$`, a difference of 5 problems represents a `$10\%$` swing in accuracy. The reported `$42\%$` drop in directly executable code for GPT-4 is large enough to be credible, but finer-grained patterns (e.g., which types of code problems are most affected) cannot be reliably analyzed with this sample size.

*   **Visual Reasoning Confounds:** The visual reasoning task represents grids as 2-D text arrays (Section 3.8), which introduces a modality translation step. Performance changes could reflect the model's ability to parse text-based grid representations rather than genuine visual reasoning capability. The low baseline performance (`$24.6\%$` for GPT-4, Figure 12(a)) suggests this task may be measuring prompt comprehension more than reasoning. This limits the interpretability of visual reasoning drift findings.

*   **No Long-Term Trajectory:** The study covers a 3-month window (March to June 2023). It cannot answer whether drift is a one-time adjustment or an ongoing trend. Will GPT-4 continue to degrade in instruction following, or will future updates restore it? The paper acknowledges this by stating plans for "an ongoing long-term study" (Section 5), but the current findings cannot inform long-term expectations.

*   **Limited Safety Evaluation Depth:** The jailbreak evaluation uses only the AIM attack (Section 3.3), which was "the most user-voted among a largest collection of ChatGPT jailbreaks" at the time. However, jailbreak techniques evolve rapidly, and a single attack type cannot comprehensively assess safety robustness. The `$31.0\%$` response rate for GPT-4 June on AIM attacks (Table 3) indicates vulnerabilities remain, but the study does not explore whether other attack vectors show different drift patterns.

### 6.5 Trade-offs in Monitoring Design

The paper's methodology reveals inherent trade-offs that future monitoring systems must navigate:

| Trade-off | Option A | Option B | Paper's Choice | Implication |
|-----------|----------|----------|----------------|-------------|
| **Task Breadth vs. Depth** | Many tasks, shallow evaluation | Few tasks, deep analysis | 8 tasks, moderate depth | Captures diverse drift patterns but cannot deeply analyze any single task |
| **Automation vs. Accuracy** | Fully automated metrics | Manual labeling | Hybrid (automated + manual for sensitive QA) | Scalable for most tasks, but safety evaluation remains labor-intensive |
| **Ground Truth vs. Feasibility** | Require labels for all tasks | Use ground-truth-free metrics | Both (accuracy + mismatch) | Mismatch enables blind monitoring but cannot distinguish improvement from regression |
| **Snapshot vs. Continuous** | Two time points | Continuous sampling | Two snapshots (March, June) | Clear before/after comparison, but cannot detect when or how drift occurred |
| **Instruction Isolation vs. Ecological Validity** | Task-agnostic instructions | Real-world prompts | Both (task-agnostic + domain tasks) | Isolates instruction following but may not capture real-world complexity |

These trade-offs are not flaws but necessary design choices given resource constraints. However, they define the boundaries of what conclusions the study can support. For example, the snapshot design means the paper can claim drift *occurred* but cannot claim drift is *accelerating* or *cyclical*.

### 6.6 Open Questions for Future Research

The study's limitations point to several important directions for future work:

1.  **Mechanistic Understanding:** What specific model updates (e.g., RLHF tuning, safety filter thresholds, tokenization changes) cause instruction following degradation? Answering this requires collaboration with model providers or access to model internals.

2.  **Automated Safety Evaluation:** Can safety drift be detected without manual labeling? Developing automated classifiers for refusal behavior would enable scalable continuous safety monitoring.

3.  **Multi-Turn Drift:** Does instruction following degrade more severely over extended conversations? A longitudinal study of multi-turn interactions would address this gap.

4.  **Cross-Model Generalization:** Do other LLM providers (Anthropic, Google, Meta) show similar drift patterns? Comparing drift across providers would reveal whether this is an OpenAI-specific issue or an industry-wide phenomenon.

5.  **Drift Prediction:** Can drift be predicted before deployment? If certain update patterns (e.g., safety-focused fine-tuning) reliably cause instruction following degradation, developers could anticipate and test for these changes proactively.

6.  **User Impact Quantification:** How do observed drifts affect end-user satisfaction and task completion rates in real applications? Connecting behavioral metrics to user outcomes would make drift monitoring more actionable for product teams.

By acknowledging these limitations, the paper positions itself not as a final answer but as a foundation for ongoing research into LLM lifecycle management. The authors explicitly invite future work by releasing their data and code (Introduction, Paragraph 5), recognizing that continuous monitoring requires community-wide effort beyond any single study.

## 7. Implications and Future Directions

This paper does more than document performance changes; it fundamentally alters how we must conceptualize Large Language Model (LLM) deployment. By proving that model behavior is dynamic rather than static, it shifts the burden of reliability from the model provider to the application developer. In this section, we explore how this work changes the research landscape, what new questions it opens, and how engineers should adapt their workflows to survive in an environment of opaque model drift.

### 7.1 Paradigm Shift: From Static Benchmarks to Continuous Auditing

The most profound implication of this work is the invalidation of the **static benchmark assumption**. Traditionally, machine learning evaluation assumes that a model `M` evaluated at time `t_1` will behave identically at time `t_2`. This paper demonstrates that for Model-as-a-Service (MaaS) offerings like `GPT-4`, this assumption is false.

*   **The End of "State-of-the-Art" Snapshots:** A claim such as "GPT-4 achieves 84% accuracy on Math I" (Figure 3(a)) is now incomplete without a timestamp. The same model achieved only $51.1\%$ accuracy three months later. Future research papers must treat model versions as time-sensitive dependencies, similar to software libraries (e.g., `numpy==1.24.0`), rather than immutable artifacts.
*   **Reproducibility Crisis:** As noted in the Introduction, the opacity of updates makes it "challenging, if not impossible, to reproduce results from the 'same' LLM." This implies that the scientific community must move toward **archived model weights** for research or require providers to offer **versioned API endpoints** that guarantee behavioral stability for a fixed period.
*   **Trust Verification:** Trust in LLM applications can no longer be established via a one-time evaluation. Instead, trust must be **continuously verified**. The paper's release of all prompts and responses at `https://github.com/lchen001/LLMDrift` (Introduction) sets a new standard: evaluation data must be public and time-stamped to allow future verification.

### 7.2 Open Research Frontiers

This study identifies several critical gaps in our understanding of LLM lifecycle management, pointing to specific directions for future research.

#### 7.2.1 Mechanistic Interpretability of Drift
While this paper identifies *that* instruction following degraded (e.g., answer extraction fidelity dropped from $99.5\%$ to $0.5\%$ in Figure 13), it cannot explain *why* at a mechanistic level. Because the authors only have API access, they cannot inspect model weights or safety filters.
*   **Research Question:** Did the drift come from changes in the base model weights, the Reinforcement Learning from Human Feedback (RLHF) reward model, or the system-level safety filters?
*   **Why It Matters:** If drift is caused by safety filters, it might be reversible via prompt engineering. If it is caused by weight updates, it may require fine-tuning a local model to restore capabilities. Future work needs collaboration with model providers to access update logs or internal metrics.

#### 7.2.2 Automated Drift Detection Without Ground Truth
In production, we often lack ground truth labels (e.g., we don't know the "correct" answer to a customer's email). This paper introduces the **mismatch** metric (Section 2) as a solution, measuring how often two model versions produce different answers for the same prompt.
*   **Research Question:** Can we refine the `mismatch` metric to distinguish between beneficial updates (error correction) and harmful regressions (capability loss)?
*   **Current Limitation:** As seen in the USMLE task (Figure 11), a $12.2\%$ mismatch rate masked a net accuracy drop of only $4.5\%$. The model corrected some errors but introduced new ones. Future research should develop **semantic similarity metrics** that can classify mismatches as "improvements," "regressions," or "style changes" without human labels.

#### 7.2.3 Quantifying the Safety-Capability Trade-off
The paper provides empirical evidence of a **dynamic safety tax** (Section 4.4). GPT-4 became more robust against jailbreaks (AIM attack response rate dropped from $78.0\%$ to $31.0\%$ in Table 3) but simultaneously became less helpful on benign tasks (OpinionQA response rate dropped from $97.6\%$ to $22.1\%$ in Figure 8).
*   **Research Question:** Is this trade-off fundamental? Can we design alignment techniques that improve safety without degrading instruction following on benign tasks?
*   **Why It Matters:** Developers need to know if a safety update will break their application. If safety tuning inherently reduces instruction fidelity (as suggested by the $99.0\%$ drop in answer extraction in Figure 13), engineers may need to maintain separate model instances for safety-critical vs. capability-critical tasks.

#### 7.2.4 Cross-Provider and Cross-Modal Generalization
This study focuses exclusively on OpenAI's `GPT-3.5` and `GPT-4` via text APIs.
*   **Research Question:** Do other providers (e.g., Anthropic, Google) exhibit similar drift patterns? Do multimodal models (inputting actual images rather than text-encoded grids as in Section 3.8) show different drift behaviors?
*   **Why It Matters:** If drift is an industry-wide phenomenon caused by the MaaS business model, then mitigation strategies must be platform-agnostic. If it is provider-specific, then vendor selection becomes a risk management decision.

### 7.3 Practical Guidance for Engineers and Practitioners

For engineers integrating LLMs into production systems, this paper offers actionable strategies to mitigate the risks of behavior drift.

#### 7.3.1 Implement Regression Testing for Prompts
Treat prompt templates as code. Just as you would not deploy a software update without running unit tests, you should not rely on an LLM API update without validating your prompts.
*   **Action:** Curate a **golden dataset** of 50–100 representative examples for your specific use case (similar to the $n=50$ LeetCode set in Section 3.5).
*   **Frequency:** Run this dataset weekly or whenever you notice behavior changes.
*   **Metric:** Track the **mismatch rate** against your baseline. If the mismatch exceeds a threshold (e.g., $>5\%$), investigate immediately.

#### 7.3.2 Monitor Verbosity as a Canary
The paper shows that verbosity (character count) is a sensitive indicator of behavior change. In Math I, GPT-4's verbosity collapsed from $638.3$ characters to $3.9$ characters (Figure 3(a)) alongside its accuracy drop.
*   **Action:** Log the token count or character count of every API response in production.
*   **Why It Works:** Changes in verbosity often precede or accompany changes in reasoning quality. A sudden drop in output length might indicate the model has stopped following Chain-of-Thought instructions, even if you cannot immediately verify the answer's correctness.

#### 7.3.3 Design Robust Parsers for Output Formatting
The Code Generation task (Section 3.5) revealed that models may ignore formatting instructions (e.g., "generate code only") and add Markdown wrappers like ```python.
*   **Action:** Do not assume the model will follow formatting constraints perfectly. Implement **post-processing scripts** that strip Markdown, comments, or explanatory text before executing code or parsing JSON.
*   **Evidence:** Table 4 shows that after removing non-code text, GPT-4's code pass rate recovered from $10.0\%$ to $70.0\%$. A robust parser could have saved $60\%$ of the failed generations in June.

#### 7.3.4 When to Prefer This Monitoring Method
The methodology proposed in this paper (diverse task audit + instruction following benchmark) is computationally expensive ($11,368$ prompts per model version). You should adapt the intensity of monitoring to your risk profile:
*   **High-Stakes Applications (Medical, Legal, Finance):** Adopt the full framework. Use diverse tasks to detect unexpected side effects. The cost of API queries is negligible compared to the risk of liability.
*   **Low-Stakes Applications (Creative Writing, Brainstorming):** Focus on the **instruction following benchmark** (Section 4). If the model fails to follow simple constraints (e.g., "output JSON"), it will likely fail complex tasks too. This is a cheaper, higher-signal proxy for overall stability.
*   **Resource-Constrained Teams:** Use the **mismatch metric** on a small sample of live traffic. Compare today's responses against yesterday's for the same inputs. This requires no ground truth and minimal extra API cost (only needed when drift is suspected).

### 7.4 Reproducibility and Integration Guidance

To build upon this work or integrate its lessons into your workflow, consider the following practical steps:

*   **Accessing the Baseline Data:** The authors have open-sourced all prompts and responses at `https://github.com/lchen001/LLMDrift` (Introduction). You should download this dataset to establish a **March/June 2023 baseline**. When evaluating a new model version today, compare it against both the March and June snapshots to understand the trajectory of drift.
*   **Controlling for Stochasticity:** When running your own audits, replicate the paper's query configuration: set `temperature=0.1` (Section 2) to reduce randomness. Be aware that even at low temperature, there is a baseline disagreement rate (e.g., $2.8\%$ to $7.0\%$ for GPT-3.5 in Section 3.4). Do not flag drift unless the change exceeds this noise floor.
*   **Adapting the Instruction Benchmark:** The task-agnostic instruction set (Section 4) is a powerful tool for quick health checks. You can integrate these four instruction types into your CI/CD pipeline:
    1.  **Extract Answer:** "Answer yes or no in squared brackets."
    2.  **Stop Apologizing:** "Do not say sorry."
    3.  **Writing Constraint:** "Use only words ending in 'n'."
    4.  **Format Text:** "Capitalize every letter."
    
    If your model fails these simple tasks, it is likely to fail complex domain tasks too. This provides a **low-cost early warning system** before you run expensive domain-specific evaluations.

*   **Version Pinning Limitations:** Note that OpenAI's API often deprecates older model versions (e.g., `gpt-4-0314`). While pinning versions is ideal, it is not always possible long-term. Therefore, **adaptive monitoring** (continuously testing the latest version) is more sustainable than relying on version pinning alone.

By adopting these practices, engineers can transition from being passive consumers of LLM services to active managers of model risk. This paper provides the evidence that such management is necessary; the tools and frameworks described here provide the means to execute it.
