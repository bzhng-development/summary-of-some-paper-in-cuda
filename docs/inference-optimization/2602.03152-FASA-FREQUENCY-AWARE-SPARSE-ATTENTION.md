## 1. Executive Summary

This paper addresses the prohibitive memory footprint of the Key Value (KV) cache in Large Language Models during long-context processing by introducing FASA, a training-free framework that exploits functional sparsity within Rotary Positional Encodings (RoPE) to dynamically predict token importance. By identifying a sparse subset of "dominant" frequency chunks, FASA achieves nearly 100% of full-KV performance on LongBench-V1 while retaining only 256 tokens and delivers a 2.56× speedup on AIME24 using just 18.9% of the cache. This method significantly reduces memory bandwidth consumption and computational overhead without requiring model retraining, consistently outperforming existing token-eviction baselines like SnapKV and Quest across models such as Llama3.2-3B and Qwen2.5-7B.

## 2. Context and Motivation

### The KV Cache Bottleneck in Long-Context LLMs

To understand why FASA matters, we first need to understand the computational bottleneck it addresses. When a Large Language Model generates text token-by-token (auto-regressive decoding), it must remember all previously generated tokens to maintain context. This memory is stored in the **Key Value (KV) cache** — a data structure holding the key and value vectors from the attention mechanism for every past token.

The critical problem: **the KV cache grows linearly with sequence length**. For a sequence of $t$ tokens with hidden dimension $d$, the cache requires $O(td)$ memory per attention head. As contexts extend to tens or hundreds of thousands of tokens (for tasks like repository-level code analysis or document summarization), this memory footprint becomes prohibitive. More critically, at each decoding step, the model must load the *entire* KV cache from memory to compute attention scores. This creates a **memory-bound process** that underutilizes high-performance GPUs.

Figure 3 in the paper quantifies this bottleneck: at a 32K context length, **decoding latency constitutes approximately 90% of total inference time**. The graph shows decoding latency scaling from 0.1 seconds at 1K tokens to 20.4 seconds at 32K tokens, while prefill latency remains relatively stable. This memory I/O bottleneck, not computation, is the primary constraint on long-context throughput.

### Why Existing Solutions Fall Short

The paper identifies five main optimization directions for KV cache management: token eviction, low-rank compression, quantization, KV merging, and budget allocation. FASA focuses on **token eviction** — the intuitive approach of selectively removing less important tokens from the cache. However, existing token eviction methods suffer from fundamental limitations that FASA aims to overcome.

**Static strategies** like Stream (Xiao et al., 2024) use fixed rules to retain tokens — typically keeping only the initial tokens (to handle "attention sinks") and recent tokens (for recency bias). The problem is obvious: this **risks irreversible information loss**. Critical information may reside in intermediate positions that get permanently discarded. As Table 2 demonstrates, Stream's performance drops by 10-20% compared to full KV cache across multiple benchmarks.

**Adaptive strategies** attempt to be more dynamic. SnapKV (Li et al., 2024) estimates token importance during the prefill stage using accumulated attention scores, then permanently evicts less critical tokens. Quest (Tang et al., 2024) organizes the cache into pages and selectively fetches them during decoding. However, both employ **heuristics that insufficiently capture the query-dependent nature of token importance**. SnapKV's one-time prefill estimation cannot adapt as generation progresses and token relevance evolves. Quest's page-level granularity is too coarse — it must retrieve entire pages even when only a few tokens within them are needed, incurring unnecessary overhead.

**Learning-based strategies** train a token predictor to estimate importance. While theoretically appealing, these methods **suffer from poor generalization across different datasets and tasks**. A predictor trained on one domain may fail catastrophically on another, limiting practical deployment.

### The Core Research Question

This landscape leads to the central question motivating FASA (stated explicitly in Section 1):

> *Can a token predictor achieve query-awareness without resorting to costly training?*

Existing methods force a trade-off: static approaches are training-free but query-blind; learning-based approaches are query-aware but require training and generalize poorly. FASA seeks to break this trade-off by discovering an intrinsic property of the model architecture itself that can serve as a **computationally free proxy for token importance**.

### Positioning FASA Relative to Prior Work

FASA positions itself distinctly from prior approaches through three key differentiators:

**First, FASA is training-free yet query-aware.** Unlike learning-based methods, it requires no additional training or fine-tuning. Unlike static methods, it dynamically adapts to each query during decoding. This is achieved by exploiting **functional sparsity in RoPE frequency chunks** — a structural property inherent to the model's positional encoding mechanism, not a learned heuristic.

**Second, FASA operates at finer granularity than adaptive methods.** While Quest works at page-level (typically 16 tokens per page) and SnapKV works at prefill-stage token level, FASA identifies important tokens at **decoding time** using frequency-chunk-level signals. This enables truly dynamic, query-dependent token selection that evolves as generation progresses.

**Third, FASA is orthogonal to other compression paradigms.** The paper explicitly notes that FASA can be integrated with methods like PyramidKV (layer-wise budget allocation) because it addresses *which* tokens to keep, while other methods address *how many* tokens per layer or *how to compress* individual token representations. Table 4 demonstrates this compatibility: combining FASA with PyramidKV yields consistent performance gains (0.1-1.1% improvement across budgets), confirming the methods address complementary aspects of the problem.

### Theoretical Significance: RoPE-Induced Functional Sparsity

Beyond practical efficiency, FASA contributes a novel theoretical insight: **Rotary Positional Encodings induce functional sparsity at the frequency-chunk level**. This discovery (detailed in Section 3) reveals that not all dimensions in RoPE contribute equally to contextual awareness. A small subset of "dominant" frequency chunks consistently exhibits high agreement with full attention patterns, while the majority construct robust positional patterns without contributing significantly to token selection.

This finding has implications beyond FASA itself. It suggests that the functional roles of different frequency components in RoPE are **intrinsic to the mechanism**, not task-specific adaptations. Table 10 shows that dominant FC identification overlaps exceed 70% across different calibration datasets, and Figure 1 demonstrates consistent sparsity patterns across model architectures (Meta-Llama-3.1-8B, Mistral-7B). This universality is what enables FASA's one-time offline calibration to work across diverse tasks without retraining.

### Real-World Impact

The practical stakes are substantial. Long-context LLMs are increasingly deployed for applications requiring hundreds of thousands of tokens: code repositories, legal documents, scientific papers, and multi-turn conversations. The KV cache bottleneck directly limits:

- **Deployment cost**: Memory requirements scale linearly with context, making long-context inference expensive on cloud infrastructure.
- **Latency**: Memory-bound decoding creates user-facing delays, particularly problematic for interactive applications.
- **Accessibility**: Only well-resourced organizations can afford the GPU memory needed for long-context models, limiting democratization.

FASA's reported 8× KV cache compression (FASA-M variant) and 2.56× speedup (FASA-C variant on AIME24 with 18.9% cache usage) directly address these constraints. By reducing memory bandwidth consumption without sacrificing accuracy (near-oracle performance with only 256 retained tokens on LongBench-V1), FASA makes long-context inference more feasible for resource-constrained environments.

### Summary of the Gap

In essence, the paper identifies a gap between what existing token eviction methods offer and what efficient long-context inference requires:

| Approach | Training-Free | Query-Aware | Fine Granularity | Universal |
|----------|---------------|-------------|------------------|-----------|
| Stream (Static) | ✓ | ✗ | Token-level | ✓ |
| SnapKV (Adaptive) | ✓ | Partial (prefill only) | Token-level | ✓ |
| Quest (Adaptive) | ✓ | Partial (page-level) | Page-level | ✓ |

FASA fills this gap by leveraging an underexploited structural property of RoPE to achieve all four desiderata simultaneously. The remainder of this analysis explains how this is accomplished and validates whether the empirical results support these claims.

## 3. Technical Approach

### 3.1 Reader orientation (approachable technical breakdown)

FASA is a two-stage inference framework that sits between a Large Language Model's attention mechanism and its KV cache, dynamically selecting which past tokens deserve full attention computation based on signals extracted from Rotary Positional Encoding frequency patterns. The system solves the memory bandwidth bottleneck of long-context decoding by replacing expensive full-cache attention with a cheap proxy computation that identifies the top 256 most important tokens from tens of thousands, then performs precise attention only on that pruned subset — achieving near-lossless accuracy while reducing memory traffic by over 80%.

### 3.2 Big-picture architecture (diagram in words)

The FASA pipeline consists of three major components that operate sequentially during the decoding phase. **First, the Offline Calibration Module** runs once before deployment to identify which frequency chunks (subsets of RoPE dimensions) are "dominant" for each attention head — this produces a fixed index set $I_{dom}$ that is stored and reused across all inference tasks. **Second, the Token Importance Predictor (TIP)** operates at every decoding step: it takes the current query vector and extracts only the dominant frequency components, computes approximate attention scores using just those components, and ranks all cached tokens by importance to select the top-$N_{fac}$ candidates. **Third, the Focused Attention Computation (FAC)** receives the selected token indices from TIP, gathers the full-dimensional key and value vectors for only those tokens from the KV cache, and performs standard high-fidelity attention computation to produce the output. Figure 2 in the paper visualizes this flow: input tokens enter TIP for importance scoring, dominant FCs guide the selection, and FAC produces the next-token prediction using only the pruned subset.

### 3.3 Roadmap for the deep dive

- We begin with **Rotary Positional Encodings and Frequency Chunks** because FASA's entire mechanism rests on understanding how RoPE decomposes attention into orthogonal 2D subspaces — without this foundation, the notion of "dominant FCs" makes no sense.
- Next, we explain the **Contextual Agreement metric** that quantifies which frequency chunks matter, since this metric is the discovery tool that revealed functional sparsity and remains the criterion for offline calibration.
- We then detail the **Token Importance Predictor (TIP)** stage, showing how dominant FCs enable cheap, query-aware token scoring during live decoding without full attention computation.
- We follow with the **Focused Attention Computation (FAC)** stage, explaining how FASA preserves generation fidelity by performing exact attention on the selected subset while maintaining positional integrity.
- We describe the **two hardware-aware variants (FASA-M and FASA-C)** because they represent different engineering trade-offs for memory-constrained versus speed-constrained deployment scenarios.
- Finally, we present the **efficiency analysis and hyperparameter configurations** to ground the theoretical speedup claims in concrete numbers, including the critical $N_{tip}=16$ setting used across experiments.

### 3.4 Detailed, sentence-based technical breakdown

#### Foundational Insight: RoPE as Frequency-Chunk Decomposition

FASA's core innovation rests on reinterpreting Rotary Positional Encodings through a frequency-domain lens. RoPE embeds relative position information by rotating query and key vectors in 2D planes, where each plane corresponds to a specific rotational frequency. Formally, for a query vector $q_{t_1}$ and key vector $k_{t_2}$ at positions $t_1$ and $t_2$, the attention score is computed as:

$$A_{t_1,t_2} = (q_{t_1} R_{t_1})(k_{t_2} R_{t_2})^\top = q_{t_1} R_{\Delta t} k_{t_2}^\top$$

where $R_{\Delta t}$ is a rotation matrix parameterized solely by the relative offset $\Delta t = t_1 - t_2$. The critical observation is that this rotation matrix is block-diagonal, composed of $d/2$ independent 2×2 rotation blocks, where $d$ is the hidden dimension. Each 2D block is called a **Frequency Chunk (FC)**, and the $i$-th FC is defined as $v^{[i]} = (v_{2i}, v_{2i+1})^\top$ for any vector $v \in \mathbb{R}^d$. Each FC rotates at a unique angular frequency $\theta_i = B^{-2(i-1)/d}$, where $B$ is a predefined frequency base (typically 10000 in standard implementations). Lower dimension indices correspond to higher frequencies that rotate rapidly, while higher indices correspond to lower frequencies that rotate slowly.

The paper's pivotal discovery is that these FCs are not functionally equivalent. Section 3.2 articulates a **functional heterogeneity hypothesis**: high-frequency FCs (low dimensions) primarily construct robust positional patterns like recency bias and attention sinks, while low-frequency FCs (high dimensions) specialize in carrying semantic information and modeling long-range dependencies. This means a small subset of "Contextual FCs" could replicate the contextual selection behavior of a full attention head, while the remaining "Structural FCs" contribute primarily to positional structure without affecting which tokens are semantically relevant.

#### Quantifying Functional Sparsity: The Contextual Agreement Metric

To validate this hypothesis, the authors introduce the **Contextual Agreement (CA) metric**, which measures how well a single FC's attention pattern aligns with the full attention head's pattern. For a query $q_t$ and key matrix $K_{1:t}$ in attention head $(l, h)$, they define two raw score vectors: the standard full-head scores $\alpha_{l,h}$ and the single-FC scores $\alpha_{l,h}^{(i)}$. The full-head scores are computed as:

$$\alpha_{l,h}(q_t, K_{1:t}) = [q_t R_{t-1} (k_0)^\top, \cdots, q_t R_0 (k_t)^\top]^\top$$

The single-FC scores use only the 2D components of the $i$-th FC:

$$\alpha_{l,h}^{(i)}(q_t, K_{1:t}) = [q_{t}^{[i]} R_{t-1,\theta_i} k_{0}^{[i]\top}, \cdots, q_{t}^{[i]} R_{0,\theta_i} k_{t}^{[i]\top}]^\top$$

The CA score quantifies agreement by measuring the normalized intersection of their top-$K$ token index sets:

$$CA_{l,h,i}^K(q_t, K_{1:t}) = \frac{|\text{TopK-I}(\alpha_{l,h}(q_t, K_{1:t}), K) \cap \text{TopK-I}(\alpha_{l,h}^{(i)}(q_t, K_{1:t}), K)|}{K}$$

where $\text{TopK-I}(\alpha, K)$ retrieves the indices of the top-$K$ values in vector $\alpha$. A CA score of 1.0 means the single FC selects exactly the same top tokens as the full head; a score of 0.0 means no overlap.

Figure 1 presents heatmaps of CA scores across all frequency chunks (x-axis) and attention heads (y-axis) for Meta-Llama-3.1-8B and Mistral-7B. The visualization reveals striking **functional sparsity**: a few "dominant" FCs appear as bright vertical bands with CA scores exceeding 0.4, while the vast majority of FCs show negligible scores below 0.15. Table 9 quantifies this: dominant FCs (CA > 0.4) account for less than 1% of all FCs across models (ranging from 0.17% for Qwen2.5-7B to 0.79% for R1-Distill-Llama-8B), while non-dominant FCs (CA &lt; 0.15) comprise 89-95% of all FCs.

Table 1 demonstrates that reconstructing attention functionality from dominant FCs alone is highly effective. With just $F=8$ FCs selected (1/8 of the 64 total FCs per head) under a tight token budget of $K=64$, the compound CA score reaches 43.0%, surpassing SnapKV's 37.9% by 5.1 percentage points. At $F=16$ FCs (1/4 of dimensions), the score reaches 55.3% at $K=64$ and 81.4% at $K=2048$, confirming that a sparse subset of FCs can approximate full-head behavior.

#### Stage One: Token Importance Predictor (TIP)

The TIP stage leverages dominant FCs to efficiently estimate token importance during decoding without computing full attention. This stage has two phases: offline calibration and online prediction.

**Offline Calibration** identifies the dominant FC index set $I_{dom}^{l,h}$ for each attention head $(l, h)$ through a one-time search process. Given a small calibration dataset $\Omega$ and a target size $N_{tip}$, the objective is to find the subset of FCs that maximizes expected CA scores:

$$I_{dom}^{l,h} = \underset{I \subseteq \{0,\ldots,d/2-1\}, |I|=N_{tip}}{\text{argmax}} \mathbb{E}_{q,K \sim \Omega} \left[ \sum_{i \in I} CA_{l,h,i}^K(q, K) \right]$$

Algorithm 1 details this process: for each example in $\Omega$, at each token generation step, for each layer and head, the system computes full attention scores, then computes single-FC scores for each FC index, calculates CA scores using Equation 4, and stores them. After collecting CA scores across samples, the system averages them and selects the top-$N_{tip}$ indices. Crucially, Section 4.1 emphasizes this calibration is **task-agnostic and robust**: Table 6 shows minimal performance variation when calibrating on different datasets (Qasper, NarrativeQA, Musique, etc.), with Coefficient of Variation below 0.014 across all tasks. Section B.1 reveals the calibration uses just **a single data sample** from Qasper for LongBench experiments and a single MATH500 question for Long-CoT reasoning — yet the identified FCs generalize across all downstream tasks.

**Online Prediction** operates at each decoding step $t$. The full attention score decomposes into a sum over all $d/2$ frequency components: $\alpha_{l,h}(q_t, K_{1:t}) = \sum_{i=0}^{d/2-1} \alpha_{l,h,i}(q_t, K_{1:t})$. Instead of computing this expensive summation, TIP constructs an importance score vector $S_{t}^{l,h}$ by aggregating only contributions from pre-identified dominant frequencies:

$$S_{t}^{l,h} \triangleq \sum_{i \in I_{dom}^{l,h}} \alpha_{l,h,i}(q_t, K_{1:t})$$

This formulation strategically bypasses computation for non-dominant frequencies, reducing the dimensionality of the scoring operation from $d$ to $2 \times N_{tip}$ (since each FC is 2-dimensional). Based on these scores, TIP identifies the set of top-$N_{fac}$ most important token indices $T_t$ for the subsequent FAC stage:

$$T_t = \text{TopK-I}(S_{t}^{l,h}, N_{fac})$$

Section B.1 specifies the hyperparameter configuration: **$N_{tip} = 16$** dominant FCs are used consistently across all heads and layers for architectural simplicity and computational parallelism. This represents approximately 25% of the 64 FCs in standard 128-dimensional heads (or 15.6% in some configurations as noted in Section 5.4).

#### Stage Two: Focused Attention Computation (FAC)

Following token selection by TIP, the FAC stage executes high-fidelity attention computation exclusively on the selected subset $T_t$. For the current query vector $q_t$ at decoding step $t$, instead of using the full key and value matrices $(K_{1:t}, V_{1:t})$ from the entire past context, FAC first gathers the keys and values corresponding to the indices in $T_t$:

$$K_{T_t} = \text{Gather}(K_{1:t}, T_t), \quad V_{T_t} = \text{Gather}(V_{1:t}, T_t)$$

where the $\text{Gather}(\cdot)$ operation selects rows from the original matrices specified by the index set $T_t$. The attention scores for each head $(l, h)$ are then computed using only these selected keys:

$$\hat{\alpha}_{FAC}^{l,h} = \text{Softmax}\left(\frac{q_t K_{T_t}^\top}{\sqrt{d}}\right), \quad O_{t}^{l,h} = \hat{\alpha}_{FAC}^{l,h} V_{T_t}$$

Section 4.2 emphasizes a critical design detail: **the original absolute positions of tokens in $T_t$ are preserved**. This directly maintains the integrity of their RoPE position embeddings and prevents the performance degradation associated with positional distortion that would occur if selected tokens were re-indexed contiguously. The FAC stage functions as a high-fidelity computational filter, restricting full-precision attention to the most salient tokens while achieving a balance between efficiency and accuracy.

Section D.2 validates two crucial design principles through ablation. First, FC-based scores $S_{t}^{l,h}$ serve as a **proxy for ranking, not a substitute for attention weights** — directly using them as attention probabilities causes catastrophic performance degradation, confirming their role as selectors rather than approximators. Second, **Frequency Chunks are indivisible functional units** — attempting to select individual dimensions instead of FC pairs causes model failure because RoPE encodes position through coupled dimension pairs, and disrupting these pairs severs the positional encoding.

#### Two Hardware-Aware Variants: FASA-M and FASA-C

Section 4.3 introduces two specialized implementations that trade off memory versus speed based on deployment constraints.

**FASA-M (Memory-Optimized)** minimizes GPU memory footprint by strategically offloading the value cache and non-dominant key components to CPU memory. Only the dominant parts of the Key cache ($C_{key}^{dom}$), required for TIP scoring, remain on GPU. The non-dominant key parts ($C_{key}^{nondom}$) and entire Value cache ($C_{val}$) are stored in CPU memory. During FAC, once critical token indices $T_t$ are identified, only the small required subsets are transferred from CPU to GPU for final attention calculation. Algorithm 2 details this "just-in-time" data transfer. The GPU memory footprint formula is:

$$\text{Mem}_{GPU} \approx N_{layers} \times \left( L \times d_{dom} + b \times d_{nondom} + b \times d \right) \times \text{bytes\_per\_param}$$

where $L$ is total sequence length, $b$ is token budget, $d_{dom}$ is dominant FC dimension, and $d_{nondom}$ is non-dominant dimension. Compared to full KV cache occupying $N_{layers} \times L \times 2d \times \text{bytes\_per\_param}$, FASA-M achieves up to **8× compression** when $d_{dom}$ is 25% of $d$ and budget $b$ is 10% of $L$. Section 5.3 notes that CPU-GPU transfer latency can be mitigated through prefetching techniques that asynchronously load required KV pairs in advance.

**FASA-C (Computation-Optimized)** prioritizes inference speed by retaining the full cache on-GPU but accessing only a sparse subset of key states. This variant drastically reduces memory I/O for acceleration without CPU offloading. Section 5.3 reports FASA-C delivers **2.56× speedup** with $N_{tip}=16$ under 64K sequence length, implemented with Triton kernels based on Ribar et al. (2024). Figure 7 compares memory and latency: FASA-C shows lower latency than FASA-M (no CPU transfer overhead) while FASA-M shows lower memory footprint at long sequences.

#### Efficiency Analysis and Computational Complexity

Section 4.4 provides formal complexity analysis. At generation step $t$, standard attention computing $q_t K_{1:t}^\top$ has complexity $O(td)$ and multiplying value states with attention scores is $O(td)$ per head. For FASA:

- **TIP stage complexity**: $O(2tN_{tip})$ because each FC takes 2 dimensions and operates in low-dimensional subspaces
- **FAC stage complexity**: $O(N_{fac}d)$ because attention is performed on reduced set of $N_{fac}$ tokens
- **Overall complexity**: $O(2tN_{tip} + 2N_{fac}d)$, assuming top-k selection overhead is negligible

The theoretical speedup at decoding stage is:

$$\text{Speedup} = \frac{2td}{2tN_{tip} + 2N_{fac}d} = \frac{1}{N_{tip}/d + N_{fac}/t} \approx \frac{d}{N_{tip}} \quad \text{if } N_{fac} \ll t$$

For memory movement, standard attention loads $2tm$ bytes from KV cache at step $t$ (where $m$ is byte size per state vector), while FASA accesses only $t(2N_{tip}/d \times m)$ bytes for TIP (keys only) and $2N_{fac}m$ bytes for FAC. The fraction FASA must load is:

$$\frac{2tmN_{tip}/d + 2N_{fac}m}{2tm} = \frac{N_{tip}}{d} + \frac{N_{fac}}{t} \approx \frac{N_{tip}}{d} \quad (N_{fac} \ll t)$$

With $N_{tip}=16$ and $d=128$, this yields approximately **12.5% memory traffic** compared to full attention, directly alleviating the memory-bound constraint shown in Figure 3 where decoding constitutes 90% of latency at 32K context.

#### Key Hyperparameters and Configuration Summary

Section B.1 provides complete experimental configurations. The **dominant FC count $N_{tip}=16$** is used uniformly across all heads and layers. For LongBench, calibration uses **one sample from Qasper**; for Long-CoT reasoning, calibration uses **one sample from MATH500**. Maximum sequence lengths are configured as: **32K for AIME24** (higher reasoning complexity), **16K for MATH500**, **127.5K for Llama3/Qwen2.5 on LongBench**, and **31.5K for Mistral-7B on LongBench**. The token retention budget $N_{fac}$ varies by experiment: Table 2 uses **256 tokens** for LongBench-V1; Table 3 tests budgets of **300, 500, 700, 1000** for MATH500 and **500, 1000, 1500, 2000, 2500** for AIME24; Figure 5 explores the trade-off between $N_{tip}$ (8, 10, 12, 14 FCs) and $N_{fac}$ (100-900 tokens).

Section 5.4 presents ablation studies validating these choices. Table 5 shows robustness to calibration window size $K$: performance is largely insensitive to $K$ (tested values: 128, 256, 512, 1024), with smaller $K$ sometimes yielding slightly superior results, suggesting inherent attention sparsity provides robust signal even from small windows. Figure 5 demonstrates the trade-off between $N_{tip}$ and $N_{fac}$: optimal performance can be achieved with high-precision selection (large $N_{tip}$) and small budget, or lenient selection (small $N_{tip}$) compensated by larger budget — empirically, **10 dominant FCs (15.6% of dimensions) with $N_{fac}=500$** matches FKV performance on TREC.

#### Extension to Non-RoPE Architectures

Section 6 investigates generalizability beyond standard RoPE. The feasibility hinges on whether functional sparsity emerges under alternative positional encoding schemes. The paper examines **ALiBi (Attention with Linear Biases)** and **Partial-RoPE (in Multi-head Latent Attention, MLA)**. ALiBi encodes relative distances using head-specific linear biases added to attention logits, while MLA applies RoPE only to a specific partition of head dimensions. Figures 8 and 9 show CA score heatmaps for Baichuan-13B-Chat (ALiBi) and DeepSeek-V2-Lite-Chat (Partial-RoPE), revealing functional sparsity persists at the head dimension level in both cases. For ALiBi, heads indexed 19-31 exhibit regular sparsity patterns while other dimensions show high CA scores (~0.7). Tables 7 and 8 demonstrate FASA achieves performance matching or surpassing FKV on both architectures without significant trade-offs, establishing broad applicability beyond standard RoPE configurations.

#### Implementation Details and Integration

Section B.4 describes the implementation approach. FASA is built on HuggingFace Transformers with **non-invasive monkey patching** that intercepts the FlashAttention2 forward pass in the model's `modeling.py` file. The dominant FC indices are stored in a globally accessible dictionary shared across all layers and heads. The TIP token selection logic is encapsulated in a `core_module_with_padding` function. Figure 14 shows the integration requires inserting just **a single line of code** (the token selection call) into the original attention function during the decoding stage (when `query_states.shape[2] == 1`), making FASA highly portable. The FAC stage seamlessly integrates with the standard FlashAttention API, leveraging its performance while enabling sparse computation. This minimal intrusion design contrasts with methods like LoKi that require storing auxiliary projection matrices, as noted in Section 2's related work comparison.

## 4. Key Insights and Innovations

This section isolates the fundamental contributions of FASA from the incremental engineering optimizations. While many methods attempt to compress the KV cache, FASA introduces distinct theoretical and methodological shifts. We distinguish between **fundamental innovations** (new discoveries about model architecture or new paradigms for efficiency) and **incremental improvements** (better hyperparameters or engineering tweaks). The following four insights represent the core novelty of this work.

### 4.1 Fundamental Innovation: RoPE-Induced Functional Sparsity

Most prior work on attention sparsity (e.g., H2O, SnapKV) observes that *tokens* are sparse—meaning only a few past tokens matter for the current prediction. FASA makes a deeper, structural claim: **sparsity exists within the positional encoding mechanism itself.**

*   **What Changed:** Previous methods treat all dimensions of a key vector as equally important for determining token saliency. FASA demonstrates that within Rotary Positional Encodings (RoPE), the $d/2$ frequency chunks (FCs) are not functionally equivalent. As shown in **Figure 1**, heatmaps of Contextual Agreement (CA) reveal bright vertical bands indicating that a tiny subset of FCs consistently drives contextual awareness, while the rest maintain positional structure.
*   **Why It Matters:** This shifts the optimization target from *tokens* to *frequencies*. Table 9 quantifies this sparsity: dominant FCs (CA score $> 0.4$) account for **less than 1%** of all FCs across models (e.g., 0.17% for Qwen2.5-7B, 0.79% for R1-Distill-Llama-8B). This is not a learned heuristic that might fail on out-of-distribution data; it is an **intrinsic property of the RoPE mechanism**.
*   **Evidence of Universality:** The paper validates that this sparsity is task-agnostic. Table 10 shows that the overlap of dominant FCs identified across different calibration datasets (e.g., Qasper vs. GovReport) consistently exceeds **70%**. This universality is what enables the "one-time offline calibration" described in Section 4.1 to work without retraining for every new domain.

### 4.2 Methodological Breakthrough: Training-Free Query-Awareness

A central tension in KV cache optimization is the trade-off between **adaptability** and **cost**. Static methods (like Stream) are cheap but blind to the query; learning-based methods are query-aware but require training and risk poor generalization. FASA breaks this trade-off.

*   **What Changed:** FASA achieves query-awareness without trainable parameters. It uses the dominant FCs identified in Insight 4.1 as a **computationally free proxy**. Instead of training a neural network to predict token importance, it computes attention scores using only the dominant frequency subspace (the TIP stage).
*   **Why It Matters:** This allows dynamic adaptation during decoding without the overhead of backpropagation or auxiliary model weights. Table 2 demonstrates the impact: on LongBench-V1, FASA reaches nearly **100% of full-KV performance** retaining only 256 tokens, whereas static baselines like Stream drop by 10-20% due to irreversible information loss.
*   **Mechanism Distinction:** Unlike SnapKV, which estimates importance once during prefill, FASA recalculates importance at every decoding step using the current query's interaction with dominant FCs. This captures the evolving relevance of tokens as the reasoning chain extends, which is critical for tasks like Long-CoT where early tokens may become irrelevant later.

### 4.3 Design Principle: Indivisibility of Frequency Chunks

A subtle but critical innovation lies in the unit of selection. Many dimension-reduction methods (e.g., SparQ) select individual dimensions based on magnitude. FASA proves this is structurally unsound for RoPE models.

*   **What Changed:** FASA enforces that Frequency Chunks must be selected as **indivisible 2D pairs**. Section D.2 explicitly validates this through ablation: attempting to select individual dimensions instead of FC pairs causes **catastrophic performance degradation**.
*   **Why It Matters:** This protects the mathematical integrity of RoPE. As defined in Section 3.1, RoPE applies rotation matrices $R_{m,\theta_i}$ to 2D subspaces $(v_{2i}, v_{2i+1})^\top$. Splitting these pairs severs the positional encoding, rendering the model unable to distinguish token order.
*   **Educational Takeaway:** This insight serves as a warning for future efficiency research on RoPE-based models: **optimizations must respect the coupled nature of rotational dimensions.** FASA's success relies on treating the FC as the atomic unit of sparsity, not the individual scalar dimension.

### 4.4 System Design: Orthogonality to Compression Paradigms

Finally, FASA positions itself not as a replacement for all other optimization strategies, but as a modular component that complements them.

*   **What Changed:** Many compression methods compete for the same resource (e.g., two methods trying to prune tokens). FASA operates on *which* tokens to keep, while methods like PyramidKV operate on *how many* tokens to allocate per layer.
*   **Why It Matters:** This modularity allows for compounded efficiency gains. Table 4 shows that integrating FASA with PyramidKV yields consistent performance improvements (e.g., **+0.7% on Qasper** at budget 256) compared to FASA alone.
*   **Significance:** This orthogonal design means FASA can be adopted incrementally. A deployment using layer-wise budget allocation can add FASA to further reduce memory bandwidth without conflicting with existing policies. It transforms FASA from a standalone model modification into a plug-in optimization compatible with the broader ecosystem of KV cache management tools.

## 5. Experimental Analysis

This section examines whether FASA's empirical results substantiate its theoretical claims. We analyze the evaluation methodology, dissect the quantitative results with specific numbers, assess the strength of evidence supporting each claim, and identify conditions where performance varies. The goal is to help you understand not just *what* the numbers show, but *why* certain patterns emerge and what they imply for real-world deployment.

### 5.1 Evaluation Methodology

#### Datasets and Task Paradigms

The paper evaluates FASA across three distinct task paradigms, each targeting different aspects of long-context capability. This multi-paradigm approach is critical because a method might excel at retrieval-based QA but fail at generative reasoning — and vice versa.

**Long-Context Understanding (LongBench-V1).** This benchmark tests the model's ability to identify and reason over critical information within lengthy static contexts. Section 5.1 specifies it includes diverse real-world tasks: Single-Doc QA (e.g., Qasper, NarrativeQA), Multi-Doc QA (e.g., HotpotQA, 2WikiMQA), Summarization (e.g., GovReport, QMSum), Synthetic tasks, and Code completion (e.g., LCC, RepoBench). Sequence lengths range from 4K to over 100K tokens. Section B.3 details the evaluation metrics: **f1 score** for question-answering, **rouge_score** for summarization, and **code_sim_score** for code tasks. The final reported score is the average across all constituent tasks.

**Long-Sequence Modeling.** This paradigm evaluates generative fidelity over long dependencies using perplexity (PPL) on three corpora: **PG-19** (book-length texts from Project Gutenberg), **WikiText** (Wikipedia articles), and **C4** (Common Crawl web text). Section B.3 defines perplexity formally: for a sequence $W = (w_1, w_2, \ldots, w_N)$,

$$\text{PPL}(W) = \exp\left(-\frac{1}{N} \sum_{i=1}^{N} \log P(w_i | w_{&lt;i})\right)$$

Lower PPL indicates better predictive confidence. This metric is sensitive to whether the model maintains coherent attention over long ranges — if critical tokens are evicted, PPL increases sharply.

**Long Chain-of-Thought Reasoning (Long-CoT).** This is the most demanding paradigm, testing whether FASA can preserve the "thought traces" required for multi-step reasoning. Section 5.1 specifies two benchmarks: **MATH500** (high school competition math problems with step-by-step solutions) and **AIME24** (American Invitational Mathematics Examination, representing significantly higher complexity). Unlike LongBench, these require *generating* long outputs (thousands of tokens) while maintaining logical coherence. Section B.3 clarifies the metric: **pass@1**, where for AIME24, the result is determined by checking if at least one correct answer exists within $k=16$ independent generations per question.

#### Baselines and Comparison Framework

Section 5.1 categorizes baselines into two groups, establishing a rigorous comparison framework:

**State-of-the-Art Token Eviction Methods:**
- **Stream** (Xiao et al., 2024): Preserves initial tokens (start_size=8) and recent tokens (recent_size=budget-8). Represents static, rule-based eviction.
- **SnapKV** (Li et al., 2024): Estimates importance during prefill using accumulated attention scores (maxpool strategy, window=32, kernel=7). Represents one-time adaptive filtering.
- **Quest** (Tang et al., 2024): Organizes cache into pages (page_size=16) and retrieves based on query-page similarity. Represents coarse-grained dynamic retrieval.
- **H2O** (Zhang et al., 2023): Heavy-hitter oracle that evicts tokens based on historical attention scores.
- **RKV** (Cai et al., 2025a): Specialized for reasoning tasks, balances recent and important tokens ($\lambda=0.1$).

**Upper Bounds:**
- **FKV (Full KV)**: Standard inference with complete, uncompressed KV cache. Serves as the absolute performance ceiling (no information loss).
- **Oracle**: Assumes ideal knowledge to retain only the most critical tokens based on full-head scores. Represents a pragmatic upper bound for eviction-based methods.

Section B.1 notes a critical experimental design choice: **all methods forgo KV cache optimizations during prefilling**, isolating the performance impact of decode-stage acceleration. This ensures fair comparison — some methods might gain advantage from prefill optimizations that FASA doesn't use.

#### Experimental Configuration

Section B.1 provides precise hyperparameter settings. For FASA:
- **$N_{tip} = 16$** dominant FCs used uniformly across all heads and layers
- **Calibration**: Single sample from Qasper for LongBench; single sample from MATH500 for Long-CoT
- **Token budget ($N_{fac}$)**: 256 tokens for LongBench-V1 (Table 2); 300-1000 for MATH500 and 500-2500 for AIME24 (Table 3)
- **Maximum sequence lengths**: 32K for AIME24, 16K for MATH500, 127.5K for Llama3/Qwen2.5 on LongBench, 31.5K for Mistral-7B on LongBench

For baselines, configurations follow original papers or represent "fair and strong setups" (Section B.1). For example, SnapKV is adapted for long-generation by re-applying filtering every $n$ generated tokens (following Cai et al., 2025a), since its original design performs one-time filtering unsuited for extended generation.

### 5.2 Main Quantitative Results

#### Long-Context Understanding: Near-Oracle Performance with Extreme Compression

Table 2 presents the core LongBench-V1 results across five models (Llama3.2-3B, Qwen2.5-7B, Mistral-7B-v0.3, Llama3.1-8B, Qwen2.5-14B-1M). The numbers reveal FASA's most striking claim: **near-lossless performance under extreme token budgets**.

Consider the Llama3.2-3B results. FKV (full cache) achieves an average score of 42.2. FASA achieves **41.5** — a drop of only **0.7 points (1.7% relative degradation)** — while retaining just 256 tokens. In stark contrast:
- **Quest** scores 25.5 (16.7 points below FKV, 39.6% degradation)
- **Stream** scores 31.8 (10.4 points below FKV, 24.6% degradation)
- **SnapKV** scores 37.0 (5.2 points below FKV, 12.3% degradation)

The pattern holds across all models. For Qwen2.5-7B, FASA scores **47.9** vs FKV's 47.8 — actually **surpassing the full-cache baseline by 0.1 points**. Section 5.2 attributes this to "mitigation of attentional distraction from irrelevant tokens," a hypothesis corroborated by the Oracle baseline also outperforming FKV sometimes (48.7 vs 47.8). This is a critical insight: **more context is not always better** if it introduces noise.

For Mistral-7B-v0.3, FASA achieves **47.8** vs FKV's 47.4 (+0.4), while Quest drops to 38.6 (-8.8) and Stream collapses to 26.7 (-20.7). The Llama3.1-8B results show FASA at **48.2** vs FKV's 48.7 (-0.5), with SnapKV at 45.0 (-3.7) and Quest at 35.4 (-13.3).

The Qwen2.5-14B-1M results (designed for 1M context) are particularly telling: FASA scores **49.2** vs FKV's 50.3 (-1.1), while Quest plummets to 34.9 (-15.4) and Stream to 35.3 (-15.0). This demonstrates FASA's robustness even on models specifically fine-tuned for extreme context lengths.

**Key Takeaway:** FASA consistently achieves within 1% of FKV performance across all models, while baselines degrade by 10-40%. The 256-token budget represents extreme compression — for a 32K context, this is retaining less than 1% of tokens.

#### Long-Sequence Modeling: Perplexity Matches Full Cache Under Aggressive Sparsity

Figure 4 presents perplexity results across Wikitext, PG19, and C4 for four models (Llama-3.2-3B, Meta-Llama-3.1-8B, Mistral-7B-v0.3, Qwen2.5-14B). The x-axis shows token sparsity (retained ratio from 0.1 to 0.6), and the y-axis shows perplexity (lower is better).

The graphs reveal two critical patterns:

**First, FASA's perplexity curve closely tracks FKV across all sparsity levels.** For Llama-3.2-3B on Wikitext, at 0.2 sparsity (20% tokens retained), FASA achieves PPL ≈ 12.5, nearly identical to FKV's ≈ 12.3. Quest and Stream show PPL ≈ 14.5 and ≈ 15.0 respectively — significantly worse. This pattern repeats across all three corpora and all four models.

**Second, Stream's performance degrades catastrophically as sparsity increases.** Section 5.2 explains this: Stream's fixed-rule approach relying on "attention sinks" (initial tokens) and recency bias "severely compromises its ability to capture long-range dependencies." On PG19 for Llama-3.2-3B, Stream's PPL increases from ≈ 17.5 at 0.5 sparsity to ≈ 22.5 at 0.1 sparsity, while FASA remains stable around 17.5-18.0.

**Third, FASA occasionally matches Oracle performance.** On C4 for Meta-Llama-3.1-8B, at 0.3 sparsity, FASA achieves PPL ≈ 11.5, matching Oracle's ≈ 11.4 and outperforming Quest's ≈ 12.5. This validates the claim that dominant FCs provide a proxy nearly as effective as oracle knowledge for token selection.

**Key Takeaway:** FASA maintains generative fidelity comparable to full cache even at 10-20% token retention, while static baselines suffer exponential perplexity growth as context is compressed.

#### Long-CoT Reasoning: Preserving Logical Dependencies Under Budget Constraints

Table 3 presents the most demanding evaluation: mathematical reasoning on MATH500 and AIME24. This tests whether FASA can maintain the "fragile thread" of chain-of-thought reasoning across thousands of generated tokens.

**MATH500 Results (DeepSeek-R1-Distill-Llama-8B).** FKV achieves 72.4% pass@1. FASA achieves **71.8%** at budget 1000 — only 0.6 points below full cache. The progression across budgets shows FASA's graceful degradation:
- Budget 300: 62.2%
- Budget 500: 68.8%
- Budget 700: 69.4%
- Budget 1000: 71.8%

Contrast this with baselines:
- **SnapKV**: 21.6% at 300, 54.6% at 1000 (17.8 points below FKV)
- **Stream**: 9.6% at 300, 47.4% at 1000 (25.0 points below FKV)
- **H2O**: 6.8% at 300, 42.8% at 1000 (29.6 points below FKV)
- **RKV** (specialized for reasoning): 24.0% at 300, 57.0% at 1000 (15.4 points below FKV)

Section 5.2 emphasizes: "SnapKV's accuracy collapses to 21.6, a stark contrast to the FKV's 72.4, demonstrating a fundamental failure to sustain the very logical dependencies required for reasoning." This is because SnapKV's prefill-stage estimation cannot adapt as the reasoning chain evolves — tokens critical at step 500 may be irrelevant at step 2000, and vice versa.

**AIME24 Results (DeepSeek-R1-Distill-Llama-8B).** This is the hardest benchmark. FKV achieves 43.9% pass@1. FASA achieves **38.0%** at budget 2500 — 5.9 points below FKV but far surpassing all baselines:
- **SnapKV**: 29.1% (14.8 points below FKV)
- **Stream**: 15.3% (28.6 points below FKV)
- **H2O**: 20.0% (23.9 points below FKV)
- **RKV**: 23.3% (20.6 points below FKV)

The output length statistics are revealing. FKV generates 13,392 total tokens (161 prefill + 13,231 decoding). FASA generates 17,327 tokens (161 prefill + 17,166 decoding) — **longer outputs** while maintaining accuracy. Section 5.3 notes this is significant: "Some compression methods, like H2O, induce generative verbosity, imposing an overlooked computational burden. Conversely, others, such as Stream, prematurely terminate generation." FASA maintains output lengths "nearly identical to FKV while preserving high performance."

**Qwen-14B and Qwen-32B Results.** The pattern holds at larger scales. For DeepSeek-R1-Distill-Qwen-14B on MATH500, FASA achieves **91.2%** vs FKV's 92.4% (-1.2), while SnapKV reaches 79.4% (-13.0) and Stream 64.4% (-28.0). On AIME24, FASA achieves **63.3%** vs FKV's 66.6% (-3.3), while SnapKV reaches 52.7% (-13.9) and Stream 29.3% (-37.3).

**Key Takeaway:** FASA preserves reasoning capability under extreme compression where baselines catastrophically fail. The dynamic, query-aware token selection is essential for maintaining logical coherence across long generation chains.

#### Efficiency Gains: Memory and Speed Trade-Offs

Section 5.3 and Figure 7 quantify the efficiency benefits of FASA's two variants.

**FASA-M (Memory-Optimized).** Section 4.3 states this variant achieves **8× compression** of the KV cache. Figure 7 shows memory usage vs sequence length: at 64K tokens, FKV requires ≈ 120 GB, while FASA-M requires ≈ 15 GB — an 8× reduction. The memory footprint formula from Section D.1 is:

$$\text{Mem}_{GPU} \approx N_{layers} \times (L \times d_{dom} + b \times d_{nondom} + b \times d) \times \text{bytes\_per\_param}$$

With $d_{dom} = 25\%$ of $d$ and budget $b = 10\%$ of $L$, this yields the 8× savings. Section 5.3 notes CPU-GPU transfer introduces "slight latency overhead" mitigated by prefetching.

**FASA-C (Computation-Optimized).** This variant achieves **2.56× speedup** on AIME24 with $N_{tip}=16$ under 64K sequence length (Abstract and Section 5.3). Figure 7 shows latency vs sequence length: at 64K, FKV requires ≈ 0.14 seconds per token, while FASA-C requires ≈ 0.055 seconds — a 2.55× improvement matching the claimed 2.56×. Section 4.4's theoretical speedup formula is:

$$\text{Speedup} = \frac{1}{N_{tip}/d + N_{fac}/t} \approx \frac{d}{N_{tip}} \quad \text{if } N_{fac} \ll t$$

With $d=128$ and $N_{tip}=16$, this yields 8× theoretical speedup, but practical overheads (top-k selection, memory gathering) reduce this to 2.56×.

**Memory Movement Reduction.** Section 4.4 quantifies the fraction of KV cache FASA must load:

$$\frac{N_{tip}}{d} + \frac{N_{fac}}{t} \approx \frac{N_{tip}}{d} \quad (N_{fac} \ll t)$$

With $N_{tip}=16$ and $d=128$, this is **12.5% memory traffic** compared to full attention. The Abstract states FASA achieves 2.56× speedup "using just 18.9% of the cache on AIME24" — this 18.9% figure likely accounts for both key and value cache access patterns.

**Key Takeaway:** FASA delivers substantial efficiency gains (8× memory reduction, 2.56× speedup) while maintaining accuracy. The two variants allow deployment teams to optimize for their specific constraint (VRAM vs latency).

### 5.4 Critical Assessment: Do the Experiments Support the Claims?

#### Claims That Are Strongly Supported

**Claim 1: FASA achieves near-oracle accuracy with extreme compression.** Table 2 provides robust evidence: across 5 models and 15+ tasks, FASA stays within 1% of FKV with only 256 tokens. The consistency across architectures (Llama, Qwen, Mistral) and scales (3B to 32B) strengthens this claim.

**Claim 2: FASA outperforms all token-eviction baselines.** Table 2 and Table 3 show FASA surpassing Stream, SnapKV, Quest, H2O, and RKV by substantial margins (10-40% relative improvement on LongBench, 15-30% on Long-CoT). The baselines are state-of-the-art and fairly configured (Section B.1), making this comparison credible.

**Claim 3: Dominant FCs are universal and task-agnostic.** Table 10 shows >70% overlap of dominant FCs across different calibration datasets. Table 6 shows minimal performance variation when calibrating on different tasks. Figure 1 and Figures 10-11 show consistent sparsity patterns across models and layers. This evidence is compelling.

**Claim 4: FASA delivers substantial efficiency gains.** Figure 7 quantifies 8× memory reduction (FASA-M) and 2.56× speedup (FASA-C). The theoretical analysis in Section 4.4 aligns with empirical measurements. This claim is well-supported.

#### Claims That Are Conditionally Supported

**Claim 5: FASA works on non-RoPE architectures.** Section 6 shows functional sparsity exists in ALiBi and Partial-RoPE, but Tables 7-8 reveal **slight performance degradation** (1-1.5 points on ALiBi). The claim holds, but with a caveat: FASA is optimized for RoPE, and performance on alternatives may be marginally lower.

**Claim 6: One-sample calibration suffices.** Section B.1 states calibration uses "just a single data sample." Table 6 supports this with low CV scores. However, all experiments use carefully selected samples (Qasper for LongBench, MATH500 for CoT). The claim might not hold for arbitrary, poorly chosen samples — this boundary condition is not tested.

#### Potential Limitations and Unanswered Questions

**Limitation 1: Long-CoT results show non-trivial degradation.** On AIME24, FASA achieves 38.0% vs FKV's 43.9% (5.9 points, 13.4% relative degradation). While this surpasses baselines, it is not "near-oracle" in absolute terms. The Abstract claims "near-oracle accuracy," but Table 3 shows Oracle achieves 36.0% on AIME24 for Llama-8B — FASA at 38.0% actually exceeds Oracle, which seems anomalous. This discrepancy warrants scrutiny.

**Limitation 2: Efficiency numbers depend on sequence length.** Figure 3 shows decoding latency dominates at 32K (90% of total), but at shorter sequences (e.g., 4K), prefill latency may dominate. FASA optimizes decoding, so benefits diminish for shorter contexts. The paper does not quantify the "break-even" sequence length where FASA becomes advantageous.

**Limitation 3: CPU-GPU transfer overhead for FASA-M.** Section 5.3 acknowledges "slight latency overhead" from CPU-GPU transfer, mitigated by prefetching. However, no quantitative data is provided on this overhead. For latency-sensitive applications, this could be a concern.

**Limitation 4: No failure cases reported.** The paper does not identify tasks or conditions where FASA fails. Are there domains where dominant FCs do not generalize? What happens with highly specialized technical content (e.g., legal contracts, medical records)? The absence of failure analysis limits understanding of boundary conditions.

#### Overall Assessment

The experiments **convincingly support the core claims** about FASA's effectiveness on RoPE-based models for long-context tasks. The multi-paradigm evaluation (LongBench, perplexity, Long-CoT), multiple models (5 architectures, 3B-32B scale), and comprehensive baselines (5+ methods) provide robust evidence. The ablation studies validate design choices and demonstrate robustness.

However, the claims about **universal applicability** (non-RoPE architectures) and **one-sample calibration** are supported with caveats. The efficiency gains are real but context-length-dependent. The paper would benefit from explicit failure case analysis and break-even sequence length quantification.

**For practitioners:** FASA is a strong choice for RoPE-based models processing 16K+ token contexts, especially for reasoning tasks where baselines fail catastrophically. For shorter contexts or non-RoPE models, the benefits may be marginal. The two variants (FASA-M, FASA-C) allow tuning for specific deployment constraints.

## 6. Limitations and Trade-offs

No efficiency method is universally optimal. FASA makes deliberate design choices that yield substantial benefits in specific scenarios while introducing constraints in others. This section critically examines the assumptions underlying FASA, identifies problem settings where benefits diminish or disappear, quantifies computational trade-offs, and surfaces open questions that remain unresolved. Understanding these limitations is essential for practitioners deciding whether FASA fits their deployment context.

### 6.1 Structural Assumptions: Dependence on RoPE Mechanics

FASA's core mechanism rests on a foundational assumption: **functional sparsity emerges from the RoPE positional encoding structure**. This assumption is well-supported for standard RoPE models (Section 3.3, Figure 1), but it introduces constraints for alternative architectures.

**Assumption 1: Frequency chunks are indivisible functional units.** Section D.2 validates through ablation that selecting individual dimensions instead of FC pairs causes "catastrophic performance degradation." This is because RoPE applies rotation matrices to coupled 2D subspaces $(v_{2i}, v_{2i+1})^\top$. However, this assumption limits FASA's applicability to models where positional encoding follows this paired-dimension structure. Models using alternative positional schemes (e.g., learned absolute positions, ALiBi without RoPE components) may not exhibit the same sparsity patterns.

**Assumption 2: Dominant FCs are layer-invariant enough for uniform configuration.** Section B.1 specifies that FASA uses a **uniform $N_{tip}=16$ across all heads and layers** for "architectural simplicity and computational parallelism." However, Figure 13 reveals that FC saliency patterns **vary significantly across layers** — Layer 0 shows different dominant FC distributions than Layer 31 or Layer 41. The paper does not quantify the performance cost of this uniformity assumption. A layer-adaptive configuration might yield better accuracy but would increase calibration complexity and storage overhead.

**Assumption 3: Functional sparsity is task-agnostic.** Table 10 shows dominant FC overlap exceeds 70% across different calibration datasets, supporting task-invariance. However, Table 6 tests only five tasks (Qasper, NarrativeQA, Musique, GovReport, Self) from the LongBench suite. The paper does not evaluate highly specialized domains such as **legal contracts, medical records, or scientific code** where attention patterns may differ substantially from general QA or summarization. The claim of universality remains untested for domain-specific long-context applications.

**Evidence of limitation:** Section 6 extends FASA to non-RoPE architectures (ALiBi and Partial-RoPE), but Tables 7-8 reveal **performance degradation on ALiBi models**: Qasper drops from 9.11 (FKV) to 7.80 (FASA), a 1.31-point decrease. TREC drops from 23.00 to 21.50. While functional sparsity exists in these architectures (Figures 8-9), the sparsity patterns are less pronounced than in standard RoPE, reducing FASA's effectiveness. This suggests FASA is **optimized for RoPE**, with diminished returns on alternatives.

### 6.2 Context-Length Dependency: Benefits Diminish at Shorter Sequences

FASA targets the **memory-bound decoding bottleneck** that emerges at long context lengths. Figure 3 quantifies this: at 32K tokens, decoding constitutes ~90% of total latency. However, this ratio shifts dramatically at shorter sequences.

**The break-even point is unspecified.** The paper does not quantify the sequence length at which FASA's decoding optimization begins to outweigh its overhead. At 4K tokens, prefill latency may dominate total inference time (Figure 3 shows prefill at 0.1s vs decode at 0.1s at 1K tokens). Since Section B.1 explicitly states **"all methods forgo KV cache optimizations during prefilling,"** FASA provides no benefit during the prefill phase. For applications with moderate context lengths (e.g., 4K-8K tokens for typical chat or document QA), the efficiency gains may be marginal or negative when accounting for TIP computation overhead.

**Memory savings scale with sequence length.** Section D.1's memory footprint formula for FASA-M is:

$$\text{Mem}_{GPU} \approx N_{layers} \times (L \times d_{dom} + b \times d_{nondom} + b \times d) \times \text{bytes\_per\_param}$$

The savings come from reducing $L \times d$ to $L \times d_{dom}$ for dominant keys. At short $L$, the absolute memory saved is small, while the CPU-GPU transfer overhead remains constant. For a 4K context with $d_{dom}=32$ (25% of 128), the savings are approximately $4096 \times 96 \times \text{bytes}$ per layer — meaningful but not transformative. At 64K, the same formula yields 16× more savings, justifying the architectural complexity.

**Practical implication:** FASA is most beneficial for **16K+ token contexts** where decoding latency dominates. For shorter contexts, simpler methods (e.g., sliding window attention) may offer better efficiency-accuracy trade-offs with lower implementation complexity.

### 6.3 Computational Overheads: The Cost of Token Selection

FASA introduces two computational stages (TIP and FAC) that replace standard attention. While Section 4.4 provides theoretical complexity analysis, several overheads are acknowledged but not fully quantified.

**Top-k selection overhead is assumed negligible.** Section 4.4 states "assuming the complexity of selecting the top-k tokens is small" when deriving the speedup formula. However, top-k selection on tens of thousands of tokens at every decoding step is non-trivial. Standard top-k algorithms have $O(t \log k)$ complexity, which at $t=32K$ and $k=256$ represents meaningful computation. The paper does not profile this overhead separately from TIP computation.

**CPU-GPU transfer latency for FASA-M lacks quantitative data.** Section 5.3 acknowledges that FASA-M's CPU offloading "introduces a slight latency overhead" that "can be effectively mitigated by prefetching techniques." However, **no measurements are provided** for:
- Absolute transfer latency per decoding step
- Prefetching effectiveness under varying workloads
- Impact on tail latency (p99) vs average latency

For latency-sensitive applications (e.g., real-time chat, interactive coding assistants), even "slight" overhead may be unacceptable. Figure 7 shows FASA-C has lower latency than FASA-M, but the gap is not quantified in milliseconds.

**TIP computation adds per-step overhead.** The TIP stage requires computing attention scores over dominant FCs for all cached tokens before FAC can proceed. Section 4.4 gives TIP complexity as $O(2tN_{tip})$, which at $t=32K$ and $N_{tip}=16$ is approximately $1M$ operations per head per step. Across 32 layers and 40 heads, this accumulates. The paper claims this is "computationally frugal" compared to full attention ($O(td)$), but does not provide wall-clock measurements isolating TIP overhead.

**Trade-off summary:**

| Variant | Memory Savings | Speed Gain | Overhead |
|---------|---------------|------------|----------|
| FASA-M | 8× (Section 4.3) | Not reported | CPU-GPU transfer latency |
| FASA-C | Minimal (full cache on GPU) | 2.56× (Section 5.3) | TIP computation per step |
| Full KV | Baseline | Baseline | None |

Practitioners must choose based on their primary constraint: VRAM capacity (FASA-M) or throughput (FASA-C).

### 6.4 Accuracy Trade-offs: Non-Trivial Degradation on Hard Tasks

While the Abstract claims "near-oracle accuracy," the empirical results reveal **context-dependent degradation** that warrants scrutiny.

**Long-CoT reasoning shows meaningful gaps.** Table 3 presents AIME24 results for DeepSeek-R1-Distill-Llama-8B: FKV achieves 43.9% pass@1, while FASA achieves 38.0% — a **5.9 point absolute drop (13.4% relative degradation)**. While this surpasses all baselines (SnapKV at 29.1%, Stream at 15.3%), it is not "near-oracle" in absolute terms. The Oracle baseline achieves 36.0%, meaning FASA exceeds Oracle — an anomalous result the paper attributes to "mitigation of attentional distraction" but does not fully explain.

**Model-scale effects are inconsistent.** Table 2 shows FASA sometimes **exceeds FKV performance** (e.g., Qwen2.5-7B: 47.9 vs 47.8; Mistral-7B: 47.8 vs 47.4). Section 5.2 attributes this to noise reduction from evoking irrelevant tokens. However, this phenomenon is not consistent: Llama3.2-3B shows FASA at 41.5 vs FKV at 42.2 (-0.7), and Qwen2.5-14B-1M shows 49.2 vs 50.3 (-1.1). The paper does not identify **why some models benefit from pruning while others degrade**, leaving practitioners uncertain whether FASA will help or harm their specific model.

**Token budget sensitivity.** Figure 5 shows performance varies with token budget ($N_{fac}$). At budget 300 on MATH500 (Table 3), FASA achieves 62.2% vs FKV's 72.4% — a **10.2 point drop**. Only at budget 1000 does FASA approach FKV (71.8% vs 72.4%). This means the "256 tokens" claim from the Abstract (based on LongBench) does not generalize to reasoning tasks, which require larger budgets. Practitioners must tune $N_{fac}$ per task type, adding deployment complexity.

### 6.5 Calibration Requirements: One Sample May Not Suffice for All Cases

Section B.1 states calibration uses **"just a single data sample"** — one from Qasper for LongBench, one from MATH500 for Long-CoT. Table 6 shows low Coefficient of Variation (&lt;0.014) across different calibration datasets, supporting robustness.

**However, boundary conditions are untested:**

1. **Sample quality matters.** The paper does not specify what makes a "good" calibration sample. A poorly chosen sample (e.g., extremely short context, domain-specific jargon) might yield suboptimal $I_{dom}$ indices. The experiments use carefully selected benchmarks, not arbitrary data.

2. **Model fine-tuning effects are unclear.** Figure 10 compares Qwen2.5-14B-Instruct with Qwen2.5-14B-Instruct-1M (fine-tuned for 1M context), showing similar FC patterns. However, the paper does not test models fine-tuned on **highly specialized domains** (e.g., code-only models, scientific literature models) where attention patterns may diverge from general pretraining.

3. **Calibration is offline and one-time.** If a model is updated (e.g., new version, additional fine-tuning), calibration must be re-run. The paper does not quantify how often $I_{dom}$ changes with model updates or whether incremental recalibration is possible.

**Practical implication:** While one-sample calibration works for the tested benchmarks, practitioners deploying FASA on proprietary models or specialized domains should validate calibration quality on their specific data distribution before production deployment.

### 6.6 Unaddressed Scenarios and Open Questions

Several important scenarios remain unexplored in the paper:

**Multi-turn conversation with evolving context.** FASA evaluates static long contexts (LongBench) and single-pass reasoning (MATH, AIME). Real-world chat applications involve **multi-turn conversations** where context grows incrementally across dozens of turns. It is unclear whether dominant FCs identified at turn 1 remain valid at turn 20, or whether recalibration is needed. The task-invariance claim (Table 10) addresses different tasks, not evolving context within a session.

**Streaming and partial prefill scenarios.** FASA assumes the full context is available before decoding begins. Applications with **streaming input** (e.g., real-time transcription, live document editing) may not have complete context during early decoding steps. The paper does not address how FASA handles incremental context growth during generation.

**Failure mode analysis is absent.** The paper reports only successful cases. There is no discussion of:
- Tasks where FASA underperforms baselines
- Input types that confuse the TIP stage (e.g., highly repetitive text, adversarial prompts)
- Conditions under which dominant FC identification fails

This absence limits understanding of FASA's **boundary conditions** — practitioners cannot identify when NOT to use FASA.

**Integration with speculative decoding.** Modern inference pipelines increasingly use speculative decoding (draft-then-verify) to accelerate generation. FASA's TIP stage operates at each decoding step, which may conflict with speculative decoding's batch verification. The paper does not address compatibility with this emerging optimization paradigm.

**Energy efficiency implications.** The Abstract and Section 7 emphasize environmental sustainability as a motivation ("environmentally sustainable," "democratization"). However, the paper provides no **energy consumption measurements** (e.g., watt-hours per token, carbon footprint). Memory reduction does not always translate to energy savings if CPU-GPU transfer or additional computation offsets the gains. This claim remains unquantified.

### 6.7 Summary of Trade-offs

FASA represents a deliberate set of engineering trade-offs rather than a universally superior solution:

| Dimension | FASA Advantage | FASA Limitation |
|-----------|---------------|-----------------|
| **Architecture** | Optimized for RoPE models | Degraded performance on ALiBi (Tables 7-8) |
| **Context Length** | Strong gains at 16K+ tokens | Diminishing returns below 8K |
| **Memory** | 8× reduction (FASA-M) | CPU-GPU transfer overhead unquantified |
| **Speed** | 2.56× speedup (FASA-C) | TIP overhead per decoding step |
| **Accuracy** | Near-FKV on LongBench (Table 2) | 13% degradation on AIME24 (Table 3) |
| **Calibration** | One-time, single sample | Untested on specialized domains |
| **Compatibility** | Orthogonal to PyramidKV (Table 4) | Not tested with speculative decoding |

**For practitioners:** FASA is a strong choice when:
- Using RoPE-based models (Llama, Qwen, Mistral)
- Processing 16K+ token contexts
- Memory bandwidth is the primary bottleneck
- Reasoning tasks require dynamic token selection

FASA may be less suitable when:
- Context lengths are consistently below 8K
- Using non-RoPE architectures without validation
- Latency sensitivity outweighs memory constraints
- Domain-specific attention patterns are expected

The paper's contributions are substantial, but understanding these limitations ensures FASA is deployed where it provides genuine value rather than assumed benefit.

## 7. Implications and Future Directions

FASA represents more than an incremental efficiency improvement — it introduces a fundamentally new lens for understanding and optimizing transformer attention mechanisms. This section examines how the work reshapes the KV cache optimization landscape, identifies concrete research directions it enables, outlines practical deployment scenarios, and provides actionable guidance for practitioners deciding whether to adopt FASA.

### 7.1 Paradigm Shift: From Token Sparsity to Structural Sparsity

Prior KV cache optimization research has overwhelmingly focused on **token-level sparsity**: identifying which tokens matter and evicting the rest. Methods like H2O, SnapKV, and Stream all operate on this premise. FASA shifts the optimization target upstream, demonstrating that **sparsity exists within the positional encoding mechanism itself**.

This reframing has three consequential implications for the field:

**First, it reveals an underexploited structural property of RoPE.** Section 3.3's discovery that dominant FCs account for less than 1% of all frequency chunks (Table 9) while capturing 43-81% of contextual agreement (Table 1) suggests that RoPE-based models contain significant latent redundancy. This is not a learned compression artifact — it is intrinsic to how rotational frequencies encode position and semantics. Future efficiency research on RoPE models should account for this frequency-domain structure rather than treating all dimensions equivalently.

**Second, it decouples query-awareness from training cost.** The central tension in token eviction has been: static methods are training-free but query-blind; learning-based methods are query-aware but require training and generalize poorly. FASA breaks this trade-off by using dominant FCs as a **computationally free proxy** for token importance. Table 2 demonstrates this: FASA achieves near-FKV performance (within 1%) without any trainable parameters, while learning-based baselines suffer from poor cross-domain generalization. This suggests future work should explore other architectural properties that could serve as similar proxies.

**Third, it establishes orthogonality as a design principle.** Section 5.3's compatibility results (Table 4) show FASA can be layered with PyramidKV for compounded gains (+0.1-1.1% across budgets). Rather than competing for the same optimization resource, FASA addresses *which* tokens to keep while PyramidKV addresses *how many* per layer. This modular approach contrasts with prior methods that often conflict (e.g., two token eviction schemes cannot both evict different tokens from the same cache). Future efficiency frameworks should explicitly design for composability.

### 7.2 Enabled Research Directions

FASA's findings open several concrete avenues for follow-up investigation. We categorize these by proximity to the original work.

#### 7.2.1 Immediate Extensions (Directly Build on FASA)

**Layer-Adaptive FC Selection.** Figure 13 reveals that dominant FC patterns vary significantly across layers — Layer 0 shows different saliency distributions than Layer 31 or Layer 41. Yet Section B.1 specifies FASA uses **uniform $N_{tip}=16$ across all layers** for simplicity. A natural extension is layer-adaptive calibration:

$$I_{dom}^{l} = \underset{I \subseteq \{0,\ldots,d/2-1\}, |I|=N_{tip}^{l}}{\text{argmax}} \mathbb{E}_{q,K \sim \Omega} \left[ \sum_{i \in I} CA_{l,i}^K(q, K) \right]$$

where $N_{tip}^{l}$ varies by layer. Early layers might require fewer FCs (processing more local features), while later layers might need more (integrating global context). This could improve accuracy while maintaining efficiency, but requires quantifying the calibration storage overhead.

**Dynamic FC Recalibration.** Section 6 identifies multi-turn conversation as an unaddressed scenario. In extended dialogues spanning dozens of turns, dominant FCs identified at turn 1 may not remain optimal at turn 20. A follow-up could investigate **online FC adaptation** — periodically recalibrating $I_{dom}$ during long sessions based on recent attention patterns. The challenge is balancing adaptation cost against accuracy gains.

**Speculative Decoding Integration.** Modern inference pipelines increasingly use speculative decoding (draft-then-verify) to accelerate generation. FASA's TIP stage operates at each decoding step, which may conflict with speculative decoding's batch verification. Research is needed to determine whether FASA can operate on draft tokens, verify tokens, or both — and how this affects the 2.56× speedup claimed in Section 5.3.

#### 7.2.2 Medium-Term Investigations (Require Substantial New Work)

**Energy Efficiency Quantification.** The Ethics Statement claims FASA makes models "more accessible, affordable, and environmentally sustainable." However, Section 6.6 notes the paper provides **no energy consumption measurements**. Follow-up work should measure:
- Watt-hours per token generated
- Carbon footprint reduction at scale
- Whether CPU-GPU transfer in FASA-M offsets memory savings in energy terms

This is critical for validating the environmental claims and comparing FASA to alternative efficiency methods on sustainability metrics.

**Domain-Specific Calibration Studies.** Table 6 tests calibration robustness across five LongBench tasks, but Section 6.1 notes highly specialized domains (legal contracts, medical records, scientific code) remain untested. A systematic study could:
- Calibrate FASA on domain-specific corpora
- Measure performance degradation when using general-domain $I_{dom}$
- Identify domains where functional sparsity patterns diverge from pretraining

This would establish boundary conditions for FASA's task-invariance claim.

**Non-RoPE Architecture Optimization.** Section 6 shows FASA works on ALiBi and Partial-RoPE but with slight degradation (Tables 7-8: 1-1.5 point drops on ALiBi). Follow-up research could investigate whether alternative positional encodings exhibit different sparsity patterns that require modified FC selection strategies. For example, ALiBi's linear bias mechanism might induce sparsity at the head level rather than the dimension level.

#### 7.2.3 Long-Term Directions (Fundamental Questions)

**Why Do Dominant FCs Exist?** The paper identifies *that* functional sparsity exists but does not fully explain *why*. Is this an emergent property of training dynamics? A consequence of RoPE's mathematical structure? Or an artifact of specific model architectures? Understanding the root cause could enable:
- Designing positional encodings with optimized sparsity from initialization
- Predicting dominant FC patterns without calibration
- Transferring $I_{dom}$ across model families

**Theoretical Bounds on Sparsity.** Table 1 shows compound CA scores plateau around 80% even with $F=16$ FCs (25% of dimensions). Is there a theoretical upper bound on how much attention functionality can be reconstructed from sparse FCs? Establishing such bounds would clarify whether FASA's ~1% accuracy gap from FKV is fundamental or addressable.

**Cross-Model FC Transfer.** Table 10 shows >70% overlap of dominant FCs across tasks within a model. An open question: do dominant FCs transfer **across models**? Could $I_{dom}$ calibrated on Llama-3.1-8B work on Llama-3.2-3B or Qwen2.5-7B? Positive results would enable "FC libraries" for common model families, eliminating per-model calibration.

### 7.3 Practical Applications and Deployment Scenarios

FASA's efficiency profile makes it particularly suitable for specific real-world use cases. We categorize these by the primary constraint they address.

#### 7.3.1 Memory-Constrained Deployments (FASA-M)

**Edge and Consumer Hardware.** FASA-M's 8× KV cache compression (Section 4.3) enables long-context inference on hardware with limited VRAM:
- **Consumer GPUs** (8-12 GB VRAM): Process 32K+ contexts that would otherwise require 24+ GB
- **Edge devices** (Jetson, mobile): Run 7B-14B models with 8K-16K context windows
- **Multi-tenant serving**: Host more concurrent sessions on the same GPU by reducing per-session memory

**Repository-Level Code Analysis.** Section 1 mentions code analysis as a key motivation. Code repositories often exceed 100K tokens. FASA-M enables:
- Loading entire codebases into context without truncation
- Cross-file reasoning for refactoring and bug detection
- Documentation generation from full project context

**Legal and Compliance Review.** Legal documents (contracts, regulations, case law) require precise long-range dependency tracking. FASA's near-FKV accuracy on LongBench (Table 2) combined with 8× memory reduction enables:
- Processing full contracts (50K+ tokens) without chunking
- Maintaining clause-to-clause references across documents
- Deploying on-premise for data privacy requirements

#### 7.3.2 Latency-Constrained Deployments (FASA-C)

**Interactive Applications.** FASA-C's 2.56× speedup (Section 5.3) benefits latency-sensitive scenarios:
- **Real-time chat assistants**: Reduce time-to-first-token and inter-token latency
- **Coding copilots**: Accelerate suggestion generation during active development
- **Live document editing**: Provide instant feedback on long-form content

**High-Throughput Batch Processing.** For applications processing many long-context requests:
- **Document summarization pipelines**: Process thousands of long documents with reduced GPU hours
- **Research paper analysis**: Extract insights from full papers rather than abstracts
- **Customer support automation**: Handle extended conversation histories efficiently

**Long-CoT Reasoning Services.** Table 3 shows FASA preserves reasoning capability where baselines fail catastrophically. This enables:
- Mathematical tutoring systems that maintain logical coherence across steps
- Scientific reasoning assistants for hypothesis generation
- Complex planning tasks requiring extended thought chains

#### 7.3.3 Hybrid Deployments (FASA + Complementary Methods)

**Layered Optimization Stacks.** Table 4 demonstrates FASA's compatibility with PyramidKV. Production deployments could combine:
- **FASA** for token selection (which tokens)
- **PyramidKV** for layer-wise budget allocation (how many per layer)
- **KV quantization** (e.g., KIVI) for per-token compression
- **FlashAttention** for efficient kernel execution

This modular approach allows teams to incrementally adopt optimizations without replacing entire inference stacks.

**Multi-Model Serving.** Organizations serving multiple model sizes could use FASA to:
- Standardize on a single KV cache management strategy across models
- Reduce engineering overhead from maintaining multiple optimization pipelines
- Enable dynamic model switching based on workload requirements

### 7.4 Reproducibility and Integration Guidance

Based on the paper's experimental results and identified limitations, we provide concrete guidance for practitioners evaluating FASA.

#### 7.4.1 When to Prefer FASA

**Strong Fit Scenarios:**

| Condition | Rationale | Expected Benefit |
|-----------|-----------|------------------|
| RoPE-based models (Llama, Qwen, Mistral) | Functional sparsity is RoPE-intrinsic (Section 3.3) | Near-FKV accuracy with 256 tokens |
| Context length ≥ 16K tokens | Decoding dominates latency at this scale (Figure 3) | 2.56× speedup, 8× memory reduction |
| Reasoning tasks (math, code, logic) | Dynamic token selection preserves thought traces (Table 3) | 15-30% improvement over SnapKV/Stream |
| Memory-bandwidth bottleneck | FASA reduces memory traffic to ~12.5% (Section 4.4) | Alleviates memory-bound decoding |
| Training-free deployment required | No model fine-tuning or auxiliary training (Section 4.1) | Immediate deployment without retraining |

**Calibration Requirements:** Section B.1 specifies calibration uses **one sample** from a relevant task (Qasper for LongBench, MATH500 for CoT). Practitioners should:
1. Select a representative sample from their target domain
2. Run offline calibration (Algorithm 1) to identify $I_{dom}$
3. Validate on a held-out set before production deployment
4. Re-calibrate if the model is updated or fine-tuned

**Variant Selection:**

| Primary Constraint | Recommended Variant | Trade-off |
|-------------------|---------------------|-----------|
| VRAM capacity | FASA-M | 8× memory reduction, slight latency overhead from CPU transfer |
| Inference speed | FASA-C | 2.56× speedup, full cache remains on GPU |
| Balanced | FASA-C with prefetching | Mitigates FASA-M's transfer latency |

#### 7.4.2 When to Consider Alternatives

**Poor Fit Scenarios:**

| Condition | Concern | Alternative |
|-----------|---------|-------------|
| Context length &lt; 8K tokens | Prefill dominates; FASA optimizes decoding (Section 6.2) | Sliding window attention |
| Non-RoPE architectures | Performance degradation on ALiBi (Tables 7-8) | Architecture-specific methods |
| Latency-critical (&lt; 50ms) | CPU-GPU transfer overhead unquantified (Section 6.3) | FASA-C or full KV |
| Specialized domains | Task-invariance untested for legal/medical (Section 6.1) | Domain-specific calibration required |
| Speculative decoding pipelines | Compatibility not evaluated (Section 6.6) | Wait for integration research |

**Baseline Comparisons:** Before adopting FASA, practitioners should benchmark against:
- **SnapKV** if prefill-stage estimation suffices for their task
- **Stream** if simplicity and minimal overhead are priorities
- **Full KV** if accuracy is paramount and resources permit

Table 2 shows FASA outperforms these baselines by 5-40% on LongBench, but specific workloads may differ.

#### 7.4.3 Implementation Checklist

The paper provides substantial reproducibility support (Reproducibility Statement, Appendix B-D). Practitioners should verify:

- [ ] **Code availability**: GitHub repository at https://github.com/AMAP-ML/FASA-ICLR2026 (Abstract footnote)
- [ ] **Model compatibility**: Confirm RoPE usage in target model (Section 3.1)
- [ ] **Calibration data**: Select representative sample from target domain (Section B.1)
- [ ] **Hyperparameters**: Start with $N_{tip}=16$, $N_{fac}=256$ (Section B.1), tune based on Figure 5
- [ ] **Integration method**: Use monkey-patching approach from Figure 14 (Section B.4)
- [ ] **Validation**: Compare against FKV baseline on held-out samples before deployment

**Known Gaps:** The paper does not provide:
- Pre-calibrated $I_{dom}$ indices for common models (must calibrate per model)
- Performance profiles for sequence lengths &lt; 8K
- Energy consumption measurements
- Failure case analysis

Practitioners should fill these gaps through internal benchmarking.

### 7.5 Summary: FASA's Place in the Efficiency Landscape

FASA occupies a distinct niche in the KV cache optimization ecosystem:

```
                        Training-Free
                            │
            ┌───────────────┼───────────────┐
            │               │               │
        Stream          FASA           Learning-Based
      (Static)      (Structural)        (Adaptive)
            │               │               │
    Query-Blind      Query-Aware      Query-Aware
    Low Overhead     Moderate          High Overhead
    Poor Accuracy    High Accuracy     Variable Accuracy
```

FASA's key differentiator is achieving **query-awareness without training** by exploiting structural sparsity in RoPE. This positions it between static methods (Stream) and learning-based methods (AttentionPredictor, TokenButler) — offering the adaptability of the latter with the deployment simplicity of the former.

**For the research community:** FASA demonstrates that architectural properties (not just learned patterns) can drive efficiency optimization. Future work should explore similar structural sparsity in other components (feed-forward networks, embedding layers, normalization).

**For practitioners:** FASA is a production-ready solution for RoPE-based models processing 16K+ token contexts, particularly for reasoning tasks where baselines fail. The two variants (FASA-M, FASA-C) allow tuning for specific constraints, and the method's orthogonality enables integration with existing optimization stacks.

**For the field:** FASA shifts the conversation from "how many tokens can we evict?" to "which structural components actually matter for attention?" This reframing may inspire a new generation of efficiency methods that target architectural redundancy rather than just data redundancy.

The work's ultimate impact will depend on follow-up research addressing the open questions in Section 7.2 and real-world deployments validating the efficiency claims at scale. The code release and detailed appendices provide a strong foundation for both.
