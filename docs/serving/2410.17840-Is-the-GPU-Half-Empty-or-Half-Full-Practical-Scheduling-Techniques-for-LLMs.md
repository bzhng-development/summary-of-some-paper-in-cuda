## 1. Executive Summary

This paper addresses the inefficiency in Large Language Model (LLM) serving systems where existing schedulers are either too complex for deployment or too simple to capture performance gains. The authors introduce `LARRY`, an engine-level scheduler, and `SAL`, a token-aware load balancer, which are easy to implement (20–30 lines of code in `vLLM`) yet outperform current techniques on production workload traces like `SW-Chat` and `SW-Code`. Specifically, `LARRY` achieves 1.8×–2.1× lower p50 Time-To-First-Token (TTFT) latency than the next-best method on single-server deployments running `Llama-3 70B` and `8B` models, demonstrating that practical "drop-in replacements" can significantly reduce latency without requiring additional prediction models.

## 2. Context and Motivation

To understand the contributions of this work, we must first establish the unique physical constraints of serving Large Language Models (LLMs) and why standard scheduling techniques from general-purpose computing do not directly apply. The core tension lies between maximizing hardware throughput and managing limited GPU memory under unpredictable workloads.

### 2.1 The Memory-Constrained Throughput Problem

LLM serving is fundamentally different from traditional request-response services because the memory required to process a request is not static. When an LLM generates text, it must store intermediate states called the **KV cache** (Key-Value cache) in GPU memory. This cache stores the projections of previous tokens so the model does not recompute them for every new token generated.

The critical constraint is that the KV cache grows linearly with the sequence length. As noted in Section 1, the KV cache for a single sequence of 8192 tokens in a Llama-3 70B model occupies approximately $2.7$ GB of GPU memory. Since GPU memory is finite, the system cannot simply run as many requests as possible; it is bounded by the total memory capacity.

This creates a scheduling dilemma centered on **batching**. LLM serving systems increase throughput by processing multiple requests simultaneously in a single batch. Figure 2 illustrates this relationship for a Llama-3 8B model on an NVIDIA A100 GPU:
*   **Low Batch Sizes:** The GPU is **memory-bound**. It spends more time loading the model weights (e.g., $16$ GB) from memory to the arithmetic units than performing calculations. Throughput is low.
*   **High Batch Sizes:** The GPU becomes **compute-bound**. The model weights are loaded once and reused across many requests, maximizing arithmetic utilization. Throughput increases.

However, increasing the batch size increases memory consumption. If the system dispatches too many requests, the growing KV caches may exceed GPU capacity. When this happens, the system must **preempt** requests (pause them and evict their KV cache). Resuming a preempted request is expensive, often requiring the KV cache to be recomputed from scratch [18, 31]. Therefore, the scheduler must walk a tightrope: dispatch enough requests to keep the GPU compute-bound, but not so many that memory exhaustion triggers costly preemptions.

### 2.2 The Two-Level Scheduling Architecture

Modern LLM serving systems typically decouple scheduling into two distinct layers, as depicted in Figure 1:

1.  **Load Balancer (Inter-Server):** This component sits in front of multiple server replicas. Its job is to route incoming requests to specific servers to ensure load is distributed evenly (data parallelism).
2.  **Engine-Level Scheduler (Intra-Server):** This component runs on each individual server. It manages the local waiting queue, deciding which requests to dispatch into the GPU batch, when to preempt running requests, and how to manage the KV cache memory.

This hierarchy matters because inefficiencies can occur at either level. A perfect engine-level scheduler cannot compensate for a load balancer that sends all memory-intensive requests to a single server. Conversely, a smart load balancer cannot fix an engine that dispatches requests in a naive order, causing **Head-Of-Line (HOL) blocking** where a large request delays many smaller ones in the queue.

### 2.3 The Complexity-Performance Trade-off in Prior Work

The paper identifies a significant gap between academic research and industrial practice regarding how these scheduling decisions are made. Table 1 summarizes the landscape of existing techniques, categorizing them by their implementation complexity and design intent.

#### Practical Systems: Simple but Suboptimal
Production systems like `vLLM` [18], `TensorRT-LLM` [33], and `SGLang` [40] typically employ simple policies:
*   **First-Come-First-Served (FCFS):** Requests are processed in arrival order. This is easy to implement but suffers from HOL blocking; a request with a long prompt can delay many short requests behind it.
*   **No-Preempt:** The system allocates the maximum possible memory for a request upfront to avoid preemption. While this avoids eviction overhead, it often leads to memory fragmentation and underutilization, as many requests finish before using their allocated maximum.
*   **General-Purpose Load Balancers:** Policies like **Round-Robin (RR)**, **Random**, or **Power of Two Choices (P2C)** are common in orchestration platforms like Kubernetes or Envoy. However, these are **LLM-agnostic**; they route based on connection counts rather than token counts or memory pressure.

These practical choices are popular because they are "drop-in replacements"—they require no changes to the underlying serving engine architecture. However, the authors note that these systems "often leave easy performance gains on the table" (Abstract).

#### Research Proposals: High Performance but High Complexity
Conversely, academic schedulers often achieve superior performance but introduce barriers to adoption:
*   **Prediction Overhead:** Techniques like **TRAIL** [26] and **LTR** [6] rely on predicting request lengths to prioritize short jobs (approximating Shortest-Remaining-Processing-Time). TRAIL requires access to model layer activations to estimate remaining tokens, which practical schedulers often cannot access [18, 23, 33]. LTR requires training an offline ranking model.
*   **System Architecture Changes:** **S3** [10] was designed before **Paged Attention** [18] became standard and requires changes to memory management beyond the scheduler. **FastServe** [35] requires proactive KV cache migration between CPU and GPU memory, which is not supported in engines like `vLLM` or `SGLang`.
*   **Obsolete Assumptions:** **PiA** [41] addresses systems that do not support **Continuous Batching** (where requests can start/stop at any iteration). Since state-of-the-art engines all support Continuous Batching, PiA addresses a problem that no longer exists in modern deployments.

### 2.4 Positioning of This Work

This paper positions itself at the intersection of performance and deployability. The authors argue that improved scheduling policies should be implementable as **"drop-in replacements"** (Abstract). This means:
1.  **No External Dependencies:** The scheduler should not require training additional prediction models (unlike LTR or TRAIL).
2.  **No Architectural Changes:** The scheduler should not require modifications to memory management or KV cache migration logic (unlike S3 or FastServe).
3.  **LLM-Aware:** Unlike general-purpose load balancers (RR, Random), the scheduler should account for LLM-specific properties like prompt length and processing phases (Prefill vs. Decode).

The motivation is driven by production reality. Figure 4 shows a trace of Queries-Per-Second (QPS) from a production environment, highlighting that load fluctuates unpredictably. A static scheduling policy cannot adapt to these shifts. The authors propose that by using metrics already available within the scheduler (such as queue length and prompt length), it is possible to achieve performance gains comparable to complex research models without the associated implementation burden. This work aims to bridge the gap where "schedulers from the literature often achieve good performance but introduce significant complexity," while practical systems are "easy to implement... but leave easy performance gains on the table" (Abstract).

## 3. Technical Approach

This section dissects the mechanics of the two proposed techniques, `LARRY` (Load-Adaptive Request Reordering) and `SAL` (Server-Aware Load Balancer). Unlike prior research that relies on complex prediction models or architectural overhauls, these techniques leverage metrics already available within standard serving engines to make scheduling decisions. We will walk through the mathematical formulation of each scheduler, the logic behind their design choices, and the specific implementation details required to integrate them into a production system like `vLLM`.

### 3.1 Reader orientation (approachable technical breakdown)
This system is a pair of lightweight scheduling algorithms designed to sit inside existing Large Language Model (LLM) serving infrastructure without requiring changes to the underlying engine or model. It solves the problem of GPU memory contention and uneven server load by dynamically reordering requests based on their prompt size and current queue depth, rather than relying on static rules or heavy machine learning predictors.

### 3.2 Big-picture architecture (diagram in words)
The solution operates across the two levels of scheduling identified in Section 2.2, inserting intelligence at both the entry point and the execution point.
*   **Incoming Request Stream:** Users submit prompts which first hit the **Load Balancer (`SAL`)**.
*   **Load Balancer (`SAL`):** This component maintains a real-time view of all available servers. It calculates a "load score" for each server based on available memory and queued tokens, then routes the request to the server with the lowest estimated wait time.
*   **Server Instance:** Each server runs a replica of the LLM and hosts an **Engine-Level Scheduler (`LARRY`)**.
*   **Engine-Level Scheduler (`LARRY`):** This component manages the local waiting queue. It assigns a priority score to every waiting request based on how long it has waited and how much memory its prompt requires. It then dispatches requests to the GPU batch in descending order of priority until memory or batch limits are reached.
*   **GPU Execution:** The GPU processes the batched requests using Continuous Batching and Paged Attention, returning tokens as they are generated.

### 3.3 Roadmap for the deep dive
To fully understand how these components interact to reduce latency, we will examine them in the following order:
*   **Engine-Level Scheduler (`LARRY`):** We start here because single-server memory management is the foundational constraint; understanding how `LARRY` mitigates Head-Of-Line (HOL) blocking locally is necessary to understand what metrics `SAL` needs to balance globally.
*   **Load Balancer (`SAL`):** Once we understand local queue dynamics, we can explain how `SAL` distributes requests across multiple servers to prevent any single `LARRY` instance from becoming overwhelmed.
*   **Implementation & Methodology:** Finally, we detail how these algorithms were integrated into `vLLM`, including specific hyperparameters, polling rates, and the necessary modifications to handle high-throughput benchmarking.

### 3.4 Detailed, sentence-based technical breakdown
This paper presents a systems engineering contribution rather than a novel machine learning architecture, focusing on heuristic-based scheduling policies that are mathematically grounded in queueing theory and memory constraints.

#### 3.4.1 Engine-Level Scheduler: LARRY
`LARRY` operates within the serving engine on a single server to decide the order in which waiting requests are dispatched to the GPU. Its core innovation is a scoring function that dynamically adjusts the priority of a request based on the current pressure on GPU memory.

**The Scoring Mechanism**
`LARRY` assigns a numeric score to every request $r$ in the waiting queue. The scheduler then sorts the queue in descending order of this score and dispatches requests until it hits a hardware limit (either GPU memory capacity or maximum batch token size). The score is calculated using Equation 1 from Section 4.1:

$$
score(r) = \alpha * wait\_time(r) - queue\_len * memory(r)
$$

Each term in this equation serves a specific purpose in balancing latency fairness against throughput efficiency:
*   `$wait\_time(r)$` represents the duration request $r$ has already spent waiting in the queue. Multiplying this by `$\alpha$` ensures that requests are not starved indefinitely; even a memory-heavy request will eventually accumulate enough score to be dispatched.
*   `$memory(r)$` estimates the GPU memory required for the request. Crucially, `LARRY` estimates this solely based on the **prompt length** (input tokens). The authors justify this approximation using Figure 5, which shows that in production workloads like `SW-Chat` and `SW-Code`, over 70% of the tokens stored in the KV cache come from the prompt. Since the prompt length is known immediately upon arrival, this avoids the need for complex output length prediction models.
*   `$queue\_len$` acts as a dynamic weighting factor for the memory cost. This is the key adaptive mechanism: when the queue is short (low `$queue\_len$`), the memory penalty is small, and requests are processed roughly in arrival order. When the queue is long (high `$queue\_len$`), the memory penalty dominates, causing `LARRY` to aggressively prioritize small requests over large ones. This prevents a few large prompts from blocking many small requests during traffic spikes, effectively mitigating Head-Of-Line (HOL) blocking.
*   `$\alpha$` is a hyperparameter that controls the trade-off between median latency and tail latency. The authors evaluate `$\alpha \in \{1, 500, 1000\}$` in Figure 9. A low value (e.g., `$\alpha = 1$`) prioritizes throughput and median latency by favoring small requests. A high value (e.g., `$\alpha = 1000$`) prioritizes waiting time, which reduces tail latency (p99) for large requests but may increase HOL blocking for smaller ones. The default recommendation is `$\alpha = 1$` for most use cases.

**Dispatching Logic**
The scheduler iterates through the sorted queue and dispatches requests sequentially. It stops dispatching when either (i) all requests are dispatched, or (ii) the next request cannot be scheduled due to insufficient memory or the batch reaching the maximum token size. This logic ensures that the GPU is kept saturated without triggering preemptions. The authors note in Section 6 that preemption rates remained below 0.1% across all experiments, indicating that `LARRY`'s proactive queue management successfully avoids memory exhaustion.

#### 3.4.2 Load Balancer: SAL
While `LARRY` optimizes a single server, `SAL` optimizes the distribution of requests across multiple servers. Standard load balancers like Round-Robin or Power of Two Choices (P2C) typically route based on connection counts or simple CPU load, which are poor proxies for LLM inference load. `SAL` instead routes based on an estimate of how long a request will wait before execution.

**The Load Estimation Function**
`SAL` quantifies the load `$load(s, r)$` that adding request $r$ would impose on server $s$. It then routes the request to the server with the minimum load. The load function is defined in Equation 2 from Section 4.2:

$$
load(s,r) = \max(\beta * (memory(r) - free\_mem(s)), \quad queued\_tokens(s,r) / max\_tokens\_per\_batch)
$$

This equation models the two primary bottlenecks in LLM serving: memory availability and compute capacity.
*   **Memory Wait Component:** The term `$\beta * (memory(r) - free\_mem(s))$` estimates the time required to free up enough GPU memory to accept the request.
    *   `$memory(r)$` is the memory required for the request's input tokens.
    *   `$free\_mem(s)$` is the currently available GPU memory on server $s$.
    *   If `$memory(r) > free\_mem(s)$`, the request must wait for other requests to finish and release memory.
    *   `$\beta$` is a scaling factor representing the approximate rate at which memory is freed. It is calculated as `$\beta = (\mu_{in} + \mu_{out}) / \mu_{out}$`, where `$\mu_{in}$` and `$\mu_{out}$` are the average input and output token counts. This ratio accounts for the fact that memory is only freed when a request completes (both prefill and decode phases). The paper provides specific values derived from traces: `$\beta_{SW-Chat} = 1365/211$` and `$\beta_{SW-Code} = 2074/27$`.
*   **Compute Wait Component:** The term `$queued\_tokens(s,r) / max\_tokens\_per\_batch$` estimates the time required to process the existing compute backlog.
    *   `$queued\_tokens(s,r)$` includes all tokens currently waiting in the queue on server $s$, plus the prefill tokens of the new request $r$.
    *   `$max\_tokens\_per\_batch$` is the maximum number of tokens the GPU can process in a single iteration.
    *   This term captures the "compute-bound" delay described in Section 2.1.
*   **The Max Operator:** The use of `$\max$` is critical. A request cannot start until *both* memory is available AND compute resources are free. Therefore, the total wait time is determined by the slower of the two constraints.

**State Tracking and Polling**
To compute these metrics, `SAL` requires real-time visibility into server state. The implementation polls each server 10 times per second to retrieve the number of queued tokens and available memory. Between polling intervals, `SAL` locally tracks changes: if it routes a request with $N$ tokens to a server, it immediately updates its local record of `$queued\_tokens$` for that server. The authors state in Section 4.2 that this polling incurs "negligible overhead," allowing for frequent updates without impacting serving performance.

#### 3.4.3 Implementation and Methodology Details
The validity of these techniques relies on their integration into a representative serving system and evaluation under realistic conditions. The authors provide specific details in Section 5 that are crucial for understanding the performance results.

**Integration into vLLM**
Both techniques were implemented inside `vLLM` [18], a state-of-the-art open-source serving engine.
*   **Code Complexity:** `LARRY` required only 20 lines of code changes, and `SAL` required 30 lines. This supports the claim that they are "drop-in replacements" (Table 1).
*   **Driver Process Modification:** A significant methodological challenge arose during implementation. The standard `vLLM` driver process handles tokenization and queue management. At high Queries-Per-Second (QPS), this process became oversubscribed, creating backpressure that prevented requests from entering the waiting queue fast enough to test the schedulers properly. To fix this, the authors offloaded data marshalling (e.g., tokenization) to a separate process. This ensures that the rate at which requests enter the waiting queue precisely reflects the workload trace, rather than being capped by the driver's CPU speed.

**Workload Traces**
The evaluation uses production traces to ensure realism, rather than synthetic distributions.
*   **QPS Trace:** Incoming request timing follows a trace of Azure Function calls [28], which is noted to be representative of model serving loads [15, 24, 39]. The authors linearly scale this trace to test system limits (Section 5).
*   **Token Lengths:** Input and output lengths are emulated using traces from Azure OpenAI [22]. Specifically, they use two datasets: `SW-Chat` (chat application) and `SW-Code` (coding copilot). Figure 6 shows the distribution of these lengths; notably, `SW-Code` has much shorter output lengths (average 27 tokens) compared to `SW-Chat` (average 211 tokens), which influences the `$\beta$` calculation in `SAL`.
*   **Emulation Method:** Since the traces only record token counts and not actual text, the system issues requests with random tokens matching the recorded lengths. The system is enforced to generate exactly the recorded output length, ensuring the computational cost matches the production trace.

**Hardware Configurations**
The experiments cover two distinct hardware scales to demonstrate generality (Section 5):
*   **Large Model Setup:** Two NVIDIA H100 GPUs running `Llama-3 70B` (with FP8 quantized weights) using 2-way tensor parallelism. This setup tests the schedulers under heavy memory pressure.
*   **Small Model Setup:** One NVIDIA A100 GPU (40GB memory) running `Llama-3 8B`. This setup tests the schedulers on commodity hardware where memory is even more constrained relative to model size.

**Evaluation Metrics**
The paper focuses on latency metrics that directly impact user experience, rather than just raw throughput.
*   **Time To First Token (TTFT):** Measures the time from request issuance to the first generated token. This is the primary metric for scheduling efficiency, as it captures queueing delay and prefill time.
*   **Normalized TTFT:** Defined as `$TTFT / input\_tokens$`. This metric accounts for the fact that users expect longer prompts to take more time to process.
*   **Total Generation Time (TGT):** The total time until the full response is generated. This is dominated by the decode phase.
*   **Serving Capacity:** Measured by scaling the QPS of the workload trace until latency degrades. The authors specifically track p50 and p95 TTFT under these scaling factors to understand how robust each scheduler is to load spikes.

By combining these specific implementation choices with the mathematical formulations of `LARRY` and `SAL`, the authors construct a rigorous testbed that isolates the impact of scheduling logic from other system variables. This allows them to claim with evidence that the performance gains observed in Sections 6 and 7 are due to the scheduling policies themselves, not underlying hardware advantages or workload anomalies.

## 4. Key Insights and Innovations

This paper's primary contribution is not merely the introduction of two new algorithms, but a fundamental re-evaluation of what constitutes an "optimal" scheduler in production LLM serving. The authors challenge the prevailing assumption in academic literature that higher performance necessarily requires complex prediction models or architectural overhauls. Below, we distill the four most significant insights that drive this work, distinguishing between incremental improvements and shifts in design philosophy.

### 4.1 Feasibility as a First-Class Design Constraint
Most prior research optimizes for theoretical performance bounds, often treating deployment complexity as a secondary concern. This paper inverts that priority, treating **"drop-in replacement"** capability as a hard constraint that filters out otherwise superior techniques.

*   **The Innovation:** The authors define a scheduler as practically viable only if it requires no changes outside the scheduler module and no upfront training of prediction models (Table 1). This constraint eliminates high-performing academic proposals like `S3` [10] (requires memory management changes) and `FastServe` [35] (requires KV cache migration logic).
*   **Why It Matters:** By enforcing this constraint, the authors reveal that the performance gap between "simple" and "complex" schedulers is smaller than previously assumed. For instance, `LARRY` outperforms `TRAIL+` (which has access to ground-truth response lengths) in tail latency on certain workloads (Section 6). This suggests that **implementation friction is a hidden cost** that often outweighs theoretical scheduling optimality.
*   **Significance:** This shifts the systems research focus from "what is the best possible schedule?" to "what is the best schedule we can actually deploy?" The fact that `LARRY` was implemented in just 20 lines of code in `vLLM` (Section 3.4.3) demonstrates that high-impact optimizations do not require massive engineering efforts.

### 4.2 Prompt Length as a Sufficient Statistic for Memory
A prevailing assumption in LLM scheduling research is that accurate **output length prediction** is necessary to optimize queue ordering (e.g., `TRAIL` [26], `LTR` [6]). This paper challenges that assumption by demonstrating that input prompt length is a sufficient proxy for memory scheduling.

*   **The Insight:** Memory consumption is dominated by the prompt, not the generated response. Figure 5 shows that in production workloads (`SW-Chat`, `SW-Code`), over 70% of the tokens stored in the KV cache come from the prompt.
*   **Mechanism:** Because the prompt length is known immediately upon request arrival, `LARRY` can estimate `$memory(r)$` without running a prediction model. This avoids the runtime overhead and potential inaccuracies of predictors like `TRAIL`'s layer activation analysis.
*   **Critical Nuance:** This insight is specific to **memory-based scheduling**. For compute-based scheduling (optimizing total generation time), output length still matters. However, since memory exhaustion is the primary cause of preemption and queue blocking in constrained GPUs, optimizing for prompt length effectively mitigates the most costly failure mode. This simplifies the scheduling problem from a dual-prediction task (input + output) to a single-known-variable task (input).

### 4.3 Dynamic Priority Weighting via Queue Depth
Standard priority schedulers typically use static weights (e.g., "always prioritize short jobs"). `LARRY` introduces a dynamic feedback loop where the priority weighting changes based on system congestion.

*   **The Innovation:** In the scoring function (Equation 1, Section 3.4.1), the memory penalty is multiplied by `$queue\_len$`.
    $$
    score(r) = \alpha * wait\_time(r) - queue\_len * memory(r)
    $$
    This means the scheduler behaves differently depending on load:
    *   **Low Load (Small `$queue\_len$`):** The memory penalty is negligible. The scheduler behaves similarly to First-Come-First-Served (FCFS), ensuring fairness when resources are abundant.
    *   **High Load (Large `$queue\_len$`):** The memory penalty dominates. The scheduler aggressively prioritizes small requests to maximize throughput and prevent Head-Of-Line (HOL) blocking.
*   **Why It's Significant:** This adaptive behavior removes the need for an explicit "load predictor." The queue length itself acts as a sensor for memory pressure. Prior work like `TRAIL` uses a static threshold `$c$` to decide when to preempt (Section 2.2), which requires tuning for specific workloads. `LARRY`'s approach is self-regulating; as the queue grows, the system naturally becomes more aggressive about clearing small jobs.
*   **Trade-off Management:** The paper explicitly analyzes the hyperparameter `$\alpha$` (Figure 9). A low `$\alpha$` (e.g., 1) optimizes for median latency but risks starving large requests. A high `$\alpha$` (e.g., 1000) protects tail latency but risks HOL blocking. The insight here is that **no single static policy is optimal**; the system must expose this knob to operators to tune for their specific Service Level Objectives (SLOs).

### 4.4 Token-Aware vs. Connection-Aware Load Balancing
In multi-server deployments, the paper exposes a fundamental mismatch between standard load balancing metrics and LLM workload characteristics.

*   **The Problem:** General-purpose load balancers like **Power of Two Choices (P2C)** route based on the number of active connections (in-flight requests). However, in LLM serving, one request with a 10,000-token prompt imposes vastly more load than one request with a 10-token prompt. Section 3.1 explains that memory consumption scales linearly with sequence length.
*   **The Innovation:** `SAL` routes based on **queued tokens** and **available memory** rather than connection counts (Equation 2, Section 3.4.2).
    $$
    load(s,r) = \max(\beta * (memory(r) - free\_mem(s)), \quad queued\_tokens(s,r) / max\_tokens\_per\_batch)
    $$
*   **Performance Impact:** Section 7 shows that while `P2C` and `Random` achieve similar performance, `SAL` consistently improves Total Generation Time (TGT) by 1.1×–1.3×. This is because `SAL` prevents scenarios where one server receives a batch of large prompts while another sits idle. By balancing the **token count**, `SAL` ensures that the compute-bound decode phase is evenly distributed, reducing inter-token latency variance.
*   **Broader Implication:** This insight extends beyond LLMs to any service where request heterogeneity is high (e.g., database queries with varying scan sizes). It argues that **connection count is a flawed proxy for load** in heterogeneous workloads, and schedulers must inspect request content (token length) to make effective routing decisions.

### 4.5 Limitations of Shortest-Remaining-Processing-Time (SRPT)
An subtle but critical finding in this work is the empirical limitation of SRPT-based policies (like `TRAIL+`) on production traces.

*   **The Finding:** While `TRAIL+` (which has perfect knowledge of remaining tokens) achieves low median latency, it suffers from **starvation of large requests**, leading to high tail latencies (p95/p99) in some deployments (Section 6).
*   **Why This Happens:** Production workloads like `SW-Chat` have correlated input and output lengths. Prioritizing short outputs inadvertently prioritizes short inputs, which `LARRY` handles more directly. However, when a large request arrives, SRPT policies continuously deprioritize it in favor of incoming small requests.
*   **Significance:** This challenges the theoretical optimality of SRPT in real-world LLM serving. It suggests that **fairness constraints (via `$wait\_time$` in `LARRY`)** are necessary to prevent tail latency explosions, even if they slightly sacrifice median throughput. This is a practical correction to theoretical scheduling models that often assume infinite queues or ignore user-perceived tail latency.

### 5.1 Evaluation Methodology

The experimental design is structured to isolate the impact of scheduling policies from other system variables. By implementing all techniques within the same serving engine (`vLLM`), the authors ensure that performance differences arise from scheduling logic rather than underlying infrastructure.

#### 5.1.4 Baselines and Evaluated Techniques

The authors compare their proposals against both practical systems and research proposals. Table 1 summarizes which techniques are evaluated.

**Engine-Level Schedulers (Section 6):**
- **`FCFS`**: First-Come-First-Served, implemented in `vLLM`, `TensorRT-LLM`, `SGLang` [18, 33, 40]
- **`No-Preempt`**: Allocates maximum memory upfront, implemented in `TensorRT-LLM`, `ORCA` [33, 37]
- **`TRAIL+`**: An upper bound on `TRAIL` [26] performance, with access to **ground-truth response lengths** at no runtime overhead. This is critical: `TRAIL+` eliminates the prediction error that would affect the actual `TRAIL` system, making it a strong baseline.
- **`LARRY`**: The proposed engine-level scheduler

Note: `S3` [10] and `PiA` [41] are excluded because they require system changes incompatible with Paged Attention and Continuous Batching, respectively (Section 2.2). `LTR` [6] and `FastServe` [35] are excluded due to their requirement for offline model training and KV cache migration logic, respectively.

**Load Balancers (Section 7):**
- **`Random`**: Samples a server uniformly at random, implemented in Istio [9] and Envoy [5]
- **`P2C`** (Power of Two Choices): Samples two servers and chooses the one with fewer in-flight requests, implemented in Envoy [5] and KNative [11]
- **`SAL`**: The proposed Server-Aware Load Balancer

Note: `Llumnix` [32] is excluded because it requires KV cache migration between servers, which is not implemented in standard `vLLM` deployments (Section 3.2).

#### 5.1.5 Evaluation Metrics

The paper focuses on latency metrics that directly impact user experience (Section 5):

- **Time To First Token (TTFT)**: Time elapsed from issuing the request until the first token is generated. This includes queueing time and the prefill phase. This is the **primary metric** for scheduling efficiency.
- **Normalized TTFT**: TTFT divided by the number of input tokens (`$TTFT / input\_tokens$`). This accounts for the fact that users expect longer prompts to take more time [1].
- **Total Generation Time (TGT)**: Total time from request issuance until the output is completely generated. This is dominated by the decode phase.
- **Serving Capacity**: Measured by linearly scaling the QPS of the workload trace and tracking how latency degrades. The authors focus on **p50 TTFT** and **p95 TTFT** under scaling to understand robustness to load spikes.

### 5.2 Single-Server Results (Engine-Level Schedulers)

Section 6 evaluates engine-level schedulers in isolation on single-server deployments where load balancing is not a factor. The results are presented in Figures 7 and 8.

#### 5.2.1 Time-To-First-Token (TTFT) Performance

`LARRY` achieves substantial improvements in TTFT latency across both hardware configurations and both workloads.

**Llama-3 70B on 2× H100 (Figure 7):**
- **p50 TTFT**: `LARRY` is **1.8×–2.1× lower** than the next-best method
- **p95 TTFT**: `LARRY` is **1.2×–1.4× lower** than the next-best method
- **Normalized p50 TTFT**: `LARRY` is **1.3×–1.5× lower** than the next-best method
- **Normalized p95 TTFT**: `LARRY` is **3.3×–5.5× lower** than the next-best method

**Llama-3 8B on 1× A100 (Figure 8):**
- Similar improvements are observed, confirming that the benefits generalize across model sizes and hardware configurations.

**Why `LARRY` Outperforms:**
The authors attribute `LARRY`'s success to avoiding **Head-Of-Line (HOL) blocking**. By prioritizing requests with smaller prompts under high memory pressure, `LARRY` allows many small requests to be dispatched quickly rather than waiting behind a few large requests. This is reflected in the dramatically lower Normalized p95 TTFT (3.3×–5.5×), which captures the experience of users with long prompts who would otherwise be starved.

**`TRAIL+` Performance:**
`TRAIL+` performs similarly to `FCFS` in terms of TTFT. This is notable because `TRAIL+` has access to **ground-truth response lengths**, which should theoretically enable optimal Shortest-Remaining-Processing-Time (SRPT) scheduling. The authors explain that `TRAIL+` is designed for workloads with **short input lengths and long output lengths**, which differs significantly from the `SW-Chat` and `SW-Code` traces used here (Section 6). Furthermore, `TRAIL+` incurred **starvation of large requests** on some deployments, leading to high tail latencies.

**`No-Preempt` Performance:**
`No-Preempt` performed so poorly on `SW-Chat` that its TTFT latencies were **beyond plot limits** (Figure 7, 8). This is because `No-Preempt` allocates maximum memory upfront, severely limiting concurrency. On `SW-Code`, `No-Preempt` did significantly better because the workload is dominated by prefills (short outputs), so processing fewer requests concurrently has a smaller impact. This highlights that `No-Preempt` is highly workload-dependent.

#### 5.2.2 Total Generation Time (TGT)

For all techniques, preemption rates were **less than 0.1%** (Section 6). This is a critical finding: it indicates that all schedulers successfully avoided memory exhaustion, and therefore TGT differences are not driven by preemption overhead.

- All methods achieve **similar TGT latencies** because TGT is dominated by the decode phase, which is less affected by queue ordering.
- `SW-Code` shows slightly more variation than `SW-Chat` because it has shorter response lengths, making differences in TTFT more significant in the overall TGT.
- `No-Preempt`'s TGT on `SW-Chat` was beyond plot limits, consistent with its poor TTFT performance.

#### 5.2.3 Serving Capacity Under Load Scaling

The authors evaluate how quality of service changes as the workload's QPS is linearly scaled up (Figures 7 and 8, bottom row).

- **`LARRY` was least sensitive to scaling the QPS up**, reflected in both p50 and p95 TTFT plots.
- Benefits are **more pronounced for p50 TTFT** because `LARRY` prioritizes requests with short prompts under heavy load, achieving high throughput and short waiting times for most requests.
- Benefits are **smaller for p95 TTFT** because this metric emphasizes the waiting times of larger requests that are deprioritized by `LARRY`.
- **`TRAIL+`** also achieves low p50 TTFTs at higher scaling factors by avoiding HOL blocking. However, since `TRAIL+` prioritizes based on output length (which is only slightly correlated with input length in these workloads), it faces higher degrees of HOL blocking than `LARRY` (Section 6).

#### 5.2.4 Ablation Study: Impact of `$\alpha$` on `LARRY`

Figure 9 presents an ablation study on the hyperparameter `$\alpha$`, which weighs waiting time against memory consumption in `LARRY`'s scoring function (Equation 1). The authors evaluate `$\alpha \in \{1, 500, 1000\}$`.

**Key Findings:**
- **Low `$\alpha$` (e.g., `$\alpha = 1$`)**: Performs better for **most latency percentiles** (p50, p95). This is because low `$\alpha$` allows `LARRY` to aggressively prioritize small requests under high memory pressure, avoiding HOL blocking.
- **High `$\alpha$` (e.g., `$\alpha = 1000$`)**: Reduces **tail latencies (p99 TTFT)**. This is because large requests are deprioritized less aggressively, preventing starvation.
- **Very High `$\alpha$` (>1000)**: `LARRY` **degenerates to FCFS**. This happens when `$\alpha$` is so high that waiting time dominates each request's score, making the memory penalty negligible. At this point, tail latency may increase again if the system struggles under frequent HOL blocking.

**Recommendation:**
The authors recommend **`$\alpha = 1$`** for most applications, as it makes `LARRY` most effective at avoiding HOL blocking. However, applications sensitive to tail latencies (e.g., p99 TTFT) should choose higher values for `$\alpha$` (Section 6).

**Trade-off Summary:**
| `$\alpha$` Value | Median Latency (p50) | Tail Latency (p99) | Behavior |
|-----------------|---------------------|-------------------|----------|
| Low (1) | Best | Higher | Aggressively prioritizes small requests |
| Medium (500) | Good | Moderate | Balanced approach |
| High (1000) | Slightly worse | Best | Protects large requests from starvation |
| Very High (>1000) | Degrades | Degrades | Degenerates to FCFS |

### 5.3 Multi-Server Results (Load Balancers)

Section 7 evaluates load balancers in combination with engine-level schedulers across multiple server instances. Figures 10 and 11 present the results.

#### 5.3.1 Time-To-First-Token (TTFT)

The load balancer has a **smaller effect on queueing delays than the engine-level schedulers** (Section 7). This makes sense: the engine-level scheduler directly controls which requests enter the GPU batch, while the load balancer only influences which server a request reaches.

**SAL Performance (in combination with `LARRY`):**
- **p50 TTFT**: **1.0×–1.5× lower** than the next-best technique
- **p95 TTFT**: **1.2×–1.3× lower** than the next-best technique
- **Normalized TTFT**: **1.0×–1.2× lower** (both p50 and p95)

**P2C vs. Random:**
- `P2C` and `Random` achieve **similar performance** across most metrics.
- This suggests that simply sampling two servers (P2C) does not provide significant benefits over uniform random selection when the metric is connection count rather than token count.

#### 5.3.2 Total Generation Time (TGT)

Unlike engine-level schedulers, **load balancers have a significant effect on TGT** (Section 7). This is because TGT is dominated by the decode phase, and the load balancer impacts **inter-token latency** by distributing tokens more evenly between servers.

**Why Load Balancing Affects TGT:**
- The authors enforce that no batch contains more than **1024 tokens** (to avoid heavily compute-bound forward passes).
- However, batches with several hundred tokens incur significantly larger runtime than batches with few tokens.
- By evenly balancing the batch size across servers, there are fewer overly full batches, which improves inter-token latency and TGT.

**SAL Performance on TGT:**
- **p50 TGT**: **1.1×–1.3× lower** than the next-best method
- **p95 TGT**: **1.1×–1.3× lower** than the next-best method

This improvement is attributed to `SAL`'s **token-aware routing strategy**, which distributes tokens more evenly than `P2C` or `Random` (Section 7).

#### 5.3.3 Serving Capacity Under Load Scaling

Similar to the single-server results, the authors evaluate how latency degrades as QPS is scaled (Figures 10 and 11, bottom row).

- **Engine-level scheduling has a larger impact on serving capacity than the load balancer**. This reinforces the finding that intra-server scheduling is the primary bottleneck.
- On `SW-Code`, all load balancers achieve similar p50 TTFT for different scaling factors.
- On H100 GPUs, `Random` and `SAL` achieve similar p50 TTFT for all scaling factors.
- For remaining evaluations, **`SAL` improves on other techniques**, while neither `P2C` nor `Random` consistently outperforms the other.

### 5.4 Critical Assessment of Experimental Claims

The experiments provide strong evidence for the paper's central claims, but there are important nuances to consider.

#### 5.4.1 Claim: "Drop-in Replacement" with Minimal Code Changes

**Supported:** The authors implemented `LARRY` in **20 lines of code** and `SAL` in **30 lines of code** within `vLLM` (Section 4). This is a verifiable, concrete metric that supports the claim of low implementation complexity. Furthermore, the techniques do not require:
- Training additional prediction models (unlike `LTR` or `TRAIL`)
- Modifications to memory management (unlike `S3`)
- KV cache migration logic (unlike `FastServe` or `Llumnix`)

**Caveat:** The authors did modify `vLLM` to offload tokenization to a separate process to handle high QPS during evaluation (Section 5). While this modification is unrelated to the scheduling logic itself, it indicates that the base `vLLM` implementation may have bottlenecks that could affect real-world deployment of any scheduler at high load.

#### 5.4.2 Claim: "Outperforms Current Techniques on Production Workload Traces"

**Supported:** The quantitative results are clear and consistent:
- `LARRY` achieves **1.8×–2.1× lower p50 TTFT** than the next-best method on single-server deployments.
- `SAL` achieves **1.1×–1.3× lower TGT** than the next-best method on multi-server deployments.
- These improvements are observed across **two different model sizes** (70B and 8B), **two different hardware configurations** (H100 and A100), and **two different workloads** (`SW-Chat` and `SW-Code`).

**Caveat:** The `TRAIL+` baseline has access to **ground-truth response lengths**, which is an upper bound on what the actual `TRAIL` system could achieve. The fact that `LARRY` outperforms `TRAIL+` in tail latency (p95/p99) on certain workloads is significant, but `TRAIL+` still achieves competitive median latency. This suggests that for applications where median latency is the primary SLO, `TRAIL+` (with accurate predictions) might still be competitive.

#### 5.4.3 Claim: "Preemption Rates Remain Low"

**Supported:** Across all experiments, preemption rates were **less than 0.1%** (Sections 6 and 7). This indicates that both `LARRY` and `SAL` successfully avoid memory exhaustion through proactive queue management, eliminating the costly KV cache recomputation overhead.

#### 5.4.4 Limitations and Failure Cases

The authors are transparent about several limitations (Section 9):

**1. Driver Process Overhead:**
The evaluation ensures that requests are added to the queue with little overhead. If a serving system incurs significant overhead when adding requests to the waiting queue, the general findings about schedulers still hold, but **performance numbers might differ** from those reported. This is an important caveat for practitioners deploying on systems with different architectural bottlenecks.

**2. Online vs. Offline Scheduling:**
This work studies the **online scheduling problem**, where requests arrive over time. It does not address **offline, throughput-oriented applications** where all requests arrive at once and the system tries to process them as quickly as possible. This is left for future work.

**3. Workload Specificity:**
The benefits of `LARRY` are most pronounced on workloads where **prompt length dominates memory consumption** (Figure 5 shows >70% of KV cache tokens are from the prompt). On workloads where output length is much larger than input length, the memory estimation based solely on prompt length may be less accurate. The `SW-Code` workload, with its short outputs (average 27 tokens), is particularly favorable to `LARRY`'s approach.

**4. Tail Latency Trade-off:**
As shown in Figure 9, optimizing for median latency (low `$\alpha$`) comes at the cost of higher tail latency for large requests. Applications with strict tail latency SLOs (e.g., p99 < X ms) may need to tune `$\alpha$` carefully, and in some cases, `LARRY` may not outperform FCFS on tail metrics if `$\alpha$` is not set appropriately.

### 5.6 Verdict on Experimental Rigor

The experiments are **well-designed and convincing** for the following reasons:

1. **Fair Comparison:** All techniques are implemented in the same serving engine (`vLLM`), eliminating confounding variables from different system architectures.
2. **Production Traces:** Using real workload traces (`SW-Chat`, `SW-Code`, Azure Function QPS) rather than synthetic distributions increases external validity.
3. **Multiple Hardware Configurations:** Testing on both H100 and A100 GPUs, with both 70B and 8B models, demonstrates generality.
4. **Ablation Studies:** The `$\alpha$` parameter study (Figure 9) provides actionable guidance for practitioners.
5. **Transparent Limitations:** The authors explicitly acknowledge the driver process overhead issue and the online vs. offline scheduling distinction.

**Areas for Improvement:**
- The evaluation does not include a comparison against `LTR` [6], which also claims to be implementable in practical systems (Table 1 marks it as a drop-in replacement). Including `LTR` would strengthen the claim that `LARRY` outperforms other research proposals.
- The multi-server evaluation uses up to 8 servers. It is unclear how `SAL` scales to larger deployments (e.g., 100+ servers), where polling overhead and state synchronization might become more significant.
- The `TRAIL+` baseline, while useful as an upper bound, does not reflect the actual performance of `TRAIL` with prediction errors. A comparison against the actual `TRAIL` system (with its predictor) would provide a more realistic baseline.

Overall, the experimental analysis provides strong evidence that `LARRY` and `SAL` deliver meaningful performance improvements with minimal implementation complexity, validating the paper's central thesis that practical "drop-in replacements" can capture significant performance gains without the complexity of prior research proposals.

## 6. Limitations and Trade-offs

While `LARRY` and `SAL` demonstrate that simple, "drop-in" schedulers can outperform complex academic proposals, this performance comes with specific constraints. Every system design involves trade-offs: by optimizing for ease of deployment and prompt-based memory estimation, the authors necessarily exclude other optimization avenues. In this section, we critically examine the boundaries of this approach. We will explore where these techniques might fail, what operational burdens they shift to the user, and what performance ceilings they accept to maintain simplicity.

### 6.1 Dependency on Prompt-Dominated Workloads

The core heuristic of `LARRY`—estimating memory consumption based solely on prompt length—relies on a specific statistical property of the workload. As shown in **Figure 5**, the production traces used in this study (`SW-Chat` and `SW-Code`) exhibit a strong skew: over $70\%$ of the tokens stored in the KV cache come from the input prompt. This validates the design choice to ignore output length prediction.

However, this is not a universal property of all LLM usage. Consider a **summarization task** where a user uploads a short document (e.g., $500$ tokens) and requests a detailed analysis (e.g., $4,000$ tokens). In this scenario:
*   **Memory Estimation Error:** `LARRY` would estimate the memory requirement based on the $500$-token prompt. The actual memory consumption will grow significantly during the decode phase as the $4,000$-token output is generated.
*   **Scheduling Consequence:** The scheduler might dispatch too many of these requests concurrently, assuming they are "small." As the outputs generate, the GPU memory could fill up unexpectedly, forcing preemptions.
*   **Comparison to Predictive Schedulers:** Techniques like `TRAIL` [26] or `LTR` [6], which attempt to predict output length, would theoretically handle this case better by accounting for the anticipated growth of the KV cache.

The authors acknowledge this in **Section 9**, noting that their findings hold for workloads where "prompt length dominates memory consumption." Practitioners deploying `LARRY` must verify that their specific use case matches this profile. If the application involves long-generation tasks (e.g., creative writing, code generation with large diffs), the memory estimation error could degrade performance, potentially negating the benefits over simpler policies like `FCFS`.

### 6.2 Operational Tuning Burden: The $\alpha$ Hyperparameter

A subtle but significant trade-off lies in the configuration of the hyperparameter `$\alpha$` within `LARRY`'s scoring function (**Equation 1**, **Section 4.1**):

$$
score(r) = \alpha * wait\_time(r) - queue\_len * memory(r)
$$

While the authors recommend `$\alpha = 1$` for most cases, **Figure 9** reveals that no single value optimizes all latency percentiles simultaneously. This introduces an operational trade-off that shifts complexity from the *code* to the *configuration*:

*   **Low `$\alpha$` (e.g., $1$):** Optimizes for median latency (p50) and throughput. This is ideal for interactive chat applications where most users care about average responsiveness. However, it risks **starvation** for large requests, leading to higher tail latencies (p99).
*   **High `$\alpha$` (e.g., $1000$):** Protects large requests from starvation, reducing tail latency. However, if set too high, the scheduler effectively degenerates to **First-Come-First-Served (FCFS)**, reintroducing the Head-Of-Line (HOL) blocking issues `LARRY` was designed to solve.

**Implication for Practitioners:** Unlike `FCFS` or `Random` balancing, which have no tunable parameters, `LARRY` requires the operator to understand their Service Level Objectives (SLOs). If an application has strict tail latency constraints (e.g., "99% of requests must finish within $X$ seconds"), the operator must tune `$\alpha$` carefully. This tuning is workload-dependent; a value that works for `SW-Chat` might not work for `SW-Code` due to differences in output length distributions (**Figure 6**). This contradicts the goal of a completely "hands-off" drop-in replacement, as optimal performance requires human intervention.

### 6.3 Scalability Constraints of Centralized Load Balancing

The `SAL` load balancer improves performance by making routing decisions based on global server state (queued tokens and free memory). To achieve this, `SAL` periodically polls each server **10 times per second** (**Section 4.2**). While the authors state this incurs "negligible overhead" in their experiments, this design introduces a scalability bottleneck that was not fully stress-tested in this work.

*   **Evaluation Scale:** The multi-server evaluation in **Section 7** scales up to **8 servers** (for A100s) and **4 servers** (for H100s). In large-scale production deployments, clusters often consist of hundreds or thousands of GPU instances.
*   **State Synchronization Overhead:** As the number of servers `$N$` increases, the load balancer must process `$10 \times N$` state updates per second. At `$N=1000$`, this becomes $10,000$ messages per second just for heartbeats.
*   **Decision Latency:** The load balancer becomes a centralized choke point. If the load balancer process itself becomes CPU-bound due to processing state updates, routing decisions could lag, causing requests to wait longer for routing than for execution.

**Comparison to Distributed Approaches:** General-purpose load balancers like **Power of Two Choices (P2C)** are decentralized; each router node only needs to sample two servers. `SAL` requires a central coordinator with a global view. While `SAL` outperforms `P2C` in **Figure 10** and **Figure 11**, the paper does not provide evidence on how `SAL` behaves at massive scale. Practitioners deploying on large clusters may need to implement hierarchical load balancing (e.g., `SAL` within clusters, `P2C` between clusters) to mitigate this constraint.

### 6.4 The "Drop-in" Performance Ceiling

The authors explicitly constrain their design to "drop-in replacements" that require no changes outside the scheduler (**Table 1**, **Section 4**). This is a strategic choice to ensure adoptability, but it inherently limits the maximum achievable performance. By refusing to modify the underlying engine architecture, `LARRY` and `SAL` cannot leverage certain high-performance techniques:

*   **KV Cache Migration:** Systems like `Llumnix` [32] and `FastServe` [35] improve load balancing and preemption recovery by migrating KV caches between servers or between CPU and GPU memory. `SAL` cannot migrate requests once routed; if a server becomes overloaded after routing, the request must wait. `LARRY` recomputes KV caches on preemption rather than swapping them to CPU memory, which is slower but requires no engine changes (**Section 2.1**).
*   **Prefill-Decode Disaggregation:** Recent architectures separate the prefill phase (compute-bound) from the decode phase (memory-bound) onto different hardware tiers. `LARRY` and `SAL` operate within a standard unified serving engine. They optimize the mixing of phases but cannot physically separate them.

**Trade-off Summary:** The authors trade **absolute performance optimality** for **deployment feasibility**. As shown in **Section 6**, `LARRY` outperforms `TRAIL+` (which has ground-truth knowledge) on tail latency, suggesting the gap is small for many workloads. However, for extreme scale or highly heterogeneous workloads, architectural changes (like KV migration) might eventually be necessary, at which point `LARRY` and `SAL` would need to be superseded by more integrated solutions.

### 6.5 Scope Limitation: Online vs. Offline Scheduling

Finally, it is crucial to distinguish the problem setting addressed by this work. **Section 9** explicitly states: "This work studies the **online scheduling problem**... We leave the evaluation of offline scenarios for future work."

*   **Online Scheduling:** Requests arrive dynamically over time (as seen in the Azure Function trace, **Figure 4**). The scheduler must make irrevocable decisions without knowing future arrivals. `LARRY` and `SAL` are designed for this low-latency, interactive setting.
*   **Offline Scheduling:** In batch processing (e.g., processing a dataset of 1 million documents overnight), all requests are known upfront. The system can optimize for pure throughput, potentially reordering requests globally to minimize fragmentation or maximize batch density in ways online schedulers cannot.

**Why This Matters:** Techniques that work well online (like prioritizing short jobs to reduce queueing delay) might not be optimal offline (where maximizing GPU utilization might require batching large jobs together). Practitioners should not apply `LARRY` to offline batch pipelines without validation, as the scoring function prioritizes latency (`$wait\_time$`) over raw throughput maximization.

### 6.6 Summary of Trade-offs

Table 3 synthesizes the key trade-offs inherent in adopting `LARRY` and `SAL`.

| Design Choice | Benefit | Limitation / Trade-off |
| :--- | :--- | :--- |
| **Prompt-Based Memory Estimation** | No prediction model needed; zero runtime overhead. | Assumes prompt dominates KV cache (**Figure 5**); may fail on long-generation tasks. |
| **`$\alpha$` Hyperparameter** | Allows tuning for median vs. tail latency (**Figure 9**). | Requires manual tuning based on SLOs; not fully "set-and-forget." |
| **Centralized Load Balancing (`SAL`)** | Better token-aware routing than `P2C` (**Section 7**). | Potential scalability bottleneck at hundreds of servers due to polling overhead (**Section 4.2**). |
| **Drop-in Replacement** | Easy deployment (20–30 lines of code); no engine changes. | Cannot use advanced features like KV cache migration (`Llumnix`, `FastServe`). |
| **Online Scheduling Focus** | Optimized for interactive latency (TTFT). | Not validated for offline batch processing throughput (**Section 9**). |

In conclusion, `LARRY` and `SAL` represent a pragmatic optimization for the majority of current LLM deployments, which are interactive, prompt-dominated, and medium-scale. However, as applications evolve toward longer generations, larger clusters, or batch-oriented processing, the assumptions underlying these techniques may require re-evaluation.

## 7. Implications and Future Directions

This work represents a pivotal shift in how we approach systems research for Large Language Model (LLM) serving. By demonstrating that simple, domain-aware heuristics can outperform complex predictive models, the authors challenge the prevailing assumption that optimal scheduling requires intricate machine learning pipelines or architectural overhauls. This section outlines how this work changes the landscape of the field, identifies concrete avenues for future research, and provides actionable guidance for practitioners looking to integrate these techniques.

### 7.1 Shifting the Research Paradigm: Deployability as a Constraint

Historically, systems research often optimizes for theoretical performance bounds, treating implementation complexity as a secondary concern. Table 1 illustrates this divide: academic proposals like `TRAIL` [26] or `S3` [10] achieve high performance but require prediction models or memory management changes, while practical systems like `vLLM` [18] use simple policies like **First-Come-First-Served (FCFS)**.

This paper redefines the optimization objective. By enforcing **"drop-in replacement"** as a hard constraint (Section 4), the authors show that:
*   **Complexity is a Cost:** The engineering burden of maintaining prediction models (like `LTR` [6]) or modifying engine internals (like `FastServe` [35]) often outweighs the marginal performance gains they offer over smarter heuristics.
*   **Domain Knowledge > General Predictors:** Instead of training a model to predict output length, `LARRY` leverages a statistical property of production workloads: prompt tokens dominate KV cache memory (Figure 5, Section 4.1). This suggests that future research should prioritize understanding workload statistics before designing complex predictors.
*   **Adaptive Heuristics > Static Policies:** The success of `LARRY`'s dynamic scoring (Equation 1) demonstrates that schedulers should react to current system state (e.g., `$queue\_len$`) rather than relying on static priorities. This encourages a move away from static configuration toward self-regulating system components.

### 7.2 Practical Integration Guidance

For engineers and system architects, this work provides a clear decision matrix for selecting scheduling policies. The choice depends on workload characteristics, cluster scale, and Service Level Objectives (SLOs).

#### When to Use `LARRY` and `SAL`
*   **Interactive Workloads:** If your application is latency-sensitive (e.g., chat bots, coding assistants), `LARRY` is superior to `FCFS`. The reduction in **Time-To-First-Token (TTFT)** (1.8×–2.1× lower p50, Section 6) directly improves user experience.
*   **Prompt-Dominated Memory:** Verify that your workload matches the profile in Figure 5, where prompt tokens constitute the majority of the KV cache. If prompts are short and generations are long (e.g., summarization), `LARRY`'s memory estimation may be inaccurate (Section 6.1).
*   **Medium-Scale Clusters:** `SAL` is ideal for clusters ranging from single servers to dozens of instances. The centralized polling mechanism (10 Hz, Section 4.2) is efficient at this scale but may require modification for hundreds of servers (Section 6.3).
*   **Limited Engineering Bandwidth:** If you cannot afford to train prediction models or modify engine internals, `LARRY` and `SAL` are the optimal choice. They require only ~50 lines of code combined (Section 4) and integrate directly into `vLLM`.

#### When to Consider Alternatives
*   **Long-Generation Tasks:** For workloads where output length significantly exceeds input length, consider schedulers that predict output length (e.g., `TRAIL` [26]), despite the complexity.
*   **Massive Scale (>100 Servers):** The centralized nature of `SAL` may become a bottleneck. In this regime, hierarchical load balancing (e.g., `SAL` within racks, **Power of Two Choices (P2C)** between racks) may be necessary.
*   **Offline Batch Processing:** `LARRY` is designed for online scheduling (Section 9). For offline throughput maximization, different policies that prioritize batch density over latency may be more appropriate.
*   **Strict Tail Latency SLOs:** If your SLO requires strict bounds on p99 latency, you must tune the `$\alpha$` hyperparameter carefully. As shown in Figure 9, the default `$\alpha = 1$` optimizes median latency but may increase tail latency. Values like `$\alpha = 1000$` may be required to protect large requests.

### 7.3 Avenues for Future Research

This work closes the gap between research and practice but opens several new questions for the community.

#### 7.3.1 Scalable State Synchronization
`SAL` relies on a central load balancer polling servers for state (Section 4.2). Future work should explore **distributed load balancing** mechanisms that retain `SAL`'s token-awareness without a single point of contention. Techniques like gossip protocols or hierarchical aggregation could allow `SAL` to scale to hundreds of servers while maintaining accurate load estimates.

#### 7.3.2 Hybrid Memory Estimation
`LARRY` assumes prompt length is a sufficient statistic for memory (Section 4.1). Future schedulers could employ **lightweight predictors** only when the prompt is short. For example, if `$input\_tokens < 100$`, the system could invoke a small regression model to estimate output length, while using `LARRY`'s heuristic for longer prompts. This hybrid approach would capture the benefits of prediction without the universal overhead.

#### 7.3.3 Integration with Disaggregated Architectures
Recent architectures propose separating the **prefill phase** (compute-bound) from the **decode phase** (memory-bound) onto different hardware tiers. `LARRY` and `SAL` operate within a unified engine. Future research should investigate how `LARRY`'s scoring logic applies to disaggregated systems. For instance, should the load balancer route based on prefill queue depth separately from decode queue depth?

#### 7.3.4 Dynamic Hyperparameter Tuning
The `$\alpha$` parameter in `LARRY` controls the trade-off between median and tail latency (Section 6.2). Currently, this requires manual tuning. Future work could implement **online adaptation**, where the system automatically adjusts `$\alpha$` based on real-time SLO violations. If p99 latency exceeds a threshold, the system could increase `$\alpha$` dynamically to protect large requests.

### 7.4 Reproduction and Implementation Notes

For researchers or engineers wishing to reproduce these results or integrate `LARRY`/`SAL` into their own systems, the following details are critical:

*   **Base System:** The authors implemented their techniques in `vLLM` [18], specifically commit `57792ed469956c7e580f982d590fcacea882570b` (Section 5.1.1). Using this exact version ensures compatibility with the reported performance numbers.
*   **Driver Process Bottleneck:** Be aware of the **driver process oversubscription** issue described in Section 5. If you observe that request ingestion rate caps out before GPU utilization is high, you may need to offload tokenization to a separate process, as the authors did. This is not a flaw in `LARRY` itself, but a prerequisite for testing schedulers at high QPS.
*   **Workload Emulation:** To fairly compare schedulers, emulate workloads using production traces (e.g., Azure OpenAI traces [22]) rather than synthetic distributions. Ensure your emulation enforces exact output lengths, as variance in generation length significantly impacts memory pressure.
*   **Monitoring Metrics:** When deploying `LARRY`, monitor **preemption rates** alongside latency. The authors report rates $&lt;0.1\%$ (Section 6). If your deployment sees higher preemption, it indicates that `$queue\_len$` or memory estimates are misaligned with your hardware capacity, requiring adjustment of the dispatching logic.

### 7.5 Conclusion

The central lesson of this work is that **simplicity, when informed by domain knowledge, is a powerful optimization tool**. By recognizing that prompt length dominates memory usage and that queue depth signals memory pressure, the authors constructed schedulers that outperform complex alternatives without the associated deployment burden. For the field of LLM serving, this suggests a path forward where future innovations are evaluated not just on theoretical throughput, but on their feasibility as "drop-in" improvements to existing infrastructure. As LLM deployments continue to scale, techniques like `LARRY` and `SAL` provide a blueprint for achieving high performance without sacrificing maintainability.
