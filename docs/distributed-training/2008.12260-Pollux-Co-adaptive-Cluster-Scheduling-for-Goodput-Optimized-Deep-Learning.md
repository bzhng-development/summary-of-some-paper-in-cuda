## 1. Executive Summary

Pollux addresses the inefficiency in deep learning cluster scheduling arising from the inter-dependence between resource allocation and training hyperparameters, introducing a co-adaptive scheduler that jointly optimizes GPU allocation, batch size, and learning rate via a new metric called `goodput`. By modeling statistical efficiency using gradient noise scale alongside system throughput, Pollux dynamically adapts jobs during training rather than relying on static user configurations or blind resource scaling. In evaluations using models like ResNet-50 on ImageNet and YOLOv3 on PASCAL-VOC, Pollux reduces average job completion times by 37–50% compared to state-of-the-art schedulers like Tiresias and Optimus (even when baselines are manually tuned), improves finish-time fairness by 1.5×–5.4×, and potentially lowers cloud training costs by 25%.

## 2. Context and Motivation

Deep learning (DL) training has become a dominant workload in shared computing environments, such as corporate datacenters and public clouds. These environments rely on clusters of expensive hardware accelerators (e.g., GPUs or TPUs) to handle resource-intensive, long-running training jobs. To manage contention among multiple users, a **cluster scheduler** mediates access to these resources. However, the interaction between scheduling decisions and the internal configuration of DL jobs creates a complex optimization problem that existing systems fail to solve efficiently.

### The Configuration Inter-Dependence Problem

In a typical DL cluster, a user submits a training job that must be configured with several critical parameters. The most significant are:
1.  **Resource Allocation:** The number of GPUs assigned to the job.
2.  **Batch Size ($M$):** The number of training examples processed before updating the model weights.
3.  **Learning Rate ($\eta$):** A scalar controlling the magnitude of each weight update.

These parameters are not independent; they are tightly coupled. The optimal choice for one depends heavily on the values of the others. For example, increasing the number of GPUs allows for a larger total batch size, which can improve **system throughput** (examples processed per second). However, simply increasing the batch size without adjusting the learning rate often degrades model quality or convergence speed. Conversely, allocating too few GPUs may leave hardware underutilized due to synchronization overhead.

This inter-dependence creates a burden on users. As noted in the Introduction, making these decisions requires expert knowledge of both the cluster hardware performance and the specific DL model architecture. In a shared cluster, the optimal configuration is also **dynamic**: it depends on the current cluster load, which changes as other jobs start and finish. A static configuration chosen at submission time is rarely optimal throughout the job's lifetime.

### The Throughput vs. Statistical Efficiency Trade-off

To understand why static configurations fail, we must distinguish between two competing objectives in DL training:

*   **System Throughput:** The number of training examples processed per unit of wall-clock time. This is generally increased by allocating more GPUs and using larger batch sizes to amortize communication costs.
*   **Statistical Efficiency:** The amount of training progress made per training example processed. This is influenced by the batch size and learning rate. Larger batch sizes often reduce statistical efficiency because the gradient estimates become less noisy, requiring more examples to reach the same convergence level.

Figure 1 in the paper illustrates this trade-off using ResNet18 on CIFAR-10. 
*   **Figure 1a** shows that job scalability (resource utilization) depends on the batch size. Without a large enough batch, adding GPUs yields diminishing returns due to synchronization overhead.
*   **Figure 1b** shows that the most efficient batch size depends on the allocated resources and the **stage of training**. 

Crucially, the paper notes that a job in a later stage of training can tolerate batch sizes **10× or larger** than early in training without degrading statistical efficiency. This implies that a configuration that is optimal at the start of training may be highly inefficient near the end. Existing schedulers typically treat jobs as static entities, missing this opportunity for dynamic optimization.

### Limitations of Prior Scheduling Approaches

The paper categorizes existing DL schedulers into two broad groups, both of which fall short of addressing the inter-dependence problem described above (Section 2.3).

#### Non-Scale-Adaptive Schedulers
Systems like **Tiresias** [22] require users to specify the number of GPUs at job submission, which remains fixed for the lifetime of the job. Others, like **Gandiva** [66], allow dynamic GPU changes but do so opportunistically (e.g., via time-sharing) without modeling how the job's performance scales with resources.
*   **Shortcoming:** These systems assume the user knows the optimal resource count beforehand. If the user guesses incorrectly, resources are wasted either through queuing (requesting too many) or slow execution (requesting too few).

#### Scale-Adaptive Schedulers
Systems like **Optimus** [52] automatically select the amount of resources for each job based on a predictive model of system throughput. They aim to minimize average job completion time by allocating more GPUs to jobs that scale well.
*   **Shortcoming:** While these schedulers optimize resource allocation, they do so **blindly to training-related configurations**. They assume the batch size and learning rate are fixed or do not account for how these parameters should be re-tuned when resources change. They optimize for system throughput but ignore **statistical efficiency**.

### The Need for Co-Adaptive Scheduling

The core motivation for Pollux is the realization that resource allocation and training hyperparameters must be optimized **jointly**. A scheduler that allocates more GPUs must also instruct the job to increase its batch size and adjust its learning rate to maintain statistical efficiency. Conversely, when cluster contention is high and fewer GPUs are available, the job should reduce its batch size to maximize progress per example.

Pollux positions itself as a **hybrid resource scheduler** that bridges this gap. It introduces a holistic metric called **`goodput`**, defined as:

$$
\text{GOODPUT}_t(\star) = \text{THROUGHPUT}(\star) \times \text{EFFICIENCY}_t(M(\star))
$$

where $\star$ represents the configuration parameters (GPU allocation, batch size, etc.). By maximizing `goodput` rather than just throughput, Pollux accounts for the diminishing returns of large batch sizes on model convergence.

This approach addresses three specific gaps in the current landscape:
1.  **Dynamic Re-optimization:** Unlike static schedulers, Pollux monitors job status during training and adapts to changes in statistical efficiency (e.g., as the model converges).
2.  **Joint Optimization:** Unlike elastic schedulers that only move GPUs, Pollux co-adapts GPU allocation, batch size, and learning rate simultaneously.
3.  **Fairness based on Progress:** By using `goodput`, Pollux promotes fairness based on useful training progress rather than raw resource share, which is a more meaningful measure for DL workloads.

In summary, the motivation stems from the inefficiency of decoupling system scheduling from training dynamics. By treating the DL job as an adaptive process rather than a static container of work, Pollux aims to reduce waste in expensive cluster environments while improving fairness and completion times.

## 3. Technical Approach

### 3.1 Reader orientation (approachable technical breakdown)
Pollux is a two-level scheduling system that sits between deep learning training jobs and a GPU cluster, continuously monitoring each job's performance and automatically adjusting both resource allocation and training parameters. It solves the problem of wasted cluster capacity by treating each training job as an adaptive process that can be reconfigured during execution, rather than a static request that must be perfectly configured at submission time.

### 3.2 Big-picture architecture (diagram in words)
The system consists of two primary components that operate at different granularities. First, a **PolluxAgent** runs alongside each individual training job, responsible for measuring that job's performance characteristics (throughput and gradient statistics) and tuning local parameters like batch size and learning rate. Second, a central **PolluxSched** service runs at the cluster level, collecting goodput models from all active PolluxAgents and computing optimal GPU allocations across all competing jobs. Information flows upward from agents to the scheduler (performance reports every 30 seconds) and downward from scheduler to agents (new resource allocations every 60 seconds), creating a feedback loop where local tuning informs global allocation and global allocation enables better local tuning.

### 3.4 Detailed, sentence-based technical breakdown

This is a systems research paper that introduces a co-adaptive scheduling architecture for deep learning clusters, with the core idea being that resource allocation and training hyperparameters must be jointly optimized using a unified performance metric called `goodput`.

#### The Goodput Metric: Unifying Throughput and Efficiency

The foundation of Pollux is a performance metric that captures both how fast a job processes data and how much training progress each processed example contributes. The paper defines `goodput` at training iteration $t$ as:

$$
\text{GOODPUT}_t(\star) = \text{THROUGHPUT}(\star) \times \text{EFFICIENCY}_t(M(\star))
$$

where $\star$ represents the configuration parameters that jointly influence throughput and batch size, and $M$ is the total batch size summed across all allocated GPUs. In plain language, this equation states that goodput equals the number of examples processed per second multiplied by a fraction (between 0 and 1) representing how useful each example is for training progress. If statistical efficiency is perfect (value of 1), then goodput equals throughput; if efficiency degrades due to a large batch size, goodput decreases even if throughput remains high.

The configuration parameters $\star$ are specifically defined as a triple $(a, m, s)$ where $a \in \mathbb{Z}^N$ is the allocation vector specifying how many GPUs are allocated from each node $n$, $m \in \mathbb{Z}$ is the per-GPU batch size, and $s \in \mathbb{Z}$ is the number of gradient accumulation steps. The total batch size is computed as:

$$
M(a, m, s) = \text{SUM}(a) \times m \times (s + 1)
$$

This formulation allows Pollux to reason about three distinct ways to increase the total batch size: allocating more GPUs (increasing $\text{SUM}(a)$), increasing the per-GPU batch size ($m$), or using gradient accumulation ($s$). Gradient accumulation is particularly important because GPU memory often limits the per-GPU batch size before the batch is large enough to overcome synchronization overhead, so Pollux can accumulate gradients over $s$ forward-backward passes before synchronizing across GPUs on the $(s+1)$-th pass.

Pollux initializes each job using the user-provided initial batch size $M_0$ and learning rate $\eta_0$, starting with a single GPU, $m = M = M_0$, $s = 0$, and $\eta = \eta_0$. The efficiency metric $\text{EFFICIENCY}_t$ is measured relative to this initial configuration, meaning $\text{EFFICIENCY}_t(M)$ is a fraction between 0 and 1 relative to $\text{EFFICIENCY}_t(M_0)$. Pollux only considers batch sizes that are at least the initial batch size ($M \geq M_0$), which ensures the system does not degrade performance below the user's baseline expectation.

#### Modeling Statistical Efficiency Using Gradient Noise Scale

The key innovation that enables Pollux to optimize goodput (rather than just throughput) is a model that predicts how statistical efficiency changes with batch size during training. The paper builds on prior work relating statistical efficiency to gradient noise scale (GNS), which measures the noise-to-signal ratio of the stochastic gradient. A larger GNS means that training parameters like batch size and learning rate can be increased to higher values with relatively less reduction in statistical efficiency.

To support adaptive optimizers like Adam and AdaGrad (not just vanilla SGD), Pollux uses the **pre-conditioned gradient noise scale (PGNS)**, denoted by $\phi_t$, which generalizes GNS to pre-conditioned stochastic gradient descent. The PGNS is expressed as:

$$
\phi_t = \frac{\text{tr}(P \Sigma P^T)}{|P g|^2}
$$

where $g$ is the true gradient, $P$ is the pre-conditioning matrix of the adaptive SGD algorithm, and $\Sigma$ is the covariance matrix of per-example stochastic gradients. For vanilla SGD without pre-conditioning, $P$ is the identity matrix and PGNS reduces to standard GNS.

The critical insight from prior work (cited as Appendix D of [46]) is that it takes $1 + \phi_t / M$ training iterations to make a similar amount of training progress across different batch sizes $M$. Using this relationship, Pollux defines a concrete expression for statistical efficiency:

$$
\text{EFFICIENCY}_t(M) = \frac{\phi_t + M_0}{\phi_t + M}
$$

In plain language, this equation measures the contribution from each training example to overall progress. If $\text{EFFICIENCY}_t(M) = E$, then training using batch size $M$ will need to process $1/E$ times as many training examples to make the same progress as using batch size $M_0$. The equation has two important properties: first, $0 < E \leq 1$ (efficiency is always positive and at most 1), and second, efficiency decreases as $M$ increases (larger batches are less efficient per example).

During training, Pollux estimates the value of $\phi_t$ at each iteration, then uses the above equation to predict $\text{EFFICIENCY}_t$ at different batch sizes without actually training at those batch sizes. This is crucial because it allows Pollux to explore configuration changes without costly trial-and-error. The measured value of $\phi_t$ varies according to training progress at iteration $t$, thus $\text{EFFICIENCY}_t(M)$ reflects the lifetime-dependent trends exhibited by true statistical efficiency.

Figure 2 in the paper validates this model across six different DL tasks (ImageNet, YOLOv3, DeepSpeech2, BERT fine-tune, CIFAR10, and Recommendation). The top row shows validation metrics versus training progress for three different batch sizes ($M_0$, an intermediate batch size, and the maximum batch size limit). The middle row shows measured statistical efficiency versus training progress for two different batch sizes, where training progress is measured in "statistical epochs" defined as $\frac{M}{|X|} \sum_t \text{EFFICIENCY}_t(M)$ where $|X|$ is the size of the training dataset. The bottom row shows measured $\text{EFFICIENCY}_t$ versus predicted $\text{EFFICIENCY}_t$ for a range of batch sizes, using $\phi_t$ measured using the median batch size from each range during an early-training epoch (roughly 1/8th of the way through training). The paper reports that although there are differences in validation curves for several DL tasks (especially in earlier epochs), they achieve similar best values across different batch sizes (±1% relative difference for all tasks except DeepSpeech2 at ±4%), which are within the plateau of high-quality models expected from large-batch training.

The paper notes that $\text{EFFICIENCY}_t$ changes during training varies from task to task and depends on specific properties like the learning rate schedule. For example, $\text{EFFICIENCY}_t$ for ImageNet, which uses step-based learning rate annealing, experiences sharp increases whenever the learning rate is annealed. BERT (a fine-tuning task starting from a pre-trained model) and Recommendation (a much smaller and shallower model) are exceptions where larger batch sizes maintain high efficiency early in training.

To estimate $\phi_t$ efficiently, Pollux follows Appendix A.1 of [46] using the pre-conditioned gradient $P g$ instead of the gradient $g$. This can be done efficiently when there are multiple data-parallel processes by using the different values of $\hat{g}^{(t)}_k$ already available on each GPU $k$. However, this method does not work when there is only a single GPU and gradient accumulation is off ($s = 0$). In this particular situation, Pollux switches to a differenced variance estimator [63] which uses consecutive gradient estimates $\hat{g}^{(t-1)}$ and $\hat{g}^{(t)}$.

The paper acknowledges that in some cases, as batch size increases, the chosen learning rate scaling rule may break down before statistical efficiency decreases, which degrades final model quality. To address these cases, the application may define a maximum batch size limit that will be respected by Pollux. The authors find that a batch size up to 32× larger works well in most cases, and limits for common models are well-studied for popular learning rate scaling rules.

#### Plug-in Learning Rate Scaling Interface

Because different training jobs require different learning rate scaling rules to adjust $\eta$ in response to changes in $M$, Pollux provides a plug-in interface that can be implemented using a function signature `SCALE_LR(M0, M) → λ`. This function is called before every model update step, and $\lambda$ is used by Pollux to scale the learning rate. The implementation of `SCALE_LR` can utilize metrics collected during training, such as the gradient noise scale. Using this interface, one can implement rules including AdaScale, square-root scaling (which prescribes that $\eta$ be scaled proportionally with $\sqrt{M}$), linear scaling (which prescribes that $\eta$ be scaled proportionally with $M$), and LEGW. This design choice allows Pollux to support a wide variety of optimizers and training algorithms without hard-coding specific scaling rules into the core system.

#### Modeling System Throughput

To predict system throughput for data-parallel DL, Pollux aims to predict the time spent per training iteration, $T_{\text{iter}}$, and then calculates throughput as:

$$
\text{THROUGHPUT}(a, m, s) = M(a, m, s) / T_{\text{iter}}(a, m, s)
$$

The paper separately models $T_{\text{grad}}$, the time in each iteration spent computing local gradient estimates, and $T_{\text{sync}}$, the time in each iteration spent averaging gradient estimates and synchronizing model parameters across all GPUs.

For $T_{\text{grad}}$, the local gradient estimates are computed using back-propagation, whose run-time scales linearly with the per-GPU batch size $m$. Thus, Pollux models $T_{\text{grad}}$ as:

$$
T_{\text{grad}}(m) = \alpha_{\text{grad}} + \beta_{\text{grad}} \cdot m
$$

where $\alpha_{\text{grad}}$ and $\beta_{\text{grad}}$ are fittable parameters learned from observed data.

For $T_{\text{sync}}$, when allocated a single GPU, no synchronization is needed and $T_{\text{sync}} = 0$. Otherwise, Pollux models $T_{\text{sync}}$ as a linear function of the number of GPUs since in data-parallelism, the amount of data sent and received from each replica is typically only dependent on the size of the gradients and/or parameters. The model includes a linear factor to account for performance retrogressions associated with using three or more GPUs, such as increasing likelihood of stragglers or network delays. Co-location of GPUs on the same node reduces network communication, which can improve $T_{\text{sync}}$, so Pollux uses different parameters depending on GPU placement. Letting $K = \text{SUM}(a)$ be the number of allocated GPUs and $N$ be the number of physical nodes occupied by at least one replica:

$$
T_{\text{sync}}(a, m) =
\begin{cases}
0 & \text{if } K = 1 \\
\alpha^{\text{local}}_{\text{sync}} + \beta^{\text{local}}_{\text{sync}} \cdot (K - 2) & \text{if } N = 1, K \geq 2 \\
\alpha^{\text{node}}_{\text{sync}} + \beta^{\text{node}}_{\text{sync}} \cdot (K - 2) & \text{otherwise}
\end{cases}
$$

where $\alpha^{\text{local}}_{\text{sync}}$ and $\beta^{\text{local}}_{\text{sync}}$ are the constant and retrogression parameters for when all processes are co-located onto the same node, and $\alpha^{\text{node}}_{\text{sync}}$ and $\beta^{\text{node}}_{\text{sync}}$ are the analogous parameters for when at least two processes are located on different nodes. The paper notes that this model for $T_{\text{sync}}$ can be extended to account for rack-level locality by adding a third pair of parameters.

Modern DL frameworks can partially overlap $T_{\text{grad}}$ and $T_{\text{sync}}$ by overlapping gradient computation with network communication. The degree of this overlap depends on structures in the specific DL model being trained, like the ordering and sizes of its layers. Assuming no overlap, then $T_{\text{iter}} = T_{\text{grad}} + T_{\text{sync}}$. Assuming perfect overlap, then $T_{\text{iter}} = \max(T_{\text{grad}}, T_{\text{sync}})$. A realistic value of $T_{\text{iter}}$ is somewhere in between these two extremes. To capture the overlap between $T_{\text{grad}}$ and $T_{\text{sync}}$, Pollux models $T_{\text{iter}}$ as:

$$
T_{\text{iter}}(a, m, 0) = \left( T_{\text{grad}}(a, m)^\gamma + T_{\text{sync}}(a)^\gamma \right)^{1/\gamma}
$$

where $\gamma \geq 1$ is a learnable parameter. This equation has the property that $T_{\text{iter}} = T_{\text{grad}} + T_{\text{sync}}$ when $\gamma = 1$, and smoothly transitions towards $T_{\text{iter}} = \max(T_{\text{grad}}, T_{\text{sync}})$ as $\gamma \to \infty$. The paper constrains $\gamma$ to be in the range $[1, 10]$ during model fitting.

For gradient accumulation, one iteration of SGD spans $s$ accumulation steps followed by one synchronization step, modeled as:

$$
T_{\text{iter}}(a, m, s) = s \times T_{\text{grad}}(a, m) + \left( T_{\text{grad}}(a, m)^\gamma + T_{\text{sync}}(a)^\gamma \right)^{1/\gamma}
$$

Figure 3 in the paper shows an example of the THROUGHPUT function fit to measured throughput values for a range of resource allocations and batch sizes across all six models from Table 1. Each DL task was implemented using PyTorch, which overlaps the backward pass' computation and communication. Gradients are synchronized with NCCL 2.7.8, which uses either ring all-reduce or tree all-reduce depending on the detected GPUs and their placements. The paper reports that across all DL tasks, the average error of the fitted model was at most 10%, indicating that it represents the observed throughput measurements well. All models except ImageNet exhibited high sensitivity to inter-node synchronization, indicating that they benefit from co-location of GPUs. YOLOv3 and BERT benefit from using gradient accumulation to increase their total batch sizes.

The paper acknowledges limits of the throughput model: it models data-parallel training throughput only in the dimensions it cares about (number and co-locality of GPUs, batch size, and gradient accumulation steps). The simple linear assumptions made may diverge from reality for specialized hardware, sophisticated synchronization algorithms, different parallelization strategies, at larger scales, or hidden resource contention not related to network used for gradient synchronization. Rather than attempting to cover all scenarios with a single throughput model, the authors designed GOODPUT$_t$ to be modular so that different equations for THROUGHPUT may be easily plugged in without interfering with the core functionalities provided by Pollux.

#### PolluxAgent: Job-Level Optimization Loop

An instance of PolluxAgent is started with each training job and runs as a Python library that is imported into DL training code. The authors integrated PolluxAgent with PyTorch, which uses all-reduce as its gradient synchronization algorithm. PolluxAgent inserts performance profiling code that measures the time taken for each iteration of training, as well as calculating the gradient noise scale.

The system throughput parameters of a training job are defined as the 7-tuple:

$$
\theta_{\text{sys}} = \left( \alpha_{\text{grad}}, \beta_{\text{grad}}, \alpha^{\text{local}}_{\text{sync}}, \beta^{\text{local}}_{\text{sync}}, \alpha^{\text{node}}_{\text{sync}}, \beta^{\text{node}}_{\text{sync}}, \gamma \right)
$$

which are required to construct the THROUGHPUT function. Together with the PGNS $\phi_t$ (for predicting EFFICIENCY$_t$) and initial batch size $M_0$, the triple $(\theta_{\text{sys}}, \phi_t, M_0)$ specifies the GOODPUT function. While $M_0$ is a constant configuration provided by the user, and $\phi_t$ can be computed according to Section 3.1, $\theta_{\text{sys}}$ is estimated by fitting the THROUGHPUT function to observed throughput values collected about the job during training.

PolluxAgent measures the time taken per iteration, $T_{\text{iter}}$, and records the tuple $(a, m, s, T_{\text{iter}})$ for all combinations of resource allocations $a$, per-GPU batch size $m$, and gradient accumulation steps $s$ encountered during its lifetime. Periodically (every 30 seconds by default), PolluxAgent fits the parameters $\theta_{\text{sys}}$ to all of the throughput data collected so far. Specifically, the system minimizes the root mean squared logarithmic error (RMSLE) between the throughput equation and the collected data triples, using L-BFGS-B optimization. The system sets constraints for each $\alpha$ and $\beta$ parameter to be non-negative, and $\gamma$ to be in the range $[1, 10]$. PolluxAgent then reports the updated values of $\theta_{\text{sys}}$ and $\phi_t$ to PolluxSched.

At the beginning of each job, throughput values have not yet been collected. To ensure that Pollux finds efficient resource allocations through systematic exploration, the system imposes several priors which bias $\theta_{\text{sys}}$ towards the belief that throughput scales perfectly with more resources, until such resource configurations are explored. In particular, the system sets $\alpha^{\text{local}}_{\text{sync}} = 0$ while the job had not used more than one GPU, $\alpha^{\text{local}}_{\text{sync}} = \beta^{\text{local}}_{\text{sync}} = 0$ while the job had not used more than one node, and $\beta^{\text{local}}_{\text{sync}} = \beta^{\text{node}}_{\text{sync}} = 0$ while the job had not used more than two GPUs. This creates the following behavior: each job starts with a single GPU and is initially assumed to scale perfectly to more GPUs. PolluxSched is then encouraged to allocate more GPUs and/or nodes to the job, naturally as part of its resource optimization, until the PolluxAgent can estimate $\theta_{\text{sys}}$ more accurately. Finally, to prevent a job from being immediately scaled out to arbitrarily many GPUs, Pollux restricts the maximum number of GPUs that can be allocated to at most twice the maximum number of GPUs the job has been allocated in its lifetime.

With $\theta_{\text{sys}}$, $\phi_t$, and $M_0$, which fully specify the DL job's GOODPUT function at its current training progress, PolluxAgent determines the most efficient per-GPU batch size and gradient accumulation steps by solving:

$$
(m^*, s^*) = \underset{m, s}{\text{argmax}} \ \text{GOODPUT}(a, m, s)
$$

where $a$ is the job's current resource allocation. Once a new configuration is found, the job will use it for its subsequent training iterations, using the plug-in LR scaling rule to adapt its learning rate appropriately. As the job's EFFICIENCY$_t$ function changes over time, PolluxAgent will periodically re-evaluate the most efficient configuration. The paper reports that finding the optimal per-GPU batch size and gradient accumulation steps by optimizing GOODPUT$_t$ takes an average of 0.4 milliseconds.

#### PolluxSched: Cluster-Wide Optimization

The PolluxSched periodically allocates (and re-allocates) resources for every job in the cluster, implemented as a service in Kubernetes. At a fixed time interval (60 seconds by default), PolluxSched runs its search algorithm, and then applies the resultant allocation matrix by creating and terminating Kubernetes Pods that run the job workers.

To determine a set of efficient cluster-wide resource allocations, PolluxSched maximizes a fitness function that is defined as a generalized (power) mean across speedups for each job:

$$
\text{FITNESS}_p(A) = \left( \frac{1}{J} \sum_{j=1}^{J} \text{SPEEDUP}_j(A_j)^p \right)^{1/p}
$$

where $A$ is an allocation matrix with each row $A_j$ being the allocation vector for a job $j$, thus $A_{jn}$ is the number of GPUs on node $n$ allocated to job $j$, and $J$ is the total number of running and pending jobs sharing the cluster.

The speedup of each job is defined as the factor of goodput improvement using a given resource allocation over using a fair-resource allocation:

$$
\text{SPEEDUP}_j(A_j) = \frac{\max_{m, s} \text{GOODPUT}_j(A_j, m, s)}{\max_{m, s} \text{GOODPUT}_j(a_f, m, s)}
$$

where GOODPUT$_j$ is the goodput of job $j$ at its current training iteration, and $a_f$ is a fair resource allocation for the job, defined to be an exclusive $1/J$ share of the cluster.

The parameter $p$ in the fitness function acts as a "fairness knob". When $p = 1$, FITNESS$_p$ is the average of SPEEDUP values across all jobs. This causes PolluxSched to allocate more GPUs to jobs that achieve a high SPEEDUP when provided with many GPUs (i.e., jobs that scale well). However, as $p \to -\infty$, FITNESS$_p$ smoothly approaches the minimum of SPEEDUP values, in which case maximizing FITNESS$_p$ promotes equal SPEEDUP between training jobs, but ignores the overall cluster goodput and resource efficiency. Thus, $p$ can be considered a fairness knob, with larger negative values being more fair. A cluster operator may select a suitable value, based on organizational priorities. In the authors' experience and results, they find that $p = -1$ achieves most goodput improvements and reasonable fairness.

Each time a job is re-allocated to a different set of GPUs, it incurs some delay to re-configure the training process. Using the popular checkpoint-restart method, the authors measured between 15 and 120 seconds of delay depending on the size of the model being trained and other initialization tasks in the training code. To prevent an excessive number of re-allocations, when PolluxSched evaluates the fitness function for a given allocation matrix, it applies a penalty for every job that needs to be re-allocated:

$$
\text{SPEEDUP}_j(A_j) \leftarrow \text{SPEEDUP}_j(A_j) \times \text{REALLOC\_FACTOR}_j(\delta)
$$

The paper defines $\text{REALLOC\_FACTOR}_j(\delta) = (T_j - R_j \delta) / (T_j + \delta)$, where $T_j$ is the age of the training job, $R_j$ is the number of re-allocations incurred by the job so far, and $\delta$ is an estimate of the re-allocation delay (set to 30 seconds by default). Intuitively, REALLOC\_FACTOR$_j(\delta)$ scales SPEEDUP$_j(A_j)$ according to the assumption that the historical average rate of re-allocations for job $j$ will continue indefinitely into the future. Thus, a job that has historically experienced a higher rate of re-allocations will be penalized more for future re-allocations. In the testbed experiments, on average, each job was re-allocated resources once every 7 minutes, resulting in an average 8% run-time overhead due to checkpoint-restarts.

When multiple distributed DL jobs share a single node, their network usage while synchronizing gradients and model parameters may interfere with each other, causing both jobs to slow down. The paper cites Xiao et al. [66] reporting up to 50% slowdown for DL jobs which compete with each other for network resources. PolluxSched mitigates this issue by disallowing different distributed jobs (each using GPUs across multiple nodes) from sharing the same node. Interference avoidance is implemented as a constraint in Pollux's search algorithm, by ensuring at most one distributed job is allocated to each node. Figure 8c in the paper shows that with interference avoidance enabled, the average job completion time is unaffected by even severe slowdowns, because network contention is completely mitigated. However, without interference avoidance, the average job completion time is 1.4× longer when the interference slowdown is 50%.

To find a good allocation matrix, PolluxSched uses a population-based search algorithm that perturbs and combines candidate allocation matrices to produce higher-value allocation matrices, and finally modifies them to satisfy node resource constraints and interference avoidance. The allocation matrix with the highest fitness score is applied to the jobs running in the cluster. During each 60-second scheduling interval, PolluxSched spent an average of 1 second on 1 vCPU computing the cluster allocations by optimizing the FITNESS$_p$ function.

Both PolluxAgent and PolluxSched require a sub-procedure that optimizes GOODPUT$_t(a, m, s)$ given a fixed $a$. The authors implemented this procedure by first sampling a range of candidate values for the total batch size $M$, then finding the smallest $s$ such that $m = \lceil M / s \rceil$ fits into GPU memory according to a user-defined upper-bound, and finally taking the configuration which results in the highest GOODPUT value.

#### Supporting Non-Adaptive Jobs and Implementation Details

In certain cases, a user may want to run a job with a fixed batch size, i.e., $M = M_0$. These jobs are well-supported by PolluxSched, which simply fixes EFFICIENCY$_t$ for that job to 1 and can continue to adapt its resource allocations based solely on its system throughput.

PolluxAgent reports its most up-to-date system throughput parameters and gradient statistics every 30 seconds. Each PolluxAgent fits its throughput model parameters on its latest observed metrics every 30 seconds, taking an average of 0.2 seconds each time. The prior-driven exploration strategy performs close (within 2-5%) to an idealized scenario in which the model is fitted offline for each job before being submitted to the cluster, according to Section 5.3.2.

The paper notes that Pollux's current throughput model does not consider accelerator heterogeneity. The authors believe that extending with Gavel's metric would allow Pollux to co-adapt for goodput in heterogeneous DL clusters.

## 4. Key Insights and Innovations

While Section 3 detailed the mechanical implementation of Pollux, this section isolates the fundamental conceptual shifts that distinguish it from prior art. The performance gains reported in Table 2 (37–50% reduction in average job completion time) are not merely the result of engineering optimization, but stem from four key innovations that redefine how cluster schedulers interact with deep learning workloads.

### 4.1 Goodput as the Primary Optimization Objective
The most fundamental innovation in Pollux is the shift from optimizing **system throughput** to optimizing **`goodput`**. Prior scale-adaptive schedulers like Optimus [52] operate on the assumption that maximizing the number of examples processed per second (throughput) is equivalent to minimizing job completion time. This assumption holds only if statistical efficiency remains constant.

Pollux challenges this assumption by introducing `goodput` (Section 3) as the unified metric:
$$
\text{GOODPUT} = \text{THROUGHPUT} \times \text{EFFICIENCY}
$$
This distinction is critical because throughput and efficiency often move in opposite directions. As shown in Figure 1b, increasing batch size improves throughput but degrades efficiency. A throughput-only scheduler (like Optimus) would blindly allocate more GPUs to increase batch size, potentially wasting resources on examples that contribute little to model convergence. Pollux, by contrast, identifies the "knee of the curve" where adding more resources yields diminishing returns in *useful* progress.

**Significance:** This is a **fundamental innovation** rather than an incremental improvement. It changes the objective function of the scheduling problem itself. The result is evident in Table 2, where Pollux outperforms Optimus+Oracle even when the baseline is given "tuned" job configurations. This proves that static tuning cannot match dynamic goodput optimization because the optimal balance between throughput and efficiency shifts during training (e.g., later stages tolerate larger batches).

### 4.2 True Co-Adaptivity vs. One-Way Elasticity
Existing elastic schedulers exhibit **one-way adaptivity**: the scheduler adjusts resource allocation, but the training job remains a static black box that must cope with the new resources. Pollux introduces **co-adaptivity**, where the scheduler and the job actively collaborate.

*   **Prior Work (One-Way):** In Optimus, the scheduler decides the GPU count, and the job simply runs with whatever batch size was configured at submission. If the scheduler allocates more GPUs than the batch size can utilize, efficiency drops.
*   **Pollux (Two-Way):** The `PolluxAgent` (Section 4.1) actively retunes the job's hyperparameters (batch size, learning rate, gradient accumulation) to match the resources provided by `PolluxSched` (Section 4.2).

This architectural insight transforms the job from a passive container into an active participant. Figure 5 illustrates this behavior: during low contention, Pollux allocates more GPUs and increases batch size (boosting throughput); during high contention, it reduces GPUs and batch size (preserving efficiency). This dynamic trade-off is impossible without the agent's ability to reconfigure the training loop on the fly.

**Significance:** This is a **systemic innovation**. It requires tight integration between the cluster manager (Kubernetes) and the training framework (PyTorch). The benefit is robustness: the system does not rely on the user guessing the correct configuration upfront, nor does it rely on the scheduler guessing the job's scaling behavior. They converge on the optimal configuration together.

### 4.3 Online Prediction of Statistical Efficiency
Prior research on Gradient Noise Scale (GNS) [46] treated statistical efficiency as a property to be analyzed offline or post-hoc. Pollux makes the novel contribution of **online estimation** of Pre-conditioned Gradient Noise Scale (PGNS) $\phi_t$ during live training (Section 3.1).

By estimating $\phi_t$ using gradients already computed during data-parallel synchronization, Pollux can predict $\text{EFFICIENCY}_t(M)$ for batch sizes $M$ that the job has *never tried*. This capability eliminates the need for expensive trial-and-error profiling. The model validates this prediction in Figure 2 (BOTTOM), showing that predicted efficiency closely matches measured efficiency across a range of batch sizes using only data from a median batch size.

**Significance:** This is a **practical enabler** for the goodput metric. Without online efficiency prediction, `goodput` would be impossible to calculate for unseen configurations. While the mathematical derivation of PGNS builds on prior theory [46], applying it as a real-time scheduling signal is a new capability that allows the scheduler to reason about convergence speed, not just hardware utilization.

### 4.4 Fairness Based on Useful Progress
Traditional fairness in cluster scheduling is often defined by resource share (e.g., each user gets 10% of GPUs) or completion time. Pollux introduces fairness based on **goodput speedup** (Section 4.2). The fitness function maximizes the generalized mean of speedups:
$$
\text{FITNESS}_p(A) = \left( \frac{1}{J} \sum_{j=1}^{J} \text{SPEEDUP}_j(A_j)^p \right)^{1/p}
$$
where $\text{SPEEDUP}_j$ compares a job's current goodput to its goodput under a fair share allocation. By tuning the parameter $p$, operators can slide between maximizing total cluster goodput ($p=1$) and ensuring equal speedup for all jobs ($p \to -\infty$).

Figure 7 demonstrates the impact of this approach. With $p=-1$, Pollux achieves significantly better finish-time fairness ($\rho$) compared to Tiresias and Optimus, with 99% of jobs achieving $\rho < 2$. This means almost no job runs more than 2× slower than it would in isolation.

**Significance:** This is a **conceptual refinement** of fairness. It acknowledges that giving two jobs the same number of GPUs is not "fair" if one job scales poorly and wastes those GPUs. Fairness should be measured in *progress made*, not *resources consumed*. This aligns scheduler incentives with user goals (getting models trained) rather than infrastructure metrics (GPU utilization).

### 4.5 Pragmatic Modeling Choices
It is important to distinguish where Pollux relies on **pragmatic engineering** rather than theoretical breakthroughs. The system throughput model (Section 3.2) uses simple linear assumptions for $T_{\text{grad}}$ and $T_{\text{sync}}$. The authors acknowledge in Section 3.2 that these assumptions "may diverge from reality for specialized hardware... or at larger scales."

However, this simplicity is a deliberate design choice. By keeping the throughput model lightweight (fitting 7 parameters via L-BFGS-B), Pollux ensures the overhead remains negligible (0.2 seconds per fit, Section 5.2). The modularity of the `GOODPUT` definition allows future work to plug in more complex throughput models without changing the core scheduling logic.

**Significance:** This highlights that Pollux's success comes from **integration** rather than perfection of individual components. The throughput model is "good enough" because the efficiency model compensates for its inaccuracies. This trade-off ensures the system is deployable today rather than remaining a theoretical construct.

## 5. Experimental Analysis

This section examines how the authors validate Pollux's claims through systematic experimentation. The evaluation is designed to answer three critical questions: Does Pollux actually reduce job completion times compared to state-of-the-art schedulers? Does it improve fairness without sacrificing overall performance? And are the system overheads acceptable for production deployment? The authors employ both real cluster testbed experiments and simulator-based studies to provide comprehensive evidence.

### 5.1 Evaluation Methodology

#### Testbed Configuration

The authors conduct experiments on a physical cluster testbed consisting of **16 nodes and 64 GPUs total** (Section 5.1). Each node is an AWS EC2 `g4dn.12xlarge` instance with:
- 4 NVIDIA T4 GPUs per node
- 48 vCPUs
- 192GB memory
- 900GB SSD

All instances are launched within the same placement group to minimize network variability. The cluster runs Kubernetes 1.18.2 with CephFS 14.2.8 for checkpoint storage, enabling the checkpoint-restart elasticity that Pollux requires for resource re-allocation.

#### Workload Construction

The evaluation uses a **synthetic workload of 160 jobs** sampled from the busiest 8-hour range (hours 3–10) of Microsoft's published DL cluster traces [31]. The original traces contain submission time, GPU count, and duration, but no information about model architectures. To create realistic jobs, the authors map each traced job to one of six DL tasks from Table 1, categorized by total GPU-hours:

| Category | GPU-Hours | Fraction of Jobs | Example Models |
|----------|-----------|------------------|----------------|
| Small (S) | 0–1 | 36% | CIFAR10, Recommendation |
| Medium (M) | 1–10 | 10% | DeepSpeech2, BERT |
| Large (L) | 10–100 | 6% | YOLOv3 |
| XLarge (XL) | 100–1000 | 2% | ImageNet |

This distribution mirrors real cluster usage patterns, where many short jobs coexist with a few long-running training tasks.

#### Baseline Schedulers

The authors compare Pollux against two state-of-the-art DL schedulers (Section 2.3, 5.1):

1. **Tiresias [22]:** A non-scale-adaptive scheduler where users specify the GPU count at submission, which remains fixed for the job's lifetime. Tiresias uses two priority queues with a PromoteKnob disabled.

2. **Optimus+Oracle [52]:** A scale-adaptive scheduler that dynamically adjusts GPU allocation based on throughput predictions. The "Oracle" variant is given the exact number of training iterations until completion (which Optimus normally must predict), removing convergence prediction as a confounding factor. The authors note that Optimus's original throughput model assumes parameter server architecture, so they replace it with Pollux's throughput model (§3.2) for fair comparison.

Both baselines use the same learning rate scaling rules as Pollux (AdaScale for SGD, square-root scaling for Adam/AdamW) to isolate scheduling differences from optimizer tuning.

#### Two Configuration Scenarios

A critical aspect of the evaluation is that the authors test **two distinct configuration scenarios** for the baseline schedulers:

1. **Well-Tuned Job Configurations:** For each job, the authors manually tune the number of GPUs and batch size by measuring time per iteration across various configurations. A GPU count is considered "valid" if using the optimal batch size achieves **50%–80% of ideal (perfectly linear) scalability** versus a single GPU. This scenario strongly favors the baselines, simulating users with expert knowledge of their models and the cluster.

2. **Realistic Job Configurations:** The number of GPUs matches the Microsoft trace exactly, and batch size is set to $M_0 \times \text{number of GPUs}$ (a common but often suboptimal user practice). This scenario reflects how users typically configure jobs without systematic profiling.

The paper emphasizes that the well-tuned scenario is "unrealistically biased in favor of the baseline schedulers" (Section 5.1), making Pollux's performance gains in this setting particularly noteworthy.

#### Pollux Configuration

Unless otherwise specified, Pollux uses the following default settings (Section 5.1):
- **Scheduling interval:** 60 seconds
- **Agent reporting interval:** 30 seconds
- **Re-allocation delay estimate ($\delta$):** 30 seconds
- **Fairness knob ($p$):** -1 (harmonic mean over speedups)

#### Metrics

The evaluation reports three primary metrics (Table 2):
- **Average Job Completion Time (JCT):** Mean time from job submission to completion across all 160 jobs.
- **Tail (99th Percentile) JCT:** Completion time of the slowest 1% of jobs, capturing worst-case latency.
- **Makespan:** Total time from the first job submission to the last job completion, measuring overall cluster utilization efficiency.

For fairness analysis (Section 5.3.1), the authors use **finish-time fairness ($\rho$)** [43], defined as the ratio of a job's JCT on shared resources to its JCT in an isolated, equally-partitioned cluster. Jobs with $\rho < 1$ are treated better-than-fair, while $\rho > 1$ indicates worse-than-fair treatment.

### 5.2 Main Quantitative Results

#### Job Completion Time Improvements

Table 2 presents the core results from the testbed experiments. The numbers are striking, particularly given that the baseline schedulers are provided with well-tuned configurations:

| Policy | Average JCT | 99%ile JCT | Makespan |
|--------|-------------|------------|----------|
| Pollux ($p=-1$) | **0.76h** | **11h** | **16h** |
| Optimus+Oracle+TunedJobs | 1.5h | 15h | 20h |
| Tiresias+TunedJobs | 1.2h | 15h | 24h |
| Optimus+Oracle (realistic) | 2.7h | 22h | 28h |
| Tiresias (realistic) | 2.8h | 25h | 31h |
| Pollux ($p=+1$) | 0.83h | 10h | 16h |
| Pollux ($p=-10$) | 0.84h | 12h | 18h |

**Key comparisons:**

- **Against well-tuned baselines:** Pollux achieves **50% shorter average JCT** than Optimus+Oracle+TunedJobs (0.76h vs 1.5h) and **37% shorter** than Tiresias+TunedJobs (0.76h vs 1.2h). This is the paper's most significant claim: Pollux outperforms baselines even when they are given ideal configurations that Pollux must discover dynamically.

- **Against realistic baselines:** The gap widens substantially. Pollux achieves **72% shorter average JCT** than Optimus+Oracle (0.76h vs 2.7h) and **73% shorter** than Tiresias (0.76h vs 2.8h). This demonstrates that Pollux's automatic configuration provides enormous value when users lack expert knowledge.

- **Tail latency:** Pollux reduces 99th percentile JCT by **27%** compared to both tuned baselines (11h vs 15h), and by **50–56%** compared to realistic baselines (11h vs 22–25h). This indicates that Pollux benefits not just average jobs but also the longest-running tasks that often dominate cluster wait times.

- **Makespan:** Pollux completes the entire workload **20% faster** than Optimus+Oracle+TunedJobs (16h vs 20h) and **33% faster** than Tiresias+TunedJobs (16h vs 24h). This reflects better overall cluster utilization.

#### Why Does Pollux Outperform Tuned Baselines?

The authors provide a mechanistic explanation through Figure 5, which shows cluster-wide GPU allocation and statistical efficiency over time. The key insight is that **static tuning cannot adapt to changing cluster conditions**:

- **During low contention (Period A):** Pollux allocates more GPUs to jobs and uses larger batch sizes, accepting lower statistical efficiency in exchange for higher throughput. This is optimal when resources are abundant.

- **During high contention (Period B):** Pollux reduces GPU allocation and batch size, increasing statistical efficiency to make progress with fewer resources. This is optimal when resources are scarce.

Figure 6 provides a job-level view, showing one ImageNet job's configuration over time:
- **Row 2 (GPUs allocated):** Varies from ~8 to ~32 GPUs depending on cluster load.
- **Row 3 (Batch size):** Ranges from ~200 to ~6400 images, adapting to GPU count.
- **Row 4 (Statistical efficiency):** Drops to ~40% during high-GPU phases but rises to ~80% during low-GPU phases.

The tuned baselines cannot make these trade-offs because their configurations are fixed at submission time. Even Optimus+Oracle, which can adjust GPU count, does not adjust batch size or learning rate, leaving potential efficiency gains on the table.

#### Fairness Results

Section 5.3.1 evaluates scheduling fairness using finish-time fairness ($\rho$). Figure 7 shows the cumulative distribution function (CDF) of $\rho$ values across all jobs:

- **Pollux ($p=1$):** Shows poor fairness similar to Tiresias+TunedJobs, with a long tail of jobs having $\rho > 4$. This is expected since $p=1$ maximizes average speedup without regard for equality.

- **Pollux ($p=-1$):** Provides the best fairness, with **99% of jobs achieving $\rho < 2$**. This means almost no job runs more than 2× slower than it would in isolation. The paper reports **1.5×–5.4× improvement in max-$\rho$** over Tiresias and Optimus.

- **Pollux ($p=-10$):** Shows slightly worse fairness overall than $p=-1$, caused by PolluxSched incurring more re-allocations while trying to equalize speedups at all times. This demonstrates that extreme fairness settings can backfire due to re-allocation overhead.

The authors note that the $\rho$ range for Pollux with $p=-1$ is similar to that reported for Themis [43], a fairness-focused scheduler, but Pollux achieves this while also providing significant performance increases (Table 2). This is important because fairness often comes at the cost of overall efficiency—a trade-off Pollux navigates successfully.

#### Effect of the Fairness Knob

Table 2 includes results for three values of $p$:

| $p$ Value | Average JCT | 99%ile JCT | Makespan | Interpretation |
|-----------|-------------|------------|----------|----------------|
| $p=+1$ | 0.83h | 10h | 16h | Maximizes average speedup, less fair |
| $p=-1$ | 0.76h | 11h | 16h | Best balance (default) |
| $p=-10$ | 0.84h | 12h | 18h | Most fair, but higher overhead |

The authors explain that $p=-1$ (harmonic mean) is more suitable than $p=1$ (arithmetic mean) for optimizing average JCT because it penalizes jobs with very low speedup more heavily. However, increasing fairness further to $p=-10$ degrades all three metrics due to excessive re-allocations. This reveals a **trade-off**: moderate fairness improves performance by preventing resource starvation, but extreme fairness incurs overhead that outweighs benefits.

### 5.3 Ablation Studies and Robustness Checks

The authors conduct several ablation studies using a discrete-time cluster simulator (Section 5.3) to isolate the impact of specific design choices. The simulator is constructed by measuring performance and gradient statistics for each model in Table 1 under 146 different GPU allocation+placement configurations, then replaying them for simulated jobs.

#### Sensitivity to Workload Intensity

Figure 8a shows results for increasing job submission rates. As expected, all three schedulers (Pollux, Optimus+Oracle+TunedJobs, Tiresias+TunedJobs) suffer longer average JCT and makespan as load increases. However, **Pollux maintains similar relative improvements across all job loads**. This indicates that Pollux's benefits are not limited to lightly-loaded clusters but persist under heavy contention—a critical property for production environments where load varies unpredictably.

#### Prior-Driven Exploration Overhead

Section 4.1 describes how Pollux uses priors to encourage exploration when throughput data is unavailable (assuming perfect scalability until proven otherwise). Section 5.3.2 evaluates whether more efficient exploration (e.g., seeding with offline-fitted models) would improve performance:

- **Short jobs (e.g., CIFAR10):** Seeding with historical data provides **2–5% reduction in JCT**.
- **Long jobs:** No significant change.

The authors conclude that prior-driven exploration has **low overhead** and performs close to an idealized offline-fitting scenario. This is important because it means Pollux does not require a separate profiling phase before jobs can run efficiently.

#### Scheduling Interval Sensitivity

Figure 8b evaluates Pollux with scheduling intervals ranging from 30 seconds to several minutes:

- **Up to 2 minutes:** Pollux performs similarly well in terms of average JCT.
- **Beyond 2 minutes:** Performance degrades.

The authors note that queuing time contributes to roughly **half of the performance degradation** observed with longer intervals, since newly-submitted jobs can only start during the next scheduling interval. However, the other half comes from less frequent resource adjustments, indicating that **Pollux benefits from relatively frequent re-optimization**. The default 60-second interval appears well-chosen.

#### Interference Avoidance

Section 4.2 describes Pollux's constraint that at most one distributed job (using GPUs across multiple nodes) can be allocated to each node, preventing network interference. Figure 8c evaluates the impact:

- **With interference avoidance:** Average JCT is unaffected even when artificial slowdowns up to 50% are injected.
- **Without interference avoidance:** Average JCT is **1.4× longer** when interference slowdown is 50%.
- **Zero interference scenario:** Pollux performs similarly with or without the constraint.

This demonstrates that the interference avoidance constraint **does not harm performance in ideal conditions** but provides substantial protection when network contention exists. The constraint is implemented as a hard constraint in the search algorithm, not a soft penalty, ensuring it is always enforced.

#### Simulator Fidelity

The authors validate their simulator against testbed results (Section 5.3):

- **Simulator vs. Testbed:** Pollux reduces average JCT by **48% and 32%** over Optimus+Oracle+TunedJobs and Tiresias+TunedJobs in simulation, compared to **50% and 37%** in the testbed.

The close match (within 2–5 percentage points) suggests the simulator accurately captures real-world dynamics, giving confidence in the broader simulator experiments that would be impractical on physical hardware.

### 5.4 Additional Applications

#### Cloud Auto-Scaling (Section 5.4.1)

The authors explore using Pollux's goodput model for cost optimization in cloud environments, where users pay for resource duration. They implement a simple auto-scaling policy: scale up nodes when $\max_{m,s} \text{GOODPUT}_t(a,m,s) / \text{SUM}(a) > U \cdot \max_{m,s} \text{GOODPUT}_t(1,m,s)$, with $U = 2/3$.

Figure 9 compares Pollux-based auto-scaling with a throughput-based auto-scaler from Or et al. [50]:

- **Pollux:** Starts with few nodes, gradually increases as statistical efficiency improves during training. Maintains high efficiency throughout (Figure 9b).
- **Or et al.:** Quickly scales to many nodes with large batch size, then remains constant. Does not account for changing efficiency.

**Results for ImageNet training:**
- **Cost reduction:** **25% cheaper** with Pollux.
- **Completion time:** Only **6% longer** than throughput-based scaling.

This reveals a new opportunity: goodput-driven scheduling can reduce cloud costs by provisioning resources when they are most effective (later training stages) rather than uniformly throughout training.

#### Hyper-Parameter Optimization (Section 5.4.2)

Table 3 shows results for HPO using the Tree-structured Parzen Estimator (TPE) algorithm on CIFAR10 with ResNet18:

| Policy | Accuracy (Top 5 trials) | Avg JCT | Makespan |
|--------|------------------------|---------|----------|
| Pollux | 95.4±0.2 | 25min | 10h |
| Baseline | 95.5±0.3 | 34min | 14h |

- **Accuracy:** Nearly identical (95.4% vs 95.5%), showing Pollux does not degrade model quality.
- **Makespan:** Pollux completes HPO **30% faster** (10h vs 14h) due to adaptive resource allocation as trials progress.

This demonstrates Pollux's applicability beyond single-job training to multi-trial HPO workloads, though the authors note that a full evaluation across different HPO algorithm types is future work.

### 5.5 System Overheads

Section 5.2 reports detailed overhead measurements:

| Component | Overhead | Frequency |
|-----------|----------|-----------|
| PolluxSched optimization | 1 second on 1 vCPU | Every 60s |
| PolluxAgent model fitting | 0.2 seconds | Every 30s |
| GOODPUT optimization | 0.4 milliseconds | Per re-configuration |
| Re-allocation frequency | Once every 7 minutes (average) | Per job |
| Checkpoint-restart overhead | 8% of run-time | Per re-allocation |

These numbers indicate that Pollux's computational overhead is **negligible** relative to the performance gains. The 8% checkpoint-restart overhead is the most significant cost, but it is amortized over the typical job duration (average JCT of 0.76h = 45.6 minutes, meaning re-allocations occur ~6 times per job on average).

### 5.6 Critical Assessment of Experimental Claims

#### Strengths

1. **Rigorous baseline comparison:** The well-tuned scenario is explicitly designed to favor baselines, making Pollux's 37–50% improvement particularly compelling. This addresses a common weakness in systems papers where baselines are poorly configured.

2. **Multiple evaluation modes:** The combination of testbed experiments (real hardware) and simulator studies (broader parameter sweeps) provides both realism and comprehensiveness.

3. **Fairness-performance trade-off analysis:** The paper does not claim Pollux is universally better at all fairness settings. Table 2 and Figure 7 show that $p=-10$ degrades performance, demonstrating intellectual honesty about limitations.

4. **Overhead transparency:** Detailed overhead measurements (Section 5.2) allow readers to assess deployment feasibility. The 8% checkpoint-restart overhead is non-trivial but acceptable given the 37–73% JCT reductions.

#### Limitations and Open Questions

1. **Accelerator heterogeneity:** Footnote 1 in Section 2.3 acknowledges that Pollux's throughput model does not consider heterogeneous accelerators (e.g., mixing V100 and T4 GPUs). The authors suggest extending with Gavel's metric [48] as future work, but this is a current limitation for clusters with mixed hardware.

2. **Scale limitations:** Section 3.2 notes that the linear throughput model assumptions "may diverge from reality... at larger scales." The evaluation is limited to 64 GPUs; performance at 1000+ GPU scales (common for large language model training) is untested.

3. **Model quality guarantees:** While Table 3 shows similar accuracy for HPO, and Figure 2 shows validation metrics within ±1–4% across batch sizes, the paper does not provide formal guarantees that Pollux's dynamic configuration never degrades final model quality. The maximum batch size limit (§3.1) mitigates this risk but relies on user-provided bounds.

4. **Checkpoint-restart dependency:** Pollux requires checkpoint-restart for resource re-allocation, which adds 8% overhead and assumes the training framework supports it. Jobs that cannot checkpoint (e.g., some reinforcement learning workloads) would not benefit from dynamic re-allocation.

5. **Workload representativeness:** The synthetic workload is derived from Microsoft traces but maps to only six model types (Table 1). While these cover common DL tasks (classification, detection, speech, NLP, recommendation), emerging workloads like diffusion models or large language model pre-training are not evaluated.

#### Do the Experiments Support the Claims?

Overall, **yes**, the experiments convincingly support the paper's main claims:

- **Claim 1 (37–50% JCT reduction vs. tuned baselines):** Supported by Table 2 with specific numbers (0.76h vs 1.2–1.5h average JCT).
- **Claim 2 (1.5×–5.4× fairness improvement):** Supported by Figure 7 and Section 5.3.1 analysis.
- **Claim 3 (25% cloud cost reduction):** Supported by Section 5.4.1, though this is a preliminary result noted as future work.
- **Claim 4 (acceptable overhead):** Supported by Section 5.2 overhead measurements showing sub-second optimization times and 8% checkpoint overhead.

The most compelling evidence is that Pollux outperforms baselines even when they are given ideal configurations. This demonstrates that **dynamic co-adaptation provides value beyond static optimization**, which is the paper's core contribution.

## 6. Limitations and Trade-offs

While Pollux demonstrates significant performance improvements over existing DL schedulers, the approach relies on several assumptions and faces practical constraints that limit its applicability in certain scenarios. This section critically examines the trade-offs inherent in Pollux's design, the scenarios it does not address, and the open questions that remain.

### 6.1 Architectural and Hardware Assumptions

#### Accelerator Heterogeneity Not Supported

Pollux's throughput model assumes homogeneous GPU clusters. As explicitly noted in Footnote 1 of Section 2.3:

> "Pollux's current throughput model does not consider accelerator heterogeneity. We believe that extending with Gavel's metric would allow Pollux to co-adapt for goodput in heterogeneous DL clusters."

This is a significant limitation for production environments where clusters often contain mixed hardware generations (e.g., V100, A100, T4 GPUs coexisting). The throughput model parameters ($\alpha_{\text{grad}}, \beta_{\text{grad}}, \alpha_{\text{sync}}, \beta_{\text{sync}}$) are fitted per-job but assume uniform GPU performance within an allocation. In heterogeneous settings, a job allocated 4 GPUs might receive 2 fast and 2 slow GPUs, breaking the linear scaling assumptions in Equation 11. The authors acknowledge this gap and suggest integrating Gavel's heterogeneity-aware metric [48] as future work, but this remains unimplemented.

#### Scale Limitations Beyond 64 GPUs

The evaluation is conducted on a 64-GPU cluster (16 nodes × 4 GPUs). Section 3.2 explicitly warns:

> "The simple linear assumptions made in Eqn. 11, although sufficiently accurate for the settings we tested, may diverge from reality for specialized hardware, sophisticated synchronization algorithms, different parallelization strategies, at larger scales, or hidden resource contention not related to network used for gradient synchronization."

At scales of 1000+ GPUs (common for large language model pre-training), several factors may invalidate the model:
- **Network topology effects**: The model distinguishes only between intra-node and inter-node synchronization ($\alpha^{\text{local}}_{\text{sync}}$ vs $\alpha^{\text{node}}_{\text{sync}}$). At larger scales, rack-level, pod-level, and datacenter-level network boundaries introduce additional latency tiers not captured.
- **Straggler probability**: The linear retrogression term $\beta_{\text{sync}} \cdot (K-2)$ may underestimate synchronization delays when $K$ is large, as the probability of stragglers increases non-linearly.
- **Gradient accumulation limits**: Section 3.2 notes GPU memory limits per-GPU batch size. For very large models (e.g., billion-parameter transformers), even gradient accumulation may not enable sufficiently large batch sizes to amortize synchronization costs at scale.

The paper does not evaluate Pollux beyond 64 GPUs, leaving its behavior at production LLM-training scales an open question.

#### Checkpoint-Restart Dependency

Pollux requires checkpoint-restart functionality to re-allocate resources dynamically. Section 4.2 reports:

> "Using the popular checkpoint-restart method, we measured between 15 and 120 seconds of delay depending on the size of the model being trained and other initialization tasks in the training code."

Section 5.2 quantifies this as **8% runtime overhead** on average, with jobs re-allocated once every 7 minutes. This introduces two constraints:

1. **Framework support**: The training framework must support checkpointing and resumption. While PyTorch and TensorFlow support this, some workloads (e.g., certain reinforcement learning setups, online learning scenarios) may not checkpoint cleanly.
2. **Overhead acceptance**: For very short jobs (e.g., CIFAR10 jobs with JCT < 10 minutes), the 15–120 second checkpoint delay represents a significant fraction of total runtime. Section 5.3.2 notes prior-driven exploration provides only 2–5% JCT reduction for short jobs, suggesting the overhead may not be justified for sub-10-minute workloads.

Jobs that cannot or will not tolerate checkpoint-restart overhead cannot benefit from Pollux's dynamic re-allocation, limiting its applicability to batch-style training workloads.

### 6.2 Modeling and Prediction Limitations

#### No Formal Model Quality Guarantees

While Section 3.1 shows validation metrics within ±1–4% across different batch sizes (Figure 2), Pollux provides no formal guarantee that dynamic configuration changes will not degrade final model quality. The system relies on:

1. **User-provided maximum batch size limits**: Section 3.1 states "the application may define a maximum batch size limit that will be respected by Pollux." This shifts the burden of ensuring model quality to the user.
2. **Learning rate scaling rules**: The plug-in interface supports AdaScale, square-root scaling, and linear scaling, but Section 2.2 acknowledges "the reasons behind [large batch size degradation of final model quality] are not completely understood."

Table 3 shows similar accuracy for HPO (95.4% vs 95.5%), but this is a single experiment on CIFAR10. The paper does not evaluate whether Pollux's dynamic batch size changes affect convergence behavior for sensitive models (e.g., models requiring specific learning rate schedules or those prone to catastrophic forgetting).

#### Throughput Model Simplifications

The throughput model (Equations 8–11) uses linear assumptions for $T_{\text{grad}}$ and $T_{\text{sync}}$. Section 3.2 acknowledges:

> "Rather than attempting to cover all scenarios with a single throughput model, we designed GOODPUT$_t$ (Eqn. 4) to be modular so that different equations for THROUGHPUT may be easily plugged in without interfering with the core functionalities provided by Pollux."

While modularity is a strength, it also reveals that the current model is intentionally simplified. Specific limitations include:

- **No overlap modeling for specific layer structures**: Equation 10 uses a single $\gamma$ parameter to model computation-communication overlap. However, Section 3.2 notes "the degree of this overlap depends on structures in the specific DL model being trained, like the ordering and sizes of its layers." A single $\gamma$ cannot capture layer-specific overlap patterns.
- **No contention modeling beyond interference avoidance**: The model assumes isolated job performance. While Section 4.2 implements interference avoidance (one distributed job per node), it does not model contention for shared resources like memory bandwidth, PCIe lanes, or CPU resources that may affect $T_{\text{grad}}$.
- **Fixed parameter synchronization**: The model assumes synchronous data-parallelism with all-reduce. It does not support asynchronous methods, parameter servers, or pipeline parallelism (e.g., GPipe [28], PipeDream [47]).

Figure 3 shows the model fits observed data with ≤10% average error, but this is on 6 DL tasks with specific architectures. The error may increase for models with unusual computation-communication patterns.

#### Statistical Efficiency Prediction Relies on PGNS Stability

The efficiency model (Equation 6) depends on accurate PGNS ($\phi_t$) estimation. Section 3.1 notes:

> "The measured value of $\phi_t$ varies according to the training progress at iteration $t$, thus EFFICIENCY$_t(M)$ reflects the lifetime-dependent trends exhibited by the true statistical efficiency."

However, several factors may destabilize $\phi_t$ estimation:

- **Single-GPU fallback**: When $K=1$ and $s=0$, Pollux switches to a differenced variance estimator [63] using consecutive gradients $\hat{g}^{(t-1)}$ and $\hat{g}^{(t)}$. This estimator may be noisier than the multi-GPU method.
- **Learning rate schedule interactions**: Section 3.1 observes "EFFICIENCY$_t$ for ImageNet, which uses step-based learning rate annealing, experiences sharp increases whenever the learning rate is annealed." The model captures this, but rapid efficiency changes may cause the scheduler to make suboptimal decisions during transition periods.
- **Optimizer-specific behavior**: PGNS generalizes GNS to adaptive optimizers, but Section 3.1 derives it "by closely following the original derivation of the GNS... starting from pre-conditioned SGD." The validity of this derivation for all adaptive optimizers (beyond Adam and AdaGrad) is not thoroughly validated.

### 6.3 Operational Trade-offs

#### Fairness vs. Performance Trade-off

Table 2 demonstrates a clear trade-off between fairness and performance:

| Fairness Knob ($p$) | Average JCT | 99%ile JCT | Makespan |
|---------------------|-------------|------------|----------|
| $p=+1$ (least fair) | 0.83h | 10h | 16h |
| $p=-1$ (default) | **0.76h** | 11h | 16h |
| $p=-10$ (most fair) | 0.84h | 12h | 18h |

Section 5.3.1 explains that $p=-10$ degrades performance because "PolluxSched incurring a larger number of re-allocations due to ignoring the cost in favor of equalizing speedups at all times." This reveals a fundamental tension:

- **High fairness ($p \to -\infty$)**: Prioritizes equal speedup across jobs but incurs more re-allocations, increasing checkpoint overhead.
- **High efficiency ($p=1$)**: Maximizes total cluster goodput but may starve jobs with poor scalability.

The default $p=-1$ achieves the best average JCT, but cluster operators must tune this parameter based on organizational priorities. The paper does not provide guidance on selecting $p$ for specific workload characteristics.

#### Scheduling Interval Sensitivity

Figure 8b shows Pollux's performance degrades with scheduling intervals beyond 2 minutes. Section 5.3.2 explains:

> "Since newly-submitted jobs can only start during the next scheduling interval, we would expect an increase in the average queuing time due to longer scheduling intervals. However, we find that queuing contributed to roughly half of the performance degradation observed, indicating that Pollux still benefits from a relatively frequent adjustment of resource allocations."

This creates an operational constraint:
- **Shorter intervals (< 60s)**: May increase scheduler CPU overhead (currently 1 second per 60s interval on 1 vCPU).
- **Longer intervals (> 2 min)**: Degrades JCT due to queuing delays and slower adaptation to cluster state changes.

The 60-second default appears well-chosen, but clusters with highly dynamic workloads (frequent job submissions/completions) may need to tune this parameter, adding operational complexity.

#### Interference Avoidance Reduces Packing Efficiency

Section 4.2 implements interference avoidance by ensuring "at most one distributed job is allocated to each node." Figure 8c shows this constraint protects against network contention but may reduce cluster packing efficiency:

- **With interference avoidance**: Average JCT unaffected by up to 50% slowdown injection.
- **Without interference avoidance**: Average JCT is 1.4× longer at 50% slowdown.

The trade-off is that nodes cannot host multiple distributed jobs simultaneously, even if they have spare GPUs. For example, a node with 4 GPUs running a 2-GPU distributed job cannot accept another 2-GPU distributed job, leaving 2 GPUs idle. This constraint is necessary for performance stability but reduces overall cluster utilization in scenarios where network interference is minimal.

### 6.4 Workload and Application Scope Limitations

#### Limited Model Diversity in Evaluation

Table 1 shows evaluation on 6 model types across common DL tasks (image classification, object detection, speech recognition, question answering, recommendation). However, several important workload categories are not evaluated:

- **Large language model pre-training**: Models with billions of parameters (e.g., GPT-scale) have different scaling characteristics than the fine-tuning tasks evaluated (BERT fine-tune is the largest NLP task).
- **Diffusion models**: Emerging generative models with different training dynamics are not covered.
- **Reinforcement learning**: RL workloads often have different resource patterns (e.g., environment simulation bottlenecks) not addressed.
- **Multi-modal models**: Models combining vision and language (e.g., CLIP, DALL-E) are not evaluated.

Section 5.4.2 evaluates HPO but notes "a full evaluation on how Pollux affects different HPO algorithm types is future work." The applicability of Pollux to diverse DL workloads beyond the 6 evaluated models remains an open question.

#### Cloud Auto-Scaling is Preliminary

Section 5.4.1 presents cloud cost reduction as a "new opportunity" but explicitly states:

> "We present some preliminary evidence using our cluster simulator, and note that a full design of an auto-scaling system based on goodput may be the subject of future work."

The 25% cost reduction for ImageNet training is promising but based on a "simple auto-scaling policy" with manually-tuned thresholds ($U=2/3$, $L=1/2$). A production cloud auto-scaler would require:
- Dynamic threshold adjustment based on spot instance availability and pricing.
- Integration with cloud provider APIs for instance provisioning.
- Handling of preemption and instance termination.

The current implementation demonstrates feasibility but is not a complete cloud solution.

#### Non-Adaptive Job Support is Limited

Section 4.2 states Pollux supports jobs with fixed batch sizes ($M=M_0$) by "fixing EFFICIENCY$_t$ for that job to 1 and can continue to adapt its resource allocations based solely on its system throughput." However, this essentially reduces Pollux to a throughput-only scheduler for these jobs, losing the goodput optimization benefit. Users who require fixed batch sizes (e.g., for reproducibility or specific convergence properties) cannot benefit from Pollux's core innovation.

### 6.5 Summary of Key Trade-offs

| Trade-off | Description | Evidence |
|-----------|-------------|----------|
| **Fairness vs. Performance** | Higher fairness ($p=-10$) increases re-allocation overhead, degrading JCT | Table 2: $p=-10$ has 11% worse average JCT than $p=-1$ |
| **Checkpoint Overhead vs. Adaptivity** | Dynamic re-allocation requires 8% runtime overhead for checkpointing | Section 5.2: 8% overhead, once per 7 minutes |
| **Packing Efficiency vs. Interference Avoidance** | One distributed job per node constraint reduces GPU packing | Section 4.2, Figure 8c |
| **Model Simplicity vs. Accuracy** | Linear throughput model may diverge at larger scales | Section 3.2: "may diverge from reality... at larger scales" |
| **Heterogeneity Support** | No support for mixed GPU types in current implementation | Footnote 1, Section 2.3 |
| **Short Job Overhead** | Checkpoint delay (15–120s) is significant fraction of short job runtime | Section 5.3.2: 2–5% improvement for short jobs with offline fitting |

### 6.6 Open Questions

Several important questions remain unanswered:

1. **How does Pollux perform at 1000+ GPU scales?** The 64-GPU evaluation does not capture network topology effects at LLM-training scales.
2. **What are the long-term model quality implications?** While short-term validation metrics are similar (±1–4%), the paper does not track whether dynamic batch size changes affect generalization over extended training.
3. **How does Pollux interact with other cluster optimizations?** The evaluation isolates Pollux but does not study interactions with gang scheduling, preemption, or priority-based scheduling.
4. **What is the optimal fairness knob ($p$) for specific workload mixes?** The paper recommends $p=-1$ but does not provide a methodology for tuning based on workload characteristics.
5. **Can the throughput model be extended to pipeline or tensor parallelism?** Current evaluation is limited to data-parallelism; emerging large model training uses hybrid parallelism strategies.

These limitations do not invalidate Pollux's contributions but highlight areas where future work can extend its applicability. The paper is transparent about these constraints, and the modular design (e.g., plug-in LR scaling, modular GOODPUT definition) provides pathways for addressing them.

## 7. Implications and Future Directions

Pollux represents a paradigm shift in how deep learning (DL) clusters are managed, moving from static resource allocation to dynamic **co-optimization** of resources and training configurations. By treating the training job as an adaptive process rather than a fixed container of work, Pollux demonstrates that significant efficiency gains (37–50% reduction in average job completion time, Table 2) are possible without requiring users to be experts in system scalability. This section outlines how this work changes the landscape of cluster scheduling, identifies specific avenues for future research, and provides practical guidance for engineers considering adoption or extension of these ideas.

### 7.1 Shifting the Scheduling Paradigm

The primary implication of Pollux is the validation of **goodput** as a superior optimization objective compared to traditional system throughput. Prior schedulers like Optimus [52] operate on the assumption that maximizing hardware utilization (throughput) minimizes completion time. Pollux disproves this by showing that throughput often comes at the cost of statistical efficiency, especially when batch sizes are increased blindly (Section 4.1).

This shift has three broader implications for the field:

*   **From Allocation to Configuration:** Traditional schedulers focus solely on assigning resources (e.g., "Job A gets 4 GPUs"). Pollux demonstrates that schedulers must also manage **training hyperparameters** (batch size, learning rate, gradient accumulation). This blurs the line between system scheduling and algorithmic tuning, suggesting future schedulers must be aware of the training loop's internal state.
*   **From Static to Dynamic Profiles:** The paper shows that a job's optimal configuration changes over time. As seen in Figure 6, a job may require different GPU counts and batch sizes at epoch 1 versus epoch 50 due to changes in gradient noise scale ($\phi_t$). This implies that profiling jobs once at submission time (as done in many production systems) is insufficient; continuous online monitoring is required.
*   **Collaborative Scheduling:** Pollux introduces a **co-adaptive architecture** where the scheduler (`PolluxSched`) and the job (`PolluxAgent`) communicate bidirectionally (Section 3.4). This contrasts with prior "black box" scheduling where the job passively accepts resources. Future systems may adopt similar agent-based architectures to expose internal job metrics (like loss curves or gradient statistics) to the cluster manager.

### 7.2 Avenues for Future Research

While Pollux establishes a strong baseline, the limitations identified in Section 6 point to specific, high-impact research directions. The modular design of Pollux (e.g., the plug-in learning rate interface and modular `GOODPUT` definition) intentionally leaves room for these extensions.

#### Heterogeneity-Aware Goodput
Pollux currently assumes homogeneous GPU clusters (Footnote 1, Section 2.3). In production environments, clusters often contain mixed hardware (e.g., V100, A100, T4).
*   **Research Opportunity:** Integrate heterogeneity-aware throughput metrics, such as those proposed by Gavel [48], into Pollux's `THROUGHPUT` model. The challenge lies in adapting the statistical efficiency model ($\text{EFFICIENCY}_t$) to account for how different accelerator architectures affect gradient noise scale ($\phi_t$).
*   **Expected Impact:** Extending Pollux to heterogeneous clusters would broaden its applicability to cloud environments where instance types vary widely.

#### Scaling Beyond 64 GPUs
The evaluation is limited to 64 GPUs (Section 5.1). The linear throughput model (Equation 11) may diverge from reality at larger scales due to complex network topologies (e.g., InfiniBand fabrics, multi-rack communication) and non-linear straggler effects.
*   **Research Opportunity:** Develop hierarchical throughput models that account for rack-level and pod-level communication costs. Additionally, investigate how `PolluxSched`'s search algorithm scales when optimizing allocations for thousands of GPUs simultaneously.
*   **Expected Impact:** Enables goodput-optimized scheduling for large language model (LLM) pre-training, which often requires hundreds or thousands of GPUs.

#### Beyond Data Parallelism
Pollux focuses on synchronous data-parallelism (Section 2.1). Modern large models often rely on pipeline parallelism (e.g., GPipe [28], PipeDream [47]) or tensor parallelism.
*   **Research Opportunity:** Extend the `GOODPUT` formulation to model the throughput and efficiency of pipeline-parallel training. This would require modeling stage imbalance and bubble efficiency in addition to batch size and learning rate.
*   **Expected Impact:** Makes goodput optimization applicable to the largest-scale model training workloads where data parallelism alone is insufficient.

#### Formal Model Quality Guarantees
Pollux relies on user-provided maximum batch size limits to prevent model quality degradation (Section 3.1). While empirical results show validation metrics within ±1–4% (Figure 2), there are no formal guarantees.
*   **Research Opportunity:** Develop theoretical bounds on how much dynamic batch size variation a model can tolerate without harming convergence or generalization. This could involve integrating convergence proofs from adaptive optimization literature (e.g., AdaScale [32]) directly into the scheduler's constraints.
*   **Expected Impact:** Increases trust in automated scheduling for critical production models where quality regression is unacceptable.

### 7.3 Practical Applications and Downstream Use Cases

The techniques introduced in Pollux have immediate applicability beyond general cluster scheduling. The authors highlight two specific downstream use cases that leverage the goodput metric.

#### Cloud Cost Optimization
In cloud environments, users pay for resource duration. Pollux's ability to delay resource scaling until statistical efficiency improves (later training stages) offers a direct cost-saving mechanism.
*   **Evidence:** Section 5.4.1 shows a Pollux-based auto-scaler reduces cloud training costs for ImageNet by **25%** compared to throughput-based auto-scaling, with only a 6% increase in completion time.
*   **Application:** Cloud providers or managed ML platforms (e.g., AWS SageMaker, Google Vertex AI) could integrate goodput models to offer "cost-optimized" training tiers that automatically scale resources based on training progress rather than fixed rules.

#### Hyper-Parameter Optimization (HPO) Acceleration
HPO workloads involve running many trials concurrently. Pollux can dynamically allocate more resources to promising trials as they progress, rather than static allocation.
*   **Evidence:** Section 5.4.2 reports Pollux completes HPO workloads **30% faster** (10h vs 14h makespan) than static baselines while achieving similar accuracy (95.4% vs 95.5%).
*   **Application:** HPO frameworks (e.g., Ray Tune, Optuna) could use Pollux's agent logic to dynamically resize trials. For example, a trial that exhibits high gradient noise scale (indicating room for larger batches) could be automatically granted more GPUs mid-training.

### 7.4 Reproduction and Integration Guidance

For researchers and engineers looking to reproduce Pollux or integrate its ideas into existing systems, the following practical considerations are critical.

#### When to Prefer Pollux
Pollux is most beneficial in environments that match its design assumptions:
*   **Workload Type:** Batch-style DL training jobs (classification, detection, NLP) that support checkpointing. It is **not** suitable for workloads that cannot checkpoint (e.g., some reinforcement learning or online learning scenarios) due to the 8% re-allocation overhead (Section 5.2).
*   **Cluster Homogeneity:** Clusters with uniform GPU types. If the cluster is highly heterogeneous, the throughput model may mispredict performance.
*   **Job Duration:** Jobs long enough to amortize checkpoint costs. For very short jobs (e.g., < 10 minutes), the 15–120 second checkpoint-restart delay (Section 4.2) may outweigh scheduling benefits. Section 5.3.2 notes only 2–5% improvement for short jobs even with offline model fitting.

#### Integration Requirements
To deploy Pollux or a similar system, the following infrastructure capabilities are required:
1.  **Checkpoint-Restart Support:** The training framework (e.g., PyTorch, TensorFlow) must support saving and loading state efficiently. Pollux uses this for resource re-allocation (Section 4.3).
2.  **Gradient Access:** The `PolluxAgent` requires access to per-GPU gradient estimates to compute PGNS ($\phi_t$). This requires hooking into the optimizer step (Section 3.1).
3.  **Kubernetes Orchestration:** The reference implementation relies on Kubernetes for pod creation/termination (Section 4.3). Adapting to other orchestrators (e.g., Slurm, YARN) would require re-implementing the resource allocation interface.

#### Reproduction Resources
The authors provide open-source artifacts to facilitate reproduction:
*   **Implementation:** The full Pollux implementation, benchmark models, and testbed scripts are available at `https://github.com/petuum/adaptdl` (Section 5.5).
*   **Results:** Raw logs and analysis scripts for the testbed experiments are provided at `https://github.com/petuum/pollux-results` (Section 5.5).
*   **Simulator:** The discrete-time cluster simulator (Section 5.3) allows researchers to test scheduling policies without access to a 64-GPU cluster, enabling broader experimentation with workload traces.

#### Recommended Starting Point
For teams interested in adopting goodput-based scheduling without replacing their entire scheduler:
1.  **Start with the Agent:** Implement the `PolluxAgent` logic (Section 4.1) within existing training jobs to collect PGNS and throughput data. This provides visibility into how statistical efficiency changes during training.
2.  **Validate Efficiency Models:** Use the collected data to validate Equation 6 ($\text{EFFICIENCY}_t(M)$) for your specific models. If the model fits well, the foundation for goodput optimization is present.
3.  **Incremental Automation:** Before full co-adaptivity, use the models to provide **recommendations** to users (e.g., "Consider increasing batch size to X at epoch Y") rather than automatic enforcement. This builds trust before enabling automatic resource re-allocation.

By treating deep learning training as a dynamic, measurable process rather than a static black box, Pollux opens the door for a new generation of "learning-aware" systems that optimize for what actually matters: model progress, not just hardware utilization.
