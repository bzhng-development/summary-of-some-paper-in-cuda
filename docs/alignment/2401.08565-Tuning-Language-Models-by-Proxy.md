## 1. Executive Summary

This paper introduces `proxy-tuning`, a decoding-time algorithm that customizes large black-box language models by applying the logit difference between a small tuned expert and its untuned anti-expert to the base model's predictions. Addressing the challenge of resource-intensive or impossible weight access for proprietary LMs, the method closes 88% of the performance gap between LLAMA2-70B and its directly tuned CHAT version using only 7B parameter proxies. The approach demonstrates generality across instruction-following, code adaptation (17–32% absolute improvement), and temporal adaptation on truly black-box models like GPT-3.5.

## 2. Context and Motivation

### The Scaling vs. Customization Dilemma
The landscape of natural language processing has shifted toward increasingly large pretrained language models (LMs). As models scale to tens or hundreds of billions of parameters, their general capabilities improve reliably. However, raw pretrained models often fail to align with specific user needs without further adaptation. For instance, a base model might complete a prompt factually but fail to follow instructions, adhere to safety guidelines, or adopt a specific domain style (e.g., medical or legal text). This necessitates a secondary phase of adaptation, commonly known as **finetuning**.

The core problem this paper addresses is the growing disconnect between the *scale* of models users want to utilize and the *feasibility* of customizing them. As noted in Section 1, tuning these models has become increasingly resource-intensive. More critically, it is becoming impossible when model weights are private. Proprietary models like GPT-4 are accessible only via API, meaning users cannot access the internal parameters required for traditional finetuning. This creates a bottleneck: users are stuck with the default behavior of large black-box models, unable to efficiently customize them for diverse applications without retraining from scratch or relying on costly proprietary finetuning services.

### Why This Problem Matters
The inability to customize large black-box models has both practical and theoretical implications:

*   **Accessibility and Democratization:** If only organizations with massive compute clusters can tune 70B parameter models, customization becomes the privilege of a few. A method that allows customization without weight access lowers the barrier to entry.
*   **Preservation of Knowledge:** Directly updating model weights (finetuning) can sometimes degrade existing capabilities, a phenomenon often referred to as the "alignment tax" or catastrophic forgetting (Section 9). There is a theoretical interest in methods that can steer model behavior without permanently altering the underlying knowledge stored in the weights.
*   **Proprietary Constraints:** As more powerful models are released as closed-source APIs, the community needs mechanisms to adapt these models without violating terms of service or requiring unauthorized access to weights. Section 7 demonstrates this by applying the method to GPT-3.5, a truly black-box model where only limited output information is available.

### Limitations of Prior Approaches
Before `proxy-tuning`, several strategies existed to adapt language models, each with distinct shortcomings regarding cost, access, or performance.

#### 1. Direct Finetuning (White-Box)
The standard approach involves updating the model's weights using gradient descent on a task-specific dataset.
*   **Requirement:** Full access to model parameters (white-box).
*   **Cost:** Computationally expensive, especially for models larger than 7B parameters.
*   **Limitation:** Impossible for proprietary models where weights are hidden. Even with access, tuning a 70B model requires significant GPU memory and time.

#### 2. Parameter-Efficient Finetuning (PEFT)
Methods like LoRA (Low-Rank Adaptation) update only a small subset of parameters while freezing the base model (Section 8).
*   **Requirement:** Still requires white-box access to load the base model and inject adapters.
*   **Limitation:** While cheaper than full finetuning, it does not solve the black-box access problem. Section D compares `proxy-tuning` to LoRA, noting that LoRA requires resources to load the large model parameters, whereas `proxy-tuning` only requires output logits.

#### 3. In-Context Learning (Prompting)
Users can guide model behavior by providing examples or instructions within the prompt (Section 8).
*   **Requirement:** Only requires API access (black-box compatible).
*   **Limitation:** Effective prompts can be very long, consuming valuable context window space and increasing inference latency and cost. Section 8 notes that curated prompts can be competitive but introduce an "inference-time computational burden."

#### 4. Decoding-Time Control (DEXPERTS)
Prior work such as DEXPERTS (Liu et al., 2021) introduced the idea of modifying output logits at generation time using "expert" and "anti-expert" models (Section 2).
*   **Mechanism:** Mathematically combines logits from multiple models to steer generation (e.g., reducing toxicity).
*   **Limitation:** Previous applications focused primarily on control attributes (like toxicity or sentiment) rather than general task performance or instruction following. Additionally, contemporary work (Mitchell et al., 2024) applied similar equations but focused on disentangling scaling effects rather than measuring effectiveness on standard benchmarks (Section 8).

### Positioning of This Work
This paper positions `proxy-tuning` as a bridge between the efficiency of decoding-time methods and the performance of direct finetuning. It distinguishes itself from prior work in three key ways:

1.  **Objective Shift:** While prior decoding-time methods (like DEXPERTS) focused on control attributes (e.g., making text less toxic), this work repurposes the technique to simulate **finetuning**. The goal is not just control, but to close the performance gap between a base model and a tuned model (Section 1).
2.  **Black-Box Compatibility:** Unlike PEFT or direct finetuning, `proxy-tuning` operates solely on output logits. Section 2 explicitly states the assumption: "For arbitrary inputs to $M$, we assume that we can access the output logits for the entire vocabulary." This makes it applicable to models where weights are private, provided logits are visible.
3.  **Empirical Generality:** Whereas contemporary analysis (Mitchell et al., 2024) used similar math for theoretical disentanglement, this paper focuses on empirical strength across diverse tasks: instruction-following, code adaptation, task-specific finetuning, and temporal adaptation (Section 1). It demonstrates that small tuned models can effectively steer much larger base models, challenging the assumption that tuning must happen at the same scale as deployment.

By leveraging the difference between a small tuned model and its untuned counterpart, the method applies the "direction" of tuning to a larger model without ever touching the larger model's weights. This offers a pathway to customize proprietary, large-scale LMs efficiently, addressing the resource and access barriers identified in Section 1.

## 3. Technical Approach

### 3.1 Reader orientation (approachable technical breakdown)
The system is a decoding-time algorithm that modifies the token probability distribution of a large base model by adding a mathematical offset derived from the difference between a small tuned model and its untuned counterpart. It solves the problem of customizing inaccessible large models by transferring the behavioral "direction" learned by a small model onto the large model's outputs without ever modifying the large model's weights.

### 3.2 Big-picture architecture (diagram in words)
The architecture consists of three language models running in parallel during the generation process: a large base model ($M$), a small tuned expert model ($M_+$), and a small untuned anti-expert model ($M_-$). Each model independently processes the current input sequence to produce raw logit scores (unnormalized probabilities) over the shared vocabulary. A combiner module then calculates the difference between the expert and anti-expert logits and adds this difference as an offset to the base model's logits before applying the softmax function to select the next token.

### 3.4 Detailed, sentence-based technical breakdown
This is a methods paper proposing a decoding-time algorithm that leverages logit arithmetic to simulate finetuning effects on black-box models. The core innovation lies in isolating the "update direction" caused by tuning a small model and applying that same directional shift to a larger model's predictions.

**The Core Mathematical Mechanism**
The fundamental operation occurs at every time step $t$ during text generation, where the goal is to compute a new probability distribution for the next token. Let $x_{<t}$ represent the sequence of tokens generated so far (the context). The system queries three models to obtain their logit scores, which are the raw, unnormalized outputs from the language modeling head before the softmax function is applied. We denote the logit scores for the base model as $s_M$, the expert model as $s_{M+}$, and the anti-expert model as $s_{M-}$. The probability distribution for the proxy-tuned model $\tilde{M}$ is calculated using the following equation, which appears as Equation (1) in Section 2:

$$
p_{\tilde{M}}(X_t | x_{<t}) = \text{softmax} [s_M(X_t | x_{<t}) + s_{M+}(X_t | x_{<t}) - s_{M-}(X_t | x_{<t})]
$$

In plain language, this formula takes the large model's original prediction and adds the difference between the small tuned model and the small untuned model. The term $(s_{M+} - s_{M-})$ represents the specific change in logits caused by the tuning process itself, effectively isolating the "tuning signal" from the underlying knowledge. By adding this signal to $s_M$, the system steers the large model toward the behavior of the expert while retaining the large model's pretrained knowledge base. An alternative interpretation groups the terms as $s_{M+} + (s_M - s_{M-})$, which views the operation as giving the small expert the benefit of the large model's scaling advantages through contrastive decoding.

**Component Roles and Configuration**
The success of this method relies on the specific configuration of the three models involved in the ensemble. The base model $M$ is the large pretrained model that the user wishes to customize, such as LLAMA2-70B, and it remains completely frozen with no parameter updates. The expert model $M_+$ is a smaller model that has been directly tuned for the target task, such as LLAMA2-7B-CHAT for instruction following or CODELLAMA-7B for coding tasks. The anti-expert model $M_-$ is the untuned counterpart to the expert, such as LLAMA2-7B, which shares the same architecture and initialization as the expert but lacks the task-specific tuning. It is critical that $M_+$ and $M_-$ are closely matched architectures so that their logit difference reflects only the effects of tuning rather than architectural biases. The paper specifies in Section 2 that these models must share the same vocabulary to ensure the logit vectors are aligned token-for-token, though techniques exist to map between vocabularies if necessary.

**Step-by-Step Generation Process**
The generation process proceeds autoregressively, meaning it generates one token at a time in a loop until a stop condition is met. At the first step, the input prompt is tokenized and fed simultaneously into the base model, the expert, and the anti-expert. Each model performs a forward pass to produce a vector of logit scores with a size equal to the vocabulary (e.g., 32,000 for LLAMA2). The system computes the offset vector by subtracting the anti-expert logits from the expert logits element-wise. This offset vector is then added to the base model's logit vector. The resulting combined logit vector is passed through a softmax function to convert the scores into a valid probability distribution summing to 1.0. The system then selects the next token using a decoding strategy, such as greedy decoding (selecting the highest probability token) or sampling. This selected token is appended to the input sequence, and the entire process repeats for the next time step $t+1$ using the updated context.

**Vocabulary and Logit Accessibility Constraints**
A strict requirement for the standard implementation is access to the full logit distribution over the vocabulary for all three models. Section 2 footnote 3 notes that tokenizers are often open-source even for closed models, making vocabulary alignment feasible. However, the paper also addresses a more constrained scenario in Section 7 involving truly black-box models like GPT-3.5, where the API provides log probabilities for only the top 5 tokens rather than the full vocabulary. In this limited-information setting, the method adapts by restricting the reweighting operation to only the available tokens. For example, in a multiple-choice setting with options {A, B, C, D}, the system computes the proxy-tuning offset using the small models and applies it only to reweigh the probabilities of those four specific tokens provided by the black-box API. This demonstrates that the method can function even when the full logit vector is not accessible, provided the tokens of interest are visible.

**Steering Strength Hyperparameter**
While the primary experiments use the standard equation with equal weighting, Section 6.2 introduces an optional hyperparameter $\alpha$ to control the intensity of the tuning effect. The modified equation becomes $s_M + \alpha(s_{M+} - s_{M-})$, where $\alpha$ scales the offset vector before it is added to the base logits. Setting $\alpha > 1$ magnifies the contrast between the expert and anti-expert, exerting stronger guidance on the base model, while $\alpha < 1$ reduces the influence of the tuning signal. The paper reports in Section 6.2 that varying $\alpha$ between 0.2 and 2.0 allows for a smooth trade-off between attributes like truthfulness and informativeness. Specifically, on TruthfulQA, increasing $\alpha$ consistently improved truthfulness scores, whereas informativeness peaked at $\alpha = 0.4$, suggesting that excessive tuning pressure might cause the model to decline answering questions too frequently.

**Computational Implications and Parallelization**
Implementing this approach introduces computational overhead because three models must perform a forward pass for every generated token instead of one. Section C.1 analyzes this runtime cost, reporting a slowdown factor of approximately 2.54× for 13B models and 1.58× for 70B models when running sequentially on single GPU setups. However, the paper clarifies that this slowdown is not inherent to the algorithm but rather a result of sequential execution. In a production environment, the three models can be deployed on separate GPUs that communicate via an all-reduce operation. In this parallel configuration, the forward passes occur simultaneously, and only the logit vectors (which are small relative to model weights) are communicated between devices. The authors note in Section C.1 that their pilot implementation of this parallel approach achieves runtime similar to a true tuned model, mitigating the latency concern for practical deployment.

## 4. Key Insights and Innovations

This section identifies the fundamental conceptual advances that distinguish `proxy-tuning` from prior work. While the mathematical mechanism (logit arithmetic) builds on DEXPERTS (Section 3), the paper's contributions extend far beyond applying an existing equation to a new task. The innovations lie in what the method reveals about model tuning, scale, and knowledge preservation.

---

### 4.1 Scale-Decoupled Tuning Transfer

**The Innovation:** The paper demonstrates that the *direction* of tuning—the logit difference between a tuned and untuned model—can be transferred across vastly different model scales. A 7B expert can effectively steer a 70B base model, closing 88% of the performance gap to a directly tuned 70B-CHAT model (Table 2, Section 3.2).

**Why This Differs from Prior Work:** Previous decoding-time methods like DEXPERTS (Liu et al., 2021) used expert models primarily for attribute control (toxicity, sentiment) at roughly similar scales. Parameter-efficient finetuning methods like LoRA require the adapter to be injected into the *same* model being deployed (Section 8). The assumption underlying most adaptation work is that tuning must occur at the deployment scale—if you want a 70B model to follow instructions, you must tune a 70B model.

**Significance:** This finding challenges a fundamental assumption in the field. Table 2 shows that proxy-tuning a 70B base model with 7B proxies achieves 88.0% on AlpacaFarm, only 2.4% behind the directly tuned 70B-CHAT model (90.4%). This suggests that the *behavioral shift* induced by tuning is somewhat scale-invariant—the small model learns a "direction" in logit space that remains meaningful when applied to a larger model's predictions. Practically, this means organizations can tune small, cheap models and deploy the steering effect on large, proprietary models they cannot directly modify.

---

### 4.2 Knowledge Preservation Through Decoding-Time Guidance

**The Innovation:** Proxy-tuning may preserve pretrained knowledge better than direct weight updates. On TruthfulQA, proxy-tuned models are *more truthful* than directly tuned CHAT models at both 13B and 70B scale (Table 3, Section 3.2).

**Why This Differs from Prior Work:** Direct finetuning is known to sometimes degrade existing capabilities—a phenomenon termed the "alignment tax" (Ouyang et al., 2022, cited in Section 9). When you update weights to improve instruction-following, you risk overwriting knowledge encoded during pretraining. Prior decoding-time methods focused on control attributes rather than measuring knowledge retention. This paper provides empirical evidence that decoding-time guidance avoids this degradation.

**Significance:** Table 3 reveals a critical detail: at 70B scale, proxy-tuned models achieve 92.3% truthfulness compared to 85.8% for directly tuned CHAT models—a 6.5% absolute improvement. The paper hypothesizes in Section 3.2 that "decoding-time algorithms may provide an avenue for better knowledge preservation, whereas direct tuning has been shown to sometimes hurt performance on knowledge-intensive tasks." This is not just a performance gain; it suggests a fundamental trade-off between *modifying weights* and *guiding outputs*. For applications where factual accuracy is critical (medical, legal, educational), proxy-tuning offers a path to alignment without sacrificing the knowledge base.

---

### 4.3 Black-Box Customization of Truly Proprietary Models

**The Innovation:** The method functions even when only the top-5 log probabilities are available, demonstrated by successfully adapting GPT-3.5 for temporal knowledge (Section 7).

**Why This Differs from Prior Work:** Most adaptation methods require either full weight access (finetuning, LoRA) or at least full logit access (standard DEXPERTS). Section 7 addresses a more constrained setting: GPT-3.5's API provides log probabilities for only the top 5 tokens at each step, and does not allow conditioning on partial responses. Prior decoding-time methods would fail under these constraints.

**Significance:** Table 7 shows that proxy-tuning improves GPT-3.5's accuracy on REALTIMEQA from 54.2% to 56.5%—a statistically significant 2.3% gain (p < 0.0001). This is notable because the expert and anti-expert (both 7B models) are *weaker* than GPT-3.5 on this task, yet contrasting their predictions still yields a positive signal. The paper identifies this as "an instance of weak-to-strong generalization" (Section 7). Practically, this proves that users can customize proprietary models without violating terms of service or requiring unauthorized weight access. As more powerful models are released as closed APIs, this capability becomes increasingly valuable.

---

### 4.4 Generality Across Adaptation Types

**The Innovation:** Proxy-tuning works consistently across four distinct adaptation scenarios: instruction-tuning, domain adaptation (code), task-specific finetuning, and temporal adaptation.

**Why This Differs from Prior Work:** Contemporary work using similar equations (Mitchell et al., 2024, cited in Section 8) focused primarily on instruction-tuning for theoretical analysis. Other decoding-time methods target narrow attributes like toxicity or sentiment. This paper systematically evaluates the method across fundamentally different adaptation goals, each with different success metrics and constraints.

**Significance:** The results span multiple sections and demonstrate consistent effectiveness:
- **Instruction-tuning (Section 3):** Closes 88–91% of the gap to directly tuned CHAT models (Table 2).
- **Code adaptation (Section 4):** Achieves 17–32% absolute improvement on coding benchmarks (Table 4).
- **Task finetuning (Section 5):** Closes 84–87% of the gap to task-specific experts, and enables base models to follow strict formatting constraints seen only by the small expert (Section 5.2).
- **Temporal adaptation (Section 7):** Improves accuracy on recent events for GPT-3.5 (Table 7).

This generality suggests that proxy-tuning captures something fundamental about how tuning changes model behavior, rather than being optimized for a single use case. The finding in Section 5.2 that proxy-tuned models follow strict formatting constraints (99.7%+ of GSM generations state the final answer after "####") is particularly notable—it shows the method can promote "extremely unlikely tokens to the top of the probability distribution," enabling pretrained models to learn originally unlikely task formats.

---

### 4.5 Distinguishing Incremental vs. Fundamental Contributions

| Contribution | Type | Reasoning |
|--------------|------|-----------|
| Logit arithmetic equation | Incremental | Builds directly on DEXPERTS (Liu et al., 2021); the math itself is not new. |
| Scale-decoupled transfer | Fundamental | Challenges the assumption that tuning must occur at deployment scale. |
| Knowledge preservation advantage | Fundamental | Provides empirical evidence for a theoretical trade-off between weight updates and output guidance. |
| Top-5 logit adaptation | Fundamental | Demonstrates functionality under constraints where prior methods would fail. |
| Cross-task generality | Fundamental | Shows the method captures general properties of tuning, not task-specific tricks. |

The paper's primary innovation is not the equation itself, but the *empirical demonstration* that small tuned models can steer large black-box models effectively across diverse tasks while preserving knowledge. This shifts the conversation from "how do we tune large models?" to "how do we transfer tuning signals across scales?"—a fundamentally different research direction with implications for accessibility, efficiency, and model safety.

## 5. Experimental Analysis

This section examines how the authors validate `proxy-tuning` across multiple adaptation scenarios. The experimental design is comprehensive, covering instruction-following, domain adaptation, task-specific finetuning, and temporal adaptation on truly black-box models. Rather than testing on a single benchmark, the authors systematically evaluate whether small tuned models can steer larger base models across fundamentally different tasks. This breadth is critical—if the method only worked for instruction-tuning, it would be a narrow trick. If it works across diverse adaptation types, it suggests a more fundamental mechanism for transferring tuning signals.

### 5.1 Evaluation Methodology

**Model Configurations**
The experiments use the LLAMA2 model family across three scales: 7B, 13B, and 70B parameters. For instruction-tuning experiments (Section 3), the configuration is consistent:
- **Base Model ($M$):** LLAMA2-13B or LLAMA2-70B (pretrained, untuned)
- **Expert ($M_+$):** LLAMA2-7B-CHAT (instruction-tuned)
- **Anti-Expert ($M_-$):** LLAMA2-7B (pretrained, untuned)

This setup isolates the effect of tuning by keeping the expert and anti-expert at the same scale (7B), differing only in their tuning state. The base model is always larger (13B or 70B), testing whether tuning signals transfer across scales.

For code adaptation (Section 4), the expert changes to CODELLAMA-7B-PYTHON, which was initialized from LLAMA2-7B and further trained on code. For task-specific finetuning (Section 5), the authors train their own 7B task experts on TriviaQA and GSM training sets. The GPT-3.5 case study (Section 7) uses LLAMA2-7B models as proxies to steer a completely separate model family.

**Datasets and Metrics**
The evaluation spans five distinct task categories, each with appropriate metrics:

| Task Category | Datasets | Primary Metric | What It Tests |
|--------------|----------|----------------|---------------|
| Instruction-following | AlpacaFarm, GSM, ToxiGen, TruthfulQA | Win rate, Accuracy, % Toxic, % Info + True | General alignment and safety |
| Code adaptation | CodexEval, DS-1000 | pass@10 | Domain-specific capability |
| Task finetuning | TriviaQA, GSM | Exact match accuracy | Task-specific formatting and reasoning |
| Temporal adaptation | REALTIMEQA | Multiple-choice accuracy | Knowledge updating for black-box models |
| Runtime analysis | Synthetic prompts | Seconds per generation | Practical deployment costs |

For TruthfulQA, the authors report both multiple-choice accuracy and open-ended performance measured as the percentage of responses that are both informative and truthful (% Info + True). Table 3 provides a finer breakdown separating informativeness (% Info) from truthfulness (% True), which is crucial for understanding the knowledge preservation claim.

For code benchmarks, the metric is pass@10, which measures the probability that at least one of 10 sampled solutions passes all test cases. This follows the standard evaluation protocol from Chen et al. (2021) and accounts for the stochastic nature of code generation.

**Baselines and Comparisons**
Each experiment includes three key comparison points:
1. **Base (untuned):** The large pretrained model without any adaptation (e.g., LLAMA2-70B)
2. **Directly tuned:** A model of the same scale as the base that has been directly finetuned (e.g., LLAMA2-70B-CHAT)
3. **Small expert alone:** The 7B tuned model used as the expert, evaluated independently

This three-way comparison allows the authors to measure both (a) how much of the tuning gap is closed relative to direct finetuning, and (b) whether proxy-tuning provides benefits beyond just using the small expert model. The "gap closed" metric is calculated as:

$$
\text{Gap Closed} = \frac{\text{Proxy-tuned} - \text{Base}}{\text{Directly tuned} - \text{Base}}
$$

This normalizes performance relative to the maximum achievable improvement from tuning.

**Experimental Setup Details**
The authors use zero-shot prompting across all models, noting in Appendix A that LLAMA2-CHAT models struggle to follow in-context example formats. Greedy decoding is used for most experiments except code generation, which uses sampling (top-$p = 0.95$, temperature = 0.8) to compute pass@10. For TruthfulQA chat models, a system prompt from the original LLAMA2 paper is applied to clarify desired behavior (Table 9, Appendix A).

### 5.2 Main Quantitative Results

**Instruction-Tuning Performance (Table 2, Section 3.2)**
The instruction-tuning results provide the strongest evidence for proxy-tuning's effectiveness. Table 2 shows performance across five evaluation settings for 13B and 70B models:

| Model | AlpacaFarm (Win %) | GSM (Acc %) | ToxiGen (% Toxic) | TruthfulQA (MC %) | TruthfulQA (Info+True %) |
|-------|-------------------|-------------|-------------------|-------------------|-------------------------|
| **13B Base** | 2.1 | 6.6 | 70.4 | 38.6 | 49.1 |
| **13B Proxy-tuned** | 83.4 | 26.4 | 0.1 | 57.4 | 82.0 |
| **13B Directly tuned** | 87.3 | 32.4 | 0.0 | 61.6 | 80.4 |
| **70B Base** | 3.7 | 9.6 | 67.4 | 42.3 | 53.9 |
| **70B Proxy-tuned** | 88.0 | 32.0 | 0.0 | 59.2 | 85.1 |
| **70B Directly tuned** | 90.4 | 51.8 | 0.0 | 68.3 | 79.6 |

Several observations stand out:

1. **Dramatic improvement over base models:** Proxy-tuning transforms nearly unusable base models into competitive instruction-followers. For example, 70B-Base achieves only 3.7% win rate on AlpacaFarm, while proxy-tuned reaches 88.0%—a 84.3 percentage point improvement.

2. **High gap closure rates:** The authors report that proxy-tuning closes 91.1% of the gap at 13B scale and 88.1% at 70B scale on average across all five settings. This means the method achieves nearly 90% of the benefit of direct finetuning without accessing the large model's weights.

3. **Proxy-tuned outperforms small expert:** In all scenarios except ToxiGen (0.1% difference), proxy-tuning the larger model outperforms the 7B expert alone. This confirms that the method successfully combines tuning benefits with larger pretraining scale.

4. **Toxicity elimination:** Base models generate toxic content 67–70% of the time on ToxiGen, while proxy-tuning reduces this to 0% at both scales—matching directly tuned models.

**Knowledge Preservation on TruthfulQA (Table 3, Section 3.2)**
Table 3 provides the critical evidence for the knowledge preservation claim by separating truthfulness from informativeness:

| Model | % Informative | % Truthful |
|-------|--------------|------------|
| **13B Base** | 90.7 | 56.9 |
| **13B Proxy-tuned** | 91.4 | 90.5 |
| **13B Directly tuned** | 93.0 | 87.3 |
| **70B Base** | 93.6 | 60.0 |
| **70B Proxy-tuned** | 92.8 | 92.3 |

The key finding: at 70B scale, proxy-tuned models achieve 92.3% truthfulness compared to 85.8% for directly tuned models—a 6.5 percentage point advantage. However, proxy-tuned models are slightly less informative (92.8% vs 93.8%). This trade-off supports the hypothesis that direct weight updates may degrade some pretrained knowledge, while decoding-time guidance preserves it. The authors note in Section 3.2 that "improvement in truthfulness suggests that decoding-time algorithms may provide an avenue for better knowledge preservation."

**Code Adaptation Results (Table 4, Section 4.2)**
For domain adaptation to code, proxy-tuning uses CODELLAMA-7B-PYTHON as the expert:

| Model | CodexEval (pass@10) | DS-1000 (pass@10) |
|-------|---------------------|-------------------|
| **7B Directly tuned** | 68.9 | 53.6 |
| **13B Base** | 33.7 | 26.2 |
| **13B Proxy-tuned** | 65.7 | 42.8 |
| **13B Directly tuned** | 78.6 | 56.9 |
| **70B Base** | 62.0 | 43.9 |
| **70B Proxy-tuned** | 70.7 | 50.6 |
| **70B Directly tuned** | 89.2 | 67.6 |

The absolute improvements are substantial: 32.0% on CodexEval and 16.6% on DS-1000 for 13B models. However, unlike instruction-tuning, proxy-tuned models do not outperform the small expert alone on code tasks. The authors hypothesize in Section 4.2 that "generic pretraining at a larger scale is not helpful when the model has already been tuned for a particular domain." This is an important boundary condition—proxy-tuning excels when the base model's general knowledge complements the expert's specialized behavior, but offers less benefit when the task requires domain-specific knowledge the base model lacks.

The most constrained evaluation involves GPT-3.5, where only top-5 log probabilities are available:

| Model | REALTIMEQA Accuracy |
|-------|---------------------|

The authors identify this as "an instance of weak-to-strong generalization" (Section 7), demonstrating that proxy-tuning can extract useful steering signals even when the proxy models are inferior to the base model.

### 5.3 Analysis and Ablation Studies

**Token-Level Influence Analysis (Section 6.1)**
The authors analyze which tokens are most affected by proxy-tuning to understand what aspects of behavior are being modified. For GSM, they compare the probability change ($\Delta_t$) for tokens on the left-hand side (LHS) versus right-hand side (RHS) of intermediate equations:

- **LHS tokens (reasoning steps):** $\Delta_t = 0.131$ on average
- **RHS tokens (factual answers):** $\Delta_t = 0.056$ on average

This difference is statistically significant (p < 0.0001), suggesting proxy-tuning contributes more to formulating reasoning steps than generating factual statements. For TruthfulQA, Table 6 shows the 10 token types whose probability increased most, including stylistic markers like "There is no scientific...", "is a common myth", and "I cannot provide". The authors conclude these findings are "consistent with the hypothesis that instruction-tuning mainly influences reasoning and style, rather than increasing the model's knowledge" (Section 6.1).

**Steering Strength Hyperparameter (Figure 2, Section 6.2)**
The optional hyperparameter $\alpha$ in the equation $s_M + \alpha(s_{M+} - s_{M-})$ allows runtime control over steering intensity. Figure 2 shows results for $\alpha \in [0.2, 2.0]$ on TruthfulQA:

- **Truthfulness:** Increases consistently with higher $\alpha$
- **Informativeness:** Peaks at $\alpha = 0.4$, then declines

This reveals a trade-off: stronger steering improves factuality but may cause the model to decline answering questions more frequently. The smooth trade-off indicates users can adjust $\alpha$ based on application needs—higher for safety-critical contexts, lower for creative tasks.

**Runtime Analysis (Table 12, Appendix C.1)**
A practical concern is computational overhead. Table 12 reports per-generation runtimes:

| Model | Slowdown Factor |
|-------|-----------------|
| **13B Proxy-tuned** | ~2.5× (sequential) |
| **70B Proxy-tuned** | ~1.5× (sequential) |

The slowdown is smaller for 70B models because the 7B proxy overhead is proportionally less significant. Crucially, the authors note this is due to sequential execution. In parallel deployment with GPU communication (all-reduce operations), their pilot implementation achieves "similar runtime compared to a true tuned model" (Section C.1). This distinction matters—the algorithmic overhead is minimal; the measured slowdown is an implementation artifact.

**Prediction Change Frequency (Figure 3, Appendix C.2)**
Figure 3 analyzes how often proxy-tuning changes the top-token prediction from the base model:

- **AlpacaFarm:** 17.3% of predictions changed
- **ToxiGen:** 24.6% of predictions changed
- **GSM:** 13.5% of predictions changed
- **TruthfulQA:** 18.0% of predictions changed

The influence is strongest on early tokens, declining as generation proceeds. This suggests potential optimization: applying proxy-tuning only to initial tokens could reduce compute costs. However, the authors caution that "the simple approach of only applying proxy-tuning to the first few tokens has limited effectiveness, due to the base model's tendency to return to endless repetition when unchecked" (Section C.2).

**Comparison with LoRA (Table 14, Table 15, Appendix D)**
The authors compare proxy-tuning to LoRA, a parameter-efficient finetuning method requiring white-box access:

| Model | TriviaQA | GSM | Training Time (relative to 7B full) |
|-------|----------|-----|-------------------------------------|
| **13B Proxy-tuned** | 55.9% | 43.9% | 1.0× (7B full finetune) |
| **13B LoRA** | 66.0% | 32.4% | 1.12× |
| **70B Proxy-tuned** | 62.7% | 53.9% | 1.0× (7B full finetune) |
| **70B LoRA** | 75.3% | 63.0% | 15.2× |

Key observations:
1. **LoRA outperforms on TriviaQA** but proxy-tuning outperforms on GSM at 13B scale (43.9% vs 32.4%).
2. **Training efficiency:** Full finetuning of a 7B expert is 15× more train-efficient than LoRA for 70B models (Table 14).
3. **Access requirements:** LoRA requires white-box access to load 70B parameters; proxy-tuning only needs output logits.

The authors hypothesize the inconsistency stems from the "size of the shift between pretraining and finetuning data" (Section D.1). TriviaQA answers are stylistically similar to base model outputs, while GSM requires learning new formatting—proxy-tuning excels at transferring formatting constraints.

---

### 5.4 Critical Assessment of Experimental Claims

**Strengths of the Experimental Design**

1. **Multi-task validation:** Testing across instruction-following, code, task-specific, and temporal adaptation demonstrates generality rather than task-specific optimization.

2. **Appropriate baselines:** The three-way comparison (base, proxy-tuned, directly tuned) allows precise measurement of gap closure. Including the small expert alone confirms the large model contributes value.

3. **Fine-grained metrics:** Separating truthfulness from informativeness (Table 3) provides evidence for the knowledge preservation claim that aggregate metrics would obscure.

4. **Black-box demonstration:** The GPT-3.5 case study proves the method works under real-world constraints where prior methods would fail.

**Limitations and Boundary Conditions**

1. **Code adaptation limitation:** Proxy-tuning does not outperform the small expert on code tasks (Table 4), suggesting the method works best when the base model's general knowledge complements rather than conflicts with the expert's specialization.

2. **Vocabulary requirement:** The method requires shared vocabulary between models (Section 2, footnote 3). While tokenizers are often open-source, this constrains which model combinations are feasible.

3. **Runtime overhead:** Even with parallelization, deploying three models instead of one increases infrastructure complexity and cost. The 1.5–2.5× sequential slowdown may be prohibitive for latency-sensitive applications.

4. **Limited GPT-3.5 evaluation:** The black-box case study uses only multiple-choice questions where 4 tokens are reweighted. Section 7 footnote 4 notes the API doesn't allow conditioning on partial responses, preventing multi-token generation tasks.

5. **No long-context evaluation:** All experiments use relatively short generations (up to 512 tokens for most tasks, 2048 for AlpacaFarm). The method's behavior on long-form content generation is untested.

**Do the Experiments Support the Claims?**

The evidence strongly supports the core claims:

| Claim | Evidence Strength | Key Supporting Data |
|-------|------------------|---------------------|
| Closes 88% of tuning gap | Strong | Table 2: 88.1% gap closed at 70B scale |
| Preserves knowledge better than direct tuning | Moderate-Strong | Table 3: 6.5% higher truthfulness at 70B |
| Works on truly black-box models | Strong | Table 7: 2.3% improvement on GPT-3.5 (p < 0.0001) |
| Combines tuning + scale benefits | Strong | Proxy-tuned outperforms small expert in 19/20 settings (Table 2) |
| Generalizes across adaptation types | Strong | Positive results across 4 task categories (Sections 3–7) |

The knowledge preservation claim is the most nuanced—while truthfulness improves, informativeness slightly decreases. This trade-off should be framed as "different knowledge retention profile" rather than universally superior.

The code adaptation results reveal an important boundary: proxy-tuning excels when behavioral alignment is needed (instruction-following, safety, formatting) but offers less benefit when domain-specific knowledge is the bottleneck. This is not a failure but a clarification of the method's appropriate use cases.

Overall, the experiments are thorough and convincingly demonstrate that proxy-tuning is a viable alternative to direct finetuning for black-box models, with the added benefit of potential knowledge preservation. The limitations identified are honest and help practitioners understand when to apply the method.

## 6. Limitations and Trade-offs

While `proxy-tuning` demonstrates compelling empirical results across multiple adaptation scenarios, the method is not universally applicable. Understanding its constraints is critical for practitioners deciding whether to adopt this approach. This section examines the technical assumptions, computational trade-offs, and unresolved questions that define the boundaries of proxy-tuning's effectiveness.

---

### 6.1 Core Technical Assumptions

**Vocabulary Alignment Requirement**
The method fundamentally requires that all three models (base, expert, anti-expert) share the same vocabulary. Section 2 footnote 3 states: "We only require that they share the same vocabulary." This is because the logit arithmetic operates token-by-token—subtracting $s_{M-}$ from $s_{M+}$ only makes sense if both vectors index the same tokens in the same order.

*   **Practical implication:** You cannot proxy-tune a LLAMA2 base model using a T5 expert, even if both are 7B parameters. The vocabulary mismatch would render the logit difference meaningless.
*   **Workaround feasibility:** The authors note that tokenizers are often open-source even for closed models (e.g., GPT-4's `tiktoken`), making vocabulary alignment feasible in many cases. However, Section 2 footnote 3 also acknowledges that "when vocabularies do not match, techniques like that of Kasai et al. (2022) could be applied"—indicating this is an unresolved engineering challenge rather than a solved problem.

**Logit Accessibility**
The standard formulation assumes access to the full logit distribution over the vocabulary for the base model $M$. Section 2 states: "For arbitrary inputs to $M$, we assume that we can access the output logits for the entire vocabulary."

*   **Reality check:** Many commercial APIs do not expose full logits. Section 7 demonstrates that proxy-tuning can function with only the top-5 log probabilities, but this severely constrains applicability. The GPT-3.5 case study was limited to multiple-choice questions where only four tokens ({A, B, C, D}) needed reweighting.
*   **Critical constraint:** Section 7 footnote 4 reveals a deeper limitation: "The API also does not allow conditioning on partial model responses; it always generates the start of a new conversational turn. This prevents us from applying proxy-tuning to any task involving generation of more than one token." This means the black-box demonstration in Section 7 cannot be extended to open-ended generation tasks—a significant restriction for practical customization of proprietary models.

**Architectural Compatibility Between Expert and Anti-Expert**
The expert $M_+$ and anti-expert $M_-$ must be closely matched architectures so that their logit difference reflects only tuning effects, not architectural biases. All experiments in Sections 3–5 use model pairs from the same family (e.g., LLAMA2-7B-CHAT vs. LLAMA2-7B, or CODELLAMA-7B vs. LLAMA2-7B).

*   **Unanswered question:** The paper does not test what happens when the expert and anti-expert differ more substantially (e.g., different initialization, different training data beyond tuning). If the logit difference captures architectural artifacts rather than pure tuning signals, the steering effect could degrade or introduce unintended behaviors.

---

### 6.2 Computational and Infrastructure Constraints

**Runtime Overhead**
Section C.1 provides the most concrete data on computational costs. When running sequentially on single-GPU setups:

| Model Scale | Slowdown Factor |
|-------------|-----------------|
| 13B proxy-tuned | ~2.5× |
| 70B proxy-tuned | ~1.5× |

The slowdown is smaller at 70B scale because the 7B proxy overhead is proportionally less significant relative to the base model's computation. However, even a 1.5× latency increase is substantial for production systems.

*   **Parallelization requirement:** The authors clarify in Section C.1 that this slowdown is "mostly due to a sequential execution of the models." In theory, deploying the three models on separate GPUs with all-reduce communication can achieve "similar runtime compared to a true tuned model." However, this requires:
    1.  Three times the GPU memory footprint (one model per device)
    2.  Low-latency GPU interconnect infrastructure
    3.  Custom deployment code to coordinate logit aggregation

*   **Infrastructure burden:** For organizations without multi-GPU deployment expertise, the sequential slowdown may be prohibitive. The paper does not provide benchmarks for the parallel implementation beyond noting it exists as a "pilot implementation."

**Training Cost Shift**
Proxy-tuning shifts the computational burden from deployment-time tuning to pre-deployment expert training. Section D.2 Table 14 compares training efficiency:

| Method | Training Time (TriviaQA) | Training Time (GSM) |
|--------|-------------------------|---------------------|
| Full 7B finetune (for proxy-tuning) | 30h 11m | 2h 35m |
| LoRA 13B | 33h 55m (1.12×) | 3h 49m (1.48×) |
| LoRA 70B | 459h 06m (15.21×) | 39h 20m (15.23×) |

*   **Trade-off:** Proxy-tuning is more train-efficient than LoRA for large models (15× faster for 70B), but this assumes you only need to train the 7B expert once. If you need many different experts for different tasks, the cumulative training cost could exceed a one-time LoRA adaptation of the large model.
*   **Unaddressed scenario:** The paper does not analyze the cost-benefit trade-off when customizing a single large model for 10+ different tasks. At what point does training multiple 7B experts exceed the cost of one LoRA adaptation?

---

### 6.3 Performance Boundary Conditions

**Code Adaptation Limitation**
Section 4.2 reveals an important boundary: proxy-tuning does not always outperform the small expert alone. On code benchmarks (Table 4):

| Model | CodexEval (pass@10) | DS-1000 (pass@10) |
|-------|---------------------|-------------------|
| 7B Directly tuned (CODELLAMA) | 68.9 | 53.6 |
| 13B Proxy-tuned | 65.7 | 42.8 |
| 70B Proxy-tuned | 70.7 | 50.6 |

The 70B proxy-tuned model (70.7%) barely exceeds the 7B expert (68.9%), and the 13B proxy-tuned model (65.7%) underperforms it. The authors hypothesize in Section 4.2 that "generic pretraining at a larger scale is not helpful when the model has already been tuned for a particular domain."

*   **Implication:** Proxy-tuning excels when the base model's general knowledge complements the expert's behavioral alignment (e.g., instruction-following, safety). It offers less benefit when the task requires domain-specific knowledge that the base model lacks. If your use case is highly specialized (medical diagnosis, legal reasoning, scientific code), a small domain-tuned expert may suffice without proxy-tuning a larger model.

**Truthfulness vs. Informativeness Trade-off**
Section 6.2 Figure 2 demonstrates that the steering strength hyperparameter $\alpha$ creates a fundamental trade-off:

*   **Truthfulness:** Increases consistently with higher $\alpha$
*   **Informativeness:** Peaks at $\alpha = 0.4$, then declines

*   **Practical consequence:** There is no single "optimal" configuration. Safety-critical applications may prioritize truthfulness (higher $\alpha$), while creative or exploratory tasks may prioritize informativeness (lower $\alpha$). Users must tune $\alpha$ for their specific use case—the default $\alpha = 1.0$ is not universally optimal.
*   **Unresolved question:** The paper does not provide guidance on how to select $\alpha$ without extensive validation data. For organizations without evaluation infrastructure, this hyperparameter introduces additional complexity.

**Token-Level Influence Decay**
Section C.2 Figure 3 shows that proxy-tuning has the largest influence on the earliest tokens in a generation, with influence declining as generation proceeds. The percentage of predictions changed varies by task:

| Task | % Predictions Changed |
|------|----------------------|
| AlpacaFarm | 17.3% |
| ToxiGen | 24.6% |
| GSM | 13.5% |
| TruthfulQA | 18.0% |

*   **Optimization opportunity:** This suggests proxy-tuning could be applied only to initial tokens to reduce compute costs. However, Section C.2 cautions: "The simple approach of only applying proxy-tuning to the first few tokens has limited effectiveness, due to the base model's tendency to return to endless repetition when unchecked."
*   **Open question:** What is the minimum number of tokens that must be proxy-tuned to maintain coherence? The paper does not provide ablation studies on partial application strategies.

---

### 6.4 Unaddressed Scenarios and Edge Cases

**Long-Form Generation**
All experiments in Sections 3–5 use relatively short generations:
*   GSM, ToxiGen, TruthfulQA: Up to 512 tokens (Appendix A.1)
*   AlpacaFarm: Up to 2048 tokens (Appendix A.1)
*   Code: Up to 512 tokens (Appendix A.2)

*   **Gap:** The method's behavior on long-form content (e.g., 10,000+ token documents, multi-turn conversations, book-length generation) is untested. Error accumulation from logit arithmetic over many time steps could degrade coherence or amplify biases in unpredictable ways.

**Multi-Turn Dialogue Consistency**
The evaluation focuses on single-turn question-answer pairs. Section 7 footnote 4 explicitly notes that the GPT-3.5 API "does not allow conditioning on partial model responses," preventing multi-turn evaluation.

*   **Critical gap:** For chatbot applications, maintaining consistency across conversation turns is essential. Proxy-tuning each turn independently could lead to contradictory statements or persona drift. The paper does not address whether the method preserves dialogue state or conversational coherence.

**Multilingual and Cross-Lingual Adaptation**
All experiments use English-language datasets and English-centric models (LLAMA2, CODELLAMA, GPT-3.5). Section 2 footnote 3 mentions vocabulary alignment as a requirement but does not test cross-lingual scenarios.

*   **Unanswered question:** Can a Spanish-tuned 7B expert proxy-tune an English 70B base model? If vocabularies overlap sufficiently, the math would work—but whether the steering signal transfers meaningfully across languages is untested.

**Adversarial Robustness**
The method modifies output logits based on external models, introducing a new attack surface. An adversary who can influence the expert or anti-expert models could potentially manipulate the base model's outputs.

*   **Security gap:** The paper does not analyze whether proxy-tuning introduces vulnerabilities to prompt injection, model poisoning, or logit manipulation attacks. For safety-critical deployments, this is a significant oversight.

---

### 6.5 Open Research Questions

**Weak-to-Strong Generalization Mechanism**
Section 7 identifies the GPT-3.5 improvement as "an instance of weak-to-strong generalization" (citing Burns et al., 2023), where weaker models provide useful steering signals for stronger models. However, the paper does not explain *why* this works.

*   **Open question:** Under what conditions do weaker experts provide positive vs. negative signals? The 2.3% improvement on GPT-3.5 is statistically significant but modest—what determines the ceiling of weak-to-strong transfer?

**Optimal Expert Selection**
The experiments use experts from the same model family as the anti-expert (LLAMA2-7B-CHAT vs. LLAMA2-7B). Section 8 mentions that contemporary work (Mitchell et al., 2024) applied similar equations but focused on theoretical disentanglement rather than empirical effectiveness.

*   **Unresolved:** Can experts from different model families (e.g., Mistral-7B expert steering LLAMA2-70B) work effectively? If vocabulary alignment is achieved, does architectural mismatch degrade performance? The paper does not test cross-family proxy-tuning.

**Composition of Multiple Experts**
Section 1 mentions the method allows users to "compose [experts] off-the-shelf with no additional training" (Section 8). However, no experiments test composing multiple experts simultaneously (e.g., one for safety, one for domain knowledge, one for style).

*   **Open question:** How do multiple expert signals combine? Do they interfere, or can they be linearly combined like the base expert/anti-expert pair? This would significantly expand the method's flexibility but remains unexplored.

**Long-Term Knowledge Drift**
Section 3.2 suggests proxy-tuning preserves knowledge better than direct tuning (Table 3 shows higher truthfulness). However, this is measured on static benchmarks.

*   **Unanswered:** Does proxy-tuning prevent knowledge degradation over time as the world evolves? The GPT-3.5 temporal adaptation (Section 7) shows short-term improvement, but whether proxy-tuning can sustain knowledge updates without periodic expert retraining is unknown.

---

### 6.6 Summary of Trade-offs

| Dimension | Proxy-Tuning Advantage | Proxy-Tuning Limitation |
|-----------|----------------------|------------------------|
| **Access** | Works with black-box models (logits only) | Requires full logits for best results; top-5 logits severely constrain tasks |
| **Compute (Training)** | 15× more efficient than LoRA for 70B models (Table 14) | Must train separate expert for each task |
| **Compute (Inference)** | Parallel deployment can match tuned model runtime (Section C.1) | Sequential deployment has 1.5–2.5× slowdown (Table 12) |
| **Knowledge** | Better truthfulness preservation (Table 3: +6.5% at 70B) | Slightly lower informativeness (Table 3: -1.0% at 70B) |
| **Performance** | Closes 88% of tuning gap (Table 2) | Does not outperform small expert on domain-specific tasks (Table 4) |
| **Flexibility** | Runtime steering control via $\alpha$ (Section 6.2) | Requires hyperparameter tuning for optimal trade-offs |
| **Vocabulary** | Open-source tokenizers often available (Section 2) | Hard constraint: models must share vocabulary |

The method is best suited for scenarios where: (1) the base model is large and black-box, (2) the adaptation goal is behavioral alignment rather than domain knowledge injection, and (3) infrastructure supports parallel model deployment. It is less suitable for highly specialized domain tasks, multi-turn dialogue systems, or latency-sensitive applications without multi-GPU infrastructure.

## 7. Implications and Future Directions

This section examines how `proxy-tuning` reshapes the landscape of language model adaptation, identifies concrete research directions it opens, and provides practical guidance for practitioners deciding when to adopt this approach. The implications extend beyond the immediate technical contribution—they challenge fundamental assumptions about where tuning effects originate and who can access model customization.

---

### 7.1 Landscape Shifts: From Weight Updates to Output Guidance

**Paradigm Change: Scale-Decoupled Adaptation**
The most significant conceptual shift this work introduces is decoupling the *scale of tuning* from the *scale of deployment*. Prior to this work, the prevailing assumption was that if you want a 70B model to follow instructions, you must tune a 70B model. Table 2 demonstrates this assumption is false: a 7B expert closes 88% of the gap to a directly tuned 70B-CHAT model.

This changes the economics of model customization:
- **Before:** Customizing a 70B model requires 70B-scale compute for tuning
- **After:** Customizing a 70B model requires only 7B-scale tuning compute, plus inference-time logit arithmetic

For organizations without access to large-scale training infrastructure, this democratizes access to large model customization. A research lab with four 80GB A100s can tune a 7B expert and deploy steering on a 70B API model—something previously impossible without weight access.

**Black-Box Models Become Customizable**
Section 7's GPT-3.5 case study demonstrates that proprietary models are no longer fixed endpoints. As long as an API exposes log probabilities (even just the top-5), users can apply behavioral steering without violating terms of service or requiring unauthorized weight access.

This has implications for:
- **Model providers:** Organizations releasing closed models may need to decide whether to expose logits as a feature
- **Users:** Customization is no longer gated by weight access, only by output visibility
- **Regulators:** Safety interventions could be applied at decoding-time without modifying deployed models

**Knowledge Preservation as a First-Class Objective**
Table 3's finding that proxy-tuned models achieve 6.5% higher truthfulness than directly tuned models at 70B scale reframes the "alignment tax" discussion. Section 9 notes that "full finetuning is an invasive approach that risks forgetting of previously learned information." This work provides empirical evidence that decoding-time guidance avoids this degradation.

For safety-critical domains (medical, legal, financial), this suggests a new design principle: **preserve knowledge in weights, steer behavior at decoding-time**. Rather than accepting knowledge loss as an inevitable cost of alignment, organizations can treat knowledge preservation as an optimization objective.

---

### 7.2 Follow-Up Research Directions

The paper explicitly raises several open questions in Sections 6 and 9. Beyond these, the method enables a broader research agenda:

**1. Cross-Family Expert Steering**
All experiments use experts from the same model family as the anti-expert (LLAMA2-7B-CHAT vs. LLAMA2-7B). Section 6.5 asks: "Can experts from different model families (e.g., Mistral-7B expert steering LLAMA2-70B) work effectively?"

*Research question:* Does architectural mismatch degrade steering quality, or is the tuning signal robust across families?
*Method:* Train experts on different base architectures (Mistral, Phi, Qwen) and measure steering effectiveness on LLAMA2 base models with vocabulary mapping.
*Significance:* If cross-family steering works, users could mix-and-match experts from any open-source model family, dramatically expanding the available expert ecosystem.

**2. Multi-Expert Composition**
Section 8 notes the method allows users to "compose [experts] off-the-shelf with no additional training." However, no experiments test composing multiple experts simultaneously.

*Research question:* Can multiple expert signals be linearly combined (e.g., one for safety, one for domain knowledge, one for style)?
*Method:* Extend Equation (1) to $s_M + \sum_i \alpha_i(s_{M+_i} - s_{M-_i})$ and measure interference vs. complementarity.
*Significance:* This would enable modular behavior specification—organizations could maintain a library of experts and compose them dynamically per request.

**3. Partial Application Strategies**
Section C.2 Figure 3 shows proxy-tuning has the largest influence on early tokens, with influence declining as generation proceeds. The authors caution that applying steering only to initial tokens causes "endless repetition when unchecked," but do not explore hybrid strategies.

*Research question:* What is the minimum subset of tokens requiring steering to maintain coherence while reducing compute?
*Method:* Ablation studies applying proxy-tuning to first-N tokens, every-K tokens, or tokens below a confidence threshold.
*Significance:* Could reduce inference costs by 30–50% while maintaining most benefits, making the method viable for latency-sensitive applications.

**4. Weak-to-Strong Generalization Mechanisms**
Section 7 identifies the GPT-3.5 improvement as "an instance of weak-to-strong generalization" but does not explain *why* weaker experts provide positive signals. The 2.3% improvement is statistically significant but modest.

*Research question:* Under what conditions do weaker experts provide positive vs. negative steering signals?
*Method:* Systematically vary expert quality (training data size, model scale, task difficulty) and measure steering signal quality.
*Significance:* Would provide theoretical grounding for when proxy-tuning is expected to work, rather than empirical discovery.

**5. Long-Form and Multi-Turn Stability**
All experiments use relatively short generations (up to 2048 tokens for AlpacaFarm). Section 6.4 notes that "error accumulation from logit arithmetic over many time steps could degrade coherence." Additionally, Section 7 footnote 4 prevents multi-turn evaluation due to API constraints.

*Research question:* Does proxy-tuning maintain coherence over 10,000+ token generations or multi-turn conversations?
*Method:* Evaluate on long-form tasks (document summarization, book-length generation) and multi-turn dialogue benchmarks.
*Significance:* Critical for chatbot and content generation applications where consistency matters.

**6. Adversarial Robustness**
Section 6.4 identifies a security gap: "An adversary who can influence the expert or anti-expert models could potentially manipulate the base model's outputs." No analysis of this attack surface exists.

*Research question:* Is proxy-tuning vulnerable to prompt injection, model poisoning, or logit manipulation attacks?
*Method:* Red-team the expert models with adversarial inputs and measure impact on base model outputs.
*Significance:* Essential for safety-critical deployments where output integrity matters.

**7. Optimal α Selection Without Validation Data**
Section 6.2 shows α creates a truthfulness-informativeness trade-off but provides no guidance on selection without evaluation infrastructure.

*Research question:* Can α be selected adaptively at runtime based on generation characteristics?
*Method:* Develop heuristics (e.g., entropy-based, confidence-based) to adjust α per-token or per-request.
*Significance:* Would reduce deployment complexity for organizations without validation datasets.

---

### 7.3 Practical Applications and Use Cases

**1. Proprietary Model Customization**
Organizations using closed APIs (GPT-4, Claude, etc.) can apply domain-specific steering without waiting for provider finetuning services.

*Example:* A healthcare company could tune a 7B model on medical dialogue and apply steering to GPT-4 for patient interactions, ensuring domain-appropriate behavior without exposing patient data to model providers.
*Requirement:* API must expose log probabilities (currently limited but growing).
*Benefit:* Customization without weight access or data sharing.

**2. Safety-Critical Knowledge Preservation**
Domains where factual accuracy is paramount (medical diagnosis, legal advice, financial guidance) benefit from the knowledge preservation advantage shown in Table 3.

*Example:* A legal assistant could use proxy-tuning to ensure instruction-following while preserving the base model's knowledge of case law and statutes, avoiding the 6.5% truthfulness degradation observed with direct tuning.
*Requirement:* Vocabulary-aligned expert model available.
*Benefit:* Alignment without knowledge loss.

**3. Rapid Behavior Prototyping**
Teams can iterate on model behavior by tuning small experts rather than large base models, reducing experimentation cycles from days to hours.

*Example:* A product team testing five different conversation styles could train five 7B experts (hours each) rather than five 70B finetunes (days each), then A/B test all styles on the same 70B base.
*Requirement:* Training infrastructure for 7B models.
*Benefit:* 15× faster iteration for 70B-scale deployment (Table 14).

**4. Temporal Adaptation for Time-Sensitive Knowledge**
Section 7 demonstrates updating model knowledge about recent events without retraining the base model.

*Example:* A news summarization system could maintain a rolling expert trained on recent articles, applied to a stable base model. When the expert becomes outdated, only the 7B expert needs retraining, not the 70B base.
*Requirement:* Access to recent training data.
*Benefit:* Continuous knowledge updates without base model retraining.

**5. Multi-Tenant Behavior Personalization**
A single deployed base model can serve multiple users with different behavioral preferences by swapping experts at inference time.

*Example:* An API provider could offer "formal," "casual," and "technical" response styles by maintaining three 7B style experts, applying the appropriate expert per request without deploying three separate 70B models.
*Requirement:* Low-latency expert switching infrastructure.
*Benefit:* One base model deployment serves multiple behavioral profiles.

**6. Compliance and Regulatory Adaptation**
Organizations in regulated industries can apply jurisdiction-specific steering without maintaining separate model deployments.

*Example:* A financial services company could apply EU-specific disclosure requirements via expert steering for European users, while applying US-specific requirements for American users, all from the same base model.
*Requirement:* Regulatory training data for expert.
*Benefit:* Compliance without model proliferation.

---

### 7.4 Integration Guidance: When to Prefer Proxy-Tuning

Based on the experimental results and limitations identified throughout this analysis, here is practical guidance for practitioners:

**Prefer Proxy-Tuning When:**

| Condition | Rationale |
|-----------|-----------|
| Base model is black-box (no weight access) | Only decoding-time method that simulates finetuning (Section 2) |
| Knowledge preservation is critical | Table 3 shows 6.5% higher truthfulness than direct tuning at 70B |
| Multiple behavioral profiles needed from same base | Swap experts at inference without redeploying base model (Section 7.3) |
| Training compute is limited | 15× more train-efficient than LoRA for 70B models (Table 14) |
| Temporal adaptation needed | Update expert without retraining base (Section 7) |
| Safety/alignment is the primary goal | Excels at behavioral steering (ToxiGen 0% toxicity, Table 2) |

**Prefer Direct Finetuning or LoRA When:**

| Condition | Rationale |
|-----------|-----------|
| Domain-specific knowledge injection needed | Table 4 shows proxy-tuning doesn't outperform small expert on code tasks |
| Latency is the primary constraint | Even parallel deployment adds infrastructure complexity (Section 6.2) |
| White-box access is available AND compute is abundant | LoRA outperforms on TriviaQA (Table 15) |
| Long-form generation required | Method untested beyond 2048 tokens (Section 6.4) |
| Multi-turn dialogue consistency required | API constraints prevent multi-turn evaluation (Section 7 footnote 4) |
| Vocabulary alignment is impossible | Hard requirement: models must share vocabulary (Section 2 footnote 3) |

**Infrastructure Requirements Checklist:**

Before adopting proxy-tuning, verify:
- [ ] Base model exposes log probabilities (full vocabulary preferred, top-5 minimum)
- [ ] Expert and anti-expert share vocabulary with base model
- [ ] GPU infrastructure supports parallel model deployment (3 models minimum)
- [ ] Latency budget accommodates 1.5–2.5× sequential slowdown or parallel coordination overhead
- [ ] Evaluation pipeline exists for selecting α hyperparameter

**Cost-Benefit Decision Framework:**

For a 70B model customization decision:

1. **Is weight access available?**
   - No → Proxy-tuning is the only viable option for finetuning-like behavior
   - Yes → Continue to step 2

2. **Is the task domain-specific knowledge or behavioral alignment?**
   - Domain knowledge → Prefer LoRA or direct finetuning (Table 4)
   - Behavioral alignment → Continue to step 3

3. **Is knowledge preservation critical?**
   - Yes → Prefer proxy-tuning (Table 3 truthfulness advantage)
   - No → Continue to step 4

4. **How many behavioral profiles are needed?**
   - One → LoRA may be simpler
   - Multiple → Proxy-tuning enables expert swapping without redeployment

This framework prioritizes the method's core strengths (black-box compatibility, knowledge preservation, multi-profile support) while acknowledging its limitations (domain knowledge injection, infrastructure complexity).

---

### 7.5 Conclusion: A New Primitive for Model Adaptation

Proxy-tuning introduces a new primitive for language model adaptation: **decoding-time steering via tuning signal transfer**. Like LoRA introduced parameter-efficient finetuning as a standard tool, proxy-tuning establishes black-box customization as a viable alternative to weight updates.

The work's significance extends beyond the immediate 88% gap-closure results. It demonstrates that:
1. Tuning effects are partially scale-invariant and transferable
2. Knowledge preservation and behavioral alignment can be optimized separately
3. Proprietary models can be customized without weight access

As the field moves toward increasingly large and increasingly closed models, decoding-time methods like proxy-tuning will become essential tools for maintaining user agency over model behavior. The authors conclude in Section 9 by encouraging "model-producing organizations to share output probabilities from their models to enable use of these methods"—a call that, if heeded, would fundamentally expand the customization ecosystem for all language model users.

Future work should address the open questions in Section 7.2, particularly cross-family steering, multi-expert composition, and adversarial robustness. If these challenges are resolved, proxy-tuning could become as standard a tool in the NLP practitioner's toolkit as prompting and finetuning are today.
