## 1. Executive Summary

This paper introduces FlexGen, a high-throughput generation engine that enables running large language models (LLMs) like OPT-175B on a single commodity GPU (e.g., 16GB NVIDIA T4) by aggregating memory from GPU, CPU, and disk (Abstract). By solving a linear programming problem to optimize tensor offloading strategies and compressing weights and KV cache to 4 bits, FlexGen achieves a generation throughput of `$1$` token/s for OPT-175B with an effective batch size of `$144$` (Abstract). This represents a `$100\times$` higher maximum throughput compared to state-of-the-art offloading systems like DeepSpeed Zero-Inference, establishing a new Pareto-optimal frontier for latency-insensitive tasks on limited hardware (Figure 1).

## 2. Context and Motivation

### The Core Problem: Memory Walls for Large Language Models

The fundamental barrier this paper addresses is the **memory capacity mismatch** between modern large language models (LLMs) and accessible hardware. Consider the OPT-175B model: it requires approximately `$325$` GB of GPU memory simply to store its weights in FP16 precision (Section 1). A single NVIDIA A100 GPU provides only `$80$` GB, meaning you would need at least **five high-end GPUs** just to load the model, before accounting for activations and the key-value (KV) cache needed during generation (Section 1).

This creates a deployment bottleneck. While research institutions and large corporations can afford multi-GPU clusters, many practical use cases — such as academic benchmarking, small-team experimentation, or edge deployment — cannot. The paper asks: *Can we run 175B-parameter models on a single commodity GPU (e.g., 16GB NVIDIA T4) without sacrificing too much throughput?*

### Why Throughput-Oriented Inference Matters

The paper distinguishes between two inference paradigms, which is critical for understanding FlexGen's design choices:

| Paradigm | Use Case | Priority | Example |
|----------|----------|----------|---------|
| **Latency-oriented** | Interactive chatbots, real-time assistants | Minimize time per token | ChatGPT responses |
| **Throughput-oriented** | Batch processing, benchmarking, data wrangling | Maximize tokens per second | HELM benchmark evaluation (Section 6.1) |

> "In addition to interactive use cases such as chatbots, LLMs are also applied to many 'back-of-house' tasks such as benchmarking, information extraction, data wrangling, and form processing... they often require running LLM inference in batches over a large number of tokens... and are less sensitive to latency." (Section 1)

This distinction is crucial because **throughput-oriented workloads allow trading latency for batch size**. If you can process 144 prompts simultaneously instead of one at a time, you can amortize expensive I/O operations across the batch, even if each individual prompt takes longer to complete. FlexGen exploits this trade-off, which prior systems did not fully optimize for (Section 1).

### Three Prior Approaches and Their Limitations

Before FlexGen, researchers pursued three main directions to reduce LLM inference resource requirements (Section 1):

**1. Model Compression (Quantization/Pruning)**
- **What it does**: Reduces the memory footprint by representing weights with fewer bits (e.g., 8-bit instead of 16-bit) or removing redundant parameters.
- **Prior work**: Works like LLM.int8() (Dettmers et al., 2022), SmoothQuant (Xiao et al., 2022), and GPTQ (Frantar et al., 2022) demonstrated compression to 8 bits or lower.
- **Where it falls short**: Even with compression, a 175B model at 4 bits still requires ~`$87.5$` GB — far exceeding a 16GB GPU's capacity. Compression alone cannot solve the problem for the largest models on commodity hardware (Section 1). Additionally, prior quantization work focused on accelerating computation via integer matrix multiplication, whereas FlexGen uses quantization primarily for **reducing I/O costs** during offloading (Section 5).

**2. Collaborative/Decentralized Inference**
- **What it does**: Distributes model layers across multiple devices or machines, with each participant holding a portion of the model.
- **Prior work**: Petals (Borzunov et al., 2022) enables collaborative inference where different GPUs across the internet host different layers.
- **Where it falls short**: This approach introduces **network communication overhead**. Every token generation step requires passing activations between participants. Section 6.3 shows that FlexGen on a single T4 GPU outperforms Petals' per-GPU throughput under all tested network conditions, because offloading to local CPU/disk is faster than network transfer for large activations. Petals also cannot use large batch sizes due to memory constraints on each participant (Section 6.3).

**3. Offloading to CPU and Disk**
- **What it does**: Stores model weights and activations in CPU RAM or disk storage, loading only what's needed for computation onto the GPU.
- **Prior work**: DeepSpeed Zero-Inference (Aminabadi et al., 2022) and Hugging Face Accelerate (HuggingFace, 2022) support offloading.
- **Where it falls short**: These systems **inherit offloading strategies from training**, not inference. They use a **row-by-row schedule** (Figure 3a in Section 4.2), which processes one batch completely before moving to the next. This causes repeated weight loading — every layer's weights must be loaded for each token generation step, creating massive I/O overhead. As a result, these systems achieve batch sizes of only 1 or 2 for OPT-175B, severely limiting throughput (Section 1, Section 6.1).

### The Two Key Challenges FlexGen Addresses

The paper identifies two specific technical challenges that prior offloading systems failed to solve optimally (Section 1):

**Challenge 1: Efficient Offloading Strategy Design**

During generative inference, three types of tensors must be managed:
- **Weights**: Model parameters (static during inference)
- **Activations**: Intermediate computation results (temporary)
- **KV Cache**: Key-value pairs from attention layers (accumulates during generation)

The offloading strategy must specify:
1. **What** tensors to offload (weights? activations? KV cache?)
2. **Where** to store them (GPU? CPU? disk?)
3. **When** to move them during the computation graph traversal

> "The batch-by-batch, token-by-token, and layer-by-layer structure of the computation forms a complex dependency graph where there are multiple ways to conduct computation. Together, these choices form a complex design space." (Section 1)

Prior systems made suboptimal choices in this space. For example, storing the KV cache on GPU (as DeepSpeed does) limits batch size because the KV cache grows with sequence length. For OPT-175B with batch size 512, the KV cache alone requires `$1.2$` TB — `$3.8\times$` the model weights (Section 3). FlexGen's insight is to **unify placement decisions** for all three tensor types and optimize them jointly via linear programming (Section 4.3).

**Challenge 2: Compression Compatible with Offloading**

Prior quantization work focused on reducing computation cost, but FlexGen targets **I/O reduction**. When offloading, the bottleneck is moving data between GPU, CPU, and disk — not the matrix multiplication itself. Compressing weights and KV cache to 4 bits reduces I/O volume by `$4\times$` compared to FP16 (Section 5).

However, combining compression with offloading introduces new considerations:
- Quantization adds compression/decompression overhead, which can negate CPU computation benefits (Section 5 notes CPU delegation is turned off when quantization is enabled).
- The KV cache, not just weights, should be compressed — a contribution FlexGen claims is novel for 175B-scale models (Section 5).

### How FlexGen Positions Itself

FlexGen positions itself as a **throughput-optimized offloading framework** that fills the gap between:
- **High-end multi-GPU systems** (expensive, not accessible to all)
- **Existing offloading systems** (suboptimal throughput due to poor scheduling)
- **Decentralized inference** (network-bound, limited batch sizes)

The key differentiators are:

1. **Zig-zag block scheduling** instead of row-by-row processing, which reuses weights across multiple batches before swapping them out (Section 4.2, Figure 3b). The paper proves this schedule is within `$2\times$` of I/O-optimal (Theorem 4.1).

2. **Linear programming-based policy search** that automatically finds optimal tensor placement given hardware specifications, rather than using fixed heuristics (Section 4.3, Equation 1).

3. **Joint compression and offloading** with 4-bit quantization for both weights and KV cache without retraining, enabling CPU-only storage for 175B models and eliminating slow disk I/O in some configurations (Section 5, Section 6.1).

4. **Explicit throughput optimization** rather than latency optimization, enabling effective batch sizes up to `$144$` for OPT-175B compared to `$1$`–`$2$` for baselines (Section 6.1, Table 2).

This positioning is validated empirically: on a single 16GB T4 GPU with 208GB CPU RAM and 1.5TB SSD, FlexGen achieves `$100\times$` higher maximum throughput than DeepSpeed Zero-Inference for OPT-175B (Abstract, Figure 1). The system also enables practical workloads like benchmarking a 30B model on 7 HELM sub-scenarios in 21 hours — a task previously infeasible on commodity hardware (Abstract, Section 6.1).

## 3. Technical Approach

### 3.1 Reader orientation (approachable technical breakdown)

FlexGen is a software system that intelligently moves different parts of a large language model between GPU memory, CPU RAM, and disk storage during inference, while automatically finding the optimal schedule for these movements. It solves the problem of running models too large for GPU memory by treating offloading as an optimization problem rather than a fixed heuristic, using a zig-zag computation pattern that reuses weights across multiple batches before swapping them out, combined with 4-bit compression to further reduce memory and I/O requirements.

### 3.2 Big-picture architecture (diagram in words)

FlexGen consists of four major components that work together during inference. First, the **Policy Search Module** takes hardware specifications (GPU memory capacity, CPU RAM size, disk bandwidth) and model parameters (number of layers, hidden dimensions) as input, then solves a linear programming problem to output an offloading policy specifying what percentage of weights, activations, and KV cache should reside on each device. Second, the **Runtime Scheduler** receives this policy and orchestrates the actual computation using a zig-zag block schedule, managing multiple CUDA streams and CPU threads to overlap I/O operations with GPU computation. Third, the **Tensor Manager** handles the physical movement of tensors between GPU, CPU, and disk, mapping disk files as virtual memory and maintaining buffers for weights, activations, and KV cache according to the policy percentages. Fourth, the **Compression Module** optionally quantizes weights and KV cache to 4-bit integers using group-wise quantization before storage, then dequantizes them back to FP16 before computation on the GPU. During execution, the Policy Search runs once at initialization, then the Runtime Scheduler coordinates the Tensor Manager and Compression Module for each inference batch, with data flowing from disk/CPU storage through decompression into GPU computation, then back to storage for the KV cache.

### 3.3 Roadmap for the deep dive

- We first explain the computational graph formulation because understanding how LLM inference decomposes into layers, tokens, and batches is prerequisite to understanding why different scheduling strategies matter.
- We then cover the offloading search space and zig-zag block schedule because this is FlexGen's core algorithmic contribution that distinguishes it from prior row-by-row systems.
- We next detail the cost model and linear programming optimization because this explains how FlexGen automatically finds good policies rather than relying on manual tuning.
- We then describe the 4-bit quantization approach because compression interacts critically with offloading decisions and enables CPU-only storage for the largest models.
- We finally cover CPU computation delegation and multi-GPU extension because these are secondary optimizations that build on the core offloading framework.

### 3.4 Detailed, sentence-based technical breakdown

This is a systems paper that presents an optimization framework for LLM inference offloading, with the core idea being that tensor placement and computation scheduling can be jointly optimized via linear programming rather than fixed heuristics.

#### The Computational Graph Formulation

FlexGen begins by formulating generative LLM inference as a graph traversal problem, which is essential for understanding why scheduling matters. Figure 2 in Section 4.1 illustrates this computational graph where each square represents the computation of one GPU batch for one transformer layer, rows correspond to different layers in the model, and columns correspond to different tokens being generated. For a model with `$l$` layers generating `$n$` tokens per prompt with batch size `$b$`, the graph contains `$l \times n$` squares that must all be traversed during inference. The graph has specific dependency constraints: a square can only be computed if all squares to its left on the same row were already computed (because token `$t$` depends on token `$t-1$`), and to compute a square on a device, all its inputs including weights, activations, and KV cache must be loaded to that same device. After computation, each square produces two outputs: activations that must be stored until the right sibling square is computed, and KV cache that must be stored until the rightmost square on the same row is computed. At any time during execution, the total size of tensors stored on any device cannot exceed that device's memory capacity. This formulation makes explicit what prior systems treated implicitly: there are multiple valid paths through this graph, and different paths have dramatically different I/O costs.

#### The Offloading Search Space

FlexGen constructs a search space over three dimensions that together define a complete offloading strategy. The first dimension is **compute schedule**, which determines the order in which squares in the computational graph are traversed. Prior systems like DeepSpeed Zero-Inference and Hugging Face Accelerate use a row-by-row schedule shown in Figure 3(a), where one batch is processed completely through all layers before moving to the next batch. This approach has a critical flaw: because every two contiguous squares in a row do not share weights, the system must repeatedly load the same weights for each batch, incurring huge I/O costs. FlexGen instead uses a column-by-column traversal converging to what the paper calls a "zig-zag block schedule" shown in Figure 3(b). In this schedule, all squares in a column share the same layer weights, so weights can stay on GPU for reuse while only activations and KV cache are loaded and unloaded. However, the system cannot traverse an entire column to the end because activations and KV cache would fill CPU and disk memory, so it stops when these tensors reach capacity and zig-zags to the next column. The paper proves in Theorem 4.1 that this zig-zag block schedule has I/O complexity within `$2\times$` of the optimal solution, with the proof appearing in Appendix A.2.

The second dimension is **tensor placement**, which specifies how to store weights, activations, and KV cache within the three-level memory hierarchy. FlexGen uses nine variables to define placement percentages: `$w_g, w_c, w_d$` for the percentage of weights on GPU, CPU, and disk respectively; `$h_g, h_c, h_d$` for activations; and `$c_g, c_c, c_d$` for KV cache. These percentages must sum to 1 for each tensor type. Within each percentage allocation, tensors can be partitioned at different granularities: model granularity (assigning entire layers to devices), layer granularity (assigning portions of tensors within a layer), or tensor granularity (assigning individual elements). FlexGen uses layer granularity for weights because it balances runtime overhead with flexibility, and tensor granularity for activations and KV cache because finer granularity enables more precise memory management.

The third dimension is **computation delegation**, which determines whether certain operations run on GPU or CPU. While CPUs are much slower than GPUs for matrix multiplication, computing attention scores on the CPU can be beneficial when the KV cache is stored on CPU. The reasoning is quantitative: if KV cache is on CPU but attention is computed on GPU, the entire KV cache of size `$b \times s \times h_1 \times 4$` bytes must be transferred to GPU, where `$b$` is batch size, `$s$` is sequence length, and `$h_1$` is hidden dimension. If attention is computed on CPU instead, only activations of size `$b \times h_1 \times 4$` bytes need to be transferred from GPU to CPU. For long sequences where `$s \geq 512$`, this reduces I/O by a factor of `$s\times$`, making CPU computation worthwhile despite slower FLOPS. However, when 4-bit quantization is enabled, CPU delegation is turned off because the compression/decompression overhead on CPU would negate the I/O benefits (Section 5).

#### The Cost Model and Linear Programming Optimization

FlexGen develops an analytical cost model to estimate execution time given algorithm parameters and hardware specifications, then uses this model to search for optimal policies. The cost model predicts two latency components: `$T_{pre}$` for the prefill stage (processing the input prompt to generate initial KV cache) and `$T_{gen}$` for the decoding stage (generating tokens step-by-step). The total latency for computing a block is estimated as:

$$T = T_{pre} \cdot l + T_{gen} \cdot (n - 1) \cdot l$$

where `$l$` is the number of layers and `$n$` is the number of tokens to generate. Assuming perfect overlapping of I/O and computation, `$T_{pre}$` is estimated as the maximum of five components:

$$T_{pre} = \max(cto_{gp}, gto_{cp}, dto_{cp}, cto_{dp}, comp_p)$$

where `$cto_{gp}$` is latency reading from CPU to GPU, `$gto_{cp}$` is writing from GPU to CPU, `$dto_{cp}$` is reading from disk to CPU, `$cto_{dp}$` is writing from CPU to disk, and `$comp_p$` is computation time during prefill. Similarly, `$T_{gen}$` uses the same structure with decoding-specific values. Each I/O term is estimated by summing the sizes of tensors being moved divided by the relevant bandwidth. For example, disk-to-CPU read latency during decoding is:

$$dto_{cg} = \frac{1}{\text{disk to CPU bandwidth}} \left( (8h_1^2 + 4h_1 \cdot h_2) \cdot w_d + 4 \cdot bls \cdot (s + \frac{n}{2}) \cdot h_1 \cdot c_d + 2 \cdot bls \cdot h_1 \cdot h_d \right)$$

where `$8h_1^2 + 4h_1 \cdot h_2$` is the size of FP16 weights for one transformer layer, `$bls$` is block size, and the subscripts `$d$` indicate the percentage stored on disk.

The policy search solves a two-level optimization problem. First, the system enumerates a few choices of `$(bls, gbs)$` tuples where `$gbs$` (GPU batch size) is typically a multiple of 4 and `$bls$` (block size) is less than 20, keeping the search space manageable. Then with fixed `$bls$` and `$gbs$`, finding the best placement `$p = (w_g, w_c, w_d, c_g, c_c, c_d, h_g, h_c, h_d)$` becomes a linear programming problem:

$$
\begin{aligned}
\min_{p} \quad & T/bls \\
\text{s.t.} \quad & \text{gpu peak memory} < \text{gpu mem capacity} \\
& \text{cpu peak memory} < \text{cpu mem capacity} \\
& \text{disk peak memory} < \text{disk mem capacity} \\
& w_g + w_c + w_d = 1 \\
& c_g + c_c + c_d = 1 \\
& h_g + h_c + h_d = 1
\end{aligned}
$$

This formulation minimizes latency per token (throughput) subject to memory capacity constraints on all three devices. The linear programming problem has only 9 variables and can be solved very quickly. In practice, the percentage variables are relaxed to continuous values between 0 and 1 even though tensors cannot be split arbitrarily, and the system manually adjusts policies if they run out of memory due to fragmentation or modeling inaccuracies (Section 4.3).

#### The 4-Bit Group-Wise Quantization

FlexGen implements fine-grained group-wise asymmetric quantization to compress both weights and KV cache to 4 bits without retraining or calibration, achieving negligible accuracy loss on OPT-175B. The quantization formula for each element `$x$` in a group is:

$$x_{quant} = \text{round}\left( \frac{x - \min}{\max - \min} \times (2^b - 1) \right)$$

where `$\min$` and `$\max$` are computed per group, and `$b = 4$` for 4-bit quantization. The system uses a group size of 64 contiguous elements along a certain dimension. For weights, grouping along the output channel dimension preserves accuracy while being runtime-efficient. For KV cache, grouping along the hidden dimension achieves similar results. The tensors are stored in quantized format and converted back to FP16 before computation, which means the quantization primarily reduces I/O costs and memory usage rather than accelerating computation through integer matrix multiplication (unlike prior work such as LLM.int8()).

This compression enables a critical configuration change: with 4-bit quantization, FlexGen can hold all OPT-175B weights in CPU memory (approximately `$87.5$` GB at 4 bits versus `$325$` GB at FP16) and eliminate disk offloading entirely. Table 2 shows this achieves 1.12 token/s throughput for OPT-175B with prompt length 512, compared to 0.69 token/s without compression. The compression overhead is significant enough on CPU that CPU computation delegation is disabled when quantization is enabled (Section 5).

#### Overlapping and Runtime Implementation

FlexGen implements overlapping of I/O and computation using multiple CUDA streams and CPU threads. Algorithm 1 in Section 4.2 shows the block schedule with overlapping, where six operations in the innermost loop can be launched in parallel: loading weights of the next layer, storing activations and cache of the previous batch, loading cache and activations of the next batch, and computing the current batch. These six logical threads have no dependencies and are synchronized at the end of each iteration. The system relies on operating systems and CUDA drivers to resolve the schedule of underlying hardware resources. This overlapping is essential for hiding I/O latency: Table 8 in Appendix A.4 shows that for OPT-175B prefill, GPU compute utilization reaches 82%, but for decoding it drops to 13% because decoding is more I/O-bound.

#### Multi-GPU Extension via Pipeline Parallelism

FlexGen extends to multiple GPUs using pipeline parallelism rather than tensor parallelism. The system equally partitions an `$l$`-layer LLM across `$m$` GPUs, reducing the problem to running an `$l/m$`-layer transformer on one GPU, then reuses the single-GPU policy search. A new for-loop is added to Algorithm 1 to combine iteration-level pipeline parallel execution with the single-device offloading runtime. Table 3 shows this achieves super-linear scaling on decoding throughput: for OPT-175B on 4 GPUs, decoding throughput increases from 0.83 token/s (1 GPU) to 3.86 token/s (4 GPUs), a `$4.65\times$` improvement rather than linear `$4\times$`. This occurs because pipeline parallelism reduces memory pressure on each GPU, enabling larger batch sizes or switching from disk offloading to CPU-only offloading. However, generation throughput (including prefill) shows sub-linear scaling due to pipeline bubbles during the prefill stage.

#### Key Configuration Numbers and Hardware Specifications

The evaluation uses specific hardware configurations that are essential for interpreting results. Table 1 specifies: NVIDIA T4 GPU with 16 GB memory, Intel Xeon CPU at 2.00GHz with 208 GB RAM, and Cloud default SSD (NVMe) with 1.5 TB capacity. The SSD read bandwidth is approximately 2 GB/s and write bandwidth is approximately 1 GB/s. For OPT-175B with input sequence length 512 and output sequence length 32, FlexGen without compression uses a GPU batch size of 32 and block size of `$32 \times 8 = 256$` (Table 2, Appendix A.4 Table 15). With 4-bit compression, the effective batch size increases to 144 (Table 2, Abstract). The policy for OPT-175B without compression places 0% weights on GPU, 50% on CPU, 50% on disk, with all activations and KV cache on CPU (Table 15). For OPT-30B, the policy places 20% weights on GPU, 80% on CPU, with all activations and KV cache on CPU, using GPU batch size 48 and block size `$48 \times 3$` (Table 4).

#### Design Choices and Trade-offs

Several design choices reflect explicit trade-offs. The zig-zag block schedule was chosen over a more I/O-optimal diagonal block schedule (described in Appendix A.2) due to practical implementation difficulty: the diagonal schedule requires efficient attention computation on non-contiguous memory, whereas the zig-zag schedule allows pre-allocating continuous KV cache buffers. The linear programming relaxation to continuous percentages was chosen despite tensors not being arbitrarily splittable because it enables fast optimization, with manual adjustment for any memory issues. CPU computation delegation was included despite slower CPU FLOPS because the I/O reduction for attention scores outweighs compute slowdown for long sequences, but this is disabled with quantization due to compression overhead. Pipeline parallelism was chosen over tensor parallelism for multi-GPU because it achieves better throughput scaling with lower communication costs, matching the throughput-oriented use case rather than latency-oriented interactive applications.

## 4. Key Insights and Innovations

FlexGen's performance gains are not merely the result of engineering optimizations but stem from fundamental rethinking of assumptions inherited from prior work. While Section 3 detailed the mechanical implementation of schedules and cost models, this section isolates the conceptual innovations that distinguish FlexGen from existing offloading and compression systems. We identify three core insights that drive its $100\times$ throughput improvement.

### 4.1 Decoupling Inference Offloading from Training Paradigms

The most significant conceptual shift in FlexGen is recognizing that **offloading strategies optimized for training are suboptimal for throughput-oriented inference**. Prior systems like DeepSpeed Zero-Inference and Hugging Face Accelerate inherit their offloading logic from training frameworks (Section 1). Training systems prioritize gradient accumulation and weight updates, typically employing a **row-by-row schedule** (Figure 3a in Section 4.2) where one batch is processed completely before the next begins.

*   **Why this matters:** In a row-by-row schedule, weights must be loaded and unloaded for every single batch because contiguous squares in the computational graph do not share weights. For a 175B model, this creates a massive I/O bottleneck that limits batch sizes to $1$ or $2$ (Table 2).
*   **FlexGen's Innovation:** FlexGen decouples the schedule from training constraints. By adopting a **zig-zag block schedule** (Figure 3b), it processes multiple batches through the same layer before moving to the next. This allows weights to remain resident in faster memory (GPU or CPU) while only activations and KV cache are swapped.
*   **Significance:** This change enables **amortization of I/O costs**. Instead of paying the weight-loading penalty for every batch, FlexGen pays it once per block. This is the primary enabler for increasing the effective batch size from $2$ (DeepSpeed) to $144$ (FlexGen with compression) for OPT-175B (Abstract, Table 2). It transforms offloading from a memory-capacity fix into a throughput optimization tool.

### 4.2 Automating Hardware Adaptation via Optimization

Prior offloading systems rely on **static heuristics** for tensor placement (e.g., "fill GPU first, then CPU, then disk"). These rules are brittle; a policy tuned for an A100 GPU may fail catastrophically on a T4, or perform poorly if SSD bandwidth varies. FlexGen introduces the insight that **tensor placement should be solved as an optimization problem** specific to the target hardware.

*   **Why this matters:** Hardware heterogeneity is extreme in commodity settings. As shown in Appendix A.4 (Table 24), changing SSD read bandwidth from $1.6$ GB/s to $0.5$ GB/s drops OPT-175B throughput from $0.69$ token/s to $0.30$ token/s. A static policy cannot adapt to this.
*   **FlexGen's Innovation:** As detailed in Section 4.3, FlexGen constructs a cost model (Equation 1) that takes hardware specifications (bandwidth, capacity) as inputs and solves a linear programming problem to find the optimal placement percentages ($w_g, w_c, w_d$, etc.).
*   **Significance:** This provides **robustness across configurations**. The system automatically adjusts whether to store weights on CPU vs. disk based on the actual bottleneck. For example, on faster SSDs, the optimizer might choose more disk offloading to free CPU RAM for larger KV caches. This automation removes the need for manual tuning, which is a major barrier for deploying LLMs on diverse commodity hardware.

### 4.3 Repurposing Quantization for I/O Elimination

Existing quantization literature (e.g., LLM.int8(), SmoothQuant) primarily targets **compute acceleration** by enabling integer matrix multiplication. FlexGen introduces the insight that for offloading systems, quantization should target **I/O volume reduction** to change the storage tier entirely.

*   **Why this matters:** Without compression, OPT-175B weights require $\approx 325$ GB in FP16 (Section 1). This exceeds typical CPU RAM (208 GB in the evaluation setup), forcing slow disk offloading. Disk I/O is orders of magnitude slower than CPU memory access.
*   **FlexGen's Innovation:** By compressing weights and KV cache to 4 bits (Section 5), FlexGen reduces the model footprint to $\approx 87.5$ GB. This fits entirely within the 208 GB CPU RAM, allowing the system to **eliminate disk offloading** for the weights.
*   **Significance:** This shifts the bottleneck from disk bandwidth to CPU memory bandwidth. Table 2 shows this increases OPT-175B throughput from $0.69$ token/s (no compression) to $1.12$ token/s (4-bit compression). Crucially, FlexGen achieves this **without retraining or calibration**, making it immediately applicable to existing models. This demonstrates that compression can be used strategically to access higher tiers of the memory hierarchy, not just to speed up arithmetic.

### 4.4 Theoretical Grounding for Heuristic Schedules

Systems papers often propose scheduling heuristics without proving their efficiency relative to an optimal solution. FlexGen provides a **theoretical bound on its scheduling efficiency**, which is rare for practical inference engines.

*   **Why this matters:** Without bounds, it is unclear if the zig-zag schedule is merely "better than row-by-row" or actually close to the best possible strategy.
*   **FlexGen's Innovation:** In Theorem 4.1 (Section 4.2), the authors prove that the I/O complexity of the zig-zag block schedule is within $2\times$ of the optimal solution. The proof (Appendix A.2) analyzes the working state of GPU batches and shows that the schedule minimizes weight loading times asymptotically.
*   **Significance:** This provides **confidence in the design space exploration**. It assures users that the performance gains are not accidental but stem from a strategy that is provably near-optimal for the given constraints. It also highlights that further gains must come from hardware improvements (bandwidth) or compression, not from radically different scheduling algorithms.

### Summary of Contributions

| Innovation | Prior Approach | FlexGen Approach | Impact |
| :--- | :--- | :--- | :--- |
| **Scheduling** | Row-by-row (Training-inherited) | Zig-zag Block (Inference-optimized) | Enables $144\times$ larger batch sizes (Table 2) |

## 5. Experimental Analysis

### 5.1 Evaluation Methodology

FlexGen's evaluation is designed to validate three core claims: (1) it achieves significantly higher throughput than existing offloading systems on commodity hardware, (2) its optimizations (scheduling, policy search, compression) each contribute meaningfully to performance, and (3) it maintains model accuracy despite aggressive compression. The experiments are carefully structured to isolate the impact of each technique while testing realistic workloads.

**Hardware Configuration.** All primary experiments run on a standardized commodity setup specified in Table 1:
- **GPU:** NVIDIA T4 with 16 GB memory
- **CPU:** Intel Xeon @ 2.00GHz with 208 GB RAM
- **Disk:** Cloud default SSD (NVMe) with 1.5 TB capacity
- **Bandwidth:** SSD read ~2 GB/s, write ~1 GB/s

This configuration is intentionally modest — a single T4 GPU is far less powerful than the A100 clusters typically used for 175B models. The 208 GB CPU RAM is critical: without compression, OPT-175B weights (325 GB in FP16) cannot fit entirely in CPU memory, forcing disk offloading. With 4-bit compression (~87.5 GB), CPU-only storage becomes feasible.

**Models and Workloads.** The evaluation uses OPT models at three scales: 6.7B, 30B, and 175B parameters (Section 6). The workload reflects throughput-oriented scenarios:
- **Input sequence length:** 512 tokens (primary), with additional experiments at 256, 1024, and variable lengths (Appendix A.4)
- **Output sequence length:** 32 tokens generated per prompt
- **Metric:** Generation throughput defined as `$\frac{b \times n}{t}$` tokens/s, where `$b$` is batch size, `$n$` is output length, and `$t$` is total time (prefill + decoding)

This metric choice is important: it measures **system-level productivity** rather than per-request latency. For batch processing tasks like benchmarking, throughput is the relevant metric.

**Baselines.** FlexGen is compared against three representative systems (Section 6):
1. **DeepSpeed Zero-Inference:** State-of-the-art offloading system that supports CPU and disk offloading. Uses row-by-row scheduling inherited from training.
2. **Hugging Face Accelerate:** Popular inference library with offloading support. Also uses row-by-row scheduling.
3. **Petals:** Decentralized collaborative inference system. Used to compare offloading vs. network-distributed approaches.

Crucially, quantization is **not enabled** for DeepSpeed and Accelerate in the 175B experiments because their quantization methods cannot preserve accuracy at that scale (Section 6, Baseline paragraph). This ensures a fair comparison: FlexGen's advantage comes from scheduling and policy optimization, not from using compression when baselines cannot.

### 5.2 Main Throughput Results

The central claim of FlexGen is that it achieves dramatically higher throughput than existing offloading systems on a single commodity GPU. Table 2 presents the generation throughput (token/s) for all systems across model sizes and sequence lengths.

**OPT-175B Results (The Primary Benchmark).** This is the most challenging case and where FlexGen's advantages are most pronounced:

| System | Seq Length 512 | Seq Length 1024 |
|--------|---------------|-----------------|
| Accelerate | 0.01 token/s | 0.01 token/s |
| DeepSpeed | 0.01 token/s | OOM |
| Petals (24 GPUs, per-GPU) | 0.08 token/s | 0.06 token/s |
| **FlexGen** | **0.69 token/s** | **0.35 token/s** |
| **FlexGen (4-bit)** | **1.12 token/s** | **0.42 token/s** |

Several observations are critical:

1. **69× to 112× Improvement:** FlexGen without compression achieves 0.69 token/s vs. 0.01 token/s for DeepSpeed/Accelerate — a 69× improvement. With 4-bit compression, this increases to 1.12 token/s, a 112× improvement. This is not incremental; it represents a qualitative shift in what is feasible on commodity hardware.

2. **Effective Batch Size Difference:** The throughput gap stems from batch size limitations. Section 6.1 states that DeepSpeed and Accelerate can only use a maximum batch size of 2 for OPT-175B due to out-of-memory issues. FlexGen uses a GPU batch size of 32 with a block size of `$32 \times 8 = 256$` (without compression) or an effective batch size of 144 (with compression). This 72× larger effective batch size directly enables the throughput improvement.

3. **Single GPU vs. 24 GPUs:** Petals requires 24 GPUs to run OPT-175B, yet its **per-GPU throughput** (0.08 token/s) is still 8.6× lower than FlexGen on a single T4 (0.69 token/s). This validates the paper's claim that local offloading can outperform decentralized inference for throughput-oriented workloads, because network communication becomes the bottleneck (Section 6.3).

4. **DeepSpeed OOM at Longer Sequences:** At sequence length 1024, DeepSpeed runs out of memory entirely, while FlexGen continues to function at 0.35 token/s. This demonstrates FlexGen's superior memory management under pressure.

**OPT-30B Results.** The 30B model fits more comfortably in CPU memory, reducing the need for disk offloading:

| System | Seq Length 512 | Seq Length 1024 |
|--------|---------------|-----------------|
| Accelerate | 0.62 token/s | 0.31 token/s |
| DeepSpeed | 0.60 token/s | 0.29 token/s |
| Petals (4 GPUs, per-GPU) | 2.84 token/s | 1.51 token/s |
| **FlexGen** | **7.32 token/s** | **3.50 token/s** |
| **FlexGen (4-bit)** | **8.70 token/s** | **3.98 token/s** |

Here, FlexGen achieves approximately 12× improvement over DeepSpeed/Accelerate. The smaller gap (compared to 175B) occurs because the 30B model places less stress on the memory hierarchy — disk offloading is not required, reducing the advantage of FlexGen's superior scheduling. Petals performs better relatively here (2.84 token/s per GPU) because the smaller model reduces communication overhead, but FlexGen still achieves 2.6× higher per-GPU throughput.

**OPT-6.7B Results.** At this scale, the model fits entirely in GPU memory:

| System | Seq Length 512 |
|--------|---------------|
| Accelerate | 25.12 token/s |
| DeepSpeed | 9.28 token/s |
| **FlexGen** | **25.26 token/s** |
| **FlexGen (4-bit)** | **29.12 token/s** |

FlexGen matches Accelerate (both use GPU-only storage) and slightly exceeds it with compression. DeepSpeed performs worse (9.28 token/s) because it has higher memory overhead and unnecessarily uses CPU offloading even when the model fits in GPU (Section 6.1). This shows FlexGen's policy search correctly identifies when offloading is not needed.

**Latency-Throughput Trade-off (Figure 1).** Figure 1 plots the Pareto frontier of latency vs. throughput for OPT-175B and OPT-30B. Key observations:

- **FlexGen dominates the frontier:** For any given latency constraint, FlexGen achieves higher throughput than baselines. Conversely, for any throughput target, FlexGen achieves lower latency.
- **100× Maximum Throughput:** The rightmost point for FlexGen (c) on OPT-175B reaches approximately 1 token/s, while DeepSpeed and Accelerate plateau at ~0.01 token/s. The figure caption explicitly states "100× higher maximum throughput for OPT-175B."
- **Configurable Trade-offs:** FlexGen can be configured for low-latency (smaller batch, more weights on GPU) or high-throughput (larger batch, aggressive offloading) modes. Baselines cannot meaningfully adjust because they hit memory limits at small batch sizes.

Appendix Table 19 provides the exact numbers for the OPT-175B Pareto frontier. For example, at approximately 4000 seconds latency, FlexGen (c) achieves 1.122 token/s with effective batch size 144, while DeepSpeed at 5024 seconds latency achieves only 0.421 token/s with batch size 64.

### 5.3 Ablation Studies: Isolating Each Optimization

Table 4 presents an ablation study that disables one optimization at a time to measure its individual contribution. This is essential for validating that each claimed innovation actually matters.

**OPT-30B Ablation (Table 4, left column):**

| Configuration | Throughput (token/s) | Policy (gbs × #gb, wg, wc) |
|--------------|---------------------|---------------------------|
| All optimizations | 7.32 | (48×3, 20, 80) |
| No policy search | 7.26 | (48×3, 0, 100) |
| No overlapping | 5.86 | — |
| No CPU compute | 4.03 | — |
| No disk | 7.32 | — (disk not used) |
| w/ DeepSpeed policy | 1.57 | (8×1, 0, 100) |

Key insights:
- **Policy search matters little for 30B:** Disabling policy search (using a suboptimal fixed policy) only reduces throughput from 7.32 to 7.26 token/s. This is because the 30B model is less constrained — many policies work reasonably well.
- **Overlapping is critical:** Disabling I/O/compute overlapping drops throughput to 5.86 token/s (20% reduction). This validates the importance of Algorithm 1's parallel execution.
- **CPU compute is significant:** Disabling CPU computation drops throughput to 4.03 token/s (45% reduction). For the 30B model, attention computation on CPU (when KV cache is on CPU) reduces I/O enough to justify slower FLOPS.
- **DeepSpeed policy is disastrous:** Using DeepSpeed's row-by-row policy in FlexGen's runtime drops throughput to 1.57 token/s — a 79% reduction. This directly proves that the scheduling innovation, not just implementation quality, drives FlexGen's advantage.

**OPT-175B Ablation (Table 4, right column):**

| Configuration | Throughput (token/s) | Policy (gbs × #gb, wg, wc) |
|--------------|---------------------|---------------------------|
| All optimizations | 0.69 | (32×8, 0, 50) |
| No policy search | 0.27 | (32×1, 0, 50) |
| No overlapping | 0.59 | — |
| No CPU compute | 0.62 | — |
| No disk | OOM | — |
| w/ DeepSpeed policy | 0.01 | (2×1, 0, 0) |

Key insights:
- **Policy search is critical for 175B:** Unlike 30B, disabling policy search drops throughput from 0.69 to 0.27 token/s (61% reduction). The reason: the 175B model is so memory-constrained that only carefully optimized policies work. The "No policy search" configuration uses only 1 GPU batch per block instead of 8, dramatically reducing amortization.
- **Disk is mandatory:** Without disk offloading, the system runs out of memory. This confirms that 325 GB of FP16 weights cannot fit in 208 GB CPU RAM plus 16 GB GPU.
- **DeepSpeed policy is catastrophic:** Using DeepSpeed's policy yields only 0.01 token/s — matching DeepSpeed's own performance. This proves that FlexGen's 69× advantage comes from the scheduling and policy innovations, not from better engineering of the same approach.

**Interpretation.** The ablation studies reveal an important pattern: **optimization importance scales with model size**. For smaller models (30B), overlapping and CPU compute matter most. For larger models (175B), policy search and disk offloading become critical. This validates FlexGen's adaptive approach — a static configuration would not work across model scales.

### 5.4 Accuracy Validation: Does Compression Hurt Quality?

A natural concern with 4-bit quantization is accuracy degradation. Table 5 evaluates accuracy on two standard benchmarks:

**Lambada (next-word prediction accuracy, higher is better):**

| Model | FP16 | 4-bit | 4-bit-S (sparse) |
|-------|------|-------|------------------|
| OPT-30B | 0.725 | 0.724 | 0.718 |
| OPT-175B | 0.758 | 0.756 | 0.756 |

**WikiText (language modeling perplexity, lower is better):**

| Model | FP16 | 4-bit | 4-bit-S (sparse) |
|-------|------|-------|------------------|
| OPT-30B | 12.72 | 12.90 | 12.90 |
| OPT-175B | 10.82 | 10.94 | 10.94 |

**Assessment:** The accuracy loss is negligible. For OPT-175B on Lambada, 4-bit quantization reduces accuracy from 0.758 to 0.756 — a 0.26% relative decrease. On WikiText, perplexity increases from 10.82 to 10.94 — a 1.1% relative increase. These differences are within the noise of typical benchmark variance.

The "4-bit-S" configuration adds sparse attention (loading only top 10% of attention values) on top of 4-bit quantization. Surprisingly, this does not further degrade accuracy for OPT-175B (0.756 for both 4-bit and 4-bit-S on Lambada). This suggests LLMs are robust to careful approximations — a finding that enables future optimizations.

**Important caveat:** The paper notes that 3-bit compression was attempted but "cannot preserve accuracy" (Section 6.2). This establishes 4-bit as a practical lower bound for OPT models without retraining. Users should not assume they can push compression further without quality loss.

### 5.5 Multi-GPU Scaling: Does Adding GPUs Help?

Table 3 evaluates FlexGen's performance when scaled to 4 GPUs using pipeline parallelism (Section 4.4). This tests whether the single-GPU optimizations translate to multi-GPU settings.

**OPT-175B on 4 GPUs:**

| Metric | FlexGen (1 GPU) | FlexGen (4 GPUs) | Speedup |
|--------|----------------|-----------------|---------|
| Generation Throughput | 0.69 token/s | 2.33 token/s | 3.38× |
| Decoding Throughput | 0.83 token/s | 3.86 token/s | 4.65× |

**Key observation: Super-linear scaling on decoding.** The decoding throughput improves 4.65× with 4 GPUs — better than the ideal 4× linear scaling. The paper explains this occurs because pipeline parallelism reduces memory pressure on each GPU, enabling:
1. Larger batch sizes per GPU
2. Switching from disk offloading to CPU-only offloading

However, **generation throughput** (which includes prefill) shows only 3.38× improvement — sub-linear scaling. The reason: "pipeline bubbles during the prefill stage" (Section 6.1). During prefill, all layers must process the full input sequence before decoding can begin, creating idle time in the pipeline. This is a known limitation of pipeline parallelism.

**Comparison to DeepSpeed (4 GPUs):**

| Model | DeepSpeed (4 GPUs) | FlexGen (4 GPUs) |
|-------|-------------------|-----------------|
| OPT-175B Generation | 0.05 token/s | 2.33 token/s |
| OPT-175B Decoding | 0.05 token/s | 3.86 token/s |

FlexGen achieves 46.6× higher generation throughput than DeepSpeed even with the same number of GPUs. This confirms that FlexGen's innovations are not specific to single-GPU settings — the scheduling and policy optimizations remain valuable in multi-GPU configurations.

### 5.6 Offloading vs. Collaborative Inference (FlexGen vs. Petals)

Section 6.3 and Figure 4 compare FlexGen (single-GPU offloading) against Petals (decentralized collaborative inference). This addresses the question: *Should I use offloading on one machine, or distribute across multiple machines?*

**Experimental Setup:**
- Petals cluster: 4 nodes with 1 T4 GPU each (same total GPU count as 4× FlexGen comparison)
- Network conditions simulated via Linux traffic control:
  - Delay: 10ms or 100ms
  - Bandwidth: 1 Gbps or 0.1 Gbps
- Model: OPT-30B, input 512 tokens, output 32 tokens
- Metric: Per-GPU throughput (Petals total throughput divided by 4)

**Results (Figure 4, right plot):**
- FlexGen (1× T4): ~7 token/s per GPU
- Petals (4× T4, 10ms delay, 1 Gbps): ~3 token/s per GPU
- Petals (4× T4, 100ms delay, 0.1 Gbps): ~0.6 token/s per GPU

**FlexGen outperforms Petals' per-GPU throughput under all tested network conditions.** The gap widens as network quality degrades. At 100ms delay and 0.1 Gbps bandwidth, Petals achieves less than 10% of FlexGen's throughput.

**Latency Comparison (Figure 4, left plot):**
Interestingly, FlexGen can achieve **lower latency** than Petals in certain conditions:
- For short output sequences (< 15 tokens) on slow networks (100ms delay), FlexGen completes faster.
- The crossover point occurs because Petals' communication overhead during prefill is proportional to input sequence length. With 512-token inputs, prefill communication dominates.

**Interpretation:** Offloading is superior for throughput-oriented batch processing on commodity hardware. Collaborative inference may be preferable for:
1. Latency-sensitive interactive applications (where FlexGen's large batch sizes hurt)
2. Scenarios where no single machine has sufficient CPU RAM or disk
3. Settings with high-bandwidth, low-latency networks (better than tested conditions)

The appendix contains several robustness experiments that test FlexGen under varying conditions.

**Different SSD Speeds (Appendix Table 24):**

**Different GPU/CPU Configurations (Appendix Table 12):**

This highlights that **CPU RAM matters as much as GPU specs** for offloading systems — a counterintuitive finding for practitioners focused on GPU selection.

### 5.8 Critical Assessment: Do the Experiments Support the Claims?

**Claim 1: "100× higher throughput than state-of-the-art offloading systems."**

**Supported.** Table 2 shows 0.69 token/s (FlexGen) vs. 0.01 token/s (DeepSpeed/Accelerate) for OPT-175B — a 69× improvement. With 4-bit compression, this reaches 1.12 token/s, a 112× improvement. The ablation study (Table 4) confirms this comes from scheduling and policy innovations, not implementation quality: using DeepSpeed's policy in FlexGen's runtime yields 0.01 token/s, matching DeepSpeed's performance.

**Claim 2: "Negligible accuracy loss with 4-bit compression."**

**Supported.** Table 5 shows < 0.3% accuracy degradation on Lambada and < 1.2% perplexity increase on WikiText for OPT-175B. These differences are within typical benchmark variance. However, the paper notes 3-bit compression fails, establishing a lower bound.

**Claim 3: "New Pareto-optimal frontier for latency-throughput trade-offs."**

**Supported.** Figure 1 shows FlexGen dominating baselines across the entire latency range. Appendix Table 19 provides exact numbers confirming FlexGen achieves higher throughput at every comparable latency point.

**Claim 4: "Outperforms decentralized collaborative inference."**

**Conditionally supported.** Figure 4 shows FlexGen outperforms Petals' per-GPU throughput under all tested network conditions. However, this comparison uses a 4-node Petals cluster vs. 1-node FlexGen. A fairer comparison might be 4-node Petals vs. 4-node FlexGen (Table 3 shows 4× FlexGen achieves 2.33 token/s, still higher than Petals' ~3 token/s total). The latency comparison is nuanced: FlexGen wins for short outputs on slow networks, but Petals may win for interactive low-latency scenarios.

**Limitations and Unanswered Questions:**

1. **Variable-length efficiency:** The 21-25% efficiency loss on variable-length batches (Table 25) is acknowledged but not solved. This matters for real-world workloads where prompts vary significantly.

2. **CPU RAM dependency:** The 3090 vs. T4 comparison (Table 12) reveals FlexGen is sensitive to CPU RAM capacity. Users with powerful GPUs but limited CPU RAM may not see expected benefits. This is not emphasized in the abstract.

3. **Prefill bottleneck:** Multi-GPU scaling shows sub-linear improvement for generation throughput due to prefill pipeline bubbles (Table 3). For workloads with long prompts and short outputs, the benefit of multi-GPU is reduced.

4. **Limited model diversity:** All experiments use OPT models. While the paper claims applicability to GPT-3, PaLM, and BLOOM (Section 6, Model paragraph), no results are shown. Different architectures may have different quantization sensitivities.

5. **No comparison to newer systems:** The paper was published in 2023. Systems like vLLM (continuous batching) or SGLang are not compared. FlexGen's advantages may be smaller against more recent inference engines.

**Overall Assessment:** The experiments are thorough and convincingly support the core claims. The ablation studies, robustness checks, and accuracy validation demonstrate scientific rigor. The limitations are honestly acknowledged. For the stated goal — high-throughput inference on commodity hardware for latency-insensitive tasks — FlexGen delivers on its promises.

## 6. Limitations and Trade-offs

FlexGen achieves remarkable throughput gains on commodity hardware, but these improvements come with explicit trade-offs and unresolved constraints. This section critically examines the assumptions underlying FlexGen's design, the scenarios where it struggles, and the open questions that remain. Understanding these limitations is essential for practitioners deciding whether FlexGen fits their use case.

### 6.1 Fundamental Assumption: Throughput Over Latency

The most significant constraint is **explicit in the paper's framing**: FlexGen targets throughput-oriented, latency-insensitive workloads (Section 1). This is not a minor design choice — it fundamentally shapes every optimization.

**What this means in practice:**

- **Large batch sizes are mandatory.** FlexGen achieves its 100× throughput improvement by using effective batch sizes up to 144 for OPT-175B (Abstract, Table 2). This directly increases per-request latency.
- **Latency-throughput trade-off is explicit.** Figure 1 and Appendix Table 19 show that to achieve maximum throughput (1.12 token/s for OPT-175B), FlexGen requires approximately 4000 seconds of latency per batch. For comparison, DeepSpeed at batch size 1 achieves ~612 seconds latency but only 0.052 token/s throughput (Table 19).
- **Interactive applications are not supported.** The paper explicitly distinguishes FlexGen from chatbot-style inference: "In addition to interactive use cases such as chatbots, LLMs are also applied to many 'back-of-house' tasks" (Section 1).

**Critical assessment:** This is an honest limitation, but practitioners should recognize that FlexGen does **not** solve the problem of running 175B models interactively on commodity hardware. It solves batch processing. If your use case requires sub-second response times, FlexGen's large batch amortization strategy works against you.

### 6.2 Variable Sequence Length Inefficiency

FlexGen assumes uniform sequence lengths within a batch. Real-world workloads rarely satisfy this assumption.

**Evidence from the paper:**

Appendix Table 25 evaluates FlexGen on HELM benchmark tasks with variable prompt and output lengths. The system handles this by padding all inputs to the maximum length in the batch — a common but inefficient approach:

| Task | Padded Throughput | Actual Throughput | Efficiency Loss |
|------|------------------|-------------------|-----------------|
| MMLU (abstract algebra) | 251.5 token/s | 188.6 token/s | 25.0% |
| xsum (summarization) | 60.5 token/s | 47.6 token/s | 21.3% |

**Why this matters:**

- **21–25% of compute is wasted** on padding tokens that produce no useful output.
- **The inefficiency scales with variance.** The paper notes: "if some sequences are very long and some sequences are short, then FlexGen will spend a lot of time on the useless computation of padding tokens" (Appendix A.4).
- **No built-in solution.** The paper acknowledges this limitation and suggests "complementary techniques from Orca (Yu et al., 2022)" for better handling (Appendix A.4). This is an admission that FlexGen does not solve variable-length batching natively.

**Critical assessment:** For benchmarking tasks with uniform lengths (like the main evaluation with 512-token inputs), this is not a problem. For production workloads with heterogeneous prompt lengths, expect significant efficiency degradation. This is a known weakness that requires external solutions.

### 6.3 CPU Memory Capacity is a Hidden Bottleneck

A counterintuitive finding from the evaluation is that **CPU RAM capacity matters more than GPU specs** for offloading-heavy workloads.

**Evidence from Appendix Table 12:**

The paper compares two hardware setups:

| Component | T4 Setup | RTX 3090 Setup |
|-----------|----------|----------------|
| GPU | 16 GB T4 | 24 GB RTX 3090 |
| CPU RAM | 208 GB | 125 GB |
| SSD | 1.5 TB | 1 TB |
| OPT-30B Throughput | 7.32 token/s | 5.73 token/s |
| OPT-175B Throughput | 0.69 token/s | 0.38 token/s |

**The surprising result:** The more powerful RTX 3090 GPU achieves **45% lower throughput** for OPT-175B than the weaker T4.

**Why this happens:**

- OPT-175B weights in FP16 require ~325 GB (Section 1).
- The T4 setup has 208 GB CPU RAM, allowing more weights to stay in CPU memory.
- The 3090 setup has only 125 GB CPU RAM, forcing more aggressive disk offloading.
- Disk I/O is the bottleneck: Appendix Table 24 shows that reducing SSD read bandwidth from 1.6 GB/s to 0.5 GB/s drops OPT-175B throughput from 0.69 to 0.30 token/s (57% reduction).

**Critical assessment:** This is a crucial insight for practitioners. When deploying FlexGen:
- **Prioritize CPU RAM over GPU upgrades** for large models requiring offloading.
- A machine with a T4 and 256 GB RAM may outperform one with an A100 and 64 GB RAM for 175B-scale inference.
- This dependency is not emphasized in the abstract or introduction, potentially misleading users focused on GPU specifications.

### 6.4 Multi-GPU Scaling Has Prefill Bottlenecks

FlexGen supports multi-GPU deployment via pipeline parallelism (Section 4.4), but the scaling benefits are asymmetric.

**Evidence from Table 3:**

| Metric | 1 GPU | 4 GPUs | Ideal 4× | Actual Speedup |
|--------|-------|--------|----------|----------------|
| OPT-175B Generation Throughput | 0.69 token/s | 2.33 token/s | 2.76 token/s | 3.38× (sub-linear) |
| OPT-175B Decoding Throughput | 0.83 token/s | 3.86 token/s | 3.32 token/s | 4.65× (super-linear) |

**Why the discrepancy:**

- **Decoding scales super-linearly** (4.65×) because pipeline parallelism reduces per-GPU memory pressure, enabling larger batch sizes or CPU-only storage.
- **Generation throughput scales sub-linearly** (3.38×) because "there are pipeline bubbles during the prefill stage" (Section 6.1).
- During prefill, all layers must process the full input sequence before decoding begins, creating idle time in the pipeline.

**Critical assessment:** For workloads with **long prompts and short outputs** (e.g., summarization with 2000-token inputs and 50-token outputs), the prefill stage dominates, and multi-GPU benefits are reduced. For workloads with **short prompts and long outputs** (e.g., creative writing), decoding dominates, and scaling is more effective. Practitioners should match their workload profile to this asymmetry.

### 6.5 Quantization Has a Practical Lower Bound

FlexGen achieves 4-bit quantization without retraining or calibration (Section 5), but there are limits to how far compression can be pushed.

**Evidence from Section 6.2:**

> "We also tried 3-bit compression but it cannot preserve accuracy."

**What this means:**

- **4-bit is a hard floor** for OPT models without fine-tuning. Users cannot assume they can reduce memory further by going to 3-bit or 2-bit.
- **KV cache compression is novel but untested at scale.** The paper claims compressing KV cache to 4-bit is a contribution (Section 5), but evaluation is limited to OPT-30B and OPT-175B on two benchmarks (Lambada, WikiText).
- **CPU compute is disabled with quantization.** Section 5 notes: "such a fine-grained group-wise quantization in FlexGen causes some overhead in compression and decompression... so we turn off the CPU delegation when enabling quantization." This removes one optimization when compression is active.

**Critical assessment:** The 4-bit result is impressive, but the 3-bit failure suggests diminishing returns. Users should not expect linear memory reductions beyond 4-bit without model-specific calibration or retraining. Additionally, the interaction between quantization and CPU delegation means some optimizations are mutually exclusive.

### 6.6 Implementation Complexity Limits Theoretical Optimality

FlexGen uses a "zig-zag block schedule" that is provably within 2× of I/O-optimal (Theorem 4.1), but the paper acknowledges a more optimal schedule exists and was not implemented.

**Evidence from Appendix A.2:**

The paper describes a "diagonal block schedule" that can achieve better I/O efficiency:

> "Despite its advantages, there are some difficulties in implementing the diagonal block schedule. The major implementation difficulty is the dynamic update of the KV cache buffer... one needs to implement efficient attention computation on non-contiguous memory."

**What this means:**

- **FlexGen is not I/O-optimal.** It is within 2× of optimal, but the gap is real.
- **Implementation complexity was prioritized over theoretical optimality.** The zig-zag schedule allows pre-allocating continuous KV cache buffers, simplifying memory management.
- **Future work could close the gap.** The diagonal schedule remains an open engineering challenge.

**Critical assessment:** This is an honest acknowledgment, but it means FlexGen's performance ceiling is not yet reached. Users should expect potential improvements from future versions, but also recognize that the current implementation trades some efficiency for practicality.

### 6.7 Hardware-Specific Cost Model Requires Calibration

FlexGen's policy search depends on a cost model that must be fitted to each hardware configuration (Section 4.3).

**Evidence from Appendix A.4:**

> "To tune the system for different architectures, we need to fit a cost model and run policy search to generate offloading policies, which can be different according to the compute capabilities, memory capacities, and memory bandwidth of different architectures."

**What this means:**

- **No universal policy exists.** A policy optimized for a T4 with 208 GB CPU RAM will not work optimally on an RTX 3090 with 125 GB CPU RAM (as shown in Table 12).
- **Profiling overhead is required.** Users must run profiling on their hardware to sample data points and fit hardware parameters before deployment.
- **Policy search can fail.** Section 4.3 notes: "sometimes a strategy from the policy search can run out of memory. In this case, we manually adjust the policy slightly."

**Critical assessment:** This reduces FlexGen's "plug-and-play" appeal. While the linear programming optimization is fast, the profiling and potential manual adjustment add deployment complexity. For organizations with heterogeneous hardware fleets, maintaining calibrated cost models for each configuration is an operational burden.

### 6.8 Limited Model Diversity in Evaluation

All experiments use OPT models (Section 6, Model paragraph). The paper claims broader applicability but provides no evidence.

**Evidence from Section 6:**

> "Although we do not evaluate other models, the offloading in FlexGen can be applied to other transformer LLMs, e.g., GPT-3, PaLM, and BLOOM because they all share a similar structure."

**What this means:**

- **No validation on GPT-style architectures.** OPT uses standard transformer blocks, but models with different attention mechanisms (e.g., grouped-query attention, multi-query attention) may have different memory profiles.
- **Quantization sensitivity may vary.** The 4-bit result is specific to OPT. Other models may require calibration or show larger accuracy degradation.
- **Embedding layers are ignored.** Footnote 1 in Section 3 states: "We ignore the embedding layer(s), which is relatively small." For models with large vocabulary sizes, embedding tables can be significant.

**Critical assessment:** This is a standard limitation in systems papers, but practitioners should validate FlexGen on their target models before deployment. The claim of broad applicability is reasonable but unproven.

### 6.9 Disk Offloading is Mandatory for 175B Without Compression

For OPT-175B without 4-bit compression, disk offloading is not optional — it is required.

**Evidence from Table 4 Ablation:**

| Configuration | OPT-175B Throughput |
|--------------|---------------------|
| All optimizations | 0.69 token/s |
| No disk | OOM |

**What this means:**

- **208 GB CPU RAM is insufficient** for 325 GB of FP16 weights.
- **Disk I/O becomes the bottleneck.** Appendix Table 24 shows disk bandwidth directly limits throughput.
- **SSD quality matters significantly.** A slow SSD can halve performance (0.69 → 0.30 token/s).

**Critical assessment:** Users without fast NVMe storage will see degraded performance. This is a hardware dependency that cannot be optimized away without compression. Organizations should prioritize SSD bandwidth when building FlexGen deployment infrastructure.

### 6.10 Summary of Trade-offs

| Trade-off | Benefit | Cost |
|-----------|---------|------|
| **Large batch sizes** | 100× throughput improvement | High per-request latency (4000+ seconds) |
| **CPU RAM dependency** | Enables offloading beyond GPU limits | CPU RAM capacity matters more than GPU specs |
| **4-bit quantization** | Eliminates disk offloading for 175B | Disables CPU compute delegation; 3-bit fails |
| **Zig-zag scheduling** | Practical implementation | Within 2× of I/O-optimal, not optimal |
| **Uniform sequence lengths** | Simplifies batching | 21–25% efficiency loss on variable lengths |
| **Pipeline parallelism** | Super-linear decoding scaling | Sub-linear generation scaling due to prefill bubbles |
| **Hardware-specific policies** | Optimal for each configuration | Requires profiling and calibration per setup |

### 6.11 Open Questions

Several questions remain unresolved:

1. **How does FlexGen compare to newer inference engines?** The paper compares against DeepSpeed and Accelerate (2022 systems). Systems like vLLM (continuous batching) or SGLang are not evaluated. FlexGen's advantages may be smaller against more recent engines.

2. **What is the true accuracy impact on downstream tasks?** Table 5 evaluates Lambada and WikiText, but these are proxy benchmarks. Real-world task performance (e.g., question answering, code generation) may show different sensitivity to 4-bit quantization.

3. **Can the diagonal block schedule be implemented efficiently?** Appendix A.2 identifies this as an open engineering challenge. If solved, it could close the 2× optimality gap.

4. **How does FlexGen interact with emerging sparse architectures?** The paper tests sparse attention as an approximation (Section 5), but does not evaluate native sparse models (e.g., Mixture of Experts).

5. **What is the energy efficiency profile?** The paper optimizes for throughput, not energy. Running a 175B model on a single T4 for 4000 seconds may consume more energy than running it on 5 A100s for 100 seconds. This trade-off is not analyzed.

### 6.12 Final Assessment

FlexGen's limitations are **well-documented and honest**. The paper does not overclaim: it explicitly targets throughput-oriented workloads, acknowledges variable-length inefficiency, and admits the zig-zag schedule is not optimal. For the stated use case — batch processing of large models on commodity hardware — FlexGen delivers on its promises.

However, practitioners should recognize:
- **FlexGen is not a universal solution.** It excels at batch benchmarking and data wrangling, not interactive applications.
- **Hardware matters more than expected.** CPU RAM and SSD bandwidth are as critical as GPU specs.
- **Deployment requires calibration.** The cost model must be fitted to each hardware configuration.

For organizations with latency-insensitive, batch-oriented workloads and limited GPU budgets, FlexGen represents a practical path to 175B-scale inference. For interactive or variable-length workloads, the trade-offs may not be acceptable.

## 7. Implications and Future Directions

FlexGen represents more than an incremental systems improvement — it fundamentally reshapes assumptions about what hardware is required for large-scale LLM inference. This section examines how the work changes the research landscape, what follow-up directions it opens, and when practitioners should adopt FlexGen versus alternative approaches.

### 7.1 Landscape Shifts: Democratizing Access to 175B-Scale Models

**Breaking the Multi-GPU Barrier.** Before FlexGen, running a 175B-parameter model required at least five A100 (80GB) GPUs with complex parallelism strategies (Section 1). FlexGen demonstrates that a single 16GB NVIDIA T4 GPU — a commodity card costing approximately $500-800 — can achieve usable throughput (1.12 token/s with 4-bit compression) for OPT-175B. This represents a **cost reduction of approximately 50×** in hardware requirements, making 175B-scale inference accessible to:

- Academic research groups without cluster access
- Small startups with limited infrastructure budgets
- Individual researchers experimenting with large models
- Educational institutions teaching LLM systems

**Reframing Offloading as Throughput Optimization.** Prior offloading systems (DeepSpeed Zero-Inference, Hugging Face Accelerate) treated offloading as a **memory capacity fix** — a fallback when models don't fit in GPU memory. FlexGen reframes offloading as a **throughput optimization tool**. By using zig-zag block scheduling (Section 4.2, Figure 3b), FlexGen amortizes I/O costs across large batches, turning what was previously a bottleneck into a manageable trade-off. This conceptual shift enables batch sizes of 144 for OPT-175B compared to 1-2 for prior systems (Table 2).

**Quantization for I/O Reduction, Not Just Compute.** Existing quantization work (LLM.int8(), SmoothQuant) primarily targets **compute acceleration** through integer matrix multiplication (Section 5). FlexGen demonstrates that quantization can instead target **I/O volume reduction** to change storage tiers entirely. By compressing OPT-175B weights from 325 GB (FP16) to ~87.5 GB (4-bit), FlexGen eliminates disk offloading and stores everything in CPU RAM. This strategic use of compression — prioritizing memory hierarchy access over arithmetic speed — opens new design spaces for systems where I/O, not compute, is the bottleneck.

**Offloading vs. Decentralized Inference.** Section 6.3 and Figure 4 show that FlexGen on a single T4 GPU outperforms Petals' per-GPU throughput under all tested network conditions. This challenges the assumption that distributed inference is necessary for large models on limited hardware. For throughput-oriented workloads, **local offloading to CPU/disk can be faster than network-distributed computation** because:
- Local SSD bandwidth (~2 GB/s) exceeds typical network bandwidth in decentralized settings
- No network latency overhead for activation transfers
- Larger batch sizes are feasible without coordination overhead

This finding suggests that for many batch processing scenarios, investing in a single machine with ample CPU RAM and fast SSD may be more cost-effective than building a multi-node cluster.

### 7.2 Follow-Up Research Directions

FlexGen's contributions open several concrete research avenues. Some are explicitly identified in the paper; others emerge from its limitations.

**Implementing the Diagonal Block Schedule.** Appendix A.2 describes a "diagonal block schedule" that is I/O-optimal asymptotically, compared to the zig-zag schedule's 2× bound (Theorem 4.1). The paper states:

> "Despite its advantages, there are some difficulties in implementing the diagonal block schedule. The major implementation difficulty is the dynamic update of the KV cache buffer... one needs to implement efficient attention computation on non-contiguous memory." (Appendix A.2)

This is a clear engineering challenge for follow-up work. Successfully implementing the diagonal schedule could close the 2× optimality gap, potentially doubling throughput for I/O-bound configurations. Key technical hurdles include:
- Dynamic KV cache buffer management without pre-allocation
- Efficient attention computation on non-contiguous memory regions
- Maintaining overlapping benefits while handling irregular memory access patterns

**Variable-Length Batching Optimization.** Appendix A.4 (Table 25) shows 21-25% efficiency loss on variable-length batches due to padding. The paper acknowledges this limitation and suggests "complementary techniques from Orca (Yu et al., 2022)" but does not implement a solution. Follow-up work could:
- Integrate continuous batching (like vLLM's approach) with FlexGen's offloading strategy
- Develop dynamic block sizing that adapts to sequence length distributions
- Create hybrid scheduling that combines zig-zag benefits with variable-length efficiency

This is particularly important for production workloads where prompt lengths vary significantly.

**Quantization Beyond 4-Bit.** Section 6.2 states: "We also tried 3-bit compression but it cannot preserve accuracy." This establishes 4-bit as a practical lower bound **without retraining or calibration**. Follow-up research could explore:
- 3-bit quantization with light fine-tuning or calibration (similar to GPTQ's approach)
- Mixed-precision strategies where sensitive layers remain at higher precision
- Per-channel or per-token quantization schemes that better preserve accuracy
- KV cache-specific quantization methods that exploit its structural properties

The paper's finding that KV cache can be compressed to 4-bit without accuracy loss (Table 5) is novel and underexplored in the quantization literature.

**Energy Efficiency Analysis.** The paper optimizes for throughput, not energy consumption. Running OPT-175B on a single T4 for 4000 seconds may consume more total energy than running it on 5 A100s for 100 seconds. This trade-off is not analyzed. Follow-up work should:
- Measure energy consumption across different hardware configurations
- Develop cost models that include energy as an optimization objective
- Identify scenarios where single-GPU offloading is more energy-efficient than multi-GPU parallelism

This is increasingly important as LLM deployment scales and environmental concerns grow.

**Extension to Emerging Architectures.** All experiments use OPT models (Section 6). Follow-up work should validate FlexGen on:
- **Mixture of Experts (MoE)** models like Mixtral, where only a subset of weights are active per token
- **Multi-Query Attention** models that have smaller KV cache footprints
- **Long-context models** (128K+ tokens) where KV cache dominates memory usage
- **Vision-language models** with different tensor structures

The offloading framework should generalize, but policy search parameters and compression strategies may need architecture-specific tuning.

**Comparison with Newer Inference Engines.** The paper compares against DeepSpeed Zero-Inference and Hugging Face Accelerate (2022 systems). Newer engines like vLLM (continuous batching), SGLang (structured generation), and TensorRT-LLM were not evaluated. Follow-up benchmarks should:
- Compare FlexGen against these newer systems on throughput-oriented workloads
- Identify which optimizations from FlexGen (zig-zag scheduling, LP-based policy search) could be integrated into newer engines
- Determine if FlexGen's advantages persist against more recent baselines

### 7.3 Practical Applications and Downstream Use Cases

FlexGen's design targets specific workload profiles. Understanding where it excels — and where it doesn't — is critical for adoption.

**Ideal Use Cases.** FlexGen is well-suited for:

| Use Case | Why FlexGen Fits | Example from Paper |
|----------|-----------------|-------------------|
| **Academic Benchmarking** | Throughput-oriented, uniform sequence lengths, latency-insensitive | HELM benchmark: 7 sub-scenarios in 21 hours (Abstract, Section 6.1) |
| **Data Wrangling** | Batch processing over document corpora, can tolerate high latency | Entity extraction, form processing (Section 1, Appendix Table 10-11) |
| **Model Evaluation Pipelines** | Need to run many prompts through large models | OPT-IML-30B evaluation on HELM tasks (Section 6.1) |
| **Offline Content Generation** | Batch generation of summaries, translations, or analyses | xsum summarization task (Appendix Table 9) |
| **Research Experimentation** | Limited GPU budget, need to test 175B-scale models | Single T4 GPU setup (Table 1) |

**Poor Fit Use Cases.** FlexGen is **not** appropriate for:

| Use Case | Why Flex Doesn't Fit | Alternative |
|----------|---------------------|-------------|
| **Interactive Chatbots** | Requires sub-second latency; FlexGen's large batches introduce seconds-to-minutes of delay | vLLM, TensorRT-LLM with small batch sizes |
| **Real-Time Assistants** | Latency-sensitive; cannot amortize I/O over large batches | Multi-GPU inference with tensor parallelism |
| **Highly Variable Prompt Lengths** | 21-25% efficiency loss from padding (Appendix Table 25) | Orca-style continuous batching |
| **GPU-Rich Environments** | If you have 5+ A100s, multi-GPU parallelism is faster | DeepSpeed-MII, FasterTransformer |
| **Limited CPU RAM** | CPU memory capacity is critical; <100 GB severely limits 175B performance | Decentralized inference (Petals) or model compression |

**Deployment Recommendations.** Based on the evaluation results:

1. **Prioritize CPU RAM over GPU specs.** Appendix Table 12 shows a T4 with 208 GB CPU RAM outperforms an RTX 3090 with 125 GB CPU RAM for OPT-175B. When building FlexGen deployment infrastructure:
   - Target 200+ GB CPU RAM for 175B-scale models
   - A weaker GPU with more CPU RAM is preferable to a stronger GPU with less

2. **Invest in fast NVMe storage.** Appendix Table 24 shows disk bandwidth directly limits throughput when disk offloading is required:
   - Local SSD (~1.6 GB/s read): 0.69 token/s for OPT-175B
   - Persistent SSD (~0.5 GB/s read): 0.30 token/s for OPT-175B (57% reduction)
   - Use 4-bit compression to eliminate disk offloading when possible

3. **Enable 4-bit compression for 175B models.** Table 2 shows compression increases OPT-175B throughput from 0.69 to 1.12 token/s (62% improvement) by eliminating disk I/O. The accuracy loss is negligible (Table 5).

4. **Profile your hardware before deployment.** Section 4.3 notes that the cost model must be fitted to each hardware configuration:
   - Run profiling to sample compute and I/O performance
   - Use the policy search to generate hardware-specific offloading policies
   - Be prepared to manually adjust policies if memory issues occur

### 7.5 Broader Impact on LLM Deployment

FlexGen's success demonstrates that **system-level innovations can match or exceed algorithmic improvements** in expanding LLM accessibility. While much attention focuses on model architecture advances (MoE, attention variants), FlexGen shows that intelligent memory management and scheduling can unlock order-of-magnitude improvements in what hardware can run which models.

This has implications for:
- **Research equity:** Smaller institutions can now experiment with 175B-scale models without cluster access
- **Environmental impact:** Running one T4 for batch jobs may be more energy-efficient than maintaining multi-GPU clusters that sit idle
- **Deployment patterns:** Edge and on-premise deployment of large models becomes more feasible when single-GPU inference is viable
- **Cost structures:** Organizations can choose between capital expenditure (buying more GPUs) and operational expenditure (longer runtime on fewer GPUs) based on their constraints

The paper's explicit focus on throughput-oriented, latency-insensitive workloads is honest and appropriate. Not every use case needs sub-second latency. For the substantial category of batch processing tasks — benchmarking, data analysis, offline content generation — FlexGen provides a practical path to large-model inference that was previously infeasible on commodity hardware.
