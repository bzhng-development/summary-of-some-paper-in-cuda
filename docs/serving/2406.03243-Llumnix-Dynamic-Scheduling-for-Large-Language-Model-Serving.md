## 1. Executive Summary

Llumnix solves the critical challenges of heterogeneity and execution unpredictability in Large Language Model (LLM) serving by introducing a runtime request rescheduling mechanism that performs live migration of requests and their GPU memory states (KV cache) across multiple model instances. By exploiting the append-only nature of the KV cache to achieve near-constant migration downtime (~20–30 ms) regardless of sequence length, the system unifies load balancing, memory de-fragmentation, and priority isolation into a single dynamic scheduling policy. Evaluations on a 16-GPU cluster using LLaMA-7B and LLaMA-30B models demonstrate that Llumnix improves P99 first-token latency by up to 15× and reduces tail decode latency by 2× compared to state-of-the-art schedulers like INFaaS, while delivering up to 36% cost savings through efficient auto-scaling.

## 2. Context and Motivation

To understand why **Llumnix** is necessary, we must first recognize a fundamental shift in how deep learning models are served. Traditional Deep Neural Network (DNN) serving systems were designed for **stateless, one-shot inference**. In those systems, a request arrives, the model processes it in a single forward pass, and the result is returned. The execution time is deterministic, and the memory footprint is known upfront. Consequently, existing schedulers (like those in Triton Server or early versions of Ray Serve) focus primarily on **dispatching**: assigning an incoming request to an instance once and letting it run to completion.

However, Large Language Models (LLMs) operate under a completely different paradigm: **autoregressive generation**. This shift introduces three specific characteristics that break the assumptions of traditional schedulers, creating a gap that Llumnix aims to fill.

### The Unique Characteristics of LLM Serving

The paper identifies three core properties of LLM workloads that make static dispatching insufficient:

1.  **Workload Heterogeneity**: Unlike traditional models where inputs are often fixed-size images or vectors, LLM prompts vary wildly. A user might ask a simple question ("What is the capital of France?") or request a summary of a 50-page document.
    *   **Input/Output Variance**: Input lengths can range from a few tokens to tens of thousands (e.g., GPT-4 Turbo supports 128k tokens). Output lengths are equally unpredictable.
    *   **Latency Sensitivity**: Applications differ in urgency. Interactive chatbots require low latency for the first token (Time-to-First-Token or **TTFT**), while offline batch jobs (like data scoring) care more about total throughput. Existing systems often treat all these requests identically, leading to poor user experiences for latency-sensitive tasks.

2.  **Execution Unpredictability**: LLM inference is iterative. The model generates one token at a time, feeding the output back as input for the next step, until an "end-of-sequence" token is produced.
    *   **Unknown Duration**: The system cannot know how many iterations (tokens) a request will need until it actually finishes. A request might stop after 10 tokens or continue for 2,000.
    *   **Dynamic Memory Growth**: As the sequence grows, the system must store the **KV cache** (Key-Value cache), which holds the intermediate attention states for all previous tokens. This memory consumption grows dynamically with every iteration. As noted in **Section 2**, for a LLaMA-2-13B model, a single request with a 4k sequence length can consume 3.2 GB of KV cache, competing directly with the model weights (26 GB) for GPU memory.

3.  **The Multi-Tenant Conflict**: Because memory demands are unknown at arrival, multiple instances in a cluster inevitably become unbalanced. Some instances may be flooded with long-running requests consuming massive memory, while others sit idle. Traditional systems cannot move a request once it has started because the cost of transferring its state (the KV cache) was historically considered too high.

### The Failure Modes of Current Approaches

The paper argues that existing LLM serving systems (such as **vLLM** [34] or **Orca** [67]) excel at optimizing a *single* instance but fail to manage the *cluster* effectively. They rely on **one-shot dispatching** policies inherited from traditional DNN serving (e.g., Round-Robin or simple load-aware balancing). This leads to three critical failure modes:

#### 1. Severe Preemptions and Tail Latency Spikes
Modern engines like vLLM use **continuous batching** and **dynamic memory allocation** (PagedAttention) to maximize throughput. However, when an instance runs out of free memory blocks due to unpredictable request growth, it must **preempt** running requests.
*   **Mechanism**: The engine stops a running request, saves its state, and swaps it out to the CPU or queue to make room for others. When the request resumes, it often incurs significant overhead from re-computation or waiting in the queue.
*   **Impact**: **Figure 3** in the paper illustrates this starkly. In an experiment with moderate memory load (62% average), **8% of requests were preempted**. More critically, preemption accounted for **70% of the P99 per-token decode latency**. One specific P99 request suffered a total preemption loss of **50 seconds** due to being swapped out twice. This creates unstable, "jittery" performance that violates Service Level Objectives (SLOs).

#### 2. Memory Fragmentation and Queuing Delays
There is an inherent conflict between **load balancing** and **memory contiguity**.
*   **The Trade-off**: To avoid preemptions, a scheduler might try to spread requests evenly across all instances (load balancing). However, this fragments the free memory. While Instance A might have 2 GB free and Instance B might have 2 GB free, a new request requiring a 3 GB contiguous block for its initial **prefill** phase cannot run on *either*.
*   **Real-World Consequence**: **Figure 5** demonstrates that even when the *total* cluster memory is sufficient to handle queued requests, those requests remain stuck in queues because no *single* instance has enough contiguous space. This "external fragmentation" causes unnecessary queuing delays, particularly for long-input requests.

#### 3. Lack of Priority Isolation
Commercial LLM services often have tiered users (e.g., free vs. "Plus" subscribers).
*   **The Gap**: Existing systems typically treat all requests in a batch equally. A low-priority batch job generating a long story can consume memory and compute bandwidth, interfering with a high-priority interactive chat request. Without the ability to dynamically isolate resources, high-priority users suffer from the same queuing delays and interference as low-priority ones.

### Positioning Relative to Existing Work

Llumnix positions itself not as a replacement for inference engines like vLLM, but as a **dynamic scheduling layer** that sits on top of them to solve the multi-instance coordination problem.

| Feature | Traditional DNN Schedulers | Single-Instance LLM Engines (e.g., vLLM) | **Llumnix** |
| :--- | :--- | :--- | : |
| **Scheduling Unit** | One-shot dispatch | One-shot dispatch | **Runtime Rescheduling (Migration)** |
| **State Handling** | Stateless | Stateful (KV Cache), but static placement | **Stateful with Live Migration** |
| **Reaction to Load** | Static (based on arrival) | Reactive only within one instance (preemption) | **Reactive across cluster (migration)** |
| **Primary Goal** | Throughput / Fairness | Single-instance Throughput | **Cluster-wide Isolation & De-fragmentation** |

The paper draws a powerful analogy to **Operating Systems**. Just as an OS performs **context switching** to move processes between CPU cores to balance load and prioritize tasks, Llumnix performs **request migration** to move LLM inference states between GPU instances.

Prior attempts to solve multi-instance scheduling, such as **INFaaS** [53], rely on smarter *dispatching* logic but still lack the ability to move a request *after* it has started execution. They cannot fix a bad dispatch decision or react to a sudden spike in memory usage mid-generation. **DeepSpeed-MII** [4] uses simple Round-Robin policies that ignore the heterogeneity of LLM workloads entirely.

Llumnix bridges this gap by introducing a **live migration mechanism** that makes moving requests cheap enough to do frequently. By exploiting the **append-only** nature of the KV cache (new tokens are added, old ones never change), Llumnix can copy the memory state in the background while the GPU continues computing, reducing the "downtime" of a migration to a constant **~20–30 ms**, regardless of whether the sequence is 100 tokens or 8,000 tokens (**Section 6.2**). This efficiency unlocks the ability to dynamically de-fragment memory, isolate high-priority users, and balance load in real-time, addressing the fundamental limitations of the "dispatch-and-forget" paradigm.

## 3. Technical Approach

This section details the architecture and mechanisms of Llumnix, a system designed to treat LLM serving instances not as static silos, but as a dynamic pool of resources capable of exchanging workloads at runtime. The core innovation is a **live migration mechanism** that moves active inference requests and their GPU memory states between instances with negligible downtime, enabling a **dynamic scheduling policy** that continuously optimizes for load balancing, memory de-fragmentation, and priority isolation.

### 3.1 Reader orientation (approachable technical breakdown)
Llumnix is a distributed scheduling layer that sits on top of existing LLM inference engines (like vLLM) to coordinate multiple GPU instances as a single, flexible cluster. It solves the problem of unpredictable memory growth and heterogeneous request priorities by allowing the system to move running requests from crowded GPUs to空闲 ones in real-time, much like an operating system moves processes between CPU cores.

### 3.2 Big-picture architecture (diagram in words)
The system is composed of three primary logical layers that interact to manage the cluster:
*   **Global Scheduler**: A central coordinator that monitors the aggregate "load" of every instance in the cluster but does not track individual requests; its sole responsibility is to make high-level decisions to dispatch new requests, trigger migrations between specific pairs of instances, or scale the cluster size up and down.
*   **Llumlets (Local Schedulers)**: Distributed agents running on each GPU instance that manage the local queue, calculate the instance's specific load metrics, decide *which* specific requests to migrate when instructed by the Global Scheduler, and coordinate the data transfer.
*   **Inference Engine (Backend)**: The underlying execution engine (e.g., vLLM) responsible for the actual GPU computation, continuous batching, and memory management (PagedAttention), which receives commands from the Llumlet to pause, resume, or transfer KV cache blocks.

### 3.3 Roadmap for the deep dive
To fully understand how Llumnix achieves its goals, we will proceed in the following logical order:
1.  **Live Migration Mechanism**: We first explain the fundamental engine that makes rescheduling possible—how the system copies massive GPU memory states while computation continues, reducing downtime to a constant ~20–30 ms.
2.  **Distributed Scheduling Architecture**: We describe how the system scales this migration capability across a cluster using a two-level hierarchy (Global Scheduler + Llumlets) to avoid bottlenecks.
3.  **Virtual Usage Abstraction**: We detail the novel metric that unifies conflicting goals (like de-fragmentation vs. load balancing) into a single number, allowing the scheduler to make simple decisions that achieve complex outcomes.
4.  **Dynamic Scheduling Policies**: We explain the specific rules for dispatching, migration triggering, and auto-scaling that utilize the virtual usage metric to handle heterogeneity and priorities.

### 3.4 Detailed, sentence-based technical breakdown

#### The Live Migration Mechanism
The cornerstone of Llumnix is its ability to migrate a running request—complete with its **KV cache** (the stored attention states for all generated tokens so far)—from a source GPU to a destination GPU without stalling the user-facing generation process.
*   **Exploiting Append-Only Semantics**: Llumnix leverages a critical property of autoregressive LLM inference: the KV cache is **append-only**. Once a token is generated and its key/value tensors are computed and stored in memory, those specific values never change; subsequent iterations only *add* new blocks for new tokens.
*   **Pipelined Copying Strategy**: Instead of stopping the request to copy the entire memory state (which would cause downtime proportional to sequence length), Llumnix pipelines the memory transfer with the ongoing computation.
    *   **Stage 0**: When migration is initiated, the source instance begins copying the existing KV cache blocks (from completed iterations) to the destination over the network while simultaneously continuing to compute the next token on the GPU.
    *   **Iterative Catch-up**: Because network copying is significantly faster than GPU inference computation, the copy process quickly catches up to the current generation head. Once the initial bulk copy is finished, the system enters a second stage to copy only the small number of new blocks generated during the first stage.
    *   **Final Handoff**: This process repeats for a minimal number of stages (typically just two). In the final stage, the source instance suspends the request for a brief moment (draining it from the current batch), copies the single remaining block generated in the previous interval, and signals completion.
*   **Constant Downtime**: As a result of this pipelining, the "downtime" (the period where the request is not generating tokens) is limited to the time required to transfer the KV cache of a *single* iteration. **Figure 10** confirms that this downtime remains constant at approximately **20–30 ms** regardless of whether the sequence length is 256 tokens or 8,000 tokens. In contrast, naive approaches like "blocking copy" or "recomputation" see downtime increase linearly with sequence length, reaching up to **3.5 seconds** for an 8k sequence on LLaMA-30B.
*   **Handshake Protocol for Correctness**: To ensure correctness amidst asynchronous computation and potential failures (e.g., a request finishing mid-migration or running out of memory on the destination), Llumnix implements a strict handshake protocol between source and destination instances (illustrated in **Figure 7**).
    *   **Pre-Allocation**: Before each copying stage, the source sends a `PRE-ALLOC` request specifying the number of blocks needed; the destination attempts to reserve this space and replies with `ACK` or `ABORT`.
    *   **Completion Check**: After each stage, the source checks if the request has naturally completed (generated an EOS token) or been preempted locally; if so, it notifies the destination to abort and release resources.
    *   **Commit**: Only after the final stage succeeds does the source release its local blocks and send a `COMMIT` signal, allowing the destination to resume execution of the request.

#### Distributed Scheduling Architecture
To support continuous rescheduling across a large cluster without creating a central bottleneck, Llumnix adopts a hierarchical architecture that separates global coordination from local execution.
*   **Separation of Concerns**: The system divides responsibilities between a **Global Scheduler** and distributed **Llumlets** (local agents), connected by a narrow interface that exchanges only aggregate metrics rather than per-request details.
*   **Global Scheduler Responsibilities**: The Global Scheduler maintains a view of the cluster's health but does not track individual requests. Its inputs are periodic load reports from each instance, and its outputs are high-level directives:
    *   **Dispatching**: Assigning new incoming requests to the "freest" instance.
    *   **Migration Triggers**: Identifying pairs of instances (Source, Destination) based on load imbalance and marking them with a migration state. It does *not* decide which specific requests to move.
    *   **Auto-scaling**: Deciding when to add or remove instances based on cluster-wide load trends.
*   **Llumlet Responsibilities**: Each GPU instance runs a Llumlet actor that handles the fine-grained logic:
    *   **Load Calculation**: Computing the instance's current "load" metric (based on the Virtual Usage concept described below) and reporting it to the Global Scheduler.
    *   **Request Selection**: When the Global Scheduler triggers a migration from this instance, the Llumlet locally decides *which* specific requests to move (preferring lower priority and shorter sequences to minimize transfer time).
    *   **Migration Execution**: Coordinating the actual KV cache transfer with the destination Llumlet and instructing the backend inference engine to pause/resume execution.
*   **Scalability Benefits**: This design ensures that the Global Scheduler's complexity scales with the number of *instances* (typically tens or hundreds), not the number of *requests* (which can be thousands per second). **Figure 16** demonstrates that while a centralized scheduler suffers scheduling stalls of up to **40 ms** under high load on 64 instances, Llumnix maintains near-zero stalls because the heavy lifting of request tracking and migration coordination is offloaded to the parallel Llumlets.

#### The Virtual Usage Abstraction
A key challenge in LLM serving is reconciling conflicting goals: **Load Balancing** (spreading requests to avoid preemption) often worsens **Memory Fragmentation** (scattering free space so no single instance can accept a large new request). Llumnix solves this by introducing a unified metric called **Virtual Usage**, which allows the scheduler to treat different scenarios as a simple load-balancing problem.
*   **Concept Definition**: Instead of measuring an instance's load solely by its physical memory consumption, Llumnix calculates a **Virtual Usage** ($V$) for each request that may differ from its actual physical size. The total load of an instance is the sum of these virtual usages.
*   **Unifying Goals via Virtual Load**: By artificially inflating or deflating the virtual usage of specific requests, Llumnix can trick a standard load-balancing policy into performing complex behaviors:
    *   **De-fragmentation**: If an instance has fragmented free space preventing a large queued request from starting, Llumnix assigns a high virtual usage to that queued request. This makes the instance appear "overloaded" to the Global Scheduler, triggering migration of *other* running requests away from this instance to create contiguous physical space.
    *   **Priority Isolation**: For high-priority requests, Llumnix adds a "headroom" value to their physical usage to calculate their virtual usage. This reserves empty space on the instance (by making it appear fuller than it is), preventing new low-priority requests from being scheduled there and thus shielding the high-priority request from interference.
    *   **Auto-scaling Drain**: To terminate an instance, Llumnix injects a "fake" request with infinite virtual usage ($\infty$). This immediately marks the instance as maximally overloaded, forcing the load balancer to migrate all real requests off it.
*   **Mathematical Formulation**: The calculation of virtual usage is formalized in **Algorithm 1**. For a request $req$ on an instance:
    *   If $req$ is queuing (specifically head-of-line), its virtual usage is set to its memory demand: $V_{req} = \text{demand}$.
    *   If $req$ is a fake request (for draining), $V_{req} = \infty$.
    *   Otherwise, $V_{req} = \text{physicalUsage} + \text{headroom}(priority)$.
    *   The headroom for a priority class $p$ is distributed among requests of that class: $\text{headroom}(p) = \frac{\text{headroomForPriority}[p]}{\text{numRequests}[p]}$.

#### Dynamic Scheduling Policies
Using the Virtual Usage metric, Llumnix implements three core policies that drive the system's dynamic behavior.
*   **Freeness Metric and Dispatching**: The system defines a **Freeness** metric ($F$) for each instance to guide dispatching and migration. This metric estimates how many more decoding iterations an instance can sustain before running out of memory:
    $$ F = \frac{M - \sum V_{req}}{B} $$
    Where $M$ is the total GPU memory, $\sum V_{req}$ is the sum of virtual usages of all requests (running and queued), and $B$ is the current batch size (representing memory consumption speed).
    *   **Dispatch Logic**: New requests are dispatched to the instance with the highest $F$. Because queued requests contribute their full demand to $\sum V_{req}$, an instance with a large queued request will have a low (or negative) $F$, naturally steering new arrivals to other instances and facilitating de-fragmentation.
*   **Migration Triggering**: The Global Scheduler periodically scans the Freeness values of all instances.
    *   It identifies **Source Candidates** (instances with $F &lt; \text{threshold}_{low}$) and **Destination Candidates** (instances with $F > \text{threshold}_{high}$).
    *   It pairs the source with the lowest $F$ to the destination with the highest $F$ and triggers a migration state.
    *   The migration continues iteratively until the source's Freeness rises above the threshold, ensuring that load is balanced dynamically in response to real-time growth, not just initial arrival.
*   **Auto-Scaling**: Llumnix manages the cluster size to maintain cost efficiency while meeting latency SLOs.
    *   **Metric**: It monitors the average Freeness of the cluster for normal-priority requests.
    *   **Thresholds**: The system maintains the average Freeness within a target range $[x, y]$. In the evaluation, these thresholds are set to **$[10, 60]$**, representing the number of future decoding steps the cluster can support.
    *   **Action**: If the average Freeness drops below $x$ (indicating high load and risk of preemption), a new instance is added. If it rises above $y$ (indicating under-utilization), an instance is selected for termination (using the "fake request" mechanism to drain it safely).
    *   **Efficiency**: Because Llumnix can rapidly saturate new instances (by migrating existing work to them) and drain terminating instances, it reacts faster to load changes than systems that rely solely on new request dispatching, leading to the observed **36% cost savings** (**Figure 15**).

#### Implementation Details
Llumnix is implemented as a standalone Python library comprising approximately **3,300 lines of code**, designed to be non-intrusive to the underlying inference engine.
*   **Backend Integration**: It currently supports **vLLM** as the backend, leveraging its PagedAttention and continuous batching capabilities. Llumnix instantiates multiple vLLM instances as **Ray actors**, utilizing Ray's distributed runtime for efficient inter-process communication and fault tolerance.
*   **Data Transfer Optimization**: For the KV cache transfer, Llumnix uses the **Gloo** collective communication library rather than NCCL. While NCCL is faster for GPU-to-GPU transfer, it does not safely support concurrent invocations required for pipelined migration alongside inference. Gloo allows safe parallel usage, and Llumnix further optimizes it by performing CPU-GPU memory copies on a separate CUDA stream to avoid blocking the inference computation.
*   **Block Fusion**: To mitigate the overhead of sending thousands of small memory blocks (e.g., a 1k sequence in LLaMA-7B results in **4,000 blocks** of 128 KB each), Llumnix fuses these blocks into a single contiguous buffer in CPU memory before transmission, significantly improving network throughput.
*   **Fault Tolerance**: The system includes robust failure handling. If the Global Scheduler fails, the system degrades gracefully to a "scheduler-bypassing" mode where frontends dispatch directly to instances (disabling migration but preserving availability). If an instance fails, its requests are aborted, and Ray automatically restarts the failed actor.

## 4. Key Insights and Innovations

Llumnix represents a paradigm shift in LLM serving architecture, moving from static "dispatch-and-forget" models to a dynamic, OS-like environment capable of runtime process migration. The following insights distinguish fundamental innovations from incremental optimizations, highlighting why Llumnix achieves order-of-magnitude improvements in tail latency and cost efficiency.

### 4.1 The "Append-Only" Property as a Migration Enabler
**Innovation Type:** Fundamental Algorithmic Insight
**Differentiation:** Prior work in distributed deep learning (e.g., Gandiva [64]) relied on checkpoint-resume mechanisms at iteration boundaries, which incur downtime proportional to the model state size. Traditional VM live migration [17] requires complex "dirty page tracing" to track memory changes during copying. Llumnix identifies a unique, domain-specific property of autoregressive LLM inference: **the KV cache is strictly append-only**. Once a token's key/value tensors are computed, they are immutable; only new blocks are added for subsequent tokens.

**Significance:**
This insight transforms the migration problem from a bandwidth-bound bottleneck into a latency-hidden operation. By exploiting immutability, Llumnix eliminates the need for dirty page tracking and enables a pipelined copy strategy where the bulk of the data transfer occurs *concurrently* with ongoing GPU computation.
*   **Performance Impact:** This reduces migration downtime from a linear function of sequence length (seconds for long contexts) to a **constant ~20–30 ms** regardless of whether the sequence is 100 or 8,000 tokens (**Figure 10**).
*   **New Capability:** It makes runtime rescheduling viable for the first time. Without this constant-time guarantee, the cost of moving a request would exceed the benefit of load balancing, forcing systems to stick with static placement and suffer the severe preemptions described in **Section 3**.

### 4.2 Virtual Usage: Unifying Conflicting Scheduling Goals
**Innovation Type:** Conceptual Abstraction
**Differentiation:** Traditional schedulers treat conflicting objectives like **load balancing** (spreading work to reduce interference) and **de-fragmentation** (consolidating work to create large contiguous blocks) as separate, often competing, optimization problems. Solving them simultaneously typically requires complex, multi-objective solvers or heuristic switching logic. Llumnix introduces **Virtual Usage**, a unified abstraction that encodes these diverse goals into a single scalar metric per request.

**Significance:**
By artificially inflating the "virtual" load of specific requests (e.g., queuing requests needing contiguous space, or high-priority requests needing headroom), Llumnix tricks a simple **load-balancing policy** into automatically executing complex behaviors like de-fragmentation and priority isolation.
*   **Simplification:** This reduces the global scheduler's logic to a single rule: "move work from high virtual load to low virtual load." The complexity is pushed to the *definition* of the load, not the *decision engine*.
*   **Effectiveness:** As shown in **Figure 12**, this approach reduces memory fragmentation from ~8% (in INFaaS++) to near **0%**, directly translating to the **15× improvement in P99 prefill latency** by eliminating queuing delays for large inputs without needing a specialized de-fragmentation algorithm.

### 4.3 Decoupled Hierarchy: Scaling Coordination via Instance-Level Metrics
**Innovation Type:** Architectural Design
**Differentiation:** A naive approach to dynamic rescheduling would involve a centralized scheduler tracking the status of every individual request across the cluster to make optimal migration decisions. However, as demonstrated in **Figure 16**, this creates a communication bottleneck where scheduling stalls reach **40 ms** on just 64 instances, negating the benefits of migration. Llumnix rejects this centralized control in favor of a **decoupled hierarchy**: a Global Scheduler that tracks only *instance-level* metrics (Virtual Usage/Freeness) and distributed *Llumlets* that handle *request-level* logic.

**Significance:**
This design ensures that the scheduling overhead scales with the number of *machines* (tens/hundreds) rather than the number of *requests* (thousands/second).
*   **Scalability:** The Global Scheduler makes coarse-grained "pairing" decisions (Source Instance $\to$ Destination Instance) based on aggregate load, while Llumlets locally decide *which* specific requests to migrate and execute the handshake protocol.
*   **Result:** This architecture maintains **near-zero scheduling stalls** even under high load, enabling the continuous, fine-grained rescheduling necessary to handle the unpredictable nature of LLM workloads described in **Section 2**.

### 4.4 Dynamic Isolation vs. Static Reservation
**Innovation Type:** Resource Management Strategy
**Differentiation:** Existing systems offering priority support typically rely on **static resource reservation** (e.g., dedicating specific GPUs to "Plus" users) or rigid admission control. These approaches are inefficient: dedicated GPUs sit idle when high-priority traffic is low, and admission control rejects work that could otherwise be processed. Llumnix introduces **dynamic isolation** via runtime migration.

**Significance:**
Instead of reserving hardware, Llumnix reserves *capacity* by dynamically evicting lower-priority requests to create "headroom" for high-priority ones only when needed.
*   **Efficiency:** This allows the system to maintain high overall utilization while still meeting strict SLOs for premium users.
*   **Performance Gain:** Evaluations show Llumnix accelerates high-priority requests by **1.5×** while increasing latency for normal requests by less than **5%** (**Figure 13**). This demonstrates that dynamic migration provides superior isolation compared to static partitioning, effectively reconciling the trade-off between multi-tenant efficiency and service differentiation.

### Summary of Contributions

| Contribution | Prior State-of-the-Art | Llumnix Innovation | Outcome |
| :--- | :--- | :--- | :--- |
| **Migration Mechanism** | Checkpoint/Resume (High Downtime) | **Append-Only Pipelining** (Constant Downtime) | Enables frequent rescheduling; ~20ms downtime. |
| **Scheduling Logic** | Multi-objective Heuristics | **Virtual Usage Abstraction** | Unifies de-frag, balancing, & priority into one metric. |
| **Architecture** | Centralized Request Tracking | **Instance-Level Decoupling** | Linear scalability; zero scheduling stalls at scale. |

These innovations collectively shift LLM serving from a static, fragile infrastructure prone to tail-latency spikes into a resilient, self-healing system that adapts to workload dynamics in real-time.

## 5. Experimental Analysis

This section dissects the empirical evaluation of Llumnix, moving beyond the headline numbers to understand *how* the system was tested, *why* specific baselines were chosen, and what the data reveals about the trade-offs in dynamic LLM serving. The experiments are designed to validate three core hypotheses: that live migration is efficient enough to be practical, that dynamic rescheduling solves the fragmentation and preemption problems of static dispatching, and that priority isolation can be achieved without starving lower-priority workloads.

### 5.1 Evaluation Methodology

To rigorously test Llumnix, the authors constructed a testbed and workload suite that mirrors the heterogeneity and unpredictability described in the motivation.

#### Testbed and Models
The experiments were conducted on a **16-GPU cluster** hosted on Alibaba Cloud, comprising 4 virtual machines. Each VM contained **4 NVIDIA A10 GPUs (24 GB memory)** connected via PCIe 4.0, with **64 Gb/s network bandwidth** between nodes. This setup is critical because it represents a realistic multi-node environment where network transfer costs for KV cache migration are non-negligible.

Two model configurations were used to stress different aspects of the system:
*   **LLaMA-7B**: Deployed on a single GPU per instance. This configuration tests the system's ability to handle high instance counts and fine-grained scheduling.
*   **LLaMA-30B**: Deployed across 4 GPUs using tensor parallelism. This tests migration overhead in a setting where the state size is larger and communication patterns are more complex.

Both models used 16-bit precision. While the base vLLM engine supported a maximum sequence length of 2k tokens in this setup, the authors argue (Section 6.1) that the system behaviors regarding fragmentation and preemption are representative of models supporting longer contexts (4k–128k), as the underlying memory dynamics remain identical.

#### Workloads and Traces
The evaluation employs both real-world traces and synthetic distributions to cover diverse usage patterns:
*   **Real Traces**: The **ShareGPT** and **BurstGPT** datasets provide realistic conversation logs. As shown in **Table 1**, these exhibit high variance: ShareGPT inputs average 306 tokens but have a P99 of 3,388, while outputs average 500 with a P99 of 1,234. This long-tail distribution is the primary driver of the memory fragmentation and preemption issues Llumnix aims to solve.
*   **Synthetic Traces**: To isolate specific variables, the authors generated traces using **Poisson** (for arrival rates) and **Gamma** (for burstiness) distributions. Input and output lengths followed power-law distributions with means of 128, 256, or 512 tokens (labeled Short, Medium, Long). Combinations like **S-L** (Short input, Long output) and **L-S** (Long input, Short output) were constructed to specifically stress the **prefill** phase (memory allocation heavy) versus the **decode** phase (compute heavy).

#### Baselines and Metrics
Llumnix was compared against three distinct baselines to isolate the value of its components:
1.  **Round-Robin**: A simple dispatching policy common in production systems. It serves as a lower bound, highlighting the dangers of ignoring load entirely.
2.  **INFaaS++**: An optimized version of **INFaaS** [53], a state-of-the-art scheduler. The authors enhanced the original INFaaS to focus on GPU memory load (the dominant resource) and included queuing request demands in its load calculation. This represents the best possible "dispatch-only" strategy.
3.  **Llumnix-base**: A variant of Llumnix with migration enabled but **priority-agnostic** (all requests treated equally). This acts as an ablation study to isolate the benefits of the priority mechanism from the benefits of migration itself.

The primary metrics focused on latency percentiles, which are crucial for user experience:
*   **End-to-End Latency**: Total time from request arrival to completion.
*   **Prefill Latency**: Time to generate the first token (dominated by queuing delay).
*   **Decode Latency**: Average time per token after the first one (dominated by compute interference and preemption).
*   **Preemption Loss**: The extra time incurred due to a request being swapped out and resumed.
*   **Cost**: Measured by the average number of active instances required to meet a latency SLO.

### 5.2 Migration Efficiency: Is the Overhead Negligible?

Before assessing cluster-wide benefits, the authors first validated the core mechanism: **Is live migration actually fast enough to justify the complexity?**

**Figure 10** presents a micro-benchmark comparing Llumnix's pipelined migration against two naive strategies: **Blocking Copy** (stop computation, copy all data, resume) and **Recompute** (abort and restart the request on the new instance).

*   **Downtime Scaling**: The results are striking. For naive approaches, downtime scales linearly with sequence length. For an 8k sequence on LLaMA-30B, recomputation takes **3.5 seconds**, effectively stalling the user for the duration of ~54 decode steps. In contrast, Llumnix maintains a constant downtime of **~20–30 ms** regardless of sequence length. This is even shorter than a single decode step.
*   **Mechanism Validation**: The graph shows that Llumnix consistently completes migration in just **two stages**. This confirms the theoretical model: because network bandwidth (Gloo) is sufficient to copy the accumulated KV cache faster than the GPU generates new tokens, the "catch-up" phase is minimal.
*   **Interference**: The right panel of Figure 10 shows that migration imposes negligible overhead on *other* running requests on the same instance. The per-step decode time increases by less than **1%** during migration. Given that migration events occupy only ~10% of the total runtime in busy clusters, the effective overhead is vanishingly small.

**Conclusion**: The experiment convincingly proves that the "append-only" optimization transforms migration from a prohibitive cost into a lightweight operation, validating the feasibility of the entire system design.

### 5.3 Serving Performance: Solving Fragmentation and Preemption

The core claim of Llumnix is that dynamic rescheduling outperforms even the smartest static dispatchers. **Figure 11** provides a comprehensive comparison across multiple workloads.

#### Real-World Traces (ShareGPT & BurstGPT)
On realistic workloads, Llumnix demonstrates substantial improvements over both Round-Robin and INFaaS++:
*   **Vs. Round-Robin**: The gap is massive. Llumnix improves mean end-to-end latency by **2×** and P99 by **2.9×**. The most dramatic gains are in **Prefill Latency**, where Llumnix is up to **34.4× faster** at the P99. This directly attributes to **de-fragmentation**: Round-Robin scatters requests, leaving no single instance with enough contiguous memory for large inputs, causing them to queue indefinitely. Llumnix migrates requests to consolidate space, allowing these large requests to start immediately.
*   **Vs. INFaaS++**: Even against a load-aware dispatcher, Llumnix wins. It improves P99 prefill latency by up to **5.5×** and P99 decode latency by **1.3×**.
    *   *Why?* INFaaS++ makes decisions only at *arrival*. If a request arrives at a seemingly empty instance but then grows to consume 80% of memory, INFaaS++ cannot react. Llumnix detects this imbalance and migrates other requests away.
    *   **Preemption Reduction**: The rightmost column of Figure 11 shows **Preemption Loss**. Llumnix reduces this loss by **84%** on average compared to Round-Robin and **70.4%** compared to INFaaS++. In many cases, Llumnix drives preemption loss to near zero. This confirms that runtime rebalancing is superior to static balancing for unpredictable workloads.

#### Synthetic Long-Tail Workloads
The bottom rows of Figure 11 (S-S, M-M, L-L, etc.) reveal how performance scales with workload variance.
*   **Fragmentation Analysis**: **Figure 12** offers a deep dive into the "M-M" trace. It plots the **Fragmentation Proportion** (wasted memory that exists in total but not contiguously) over time. INFaaS++ consistently wastes **~7.9%** of cluster memory due to fragmentation, whereas Llumnix keeps this near **0.7%** (a **92% reduction**).
*   **Impact on Latency**: This reduction in wasted space directly translates to the **14.8× improvement in P99 prefill latency** observed in the L-L (Long-Long) trace. When inputs are long, the need for contiguous blocks is critical; Llumnix's ability to defragment dynamically ensures these requests are never stuck waiting for space that technically exists but is scattered.

### 5.4 Priority Isolation: Dynamic vs. Static

Can Llumnix protect VIP users without starving everyone else? **Figure 13** evaluates this by assigning **10% of requests** as "High Priority" and subjecting the system to bursty workloads (varying Gamma Coefficient of Variance, or CV).

*   **High-Priority Gains**: As burstiness increases (higher CV), interference typically worsens. However, Llumnix maintains stable latencies for high-priority requests.
    *   **Prefill**: Mean prefill latency for high-priority requests improves by **2.9× to 8.6×** compared to the priority-agnostic baseline. This is achieved by migrating lower-priority requests away to clear the queue for VIPs.
    *   **Decode**: High-priority decode latency improves by **1.2× to 1.5×**. The "Decode Execution Time" chart confirms this is due to reduced interference (lower instance load), not just queue jumping.
*   **Impact on Normal Requests**: Crucially, the second row of Figure 13 shows that **normal requests suffer minimally**. Their mean latency increases by only **4.5%**, prefill by **13%**, and decode by **2%**.
*   **Insight**: This demonstrates the efficiency of **dynamic isolation**. Unlike static reservation (where dedicated GPUs sit idle when VIP traffic is low), Llumnix only evicts normal requests when necessary. The system effectively "borrows" capacity from normal users during spikes and returns it immediately after, achieving strong isolation with negligible collateral damage.

### 5.5 Auto-Scaling and Cost Efficiency

LLM serving costs are driven by the number of active GPUs. Llumnix claims to save costs by scaling more efficiently. **Figure 14** and **Figure 15** validate this.

*   **Responsiveness**: In Figure 14, as request rates increase, Llumnix maintains lower latencies than INFaaS++ at the same scale. More importantly, it scales up and down faster. Because Llumnix can **migrate** existing requests to a new instance (saturating it immediately) or drain an instance (by marking it with infinite virtual usage), it doesn't have to wait for natural request arrivals/completions to balance the cluster.
*   **Cost Savings**: **Figure 15** plots P99 prefill latency against the average number of instances used. To achieve a target latency of roughly **5 seconds** (red dashed line):
    *   **INFaaS++** requires ~14 instances on average.
    *   **Llumnix** achieves the same latency with only ~9 instances.
    *   This represents a **36.49% cost saving**.
*   **Trade-off**: The experiment shows that Llumnix can operate at higher utilization levels without violating SLOs because its dynamic rescheduling prevents the tail-latency spikes that usually force operators to over-provision resources as a safety buffer.

### 5.6 Scalability and Robustness

Finally, does the distributed architecture hold up under pressure? **Figure 16** presents a stress test on **64 instances** (simulated, as the physical testbed was limited to 16).

*   **Centralized Bottleneck**: A baseline centralized scheduler (tracking every request) experiences scheduling stalls of up to **40 ms** per iteration under high load. This stall time is comparable to the actual inference time, causing a **1.7× slowdown** in total throughput.
*   **Llumnix Scalability**: Llumnix exhibits **near-zero scheduling stalls** even at high request rates. By offloading request-level logic to local Llumlets and communicating only aggregate "Freeness" metrics globally, the system avoids the communication bottleneck. This confirms that the hierarchical design is essential for scaling beyond a few nodes.

### 5.7 Critical Assessment

The experimental analysis is robust and convincingly supports the paper's claims.
*   **Strengths**: The use of both real and synthetic traces effectively isolates the specific problems (fragmentation vs. preemption). The comparison against an optimized INFaaS++ baseline strengthens the claim that *migration* (not just better dispatching) is the key differentiator. The ablation using `Llumnix-base` clearly separates the benefits of migration from priority handling.
*   **Limitations**: The evaluation is limited to LLaMA models up to 30B parameters. While the authors argue the mechanics scale, the network overhead for migrating state for massive models (e.g., 70B+) on slower interconnects (e.g., standard Ethernet vs. InfiniBand) might warrant further investigation. Additionally, the fault tolerance section mentions aborting requests on instance failure; while acceptable for stateless retries, the impact on long-running batch jobs (which might lose significant progress) is noted but not deeply quantified in terms of retry storms.
*   **Verdict**: The data unequivocally shows that Llumnix solves the "dispatch-and-forget" limitation. By reducing P99 latencies by an order of magnitude and cutting costs by over a third, it demonstrates that treating LLM serving as a dynamic, OS-like environment is not just theoretically sound but practically superior.

## 6. Limitations and Trade-offs

While Llumnix demonstrates significant improvements in tail latency and cost efficiency, its design relies on specific assumptions about the underlying hardware, model architecture, and workload characteristics. Understanding these constraints is crucial for determining where Llumnix excels and where it might face challenges in production environments.

### 6.1 Hardware and Network Dependencies
The efficacy of Llumnix's live migration mechanism is fundamentally tied to the ratio between **network bandwidth** and **GPU inference speed**.
*   **The Bandwidth-Speed Ratio**: The "append-only" pipelining strategy works because the system can copy the KV cache of completed tokens faster than the GPU generates new ones. As noted in **Section 5.2**, the migration typically completes in just two stages because the copying is "sufficiently fast."
    *   **Constraint**: This assumption holds well in the evaluated testbed (64 Gb/s network bandwidth between A10 GPUs). However, in clusters with slower interconnects (e.g., standard 10/25 GbE) or when serving massive models where the KV cache per token is enormous (e.g., models with very large hidden dimensions or many layers), the copy speed might fall behind the generation speed.
    *   **Consequence**: If the network cannot keep up, the number of "catch-up" stages would increase, potentially extending the migration duration and increasing the risk that the request finishes or is preempted before migration completes. The paper does not explicitly quantify the breaking point where network latency renders migration inefficient.
*   **Communication Library Limitations**: In **Section 5**, the authors note they use **Gloo** instead of **NCCL** for data transfer. While NCCL is generally faster for GPU-to-GPU communication, it is unsafe for concurrent invocations required by the pipelined migration design.
    *   **Trade-off**: Llumnix sacrifices raw transfer speed (by using Gloo and copying data via CPU memory) to ensure correctness and non-blocking operation. While the evaluation shows this overhead is negligible for the tested models, this CPU-mediated copy path could become a bottleneck for extremely large context windows where the KV cache size exceeds CPU memory bandwidth capabilities.

### 6.2 Model and Context Length Scalability
The evaluation focuses on **LLaMA-7B** and **LLaMA-30B** with a maximum sequence length of **2k tokens** (due to the specific vLLM version used).
*   **Extrapolation to Long Contexts**: The authors argue in **Section 6.1** that the results are representative for models supporting 4k to 128k contexts because the *mechanisms* of fragmentation and preemption are identical.
    *   **Uncertainty**: While the *logic* holds, the *magnitude* of the state transfer changes drastically. Migrating an 8k sequence is manageable; migrating a 128k sequence involves gigabytes of state. Although the downtime remains constant (determined only by the last iteration), the **background bandwidth consumption** during the multi-stage copy would be sustained for a much longer period. This could saturate the network link for extended durations, potentially interfering with other system traffic or tensor-parallel communication if not carefully isolated.
*   **Model Architecture Specificity**: Llumnix exploits the **autoregressive, append-only** nature of standard Transformer decoders.
    *   **Limitation**: This approach may not directly apply to non-autoregressive models, models with bidirectional attention mechanisms that modify past states during inference, or speculative decoding setups where tokens are verified and potentially rejected (which would invalidate the "append-only" assumption for the rejected tokens). The paper does not address how Llumnix would handle such emerging inference paradigms.

### 6.3 Fault Tolerance and Progress Loss
The system's approach to failure involves a trade-off between availability and progress preservation.
*   **Request Abortion on Failure**: As described in **Section 5** (Implementation), if an instance (or its co-located `llumlet`) fails, the requests running on it are **aborted**.
    *   **Impact on Workloads**: For interactive, short-latency requests (chatbots), a retry is often acceptable. However, for long-running offline tasks (e.g., summarizing a book or generating code for a large project) that have already run for hundreds of iterations, an abort means losing all computed progress.
    *   **Missing Mechanism**: Unlike traditional checkpointing systems that save state to persistent storage to survive node failures, Llumnix relies on the liveness of the source instance during migration. If the source dies *during* migration, the handshake protocol aborts the process, and the request is lost. The paper acknowledges this in **Section 5** but does not propose a mechanism for persistent checkpointing of the KV cache, which remains an open challenge for long-duration jobs in this architecture.

### 6.4 Scheduling Granularity and Overhead
While the distributed architecture solves the scalability bottleneck of centralized schedulers, it introduces a layer of indirection.
*   **Coarse-Grained Global Decisions**: The Global Scheduler makes decisions based on **instance-level metrics** (Freeness) and pairs instances, leaving the specific request selection to the local `llumlet`.
    *   **Potential Sub-optimality**: This decoupling means the Global Scheduler does not have a global view of *which* specific requests are causing the imbalance. The local `llumlet` uses simple heuristics (preferring lower priority and shorter sequences) to select requests. In complex scenarios where a specific combination of long, high-priority requests causes a bottleneck, the local heuristic might not make the globally optimal choice for de-fragmentation, potentially requiring multiple migration rounds to resolve.
*   **Migration Frequency**: The system triggers migration periodically. If the workload dynamics change faster than the migration interval, the system might react too slowly to prevent preemptions. Conversely, triggering migration too frequently increases the overhead of the handshake protocol and network setup. The paper identifies the existence of thresholds but does not deeply explore the sensitivity of performance to the **migration trigger frequency** or the specific tuning of the Freeness thresholds ($x, y$) across vastly different hardware configurations.

### 6.5 Multi-Model and Heterogeneous Clusters
The current evaluation and design assume a **homogeneous cluster** serving a **single model type** (or identical replicas).
*   **Heterogeneity Gap**: In real-world deployments, a cluster often serves multiple model variants (e.g., LLaMA-7B, LLaMA-70B, and specialized fine-tuned models) on heterogeneous hardware (e.g., mixing A10, A100, and H100 GPUs).
    *   **Limitation**: Llumnix's migration mechanism assumes the source and destination instances have compatible memory layouts and model weights. Migrating a request from an A10 to an H100, or between different model architectures, is not addressed. As noted in **Section 7** (Related Work), extending Llumnix to support multiple model types is planned future work, but currently, the system cannot exploit migration to balance load across heterogeneous model pools.

### 6.6 Summary of Trade-offs

| Aspect | Benefit | Trade-off / Limitation |
| :--- | :--- | :--- |
| **Migration Strategy** | Near-zero downtime (~20ms) | Relies on network bandwidth > generation speed; uses slower Gloo/CPU copy path. |
| **Fault Tolerance** | High availability via fast restart | **No progress preservation**; long-running requests are aborted on instance failure. |
| **Scheduling Logic** | Scalable (instance-level metrics) | Local request selection heuristics may be sub-optimal compared to global knowledge. |
| **Workload Scope** | Optimized for autoregressive decoding | Not directly applicable to non-autoregressive or bidirectional inference patterns. |
| **Cluster Homogeneity** | Simplifies migration logic | Cannot currently migrate across heterogeneous hardware or different model architectures. |

In conclusion, Llumnix represents a powerful shift towards dynamic, OS-like scheduling for LLMs, but it is not a silver bullet. Its success depends on a high-speed network fabric, homogeneous model deployments, and workloads where request retries are an acceptable cost of failure. Future iterations must address persistent state management for long jobs and extend the migration protocol to support heterogeneous environments to fully realize the vision of "LLM serving as Unix."

## 7. Implications and Future Directions

The introduction of **Llumnix** marks a pivotal transition in the architecture of Large Language Model (LLM) serving systems. By successfully demonstrating that **runtime request migration** is not only feasible but highly efficient, Llumnix challenges the long-standing "dispatch-and-forget" paradigm that has dominated both traditional DNN serving and early LLM infrastructure. This section explores how this work reshapes the field, the new research avenues it opens, and practical guidelines for adopting these techniques.

### 7.1 Shifting the Paradigm: From Static Dispatch to Dynamic Orchestration

The most profound implication of Llumnix is the conceptual re-framing of an LLM cluster. Prior to this work, a cluster of GPU instances was viewed as a collection of static silos: once a request was assigned to a GPU, it was bound to that hardware until completion. This static view forced system designers to rely on **predictive heuristics** (guessing request length) or **over-provisioning** (adding extra GPUs to handle worst-case fragmentation) to maintain Service Level Objectives (SLOs).

Llumnix proves that LLM serving shares fundamental characteristics with **modern Operating Systems (OS)**:
*   **Context Switching:** Just as an OS moves processes between CPU cores to balance load and prioritize tasks, Llumnix moves inference requests between GPUs.
*   **Dynamic Memory Management:** The system treats GPU memory not as fixed partitions but as a fluid, cluster-wide resource pool that can be defragmented in real-time.
*   **Isolation via Scheduling:** Instead of relying on physical isolation (dedicated servers for VIP users), Llumnix achieves logical isolation through dynamic eviction, similar to how OS schedulators handle real-time threads.

This shift implies that future serving systems must be **reactive** rather than **predictive**. The complexity of the scheduler moves from trying to perfectly predict request behavior at arrival (an impossible task given LLM stochasticity) to continuously monitoring and correcting imbalances at runtime. This reduces the need for conservative over-provisioning, directly translating to the **36% cost savings** observed in the evaluation.

### 7.2 Enabling New Research Directions

Llumnix unlocks several promising avenues for future research by solving the foundational problem of state transfer:

#### A. Cross-Model and Heterogeneous Migration
Currently, Llumnix migrates requests between instances running the **same model weights**. A natural extension is **heterogeneous migration**, where a request starts on a smaller, cheaper model (or a quantized version) and migrates to a larger, more powerful instance if the context grows complex or if priority increases.
*   *Research Question:* Can we develop translation layers that allow a KV cache generated by a 7B model to be seamlessly resumed on a 70B model (perhaps via distillation maps) or across different hardware architectures (e.g., A10 to H100)?

#### B. Integration with Speculative Decoding
Speculative decoding (using a small "draft" model to propose tokens verified by a large "target" model) is becoming standard for acceleration.
*   *Opportunity:* Llumnix's migration mechanism could be adapted to move the **verification state** between nodes. For instance, if a draft model on a local CPU generates a long sequence, the heavy verification step could be migrated to a less loaded GPU cluster without restarting the generation, optimizing the use of specialized verification hardware.

#### C. Persistent State and Fault Tolerance
As noted in the limitations, Llumnix currently aborts requests upon instance failure. The efficient pipelined copying mechanism suggests a path toward **low-overhead checkpointing**.
*   *Future Work:* Instead of copying to another active instance for load balancing, the same pipeline could copy state to **persistent storage** (NVMe or remote memory) at regular intervals. This would enable "pause-and-resume" capabilities for multi-hour batch jobs, making LLMs viable for long-running scientific or data-processing tasks where losing progress is unacceptable.

#### D. Geo-Distributed Serving
If migration overhead can be further reduced or hidden, the scope could expand from a single datacenter to **geo-distributed clusters**.
*   *Vision:* A user in Europe could start a session on a local edge node and, as the conversation deepens or requires more compute, have their state migrated to a central cloud region with specialized accelerators, all without the user perceiving a disconnect.

### 7.3 Practical Applications and Downstream Use Cases

The capabilities introduced by Llumnix directly address pain points in several high-value commercial and scientific domains:

*   **Tiered SaaS Platforms (e.g., ChatGPT Plus):**
    Companies offering freemium LLM services can now guarantee strict latency SLOs for premium subscribers without dedicating entire GPU clusters to them. Llumnix allows the system to dynamically create "fast lanes" by evicting free-tier traffic during spikes, maximizing hardware utilization while preserving the premium experience.

*   **Long-Context Enterprise Agents:**
    Enterprise applications often involve agents that summarize massive documents (legal contracts, codebases) or engage in multi-turn debugging sessions. These workloads are prone to the **memory fragmentation** that Llumnix solves. By ensuring that large contiguous memory blocks are always available via defragmentation, enterprises can deploy longer-context models more reliably without fearing "out-of-memory" stalls.

*   **Bursty Interactive Workloads:**
    Applications like gaming NPCs or real-time translation services experience extreme traffic bursts. Llumnix's rapid **auto-scaling** (saturation and draining of instances via migration) allows infrastructure to scale up instantly to meet demand and scale down just as quickly when the burst subsides, optimizing cloud spend.

*   **Multi-Tenant Research Clouds:**
    University or corporate clusters shared by multiple research teams often suffer from the "noisy neighbor" problem. Llumnix provides a mechanism to enforce fair sharing or priority access dynamically, ensuring that a large batch job from one team does not starve an interactive experiment from another.

### 7.4 Reproduction and Integration Guidance

For practitioners and researchers looking to adopt or build upon Llumnix, the following guidelines clarify when and how to utilize this approach:

#### When to Prefer Llumnix Over Alternatives
*   **Choose Llumnix if:**
    *   Your workload exhibits **high variance** in input/output lengths (e.g., mixing chat with document summarization).
    *   You have **strict tail-latency SLOs** (P99) that are currently being violated by preemptions or queuing.
    *   You operate a **multi-tenant cluster** requiring priority isolation without static partitioning.
    *   Your network interconnect is high-bandwidth (e.g., ≥25 GbE, ideally InfiniBand or proprietary cloud interconnects like Alibaba's 64 Gb/s setup).
*   **Stick to Static Dispatch (e.g., vLLM standalone, Round-Robin) if:**
    *   Your workloads are **homogeneous** (e.g., all requests are short chat prompts with similar lengths).
    *   You are deploying on **low-bandwidth networks** where KV cache transfer would saturate the link and degrade overall throughput.
    *   Your application cannot tolerate **any** request abortion in the event of a node failure (until persistent checkpointing is added).

#### Integration Strategy
Llumnix is designed as a **non-intrusive scheduling layer**.
1.  **Backend Compatibility:** It currently integrates with **vLLM**. To adopt it, organizations should standardize on vLLM as their inference engine. The integration requires running Llumnix as a sidecar or wrapper that manages multiple vLLM instances via **Ray actors**.
2.  **Configuration Tuning:** The effectiveness of Llumnix depends on tuning the **Freeness thresholds** ($x, y$) for auto-scaling.
    *   *Guideline:* Start with the paper's default range of **[10, 60]** (representing future decode steps). Monitor the **migration frequency**; if migrations are too frequent (causing network noise), widen the gap between thresholds. If preemptions still occur, lower the lower bound $x$.
3.  **Priority Setup:** To leverage priority isolation, applications must tag requests with priority levels. The system requires profiling to determine the **headroom** needed for high-priority requests to achieve "interference-free" speeds (as done in Section 6.4 with the 1,600 token target).

#### Reproducibility Note
The authors have released the code at `https://github.com/AlibabaPAI/llumnix`.
*   **Artifact vs. Production:** Be aware that the `osdi24ae` branch contains the research prototype used for the paper's experiments. The `main` branch contains the refactored, production-ready version with improved fault tolerance and API support.
*   **Hardware Requirements:** To reproduce the results faithfully, a multi-node setup is essential. The benefits of Llumnix (migration, defragmentation) **do not manifest on a single GPU**. A minimum of 4 GPUs across at least 2 nodes is recommended to observe the network transfer dynamics and cross-instance load balancing.

In summary, Llumnix provides the missing "control plane" for dynamic LLM serving. By treating GPU clusters as fluid, manageable resources rather than static compute blocks, it paves the way for more efficient, reliable, and cost-effective generative AI infrastructure.
