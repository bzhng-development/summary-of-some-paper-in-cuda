## 1. Executive Summary

This paper introduces **PrefillOnly**, the first LLM inference engine specifically designed for "prefill-only" workloads—emerging discriminative tasks like recommendation and credit verification where the model generates only a single output token. By leveraging the fact that these requests do not require multi-token decoding, PrefillOnly employs **hybrid prefilling** to discard most KV caches and **continuous Job Completion Time (JCT) calibration** to optimize scheduling, thereby reducing the active GPU memory footprint and enabling deterministic latency prediction. Evaluated on workloads such as post recommendation and credit verification using models like **Llama-3.1-8B** and **Qwen-32B**, the system handles up to **4× higher queries per second** than baseline engines (e.g., vLLM with tensor or pipeline parallelism) while maintaining lower average and P99 latency, effectively solving the throughput bottlenecks caused by unnecessary memory retention in traditional generative engines.

## 2. Context and Motivation

To understand the necessity of **PrefillOnly**, we must first recognize a fundamental shift in how Large Language Models (LLMs) are being deployed in production environments. While the public perception of LLMs is dominated by generative applications like ChatGPT or GitHub Copilot—where the model writes paragraphs of text—the industry is increasingly adopting LLMs for **discriminative tasks**. These include recommendation systems, credit verification, and data labeling.

### The Emergence of "Prefill-Only" Workloads

In traditional generative tasks, an LLM request consists of two distinct phases:
1.  **Prefilling:** The model processes the entire input prompt (which can be thousands of tokens) to generate the *first* output token.
2.  **Decoding:** The model iteratively generates subsequent tokens, one by one, reusing previously computed states until an end-of-sequence token is reached.

However, in discriminative tasks, the goal is often binary or categorical classification (e.g., "Should we recommend this document? Yes/No" or "Is this credit application valid? Approved/Rejected"). As described in **Section 2.3**, these applications constrain the LLM to generate exactly **one single output token**.

> "To determine whether this document should be recommended, the LLM only needs to generate one output token (i.e., Yes or No)." (**Section 1**)

Because the inference stops immediately after the first token is generated, the **decoding phase is effectively non-existent**. The entire computational workload consists solely of the prefilling phase. The authors term this the **prefill-only workload**.

This distinction is critical because the architectural assumptions of existing LLM engines are built around the expectation of long decoding sequences. When applied to prefill-only workloads, these assumptions lead to significant inefficiencies.

### The Core Problem: Mismatched Architectural Assumptions

Existing state-of-the-art inference engines (such as vLLM, Orca, or DistServe) are optimized for generative workflows where the **Key-Value (KV) cache** is a precious resource that must be preserved.

#### 1. The KV Cache Bottleneck
In standard LLM inference, the **KV cache** stores the intermediate attention keys and values for every token processed. This allows the model to avoid recomputing the entire history when generating the *next* token during the decoding phase.
*   **Standard Assumption:** The engine must store the KV cache for *all layers* and *all tokens* in GPU memory to support arbitrary-length decoding.
*   **The Reality for Prefill-Only:** Since the request terminates after one token, the KV cache is **never reused**. Storing it is a waste of memory.

As noted in **Section 2.5**, this unnecessary retention creates a hard limit on the **Maximum Input Length (MIL)** a single GPU can handle. For example, on an NVIDIA A100 (40GB) with a Qwen-32B model, the MIL is capped at roughly **11,000 tokens** because the KV cache consumes the majority of the memory.

When applications require longer contexts (e.g., a user profile containing months of browsing history, totaling 40,000–60,000 tokens as seen in the **Credit Verification** dataset in **Table 1**), existing engines are forced to adopt workaround strategies that severely degrade performance:
*   **Chunked Prefilling:** Processing the input in small chunks to fit memory. This reduces throughput by **14%** due to inefficient attention kernel execution (**Section 2.5**).
*   **Tensor/Pipeline Parallelism:** Splitting the model across multiple GPUs. While this increases capacity, it introduces expensive communication overheads (all-reduce operations) that under-utilize GPU computation and lower overall throughput (**Section 2.5**).

#### 2. The Scheduling Uncertainty Gap
The second major inefficiency lies in request scheduling.
*   **Standard Assumption:** In generative tasks, the output length is stochastic (random). An engine cannot know how long a request will take to complete (Job Completion Time, or **JCT**) until it finishes. Consequently, engines use simple, JCT-agnostic policies like First-Come-First-Serve (FCFS).
*   **The Reality for Prefill-Only:** The output length is fixed at **one token**. Therefore, the JCT is **deterministic** and can be precisely calculated before the request starts, based purely on the input length and prefix cache status.

Existing engines fail to leverage this determinism, missing the opportunity to use optimal scheduling algorithms like **Shortest Remaining Job First (SRJF)**, which could significantly reduce average latency and improve cache utilization.

### Why Existing Solutions Fall Short

One might ask: *Why not simply modify existing engines to delete the KV cache after the first token?*

The paper identifies two subtle but critical barriers that prevent naive modifications from working:

1.  **Intermediate Tensor Spikes:** Even if you discard the KV cache, the GPU memory usage during the prefill phase is dominated not by the KV cache, but by **intermediate activation tensors** generated by the linear layers (MLP modules).
    *   **Figure 3** and **Figure 4** illustrate that for a Llama-3.1-8B model, the intermediate tensors in the MLP module can be **14× larger** than the KV cache of a single layer.
    *   Simply dropping the KV cache only increases the maximum input length by **1.6×**, which is insufficient for long-context workloads (**Section 2.6**).

2.  **Prefix Caching Conflicts:** Many discriminative tasks involve repeated queries with similar prefixes (e.g., the same user profile evaluated against 50 different documents). Existing engines use **prefix caching** to store the KV state of common prefixes to accelerate these requests.
    *   If an engine blindly discards *all* KV caches, it destroys the prefix cache, forcing recomputation for every request and skyrocketing latency.
    *   Existing engines lack a mechanism to selectively discard only the *suffix* (the unique part of the current request) while preserving the *prefix* (the shared history) without complex engineering changes.

### Positioning of PrefillOnly

**PrefillOnly** positions itself not as a general-purpose LLM engine, but as a specialized runtime tailored explicitly for this emerging class of discriminative, single-token workloads. It differentiates itself from prior work in three key dimensions:

| Feature | Traditional Engines (vLLM, Orca) | Naive Modification | **PrefillOnly** |
| :--- | :--- | :--- | : |
| **KV Cache Strategy** | Store all layers for all tokens. | Discard all caches. | **Hybrid Prefilling:** Stores only necessary prefix; discards suffix; chunks non-attention layers to minimize peak memory. |
| **Memory Bottleneck** | Limited by total KV cache size. | Limited by intermediate tensor spikes. | **Optimized:** Reduces peak memory by chunking linear layers, enabling **5× longer inputs** without parallelism (**Section 7.2**). |
| **Scheduling Logic** | JCT-agnostic (FCFS) due to uncertainty. | Static JCT estimation. | **Continuous JCT Calibration:** Dynamically recalculates JCT based on real-time prefix cache availability to maximize cache hits. |
| **Target Workload** | Generative (Chat, Code). | N/A (Inefficient). | **Discriminative** (Recommendation, Verification). |

The significance of this work is both theoretical and practical. Theoretically, it challenges the dogma that LLM serving must always prioritize decoding optimization. Practically, it unlocks the ability to run complex, long-context discriminative tasks on commodity hardware without the prohibitive costs of multi-GPU parallelism, enabling throughput scales of **tens of thousands of queries per second** required by modern recommendation systems (**Section 2.4**).

By addressing the specific memory and scheduling mismatches of prefill-only workloads, **PrefillOnly** achieves up to **4× higher throughput** than baseline engines while maintaining lower latency, effectively bridging the gap between the capabilities of modern LLMs and the efficiency requirements of high-scale industrial applications.

## 3. Technical Approach

### 3.1 Reader orientation (approachable technical breakdown)
PrefillOnly is a specialized LLM inference engine designed to execute "prefill-only" workloads—tasks where the model processes a long input but generates exactly one output token—by fundamentally altering how GPU memory is managed and how requests are scheduled. It solves the dual problems of memory exhaustion on long inputs and inefficient request ordering by introducing a "hybrid prefilling" technique that chunks memory-heavy operations without slowing down attention, and a "continuous JCT calibration" scheduler that dynamically re-ranks waiting requests to maximize cache reuse.

### 3.2 Big-picture architecture (diagram in words)
The system operates as a centralized scheduler coordinating multiple executor processes, functioning through a clear three-stage pipeline:
1.  **Profile Run & Initialization:** Before serving live traffic, the system performs a dry-run with a fake request of the user-defined Maximum Input Length (MIL) to measure peak GPU memory usage; it then partitions the remaining GPU memory exclusively for storing prefix KV caches.
2.  **Request Ingestion & Queueing:** Incoming HTTP requests are tokenized and placed into a central waiting queue managed by a scheduler process, where they await selection based on a dynamically calculated score rather than arrival time.
3.  **Dynamic Scheduling & Execution:** In each scheduling step, the scheduler iterates through the waiting queue to recalculate the Job Completion Time (JCT) for every request based on current prefix cache availability, selects the single request with the lowest adjusted JCT, and dispatches it to an executor process which runs the inference using **hybrid prefilling** to minimize peak memory and discards the suffix KV cache immediately after completion.

### 3.3 Roadmap for the deep dive
*   **Hybrid Prefilling Mechanism:** We first explain the core memory optimization that allows PrefillOnly to handle inputs up to 5× longer than baselines by distinguishing between attention and non-attention layers.
*   **Memory Management & Suffix Discarding:** We detail how the system selectively retains prefix caches for reuse while aggressively discarding suffix caches to free up space, a strategy enabled by the hybrid approach.
*   **Continuous JCT Calibration:** We dissect the scheduling algorithm that replaces static First-Come-First-Serve logic with a dynamic, cache-aware prioritization system to minimize latency.
*   **Implementation Specifics:** We conclude with the practical engineering choices, including the use of `torch.compile` and specific hyperparameters like the fairness factor $\lambda$, that make these theoretical contributions deployable.

### 3.4 Detailed, sentence-based technical breakdown

This paper presents a systems engineering solution that re-architects the LLM inference pipeline specifically for single-token output workloads, leveraging the insight that the absence of a decoding phase allows for aggressive memory reclamation and deterministic scheduling that are impossible in generative settings.

#### The Core Innovation: Hybrid Prefilling
The primary bottleneck in handling long inputs for prefill-only workloads is not just the KV cache, but the massive **intermediate activation tensors** generated by the linear layers (specifically the Multi-Layer Perceptron or MLP modules) during the forward pass.
*   In standard LLM architectures like Llama, the MLP module expands the hidden dimension significantly (e.g., from 4096 to 28672 in Llama-3.1-8B), creating intermediate tensors that are **14× larger** than the KV cache of a single layer, as illustrated in **Figure 4**.
*   Traditional "chunked prefilling" attempts to solve memory limits by processing the input in small segments, but this forces the attention mechanism to also operate in chunks, which disrupts the highly optimized attention kernels and reduces throughput by approximately **14%** (**Section 2.5**).
*   PrefillOnly introduces **hybrid prefilling**, a technique that treats attention layers and non-attention (linear) layers differently to maximize memory efficiency without sacrificing compute performance.
*   Specifically, the system processes **non-attention layers** (the MLP modules) chunk-by-chunk, computing the activation for a small segment of tokens, writing the result to a pre-allocated output buffer, and immediately freeing the large intermediate tensor before moving to the next chunk.
*   Conversely, the system processes **attention layers** normally (in a single pass over the full sequence), avoiding the performance penalty of chunked attention kernels while relying on the fact that attention layers do not generate the massive intermediate spikes seen in MLP modules.
*   This hybrid approach is mathematically valid because linear layers operate independently on each token (or chunk of tokens) without requiring global context, whereas attention layers require global context but have a much smaller memory footprint per token compared to the MLP intermediates.
*   By implementing this via `torch.compile`, the system dynamically rewrites the computation graph to group consecutive linear operations into a "virtual layer" that is executed iteratively over chunks, concatenating the results into a pre-allocated tensor to avoid doubling memory usage during the concatenation step (**Section 4.3**).
*   The result of this design is a drastic reduction in peak GPU memory usage, allowing the system to handle a Maximum Input Length (MIL) of **130,000 tokens** on an NVIDIA L4 GPU, compared to only **24,000 tokens** for standard PagedAttention and **46,000 tokens** for standard chunked prefill (**Table 2**).

#### Suffix KV Cache Discarding and Prefix Preservation
While hybrid prefilling reduces the *active* memory during computation, PrefillOnly must also manage the *stored* memory (KV caches) to support long contexts and prefix reuse simultaneously.
*   The system employs a strategy called **suffix KV cache discarding**, where it distinguishes between the **prefix** (the shared part of the prompt, such as a user profile) and the **suffix** (the unique part, such as a specific document being evaluated).
*   Upon completion of a request, the engine immediately discards (or optionally offloads to CPU) the KV cache entries corresponding to the suffix tokens, as these will never be reused in a prefill-only workload that terminates after one token.
*   However, the KV cache entries for the prefix tokens are retained in the GPU's high-speed memory to serve as a **prefix cache** for subsequent requests that share the same initial context.
*   This selective retention is critical: blindly discarding all caches would destroy the benefits of prefix caching, while retaining all caches would quickly exhaust GPU memory given the long input lengths typical of these workloads (e.g., 40,000–60,000 tokens for credit verification, per **Table 1**).
*   The implementation leverages existing abstractions for **sliding window attention** found in vLLM, repurposing them to define a "window" that effectively covers only the prefix, allowing the suffix to be truncated from the cache table without modifying low-level hardware kernels (**Section 5.1**).
*   This approach enables PrefillOnly to support workloads that require both extreme context lengths and high cache hit rates, achieving a **5× increase** in maximum input length compared to non-parallelized baselines without incurring the communication overheads of tensor or pipeline parallelism (**Section 7.2**).

#### Continuous JCT Calibration for Cache-Aware Scheduling
The second pillar of PrefillOnly is its scheduling logic, which exploits the deterministic nature of prefill-only workloads to optimize latency and cache utilization.
*   In traditional generative serving, the Job Completion Time (JCT) is unpredictable because the output length is stochastic; however, in prefill-only workloads, the output length is fixed at **one token**, making the JCT a deterministic function of the number of tokens that must be computed.
*   The core challenge is that the effective computation load depends on **prefix cache hits**: if a request's prefix is already in the GPU cache, its JCT is proportional only to the suffix length; if the cache is missed, the JCT is proportional to the total input length.
*   Standard Shortest Remaining Job First (SRJF) schedulers fail here because they calculate JCT once upon arrival; if the cache state changes (e.g., a relevant prefix is loaded or evicted) while the request is waiting, the initial JCT estimate becomes stale, leading to suboptimal ordering (**Section 6.2**).
*   PrefillOnly solves this with **continuous JCT calibration**, an algorithm that re-evaluates the JCT of *every* waiting request immediately before each scheduling decision.
*   The system estimates the JCT for a request $r$ using a proxy metric based on the number of cache-miss tokens, defined as $n_{\text{miss}} = n_{\text{input}} - n_{\text{cached}}$, where $n_{\text{input}}$ is the total token count and $n_{\text{cached}}$ is the count of tokens found in the current prefix cache.
*   Empirical profiling shows this proxy has a Pearson correlation coefficient of **0.987** with actual execution time on an A100 GPU, making it a highly accurate predictor without needing complex runtime profiling (**Section 6.3**).
*   To prevent starvation of long requests that rarely hit the cache, the system introduces a fairness adjustment to the scheduling score, calculated as:
    $$ \text{score} = \text{get\_jct}(n_{\text{input}}, n_{\text{cached}}) - \lambda \cdot T_{\text{queue}} $$
    where $T_{\text{queue}}$ is the time the request has spent waiting, and $\lambda$ is a hyperparameter controlling the trade-off between average latency and worst-case (P99) latency.
*   By default, PrefillOnly sets $\lambda = 500$; increasing this value prioritizes older requests more aggressively, improving P99 latency at the cost of slightly higher average latency, as demonstrated in the cumulative distribution function (CDF) analysis in **Figure 11**.
*   This continuous recalibration ensures that if a request $D$ shares a prefix with a just-completed request $A$, its JCT estimate drops instantly, causing the scheduler to prioritize $D$ over other waiting requests, thereby maximizing the probability that the cached prefix of $A$ is reused before it is evicted (**Figure 5**).

#### System Workflow and Configuration
The operational flow of PrefillOnly integrates these components into a cohesive serving loop designed for high-throughput discriminative tasks.
*   **Initialization Phase:** The user specifies the target Maximum Input Length (MIL); the system runs a "profile run" with a dummy request of this length to measure the peak memory consumption of the hybrid prefilling process, then dedicates the remaining GPU memory to the prefix cache pool (**Section 3.1**).
*   **Request Handling:** Requests arrive via an HTTP server compatible with the OpenAI API, are tokenized, and sent via ZeroMQ RPC to the scheduler process, which maintains the global waiting queue.
*   **Scheduling Loop:** The scheduler operates in discrete steps; in each step, it iterates through the queue, computes the calibrated score for each request using the current cache state, and selects the single request with the minimum score for execution.
*   **Execution:** The selected request is sent to an executor process, which performs the forward pass using hybrid prefilling; upon generating the single output token (and its probability distribution), the executor discards the suffix KV cache and returns the result to the user.
*   **Parallelism Strategy:** Unlike baselines that use tensor or pipeline parallelism to fit long contexts, PrefillOnly scales by running multiple independent instances (one per GPU) and using **user-ID-based routing** to direct all requests from a specific user to the same instance, thereby naturally co-locating prefixes to maximize cache hits without inter-GPU communication overhead (**Section 7.1**).
*   This architecture allows PrefillOnly to achieve **4× higher queries per second (QPS)** than baseline engines like vLLM with tensor parallelism on high-end hardware (H100), while maintaining lower average and P99 latencies, particularly under high-load conditions where baseline systems suffer from communication bottlenecks or cache thrashing (**Figure 6** and **Figure 8**).

## 4. Key Insights and Innovations

The contributions of **PrefillOnly** extend beyond simple engineering optimizations; they represent a fundamental rethinking of LLM serving architecture when the assumption of "arbitrary length generation" is removed. The paper identifies that the inefficiencies in current systems are not merely due to lack of tuning, but stem from structural mismatches between generative engine designs and discriminative workload requirements. Below are the core innovations that distinguish this work from prior art.

### 4.1 Fundamental Decoupling of Attention and Linear Memory Profiles
**Innovation Type:** *Architectural Paradigm Shift*

Prior LLM serving engines (e.g., vLLM, Orca) treat the model forward pass as a monolithic memory operation. They assume that if a request fits in memory, all layers must be processed with the same memory strategy. If memory is tight, they apply **chunked prefilling** globally, which forces the attention mechanism—a highly optimized, memory-bandwidth-bound operation—to operate on small, inefficient chunks. This results in a direct trade-off: you can fit longer inputs, but you sacrifice **14%** of your throughput due to kernel inefficiency (**Section 2.5**).

**PrefillOnly** introduces the novel insight that **memory bottlenecks in LLMs are layer-specific**, not model-wide.
*   **The Distinction:** The paper proves that for modern architectures like Llama, the GPU memory spike during prefilling is dominated by intermediate activation tensors in the **linear layers (MLP)**, which are **14× larger** than the KV cache of a single layer (**Figure 4**). Conversely, attention layers have a relatively small memory footprint but are extremely sensitive to chunking.
*   **The Breakthrough:** By decoupling the execution strategy, PrefillOnly applies chunking *only* to the memory-heavy linear layers while keeping the attention layers intact. This is a **fundamental innovation** because it breaks the traditional coupling where memory reduction techniques inevitably degrade compute efficiency.
*   **Significance:** This allows the system to achieve a **7.9× increase** in Maximum Input Length (MIL) compared to vanilla baselines and an **8.7× increase** over standard chunked prefill (**Figure 10**) *without* the throughput penalty associated with chunked attention. It effectively creates a "free lunch" in memory capacity that was previously thought impossible without resorting to multi-GPU parallelism.

### 4.2 Dynamic, Cache-State-Dependent Scheduling
**Innovation Type:** *Algorithmic Advance in Resource Orchestration*

Traditional scheduling in LLM serving is constrained by uncertainty. Because generative requests have stochastic output lengths, engines cannot know a request's Job Completion Time (JCT) in advance. Consequently, they rely on **JCT-agnostic** policies like First-Come-First-Serve (FCFS) or static batching. Even in systems that support prefix caching, the scheduler typically makes a decision based on the cache state *at arrival time*.

**PrefillOnly** leverages the deterministic nature of prefill-only workloads (output length = 1) to introduce **Continuous JCT Calibration**, a scheduling paradigm that treats JCT as a dynamic variable rather than a static attribute.
*   **The Distinction:** In prior work, if Request A arrives and misses the cache, it is assigned a high JCT and placed at the back of the queue. If, while waiting, Request B finishes and loads the exact prefix Request A needs, standard schedulers do not react; Request A remains at the back, and by the time it runs, the cache might be evicted by Request C.
*   **The Breakthrough:** PrefillOnly recalculates the JCT of *every* waiting request before *every* scheduling decision. It uses the number of cache-miss tokens ($n_{\text{input}} - n_{\text{cached}}$) as a high-fidelity proxy (correlation **0.987**) for execution time (**Section 6.3**). This allows the scheduler to instantly promote a request the moment its required prefix enters the GPU, ensuring the cache is utilized before eviction.
*   **Significance:** This transforms the scheduler from a passive queue manager into an active **cache coherence optimizer**. As illustrated in **Figure 5**, this approach increases cache hit rates compared to both FCFS and static Shortest-Job-First (SRJF), directly translating to lower average latency and higher effective throughput in workloads with high prefix reuse (like recommendation systems). It solves the "stale estimate" problem that has plagued cache-aware scheduling in uncertain environments.

### 4.3 The "Single-GPU Long Context" Capability
**Innovation Type:** *New System Capability / Cost Efficiency*

Before PrefillOnly, handling extremely long contexts (e.g., 60,000+ tokens for credit verification) on a single GPU was effectively impossible without severe performance degradation. The standard industry solution was **Tensor Parallelism (TP)** or **Pipeline Parallelism (PP)**, which splits the model across multiple GPUs. While this increases capacity, it introduces expensive inter-GPU communication (all-reduce operations) that creates "bubbles" of idle time and drastically reduces overall throughput, especially when high-speed interconnects like NVLink are unavailable.

**PrefillOnly** demonstrates that for prefill-only workloads, **parallelism is often an anti-pattern**.
*   **The Distinction:** Existing engines view long context as a "capacity problem" solved by adding more GPUs. PrefillOnly views it as a "memory management problem" solvable by smarter deletion. By discarding the suffix KV cache and chunking only the MLP layers, it enables a single GPU to handle inputs that previously required 2–4 GPUs.
*   **The Breakthrough:** The system achieves a **4× higher query-per-second (QPS)** throughput compared to tensor-parallel baselines on H100 clusters (**Figure 8**). Crucially, it maintains this lead even when the baseline uses NVLink, proving that the communication overhead of parallelism outweighs the benefits of distributed memory for single-token workloads.
*   **Significance:** This is a major economic and architectural insight. It implies that for the burgeoning market of discriminative LLM applications (recommendation, verification, labeling), organizations do not need to invest in complex, high-bandwidth multi-GPU clusters. They can achieve superior performance on commodity, single-GPU instances, reducing infrastructure costs and complexity by an order of magnitude.

### 4.4 Reframing the KV Cache: From "Asset" to "Liability"
**Innovation Type:** *Conceptual Reframing*

In the entire history of LLM serving research, the KV cache has been treated as a precious **asset** to be preserved, compressed, or paged, because it is essential for accelerating the decoding phase. The entire design of systems like vLLM (PagedAttention) revolves around maximizing KV cache retention.

**PrefillOnly** flips this dogma on its head for prefill-only workloads, treating the KV cache as a **liability**.
*   **The Distinction:** Prior optimization efforts focus on *compression* (quantization) or *swapping* (offloading to CPU) to keep the cache available. PrefillOnly argues that for 99% of the tokens in a prefill-only request (the suffix), the cache has **zero future value**. Keeping it is not just wasteful; it is actively harmful as it blocks memory for other requests.
*   **The Breakthrough:** The system introduces **Suffix KV Cache Discarding** as a first-class citizen of the inference pipeline. Unlike naive approaches that might discard everything (breaking prefix caching) or keep everything (limiting length), PrefillOnly surgically removes only the useless suffix while preserving the useful prefix.
*   **Significance:** This conceptual shift enables the other technical innovations. It is the prerequisite that allows the "Single-GPU Long Context" capability to exist. It challenges the field to reconsider whether "universal" serving engines are truly optimal, suggesting that specialized runtimes that discard traditional assumptions can unlock performance regimes inaccessible to general-purpose systems.

## 5. Experimental Analysis

To validate the claims that **PrefillOnly** can handle significantly longer contexts and higher throughput than existing engines, the authors conduct a rigorous evaluation across diverse hardware configurations, model sizes, and workload characteristics. The experiments are designed not just to show "better performance," but to isolate the specific contributions of **hybrid prefilling** (memory efficiency) and **continuous JCT calibration** (scheduling efficiency).

### 5.1 Evaluation Methodology

The evaluation strategy moves beyond standard NLP benchmarks (which focus on accuracy) to focus entirely on **system performance metrics** relevant to high-scale serving: throughput (Queries Per Second, QPS), latency (mean and P99), and Maximum Input Length (MIL).

#### Datasets and Workloads
Since existing public datasets focus on model accuracy rather than engine stress-testing, the authors construct two simulated datasets representing distinct discriminative use cases (**Table 1**):

1.  **Post Recommendation (Prefix-Heavy Workload):**
    *   **Scenario:** A social media platform recommending posts based on user history.
    *   **Structure:** 20 users, each with a profile length of **11,000–17,000 tokens** (simulating months of browsing history). For each user, 50 requests are generated (one per candidate post), where the post content is short (**~150 tokens**).
    *   **Goal:** This dataset tests the system's ability to leverage **prefix caching**. Since the user profile is shared across 50 requests, a smart scheduler should recognize the cache hit opportunity for subsequent requests from the same user.
    *   **Scale:** Total of **14,000,000 tokens**.

2.  **Credit Verification (Long-Context Workload):**
    *   **Scenario:** A bank verifying creditworthiness based on transaction history.
    *   **Structure:** 60 users, each with a massive credit history of **40,000–60,000 tokens** (simulating 10 months of data). Each user generates only **1 request**.
    *   **Goal:** This dataset stresses the **Maximum Input Length (MIL)** capabilities. It tests whether the system can handle extreme context lengths on a single GPU without resorting to slow parallelization strategies.
    *   **Scale:** Total of **3,000,000 tokens**.

**Request Arrival:** Requests arrive according to a **Poisson process**, allowing the authors to vary the load (QPS) systematically to observe performance under low, medium, and high contention.

#### Hardware and Model Configurations
The experiments cover a spectrum of hardware to ensure generalizability (**Table 3**):
*   **Low-End:** 2× NVIDIA L4 (24 GB) running **Llama-3.1-8B**.
*   **Mid-Range:** 2× NVIDIA A100 (40 GB) running **Qwen-32B** (FP8 quantized).
*   **High-End:** 2× NVIDIA H100 (80 GB) running **Llama-3.3-70B** (FP8 quantized), tested both with and without **NVLink** high-speed interconnects.

*Note on Routing:* For non-parallelized setups (PrefillOnly, PagedAttention, Chunked Prefill), the authors run one engine instance per GPU and use **user-ID-based routing**. This ensures that all requests from a specific user go to the same GPU, maximizing the potential for prefix cache hits without needing complex distributed cache coherence protocols.

#### Baselines
PrefillOnly is compared against four distinct baselines implemented in the state-of-the-art **vLLM** engine:
1.  **PagedAttention:** The standard vLLM configuration using First-Come-First-Serve (FCFS) scheduling.
2.  **Chunked Prefill:** Processes inputs in small chunks to fit memory but suffers from attention kernel inefficiency.
3.  **Pipeline Parallelism (PP=2):** Splits model layers across two GPUs.
4.  **Tensor Parallelism (TP=2):** Splits model weights across two GPUs, requiring expensive all-reduce communication.

*Crucial Detail:* All baselines have **prefix caching enabled** to ensure a fair comparison. However, as shown in **Table 2**, some baselines (PagedAttention, Chunked Prefill) simply **cannot run** the Credit Verification workload on certain GPUs because their MIL is too low (marked as `WL2: ×`).

### 5.2 Quantitative Results

The experimental results strongly support the paper's core claims, demonstrating substantial gains in both capacity and throughput.

#### Claim 1: PrefillOnly Enables Much Longer Contexts on Single GPUs
The most immediate benefit of **hybrid prefilling** and **suffix discarding** is the ability to handle inputs that crash other systems.

*   **Maximum Input Length (MIL):** As detailed in **Table 2**, PrefillOnly drastically expands the feasible context window.
    *   On an **NVIDIA L4 (24GB)** with Llama-3.1-8B:
        *   Standard PagedAttention fails beyond **24,000 tokens**.
        *   Chunked Prefill reaches **46,000 tokens**.
        *   **PrefillOnly reaches 130,000 tokens**—a **5.4× improvement** over the standard baseline and **2.8×** over chunked prefill.
    *   On an **NVIDIA A100 (40GB)** with Qwen-32B:
        *   Standard PagedAttention caps at **11,000 tokens** (insufficient for the Credit Verification workload).
        *   **PrefillOnly reaches 87,000 tokens**, successfully enabling the long-context workload where the baseline fails completely.

*   **Ablation of Hybrid Prefilling:** **Figure 10** isolates the contribution of the hybrid prefilling technique itself. On a Qwen-2.5-32B model (A100), vanilla vLLM handles ~15k tokens. Adding standard chunking helps slightly but hurts performance. PrefillOnly's specific combination of **chunking + preallocation + in-place computation** pushes the limit to **~130k tokens** (an **8.7× increase** over chunked prefill alone) without the throughput penalty typically associated with chunking.

#### Claim 2: Higher Throughput (QPS) Without Latency Inflation
The primary metric for serving engines is the trade-off between how many requests they can handle (QPS) and how fast they respond (Latency).

*   **Throughput Gains:** Across all hardware setups, PrefillOnly sustains significantly higher QPS before latency spikes.
    *   **Post Recommendation (H100 w/o NVLink):** As shown in **Figure 6(c)**, PrefillOnly maintains low latency up to **~40 QPS**. In contrast, the Tensor Parallel baseline saturates much earlier, and the Chunked Prefill baseline sees latency explode as the prefix cache thrashes.
    *   **Credit Verification (H100):** In **Figure 8**, the authors compare request throughput directly. Even with NVLink enabled (which accelerates Tensor Parallelism), **PrefillOnly achieves the highest throughput**. The Tensor Parallel baseline suffers from "GPU idle time" due to all-reduce communication bubbles, while PrefillOnly keeps the GPU fully utilized on computation.
    *   **Magnitude:** The abstract and **Section 7.2** claim up to **4× higher QPS**. Specifically, in the Post Recommendation scenario on H100s, PrefillOnly handles roughly **4× the load** of the Chunked Prefill baseline at the same latency threshold.

*   **Latency Performance:**
    *   **Average Latency:** **Figure 6** shows that at high QPS, PrefillOnly's average latency is significantly lower than all baselines. This is attributed to the **continuous JCT calibration** preventing cache thrashing.
    *   **P99 Latency (Tail Latency):** A common concern with optimization-focused schedulers is that they might starve long requests, hurting tail latency. **Figure 7** disproves this. PrefillOnly maintains lower P99 latency than baselines across the board.
    *   **Fairness Trade-off:** **Figure 11** analyzes the impact of the fairness hyperparameter $\lambda$.
        *   With $\lambda = 0$ (no fairness adjustment), the system optimizes purely for speed, resulting in excellent average latency but potentially higher P99 for unlucky long requests.
        *   Increasing $\lambda$ to **500** (the default) or **2000** shifts the Cumulative Distribution Function (CDF) curve, significantly improving P99 latency at the cost of a slight increase in average latency. This confirms the mechanism works as intended: it allows operators to tune the system for strict Service Level Objectives (SLOs).

#### Claim 3: Superior Cache Utilization via Continuous JCT Calibration
The Post Recommendation dataset specifically targets the scheduling algorithm.

*   **Cache Throttling:** In **Figure 9**, the throughput of the **Chunked Prefill** baseline drops precipitously as QPS increases. The authors attribute this to **prefix cache throttling**: the naive scheduler admits too many requests that miss the cache, filling the memory with useless suffixes and evicting useful prefixes.
*   **PrefillOnly's Stability:** PrefillOnly maintains high throughput because its **continuous JCT calibration** dynamically identifies requests that *will* hit the cache (e.g., the 2nd, 3rd, ... 50th request for a user) and prioritizes them. This keeps the valuable prefix resident in GPU memory, avoiding the recomputation penalty.
*   **Parallelism Overhead:** While Tensor/Pipeline parallelism avoids cache throttling by having more total memory across GPUs, their throughput is capped by communication overhead. **Figure 8** clearly shows that even with NVLink, the communication cost prevents parallelized baselines from matching PrefillOnly's pure compute efficiency.

### 5.3 Critical Assessment and Limitations

The experiments are robust and convincingly support the paper's claims, but several nuances and conditional results are worth noting.

#### Strengths of the Evaluation
1.  **Realistic Stress Testing:** By simulating workloads with 60k token contexts, the authors expose failure modes (OOM errors) in baselines that standard benchmarks (like ShareGPT) would miss.
2.  **Hardware Diversity:** Testing on L4, A100, and H100 (with and without NVLink) demonstrates that the benefits are not limited to a specific GPU architecture. The fact that PrefillOnly wins even *with* NVLink is a strong indictment of the communication overhead in tensor parallelism for this specific workload.
3.  **Isolation of Variables:** The ablation study in **Figure 10** and the fairness analysis in **Figure 11** provide clear evidence that *both* the memory optimization (hybrid prefilling) and the scheduling logic (JCT calibration) are necessary for the observed gains.

#### Conditional Results and Trade-offs
1.  **Low-QPS Latency Penalty:** **Figure 6** reveals a subtle trade-off. At very **low QPS** (e.g., &lt; 5 QPS on H100), **Tensor Parallelism** sometimes achieves *lower* latency than PrefillOnly.
    *   *Reasoning:* Tensor parallelism splits the compute across two GPUs, reducing the *time-to-first-token* for a single request. PrefillOnly runs on a single GPU, so the raw compute time for one massive request is slightly higher.
    *   *Implication:* PrefillOnly is optimized for **throughput** (high scale), not necessarily for minimizing latency for a single isolated request. However, as soon as load increases (which is the target use case for recommendation systems), PrefillOnly immediately surpasses parallelism.
2.  **Dependency on Prefix Reuse:** The scheduling benefits are most pronounced in the **Post Recommendation** workload where prefix reuse is high (50 requests per user). In the **Credit Verification** workload (1 request per user), the scheduling advantage is less about cache hits and more about avoiding memory fragmentation and OOM errors. The system does not magically create cache hits if the workload has no shared prefixes, though the memory savings from suffix discarding still apply.
3.  **Implementation Complexity:** While the paper notes the implementation is ~4.6k lines of Python on top of vLLM, the reliance on `torch.compile` for hybrid prefilling introduces a dependency on PyTorch's graph compilation stability. The paper does not report compilation overhead times, which could be non-trivial for dynamic workloads, though presumably amortized over long-serving periods.

#### Missing Elements
*   **Energy Efficiency:** The paper focuses on latency and throughput but does not explicitly measure power consumption. Given that PrefillOnly avoids inter-GPU communication (which is energy-intensive) and reduces memory traffic, one could infer better energy efficiency, but this is not quantified.
*   **Multi-Tenant Interference:** The evaluation assumes user-ID routing to single instances. It does not explicitly test a "noisy neighbor" scenario where multiple tenants share a GPU without clean prefix separation, though the JCT calibration is theoretically robust against this.

### Conclusion of Experimental Analysis
The evaluation provides compelling evidence that **PrefillOnly** successfully bridges the gap between LLM capabilities and the efficiency requirements of discriminative tasks. By demonstrating a **5× increase in maximum input length** and up to **4× higher throughput** compared to state-of-the-art parallelized baselines, the experiments validate the core hypothesis: *general-purpose generative engines are fundamentally mismatched for prefill-only workloads.* The results confirm that treating the KV cache as a disposable resource (for suffixes) and leveraging deterministic job lengths for scheduling yields superior performance, particularly in the high-scale, long-context regimes required by modern recommendation and verification systems.

## 6. Limitations and Trade-offs

While **PrefillOnly** demonstrates significant performance gains for discriminative tasks, its design is highly specialized. This specialization introduces specific assumptions, trade-offs, and scenarios where the system may underperform or fail to provide benefits compared to general-purpose engines. Understanding these limitations is crucial for determining when to deploy PrefillOnly versus a standard generative engine.

### 6.1 Strict Workload Assumptions: The "Single-Token" Constraint
The most fundamental limitation of PrefillOnly is its rigid assumption that **every request generates exactly one output token**.
*   **The Assumption:** The entire architecture—from memory management (suffix discarding) to scheduling (deterministic JCT)—relies on the inference process terminating immediately after the prefill phase. As stated in **Section 1**, the system is designed for workloads where "the LLM generates only a single output token."
*   **The Consequence:** If a application requires even a short chain of thought (e.g., generating a 5-token explanation before a "Yes/No" label) or multi-step reasoning, PrefillOnly's optimizations become detrimental.
    *   **Suffix Discarding Failure:** The **suffix KV cache discarding** mechanism (**Section 5.1**) aggressively deletes the KV state of the input sequence after the first token is generated. In a multi-token generation scenario, this would force the engine to recompute the entire context for the *second* token, resulting in catastrophic latency penalties (quadratic recomputation) compared to standard engines that retain the cache.
    *   **Scheduling Mismatch:** The **continuous JCT calibration** (**Section 6.3**) calculates job completion time based on the formula $JCT \propto (n_{\text{input}} - n_{\text{cached}})$. This formula assumes the decoding phase is negligible (1 token). If the output length becomes stochastic (e.g., 10–100 tokens), the JCT estimate becomes inaccurate, potentially breaking the fairness guarantees and efficiency of the Shortest Remaining Job First (SRJF) scheduler.
*   **Mitigation Gap:** The paper explicitly notes in **Section 9** that extending PrefillOnly to handle mixed workloads (prefill + decode) is left for future work. Currently, it cannot natively support applications that blend discriminative and generative tasks without significant architectural re-engineering.

### 6.2 The Low-Load Latency Penalty
Contrary to the intuition that "optimization always equals lower latency," PrefillOnly exhibits a **latency disadvantage under low query-per-second (QPS)** conditions when compared to parallelized baselines.
*   **The Evidence:** In **Figure 6** (specifically panels d and h for H100 with NVLink), the **Tensor Parallelism** baseline achieves lower average latency than PrefillOnly at very low loads (e.g., &lt; 5 QPS).
*   **The Reasoning:**
    *   **Parallelism Advantage:** Tensor parallelism splits the model weights and computation across multiple GPUs. For a *single* isolated request, this reduces the wall-clock time of the forward pass because two GPUs are working simultaneously on the matrix multiplications.
    *   **PrefillOnly Constraint:** PrefillOnly intentionally avoids inter-GPU communication to maximize throughput. It runs the entire model on a **single GPU** per instance (**Section 7.1**). Therefore, for a single request, it cannot leverage the aggregate compute power of multiple GPUs.
*   **The Trade-off:** PrefillOnly optimizes for **throughput** (total requests handled per second) at the expense of **single-request latency** in low-concurrency scenarios. The communication overhead of tensor parallelism only becomes a bottleneck when the system is saturated; under light load, that overhead is negligible, and the compute speedup dominates.
*   **Implication:** Organizations with low-traffic discriminative applications (e.g., a credit check system used only a few times an hour) might experience *slower* response times with PrefillOnly than with a standard tensor-parallel setup. The benefits of PrefillOnly only materialize when the system is under moderate-to-high load, where its ability to avoid communication bubbles allows it to sustain much higher QPS.

### 6.3 Dependency on Prefix Reuse for Scheduling Gains
While the **memory efficiency** of PrefillOnly (via hybrid prefilling) benefits *all* long-context requests, its **scheduling advantages** are heavily dependent on the presence of **prefix reuse**.
*   **The Mechanism:** The **continuous JCT calibration** (**Section 6.3**) improves performance by dynamically re-ranking requests that share a prefix with a recently executed request. As shown in **Figure 5**, this allows the system to "strike while the iron is hot," executing Request D immediately after Request A if they share a prefix, thus avoiding cache eviction.
*   **The Limitation:** In workloads where prefixes are unique for every request (low cache hit potential), the dynamic re-ranking provides minimal benefit over a static SRJF scheduler.
    *   **Evidence:** The **Credit Verification** dataset (**Table 1**) consists of 60 users with 1 request each. In this scenario, there is no opportunity for prefix reuse between different users. While PrefillOnly still wins here due to its ability to handle the long context without parallelism (**Section 7.2**), the *scheduling algorithm itself* does not yield the same relative improvement seen in the **Post Recommendation** dataset (where 50 requests share the same user profile).
*   **Edge Case:** If a workload has high variance in input lengths but zero prefix overlap, the complexity of continuously recalculating JCT for every waiting request (Algorithm 1, **Section 6.3**) adds scheduler overhead without the corresponding payoff of increased cache hits. The paper does not quantify this scheduler overhead in isolation, but it is a non-zero cost.

### 6.4 Implementation Constraints and Compilation Overhead
PrefillOnly relies on specific software features that introduce potential deployment constraints and unquantified overheads.
*   **Reliance on `torch.compile`:** The **hybrid prefilling** mechanism is implemented by rewriting the computation graph using PyTorch's `torch.compile` (**Section 4.3**).
    *   **Compilation Latency:** Graph compilation is not instantaneous. While the paper mentions implementing the logic in Python, it does not report the **compilation time** or the "cold start" latency when a new model or input shape is encountered. In dynamic serving environments where models might change or input shapes vary wildly, this compilation overhead could introduce jitter or initial latency spikes.
    *   **Graph Stability:** `torch.compile` can sometimes struggle with highly dynamic control flows or specific custom kernels. If the underlying LLM architecture uses non-standard layers that `torch.compile` cannot efficiently fuse or chunk, the hybrid prefilling optimization might degrade or fail, reverting to slower paths.
*   **Fairness Hyperparameter Sensitivity:** The system requires tuning the fairness parameter $\lambda$ to balance average latency and P99 latency (**Figure 11**).
    *   **Operational Burden:** While the default $\lambda = 500$ works well in the evaluated traces, the optimal value depends on the specific distribution of request lengths and arrival rates. In a production environment with shifting traffic patterns, a static $\lambda$ might lead to suboptimal fairness (starvation of long requests) or unnecessary latency inflation. The paper acknowledges this trade-off but does not propose an adaptive mechanism to auto-tune $\lambda$.

### 6.5 Discarded Computation and Offloading Gaps
The current implementation makes a hard choice to **discard** suffix KV caches rather than offload them, which limits future flexibility.
*   **Lost Reuse Opportunities:** By discarding the suffix KV cache (**Section 5.1**), PrefillOnly permanently loses the computation associated with those tokens.
    *   **Scenario:** Consider a scenario where a user's session evolves. If a request comes in later that matches the *suffix* of a previous request (but not the prefix), PrefillOnly cannot leverage any cached state, whereas a system with larger memory or CPU offloading might have retained it.
*   **Offloading as Future Work:** The paper explicitly states in **Section 9** that offloading KV caches to CPU memory (e.g., via LMCache) is left for future work.
    *   **Current Limitation:** Without CPU offloading, PrefillOnly is strictly bound by the **GPU memory capacity** for its prefix cache pool. If the working set of unique prefixes exceeds the GPU memory reserved after the profile run (**Section 3.1**), the system will thrash, evicting useful prefixes just like any other engine. It solves the *active* memory spike during computation, but not the *capacity* limit for storing diverse prefixes.

### 6.6 Summary of Trade-offs

| Feature | Benefit | Trade-off / Limitation |
| :--- | :--- | :--- |
| **Single-Token Focus** | Enables aggressive suffix discarding & deterministic scheduling. | **Fails completely** for multi-token generation or chain-of-thought tasks. |
| **No Inter-GPU Parallelism** | Eliminates communication overhead; maximizes throughput at scale. | **Higher latency** for single/isolated requests compared to Tensor Parallelism (**Figure 6**). |
| **Continuous JCT Calibration** | Maximizes prefix cache hits in high-reuse workloads. | **Overhead with no gain** in zero-reuse workloads; requires tuning of $\lambda$ for fairness. |
| **`torch.compile` Implementation** | Allows flexible graph rewriting without kernel changes. | **Unquantified compilation overhead**; potential compatibility issues with non-standard layers. |
| **Suffix Discarding** | Drastically reduces active memory footprint. | **Permanently loses computation**; no CPU offloading support yet (**Section 9**). |

In conclusion, **PrefillOnly** is not a universal replacement for general-purpose LLM engines. It is a specialized tool optimized for a specific emerging niche: **high-throughput, long-context, single-token discriminative tasks with significant prefix reuse**. Deploying it outside this narrow band—such as for generative chat, low-traffic APIs, or workloads with unique prefixes for every request—would likely result in performance degradation or functional failure.

## 7. Implications and Future Directions

The introduction of **PrefillOnly** marks a pivotal shift in the architecture of LLM serving systems, challenging the long-held assumption that inference engines must be universally optimized for arbitrary-length generation. By demonstrating that discriminative tasks constitute a distinct workload class with unique memory and scheduling properties, this work forces a re-evaluation of how we deploy LLMs in production. The implications extend beyond immediate performance gains, reshaping the economic and architectural landscape of AI infrastructure while opening new avenues for specialized system research.

### 7.1 Reshaping the LLM Serving Landscape

**From "One-Size-Fits-All" to Specialized Runtimes**
For the past decade, the dominant paradigm in deep learning serving has been the development of general-purpose engines (e.g., TensorFlow Serving, Triton, vLLM) capable of handling diverse model architectures and output lengths. **PrefillOnly** disrupts this by proving that **specialization yields order-of-magnitude gains**.
*   **The Paradigm Shift:** The paper demonstrates that treating the KV cache as a disposable resource for suffix tokens—heresy in generative contexts—is the key to unlocking long-context efficiency. This suggests that the future of LLM infrastructure may not be a single monolithic engine, but a **polyglot serving stack** where requests are routed to specialized runtimes based on their workload profile (generative vs. discriminative).
*   **Economic Impact:** The ability to handle **60,000+ token contexts on a single NVIDIA L4 GPU** (**Table 2**) fundamentally alters the cost structure of LLM applications. Previously, such contexts required multi-GPU tensor parallelism (e.g., 2–4 H100s), incurring high hardware costs and complex networking requirements. PrefillOnly enables organizations to replace expensive, interconnected GPU clusters with commodity, single-GPU instances, potentially reducing infrastructure costs by **10×** for recommendation and verification pipelines.

**Redefining the Role of Parallelism**
The evaluation results (**Figure 8**) deliver a counter-intuitive but critical lesson: **parallelism is not always the answer to scale**.
*   **The Communication Bottleneck:** In high-throughput prefill-only workloads, the communication overhead of tensor parallelism (all-reduce operations) creates "bubbles" of idle GPU time that outweigh the benefits of distributed computation. PrefillOnly shows that for single-token outputs, **vertical scaling** (optimizing memory usage on one GPU) is superior to **horizontal scaling** (splitting the model across GPUs).
*   **Architectural Consequence:** This finding suggests that future data center designs for discriminative AI might prioritize **high-density single-GPU servers** over tightly coupled multi-GPU nodes, simplifying network topology and cooling requirements.

### 7.2 Enabling Follow-Up Research

The success of PrefillOnly highlights several under-explored areas in systems research, providing a clear roadmap for future innovation:

**1. Hybrid Workload Orchestration**
The most immediate open question is how to serve **mixed workloads** (generative chat + discriminative classification) on the same cluster.
*   **Research Direction:** Future systems could implement a **dynamic dispatcher** that routes single-token requests to a PrefillOnly-like engine and multi-token requests to a standard decoder-optimized engine (like vLLM).
*   **Challenge:** This requires solving the **resource fragmentation problem**: how to partition GPU memory between a "prefix-cache-heavy" pool (for discriminative tasks) and a "KV-cache-retentive" pool (for generative tasks) without starving either workload. The **profile run** mechanism in PrefillOnly (**Section 3.1**) could be extended to dynamically adjust these partitions based on real-time traffic mix.

**2. CPU Offloading and Hierarchical Caching**
PrefillOnly currently **discards** suffix KV caches (**Section 5.1**), permanently losing that computation. The authors note in **Section 9** that offloading to CPU is a natural next step.
*   **Research Direction:** Integrating **hierarchical caching** (GPU L1 → CPU RAM L2 → Disk L3) specifically for discriminative workloads. Since the access pattern for suffixes is predictable (rarely reused immediately but potentially useful for session continuity), aggressive compression and CPU offloading could further expand the effective context window beyond GPU limits.
*   **Connection to Prior Work:** This builds on systems like **LMCache** [33] but tailors the eviction policy to the "single-token termination" property, potentially allowing for **infinite context** sessions where only the active prefix resides in GPU memory.

**3. Compiler-Driven Memory Optimization**
The implementation of **hybrid prefilling** via `torch.compile` (**Section 4.3**) demonstrates the power of graph-level rewriting over kernel-level hacking.
*   **Research Direction:** This invites research into **automatic layer analysis** where compilers statically identify "memory-heavy" layers (like MLPs with large expansion factors) and automatically apply chunking, while leaving "compute-sensitive" layers (like Attention) intact.
*   **Generalization:** Such techniques could benefit not just LLMs, but other transformer-based models (e.g., Vision Transformers) where intermediate activation spikes limit batch sizes, suggesting a broader applicability of the "hybrid execution" concept.

**4. Adaptive Fairness Mechanisms**
The reliance on a static fairness hyperparameter $\lambda$ (**Section 6.3**) to prevent starvation is a pragmatic but imperfect solution.
*   **Research Direction:** Developing **reinforcement learning-based schedulers** that dynamically adjust $\lambda$ based on observed queue depths and latency SLOs. Instead of a fixed trade-off, the system could learn to prioritize cache hits during low-load periods and enforce strict fairness during traffic spikes, optimizing for a composite utility function rather than a static rule.

### 7.3 Practical Applications and Downstream Use Cases

The capabilities unlocked by PrefillOnly directly address bottlenecks in several high-value industrial applications:

*   **Real-Time Personalized Recommendation:**
    *   **Current Limitation:** Modern recommenders often truncate user history to fit within 4k–8k token limits to avoid OOM errors or latency spikes.
    *   **PrefillOnly Impact:** With support for **130,000+ tokens** on commodity hardware, systems can ingest a user's **entire browsing history** (months of data) as context. This enables truly long-term personalization, where the model considers behavior from weeks ago rather than just the last few clicks, significantly improving recommendation relevance.
    *   **Scale:** The **4× throughput gain** allows social media platforms to score **tens of thousands of candidates per user** in real-time, enabling finer-grained ranking without resorting to cheap heuristic filters.

*   **Automated Credit and Fraud Verification:**
    *   **Current Limitation:** Financial institutions often rely on separate, specialized models for different data sources (transaction logs, credit bureau reports, employment history) because combining them exceeds context limits.
    *   **PrefillOnly Impact:** The ability to process **40,000–60,000 token** credit histories (**Table 1**) in a single forward pass allows for **holistic risk assessment**. A single LLM prompt can reason over the correlation between a transaction from six months ago and a recent address change, detecting complex fraud patterns that siloed models miss.
    *   **Cost Efficiency:** Banks can run these heavy analyses on **NVIDIA L4** instances rather than expensive H100 clusters, making comprehensive AI-driven verification economically viable for smaller transactions.

*   **Large-Scale Data Labeling and Moderation:**
    *   **Current Limitation:** Content moderation often requires analyzing long threads or document chains. Current engines struggle to batch these long inputs efficiently.
    *   **PrefillOnly Impact:** The **suffix discarding** mechanism allows moderation systems to process massive volumes of long-form content (e.g., legal documents, medical records) with minimal memory footprint. The deterministic scheduling ensures predictable latency, which is critical for meeting regulatory SLAs in finance and healthcare.

### 7.4 Reproducibility and Integration Guidance

For practitioners considering the adoption of PrefillOnly or similar techniques, the following guidance outlines when and how to integrate this approach:

**When to Prefer PrefillOnly:**
*   **Workload Profile:** Your application generates **exactly one output token** (or a fixed, very small number) per request. Examples: Binary classification (Yes/No), multiple-choice selection, or log-probability scoring.
*   **Context Length:** Your input prompts are **long (>8k tokens)** and approach the memory limits of standard engines.
*   **Prefix Reuse:** Your workload exhibits **high prefix redundancy** (e.g., same user profile queried against multiple items, same system prompt used for many documents). The **continuous JCT calibration** provides maximal benefit here.
*   **Throughput Priority:** Your primary metric is **Queries Per Second (QPS)** at scale, rather than minimizing latency for a single isolated request.

**When to Stick with Standard Engines (e.g., vLLM, TGI):**
*   **Generative Tasks:** If your application requires generating paragraphs, code, or chain-of-thought reasoning (>10 tokens), PrefillOnly's suffix discarding will cause **catastrophic recomputation** for every subsequent token.
*   **Low-Traffic/Low-Latency Sensitivity:** If your system operates at very low QPS (&lt;5 requests/sec) and requires the absolute lowest latency for individual requests, **Tensor Parallelism** on high-end GPUs may still outperform PrefillOnly due to raw compute splitting (**Figure 6**).
*   **Zero Prefix Reuse:** If every request has a completely unique prefix (no shared context), the scheduling advantages of PrefillOnly diminish, though the memory savings from hybrid prefilling still apply.

**Integration Steps:**
1.  **Codebase:** PrefillOnly is implemented as a fork of **vLLM** with ~4.6k lines of Python. Integration requires adopting their modified scheduler and enabling the `torch.compile` based hybrid prefilling graph.
2.  **Configuration:**
    *   Set the **Maximum Input Length (MIL)** based on your worst-case scenario.
    *   Run the **profile phase** to determine the memory split between active computation and prefix cache.
    *   Tune the fairness parameter $\lambda$ (default 500) based on your P99 latency requirements vs. average throughput goals (**Figure 11**).
3.  **Routing:** Implement **user-ID-based routing** at the load balancer level to ensure requests with shared prefixes land on the same GPU instance, maximizing the efficacy of the prefix cache.

In summary, **PrefillOnly** does not merely optimize an existing process; it validates a new architectural primitive for the AI era. As LLMs continue to migrate from novelty chatbots to the backbone of industrial decision-making, the shift toward **workload-specific engines** exemplified by this paper will likely become the standard for efficient, scalable deployment.
